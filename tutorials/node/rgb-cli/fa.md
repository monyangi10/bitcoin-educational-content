در این آموزش، ما فرآیند گام‌به‌گام نوشتن یک Contract را با استفاده از ابزار خط فرمان `RGB` که توسط انجمن LNP/BP ایجاد شده است، دنبال خواهیم کرد. هدف این است که نشان دهیم چگونه CLI را نصب و مدیریت کنیم، یک Schema را کامپایل کنیم، Interface و Interface Implementation را وارد کنیم و سپس یک دارایی RGB صادر کنیم. همچنین به منطق زیرین، از جمله کامپایل و اعتبارسنجی وضعیت، خواهیم پرداخت. در پایان این آموزش، شما باید قادر باشید این فرآیند را بازتولید کرده و قراردادهای RGB خود را ایجاد کنید.


## یادآوری پروتکل RGB


RGB پروتکلی است که بر روی Bitcoin اجرا می‌شود و عملکرد Smart contract و مدیریت دارایی دیجیتال را شبیه‌سازی می‌کند، بدون اینکه Blockchain که بر اساس آن است را بارگذاری کند. برخلاف قراردادهای هوشمند متعارف On-Chain (مانند اتریوم، به عنوان مثال)، RGB به یک سیستم "*Client-side Validation*" متکی است: اکثر داده‌ها و تاریخچه وضعیت‌ها به‌طور انحصاری توسط شرکت‌کنندگان درگیر مبادله و ذخیره می‌شوند، در حالی که Bitcoin Blockchain تنها تعهدات رمزنگاری کوچک را میزبانی می‌کند (از طریق مکانیزم‌هایی مانند *Tapret* یا *Opret*). در پروتکل RGB، بنابراین Bitcoin Blockchain تنها به عنوان یک سرور زمان‌سنجی و سیستم حفاظت Double-spending عمل می‌کند.


یک RGB Contract به صورت یک ماشین حالت تکاملی ساختار یافته است. این فرآیند با یک Genesis آغاز می‌شود که حالت اولیه را تعریف می‌کند (به عنوان مثال، توصیف Supply، تیکر یا دیگر متادیتا) بر اساس یک Schema که به صورت دقیق تایپ و کامپایل شده است. انتقالات حالت و در صورت لزوم، گسترش‌های حالت برای تغییر یا گسترش این حالت اعمال می‌شوند. هر عملیات، چه انتقال دارایی‌های قابل تعویض (RGB20) و چه ایجاد دارایی‌های منحصر به فرد (RGB21)، شامل *مهر و موم‌های یک‌بار مصرف* است. این‌ها UTXOهای Bitcoin را به حالت‌های off-chain متصل می‌کنند و از دوبار خرج کردن جلوگیری می‌کنند، در حالی که محرمانگی و مقیاس‌پذیری را تضمین می‌کنند.


برای کسب اطلاعات بیشتر در مورد نحوه عملکرد پروتکل RGB، توصیه می‌کنم این دوره آموزشی جامع را بگذرانید:


https://planb.academy/courses/3ce1d37c-05ba-4f54-aa15-7586d37b2bb7

منطق داخلی RGB بر اساس کتابخانه‌های Rust است که شما به عنوان توسعه‌دهندگان می‌توانید آن‌ها را به پروژه‌های خود وارد کنید تا بخش *Client-side Validation* را مدیریت کنید. علاوه بر این، تیم LNP/BP در حال کار بر روی بایندینگ‌ها برای زبان‌های دیگر است، اما این هنوز نهایی نشده است. علاوه بر این، نهادهای دیگری مانند Bitfinex در حال توسعه استک‌های یکپارچه‌سازی خود هستند، اما در یک آموزش دیگر در مورد این‌ها صحبت خواهیم کرد. در حال حاضر، `RGB` CLI مرجع رسمی است، حتی اگر نسبتاً ناپخته باقی بماند.


## نصب و ارائه ابزار RGB CLI


دستور اصلی به سادگی `RGB` نامیده می‌شود. این طراحی به گونه‌ای است که یادآور `git` باشد، با مجموعه‌ای از زیر-دستورات برای دستکاری قراردادها، فراخوانی آن‌ها، صدور دارایی‌ها و غیره. Bitcoin Wallet در حال حاضر یکپارچه نشده است، اما در نسخه‌ای قریب‌الوقوع (0.11) خواهد بود. این نسخه بعدی به کاربران امکان می‌دهد تا کیف پول‌های خود را (از طریق توصیف‌گرها) مستقیماً از `RGB` ایجاد و مدیریت کنند، از جمله تولید PSBT، سازگاری با سخت‌افزار خارجی (مثلاً یک Hardware Wallet) برای امضا، و تعامل با نرم‌افزارهایی مانند Sparrow. این امر سناریوی صدور و انتقال دارایی را به‌طور کامل ساده‌تر خواهد کرد.


### نصب از طریق Cargo


ما ابزار را در Rust نصب می‌کنیم با:


```bash
cargo install rgb-contracts --all-features
```


(توجه: جعبه به نام `RGB-contracts` نامیده می‌شود و فرمان نصب شده به نام `RGB` خواهد بود. اگر جعبه‌ای به نام `RGB` از قبل وجود داشت، ممکن بود تداخلی رخ دهد، به همین دلیل این نام انتخاب شده است)


نصب تعداد زیادی وابستگی را کامپایل می‌کند (مانند تجزیه فرمان، یکپارچه‌سازی Electrum، مدیریت اثبات‌های دانش صفر و غیره).


پس از اتمام نصب، :


```bash
rgb
```


اجرای `RGB` (بدون آرگومان) لیستی از زیر-دستورات موجود را نمایش می‌دهد، مانند `interfaces`، `Schema`، `import`، `export`، `issue`، `Invoice`، `transfer` و غیره. شما می‌توانید دایرکتوری ذخیره‌سازی محلی را تغییر دهید (یک Stash که شامل تمام لاگ‌ها، شماتیک‌ها و پیاده‌سازی‌ها است)، شبکه را انتخاب کنید (Testnet، Mainnet) یا سرور Electrum خود را پیکربندی کنید.


![RGB-CLI](assets/fr/01.webp)


### اولین مرور کنترل‌ها


وقتی فرمان زیر را اجرا می‌کنید، خواهید دید که یک `RGB20` Interface به‌طور پیش‌فرض از قبل یکپارچه شده است:


```bash
rgb interfaces
```


اگر این Interface یکپارچه نشده است، کلون کنید:


```bash
git clone https://github.com/RGB-WG/rgb-interfaces
```


کامپایلش کن:


```bash
cargo run
```


سپس Interface مورد نظر خود را وارد کنید:


```bash
rgb import interfaces/RGB20.rgb
```


![RGB-CLI](assets/fr/02.webp)


با این حال، به ما گفته شده است که هنوز هیچ Schema به نرم‌افزار وارد نشده است. همچنین هیچ Contract در Stash وجود ندارد. برای دیدن آن، فرمان زیر را اجرا کنید:


```bash
rgb schemata
```


سپس می‌توانید مخزن را کلون کنید تا برخی از شماتیک‌ها را بازیابی کنید:


```bash
git clone https://github.com/RGB-WG/rgb-schemata
```


![RGB-CLI](assets/fr/03.webp)


این مخزن شامل چندین فایل Rust در دایرکتوری `src/` خود است (برای مثال `nia.rs`) که طرح‌واره‌ها را تعریف می‌کنند (NIA برای "*دارایی غیرقابل تورم*", UDA برای "*دارایی دیجیتال منحصر به فرد*" و غیره). برای کامپایل، می‌توانید سپس اجرا کنید:


```bash
cd rgb-schemata
cargo run
```


این چندین فایل `.RGB` و `.rgba` تولید می‌کند که با طرح‌های کامپایل شده مطابقت دارند. به عنوان مثال، شما `NonInflatableAsset.RGB` را پیدا خواهید کرد.

## ایجاد Contract (صدور)


دو رویکرد برای ایجاد یک دارایی جدید وجود دارد:




- یا از یک اسکریپت یا کدی در Rust استفاده می‌کنیم که با پر کردن فیلدهای Schema (Global State، Owned States، و غیره) یک Contract می‌سازد و یک فایل `.RGB` یا `.rgba` تولید می‌کند؛
- یا از زیر فرمان `issue` به طور مستقیم استفاده کنید، با یک فایل YAML (یا TOML) که ویژگی‌های token را توصیف می‌کند.


می‌توانید مثال‌ها را در Rust در پوشه `examples` پیدا کنید، که نشان می‌دهند چگونه یک `ContractBuilder` بسازید، `Global State` (نام دارایی، نماد، Supply، تاریخ، و غیره) را پر کنید، Owned State را تعریف کنید (که به کدام UTXO اختصاص داده شده است)، سپس همه این‌ها را به یک *Contract Consignment* که می‌توانید صادر کنید، اعتبارسنجی کنید و به یک Stash وارد کنید، کامپایل کنید.


راه دیگر این است که به صورت دستی یک فایل YAML را ویرایش کنید تا `ticker`، `name`، `Supply` و غیره را سفارشی کنید. فرض کنید فایل `RGB20-demo.yaml` نام دارد. می‌توانید مشخص کنید:




- `spec`: نماد، نام، دقت؛
- `terms`: یک فیلد برای اطلاعیه‌های قانونی ;
- `issuedSupply` : مقدار token صادر شده ؛
- `assignments`: نشان‌دهنده Single-Use Seal (*Seal Definition*) و مقدار آزاد شده است.


در اینجا یک مثال از یک فایل YAML برای ایجاد آمده است:


```yaml
interface: RGB20Fixed
globals:
spec:
ticker: Plan ₿ Academy
name: Plan ₿ Academy
details: "Pay attention: the asset has no value"
precision: 2
terms:
text: >
SUBJECT TO, AND WITHOUT IN ANY WAY LIMITING, THE REPRESENTATIONS AND WARRANTIES OF ANY SELLER. PROPERTY IS BEING SOLD “AS IS”...
media: ~
issuedSupply: 100000000
assignments:
assetOwner:
seal: tapret1st:b449f7eaa3f98c145b27ad0eeb7b5679ceb567faef7a52479bc995792b65f804:1
amount: 100000000 # this is 1 million (we have two digits for cents)
```


![RGB-CLI](assets/fr/05.webp)


سپس به سادگی فرمان را اجرا کنید :


```bash
rgb issue '<SchemaID>' ssi:<Issuer> rgb20-demo.yaml
```


![RGB-CLI](assets/fr/06.webp)


در مورد من، شناسه منحصر به فرد Schema (که باید در گیومه تک قرار گیرد) `RDYhMTR!9gv8Y2GLv9UNBEK1hcrCmdLDFk9Qd5fnO8k` است و من هیچ صادرکننده‌ای قرار نداده‌ام. بنابراین سفارش من به این صورت است:


```txt
rgb issue 'RDYhMTR!9gv8Y2GLv9UNBEK1hcrCmdLDFk9Qd5fnO8k' ssi:anonymous rgb20-demo.yaml
```


اگر شناسه Schema را نمی‌دانید، فرمان زیر را اجرا کنید:


```bash
rgb schemata
```


CLI پاسخ می‌دهد که یک Contract جدید صادر شده و به Stash اضافه شده است. اگر فرمان زیر را تایپ کنیم، می‌بینیم که اکنون یک Contract اضافی وجود دارد که مربوط به همان است که تازه صادر شده است:


```bash
rgb contracts
```


![RGB-CLI](assets/fr/07.webp)


سپس، فرمان بعدی وضعیت‌های جهانی (نام، نماد، Supply...) و فهرست وضعیت‌های مالکیت‌شده، یعنی تخصیص‌ها (برای مثال، 1 میلیون توکن `Plan ₿ Academy` تعریف‌شده در UTXO `b449f7eaa3f98c145b27ad0eeb7b5679ceb567faef7a52479bc995792b65f804:1`) را نمایش می‌دهد.


```bash

```

## صادرات، واردات و اعتبارسنجی


برای به اشتراک‌گذاری این Contract با کاربران دیگر، می‌توان آن را از Stash به یک :


```bash
rgb export '<ContractId>' myContractPBN.rgb
```


![RGB-CLI](assets/fr/09.webp)


فایل `myContractPBN.RGB` می‌تواند به کاربر دیگری منتقل شود، که می‌تواند آن را با فرمان زیر به Stash خود اضافه کند:


```bash
rgb import myContractPBN.rgb
```


در هنگام واردات، اگر یک *Contract Consignment* ساده باشد، پیغام "`Importing Consignment RGB`" را دریافت خواهیم کرد. اگر یک *State Transition Consignment* بزرگتر باشد، دستور متفاوت خواهد بود (`RGB accept`).


برای اطمینان از اعتبار، می‌توانید از تابع اعتبارسنجی محلی نیز استفاده کنید. به عنوان مثال، می‌توانید اجرا کنید:


```bash
rgb validate myContract.rgb
```

## انتقال و PSBT


برای انجام انتقال، باید یک Bitcoin Wallet محلی را برای مدیریت تعهدات `Tapret` یا `Opret` دستکاری کنید.


### generate و Invoice


در اکثر موارد، تعامل بین شرکت‌کنندگان در یک Contract (مثلاً Alice و Bob) از طریق تولید یک Invoice صورت می‌گیرد. اگر Alice بخواهد Bob چیزی را اجرا کند (انتقال یک token، صدور مجدد، یک اقدام در یک DAO، و غیره)، Alice یک Invoice ایجاد می‌کند که دستورالعمل‌های خود را به Bob جزئیات می‌دهد. بنابراین ما داریم:




- **Alice** (صادرکننده Invoice) ;
- **Bob** (که Invoice را دریافت و اجرا می‌کند).


برخلاف سایر اکوسیستم‌ها، یک RGB Invoice به مفهوم پرداخت محدود نمی‌شود. می‌تواند هر درخواستی که به Contract مرتبط است را در بر گیرد: لغو یک کلید، رأی‌گیری، ایجاد یک حکاکی (*حکاکی*) بر روی یک NFT، و غیره. عملیات مربوطه می‌تواند در Contract Interface توصیف شود. عملیات مربوطه می‌تواند در Contract Interface توصیف شود.


دستور زیر یک RGB Invoice تولید می‌کند:


```bash
$ rgb invoice $CONTRACT -i $INTERFACE $ACTION $STATE $SEAL
```


با:




- `$Contract`: شناسه Contract (*ContractId*) ;
- `$Interface`: Interface مورد استفاده (مثلاً `RGB20`) ;
- `$ACTION`: نام عملیات مشخص شده در Interface (برای انتقال ساده قابل تعویض token، می‌تواند "انتقال" باشد). اگر Interface از قبل یک عملیات پیش‌فرض ارائه می‌دهد، نیازی نیست که دوباره آن را اینجا وارد کنید؛
- `$STATE`: داده وضعیت برای انتقال (برای مثال، مقدار توکن‌ها اگر یک token قابل تعویض منتقل شود)؛
- `$Seal`: Single-Use Seal گیرنده (Alice)، یعنی یک ارجاع صریح به UTXO. Bob از این اطلاعات برای ساخت Witness Transaction استفاده خواهد کرد و خروجی مربوطه سپس به Alice تعلق خواهد داشت (در *blinded UTXO* یا به صورت رمزنگاری نشده).


به عنوان مثال، با دستورات زیر


```bash
alice$ CONTRACT='iZgIN9EL-2H21UgQ-x!A3uJc-WwXhCSm-$9Lwcc1-v!mUkKY'
alice$ MY_UTXO=4960acc21c175c551af84114541eace09c14d3a1bb184809f7b80916f57f9ef8:1
alice$ rgb invoice $CONTRACT -i RGB20 --amount 100 $MY_UTXO
```


CLI به generate و Invoice مانند :


```bash
rgb:iZgIN9EL-2H21UgQ-x!A3uJc-WwXhCSm-$9Lwcc1-v!mUkKY/RGB20/100+utxob:zlVS28Rb-...
```


می‌توان آن را از طریق هر کانالی (متن، کد QR، و غیره) به Bob منتقل کرد.


### انجام انتقال


برای انتقال از این Invoice :




- Bob (که توکن‌ها را در Stash خود نگه می‌دارد) دارای یک Bitcoin Wallet است. او نیاز دارد یک تراکنش Bitcoin (به صورت یک PSBT، مثلاً `tx.PSBT`) آماده کند که UTXOهایی را که توکن‌های مورد نیاز RGB در آن‌ها قرار دارند، به علاوه یک UTXO برای ارز (Exchange) خرج کند؛
- Bob فرمان زیر را اجرا می‌کند:


```bash
bob$ rgb transfer tx.psbt $INVOICE consignment.rgb
```




- این یک فایل `Consignment.RGB` تولید می‌کند که شامل :
 - تاریخچه انتقال که به Alice اثبات می‌کند که توکن‌ها معتبر هستند؛
 - انتقال جدیدی که توکن‌ها را به Alice's Single-Use Seal منتقل می‌کند؛
 - یک Witness Transaction (امضا نشده).
- Bob این فایل `Consignment.RGB` را به Alice ارسال می‌کند (از طریق ایمیل، یک سرور اشتراک‌گذاری یا یک پروتکل RGB-RPC، Storm، و غیره)؛
- Alice دریافت می‌کند `Consignment.RGB` و آن را در Stash خود می‌پذیرد:


```bash
alice$ rgb accept consignment.rgb
```




- CLI اعتبار انتقال را بررسی می‌کند و آن را به Stash از Alice اضافه می‌کند. اگر نامعتبر باشد، فرمان با پیام‌های خطای دقیق شکست می‌خورد. در غیر این صورت، موفق می‌شود و گزارش می‌دهد که تراکنش نمونه هنوز در شبکه Bitcoin پخش نشده است (Bob منتظر نور Green از Alice است)؛
- به عنوان تأیید، فرمان `accept` یک امضا (*فیش حقوقی*) را برمی‌گرداند که Alice می‌تواند به Bob ارسال کند تا به او نشان دهد که او *Consignment* را تأیید کرده است؛
- سپس Bob می‌تواند تراکنش Bitcoin خود را امضا و منتشر کند (`--publish`):


```bash
bob$ rgb check <sig> && wallet sign --publish tx.psbt
```
