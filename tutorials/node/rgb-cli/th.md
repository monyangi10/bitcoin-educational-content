ในบทแนะนำนี้ เราจะทำตามขั้นตอนการเขียนสัญญาโดยใช้เครื่องมือบรรทัดคำสั่ง `rgb` ที่สร้างโดยสมาคม LNP/BP วัตถุประสงค์คือเพื่อแสดงวิธีการติดตั้งและจัดการ CLI, คอมไพล์ Schema, นำเข้า Interface และ Interface Implementation, และจากนั้นออกสินทรัพย์ RGB เราจะดูที่ตรรกะพื้นฐาน รวมถึงการคอมไพล์และการตรวจสอบสถานะ ในตอนท้ายของบทแนะนำนี้ คุณควรจะสามารถทำซ้ำกระบวนการและสร้างสัญญา RGB ของคุณเองได้


## การแจ้งเตือนโปรโตคอล RGB


RGB เป็นโปรโตคอลที่ทำงานบน Bitcoin และจำลองการทำงานของสมาร์ทคอนแทรคและการจัดการสินทรัพย์ดิจิทัล โดยไม่ทำให้บล็อกเชนที่เป็นฐานนั้นมีภาระมากเกินไป แตกต่างจากสมาร์ทคอนแทรค on-chain แบบดั้งเดิม (เช่น บน Ethereum) RGB พึ่งพาระบบ "*การตรวจสอบฝั่งไคลเอนต์*" ซึ่งข้อมูลและประวัติสถานะส่วนใหญ่จะถูกแลกเปลี่ยนและจัดเก็บโดยผู้เข้าร่วมที่เกี่ยวข้องเท่านั้น ในขณะที่บล็อกเชน Bitcoin จะเก็บเฉพาะข้อผูกพันทางการเข้ารหัสขนาดเล็ก (ผ่านกลไกเช่น *Tapret* หรือ *Opret*) ในโปรโตคอล RGB บล็อกเชน Bitcoin จึงทำหน้าที่เป็นเซิร์ฟเวอร์ประทับเวลาและระบบป้องกัน double-spending เท่านั้น


สัญญา RGB ถูกจัดโครงสร้างเหมือนเครื่องสถานะวิวัฒนาการ เริ่มต้นด้วย Genesis ที่กำหนดสถานะเริ่มต้น (อธิบายเช่น การจัดหา, ตั๋ว หรือเมตาดาต้าอื่น ๆ) ตาม Schema ที่ถูกพิมพ์และคอมไพล์อย่างเคร่งครัด การเปลี่ยนสถานะและหากจำเป็น การขยายสถานะจะถูกนำมาใช้เพื่อปรับเปลี่ยนหรือขยายสถานะนี้ แต่ละการดำเนินการ ไม่ว่าจะเป็นการโอนสินทรัพย์ที่สามารถทดแทนได้ (RGB20) หรือการสร้างสินทรัพย์ที่ไม่ซ้ำกัน (RGB21) จะเกี่ยวข้องกับ *ตราประทับใช้ครั้งเดียว* ซึ่งเชื่อมโยง Bitcoin UTXOs กับสถานะ off-chain และป้องกันการใช้จ่ายซ้ำซ้อน ในขณะเดียวกันก็รับรองความลับและความสามารถในการขยายตัว


หากต้องการเรียนรู้เพิ่มเติมเกี่ยวกับวิธีการทำงานของโปรโตคอล RGB ฉันขอแนะนำให้คุณเข้ารับการอบรมหลักสูตรที่ครอบคลุมนี้:


https://planb.academy/courses/3ce1d37c-05ba-4f54-aa15-7586d37b2bb7

ตรรกะภายในของ RGB อิงตามไลบรารี Rust ที่คุณในฐานะนักพัฒนาสามารถนำเข้าไปยังโปรเจกต์ของคุณเพื่อจัดการส่วน *Client-side Validation* นอกจากนี้ ทีม LNP/BP กำลังทำงานเกี่ยวกับการผูกสำหรับภาษาอื่น ๆ แต่ยังไม่ได้สรุป นอกจากนี้ หน่วยงานอื่น ๆ เช่น Bitfinex กำลังพัฒนาสแต็คการรวมของตนเอง แต่เราจะพูดถึงสิ่งเหล่านี้ในบทเรียนอื่น สำหรับตอนนี้ `rgb` CLI เป็นข้อมูลอ้างอิงอย่างเป็นทางการ แม้ว่าจะยังคงไม่ค่อยสมบูรณ์นักก็ตาม


## การติดตั้งและการนำเสนอเครื่องมือ rgb CLI


คำสั่งหลักเรียกว่า `rgb` ถูกออกแบบมาให้คล้ายกับ `git` โดยมีชุดคำสั่งย่อยสำหรับการจัดการสัญญา เรียกใช้งาน ออกสินทรัพย์ และอื่นๆ Bitcoin Wallet ยังไม่ได้ถูกรวมเข้ามาในขณะนี้ แต่จะมีในเวอร์ชันที่ใกล้จะออก (0.11) เวอร์ชันถัดไปนี้จะทำให้ผู้ใช้สามารถสร้างและจัดการกระเป๋าเงินของตนเอง (ผ่าน descriptors) ได้โดยตรงจาก `rgb` รวมถึงการสร้าง PSBT ความเข้ากันได้กับฮาร์ดแวร์ภายนอก (เช่น ฮาร์ดแวร์ wallet) สำหรับการลงนาม และการทำงานร่วมกับซอฟต์แวร์เช่น Sparrow สิ่งนี้จะทำให้การออกและโอนสินทรัพย์ทั้งหมดง่ายขึ้น


### การติดตั้งผ่าน Cargo


เราติดตั้งเครื่องมือใน Rust ด้วย :


```bash
cargo install rgb-contracts --all-features
```


(หมายเหตุ: crate นี้เรียกว่า `rgb-contracts` และคำสั่งที่ติดตั้งจะถูกตั้งชื่อว่า `rgb` หากมี crate ที่ชื่อ `rgb` อยู่แล้ว อาจเกิดการชนกันได้ ดังนั้นจึงตั้งชื่อนี้)


การติดตั้งจะคอมไพล์การพึ่งพาจำนวนมาก (เช่น การแยกวิเคราะห์คำสั่ง, การรวม Electrum, การจัดการหลักฐานความรู้เป็นศูนย์, เป็นต้น)


เมื่อติดตั้งเสร็จสิ้น, the :


```bash
rgb
```


การรัน `rgb` (โดยไม่มีอาร์กิวเมนต์) จะแสดงรายการคำสั่งย่อยที่มีอยู่ เช่น `interfaces`, `schema`, `import`, `export`, `issue`, `invoice`, `transfer`, เป็นต้น คุณสามารถเปลี่ยนไดเรกทอรีการจัดเก็บข้อมูลในเครื่อง (ที่เก็บที่ถือบันทึกทั้งหมด แผนผัง และการดำเนินการ) เลือกเครือข่าย (testnet, mainnet) หรือกำหนดค่าเซิร์ฟเวอร์ Electrum ของคุณได้


![RGB-CLI](assets/fr/01.webp)


### ภาพรวมแรกของการควบคุม


เมื่อคุณรันคำสั่งต่อไปนี้ คุณจะเห็นว่าอินเทอร์เฟซ `RGB20` ได้ถูกรวมเข้ามาโดยค่าเริ่มต้นแล้ว:


```bash
rgb interfaces
```


หากอินเทอร์เฟซนี้ไม่ได้ถูกรวมเข้าด้วยกัน ให้โคลน:


```bash
git clone https://github.com/RGB-WG/rgb-interfaces
```


รวบรวมมัน:


```bash
cargo run
```


จากนั้นนำเข้าชุดคำสั่งที่คุณเลือก:


```bash
rgb import interfaces/RGB20.rgb
```


![RGB-CLI](assets/fr/02.webp)


อย่างไรก็ตาม เราได้รับแจ้งว่ายังไม่มีการนำเข้า schema ใดๆ เข้าสู่ซอฟต์แวร์ และไม่มีสัญญาในคลังด้วย หากต้องการดู ให้รันคำสั่ง :


```bash
rgb schemata
```


จากนั้นคุณสามารถโคลนที่เก็บเพื่อดึงข้อมูลแผนผังบางอย่าง:


```bash
git clone https://github.com/RGB-WG/rgb-schemata
```


![RGB-CLI](assets/fr/03.webp)


ที่เก็บนี้มีไฟล์ Rust หลายไฟล์ในไดเรกทอรี `src/` (เช่น `nia.rs`) ซึ่งกำหนดสคีมา (NIA สำหรับ "*Non Inflatable Asset*", UDA สำหรับ "*Unique Digital Asset*", ฯลฯ) ในการคอมไพล์ คุณสามารถรัน :


```bash
cd rgb-schemata
cargo run
```


สิ่งนี้จะสร้างไฟล์ `.rgb` และ `.rgba` หลายไฟล์ที่สอดคล้องกับแผนผังที่คอมไพล์ ตัวอย่างเช่น คุณจะพบ `NonInflatableAsset.rgb`


### นำเข้า Schema และ Interface Implementation


คุณสามารถนำเข้าภาพร่างลงใน `rgb` ได้แล้ว :


```bash
rgb import schemata/NonInflatableAssets.rgb
```


![RGB-CLI](assets/fr/04.webp)


สิ่งนี้จะเพิ่มไปยังที่เก็บชั่วคราวในเครื่อง หากเรารันคำสั่งต่อไปนี้ เราจะเห็นว่า schema ปรากฏขึ้นแล้ว:


```bash
rgb schemata
```


## การสร้าง Contract (การออก)


มีสองวิธีในการสร้างสินทรัพย์ใหม่:




- ไม่ว่าเราจะใช้สคริปต์หรือโค้ดใน Rust ที่สร้าง Contract โดยการเติมข้อมูลในฟิลด์สคีมา (global state, Owned States, ฯลฯ) และสร้างไฟล์ `.rgb` หรือ `.rgba`;
- หรือใช้คำสั่งย่อย `issue` โดยตรง พร้อมกับไฟล์ YAML (หรือ TOML) ที่อธิบายคุณสมบัติของ token


คุณสามารถค้นหาตัวอย่างใน Rust ในโฟลเดอร์ `examples` ซึ่งแสดงวิธีการสร้าง `ContractBuilder`, กรอก `global state` (ชื่อสินทรัพย์, ตัวย่อ, จำนวน, วันที่, ฯลฯ), กำหนด Owned State (ซึ่ง UTXO ถูกกำหนดให้), จากนั้นรวบรวมทั้งหมดนี้เป็น *contract consignment* ที่คุณสามารถส่งออก, ตรวจสอบความถูกต้อง และนำเข้าไปยัง stash ได้


อีกวิธีหนึ่งคือการแก้ไขไฟล์ YAML ด้วยตนเองเพื่อปรับแต่ง `ticker`, `name`, `supply` และอื่นๆ สมมติว่าไฟล์นี้ชื่อ `RGB20-demo.yaml` คุณสามารถระบุ :




- `spec`: ตัวย่อ, ชื่อ, ความละเอียด ;
- `terms`: ฟิลด์สำหรับประกาศทางกฎหมาย ;
- `issuedSupply` : ปริมาณของ token ที่ออก;
- `assignments`: ระบุ Seal แบบใช้ครั้งเดียว (*คำจำกัดความของซีล*) และจำนวนที่ปลดล็อก


นี่คือตัวอย่างไฟล์ YAML ที่จะสร้าง:


```yaml
interface: RGB20Fixed
globals:
spec:
ticker: Plan ₿ Academy
name: Plan ₿ Academy
details: "Pay attention: the asset has no value"
precision: 2
terms:
text: >
SUBJECT TO, AND WITHOUT IN ANY WAY LIMITING, THE REPRESENTATIONS AND WARRANTIES OF ANY SELLER. PROPERTY IS BEING SOLD “AS IS”...
media: ~
issuedSupply: 100000000
assignments:
assetOwner:
seal: tapret1st:b449f7eaa3f98c145b27ad0eeb7b5679ceb567faef7a52479bc995792b65f804:1
amount: 100000000 # this is 1 million (we have two digits for cents)
```


![RGB-CLI](assets/fr/05.webp)


จากนั้นเพียงแค่รันคำสั่ง :


```bash
rgb issue '<SchemaID>' ssi:<Issuer> rgb20-demo.yaml
```


![RGB-CLI](assets/fr/06.webp)


ในกรณีของฉัน ตัวระบุสคีมาเฉพาะ (ให้อยู่ในเครื่องหมายอัญประกาศเดี่ยว) คือ `RDYhMTR!9gv8Y2GLv9UNBEK1hcrCmdLDFk9Qd5fnO8k` และฉันยังไม่ได้ใส่ผู้ออก ดังนั้นคำสั่งของฉันคือ :


```txt
rgb issue 'RDYhMTR!9gv8Y2GLv9UNBEK1hcrCmdLDFk9Qd5fnO8k' ssi:anonymous rgb20-demo.yaml
```


หากคุณไม่ทราบ ID สคีมา ให้รันคำสั่ง :


```bash
rgb schemata
```


CLI ตอบกลับว่ามีการออกสัญญาใหม่และเพิ่มเข้าไปในคลังแล้ว หากเราพิมพ์คำสั่งต่อไปนี้ เราจะเห็นว่ามีสัญญาเพิ่มเติม ซึ่งตรงกับสัญญาที่เพิ่งออก:


```bash
rgb contracts
```


![RGB-CLI](assets/fr/07.webp)


จากนั้น คำสั่งถัดไปจะแสดงสถานะทั่วโลก (ชื่อ, ตัวย่อ, อุปทาน...) และรายการของสถานะที่เป็นเจ้าของ, เช่น การจัดสรร (ตัวอย่างเช่น, 1 ล้านโทเค็น `Plan ₿ Academy` ที่กำหนดใน UTXO `b449f7eaa3f98c145b27ad0eeb7b5679ceb567faef7a52479bc995792b65f804:1`)


```bash
rgb state '<ContractId>'
```


![RGB-CLI](assets/fr/08.webp)


## ส่งออก, นำเข้า และการตรวจสอบความถูกต้อง


ในการแชร์สัญญานี้กับผู้ใช้อื่น สามารถส่งออกจากที่เก็บไปยัง:


```bash
rgb export '<ContractId>' myContractPBN.rgb
```


![RGB-CLI](assets/fr/09.webp)


ไฟล์ `myContractPBN.rgb` สามารถส่งต่อให้ผู้ใช้อื่นได้ ซึ่งผู้ใช้นั้นสามารถเพิ่มไฟล์ลงในคลังของเขาด้วยคำสั่ง :


```bash
rgb import myContractPBN.rgb
```


ในการนำเข้า หากเป็น *contract consignment* แบบง่าย เราจะได้รับข้อความ "`Importing consignment rgb`" หากเป็น *state transition consignment* ที่ใหญ่กว่า คำสั่งจะต่างออกไป (`rgb accept`)


เพื่อให้แน่ใจว่าถูกต้อง คุณยังสามารถใช้ฟังก์ชันการตรวจสอบความถูกต้องในท้องถิ่นได้อีกด้วย ตัวอย่างเช่น คุณสามารถรัน :


```bash
rgb validate myContract.rgb
```


### การใช้งาน, การตรวจสอบและการแสดงผล Stash


ในฐานะที่เป็นการเตือนความจำ สต๊อคเป็นคลังสินค้าท้องถิ่นของสคีมา, อินเตอร์เฟซ, การดำเนินการ และสัญญา (Genesis + การเปลี่ยนผ่าน) ทุกครั้งที่คุณรัน "import" คุณจะเพิ่มองค์ประกอบลงในสต๊อค คุณสามารถดูรายละเอียดของสต๊อคนี้ได้ด้วยคำสั่ง :


```bash
rgb dump
```


![RGB-CLI](assets/fr/10.webp)


สิ่งนี้จะ generate โฟลเดอร์ที่มีรายละเอียดของของทั้งหมด


## โอนและ PSBT


ในการดำเนินการโอน คุณจะต้องจัดการ Bitcoin wallet ในพื้นที่เพื่อจัดการกับข้อผูกพัน `Tapret` หรือ `Opret`


### สร้างใบแจ้งหนี้


ในกรณีส่วนใหญ่ การโต้ตอบระหว่างผู้เข้าร่วมในสัญญา (เช่น Alice และ Bob) เกิดขึ้นผ่านการสร้างใบแจ้งหนี้ หาก Alice ต้องการให้ Bob ดำเนินการบางอย่าง (การโอน token การออกใหม่ การดำเนินการใน DAO เป็นต้น) Alice จะสร้างใบแจ้งหนี้ที่มีรายละเอียดคำแนะนำของเธอไปยัง Bob ดังนั้นเราจึงมี :




- Alice** (ผู้ออกใบแจ้งหนี้) ;
- Bob** (ผู้ที่ได้รับและดำเนินการตามใบแจ้งหนี้)


ไม่เหมือนกับระบบนิเวศอื่น ๆ ใบแจ้งหนี้ RGB ไม่จำกัดเฉพาะแนวคิดเรื่องการชำระเงินเท่านั้น มันสามารถฝังคำขอใด ๆ ที่เชื่อมโยงกับสัญญา: เพิกถอนคีย์, ลงคะแนน, สร้างการแกะสลัก (*engraving*) บน NFT เป็นต้น การดำเนินการที่เกี่ยวข้องสามารถอธิบายได้ในอินเทอร์เฟซของสัญญา การดำเนินการที่เกี่ยวข้องสามารถอธิบายได้ในอินเทอร์เฟซของสัญญา


คำสั่งต่อไปนี้สร้างใบแจ้งหนี้ RGB:


```bash
$ rgb invoice $CONTRACT -i $INTERFACE $ACTION $STATE $SEAL
```


ด้วย :




- `$CONTRACT`: Contract identifier (*ContractId*) ;
- `$INTERFACE`: อินเทอร์เฟซที่จะใช้ (เช่น `RGB20`) ;
- `$ACTION`: ชื่อของการดำเนินการที่ระบุในอินเทอร์เฟซ (สำหรับการโอน token แบบง่าย ๆ อาจเป็น "Transfer") หากอินเทอร์เฟซมีการดำเนินการเริ่มต้นอยู่แล้ว คุณไม่จำเป็นต้องป้อนอีกครั้งที่นี่;
- `$STATE`: ข้อมูลสถานะที่จะถูกโอน (เช่น จำนวนโทเค็นหากมีการโอน token ที่สามารถแลกเปลี่ยนได้);
- `$SEAL`: Alice (ผู้รับผลประโยชน์) ใช้ Seal เพียงครั้งเดียว, กล่าวคือ การอ้างอิงโดยตรงถึง UTXO. Bob จะใช้ข้อมูลนี้เพื่อสร้างธุรกรรมพยาน, และผลลัพธ์ที่สอดคล้องกันจะเป็นของ Alice (ในรูปแบบ *blinded UTXO* หรือรูปแบบที่ไม่เข้ารหัส).


ตัวอย่างเช่น ด้วยคำสั่งต่อไปนี้


```bash
alice$ CONTRACT='iZgIN9EL-2H21UgQ-x!A3uJc-WwXhCSm-$9Lwcc1-v!mUkKY'
alice$ MY_UTXO=4960acc21c175c551af84114541eace09c14d3a1bb184809f7b80916f57f9ef8:1
alice$ rgb invoice $CONTRACT -i RGB20 --amount 100 $MY_UTXO
```


CLI จะ generate ใบแจ้งหนี้เช่น:


```bash
rgb:iZgIN9EL-2H21UgQ-x!A3uJc-WwXhCSm-$9Lwcc1-v!mUkKY/RGB20/100+utxob:zlVS28Rb-...
```


สามารถส่งไปยัง Bob ได้ผ่านช่องทางใดก็ได้ (ข้อความ, รหัส QR, ฯลฯ)


### กำลังทำการโอน


เพื่อโอนจากใบแจ้งหนี้นี้ :




- Bob (ผู้ที่ถือโทเค็นในคลังของเขา) มี Bitcoin wallet เขาจำเป็นต้องเตรียมธุรกรรม Bitcoin (ในรูปแบบของ PSBT, เช่น `tx.psbt`) ซึ่งใช้จ่าย UTXOs ที่มีโทเค็น RGB ที่ต้องการอยู่, บวกกับหนึ่ง UTXO สำหรับสกุลเงิน (แลกเปลี่ยน);
- Bob ดำเนินการคำสั่งต่อไปนี้:


```bash
bob$ rgb transfer tx.psbt $INVOICE consignment.rgb
```




- สิ่งนี้จะสร้างไฟล์ `consignment.rgb` ซึ่งประกอบด้วย :
 - ประวัติการเปลี่ยนแปลงที่พิสูจน์ให้ Alice ว่าโทเค็นนั้นเป็นของแท้;
 - การเปลี่ยนแปลงใหม่ที่โอนโทเค็นไปยัง Alice's Single-use Seal ;
 - พยานธุรกรรม (ยังไม่ได้ลงนาม)
- Bob ส่งไฟล์ `consignment.rgb` นี้ไปยัง Alice (ทางอีเมล, เซิร์ฟเวอร์แชร์หรือโปรโตคอล RGB-RPC, Storm, เป็นต้น);
- Alice รับ `consignment.rgb` และยอมรับในคลังของตัวเอง :


```bash
alice$ rgb accept consignment.rgb
```




- CLI ตรวจสอบความถูกต้องของการเปลี่ยนผ่านและเพิ่มไปยังคลังของ Alice หากไม่ถูกต้อง คำสั่งจะล้มเหลวพร้อมข้อความแสดงข้อผิดพลาดโดยละเอียด มิฉะนั้นจะสำเร็จ และรายงานว่าธุรกรรมตัวอย่างยังไม่ได้ถูกกระจายบนเครือข่าย Bitcoin (Bob กำลังรอสัญญาณไฟเขียวจาก Alice);
- เพื่อเป็นการยืนยัน คำสั่ง `accept` จะส่งคืนลายเซ็น (*payslip*) ซึ่ง Alice สามารถส่งให้กับ Bob เพื่อแสดงให้เขาเห็นว่าเธอได้ตรวจสอบ *consignment* แล้ว;
- Bob สามารถลงนามและเผยแพร่ (`--publish`) ธุรกรรม Bitcoin ของเขา:


```bash
bob$ rgb check <sig> && wallet sign --publish tx.psbt
```




- ทันทีที่ธุรกรรมนี้ได้รับการยืนยัน on-chain การเป็นเจ้าของสินทรัพย์จะถือว่าถูกโอนไปยัง Alice wallet ของ Alice ซึ่งตรวจสอบ mining ของธุรกรรม เห็น Owned State ใหม่ปรากฏในคลังของมัน


คุณตอนนี้รู้วิธีการออกและโอนสัญญา RGB แล้ว หากคุณพบว่าบทแนะนำนี้มีประโยชน์ ฉันจะรู้สึกขอบคุณมากหากคุณกดนิ้วโป้งสีเขียวด้านล่าง โปรดอย่าลังเลที่จะแบ่งปันบทความนี้บนเครือข่ายสังคมของคุณ ขอบคุณมาก!
