इस ट्यूटोरियल में, हम LNP/BP एसोसिएशन द्वारा बनाए गए कमांड-लाइन टूल `RGB` का उपयोग करके Contract लिखने की चरण-दर-चरण प्रक्रिया का पालन करेंगे। इसका उद्देश्य यह दिखाना है कि CLI को कैसे इंस्टॉल और मैनिपुलेट किया जाए, Schema को कैसे संकलित किया जाए, Interface और Interface Implementation को कैसे आयात किया जाए और फिर RGB एसेट कैसे जारी किया जाए। हम संकलन और स्थिति सत्यापन सहित अंतर्निहित तर्क को भी देखेंगे। इस ट्यूटोरियल के अंत में, आपको प्रक्रिया को पुन: पेश करने और अपने स्वयं के RGB अनुबंध बनाने में सक्षम होना चाहिए।


## RGB प्रोटोकॉल अनुस्मारक


RGB एक प्रोटोकॉल है जो Bitcoin के शीर्ष पर चलता है और Smart contract कार्यक्षमता और डिजिटल एसेट प्रबंधन का अनुकरण करता है, बिना Blockchain को ओवरलोड किए जिस पर यह आधारित है। पारंपरिक On-Chain स्मार्ट कॉन्ट्रैक्ट्स (जैसे कि एथेरियम पर, उदाहरण के लिए) के विपरीत, RGB एक "*Client-side Validation*" सिस्टम पर निर्भर करता है: अधिकांश डेटा और स्थिति इतिहास का आदान-प्रदान और भंडारण विशेष रूप से शामिल प्रतिभागियों द्वारा किया जाता है, जबकि Bitcoin Blockchain केवल छोटी क्रिप्टोग्राफ़िक प्रतिबद्धताओं (जैसे *टैप्रेट* या *ओप्रेट* जैसे तंत्रों के माध्यम से) को होस्ट करता है। RGB प्रोटोकॉल में, Bitcoin Blockchain इसलिए केवल टाइम-स्टैम्पिंग सर्वर और Double-spending सुरक्षा प्रणाली के रूप में कार्य करता है।


RGB Contract को एक विकासवादी राज्य मशीन की तरह संरचित किया गया है। यह एक Genesis से शुरू होता है जो एक सख्त टाइप और संकलित Schema के अनुसार प्रारंभिक स्थिति (उदाहरण के लिए, Supply, टिकर या अन्य मेटाडेटा का वर्णन) को परिभाषित करता है। राज्य संक्रमण और, यदि आवश्यक हो, तो राज्य विस्तार को इस स्थिति को संशोधित या विस्तारित करने के लिए लागू किया जाता है। प्रत्येक ऑपरेशन, चाहे वह फ़ंजिबल एसेट्स (RGB20) को स्थानांतरित करना हो या अद्वितीय एसेट्स (RGB21) बनाना हो, इसमें *सिंगल-यूज़ सील* शामिल हैं। ये Bitcoin UTXO को off-chain राज्यों से जोड़ते हैं और गोपनीयता और मापनीयता सुनिश्चित करते हुए दोहरे खर्च को रोकते हैं।


RGB प्रोटोकॉल कैसे काम करता है, इसके बारे में अधिक जानने के लिए, मैं आपको यह व्यापक प्रशिक्षण पाठ्यक्रम लेने की सलाह देता हूं:


https://planb.network/courses/3ce1d37c-05ba-4f54-aa15-7586d37b2bb7

RGB का आंतरिक तर्क Rust लाइब्रेरी पर आधारित है जिसे आप, डेवलपर्स के रूप में, *Client-side Validation* भाग को प्रबंधित करने के लिए अपनी परियोजनाओं में आयात कर सकते हैं। इसके अलावा, LNP/BP टीम अन्य भाषाओं के लिए बाइंडिंग पर काम कर रही है, लेकिन इसे अभी तक अंतिम रूप नहीं दिया गया है। इसके अलावा, बिटफ़ाइनेक्स जैसी अन्य संस्थाएँ अपने स्वयं के एकीकरण स्टैक विकसित कर रही हैं, लेकिन हम इनके बारे में किसी अन्य ट्यूटोरियल में बात करेंगे। फिलहाल, `RGB` CLI आधिकारिक संदर्भ है, भले ही यह अपेक्षाकृत अपरिष्कृत हो।


## RGB CLI उपकरण की स्थापना और प्रस्तुति


मुख्य कमांड को बस `RGB` कहा जाता है। इसे `git` की याद दिलाने के लिए डिज़ाइन किया गया है, जिसमें अनुबंधों में हेरफेर करने, उन्हें लागू करने, संपत्ति जारी करने आदि के लिए उप-कमांड का एक सेट है। Bitcoin Wallet वर्तमान में एकीकृत नहीं है, लेकिन एक आसन्न संस्करण (0.11) में होगा। यह अगला संस्करण उपयोगकर्ताओं को `RGB` से सीधे अपने वॉलेट (वर्णनकर्ताओं के माध्यम से) बनाने और प्रबंधित करने में सक्षम करेगा, जिसमें PSBT पीढ़ी, हस्ताक्षर करने के लिए बाहरी हार्डवेयर (जैसे Hardware Wallet) के साथ संगतता और स्पैरो जैसे सॉफ़्टवेयर के साथ अंतर-संचालन शामिल है। यह संपूर्ण परिसंपत्ति जारी करने और हस्तांतरण परिदृश्य को सरल करेगा।


### कार्गो के माध्यम से स्थापना


हम Rust में उपकरण स्थापित करते हैं:


```bash
cargo install rgb-contracts --all-features
```


(नोट: क्रेट को `RGB-contracts` कहा जाता है, और स्थापित कमांड का नाम `RGB` होगा। यदि `RGB` नाम का क्रेट पहले से मौजूद है, तो टकराव हो सकता है, इसलिए यह नाम रखा गया है)


यह संस्थापन बड़ी संख्या में निर्भरताओं (जैसे कमांड पार्सिंग, इलेक्ट्रम एकीकरण, शून्य-ज्ञान प्रमाण प्रबंधन, आदि) को संकलित करता है।


एक बार स्थापना पूर्ण हो जाने पर, :


```bash
rgb
```


`RGB` (बिना तर्कों के) चलाने पर उपलब्ध उप-कमांडों की सूची प्रदर्शित होती है, जैसे `इंटरफेस`, `Schema`, `इम्पोर्ट`, `एक्सपोर्ट`, `इश्यू`, `Invoice`, `ट्रांसफर`, आदि। आप स्थानीय भंडारण निर्देशिका (एक Stash जो सभी लॉग, स्कीमैटिक्स और कार्यान्वयन रखती है) को बदल सकते हैं, नेटवर्क (Testnet, Mainnet) चुन सकते हैं या अपने इलेक्ट्रम सर्वर को कॉन्फ़िगर कर सकते हैं।


![RGB-CLI](assets/fr/01.webp)


### नियंत्रणों का पहला अवलोकन


जब आप निम्नलिखित कमांड चलाएंगे, तो आप देखेंगे कि `RGB20` Interface पहले से ही डिफ़ॉल्ट रूप से एकीकृत है:


```bash
rgb interfaces
```


यदि यह Interface एकीकृत नहीं है, तो क्लोन करें:


```bash
git clone https://github.com/RGB-WG/rgb-interfaces
```


इसे संकलित करें:


```bash
cargo run
```


फिर अपनी पसंद का Interface आयात करें:


```bash
rgb import interfaces/RGB20.rgb
```


![RGB-CLI](assets/fr/02.webp)


हालाँकि, हमें बताया गया है कि सॉफ़्टवेयर में अभी तक कोई Schema आयात नहीं किया गया है। न ही Stash में कोई Contract है। इसे देखने के लिए, कमांड चलाएँ:


```bash
rgb schemata
```


फिर आप कुछ निश्चित योजनाओं को पुनः प्राप्त करने के लिए रिपोजिटरी को क्लोन कर सकते हैं:


```bash
git clone https://github.com/RGB-WG/rgb-schemata
```


![RGB-CLI](assets/fr/03.webp)


इस रिपॉजिटरी में, इसकी `src/` निर्देशिका में, कई Rust फ़ाइलें (उदाहरण के लिए `nia.rs`) हैं जो स्कीमा परिभाषित करती हैं (NIA "*Non Inflatable Asset*" के लिए, UDA "*Unique Digital Asset*" के लिए, आदि)। संकलन करने के लिए, आप चला सकते हैं:


```bash
cd rgb-schemata
cargo run
```


यह संकलित स्कीमेटिक्स के अनुरूप कई `.RGB` और `.rgba` फ़ाइलें उत्पन्न करता है। उदाहरण के लिए, आपको `NonInflatableAsset.RGB` मिलेगा।


### Schema और Interface Implementation का आयात


अब आप योजनाबद्ध को `RGB` में आयात कर सकते हैं:


```bash
rgb import schemata/NonInflatableAssets.rgb
```


![RGB-CLI](assets/fr/04.webp)


यह इसे स्थानीय Stash में जोड़ता है। यदि हम निम्न आदेश चलाते हैं, तो हम देखते हैं कि Schema अब प्रकट होता है:


```bash
rgb schemata
```


## Contract निर्माण (जारी करना)


नई परिसंपत्ति बनाने के दो तरीके हैं:




- या तो हम Rust में एक स्क्रिप्ट या कोड का उपयोग करते हैं जो Schema फ़ील्ड (Global State, स्वामित्व वाले राज्य, आदि) को पॉप्युलेट करके Contract बनाता है और `.RGB` या `.rgba` फ़ाइल का उत्पादन करता है;
- या फिर टोकन के गुणों का वर्णन करने वाली YAML (या TOML) फ़ाइल के साथ सीधे `issue` उप-कमांड का उपयोग करें।


आप `उदाहरण` फ़ोल्डर में Rust में उदाहरण पा सकते हैं, जो दर्शाते हैं कि आप `ContractBuilder` का निर्माण कैसे करते हैं, `Global State` (परिसंपत्ति का नाम, टिकर, Supply, दिनांक, आदि) को कैसे भरते हैं, Owned State को परिभाषित करते हैं (किस UTXO को यह सौंपा गया है), फिर इन सभी को एक *Contract Consignment* में संकलित करते हैं जिसे आप निर्यात कर सकते हैं, मान्य कर सकते हैं और एक Stash में आयात कर सकते हैं।


दूसरा तरीका `टिकर`, `नाम`, `Supply`, इत्यादि को अनुकूलित करने के लिए YAML फ़ाइल को मैन्युअल रूप से संपादित करना है। मान लीजिए कि फ़ाइल का नाम `RGB20-demo.yaml` है। आप निर्दिष्ट कर सकते हैं:




- `spec`: टिकर, नाम, परिशुद्धता;
- `शर्तें`: कानूनी नोटिस के लिए एक क्षेत्र;
- `issuedSupply` : जारी किए गए टोकन की मात्रा;
- `असाइनमेंट`: Single-Use Seal (*Seal Definition*) और अनलॉक की गई मात्रा को इंगित करता है।


यहां एक YAML फ़ाइल बनाने का उदाहरण दिया गया है:


```yaml
interface: RGB20Fixed
globals:
spec:
ticker: PBN
name: Plan B Network
details: "Pay attention: the asset has no value"
precision: 2
terms:
text: >
SUBJECT TO, AND WITHOUT IN ANY WAY LIMITING, THE REPRESENTATIONS AND WARRANTIES OF ANY SELLER. PROPERTY IS BEING SOLD “AS IS”...
media: ~
issuedSupply: 100000000
assignments:
assetOwner:
seal: tapret1st:b449f7eaa3f98c145b27ad0eeb7b5679ceb567faef7a52479bc995792b65f804:1
amount: 100000000 # this is 1 million (we have two digits for cents)
```


![RGB-CLI](assets/fr/05.webp)


फिर बस कमांड चलाएँ:


```bash
rgb issue '<SchemaID>' ssi:<Issuer> rgb20-demo.yaml
```


![RGB-CLI](assets/fr/06.webp)


मेरे मामले में, अद्वितीय Schema पहचानकर्ता (एकल उद्धरण चिह्नों में संलग्न) `RDYhMTR!9gv8Y2GLv9UNBEK1hcrCmdLDFk9Qd5fnO8k` है और मैंने कोई जारीकर्ता नहीं डाला है। इसलिए मेरा आदेश है:


```txt
rgb issue 'RDYhMTR!9gv8Y2GLv9UNBEK1hcrCmdLDFk9Qd5fnO8k' ssi:anonymous rgb20-demo.yaml
```


यदि आपको Schema आईडी नहीं पता है, तो कमांड चलाएँ:


```bash
rgb schemata
```


CLI उत्तर देता है कि एक नया Contract जारी किया गया है और Stash में जोड़ा गया है। यदि हम निम्नलिखित कमांड टाइप करते हैं, तो हम देखते हैं कि अब एक अतिरिक्त Contract है, जो अभी जारी किए गए Contract के अनुरूप है:


```bash
rgb contracts
```


![RGB-CLI](assets/fr/07.webp)


फिर, अगला आदेश वैश्विक स्थितियाँ (नाम, टिकर, Supply...) और स्वामित्व वाली स्थितियों की सूची, यानी आवंटन (उदाहरण के लिए, UTXO `b449f7eaa3f98c145b27ad0eeb7b5679ceb567faef7a52479bc995792b65f804:1` में परिभाषित 1 मिलियन `PBN` टोकन) प्रदर्शित करता है।


```bash
rgb state '<ContractId>'
```


![RGB-CLI](assets/fr/08.webp)


## निर्यात, आयात और सत्यापन


इस Contract को अन्य उपयोगकर्ताओं के साथ साझा करने के लिए, इसे Stash से निर्यात किया जा सकता है:


```bash
rgb export '<ContractId>' myContractPBN.rgb
```


![RGB-CLI](assets/fr/09.webp)


`myContractPBN.RGB` फ़ाइल को किसी अन्य उपयोगकर्ता को दिया जा सकता है, जो इसे निम्न आदेश द्वारा अपने Stash में जोड़ सकता है:


```bash
rgb import myContractPBN.rgb
```


आयात करते समय, यदि यह एक साधारण *Contract Consignment* है, तो हमें "`Consignment RGB आयात किया जा रहा है`" संदेश मिलेगा। यदि यह एक बड़ा *State Transition Consignment* है, तो कमांड अलग होगा (`RGB स्वीकार करें`)।


वैधता सुनिश्चित करने के लिए, आप स्थानीय सत्यापन फ़ंक्शन का भी उपयोग कर सकते हैं। उदाहरण के लिए, आप चला सकते हैं:


```bash
rgb validate myContract.rgb
```


### Stash उपयोग, सत्यापन और प्रदर्शन


याद दिला दें कि Stash स्कीमा, इंटरफेस, कार्यान्वयन और अनुबंधों (Genesis + संक्रमण) की एक स्थानीय सूची है। हर बार जब आप "आयात" चलाते हैं, तो आप Stash में एक तत्व जोड़ते हैं। इस Stash को कमांड के साथ विस्तार से देखा जा सकता है:


```bash
rgb dump
```


![RGB-CLI](assets/fr/10.webp)


इससे संपूर्ण Stash के विवरण वाला एक फ़ोल्डर बन जाएगा।


## स्थानांतरण और PSBT


स्थानांतरण करने के लिए, आपको `Tapret` या `Opret` प्रतिबद्धताओं को प्रबंधित करने के लिए स्थानीय Bitcoin Wallet में हेरफेर करने की आवश्यकता होगी।


### generate और Invoice


ज़्यादातर मामलों में, Contract में प्रतिभागियों (जैसे कि ऐलिस और बॉब) के बीच बातचीत Invoice के निर्माण के ज़रिए होती है। अगर ऐलिस चाहती है कि बॉब कुछ निष्पादित करे (टोकन ट्रांसफ़र, रीइश्यू, DAO में कोई कार्रवाई, आदि), तो ऐलिस बॉब को दिए गए अपने निर्देशों का विवरण देते हुए एक Invoice बनाती है। तो हमारे पास है:




- ऐलिस** (Invoice जारीकर्ता) ;
- बॉब** (जो Invoice प्राप्त करता है और उसे निष्पादित करता है)।


अन्य पारिस्थितिकी तंत्रों के विपरीत, RGB Invoice भुगतान की धारणा तक सीमित नहीं है। यह Contract से जुड़े किसी भी अनुरोध को एम्बेड कर सकता है: कुंजी को रद्द करना, वोट करना, NFT पर उत्कीर्णन (* उत्कीर्णन *) बनाना, आदि। संबंधित ऑपरेशन को Contract Interface में वर्णित किया जा सकता है। संबंधित ऑपरेशन को Contract Interface में वर्णित किया जा सकता है।


निम्न आदेश RGB Invoice उत्पन्न करता है:


```bash
$ rgb invoice $CONTRACT -i $INTERFACE $ACTION $STATE $SEAL
```


साथ :




- `$Contract`: Contract पहचानकर्ता (*ContractId*) ;
- `$Interface`: उपयोग किया जाने वाला Interface (उदाहरणार्थ `RGB20`) ;
- `$ACTION`: Interface में निर्दिष्ट ऑपरेशन का नाम (सरल फंगिबल टोकन ट्रांसफर के लिए, यह "ट्रांसफर" हो सकता है)। यदि Interface पहले से ही एक डिफ़ॉल्ट एक्शन प्रदान करता है, तो आपको इसे यहाँ फिर से दर्ज करने की आवश्यकता नहीं है;
- `$STATE`: स्थानांतरित किया जाने वाला स्थिति डेटा (उदाहरण के लिए, यदि एक फंजिबल टोकन स्थानांतरित किया जाता है तो टोकन की मात्रा);
- `$Seal`: लाभार्थी (ऐलिस) का Single-Use Seal, यानी UTXO का स्पष्ट संदर्भ। बॉब इस जानकारी का उपयोग Witness Transaction बनाने के लिए करेगा, और उसके बाद संबंधित आउटपुट ऐलिस (*blinded UTXO* या अनएन्क्रिप्टेड फॉर्म में) का होगा।


उदाहरण के लिए, निम्नलिखित आदेशों के साथ


```bash
alice$ CONTRACT='iZgIN9EL-2H21UgQ-x!A3uJc-WwXhCSm-$9Lwcc1-v!mUkKY'
alice$ MY_UTXO=4960acc21c175c551af84114541eace09c14d3a1bb184809f7b80916f57f9ef8:1
alice$ rgb invoice $CONTRACT -i RGB20 --amount 100 $MY_UTXO
```


CLI, generate और Invoice की तरह होगा:


```bash
rgb:iZgIN9EL-2H21UgQ-x!A3uJc-WwXhCSm-$9Lwcc1-v!mUkKY/RGB20/100+utxob:zlVS28Rb-...
```


इसे किसी भी चैनल (टेक्स्ट, क्यूआर कोड, आदि) के माध्यम से बॉब तक प्रेषित किया जा सकता है।


### स्थानांतरण करना


इस Invoice से स्थानांतरित करने के लिए:




- बॉब (जो अपने Stash में टोकन रखता है) के पास Bitcoin Wallet है। उसे एक Bitcoin लेनदेन (PSBT के रूप में, उदाहरण के लिए `tx.PSBT`) तैयार करने की आवश्यकता है जो UTXOs को उन जगहों पर खर्च करता है जहाँ आवश्यक RGB टोकन स्थित हैं, साथ ही मुद्रा के लिए एक UTXO (Exchange) ;
- बॉब निम्नलिखित आदेश निष्पादित करता है:


```bash
bob$ rgb transfer tx.psbt $INVOICE consignment.rgb
```




- यह एक `Consignment.RGB` फ़ाइल उत्पन्न करता है जिसमें निम्न शामिल है:
 - संक्रमण इतिहास एलिस को साबित करता है कि टोकन वास्तविक हैं;
 - नया संक्रमण जो टोकन को ऐलिस के Single-Use Seal में स्थानांतरित करता है;
 - एक Witness Transaction (हस्ताक्षर रहित).
- बॉब यह `Consignment.RGB` फ़ाइल ऐलिस को भेजता है (ई-मेल, शेयरिंग सर्वर या RGB-RPC प्रोटोकॉल, स्टॉर्म, आदि द्वारा);
- ऐलिस `Consignment.RGB` प्राप्त करता है और इसे अपने Stash में स्वीकार करता है:


```bash
alice$ rgb accept consignment.rgb
```




- CLI संक्रमण की वैधता की जाँच करता है और इसे ऐलिस के Stash में जोड़ता है। यदि अमान्य है, तो कमांड विस्तृत त्रुटि संदेशों के साथ विफल हो जाता है। अन्यथा, यह सफल होता है, और रिपोर्ट करता है कि नमूना लेनदेन अभी तक Bitcoin नेटवर्क पर प्रसारित नहीं हुआ है (बॉब ऐलिस के Green प्रकाश की प्रतीक्षा कर रहा है);
- पुष्टि के रूप में, `स्वीकार` कमांड एक हस्ताक्षर (*पेस्लिप*) लौटाता है जिसे ऐलिस बॉब को यह दिखाने के लिए भेज सकती है कि उसने *Consignment* को मान्य कर दिया है;
- इसके बाद बॉब अपने Bitcoin लेनदेन पर हस्ताक्षर कर उसे प्रकाशित (`--publish`) कर सकता है:


```bash
bob$ rgb check <sig> && wallet sign --publish tx.psbt
```




- जैसे ही इस लेन-देन की पुष्टि हो जाती है, परिसंपत्ति का On-Chain, Ownership एलिस को हस्तांतरित माना जाता है। एलिस का Wallet, लेन-देन के Mining की निगरानी करता है, और अपने Stash में नया Owned State देखता है।


अब आप जानते हैं कि RGB Contract को कैसे जारी और स्थानांतरित किया जाता है। अगर आपको यह ट्यूटोरियल उपयोगी लगा, तो मैं बहुत आभारी रहूंगा यदि आप नीचे Green थंब लगाएंगे। कृपया इस लेख को अपने सोशल नेटवर्क पर साझा करने के लिए स्वतंत्र महसूस करें। आपका बहुत-बहुत धन्यवाद!
