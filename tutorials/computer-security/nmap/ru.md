## 1 - Введение в Nmap: Что такое Nmap?



### I. Презентация



В этом первом разделе мы рассмотрим инструмент для сканирования сетей Nmap. Мы рассмотрим ключевые Elements, которые необходимо знать об этом инструменте и о том, как он работает в целом. Это поможет нам лучше понять остальную часть учебника.



### II. Знакомство с инструментом Nmap



Nmap, что означает _Network Mapper_, - это бесплатный инструмент с открытым исходным кодом, используемый для **обнаружения, отображения и аудита безопасности сети**. Он также может использоваться для других задач, таких как **инвентаризация, диагностика или наблюдение за сетью**.



С его помощью можно определить, активны ли и доступны ли хосты в целевой сети, какие сетевые службы открыты, какие версии и технологии используются, а также получить другую полезную аналитическую информацию. Nmap можно использовать как для сканирования отдельной службы на конкретной машине, так и на больших участках сети, вплоть до всего Интернета.



Сильных сторон у Nmap много:





- Мощный и гибкий**: Nmap может сканировать большие сети и использовать передовые методы обнаружения. Он поддерживает UDP, TCP, ICMP, IPv4 и IPv6 и может выполнять определение версий, сканирование уязвимостей или взаимодействие с конкретными протоколами. Его архитектура является модульной, в частности, благодаря скриптам NSE (Nmap Scripting Engine), которые мы рассмотрим далее в этом руководстве.
- Простота использования**: официальная документация многочисленна и отличается высочайшим качеством. Многочисленные ресурсы сообщества также помогут вам начать работу.
- Популярность и долговечность**: Nmap является эталоном в своей области с 1998 года. Текущая версия на момент обновления - 7.95. Несмотря на существование других инструментов для решения конкретных задач, Nmap остается обязательным инструментом для картографирования и анализа сетей.



**Нмап в кино*



Nmap - один из немногих инструментов безопасности, получивших определенную известность среди широкой публики. Он появляется в фильме _Matrix Reloaded_, в знаменательной сцене, где Тринити использует его для взлома системы:

### III. Работа на высоком уровне



Nmap доступен для Linux, Windows и macOS. Он написан в основном на языках C, C++ и Lua (для скриптов NSE). В основном он используется в командной строке, хотя графические интерфейсы, такие как Zenmap, также доступны. Тем не менее, мы настоятельно рекомендуем вам начать с командной строки, чтобы лучше понять, как она работает.



Простой пример:



```
nmap 192.168.10.13 10.10.10.0/24 -sV -sC --top-ports 250
```



Эта команда будет подробно описана позже. В этом руководстве мы будем использовать Nmap в Linux, но его применение в других системах аналогично. В Windows Nmap использует библиотеку **Npcap** (заменившую устаревшую WinPcap) для перехвата и инъекции сетевых пакетов.



Nmap используется как обычный двоичный файл, такой как `ls` или `ip`. Некоторые дополнительные функции могут потребовать повышенных прав, так как инструмент иногда манипулирует пакетами нетрадиционными способами, чтобы спровоцировать определенные реакции на целевых системах (в частности, для обнаружения сервисов или уязвимостей).



### IV. Последствия использования Nmap



Прежде чем использовать Nmap, необходимо знать о его потенциальном влиянии на сети и системы:





- Он может отправлять **тысячи или даже миллионы пакетов** за короткий промежуток времени, что может насытить определенные сетевые инфраструктуры.
- Он может generate **деформированные или нестандартные** пакеты, способные вывести из строя определенное оборудование (особенно промышленные системы).
- Он может создавать **поведение, похожее на атаку**, что может вызвать предупреждения в системах безопасности (брандмауэрах, IDS/IPS и т. д.).



Вообще говоря, **Nmap - очень разговорчивый инструмент**, поскольку он генерирует много трафика, чтобы извлечь как можно больше информации. Поэтому рекомендуется полностью понять, как он работает, прежде чем использовать его в чувствительных или производственных средах.



### V. Заключение



В этом разделе мы познакомимся с программой Nmap и ее основными возможностями. Мы увидели, что это важный, мощный и гибкий инструмент для картографирования сетей. Мы также обсудили, как он работает и какие меры предосторожности необходимо соблюдать при его использовании, чтобы подготовить почву для последующих частей руководства.



## 2 - Зачем использовать Nmap?



### I. Презентация



В этом разделе мы рассмотрим основные области применения инструмента для сканирования сетей Nmap. Мы увидим, что этот инструмент широко используется во многих контекстах и профессиях, и что его наличие в вашем наборе инструментов и умение им владеть - это, безусловно, полезный навык.



### II. Использование Nmap для диагностики и контроля



Nmap можно использовать для диагностики сети и, в более широком смысле, для мониторинга. Подобно тому, как с помощью ping можно определить, взаимодействуют ли два хоста, с помощью Nmap можно быстро определить, активен ли хост или работает ли определенная служба. Благодаря [Nmap] (https://www.it-connect.fr/cours/nmap-cartographie-reseau-scan-de-vulnerabilites/ "Nmap") мы можем получить точные данные о времени отклика хоста, маршруте следования пакетов, отклике конкретной службы и т. д.



Следующую команду и результат можно использовать, например, чтобы быстро узнать, активен ли веб-сервер на хосте **192.168.1.18** и правильно ли он отвечает:



```
nmap --open -p 80 192.168.1.18
```



![nmap-image](assets/fr/02.webp)



*Используйте Nmap для получения информации о состоянии веб-сервиса с удаленного сервера.*



Таким образом, использование Nmap выходит за рамки знаменитого "пинг-теста" на этапах отладки или диагностики. Позже мы увидим, что существует несколько методов, используемых Nmap для определения того, какой сервис прослушивает данный порт, включая его версию.



### III. Использование Nmap для составления карты сети



Основной задачей этого инструмента является составление карты сети. Его можно использовать как в локальной сети, так и в нескольких сетях, подсетях и VLAN, чтобы составить список всех доступных хостов и служб. Nmap делает эту задачу намного быстрее и эффективнее, чем любой ручной метод.



Например, следующая команда может быть использована для быстрого определения активных хостов в сети **192.168.1.0/24**:



```
nmap -sn 192.168.1.0/24
```



*Примечание: опция `-sP` устарела и была заменена на `-sn`.*



![nmap-image](assets/fr/03.webp)



*Использование Nmap для обнаружения доступных хостов в сети*



Позже мы увидим, что существует несколько методов, используемых Nmap для определения того, можно ли считать хост "активным", даже если он априори не предоставляет никаких услуг.



### IV. Использование Nmap для оценки политики фильтрации



Преимущество Nmap - в фактичности: его результаты позволяют сделать конкретные выводы, в отличие от любого архитектурного документа или политики фильтрации. Это ключевой инструмент для оценки эффективности разделения информационных систем, поскольку он позволяет **проверить, действительно ли применяется политика фильтрации**.



В корпоративной сети лучшие практики предписывают сегментировать системы в зависимости от их роли, важности или местоположения. Правила фильтрации (часто реализуемые с помощью брандмауэров) должны ограничивать связь между зонами. Но эти конфигурации часто сложны и чреваты ошибками.



Поэтому для подтверждения соблюдения политики нет ничего лучше конкретного теста. Например, можно проверить, что чувствительные службы администрирования (SSH, WinRM, MSSQL, MySQL и т. д.) недоступны из пользовательской зоны.



Использование **Nmap для проверки политики фильтрации** должно быть систематическим, прежде чем такая политика будет запущена в производство. К сожалению, этой проверкой часто пренебрегают.



По моему опыту, многие ошибки конфигурации остаются незамеченными при отсутствии тестов проверки. Простая ошибка в диапазоне IP-адресов или недосмотр в правиле могут сделать уязвимой якобы изолированную зону.



### V. Использование Nmap для аудита безопасности и тестирования на проникновение



Nmap обладает **многими полезными функциями для оценки безопасности**, тестирования на проникновение (пентесты), а также, к сожалению, для злоумышленников.



Функции обнаружения сети очень важны для злоумышленника, которому необходимо понять топологию сети после ее первоначального взлома. Но Nmap предлагает гораздо больше, чем это: в него интегрирован движок сценариев, **многие из которых предназначены для обнаружения уязвимостей**.



Например, с помощью этой команды можно проверить, разрешает ли служба FTP анонимное подключение, без необходимости подключаться вручную:



```
nmap --script ftp-anon -p 21 192.168.1.18
```



![nmap-image](assets/fr/04.webp)



*Использование сценария NSE для проверки анонимной FTP-аутентификации через Nmap.*



Обнаружение уязвимостей с помощью Nmap - один из первых этапов аудита или теста на проникновение. Оно позволяет быстро определить слабые места и оптимизировать усилия по анализу.



Но будьте осторожны: **инструменты сканирования уязвимостей имеют свои пределы**. Nmap охватывает лишь часть угроз и не гарантирует, что система безопасна, если в ней не обнаружено никаких проблем. Поэтому важно **понимать, как работают используемые скрипты**, и не полагаться только на их вердикт.



### VI. Заключение



Мы убедились, что освоение Nmap может охватить широкий спектр областей применения, от диагностики и мониторинга до картографирования, оценки политик безопасности и сканирования уязвимостей. В следующем разделе мы перейдем к деталям и установим Nmap.

### IV. Локальные привилегии, необходимые для использования Nmap



Но, кстати, при использовании Nmap **необходимо ли иметь повышенные локальные привилегии в системе?** Ответ: **это зависит**.



В самом базовом виде, т.е. не заходя далеко в использовании своих опций, Nmap не обязательно требует привилегированных прав. Однако вскоре вы поймете, что лучше использовать Nmap в привилегированном контексте ("root" в Linux, "administrator" или эквивалент в Windows), чтобы иметь возможность использовать его в полной мере, рискуя получить сообщение об ошибке, подобное этому:



![nmap-image](assets/fr/10.webp)



сообщение об ошибке в Linux, когда опции Nmap требуют прав root._



Будь то Linux или Windows, существует множество случаев, когда Nmap запросит у вас привилегированный доступ. Основные причины следующие (список неполный):





- Конструирование "сырых" сетевых пакетов**: Nmap способен использовать широкий спектр методов сканирования, включая продвинутые манипуляции с пакетами и их построение. Так, например, происходит, когда мы хотим выполнить сканирование TCP SYN, в котором не соблюдается классическое _трехстороннее рукопожатие_ при обмене TCP. Для этого Nmap должен использовать функции, отличные от тех, которые являются родными для операционных систем и которые знают только о том, как уважать хорошую практику сетевых коммуникаций (он обращается к библиотекам "Npcap" и "libcap", о которых говорилось выше). Именно благодаря тому, что Nmap делает все не так, как принято, ему удается получить определенную информацию об ОС, сервисах и некоторых уязвимостях.





- Прослушивание сетевого трафика**: некоторые опции Nmap требуют прослушивания сети для получения определенной информации. Это действие считается чувствительным в операционных системах, поскольку позволяет прослушивать коммуникации других приложений в системе. Как и Wireshark, Nmap нуждается в определенных привилегиях, которые легче получить, находясь непосредственно в привилегированной сессии.





- Прослушивание на привилегированных портах**: в операционных системах порты от 0 до 1024 (TCP и UDP) считаются привилегированными, то есть зарезервированными для очень специфического использования и поэтому защищенными. Хотя сегодня эта причина несколько устарела, для прослушивания этих портов по-прежнему необходимо иметь определенные привилегии, что может потребоваться Nmap в зависимости от того, как он будет использоваться.





- Отправка UDP-пакетов:** Аналогично, прослушивание сетевых приложений на UDP-портах (протокол без статического состояния) требует привилегированных прав в операционных системах. Поэтому для выполнения UDP-сканирования, для которого Nmap должен будет прослушать ответ, чтобы проанализировать ответы на свои сканирования, потребуется привилегированная сессия.




Если быть более точным, то, по крайней мере, в Linux, можно запустить Nmap со всеми его функциями и опциями, не являясь при этом root'ом. Это достигается путем предоставления бинарному файлу нужных _возможностей_. Однако это требует сложных манипуляций и может быть не так практично, как запуск Nmap напрямую с привилегиями. Кроме того, этот подход менее распространен и при неправильной настройке может вызвать проблемы с безопасностью.



Однако это немного отходит от нашего учебника по Nmap, поэтому мы пока обойдемся без этого.



В остальной части этого руководства считайте, что Nmap всегда запускается от имени "root" (из оболочки от имени "root" или с помощью команды "sudo") или в терминале администратора под Windows, даже если это не указано. В противном случае вы можете ощутить тонкие, но заметные отличия от данного руководства.



### V. Заключение



Вот и все! Теперь Nmap установлен в нашей операционной системе и готов к работе, не требуя дополнительной настройки. Этот раздел завершает знакомство с Nmap и его презентацию. Я надеюсь, что у вас от него развязались рты, и вам не терпится приступить к практике.



Если говорить более серьезно, то теперь мы лучше представляем себе, что такое инструмент Nmap mapping, каковы его наиболее распространенные применения, а также его ограничения. Давайте двигаться дальше!

### II. Сканирование TCP-портов хоста с помощью Nmap



#### A. Первое сканирование TCP-портов с помощью Nmap



Сейчас мы выполним наше первое сканирование с помощью Nmap. Наша цель проста: мы хотим выяснить, какие службы открыты веб-сервером, который мы только что развернули, чтобы увидеть, нет ли чего-нибудь неожиданного, например, службы администрирования, которая не должна быть доступна, или открытие порта приложения, которое, как мы думали, было выведено из эксплуатации.



В моем примере сканируемый хост имеет IP Address "192.168.1.18":



```
nmap 192.168.1.18
```



Вот возможный результат. Мы видим классическое возвращение Nmap с большим количеством информации:



![nmap-image](assets/fr/11.webp)



результаты простого сканирования TCP, выполненного с помощью Nmap._



Бегло взглянув на этот результат, мы понимаем, что порты TCP/22 и TCP/80 доступны на этом хосте.



По умолчанию, если вы не укажете, Nmap будет сканировать только TCP-порты.



#### B. Понимание результатов простого сканирования Nmap



Однако давайте сделаем еще один шаг вперед, чтобы понять этот вывод: каждая строка важна, во-первых, чтобы понять, что только что было сделано, а во-вторых, чтобы правильно интерпретировать результаты сканирования.



Первая строка - это напоминание о версии и дате сканирования (в конце концов, это полезно для ведения журнала и архивации!):



```
Starting Nmap 7.94SVN ( https://nmap.org ) at 2024-04-29 21:59 CEST
```



Вторая строка показывает результаты начала сканирования для хоста "debian.home (192.168.1.19)". Эта информация будет полезна, когда мы начнем сканировать несколько хостов одновременно:



```
Nmap scan report for debian.home (192.168.1.19)
```



Третья строка сообщает нам, что рассматриваемый хост "Up", то есть активен:



```
Host is up (0.00022s latency).
```



Наконец, Nmap сообщает нам, что 998 TCP-портов, идентифицированных как закрытые, не отображаются в окне:



```
Not shown: 998 closed tcp ports (conn-refused)
```



Это избавляет нас от почти 1 000 строк вывода, выглядящих как:



```
1/tcp closed
2/tcp closed
3/tcp closed
…
```



Спасибо ему за то, что избавил нас от этого!



Почему 998 "закрытых" портов? Если добавить 2 открытых порта, то получится 1000, и это количество портов, которые Nmap будет сканировать в своей конфигурации по умолчанию, а не 65535 существующих TCP-портов! Позже мы увидим, что это можно легко и просто настроить. Но если на целевом узле есть служба, прослушивающая довольно экзотический порт, такое сканирование "по умолчанию" ее не обнаружит.



После этой информации мы находим самое интересное: таблицу, организованную в соответствии с тремя колонками "PORT - STATE - SERVICE":





- Первый столбец "PORT" просто указывает на целевой порт/протокол, и здесь важно обратить внимание на то, является ли это TCP-портом (`<port>/tcp`) или UDP (`<port>/udp`).





- В столбце "STATE" указывается статус сетевой службы, обнаруженной на этом порту и определенной Nmap на основе полученного ответа. Это может быть "открыто", "закрыто", "отфильтровано" или "неизвестно". Позже мы увидим, как Nmap различает эти состояния.





- В столбце "SERVICE" указана служба, открытая на данном порту. Обратите внимание, что здесь мы не использовали активные опции обнаружения сервисов. Nmap полагается на локальную ссылку между портом/протоколом и предположительно назначенным сервисом: файл "/etc/services"




Если вы посмотрите на файл "/etc/services" в системе Linux, вы найдете ссылку "port/protocol - service", похожую на ту, которую отображает Nmap:



![nmap-image](assets/fr/12.webp)



извлекает содержимое файла "/etc/services" в Linux._



Важно понимать, что на данный момент Nmap не выполнял активного обнаружения сервисов. Например, в этом случае он не смог бы обнаружить службу SSH за портом TCP/80. Отсюда следует, что важно знать, как использовать правильные опции - это уже скоро!



Знание того, как интерпретировать результаты работы Nmap, является важной частью освоения этого инструмента. Хорошая новость заключается в том, что эти результаты будут в основном одинаковыми, какие бы опции вы ни использовали.



#### C. Под капотом: анализ сети с помощью Wireshark



Если вы внимательно посмотрите на то, что происходит в сети Interface хоста, сканирующего сервер, или на самом сервере, действия Nmap станут намного понятнее. Именно этим мы и займемся.



То, что я могу показать вам здесь, - лишь часть того, что видно в Wireshark. Если вы хотите пойти дальше, не стесняйтесь сами сделать захват сети во время сканирования, а затем просмотреть то, что было захвачено.



В этом тесте мой сканирующий хост (192.168.1.18) и целевой хост (192.168.1.19) находятся в одной локальной сети.



Nmap начинает с того, что выясняет, активен ли целевой хост в локальной сети, отправляя ARP-запрос. Если он получает ответ, он знает, что хост активен, и начинает сканирование сети:



![nmap-image](assets/fr/13.webp)



_ARP-запрос, выдаваемый Nmap для определения наличия целевого узла в локальной сети._



Если сканируемый хост находится в удаленной сети, Nmap начинает с отправки ping-запроса и пытается связаться с некоторыми из наиболее часто открываемых портов (TCP/80, TCP/443):



![nmap-image](assets/fr/14.webp)



ping-запрос, выдаваемый Nmap для определения доступности целевого узла в удаленной сети



Если он получает ответ на любой из этих тестов, то считает цель активной.



После того как Nmap определит, что цель активна, он попытается разрешить ее доменное имя с помощью DNS-сервера, настроенного на сетевой карте:



![nmap-image](assets/fr/15.webp)



разрешение DNS на цели сканирования Nmap



Теперь, когда Nmap определил свою цель и знает, что она активна, он начинает сканирование TCP-портов:



![nmap-image](assets/fr/16.webp)



передача пакета TCP SYN и прием RST/ACK во время сканирования Nmap



Для этого на каждый TCP-порт в диапазоне по умолчанию он **отправляет пакеты TCP SYN и ждет ответа**. На скриншоте выше он получает от сканируемого сервера пакеты TCP RST/ACK, означающие "проезжайте, здесь нечего смотреть" - другими словами, эти порты закрыты. Как мы видим из результата, так будет с большинством просканированных портов. За двумя исключениями:



![nmap-image](assets/fr/17.webp)



ответ на пакет TCP SYN, отправленный на порт 22, активный на объекте сканирования



На скриншоте выше мы видим пакет TCP SYN/ACK, отправленный целевым хостом**. Порт активен и предоставляет сервис. Nmap подтверждает получение ответа, а затем разрывает соединение (TCP RST/ACK). **Таким образом он узнал, что порт TCP/22 активен**.



Здесь мы увидели, что Nmap соблюдает "трехстороннее рукопожатие" при сканировании TCP-сети. По соображениям производительности можно попросить его не отвечать на ответ сервера, что позволит сэкономить несколько тысяч пакетов при сканировании большой сети. Но мы рассмотрим эти возможности и оптимизации позже в учебнике.



Теперь мы лучше представляем себе, как выполнять сканирование TCP и что на самом деле происходит при его выполнении. Мы также увидели, что по умолчанию Nmap выполняет сканирование TCP-портов на ограниченном количестве портов.



### III. Сканирование UDP-портов с помощью Nmap



#### A. Первое сканирование портов UDP с помощью Nmap



Теперь давайте посмотрим, как сканировать UDP-порты хоста. Как мы уже видели, по умолчанию Nmap всегда сканирует TCP-порты. Это может означать, что мы упускаем много информации, если не знаем об этом.



Напоминаю, что для целей этого теста мой сканирующий хост (192.168.1.18) и целевой хост (192.168.1.19) находятся в одной локальной сети.



```
nmap -sU 192.168.1.19
```



Здесь полученный результат имеет тот же формат, что и при сканировании TCP, но активные сервисы отображаются в формате `<порт>/UDP`, как и было запрошено!



![nmap-image](assets/fr/18.webp)



результат простого UDP-сканирования, выполненного с помощью Nmap._



Опция "-sU" используется для указания Nmap, что вы хотите работать с UDP, а не с TCP, как по умолчанию.



Кстати, вы наверняка заметили, что Nmap требует права "root" для UDP-сканирования, как уже упоминалось ранее в руководстве.



примечание: Начиная с последних версий Nmap, для получения достоверных результатов рекомендуется выполнять UDP-сканирование с правами администратора, поскольку некоторые функции требуют необработанного доступа к сетевым сокетам._



Сканирование UDP может занять очень много времени (1100 секунд для сканирования 1000 портов в моем примере) из-за отсутствия "трехстороннего рукопожатия" в UDP, что означает, что Nmap будет ждать ответа на каждый отправленный UDP-пакет и определит порт как "закрытый" только в случае отсутствия ответа через определенное время (таймаут). Этот ответ от сканируемых узлов не является систематическим и часто ограничен по количеству ответов в секунду, чтобы избежать некоторых атак усиления. В отличие от TCP, где ответ от сканируемого узла поступает немедленно, независимо от того, открыт порт или закрыт. Позже мы увидим, как это оптимизировать.



Вторая трудность с UDP заключается в том, что **службы не всегда отвечают на входящие пакеты**, просто потому, что это не всегда необходимо и таков принцип работы UDP. Когда это происходит, и не получено ICMP "port unreachable", служба помечается Nmap как "open|filtered", как показано на скриншоте выше.



#### B. Под капотом: анализ сети с помощью Wireshark



Как и в случае с TCP-сканированием, давайте подробнее рассмотрим, что происходит на сетевом уровне во время UDP-сканирования с помощью анализа Wireshark. Поведение Nmap при определении активности хоста такое же.



Единственное реальное отличие при сканировании UDP заключается в том, что Nmap не будет ждать "трехстороннего рукопожатия", поскольку этот механизм не существует в UDP (протокол без статического состояния):



![nmap-image](assets/fr/19.webp)



передача пакетов uDP и прием ICMP (порт недоступен) во время сканирования Nmap



На скриншоте выше видно, что Nmap отправляет большое количество UDP-пакетов и получает в ответ на большинство из них пакет ICMP "Destination unreachable (Port unreachable)". Это нормально, так как это соответствующий ответ, определенный в [RFC 1122] (https://www.freesoft.org/CIE/RFC/1122/41.htm "RFC 1122"), когда UDP-порт недостижим:



![nmap-image](assets/fr/20.webp)



выдержка из RFC 1122._



Давайте посмотрим на этот снимок Wireshark, который показывает **три возможных сценария** в UDP:



![nmap-image](assets/fr/21.webp)



захват сети во время сканирования UDP на разных портах с помощью Nmap._



Три случая следующие:





- Первый Exchange состоит из пакетов №№. 3, 4 и 8, 9. Nmap отправляет UDP-пакеты на классический порт SNMP и поэтому **конструирует пакеты в соответствии с протоколом заранее**. Затем он получает ответ от сервера (пакеты № 8 и 9). Результат: Nmap получил ответ, служба "открыта".





- Второй Exchange состоит из пакетов 6 и 7. Nmap отправляет "пустой" UDP-пакет (без структуры протокола) на порт UDP/165 и получает в ответ ICMP-пакет: "Destination unreachable (Port unreachable)". Результат: Nmap получил (отрицательный) ответ, хост работает, но служба, к которой он пытался обратиться, не работает на этом порту, который будет помечен как "закрытый".





- Последний Exchange состоит из пакета № 12: Nmap отправляет "пустой" UDP-пакет на порт UDP/1235. Никакого ответа, даже явного отказа от сканируемого узла, нет. Результат: Nmap помечает порт как "открытый|фильтрованный", поскольку не может определить, связано ли это с наличием брандмауэра, настроенного на отсутствие ответа, или с активной службой UDP, которая все равно не отвечает (в UDP это не обязательно).




Вот результат, показанный Nmap в этих трех случаях:



![nmap-image](assets/fr/22.webp)



возможные результаты UDP-сканирования, выполненного с помощью Nmap._



Теперь мы лучше представляем себе, как выполнять UDP-сканирование и что на самом деле происходит при его выполнении. До сих пор мы использовали Nmap очень простым способом, не решая, какие порты сканировать, но теперь все изменится!



### IV. Тонкая настройка сканирования портов с помощью Nmap



#### A. Напоминание о поведении Nmap по умолчанию



Как мы уже видели, Nmap сам выбирает количество и порты для сканирования, если вы не указываете никаких опций. Это конфигурация "по умолчанию", используемая Nmap, когда вы не указываете ему, что именно делать. Эти параметры по умолчанию предназначены для того, чтобы дать представление об основных открытых портах, которые выбираются по частоте использования (наиболее распространенные или частые порты), а не в числовом порядке (порт 1, 2, 3 и т. д.), а также для того, чтобы не начинать сканирование 65535 возможных портов, если вы не укажете соответствующие параметры, что было бы слишком долго и многословно для случая использования "по умолчанию".



**Как выбираются эти порты?



1000 портов, сканируемых в режиме по умолчанию, выбираются в зависимости от частоты их появления. Эта статистика ведется Nmap и обновляется так же, как и сам бинарник и его скрипты (модули). Вы можете самостоятельно просмотреть эту статистику в файле "/usr/shares/nmap/nmap-services":



![nmap-image](assets/fr/23.webp)



извлекается из файла "/usr/shares/nmap/nmap-services"._



Здесь, в третьем столбце, мы видим то, что выглядит как вероятности (от 0 до 1) или процентное распределение. Это частота встречаемости каждой пары порт/протокол. Видно, что самые известные порты (FTP, SSH, TELNET и SMTP в данном примере) имеют гораздо большее значение, чем остальные.



#### B. Точное указание целевых портов для сканирования Nmap



Однако в реальном мире нам может понадобиться просканировать только определенный порт, или несколько портов, или определенный диапазон портов. Nmap позволяет легко сделать именно это, причем единообразно для сканирования как UDP, так и TCP.



**Сканировать определенный порт с помощью Nmap**



Если мы хотим просканировать только один порт, а не 1000, мы можем указать этот порт с помощью опции Nmap "-p" или "--port":



```
# Scan a single TCP port with Nmap
nmap 192.168.1.19 -p 80

# Scan a single UDP port with Nmap
nmap -sU 192.168.1.19 -p 161
```



В результате сканирование будет происходить гораздо быстрее, а Nmap будет испускать только пакеты, необходимые для определения активности хоста, а затем - доступности указанного порта. Это экономит время, если вам нужно провести быстрый тест, чтобы проверить, работает ли веб-сервис на вашем сайте-витрине.



**Сканируйте несколько портов с помощью Nmap**



Таким же образом мы можем указать Nmap несколько портов, используя одну и ту же опцию и объединяя указанные порты запятой:



```

# Scan several TCP ports with Nmap

nmap 192.168.1.19 -p 80,10999,22,23,1345

# Scan several UDP ports with Nmap

nmap -sU 192.168.1.19 -p 161,23,69

```



Независимо от порядка, Nmap проверит все эти порты, и только те, которые находятся на целевом хосте. Вы заметите в выводе Nmap, что он **явно сообщает нам обо всех портах и их статусе**, даже если они "закрыты". В отличие от поведения по умолчанию, где полный вывод занял бы слишком много места:



![nmap-image](assets/fr/24.webp)



*Результат сканирования Nmap TCP на указанных портах.*



** Сканирование ряда портов



Если количество сканируемых портов слишком велико, можно указать их по диапазону, например:



```

# Scan TCP ports from 1000 to 2000 with Nmap

nmap 192.168.1.19 -p 1000-2000

# Scan UDP ports from 1000 to 2000 with Nmap

nmap -sU 192.168.1.19 -p 100-150

```



Конечно, вы можете смешивать и сочетать их по своему усмотрению, например:



```

# Scan TCP ports 22,80, 3389 and from 1000 to 2000 with Nmap

nmap 192.168.1.19 -p 22,80,1000-2000,3389

```



** Сканирование портов TCP и UDP



Вы также можете одновременно выполнять сканирование UDP и TCP на выбранных портах:



```

# Scan the list of 1000 default ports, in TCP and UDP

sudo nmap 192.168.1.19 -sT -sU

# Scan only UDP/161 and TCP/22

sudo nmap 192.168.1.19 -sT -sU -p U:161,T:22

```



В последнем примере вы заметите присутствие "U:" для обозначения UDP-порта и "T:" для обозначения TCP-порта. Вот возможный результат этого типа сканирования:



![nmap-image](assets/fr/25.webp)



*Результат сканирования TCP и UDP портов с помощью Nmap.*



Вот это интересный способ персонализации сканов!



** Просканируйте все порты



Наконец, можно указать Nmap гораздо большие или меньшие диапазоны. Мы видели, что список, выбранный Nmap по умолчанию, содержит 1000 портов. Мы также можем запросить 100 наиболее часто встречающихся портов или 200, используя опцию "--top-ports":



```

# Scan the top 100 most common ports with Nmap

nmap 192.168.1.19 --top-ports 100

# Scan the top 200 most common ports with Nmap

nmap 192.168.1.19 --top-ports 200

```



Наконец, мы можем попросить его просканировать все возможные порты (все 65535), используя обозначение "-p-":



```

# Scan all TCP ports from 1 to 65535 with Nmap

nmap 192.168.1.19 -p-

```



Последний способ займет больше времени, особенно в случае с UDP, но вы будете уверены, что не пропустите ни одного открытого порта.



*Примечание: Опция "-p-" рекомендуется для сканирования всех TCP-портов. Для сканирования UDP рекомендуется ограничить количество портов по соображениям производительности, так как полное сканирование всех UDP-портов может занять очень много времени.*



Позже мы рассмотрим, как оптимизировать скорость сканирования Nmap под наши нужды, что будет особенно полезно при сканировании всех TCP и UDP портов.

Мы также подробно рассмотрели, что происходит на сетевом уровне и **как Nmap определяет, активен ли TCP или UDP порт или нет**. ## 5 - Картирование и обнаружение сетей с помощью Nmap



### I. Презентация



В этом разделе мы узнаем, как использовать инструмент сканирования сети Nmap для создания карты сети. Мы увидим, насколько эффективным он может быть в этой задаче, используя различные опции. Наконец, мы рассмотрим различные способы указания целей сканирования в Nmap.



В частности, мы будем использовать то, что узнали в предыдущем разделе о том, как Nmap определяет, является ли хост активным и достижимым.



Как уже говорилось во введении к Nmap, это сетевой маппер. Как таковой, это идеальный инструмент для составления списка доступных хостов в сети, как локальной, так и удаленной.



**Возвращение автора:*



Как аудитор кибербезопасности и пентестер, я систематически использую Nmap при проведении внутренних тестов на проникновение, чтобы узнать, где я нахожусь, кто мои соседи в локальной сети и какие другие сети доступны, а также системы, расположенные в них. Моя цель проста: составить карту сети, определить размер информационной системы и, в частности, очертить ее поверхность атаки.



Картирование сети также может быть полезным в контексте диагностики сети, наблюдения, составления карты активов (вы действительно уверены, что ваша ИБ состоит только из того, что есть в Active Directory или в вашей GLPI/OCS Inventory? Его также можно использовать для обнаружения присутствия теневых ИТ в вашей информационной системе.



### II. Использование Nmap для сканирования сетевого диапазона



#### A. Обнаружение сети с помощью сканирования Nmap



Теперь мы хотим перейти на более высокую ступень и проанализировать всю нашу локальную сеть. Нет ничего проще: все, что нам нужно сделать, это повторить команды, которые мы видели в предыдущем разделе, но указать CIDR вместо простого IP Address.



CIDR (**Classless Inter Domain Routing**) - это "классическая" нотация для указания диапазона сетей и их протяженности (с помощью маски). Например, "192.168.0.0/24" - это "перевод" десятичной нотации маски "255.255.255.0".



Чтобы использовать Nmap, указав CIDR, мы можем использовать его следующим образом:



```
# Scan a CIDR
nmap 192.168.0.0/24
```



Также, как и в случае с портами в предыдущем разделе, можно указать несколько хостов, несколько сетей или диапазон:



```
# Scan several networks at once via their CIDR
nmap 192.168.0.0/24 192.168.1.0/24

# Scan several hosts via their IP
nmap 192.168.1.2 192.168.1.3 192.168.1.10-20

# Mix of both
nmap 192.168.0.0/24 192.168.1.3 192.168.1.10-20
```



Вот пример результатов, которые мы можем получить при сканировании сети:



![nmap-image](assets/fr/26.webp)



результаты сканирования Nmap для создания карты нескольких сетей



В частности, мы видим несколько активных хостов, и каждый раздел хоста начинается с такой строки:



```

```



Это позволяет нам четко определить, к какому хосту относятся следующие результаты. Последняя строка также важна:



```

```



Мы знаем, что в просканированных сетях Nmap обнаружил 5 активных узлов.



#### B. Под капотом: анализ сети с помощью Wireshark



Сейчас мы подробнее рассмотрим, что происходит на сетевом уровне во время обнаружения сети с помощью Nmap.



Как мы видели в предыдущем разделе, по умолчанию Nmap использует протокол ARP для обнаружения присутствия хостов в локальной сети:



![nmap-image](assets/fr/27.webp)



пакеты aRP, захваченные при сканировании локальной сети с помощью Nmap и его опций по умолчанию



Таким образом, он способен обнаружить практически все узлы в локальной сети, поскольку ответ на ARP-запрос обычно дают все активные в сети узлы, и он никак не выглядит подозрительным.



Для работы с удаленными сетями Nmap использует комбинацию методов:



![nmap-image](assets/fr/28.webp)



перехват iCMP и TCP-пакетов при сканировании удаленной сети с помощью Nmap и его опций по умолчанию



Если быть более точным, Nmap использует ICMP-эхо-пакет (классический случай pinging) и ICMP Timestamp-пакет, обычно используемый для расчета времени прохождения пакетов. Он надеется получить ответ от узлов в удаленных сетях.



Но это еще не все. На приведенном выше снимке Wireshark видно, что пакеты **TCP SYN** систематически отправляются на порты TCP/443 всех потенциальных хостов в сканируемых сетях, а также пакеты **TCP ACK** на порт TCP/80.



**Зачем отправлять TCP-пакеты на порты в рамках обнаружения сети?



Отправка SYN-пакета на заданный порт позволяет Nmap **определить, прослушивается ли служба на этом порту**. Если хост отвечает на SYN-пакет пакетом SYN/ACK, это означает, что он активен и что служба прослушивает этот порт. Поэтому Nmap пытается попытать счастья на этом сервисе, **даже если ответа на пинг не получено**.



Отправка ACK-пакета на заданный порт позволяет Nmap **определить, присутствует ли на этом хосте брандмауэр**. Если хост отвечает на ACK-пакет пакетом RST (Reset), это указывает на то, что на этом хосте, вероятно, присутствует брандмауэр, который блокирует незапрашиваемый трафик. Таким образом, хост выдает свое присутствие в сети, даже если он не отвечает на другие запросы.



Однако важно отметить, что обнаружение брандмауэра с помощью этой техники может быть не во всех случаях абсолютно надежным. Некоторые узлы могут не принимать ответы generate RST по причинам, не связанным с наличием брандмауэра, например, из-за специфических настроек службы или операционной системы. Кроме того, современные брандмауэры могут быть настроены на то, чтобы не отвечать на попытки обнаружения такого типа.



Теперь мы проделали долгий путь и можем выполнять базовое обнаружение сети. Теперь мы рассмотрим еще несколько опций, которые дадут нам больше контроля над поведением Nmap.



### III. Обнаружение сети без сканирования портов с помощью Nmap



Как вы могли заметить, по умолчанию Nmap **проводит сканирование портов после обнаружения активного хоста**, что добавляет огромное количество пакетов и ожидание ответов на наше сканирование. Если в вашей сети 5 хостов, Nmap попытается проверить состояние около 5 000 портов, что займет больше времени.



Однако с помощью опций Nmap можно выполнить **только обнаружение активных хостов** в сети, не обнаруживая их сервисы.



Если мы хотим знать только, какие хосты доступны, без какой-либо информации о сервисах и портах, которые они открывают, мы можем использовать опцию "-sn", чтобы выполнить **только сканирование с помощью ICMP Echo (ping) и ARP-запросов**. Другими словами, полностью отключить сканирование портов:



```
# Scan a CIDR in Echo ICMP
nmap 192.168.1.0/24 -sn
```



Вот результат обнаружения сети Nmap, выполненного без сканирования портов:



![nmap-image](assets/fr/29.webp)



Результат обнаружения сети без сканирования портов с помощью Nmap.



Мы уже упоминали о возможных ограничениях использования только ICMP для обнаружения хостов (для удаленных сетей). Поэтому Nmap также использует несколько трюков, которые могут выдать наличие брандмауэра или определенной службы на хостах. С помощью опций мы можем повторно использовать эти трюки и даже расширить их, не прибегая к полному сканированию портов каждого обнаруженного хоста.



Для этого мы будем использовать опции "-PS" (TCP SYN) и "-PA" (TCP ACK), которые позволят нам указать порты, к которым мы хотим присоединиться в процессе обнаружения хоста, а также опцию "-PP":



```
# Scan specific ports on a CIDR
nmap -sn -PP -PS22,3389,445,139 -PA80 192.168.1.0/24
```



Это сканирование обеспечивает более полное обнаружение хоста, чем при использовании параметров по умолчанию.



Мы начинаем получать достаточно полные команды, использующие множество опций. Это происходит потому, что мы знаем, как работает Nmap, и ограничения его опций "по умолчанию", которые иногда могут заставить нас потерять время или пропустить важные Elements. В этом и заключается смысл того, чтобы потратить время на его освоение!



Чтобы детализировать варианты нашего последнего заказа:





- "`-sn`: отключает сканирование портов для каждого активного узла, обнаруженного Nmap.





- "`-PP`": включает ICMP-эхо (ping-сканирование) для обнаружения хоста.





- "`-PS <PORT>`": отправьте TCP SYN-пакет на указанный порт (порты), чтобы обнаружить любой активный сервис, выдающий присутствие хоста, который не ответил на ping.





- "`-PA <PORT>`": отправьте TCP ACK-пакет на указанный порт (порты), чтобы обнаружить любой активный брандмауэр, выдающий присутствие хоста, который не ответил на ping.




В примере выше я указал порты, которые, по моему мнению, наиболее часто открываются в контекстах Nmap, для опции "-PS". Затем эти различные порты будут протестированы на каждом хосте, но не для того, чтобы проверить, действительно ли активна служба, на которой они размещены, а для того, чтобы выяснить, позволяет ли это обнаружить хост, который не ответил на наш ICMP Echo, оставаясь при этом активным (через ответ от службы или брандмауэра хоста).



Вот что можно увидеть на снимке сети, сделанном во время такого сканирования, в данном случае на одном целевом узле:



![nmap-image](assets/fr/30.webp)



пакеты, отправленные Nmap во время расширенного обнаружения сети, без сканирования портов



Мы находим наши пакеты TCP SYN, TCP ACK на порту TCP/80 и ICMP echo. Nmap выполнит все эти тесты для каждого узла, на который направлено наше сканирование на обнаружение сети.



### IV. Использование файла с активами для нацеливания с помощью Nmap



В реальных информационных системах, состоящих из десятков и сотен сетей, подсетей и виртуальных локальных сетей, определение целей может быстро оказаться сложным. Поэтому проще использовать файл в качестве источника для Nmap, чем указывать их по одному в командной строке.



Для начала создайте простой файл, содержащий по одной записи в строке:



![nmap-image](assets/fr/31.webp)

```
# Scan a list of targets contained in a file
```

Последний важный совет, которым я хотел бы с вами поделиться, касается **исключения хоста или сети в процессе сканирования**. Необходимость исключения хоста может потребоваться в ряде случаев, особенно если мы хотим быть уверены, что **чувствительный компонент информационной системы не будет нарушен или выведен из строя в результате сканирования**.



Частыми примерами таких потребностей являются случаи, когда компания владеет промышленным (PLC) или медицинским оборудованием. Такое оборудование иногда плохо спроектировано и совсем не предназначено для приема плохо отформатированных пакетов или слишком большого их количества. По очевидным причинам, связанным с доступностью или риском для бизнеса и людей, предпочтительнее исключить их из тестирования.



Чтобы исключить IP-адреса или сети из нашего сканирования, мы можем использовать опцию Nmap "--exclude", например:



```
# Exclude an IP address in a CIDR scan
nmap 192.168.1.0/24 --exclude 192.168.1.140
```



В этом примере я сканирую сеть "192.168.1.0/24", но исключаю расположенный в ней хост "192.168.1.140". Nmap не будет отправлять пакеты на этот хост. Еще один пример с исключением из подсети:



```
# Exclude a subnet in a CIDR scan
nmap 10.0.0.0/16 --exclude 10.0.100.0/24
```



Аналогично, я сканирую большую сеть "10.0.0.0/16", но сеть "10.0.100.0/24" сканироваться не будет. Опять же, я рекомендую использовать опцию "-sL -n", чтобы получить очень четкое представление о том, какие хосты будут просканированы, а какие исключены из сканирования, особенно если вы работаете в конфиденциальном контексте.



### V. Обнаружение сети и сканирование портов



Теперь мы можем объединить то, что мы узнали в этом разделе, с тем, что мы узнали в предыдущем разделе о параметрах сканирования портов. По умолчанию, как мы видели, Nmap будет сканировать 1000 наиболее часто используемых портов на каждом активном узле, который он обнаружит. Мы видели, как предотвратить это поведение, если оно нам не нужно, но мы можем полностью контролировать его и даже расширить, если это соответствует нашим потребностям.



Например, следующая команда проверит наличие прослушивающей службы на порту TCP/22 на каждом сканируемом узле:



```
# Scan TCP/22 on a CIDR
nmap 192.168.0.0/24 192.168.1.0/24 -p 22
```



Сначала Nmap выполнит обнаружение сети, чтобы составить список активных узлов, и для каждого из них проверит наличие службы на порту TCP/22.



Таким же образом мы можем выполнить полное сканирование всех TCP-портов на каждом хосте, обнаруженном в сети "192.168.0.0/24", исключая, например, хост "192.168.0.4":



```
# Port scan of a CIDR with exclusion
nmap 192.168.0.0/24 --exclude 192.168.0.4 -p-
```



Вы можете комбинировать все варианты, которые мы изучили до сих пор, в соответствии со своими потребностями.



### VI. Заключение



В этом разделе мы рассмотрели, как использовать Nmap для создания карты сети с помощью различных опций. Теперь мы хорошо понимаем цели нашего сканирования, а также поведение Nmap при сканировании портов и метод обнаружения хостов. И самое главное, мы знаем, как Nmap ведет себя по умолчанию и каковы его ограничения, а также как использовать его основные опции, чтобы пойти дальше.



В следующем разделе мы рассмотрим механизмы и опции для обнаружения версий сервисов и операционных систем, сканируемых Nmap.




## 6 - Определение версий служб и операционных систем с помощью Nmap



### I. Презентация



В этом разделе мы узнаем, как использовать Nmap для обнаружения и точного определения версий служб и операционных систем, используемых сканируемыми узлами. Мы подробно рассмотрим, как Nmap решает эту задачу, а также ограничения инструмента, чтобы лучше понять и интерпретировать его результаты.



Как мы уже видели в предыдущих разделах этого руководства, по умолчанию Nmap не смотрит, какой сервис открыт на портах, которые он сканирует и считает открытыми. Так что если вы слушаете веб-службу на порту TCP/22, Nmap продолжит сообщать о ней как об открытой, но как о службе `SSH`. Это происходит потому, что для поиска связи между портом/протоколом и именем службы (файл `/etc/services/`) используется локальная для вашей системы [база данных](https://www.it-connect.fr/cours-tutoriels/administration-systemes/stockage/bdd/).



В большинстве случаев [Nmap](https://www.it-connect.fr/cours/nmap-cartographie-reseau-scan-de-vulnerabilites/) предоставит вам правильную информацию, поскольку в производственной среде такие случаи встречаются редко. Однако в остальных случаях будут возникать ситуации, когда классический сервис ([SSH](https://www.it-connect.fr/cours/comprendre-et-maitriser-ssh/), HTTP и т. д.) открывается на неклассическом порту (например, 2022 для сервиса SSH), и в этом случае Nmap не найдет соответствия в своей локальной базе данных или найдет несоответствие в реальности, и вы упустите важную информацию.



К счастью, Nmap предлагает очень точные опции и механизмы для обнаружения того, какой именно сервис может скрываться за открытым портом. У него даже есть база данных запросов и сигнатур для определения точных технологий и версий. Помимо служб, Nmap также может определить используемую технологию и ее версию.



Именно это мы и рассмотрим в данном разделе.



### II. Как обнаружить технологию или версию



#### A. Напоминание о том, как идентифицировать технологию или версию



Идентификация технологии и версии включает в себя поиск имени сервиса, CMS, приложения или программного обеспечения, прослушивающего целевой порт. Например, веб-страница управляется CMS (`WordPress`), запускается веб-службой (`Apache`, IIS, Nginx) и размещается на сервере (Linux или Windows). Но как узнать, какой веб-сервис запущен?



Классической методикой получения этой информации является _баннерный захват_, который заключается в том, чтобы найти место, где служба отображает эту информацию, и прочитать данные. Очень часто в стандартной конфигурации или обработке сервисы отображают свое имя и даже версию в качестве первого ответа после подключения.



![nmap-image](assets/fr/32.webp)



отображать версию, как только служба FTP установит TCP-соединение



Здесь мы видим, что простое TCP-соединение с этой службой через `telnet` приводит к TCP-пакету, содержащему ее технологию и версию.



Получив представление о типе службы, с которой вы имеете дело, вы также можете отправлять ей определенные команды или запросы, чтобы получить от нее информацию. Эти запросы/команды можно отправлять и вслепую (не будучи уверенным, что они относятся к нужному типу служб), в надежде, что один из них вызовет ответ от соответствующей службы.



В других, более сложных случаях, необходимо отправить определенные пакеты, чтобы вызвать реакцию, например ошибку, которая предоставит подробную информацию об используемой версии или технологии.



Как вы можете себе представить, Nmap будет использовать все эти методы, чтобы попытаться определить точный тип сервиса, размещенного на порту, а также название его технологии и версию.



#### B. Понимание зондов и совпадений Nmap



Для выполнения всех этих проверок на каждом сканируемом порту Nmap использует локальную базу данных, которая часто обновляется (как и бинарники или их модули). Это текстовый файл из нескольких тысяч строк: `/usr/share/nmap/nmap-service-probes`.



Этот файл состоит из множества записей, организованных по двум основным принципам:





- `Probe`: это определение пакета, который Nmap отправит в попытке спровоцировать реакцию со стороны идентифицируемого сервиса. Думайте об этом как о слепой попытке типа _Hello? Guten Tag? Привет? Эм... Может быть, Buenos Dias? Как только целевой сервис получит зонд, который он понимает (т.е. говорит по правильному протоколу), он ответит Nmap, который получит подтверждение типа сервиса.





- Match": это регулярные выражения, которые Nmap применяет к полученному ответу. Если отправка HTTP GET-запроса вызвала ответ от сервиса, он применит к этому ответу десятки регулярных выражений, чтобы определить наличие, например, слов `Apache`, `Nginx`, `Microsoft IIS` и т. д.




Есть еще несколько директив для конкретных случаев, но основными для понимания работы Nmap и настройки его использования являются эти. Чтобы сделать эту теоретическую часть более конкретной, приведем пример:



![nmap-image](assets/fr/33.webp)



пример зонда в файле `/usr/share/nmap/nmap-service-probes` в Nmap



В первой строке этого примера мы видим простой для понимания зонд с именем `GetRequest`. Это TCP-пакет, содержащий пустой HTTP GET-запрос к корню веб-сервиса с использованием HTTP/1.0, за которым следует перевод строки и пустая строка.



Строка `ports` указывает Nmap, на какой порт отправлять данный зонд. Это позволяет определить приоритет тестов и сэкономить время.



Наконец, у нас есть два примера `match`. Первый, например, классифицирует сканируемый веб-сервис как `ajp13`, если регулярное выражение, содержащееся в этой строке, совпадает с полученным ответом сервиса.



Чтобы помочь вам понять, как могут выглядеть зонды, вот список некоторых зондов, которые вы найдете в этом файле (всего их 188 на момент написания этого руководства).



![nmap-image](assets/fr/34.webp)



пример нескольких зондов, используемых Nmap и представленных в файле `/usr/share/nmap/nmap-service-probes`._



Первые две (`NULL` и `GenericLines`) представляют особый интерес, поскольку они просто посылают пустой TCP-пакет или пакет, содержащий разрыв строки. Серверные службы часто объявляют о себе сразу же после получения соединения, без каких-либо конкретных действий, команд или запросов со стороны клиента.



Вот случай немного более сложного _соответствия_:



```
# Match Nginx + version in an error 400 page
match ssl/http m|^HTTP/1.1 400 Bad Request\r\n.*?Server: nginx/([\d.]+)[^\r\n]*?\r\n.*<title>400 The plain HTTP request was sent to HTTPS port</title>|s p/nginx/ v/$1/ cpe:/a:igor_sysoev:nginx:$1/
```



Точное регулярное выражение содержится здесь между `m|` и `|`, которые отделяют любое регулярное выражение в этом файле. Пожалуйста, найдите время, чтобы прочитать этот пример целиком. Вы заметите выделение в регулярном выражении: `([\d.]+)`, используемое для выделения версии. В этом примере также определены другие Elements, такие как название продукта `p/nginx/`, извлекаемая версия `v/$1/` и CPE с версией `cpe:/a:igor_sysoev:nginx:$1/`.



CPE (Common Platform Enumeration) - это стандартизированная система обозначений, используемая для идентификации и описания программного и аппаратного обеспечения. Этот формат позволяет более эффективно управлять уязвимостями и конфигурациями безопасности, а главное - унифицировать способ их представления, независимо от конкретного продукта. Вот два примера CPE: `cpe:/o:microsoft:windows_8.1:r1` и `cpe:/a:apache:http_server:2.4.35`



Здесь мы четко определяем их типы: `o` для ОС, `a` для приложений, вендоров, продуктов и версий.



Таким образом, в случае _соответствия_ одному из этих регулярных выражений мы получим не только имя службы, но и ее версию и точный CPE, что облегчит поиск CVE, затрагивающих эту версию. Вы найдете эту информацию в стандартном выводе Nmap, и увидите, что она очень полезна для других целей, которые мы рассмотрим в нескольких разделах.



Точный синтаксис _matches_ и, в более общем смысле, директив в файле `/usr/share/nmap/nmap-service-probes` на этом не заканчивается и может показаться довольно сложным, если вы не привыкли работать с Nmap и его результатами. Тем не менее, вы должны хотя бы помнить о его существовании и общих принципах работы, что пригодится вам в дальнейшем, когда вы захотите понять или отладить результат, настроить сканирование или даже внести свой вклад в развитие Nmap.



### III. Использование Nmap для определения версий



Теперь мы будем использовать всю эту сложную механику _Probe_ и _Match_ с помощью простой опции: `-sV`. Это просто говорит Nmap: попытайтесь выяснить, какие именно сервисы и версии портов вы установили как открытые.



```
# Enable service and version detection
nmap 192.168.1.0/24 -sV
```



Вот полный пример результата выполнения такой команды:

Здесь мы видим, что Nmap удалось определить все версии сетевых служб, открытых этой целью, и отобразить эту информацию в новом столбце `VERSION`. Можно увидеть довольно точную информацию, вплоть до операционной системы, если эта информация является частью восстановленной сигнатуры.



Для детального понимания того, что происходит во время сканирования уязвимостей, мы можем использовать опцию `--version-trace`. Это обеспечит просмотр в режиме отладки, отображая _зонд_, который привел к обнаружению:

Мы можем четко видеть, какие _зонды_ использовались для определения технологии и версии (в данном случае это `NULL` и `GetRequest` _зонды_), а также полученную информацию.



### IV. Освоение тестов и точность обнаружения



Сейчас мы вернемся к директиве в файле `/usr/share/nmap/nmap-service-probes`, которую мы не рассматривали ранее:



![nmap-image](assets/fr/36.webp)



директива `rarity` для зондов в файле `/usr/share/nmap/nmap-service-probes`._



Эта директива используется для указания редкости (т.е. приоритета/вероятности), связанной с _зондом_. Эта нотация от 1 до 9 позволяет контролировать полноту анализа, выполняемого Nmap при отправке _Probes_. В системе обозначений Nmap, редкость 1 предоставляет информацию в подавляющем большинстве случаев, в то время как редкость 8 или 9 представляет собой особый случай, специфичный для конфигурации или сервиса, который редко встречается.



Чтобы было понятнее, в стандартном случае Nmap отправит каждому сервису, который будет идентифицирован, _пробы_, имеющие редкость от 1 до 7. Чтобы дать вам лучшее представление о распределении _зондов_ по _редкости_, вот их количество:



```
$ grep -E "^rarity" nmap-service-probes |sort |uniq -c

6 rarity 1
1 rarity 2
3 rarity 3
8 rarity 4
9 rarity 5
13 rarity 6
8 rarity 7
81 rarity 8
54 rarity 9
```



Может показаться нелогичным, что зондов `rarity` 8 и 9 больше, чем остальных. Это происходит потому, что Зонды раритета 1 являются общими и работают в большинстве случаев, независимо от сервиса (вспомните Зонд `NULL`, который просто отправляет пустой TCP-пакет). В то время как более сложные зонды практически уникальны для каждого сервиса.



Если мы хотим вручную управлять зондами, которые мы хотим использовать в нашей проверке версий, мы можем использовать опцию `--version-intensity`. Вот два примера:



```
# Less accurate version detection than default
nmap 192.168.1.0/24 -sV --version-intensity 2

# Very deep detection, using all existing Probes
nmap 192.168.1.0/24 -sV --version-intensity 9
```



Чтобы закончить эту тему, вот пример _Probe_ 9 и 8:



![nmap-image](assets/fr/37.webp)



примеры зондов с редкостью 8 и 9 в файле `/usr/share/nmap/nmap-service-probes`._



Эти два _зонда_ обнаруживают серверы Quake1 и Quake2 (видеоигры). Интересны с ностальгической точки зрения, но вряд ли пригодятся в повседневной жизни.



В зависимости от ваших потребностей в точности или скорости, помните, что этот принцип "редкости" существует и может повлиять на результат.



### V. Использование Nmap для обнаружения операционных систем



Сейчас мы рассмотрим, как Nmap может помочь нам определить операционные системы хостов, сканируемых и обнаруживаемых в сети. Для этого используйте опцию Nmap `-O` (для `OS Scan`).



```
# Enable OS Scan
nmap -O 10.10.10.0/24
```

Для этого Nmap будет использовать множество техник, которые работают очень похоже на _Probes_ и _Matches_ для определения технологий и версий. Основное отличие заключается в том, что Nmap будет использовать достаточно "низкоуровневые" параметры ICMP, TCP, UDP и других протоколов. Вот два тестовых примера для обнаружения операционной системы Microsoft Windows 11:



![nmap-image](assets/fr/39.webp)

## 7 - Анализ безопасности: обнаружение уязвимостей



### I. Презентация



В этом разделе мы узнаем, как использовать инструмент сетевого сканирования Nmap для обнаружения и анализа уязвимостей на объектах сканирования. В частности, мы рассмотрим различные опции, доступные для выполнения этой задачи, и изучим пределы возможностей инструмента, чтобы лучше понять и интерпретировать его результаты.



В этом первом разделе мы познакомимся со сканером уязвимостей Nmap и посмотрим, как использовать основные опции обнаружения уязвимостей. В следующих разделах мы более подробно рассмотрим, как работает эта функция и как ее можно настроить.



### II. Использование Nmap для обнаружения уязвимостей



Теперь мы хотим использовать сетевой сканер Nmap для обнаружения уязвимостей в службах и системах нашей информационной системы. Это означает, что помимо обнаружения активных узлов, перечисления открытых сервисов и определения версий и технологий, Nmap будет искать уязвимости.



Для этого Nmap использует скрипты NSE (_Nmap Scripting Engine_), которые можно рассматривать как модули, обеспечивающие гранулярный подход к тестированию.



С помощью правильных опций мы попросим Nmap использовать различные NSE-скрипты на каждом обнаруженном сервисе, что позволит нам обнаружить:





- Ошибки конфигурации ;





- Дополнительные и более продвинутые версии и открытия для ОС;





- Известные уязвимости (CVE) ;





- Слабые идентификаторы;





- Характерный Elements вредоносной программы;





- Возможности отказа в обслуживании ;





- И т.д.




Как видите, сценарии NSE значительно расширяют возможности Nmap в плане сетевых операций, которые он может выполнять. И это позволяет выполнять гораздо более сложные задачи, чем когда-либо прежде. Хорошая новость заключается в том, что, как обычно, эти возможности можно использовать просто через опцию и в контексте по умолчанию. Именно это мы и рассмотрим далее.



### III. Пример сканирования уязвимостей



Сценарии NSE можно использовать при использовании Nmap для сканирования одного порта на хосте, всех служб на этом хосте или всех служб, обнаруженных в нескольких сетях. Поэтому мы можем использовать опции, которые мы увидим, во всех контекстах, которые мы изучили до сих пор.



Чтобы включить сканирование уязвимостей через Nmap, нам нужно использовать опцию `-sC`:



```
# Enable vulnerability scanning during a scan
nmap -sC 10.10.10.152
```



Помните, что по умолчанию, если вы ничего не укажете, Nmap будет сканировать только 1000 наиболее распространенных портов. Он не обнаружит уязвимости на более экзотических портах, которые могут быть открыты вашими целями.



Прежде чем использовать эту функциональность в производственной информационной системе, я приглашаю вас продолжить чтение этого руководства. В следующих разделах мы рассмотрим, как лучше контролировать влияние и типы тестов, которые будут выполняться.



Повторно используя полученные ранее знания, мы можем, например, более полно просканировать все TCP-порты цели:



```
# Enable vulnerability scanning on all ports
nmap -sC -p- 10.10.10.152
```

Здесь мы видим отображение дополнительной информации, представляющей интерес в контексте анализа уязвимостей:





- Доступ к службе FTP может быть анонимным, и она не защищена аутентификацией. Сценарий NSE, отвечающий за эту проверку, сообщает нам об этом и даже отображает часть древовидной структуры службы FTP. Здесь мы видим, что у нас есть доступ к директории `C` системы Windows!


### IV. Понимание ограничений Nmap при сканировании уязвимостей



Сразу оговоримся: Nmap не способен провести полный тест на проникновение в вашу информационную систему или смоделировать операцию Red Team. У него есть несколько ограничений, о которых нужно знать, чтобы у вас не возникло ложного чувства безопасности:





- Ограниченный охват**: хотя скрипты Nmap NSE являются мощными, их тестовый охват может быть ограничен по сравнению с другими специализированными инструментами обнаружения уязвимостей. Некоторые уязвимости могут быть не охвачены доступными сценариями NSE, например уязвимости Active Directory, раскрытие конфиденциальных данных или более сложные случаи уязвимых веб-приложений.





- Сложность уязвимости**: некоторые типы уязвимостей может быть сложно обнаружить с помощью скриптов NSE из-за их сложности. Например, уязвимости, требующие сложного взаимодействия с удаленным сервисом, могут быть неэффективно обнаружены Nmap (как в случае избыточных разрешений в файловом ресурсе или недостатка контроля разрешений в веб-приложении).





- Пассивное обнаружение**: Для обнаружения уязвимостей Nmap ориентируется в основном на активное сканирование, что означает, что он может неэффективно обнаруживать потенциальные уязвимости без установления активного соединения с целевыми узлами. Поэтому уязвимости, которые не проявляются во время активного сканирования, могут быть пропущены (как в случае с инъекцией кода в веб-приложении).





- Зависимость от обновлений**: База данных Nmap [https://www.it-connect.fr/cours-tutoriels/administration-systemes/stockage/bdd/] скриптов NSE постоянно пополняется, но между обнаружением новой уязвимости и добавлением соответствующего скрипта в Nmap может быть задержка. В результате Nmap не всегда может быть в курсе последних уязвимостей.





- Ложноположительные и ложноотрицательные результаты**: как и любой инструмент безопасности, скрипты Nmap NSE могут выдавать ложноположительные (ложные предупреждения об уязвимостях) или ложноотрицательные (реальные уязвимости не обнаружены) результаты. Это следует иметь в виду при анализе результатов Nmap.




Поэтому важно понимать, что делает и чего не делает Nmap, а также знать, как интерпретировать его результаты. В частности, мы видели в этом руководстве, что опции по умолчанию могут привести к тому, что мы упустим важные Elements, которые могут быть обнаружены при тщательном использовании.



Будь вы системным администратором сети, инженером по безопасности или даже CISO, использование Nmap позволит вам получить представление о состоянии безопасности информационной системы. Это важный первый шаг в обеспечении безопасности системы, который может регулярно выполняться ИТ-командой. Однако он не должен заменять вмешательство и консультации экспертов по [кибербезопасности] (https://www.it-connect.fr/cours-tutoriels/securite-informatique/), которые смогут выявить слабые места гораздо полнее, чем Nmap.

### II. Концепция скриптов Nmap NSE



Сценарии Nmap NSE позволяют очень гибко расширять его возможности. Они написаны на LUA, языке сценариев, который проще в обращении и доступе, чем C или C#, используемые в Nmap. Преимущество использования LUA-скрипта вместе с Nmap, а не отдельного инструмента, заключается в том, что он позволяет нам воспользоваться скоростью выполнения и стандартными возможностями Nmap (обнаружение хоста, порта, версии и т. д.).



Эти скрипты распределены по категориям, и один скрипт может относиться к нескольким категориям:



| Catégorie       | Description |
|----------------|-------------|
| **auth**       | Contient les scripts relatifs à l’authentification sur des services, dont l’accès anonyme ou l’énumération des utilisateurs. Exemples: `oracle-enum-users`, `ftp-anon`. |
| **broadcast**  | Contient les scripts relatifs aux opérations de broadcast sur le réseau, notamment en vue d’exploiter et de découvrir certains services, hôtes ou protocoles reposant sur le broadcast (IPv6, wake on lan, IGMP, etc.). Exemples: `broadcast-dhcp6-discover`, `broadcast-ospf2-discover`. |
| **brute**      | Contient les scripts relatifs aux opérations de brute force de l’authentification sur les services (brute force [SSH](https://www.it-connect.fr/cours/comprendre-et-maitriser-ssh/), MSSQL, etc.). Exemples: `ssh-brute`, `vnc-brute`. |
| **default**    | Contient les scripts utilisés dans le cas par défaut (utilisation de `-sC`). Plusieurs critères sont utilisés afin de valider l’entrée d’un script dans cette catégorie dont la vitesse d’exécution, la structure de la sortie, la fiabilité du test, le caractère “intrusif” ou “risqué”, etc. |
| **discovery**  | Contient les scripts relatifs à la découverte avancée du réseau et des services. On y retrouve par exemple l’énumération du contenu d’un partage SMB, d’une version d’un service VNC, des requêtes SNMP, etc. Exemples: `mysql-info`, `http-security-headers`. |
| **dos**        | Contient les scripts pouvant causer un déni de service. Il peut s’agir de scripts créés pour exploiter une vulnérabilité de type déni de service ou alors de scripts ayant pour effet de bord un déni de service. Prudence donc (ils sont exclus de la catégorie `default`). Exemples: `http-slowloris`, `ipv6-ra-flood`. |
| **exploit**    | Contient les scripts créés pour exploiter de manière directe une vulnérabilité. Exemples: `http-shellsock`, `smb-vuln-ms08-067`. |
| **external**   | Contient les scripts qui nécessitent l’utilisation d’une ressource tierce, comme une base d’information en ligne. Cela indique notamment une tentative de connexion vers l’extérieur (attention à la confidentialité). Exemples: `whois-ip`, `dns-blacklist`, `ip-geolocation-geoplugin`. |
| **fuzzer**     | Contient les scripts conçus pour envoyer des trames, paquets ou paramètres inattendus par un service. Cela permet notamment de causer des erreurs ou dysfonctionnements afin d’obtenir des pistes de vulnérabilité ou des informations techniques. Exemples: `dns-fuzz`, `http-form-fuzzer`. |
| **intrusive**  | Contient les scripts qui sont catégorisés comme “risqués” d’un point de vue disponibilité, ou détection. Ils peuvent provoquer un crash du système ou être détectés comme malveillant par une solution de sécurité. Il s’agit de la catégorie inverse de `safe`. Exemples: `smtp-brute`, `smb-vuln-ms08-067`, `smb-psexec`. |
| **malware**    | Contient les scripts conçus pour détecter la présence d’élément caractéristique d’un malware, tel qu’un port en écoute communément utilisé par une backdoor connue. Exemples: `ftp-proftpd-backdoor`, `smtp-strangeport`. |
| **safe**       | Contient les scripts qui sont considérés comme sûrs d’un point de vue détection ou stabilité. Il s’agit de la catégorie inverse de `intrusive` et elle contient en grande majorité des scripts avancés d’identification de version ou de relevé d’élément de configuration. Exemples: `html-title`, `smb2-security-mode`, `ms-sql-info`. |
| **version**    | Contient les scripts qui permettent une détection avancée de version. Ils peuvent être utilisés en complément des Probes et Matchs étudiés précédemment quand la détection d’une version nécessite des opérations un peu plus complexes. Exemples: `http-php-version`, `vmware-version`. |
| **vuln**       | Contient les scripts conçus pour détecter la présence de vulnérabilité connue (CVE) sans pour autant les exploiter (à l’inverse de la catégorie `exploit`). Ils se contentent en général de rapporter le statut “vulnérable” ou non d’un service. Exemples: `smb-vuln-ms17-010` (eternal blue), `http-phpmyadmin-dir-traversal`. |


Технически, категории, к которым относится скрипт, указываются непосредственно в его коде.



![nmap-image](assets/fr/41.webp)



категории сценариев nSE `ftp-anon`._

### IV. Использование скриптов Nmap для NSE



Теперь мы научимся проводить сканирование уязвимостей, тщательно выбирая интересующие нас сценарии NSE.



#### A. Выберите сценарии по категориям



Для начала мы можем выбрать выполнение всех скриптов, относящихся к определенной категории. Мы должны указать эту категорию или эти категории Nmap с помощью аргумента `--script <category>`:



```
# Use default NSE scripts
nmap --script default 10.10.10.152
```



Эта первая команда эквивалентна команде `nmap -sC`. По умолчанию Nmap будет выбирать скрипты из категории `default`, но это только для примера. Следующая команда, например, будет использовать все скрипты из категории `discovery`:



```
# Use NSE scripts from the "discovery" category
nmap --script discovery 10.10.10.152
```



Как мы уже видели, некоторые категории позволяют быстро определить, что делают соответствующие скрипты NSE (`discovery`, `vuln`, `exploit`), в то время как другие определяют уровень риска, обнаружения или стабильности проводимого теста. Если мы находимся в деликатном контексте и не очень хорошо представляем себе различные действия, выполняемые выбранным нами скриптом, мы можем объединить выборки и выбрать только те скрипты, которые относятся к категориям `discovery` и `afe`:



```
# Use scripts from multiple categories
nmap --script "discovery and safe" 10.10.10.152
```



Если вы абсолютно и явно хотите исключить скрипты из категорий `dos` и `intrusive`, вы можете использовать следующую нотацию:



```
# Exclude categories
nmap --script "not intrusive and not dos" 10.10.10.152
```



Обратите внимание, что указание условий исключения, как указано выше, приведет к использованию всех остальных категорий, которые не исключены явно. Чтобы быть более справедливыми, мы могли бы указать, например:



```
# Include scripts from the "vuln" category except those that are too risky
nmap --script "vuln and not intrusive and not dos" 10.10.10.152

# Same thing, but only targeting the HTTP protocol
nmap --script "(http and vuln) and not intrusive and not dos" 10.10.10.152
```



Вот несколько примеров того, как следует обращаться с NSE-скриптами по категориям, особенно при использовании Nmap для анализа уязвимостей в реальных условиях.



#### B. Выберите сценарии как единое целое



Мы также можем выбрать выполнение одного конкретного теста во время анализа, теста, соответствующего сценарию NSE. Для этого необходимо указать имя скрипта в параметре `-script <name>`. В качестве примера можно взять скрипт `ftp-anon.nse`:



```
# Use an NSE script and a specific port
nmap --script ftp-anon -p 21 10.10.10.152
```



Тогда мы получим очень точный результат:



![nmap-image](assets/fr/45.webp)

Мы видим результат выполнения сценария `ftp-anon` на порту 21, и ни на каком другом порту, поскольку мы указали опцию `-p 21`. Мы также могли бы выполнить базовое сканирование портов, запустив сценарий `ftp-anon` NSE только на обнаруженных службах FTP:



```
# Use a specific NSE script
nmap --script ftp-anon 10.10.10.152
```



Таким образом, Nmap также выполнил бы этот тест анонимного соединения, если бы обнаружил службу FTP на другом порту.



Для получения краткого описания того, что делает скрипт NSE, вы можете воспользоваться опцией `--script-help`, упомянутой выше:

#### C. Управление аргументами сценария



В процессе использования Nmap вы столкнетесь с некоторыми скриптами NSE, которым для корректной работы требуются входные аргументы. Сейчас мы рассмотрим, как передать аргументы этим скриптам через опции Nmap.



В качестве примера возьмем скрипт `ssh-brute, который позволяет выполнить атаку грубой силы на службу SSH.



Классическая атака грубой силы заключается в проверке нескольких паролей (иногда миллионов) в попытке аутентифицироваться в определенной учетной записи. Пробуя такое количество паролей, злоумышленник делает ставку на вероятность того, что пользователь использовал слабый пароль из словаря паролей, используемого для атаки.



Этот скрипт имеет опции "по умолчанию", которые мы можем настроить в соответствии с нашим контекстом. Например, в контексте данной атаки мы можем предоставить Nmap список пользователей и словарь паролей, который будет использоваться. Насколько я знаю, невозможно просто перечислить аргументы, необходимые для скрипта, поэтому самый надежный способ - посетить официальный сайт Nmap. Прямую ссылку на документацию к скрипту NSE можно получить в ответ на опцию `--script-help`:



![nmap-image](assets/fr/47.webp)



результат отображения справки для скрипта NSE`ssh-brute со ссылкой на nmap.org._



Перейдя по указанной ссылке, мы попадаем на эту страницу сайта [https://nmap.org](https://nmap.org/):



![nmap-image](assets/fr/48.webp)



список аргументов, которые могут быть переданы Nmap'овскому скрипту `ssh-brute` NSE



Здесь мы имеем четкое представление об аргументах, которые могут быть использованы, основными из которых в нашем контексте являются `passdb` (файл, содержащий список паролей) и `userdb` (файл, содержащий список пользователей). Документация здесь относится к внутренним библиотекам Nmap, так как эти механизмы перебора и связанные с ними опции взаимно унифицированы для использования в нескольких скриптах (`ssh-brute`, `mysql-brute`, `mssql-brute` и т.д.) и поэтому будут иметь более или менее одинаковые аргументы:



```

# Create a file containing my password list
echo "123456" > /tmp/passlist
echo "NomEntreprise75" >> /tmp/passlist
echo "changezmoi" >> /tmp/passlist

# Run an SSH brute force via Nmap network scan
nmap --script ssh-brute --script-args userdb=/tmp/userlist,passdb=/tmp/passlist 10.10.10.245 192.168.1.19
```



Как видно из последней команды, мы можем указать необходимые аргументы для скрипта Nmap с помощью опции `--scripts-args key=value,key=value`. Вот возможный результат вывода Nmap при выполнении SSH-перебора с помощью NSE-скрипта `ssh-brute:



![nmap-image](assets/fr/49.webp)

Как видите, в интерактивном выводе (выводе терминала) информация, генерируемая скриптами NSE, имеет префикс `NSE: [имя скрипта]`, что облегчает ее поиск. В рамках обычного отображения результатов работы Nmap мы просто имеем сводку, указывающую, были ли обнаружены слабые идентификаторы (включая пароли, помните).



Чтобы сделать еще один шаг вперед и напомнить вам, что все это можно использовать в дополнение ко всем опциям, которые мы уже рассмотрели, вот команда, которая обнаружит сеть `10.10.10.0/24`, просканирует 2000 наиболее часто используемых TCP-портов и выполнит поиск анонимного доступа к FTP-сервисам и грубую силу к SSH-сервисам:



```
# Example of a complete command using multiple scripts
nmap --top-ports 2000 10.10.10.0/24 --script ftp-anon,ssh-brute --script-args userdb=/tmp/userlist,passdb=/tmp/passlist
```



Это лишь один пример из множества доступных скриптов и их опций. Но теперь у нас есть лучшее представление о том, как разобраться со скриптами NSE, требуют ли они аргументов и как передать эти аргументы в Nmap.

## 9 - Управление выходными данными Nmap




### I. Презентация



В этом разделе мы рассмотрим выходные данные, создаваемые Nmap, и, в частности, различные варианты форматирования этих данных. Мы увидим, что Nmap может создавать несколько форматов выходных данных для различных нужд, и это тоже является одним из сильных сторон этого инструмента.



По умолчанию Nmap предлагает подробный обзор результатов сканирования и тестов. Сюда входят просканированные хосты и службы, обнаруженные доступными, информация об открытых портах, их состоянии и версии. Кроме того, в выводе терминала доступны детали сценариев NSE. Однако даже при четком форматировании информации этот вывод может быстро стать объемным, что может затруднить поиск точной информации в результатах.



### II. Освоение выходных форматов Nmap



#### A. Сохранить результаты сканирования в текстовом файле



Чтобы облегчить работу, [Nmap](https://www.it-connect.fr/cours/nmap-cartographie-reseau-scan-de-vulnerabilites/) позволяет легко сохранить результаты в текстовом файле. Это может быть полезно для архивирования, сравнения с другими тестами, а также для просмотра вывода с помощью специализированных текстовых редакторов или скриптовых языков, таких как Sublime text, [PowerShell](https://www.it-connect.fr/cours-tutoriels/administration-systemes/scripting/powershell/), Python, grep, sed и т. д. Чтобы сохранить стандартный вывод Nmap в текстовом файле, мы можем использовать опцию `-oN <filename>` ("N" в слове "normal"):



```
# Save Nmap output to a file
nmap 10.10.10.0/24 -sV -oN nmap_scan_10.10.10.0_24.txt
```



Ничего удивительного, ведь Nmap выведет свой обычный стандартный вывод не только в терминал, но и в указанный файл.



#### B. Вывод generate Nmap в сжатом формате



Существует также второй формат вывода в стиле "text", который может быть легко интерпретирован человеком: формат "greppable".



Этот формат был создан для обеспечения "сжатого" представления вывода Nmap, структурированного таким образом, чтобы облегчить его обработку такими инструментами, как `grep`. Давайте рассмотрим пример такого типа вывода:



![nmap-image](assets/fr/50.webp)



nmap сканирование сети и вывод в формате "greppable"._



Здесь я выполнил обнаружение сети, а также сканирование портов и анализ технологий и версий в сети /24, затем сохранил результаты в файл в формате "greppable". В итоге у меня получился файл, содержащий 2 строки на каждый активный хост:





- Первая строка сообщает мне, что такой-то и такой-то хост находится в состоянии _Up_;





- Вторая строка сообщает мне, какие порты были просканированы, их статус и информацию о технологии и версии, полученную в очень специфическом формате: `<порт>/<статус/<протокол>//<сервис>//<версия>/,`




Такое форматирование с фиксированным разделителем позволяет быстро обрабатывать данные средствами обработки текстов, такими как `grep`, или скриптами и языками программирования. Например, следующая команда позволяет мне легко получить информацию о хосте `10.10.10.5` в случае огромного сканирования, выполненного Nmap, вывод которого было бы трудно просмотреть:



```
# Filter by IP address in the Nmap "greppable" file
grep '10.10.10.5' nmap_10.10.10.0.gnmap
Host: 10.10.10.5 () Status: Up
Host: 10.10.10.5 () Ports: 21/open/tcp//ftp//Microsoft ftpd/, 80/open/tcp//http//Microsoft IIS httpd 7.5/ Ignored State: filtered (998)
```



И наоборот, я могу легко перечислить все хосты, у которых открыт порт 21, поскольку порты и IP находятся в одной строке:



```
# Filter by open port in the Nmap "greppable" file
grep '21/open' nmap_10.10.10.0.gnmap
Host: 10.10.10.5 () Ports: 21/open/tcp//ftp//Microsoft ftpd/, 80/open/tcp//http//Microsoft IIS httpd 7.5/ Ignored State: filtered (998)

Host: 10.10.10.152 () Ports: 21/open/tcp//ftp//Microsoft ftpd/, 80/open/tcp//http//Indy httpd 18.1.37.13946 (Paessler PRTG bandwidth monitor)/, 135/open/tcp//msrpc//Microsoft Windows RPC/, 139/open/tcp//netbios-ssn//Microsoft Windows netbios-ssn/, 445/open/tcp//microsoft-ds//Microsoft Windows Server 2008 R2 - 2012 microsoft-ds/ Ignored State: closed (995)
```



Чтобы получить generate, нужно использовать опцию `-oG <filename>.gnmap` (буква "G" в слове "grep"). По привычке я использую для такого файла расширение `.gnmap`, но вы можете использовать любое другое:



```
# Save Nmap output to a file in "greppable" format
nmap 10.10.10.0/24 -sV -oG nmap_scan_10.10.10.0_24.gnmap
```



Этот формат можно использовать для самых разных целей, и он особенно полезен для быстрого создания сценариев/сортировки. Тем не менее от него стараются отказываться в пользу формата, который мы рассмотрим далее.



примечание: формат `-oG` greppable официально устарел начиная с Nmap 7.90. Он все еще может использоваться для совместимости. Он все еще может использоваться для совместимости, но рекомендуется использовать XML или обычный формат для любой разработки или автоматического разбора._



#### C. Формат XML для вывода Nmap



Последний формат, о котором стоит упомянуть в этом учебнике, - XML. В отличие от двух предыдущих форматов, этот предназначен для чтения не человеком, а другими инструментами или скриптами.



XML (_eXtensible Markup Language_) - это язык разметки, используемый для хранения и транспортировки данных, предлагающий иерархическую структуру с помощью пользовательских тегов.



В Nmap формат XML используется для создания generate подробных отчетов о проведенном сканировании, включающих информацию о хостах, портах и обнаруженных уязвимостях, а также дополнительную информацию, не отображаемую в стандартном выводе Nmap.



Чтобы получить выходной файл generate в формате XML, нужно использовать опцию `-oX` ("O" от "XML"):



```
# Save Nmap output to a file in XML format
nmap 10.10.10.0/24 -sV -oX nmap_scan_10.10.10.0_24.xml
```



В результате вы получите стандартный вывод Nmap в терминале, а также файл в формате XML в текущем каталоге.



Конечно, формат XML не предназначен для чтения и интерпретации человеком. Тем не менее, если вы хотите выполнять сценарии или автоматический анализ на этом формате вывода Nmap, вам все равно нужно иметь представление об используемых тегах и структуре. Например, вот часть содержимого XML-файла, созданного Nmap, в котором показаны результаты сканирования 1 хоста:



![nmap-image](assets/fr/51.webp)



пример XML-записи для 1 хоста во время сканирования Nmap



Здесь много информации, и нас особенно интересуют два открытых порта:



```
<port protocol="tcp" portid="21"><state state="open" reason="syn-ack" reason_ttl="0"/><service name="ftp" product="Microsoft ftpd" ostype="Windows" method="probed" conf="10"><cpe>cpe:/a:microsoft:ftp_service</cpe><cpe>cpe:/o:microsoft:windows</cpe></service></port>
<port protocol="tcp" portid="80"><state state="open" reason="syn-ack" reason_ttl="0"/><service name="http" product="Microsoft IIS httpd" version="7.5" ostype="Windows" method="probed" conf="10"><cpe>cpe:/a:microsoft:internet_information_services:7.5</cpe><cpe>cpe:/o:microsoft:windows</cpe></service></port>
```



Мы понимаем, что такой формат облегчает автоматизированный разбор результатов, поскольку каждая часть информации аккуратно располагается в специальном теге или атрибуте. В частности, мы находим информацию, с которой раньше не сталкивались: CPE.



```
cpe>cpe:/a:microsoft:internet_information_services:7.5</cpe><cpe>cpe:/o:microsoft:windows</cpe></service></port>
<cpe>cpe:/a:microsoft:internet_information_services:7.5</cpe><cpe>cpe:/o:microsoft:windows</cpe></service></port>
```



Мы кратко упоминали о CPE в разделе 2 модуля 2, и эта информация определяется в совпадениях при определении версии. Nmap использует свои механизмы обнаружения сервисов, технологий и версий для поиска соответствующего CPE.



Это позволяет нам повторно использовать эту информацию в базах данных и приложениях, которые ее используют. В частности, я имею в виду базу данных NVD, которая ссылается на CVE. Для каждого CVE в ней указаны компьютеры CPE, затронутые уязвимостью. Вот пример CVE, касающегося `a:microsoft:internet_information_services:7.5` из базы данных NVD:



![nmap-image](assets/fr/52.webp)



наличие CPE в сведениях о CVE в базе данных NVD



Теперь мы лучше понимаем преимущества этого формата, который предлагает очень четкую структуру информации и содержит все данные, собранные или обработанные Nmap.



Как рефлекс, я систематически сохраняю свои сканы Nmap во всех трех форматах сразу. Это возможно благодаря опции `-oA <имя>` ("A" означает "All"), которая создает файл `<имя>.nmap`, файл `<имя>.xml` и файл `<имя>.gnmap`. Таким образом, я буду уверен, что ничего не пропадет, когда мне понадобится просмотреть результаты.



С этими тремя форматами у вас должно быть все необходимое для сохранения и последующей автоматизированной обработки результатов Nmap. Мы снова будем использовать формат XML в следующем разделе, когда рассмотрим использование Nmap с другими инструментами безопасности.



#### III. Создание HTML-отчета по результатам сканирования Nmap



Формат XML открывает множество возможностей, и не в последнюю очередь он может служить основой для создания отчета в формате HTML, который будет более наглядным для просмотра.



Чтобы преобразовать файл Nmap в формате XML в веб-страницу, мы воспользуемся инструментом `xsltproc`, который нам нужно будет сначала установить:



```
# Install the xsltproc tool
sudo apt install xsltproc
```



После установки этого инструмента просто предоставьте ему XML-файл, который нужно преобразовать, и имя HTML-отчета, который нужно создать:



```
# Create an Nmap HTML report from XML
xsltproc nmap_10.10.10.0.xml -o "Nmap – rapport web 05-2024.html"

# Open the .html file with Firefox
firefox "Nmap – rapport web 05-2024.html"
```



В результате мы получим весь скан, красиво структурированный, с несколькими цветами и кликабельными ссылками в оглавлении!



![nmap-image](assets/fr/53.webp)



извлечение из отчета о сканировании Nmap в формате HTML, сгенерированного xsltproc._



В общем случае XML-файл, сохраненный Nmap, содержит ссылку на другой файл в формате XSL:



```
<?xml-stylesheet href="/usr/share/nmap/nmap.xsl" type="text/xsl"?>
```



Таким образом, преобразование в HTML - это функция, предоставляемая и облегчаемая Nmap, `xsltproc` является общепринятым и признанным инструментом для выполнения этой задачи (который не входит в набор инструментов Nmap).



XSLT (_Extensible Stylesheet Language Transformations_) - это подмножество XSL, позволяющее отображать XML-данные на веб-странице и "преобразовывать" их, параллельно используя стили XSL, в читабельную, отформатированную информацию в формате HTML.



источник: [helpx.adobe.com/](https://helpx.adobe.com/fr/dreamweaver/using/xml-xslt.html)_



Уровень информации в отчете эквивалентен XML-формату Nmap и выше, чем в стандартном выводе терминала (_interactive output_).



### IV. Управление уровнем многословности Nmap



Сейчас мы рассмотрим несколько вариантов Debugger Nmap или отслеживания его работы.



Первая опция, которую мы должны упомянуть, - это опция `-v`, которая увеличивает многословность Nmap. Вот пример:

При сканировании множества хостов и портов вывод терминала будет трудно использовать из-за большого количества отображаемой информации. По этой причине данную опцию следует использовать в сочетании с рассмотренными ранее опциями, которые позволяют сохранять стандартный вывод Nmap в файл. При длительном сканировании, когда отображение данных может происходить медленно, это позволяет не следить за текущей активностью Nmap и не знать, что и в каком темпе происходит. ## 11 - Повышение производительности сканирования сети



### I. Презентация



В этой главе мы узнаем, как оптимизировать скорость сканирования сети с помощью Nmap, используя различные специфические опции. В частности, мы узнаем о внутреннем устройстве Nmap, начиная с управления _timeout_ и заканчивая предварительно сохраненными конфигурациями инструмента.



Теперь, когда мы хорошо рассмотрели возможности Nmap, давайте поближе познакомимся с этим зверем и его мощью. Если вы когда-нибудь использовали этот инструмент в больших сетях, то наверняка замечали, что некоторые сканирования могут занимать много времени, несмотря на всю мощь инструмента. И на то есть причины: простая команда `nmap` с несколькими опциями может generate миллионы пакетов, направленных на сотни тысяч потенциальных систем и сервисов.



Более того, некоторые конфигурации сетевого оборудования могут намеренно устанавливать более низкую скорость (количество пакетов в секунду), рискуя отклонить ваши пакеты или запретить ваш IP Address по соображениям безопасности.



В зависимости от контекста, возможно, стоит попытаться оптимизировать все это, как мы увидим в этой главе.



В любом случае, вы можете проверить значения по умолчанию параметров, которые мы будем рассматривать, а также то, были ли правильно учтены опции, которые вы собираетесь использовать, с помощью Nmap debug (опция `-d`, рассмотренная в предыдущей главе):



![nmap-image](assets/fr/63.webp)



просмотр опций тайминга с помощью опции `-d` в Nmap._



### II. Управление скоростью сканирования Nmap



#### A. Управление распараллеливанием



По умолчанию Nmap использует распараллеливание при сканировании для его оптимизации, и все параметры, которые он использует, могут быть изменены с помощью различных опций. Однако случаи, когда необходимо изменять эти параметры, довольно редки, поэтому мы не будем подробно рассматривать их в этом руководстве:





- `--min-hostgroup/max-hostgroup <size>`: размер групп параллельного сканирования хостов.





- `--min-parallelism/max-parallelism <numprobes>`: распараллеливание зондов.





- `--scan-delay/--max-scan-delay <time>`: настраивает задержку между зондами.




Просто знайте, что они существуют и могут быть использованы.



#### B. Управление количеством пакетов в секунду



По умолчанию Nmap сам регулирует количество отправляемых пакетов в секунду в зависимости от реакции сети. Однако есть возможность принудительно изменить эту настройку, задав максимальное и/или минимальное значение количества пакетов в секунду. Эта настройка выполняется с помощью опций `--min-rate <number>` и `--max-rate <number>`, где `number` представляет собой количество пакетов в секунду. Пример:



```
# Limit the scan speed to 300 packets per second
nmap -sV 10.10.10.0/24 --max-rate 300
```



Эти опции позволяют настроить скорость сканирования в соответствии с вашими потребностями, либо ускорить процесс, либо ограничить используемую пропускную способность. Последний случай (ограничение скорости сканирования), скорее всего, приведет вас к этим опциям, особенно если вы испытываете сетевые задержки при использовании Nmap (что случается довольно редко).



### III. Управление сбоями соединений и таймаутами



Еще один параметр, с которым мы можем играть, чтобы оптимизировать скорость сканирования Nmap (или гарантировать большую точность), касается _timeout_ и _retry_.



Для _timeouts_ - это **таймаут отсутствия ответа**, после которого Nmap прекратит ожидание ответа и будет считать службу или хост недоступными. Для _retry_ - это **количество последовательных попыток выполнения операции**, которые Nmap предпримет, прежде чем двигаться дальше.



Как и в случае с распараллеливанием, управление _таймаутами_ и _повторными_ попытками может быть применено к фазам обнаружения хоста или сервиса:





- `--min-rtt-timeout/max-rtt-timeout/initial-rtt-timeout <time>`: задает время обхода Exchange. Опять же, этот параметр фактически вычисляется и адаптируется на лету во время сканирования. Маловероятно, что вам понадобится его использовать, так как Nmap рассчитывает это время на лету в зависимости от реакции сети.





- `--max-retries <number>`: ограничивает количество повторных передач пакета при сканировании порта. По умолчанию Nmap может выполнить до 10 повторных попыток для одного сервиса, особенно если он обнаружит задержки или потери на сетевом уровне, но в большинстве случаев выполняется только одна.





- `--host-timeout <time>`: задает максимальное время, которое Nmap проведет на хосте для всех своих операций, включая сканирование портов, обнаружение сервисов и любые другие операции, связанные с этим хостом. Если этот интервал времени будет превышен без какого-либо ответа или **завершения операций**, Nmap покинет этот хост, не отобразив никаких результатов относительно него, и перейдет к следующему в своем списке. Это позволяет контролировать максимальное время, которое Nmap проводит на данном хосте, избегая застревания на непокорных хостах и оптимизируя общее время сканирования.




В своей повседневной работе я использую опции `--max-retries` и `--host-timeout` для оптимизации сканирования:



```
# Optimization of a scan with 0 additional attempts and a timeout of 15 minutes per host
nmap -sV -sC 10.10.10.0/24 --max-retries 0 --host-timeout 15m
```



Эти параметры обеспечивают дополнительную гибкость при настройке поведения сканирования в соответствии с конкретными потребностями и условиями сети. Однако вы должны знать об их последствиях с точки зрения нагрузки на сканируемые узлы и потенциальной потери точности.



### IV. Использование подготовленных конфигураций



Различные опции, рассмотренные в этой главе, могут использоваться как по отдельности, так и в составе готовых конфигураций, предлагаемых Nmap. Опция, которая позволяет использовать эти _шаблоны_ (шаблоны конфигурации), - это `-T <номер>` или `-T <имя>`. Существует 5 уровней шаблонов:



```
-T<0-5>: Set timing template (higher is faster)
```



По умолчанию Nmap использует _шаблон_ 3 (_нормальный_), которого обычно достаточно.



Со своей стороны, я обычно работаю в условиях, когда мне нужно быть достаточно быстрым (оставаясь при этом максимально полным), и я часто использую опцию `-T4`.



```
# Use Nmap for a network scan with preset T4 (with debug)
nmap 10.10.10.0/24 -sV --top-ports 2000 -T4 -d
```

## 12 - Безопасность и конфиденциальность данных при использовании Nmap



### I. Презентация



В этой главе мы рассмотрим ряд передовых методов, которые следует применять для обеспечения безопасности и, прежде всего, конфиденциальности данных, создаваемых, обрабатываемых и хранимых в Nmap.



Использование Nmap в информационной системе может быть быстро классифицировано как наступательное действие. Следовательно, необходимо принять ряд мер предосторожности, чтобы действовать в рамках закона, гарантируя безопасность предполагаемых целей, собранных данных и системы, используемой для сканирования.



### II. Получение соответствующих разрешений



Перед сканированием сети или системы убедитесь, что вы получили соответствующие разрешения. Сканирование систем на наличие уязвимостей (`NSE-скрипты`) без разрешения может быть незаконным и иметь юридические последствия, особенно если безопасность информационных систем не входит в ваши официальные полномочия.





- Напоминаем: [Уголовный кодекс: Глава III: Нападения на автоматизированные системы обработки данных](https://www.legifrance.gouv.fr/codes/article_lc/LEGIARTI000030939438/)




### III. Защита конфиденциальных данных



Результаты, выдаваемые Nmap, могут считаться конфиденциальными, особенно если они содержат информацию о слабых местах в информационной системе, которые могут быть использованы злоумышленником. Но также и в тех случаях, когда они касаются систем, доступ к которым открыт не для всех (например, конфиденциальных, промышленных, медицинских или [резервных] информационных систем (https://www.it-connect.fr/cours-tutoriels/administration-systemes/autres/sauvegarde/)).



Мы также видели, что в зависимости от используемых скриптов NSE результаты сканирования [Nmap](https://www.it-connect.fr/cours/nmap-cartographie-reseau-scan-de-vulnerabilites/) также могут содержать идентификаторы.



Таким образом, злоумышленник, получивший доступ к результатам сканирования, будет иметь в своем распоряжении карту информационной системы и множество технической информации, не выполняя этих действий самостоятельно и рискуя быть обнаруженным.



Поэтому при использовании Nmap важно следить за тем, чтобы не собирать и не хранить конфиденциальную информацию, включая, но не ограничиваясь, следующее:





- Шифрование выходных данных: если вам нужно хранить или передавать результаты сканирования Nmap, обязательно зашифруйте их, чтобы обеспечить конфиденциальность данных. Это позволит предотвратить несанкционированный перехват конфиденциальной информации. В идеале данные должны быть зашифрованы сразу после того, как они покинут систему, используемую для выполнения сканирования (для начала лучше всего использовать ZIP-архив, зашифрованный надежным паролем).





- Настройте контроль доступа: убедитесь, что только уполномоченные лица имеют доступ к результатам сканирования Nmap, где они будут храниться. Установите соответствующие средства контроля доступа, чтобы защитить конфиденциальную информацию от несанкционированного доступа.





- Бдительность при работе с данными: при передаче, копировании или обработке данных сканирования строго следите за безопасностью данных. Это значит: не оставляйте их в директории `Download` на рабочей станции, подключенной к Интернету, не позволяйте им проходить через внутреннее приложение HTTP file Exchange, не оставляйте блокнот открытым, не заблокировав рабочую станцию во время обеденного перерыва, и т. д.




### IV. Управление агрессивными сканами



Как мы уже видели в этом учебнике, Nmap может быть очень многословным на сетевом уровне. Он также может посылать пакеты, которые неправильно отформатированы, и не строго соблюдать структуру протокола в сетевых кадрах и пакетах, которые он генерирует. Все эти действия могут оказывать влияние на определенные системы и службы, иногда доводя до сбоев или перенасыщения системных и сетевых ресурсов.



Чтобы избежать любых инцидентов, вам необходимо освоить поведение Nmap и знать, как адаптировать его к контексту, в котором он используется, с помощью различных опций, рассмотренных в этом руководстве. Мы не обязательно будем использовать Nmap одинаково в информационной системе, содержащей промышленное [аппаратное] обеспечение (https://www.it-connect.fr/actualites/actu-materiel/), и в пользовательской сети, состоящей из систем Windows, защищенных локальным брандмауэром, или в ядре сети.



Надеемся, что различные уроки этого руководства научили вас осваивать и анализировать поведение Nmap, но лучший способ научиться - это делать. Поэтому убедитесь, что вы знакомы с опциями Nmap, которые будете использовать.



### V. Защита системы сканирования



В первой главе мы увидели, что в большинстве случаев Nmap необходимо запускать от имени `root` или локального администратора. Это связано с тем, что он выполняет сетевые операции, иногда на довольно низком уровне, с помощью сетевых библиотек, которые требуют высоких и рискованных разрешений с точки зрения стабильности системы или конфиденциальности других приложений.



В результате Nmap может рассматриваться как чувствительный компонент системы, на которой он установлен. Обязательно используйте последнюю версию Nmap, поскольку старые версии могут содержать известные уязвимости в системе безопасности. Использование актуальной версии позволяет минимизировать риски, связанные с использованием этого инструмента.



Если вы решили использовать Nmap не через сессию от имени `root`, а предоставив определенные привилегии привилегированному пользователю, чтобы у него было все необходимое для использования Nmap (`sudo` или _capabilities_), имейте в виду, что Nmap может быть использован как часть полного повышения привилегий:

Также крайне нежелательно устанавливать Nmap на системы, не предназначенные для сканирования сети. В частности, я имею в виду серверы или рабочие станции. ### II. Сканирование TCP SYN или "полуоткрытое сканирование



Первый тип TCP-сканирования, который мы рассмотрим, - это `TCP SYN Scan`, также известный как `Half Open Scan`. Если вы помните сканирование сети, которое мы проводили после первого сканирования портов, это тип сканирования, используемый по умолчанию в [Nmap](https://www.it-connect.fr/cours/nmap-cartographie-reseau-scan-de-vulnerabilites/) при запуске с правами root.



Перевод поможет вам понять, как работает это сканирование. На самом деле, при сканировании TCP SYN на каждый целевой порт будет отправлен пакет TCP SYN, который является первым пакетом, отправляемым клиентом (запрашивающим установление соединения) в рамках знаменитого _трехстороннего рукопожатия_ TCP. Обычно, если порт открыт на целевом сервере и за ним работает служба, сервер отправляет ответный пакет TCP SYN/ACK, чтобы подтвердить SYN клиента и инициализировать TCP-соединение. Этот ответ принимает форму TCP-пакета с флагами SYN и ACK, установленными в 1, что позволяет нам подтвердить, что порт открыт и ведет к службе.



С другой стороны, если порт закрыт, сервер пошлет нам пакет `TCP` с флагами RST и ACK, установленными в 1, чтобы прервать запрос на соединение, так что мы будем знать, что за этим портом нет активного сервиса:

![nmap-image](assets/fr/68.webp)

В первой строке мы видим, что источник сканирования отправляет TCP-пакет на хост `10.10.10.203` на порт TCP/80. В этом TCP-пакете флаг SYN установлен в 1, чтобы указать, что это запрос на инициализацию TCP-соединения.



Затем, во второй строке, мы видим, что цель отвечает `TCP SYN/ACK`, что означает, что она согласна инициализировать соединение и, следовательно, принимать потоки на порт TCP/80. Таким образом, мы можем сделать вывод, что порт TCP/80 открыт и что на сканируемом сервере присутствует веб-сервер.

![nmap-image](assets/fr/70.webp)

Сканируемый хост возвращает `TCP RST/ACK` в ответ на мой `TCP SYN`, когда порт не открыт.



Как уже упоминалось, при запуске Nmap из привилегированного терминала режим TCP SYN Scan используется по умолчанию, и может быть принудительно установлен с помощью опции `-sS` (`scan SYN`):



```
# Execution of a TCP Syn Scan_
nmap -sS 192.168.1.15
```




По соображениям скорости наиболее часто используется `TCP SYN Scan`. С другой стороны, неспособность клиента завершить _трехстороннее рукопожатие_ (т. е. не отправить ACK после SYN/ACK сервера) может показаться подозрительной, если она наблюдается слишком часто на одном сервере или от одного и того же источника в сети. Действительно, нормальным поведением клиента после получения пакета TCP SYN/ACK в ответ на TCP SYN является отправка `подтверждения` (ACK), после чего начинается Exchange.



Тем не менее, он обеспечивает несколько более быстрое сканирование, поскольку не утруждает себя отправкой ACK для каждого положительного ответа. Преимуществом SYN Scan является его скорость, так как на каждый сканируемый порт отправляется только один пакет, за счет большей вероятности обнаружения.



Кроме того, TCP SYN scan способен определить, фильтруется (защищается) ли порт брандмауэром. Фактически, брандмауэр перед целевым узлом можно обнаружить по тому, как он ведет себя при получении пакета TCP SYN на порт, который он должен защищать. Он просто не отвечает. Однако, как мы уже видели, в обоих случаях (открытый или закрытый порт) ответ от хоста есть. Это третье поведение покажет наличие брандмауэра между сканируемым узлом и машиной, выполняющей сканирование. Вот результат, который может выдать Nmap, когда сканируемый порт фильтруется брандмауэром:



![nmap-image](assets/fr/71.webp)

Когда мы выполняем захват сети во время сканирования, мы видим, что ответа нет:



![nmap-image](assets/fr/72.webp)



захват сети во время сканирования TCP SYN для порта, отфильтрованного брандмауэром



Разница между закрытым и отфильтрованным портом заключается в следующем: отфильтрованный порт - это порт, защищенный брандмауэром, а закрытый порт - это порт, на котором не работает ни одна служба и который поэтому не может обрабатывать наши TCP-пакеты. Некоторые типы TCP-сканирования, такие как TCP SYN scan, способны определить, фильтруется ли порт, в то время как другие типы сканирования этого не делают.



### III. Сканирование с помощью TCP-соединения или полное открытое сканирование



Второй тип TCP-сканирования - это `TCP Connect scan`, также известный как _Full Open Scan_. Он работает так же, как и сканирование TCP SYN, но на этот раз возвращает `TCP ACK` после положительного ответа от сервера (SYN/ACK). Именно поэтому он называется `Full Open`, так как соединение полностью открывается и инициируется на каждом порту, открытом во время сканирования, таким образом соблюдая TCP _Three Way Handshake_:

![nmap-image](assets/fr/74.webp)

Мы видим, что первым TCP-пакетом клиент отправляет `TCP SYN`, а сервер отвечает `TCP SYN/ACK`, указывая, что порт открыт и содержит активную службу. Чтобы имитировать легитимного клиента, Nmap отправит серверу ответный сигнал `TCP ACK`. И наоборот, при сканировании закрытого порта:



![nmap-image](assets/fr/75.webp)

При использовании Nmap опция `-sT` (`scan Connect`) используется для выполнения сканирования TCP-соединения. Обратите внимание, что при использовании Nmap из непривилегированной сессии этот режим сканирования TCP используется по умолчанию:



```
# Execution of a TCP Connect Scan

```



Сканирование `TCP Connect Scan` имитирует более легитимный запрос на соединение, а его поведение больше всего напоминает поведение лямбда-клиента, так что обнаружить сканирование по уменьшенному количеству портов сложнее. Однако он работает медленнее, поскольку полностью инициализирует каждое TCP-соединение на открытых портах сканируемой машины.



Сканирование Nmap 10 000 портов все равно будет легко обнаружить, если установлены службы обнаружения и защиты от сетевых вторжений (IDS, IPS, EDR). Если злоумышленник хочет остаться незамеченным, он сосредоточится на небольшом количестве стратегически выбранных портов, таких как 445 (SMB) или 80 (HTTP), которые часто открыты на серверах и представляют собой распространенные уязвимости.



Поскольку TCP Connect Scan ожидает ответа в обоих случаях, он также может обнаружить наличие брандмауэра, который может фильтровать порты на целевом узле.



### IV. Сканирование TCP FIN или "скрытое сканирование

Если у сервера нет активного TCP-соединения с клиентом, он посылает `RST/ACK`. Таким образом, мы можем различать открытые и закрытые порты, посылая пакеты `TCP FIN` на набор портов:

Мы видим, что клиент посылает один или два пакета, чтобы прервать TCP-соединение, а сервер не отвечает. Он просто принимает конец соединения и прекращает связь.



Вот что мы видим теперь при сканировании закрытого порта:



![nmap-image](assets/fr/78.webp)



захват сети во время сканирования TCP FIN для закрытого порта



Мы видим, что сервер посылает в ответ пакет `TCP RST/ACK`, поэтому есть разница в поведении между открытым и закрытым портом, и мы можем перечислить открытые порты на сервере, отправив пакет TCP FIN. В Nmap для выполнения сканирования TCP FIN используется опция `-sF` (`scan FIN`):



```
# Execution of a TCP Fin Scan
nmap -sF 192.168.1.15
```



TCP FIN Scan не работает на хостах Windows, поскольку ОС склонна игнорировать пакеты TCP FIN, когда они отправляются на не открытые порты. Поэтому, если вы запустите TCP FIN Scan на хосте Windows, у вас сложится впечатление, что все порты закрыты.



Поэтому важно знать несколько методов сканирования и понимать разницу между ними.



Поскольку в любом случае TCP FIN не будет ждать ответа, он не сможет обнаружить наличие брандмауэра между целевым узлом и источником сканирования.



Вот пример результата сканирования TCP FIN в Nmap:



![nmap-image](assets/fr/79.webp)



результаты сканирования TCP FIN программой Nmap._



Фактически, отсутствие ответа от хоста на определенном порту может означать, что порт заблокирован, но также и то, что он открыт и активен.



Это сканирование называют "скрытым", так как оно не передает много трафика generate и обычно не вызывает регистрации в целевых системах. С его помощью можно незаметно обнаружить порты в сети, не поднимая тревоги. Однако, как уже говорилось выше, его эффективность может варьироваться в зависимости от целевой системы, как и его дискретность в зависимости от конфигурации оборудования безопасности.

## 14 - Сканирование портов по TCP: XMAS, Null и ACK



### I. Презентация



В этом разделе мы продолжим изучать различные методы сканирования TCP, предлагаемые Nmap. Здесь мы рассмотрим методы `XMAS`, `Null` и `ACK`, которые используют специфические особенности TCP для получения информации о портах и сервисах, открытых на данной цели.



### II. Сканирование TCP XMAS



XMAS Scan TCP немного необычен тем, что он совсем не имитирует обычное поведение пользователей или машин в сети. На самом деле, XMAS Scan отправляет TCP-пакеты с флагами `URG` (Urgent), `PSH` (Push) и `FIN` (Finish), установленными на 1, чтобы обойти определенные брандмауэры или механизмы фильтрации.



Название XMAS происходит от того, что видеть эти флаги включенными - необычно. Когда все три флага устанавливаются одновременно в TCP-пакете, он выглядит как зажженная рождественская елка:



![nmap-image](assets/fr/80.webp)



флаги TCP, используемые при сканировании XMAS



Не вдаваясь в подробности роли этих флагов, важно знать, что при отправке пакета с этими тремя включенными флагами активная служба, находящаяся за целевым портом, не вернет ни одного пакета. С другой стороны, если порт закрыт, мы получим пакет TCP RST/ACK. Теперь мы сможем различать поведение открытого и закрытого порта при перечислении портов на машине:



![nmap-image](assets/fr/81.webp)



диаграмма поведения сканера tCP XMAS Scan для открытых и закрытых портов



Следуя той же логике, сканирование сети на порт TCP/80 хоста с активным веб-сервером показывает следующее поведение при обнаружении открытого порта (источник сканирования `10.10.14.84`):



![nmap-image](assets/fr/82.webp)



захват сети во время сканирования TCP XMAS на предмет открытого порта



Мы видим, что источник сканирования отправляет два TCP XMAS пакета (с флагами `FIN`, `PSH` и `URG`, установленными на 1) на хост `10.10.10.203` и не получает ответа от цели, что говорит о том, что порт открыт. И наоборот, при выполнении `TCP XMAS Scan` на закрытом порту наблюдается следующий результат:



![nmap-image](assets/fr/83.webp)



захват сети во время сканирования TCP XMAS для закрытого порта



Ответом на наш TCP-пакет будет `TCP RST/ACK`, указывающий на то, что порт закрыт. Чтобы использовать эту технику в Nmap, опция `-sX` (`scan XMAS`) позволяет выполнить TCP XMAS Scan:



```bash
# Execution of a TCP XMAS Scan
nmap -sX 192.168.1.15
```



Важно отметить, что сканирование TCP XMAS не способно обнаружить брандмауэры, которые могут находиться между целевой и сканирующей машинами, в отличие от некоторых других типов сканирования, таких как TCP SYN или Connect. Действительно, активный брандмауэр между двумя хостами обеспечит отсутствие TCP-возврата, если целевой порт отфильтрован (т. е. защищен брандмауэром). Поэтому в случае отсутствия ответа невозможно узнать, защищен ли порт брандмауэром или открыт и активен. Также следует знать, что, как и при проверке TCP FIN, некоторые приложения или операционные системы, такие как Windows, могут исказить результаты этого типа проверки.



примечание: поддержка сканирования XMAS/FIN/NULL на последних версиях Windows остается ограниченной, и результаты могут быть непоследовательными на этом типе целей. (Обновление 2025 г.)_



### III. Нулевое сканирование TCP



В отличие от TCP XMAS scan, TCP Null scan будет отправлять пакеты TCP scan со всеми флагами, установленными в 0. Это тоже поведение, которое никогда не встретится в обычном Exchange между машинами, поскольку отправка TCP-пакета без флага не оговорена в RFC, описывающем протокол TCP. Поэтому его легче обнаружить.



Как и проверка TCP XMAS, эта проверка может помешать работе некоторых брандмауэров или модулей фильтрации, позволяя пакетам проходить через них:



![nmap-image](assets/fr/84.webp)



диаграмма поведения tCP Null Scan для открытых и закрытых портов



Вот что можно увидеть в сети во время сканирования TCP Null на открытом порту:



![nmap-image](assets/fr/85.webp)



захват сети во время сканирования TCP Null на наличие открытого порта



Сканирующая машина отправляет пакет без флага (`[<None>]` в Wireshark), не получая никакого ответа от сервера. И наоборот, когда целевой порт закрыт:



![nmap-image](assets/fr/86.webp)



захват сети во время сканирования TCP Null для закрытого порта



Чтобы выполнить сканирование TCP Null с помощью Nmap, просто используйте опцию `-sN` (`scan Null`):



```bash
# Execution of a TCP Null Scan
nmap -sN 192.168.1.15
```



Поскольку реакция при открытом порте и при активном брандмауэре (отсутствие обратной связи с сервером в обоих случаях) идентична, TCP Null scan не может определить наличие брандмауэра. Более того, брандмауэр может даже фальсифицировать результат, предполагая, что порт открыт, поскольку он не отвечает на TCP-пакеты без флагов, даже если порт отфильтрован. Это важная информация, которую следует учитывать при использовании сканеров, не способных отличить открытый порт от отфильтрованного (защищенного брандмауэром), таких как сканеры `TCP Null`, `XMAS` или `FIN`, чтобы оставаться последовательным в интерпретации полученных результатов.



### IV. Сканирование TCP ACK



Сканирование TCP ACK используется для обнаружения наличия брандмауэра на целевом узле или между ним и источником сканирования.



В отличие от других сканирований, сканирование TCP ACK не пытается определить, какие порты открыты на хосте, а скорее определить, активна ли система фильтрации, отвечая для каждого порта `filtered` или `unfiltered`. Некоторые TCP-сканеры, такие как `TCP SYN` или `TCP Connect`, могут делать и то, и другое одновременно, в то время как другие, такие как `TCP FIN` или `TCP XMAS`, вообще не могут определить наличие фильтрации. Вот почему сканирование TCP ACK может быть полезным:



![nmap-image](assets/fr/87.webp)

Для выполнения этого типа сканирования мы будем использовать опцию `-sA` в Nmap. Вот результат сканирования TCP ACK, если порт отфильтрован, т. е. заблокирован и защищен брандмауэром:



![nmap-image](assets/fr/88.webp)



отображение nmap во время сканирования TCP ACK._



Пример результата для хоста с брандмауэром и без него. Nmap возвращает `фильтры` на портах TCP/80 и TCP/81 хоста `10.10.10.203`. При анализе сети с помощью Wireshark поведение выглядит следующим образом:



![nmap-image](assets/fr/89.webp)



захват сети во время сканирования TCP ACK для порта, не отфильтрованного брандмауэром



Если на целевой машине присутствует брандмауэр, ничего не возвращается.



Чтобы запустить это сканирование с помощью Nmap, используйте опцию `-sA` (`scan ACK`):



```bash
# Execution of a TCP ACK Scan
nmap -sA 192.168.1.15
```

## 15 - Nmap CheatSheet - краткое описание команд учебника



### I. Презентация



Вот краткое описание многочисленных команд Nmap и вариантов их использования, чтобы вы могли быстро найти и использовать их в повседневной работе.



### II. Nmap: Шпаргалка IT-Connect



Вот шпаргалка по представленным командам. Эта страница облегчает поиск наиболее распространенных вариантов использования Nmap.





- Сканирование портов




```bash
# Display installed Nmap version
nmap --version

# Scan for open specific ports on a single IP address
nmap --open -p 80 192.168.1.18

# Scan TCP ports on a selection of ports
nmap 192.168.1.19 -p 80
nmap 192.168.1.19 -p 22,80,1000-2000,3389

# Scan UDP services on an IP address
nmap -sU 192.168.1.19

# Scan UDP services on specific ports
nmap -sU 192.168.1.19 -p 161,23

# Scan the 100 most commonly used TCP ports
nmap 192.168.1.19 --top-ports 100

# Scan all ports on an IP address
nmap 192.168.1.19 -p-

# Scan multiple subnets with specific ports
nmap 192.168.0.0/24 192.168.1.0/24 -p 22

# Scan a subnet while excluding a specific IP address, scan all ports
nmap 192.168.0.0/24 --exclude 192.168.0.4 -p-
```





- Обнаружение активных узлов




```bash
# Scan on CIDR or IP ranges
nmap 192.168.0.0/24
nmap 192.168.0.0/24 192.168.1.0/24
nmap 192.168.1.2 192.168.1.3 192.168.1.10-20
nmap 192.168.0.0/24 192.168.1.3 192.168.1.10-20

# Host discovery scan (Ping Scan) on a network
nmap -sn 192.168.1.0/24
```



примечание: Опция `-sP` устарела уже несколько лет назад и должна быть заменена на `-sn`. (Обновление 2025 г.)_



```bash
# Host discovery scan without port scan
nmap 192.168.1.0/24 -sn

# Host discovery scan on a local network using various probe techniques
nmap -sn -PP -PS22,3389,445,139 -PA80 192.168.1.0/24

# Scan hosts listed in a text file
nmap -iL /tmp/mesCibles.txt

# Network scan with a specific IP excluded
nmap 192.168.1.0/24 --exclude 192.168.1.140

# Subnet scan excluding another subnet
nmap 10.0.0.0/16 --exclude 10.0.100.0/24
```





- Обнаружение версии




```bash
# Enable version detection
nmap 192.168.1.0/24 -sV

# Version detection + advanced trace
nmap 192.168.1.0/24 -sV --version-trace

# Version detection with maximum probe rarity of 2
nmap 192.168.1.0/24 -sV --version-intensity 2

# Version detection with all probes
nmap 192.168.1.0/24 -sV --version-intensity 9

# Enable OS detection
nmap -O 10.10.10.0/24
```





- Скрипты NSE: поиск уязвимостей




```bash
# Run default NSE scripts
nmap -sC 10.10.10.152

# Scan all TCP ports with default NSE scripts
nmap -sC -p- 10.10.10.152

# Display help for script by name
nmap --script-help=ftp-*

# Display help for a category
nmap --script-help=discovery

# Use NSE scripts by category
nmap --script default 10.10.10.152
nmap --script discovery 10.10.10.152

# Inclusion/exclusion filter for NSE script categories
nmap --script "discovery and safe" 10.10.10.152
nmap --script "not intrusive and not dos" 10.10.10.152
nmap --script "vuln and not intrusive and not dos" 10.10.10.152
nmap --script "(http and vuln) and not intrusive and not dos" 10.10.10.152

# Run a specific NSE script
nmap --script ftp-anon -p 21 10.10.10.152
nmap --script ftp-anon 10.10.10.152

# Pass arguments to an NSE script
nmap --script ssh-brute --script-args userdb=/tmp/userlist,passdb=/tmp/passlist 10.10.10.245 192.168.1.19
```





- Управление выходом Nmap




```bash
# Save scan to text file
nmap 10.10.10.0/24 -sV -oN nmap_scan_10.10.10.0_24.txt

# Save scan to condensed text file
nmap 10.10.10.0/24 -sV -oG nmap_scan_10.10.10.0_24.gnmap

# Save scan to XML file
nmap 10.10.10.0/24 -sV -oX nmap_scan_10.10.10.0_24.xml
```





- Оптимизация производительности




```bash
# Version detection scan with max rate of 300 packets per second
nmap -sV 10.10.10.0/24 --max-rate 300

# Version detection scan with default scripts, no retry, max host timeout 15min
nmap -sV -sC 10.10.10.0/24 --max-retries 0 --host-timeout 15m

# Version detection scan with the 2000 most common ports, aggressive scan speed (T4), debug output
nmap 10.10.10.0/24 -sV --top-ports 2000 -T4 -d
```





- Типы сканирования TCP




```bash
# TCP SYN scan (fast, less stealthy)
nmap -sS 192.168.1.15

# TCP Connect scan (classic)
nmap -sT 192.168.1.15

# TCP FIN scan (stealth)
nmap -sF 192.168.1.15

# TCP XMAS scan
nmap -sX 192.168.1.15

# TCP Null scan
nmap -sN 192.168.1.15

# TCP ACK scan (detect firewall)
nmap -sA 192.168.1.15
```
