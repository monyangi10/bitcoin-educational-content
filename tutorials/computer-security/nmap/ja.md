## 1 - Nmapの紹介：Nmapとは？



### I.プレゼンテーション



この最初のセクションでは、Nmapネットワーク・スキャン・ツールについて見ていこう。このツールについて知っておくべき重要なElementsと、一般的にどのように動作するかを見ていく。これはチュートリアルの残りの部分をよりよく理解するのに役立ちます。



### II.Nmapツールの紹介



Nmapは_Network Mapper_の略で、**ネットワークの発見、マッピング、セキュリティ監査**に使われるフリーのオープンソースツールです。また、**ネットワークのインベントリ、診断、監視**などの他のタスクにも使用できます。



対象となるネットワーク上のホストがアクティブで到達可能かどうか、どのネットワークサービスが公開されているか、どのバージョンや技術が使用されているか、その他の有用な分析情報を判断することができる。Nmapは、特定のマシン上の単一のサービスをスキャンするのにも、インターネット全体までのネットワークの広い範囲をスキャンするのにも使える。



Nmapの強みはたくさんある：





- **強力で柔軟**：Nmapは大規模なネットワークをスキャンし、高度な検出技術を使用することができる。UDP、TCP、ICMP、IPv4、IPv6をサポートし、バージョン検出、脆弱性スキャン、プロトコル固有のインタラクションを実行できる。Nmapのアーキテクチャは、特にNSE（Nmap Scripting Engine）スクリプトのおかげでモジュール化されています。
- 使いやすさ：公式ドキュメントは豊富で高品質。数多くのコミュニティ・リソースも利用できます。
- 人気と長寿：Nmapは1998年以来、この分野でリファレンスとなっている。現在のバージョンは7.95である。特定のタスクのために他のツールも存在するが、Nmapはネットワーク・マッピングと解析の必需品であり続けている。



**映画館でのNmap**



Nmapは、一般市民の間で一定の知名度を獲得した数少ないセキュリティ・ツールのひとつである。映画『マトリックス・リローデッド』（原題：Matrix Reloaded）では、トリニティがNmapを使ってシステムに侵入するシーンが登場する：

### III.ハイレベルの操作



NmapはLinux、Windows、macOSで利用できる。主にC、C++、Lua（NSEスクリプト用）で書かれている。主にコマンドラインで使用されるが、Zenmapのようなグラフィカル・インターフェースも利用できる。しかし、どのように動作するかをよりよく理解するために、コマンドラインから始めることを強くお勧めします。



簡単な例だ：



```
nmap 192.168.10.13 10.10.10.0/24 -sV -sC --top-ports 250
```



このコマンドについては後で詳しく説明する。このチュートリアルでは、Linux上でNmapを使用するが、その使用方法は他のシステムでも同様である。Windowsでは、Nmapはネットワーク・パケットをキャプチャして注入するために、**Npcap**ライブラリ（現在は廃止されているWinPcapの代わり）に依存している。



Nmapは、`ls`や`ip`のような従来のバイナリのように使用する。このツールは、ターゲットシステム上で特定の反応を引き起こすために（特にサービスや脆弱性の検出のために）、パケットを型破りな方法で操作することがあるため、一部の高度な機能には昇格した権限が必要な場合がある。



### IV.Nmapの使用による影響



Nmapを使う前に、それがネットワークやシステムに与える潜在的な影響を知っておくことが不可欠だ：





- 短時間に数千、数百万のパケットを送信することが可能であり、特定のネットワーク・インフラを飽和させる可能性がある。
- 不正なパケットや非標準のパケットをgenerateすることができ、特定の機器（特に産業用システム）を混乱させる可能性がある。
- これは、セキュリティ・システム（ファイアウォール、IDS/IPSなど）のアラートをトリガーすることができる**攻撃のような振る舞い**を生成することができる。



一般的に言って、**Nmapは非常におしゃべりなツール**である。したがって、機密環境や本番環境で使用する前に、どのように動作するかを十分に理解することをお勧めする。



### V.結論



このセクションでは、Nmapとその主な機能について紹介する。Nmapは必要不可欠で、強力かつ柔軟なネットワークマッピングツールであることがわかった。また、Nmapがどのように動作するのか、そして使用する際の注意点についても説明し、このチュートリアルの次の部分のための準備を整えました。



## 2 - なぜNmapを使うのか？



### I.プレゼンテーション



このセクションでは、Nmapネットワーク・スキャン・ツールの主な使い方を見ていこう。Nmapは多くの文脈や職業で広く使われているツールであり、これをツールボックスに入れておくことや、使いこなす方法を知っておくことは、間違いなく有用なスキルであることがわかるだろう。



### II.診断と監視のためのNmapの使用



Nmapは、ネットワーク診断や、より広い意味での監視に使うことができる。pingを使って2つのホストが通信しているかどうかを判断するのと同じように、Nmapを使ってホストがアクティブかどうか、あるいは特定のサービスが稼働しているかどうかを素早く判断することができる。Nmap](https://www.it-connect.fr/cours/nmap-cartographie-reseau-scan-de-vulnerabilites/ "Nmap")のおかげで、ホストの応答時間、パケットが通った経路、特定のサービスによる応答などに関する正確なデータを得ることができる。



例えば、ホスト**192.168.1.18**上のウェブ・サーバーがアクティブかどうか、正しく応答しているかどうかを素早く調べるには、次のコマンドとその結果を使うことができる：



```
nmap --open -p 80 192.168.1.18
```



![nmap-image](assets/fr/02.webp)



**Nmapを使ってリモートサーバーからウェブサービスのステータスを取得する。**



つまり、Nmapを使うことは、デバッグや診断の段階で有名な「pingテスト」よりもさらに進んだことを意味する。Nmapが、あるポートでリッスンしているサービスを特定するために、そのバージョンも含めて、いくつかの方法があることは後で説明する。



### III.ネットワーク・マッピングにNmapを使う



ネットワークマッパーとして、ネットワークマッピングがこのツールの主な目的です。ローカルネットワーク内でも、複数のネットワーク、サブネット、VLAN間でも使用でき、到達可能なすべてのホストとサービスをリストアップする。Nmapは、この作業を手作業よりもはるかに速く、効率的にします。



例えば、以下のコマンドは、**192.168.1.0/24**ネットワーク上のアクティブなホストを素早く特定するために使用できる：



```
nmap -sn 192.168.1.0/24
```



**注意：** `-sP` オプションは廃止され、`-sn` に置き換えられました。



![nmap-image](assets/fr/03.webp)



*Nmapを使ってネットワーク上の到達可能なホストを発見する*。



Nmapが、たとえ事前にサービスを公開していなくても、そのホストが「アクティブ」とみなせるかどうかを判断するために使用するいくつかの方法があることは、後で説明する。



### IV.Nmapを使ってフィルタリングポリシーを評価する



Nmapには事実に基づいているという利点があります。その結果、アーキテクチャ文書やフィルタリングポリシーとは異なり、具体的な調査結果を確立することができます。フィルタリングポリシーが実際に適用されているかどうかを検証できるため**、情報システムのコンパートメント化の有効性を評価するための重要なツールである**。



企業ネットワークでは、ベストプラクティスとして、システムをその役割、重要度、場所によってセグメント化することが決められている。フィルタリングルール（多くの場合、ファイアウォールによって実装される）は、ゾーン間の通信を制限しなければならない。しかし、これらのコンフィギュレーションはしばしば複雑で、エラーが発生しやすい。



そのため、ポリシーが尊重されていることを検証するには、具体的なテストに勝るものはない。例えば、機密性の高い管理サービス（SSH、WinRM、MSSQL、MySQLなど）がユーザーゾーンからアクセスできないことを確認することができます。



フィルタリングポリシー**をテストするために**Nmapを使用することは、そのようなポリシーが実運用される前に体系化されるべきである。残念ながら、このチェックは軽視されがちである。



私の経験では、検証テストがない場合、多くの設定ミスに気づかない。IPレンジの単純なエラーやルールの見落としによって、隔離されているはずのゾーンが脆弱なままになってしまうことがある。



### V.セキュリティ監査と侵入テストのためのNmapの使用



Nmapは、セキュリティ評価、侵入テスト（ペンテスト）、そして残念ながら攻撃者にとっても便利な機能をたくさん持っている。



ネットワーク・ディスカバリー機能は、攻撃者にとって非常に重要であり、攻撃者は最初の侵害の後、ネットワークのトポロジーを理解する必要がある。しかし、Nmapはこれ以上のものを提供している。Nmapはスクリプトエンジンを統合しており、**その多くは脆弱性検知**に特化している。



例えば、このコマンドは、手動で接続しなくても、FTPサービスが匿名接続を許可しているかどうかをチェックするのに使うことができる：



```
nmap --script ftp-anon -p 21 192.168.1.18
```



![nmap-image](assets/fr/04.webp)



*Nmap.*による匿名FTP認証のチェックにNSEスクリプトを使用する。



Nmapによる脆弱性の検出は、監査や侵入テストの最初のステップの1つである。特定の弱点を迅速に特定し、分析作業を最適化することができます。



しかし、注意してほしい： **脆弱性スキャンツールには限界がある**。Nmapは脅威のほんの一部しかカバーしておらず、問題が検出されなければシステムが安全であるとは保証していない。したがって、**使用されているスクリプトがどのように動作するかを理解する**ことが不可欠であり、スクリプトの判定だけに頼らないことである。



### VI.結論



Nmapを使いこなすことで、診断やモニタリングからマッピング、セキュリティ・ポリシーの評価、脆弱性スキャンまで、幅広いユースケースをカバーできることを見てきた。次のセクションでは、Nmapのインストールについて詳しく説明する。

### IV.Nmapの使用に必要なローカル権限



ところで、Nmapを使う場合、**システム上で昇格したローカル権限が必要だろうか？** **場合による**。



ごく基本的な形、つまりオプションをあまり使い込まない状態では、Nmapは必ずしも特権的な権限を必要としない。しかし、このようなエラーメッセージが表示される危険を冒してまで、Nmapをその能力をフルに発揮させるためには、特権的なコンテキスト(Linuxでは "root"、Windowsでは "administrator "または同等のもの)でNmapを使用した方がよいことに、すぐに気がつくだろう：



![nmap-image](assets/fr/10.webp)



Linuxで、Nmapのオプションにroot権限が必要な場合のエラーメッセージ。



LinuxであろうとWindowsであろうと、Nmapが特権アクセスを要求してくるケースは多い。主な理由は以下の通りである(網羅的なリストではない)：





- **生の**ネットワークパケットを構築する：Nmapは、高度なパケット操作や構築を含む、幅広いスキャン方法を実行できる。例えば、TCP SYNスキャンを実行する場合である。TCP SYNスキャンは、TCP交換の古典的な三者間ハンドシェークを尊重しない。これを実行するために、Nmapは、ネットワーク通信における適切な慣行を尊重する方法しか知らないオペレーティングシステムのネイティブ関数以外の関数を使用する必要がある(上で見た「Npcap」と「libcap」ライブラリを呼び出す)。Nmapが**標準的な**方法で物事を行わないからこそ、OSやサービス、そして特定の脆弱性に関する特定の情報を推測することができるのだ。





- ネットワークトラフィックをリッスンする：Nmapのオプションの中には、特定の情報を取得するためにネットワークをリッスンする必要があるものがある。この動作は、システム上の他のアプリケーションの通信を盗聴することもできるため、オペレーティングシステム上ではセンシティブな動作とみなされる。Wiresharkと同じように、Nmapはこれを実行するために特定の特権を必要とするが、この特権は特権セッションに直接入ることで簡単に取得できる。





- **特権ポートでのリスニング**：オペレーティングシステムでは、0から1024までのポート（TCPおよびUDP）は特権ポートであると言われている。これは今日ではやや時代遅れの理由ではあるが、これらのポートをリッスンするためには、依然として特定の特権が必要である。





- **UDPパケットの送信:** 同様に、UDPポート(ステートレスプロトコル)でネットワークアプリケーションをリッスンするには、オペレーティングシステムの特権権限が必要である。そのため、UDPスキャンを実行する場合、特権セッションが必要となる。このスキャンでは、Nmapがスキャンに対する応答を分析するために、応答を待ち受ける必要がある。




より正確には、少なくともLinuxでは、実際にrootになることなく、Nmapのすべての機能とオプションを実行することが可能である。これは、バイナリに適切な_capabilities_を付与することで実現できる。しかし、これには高度な操作が必要であり、権限を使って直接Nmapを実行するほど実用的ではないかもしれない。また、この方法はあまり一般的ではなく、設定を誤るとセキュリティ上の問題を引き起こす可能性がある。



しかし、これはNmapのチュートリアルから少し外れるので、今回は割愛する。



このチュートリアルの残りの部分では、Nmapは常に "root "として(シェルから "root "として、または "sudo "コマンドで)、またはWindowsの管理者ターミナルで実行されるものと仮定する。そうしないと、チュートリアルとは微妙に、しかし顕著な違いが生じるかもしれない。

## 4 - NmapによるTCPとUDPのポートスキャン



### I.プレゼンテーション

### II.NmapによるホストのTCPポートのスキャン



#### A.Nmapによる最初のTCPポートスキャン



これからNmapを使って最初のスキャンを実行する。アクセスできないはずの管理サービスや、停止したと思っていたアプリケーションのポートが公開されているなど、予期せぬことがないか確認するためだ。



私の例では、スキャンされるホストはIP Address "192.168.1.18 "を持っている：



```
nmap 192.168.1.18
```



可能性のある結果はこうだ。典型的なNmapのリターンには多くの情報が含まれている：



![nmap-image](assets/fr/11.webp)



Nmap._で実行した単純なTCPスキャンの結果。



この結果をざっと見て、ポートTCP/22とTCP/80がこのホスト上でアクセス可能であることがわかる。



デフォルトでは、また何も指示しなければ、NmapはTCPポートのみをスキャンする。



#### B.単純なNmapスキャンの結果を理解する



しかし、この出力を理解するために、もう一歩踏み込んでみよう。それぞれの行は、第一に今行われたことを知るために、第二にスキャン結果そのものを正しく解釈するために重要である。



最初の行は、基本的にスキャンのバージョンと日付のリマインダーである（結局のところ、ログとアーカイブに役立つ！）：



```

```



2行目は、ホスト「debian.home (192.168.1.19)」のスキャン開始結果を示す。この情報は、複数のホストのスキャンを同時に開始するときに役立つだろう：



```
Nmap scan report for debian.home (192.168.1.19)
```



3行目は、問題のホストが "Up"、つまりアクティブであることを示している：



```
Host is up (0.00022s latency).
```



最後に、Nmapは、クローズされたと識別された998個のTCPポートが.NET Frameworkに表示されないことを教えてくれる：



```
Not shown: 998 closed tcp ports (conn-refused)
```



これで、.NET Frameworkのような出力が1,000行近く節約できる：



```
1/tcp closed
2/tcp closed
3/tcp closed
…
```



私たちを助けてくれた彼に感謝する！



なぜ998個の「クローズド」ポートなのか？2つのオープンポートを加えると1000になるが、これはNmapがデフォルト設定でスキャンするポート数であり、既存のTCPポート65535個ではない！後で説明するが、これは完全に、簡単にカスタマイズできる。しかし、対象となるホストが、かなりエキゾチックなポートをリッスンしているサービスを持っている場合、この "デフォルト "スキャンでは、それを発見することはできない。



この情報に続いて、最も興味深いのは、「PORT - STATE - SERVICE」という3つの列に従って整理された表である：





- 最初の "PORT "カラムは単に対象となるポート/プロトコルを示し、ここで重要なのは、それがTCPポート（`<port>/tcp`）かUDPポート（`<port>/udp`）かを見ることである。





- STATE "列は、このポートで発見され、得られた応答に基づいてNmapが決定したネッ トワークサービスのステータスを示す。これは、"open"、"closed"、"filtered"、"unknown "のいずれかである。Nmapがこれらの異なるステータスをどのように区別するかについては、後で説明する。





- SERVICE」列は、当該ポートで公開されているサービスを示す。しかし、ここでは、アクティブなサービス発見オプションを使用していないことに注意されたい。Nmapは、ポート/プロトコルと割り当てられたと思われるサービス間のローカル参照に依存している。




Linuxシステムの「/etc/services」ファイルを見てみると、Nmapが表示するのと同じような「port/protocol - service」リンクが見つかるだろう：



![nmap-image](assets/fr/12.webp)



Linuxの"/etc/services "ファイルの内容を展開する。



今のところ、Nmapはアクティブなサービス検出を行っていないことを理解しておくことが重要である。例えば、TCP/80ポートの背後にあるSSHサービスを特定することはできない。したがって、正しいオプションの使い方を知ることが重要である！



Nmapの出力をどのように解釈するかを知ることは、このツールを使いこなすための重要な要素である。良い知らせは、どんなオプションを使っても、この出力はほとんど同じであるということだ。



#### C.ボンネットの中：Wiresharkによるネットワーク分析



サーバーをスキャンしているホストのネットワークInterface、またはサーバー自体のネットワークInterfaceで何が起こっているかをよく見てみると、Nmapの動作がより明確になる。ここでやろうとしていることは、そういうことだ。



ここでお見せできるのは、Wiresharkで見えるもののほんの一部です。さらに詳しく知りたい場合は、スキャン中に自分でネットワークをキャプチャして、キャプチャされたものをブラウズしてみてください。



このテストでは、私のスキャンホスト（192.168.1.18）と私のターゲットホスト（192.168.1.19）は同じローカルネットワーク上にある。



Nmapはまず、ARPリクエストを送信して、ターゲットホストがローカルネットワーク上でアクティブかどうかを調べる。応答を受信すると、ホストがアクティブであることがわかり、ネットワークスキャンを開始する：



![nmap-image](assets/fr/13.webp)



Nmapがターゲットホストがローカルネットワーク上に存在するかどうかを判断するために発行するARPリクエスト。



スキャン対象のホストがリモートネットワーク上にある場合、Nmapはまずpingリクエストを送信し、最も頻繁に公開されるポート（TCP/80、TCP/443）に到達しようとする：



![nmap-image](assets/fr/14.webp)



ターゲットホストがリモートネットワーク上で到達可能かどうかを判断するためにNmapが発行するpingリクエスト



これらのテストのいずれかに応答があれば、ターゲットがアクティブであるとみなす。



Nmapは、ターゲットがアクティブであると判断すると、そのドメイン名をネットワークカードに設定されたDNSサーバーで解決しようとする：



![nmap-image](assets/fr/15.webp)

Nmapはターゲットを特定し、それがアクティブであることを知ったので、TCPポートのスキャンを開始する：



![nmap-image](assets/fr/16.webp)



Nmapスキャン中のtCP SYNパケット送信とRST/ACK受信



これを行うには、デフォルト範囲の各TCPポートで、**TCP SYNパケットを送信し、応答を待つ**。上のスクリーンショットでは、スキャンされたサーバーからTCP RST/ACKパケットを受信している。結果で見たように、これはスキャンされたポートのほとんどに当てはまる。2つの例外がある：



![nmap-image](assets/fr/17.webp)



ポート22で送信されたTCP SYNパケットに対する応答。



上のスクリーンショットでは、ターゲット・ホスト**からTCP SYN/ACKパケットが送信されている。ポートはアクティブで、サービスを公開している。Nmapは応答の受信を確認し、接続を終了する(TCP RST/ACK)。 **このようにして、TCP/22ポートがアクティブであることがわかった。



ここでは、NmapがTCPネットワークをスキャンする際に「3ウェイ・ハンドシェイク」を尊重することを見てきた。パフォーマンス上の理由から、サーバーのリターンに対して応答しないように要求することが可能であり、その結果、大規模なネットワークをスキャンする場合、数千パケットを節約することができる。しかし、これらのオプションや最適化については、チュートリアルの後半で見ていくことにしよう。



これで、TCPスキャンの方法と、TCPスキャンが実行されたときに実際に何が起こるかについて、よりよく理解できるようになった。また、デフォルトでは、Nmapは限られた数のポートに対してTCPポートスキャンを実行することもわかった。



### III.NmapでUDPポートをスキャンする



#### A.Nmapによる最初のUDPポートスキャン



それでは、ホストのUDPポートをスキャンする方法を見てみよう。これまで見てきたように、デフォルトでは、Nmapは常にTCPポートをスキャンする。これは、意識しなければ、多くの情報を見逃してしまうことを意味する。



注意点として、このテストではスキャンホスト（192.168.1.18）とターゲットホスト（192.168.1.19）は同じローカルネットワーク上にある。



```
nmap -sU 192.168.1.19
```



ここで得られる返り値は、TCPスキャンと同じフォーマットであるが、表示される アクティブなサービスは、要求されたとおり、`<port>/UDP`である！



![nmap-image](assets/fr/18.webp)



Nmap._で実行した単純なUDPスキャンの結果。



sU "オプションは、Nmapに、デフォルトのTCPではなく、UDPで作業したいことを伝えるために使用する。



ところで、チュートリアルの前半で述べたように、NmapがUDPスキャンを行うには「root」権限が必要であることにお気づきだろう。



注意: Nmapの最新バージョンでは、一部の機能がネットワーク・ソケットへの生のアクセスを必要とするため、信頼性の高い結果を得るためには、常に管理者権限でUDPスキャンを実行することをお勧めします。



UDPスキャンには、非常に長い時間がかかることがある（私の例では、1000ポートをスキャンするのに1100秒）。これは、UDPに「Three Way Handshake（三者間ハンドシェイク）」がないためで、NmapはUDPパケットを送信するたびに応答を待ち、一定時間（タイムアウト）経過しても応答がない場合にのみ、そのポートを「クローズ」と判断する。スキャンされたホストからのこの応答は系統的ではなく、ある種の増幅攻撃を避けるために、1秒あたりの応答数が制限されることが多い。これはTCPとは対照的で、ポートが開いていようが閉じていようが、スキャンされたホストからの応答は即座に返ってくる。これを最適化する方法は後で説明する。



UDPの2つ目の難点は、**サービスが常に着信パケットに応答するわけではない**ということである。このような場合、ICMPの "port unreachable "を受信しないと、上のスクリーンショットのように、Nmapはサービスを "open|filtered "とマークする。

### IV.Nmapでポートスキャンを微調整する



#### A.Nmapのデフォルト動作の注意点



これまで見てきたように、何もオプションを指定しなければ、Nmap自身がスキャンする番号とポートを選択する。これは、Nmapに何をすべきかを正確に指示しない場合に、Nmapが使用する「デフォルト」設定である。これらのデフォルトのオプションは、公開されている主なポートを知ることができるように設計されており、これらのポートは、数値順（ポート1、2、3など）ではなく、公開頻度順（最も一般的なポートまたは頻繁に使用されるポート）で選択される。また、適切なオプションを指定しないと、65535個の可能なポートのスキャンを開始しないようにするためでもある。



**これらのポートはどのように選ばれるのですか？**



デフォルトモードでスキャンされる1000個のポートは、出現頻度に応じて選択される。これらの統計はNmapによって管理され、バイナリ自身やそのスクリプト（モジュール）と同じ方法で更新される。これらの統計は、"/usr/shares/nmap/nmap-services "ファイルで自分で見ることができる：



![nmap-image](assets/fr/23.webp)



ファイル"/usr/shares/nmap/nmap-services"._から抽出した。



ここでは、3列目に確率（0と1の間）またはパーセンテージ分布のようなものが表示されている。これは、各ポート/プロトコルのペアの出現頻度である。最もよく知られているポート（この抽出では、FTP、SSH、TELNET、SMTP）は、他のポートよりもはるかに高い値を持っていることがわかります。



#### B.Nmapスキャンのターゲット・ポートを正確に指定する。



しかし、現実の世界では、特定のポートだけをスキャンしたり、いくつかのポートをスキャンしたり、特定の範囲のポートをスキャンしたりする必要があるかもしれない。Nmapを使えば、UDPとTCPの両方のスキャンを統一的な方法で簡単に行うことができる。



**Nmap**で特定のポートをスキャンする。



1000ではなく、1つのポートをスキャンしたい場合は、Nmapの"-p "または"--port "オプションでこのポートを指定することができる：



```
# Scan a single TCP port with Nmap
nmap 192.168.1.19 -p 80

# Scan a single UDP port with Nmap
nmap -sU 192.168.1.19 -p 161
```



その結果、スキャンはより高速になり、Nmapはホストがアクティブかどうか、そして指定されたポートに到達可能かどうかを検出するのに必要なパケットだけを送信する。これは、ショーケースサイトのウェブサービスがまだ稼働しているかどうかの簡単なテストを実行したい場合、時間の節約になる。



**Nmap**で複数のポートをスキャンする。



同じように、Nmapに複数のポートを指定するには、同じオプションを使い、指定したポートをカンマで連結する：



```

# Scan several TCP ports with Nmap

nmap 192.168.1.19 -p 80,10999,22,23,1345

# Scan several UDP ports with Nmap

nmap -sU 192.168.1.19 -p 161,23,69

```



順序に関係なく、Nmapはこれらのポートをすべてチェックし、ターゲットとなるホスト上のものだけをチェックする。Nmapの出力では、**明示的にすべてのポートとそのステータス**を伝えていることに気づくだろう。デフォルトの動作とは異なり、この完全な出力はあまりにも多くのスペースを取ってしまうだろう：



![nmap-image](assets/fr/24.webp)



**指定されたポートに対するNmap TCPスキャンの結果。**



**ポートの範囲をスキャンする**



スキャンしたいポートの数が多すぎる場合は、例えば ：



```

# Scan TCP ports from 1000 to 2000 with Nmap

nmap 192.168.1.19 -p 1000-2000

# Scan UDP ports from 1000 to 2000 with Nmap

nmap -sU 192.168.1.19 -p 100-150

```



もちろん、好きなように組み合わせてもいい：



```

# Scan TCP ports 22,80, 3389 and from 1000 to 2000 with Nmap

nmap 192.168.1.19 -p 22,80,1000-2000,3389

```



**TCPおよびUDPポートスキャン**



また、選択したポートに対して、UDPとTCPのスキャンを同時に実行することもできる：



```

# Scan the list of 1000 default ports, in TCP and UDP

sudo nmap 192.168.1.19 -sT -sU

# Scan only UDP/161 and TCP/22

sudo nmap 192.168.1.19 -sT -sU -p U:161,T:22

```



この最後の例では、UDPポートを示す "U: "とTCPポートを示す "T: "があることに気づくだろう。以下は、このタイプのスキャンの可能な出力である：



![nmap-image](assets/fr/25.webp)



*Nmap.*によるTCPおよびUDPポートのスキャン結果



これはスキャンをカスタマイズする面白い方法だ！



**すべてのポートをスキャン**



最後に、Nmapにもっと大きな範囲や小さな範囲を指定することも可能である。Nmapが選択するデフォルトのリストには、1000個のポートが含まれていることを見てきた。また、「-top-ports」オプションを使って、最も頻繁に使用される上位100ポート、または上位200ポートを求めることもできる：



```

# Scan the top 100 most common ports with Nmap

nmap 192.168.1.19 --top-ports 100

# Scan the top 200 most common ports with Nmap

nmap 192.168.1.19 --top-ports 200

```



最後に、"-p-"表記を使って、すべての可能なポート（65535すべて）をスキャンするように頼むことができる：



```

# Scan all TCP ports from 1 to 65535 with Nmap

nmap 192.168.1.19 -p-

```



後者の場合、特にUDPの場合は時間がかかるが、開いているポートを見逃すことはないだろう。



*注："-p-"オプションは、すべてのTCPポートをスキャンする場合に推奨される方法である。UDPポートの完全なスキャンには非常に長い時間がかかるためである*。



チュートリアルの後半で、Nmapスキャンの速度をニーズに合わせて最適化する方法を紹介するが、これは特にすべてのTCPポートとUDPポートのスキャンに役立つだろう。

CIDR（**Classless Inter Domain Routing**）は、ネットワーク範囲とその範囲（マスクを使用）を指定するための「古典的な」表記法です。例えば、"192.168.0.0/24 "は10進数マスク表記 "255.255.255.0 "の "翻訳 "です。

```
nmap 192.168.0.0/24 192.168.1.0/24

# Scan several hosts via their IP
nmap 192.168.1.2 192.168.1.3 192.168.1.10-20

# Mix of both
nmap 192.168.0.0/24 192.168.1.3 192.168.1.10-20
```

これから、Nmapを使ってネットワーク・ディスカバリーを行う際に、ネットワーク・レベルで何が起こるかを詳しく見ていこう。



前のセクションで見たように、デフォルトでは、NmapはARPプロトコルを使ってローカルネットワーク上のホストの存在を検出する：



![nmap-image](assets/fr/27.webp)

ARPリクエストに対する応答は、一般的にネットワーク上でアクティブなすべてのホストから提供され、何ら不審に見えないため、ローカルネットワーク上の事実上すべてのホストを検出することができる。



リモート・ネットワークでは、Nmapはいくつかのテクニックを組み合わせて使う：



![nmap-image](assets/fr/28.webp)

より正確には、NmapはICMPエコー・パケット（Ping送信の典型的なケース）とICMP Timestampパケットを使う。Nmapは、リモートネットワーク上のホストから応答を得ることを期待している。



しかし、それだけではありません。上のWiresharkのキャプチャを見ると、**TCP SYN**パケットが、TCP/80ポート上の**TCP ACK**パケットと同様に、スキャンされるネットワーク上のすべての潜在的なホストのTCP/443ポート上に組織的に送信されていることがわかる。



**なぜネットワーク・ディスカバリーの一環としてポートにTCPパケットを送るのか？**



指定されたポートにSYNパケットを送信することで、Nmapは**そのポートでサービスがリッスンしているかどうかを判断できる**。ホストがSYNパケットにSYN/ACKパケットで応答した場合、そのホストはアクティブであり、そのポートでサービスがリッスンしていることを示す。したがって、Nmapは、**pingに対する応答が得られなくても**、このサービスで運試しを行う。



与えられたポートにACKパケットを送信することで、Nmapはそのホストにファイアウォールが存在するかどうかを**判定することができる**。ホストがACKパケットにRST(リセット)パケットで応答した場合、これはおそらくそのホストにファイアウォールが存在し、未承諾トラフィックをブロックしていることを示す。そのため、そのホストは、たとえ他のリクエストに応答しなかったとしても、ネットワーク上の存在を裏切ることになる。



しかし、このテクニックを使ったファイアウォール検知は、すべてのケースで完全に信頼できるとは限らないことに注意することが重要である。ホストによっては、特定のサービスやオペレーティングシステムの設定など、ファイアウォールの存在以外の理由でRST応答をgenerateするかもしれない。さらに、最近のファイアウォールは、このタイプの発見の試みに応答しないように設定できる。



ここまでくれば、基本的なネットワーク・ディスカバリーができるようになる。これからは、Nmapの動作をよりコントロールできるようにするために、さらにいくつかのオプションを見ていこう。



### III.Nmapによるポートスキャンを使わないネットワーク・ディスカバリー



お気づきかもしれないが、デフォルトでは、Nmapはアクティブなホスト**を検出した後にポートスキャンを実行する**。ネットワーク上に5台のホストがある場合、Nmapは約5,000個のポートの状態をチェックしようとするので、時間がかかる。



しかし、Nmapのオプションを使って、ネットワーク上のアクティブなホスト**を発見するだけで、そのサービスを発見しないことも可能である**。



どのホストが到達可能かを知りたいだけで、そのホストが公開しているサービスやポートに関する情報が必要ない場合は、「-sn」オプションを使って、**ICMPエコー（ping）とARPリクエスト**を使ったスキャンだけを実行することができる。言い換えれば、ポートスキャンを完全に無効にする：



```
# Scan a CIDR in Echo ICMP
nmap 192.168.1.0/24 -sn
```



以下は、ポートスキャンを行わずにNmapによるネットワーク・ディスカバリーを行った結果である：



![nmap-image](assets/fr/29.webp)



Nmapでポートスキャンを行わずにネットワークを検出した結果。



ホスト検出にICMPだけを使うことの限界はすでに述べた(リモートネットワークの場合)。そのため、Nmapはホスト上のファイアウォールや特定のサービスの存在を裏切ることができるいくつかのトリックも使用している。オプションの助けを借りて、これらのトリックを再利用したり、拡張したりすることができる。



そのために、"-PS"（TCP SYN）と"-PA"（TCP ACK）オプションを使用する。"-PP "オプションと同様に、ホスト・ディスカバリーの一部として参加したいポートを指定することができる：



```
# Scan specific ports on a CIDR
nmap -sn -PP -PS22,3389,445,139 -PA80 192.168.1.0/24
```



このスキャンは、デフォルトのオプションの場合よりも、ホストの発見が少し完全なものになることをすでに保証している。



複数のオプションを使って、かなり包括的なコマンドを使えるようになってきた。これは、Nmapがどのように機能するのか、そして「デフォルト」のオプションの限界を知っているからである。時間をかけてNmapをマスターする意味はそこにある！



前回の注文のオプションを詳しく説明する：





- 「sn`：Nmapが発見した各アクティブホストのポートスキャンを無効にする。





- "`-PP`: ホスト発見のためのICMPエコー(pingスキャン)を有効にする。





- "`-PS <PORT>`": 指定されたポートにTCP SYNパケットを送り、pingに応答しなかったホストの存在を裏切るアクティブなサービスを検出する。





- "`-PA <PORT>`": TCP ACKパケットを指定されたポートに送り、pingに応答しなかったホストの存在を裏切るアクティブなファイアウォールを検出する。




上の例では、「-PS」オプションに、Nmapのコンテキストで最も頻繁に公開されていると思われるポートを指定している。これらの異なるポートを各ホストでテストし、そのホストがホストしているサービスが本当にアクティブかどうかを確認するのではなく、（サービスまたはホストのファイアウォールからの応答によって）まだアクティブでありながら、ICMPエコーに応答していないホストを発見できるかどうかを確認する。



このようなスキャンの際に撮影されたネットワーク・キャプチャー（この場合は1台のターゲット・ホスト上の抽出）を見てみよう：



![nmap-image](assets/fr/30.webp)



高度なネットワーク・ディスカバリーの際にNmapが送信するパケット。



TCP SYNパケット、ポートTCP/80上のTCP ACK、ICMPエコーを見つける。Nmapは、ネットワーク発見スキャンの対象となる各ホストに対して、これらのテストをすべて実行する。



### IV.Nmapでターゲットとする資産のファイルを使う



何十、何百ものネットワーク、サブネット、VLANで構成されていることもある現実の情報システムでは、ターゲットの指定はすぐに複雑になる。そのため、コマンドラインで1つ1つ指定するよりも、Nmapのソースとしてファイルを使う方が簡単なのだ。



まずはじめに、1行に1つのエントリーを含むシンプルなファイルを作成する：



![nmap-image](assets/fr/31.webp)



1行に1つのターゲット(ホストまたはネットワーク)を含むファイル



次に、これまで見てきたすべてのNmapオプションを使い、「-iL <path/file>」オプションを指定する：



```
# Scan a list of targets contained in a file
nmap -iL /tmp/mesCibles.txt
```



Nmapは、このファイルに含まれるすべてのターゲットをスキャン対象に含める。



すべてのターゲットが考慮されていることを確認したい場合は、"-sL -n "オプションを使うことができる。Nmapは、ネットワーク上にパケットを送信することなく、ファイル内のCIDRとホストだけを解釈して表示する：



```
# Display targets contained in a file
nmap -iL /tmp/mesCibles.txt -sL -n

Starting Nmap 7.94SVN ( https://nmap.org ) at 2024-05-01 14:52 CEST
Nmap scan report for 192.168.0.0
Nmap scan report for 192.168.0.1
Nmap scan report for 192.168.0.2
Nmap scan report for 192.168.0.3
Nmap scan report for 192.168.0.4
Nmap scan report for 192.168.0.5
Nmap scan report for 192.168.0.6
Nmap scan report for 192.168.0.7
Nmap scan report for 192.168.0.8
Nmap scan report for 192.168.0.9
Nmap scan report for 192.168.0.10
Nmap scan report for 192.168.0.11
Nmap scan report for 192.168.0.12
```



これにより、スキャンするホストのリストが正確であることが保証される。



最後に重要なヒントとして、スキャンの一部としての**ホストまたはネットワークの除外**に関 する話をしたい。特に、**情報システムの機密コンポーネントがスキャンによって妨害されたり、混乱させられたりしないようにしたい場合**には、ホストを除外する必要があります。



このようなニーズのよくある例は、企業が産業用（PLC）機器やヘルスケア機器を所有している場合である。そのような機器は設計が不十分なことがあり、書式の悪いパケットや多すぎるパケットを受信することを全く想定していない。可用性やビジネス／人的リスクの明らかな理由から、テストから除外することが望ましい。



IPアドレスやネットワークをスキャン対象から除外するには、Nmapの「--exclude」オプションを使うことができる：



```
# Exclude an IP address in a CIDR scan
nmap 192.168.1.0/24 --exclude 192.168.1.140
```



この例では、ネットワーク「192.168.1.0/24」をスキャンしているが、そこにあるホスト「192.168.1.140」は除外している。Nmapはこのホストにパケットを送らない。サブネット除外を使った別の例：



```
# Exclude a subnet in a CIDR scan
nmap 10.0.0.0/16 --exclude 10.0.100.0/24
```



同様に、大きなネットワーク "10.0.0.0/16 "をスキャンするが、ネットワーク "10.0.100.0/24 "はスキャンされない。繰り返しになるが、どのホストがスキャンされ、どのホストがスキャンから除外されるかを非常に明確に把握するために、"-sL -n "オプションを使うことをお勧めする。



### V.ネットワーク・ディスカバリーとポート・スキャン



このセクションで学んだことを、前のセクションでポートスキャンオプションについて学んだことと組み合わせることができる。デフォルトでは、Nmapは発見したすべてのアクティブホストで、最も頻繁に使用される1000個のポートをスキャンすることがわかった。この動作が必要でなければ、この動作を防ぐ方法を見たが、完全に制御することもできるし、必要であれば拡張することもできる。



例えば、以下のコマンドは、スキャンされた各ホスト上のポートTCP/22でリッスンしているサービスの存在をチェックする：



```
# Scan TCP/22 on a CIDR
nmap 192.168.0.0/24 192.168.1.0/24 -p 22
```



Nmapは、まずネットワーク・ディスカバリーを実行してアクティブなホストをリストアップし、その各ホストについて、ポートTCP/22にサービスが存在するかどうかをチェックする。



同様に、「192.168.0.0/24」ネットワーク上で発見されたすべてのホストのすべてのTCPポートを、例えばホスト「192.168.0.4」を除いてフルスキャンすることができる：



```
# Port scan of a CIDR with exclusion
nmap 192.168.0.0/24 --exclude 192.168.0.4 -p-
```



これまで学んできたすべてのオプションを、自分のニーズに合わせて自由に組み合わせることができる。

## 6 - Nmapによるサービスとオペレーティング・システムのバージョンの検出



### I.プレゼンテーション



このセクションでは、Nmapを使用して、スキャンされたホストが使用しているサービスとオペレーティングシステムのバージョンを発見し、正確に検出する方法を学ぶ。Nmapがどのようにしてこのタスクを達成するのか、また、その結果をよりよく理解し解釈するために、ツールの制限についても詳しく見ていく。



このチュートリアルの前のセクションで見てきたように、デフォルトでは、Nmapはスキャンしてオープンとみなされたポートで公開されているサービスを確認しない。したがって、ポートTCP/22でWebサービスをリッスンしている場合、Nmapはそれをオープンとして報告し続けるが、`SSH`サービスとして報告する。これは、ポート/プロトコルとサービス名(`/etc/services/`ファイル)の関係を探すために、システムローカルの[データベース](https://www.it-connect.fr/cours-tutoriels/administration-systemes/stockage/bdd/)を使用するためである。



ほとんどの場合、[Nmap](https://www.it-connect.fr/cours/nmap-cartographie-reseau-scan-de-vulnerabilites/)が正しい情報を提供してくれる。しかし、残りのケースは、古典的なサービス([SSH](https://www.it-connect.fr/cours/comprendre-et-maitriser-ssh/)、HTTPなど)が古典的でないポート(たとえば、SSHサービスの2022)で公開されている状況である。この場合、Nmapはローカルデータベースで一致するものを見つけられないか、現実に一致しないものを見つけ、重要な情報を見逃してしまう。



幸いなことに、Nmapは、開いているポートの背後に隠れているかもしれない正確なサービスを発見するための、非常に正確なオプションとメカニズムを提供している。さらに、正確な技術とバージョンを特定するためのクエリとシグネチャのデータベースまで備えている。サービスに加え、Nmapは使用されているテクノロジーとそのバージョンも特定できる。



それがこのセクションで見ていくことになる。



### II.技術やバージョンの検出方法



#### A.技術またはバージョンの識別方法の注意事項



テクノロジーとバージョンを識別するには、対象となるポートでリッスンしているサービス、CMS、アプリケーション、ソフトウェアの名前を取得する必要があります。例えば、ウェブページはCMS（`WordPress`）によって管理され、ウェブサービス（`Apache`、IIS、Nginx）によって実行され、サーバー（LinuxまたはWindows）によってホストされています。しかし、どのウェブサービスが稼働しているかはどうやって知るのでしょうか？



この情報を見つける古典的な方法は「バナー・グラビング（banner grab）」であり、問題のサービスがこの情報を表示している場所を特定し、そのデータを読み取るだけである。多くの場合、デフォルトの設定や処理では、サービスは接続後の最初のレスポンスとして名前とバージョンを表示します。



![nmap-image](assets/fr/32.webp)



FTPサービスによってTCPコネクションが確立されると、すぐにバージョンを表示する。



ここでは、`telnet`を経由してこのサービスに単純なTCP接続をすると、そのサービ スのテクノロジーとバージョンを含むTCPパケットになることがわかる。



相手のサービスの種類がわかったら、そのサービスに対して特定のコマンドやリクエストを送り、情報を引き出すこともできる。これらのリクエストやコマンドをやみくもに（正しいタイプのサービスであることを確信することなく）送ることもできる。



また、より高度なケースでは、特定のパケットを送信してエラーなどの反応を起こさせ、使用されているバージョンや技術に関する詳細な情報を提供する必要がある。



ご想像のとおり、Nmapはこれらすべてのテクニックを使って、ポートでホストされているサービスの正確なタイプや、その技術名、バージョンを特定しようとする。



#### B.Nmapのプローブとマッチを理解する



スキャンされた各ポートでこれらのチェックをすべて実行するために、Nmapは（バイナリやそのモジュールと同様に）頻繁に更新されるローカルデータベースを使用する。これは数千行のテキストファイルである：usr/share/nmap/nmap-service-probes`である。



このファイルは、2つの主要なガイドラインを中心に構成された多数の項目で構成されている：





- Probe`：これはNmapが識別されるべきサービスからの反応を引き起こすために送信するパケットの定義である。これは、_Hello?Guten Tag?こんにちは？ブエノス・ディアスかな？対象となるサービスが、自分が理解できる（つまり正しいプロトコルを話す）プローブを受信するとすぐに、Nmapに応答する。





- Match": Nmapが取得したレスポンスに適用する正規表現である。HTTP GETリクエストを送信することで、サービスからのレスポンスが返された場合、このレスポンスに対して何十もの正規表現を適用し、例えば`Apache`、`Nginx`、`Microsoft IIS`などの単語の存在を特定する。




特定のケースに対応するディレクティブは他にもいくつかあるが、Nmapの動作を理解し、その使い方をカスタマイズするための主なディレクティブはこれらである。この理論の部分をより具体的にするために、ここに例を示す：



![nmap-image](assets/fr/33.webp)



Nmapの `/usr/share/nmap/nmap-service-probes` ファイルにあるプローブの例。



この例の最初の行には、`GetRequest`というわかりやすいProbeがある。これは、HTTP/1.0を使ったウェブサービスルートへの空のHTTP GETリクエストを含むTCPパケットであり、その後に改行と空行が続く。



ports` 行は、Nmap にプローブを送信するポートを指定する。これにより、テストに優先順位をつけ、時間を節約することができる。



最後に `match` の例を2つ示す。最初の例は、この行に含まれる正規表現が受信したサービスのレスポンスにマッチした場合、スキャンされたウェブサービスを `ajp13` に分類する。



プローブがどのようなものかを理解するために、このファイルにあるプローブのリストを以下に示します（このチュートリアルを書いている時点では全部で188あります）。



![nmap-image](assets/fr/34.webp)



Nmapが使用し、ファイル `/usr/share/nmap/nmap-service-probes`._ に格納されているプローブの例。



最初の2つ(`NULL`と`GenericLines`と呼ばれる)は、単に空のTCPパケットを送信するか、改行を 含んだパケットを送信するので、ここで特に注目される。サーバーサービスは、クライアントからの特別なアクションやコマンド、リクエストな しに、コネクションを受信すると同時に自分自身を正確にアナウンスすることがよくあ る。



もう少し複雑な_match_のケースを紹介しよう：



```
# Match Nginx + version in an error 400 page
match ssl/http m|^HTTP/1.1 400 Bad Request\r\n.*?Server: nginx/([\d.]+)[^\r\n]*?\r\n.*<title>400 The plain HTTP request was sent to HTTPS port</title>|s p/nginx/ v/$1/ cpe:/a:igor_sysoev:nginx:$1/
```



m|` と `|` の間に正確な正規表現が含まれています。この例全体をじっくり読んでください。正規表現の中に、バージョンを分離するために使われる `([\d.]+)` という選択があることに気づくでしょう。この例では、製品名 `p/nginx/`、取得したバージョン `v/$1/`、バージョン `cpe:/a:igor_syssoev:nginx:$1/`のCPEなど、他のElementsも定義しています。



CPE（Common Platform Enumeration）は、ソフトウェアとハードウェアを識別し、記述するために使用される標準化された表記体系である。このフォーマットにより、脆弱性やセキュリティ設定をより効率的に管理することができる。cpe:/o:microsoft:windows_8.1:r1`と`cpe:/a:apache:http_server:2.4.35`である。



ここでは、OSを表す「o」、アプリケーションを表す「a」、ベンダー、製品、バージョンを明確に区別する。



そのため、これらの正規表現で_match_された場合、サービス名だけでなく、バージョンと正確なCPEも取得し、このバージョンに影響するCVEを見つけやすくする。この情報はNmapの標準出力で見ることができ、いくつかのセクションで取り上げる他の目的にも非常に有用であることがわかるだろう。



matches_の正確な構文と、より一般的には`/usr/share/nmap/nmap-service-probes`ファイル内のディレクティブの構文はこれだけにとどまらない。しかし、少なくともこのファイルの存在と一般的な操作方法については覚えておいた方がよい。後々、結果を理解したり、デバッグしたり、スキャンをカスタマイズしたり、あるいはNmapの開発に貢献したりするときに役に立つだろう。

### I.プレゼンテーション

### II.Nmapを使って脆弱性を検出する



Nmapネットワークスキャナーを使って、情報システムのサービスとシステムの脆弱性を検出したい。つまり、アクティブなホストを発見し、公開されているサービスを列挙し、バージョンと技術を検出することに加えて、Nmapは脆弱性を探すことになる。



これを実現するために、NmapはNSE（_Nmap Scripting Engine）スクリプトに依存している。NSEスクリプトは、テストへのきめ細かなアプローチを可能にするモジュールとみなすことができる。



適切なオプションを使えば、Nmapに、発見された各サービスでさまざまなNSEスクリプトを使うよう依頼することができる：





- コンフィギュレーションの不具合 ；





- さらに進化したバージョンとOSの発見；





- 既知の脆弱性 (CVEs) ；





- 弱い識別子；





- マルウェア感染の特徴的なElements ；





- サービス拒否の可能性；





- などなど。




おわかりのように、NSEスクリプトは、実行可能なネットワーク操作という点で、Nmapの能力を大幅に拡張する。そして、これまでよりもはるかに高度なタスクを実行できるようになった。良い知らせは、いつものように、これらの機能を単にオプションを介して、デフォルトのコンテキストで使用できることである。これが次に見るものだ。



### III.脆弱性スキャンの例



NSEスクリプトは、Nmapを使用してホストの単一ポート、そのホストの全サービス、または複数のネットワークで検出された全サービスをスキャンする場合に使用できる。そのため、これから紹介するオプションは、これまでに調べたすべてのコンテキストで使用できる。

本番の情報システムでこの機能を使う前に、このチュートリアルを読み進めてください。以下の節では、実行されるテストの影響や種類をうまく制御する方法について見ていきます。

### IV.脆弱性スキャンにおけるNmapの限界を理解する



はっきりさせておこう。Nmapは、情報システムの完全な侵入テストを実施したり、レッド・チームの活動をシミュレートしたりすることはできない。Nmapにはいくつかの制限があり、誤った安心感を持たないようにするためには、その制限を認識しておく必要がある：





- **限定されたカバレッジ**：NmapのNSEスクリプトは強力ですが、他の専門的な脆弱性発見ツールと比較すると、テストカバレッジが限定される場合があります。Active Directoryの脆弱性、機密データの暴露、脆弱なWebアプリケーションの高度なケースなど、利用可能なNSEスクリプトではカバーできない脆弱性もあります。





- **脆弱性の複雑さ**：ある種の脆弱性は、その複雑さゆえにNSEスクリプトを使用して検出することが困難な場合がある。たとえば、リモートサービスとの複雑なインタラクションを必要とする脆弱性は、Nmapでは効果的に検出できない可能性がある（ファイル共有の過剰なパーミッションやWebアプリケーションのパーミッション制御の欠陥の場合など）。





- **受動的検知**：Nmapは、脆弱性を検出するために主にアクティブスキャンに重点を置いているため、ターゲットホストとのアクティブな接続を確立しなければ、潜在的な脆弱性を効果的に検出できない可能性がある。そのため、アクティブスキャン中に顕在化しない脆弱性は見逃される可能性がある（ウェブアプリケーションのコードインジェクションの場合など）。





- **更新への依存**：NmapのNSEスクリプトの[データベース](https://www.it-connect.fr/cours-tutoriels/administration-systemes/stockage/bdd/)は常に進化しているが、新しい脆弱性が発見されてから対応するスクリプトがNmapに追加されるまでに時間がかかることがある。その結果、Nmapが常に最新の脆弱性に対応しているとは限らない。





- 偽陽性と偽陰性：どのセキュリティツールでもそうであるように、NmapのNSEスクリプトは偽陽性（偽の脆弱性警告）や偽陰性（検出されなかった本当の脆弱性）を生成する可能性がある。これは、Nmapの結果を分析する際に留意すべきことである。




そのため、Nmapが何をし、何をしないかを理解し、同様にその結果をどのように解釈するかを知ることが重要である。特に、このチュートリアルを通して、デフォルトのオプションは、注意深く使用することで発見できる重要なElementsを見逃す可能性があることを見てきた。



ネットワーク・システム管理者であれ、セキュリティ・エンジニアであれ、あるいはCISOであれ、Nmapを使えば情報システムのセキュリティ状況の概要を把握することができる。これは、システムの安全性を確保するための重要な第一歩であり、ITチームが定期的に実施することができる。しかし、Nmapよりもはるかに包括的に弱点を発見することができる[サイバーセキュリティ]専門家（https://www.it-connect.fr/cours-tutoriels/securite-informatique/）の介入や助言に取って代わるべきものではない。

## 8 - NmapのNSEスクリプトを使う



### I.プレゼンテーション



このセクションでは、NSE（_Nmap Scripting Engine）スクリプトについて詳しく見ていく。特に、スクリプトがこのツールの大きな長所の1つである理由、スクリプトの動作方法、既存のスクリプトの閲覧と使用方法について説明する。



このセクションは、Nmapの脆弱性スキャン機能の基本的な使い方を学んだ前回のチュートリアルに続くものである。ここでは、[Nmap](https://www.it-connect.fr/cours/nmap-cartographie-reseau-scan-de-vulnerabilites/)がどのように機能するかを詳しく見ていくことで、より正確で制御されたスキャンを再び実行できるようにする。



### II.Nmap NSEスクリプトのコンセプト



NmapのNSEスクリプトを使えば、非常に柔軟な方法で機能を拡張することができる。これらのスクリプトは、Nmapが使用するCやC#よりも扱いやすく、アクセスしやすいスクリプト言語であるLUAで記述されている。スタンドアロンツールではなく、NmapとともにLUAスクリプトを使用する利点は、Nmapの実行速度と標準機能（ホスト、ポート、バージョンの検出など）を活用できることだ。



これらのスクリプトはカテゴリー別に分類されており、1つのスクリプトが複数のカテゴリーに属することもある：



| Catégorie       | Description |
|----------------|-------------|
| **auth**       | Contient les scripts relatifs à l’authentification sur des services, dont l’accès anonyme ou l’énumération des utilisateurs. Exemples: `oracle-enum-users`, `ftp-anon`. |
| **broadcast**  | Contient les scripts relatifs aux opérations de broadcast sur le réseau, notamment en vue d’exploiter et de découvrir certains services, hôtes ou protocoles reposant sur le broadcast (IPv6, wake on lan, IGMP, etc.). Exemples: `broadcast-dhcp6-discover`, `broadcast-ospf2-discover`. |
| **brute**      | Contient les scripts relatifs aux opérations de brute force de l’authentification sur les services (brute force [SSH](https://www.it-connect.fr/cours/comprendre-et-maitriser-ssh/), MSSQL, etc.). Exemples: `ssh-brute`, `vnc-brute`. |
| **default**    | Contient les scripts utilisés dans le cas par défaut (utilisation de `-sC`). Plusieurs critères sont utilisés afin de valider l’entrée d’un script dans cette catégorie dont la vitesse d’exécution, la structure de la sortie, la fiabilité du test, le caractère “intrusif” ou “risqué”, etc. |
| **discovery**  | Contient les scripts relatifs à la découverte avancée du réseau et des services. On y retrouve par exemple l’énumération du contenu d’un partage SMB, d’une version d’un service VNC, des requêtes SNMP, etc. Exemples: `mysql-info`, `http-security-headers`. |
| **dos**        | Contient les scripts pouvant causer un déni de service. Il peut s’agir de scripts créés pour exploiter une vulnérabilité de type déni de service ou alors de scripts ayant pour effet de bord un déni de service. Prudence donc (ils sont exclus de la catégorie `default`). Exemples: `http-slowloris`, `ipv6-ra-flood`. |
| **exploit**    | Contient les scripts créés pour exploiter de manière directe une vulnérabilité. Exemples: `http-shellsock`, `smb-vuln-ms08-067`. |
| **external**   | Contient les scripts qui nécessitent l’utilisation d’une ressource tierce, comme une base d’information en ligne. Cela indique notamment une tentative de connexion vers l’extérieur (attention à la confidentialité). Exemples: `whois-ip`, `dns-blacklist`, `ip-geolocation-geoplugin`. |
| **fuzzer**     | Contient les scripts conçus pour envoyer des trames, paquets ou paramètres inattendus par un service. Cela permet notamment de causer des erreurs ou dysfonctionnements afin d’obtenir des pistes de vulnérabilité ou des informations techniques. Exemples: `dns-fuzz`, `http-form-fuzzer`. |
| **intrusive**  | Contient les scripts qui sont catégorisés comme “risqués” d’un point de vue disponibilité, ou détection. Ils peuvent provoquer un crash du système ou être détectés comme malveillant par une solution de sécurité. Il s’agit de la catégorie inverse de `safe`. Exemples: `smtp-brute`, `smb-vuln-ms08-067`, `smb-psexec`. |
| **malware**    | Contient les scripts conçus pour détecter la présence d’élément caractéristique d’un malware, tel qu’un port en écoute communément utilisé par une backdoor connue. Exemples: `ftp-proftpd-backdoor`, `smtp-strangeport`. |
| **safe**       | Contient les scripts qui sont considérés comme sûrs d’un point de vue détection ou stabilité. Il s’agit de la catégorie inverse de `intrusive` et elle contient en grande majorité des scripts avancés d’identification de version ou de relevé d’élément de configuration. Exemples: `html-title`, `smb2-security-mode`, `ms-sql-info`. |
| **version**    | Contient les scripts qui permettent une détection avancée de version. Ils peuvent être utilisés en complément des Probes et Matchs étudiés précédemment quand la détection d’une version nécessite des opérations un peu plus complexes. Exemples: `http-php-version`, `vmware-version`. |
| **vuln**       | Contient les scripts conçus pour détecter la présence de vulnérabilité connue (CVE) sans pour autant les exploiter (à l’inverse de la catégorie `exploit`). Ils se contentent en général de rapporter le statut “vulnérable” ou non d’un service. Exemples: `smb-vuln-ms17-010` (eternal blue), `http-phpmyadmin-dir-traversal`. |


技術的には、スクリプトが属するカテゴリーは、そのコードに直接示される。

### IV.NmapのNSEスクリプトを使う



では、興味のあるNSEスクリプトを注意深く選びながら、脆弱性スキャンを実行する方法を学んでいこう。



#### A.カテゴリー別にスクリプトを選択



まず始めに、特定のカテゴリーに属するすべてのスクリプトを実行するように選択することができる。引数 `--script <category>` で、このカテゴリーまたはこれらのカテゴリーをNmapに指定する必要がある：



```
# Use default NSE scripts
nmap --script default 10.10.10.152
```



この最初のコマンドは `nmap -sC` コマンドに相当する。デフォルトでは、Nmapは`default`カテゴリのスクリプトを選択するが、これは議論のためのものである。次のコマンドは、例えば、`discovery` カテゴリのすべてのスクリプトを使用する：



```
# Use NSE scripts from the "discovery" category
nmap --script discovery 10.10.10.152
```



これまで見てきたように、関連するNSEスクリプトが何を行うかをすばやく識別できるカ テゴリ（`discovery`、`vuln`、`exploit`）もあれば、実行するテストのリスク、検出、安 定性のレベルを定義するカテゴリもある。スクリプトの選択によって実行されるさまざまなアクションを十分に把握できないような微妙な状況であれば、`discovery`と`safe`のカテゴリーに属するスクリプトだけを選択するように、選択項目を組み合わせることもできる：



```
# Use scripts from multiple categories
nmap --script "discovery and safe" 10.10.10.152
```



dos`と`intrusive`のカテゴリーからスクリプトを絶対的かつ明示的に除外したい場合は、以下の表記を使うことができる：



```
# Exclude categories
nmap --script "not intrusive and not dos" 10.10.10.152
```



上記のように除外条件を指定すると、明示的に除外されていない他のすべてのカテゴリーが使用されることに注意してください。より公平を期すために、例えば次のように指定することもできる：



```
# Include scripts from the "vuln" category except those that are too risky
nmap --script "vuln and not intrusive and not dos" 10.10.10.152

# Same thing, but only targeting the HTTP protocol
nmap --script "(http and vuln) and not intrusive and not dos" 10.10.10.152
```



ここでは、特にNmapを実際のコンテキストで脆弱性分析に使用する場合に、NSEスクリプトをカテゴリ別にどのように扱うかの例をいくつか紹介する。



#### B.ユニットとしてスクリプトを選択



分析中に特定のテスト（NSEスクリプトに対応するテスト）を1つだけ実行することもできます。そのためには、スクリプトの名前を `-script <name>` パラメータで指定する必要がある。ftp-anon.nse`スクリプトを例に挙げる：



```
# Use an NSE script and a specific port
nmap --script ftp-anon -p 21 10.10.10.152
```



そうすれば、非常に正確な結果が得られる：



![nmap-image](assets/fr/45.webp)



Nmap._経由でFTPポートにNSEの`ftp-anon`スクリプトを使用した結果。



ポート21で `ftp-anon` スクリプトを実行した結果が表示されるが、`-p 21` オプションを指定したため、他のポートは表示されない。基本的なポートスキャンを実行し、検出されたFTPサービスに対してのみ `ftp-anon` NSEスクリプトを実行することもできた：



```
# Use a specific NSE script
nmap --script ftp-anon 10.10.10.152
```



したがって、Nmapは、別のポートでFTPサービスを見つけた場合にも、この匿名接続テストを実行したことになる。



NSEスクリプトが何をするのかを簡単に説明するには、前述の`--script-help`オプションを使用することができます：



![nmap-image](assets/fr/46.webp)



NSE スクリプト `sshv1`._ の結果を表示します。



要するに、これまで使ってきたネットワーク・ディスカバリーのオプション、サービス、バージョン、テクノロジーをもう一度すべて再利用できるのだ！



#### C.スクリプト引数の管理



Nmapを使用していると、正しく機能するために入力引数を必要とするNSEスクリプトに出くわすことがある。ここでは、Nmapのオプションを使ってこれらのスクリプトに引数を渡す方法を見ていく。



例として、SSHサービスに対してブルートフォースアタックを実行できる `ssh-brute` スクリプトを見てみよう。



古典的なブルートフォース攻撃は、特定のアカウントへの認証を試みて、いくつかのパスワー ド（時には数百万）をテストするものである。これほど多くのパスワードを試すことで、攻撃者は、ユーザーが攻撃に使われたパスワード辞書の中で弱いパスワードを使った確率に賭ける。



このスクリプトには「デフォルト」のオプションが用意されているが、文脈に合わせてカスタマイズすることができる。例えば、この攻撃の文脈では、使用するユーザーのリストとパスワード辞書をNmapに提供することができる。私の知る限り、スクリプトに必要な引数を簡単にリストアップすることはできないので、最も確実な方法は、Nmapの公式ウェブサイトを訪問することである。NSEスクリプトのドキュメントへの直接リンクは、`--script-help`オプションに応答して得ることができる：



![nmap-image](assets/fr/47.webp)



NSE `ssh-brute` スクリプトのヘルプを nmap.org._ へのリンクとともに表示した結果。



表示されたリンクをクリックすると、サイト[https://nmap.org](https://nmap.org/)のこのウェブページにアクセスします：



![nmap-image](assets/fr/48.webp)



Nmap の `ssh-brute` NSE スクリプトに渡すことができる引数のリスト。



ここでは、使用可能な引数を明確に示している。主な引数は `passdb` (パスワードのリストを含むファイル) と `userdb` (ユーザーのリストを含むファイル) である。これらのブルートフォースメカニズムと関連するオプションは、複数のスクリプト（`ssh-brute`、`mysql-brute`、`mssql-brute`など）で統一的に使用できるように相互化されているため、多かれ少なかれ同じ引数を持つことになる：



```
# Create a file containing my user list
echo "root" > /tmp/userlist

# Create a file containing my password list
echo "123456" > /tmp/passlist
echo "NomEntreprise75" >> /tmp/passlist
echo "changezmoi" >> /tmp/passlist

# Run an SSH brute force via Nmap network scan
nmap --script ssh-brute --script-args userdb=/tmp/userlist,passdb=/tmp/passlist 10.10.10.245 192.168.1.19
```



この最後のコマンドでわかるように、`--scripts-args key=value,key=value`オプションを使用して、Nmapスクリプトに必要な引数を指定することができる。以下は、`ssh-brute` NSEスクリプトを使ってSSHブルートフォースを実行したときのNmapの出力結果の可能性である：



![nmap-image](assets/fr/49.webp)



Nmap._によるSSHブルートフォース実行結果



ご覧のように、NSEスクリプトによって生成された情報は、対話型出力(端末出力)の先頭に`NSE: [スクリプト名]`が付くので、見つけやすくなっている。通常のNmap結果の表示では、弱い識別子が発見されたかどうか（パスワードを含む）を示すサマリーが表示されるだけである。



さらに一歩進んで、すでに見てきたすべてのオプションに加えて、これらすべてを使用することができることを思い出させるために、`10.10.10.0/24`ネットワークを発見し、最も頻繁に使用される2000のTCPポートをスキャンし、FTPサービスの匿名アクセス検索とSSHサービスのブルートフォースキャンペーンを実行するコマンドを以下に示す：



```
# Example of a complete command using multiple scripts
nmap --top-ports 2000 10.10.10.0/24 --script ftp-anon,ssh-brute --script-args userdb=/tmp/userlist,passdb=/tmp/passlist
```



これは、利用可能なスクリプトとそのオプションのほんの一例に過ぎない。しかし、これでNSEスクリプトの扱い方、スクリプトが引数を必要とするかどうか、そしてこれらの引数をNmapに渡す方法について、よりよく理解できるようになった。

## 9 - Nmap出力データの管理




### I.プレゼンテーション



このセクションでは、Nmapが生成する出力について、特にこの出力をフォーマットするための様々なオプションについて見ていく。Nmapは、さまざまなニーズに合わせていくつかの出力形式を作成することができ、これもこのツールの大きな強みの1つであることがわかるだろう。



デフォルトでは、Nmapは実行したスキャンとテストの結果を詳細に表示する。これには、スキャンされたホストやサービス、アクセス可能として検出されたもの、オープン・ポートの詳細、そのステータス、バージョンなどが含まれる。さらに、NSEスクリプトの詳細もターミナル出力で確認できる。しかし、この出力は、情報を明確にフォーマットしても、すぐに膨大な量になり、結果から正確な情報を見つけるのが難しくなることがある。



### II.Nmapの出力フォーマットをマスターする



#### A.スキャン結果をテキストファイルに保存する



より簡単にするために、[Nmap](https://www.it-connect.fr/cours/nmap-cartographie-reseau-scan-de-vulnerabilites/)では出力をテキストファイルに保存することが非常に簡単にできる。これはアーカイブや他のテストとの比較に役立つだけでなく、Sublime text、[PowerShell](https://www.it-connect.fr/cours-tutoriels/administration-systemes/scripting/powershell/)、Python、grep、sedなどのような特殊なワープロツールやスクリプト言語でこの出力を参照するのにも便利である。Nmapの標準出力をテキストファイルに保存するには、`-oN <ファイル名>` オプション（"normal "の "N"）を使用する：



```
# Save Nmap output to a file
nmap 10.10.10.0/24 -sV -oN nmap_scan_10.10.10.0_24.txt
```



Nmapは通常の標準出力をターミナルに表示するが、指定したファイルにも表示する。



#### B.generate Nmap出力の凝縮されたフォーマット



人間が簡単に解釈できる「テキスト」スタイルには、「グレッパブル」フォーマットという第二の出力フォーマットもある。



このフォーマットは、`grep`のようなツールで処理しやすいように構造化された、Nmapの出力を "凝縮した "ビューを提供するために作成された。このタイプの出力の例を見てみよう：



![nmap-image](assets/fr/50.webp)



nmap ネットワークスキャンと「grepable」フォーマットでの出力。



ここでは、ネットワーク・ディスカバリーとポート・スキャンを実行し、/24ネットワーク上のテクノロジーとバージョンを分析した。最終的に、アクティブなホストごとに2行のファイルができた：





- 最初の行は、このようなホストが_Up_していると教えてくれる；





- 2行目は、どのポートがスキャンされたか、そのステータス、そして非常に特殊なフォーマットで取得されたテクノロジーとバージョン情報を教えてくれる：<ポート>/<ステータス>/<プロトコル>/<サービス>/<バージョン>/,`」。




この固定区切り文字によるフォーマットは、`grep`のようなワープロツールや、スクリプトやプログラミング言語による迅速な処理を可能にする。例えば、以下のコマンドは、Nmapによって実行された巨大なスキャンの場合、その出力をブラウズするのが難しいが、ホスト`10.10.10.5`に関する情報を簡単に取り出すことができる：



```
# Filter by IP address in the Nmap "greppable" file
grep '10.10.10.5' nmap_10.10.10.0.gnmap
Host: 10.10.10.5 () Status: Up
Host: 10.10.10.5 () Ports: 21/open/tcp//ftp//Microsoft ftpd/, 80/open/tcp//http//Microsoft IIS httpd 7.5/ Ignored State: filtered (998)
```



逆に、ポートとIPが同じ行にあるので、ポート21を開いているすべてのホストを簡単にリストアップすることもできる：



```
# Filter by open port in the Nmap "greppable" file
grep '21/open' nmap_10.10.10.0.gnmap
Host: 10.10.10.5 () Ports: 21/open/tcp//ftp//Microsoft ftpd/, 80/open/tcp//http//Microsoft IIS httpd 7.5/ Ignored State: filtered (998)

Host: 10.10.10.152 () Ports: 21/open/tcp//ftp//Microsoft ftpd/, 80/open/tcp//http//Indy httpd 18.1.37.13946 (Paessler PRTG bandwidth monitor)/, 135/open/tcp//msrpc//Microsoft Windows RPC/, 139/open/tcp//netbios-ssn//Microsoft Windows netbios-ssn/, 445/open/tcp//microsoft-ds//Microsoft Windows Server 2008 R2 - 2012 microsoft-ds/ Ignored State: closed (995)
```



このような出力をgenerateするためには、`-oG <ファイル名>.gnmap`オプション（"grep "の "G"）を使う必要がある。私は習慣的に、このようなファイルには`.gnmap`という拡張子を使うが、好きな方を使えばいい：



```
# Save Nmap output to a file in "greppable" format
nmap 10.10.10.0/24 -sV -oG nmap_scan_10.10.10.0_24.gnmap
```



このフォーマットはさまざまな目的に使用でき、特に迅速なスクリプト作成／ソートには便利である。とはいえ、次に紹介する形式に取って代わられる傾向にある。



note: Nmap 7.90以降、`-oG`グレッパブルフォーマットは公式に非推奨となった。互換性のために使用することはできる。しかし、開発や自動パースにはXMLまたは通常のフォーマットを使用することを推奨する。



#### C.Nmap出力のXMLフォーマット



このチュートリアルで言及する価値のある最後のフォーマットはXMLである。前の2つのフォーマットとは異なり、このフォーマットは人間が読むためのものではなく、他のツールやスクリプトが読むためのものです。



XML（_eXtensible Markup Language_）は、データの保存と転送に使われるマークアップ言語で、カスタムタグによる階層構造を提供する。



Nmapでは、XML形式を使用して、ホスト、ポート、検出された脆弱性に関する情報、および標準のNmap出力では表示されない追加情報を含む、実行されたスキャンに関する詳細なレポートをgenerateに出力する。



XML形式の出力ファイルをgenerateするためには、`-oX`オプション（"XML "の "O"）を使う必要がある：



```
# Save Nmap output to a file in XML format
nmap 10.10.10.0/24 -sV -oX nmap_scan_10.10.10.0_24.xml
```



その結果、Nmapの標準出力が端末に出力され、XML形式のファイルがカレントディレクトリに出力される。



もちろん、XMLフォーマットは人間が読んで解釈するようには設計されていない。とはいえ、この形式のNmap出力でスクリプトや自動分析を行う場合、使用されるタグや構造を把握しておく必要がある。例えば、Nmapが作成したXMLファイルの内容の一部を以下に示す：



![nmap-image](assets/fr/51.webp)



Nmapスキャン中の1ホストのXMLレコードの例



ここには多くの情報があり、特に2つのオープンポートに興味がある：



```
<port protocol="tcp" portid="21"><state state="open" reason="syn-ack" reason_ttl="0"/><service name="ftp" product="Microsoft ftpd" ostype="Windows" method="probed" conf="10"><cpe>cpe:/a:microsoft:ftp_service</cpe><cpe>cpe:/o:microsoft:windows</cpe></service></port>
<port protocol="tcp" portid="80"><state state="open" reason="syn-ack" reason_ttl="0"/><service name="http" product="Microsoft IIS httpd" version="7.5" ostype="Windows" method="probed" conf="10"><cpe>cpe:/a:microsoft:internet_information_services:7.5</cpe><cpe>cpe:/o:microsoft:windows</cpe></service></port>
```



このフォーマットでは、各情報が専用のタグや属性名できちんと整理されているため、結果の自動解析が容易になると理解している。特に、CPEというこれまでになかった情報を見つけることができた。



```
cpe>cpe:/a:microsoft:internet_information_services:7.5</cpe><cpe>cpe:/o:microsoft:windows</cpe></service></port>

```

これにより、この情報を使用するデータベースやアプリケーションで情報を再利用することができる。私が特に考えているのは、CVEを参照するNVDデータベースである。各CVEについて、その脆弱性の影響を受けるCPEが含まれています。以下は、NVDデータベースの`a:microsoft:internet_information_services:7.5`に関するCVEの例です：



![nmap-image](assets/fr/52.webp)



NVDデータベースのCVEの詳細にCPEが存在すること。



Nmapによって収集または処理されたすべてのデータを含む、非常に明確な情報構造を提供するこのフォーマットの利点を、私たちはよりよく理解できるようになった。



反射的に、私はNmapスキャンを一度に3つのフォーマットで保存するようにしている。これは、`-oA <名前>`オプション（"A "は "All "の意味）により可能で、`<名前>.nmap`ファイル、`<名前>.xml`ファイル、`<名前>.gnmap`ファイルを作成する。こうしておけば、結果を見返したいと思ったときに、何もなくなってしまうことはないだろう。



これら3つのフォーマットで、Nmapの結果を保存し、最終的に自動化された方法で処理するために必要なものはすべて揃うはずである。次のセクションでは、Nmapを他のセキュリティツールと組み合わせて使用する場合に、XMLフォーマットを再び使用する。



#### III.NmapスキャンからHTMLレポートを生成する



XMLフォーマットは多くの可能性を提供するが、特にHTMLフォーマットでレポートを作成する際のベースとなる。

```

# Open the .html file with Firefox
firefox "Nmap – rapport web 05-2024.html"
```

### I.プレゼンテーション

いずれにせよ、Nmap debug (前の章で見たオプション `-d`) を使って、これから見るパラメーターのデフォルト値や、使おうとしているオプションが正しく考慮されているかどうかをチェックすることができる：



![nmap-image](assets/fr/63.webp)

### II.Nmapスキャンの速度を管理する



#### A.並列化の管理



デフォルトでは、Nmapはスキャンを最適化するために並列化を使用しており、使用するすべてのパラメータは様々なオプションで変更することができる。しかし、実際にこれらのパラメータを変更する必要があるケースはかなり稀であるため、このチュートリアルでは詳しく説明しない：





- min-hostgroup/max-hostgroup <size>`: パラレルホストスキャングループのサイズ。





- min-parallelism/max-parallelism <numprobes>`: プローブの並列化。





- --scan-delay/--max-scan-delay <time>`: プローブ間の遅延を調整する。




ただ、それが存在し、使用できることを知っているだけだ。



#### B.秒間パケット数の管理



デフォルトでは、Nmap自身がネットワークのレスポンスに応じて1秒あたりの送信パケット数を調整する。しかし、1秒あたりのパケット数の上限値や下限値を定義することで、この設定を強制することができる。この設定は `--min-rate <number>` と `--max-rate <number>` オプションを使って行う。例



```
# Limit the scan speed to 300 packets per second
nmap -sV 10.10.10.0/24 --max-rate 300
```



これらのオプションを使用すると、特定のニーズに応じてスキャン速度を調整し、プロセスを高速化したり、使用する帯域幅を制限したりすることができる。後者の場合（スキャン速度の制限）は、特にNmapの使用時にネットワーク遅延が発生する場合（かなり稀なケースだが）、これらのオプションにたどり着く可能性が高い。



### III.接続障害とタイムアウトの管理



Nmapスキャンの速度を最適化する（あるいは精度を高める）ために、_timeout_と_retry_というパラメーターを使うことができる。



タイムアウト(*timeouts*)の場合、これは**応答なしタイムアウト**(*no response timeout*)であり、これを過ぎるとNmapは応答待ちを停止し、サービスまたはホストに到達不能とみなす。*retry*の場合、これはNmapが次に進む前に実行する**操作の連続試行回数**である。



並列化と同様に、_timeout_と_retry_の管理は、ホストやサービスの発見段階に適用することができる：





- min-rtt-timeout/max-rtt-timeout/initial-rtt-timeout <time>`: Exchangeのラウンドトリップタイムを指定する。繰り返しますが、このパラメータはスキャン中にその場で計算され、適応されます。Nmapはネットワークの反応に応じて、その場でこの時間を計算するので、使う必要はまずないだろう。





- max-retries <number>`: ポートスキャン中のパケットの再送回数を制限する。デフォルトでは、Nmapは特にネットワークレベルで遅延やロスを発見した場合、1つのサービスに対して最大10回の再試行を行うことができるが、ほとんどの場合、実行されるのは1回だけである。





- `--host-timeout <time>`: Nmapが、ポートスキャン、サービス検出、そのホストに関連するその他の操作を含むすべての操作のために、そのホストに費やす最大時間を指定する。応答や**操作の完了**がないままこの時間間隔を超えると、Nmapはそのホストに関する結果を表示することなくそのホストを放棄し、次のリストに移る。これにより、Nmapが指定されたホストに費やす最大時間を制御することができ、不従順なホストで立ち往生することを避け、スキャン時間全体を最適化することができる。




日常業務では、`--max-retries`オプションと`--host-timeout`オプションを使ってスキャンを最適化している：



```
# Optimization of a scan with 0 additional attempts and a timeout of 15 minutes per host
nmap -sV -sC 10.10.10.0/24 --max-retries 0 --host-timeout 15m
```



これらのパラメータは、特定のニーズやネットワーク条件に合わせてスキャンの動作を調整するための、さらなる柔軟性を提供する。しかし、スキャンされるホストの負荷や精度の潜在的な損失という点で、その影響に注意する必要がある。

この章で見てきた様々なオプションは、個別に使うこともできるし、Nmapが提供する既製の設定の一部として使うこともできる。これらの_templates_(設定テンプレート)を使えるようにするオプションは `-T <number>` または `-T <name>` である。使用可能な_templates_のレベルは5つある：



```

```

## 12 - Nmap使用時のデータセキュリティと機密性



### I.プレゼンテーション



この章では、Nmapによって生成、処理、保存されるデータのセキュリティと、何よりも機密性に関して採用すべきグッドプラクティスを紹介する。



情報システム内でのNmapの使用は、すぐに攻撃的行為に分類される。そのため、意図したターゲット、収集したデータ、スキャンに使用したシステムのセキュリティを保証しつつ、法的な枠組みの中で行動するためには、多くの予防措置を講じる必要がある。



### II.適切な認可の取得



ネットワークやシステムをスキャンする前に、適切な承認を得ていることを確認してください。特に、情報システムセキュリティがあなたの正式な任務の一部でない場合、無許可でシステムの脆弱性をスキャンすること（「NSEスクリプト」）は違法であり、法的な結果を招く可能性があります。





- 備忘録として：[刑法：第三章：自動データ処理システムへの攻撃](https://www.legifrance.gouv.fr/codes/article_lc/LEGIARTI000030939438/)




### III.機密データの保護



Nmapが生成する結果は、特に攻撃者に悪用される可能性のある情報システムの弱点に関する情報を含んでいる場合、センシティブ（機密）であると考えられる。また、誰もがアクセスできないシステム（機密、産業、医療、[バックアップ]情報システム(https://www.it-connect.fr/cours-tutoriels/administration-systemes/autres/sauvegarde/)など）にも関係する。



また、使用するNSEスクリプトによっては、[Nmap](https://www.it-connect.fr/cours/nmap-cartographie-reseau-scan-de-vulnerabilites/)のNSEスキャン結果にも識別子が含まれることがあることもわかった。



したがって、これらのスキャン結果にアクセスすることに成功した悪意のある個人は、自分でこれらの行為を行わなくても、情報システムの地図と豊富な技術情報を手にすることができ、発見される危険性がある。



したがって、Nmapを使用する際には、以下のような機密情報を不適切に収集したり保存したりしないように注意することが重要である：





- 出力データの暗号化：Nmapスキャンの結果を保存または送信する必要がある場合は、データの機密性を保護するために必ず暗号化してください。これにより、機密情報の不正な傍受を防ぐことができる。理想的には、スキャンを実行するために使用されたシステムからデータが出るとすぐに暗号化すべきである(強力なパスワードで暗号化されたZIPアーカイブは非常に良い手始めである)。





- アクセス制御の設定：Nmapスキャンの結果が保存される場所には、許可された人だけがアクセスできるようにする。機密情報を不正アクセスから保護するために、適切なアクセス制御を設定します。





- データ取り扱い時の注意：スキャンデータを転送、コピー、処理する場合は、データのセキュリティを厳重に管理してください。つまり、インターネットに接続されたワークステーションの `Download` ディレクトリに放置しない、内部の HTTP ファイル Exchange アプリケーションを経由させない、昼休みにワークステーションをロックせずにメモ帳を開いたままにしない、などです。




### IV.アグレッシブスキャンの管理



このチュートリアルで見てきたように、Nmapはネットワークレベルでは非常に冗長である。また、適切にフォーマットされていないパケットを送信したり、生成するネットワークフレームやパケットのプロトコル構造を厳密に無視したりすることもある。これらの動作はすべて、特定のシステムやサービスに影響を及ぼし、時にはシステムやネットワーク・リソースの誤動作や飽和を引き起こす可能性がある。



インシデントを回避するためには、Nmapの動作をマスターし、このチュートリアルで説明するさまざまなオプションを使用して、Nmapを使用するコンテキストに適合させる方法を知っておく必要がある。産業用[ハードウェア](https://www.it-connect.fr/actualites/actu-materiel/)を含む情報システムでは、ローカルのファイアウォールやネットワークコアで保護されたWindowsシステムで構成されたユーザーネットワークと同じようにNmapを使用するとは限りません。



このチュートリアルの様々なレッスンで、Nmapの動作をマスターし、分析する方法を学んだと思うが、学ぶ最善の方法は、実際にやってみることである。そのため、使用するNmapのオプションに精通していることを確認してほしい。



### V.スキャンシステムの保護



最初の章では、ほとんどの場合、Nmapは`root`またはローカル管理者として実行する必要があることを説明した。これは、Nmapがネットワークライブラリを通じて、時にはかなり低いレベルでネットワーク操作を行うためであり、システムの安定性や他のアプリケーションの機密性の観点から、高くて危険なパーミッションが必要となる。



その結果、Nmapはインストールされているシステムの機密コンポーネントとみなされる可能性がある。古いバージョンには既知のセキュリティ脆弱性が含まれている可能性があるため、Nmapは必ず最新バージョンを使用すること。最新バージョンを使用することで、ツールの使用に伴うリスクを最小限に抑えることができる。



Nmapを`root`としてのセッションではなく、特権ユーザーに特定の特権を与えて、そのユーザーがNmapを使用するために必要なものすべて（`sudo`または_capabilities_）を持つようにして使用する場合は、Nmapが完全な特権昇格の一部として使用される可能性があることに注意すること：



![nmap-image](assets/fr/65.webp)



sudo`によるNmap権限の昇格。



ここでは、`sudo`を通してNmapコマンドを使っているが、これだとシステム上で`root`として対話型シェルを使うことができる。



また、ネットワークスキャンを実行するように設計されていないシステムにNmapをインストールするのは、非常に好ましくない。特にサーバーやワークステーションを考えている。一方では、特権昇格の潜在的なベクトルを追加することになるが、何よりも攻撃者に攻撃ツールへの楽なアクセスを与えることになる。



最後に、スキャンに使用するシステムのセキュリティは、それ自体が侵入や情報漏洩の媒介とならないよう、より広範に確保されなければならない。システム管理者としては、自分のワークステーションではなく、理想的には寿命の限られた専用システムを使う方がよい。



### VI.結論



結論として、Nmapを実運用や本番環境で使用する前に、Nmapを正しくマスターしていることを確認し、その結果を処理・管理する際には用心すること。せっかく自社のセキュリティを向上させるためにNmapを導入したのに、損害を与えたり、データを漏えいさせたり、侵害を助長するようなことがあってはならない。

### II.TCP SYNスキャンまたは「ハーフオープンスキャン



TCPスキャンの最初のタイプは、`TCP SYNスキャン`であり、`ハーフオープンスキャン`としても知られている。最初のポートスキャンの後に行ったネットワークスキャンを覚えているなら、これは[Nmap](https://www.it-connect.fr/cours/nmap-cartographie-reseau-scan-de-vulnerabilites/)がroot権限で実行したときにデフォルトで使用するスキャンのタイプである。



この訳は、このスキャンがどのように動作するかを理解するのに役立つだろう。実際、TCP SYNスキャンは、各ターゲットポートにTCP SYNパケットを送信する。これは、有名な_Three way handshake_ TCPの一部として、クライアント（接続の確立を要求するもの）が送信する最初のパケットである。通常、ターゲットとなるサーバーでポートが開いていて、その背後でサービスが動作している場合、サーバーはクライアントのSYNを検証し、TCP接続を初期化するためにTCP SYN/ACKパケットを送り返します。この応答は、SYNフラグとACKフラグが1に設定されたTCPパケットの形をとり、ポートが開いていてサービスにつながっていることを確認できる。



一方、ポートが閉じている場合、サーバーは接続要求を終了するために、RSTフラグとACKフラグを1に設定した`TCP`パケットを送信してくるので、このポートの後ろではどのサービスもアクティブになっていないことがわかる：



![nmap-image](assets/fr/67.webp)

TCP SYNスキャン`をより具体的に見るために、ポートTCP/80のスキャンを、このポートでアクティブなウェブサーバーがあるホストに対して実行した。Wiresharkでネットワークスキャンを実行すると、次のような流れが確認できる（スキャン元：`10.10.14.84`）：

スキャンされたホストは、ポートがオープンされていないときに、私の `TCP SYN`に対して`TCP RST/ACK`を返す。



前述のように、特権端末からNmapを実行する場合、TCP SYNスキャンはデフォルトのモードであり、`-sS`（`scan SYN`）オプションで強制することができる：



```

```




TCP SYNスキャン`は、速度の理由から最も一般的に使用されるスキャンである。一方、クライアントが_Three Way Handshake_を完了しない(すなわち、サーバ のSYN/ACKの後にACKを送信しない)ことは、あるサーバやネットワーク上の 同じソースから何度も観察されると、疑わしいと思われるかもしれない。実際、TCP SYNに対する応答としてTCP SYN/ACKパケットを受信した後のクライアン トの通常の動作は、「確認」(ACK)を送信し、それからExchangeを開始することである。



とはいえ、肯定的な応答ごとにACKを送信する手間が省けるため、スキャン速度は若干速くなる。SYN Scanの利点は、検出の可能性が高くなる代わりに、スキャンするポートごとに1つのパケットしか送信されないので、その速さである。



さらに、TCP SYNスキャンは、ポートがファイアウォールによってフィルタリング（保護）されているかどうかを検出することができる。実際、ターゲットホストの前にあるファイアウォールは、保護されているはずのポートでTCP SYNパケットを受信したときの動作によって検出できる。単に応答しないだけだ。しかし、これまで見てきたように、どちらの場合（ポートが開いているか閉じているか）でも、ホストからの応答があります。この3つ目の動作は、スキャンされたホストとスキャンを実行しているマシンとの間にファイアウォールが存在することを明らかにする。以下は、スキャンされたポートがファイアウォールによってフィルタされた場合にNmapが返す結果である：

### III.TCPコネクトスキャンまたはフルオープンスキャン

TCP Connect Scan`は、ラムダクライアントに最も近い動作で、より正当な接続要求をシミュレートするため、ポート数を減らしてもスキャンを見破られにくい。しかし、スキャンされたマシンのオープンポート上のすべてのTCPコネクションを完全に初期化するので、より遅い。

### IV.TCP FINスキャンまたは「ステルススキャン



TCP FINスキャン」は、「ステルス・スキャン」とも呼ばれ、クライアントがTCP接続を 終了するときの動作を利用して、開いているポートを検出する。



TCPでは、セッションの終了とは、FINフラグを1に設定したTCPパケットを送信することを意味する。通常のExchangeでは、サーバーはクライアントとのすべての通信を停止する(応答がない)。サーバーがクライアントとアクティブなTCPコネクションを持っていない場合、`RST/ACK`を送信する。従って、あるポートに`TCP FIN`パケットを送ることで、開いているポートと閉じているポートを区別することができる：



![nmap-image](assets/fr/76.webp)



開いているポートと閉じているポートのtCP FINスキャン動作図



ステルス・スキャン中にネットワークをキャプチャしてみると、スキャンされたポートが開いているときにこのように表示された：



![nmap-image](assets/fr/77.webp)



オープンポートのTCP FINスキャン中のネットワークキャプチャ



クライアントがTCPコネクションを終了するために1つか2つのパケットを送信し、サーバーがそれに応答しないことがわかる。サーバーは単に接続の終了を受け入れ、通信を停止する。



クローズド・ポートをスキャンするとこうなる：



![nmap-image](assets/fr/78.webp)



閉じたポートに対するTCP FINスキャン中のネットワークキャプチャ



サーバは`TCP RST/ACK`パケットを送り返すので、開いているポートと閉じているポートでは動作に違いがあり、TCP FINパケットを送信することでサーバの開いているポートをリストアップできることがわかる。Nmapを使用して、`-sF`（`scan FIN`）オプションを使用してTCP FINスキャンを実行する：



```
# Execution of a TCP Fin Scan
nmap -sF 192.168.1.15
```



TCP FINスキャンはWindowsホストでは動作しません。なぜなら、OSはTCP FINパケットがオープンされていないポートに送信された場合、無視する傾向があるからです。そのため、Windowsホスト上でTCP FINスキャンを実行すると、すべてのポートが閉じられているような印象を受ける。



だからこそ、いくつかのスキャン方法を熟知し、それぞれの違いを理解することが重要なのだ。



いずれの場合も、TCP FINは応答を待たないので、ターゲットホストとスキャンソース間のファイアウォールの存在を検出できない。



以下は、NmapのTCP FINスキャン結果の例である：



![nmap-image](assets/fr/79.webp)



Nmap._によるTCP FINスキャンの結果。



実際、あるポートでホストからの応答がないということは、そのポートがフィルタリングされていることを意味するだけでなく、そのポートが開いていてアクティブであることを意味することもある。



このスキャンは「ステルス」と呼ばれ、generateのトラフィックがそれほど多くなく、一般にターゲットとなるシステムでロギングを引き起こさないためである。アラームを発生させることなく、ネットワーク上のポートを目立たないように発見するために使用できる。しかし、前述のように、その効果はターゲット・システムによって異なり、セキュリティ機器の構成によってその裁量が変わることもある。



### V.結論



Nmapが提供するさまざまなTCPスキャンタイプに関する2つの章のうち、最初の章はここまでである！次の章では、XMAS、Null、ACKのTCPスキャンタイプについて見ていく。





## 14 - TCP経由のポートスキャン：XMAS、Null、ACK



### I.プレゼンテーション



このセクションでは、Nmapが提供する様々なTCPスキャン方法について引き続き調べる。ここでは、`XMAS`メソッド、`Null`メソッド、`ACK`メソッドを取り上げる。これらのメソッドでは、TCP固有の機能を使用して、指定したターゲット上で開いているポートやサービスに関する情報を取得する。



### II.TCP XMASスキャン



XMAS ScanのTCPは少し変わっていて、ネットワーク上の通常のユーザーやマシンの動作をまったくシミュレートしない。実際、XMASスキャンは特定のファイアウォールやフィルタリング機構をバイパスするために、`URG`（緊急）、`PSH`（プッシュ）、`FIN`（終了）のフラグを1に設定してTCPパケットを送信する。



XMASという名前は、これらのフラグがオンになっているのが珍しいことに由来する。TCPパケットの中で3つのフラグが同時にセットされると、点灯したクリスマスツリーのように見える：



![nmap-image](assets/fr/80.webp)



XMASスキャンで使用されるtCPフラグ



これらのフラグの役割についてここで詳しく説明することはしないが、これら3つのフラグを有効にしてパケットを送信する場合、ターゲットポートの背後でアクティブなサービスはパケットを返さないことを知っておくことが重要である。一方、ポートが閉じている場合は、TCP RST/ACKパケットを受信する。これで、マシンのポートをリストアップするときに、開いているポートと閉じているポートの動作を区別できるようになる：



![nmap-image](assets/fr/81.webp)



開いているポートと閉じているポートのtCP XMASスキャン動作図



同じロジックに従って、アクティブなウェブ・サーバーがあるホストのポートTCP/80をネットワーク・スキャンすると、オープン・ポートを検出したときに次のような動作を示す（スキャン・ソースは`10.10.14.84`）：



![nmap-image](assets/fr/82.webp)



オープンポートのTCP XMASスキャン中のネットワークキャプチャ



スキャン元がホスト`10.10.10.203`に2つのTCP XMASパケット（フラグ `FIN`、`PSH`、`URG`を1に設定）を送信し、ターゲットからの応答がないことから、ポートが開かれていることがわかる。逆に、閉じたポートに対して `TCP XMAS Scan` を実行すると、以下のような結果が得られる：



![nmap-image](assets/fr/83.webp)



閉じたポートに対するTCP XMASスキャン中のネットワークキャプチャ



TCPパケットに対する応答は`TCP RST/ACK`で、ポートが閉じていることを示す。このテクニックをNmapで使用するには、`-sX` (`scan XMAS`) オプションを使用すると、TCP XMASスキャンを実行できる：



```bash
# Execution of a TCP XMAS Scan
nmap -sX 192.168.1.15
```



TCP SYNやコネクトのような他のタイプのスキャンとは異なり、TCP XMASスキャンはターゲットとスキャンマシンの間に存在する可能性のあるファイアウォールを検出できないことに注意することが重要である。実際、2つのホスト間にアクティブなファイアウォールがあると、ターゲットポートがフィルタリングされている(すなわち、ファイアウォールで保護されている)場合、TCPリターンが行われないことが保証される。したがって、応答がない場合、そのポートがファイアウォールによって保護されているのか、それとも開いていてアクティブなのかを知ることはできない。また、TCP FINスキャンと同様に、特定のアプリケーションやWindowsなどのオペレーティングシステムが、この種のスキャンの結果を歪める可能性があることにも注意すべきである。



注：Windowsの最近のバージョンにおけるXMAS/FIN/NULLスキャンのサポートはまだ限定的であり、この種のターゲットでは結果に一貫性がない可能性がある。(2025年更新)_。



### III.TCPヌルスキャン



TCP XMASスキャンとは対照的に、TCP Nullスキャンは、すべてのフラグを0に設定してTCPスキャンパケットを送信する。フラグなしでTCPパケットを送信することは、TCPプロトコルを記述するRFCで規定されていないため、これもマシン間の通常のExchangeでは決して見られない動作である。そのため、より簡単に検出することができます。



TCP XMASスキャンと同様に、このスキャンは特定のファイアウォールやフィルタリングモジュールに干渉し、パケットを通過させる可能性がある：



![nmap-image](assets/fr/84.webp)



開いているポートと閉じているポートのtCP Null Scan動作図



開いているポートのTCP Nullスキャン中にネットワーク上に表示されるものを以下に示す：



![nmap-image](assets/fr/85.webp)



オープンポートのTCP Nullスキャン中のネットワークキャプチャ



スキャンマシンはサーバーからの応答なしにフラグのないパケット（Wiresharkでは`[<None>]`）を送信する。逆に、ターゲットポートが閉じられている場合は：



![nmap-image](assets/fr/86.webp)



閉じたポートに対するTCP Nullスキャン中のネットワークキャプチャ



NmapでTCP Nullスキャンを実行するには、単に`-sN` (`scan Null`)オプションを使用する：



```bash
# Execution of a TCP Null Scan
nmap -sN 192.168.1.15
```



ポートが開いているときとファイアウォールがアクティブなときの応答は同じであるため（どちらの場合もサーバからのフィードバックはない）、TCP Nullスキャンはファイアウォールの存在を検出することができない。さらに、ファイアウォールは、ポートがフィルタリングされているにもかかわらず、フラグのないTCPパケットに応答しないため、ポートが開いていることを示唆して結果を改ざんすることさえある。これは、`TCP Null`スキャン、`XMAS`スキャン、`FIN`スキャンなど、ポートが開いているかフィルタリングされているか（ファイアウォールで保護されているか）を区別できないスキャンを使用する際に、得られた結果の解釈に一貫性を保つために注意すべき重要な情報である。



### IV.TCP ACKスキャン



TCP ACKスキャンは、ターゲットホスト上またはターゲットとスキャンソース間のファイアウォールの存在を検出するために使用される。



他のスキャンとは異なり、TCP ACKスキャンは、ホスト上でどのポートが開かれてい るかを識別しようとするのではなく、フィルタリングシステムが有効かどうかを識別 し、各ポートに対して`filtered`または`unfiltered`で応答する。TCP SYN`や`TCP Connect`のようないくつかのTCPスキャンは、この2つを同時に行 うことができるが、`TCP FIN`や`TCP XMAS`のような他のスキャンは、フィルタリングの存 在をまったく判断することができない。これが、TCP ACKスキャンが有用な理由である：



![nmap-image](assets/fr/87.webp)



フィルタリングされたポートとフィルタリングされていないポートのtCP ACKスキャン動作図



Nmapの`-sA`オプションを使用して、このタイプのスキャンを実行する。ポートがフィルタリングされている場合、つまり、ファイアウォールでブロックされ、保護されている場合のTCP ACKスキャンの結果である：



![nmap-image](assets/fr/88.webp)



TCP ACKスキャン中のnmapの表示._.



ファイアウォールがあるホストとないホストの結果の例。Nmapはホスト`10.10.10.203`のポートTCP/80とTCP/81で`filtered`を返す。Wiresharkを使ったネットワーク解析では、以下のような動作になる：



![nmap-image](assets/fr/89.webp)



ファイアウォールによってフィルタリングされていないポートに対するTCP ACKスキャン中のネットワークキャプチャ



ファイアウォールが存在する場合、ターゲット・マシンは何も返さない。



Nmapでこのスキャンを起動するには、`-sA` (`scan ACK`) オプションを使用する：



```bash
# Execution of a TCP ACK Scan
nmap -sA 192.168.1.15
```

すでに紹介したものに加えて、TCP経由のスキャンの3つの異なる方法について見てきた。これらの異なる方法は、特に侵入テストやレッドチーム活動のような、慎重さが要求される非常に特殊な状況や文脈で使用される。



## 15 - Nmap CheatSheet - チュートリアルコマンドのまとめ



### I.プレゼンテーション



ここでは、Nmapの多くのコマンドとユースケースを簡単にまとめ、日常ですぐに見つけて再利用できるようにする。



### II.Nmap：チートシート IT-Connect



ここに、紹介するコマンドのチートシートがある。このページでは、Nmapの最も一般的な使い方を簡単に見つけることができる。





- ポートスキャン




```bash
# Display installed Nmap version
nmap --version

# Scan for open specific ports on a single IP address
nmap --open -p 80 192.168.1.18

# Scan TCP ports on a selection of ports
nmap 192.168.1.19 -p 80
nmap 192.168.1.19 -p 22,80,1000-2000,3389

# Scan UDP services on an IP address
nmap -sU 192.168.1.19

# Scan UDP services on specific ports
nmap -sU 192.168.1.19 -p 161,23

# Scan the 100 most commonly used TCP ports
nmap 192.168.1.19 --top-ports 100

# Scan all ports on an IP address
nmap 192.168.1.19 -p-

# Scan multiple subnets with specific ports
nmap 192.168.0.0/24 192.168.1.0/24 -p 22

# Scan a subnet while excluding a specific IP address, scan all ports
nmap 192.168.0.0/24 --exclude 192.168.0.4 -p-
```





- アクティブなホストの発見




```bash
# Scan on CIDR or IP ranges
nmap 192.168.0.0/24
nmap 192.168.0.0/24 192.168.1.0/24
nmap 192.168.1.2 192.168.1.3 192.168.1.10-20
nmap 192.168.0.0/24 192.168.1.3 192.168.1.10-20

# Host discovery scan (Ping Scan) on a network
nmap -sn 192.168.1.0/24
```



note: `-sP` オプションは数年前から廃止されており、`sn` に置き換える必要があります。(2025年更新)_。



```bash
# Host discovery scan without port scan
nmap 192.168.1.0/24 -sn

# Host discovery scan on a local network using various probe techniques
nmap -sn -PP -PS22,3389,445,139 -PA80 192.168.1.0/24

# Scan hosts listed in a text file
nmap -iL /tmp/mesCibles.txt

# Network scan with a specific IP excluded
nmap 192.168.1.0/24 --exclude 192.168.1.140

# Subnet scan excluding another subnet
nmap 10.0.0.0/16 --exclude 10.0.100.0/24
```





- バージョン検出




```bash
# Enable version detection
nmap 192.168.1.0/24 -sV

# Version detection + advanced trace
nmap 192.168.1.0/24 -sV --version-trace

# Version detection with maximum probe rarity of 2
nmap 192.168.1.0/24 -sV --version-intensity 2

# Version detection with all probes
nmap 192.168.1.0/24 -sV --version-intensity 9

# Enable OS detection
nmap -O 10.10.10.0/24
```





- NSEスクリプト：脆弱性を探す




```bash
# Run default NSE scripts
nmap -sC 10.10.10.152

# Scan all TCP ports with default NSE scripts
nmap -sC -p- 10.10.10.152

# Display help for script by name
nmap --script-help=ftp-*

# Display help for a category
nmap --script-help=discovery

# Use NSE scripts by category
nmap --script default 10.10.10.152
nmap --script discovery 10.10.10.152

# Inclusion/exclusion filter for NSE script categories
nmap --script "discovery and safe" 10.10.10.152
nmap --script "not intrusive and not dos" 10.10.10.152
nmap --script "vuln and not intrusive and not dos" 10.10.10.152
nmap --script "(http and vuln) and not intrusive and not dos" 10.10.10.152

# Run a specific NSE script
nmap --script ftp-anon -p 21 10.10.10.152
nmap --script ftp-anon 10.10.10.152

# Pass arguments to an NSE script
nmap --script ssh-brute --script-args userdb=/tmp/userlist,passdb=/tmp/passlist 10.10.10.245 192.168.1.19
```





- Nmapの出力管理




```bash
# Save scan to text file
nmap 10.10.10.0/24 -sV -oN nmap_scan_10.10.10.0_24.txt

# Save scan to condensed text file
nmap 10.10.10.0/24 -sV -oG nmap_scan_10.10.10.0_24.gnmap

# Save scan to XML file
nmap 10.10.10.0/24 -sV -oX nmap_scan_10.10.10.0_24.xml
```





- パフォーマンスの最適化




```bash
# Version detection scan with max rate of 300 packets per second
nmap -sV 10.10.10.0/24 --max-rate 300

# Version detection scan with default scripts, no retry, max host timeout 15min
nmap -sV -sC 10.10.10.0/24 --max-retries 0 --host-timeout 15m

# Version detection scan with the 2000 most common ports, aggressive scan speed (T4), debug output
nmap 10.10.10.0/24 -sV --top-ports 2000 -T4 -d
```





- TCPスキャンの種類




```bash
# TCP SYN scan (fast, less stealthy)
nmap -sS 192.168.1.15

# TCP Connect scan (classic)
nmap -sT 192.168.1.15

# TCP FIN scan (stealth)
nmap -sF 192.168.1.15

# TCP XMAS scan
nmap -sX 192.168.1.15

# TCP Null scan
nmap -sN 192.168.1.15

# TCP ACK scan (detect firewall)
nmap -sA 192.168.1.15
```



これらのコマンドが役に立つことを願っている。スキャンの対象をあなたのコンテキストに合わせ、実行されるテストを完全にマスターするために公式ドキュメントを参照することをお忘れなく。
