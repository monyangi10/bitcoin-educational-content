ยินดีต้อนรับสู่บทแนะนำเบื้องต้นเกี่ยวกับ Nmap ที่ออกแบบมาสำหรับผู้ที่ต้องการเชี่ยวชาญเครื่องมือสแกนเครือข่ายที่ทรงพลังนี้ เป้าหมายคือการให้ความรู้พื้นฐานที่คุณต้องการเพื่อใช้ Nmap อย่างมีประสิทธิภาพในแต่ละวัน



Nmap เป็นเครื่องมืออเนกประสงค์ที่ใช้กันอย่างแพร่หลายโดยผู้เชี่ยวชาญด้าน IT เครือข่าย และความปลอดภัยทางไซเบอร์สำหรับการวินิจฉัย การค้นหาเครือข่าย และการตรวจสอบความปลอดภัย บทแนะนำนี้มุ่งเป้าไปที่ผู้ที่ยังใหม่กับสาขาเหล่านี้และต้องการเรียนรู้พื้นฐานของ Nmap ขอแนะนำให้มีความรู้พื้นฐานเกี่ยวกับการจัดการระบบและเครือข่าย



คุณจะได้เรียนรู้พื้นฐานของ Nmap วิธีการสแกนพอร์ต การระบุโฮสต์ที่ใช้งานอยู่บนเครือข่าย การตรวจจับเวอร์ชันของบริการและระบบปฏิบัติการ การสแกนหาช่องโหว่ และอื่นๆ อีกมากมาย แต่ละส่วนประกอบด้วยคำอธิบายที่ละเอียดและตัวอย่างการใช้งานจริงเพื่อช่วยให้คุณเชี่ยวชาญการใช้ Nmap ในบริบทที่หลากหลาย



เมื่อสิ้นสุดบทแนะนำนี้ คุณจะมีความเข้าใจที่มั่นคงเกี่ยวกับ Nmap และสามารถใช้งานได้อย่างมีประสิทธิภาพเพื่อปรับปรุงความปลอดภัยและการจัดการเครือข่ายของคุณ ขอให้สนุกกับการอ่าน



## 1 - บทนำสู่ Nmap: Nmap คืออะไร?



### I. การนำเสนอ



ในส่วนแรกนี้ เราจะมาดูเครื่องมือสแกนเครือข่าย Nmap เราจะดูที่องค์ประกอบสำคัญที่คุณจำเป็นต้องรู้เกี่ยวกับเครื่องมือนี้และวิธีการทำงานโดยทั่วไป ซึ่งจะช่วยให้เราเข้าใจส่วนที่เหลือของบทแนะนำได้ดียิ่งขึ้น



### II. แนะนำเครื่องมือ Nmap



Nmap, สำหรับ _Network Mapper_, เป็นเครื่องมือฟรีและโอเพ่นซอร์สที่ใช้สำหรับ **การค้นพบเครือข่าย, การทำแผนที่และการตรวจสอบความปลอดภัย** นอกจากนี้ยังสามารถใช้สำหรับงานอื่นๆ เช่น **การตรวจนับเครือข่าย, การวินิจฉัยหรือการควบคุมดูแล**



มันสามารถระบุได้ว่าโฮสต์บนเครือข่ายเป้าหมายมีการใช้งานและสามารถเข้าถึงได้หรือไม่ บริการเครือข่ายใดที่ถูกเปิดเผย เวอร์ชันและเทคโนโลยีใดที่ถูกใช้งาน และข้อมูลการวิเคราะห์อื่น ๆ ที่เป็นประโยชน์ Nmap สามารถใช้สแกนบริการเดียวบนเครื่องเฉพาะ หรือครอบคลุมพื้นที่ขนาดใหญ่ของเครือข่าย จนถึงอินเทอร์เน็ตทั้งหมด



จุดแข็งของ Nmap มีมากมาย:





- ทรงพลังและยืดหยุ่น**: Nmap สามารถสแกนเครือข่ายขนาดใหญ่และใช้เทคนิคการตรวจจับขั้นสูง รองรับ UDP, TCP, ICMP, IPv4 และ IPv6 และสามารถตรวจจับเวอร์ชัน สแกนหาช่องโหว่ หรือโต้ตอบเฉพาะโปรโตคอลได้ สถาปัตยกรรมของมันเป็นแบบโมดูลาร์ โดยเฉพาะอย่างยิ่งต้องขอบคุณสคริปต์ NSE (Nmap Scripting Engine) ซึ่งเราจะมาดูในภายหลังในบทเรียนนี้
- ความง่ายในการใช้งาน**: เอกสารอย่างเป็นทางการมีอยู่อย่างมากมายและมีคุณภาพสูงสุด นอกจากนี้ยังมีแหล่งข้อมูลจากชุมชนจำนวนมากที่พร้อมช่วยเหลือคุณในการเริ่มต้น
- ความนิยมและความยืนยาว**: Nmap ได้เป็นที่อ้างอิงในสาขาของมันตั้งแต่ปี 1998 เวอร์ชันปัจจุบัน ณ เวลาที่อัปเดตนี้คือ 7.95 แม้ว่าจะมีเครื่องมืออื่น ๆ สำหรับงานเฉพาะทาง Nmap ยังคงเป็นสิ่งที่ต้องมีสำหรับการทำแผนที่เครือข่ายและการวิเคราะห์



**Nmap ที่โรงภาพยนตร์**



Nmap เป็นหนึ่งในเครื่องมือรักษาความปลอดภัยไม่กี่ตัวที่ได้รับชื่อเสียงในหมู่ประชาชนทั่วไป มันปรากฏในภาพยนตร์ _Matrix Reloaded_ ในฉากที่เป็นสัญลักษณ์ที่ Trinity ใช้มันเพื่อแฮ็กเข้าสู่ระบบ:

เขายังปรากฏในผลงานภาพยนตร์เรื่องอื่น ๆ ด้วย



**ความคิดเห็น**



ในฐานะผู้ดูแลระบบและต่อมาเป็นผู้ตรวจสอบความปลอดภัยทางไซเบอร์และผู้ทดสอบการเจาะระบบ **ฉันใช้ Nmap เกือบทุกวัน** และฉัน **แนะนำเป็นประจำ** ให้กับผู้ดูแลระบบที่ต้องการเสริมสร้างความเข้าใจในเครือข่ายและปรับปรุงความสามารถในการวินิจฉัยของพวกเขา



### III. การดำเนินงานระดับสูง



Nmap มีให้ใช้งานสำหรับ Linux, Windows และ macOS มันถูกเขียนขึ้นด้วยภาษา C, C++ และ Lua (สำหรับสคริปต์ NSE) เป็นหลัก มันถูกใช้งานหลักบนบรรทัดคำสั่ง แม้ว่าจะมีอินเทอร์เฟซแบบกราฟิกเช่น Zenmap ให้ใช้งานด้วยก็ตาม อย่างไรก็ตาม เราขอแนะนำอย่างยิ่งให้คุณเริ่มต้นด้วยบรรทัดคำสั่งเพื่อให้เข้าใจการทำงานได้ดียิ่งขึ้น



ตัวอย่างง่าย ๆ:



```
nmap 192.168.10.13 10.10.10.0/24 -sV -sC --top-ports 250
```



คำสั่งนี้จะอธิบายอย่างละเอียดในภายหลัง ในบทแนะนำนี้ เราจะใช้ Nmap บน Linux แต่การใช้งานจะคล้ายกันในระบบอื่น ๆ บน Windows, Nmap อาศัยไลบรารี **Npcap** (แทนที่ WinPcap ที่ล้าสมัยแล้ว) เพื่อจับและฉีดแพ็กเก็ตเครือข่าย



Nmap ถูกใช้งานเหมือนกับไบนารีทั่วไป เช่น `ls` หรือ `ip` ฟีเจอร์ขั้นสูงบางอย่างอาจต้องการสิทธิ์ที่สูงขึ้น เนื่องจากเครื่องมือนี้บางครั้งจัดการแพ็กเก็ตในวิธีที่ไม่ธรรมดาเพื่อกระตุ้นปฏิกิริยาเฉพาะบนระบบเป้าหมาย (โดยเฉพาะสำหรับการตรวจจับบริการหรือช่องโหว่)



### IV. ผลกระทบจากการใช้ Nmap



ก่อนใช้ Nmap จำเป็นต้องตระหนักถึงผลกระทบที่อาจเกิดขึ้นกับเครือข่ายและระบบ:





- มันสามารถส่ง **หลายพันหรือแม้กระทั่งหลายล้านแพ็กเก็ต** ในช่วงเวลาสั้น ๆ ซึ่งสามารถทำให้โครงสร้างพื้นฐานเครือข่ายบางอย่างอิ่มตัวได้
- มันสามารถ generate **แพ็กเก็ตที่ผิดรูปหรือไม่เป็นมาตรฐาน** ซึ่งอาจรบกวนอุปกรณ์บางอย่าง (โดยเฉพาะระบบอุตสาหกรรม)
- มันสามารถสร้าง **พฤติกรรมคล้ายการโจมตี** ซึ่งสามารถกระตุ้นการแจ้งเตือนในระบบรักษาความปลอดภัย (ไฟร์วอลล์, IDS/IPS, ฯลฯ)



โดยทั่วไปแล้ว **Nmap เป็นเครื่องมือที่พูดมาก** เนื่องจากมันสร้างการจราจรจำนวนมากเพื่อดึงข้อมูลให้ได้มากที่สุด ดังนั้นจึงควรทำความเข้าใจอย่างถ่องแท้ว่ามันทำงานอย่างไรก่อนที่จะใช้ในสภาพแวดล้อมที่ละเอียดอ่อนหรือการผลิต



### บทสรุป



ส่วนนี้จะแนะนำ Nmap และคุณสมบัติหลักของมัน เราได้เห็นแล้วว่ามันเป็นเครื่องมือการทำแผนที่เครือข่ายที่จำเป็น ทรงพลัง และยืดหยุ่น นอกจากนี้เรายังได้พูดคุยเกี่ยวกับวิธีการทำงานและข้อควรระวังที่คุณต้องปฏิบัติเมื่อใช้งาน เพื่อเตรียมพร้อมสำหรับส่วนต่อไปของบทแนะนำ



## 2 - ทำไมต้องใช้ Nmap?



### I. การนำเสนอ



ในส่วนนี้ เราจะมาดูการใช้งานหลักของเครื่องมือสแกนเครือข่าย Nmap เราจะเห็นว่าเป็นเครื่องมือที่ใช้กันอย่างแพร่หลายในหลายบริบทและอาชีพ และการมีเครื่องมือนี้ในกล่องเครื่องมือของคุณและรู้วิธีการใช้งานอย่างชำนาญนั้นเป็นทักษะที่มีประโยชน์อย่างแน่นอน



### II. การใช้ Nmap สำหรับการวินิจฉัยและการควบคุมดูแล



Nmap สามารถใช้สำหรับการวินิจฉัยเครือข่ายและในวงกว้างสำหรับการตรวจสอบ ในลักษณะเดียวกับที่การ ping สามารถใช้เพื่อตรวจสอบว่าสองโฮสต์กำลังสื่อสารกันหรือไม่ Nmap สามารถใช้เพื่อตรวจสอบอย่างรวดเร็วว่าโฮสต์ทำงานอยู่หรือไม่ หรือบริการเฉพาะทำงานอยู่หรือไม่ ขอบคุณ [Nmap](https://www.it-connect.fr/cours/nmap-cartographie-reseau-scan-de-vulnerabilites/ "Nmap") เราสามารถได้รับข้อมูลที่แม่นยำเกี่ยวกับเวลาตอบสนองของโฮสต์ เส้นทางที่แพ็กเก็ตเดินทาง การตอบสนองที่ทำโดยบริการเฉพาะ เป็นต้น



คำสั่งและผลลัพธ์ต่อไปนี้สามารถใช้ได้ เช่น เพื่อค้นหาอย่างรวดเร็วว่าเว็บเซิร์ฟเวอร์บนโฮสต์ **192.168.1.18** กำลังทำงานและตอบสนองอย่างถูกต้องหรือไม่:



```
nmap --open -p 80 192.168.1.18
```



![nmap-image](assets/fr/02.webp)



*ใช้ Nmap เพื่อดึงสถานะบริการเว็บจากเซิร์ฟเวอร์ระยะไกล*



ดังนั้น การใช้ Nmap จึงไปไกลกว่าการทดสอบ "ping test" ที่มีชื่อเสียงในระหว่างการดีบักหรือขั้นตอนการวินิจฉัย เราจะเห็นในภายหลังว่ามีหลายวิธีที่ Nmap ใช้ในการระบุว่าบริการใดกำลังฟังอยู่บนพอร์ตที่กำหนด รวมถึงเวอร์ชันของมันด้วย



### III. การใช้ Nmap สำหรับการทำแผนที่เครือข่าย



ในฐานะที่เป็น _Network Mapper_ การทำแผนที่เครือข่ายเป็นวัตถุประสงค์หลักของเครื่องมือนี้ สามารถใช้ภายในเครือข่ายท้องถิ่น หรือข้ามเครือข่ายหลายเครือข่าย ซับเน็ต และ VLANs เพื่อแสดงรายการโฮสต์และบริการที่สามารถเข้าถึงได้ทั้งหมด Nmap ทำให้ภารกิจนี้รวดเร็วและมีประสิทธิภาพมากกว่าวิธีการแบบแมนนวลใดๆ



ตัวอย่างเช่น คำสั่งต่อไปนี้สามารถใช้เพื่อระบุโฮสต์ที่ใช้งานอยู่บนเครือข่าย **192.168.1.0/24** ได้อย่างรวดเร็ว:



```
nmap -sn 192.168.1.0/24
```



*หมายเหตุ: ตัวเลือก `-sP` ล้าสมัยแล้วและถูกแทนที่ด้วย `-sn`.*



![nmap-image](assets/fr/03.webp)



*การใช้ Nmap เพื่อค้นหาโฮสต์ที่เข้าถึงได้บนเครือข่าย*



เราจะเห็นในภายหลังว่ามีหลายวิธีที่ใช้โดย Nmap เพื่อกำหนดว่าโฮสต์สามารถถือว่า "active" ได้หรือไม่ แม้ว่ามันจะไม่ได้เปิดเผยบริการใด ๆ ล่วงหน้าก็ตาม



### IV. การใช้ Nmap ในการประเมินนโยบายการกรอง



Nmap มีข้อได้เปรียบในการเป็นข้อเท็จจริง: ผลลัพธ์ของมันทำให้สามารถสร้างข้อค้นพบที่เป็นรูปธรรมได้ ซึ่งแตกต่างจากเอกสารสถาปัตยกรรมหรือนโยบายการกรองใดๆ มันเป็นเครื่องมือสำคัญสำหรับการประเมินประสิทธิภาพของการแบ่งส่วนระบบสารสนเทศ เนื่องจากช่วยให้คุณ **ตรวจสอบว่านโยบายการกรองถูกนำมาใช้จริงหรือไม่**



ในเครือข่ายองค์กร แนวทางปฏิบัติที่ดีที่สุดกำหนดให้ระบบถูกแบ่งส่วนตามบทบาท ความสำคัญ หรือสถานที่ กฎการกรอง (มักจะดำเนินการผ่านไฟร์วอลล์) ต้องจำกัดการสื่อสารระหว่างโซน แต่การกำหนดค่าเหล่านี้มักจะซับซ้อนและมีแนวโน้มที่จะเกิดข้อผิดพลาด



ดังนั้น เพื่อยืนยันว่านโยบายได้รับการปฏิบัติตาม ไม่มีอะไรดีไปกว่าการทดสอบที่เป็นรูปธรรม ตัวอย่างเช่น คุณสามารถตรวจสอบว่าบริการการจัดการที่ละเอียดอ่อน (SSH, WinRM, MSSQL, MySQL, ฯลฯ) ไม่สามารถเข้าถึงได้จากโซนผู้ใช้



การใช้ **Nmap เพื่อตรวจสอบนโยบายการกรอง** ควรทำอย่างเป็นระบบก่อนที่จะนำนโยบายดังกล่าวไปใช้จริง น่าเสียดายที่การตรวจสอบนี้มักถูกละเลย



จากประสบการณ์ของฉัน ข้อผิดพลาดในการกำหนดค่าหลายอย่างมักจะไม่ถูกสังเกตเห็นเมื่อไม่มีการทดสอบการตรวจสอบความถูกต้อง ข้อผิดพลาดง่ายๆ ในช่วง IP หรือการมองข้ามกฎสามารถทำให้โซนที่ควรจะถูกแยกออกนั้นมีความเสี่ยงได้



### V. การใช้ Nmap สำหรับการตรวจสอบความปลอดภัยและการทดสอบการเจาะระบบ



Nmap มี**คุณสมบัติที่มีประโยชน์มากมายสำหรับการประเมินความปลอดภัย**, การทดสอบเจาะระบบ (pentests) และน่าเสียดายที่ยังมีประโยชน์สำหรับผู้โจมตีด้วยเช่นกัน



ฟังก์ชันการค้นหาเครือข่ายมีความสำคัญอย่างยิ่งสำหรับผู้โจมตีที่ต้องการทำความเข้าใจโครงสร้างเครือข่ายหลังจากการบุกรุกครั้งแรก แต่ Nmap มีมากกว่านั้น: มันรวมเอ็นจินการเขียนสคริปต์ **ซึ่งหลายอย่างทุ่มเทให้กับการตรวจจับช่องโหว่**



ตัวอย่างเช่น คำสั่งนี้สามารถใช้เพื่อตรวจสอบว่า FTP service อนุญาตการเชื่อมต่อแบบไม่ระบุตัวตนหรือไม่ โดยไม่ต้องเชื่อมต่อด้วยตนเอง:



```
nmap --script ftp-anon -p 21 192.168.1.18
```



![nmap-image](assets/fr/04.webp)



*ใช้สคริปต์ NSE เพื่อตรวจสอบการยืนยันตัวตน FTP แบบไม่ระบุตัวตนผ่าน Nmap.*



การตรวจจับช่องโหว่ Nmap เป็นหนึ่งในขั้นตอนแรกของการตรวจสอบหรือการทดสอบการเจาะระบบ มันช่วยให้คุณสามารถระบุจุดอ่อนบางอย่างได้อย่างรวดเร็วและเพิ่มประสิทธิภาพในการวิเคราะห์ของคุณ



แต่ระวัง: **เครื่องมือสแกนหาช่องโหว่มีข้อจำกัดของมัน** Nmap ครอบคลุมเพียงเศษเสี้ยวของภัยคุกคาม และไม่ได้รับประกันว่าระบบจะปลอดภัยหากไม่พบปัญหา ดังนั้นจึงจำเป็นต้อง **เข้าใจวิธีการทำงานของสคริปต์ที่ใช้** และไม่ควรพึ่งพาคำตัดสินของมันเพียงอย่างเดียว



### VI. บทสรุป



เราได้เห็นแล้วว่าการเชี่ยวชาญ Nmap สามารถครอบคลุมการใช้งานที่หลากหลาย ตั้งแต่การวินิจฉัยและการตรวจสอบไปจนถึงการทำแผนที่ การประเมินนโยบายความปลอดภัย และการสแกนหาช่องโหว่ ในส่วนถัดไป เราจะลงลึกในรายละเอียดและติดตั้ง Nmap




## 3 - การติดตั้งและกำหนดค่า Nmap



### I. การนำเสนอ



ในส่วนนี้ เราจะเรียนรู้วิธีการติดตั้งเครื่องมือสแกนเครือข่าย Nmap บนระบบปฏิบัติการ Linux และ Windows เมื่อสิ้นสุดส่วนนี้ เราจะมีทุกอย่างที่เราต้องการเพื่อเริ่มใช้ Nmap สำหรับโมดูลในอนาคต สุดท้าย เราจะดูว่ามันอาจต้องการสิทธิพิเศษอะไรบ้างเมื่อใช้งานและทำไม



### II. การติดตั้ง Nmap บน Linux



Nmap ได้รับการออกแบบมาเพื่อใช้งานบนระบบปฏิบัติการ GNU/Linux ตั้งแต่แรก ด้วยเหตุนี้ และด้วยความยาวนานและความนิยมของมัน คุณจะพบมันในที่เก็บข้อมูลอย่างเป็นทางการทั้งหมดของการแจกจ่าย Unix หลัก ๆ ในบทช่วยสอนนี้ ฉันจะใช้ระบบปฏิบัติการที่ใช้ Debian [Kali Linux](https://www.it-connect.fr/cours/debuter-avec-kali-linux/ "Kali Linux") แต่คุณสามารถใช้มันในลักษณะเดียวกันจาก Debian แบบคลาสสิก, CentOS, Red Hat หรืออะไรก็ตาม!



ภายใต้ Debian เพื่อตรวจสอบว่า Nmap มีอยู่ในที่เก็บข้อมูลปัจจุบันของคุณหรือไม่ คุณสามารถใช้คำสั่งต่อไปนี้:



```
# Debian and derivatives
$ sudo apt search ^nmap$
Sorting... Done
Full Text Search... Done
nmap/kali-rolling,now 7.94+git20230807.3be01efb1+dfsg-2+kali1 amd64
The Network Mapper

# Red Hat and derivatives
$ dnf search '^nmap$'
```



คำตอบที่นี่บ่งบอกอย่างชัดเจนว่าแพ็กเกจ "nmap" มีอยู่ในที่เก็บ (ที่นี่คือของ Kali [Linux](https://www.it-connect.fr/cours-tutoriels/administration-systemes/linux/ "Linux")). จากนี้ไป คุณสามารถติดตั้ง Nmap ผ่านคำสั่งติดตั้งตามปกติ ไม่มีอะไรที่น่ากังวลในขณะนี้ 🙂:



```
# Debian and derivatives
sudo apt install nmap

# Red Hat and derivatives
sudo dnf install nmap
```



เพื่อตรวจสอบว่า Nmap ติดตั้งอย่างถูกต้องหรือไม่ เราจะแสดงเวอร์ชันของมัน:



```
nmap --version
```



นี่คือผลลัพธ์ที่คาดหวัง:



![nmap-image](assets/fr/05.webp)



ผลลัพธ์ของการแสดงเวอร์ชันปัจจุบันของ Nmap._



โปรดสังเกตการมีอยู่ในหน้าจอนี้ของไลบรารี "libpcap" (_Packet Capture Library_) และเวอร์ชันของมัน ไลบรารี "libpcap" ยังถูกใช้โดย Wireshark และถูกใช้โดย Nmap เพื่อสร้างและจัดการแพ็กเก็ตและฟังการจราจรของเครือข่าย



### III การติดตั้ง Nmap บน Windows



ในการติดตั้งบนระบบปฏิบัติการ Windows ให้เริ่มต้นโดยการดาวน์โหลดไบนารีจากเว็บไซต์ทางการ (และไม่มีที่อื่น!):





- หน้าเพจดาวน์โหลด Nmap บนเว็บไซต์ทางการ: [https://nmap.org/download.html#windows](https://nmap.org/download.html#windows)




จากนั้นคุณจะต้องดาวน์โหลดไฟล์ไบนารีที่ชื่อว่า `nmap-<VERSION>-setup.exe`:



![nmap-image](assets/fr/06.webp)



หน้าดาวน์โหลดไฟล์ติดตั้ง nmap สำหรับ Windows



เมื่อคุณมีไบนารีนี้ในระบบของคุณแล้ว เพียงแค่เรียกใช้งานเพื่อทำการติดตั้ง Nmap นี่เป็นการติดตั้งที่ง่ายดาย และคุณสามารถปล่อยให้ตัวเลือกทั้งหมดเป็นค่าเริ่มต้นได้



ปฏิกิริยาตอบสนองของฉันคือการยกเลิกการเลือกช่อง "zenmap (GUI Frontend)" นี่คือ Interface แบบกราฟิกสำหรับ Nmap ที่ฉันไม่ได้ใช้และจะไม่ครอบคลุมในบทแนะนำนี้ แต่รู้สึกอิสระที่จะลองใช้เมื่อคุณเชี่ยวชาญเครื่องมือบรรทัดคำสั่ง Nmap แล้ว!



![nmap-image](assets/fr/07.webp)



ตัวเลือกการยกเลิกการเลือก Zenmap เมื่อทำการติดตั้ง Nmap บน Windows



เมื่อสิ้นสุดการติดตั้ง Nmap จะมีการเสนอการติดตั้งครั้งที่สอง: นั่นคือไลบรารี "Npcap":



![nmap-image](assets/fr/08.webp)



การติดตั้งไลบรารี "Npcap" เมื่อทำการติดตั้ง Nmap บน Windows



นี่คือไลบรารีที่ Nmap ใช้เพื่อจัดการการสื่อสารเครือข่าย เช่น การสร้าง การส่ง และการรับแพ็กเก็ตเครือข่าย คุณอาจเคยเจอไลบรารีนี้แล้วหากคุณใช้ Wireshark บน Windows เนื่องจากมันก็ใช้และต้องการการติดตั้งเช่นกัน



เช่นเดียวกับ Linux คุณสามารถตรวจสอบว่า Nmap ติดตั้งอยู่หรือไม่โดยการเปิด Command Prompt หรือ [Powershell] terminal (https://www.it-connect.fr/cours-tutoriels/administration-systemes/scripting/powershell/ "Powershell") และพิมพ์คำสั่งต่อไปนี้:



```
nmap --version
```



นี่คือผลลัพธ์ที่คาดหวัง:



![nmap-image](assets/fr/09.webp)



ผลลัพธ์ของการแสดงเวอร์ชันปัจจุบันของ Nmap._



Nmap ได้ติดตั้งบน Windows แล้ว คุณสามารถใช้งานได้เหมือนกับบน Linux โดยทำตามบทแนะนำนี้



### IV. สิทธิพิเศษในท้องถิ่นที่จำเป็นในการใช้ Nmap



แต่โดยวิธีการ เมื่อใช้ Nmap **จำเป็นต้องมีสิทธิพิเศษในระบบหรือไม่?** คำตอบคือ: **ขึ้นอยู่กับ**



ในรูปแบบพื้นฐานที่สุด กล่าวคือ โดยไม่ต้องใช้ตัวเลือกต่างๆ มากนัก Nmap ไม่จำเป็นต้องใช้สิทธิพิเศษ อย่างไรก็ตาม คุณจะพบในไม่ช้าว่าการใช้ Nmap ในบริบทที่มีสิทธิพิเศษ ("root" ใน Linux, "administrator" หรือเทียบเท่าใน Windows) จะดีกว่าเพื่อให้สามารถใช้งานได้อย่างเต็มประสิทธิภาพ โดยมีความเสี่ยงที่จะได้รับข้อความแสดงข้อผิดพลาดเช่นนี้:



![nmap-image](assets/fr/10.webp)



ข้อความแสดงข้อผิดพลาดภายใต้ Linux เมื่อ Nmap ตัวเลือกต้องการสิทธิ์ของ root._



ไม่ว่าจะเป็นบน Linux หรือ Windows มีหลายกรณีที่ Nmap จะขอให้คุณเข้าถึงสิทธิพิเศษ เหตุผลหลักมีดังนี้ (รายการไม่ครบถ้วน):





- การสร้างแพ็กเก็ตเครือข่าย "ดิบ"**: Nmap มีความสามารถในการสแกนหลากหลายวิธี รวมถึงการจัดการและการสร้างแพ็กเก็ตขั้นสูง ตัวอย่างเช่น เมื่อเราต้องการทำการสแกน TCP SYN ซึ่งไม่เคารพการจับมือสามทาง (*Three-way handshake*) แบบคลาสสิกของการแลกเปลี่ยน TCP ในการทำเช่นนี้ Nmap จำเป็นต้องใช้ฟังก์ชันอื่นนอกเหนือจากที่มีอยู่ในระบบปฏิบัติการ ซึ่งรู้เพียงวิธีการปฏิบัติตามแนวทางที่ดีในการสื่อสารเครือข่าย (มันเรียกใช้ไลบรารี "Npcap" และ "libcap" ที่กล่าวถึงข้างต้น) เนื่องจาก Nmap ไม่ทำสิ่งต่างๆ ในวิธี "มาตรฐาน" ทำให้สามารถสรุปข้อมูลบางอย่างเกี่ยวกับระบบปฏิบัติการ บริการ และช่องโหว่บางประการได้





- ฟังการรับส่งข้อมูลเครือข่าย**: บางตัวเลือกของ Nmap จำเป็นต้องฟังเครือข่ายเพื่อดึงข้อมูลบางอย่าง การกระทำนี้ถือว่าอ่อนไหวบนระบบปฏิบัติการ เนื่องจากยังอนุญาตให้คุณฟังการสื่อสารของแอปพลิเคชันอื่น ๆ บนระบบได้ เช่นเดียวกับ Wireshark, Nmap ต้องการสิทธิพิเศษเฉพาะในการทำเช่นนี้ ซึ่งง่ายต่อการได้รับโดยการอยู่ในเซสชันที่มีสิทธิพิเศษโดยตรง





- การฟังบนพอร์ตที่มีสิทธิพิเศษ**: บนระบบปฏิบัติการ พอร์ตตั้งแต่ 0 ถึง 1024 (ทั้ง TCP และ UDP) ถือว่าเป็นพอร์ตที่มีสิทธิพิเศษ กล่าวคือ พอร์ตเหล่านี้ถูกสงวนไว้สำหรับการใช้งานที่เฉพาะเจาะจงและได้รับการป้องกัน แม้ว่านี่จะเป็นเหตุผลที่ค่อนข้างล้าสมัยในปัจจุบัน แต่ก็ยังจำเป็นต้องมีสิทธิพิเศษบางอย่างในการฟังบนพอร์ตเหล่านี้ ซึ่ง Nmap อาจต้องทำขึ้นอยู่กับวิธีการใช้งาน





- การส่งแพ็กเก็ต UDP:** ในทำนองเดียวกัน การฟังแอปพลิเคชันเครือข่ายบนพอร์ต UDP (โปรโตคอลที่ไม่มีสถานะ) ต้องการสิทธิพิเศษบนระบบปฏิบัติการ ดังนั้นจะต้องมีเซสชันที่มีสิทธิพิเศษหากคุณต้องการทำการสแกน UDP ซึ่ง Nmap จะต้องฟังการตอบสนองเพื่อวิเคราะห์การตอบกลับต่อการสแกนของมัน




เพื่อให้แม่นยำยิ่งขึ้น เป็นไปได้อย่างน้อยภายใต้ระบบ Linux ที่จะรัน Nmap พร้อมฟังก์ชันและตัวเลือกทั้งหมดโดยไม่ต้องเป็น root จริงๆ สิ่งนี้ทำได้โดยการให้ _capabilities_ ที่ถูกต้องแก่ไบนารี อย่างไรก็ตาม วิธีนี้ต้องการการจัดการขั้นสูงและอาจไม่สะดวกเท่ากับการรัน Nmap โดยตรงด้วยสิทธิพิเศษ นอกจากนี้ วิธีนี้ยังไม่ค่อยพบและอาจก่อให้เกิดปัญหาด้านความปลอดภัยหากกำหนดค่าผิดพลาด



อย่างไรก็ตาม นี่เป็นการเบี่ยงเบนเล็กน้อยจากบทเรียน Nmap ของเรา ดังนั้นเราจะละเว้นมันไว้ก่อน



สำหรับบทแนะนำที่เหลือนี้ ให้สมมติว่า Nmap ถูกเรียกใช้เสมอในฐานะ "root" (จากเชลล์ในฐานะ "root" หรือผ่านคำสั่ง "sudo") หรือในเทอร์มินัลผู้ดูแลระบบภายใต้ Windows แม้ว่าจะไม่ได้ระบุไว้ก็ตาม มิฉะนั้น คุณอาจพบความแตกต่างที่ละเอียดอ่อนแต่สังเกตได้จากบทแนะนำ



### V. บทสรุป



นั่นแหละ! Nmap ได้ติดตั้งบนระบบปฏิบัติการของเราเรียบร้อยแล้วและพร้อมใช้งาน โดยไม่ต้องการการกำหนดค่าเพิ่มเติม ส่วนนี้สรุปการแนะนำและการนำเสนอของ Nmap หวังว่ามันจะทำให้คุณน้ำลายสอ และคุณพร้อมที่จะเริ่มฝึกฝนแล้ว



ในเชิงที่จริงจังมากขึ้น ตอนนี้เรามีความเข้าใจที่ดีขึ้นเกี่ยวกับเครื่องมือการทำแผนที่ Nmap และการใช้งานที่พบบ่อยที่สุด รวมถึงข้อจำกัดของมัน ไปต่อกันเถอะ!




## 4 - การสแกนพอร์ต TCP และ UDP ด้วย Nmap



### I. การนำเสนอ



ในส่วนนี้ เราจะเรียนรู้วิธีการสแกนพอร์ตครั้งแรกโดยใช้เครื่องมือสแกนเครือข่าย Nmap เราจะดูวิธีการใช้มันเพื่อรวบรวมรายชื่อบริการเครือข่ายที่เปิดเผยบนโฮสต์ ไม่ว่าจะใช้โปรโตคอล TCP หรือ UDP



ตั้งแต่นี้เป็นต้นไป โปรดจำไว้ว่าต้องสแกนเฉพาะโฮสต์ในสภาพแวดล้อมที่มีการควบคุมซึ่งคุณได้รับอนุญาตอย่างชัดเจนเท่านั้น





- เพื่อเป็นการเตือนความจำ: [ประมวลกฎหมายอาญา: บทที่ III: การโจมตีระบบประมวลผลข้อมูลอัตโนมัติ](https://www.legifrance.gouv.fr/codes/article_lc/LEGIARTI000030939438/)




**หากคุณไม่มีอยู่ในมือ**, ฉันขอแนะนำโซลูชันฟรีต่อไปนี้ ซึ่งเป็นสิ่งที่ใช่เลย!





- [Hack The Box](https://app.hackthebox.com/ "Hack The Box"): แพลตฟอร์มฝึกอบรมการแฮ็ก Hack The Box มอบระบบที่มีช่องโหว่ให้คุณโจมตีตามที่คุณต้องการอย่างต่อเนื่อง มีระบบหลายร้อยระบบให้ใช้งาน แต่มีเครื่องใหม่ 20 เครื่องที่ให้บริการฟรีตลอดทั้งปี โดยเข้าถึงผ่าน OpenVPN VPN





- [Vulnhub](https://www.vulnhub.com/ "Vulnhub"): แพลตฟอร์มนี้มีระบบที่มีช่องโหว่โดยเจตนาจำนวนมากให้ดาวน์โหลด ซึ่งสามารถใช้ผ่าน VirtualBox (ซึ่งเป็นโซลูชันฟรีเช่นกัน) หรือวิธีการอื่น ๆ เมื่อดาวน์โหลดแล้ว ไม่จำเป็นต้องใช้ VPN - ทุกอย่างเป็นแบบโลคอล




นอกจากนี้ คุณยังสามารถ **สร้างเครื่องเสมือน** บนระบบปฏิบัติการที่คุณชื่นชอบและติดตั้งบริการต่างๆ บนเครื่องนั้นเพื่อใช้เป็นเป้าหมายทดสอบ ข้อดีของวิธีนี้คือคุณจะสามารถเห็นสิ่งที่เกิดขึ้นบนฝั่งเซิร์ฟเวอร์ระหว่างการสแกน โดยเฉพาะอย่างยิ่งเมื่อใช้ Wireshark และมีส่วนร่วมในการจัดการไฟร์วอลล์ท้องถิ่นเมื่อเราทำการทดสอบขั้นสูงมากขึ้น



มาลงมือทำกันเถอะ!



### II. การสแกนพอร์ต TCP ของโฮสต์ผ่าน Nmap



#### A. การสแกนพอร์ต TCP ครั้งแรกด้วย Nmap



ตอนนี้เรากำลังจะทำการสแกนครั้งแรกผ่าน Nmap วัตถุประสงค์ของเราที่นี่ง่ายมาก: เราต้องการค้นหาว่ามีบริการใดบ้างที่ถูกเปิดเผยโดยเว็บเซิร์ฟเวอร์ที่เราเพิ่งปรับใช้ เพื่อดูว่ามีสิ่งใดที่ไม่คาดคิดหรือไม่ เช่น บริการการจัดการที่ไม่ควรเข้าถึงได้ หรือการเปิดเผยพอร์ตของแอปพลิเคชันที่เราคิดว่าได้ยกเลิกการใช้งานไปแล้ว



ในตัวอย่างของฉัน โฮสต์ที่จะสแกนมีที่อยู่ IP "192.168.1.18":



```
nmap 192.168.1.18
```



นี่คือผลลัพธ์ที่เป็นไปได้ เราเห็นการกลับมาของ Nmap แบบคลาสสิกที่มีข้อมูลมากมาย:



![nmap-image](assets/fr/11.webp)



ผลลัพธ์ของการสแกน TCP แบบง่ายที่ดำเนินการด้วย Nmap._



จากการตรวจสอบผลลัพธ์อย่างรวดเร็ว เราเข้าใจว่า ports TCP/22 และ TCP/80 สามารถเข้าถึงได้บนโฮสต์นี้



ตามค่าเริ่มต้น และหากคุณไม่ได้บอกให้มันทำอย่างอื่น Nmap จะสแกนเฉพาะพอร์ต TCP เท่านั้น



#### ข. การทำความเข้าใจผลลัพธ์ของการสแกน Nmap แบบง่าย



อย่างไรก็ตาม มาทำความเข้าใจผลลัพธ์นี้ให้ลึกซึ้งขึ้นอีกขั้น: แต่ละบรรทัดมีความสำคัญ อย่างแรกเพื่อรู้ว่าเพิ่งทำอะไรไป และอย่างที่สองเพื่อแปลผลการสแกนได้อย่างถูกต้อง



บรรทัดแรกเป็นการเตือนเกี่ยวกับเวอร์ชันการสแกนและวันที่ (มีประโยชน์สำหรับการบันทึกและเก็บถาวรในที่สุด!):



```
Starting Nmap 7.94SVN ( https://nmap.org ) at 2024-04-29 21:59 CEST
```



บรรทัดที่สองแสดงผลการสแกนเริ่มต้นสำหรับโฮสต์ "debian.home (192.168.1.19)" ข้อมูลนี้จะมีประโยชน์เมื่อเราเริ่มสแกนหลายโฮสต์ในเวลาเดียวกัน:



```
Nmap scan report for debian.home (192.168.1.19)
```



บรรทัดที่สามบอกเราว่าโฮสต์ที่กล่าวถึงคือ "Up" หรือก็คือใช้งานอยู่:



```
Host is up (0.00022s latency).
```



ในที่สุด Nmap แจ้งให้เราทราบว่า 998 พอร์ต TCP ที่ระบุว่าปิดจะไม่แสดงใน:



```
Not shown: 998 closed tcp ports (conn-refused)
```



สิ่งนี้ช่วยให้เราประหยัดบรรทัดผลลัพธ์ได้เกือบ 1,000 บรรทัดที่มีลักษณะดังนี้:



```
1/tcp closed
2/tcp closed
3/tcp closed
…
```



ขอบคุณเขาที่ช่วยให้เราไม่ต้องเจอสิ่งนี้!



ทำไมถึงมีพอร์ต "ปิด" 998 พอร์ต? การเพิ่มพอร์ตที่เปิด 2 พอร์ตจะรวมเป็น 1000 ซึ่งเป็นจำนวนพอร์ตที่ Nmap จะสแกนในค่าคอนฟิกูเรชันเริ่มต้น ไม่ใช่ 65535 พอร์ต TCP ที่มีอยู่! เราจะเห็นในภายหลังว่าสิ่งนี้สามารถปรับแต่งได้อย่างสมบูรณ์และง่ายดาย แต่ถ้าโฮสต์เป้าหมายมีบริการที่ฟังอยู่บนพอร์ตที่ค่อนข้างแปลกใหม่ การสแกน "เริ่มต้น" นี้จะไม่สามารถค้นพบได้



จากข้อมูลนี้ เราพบสิ่งที่น่าสนใจที่สุด: ตารางที่จัดเรียงตามสามคอลัมน์ "PORT - STATE - SERVICE":





- คอลัมน์ "PORT" แรกแสดงถึงพอร์ต/โปรโตคอลที่เป็นเป้าหมาย และสิ่งสำคัญที่นี่คือการดูว่ามันเป็นพอร์ต TCP (`<port>/tcp`) หรือ UDP (`<port>/udp`)





- คอลัมน์ "STATE" ระบุสถานะของบริการเครือข่ายที่ค้นพบในพอร์ตนี้และกำหนดโดย Nmap บนพื้นฐานของการตอบสนองที่ได้รับ ซึ่งสามารถเป็น "open", "closed", "filtered" หรือ "unknown" เราจะเห็นในภายหลังว่า Nmap แยกแยะระหว่างสถานะเหล่านี้อย่างไร





- คอลัมน์ "SERVICE" แสดงถึงบริการที่เปิดเผยบนพอร์ตที่กล่าวถึง โปรดทราบว่าเราไม่ได้ใช้ตัวเลือกการค้นหาบริการแบบแอคทีฟที่นี่ Nmap อาศัยการอ้างอิงภายในระหว่างพอร์ต/โปรโตคอลและบริการที่ถูกกำหนดไว้: ไฟล์ "/etc/services"




หากคุณดูที่ไฟล์ "/etc/services" บนระบบ Linux คุณจะพบลิงก์ "port/protocol - service" ที่คล้ายกับที่แสดงโดย Nmap:



![nmap-image](assets/fr/12.webp)



ดึงเนื้อหาของไฟล์ "/etc/services" ภายใต้ Linux._



สิ่งสำคัญคือต้องเข้าใจว่า ณ ขณะนี้ Nmap ยังไม่ได้ดำเนินการค้นหาบริการที่ใช้งานอยู่ ตัวอย่างเช่น มันจะไม่สามารถระบุบริการ SSH ที่อยู่เบื้องหลังพอร์ต TCP/80 ได้หากเป็นกรณีนี้ ดังนั้นความสำคัญของการรู้วิธีใช้ตัวเลือกที่ถูกต้อง - กำลังจะมาเร็ว ๆ นี้!



การรู้วิธีตีความผลลัพธ์ของ Nmap เป็นส่วนสำคัญของการเชี่ยวชาญเครื่องมือนี้ ข่าวดีก็คือผลลัพธ์นี้จะเหมือนกันเป็นส่วนใหญ่ ไม่ว่าคุณจะใช้ตัวเลือกใดก็ตาม



#### C. ภายใต้ประทุน: การวิเคราะห์เครือข่ายผ่าน Wireshark



หากคุณพิจารณาอย่างใกล้ชิดถึงสิ่งที่เกิดขึ้นบนเครือข่าย Interface ของโฮสต์ที่สแกนเซิร์ฟเวอร์ หรือบนเครือข่ายของเซิร์ฟเวอร์เอง การกระทำของ Nmap จะชัดเจนยิ่งขึ้น นั่นคือสิ่งที่เราจะทำที่นี่



สิ่งที่ฉันสามารถแสดงให้คุณเห็นที่นี่เป็นเพียงส่วนหนึ่งของสิ่งที่มองเห็นได้ใน Wireshark หากคุณต้องการไปต่อ ลองทำการจับภาพเครือข่ายด้วยตัวเองระหว่างการสแกน แล้วเรียกดูสิ่งที่ถูกจับภาพไว้



ในการทดสอบนี้ โฮสต์สแกนของฉัน (192.168.1.18) และโฮสต์เป้าหมายของฉัน (192.168.1.19) อยู่บนเครือข่ายท้องถิ่นเดียวกัน



Nmap เริ่มต้นด้วยการตรวจสอบว่าโฮสต์เป้าหมายทำงานอยู่บนเครือข่ายท้องถิ่นหรือไม่โดยการส่งคำขอ ARP หากได้รับการตอบกลับ จะทราบว่าโฮสต์ทำงานอยู่และเริ่มการสแกนเครือข่าย:



![nmap-image](assets/fr/13.webp)



_คำขอ ARP ที่ออกโดย Nmap เพื่อตรวจสอบว่าโฮสต์เป้าหมายมีอยู่ในเครือข่ายท้องถิ่นหรือไม่._



หากโฮสต์ที่จะสแกนอยู่บนเครือข่ายระยะไกล Nmap จะเริ่มต้นด้วยการส่งคำขอ ping และพยายามเข้าถึงพอร์ตที่เปิดเผยบ่อยที่สุดบางพอร์ต (TCP/80, TCP/443):

ตอนนี้ที่ Nmap ได้ระบุเป้าหมายและรู้ว่ามันทำงานอยู่ มันเริ่มการสแกนพอร์ต TCP:



![nmap-image](assets/fr/16.webp)



การส่งแพ็กเก็ต tCP SYN และการรับ RST/ACK ระหว่างการสแกน Nmap



ในการทำเช่นนี้ มันจะ **ส่งแพ็กเก็ต TCP SYN และรอการตอบกลับ** ในแต่ละพอร์ต TCP ในช่วงค่าเริ่มต้นของมัน ในภาพหน้าจอด้านบน มันได้รับแพ็กเก็ต TCP RST/ACK จากเซิร์ฟเวอร์ที่ถูกสแกน ซึ่งหมายถึง "ไปต่อ ไม่มีอะไรให้ดูที่นี่" - กล่าวอีกนัยหนึ่งคือ พอร์ตเหล่านี้ถูกปิด ดังที่เราเห็นในผลลัพธ์ นี่จะเป็นกรณีสำหรับพอร์ตส่วนใหญ่ที่ถูกสแกน ยกเว้นสองกรณี:



![nmap-image](assets/fr/17.webp)



ตอบสนองต่อแพ็กเก็ต TCP SYN ที่ส่งไปยังพอร์ต 22 ซึ่งทำงานอยู่บนเป้าหมายการสแกน



ในภาพหน้าจอด้านบน เราเห็นแพ็กเก็ต TCP SYN/ACK ที่ส่งโดยโฮสต์เป้าหมาย พอร์ตนั้นทำงานและเปิดเผยบริการ Nmap รับทราบการตอบรับ จากนั้นยุติการเชื่อมต่อ (TCP RST/ACK) **นี่คือวิธีที่มันรู้ว่าพอร์ต TCP/22 ทำงานอยู่**



เราได้เห็นแล้วว่า Nmap เคารพ "Three Way Handshake" เมื่อทำการสแกนเครือข่าย TCP ด้วยเหตุผลด้านประสิทธิภาพ เราสามารถสั่งให้มันไม่ตอบสนองต่อการตอบกลับของเซิร์ฟเวอร์ได้ ซึ่งจะช่วยประหยัดแพ็กเก็ตได้หลายพันเมื่อทำการสแกนเครือข่ายขนาดใหญ่ แต่เราจะมาดูตัวเลือกและการปรับแต่งเหล่านี้ในภายหลังในบทแนะนำ



ตอนนี้เรามีความเข้าใจที่ดีขึ้นเกี่ยวกับวิธีการทำการสแกน TCP และสิ่งที่เกิดขึ้นจริงเมื่อมันถูกดำเนินการ เราได้เห็นแล้วด้วยว่า โดยค่าเริ่มต้น Nmap จะทำการสแกนพอร์ต TCP ในจำนวนพอร์ตที่จำกัด



### III. การสแกนพอร์ต UDP ด้วย Nmap



#### A. การสแกนพอร์ต UDP ครั้งแรกด้วย Nmap



ตอนนี้มาดูกันว่าจะสแกนพอร์ต UDP ของโฮสต์ได้อย่างไร ตามที่เราได้เห็นแล้ว โดยค่าเริ่มต้น Nmap จะสแกนพอร์ต TCP เสมอ ซึ่งอาจหมายถึงการพลาดข้อมูลจำนวนมากหากเราไม่ทราบถึงมัน



เพื่อเป็นการเตือนความจำ สำหรับวัตถุประสงค์ของการทดสอบนี้ โฮสต์สแกนของฉัน (192.168.1.18) และโฮสต์เป้าหมายของฉัน (192.168.1.19) อยู่บนเครือข่ายท้องถิ่นเดียวกัน



```
nmap -sU 192.168.1.19
```



ที่นี่ ผลตอบแทนที่ได้รับมีรูปแบบเดียวกันกับการสแกน TCP แต่บริการที่ใช้งานจะแสดงในรูปแบบ `<port>/UDP` ตามที่ร้องขอ!



![nmap-image](assets/fr/18.webp)



ผลลัพธ์ของการสแกน UDP แบบง่ายที่ดำเนินการด้วย Nmap._



ตัวเลือก "-sU" ใช้เพื่อบอก Nmap ว่าคุณต้องการทำงานบน UDP แทนที่จะเป็น TCP ซึ่งเป็นค่าเริ่มต้น



อย่างไรก็ตาม คุณอาจสังเกตเห็นว่า Nmap ต้องการสิทธิ์ "root" สำหรับการสแกน UDP ตามที่กล่าวไว้ก่อนหน้านี้ในบทแนะนำ



หมายเหตุ: ตั้งแต่เวอร์ชันล่าสุดของ Nmap ขอแนะนำให้รันการสแกน UDP ด้วยสิทธิ์ของผู้ดูแลระบบเสมอเพื่อให้ได้ผลลัพธ์ที่เชื่อถือได้ เนื่องจากคุณสมบัติบางอย่างต้องการการเข้าถึงซ็อกเก็ตเครือข่ายแบบ raw



การสแกน UDP อาจใช้เวลานานมาก (1100 วินาทีในการสแกน 1000 พอร์ตในตัวอย่างของฉัน) เนื่องจากไม่มี "Three Way Handshake" ใน UDP ซึ่งหมายความว่า Nmap จะรอการตอบกลับสำหรับแต่ละแพ็กเก็ต UDP ที่ส่งไป และจะกำหนดว่าพอร์ต "ปิด" ก็ต่อเมื่อไม่มีการตอบกลับหลังจากเวลาที่กำหนด (timeout) การตอบสนองจากโฮสต์ที่ถูกสแกนไม่เป็นระบบและมักถูกจำกัดในแง่ของจำนวนการตอบสนองต่อวินาที เพื่อหลีกเลี่ยงการโจมตีแบบขยายบางประเภท ซึ่งแตกต่างจาก TCP ที่มีการตอบสนองทันทีจากโฮสต์ที่ถูกสแกน ไม่ว่าพอร์ตจะเปิดหรือปิด เราจะเห็นวิธีการปรับปรุงสิ่งนี้ในภายหลัง



ความยากลำบากประการที่สองของ UDP คือ **บริการไม่ได้ตอบสนองต่อแพ็กเก็ตที่เข้ามาเสมอไป** เพราะบางครั้งมันไม่จำเป็นและเป็นหลักการของ UDP เมื่อเป็นกรณีนี้ และไม่ได้รับ ICMP "port unreachable" บริการจะถูกทำเครื่องหมายว่า "open|filtered" โดย Nmap ดังที่แสดงในภาพหน้าจอด้านบน



#### B. ภายใต้ประทุน: การวิเคราะห์เครือข่ายผ่าน Wireshark



เช่นเดียวกับการสแกน TCP ของเรา มาดูรายละเอียดเพิ่มเติมเกี่ยวกับสิ่งที่เกิดขึ้นในระดับเครือข่ายระหว่างการสแกน UDP โดยใช้การวิเคราะห์ Wireshark พฤติกรรมของ Nmap ในการตรวจสอบว่าโฮสต์ทำงานอยู่หรือไม่ก็เหมือนกัน



ความแตกต่างที่แท้จริงเมื่อสแกน UDP คือ Nmap จะไม่รอ "Three Way Handshake" เนื่องจากกลไกนี้ไม่มีอยู่ใน UDP (โปรโตคอลที่ไม่มีสถานะ):



![nmap-image](assets/fr/19.webp)



การส่งแพ็กเก็ต uDP และการรับ ICMP (พอร์ตไม่สามารถเข้าถึงได้) ระหว่างการสแกน Nmap



เราสามารถเห็นได้จากภาพหน้าจอด้านบนว่า Nmap จะส่งแพ็กเก็ต UDP จำนวนมาก และได้รับแพ็กเก็ต ICMP "Destination unreachable (Port unreachable)" เป็นการตอบกลับสำหรับแพ็กเก็ตส่วนใหญ่ นี่เป็นเรื่องปกติ เนื่องจากเป็นการตอบกลับที่เหมาะสมตามที่กำหนดโดย [RFC 1122](https://www.freesoft.org/CIE/RFC/1122/41.htm "RFC 1122") เมื่อพอร์ต UDP ไม่สามารถเข้าถึงได้:



![nmap-image](assets/fr/20.webp)



ดึงข้อมูลจาก RFC 1122._



มาดูการจับภาพ Wireshark นี้อย่างละเอียดกัน ซึ่งแสดงให้เห็น **สามสถานการณ์ที่เป็นไปได้** ใน UDP:



![nmap-image](assets/fr/21.webp)



การจับภาพเครือข่ายระหว่างการสแกน UDP บนพอร์ตต่าง ๆ ด้วย Nmap._



กรณีทั้งสามมีดังนี้:





- การแลกเปลี่ยนครั้งแรกประกอบด้วยแพ็กเก็ตหมายเลข 3, 4 และ 8, 9 Nmap ส่งแพ็กเก็ต UDP บนพอร์ต SNMP แบบคลาสสิกและดังนั้น **สร้างแพ็กเก็ตที่สอดคล้องกับโปรโตคอลล่วงหน้า** จากนั้นจะได้รับการตอบกลับจากเซิร์ฟเวอร์ (แพ็กเก็ตหมายเลข 8 และ 9) ผลลัพธ์: Nmap ได้รับการตอบกลับ บริการ "เปิด"





- การแลกเปลี่ยนครั้งที่สองประกอบด้วยแพ็กเก็ต 6 และ 7 Nmap ส่งแพ็กเก็ต UDP "ว่าง" (ไม่มีโครงสร้างโปรโตคอล) ไปยังพอร์ต UDP/165 และได้รับแพ็กเก็ต ICMP ตอบกลับ: "Destination unreachable (Port unreachable)" ผลลัพธ์: Nmap ได้รับการตอบกลับ (เชิงลบ) โฮสต์ยังคงทำงานอยู่ แต่บริการที่พยายามเข้าถึงไม่ทำงานบนพอร์ตนี้ ซึ่งจะถูกทำเครื่องหมายว่า "ปิด"





- การแลกเปลี่ยนครั้งสุดท้ายประกอบด้วยแพ็กเก็ตหมายเลข 12: Nmap ส่งแพ็กเก็ต UDP "ว่าง" ไปยังพอร์ต UDP/1235 ไม่มีการตอบกลับ แม้แต่การปฏิเสธอย่างชัดเจนจากโฮสต์ที่ถูกสแกน ผลลัพธ์: Nmap ทำเครื่องหมายพอร์ตว่า "open|filtered" เนื่องจากไม่สามารถบอกได้ว่านี่เป็นเพราะการมีอยู่ของไฟร์วอลล์ที่ถูกกำหนดค่าไม่ให้ตอบสนอง หรือเป็นบริการ UDP ที่ใช้งานอยู่ซึ่งไม่ตอบสนองอยู่ดี (ไม่บังคับใน UDP)




นี่คือผลลัพธ์ที่แสดงโดย Nmap ตามกรณีทั้งสามนี้:



![nmap-image](assets/fr/22.webp)



ผลลัพธ์ที่เป็นไปได้ของการสแกน UDP ที่ดำเนินการผ่าน Nmap._



ตอนนี้เรามีความเข้าใจที่ดีขึ้นเกี่ยวกับวิธีการทำการสแกน UDP และสิ่งที่เกิดขึ้นจริงเมื่อทำการสแกน จนถึงตอนนี้เราได้ใช้ Nmap ในวิธีที่ง่ายมาก โดยไม่ได้ตัดสินใจจริงๆ ว่าจะสแกนพอร์ตใด แต่สิ่งนั้นกำลังจะเปลี่ยนไป!



### IV. การปรับแต่งการสแกนพอร์ตด้วย Nmap



#### A. การเตือนเกี่ยวกับพฤติกรรมเริ่มต้นของ Nmap



ตามที่เราได้เห็น Nmap เองจะเลือกหมายเลขและพอร์ตที่จะสแกนหากคุณไม่ได้ระบุตัวเลือกใดๆ นี่คือการตั้งค่า "ค่าเริ่มต้น" ที่ใช้โดย Nmap เมื่อคุณไม่ได้บอกให้มันทำอะไรอย่างชัดเจน ตัวเลือกค่าเริ่มต้นเหล่านี้ถูกออกแบบมาเพื่อให้ทราบถึงพอร์ตหลักที่เปิดเผย โดยเลือกจากความถี่ของการเปิดเผย (พอร์ตที่พบมากหรือบ่อยที่สุด) แทนที่จะเป็นตามลำดับตัวเลข (พอร์ต 1, 2, 3, ฯลฯ) และยังเพื่อหลีกเลี่ยงการเริ่มสแกนพอร์ตที่เป็นไปได้ทั้งหมด 65535 พอร์ตหากคุณไม่ได้ระบุตัวเลือกที่เหมาะสม ซึ่งจะยาวและซับซ้อนเกินไปสำหรับกรณีการใช้งาน "ค่าเริ่มต้น"



**พอร์ตเหล่านี้ถูกเลือกอย่างไร?**



พอร์ต 1000 ที่ถูกสแกนในโหมดเริ่มต้นถูกเลือกตามความถี่ในการเกิดขึ้น สถิติเหล่านี้ถูกดูแลโดย Nmap และอัปเดตในลักษณะเดียวกับไบนารีเองและสคริปต์ของมัน (โมดูล) คุณสามารถดูสถิติเหล่านี้ได้ด้วยตัวเองในไฟล์ "/usr/shares/nmap/nmap-services":



![nmap-image](assets/fr/23.webp)



ดึงข้อมูลจากไฟล์ "/usr/shares/nmap/nmap-services"._



ที่นี่ ในคอลัมน์ที่สาม เราเห็นสิ่งที่ดูเหมือนความน่าจะเป็น (ระหว่าง 0 และ 1) หรือการกระจายเปอร์เซ็นต์ นี่คือความถี่ของการเกิดขึ้นของแต่ละคู่พอร์ต/โปรโตคอล เราสามารถเห็นได้ว่าพอร์ตที่รู้จักกันดีที่สุด (FTP, SSH, TELNET และ SMTP ในการสกัดนี้) มีค่าที่สูงกว่าพอร์ตอื่นๆ มาก



#### B. ระบุพอร์ตเป้าหมายอย่างแม่นยำสำหรับการสแกน Nmap



อย่างไรก็ตาม ในโลกแห่งความเป็นจริง เราอาจจำเป็นต้องสแกนเฉพาะพอร์ตใดพอร์ตหนึ่ง หรือหลายพอร์ต หรือช่วงพอร์ตเฉพาะ Nmap ทำให้การทำเช่นนั้นเป็นเรื่องง่าย ในรูปแบบที่เป็นเอกภาพสำหรับการสแกนทั้ง UDP และ TCP



**สแกนพอร์ตเฉพาะผ่าน Nmap**



หากเราต้องการสแกนพอร์ตเดียว และไม่ใช่ 1000 พอร์ต เราสามารถระบุพอร์ตนี้ผ่านตัวเลือก "-p" หรือ "--port" ของ Nmap:



```
# Scan a single TCP port with Nmap
nmap 192.168.1.19 -p 80

# Scan a single UDP port with Nmap
nmap -sU 192.168.1.19 -p 161
```



ผลลัพธ์คือ การสแกนจะเร็วขึ้นอย่างเป็นธรรมชาติและ Nmap จะส่งเฉพาะแพ็กเก็ตที่จำเป็นในการตรวจสอบว่าโฮสต์ยังทำงานอยู่หรือไม่ จากนั้นตรวจสอบว่าพอร์ตที่ระบุสามารถเข้าถึงได้หรือไม่ ซึ่งจะช่วยประหยัดเวลา หากคุณต้องการทดสอบอย่างรวดเร็วเพื่อดูว่าบริการเว็บบนไซต์โชว์เคสของคุณยังคงทำงานอยู่หรือไม่



**สแกนพอร์ตหลายพอร์ตผ่าน Nmap**



ในลักษณะเดียวกัน เราสามารถระบุพอร์ตหลายพอร์ตไปยัง Nmap โดยใช้ตัวเลือกเดียวกันและเชื่อมต่อพอร์ตที่ระบุด้วยเครื่องหมายจุลภาค:



```

# Scan several TCP ports with Nmap

nmap 192.168.1.19 -p 80,10999,22,23,1345

# Scan several UDP ports with Nmap

nmap -sU 192.168.1.19 -p 161,23,69

```



ไม่ว่าจะเป็นลำดับใด Nmap จะตรวจสอบพอร์ตทั้งหมดเหล่านี้ และเฉพาะพอร์ตที่อยู่บนโฮสต์เป้าหมายเท่านั้น คุณจะสังเกตเห็นในผลลัพธ์ของ Nmap ว่ามัน **บอกเราทุกพอร์ตและสถานะของพอร์ตเหล่านั้นอย่างชัดเจน** แม้ว่าพอร์ตเหล่านั้นจะ "ปิด" ก็ตาม ซึ่งแตกต่างจากพฤติกรรมเริ่มต้นที่ผลลัพธ์ทั้งหมดนี้จะใช้พื้นที่มากเกินไป:



![nmap-image](assets/fr/24.webp)



*ผลลัพธ์ของการสแกน TCP Nmap บนพอร์ตที่ระบุ*



**สแกนพอร์ตในช่วง**



หากจำนวนพอร์ตที่คุณต้องการสแกนมีมากเกินไป คุณสามารถระบุพอร์ตเหล่านั้นเป็นช่วงได้ เช่น:



```

# Scan TCP ports from 1000 to 2000 with Nmap

nmap 192.168.1.19 -p 1000-2000

# Scan UDP ports from 1000 to 2000 with Nmap

nmap -sU 192.168.1.19 -p 100-150

```



แน่นอน คุณสามารถผสมและจับคู่ตามที่คุณเห็นว่าเหมาะสม ตัวอย่างเช่น:



```

# Scan TCP ports 22,80, 3389 and from 1000 to 2000 with Nmap

nmap 192.168.1.19 -p 22,80,1000-2000,3389

```



**การสแกนพอร์ต TCP และ UDP**



คุณยังสามารถทำการสแกน UDP และ TCP พร้อมกันได้บนพอร์ตที่เลือก:



```

# Scan the list of 1000 default ports, in TCP and UDP

sudo nmap 192.168.1.19 -sT -sU

# Scan only UDP/161 and TCP/22

sudo nmap 192.168.1.19 -sT -sU -p U:161,T:22

```



คุณจะสังเกตเห็นในตัวอย่างสุดท้ายนี้ว่ามีการใช้ "U:" เพื่อระบุพอร์ต UDP และ "T:" เพื่อระบุพอร์ต TCP นี่คือตัวอย่างผลลัพธ์ที่เป็นไปได้ของการสแกนประเภทนี้:



![nmap-image](assets/fr/25.webp)



*ผลลัพธ์ของการสแกนพอร์ต TCP และ UDP ด้วย Nmap.*



นั่นเป็นวิธีที่น่าสนใจในการปรับแต่งการสแกนของคุณ!



**สแกนพอร์ตทั้งหมด**



ในที่สุดก็สามารถระบุช่วงที่ใหญ่กว่าหรือเล็กกว่ามากให้กับ Nmap ได้ เราได้เห็นแล้วว่ารายการเริ่มต้นที่เลือกโดย Nmap มีพอร์ต 1000 พอร์ต เราสามารถขอพอร์ตที่ใช้บ่อยที่สุด 100 พอร์ต หรือ 200 พอร์ต โดยใช้ตัวเลือก "--top-ports":



```

# Scan the top 100 most common ports with Nmap

nmap 192.168.1.19 --top-ports 100

# Scan the top 200 most common ports with Nmap

nmap 192.168.1.19 --top-ports 200

```



ในที่สุด เราสามารถขอให้มันสแกนพอร์ตทั้งหมดที่เป็นไปได้ (ทั้งหมด 65535) โดยใช้สัญลักษณ์ "-p-":



```

# Scan all TCP ports from 1 to 65535 with Nmap

nmap 192.168.1.19 -p-

```



สิ่งหลังจะใช้เวลานานกว่า โดยเฉพาะกับ UDP แต่คุณจะมั่นใจได้ว่าจะไม่พลาดพอร์ตที่เปิดอยู่



*หมายเหตุ: ตัวเลือก "-p-" เป็นวิธีที่แนะนำสำหรับการสแกนพอร์ต TCP ทั้งหมด สำหรับการสแกน UDP ขอแนะนำให้จำกัดจำนวนพอร์ตเพื่อเหตุผลด้านประสิทธิภาพ เนื่องจากการสแกนพอร์ต UDP ทั้งหมดอาจใช้เวลานานมาก*



ในภายหลังในบทแนะนำ เราจะได้เห็นวิธีการเพิ่มประสิทธิภาพความเร็วของการสแกน Nmap ให้เหมาะสมกับความต้องการของเรา ซึ่งจะมีประโยชน์อย่างยิ่งสำหรับการสแกนบนพอร์ต TCP และ UDP ทั้งหมด



### บทสรุป



ในส่วนนี้ เราได้มีโอกาสฝึกปฏิบัติจริง ดังนั้นเราจึงรู้แล้วว่า **วิธีการใช้ Nmap ในการสแกนพอร์ต TCP และ UDP ของโฮสต์ในแบบพื้นฐาน** เราได้ดูรายละเอียดเกี่ยวกับสิ่งที่เกิดขึ้นในระดับเครือข่ายและ **วิธีที่ Nmap กำหนดว่าพอร์ต TCP หรือ UDP นั้นทำงานอยู่หรือไม่** สุดท้าย เรารู้วิธีการเลือกพอร์ตที่ต้องการสแกนอย่างละเอียดและ **ตัวเลือกเริ่มต้นของ Nmap ทำอะไรบ้าง** ในส่วนต่อไปนี้ เราจะนำความรู้นี้มาใช้ในการสแกนเครือข่ายทั้งหมด รวมถึงการทำแผนที่ทั่วโลกและการค้นพบเครือข่าย




## 5 - การทำแผนที่เครือข่ายและการค้นพบด้วย Nmap



### I. การนำเสนอ



ในส่วนนี้ เราจะเรียนรู้วิธีการใช้เครื่องมือสแกนเครือข่าย Nmap เพื่อทำแผนที่เครือข่ายของคุณ เราจะเห็นว่ามันมีประสิทธิภาพเพียงใดในงานนี้ ผ่านตัวเลือกต่างๆ ของมัน สุดท้าย เราจะดูวิธีการต่างๆ ในการระบุเป้าหมายของการสแกนของเราไปยัง Nmap



โดยเฉพาะอย่างยิ่ง เราจะใช้สิ่งที่เราได้เรียนรู้ในส่วนก่อนหน้าเกี่ยวกับวิธีที่ Nmap กำหนดว่าโฮสต์นั้นใช้งานอยู่และสามารถเข้าถึงได้หรือไม่



ตามที่กล่าวไว้ในบทนำของ Nmap นี่คือ Network Mapper ดังนั้นจึงเป็นเครื่องมือที่สมบูรณ์แบบสำหรับการจัดทำรายการโฮสต์ที่เข้าถึงได้บนเครือข่าย ไม่ว่าจะเป็นเครือข่ายภายในหรือภายนอก



**การกลับมาของผู้เขียน:**



ในความเป็นจริง ในฐานะผู้ตรวจสอบความปลอดภัยทางไซเบอร์และผู้ทดสอบการเจาะระบบ ฉันใช้ Nmap อย่างเป็นระบบเมื่อดำเนินการทดสอบการเจาะระบบภายในเพื่อค้นหาว่าฉันอยู่ที่ไหน ใครคือเพื่อนบ้านของฉันบนเครือข่ายท้องถิ่นและเครือข่ายอื่นใดที่สามารถเข้าถึงได้ รวมถึงระบบที่ตั้งอยู่บนเครือข่ายเหล่านั้น วัตถุประสงค์ของฉันนั้นง่าย: เพื่อทำแผนที่เครือข่าย กำหนดขนาดของระบบสารสนเทศ และโดยเฉพาะอย่างยิ่ง ร่างพื้นผิวการโจมตีของมัน



การทำแผนที่เครือข่ายยังสามารถเป็นประโยชน์ในบริบทของการวินิจฉัยเครือข่าย, การควบคุมดูแล, การทำแผนที่สินทรัพย์ (คุณแน่ใจจริงหรือว่าระบบสารสนเทศของคุณประกอบด้วยสิ่งที่อยู่ใน Active Directory หรือใน GLPI/OCS Inventory เท่านั้น? นอกจากนี้ยังสามารถใช้เพื่อตรวจจับการมีอยู่ของ Shadow IT ในระบบสารสนเทศของคุณได้อีกด้วย



### II. การใช้ Nmap เพื่อสแกนช่วงเครือข่าย



#### A. การค้นหาเครือข่ายด้วยการสแกน Nmap



ตอนนี้เราต้องการเพิ่มระดับและวิเคราะห์เครือข่ายท้องถิ่นทั้งหมดของเรา ไม่มีอะไรจะง่ายไปกว่านี้: สิ่งที่เราต้องทำคือใช้คำสั่งที่เราเห็นในส่วนก่อนหน้าอีกครั้ง แต่ระบุ CIDR แทนที่จะเป็นที่อยู่ IP แบบง่าย ๆ



CIDR (**Classless Inter Domain Routing**) คือการเขียนแบบ "คลาสสิก" สำหรับการระบุช่วงเครือข่ายและขอบเขตของมัน (โดยใช้มาสก์) ตัวอย่างเช่น "192.168.0.0/24" เป็น "การแปล" ของการเขียนมาสก์แบบทศนิยม "255.255.255.0"



ในการใช้ Nmap โดยการระบุ CIDR เราสามารถใช้งานได้ดังนี้:



```
# Scan a CIDR
nmap 192.168.0.0/24
```



นอกจากนี้ยังสามารถ, เช่นเดียวกับพอร์ตในส่วนก่อนหน้า, ระบุโฮสต์หลายตัว, เครือข่ายหลายเครือข่าย, หรือช่วง:



```
# Scan several networks at once via their CIDR
nmap 192.168.0.0/24 192.168.1.0/24

# Scan several hosts via their IP
nmap 192.168.1.2 192.168.1.3 192.168.1.10-20

# Mix of both
nmap 192.168.0.0/24 192.168.1.3 192.168.1.10-20
```



นี่คือตัวอย่างของผลลัพธ์ที่เราอาจได้รับเมื่อทำการสแกนบนเครือข่าย:



![nmap-image](assets/fr/26.webp)



ผลลัพธ์ของการสแกน Nmap เพื่อทำแผนที่เครือข่ายหลายเครือข่าย



โดยเฉพาะอย่างยิ่ง เราจะเห็นโฮสต์ที่ใช้งานอยู่หลายตัว และแต่ละส่วนของโฮสต์จะเริ่มต้นด้วยบรรทัดเช่นนี้:



```
Nmap scan report for <name> (<IP>)
```



สิ่งนี้ทำให้เรามองเห็นได้ชัดเจนว่าผลลัพธ์ต่อไปนี้อ้างอิงถึงโฮสต์ใด บรรทัดสุดท้ายก็สำคัญเช่นกัน:



```
Nmap done: 512 IP addresses (5 hosts up) scanned in 21.43 seconds
```



เราทราบว่า ในเครือข่ายที่สแกน Nmap ค้นพบโฮสต์ที่ใช้งานอยู่ 5 โฮสต์



#### B. ภายใต้ประทุน: การวิเคราะห์เครือข่ายผ่าน Wireshark



ตอนนี้เราจะมาดูรายละเอียดเพิ่มเติมเกี่ยวกับสิ่งที่เกิดขึ้นในระดับเครือข่ายระหว่างการค้นหาเครือข่ายที่ดำเนินการผ่าน Nmap



ตามที่เราเห็นในส่วนก่อนหน้า โดยค่าเริ่มต้น Nmap จะใช้โปรโตคอล ARP เพื่อตรวจจับการมีอยู่ของโฮสต์บนเครือข่ายท้องถิ่น:



![nmap-image](assets/fr/27.webp)



แพ็กเก็ต aRP ที่จับได้เมื่อสแกนเครือข่ายท้องถิ่นโดยใช้ Nmap และตัวเลือกเริ่มต้น



ดังนั้นจึงสามารถตรวจจับโฮสต์ทั้งหมดบนเครือข่ายท้องถิ่นได้ เนื่องจากการตอบสนองต่อคำขอ ARP มักจะได้รับจากโฮสต์ทั้งหมดที่ใช้งานอยู่บนเครือข่ายและไม่ปรากฏว่าน่าสงสัยแต่อย่างใด



สำหรับเครือข่ายระยะไกล Nmap ใช้การผสมผสานของเทคนิค:



![nmap-image](assets/fr/28.webp)



iCMP และ TCP แพ็กเก็ตที่ถูกจับเมื่อทำการสแกนเครือข่ายระยะไกลโดยใช้ Nmap และตัวเลือกเริ่มต้น



เพื่อให้แม่นยำยิ่งขึ้น Nmap ใช้แพ็กเก็ต ICMP echo (กรณีคลาสสิกของการ ping) และแพ็กเก็ต ICMP Timestamp ซึ่งมักใช้ในการคำนวณเวลาการส่งผ่านของแพ็กเก็ต โดยหวังว่าจะได้รับการตอบสนองจากโฮสต์บนเครือข่ายระยะไกล



แต่มันมีอะไรมากกว่านั้น คุณสามารถเห็นใน Wireshark capture ข้างต้นว่าแพ็กเก็ต **TCP SYN** ถูกส่งอย่างเป็นระบบไปยังพอร์ต TCP/443 ของโฮสต์ที่เป็นไปได้ทุกตัวบนเครือข่ายที่จะสแกน เช่นเดียวกับแพ็กเก็ต **TCP ACK** บนพอร์ต TCP/80



**ทำไมต้องส่งแพ็กเก็ต TCP ไปยังพอร์ตต่าง ๆ เป็นส่วนหนึ่งของการค้นหาเครือข่าย?**



การส่งแพ็กเก็ต SYN ไปยังพอร์ตที่กำหนดช่วยให้ Nmap สามารถ **ตรวจสอบว่ามีบริการกำลังฟังอยู่บนพอร์ตนั้นหรือไม่** หากโฮสต์ตอบกลับแพ็กเก็ต SYN ด้วยแพ็กเก็ต SYN/ACK นี่บ่งชี้ว่าโฮสต์นั้นกำลังทำงานและมีบริการกำลังฟังอยู่บนพอร์ตนั้น Nmap จึงลองเสี่ยงโชคกับบริการนี้ **แม้ว่าจะไม่ได้รับการตอบกลับจากการพิงก์ก็ตาม**



การส่ง ACK packet ไปยังพอร์ตที่กำหนดช่วยให้ Nmap **ตรวจสอบได้ว่ามีไฟร์วอลล์อยู่บนโฮสต์นั้นหรือไม่** หากโฮสต์ตอบสนองต่อ ACK packet ด้วย RST (Reset) packet นี่บ่งชี้ว่าอาจมีไฟร์วอลล์อยู่บนโฮสต์นั้นและกำลังบล็อกทราฟฟิกที่ไม่ได้ร้องขอ โฮสต์จึงเผยให้เห็นการมีอยู่บนเครือข่าย แม้ว่าจะไม่ได้ตอบสนองต่อคำขออื่น ๆ ก็ตาม



อย่างไรก็ตาม สิ่งสำคัญคือต้องทราบว่าการตรวจจับไฟร์วอลล์โดยใช้เทคนิคนี้อาจไม่เชื่อถือได้อย่างสมบูรณ์ในทุกกรณี โฮสต์บางตัวอาจส่งการตอบสนอง generate RST ด้วยเหตุผลอื่นที่ไม่ใช่การมีอยู่ของไฟร์วอลล์ เช่น การกำหนดค่าบริการหรือระบบปฏิบัติการเฉพาะ นอกจากนี้ ไฟร์วอลล์สมัยใหม่สามารถกำหนดค่าไม่ให้ตอบสนองต่อความพยายามในการค้นพบประเภทนี้ได้



ตอนนี้เราได้ก้าวมาไกลและสามารถทำการค้นหาเครือข่ายพื้นฐานได้แล้ว ตอนนี้เราจะมาดูตัวเลือกเพิ่มเติมอีกสองสามตัวที่จะให้เราควบคุมพฤติกรรมของ Nmap ได้มากขึ้น



### III. การค้นพบเครือข่ายโดยไม่ต้องสแกนพอร์ตด้วย Nmap



ตามที่คุณอาจสังเกตเห็น โดยค่าเริ่มต้น Nmap **จะทำการสแกนพอร์ตหลังจากค้นพบโฮสต์ที่ใช้งานอยู่** ซึ่งจะเพิ่มจำนวนแพ็กเก็ตอย่างมากและต้องรอการตอบสนองต่อการสแกนของเรา หากคุณมี 5 โฮสต์ในเครือข่ายของคุณ Nmap จะพยายามตรวจสอบสถานะของพอร์ตประมาณ 5,000 พอร์ต ซึ่งจะใช้เวลานานขึ้น



อย่างไรก็ตาม เป็นไปได้ที่จะใช้ตัวเลือกของ Nmap เพื่อทำการ **ค้นหาเฉพาะโฮสต์ที่ใช้งานอยู่** บนเครือข่าย โดยไม่ต้องค้นหาบริการของพวกเขา



หากเราต้องการทราบเพียงว่าโฮสต์ใดสามารถเข้าถึงได้ โดยไม่ต้องการข้อมูลเกี่ยวกับบริการและพอร์ตที่พวกเขาเปิดเผย เราสามารถใช้ตัวเลือก "-sn" เพื่อทำ **การสแกนโดยใช้ ICMP Echo (ping) และคำขอ ARP เท่านั้น** กล่าวคือ ปิดการสแกนพอร์ตทั้งหมด:



```
# Scan a CIDR in Echo ICMP
nmap 192.168.1.0/24 -sn
```



นี่คือผลลัพธ์ของการค้นพบเครือข่าย Nmap ที่ดำเนินการโดยไม่ใช้การสแกนพอร์ต:



![nmap-image](assets/fr/29.webp)



ผลลัพธ์ของการค้นพบเครือข่ายโดยไม่ใช้การสแกนพอร์ตกับ Nmap.



เราได้กล่าวถึงข้อจำกัดที่อาจเกิดขึ้นจากการใช้ ICMP เพียงอย่างเดียวสำหรับการค้นหาโฮสต์ (สำหรับเครือข่ายระยะไกล) นั่นคือเหตุผลที่ Nmap ใช้กลเม็ดบางอย่างที่สามารถเปิดเผยการมีอยู่ของไฟร์วอลล์หรือบริการเฉพาะบนโฮสต์ ด้วยความช่วยเหลือของตัวเลือกต่างๆ เราสามารถนำกลเม็ดเหล่านี้กลับมาใช้ใหม่และขยายเพิ่มเติมได้ โดยไม่จำเป็นต้องเริ่มต้นใหม่ด้วยการสแกนพอร์ตทั้งหมดของทุกโฮสต์ที่ค้นพบ



ในการทำเช่นนี้ เราจะใช้ตัวเลือก "-PS" (TCP SYN) และ "-PA" (TCP ACK) ซึ่งจะช่วยให้เราระบุพอร์ตที่เราต้องการเข้าร่วมเป็นส่วนหนึ่งของการค้นหาโฮสต์ของเรา รวมถึงตัวเลือก "-PP":



```
# Scan specific ports on a CIDR
nmap -sn -PP -PS22,3389,445,139 -PA80 192.168.1.0/24
```



การสแกนนี้ทำให้การค้นหาโฮสต์มีความสมบูรณ์มากขึ้นเล็กน้อยเมื่อเทียบกับตัวเลือกเริ่มต้นแล้ว



เราเริ่มที่จะได้รับคำสั่งที่ครอบคลุมมากขึ้น โดยใช้ตัวเลือกหลายตัว นี่เป็นเพราะเรารู้ว่า Nmap ทำงานอย่างไรและข้อจำกัดของตัวเลือก "ค่าเริ่มต้น" ซึ่งบางครั้งอาจทำให้เราเสียเวลา หรือพลาดองค์ประกอบสำคัญไป นั่นคือเหตุผลทั้งหมดที่เราต้องใช้เวลาในการเรียนรู้ให้เชี่ยวชาญ!



เพื่อแสดงรายละเอียดตัวเลือกของคำสั่งซื้อครั้งล่าสุดของเรา:





- "`-sn`: ปิดการสแกนพอร์ตสำหรับแต่ละโฮสต์ที่ใช้งานที่ค้นพบโดย Nmap.





- "`-PP`: เปิดใช้งาน ICMP echo (ping scan) สำหรับการค้นหาโฮสต์





- "`-PS <PORT>`": ส่งแพ็กเก็ต TCP SYN ไปยังพอร์ตที่ระบุเพื่อตรวจจับบริการที่ใช้งานอยู่ซึ่งบ่งบอกถึงการมีอยู่ของโฮสต์ที่ไม่ได้ตอบสนองต่อการพิง





- "`-PA <PORT>`": ส่งแพ็กเก็ต TCP ACK ไปยังพอร์ตที่ระบุเพื่อตรวจจับไฟร์วอลล์ที่ใช้งานอยู่ซึ่งบ่งบอกถึงการมีอยู่ของโฮสต์ที่ไม่ได้ตอบสนองต่อการพิง




ในตัวอย่างข้างต้น ฉันระบุพอร์ตที่ฉันพิจารณาว่ามักจะถูกเปิดเผยบ่อยที่สุดในบริบท Nmap ของฉันสำหรับตัวเลือก "-PS" พอร์ตต่างๆ เหล่านี้จะถูกทดสอบบนโฮสต์แต่ละตัว ไม่ใช่เพื่อดูว่าบริการที่พวกเขาโฮสต์นั้นใช้งานอยู่จริงหรือไม่ แต่เพื่อดูว่าสิ่งนี้ช่วยให้เราค้นพบโฮสต์ที่ไม่ได้ตอบสนองต่อ ICMP Echo ของเราในขณะที่ยังคงใช้งานอยู่หรือไม่ (ผ่านการตอบสนองจากบริการหรือไฟร์วอลล์ของโฮสต์)



นี่คือสิ่งที่สามารถเห็นได้ในข้อมูลที่จับได้จากเครือข่ายในขณะที่ทำการสแกน ในกรณีนี้เป็นการดึงข้อมูลจากโฮสต์เป้าหมายเดียว:



![nmap-image](assets/fr/30.webp)



แพ็กเก็ตที่ส่งโดย Nmap ระหว่างการค้นพบเครือข่ายขั้นสูง โดยไม่ใช้การสแกนพอร์ต



เราพบแพ็กเก็ต TCP SYN ของเรา, TCP ACK บนพอร์ต TCP/80 และ ICMP echo ของเรา Nmap จะทำการทดสอบทั้งหมดนี้สำหรับแต่ละโฮสต์ที่เป็นเป้าหมายโดยการสแกนการค้นหาเครือข่ายของเรา



### IV. การใช้ไฟล์ของสินทรัพย์เพื่อกำหนดเป้าหมายด้วย Nmap



การระบุเป้าหมายสามารถกลายเป็นเรื่องซับซ้อนได้อย่างรวดเร็วในระบบสารสนเทศในชีวิตจริง ซึ่งบางครั้งอาจประกอบด้วยเครือข่ายหลายสิบหรือหลายร้อยเครือข่าย, ซับเน็ต และ VLAN นี่คือเหตุผลว่าทำไมการใช้ไฟล์เป็นแหล่งข้อมูลสำหรับ Nmap จึงง่ายกว่าการระบุทีละรายการบนบรรทัดคำสั่ง



ในการเริ่มต้น ให้สร้างไฟล์ง่ายๆ ที่มีหนึ่งรายการต่อบรรทัด:



![nmap-image](assets/fr/31.webp)



ไฟล์ที่มีเป้าหมายหนึ่งรายการ (โฮสต์หรือเครือข่าย) ต่อบรรทัด



ถัดไป เราสามารถใช้ตัวเลือก Nmap ทั้งหมดที่เห็นจนถึงตอนนี้และระบุตัวเลือก "-iL <path/file>":



```
# Scan a list of targets contained in a file
nmap -iL /tmp/mesCibles.txt
```



Nmap จะรวมเป้าหมายทั้งหมดที่อยู่ในไฟล์ของเราในการสแกน



หากคุณต้องการให้แน่ใจว่าเป้าหมายทั้งหมดของคุณจะถูกนำมาพิจารณา คุณสามารถใช้ตัวเลือก "-sL -n" ได้ Nmap จะทำการตีความเฉพาะ CIDRs และโฮสต์ในไฟล์และแสดงให้คุณเห็น โดยไม่ส่งแพ็กเก็ตใดๆ ผ่านเครือข่าย:



```
# Display targets contained in a file
nmap -iL /tmp/mesCibles.txt -sL -n

Starting Nmap 7.94SVN ( https://nmap.org ) at 2024-05-01 14:52 CEST
Nmap scan report for 192.168.0.0
Nmap scan report for 192.168.0.1
Nmap scan report for 192.168.0.2
Nmap scan report for 192.168.0.3
Nmap scan report for 192.168.0.4
Nmap scan report for 192.168.0.5
Nmap scan report for 192.168.0.6
Nmap scan report for 192.168.0.7
Nmap scan report for 192.168.0.8
Nmap scan report for 192.168.0.9
Nmap scan report for 192.168.0.10
Nmap scan report for 192.168.0.11
Nmap scan report for 192.168.0.12
```



สิ่งนี้ช่วยให้มั่นใจได้ว่ารายการโฮสต์ที่จะสแกนนั้นถูกต้อง



เคล็ดลับสำคัญสุดท้ายที่ฉันอยากจะแบ่งปันกับคุณเกี่ยวข้องกับ **การยกเว้นโฮสต์หรือเครือข่ายเป็นส่วนหนึ่งของการสแกน** ความจำเป็นในการยกเว้นโฮสต์อาจจำเป็นในหลายกรณี โดยเฉพาะอย่างยิ่งหากเราต้องการให้แน่ใจว่า **ส่วนประกอบที่ละเอียดอ่อนของระบบสารสนเทศจะไม่ถูกรบกวนหรือขัดจังหวะโดยการสแกนของเรา**



ตัวอย่างที่พบบ่อยของความต้องการดังกล่าวคือเมื่อบริษัทเป็นเจ้าของอุปกรณ์อุตสาหกรรม (PLC) หรืออุปกรณ์ดูแลสุขภาพ อุปกรณ์ดังกล่าวบางครั้งถูกออกแบบมาไม่ดี และไม่ได้ตั้งใจให้รับแพ็กเก็ตที่มีรูปแบบไม่ดี หรือรับมากเกินไป ด้วยเหตุผลที่ชัดเจนเกี่ยวกับความพร้อมใช้งานหรือความเสี่ยงทางธุรกิจ/มนุษย์ จึงควรที่จะยกเว้นพวกมันจากการทดสอบ



หากต้องการยกเว้นที่อยู่ IP หรือเครือข่ายจากการสแกนของเรา เราสามารถใช้ตัวเลือก "--exclude" ของ Nmap ได้ เช่น:



```
# Exclude an IP address in a CIDR scan
nmap 192.168.1.0/24 --exclude 192.168.1.140
```



ในตัวอย่างนี้ ฉันกำลังสแกนเครือข่าย "192.168.1.0/24" แต่ยกเว้นโฮสต์ "192.168.1.140" ที่ตั้งอยู่ที่นั่น จะไม่มีการส่งแพ็กเก็ตโดย Nmap ไปยังโฮสต์นี้ อีกตัวอย่างหนึ่งที่มีการยกเว้นซับเน็ต:



```
# Exclude a subnet in a CIDR scan
nmap 10.0.0.0/16 --exclude 10.0.100.0/24
```



ในทำนองเดียวกัน ฉันสแกนเครือข่ายขนาดใหญ่ "10.0.0.0/16" แต่เครือข่าย "10.0.100.0/24" จะไม่ถูกสแกน อีกครั้ง ฉันแนะนำให้ใช้ตัวเลือก "-sL -n" เพื่อให้ได้ภาพที่ชัดเจนมากว่ามีโฮสต์ใดบ้างที่จะถูกสแกนและโฮสต์ใดบ้างที่จะถูกยกเว้นจากการสแกน โดยเฉพาะอย่างยิ่งหากคุณกำลังดำเนินการในบริบทที่ละเอียดอ่อน



### V. การค้นพบเครือข่ายและการสแกนพอร์ต



ตอนนี้เราสามารถรวมสิ่งที่เราได้เรียนรู้ในส่วนนี้กับสิ่งที่เราได้เรียนรู้ในส่วนก่อนหน้าเกี่ยวกับตัวเลือกการสแกนพอร์ต โดยค่าเริ่มต้น เราได้เห็นว่า Nmap จะสแกนพอร์ตที่พบบ่อยที่สุด 1000 พอร์ตบนโฮสต์ที่ใช้งานอยู่ทุกตัวที่ค้นพบ เราได้เห็นวิธีป้องกันพฤติกรรมนี้หากเราไม่ต้องการ แต่เราสามารถควบคุมมันได้อย่างเต็มที่ และแม้กระทั่งขยายมันหากมันเหมาะสมกับความต้องการของเรา



ตัวอย่างเช่น คำสั่งต่อไปนี้จะตรวจสอบการมีอยู่ของบริการที่กำลังฟังบนพอร์ต TCP/22 บนโฮสต์ที่สแกนแต่ละตัว:



```
# Scan TCP/22 on a CIDR
nmap 192.168.0.0/24 192.168.1.0/24 -p 22
```



Nmap จะทำการค้นหาเครือข่ายก่อนเพื่อแสดงรายการโฮสต์ที่ใช้งานอยู่ และสำหรับแต่ละโฮสต์นั้น ตรวจสอบว่ามีบริการอยู่บนพอร์ต TCP/22



ในทำนองเดียวกัน เราสามารถทำการสแกน TCP พอร์ตทั้งหมดบนทุกโฮสต์ที่ค้นพบในเครือข่าย "192.168.0.0/24" โดยยกเว้นโฮสต์ "192.168.0.4" ตัวอย่างเช่น:



```
# Port scan of a CIDR with exclusion
nmap 192.168.0.0/24 --exclude 192.168.0.4 -p-
```



คุณมีอิสระในการรวมตัวเลือกทั้งหมดที่เราได้เรียนรู้มาเพื่อให้เหมาะกับความต้องการของคุณเอง



### บทสรุป



ในส่วนนี้ เราได้เห็นวิธีการใช้ Nmap เพื่อทำแผนที่เครือข่ายโดยใช้ตัวเลือกต่างๆ ตอนนี้เรามีความเข้าใจอย่างละเอียดเกี่ยวกับเป้าหมายของการสแกนของเรา รวมถึงพฤติกรรมการสแกนพอร์ตและวิธีการค้นหาโฮสต์ของ Nmap และที่สำคัญที่สุด เรารู้แล้วว่าพฤติกรรมและข้อจำกัดเริ่มต้นของ Nmap คืออะไร และวิธีการใช้ตัวเลือกหลักเพื่อก้าวไปข้างหน้า



ในส่วนถัดไป เราจะพิจารณากลไกและตัวเลือกสำหรับการค้นหาเวอร์ชันของบริการและระบบปฏิบัติการที่สแกนโดย Nmap.




## 6 - การตรวจจับเวอร์ชันของบริการและระบบปฏิบัติการด้วย Nmap



### I. การนำเสนอ



ในส่วนนี้ เราจะเรียนรู้วิธีการใช้ Nmap เพื่อค้นหาและตรวจจับเวอร์ชันของบริการและระบบปฏิบัติการที่ใช้โดยโฮสต์ที่ถูกสแกนอย่างแม่นยำ เราจะพิจารณาอย่างละเอียดว่า Nmap ทำงานนี้อย่างไร รวมถึงข้อจำกัดของเครื่องมือเพื่อให้เข้าใจและตีความผลลัพธ์ได้ดียิ่งขึ้น



ตามที่เราได้เห็นในส่วนก่อนหน้าของบทแนะนำนี้ โดยค่าเริ่มต้น Nmap จะไม่ตรวจสอบว่ามีบริการใดที่เปิดเผยบนพอร์ตที่มันสแกนและพิจารณาว่าเปิดอยู่ ดังนั้นหากคุณกำลังฟังบริการเว็บบนพอร์ต TCP/22 Nmap จะยังคงรายงานว่ามันเปิดอยู่ แต่เป็นบริการ `SSH` นี่เป็นเพราะมันใช้ [ฐานข้อมูล](https://www.it-connect.fr/cours-tutoriels/administration-systemes/stockage/bdd/) ที่อยู่ในระบบของคุณเพื่อค้นหาความสัมพันธ์ระหว่างพอร์ต/โปรโตคอลและชื่อของบริการ (ไฟล์ `/etc/services/`)



ในกรณีส่วนใหญ่ [Nmap](https://www.it-connect.fr/cours/nmap-cartographie-reseau-scan-de-vulnerabilites/) จะให้ข้อมูลที่ถูกต้องแก่คุณ เนื่องจากเป็นเรื่องที่หาได้ยากในสภาพแวดล้อมการผลิตที่จะพบกรณีดังกล่าว อย่างไรก็ตาม กรณีที่เหลือจะเป็นสถานการณ์ที่บริการคลาสสิก ([SSH](https://www.it-connect.fr/cours/comprendre-et-maitriser-ssh/), HTTP, ฯลฯ) ถูกเปิดเผยบนพอร์ตที่ไม่ใช่คลาสสิก (เช่น 2022 สำหรับบริการ SSH) ซึ่งในกรณีนี้ Nmap จะไม่พบการจับคู่ในฐานข้อมูลท้องถิ่น หรือไม่ตรงกับความเป็นจริง และคุณจะพลาดข้อมูลสำคัญไป



โชคดีที่ Nmap มีตัวเลือกและกลไกที่แม่นยำมากสำหรับการค้นหาว่าบริการใดที่อาจซ่อนอยู่หลังพอร์ตที่เปิดอยู่ มันยังมีฐานข้อมูลของการสืบค้นและลายเซ็นเพื่อระบุเทคโนโลยีและเวอร์ชันที่แน่นอน นอกจากบริการแล้ว Nmap ยังสามารถระบุเทคโนโลยีที่ใช้และเวอร์ชันของมันได้อีกด้วย



นั่นคือสิ่งที่เราจะพิจารณาในส่วนนี้



### II. วิธีการตรวจจับเทคโนโลยีหรือเวอร์ชัน



#### A. การเตือนความจำเกี่ยวกับวิธีการระบุเทคโนโลยีหรือเวอร์ชัน



การระบุเทคโนโลยีและเวอร์ชันเกี่ยวข้องกับการดึงชื่อของบริการ, CMS, แอปพลิเคชัน หรือซอฟต์แวร์ที่กำลังฟังอยู่บนพอร์ตที่ถูกเป้าหมาย ตัวอย่างเช่น หน้าเว็บถูกจัดการโดย CMS (`WordPress`), รันโดยบริการเว็บ (`Apache`, IIS, Nginx) และโฮสต์โดยเซิร์ฟเวอร์ (Linux หรือ Windows) แต่คุณจะรู้ได้อย่างไรว่าบริการเว็บใดกำลังทำงานอยู่?



วิธีการแบบคลาสสิกในการค้นหาข้อมูลนี้คือ _banner grabbing_ ซึ่งประกอบด้วยการค้นหาว่าบริการที่เกี่ยวข้องแสดงข้อมูลนี้ที่ใดและอ่านข้อมูลนั้น บ่อยครั้งมาก ในการกำหนดค่าหรือการประมวลผลเริ่มต้น บริการจะแสดงชื่อและแม้แต่เวอร์ชันของตนเป็นการตอบสนองแรกหลังจากการเชื่อมต่อ



![nmap-image](assets/fr/32.webp)



แสดงเวอร์ชันทันทีที่มีการเชื่อมต่อ TCP โดยบริการ FTP



ที่นี่เราจะเห็นว่าการเชื่อมต่อ TCP แบบง่ายไปยังบริการนี้ผ่าน `telnet` ส่งผลให้เกิดแพ็กเก็ต TCP ที่มีเทคโนโลยีและเวอร์ชันของมัน



เมื่อคุณมีแนวคิดเกี่ยวกับประเภทของบริการที่คุณกำลังจัดการ คุณยังสามารถส่งคำสั่งหรือคำขอเฉพาะไปยังบริการนั้นเพื่อดึงข้อมูลจากมันได้ คำขอ/คำสั่งเหล่านี้สามารถส่งไปโดยไม่รู้แน่ชัด (โดยไม่มั่นใจว่าเป็นประเภทบริการที่ถูกต้อง) โดยหวังว่าหนึ่งในนั้นจะกระตุ้นให้บริการที่เกี่ยวข้องตอบสนองกลับมา



ในกรณีอื่นๆ ที่ซับซ้อนมากขึ้น จำเป็นต้องส่งแพ็กเก็ตเฉพาะเพื่อกระตุ้นให้เกิดปฏิกิริยา เช่น ข้อผิดพลาด ซึ่งจะให้ข้อมูลรายละเอียดเกี่ยวกับเวอร์ชันหรือเทคโนโลยีที่ใช้



ตามที่คุณสามารถจินตนาการได้ Nmap จะใช้เทคนิคทั้งหมดเหล่านี้เพื่อพยายามระบุประเภทที่แน่นอนของบริการที่โฮสต์บนพอร์ต รวมถึงชื่อของเทคโนโลยีและเวอร์ชันของมันด้วย



#### B. การทำความเข้าใจโพรบและการจับคู่ Nmap



ในการดำเนินการตรวจสอบทั้งหมดนี้บนพอร์ตที่สแกนแต่ละพอร์ต Nmap ใช้ฐานข้อมูลท้องถิ่นซึ่งมีการอัปเดตบ่อยครั้ง (เช่นเดียวกับไบนารีหรือโมดูลของมัน) นี่คือไฟล์ข้อความที่มีหลายพันบรรทัด: `/usr/share/nmap/nmap-service-probes`.



ไฟล์นี้ประกอบด้วยหลายรายการ ซึ่งทั้งหมดจัดระเบียบตามแนวทางหลักสองข้อ:





- `Probe`: นี่คือคำจำกัดความของแพ็กเก็ตที่ Nmap จะส่งในความพยายามที่จะกระตุ้นปฏิกิริยาจากบริการที่จะถูกระบุ คิดซะว่าเป็นความพยายามแบบสุ่มๆ เช่น _Hello? Guten Tag? Hello? Um... Buenos Dias perhaps?_ ทันทีที่บริการเป้าหมายได้รับ probe ที่เข้าใจ (เช่น พูดคุยด้วยโปรโตคอลที่ถูกต้อง) มันจะตอบกลับไปยัง Nmap ซึ่งจะทำให้มีการยืนยันประเภทของบริการนั้นๆ





- ตรงกัน": เหล่านี้คือ regular expressions ที่ Nmap ใช้กับการตอบสนองที่ได้รับ หากการส่งคำขอ HTTP GET ได้กระตุ้นการตอบสนองจากบริการ มันจะใช้ regular expressions หลายสิบรายการกับการตอบสนองนี้เพื่อระบุการมีอยู่ของ, ตัวอย่างเช่น, คำว่า `Apache`, `Nginx`, `Microsoft IIS`, เป็นต้น




มีคำสั่งอื่น ๆ สำหรับกรณีเฉพาะอยู่บ้าง แต่คำสั่งหลักสำหรับการทำความเข้าใจว่า Nmap ทำงานอย่างไรและการปรับแต่งการใช้งานคือสิ่งเหล่านี้ เพื่อทำให้ทฤษฎีส่วนนี้ชัดเจนขึ้น นี่คือตัวอย่าง:



![nmap-image](assets/fr/33.webp)



ตัวอย่างของ Probe ในไฟล์ `/usr/share/nmap/nmap-service-probes` ของ Nmap



ในบรรทัดแรกของตัวอย่างนี้ เราจะเห็น Probe ที่เข้าใจได้ง่ายชื่อว่า `GetRequest` นี่คือแพ็กเก็ต TCP ที่มีคำขอ HTTP GET ว่างไปยังรากของบริการเว็บโดยใช้ HTTP/1.0 ตามด้วยการป้อนบรรทัดและบรรทัดว่าง



บรรทัด `ports` บอก Nmap ว่าจะส่ง Probe นี้ไปที่พอร์ตใด ซึ่งช่วยให้คุณสามารถจัดลำดับความสำคัญของการทดสอบและประหยัดเวลา



ในที่สุด เรามีตัวอย่างของ `match` สองตัวอย่าง ตัวอย่างแรกจะจัดประเภทบริการเว็บที่สแกนเป็น `ajp13` หากนิพจน์ปกติที่อยู่ในบรรทัดนี้ตรงกับการตอบสนองของบริการที่ได้รับ



เพื่อช่วยให้คุณเข้าใจว่า Probes มีลักษณะอย่างไร นี่คือรายการของ Probes บางส่วนที่คุณจะพบในไฟล์นี้ (มีทั้งหมด 188 รายการ ณ เวลาที่เขียนบทแนะนำนี้)



![nmap-image](assets/fr/34.webp)



ตัวอย่างของ Probes หลายตัวที่ใช้โดย Nmap และมีอยู่ในไฟล์ `/usr/share/nmap/nmap-service-probes`._



สองตัวแรก (เรียกว่า `NULL` และ `GenericLines`) น่าสนใจเป็นพิเศษที่นี่ เนื่องจากพวกมันเพียงแค่ส่งแพ็กเก็ต TCP ว่างเปล่าหรือแพ็กเก็ตที่มีการขึ้นบรรทัดใหม่ บริการเซิร์ฟเวอร์มักจะประกาศตัวเองทันทีที่ได้รับการเชื่อมต่อ โดยไม่ต้องมีการกระทำ คำสั่ง หรือคำขอเฉพาะจากลูกค้า



นี่คือตัวอย่างของ _match_ ที่ซับซ้อนกว่าเล็กน้อย:



```
# Match Nginx + version in an error 400 page
match ssl/http m|^HTTP/1.1 400 Bad Request\r\n.*?Server: nginx/([\d.]+)[^\r\n]*?\r\n.*<title>400 The plain HTTP request was sent to HTTPS port</title>|s p/nginx/ v/$1/ cpe:/a:igor_sysoev:nginx:$1/
```



นิพจน์ปกติที่แน่นอนจะอยู่ระหว่าง `m|` และ `|` ซึ่งใช้เพื่อกำหนดขอบเขตของนิพจน์ปกติในไฟล์นี้ โปรดใช้เวลาอ่านตัวอย่างนี้ทั้งหมด คุณจะสังเกตเห็นการเลือกในนิพจน์ปกติ: `([\d.]+)` ซึ่งใช้เพื่อแยกเวอร์ชัน ตัวอย่างนี้ยังนิยามองค์ประกอบอื่น ๆ เช่น ชื่อผลิตภัณฑ์ `p/nginx/`, เวอร์ชันที่ดึงมา `v/$1/` และ CPE พร้อมเวอร์ชัน `cpe:/a:igor_sysoev:nginx:$1/`.



CPE (Common Platform Enumeration) คือระบบสัญกรณ์มาตรฐานที่ใช้ในการระบุและอธิบายซอฟต์แวร์และฮาร์ดแวร์ รูปแบบนี้ช่วยให้การจัดการช่องโหว่และการกำหนดค่าความปลอดภัยมีประสิทธิภาพมากขึ้น และที่สำคัญที่สุดคือเป็นวิธีการที่เป็นเอกภาพในการแสดงผล ไม่ว่าจะเป็นผลิตภัณฑ์ใดก็ตาม ต่อไปนี้คือตัวอย่างสองตัวอย่างของ CPE: `cpe:/o:microsoft:windows_8.1:r1` และ `cpe:/a:apache:http_server:2.4.35`



ที่นี่เราระบุประเภทของพวกเขาอย่างชัดเจน `o` สำหรับ OS, `a` สำหรับแอปพลิเคชัน, ผู้ขาย, ผลิตภัณฑ์, และเวอร์ชัน



ดังนั้น ในกรณีที่มี _match_ กับหนึ่งใน regular expressions เหล่านี้ เราจะดึงข้อมูลไม่เพียงแค่ชื่อของบริการเท่านั้น แต่ยังรวมถึงเวอร์ชันและ CPE ที่แน่นอนด้วย ทำให้ง่ายต่อการค้นหา CVE ที่มีผลกระทบต่อเวอร์ชันนี้ คุณจะพบข้อมูลนี้ในผลลัพธ์มาตรฐานของ Nmap และคุณจะเห็นว่ามันมีประโยชน์มากสำหรับวัตถุประสงค์อื่น ๆ ที่เราจะครอบคลุมในอีกไม่กี่ส่วนถัดไป



ไวยากรณ์ที่แน่นอนของ _matches_ และโดยทั่วไปของคำสั่งในไฟล์ `/usr/share/nmap/nmap-service-probes` ไม่ได้หยุดเพียงแค่นั้น และอาจดูซับซ้อนหากคุณไม่คุ้นเคยกับการจัดการ Nmap และผลลัพธ์ของมัน อย่างไรก็ตาม คุณควรจำไว้อย่างน้อยถึงการมีอยู่และการทำงานทั่วไป ซึ่งจะมีประโยชน์ในภายหลังเมื่อคุณต้องการทำความเข้าใจหรือแก้ไขข้อบกพร่องของผลลัพธ์ ปรับแต่งการสแกน หรือแม้กระทั่งมีส่วนร่วมในการพัฒนา Nmap



### III. การใช้ Nmap เพื่อตรวจจับเวอร์ชัน



ตอนนี้เราจะใช้กลไก _Probe_ และ _Match_ ที่ซับซ้อนทั้งหมดนี้ผ่านตัวเลือกง่ายๆ: `-sV` นี่จะบอก Nmap อย่างง่ายๆ: พยายามค้นหาว่าบริการและเวอร์ชันที่แน่นอนของพอร์ตที่คุณตั้งค่าให้เปิดคืออะไร



```
# Enable service and version detection
nmap 192.168.1.0/24 -sV
```



นี่คือตัวอย่างที่สมบูรณ์ของผลลัพธ์จากคำสั่งดังกล่าว:



![nmap-image](assets/fr/35.webp)



ผลลัพธ์ของการตรวจจับเวอร์ชันของแอปพลิเคชันที่เปิดเผยบนเครือข่ายโดย Nmap



ที่นี่เราสามารถเห็นว่า Nmap ได้ระบุเวอร์ชันทั้งหมดของบริการเครือข่ายที่เปิดเผยโดยเป้าหมายนี้ และแสดงข้อมูลนี้ในคอลัมน์ `VERSION` ใหม่ สามารถเห็นข้อมูลที่ค่อนข้างแม่นยำได้ แม้กระทั่งระบบปฏิบัติการ หากข้อมูลนี้เป็นส่วนหนึ่งของลายเซ็นที่กู้คืนได้



เพื่อทำความเข้าใจในรายละเอียดว่าเกิดอะไรขึ้นระหว่างการสแกนหาช่องโหว่ เราสามารถใช้ตัวเลือก `--version-trace` ซึ่งจะให้มุมมองโหมดดีบัก โดยแสดง _Probe_ ที่นำไปสู่การตรวจพบ:



```
# Enable version detection debug
nmap 192.168.1.0/24 -sV --version-trace
```



ผลลัพธ์คือ เราจะมีข้อมูลจำนวนมากที่ต้องจัดการ พยายามระบุบรรทัดที่ขึ้นต้นด้วย `Service scan Hard match` จากนั้นคุณจะเห็นบรรทัดเช่นนี้:



```
Service scan hard match (Probe NULL matched with NULL line 789): 10.10.10.187:21 is ftp. Version: |vsftpd|3.0.3||
Service scan hard match (Probe NULL matched with NULL line 3525): 10.10.10.187:22 is ssh. Version: |OpenSSH|7.4p1 Debian 10+deb9u7|protocol 2.0|
Service scan hard match (Probe GetRequest matched with GetRequest line 10510): 10.10.10.187:80 is http. Version: |Apache httpd|2.4.25|(Debian)|
```



เราสามารถเห็นได้อย่างชัดเจนว่า _Probes_ ใดถูกใช้ในการตรวจจับเทคโนโลยีและเวอร์ชัน (ในกรณีนี้คือ `NULL` และ `GetRequest` _Probes_) รวมถึงข้อมูลที่ดึงมาได้



### IV. การทดสอบและความแม่นยำในการตรวจจับขั้นสูง



เราจะกลับไปที่คำสั่งในไฟล์ `/usr/share/nmap/nmap-service-probes` ที่เรายังไม่ได้ดูมาก่อน:



![nmap-image](assets/fr/36.webp)



ตรวจสอบคำสั่ง `rarity` ในไฟล์ `/usr/share/nmap/nmap-service-probes`._



คำสั่งนี้ใช้เพื่อระบุความหายาก (เช่น ลำดับความสำคัญ/ความน่าจะเป็น) ที่เกี่ยวข้องกับ _Probe_ สัญลักษณ์นี้จาก 1 ถึง 9 ช่วยให้คุณควบคุมความสมบูรณ์ของการวิเคราะห์ที่ดำเนินการโดย Nmap เมื่อส่ง _Probes_ ในระบบ "สัญลักษณ์" ของ Nmap ความหายากที่ระดับ 1 ให้ข้อมูลในกรณีส่วนใหญ่ ในขณะที่ความหายากที่ระดับ 8 หรือ 9 แสดงถึงกรณีพิเศษมาก เฉพาะกับการกำหนดค่าหรือบริการที่พบได้ยาก



เพื่อให้ชัดเจนยิ่งขึ้น ในกรณีเริ่มต้น Nmap จะส่ง _Probes_ ที่มีความหายากตั้งแต่ 1 ถึง 7 ไปยังแต่ละบริการเพื่อระบุ เพื่อให้คุณมีความเข้าใจที่ดียิ่งขึ้นเกี่ยวกับการกระจายของ _Probes_ ตาม _rarity_ นี่คือจำนวนของพวกมัน:



```
$ grep -E "^rarity" nmap-service-probes |sort |uniq -c

6 rarity 1
1 rarity 2
3 rarity 3
8 rarity 4
9 rarity 5
13 rarity 6
8 rarity 7
81 rarity 8
54 rarity 9
```



อาจดูขัดแย้ง แต่มี `rarity` 8 และ 9 มากกว่าส่วนที่เหลือ นี่เป็นเพราะว่า Probes ที่มีความหายากระดับ 1 นั้นเป็นแบบทั่วไปและทำงานได้ในกรณีส่วนใหญ่ ไม่ว่าจะเป็นบริการใด (จำ Probe `NULL` ที่เพียงแค่ส่งแพ็กเก็ต TCP ว่างเปล่า) ในขณะที่ Probes ที่ซับซ้อนกว่านั้นเกือบจะเป็นเอกลักษณ์เฉพาะสำหรับแต่ละบริการ



หากเราต้องการจัดการ Probes ด้วยตนเองที่เราต้องการใช้ในการสแกนเวอร์ชัน เราสามารถใช้ตัวเลือก `--version-intensity` ได้ นี่คือตัวอย่างสองตัวอย่าง:



```
# Less accurate version detection than default
nmap 192.168.1.0/24 -sV --version-intensity 2

# Very deep detection, using all existing Probes
nmap 192.168.1.0/24 -sV --version-intensity 9
```



เพื่อจบในหัวข้อนี้ นี่คือตัวอย่างของ _Probe_ 9 และ 8:



![nmap-image](assets/fr/37.webp)



ตัวอย่างของ Probe ที่มีความหายากระดับ 8 และ 9 ในไฟล์ `/usr/share/nmap/nmap-service-probes`._



_Probes_ สองตัวนี้ตรวจจับเซิร์ฟเวอร์ Quake1 และ Quake2 (วิดีโอเกม) น่าสนใจสำหรับด้านที่ทำให้นึกถึงอดีต แต่ไม่น่าจะมีประโยชน์มากนักในชีวิตประจำวัน



ขึ้นอยู่กับความต้องการของคุณในด้านความแม่นยำหรือความเร็ว จำไว้ว่ามีหลักการ `rarity` นี้อยู่และสามารถส่งผลต่อผลลัพธ์ได้



### V. การใช้ Nmap เพื่อตรวจจับระบบปฏิบัติการ



ตอนนี้เราจะมาดูว่า Nmap สามารถช่วยเราตรวจจับระบบปฏิบัติการของโฮสต์ที่ถูกสแกนและตรวจพบในเครือข่ายได้อย่างไร ในการทำเช่นนี้ ให้ใช้ตัวเลือก `-O` (สำหรับ `OS Scan`) ของ Nmap



```
# Enable OS Scan
nmap -O 10.10.10.0/24
```



นี่คือตัวอย่างของผลลัพธ์ ที่นี่ Nmap บอกเราว่ามันน่าจะเป็นระบบปฏิบัติการ Linux และให้สถิติต่างๆ เกี่ยวกับเวอร์ชันที่แน่นอนของมันแก่เรา



![nmap-image](assets/fr/38.webp)



การตรวจจับความน่าจะเป็นของการระบุระบบปฏิบัติการโดย Nmap



ในการบรรลุเป้าหมายนี้ Nmap จะใช้เทคนิคหลากหลายที่ทำงานในลักษณะที่คล้ายคลึงกับ _Probes_ และ _Matches_ สำหรับการตรวจจับเทคโนโลยีและเวอร์ชัน ความแตกต่างหลักคือ Nmap จะใช้พารามิเตอร์ "ระดับต่ำ" ของ ICMP, TCP, UDP และโปรโตคอลอื่น ๆ ต่อไปนี้คือตัวอย่างการทดสอบสองตัวอย่างสำหรับการตรวจจับระบบปฏิบัติการ Microsoft Windows 11:



![nmap-image](assets/fr/39.webp)



ตัวอย่างของการทดสอบที่ดำเนินการโดย Nmap เพื่อการตรวจจับระบบปฏิบัติการ Windows 11



ลองมาดูกันเถอะว่า การทดสอบเหล่านี้ยากมากที่จะตีความ และเราจะไม่พยายามทำความเข้าใจอย่างลึกซึ้งในบริบทของบทเรียนเบื้องต้น Nmap หากคุณต้องการเจาะลึกในหัวข้อนี้ ไฟล์ที่มีข้อมูลนี้คือ `/usr/share/nmap/nmap-os-db`.



อย่างไรก็ตาม คุณจำเป็นต้องทราบว่าการตรวจจับระบบปฏิบัติการเป็นเพียงความน่าจะเป็นที่กำหนดโดย Nmap มากกว่าความแน่นอน



### VI. บทสรุป



ในส่วนนี้ เราได้เรียนรู้วิธีการใช้ตัวเลือกของ Nmap สำหรับการตรวจจับเทคโนโลยี เวอร์ชัน และระบบปฏิบัติการของโฮสต์และบริการที่สแกน เรามีความเข้าใจที่ดีเกี่ยวกับวิธีที่ Nmap ใช้ในการรับข้อมูลนี้จากระยะไกลแล้ว เรายังได้ทบทวนตัวเลือกสำหรับการจัดการความละเอียดและความแม่นยำของการทดสอบ รวมถึงข้อจำกัดของเครื่องมือในหัวข้อเหล่านี้ด้วย



ในส่วนถัดไป เราจะเรียนรู้วิธีใช้สคริปต์ NSE ของ Nmap เพื่อทำการวิเคราะห์ความปลอดภัยของระบบสารสนเทศของเรา ใช้เวลาในการอ่านทบทวนส่วนสุดท้ายหากจำเป็น และอย่าลังเลที่จะฝึกฝนและสำรวจเครื่องมือนี้ด้วยตัวคุณเองเพื่อให้เข้าใจสิ่งที่เราได้เรียนรู้มาจนถึงตอนนี้ได้ดียิ่งขึ้น




## 7 - การวิเคราะห์ความปลอดภัย: การตรวจจับช่องโหว่



### I. การนำเสนอ



ในส่วนนี้ เราจะเรียนรู้วิธีการใช้เครื่องมือสแกนเครือข่าย Nmap เพื่อตรวจจับและวิเคราะห์ช่องโหว่บนเป้าหมายของการสแกนของเรา โดยเฉพาะอย่างยิ่ง เราจะดูตัวเลือกต่างๆ ที่มีให้เพื่อทำงานนี้ให้สำเร็จ และศึกษาขีดจำกัดของความสามารถของเครื่องมือเพื่อให้เข้าใจและตีความผลลัพธ์ได้ดียิ่งขึ้น



ในส่วนแรกนี้ เราจะมาดูที่เครื่องสแกนหาช่องโหว่ของ Nmap และดูวิธีการใช้ตัวเลือกการตรวจจับช่องโหว่พื้นฐาน ในส่วนถัดไป เราจะมาดูรายละเอียดเพิ่มเติมเกี่ยวกับวิธีการทำงานของฟีเจอร์นี้ และวิธีการปรับแต่งให้เหมาะสม



### II. การใช้ Nmap เพื่อตรวจหาช่องโหว่



ตอนนี้เราต้องการใช้เครื่องสแกนเครือข่าย Nmap เพื่อตรวจจับช่องโหว่ในบริการและระบบของระบบสารสนเทศของเรา ซึ่งหมายความว่านอกจากการค้นหาโฮสต์ที่ใช้งานอยู่ การระบุบริการที่เปิดเผย และการตรวจจับเวอร์ชันและเทคโนโลยีแล้ว Nmap จะค้นหาช่องโหว่ด้วย



ในการบรรลุเป้าหมายนี้ Nmap อาศัยสคริปต์ NSE (_Nmap Scripting Engine_) ซึ่งสามารถมองว่าเป็นโมดูลที่ช่วยให้สามารถทดสอบได้อย่างละเอียด



ด้วยตัวเลือกที่ถูกต้อง เราจะขอให้ Nmap ใช้สคริปต์ NSE ต่างๆ ของมันกับแต่ละบริการที่ค้นพบ ทำให้เราสามารถค้นพบ:





- ข้อบกพร่องในการกำหนดค่า ;





- การค้นพบเวอร์ชันและระบบปฏิบัติการเพิ่มเติมและขั้นสูงมากขึ้น ;





- ช่องโหว่ที่รู้จัก (CVEs) ;





- ตัวระบุที่อ่อนแอ ;





- ลักษณะเฉพาะของการติดมัลแวร์ ;





- ความเป็นไปได้ของการปฏิเสธการให้บริการ ;





- เป็นต้น




ตามที่คุณเห็น สคริปต์ NSE ขยายขีดความสามารถของ Nmap อย่างมากในแง่ของการดำเนินงานเครือข่ายที่สามารถทำได้ และสิ่งนี้เพื่อทำงานที่ซับซ้อนมากขึ้นกว่าเดิม ข่าวดีก็คือ ตามปกติแล้ว ฟีเจอร์เหล่านี้สามารถใช้งานได้ง่ายๆ ผ่านทางออปชั่นและในบริบทเริ่มต้น นี่คือสิ่งที่เราจะเห็นต่อไป



### III. ตัวอย่างของการสแกนหาช่องโหว่



สคริปต์ NSE สามารถใช้เมื่อใช้ Nmap เพื่อสแกนพอร์ตเดียวบนโฮสต์ บริการทั้งหมดบนโฮสต์นั้น หรือบริการทั้งหมดที่ตรวจพบในหลายเครือข่าย ดังนั้นเราจึงสามารถใช้ตัวเลือกที่เรากำลังจะเห็นในทุกบริบทที่เราได้ศึกษาไปแล้ว



ในการเปิดใช้งานการสแกนช่องโหว่ผ่านการสแกน Nmap เราจำเป็นต้องใช้ตัวเลือก `-sC`:



```
# Enable vulnerability scanning during a scan
nmap -sC 10.10.10.152
```



โปรดจำไว้ว่าตามค่าเริ่มต้น หากคุณไม่ได้ระบุอะไร Nmap จะสแกนเฉพาะพอร์ตที่พบมากที่สุด 1000 พอร์ตเท่านั้น มันจะไม่ตรวจจับช่องโหว่บนพอร์ตที่แปลกใหม่กว่าที่เป้าหมายของคุณอาจเปิดเผย



ก่อนที่จะใช้ฟังก์ชันนี้ในระบบข้อมูลการผลิต ฉันขอเชิญคุณอ่านบทแนะนำต่อไป ในส่วนต่อไปนี้ เราจะพิจารณาวิธีการควบคุมผลกระทบและประเภทของการทดสอบที่จะดำเนินการให้ดียิ่งขึ้น



โดยการนำสิ่งที่เราได้เรียนรู้มาก่อนหน้านี้มาใช้ซ้ำ เราสามารถทำให้ครอบคลุมมากขึ้นและสแกนพอร์ต TCP ทั้งหมดของเป้าหมายได้ เช่น:



```
# Enable vulnerability scanning on all ports
nmap -sC -p- 10.10.10.152
```



นี่คือผลลัพธ์ของการสแกน Nmap โดยใช้สคริปต์ NSE:



![nmap-image](assets/fr/40.webp)



ตัวอย่างผลลัพธ์ของการสแกนหาช่องโหว่บนโฮสต์ผ่าน Nmap._



ที่นี่เราจะเห็นการแสดงข้อมูลเพิ่มเติมที่น่าสนใจในบริบทของการวิเคราะห์ช่องโหว่:





- บริการ FTP สามารถเข้าถึงได้โดยไม่ระบุตัวตน และไม่ได้รับการป้องกันด้วยการตรวจสอบสิทธิ์ สคริปต์ NSE ที่รับผิดชอบการตรวจสอบนี้บอกเราว่าเป็นเช่นนั้น และยังแสดงโครงสร้างบางส่วนของบริการ FTP ให้เราเห็นอีกด้วย ที่นี่เราจะเห็นว่าเราสามารถเข้าถึงไดเรกทอรี `C` ของระบบ Windows ได้!





- สคริปต์ NSE ที่รับผิดชอบการดึงข้อมูลเว็บเซอร์วิสขั้นสูงจะแสดงชื่อหน้าซึ่งช่วยให้เราเข้าใจได้ดีขึ้นว่าเว็บเซอร์วิสนั้นโฮสต์อะไรอยู่





- เรายังมีการวิเคราะห์ขนาดเล็กของการกำหนดค่าบริการ SMB (สคริปต์ `smb2-time`, `smb-security-mode` และ `smb2-security-mode`) ข้อมูลจะแสดงแตกต่างกันเล็กน้อย หลังจากผลการสแกนเครือข่าย เพื่อให้อ่านได้ง่ายขึ้น โดยเฉพาะอย่างยิ่ง ข้อมูลนี้บ่งชี้ถึงการไม่มีลายเซ็นการแลกเปลี่ยน SMB จุดอ่อนในการกำหนดค่านี้ทำให้เป้าหมายสามารถถูกใช้ในการโจมตี SMB Relay ซึ่งเป็นข้อบกพร่องด้านความปลอดภัยที่สำคัญที่มักถูกใช้ในการทดสอบการบุกรุก/การโจมตีทางไซเบอร์




แน่นอนว่านี่เป็นเพียงตัวอย่างหนึ่งเท่านั้น Nmap มีสคริปต์ NSE สำหรับบริการหลายอย่าง โดยมุ่งเป้าไปที่ช่องโหว่ที่หลากหลาย เราจะมาดูความเป็นไปได้เหล่านี้อย่างละเอียดในส่วนถัดไป



เพื่อสรุปการแนะนำการสแกนหาช่องโหว่นี้ นี่คือคำสั่งที่สมบูรณ์สำหรับการค้นหาเครือข่าย การสแกนพอร์ต TCP การตรวจจับเวอร์ชันและช่องโหว่:



```
# Complete and realistic vulnerability scan command
nmap -sV -sC -p- 192.168.0.0/24 192.168.1.13 192.168.2.10-20 --exclude 192.168.0.4
```



นี่คือคำสั่งที่เริ่มดูเหมือนกรณีการใช้งาน Nmap ที่สมจริงมากขึ้น!



### IV. การทำความเข้าใจข้อจำกัดของ Nmap ในการสแกนหาช่องโหว่



ขอให้ชัดเจน: Nmap ไม่สามารถดำเนินการทดสอบการเจาะระบบของระบบข้อมูลของคุณได้อย่างสมบูรณ์ หรือจำลองการปฏิบัติการของทีมแดง มันมีข้อจำกัดหลายประการที่คุณต้องทราบเพื่อไม่ให้เกิดความรู้สึกปลอดภัยที่ผิดพลาด:





- การครอบคลุมที่จำกัด**: แม้ว่า NSE scripts ของ Nmap จะมีความสามารถสูง แต่การครอบคลุมการทดสอบอาจจำกัดเมื่อเทียบกับเครื่องมือค้นหาช่องโหว่เฉพาะทางอื่น ๆ บางช่องโหว่อาจไม่ได้รับการครอบคลุมโดย NSE scripts ที่มีอยู่ เช่น ช่องโหว่ของ Active Directory การเปิดเผยข้อมูลที่ละเอียดอ่อน หรือกรณีที่ซับซ้อนมากขึ้นของแอปพลิเคชันเว็บที่มีช่องโหว่





- ความซับซ้อนของช่องโหว่**: ช่องโหว่บางประเภทอาจตรวจจับได้ยากโดยใช้สคริปต์ NSE เนื่องจากความซับซ้อนของมัน ตัวอย่างเช่น ช่องโหว่ที่ต้องการการโต้ตอบที่ซับซ้อนกับบริการระยะไกลอาจไม่ถูกตรวจจับได้อย่างมีประสิทธิภาพโดย Nmap (เช่นในกรณีของการอนุญาตที่มากเกินไปในการแชร์ไฟล์หรือข้อบกพร่องในการควบคุมการอนุญาตในเว็บแอปพลิเคชัน)





- การตรวจจับแบบพาสซีฟ**: Nmap มุ่งเน้นไปที่การสแกนแบบแอคทีฟเพื่อค้นหาช่องโหว่เป็นหลัก ซึ่งหมายความว่าอาจไม่สามารถตรวจจับช่องโหว่ที่อาจเกิดขึ้นได้อย่างมีประสิทธิภาพหากไม่ได้สร้างการเชื่อมต่อแบบแอคทีฟกับโฮสต์เป้าหมาย ช่องโหว่ที่ไม่ปรากฏตัวในระหว่างการสแกนแบบแอคทีฟอาจถูกมองข้ามไป (เช่นในกรณีของการฉีดโค้ดในเว็บแอปพลิเคชัน)





- การพึ่งพาการอัปเดต**: ฐานข้อมูลของ Nmap [สคริปต์ NSE](https://www.it-connect.fr/cours-tutoriels/administration-systemes/stockage/bdd/) กำลังพัฒนาอย่างต่อเนื่อง แต่ก็อาจมีความล่าช้าระหว่างการค้นพบช่องโหว่ใหม่และการเพิ่มสคริปต์ที่สอดคล้องกันใน Nmap ดังนั้น Nmap อาจไม่ทันสมัยเสมอไปกับช่องโหว่ล่าสุด





- ผลบวกลวงและผลลบลวง**: เช่นเดียวกับเครื่องมือรักษาความปลอดภัยอื่น ๆ สคริปต์ NSE ของ Nmap อาจสร้างผลบวกลวง (การแจ้งเตือนช่องโหว่ที่ผิดพลาด) หรือผลลบลวง (ช่องโหว่จริงที่ไม่ได้ตรวจพบ) นี่คือสิ่งที่ควรคำนึงถึงเมื่อวิเคราะห์ผลลัพธ์ของ Nmap




ดังนั้นจึงเป็นเรื่องสำคัญที่จะต้องเข้าใจว่า Nmap ทำอะไรและไม่ทำอะไร และเช่นเดียวกันต้องรู้วิธีการตีความผลลัพธ์ของมัน โดยเฉพาะอย่างยิ่ง เราได้เห็นตลอดบทแนะนำนี้ว่า ตัวเลือกเริ่มต้นอาจทำให้เราพลาดองค์ประกอบสำคัญที่สามารถค้นพบได้ด้วยการใช้งานอย่างระมัดระวัง



ไม่ว่าคุณจะเป็นผู้ดูแลระบบเครือข่าย วิศวกรความปลอดภัย หรือแม้แต่ CISO การใช้ Nmap จะช่วยให้คุณมีภาพรวมของสถานะความปลอดภัยของระบบสารสนเทศ นี่เป็นขั้นตอนแรกที่สำคัญในการรักษาความปลอดภัยของระบบ ซึ่งสามารถดำเนินการได้เป็นประจำโดยทีมไอที อย่างไรก็ตาม ไม่ควรแทนที่การแทรกแซงและคำแนะนำของ [ผู้เชี่ยวชาญด้านความปลอดภัยทางไซเบอร์](https://www.it-connect.fr/cours-tutoriels/securite-informatique/) ซึ่งจะสามารถค้นพบจุดอ่อนอย่างครอบคลุมมากกว่า Nmap



### V. บทสรุป



ในส่วนแรกของโมดูล 3 นี้ เราได้แนะนำการสแกนหาช่องโหว่ผ่านทาง Nmap เราได้เรียนรู้วิธีการใช้ตัวเลือกหลักในการทำงานนี้แล้ว แต่เราก็รู้ถึงขีดจำกัดของการฝึกหัดนี้ด้วย ในส่วนถัดไป เราจะมาดูฟังก์ชันนี้อย่างละเอียดขึ้น โดยใช้สคริปต์ NSE เพื่อขยายพลังของ Nmap ให้มากขึ้นถึงสิบเท่า



## 8 - การใช้สคริปต์ NSE ของ Nmap



### I. การนำเสนอ



ในส่วนนี้ เราจะพิจารณาอย่างละเอียดเกี่ยวกับสคริปต์ NSE (_Nmap Scripting Engine_) โดยเฉพาะเราจะดูว่าทำไมมันถึงเป็นหนึ่งในจุดแข็งที่ยอดเยี่ยมของเครื่องมือนี้ วิธีการทำงาน และวิธีการเรียกดูและใช้สคริปต์ที่มีอยู่มากมาย



ส่วนนี้เป็นการต่อเนื่องจากบทเรียนก่อนหน้านี้ ซึ่งเราได้เรียนรู้วิธีการใช้ฟีเจอร์การสแกนหาช่องโหว่ของ Nmap ในรูปแบบพื้นฐาน ตอนนี้เราจะมาดูรายละเอียดเพิ่มเติมเกี่ยวกับการทำงานของ [Nmap](https://www.it-connect.fr/cours/nmap-cartographie-reseau-scan-de-vulnerabilites/) ในแง่นี้ เพื่อที่เราจะสามารถดำเนินการสแกนที่มีความแม่นยำและควบคุมได้มากขึ้นอีกครั้ง



### II. แนวคิดของสคริปต์ Nmap NSE



สคริปต์ NSE ของ Nmap ช่วยให้คุณสามารถขยายความสามารถของมันได้อย่างยืดหยุ่นสูง พวกมันถูกเขียนด้วยภาษา LUA ซึ่งเป็นภาษาสคริปต์ที่ง่ายต่อการจัดการและเข้าถึงมากกว่าภาษา C หรือ C# ที่ใช้โดย Nmap ข้อดีของการใช้สคริปต์ LUA กับ Nmap แทนที่จะเป็นเครื่องมือแบบสแตนด์อโลนคือมันช่วยให้เราสามารถใช้ประโยชน์จากความเร็วในการประมวลผลและคุณสมบัติมาตรฐานของ Nmap (การค้นหาโฮสต์ พอร์ต และเวอร์ชัน ฯลฯ)



สคริปต์เหล่านี้ถูกจัดระเบียบตามหมวดหมู่ และสคริปต์เดียวอาจอยู่ในมากกว่าหนึ่งหมวดหมู่:





| หมวดหมู่ | คำอธิบาย |
|----------------|-------------|
| **auth** | ประกอบด้วยสคริปต์ที่เกี่ยวกับการตรวจสอบสิทธิ์บนบริการต่างๆ รวมถึงการเข้าถึงแบบไม่ระบุตัวตนหรือการแจกแจงรายชื่อผู้ใช้ ตัวอย่าง: `oracle-enum-users`, `ftp-anon` |
| **broadcast** | ประกอบด้วยสคริปต์ที่เกี่ยวกับการดำเนินการกระจายสัญญาณ (broadcast) บนเครือข่าย โดยเฉพาะเพื่อแสวงหาประโยชน์และค้นหาบริการ โฮสต์ หรือโปรโตคอลบางอย่างที่อาศัยการกระจายสัญญาณ (IPv6, wake on lan, IGMP ฯลฯ) ตัวอย่าง: `broadcast-dhcp6-discover`, `broadcast-ospf2-discover` |
| **brute** | ประกอบด้วยสคริปต์ที่เกี่ยวกับการดำเนินการสุ่มรหัสผ่าน (brute force) เพื่อตรวจสอบสิทธิ์บนบริการต่างๆ (brute force [SSH](https://www.it-connect.fr/cours/comprendre-et-maitriser-ssh/), MSSQL ฯลฯ) ตัวอย่าง: `ssh-brute`, `vnc-brute` |
| **default** | ประกอบด้วยสคริปต์ที่ใช้ในกรณีเริ่มต้น (การใช้ `-sC`) มีการใช้เกณฑ์หลายประการเพื่อตรวจสอบการเข้าสู่หมวดหมู่นี้ของสคริปต์ รวมถึงความเร็วในการทำงาน โครงสร้างของผลลัพธ์ ความน่าเชื่อถือของการทดสอบ ลักษณะการ "บุกรุก" หรือ "ความเสี่ยง" เป็นต้น |
| **discovery** | ประกอบด้วยสคริปต์ที่เกี่ยวกับการค้นหาเครือข่ายและบริการขั้นสูง ตัวอย่างเช่น การแจกแจงเนื้อหาของ SMB share, เวอร์ชันของบริการ VNC, การสอบถาม SNMP เป็นต้น ตัวอย่าง: `mysql-info`, `http-security-headers` |
| **dos** | ประกอบด้วยสคริปต์ที่อาจทำให้เกิดการปฏิเสธการให้บริการ (denial of service) อาจเป็นสคริปต์ที่สร้างขึ้นเพื่อแสวงหาประโยชน์จากช่องโหว่ประเภทการปฏิเสธการให้บริการ หรือสคริปต์ที่มีผลข้างเคียงเป็นการปฏิเสธการให้บริการ ดังนั้นควรใช้ความระมัดระวัง (สคริปต์เหล่านี้จะถูกแยกออกจากหมวดหมู่ `default`) ตัวอย่าง: `http-slowloris`, `ipv6-ra-flood` |
| **exploit** | ประกอบด้วยสคริปต์ที่สร้างขึ้นเพื่อแสวงหาประโยชน์จากช่องโหว่โดยตรง ตัวอย่าง: `http-shellsock`, `smb-vuln-ms08-067` |
| **external** | ประกอบด้วยสคริปต์ที่ต้องใช้ทรัพยากรจากภายนอก เช่น ฐานข้อมูลข้อมูลออนไลน์ สิ่งนี้บ่งชี้ถึงความพยายามในการเชื่อมต่อไปยังภายนอก (โปรดระวังเรื่องความเป็นส่วนตัว) ตัวอย่าง: `whois-ip`, `dns-blacklist`, `ip-geolocation-geoplugin` |
| **fuzzer** | ประกอบด้วยสคริปต์ที่ออกแบบมาเพื่อส่งเฟรม แพ็กเก็ต หรือพารามิเตอร์ที่บริการไม่คาดคิด สิ่งนี้ช่วยให้สามารถทำให้เกิดข้อผิดพลาดหรือการทำงานผิดปกติเพื่อรับเบาะแสของช่องโหว่หรือข้อมูลทางเทคนิค ตัวอย่าง: `dns-fuzz`, `http-form-fuzzer` |
| **intrusive** | ประกอบด้วยสคริปต์ที่ถูกจัดประเภทเป็น "ความเสี่ยง" จากมุมมองของความพร้อมใช้งานหรือการตรวจจับ สิ่งเหล่านี้อาจทำให้ระบบค้างหรือถูกตรวจพบว่าเป็นอันตรายโดยโซลูชันความปลอดภัย ซึ่งเป็นหมวดหมู่ที่ตรงกันข้ามกับ `safe` ตัวอย่าง: `smtp-brute`, `smb-vuln-ms08-067`, `smb-psexec` |
| **malware** | ประกอบด้วยสคริปต์ที่ออกแบบมาเพื่อตรวจหาองค์ประกอบที่มีลักษณะเฉพาะของมัลแวร์ เช่น พอร์ตที่เปิดรอการเชื่อมต่อซึ่งมักใช้โดย backdoor ที่รู้จัก ตัวอย่าง: `ftp-proftpd-backdoor`, `smtp-strangeport` |
| **safe** | ประกอบด้วยสคริปต์ที่ถือว่าปลอดภัยจากมุมมองของการตรวจจับหรือความเสถียร ซึ่งเป็นหมวดหมู่ที่ตรงกันข้ามกับ `intrusive` และส่วนใหญ่ประกอบด้วยสคริปต์ขั้นสูงสำหรับการระบุเวอร์ชันหรือการสำรวจองค์ประกอบการกำหนดค่า ตัวอย่าง: `html-title`, `smb2-security-mode`, `ms-sql-info` |
| **version** | ประกอบด้วยสคริปต์ที่ช่วยในการตรวจหาเวอร์ชันขั้นสูง สามารถใช้เสริมกับ Probes และ Matchs ที่ศึกษาก่อนหน้านี้ เมื่อการตรวจหาเวอร์ชันต้องมีการดำเนินการที่ซับซ้อนขึ้นเล็กน้อย ตัวอย่าง: `http-php-version`, `vmware-version` |
| **vuln** | ประกอบด้วยสคริปต์ที่ออกแบบมาเพื่อตรวจหาช่องโหว่ที่รู้จัก (CVE) โดยไม่แสวงหาประโยชน์จากช่องโหว่นั้น (ตรงข้ามกับหมวดหมู่ `exploit`) โดยทั่วไปจะทำเพียงการรายงานสถานะ " vulnerable" (มีช่องโหว่) หรือไม่ของบริการ ตัวอย่าง: `smb-vuln-ms17-010` (eternal blue), `http-phpmyadmin-dir-traversal` |

ในทางเทคนิค หมวดหมู่ที่สคริปต์เป็นของจะถูกระบุโดยตรงในโค้ดของมัน



![nmap-image](assets/fr/41.webp)



หมวดหมู่สคริปต์ nSE `ftp-anon`._

![nmap-image](assets/fr/43.webp)



รายการสคริปต์ NSE Nmap ที่มีชื่อขึ้นต้นด้วย `ftp-`._



Nmap ไม่มีตัวเลือกสำหรับการเรียกดูและแสดงรายการสคริปต์ NSE; คุณสามารถใช้คำสั่ง `--script-help` ตามด้วยชื่อของหมวดหมู่หรือคำ:



```
# List all scripts whose name starts with "ftp-"
nmap --script-help=ftp-*

# List all scripts from the "discovery" category
nmap --script-help=discovery
```



อย่างไรก็ตาม ผลลัพธ์จะเป็นชื่อของแต่ละสคริปต์และคำอธิบาย ซึ่งไม่เหมาะสมหากการค้นหานำเสนอหลายสิบสคริปต์:



![nmap-image](assets/fr/44.webp)



ผลลัพธ์จากการใช้คำสั่ง `--script-help` ของ Nmap



ในความคิดของฉัน วิธีที่มีประสิทธิภาพที่สุดคือการใช้คำสั่ง Linux แบบคลาสสิกในไดเรกทอรี `/usr/share/nmap/scripts/`:



```
# List scripts targeting the "ssh" service
ls -al /usr/share/nmap/scripts/ssh*

# List scripts from the "dos" category
grep -rl '"dos"' /usr/share/nmap/scripts/
```



อย่าลังเลที่จะเรียกดูโค้ดของโมดูลที่คุณสนใจ เพื่อทำความเข้าใจให้ดียิ่งขึ้นว่า NSE script ทำงานอย่างไร



### IV. การใช้สคริปต์ NSE ของ Nmap



ตอนนี้เราจะเรียนรู้วิธีการดำเนินการสแกนหาช่องโหว่โดยการเลือกสคริปต์ NSE ที่เราสนใจอย่างรอบคอบ



#### A. เลือกสคริปต์ตามหมวดหมู่



ในการเริ่มต้น เราสามารถเลือกที่จะรันสคริปต์ทั้งหมดที่อยู่ในหมวดหมู่เฉพาะได้ เราจำเป็นต้องระบุหมวดหมู่นี้หรือหมวดหมู่เหล่านี้ให้กับ Nmap ด้วยอาร์กิวเมนต์ `--script <category>`:



```
# Use default NSE scripts
nmap --script default 10.10.10.152
```



คำสั่งแรกนี้เทียบเท่ากับคำสั่ง `nmap -sC` โดยปกติแล้ว Nmap จะเลือกสคริปต์ในหมวดหมู่ `default` แต่ก็เพื่อการอภิปรายเท่านั้น คำสั่งถัดไป ตัวอย่างเช่น จะใช้สคริปต์ทั้งหมดในหมวดหมู่ `discovery`:



```
# Use NSE scripts from the "discovery" category
nmap --script discovery 10.10.10.152
```



ตามที่เราได้เห็น บางหมวดหมู่ช่วยให้เราสามารถระบุได้อย่างรวดเร็วว่าสคริปต์ NSE ที่เกี่ยวข้องทำอะไร (`discovery`, `vuln`, `exploit`) ในขณะที่หมวดหมู่อื่นๆ กำหนดระดับความเสี่ยง การตรวจจับ หรือความเสถียรของการทดสอบที่ดำเนินการ หากเราอยู่ในบริบทที่อ่อนไหวและไม่มีความเข้าใจที่ดีเกี่ยวกับการกระทำต่างๆ ที่ดำเนินการโดยการเลือกสคริปต์ของเรา เราสามารถเลือกที่จะรวมการเลือกเพื่อเลือกเฉพาะสคริปต์ที่อยู่ในหมวดหมู่ `discovery` และ `safe`:



```
# Use scripts from multiple categories
nmap --script "discovery and safe" 10.10.10.152
```



หากคุณต้องการที่จะยกเว้นสคริปต์จากหมวดหมู่ `dos` และ `intrusive` อย่างเด็ดขาดและชัดเจน คุณสามารถใช้สัญกรณ์ต่อไปนี้:



```
# Exclude categories
nmap --script "not intrusive and not dos" 10.10.10.152
```



โปรดทราบว่าการระบุเงื่อนไขการยกเว้นตามที่กล่าวมาข้างต้นจะส่งผลให้มีการใช้หมวดหมู่อื่นๆ ทั้งหมดที่ไม่ได้ถูกยกเว้นอย่างชัดเจน เพื่อให้เป็นธรรมมากขึ้น เราอาจระบุได้ เช่น:



```
# Include scripts from the "vuln" category except those that are too risky
nmap --script "vuln and not intrusive and not dos" 10.10.10.152

# Same thing, but only targeting the HTTP protocol
nmap --script "(http and vuln) and not intrusive and not dos" 10.10.10.152
```



นี่คือตัวอย่างบางส่วนของวิธีการจัดการสคริปต์ NSE ตามหมวดหมู่ โดยเฉพาะเมื่อใช้ Nmap สำหรับการวิเคราะห์ช่องโหว่ในบริบทของชีวิตจริง



#### B. เลือกสคริปต์เป็นหน่วย



เรายังสามารถเลือกที่จะดำเนินการทดสอบเฉพาะเจาะจงเพียงครั้งเดียวระหว่างการวิเคราะห์ ซึ่งเป็นการทดสอบที่สอดคล้องกับสคริปต์ NSE ในการทำเช่นนี้ เราจำเป็นต้องระบุชื่อของสคริปต์ในพารามิเตอร์ `-script <name>` โดยใช้ตัวอย่างสคริปต์ `ftp-anon.nse`:



```
# Use an NSE script and a specific port
nmap --script ftp-anon -p 21 10.10.10.152
```



จากนั้นเราก็มีผลลัพธ์ที่แม่นยำมาก:



![nmap-image](assets/fr/45.webp)



ผลลัพธ์จากการใช้สคริปต์ NSE `ftp-anon` บนพอร์ต FTP ผ่าน Nmap._



เราเห็นผลลัพธ์จากการรันสคริปต์ `ftp-anon` บนพอร์ต 21 และไม่มีพอร์ตอื่น เพราะเราได้ระบุออปชัน `-p 21` เราอาจทำการสแกนพอร์ตพื้นฐาน โดยการรันสคริปต์ NSE `ftp-anon` เฉพาะบนบริการ FTP ที่ค้นพบ:



```
# Use a specific NSE script
nmap --script ftp-anon 10.10.10.152
```



ดังนั้น Nmap ก็จะได้ดำเนินการทดสอบการเชื่อมต่อแบบไม่ระบุชื่อเช่นกัน หากพบบริการ FTP บนพอร์ตอื่น



สำหรับคำอธิบายสั้น ๆ เกี่ยวกับสิ่งที่สคริปต์ NSE ทำ คุณสามารถใช้ตัวเลือก `--script-help` ที่กล่าวถึงข้างต้น:



![nmap-image](assets/fr/46.webp)



ช่วยแสดงผลลัพธ์สำหรับสคริปต์ NSE `sshv1`._



โดยสรุป เราสามารถนำตัวเลือกการค้นหาเครือข่าย บริการ เวอร์ชัน และเทคโนโลยีทั้งหมดที่เราเคยใช้กลับมาใช้ใหม่ได้อีกครั้ง!



#### C. การจัดการอาร์กิวเมนต์ของสคริปต์



ในระหว่างการใช้ Nmap คุณจะพบกับสคริปต์ NSE บางตัวที่ต้องการอาร์กิวเมนต์อินพุตเพื่อให้ทำงานได้อย่างถูกต้อง ตอนนี้เราจะมาดูวิธีการส่งอาร์กิวเมนต์ไปยังสคริปต์เหล่านี้ผ่านตัวเลือกของ Nmap



ตัวอย่างเช่น ลองพิจารณาสคริปต์ `ssh-brute` ซึ่งช่วยให้คุณสามารถทำการโจมตีแบบ brute force บนบริการ SSH ได้



การโจมตีแบบ brute force แบบคลาสสิกประกอบด้วยการทดสอบรหัสผ่านหลาย ๆ อัน (บางครั้งเป็นล้าน) ในความพยายามที่จะยืนยันตัวตนกับบัญชีเฉพาะ โดยการพยายามรหัสผ่านจำนวนมากเช่นนี้ ผู้โจมตีเดิมพันกับความน่าจะเป็นที่ผู้ใช้ได้ใช้รหัสผ่านที่อ่อนแอในพจนานุกรมรหัสผ่านที่ใช้สำหรับการโจมตี



สคริปต์นี้มีตัวเลือก "default" ซึ่งเราสามารถปรับแต่งให้เหมาะสมกับบริบทของเราได้ ในบริบทของการโจมตีนี้ ตัวอย่างเช่น เราสามารถให้ Nmap พร้อมกับรายชื่อผู้ใช้และพจนานุกรมรหัสผ่านที่จะใช้ เท่าที่ฉันทราบ ยังไม่มีวิธีง่าย ๆ ในการแสดงรายการอาร์กิวเมนต์ที่จำเป็นสำหรับสคริปต์ ดังนั้นวิธีที่น่าเชื่อถือที่สุดคือการเยี่ยมชมเว็บไซต์ทางการของ Nmap ลิงก์ตรงไปยังเอกสารสำหรับสคริปต์ NSE สามารถรับได้จากการตอบสนองต่อออปชัน `--script-help`:



![nmap-image](assets/fr/47.webp)



ผลลัพธ์ของการแสดงความช่วยเหลือสำหรับสคริปต์ NSE `ssh-brute` พร้อมลิงก์ไปยัง nmap.org.



โดยการคลิกที่ลิงก์ที่ระบุ เรามาถึงหน้าเว็บนี้ของเว็บไซต์ [https://nmap.org](https://nmap.org/):



![nmap-image](assets/fr/48.webp)



รายการอาร์กิวเมนต์ที่สามารถส่งไปยังสคริปต์ NSE `ssh-brute` ของ Nmap



ที่นี่เรามีมุมมองที่ชัดเจนของอาร์กิวเมนต์ที่สามารถใช้ได้ โดยหลักๆ ในบริบทของเราคือ `passdb` (ไฟล์ที่มีรายการรหัสผ่าน) และ `userdb` (ไฟล์ที่มีรายการผู้ใช้) เอกสารที่นี่อ้างถึงไลบรารีภายใน Nmap เนื่องจากกลไกการโจมตีด้วยกำลังดุร้ายและตัวเลือกที่เกี่ยวข้องเหล่านี้ถูกทำให้เป็นมาตรฐานเพื่อให้สามารถใช้ได้อย่างสม่ำเสมอในหลายสคริปต์ (`ssh-brute`, `mysql-brute`, `mssql-brute`, ฯลฯ) และดังนั้นจะมีอาร์กิวเมนต์ที่เหมือนกันมากหรือน้อย:



```
# Create a file containing my user list
echo "root" > /tmp/userlist

# Create a file containing my password list
echo "123456" > /tmp/passlist
echo "NomEntreprise75" >> /tmp/passlist
echo "changezmoi" >> /tmp/passlist

# Run an SSH brute force via Nmap network scan
nmap --script ssh-brute --script-args userdb=/tmp/userlist,passdb=/tmp/passlist 10.10.10.245 192.168.1.19
```



ตามที่คุณเห็นในคำสั่งสุดท้ายนี้ เราสามารถระบุอาร์กิวเมนต์ที่จำเป็นสำหรับสคริปต์ Nmap โดยใช้ตัวเลือก `--scripts-args key=value,key=value` นี่คือตัวอย่างผลลัพธ์ที่เป็นไปได้ของการทำงาน Nmap เมื่อทำการโจมตี SSH brute force ผ่านสคริปต์ `ssh-brute` ของ NSE:



![nmap-image](assets/fr/49.webp)



ผลลัพธ์ของการโจมตี SSH แบบเดาสุ่มผ่าน Nmap._



ตามที่คุณเห็น ข้อมูลที่สร้างโดยสคริปต์ NSE จะมีคำนำหน้าเป็น `NSE: [script name]` ในผลลัพธ์แบบโต้ตอบ (ผลลัพธ์ในเทอร์มินัล) ทำให้ง่ายต่อการค้นหา ภายในการแสดงผลลัพธ์ของ Nmap ตามปกติ เรามีเพียงสรุปที่บ่งบอกว่ามีการค้นพบตัวระบุที่อ่อนแอหรือไม่ (รวมถึงรหัสผ่าน จำไว้ด้วย)



เพื่อก้าวไปอีกขั้น และเพื่อเตือนคุณว่าสิ่งเหล่านี้สามารถใช้เพิ่มเติมจากตัวเลือกทั้งหมดที่เราได้พิจารณาแล้ว นี่คือคำสั่งที่จะค้นหาเครือข่าย `10.10.10.0/24` สแกนพอร์ต TCP ที่ใช้บ่อยที่สุด 2000 พอร์ต และทำการค้นหาการเข้าถึงแบบไม่ระบุตัวตนบนบริการ FTP และแคมเปญ brute force บนบริการ SSH:



```
# Example of a complete command using multiple scripts
nmap --top-ports 2000 10.10.10.0/24 --script ftp-anon,ssh-brute --script-args userdb=/tmp/userlist,passdb=/tmp/passlist
```



นี่เป็นเพียงตัวอย่างหนึ่งของสคริปต์ที่มีอยู่มากมายและตัวเลือกของมัน แต่ตอนนี้เรามีความเข้าใจที่ดีขึ้นเกี่ยวกับวิธีการจัดการกับสคริปต์ NSE ไม่ว่าจะต้องการอาร์กิวเมนต์หรือไม่และวิธีการส่งอาร์กิวเมนต์เหล่านี้ไปยัง Nmap



### บทสรุป



ในส่วนนี้ เราได้เรียนรู้วิธีการใช้สคริปต์ NSE ของ Nmap เพื่อทำงานต่างๆ ฉันขอเชิญคุณใช้เวลาในการค้นพบหมวดหมู่ต่างๆ ของสคริปต์และตัวสคริปต์เอง เพื่อดูว่าพวกมันสามารถทำการทดสอบได้มากเพียงใด



ตลอดหลายส่วนที่ผ่านมา เราได้สะสมตัวเลือกการค้นพบ การสแกน และการนับจำนวนที่มีความก้าวหน้ามากขึ้นเรื่อยๆ จนถึงตอนนี้ คุณควรจะทราบแล้วว่าการแสดงผลและผลลัพธ์ของ Nmap เริ่มมีความครอบคลุมมากขึ้น บางครั้งอาจจะมีรายละเอียดมากเกินไปสำหรับเทอร์มินัลของเรา ในส่วนถัดไป เราจะเรียนรู้วิธีการจัดการกับผลลัพธ์นี้ โดยเฉพาะอย่างยิ่งการจัดเก็บในไฟล์ในรูปแบบต่างๆ






## 9 - การจัดการข้อมูลผลลัพธ์ Nmap




### I. การนำเสนอ



ในส่วนนี้ เราจะมาดูผลลัพธ์ที่ผลิตโดย Nmap และโดยเฉพาะอย่างยิ่งตัวเลือกต่างๆ สำหรับการจัดรูปแบบผลลัพธ์นี้ เราจะเห็นว่า Nmap สามารถผลิตรูปแบบผลลัพธ์ได้หลายแบบเพื่อตอบสนองความต้องการที่แตกต่างกัน และนี่ก็เป็นหนึ่งในจุดแข็งที่ยอดเยี่ยมของเครื่องมือนี้เช่นกัน



ตามค่าเริ่มต้น Nmap ให้มุมมองรายละเอียดของผลลัพธ์จากการสแกนและทดสอบที่ดำเนินการ ซึ่งรวมถึงโฮสต์และบริการที่ถูกสแกน, โฮสต์และบริการที่ตรวจพบว่าสามารถเข้าถึงได้, รายละเอียดของพอร์ตที่เปิด, สถานะและเวอร์ชันของพอร์ตเหล่านั้น นอกจากนี้ รายละเอียดของสคริปต์ NSE ยังมีให้ในผลลัพธ์ของเทอร์มินัลด้วย อย่างไรก็ตาม ผลลัพธ์นี้สามารถกลายเป็นข้อมูลจำนวนมากได้อย่างรวดเร็ว แม้ว่าจะมีการจัดรูปแบบข้อมูลอย่างชัดเจน ซึ่งอาจทำให้ยากต่อการค้นหาข้อมูลที่แม่นยำในผลลัพธ์



### II. การเชี่ยวชาญรูปแบบการส่งออก Nmap



#### A. บันทึกผลการสแกนในไฟล์ข้อความ



เพื่อให้ง่ายขึ้น [Nmap](https://www.it-connect.fr/cours/nmap-cartographie-reseau-scan-de-vulnerabilites/) ทำให้ง่ายมากในการบันทึกผลลัพธ์ลงในไฟล์ข้อความ ซึ่งสามารถเป็นประโยชน์สำหรับการเก็บถาวร การเปรียบเทียบกับการทดสอบอื่น ๆ แต่ยังสำหรับการเรียกดูผลลัพธ์นี้ด้วยเครื่องมือประมวลผลคำเฉพาะทางหรือภาษาสคริปต์ เช่น Sublime text, [PowerShell](https://www.it-connect.fr/cours-tutoriels/administration-systemes/scripting/powershell/), Python, grep, sed เป็นต้น เพื่อจัดเก็บผลลัพธ์มาตรฐานของ Nmap ลงในไฟล์ข้อความ เราสามารถใช้ตัวเลือก `-oN <filename>` (ตัว "N" ใน "normal"):



```
# Save Nmap output to a file
nmap 10.10.10.0/24 -sV -oN nmap_scan_10.10.10.0_24.txt
```



ไม่น่าแปลกใจเลยที่ Nmap จะแสดงผลลัพธ์มาตรฐานตามปกติในเทอร์มินัลของเรา แต่ยังแสดงในไฟล์ที่ระบุด้วย



#### ข. สร้างผลลัพธ์ Nmap ในรูปแบบย่อ



นอกจากนี้ยังมีรูปแบบเอาต์พุตที่สองในสไตล์ "text" ที่สามารถตีความได้ง่ายโดยมนุษย์: รูปแบบ "greppable"



รูปแบบนี้ถูกสร้างขึ้นเพื่อให้มุมมอง "ย่อ" ของผลลัพธ์ Nmap ซึ่งมีโครงสร้างในลักษณะที่เอื้อต่อการประมวลผลโดยเครื่องมือเช่น `grep` ลองมาดูตัวอย่างของผลลัพธ์ประเภทนี้:



![nmap-image](assets/fr/50.webp)



สแกนเครือข่าย nmap และส่งออกในรูปแบบ "greppable"



ที่นี่ ฉันได้ทำการค้นหาเครือข่ายรวมถึงการสแกนพอร์ตและการวิเคราะห์เทคโนโลยีและเวอร์ชันบนเครือข่าย /24 จากนั้นจัดเก็บผลลัพธ์ในไฟล์ในรูปแบบ "greppable" ฉันจบลงด้วยไฟล์ที่มี 2 บรรทัดต่อโฮสต์ที่ใช้งานอยู่:





- บรรทัดแรกบอกฉันว่าโฮสต์นั้น _Up_;





- บรรทัดที่สองบอกฉันว่าพอร์ตใดบ้างที่ถูกสแกน สถานะของพอร์ต และข้อมูลเทคโนโลยีและเวอร์ชันที่ดึงมาในรูปแบบที่เฉพาะเจาะจงมาก: `<port>/<status/<protocol>//<service>//<version>/,`




การจัดรูปแบบนี้ด้วยตัวคั่นแบบคงที่ช่วยให้การประมวลผลรวดเร็วขึ้นโดยเครื่องมือประมวลผลคำ เช่น `grep` หรือภาษาสคริปต์และการเขียนโปรแกรม คำสั่งต่อไปนี้, ตัวอย่างเช่น, ช่วยให้ฉันดึงข้อมูลเกี่ยวกับโฮสต์ `10.10.10.5` ได้อย่างง่ายดายในกรณีที่มีการสแกนขนาดใหญ่ที่ดำเนินการโดย Nmap ซึ่งผลลัพธ์จะยากต่อการเรียกดู:



```
# Filter by IP address in the Nmap "greppable" file
grep '10.10.10.5' nmap_10.10.10.0.gnmap
Host: 10.10.10.5 () Status: Up
Host: 10.10.10.5 () Ports: 21/open/tcp//ftp//Microsoft ftpd/, 80/open/tcp//http//Microsoft IIS httpd 7.5/ Ignored State: filtered (998)
```



ในทางกลับกัน ฉันยังสามารถแสดงรายการโฮสต์ทั้งหมดที่มีพอร์ต 21 เปิดได้อย่างง่ายดาย เนื่องจากพอร์ตและ IP อยู่ในบรรทัดเดียวกัน:



```
# Filter by open port in the Nmap "greppable" file
grep '21/open' nmap_10.10.10.0.gnmap
Host: 10.10.10.5 () Ports: 21/open/tcp//ftp//Microsoft ftpd/, 80/open/tcp//http//Microsoft IIS httpd 7.5/ Ignored State: filtered (998)

Host: 10.10.10.152 () Ports: 21/open/tcp//ftp//Microsoft ftpd/, 80/open/tcp//http//Indy httpd 18.1.37.13946 (Paessler PRTG bandwidth monitor)/, 135/open/tcp//msrpc//Microsoft Windows RPC/, 139/open/tcp//netbios-ssn//Microsoft Windows netbios-ssn/, 445/open/tcp//microsoft-ds//Microsoft Windows Server 2008 R2 - 2012 microsoft-ds/ Ignored State: closed (995)
```



เพื่อให้ได้ผลลัพธ์ generate เช่นนี้ เราจำเป็นต้องใช้ตัวเลือก `-oG <filename>.gnmap` (ตัว "G" ใน "grep") โดยปกติแล้ว ฉันใช้ส่วนขยาย `.gnmap` สำหรับไฟล์ประเภทนี้ แต่คุณสามารถใช้ส่วนขยายใดก็ได้ตามที่คุณต้องการ:



```
# Save Nmap output to a file in "greppable" format
nmap 10.10.10.0/24 -sV -oG nmap_scan_10.10.10.0_24.gnmap
```



รูปแบบนี้สามารถใช้สำหรับวัตถุประสงค์ที่หลากหลายและมีประโยชน์อย่างยิ่งสำหรับการเขียนสคริปต์/การจัดเรียงอย่างรวดเร็ว อย่างไรก็ตาม มันมีแนวโน้มที่จะถูกละทิ้งเพื่อสนับสนุนรูปแบบที่เราจะพิจารณาต่อไป



หมายเหตุ: รูปแบบ `-oG` ที่สามารถค้นหาได้ถูกยกเลิกอย่างเป็นทางการตั้งแต่ Nmap 7.90 ยังคงสามารถใช้งานได้เพื่อความเข้ากันได้ ยังคงสามารถใช้งานได้เพื่อวัตถุประสงค์ในการเข้ากันได้ แต่แนะนำให้ใช้รูปแบบ XML หรือรูปแบบปกติสำหรับการพัฒนาหรือการแยกวิเคราะห์อัตโนมัติ



#### C. รูปแบบ XML สำหรับเอาต์พุต Nmap



รูปแบบสุดท้ายที่ควรกล่าวถึงในบทแนะนำนี้คือ XML ซึ่งแตกต่างจากสองรูปแบบก่อนหน้านี้ รูปแบบนี้ไม่ได้ออกแบบมาให้อ่านโดยมนุษย์ แต่โดยเครื่องมือหรือสคริปต์อื่น ๆ



XML (_eXtensible Markup Language_) เป็นภาษามาร์กอัปที่ใช้ในการจัดเก็บและขนส่งข้อมูล โดยมีโครงสร้างแบบลำดับชั้นผ่านแท็กที่กำหนดเอง



ภายใน Nmap รูปแบบ XML ถูกใช้เพื่อรายงานรายละเอียด generate เกี่ยวกับการสแกนที่ดำเนินการ รวมถึงข้อมูลเกี่ยวกับโฮสต์ พอร์ต และช่องโหว่ที่ตรวจพบ รวมถึงข้อมูลเพิ่มเติมที่ไม่ได้แสดงในผลลัพธ์มาตรฐานของ Nmap



ในการสร้างไฟล์เอาต์พุตในรูปแบบ XML สำหรับ generate เราจำเป็นต้องใช้ตัวเลือก `-oX` ("O" จาก "XML"):



```
# Save Nmap output to a file in XML format
nmap 10.10.10.0/24 -sV -oX nmap_scan_10.10.10.0_24.xml
```



ผลลัพธ์คือเอาต์พุตมาตรฐานของ Nmap ในเทอร์มินัลของคุณ รวมถึงไฟล์ในรูปแบบ XML ในไดเรกทอรีปัจจุบันของคุณ



แน่นอนว่า รูปแบบ XML ไม่ได้ถูกออกแบบมาให้อ่านและตีความโดยมนุษย์ อย่างไรก็ตาม หากคุณต้องการทำการสคริปต์หรือการวิเคราะห์อัตโนมัติในรูปแบบผลลัพธ์ของ Nmap คุณยังคงต้องมีความเข้าใจเกี่ยวกับแท็กและโครงสร้างที่ใช้ ตัวอย่างเช่น นี่คือส่วนหนึ่งของเนื้อหาไฟล์ XML ที่สร้างโดย Nmap ซึ่งแสดงผลการสแกนสำหรับ 1 โฮสต์:



![nmap-image](assets/fr/51.webp)



ตัวอย่างของบันทึก XML สำหรับ 1 โฮสต์ระหว่างการสแกน Nmap



มีข้อมูลมากมายที่นี่ และเราสนใจเป็นพิเศษในพอร์ตเปิดสองพอร์ต:



```
<port protocol="tcp" portid="21"><state state="open" reason="syn-ack" reason_ttl="0"/><service name="ftp" product="Microsoft ftpd" ostype="Windows" method="probed" conf="10"><cpe>cpe:/a:microsoft:ftp_service</cpe><cpe>cpe:/o:microsoft:windows</cpe></service></port>
<port protocol="tcp" portid="80"><state state="open" reason="syn-ack" reason_ttl="0"/><service name="http" product="Microsoft IIS httpd" version="7.5" ostype="Windows" method="probed" conf="10"><cpe>cpe:/a:microsoft:internet_information_services:7.5</cpe><cpe>cpe:/o:microsoft:windows</cpe></service></port>
```



เราเข้าใจว่ารูปแบบนี้จะช่วยให้การแยกวิเคราะห์ผลลัพธ์โดยอัตโนมัติง่ายขึ้น เนื่องจากข้อมูลแต่ละชิ้นถูกจัดเรียงอย่างเรียบร้อยในแท็กหรือแอตทริบิวต์ที่มีชื่อเฉพาะ โดยเฉพาะอย่างยิ่ง เราพบข้อมูลชิ้นหนึ่งที่เราไม่เคยเจอมาก่อน: CPE.



```
cpe>cpe:/a:microsoft:internet_information_services:7.5</cpe><cpe>cpe:/o:microsoft:windows</cpe></service></port>
<cpe>cpe:/a:microsoft:internet_information_services:7.5</cpe><cpe>cpe:/o:microsoft:windows</cpe></service></port>
```



เราได้กล่าวถึง CPE อย่างสั้น ๆ ในส่วนที่ 2 ของโมดูล 2 และข้อมูลนี้ถูกกำหนดในแมตช์ระหว่างการตรวจจับเวอร์ชัน Nmap ใช้กลไกการบริการ เทคโนโลยี และการตรวจจับเวอร์ชันเพื่อค้นหา CPE ที่เกี่ยวข้อง



สิ่งนี้ช่วยให้เราสามารถนำข้อมูลนี้กลับมาใช้ใหม่กับฐานข้อมูลและแอปพลิเคชันที่ใช้งานได้ ฉันกำลังคิดถึงฐานข้อมูล NVD โดยเฉพาะ ซึ่งอ้างอิงถึง CVE สำหรับแต่ละ CVE จะมี CPE ที่ได้รับผลกระทบจากช่องโหว่ นี่คือตัวอย่างของ CVE ที่เกี่ยวข้องกับ `a:microsoft:internet_information_services:7.5` จากฐานข้อมูล NVD:



![nmap-image](assets/fr/52.webp)



การมีอยู่ของ CPE ในรายละเอียดของ CVE ในฐานข้อมูล NVD



ตอนนี้เรามีความเข้าใจที่ดีขึ้นเกี่ยวกับประโยชน์ของรูปแบบนี้ ซึ่งมีโครงสร้างข้อมูลที่ชัดเจนมากและมีข้อมูลทั้งหมดที่รวบรวมหรือประมวลผลโดย Nmap



ในฐานะปฏิกิริยาตอบสนอง ฉันบันทึกการสแกน Nmap ของฉันในทั้งสามรูปแบบพร้อมกันอย่างเป็นระบบ สิ่งนี้สามารถทำได้โดยใช้ตัวเลือก `-oA <name>` ("A" สำหรับ "All") ซึ่งจะสร้างไฟล์ `<name>.nmap` ไฟล์ `<name>.xml` และไฟล์ `<name>.gnmap` ด้วยวิธีนี้ฉันมั่นใจได้ว่าฉันจะไม่ขาดอะไรเมื่อฉันต้องการกลับไปดูผลลัพธ์อีกครั้ง



ด้วยรูปแบบทั้งสามนี้ คุณควรมีทุกสิ่งที่จำเป็นในการบันทึกและประมวลผลผลลัพธ์ Nmap ในแบบอัตโนมัติ เราจะใช้รูปแบบ XML อีกครั้งในส่วนถัดไป เมื่อเราดูการใช้ Nmap กับเครื่องมือรักษาความปลอดภัยอื่น ๆ



#### III. การสร้างรายงาน HTML จากการสแกน Nmap



รูปแบบ XML มีความเป็นไปได้มากมาย โดยเฉพาะอย่างยิ่งการทำหน้าที่เป็นพื้นฐานสำหรับการสร้างรายงานในรูปแบบ HTML ซึ่งจะทำให้การเรียกดูมีความน่าดึงดูดยิ่งขึ้น



ในการแปลงไฟล์ Nmap ในรูปแบบ XML ให้เป็นหน้าเว็บ เราจะใช้เครื่องมือ `xsltproc` ซึ่งเราจำเป็นต้องติดตั้งก่อน:



```
# Install the xsltproc tool
sudo apt install xsltproc
```



เมื่อติดตั้งเครื่องมือนี้แล้ว เพียงแค่ให้ไฟล์ XML ที่ต้องการแปลงและชื่อของรายงาน HTML ที่จะสร้าง:



```
# Create an Nmap HTML report from XML
xsltproc nmap_10.10.10.0.xml -o "Nmap – rapport web 05-2024.html"

# Open the .html file with Firefox
firefox "Nmap – rapport web 05-2024.html"
```



ผลลัพธ์คือ เราจะมีการสแกนทั้งหมดที่มีโครงสร้างอย่างดี พร้อมด้วยสีสันเล็กน้อยและลิงก์ที่คลิกได้ในสารบัญ!



![nmap-image](assets/fr/53.webp)



ดึงข้อมูลจากรายงานการสแกน Nmap ในรูปแบบ HTML ที่สร้างโดย xsltproc._



โดยทั่วไปแล้ว ไฟล์ XML ที่บันทึกโดย Nmap จะมีการอ้างอิงไปยังไฟล์อื่นในรูปแบบ XSL:



```
<?xml-stylesheet href="/usr/share/nmap/nmap.xsl" type="text/xsl"?>
```



การแปลงเป็น HTML จึงเป็นฟังก์ชันที่จัดหาและอำนวยความสะดวกโดย Nmap โดยที่ `xsltproc` เป็นเครื่องมือที่พบได้ทั่วไปและได้รับการยอมรับสำหรับการทำงานนี้ (ซึ่งไม่ได้มาจากชุดเครื่องมือ Nmap)



XSLT (_Extensible Stylesheet Language Transformations_) เป็นชุดย่อยของ XSL ที่ช่วยให้ข้อมูล XML สามารถแสดงบนหน้าเว็บและ "แปลง" ควบคู่ไปกับสไตล์ XSL ให้เป็นข้อมูลที่อ่านได้และมีรูปแบบในรูปแบบ HTML



แหล่งที่มา: [helpx.adobe.com/](https://helpx.adobe.com/fr/dreamweaver/using/xml-xslt.html)_



ระดับของข้อมูลในรายงานเทียบเท่ากับรูปแบบ XML ของ Nmap และสูงกว่าการแสดงผลทางเทอร์มินัลมาตรฐาน (_interactive output_)

![nmap-image](assets/fr/54.webp)



เอาต์พุตแบบละเอียดของ nmap โดยใช้ `-v`._ ตัวเลือก



ในการสแกนที่มุ่งเป้าไปที่โฮสต์และพอร์ตจำนวนมาก การแสดงผลในเทอร์มินัลจะยากต่อการใช้งานเนื่องจากปริมาณข้อมูลที่แสดง ด้วยเหตุนี้ ตัวเลือกนี้ควรใช้ร่วมกับตัวเลือกที่เห็นก่อนหน้านี้ ซึ่งช่วยให้คุณจัดเก็บผลลัพธ์มาตรฐานของ Nmap ในไฟล์ ข้อมูลที่เกี่ยวข้องกับการใช้ความละเอียดจะไม่รวมอยู่ในไฟล์ผลลัพธ์นี้ ดังที่คุณเห็นจากตัวอย่างข้างต้น ความละเอียดนี้ช่วยให้คุณติดตามการกระทำและการค้นพบของ Nmap ได้อย่างชัดเจนและตรงไปตรงมา สำหรับการสแกนที่ยาวนานขึ้นซึ่งการแสดงข้อมูลอาจช้าลง การทำเช่นนี้จะช่วยหลีกเลี่ยงการไม่รู้ถึงกิจกรรมปัจจุบันของ Nmap และรู้ว่าสิ่งต่าง ๆ กำลังดำเนินไปและในอัตราใด หากต้องการเพิ่มความละเอียดอีกระดับ คุณสามารถใช้ตัวเลือก `-vv`



เพื่อติดตามกิจกรรมของ Nmap เพิ่มเติมระหว่างการสแกน คุณสามารถใช้ตัวเลือก `--packet-trace` ได้ ด้วยตัวเลือก `-v` เราจะได้รับบันทึกสดของพอร์ตที่เปิดทั้งหมดที่ค้นพบโดย Nmap ในขณะที่ใช้ตัวเลือกนี้ เราจะได้รับบรรทัดบันทึกสำหรับแต่ละแพ็กเก็ตที่ส่งไปยังพอร์ต ซึ่งจะทำให้เกิดผลลัพธ์ที่มีความละเอียดมาก แต่ช่วยให้สามารถตรวจสอบกิจกรรมของ Nmap ได้อย่างละเอียด นี่คือตัวอย่าง:



![nmap-image](assets/fr/55.webp)



การตรวจสอบกิจกรรมของ Nmap อย่างละเอียดผ่าน `--packet-trace`._



อีกครั้ง ข้อมูลนี้จะไม่ถูกบันทึกในไฟล์ผลลัพธ์ที่ผลิตโดย Nmap หากมีการใช้ตัวเลือก `-oN`, `-oG`, `-oX` หรือ `-oA`



ในที่สุด Nmap ยังมีตัวเลือกการดีบักสองตัวเลือก: `-d` และ `-dd` ตัวเลือกเหล่านี้ทำงานคล้ายกับตัวเลือกความละเอียด `-v` แต่เพิ่มข้อมูลทางเทคนิคเพิ่มเติม เช่น สรุปพารามิเตอร์ทางเทคนิคที่จุดเริ่มต้นของการสแกน:



![nmap-image](assets/fr/56.webp)



ตัวเลือกการตั้งเวลาแสดงในมุมมองการดีบักของ Nmap



ในส่วนถัดไป เราจะมาดูว่า "ตัวเลือกการตั้งเวลา" คืออะไร และทำไมถึงควรรู้จักพวกมัน



สุดท้ายนี้ หากคุณต้องการเพียงภาพรวมพื้นฐานและสังเคราะห์ของความคืบหน้าของการสแกน Nmap คุณสามารถใช้ตัวเลือก `--stats-every 5s` ได้ "5s" ที่นี่หมายถึง 5 วินาทีและสามารถปรับเปลี่ยนให้เหมาะสมกับความต้องการของคุณได้ นี่คือความถี่ที่เราจะได้รับข้อมูลย้อนกลับจาก Nmap เกี่ยวกับความคืบหน้าของมัน:



![nmap-image](assets/fr/57.webp)



ข้อมูลที่แสดงโดยตัวเลือก `--stats-every` ของ Nmap



โดยเฉพาะอย่างยิ่ง เราสามารถรับเปอร์เซ็นต์ของความคืบหน้า รวมถึงการบ่งชี้ของเฟสที่อยู่ในขณะนั้น: เฟสการค้นหาโฮสต์ผ่าน [ping](https://www.it-connect.fr/le-ping-pour-les-debutants/), เฟสการค้นหาพอร์ต TCP ที่เปิดเผย เป็นต้น ข้อมูลนี้ยังสามารถได้รับในผลลัพธ์ของเทอร์มินัลโดยการกด "Enter" ระหว่างการสแกน



อย่างไรก็ตาม Nmap ไม่เก่งในการประมาณระยะเวลาที่งานจะใช้ ไม่ใช่น้อยเพราะมันไม่รู้ล่วงหน้าว่าจะต้องสแกนโฮสต์และบริการจำนวนเท่าใด



### บทสรุป



ในส่วนนี้ เราได้พิจารณาตัวเลือกหลายอย่างสำหรับการบันทึกผลการสแกน Nmap ในรูปแบบไฟล์ต่างๆ ซึ่งจะมีประโยชน์มาก เนื่องจากในบริบทที่เป็นจริง ผลการสแกนอาจมีหลายร้อยหรือแม้กระทั่งหลายพันบรรทัด! เรายังได้เห็นวิธีการเพิ่มระดับความละเอียดของ Nmap เพื่อวัตถุประสงค์ในการดีบักหรือเพื่อรับรายงานความคืบหน้าของการสแกนอีกด้วย



รูปแบบ XML จะมีประโยชน์อย่างยิ่งในส่วนถัดไป ซึ่งเราจะดูเครื่องมือบางอย่างที่สามารถทำงานร่วมกับผลลัพธ์ของ Nmap ได้




## 10 - การใช้ Nmap ร่วมกับเครื่องมือรักษาความปลอดภัยอื่น ๆ



### I. การนำเสนอ



ในส่วนนี้ เราจะมาดูการใช้งานคลาสสิกบางประการของ Nmap ร่วมกับเครื่องมือรักษาความปลอดภัยแบบฟรีและโอเพ่นซอร์สอื่นๆ โดยเฉพาะอย่างยิ่ง เราจะใช้สิ่งที่เราได้เรียนรู้ในส่วนก่อนหน้าเพื่อเพิ่มพลังและประสิทธิภาพของ Nmap ให้ดียิ่งขึ้น



ความสามารถในการบันทึกผลการสแกน Nmap ในรูปแบบ XML ทำให้ข้อมูลสามารถใช้งานร่วมกับเครื่องมืออื่นๆ ได้มากมาย เนื่องจากภาษาโปรแกรมและสคริปต์เกือบทั้งหมดในปัจจุบันมีไลบรารีที่สามารถแยกวิเคราะห์ XML ได้ ทำให้การประมวลผลข้อมูลนี้ง่ายขึ้นมาก เครื่องมือหลายตัว โดยเฉพาะอย่างยิ่งเครื่องมือที่มุ่งเน้นด้านความปลอดภัยเชิงรุก มีฟังก์ชันสำหรับประมวลผลรูปแบบ XML ที่สร้างโดย Nmap ลองมาดูรายละเอียดเพิ่มเติมกันเถอะ



ฉันจะกล่าวถึงเครื่องมือที่มีลักษณะเป็นการโจมตีบางอย่างโดยไม่ลงรายละเอียดเกี่ยวกับวิธีการใช้งานหรือการทำงานของมัน ฉันจะถือว่าผู้อ่านคุ้นเคยกับการใช้งานพื้นฐานของมันแล้วและเครื่องมือเหล่านี้พร้อมใช้งานแล้ว ส่วนนี้จะเป็นที่สนใจเป็นพิเศษสำหรับ [ผู้เชี่ยวชาญด้านความปลอดภัยทางไซเบอร์](https://www.it-connect.fr/cours-tutoriels/securite-informatique/), ผู้ที่อยู่ระหว่างการฝึกอบรม หรือผู้ที่ตัดสินใจที่จะเจาะลึกในหัวข้อนี้



### II. การนำเข้าผลลัพธ์ Nmap สู่ Metasploit



เครื่องมือแรกที่เราจะพิจารณาสำหรับการนำข้อมูล Nmap กลับมาใช้ใหม่ในด้านความปลอดภัยเชิงรุกและการวิจัยช่องโหว่คือ Metasploit.



Metasploit เป็นเฟรมเวิร์กสำหรับการโจมตีและการใช้ประโยชน์จากช่องโหว่ เป็นโซลูชันฟรีและเป็นเครื่องมือที่ได้รับการยอมรับซึ่งประกอบด้วยโมดูลจำนวนมากที่เขียนด้วย Ruby หรือ Python สิ่งเหล่านี้ช่วยให้สามารถใช้ประโยชน์จากช่องโหว่ ดำเนินการโจมตี สร้างแบ็คดอร์ จัดการการเรียกกลับ (ฟังก์ชัน C&C หรือ Communication and Control) และใช้งานทุกอย่างได้อย่างสม่ำเสมอ



โดยเฉพาะอย่างยิ่ง โครงสร้างการทำงานที่เป็นที่รู้จักและใช้งานอย่างแพร่หลายนี้สามารถทำงานร่วมกับฐานข้อมูล postgreSQL [database](https://www.it-connect.fr/cours-tutoriels/administration-systemes/stockage/bdd/) ซึ่งมีการจัดเก็บโฮสต์, พอร์ต, บริการ, ข้อมูลการยืนยันตัวตน และอื่น ๆ





- เอกสารอย่างเป็นทางการของ Metasploit: [https://docs.metasploit.com/](https://docs.metasploit.com/)




นี่คือจุดที่ Nmap และผลลัพธ์ของมันมีบทบาท เนื่องจากรูปแบบ XML ของผลลัพธ์ Nmap สามารถนำเข้าไปยังฐานข้อมูลของ Metasploit ได้อย่างง่ายดายเพื่อเติมฐานข้อมูลของโฮสต์และบริการ ซึ่งสามารถกำหนดเป็นเป้าหมายสำหรับการโจมตีต่างๆ ได้อย่างรวดเร็ว



เมื่ออยู่ใน Metasploit interactive shell ของฉัน ฉันเริ่มต้นด้วยการสร้าง workspace ซึ่งเป็นพื้นที่เฉพาะสำหรับสภาพแวดล้อมของฉันในวันนั้น:



```
# Create a Metasploit workspace
msf6 > workspace -a SI_siege
```



เมื่อสร้างพื้นที่ทำงานของฉันเสร็จแล้ว เราจำเป็นต้องตรวจสอบว่าการสื่อสารกับฐานข้อมูลทำงานได้:



```
# Retrieve the database status
msf6 > db_status
[*] Connected to msf. Connection type: postgresql.
```



ในที่สุด เราสามารถใช้คำสั่ง Metasploit `db_import` เพื่อ import การสแกน Nmap ของเราในรูปแบบ XML:



```
# Import a Nmap XML file into the database
msf6> db_import /tmp/nmap_10.10.10.0.xml
```



นี่คือผลลัพธ์จากการดำเนินการคำสั่งทั้งหมดนี้:



![nmap-image](assets/fr/58.webp)



นำเข้า Nmap สแกนในรูปแบบ XML เข้าสู่ฐานข้อมูล Metasploit



ที่นี่คุณสามารถเห็นได้ว่าแต่ละโฮสต์ถูกนำเข้า พร้อมกับบริการของมัน ข้อมูลนี้สามารถแสดงผลได้ผ่านคำสั่ง `services` หรือ `services -p <port>` สำหรับบริการเฉพาะ:



![nmap-image](assets/fr/59.webp)



รายการบริการที่นำเข้าจากไฟล์ XML ไปยังฐานข้อมูล Metasploit



ในที่สุด เราก็สามารถนำข้อมูลนี้กลับมาใช้ใหม่ในโมดูลได้อย่างรวดเร็วและง่ายดายด้วยตัวเลือก `-R` ซึ่งจะ "แปลง" รายการบริการที่ได้รับเป็นอินพุตสำหรับคำสั่ง `RHOSTS` ซึ่งใช้เพื่อระบุเป้าหมายของการโจมตีที่จะดำเนินการ นี่คือตัวอย่างกับโมดูล `ssh_login` ซึ่งช่วยให้คุณสามารถดำเนินการโจมตีแบบ brute force บนบริการ [SSH](https://www.it-connect.fr/cours/comprendre-et-maitriser-ssh/):



![nmap-image](assets/fr/60.webp)



ใช้ตัวเลือก `services -R` เพื่อนำเข้าบริการที่ระบุเป็นเป้าหมายของการโจมตี



นี่เป็นเพียงตัวอย่างเล็ก ๆ ของสิ่งที่สามารถทำได้ด้วยข้อมูล Nmap ใน Metasploit แต่ก็ให้คุณเห็นภาพเล็ก ๆ ว่าข้อมูลนี้สามารถนำกลับมาใช้ใหม่ได้อย่างรวดเร็วและง่ายดายเพียงใดในฐานะส่วนหนึ่งของการทดสอบการเจาะระบบ การสแกนหาช่องโหว่ หรือการโจมตีทางไซเบอร์ นอกจากนี้ยังควรกล่าวถึงว่า Nmap สามารถรันได้โดยตรงจาก Metasploit เพื่อที่จะนำผลลัพธ์เข้าสู่ฐานข้อมูล (คำสั่ง `db_nmap`) ซึ่งเป็นอีกหัวข้อที่น่าสนใจที่จะครอบคลุม!



### III. การใช้ Nmap กับเครื่องสแกนเว็บ Aquatone



เครื่องมือที่สองที่ฉันอยากจะแนะนำในส่วนนี้เกี่ยวกับการใช้ผลลัพธ์ของ Nmap สำหรับการรักษาความปลอดภัยเชิงรุกและการวิเคราะห์ช่องโหว่คือ Aquatone.



Aquatone เป็นเครื่องมือสแกนเว็บที่ออกแบบมาเพื่อสำรวจแอปพลิเคชันเว็บบนเครือข่ายอย่างมีประสิทธิภาพ มันมีฟีเจอร์ขั้นสูงสำหรับการค้นหาบริการเว็บ การระบุโดเมนย่อย การวิเคราะห์พอร์ต และการระบุลักษณะเฉพาะของแอปพลิเคชันเว็บ ทั้งหมดนี้ถูกนำเสนออย่างชัดเจนและกระชับในรูปแบบรายงาน HTML, JSON และข้อความ เพื่อการวิเคราะห์ความปลอดภัยเว็บที่ง่ายดาย



เช่นเดียวกับ Metasploit, Aquatone สามารถประมวลผลรูปแบบ XML ของ Nmap ได้โดยตรงและใช้เป็นเป้าหมายสำหรับการสแกน โดยเฉพาะอย่างยิ่ง มันสามารถดึงเฉพาะโฮสต์และบริการที่สนใจ (บริการเว็บ) จากข้อมูลทั้งหมดที่รายงาน Nmap อาจมีได้





- ลิงก์เครื่องมือ: [Github - Michenriksen/aquatone](https://github.com/michenriksen/aquatone)




ในการใช้เอาต์พุต XML ของ Nmap กับ Aquatone เพียงแค่ส่งไฟล์ XML ใน pipe ที่จะถูกใช้โดย Aquatone นี่คือตัวอย่าง:



```
# Send the Nmap XML output to Aquatone
cat /tmp/nmap_10.10.10.0.xml | ./aquatone -nmap
```



ที่ที่ Aquatone ปกติจะทำการค้นหาพอร์ตบนโฮสต์เพื่อค้นหาบริการเว็บ ในบริบทนี้มันจะพึ่งพาผลลัพธ์ของ Nmap เท่านั้น ซึ่งได้ทำการดำเนินการนี้ไปแล้ว จึงช่วยประหยัดเวลา:



![nmap-image](assets/fr/61.webp)



การใช้ Nmap ให้ผลลัพธ์ในรูปแบบ XML ด้วย `aquatone`._



เพื่อเป็นข้อมูลของคุณ นี่คือข้อความที่คัดมาจากรายงานที่จัดทำโดย Aquatone:



![nmap-image](assets/fr/62.webp)



ตัวอย่างของรายงาน `aquatone`



ส่วนตัวแล้ว ฉันมักใช้ Aquatone เพื่อดูภาพรวมอย่างรวดเร็วเกี่ยวกับประเภทของเว็บไซต์ที่มีอยู่ในเครือข่าย โดยเฉพาะอย่างยิ่งต้องขอบคุณฟังก์ชันการถ่ายภาพหน้าจอของมัน



ที่นี่อีกครั้ง การมีรายงาน Nmap ที่สมบูรณ์ในรูปแบบ XML ช่วยประหยัดเวลาและทำให้ง่ายต่อการนำไปใช้ซ้ำในเครื่องมืออื่น



### IV. บทสรุป



ตัวอย่างทั้งสองนี้แสดงให้เห็นอย่างชัดเจนว่า XML format ของ Nmap ทำให้เครื่องมืออื่น ๆ ใช้ผลลัพธ์ได้ง่าย เนื่องจากเป็นรูปแบบข้อมูลที่มีโครงสร้างและใช้งานง่าย มีเครื่องมืออื่น ๆ อีกมากมายที่สามารถประมวลผลผลลัพธ์เหล่านี้ได้ เช่น เครื่องมือรายงานอัตโนมัติ การแสดงผลในรูปแบบกราฟิก หรือเครื่องมือสแกนหาช่องโหว่ที่ซับซ้อนและเป็นกรรมสิทธิ์มากขึ้น



แน่นอน คุณยังสามารถพัฒนาสคริปต์และเครื่องมือของคุณเองใน Python, [PowerShell](https://www.it-connect.fr/cours-tutoriels/administration-systemes/scripting/powershell/) หรือภาษาอื่นใดที่มีไลบรารีการแยกวิเคราะห์ XML เพื่อจัดการและนำข้อมูลผลลัพธ์ Nmap กลับมาใช้ใหม่ตามที่คุณเห็นว่าเหมาะสม



ส่วนนี้นำเรามาถึงจุดสิ้นสุดของโมดูลการสอนเกี่ยวกับการใช้ Nmap ขั้นสูงมากขึ้น โดยเฉพาะสำหรับการสแกนช่องโหว่ผ่านสคริปต์ NSE



ส่วนถัดไปของบทแนะนำจะมุ่งเน้นไปที่สิ่งอื่น ๆ รวมถึงแนวทางปฏิบัติที่ดีที่สุดเพิ่มเติมที่มีความเทคนิคมากขึ้นและเคล็ดลับเกี่ยวกับการสแกนเฉพาะที่ Nmap สามารถทำได้ นอกจากนี้เรายังจะดูตัวเลือกการเพิ่มประสิทธิภาพการสแกน ซึ่งมีประโยชน์อย่างยิ่งเมื่อสแกนเครือข่ายขนาดใหญ่




## 11 - ปรับปรุงประสิทธิภาพการสแกนเครือข่าย



### I. การนำเสนอ



ในบทนี้ เราจะได้เรียนรู้วิธีการเพิ่มประสิทธิภาพความเร็วของการสแกนเครือข่ายที่ดำเนินการด้วย Nmap โดยใช้ตัวเลือกเฉพาะต่างๆ โดยเฉพาะอย่างยิ่ง เราจะได้เรียนรู้เพิ่มเติมเกี่ยวกับการทำงานภายในของ Nmap ตั้งแต่การจัดการ _timeout_ ไปจนถึงการกำหนดค่าที่บันทึกไว้ล่วงหน้าของเครื่องมือ



ตอนนี้ที่เราได้ดูคุณสมบัติของ Nmap อย่างละเอียดแล้ว มาทำความเข้าใจกับเครื่องมือนี้และพลังของมันกันเถอะ หากคุณเคยใช้เครื่องมือนี้บนเครือข่ายขนาดใหญ่ คุณอาจสังเกตเห็นว่าการสแกนบางอย่างอาจใช้เวลานาน แม้ว่าจะมีพลังของเครื่องมือก็ตาม และมีเหตุผลที่ดี: คำสั่ง `nmap` ง่ายๆ พร้อมตัวเลือกไม่กี่ตัวสามารถทำให้ generate ส่งแพ็กเก็ตนับล้านที่มุ่งเป้าไปยังระบบและบริการที่เป็นไปได้นับแสน



ยิ่งไปกว่านั้น การกำหนดค่าของอุปกรณ์เครือข่ายบางอย่างอาจตั้งใจลดอัตราความเร็ว (จำนวนแพ็กเก็ตต่อวินาที) ซึ่งอาจเสี่ยงต่อการปฏิเสธแพ็กเก็ตของคุณหรือแบนที่อยู่ IP ของคุณด้วยเหตุผลด้านความปลอดภัย



ขึ้นอยู่กับบริบท อาจคุ้มค่าที่จะลองและปรับปรุงทั้งหมดนี้ ดังที่เราจะเห็นในบทนี้



ในกรณีใด ๆ คุณสามารถตรวจสอบค่าเริ่มต้นของพารามิเตอร์ที่เรากำลังจะดู รวมถึงตัวเลือกที่คุณจะใช้ว่าถูกนำมาพิจารณาอย่างถูกต้องหรือไม่ ผ่านการดีบัก Nmap (ตัวเลือก `-d` ที่เห็นในบทก่อนหน้า):



![nmap-image](assets/fr/63.webp)



ดูตัวเลือกการตั้งเวลาได้ผ่านตัวเลือก `-d` ของ Nmap



### II. การจัดการความเร็วของการสแกน Nmap



#### A. การจัดการการประมวลผลแบบขนาน



ตามค่าเริ่มต้น Nmap ใช้การประมวลผลแบบขนานในการสแกนเพื่อเพิ่มประสิทธิภาพ และพารามิเตอร์ทั้งหมดที่ใช้สามารถแก้ไขได้ผ่านตัวเลือกต่างๆ อย่างไรก็ตาม กรณีที่จำเป็นต้องแก้ไขพารามิเตอร์เหล่านี้จริงๆ นั้นค่อนข้างหายาก ดังนั้นเราจะไม่ลงรายละเอียดในบทแนะนำนี้:





- `--min-hostgroup/max-hostgroup <size>`: ขนาดของกลุ่มสแกนโฮสต์แบบขนาน





- `--min-parallelism/max-parallelism <numprobes>`: การประมวลผลแบบขนานของ Probes.





- `--scan-delay/--max-scan-delay <time>`: ปรับการหน่วงเวลาระหว่าง Probes.




เพียงแค่รู้ว่าพวกมันมีอยู่และสามารถใช้งานได้



#### B. การจัดการจำนวนแพ็กเก็ตต่อวินาที



ตามค่าเริ่มต้น Nmap จะปรับจำนวนแพ็กเก็ตต่อวินาทีที่ส่งตามการตอบสนองของเครือข่ายโดยอัตโนมัติ แต่สามารถบังคับการตั้งค่านี้ได้โดยการกำหนดค่าต่ำสุดและ/หรือค่าสูงสุดที่ต้องการในแง่ของจำนวนแพ็กเก็ตต่อวินาที การตั้งค่านี้ทำได้โดยใช้ตัวเลือก `--min-rate <number>` และ `--max-rate <number>` โดยที่ `number` แทนจำนวนแพ็กเก็ตต่อวินาที ตัวอย่าง:



```
# Limit the scan speed to 300 packets per second
nmap -sV 10.10.10.0/24 --max-rate 300
```



ตัวเลือกเหล่านี้ช่วยให้คุณปรับความเร็วของการสแกนตามความต้องการเฉพาะของคุณ ไม่ว่าจะเพื่อเร่งกระบวนการหรือจำกัดแบนด์วิดท์ที่ใช้ กรณีหลัง (การจำกัดความเร็วของการสแกน) เป็นกรณีที่มีแนวโน้มมากที่สุดที่จะนำคุณไปสู่ตัวเลือกเหล่านี้ โดยเฉพาะอย่างยิ่งหากคุณประสบกับความล่าช้าของเครือข่ายเมื่อใช้ Nmap (ซึ่งค่อนข้างหายาก)



### III. การจัดการความล้มเหลวของการเชื่อมต่อและการหมดเวลา



อีกพารามิเตอร์หนึ่งที่เราสามารถปรับเพื่อเพิ่มความเร็วในการสแกน Nmap (หรือรับประกันความแม่นยำที่มากขึ้น) เกี่ยวข้องกับ _timeout_ และ _retry_



สำหรับ _timeouts_ นี่คือ **การหมดเวลาที่ไม่มีการตอบสนอง** หลังจากนั้น Nmap จะหยุดรอการตอบสนองและพิจารณาว่าบริการหรือโฮสต์ไม่สามารถเข้าถึงได้ สำหรับ _retry_ นี่คือ **จำนวนครั้งของความพยายามติดต่อกันในการดำเนินการ** ที่ Nmap จะทำก่อนที่จะดำเนินการต่อไป



เช่นเดียวกับการประมวลผลแบบขนาน การจัดการ _timeout_ และ _retry_ สามารถนำไปใช้กับขั้นตอนการค้นหาโฮสต์หรือบริการ:





- `--min-rtt-timeout/max-rtt-timeout/initial-rtt-timeout <time>`: ระบุเวลาการเดินทางไปกลับของการแลกเปลี่ยนข้อมูล พารามิเตอร์นี้จะถูกคำนวณและปรับเปลี่ยนโดยอัตโนมัติระหว่างการสแกนอีกครั้ง เป็นไปได้ยากที่คุณจะต้องใช้มัน เนื่องจาก Nmap คำนวณเวลานี้โดยอัตโนมัติตามการตอบสนองของเครือข่าย





- `--max-retries <number>`: จำกัดจำนวนครั้งของการส่งแพ็กเก็ตซ้ำระหว่างการสแกนพอร์ต โดยค่าเริ่มต้น Nmap สามารถทำได้สูงสุดถึง 10 ครั้งสำหรับบริการเดียว โดยเฉพาะอย่างยิ่งหากพบความล่าช้าหรือการสูญเสียในระดับเครือข่าย แต่ในกรณีส่วนใหญ่จะทำเพียงครั้งเดียวเท่านั้น





- `--host-timeout <time>`: ระบุเวลาสูงสุดที่ Nmap จะใช้กับโฮสต์สำหรับการดำเนินการทั้งหมด รวมถึงการสแกนพอร์ต การตรวจจับบริการ และการดำเนินการอื่น ๆ ที่เกี่ยวข้องกับโฮสต์นั้น หากเกินช่วงเวลานี้โดยไม่มีการตอบสนองหรือ **การดำเนินการเสร็จสิ้น** Nmap จะละทิ้งโฮสต์นี้โดยไม่แสดงผลลัพธ์ใด ๆ เกี่ยวกับมัน และจะย้ายไปยังโฮสต์ถัดไปในรายการของมัน วิธีนี้ช่วยให้คุณควบคุมเวลาสูงสุดที่ Nmap ใช้กับโฮสต์ที่กำหนด หลีกเลี่ยงการติดอยู่กับโฮสต์ที่ไม่ตอบสนองและเพิ่มประสิทธิภาพเวลาการสแกนโดยรวม




ในงานประจำวันของฉัน ฉันใช้ตัวเลือก `--max-retries` และ `--host-timeout` เพื่อเพิ่มประสิทธิภาพการสแกนของฉัน:



```
# Optimization of a scan with 0 additional attempts and a timeout of 15 minutes per host
nmap -sV -sC 10.10.10.0/24 --max-retries 0 --host-timeout 15m
```



พารามิเตอร์เหล่านี้มอบความยืดหยุ่นเพิ่มเติมสำหรับการปรับพฤติกรรมการสแกนให้ตรงกับความต้องการเฉพาะและสภาพเครือข่าย อย่างไรก็ตาม คุณจำเป็นต้องตระหนักถึงผลกระทบของพารามิเตอร์เหล่านี้ในแง่ของภาระที่เกิดขึ้นกับโฮสต์ที่ถูกสแกนและความแม่นยำที่อาจสูญเสียไป



### IV. การใช้การกำหนดค่าที่เตรียมไว้



ตัวเลือกต่างๆ ที่เราได้เห็นในบทนี้สามารถใช้ได้ทั้งแบบเดี่ยวหรือเป็นส่วนหนึ่งของการกำหนดค่าที่เตรียมไว้ล่วงหน้าโดย Nmap ตัวเลือกที่เปิดใช้งาน _templates_ (แม่แบบการกำหนดค่า) เหล่านี้คือ `-T <number>` หรือ `-T <name>` มีระดับ _templates_ ที่ใช้งานได้ 5 ระดับ:



```
-T<0-5>: Set timing template (higher is faster)
```



ตามค่าเริ่มต้น Nmap ใช้ _template_ 3 (_normal_) ซึ่งโดยทั่วไปเพียงพอแล้ว



สำหรับฉัน โดยทั่วไปแล้วฉันทำงานในบริบทที่ฉันต้องการความรวดเร็วพอสมควร (ในขณะที่ยังคงความสมบูรณ์ให้มากที่สุด) และฉันมักจะใช้ตัวเลือก `-T4` บ่อยๆ



```
# Use Nmap for a network scan with preset T4 (with debug)
nmap 10.10.10.0/24 -sV --top-ports 2000 -T4 -d
```



นี่คือข้อมูลการดีบักสำหรับการสแกนนี้ที่แสดงให้เราเห็น:



![nmap-image](assets/fr/64.webp)



การใช้การตั้งค่า `-T4` ระหว่างการสแกน Nmap



### V. บทสรุป



ในบทนี้ เราได้พิจารณาเทคนิคและตัวเลือกต่างๆ ที่คุณสามารถใช้เพื่อจัดการพลังงาน ความก้าวร้าว และประสิทธิภาพของ Nmap ตัวเลือกเหล่านี้มีประโยชน์อย่างยิ่งเมื่อทำการสแกนเครือข่ายขนาดใหญ่ และไม่ค่อยใช้เพื่อวัตถุประสงค์ในการล่องหน



ในบทถัดไป เราจะพิจารณาวิธีปฏิบัติที่ดีที่สุดบางประการในการใช้ Nmap และการรับรองความปลอดภัยของมัน




## 12 - ความปลอดภัยของข้อมูลและการรักษาความลับเมื่อใช้ Nmap



### I. การนำเสนอ



ในบทนี้ เราจะพิจารณาถึงแนวปฏิบัติที่ดีหลายประการที่ควรนำมาใช้เกี่ยวกับความปลอดภัยและเหนือสิ่งอื่นใดคือความลับของข้อมูลที่ผลิต ประมวลผล และจัดเก็บโดย Nmap



การใช้ Nmap ภายในระบบสารสนเทศสามารถจัดประเภทได้อย่างรวดเร็วว่าเป็นการกระทำเชิงรุก ดังนั้นจึงจำเป็นต้องมีการดำเนินการป้องกันหลายประการเพื่อให้สามารถดำเนินการภายในกรอบกฎหมาย ในขณะที่รับประกันความปลอดภัยของเป้าหมายที่ตั้งใจไว้ ข้อมูลที่รวบรวม และระบบที่ใช้สำหรับการสแกน



### II. การขอรับการอนุมัติที่เหมาะสม



ก่อนที่จะทำการสแกนเครือข่ายหรือระบบ ควรตรวจสอบให้แน่ใจว่าคุณได้รับการอนุญาตที่เหมาะสมแล้ว การสแกนระบบเพื่อหาช่องโหว่ (`NSE scripts`) โดยไม่ได้รับอนุญาตอาจเป็นการกระทำที่ผิดกฎหมาย และอาจมีผลทางกฎหมาย โดยเฉพาะอย่างยิ่งหากความปลอดภัยของระบบสารสนเทศไม่ใช่ส่วนหนึ่งของหน้าที่ทางการของคุณ





- เพื่อเป็นการเตือนความจำ: [ประมวลกฎหมายอาญา: บทที่ III: การโจมตีระบบประมวลผลข้อมูลอัตโนมัติ](https://www.legifrance.gouv.fr/codes/article_lc/LEGIARTI000030939438/)




### III. การปกป้องข้อมูลที่ละเอียดอ่อน



ผลลัพธ์ที่ผลิตโดย Nmap สามารถถือว่าเป็นข้อมูลที่ละเอียดอ่อน โดยเฉพาะเมื่อมีข้อมูลเกี่ยวกับจุดอ่อนในระบบสารสนเทศที่อาจถูกโจมตีโดยผู้ไม่หวังดีได้ แต่ยังรวมถึงเมื่อเกี่ยวข้องกับระบบที่ไม่สามารถเข้าถึงได้โดยทุกคน (เช่น ระบบสารสนเทศที่ละเอียดอ่อน, อุตสาหกรรม, การดูแลสุขภาพ หรือ [สำรองข้อมูล](https://www.it-connect.fr/cours-tutoriels/administration-systemes/autres/sauvegarde/)).



เราได้เห็นแล้วว่า ขึ้นอยู่กับสคริปต์ NSE ที่ใช้ ผลลัพธ์การสแกน NSE ของ [Nmap](https://www.it-connect.fr/cours/nmap-cartographie-reseau-scan-de-vulnerabilites/) อาจมีตัวระบุได้เช่นกัน



ดังนั้น บุคคลที่มีเจตนาร้ายซึ่งสามารถเข้าถึงผลการสแกนเหล่านี้ได้ จะมีแผนที่ของระบบสารสนเทศและข้อมูลทางเทคนิคจำนวนมากอยู่ในมือ โดยไม่ต้องดำเนินการเหล่านี้ด้วยตนเอง ซึ่งเสี่ยงต่อการถูกตรวจพบ



ดังนั้นจึงเป็นเรื่องสำคัญที่จะต้องระมัดระวังไม่ให้มีการเก็บรวบรวมหรือจัดเก็บข้อมูลที่ละเอียดอ่อนอย่างไม่เหมาะสมเมื่อใช้ Nmap ซึ่งรวมถึงแต่ไม่จำกัดเพียงข้อมูลต่อไปนี้:





- การเข้ารหัสข้อมูลที่ส่งออก: หากคุณจำเป็นต้องจัดเก็บหรือส่งผลลัพธ์ของการสแกน Nmap ของคุณ อย่าลืมเข้ารหัสเพื่อปกป้องความลับของข้อมูล วิธีนี้จะป้องกันการดักจับข้อมูลที่ละเอียดอ่อนโดยไม่ได้รับอนุญาต โดยอุดมคติแล้ว ข้อมูลควรถูกเข้ารหัสทันทีที่ออกจากระบบที่ใช้ในการสแกน (ไฟล์ ZIP ที่เข้ารหัสด้วยรหัสผ่านที่แข็งแกร่งเป็นการเริ่มต้นที่ดีมาก)





- ตั้งค่าการควบคุมการเข้าถึง: ตรวจสอบให้แน่ใจว่ามีเพียงผู้ที่ได้รับอนุญาตเท่านั้นที่สามารถเข้าถึงผลลัพธ์ของการสแกน Nmap ของคุณในที่ที่พวกเขาจะถูกจัดเก็บ ตั้งค่าการควบคุมการเข้าถึงที่เหมาะสมเพื่อปกป้องข้อมูลที่ละเอียดอ่อนจากการเข้าถึงโดยไม่ได้รับอนุญาต





- ความระมัดระวังเมื่อจัดการข้อมูล: เมื่อมีการส่งผ่าน คัดลอก หรือประมวลผลข้อมูลการสแกน ให้แน่ใจว่าคุณควบคุมความปลอดภัยของข้อมูลอย่างเข้มงวด ซึ่งหมายความว่า: อย่าปล่อยให้ข้อมูลอยู่ในไดเรกทอรี `Download` ของเวิร์กสเตชันที่เชื่อมต่อกับอินเทอร์เน็ต อย่าให้ข้อมูลผ่านแอปพลิเคชันแลกเปลี่ยนไฟล์ HTTP ภายในของคุณ อย่าปล่อยให้ Notepad ของคุณเปิดโดยไม่ล็อกเวิร์กสเตชันในช่วงพักกลางวัน เป็นต้น




### IV. การจัดการการสแกนที่ก้าวร้าว



ตามที่เราได้เห็นตลอดบทแนะนำนี้ Nmap สามารถมีความยาวมากในระดับเครือข่าย นอกจากนี้ยังสามารถส่งแพ็กเก็ตที่ไม่ได้รับการจัดรูปแบบอย่างถูกต้อง และไม่เคารพโครงสร้างโปรโตคอลอย่างเคร่งครัดในเฟรมและแพ็กเก็ตเครือข่ายที่มันสร้างขึ้น การกระทำทั้งหมดนี้สามารถมีผลกระทบต่อระบบและบริการบางอย่าง บางครั้งถึงขั้นทำให้เกิดการทำงานผิดปกติหรือการอิ่มตัวของทรัพยากรระบบและเครือข่าย



เพื่อหลีกเลี่ยงเหตุการณ์ใดๆ คุณจำเป็นต้องเชี่ยวชาญพฤติกรรมของ Nmap และรู้วิธีปรับให้เข้ากับบริบทที่ใช้งาน โดยใช้ตัวเลือกต่างๆ ที่กล่าวถึงในบทแนะนำนี้ เราอาจจะไม่ใช้ Nmap ในลักษณะเดียวกันบนระบบสารสนเทศที่มี [ฮาร์ดแวร์](https://www.it-connect.fr/actualites/actu-materiel/) อุตสาหกรรมเหมือนในเครือข่ายผู้ใช้ที่ประกอบด้วยระบบ Windows ที่ได้รับการป้องกันโดยไฟร์วอลล์ท้องถิ่นหรือในแกนเครือข่าย



หวังว่าบทเรียนต่างๆ ในบทแนะนำนี้จะสอนให้คุณรู้วิธีควบคุมและวิเคราะห์พฤติกรรมของ Nmap แต่การเรียนรู้ที่ดีที่สุดคือการลงมือทำ ดังนั้นให้แน่ใจว่าคุณคุ้นเคยกับตัวเลือกของ Nmap ที่คุณจะใช้



### V. การปกป้องระบบสแกน



ในบทแรก เราได้เห็นว่าในกรณีส่วนใหญ่ Nmap จำเป็นต้องรันในฐานะ `root` หรือผู้ดูแลระบบท้องถิ่น นี่เป็นเพราะมันดำเนินการเครือข่าย บางครั้งในระดับที่ค่อนข้างต่ำ ผ่านไลบรารีเครือข่าย ซึ่งต้องการสิทธิ์ที่สูงและเสี่ยงจากมุมมองของความเสถียรของระบบหรือความลับของแอปพลิเคชันอื่น ๆ



ผลลัพธ์คือ Nmap สามารถถูกมองว่าเป็นส่วนประกอบที่อ่อนไหวของระบบที่ติดตั้งอยู่ อย่าลืมใช้เวอร์ชันล่าสุดของ Nmap เนื่องจากเวอร์ชันเก่าอาจมีช่องโหว่ด้านความปลอดภัยที่ทราบอยู่แล้ว การใช้เวอร์ชันที่ทันสมัยจะช่วยลดความเสี่ยงที่เกี่ยวข้องกับการใช้เครื่องมือนี้



หากคุณเลือกใช้ Nmap โดยไม่ผ่านเซสชันในฐานะ `root` แต่โดยการให้สิทธิพิเศษเฉพาะแก่ผู้ใช้ที่มีสิทธิพิเศษเพื่อให้เขามีทุกสิ่งที่จำเป็นในการใช้ Nmap (`sudo` หรือ _capabilities_) โปรดทราบว่า Nmap สามารถใช้เป็นส่วนหนึ่งของการยกระดับสิทธิพิเศษทั้งหมด:



![nmap-image](assets/fr/65.webp)



การยกระดับสิทธิ์ของ Nmap ผ่าน `sudo`._



ที่นี่ ฉันกำลังใช้คำสั่ง Nmap ผ่าน `sudo` แต่สิ่งนี้ทำให้ฉันสามารถเข้าถึงเชลล์แบบโต้ตอบในฐานะ `root` บนระบบ ซึ่งไม่ใช่เป้าหมายเดิม



การติดตั้ง Nmap บนระบบที่ไม่ได้ออกแบบมาเพื่อทำการสแกนเครือข่ายก็ไม่แนะนำอย่างยิ่งเช่นกัน ฉันคิดถึงเซิร์ฟเวอร์หรือเวิร์กสเตชันเป็นพิเศษ ในแง่หนึ่ง สิ่งนี้จะเพิ่มช่องทางที่เป็นไปได้สำหรับการยกระดับสิทธิพิเศษ แต่เหนือสิ่งอื่นใด มันจะทำให้ผู้โจมตีเข้าถึงเครื่องมือโจมตีได้อย่างง่ายดาย



ในที่สุด ความปลอดภัยของระบบที่ใช้สำหรับการสแกนต้องได้รับการประกันในวงกว้างมากขึ้น เพื่อไม่ให้มันกลายเป็นช่องทางสำหรับการบุกรุกหรือการรั่วไหลของข้อมูลเอง ในฐานะผู้ดูแลระบบ ควรใช้ระบบเฉพาะทาง ซึ่งในอุดมคติควรมีอายุการใช้งานที่จำกัด แทนที่จะใช้สถานีงานของคุณเอง



### VI. บทสรุป



โดยสรุป ให้แน่ใจว่าคุณได้เรียนรู้ Nmap อย่างถ่องแท้ก่อนที่จะใช้งานในสภาพแวดล้อมจริงหรือในสภาวะการผลิต และควรระมัดระวังเมื่อประมวลผลและจัดการผลลัพธ์ของมัน จะเป็นเรื่องน่าเสียดายหากเกิดความเสียหาย ข้อมูลรั่วไหล หรือเกิดการประนีประนอม เมื่อวิธีการเริ่มต้นนั้นมีเป้าหมายเพื่อปรับปรุงความปลอดภัยของบริษัทของคุณ



## 13 - การสแกนพอร์ตผ่าน TCP: SYN, Connect และ FIN



### I. การนำเสนอ

แนวคิดที่นี่คือการอธิบายวิธีการบางอย่างเหล่านี้ เพื่อช่วยให้คุณเข้าใจถึงความแตกต่าง ข้อดี และข้อจำกัดของพวกมัน คุณจะเห็นว่า ขึ้นอยู่กับบริบทหรือสิ่งที่คุณต้องการทราบ การเลือกใช้ตัวเลือกใดตัวเลือกหนึ่งอาจดีกว่า



### II. การสแกน TCP SYN หรือ "การสแกนแบบ Half Open"



ประเภทแรกของการสแกน TCP ที่เราจะพิจารณาคือ `TCP SYN Scan` หรือที่รู้จักกันในชื่อ `Half Open Scan` หากคุณจำการสแกนเครือข่ายที่เราทำหลังจากการสแกนพอร์ตครั้งแรก นี่คือประเภทของการสแกนที่ใช้โดยค่าเริ่มต้นโดย [Nmap](https://www.it-connect.fr/cours/nmap-cartographie-reseau-scan-de-vulnerabilites/) เมื่อรันด้วยสิทธิ์ root



การแปลนี้จะช่วยให้คุณเข้าใจว่าการสแกนนี้ทำงานอย่างไร ในความเป็นจริง การสแกน TCP SYN จะส่งแพ็กเก็ต TCP SYN ไปยังพอร์ตที่ถูกกำหนดเป้าหมายแต่ละพอร์ต ซึ่งเป็นแพ็กเก็ตแรกที่ส่งโดยไคลเอนต์ (ผู้ที่ร้องขอการเชื่อมต่อ) เป็นส่วนหนึ่งของ _Three way handshake_ TCP ที่มีชื่อเสียง โดยปกติแล้ว หากพอร์ตเปิดอยู่บนเซิร์ฟเวอร์เป้าหมาย โดยมีบริการทำงานอยู่เบื้องหลัง เซิร์ฟเวอร์จะส่งแพ็กเก็ต TCP SYN/ACK กลับมาเพื่อยืนยัน SYN ของไคลเอนต์และเริ่มต้นการเชื่อมต่อ TCP การตอบสนองนี้จะอยู่ในรูปแบบของแพ็กเก็ต TCP ที่มีการตั้งค่าสถานะ SYN และ ACK เป็น 1 ทำให้เราสามารถยืนยันได้ว่าพอร์ตเปิดและนำไปสู่บริการ



ในทางกลับกัน หากพอร์ตถูกปิด เซิร์ฟเวอร์จะส่งแพ็กเก็ต `TCP` ที่มีการตั้งค่าสถานะ RST และ ACK เป็น 1 เพื่อยกเลิกคำขอการเชื่อมต่อ ดังนั้นเราจะรู้ว่าไม่มีบริการใดที่ดูเหมือนจะทำงานอยู่หลังพอร์ตนี้:



![nmap-image](assets/fr/67.webp)



แผนภาพพฤติกรรมการสแกน tCP SYN สำหรับพอร์ตที่เปิดและปิด



เพื่อให้ได้มุมมองที่ชัดเจนยิ่งขึ้นเกี่ยวกับ `TCP SYN Scan` ฉันได้ทำการสแกนพอร์ต TCP/80 ไปยังโฮสต์ที่มีเว็บเซิร์ฟเวอร์ที่ใช้งานอยู่บนพอร์ตนี้ การรันการสแกนเครือข่ายด้วย Wireshark เราสามารถเห็นการไหลของข้อมูลดังต่อไปนี้ (แหล่งที่มาของการสแกน: `10.10.14.84`):

การจับภาพเครือข่ายระหว่างการสแกน TCP SYN สำหรับพอร์ตที่เปิดอยู่



ในบรรทัดแรก เราเห็นว่าแหล่งที่มาของการสแกนกำลังส่งแพ็กเก็ต TCP ไปยังโฮสต์ `10.10.10.203` บนพอร์ต TCP/80 ในแพ็กเก็ต TCP นี้ ธง SYN ถูกตั้งค่าเป็น 1 เพื่อระบุว่านี่คือคำขอเริ่มต้นการเชื่อมต่อ TCP



จากนั้น ในบรรทัดที่สอง เราจะเห็นว่าเป้าหมายตอบกลับด้วย `TCP SYN/ACK` ซึ่งหมายความว่ามันยอมรับที่จะเริ่มต้นการเชื่อมต่อและดังนั้นจึงยอมรับการรับสตรีมบนพอร์ต TCP/80 เราจึงสามารถสรุปได้ว่าพอร์ต TCP/80 เปิดอยู่และมีเว็บเซิร์ฟเวอร์อยู่บนเซิร์ฟเวอร์ที่ถูกสแกน



โฮสต์ของเราจะส่งแพ็กเก็ต RST กลับไปเพื่อปิดการเชื่อมต่อ ทำให้โฮสต์ที่ถูกสแกนไม่ต้องรักษาการเชื่อมต่อ TCP ที่เปิดรอการตอบกลับ ในกรณีของการสแกนหลายพอร์ต การไม่ปิดการเชื่อมต่อ TCP อาจนำไปสู่การปฏิเสธการให้บริการ โดยทำให้จำนวนการเชื่อมต่อที่รอการตอบกลับที่เซิร์ฟเวอร์สามารถรักษาไว้เต็ม (ดู [Wikipedia - Syn flood](https://fr.wikipedia.org/wiki/SYN_flood))



ใน Wireshark คุณจะสามารถดูสถานะของ TCP flags สำหรับการทดสอบแต่ละครั้งที่เราดำเนินการได้ ซึ่งจะแสดงว่าข้อมูลแพ็กเก็ตนั้นเป็นแพ็กเก็ต SYN, SYN/ACK, ACK, ฯลฯ:



![nmap-image](assets/fr/69.webp)

ในทางกลับกัน ฉันได้ทำการทดสอบเดียวกันระหว่างเครื่องทั้งสอง แต่คราวนี้สแกนพอร์ต TCP/81 ซึ่งไม่มีบริการใดๆ ทำงานอยู่ (แหล่งที่มาของการสแกน: `10.10.14.84`):



![nmap-image](assets/fr/70.webp)

โฮสต์ที่สแกนส่งกลับ `TCP RST/ACK` เพื่อตอบสนองต่อ `TCP SYN` ของฉันเมื่อพอร์ตไม่เปิด



ตามที่กล่าวไว้ เมื่อรัน Nmap จากเทอร์มินัลที่มีสิทธิพิเศษ โหมดเริ่มต้นจะเป็น TCP SYN Scan และสามารถบังคับใช้ได้ผ่านตัวเลือก `-sS` (`scan SYN`):



```
# Execution of a TCP Syn Scan_
nmap -sS 192.168.1.15
```




การสแกน `TCP SYN Scan` เป็นการสแกนที่ใช้บ่อยที่สุดเนื่องจากความเร็ว ในทางกลับกัน การที่ไคลเอนต์ไม่สามารถทำ _Three Way Handshake_ ให้เสร็จสมบูรณ์ (เช่น ไม่ส่ง ACK หลังจากที่เซิร์ฟเวอร์ส่ง SYN/ACK) อาจดูน่าสงสัยหากมีการสังเกตเห็นบ่อยเกินไปบนเซิร์ฟเวอร์หรือจากแหล่งเดียวกันในเครือข่าย อันที่จริง พฤติกรรมปกติของไคลเอนต์หลังจากได้รับแพ็กเก็ต TCP SYN/ACK เพื่อตอบสนองต่อ TCP SYN คือการส่ง `acknowledgement` (ACK) และเริ่มการแลกเปลี่ยนข้อมูล



อย่างไรก็ตาม มันให้การสแกนที่เร็วขึ้นเล็กน้อย เนื่องจากไม่ต้องส่ง ACK สำหรับการตอบสนองที่เป็นบวกแต่ละครั้ง ข้อดีของ SYN Scan คือความเร็ว เนื่องจากส่งเพียงแพ็กเก็ตเดียวต่อพอร์ตที่จะสแกน โดยแลกกับโอกาสในการถูกตรวจจับที่มากขึ้น



นอกจากนี้ TCP SYN scan ยังสามารถตรวจจับได้ว่าพอร์ตถูกกรอง (ป้องกัน) โดยไฟร์วอลล์หรือไม่ ในความเป็นจริง ไฟร์วอลล์ที่อยู่หน้าฮอสต์เป้าหมายสามารถตรวจจับได้จากวิธีที่มันตอบสนองเมื่อได้รับแพ็กเก็ต TCP SYN บนพอร์ตที่มันควรจะป้องกัน มันจะไม่ตอบสนอง อย่างไรก็ตาม ตามที่เราได้เห็น ในทั้งสองกรณี (พอร์ตเปิดหรือปิด) จะมีการตอบสนองจากฮอสต์ พฤติกรรมที่สามนี้จะเผยให้เห็นถึงการมีอยู่ของไฟร์วอลล์ระหว่างฮอสต์ที่ถูกสแกนและเครื่องที่ทำการสแกน นี่คือผลลัพธ์ที่ Nmap สามารถคืนค่าได้เมื่อพอร์ตที่ถูกสแกนถูกกรองโดยไฟร์วอลล์:



![nmap-image](assets/fr/71.webp)



nmap แสดงผลเมื่อสแกนพอร์ตที่ถูกกรอง



เมื่อเราทำการจับภาพเครือข่ายในขณะที่สแกน เราจะเห็นได้จริงว่าไม่มีการตอบสนอง:



![nmap-image](assets/fr/72.webp)



การจับภาพเครือข่ายระหว่างการสแกน TCP SYN สำหรับพอร์ตที่ถูกกรองโดยไฟร์วอลล์



ความแตกต่างระหว่างพอร์ตที่ปิดและพอร์ตที่ถูกกรองมีดังนี้: พอร์ตที่ถูกกรองคือพอร์ตที่ได้รับการป้องกันโดยไฟร์วอลล์ ในขณะที่พอร์ตที่ปิดคือพอร์ตที่ไม่มีบริการใดๆ ทำงานอยู่และไม่สามารถประมวลผลแพ็กเก็ต TCP ของเราได้ การสแกน TCP บางประเภท เช่น การสแกน TCP SYN สามารถตรวจจับได้ว่าพอร์ตถูกกรองหรือไม่ ในขณะที่การสแกนประเภทอื่นไม่สามารถทำได้



### III. การสแกน TCP Connect หรือการสแกน Full Open



ประเภทที่สองของการสแกน TCP คือ `TCP Connect scan` หรือที่รู้จักกันในชื่อ _Full Open Scan_ มันทำงานในลักษณะเดียวกับการสแกน TCP SYN แต่ครั้งนี้จะส่งคืน `TCP ACK` หลังจากได้รับการตอบสนองเชิงบวกจากเซิร์ฟเวอร์ (SYN/ACK) นี่คือเหตุผลที่เรียกว่า `Full Open` เนื่องจากการเชื่อมต่อถูกเปิดและเริ่มต้นอย่างเต็มที่ในทุกพอร์ตที่เปิดระหว่างการสแกน ดังนั้นจึงเคารพ TCP _Three Way Handshake_:



![nmap-image](assets/fr/73.webp)



แผนภาพพฤติกรรมการสแกนการเชื่อมต่อ tCP สำหรับพอร์ตที่เปิดและปิด



นี่คือสิ่งที่สามารถเห็นได้เมื่อมีการส่งข้อมูลผ่านเครือข่ายระหว่างการ `TCP Connect scan` ที่มุ่งเป้าไปยังพอร์ตที่เปิดอยู่:



![nmap-image](assets/fr/74.webp)



การดักจับข้อมูลเครือข่ายระหว่างการสแกน TCP Connect เพื่อหาพอร์ตที่เปิดอยู่



เราจะเห็นว่าแพ็กเก็ต TCP แรกที่ส่งคือ `TCP SYN` ที่ส่งโดยไคลเอนต์ และเซิร์ฟเวอร์จะตอบกลับด้วย `TCP SYN/ACK` ซึ่งบ่งบอกว่าพอร์ตเปิดและมีบริการที่ใช้งานอยู่ เพื่อจำลองไคลเอนต์ที่ถูกต้อง Nmap จะส่ง `TCP ACK` กลับไปยังเซิร์ฟเวอร์ ในทางกลับกัน เมื่อสแกนพอร์ตที่ปิด:



![nmap-image](assets/fr/75.webp)



การจับภาพเครือข่ายระหว่างการสแกน TCP Connect สำหรับพอร์ตที่ปิด



โปรดทราบว่าการตอบสนองของเซิร์ฟเวอร์ต่อแพ็กเก็ต `SYN` ของเราเป็นแพ็กเก็ต `TCP RST/ACK` อีกครั้ง ดังนั้นเราสามารถสรุปได้ว่าพอร์ตถูกปิดและไม่มีบริการใด ๆ ทำงานอยู่บนพอร์ตนั้น



เมื่อใช้ Nmap ตัวเลือก `-sT` (`scan Connect`) จะถูกใช้เพื่อทำการสแกน TCP Connect โปรดทราบว่าเมื่อใช้ Nmap จากเซสชันที่ไม่มีสิทธิพิเศษ นี่คือโหมดการสแกน TCP เริ่มต้น:



```
# Execution of a TCP Connect Scan
nmap -sT 192.168.1.15
```



`TCP Connect Scan` จำลองคำขอการเชื่อมต่อที่ถูกต้องมากขึ้น โดยมีพฤติกรรมที่คล้ายกับไคลเอนต์ lambda มากที่สุด ดังนั้นจึงยากที่จะสังเกตเห็นการสแกนบนจำนวนพอร์ตที่ลดลง อย่างไรก็ตาม มันช้ากว่าเนื่องจากจะเริ่มต้นการเชื่อมต่อ TCP ทุกครั้งบนพอร์ตที่เปิดของเครื่องที่ถูกสแกนอย่างสมบูรณ์



การสแกน Nmap ของพอร์ต 10,000 พอร์ตจะยังคงสามารถตรวจจับได้ง่ายหากมีการติดตั้งบริการตรวจจับและป้องกันการบุกรุกเครือข่าย (IDS, IPS, EDR) เมื่อผู้โจมตีต้องการรักษาโปรไฟล์ต่ำ เขามักจะมุ่งเน้นไปที่พอร์ตจำนวนเล็กน้อยที่เลือกอย่างมีกลยุทธ์ เช่น 445 (SMB) หรือ 80 (HTTP) ซึ่งมักจะเปิดบนเซิร์ฟเวอร์และมีช่องโหว่ทั่วไป



เนื่องจาก TCP Connect Scan คาดหวังการตอบสนองในทั้งสองกรณี มันจึงสามารถตรวจจับการมีอยู่ของไฟร์วอลล์ที่อาจกรองพอร์ตบนโฮสต์เป้าหมายได้เช่นกัน



### IV. การสแกน TCP FIN หรือ "Stealth Scan



การสแกน `TCP FIN Scan` หรือที่รู้จักกันในชื่อ _Stealth Scan_ ใช้พฤติกรรมของไคลเอนต์ที่ยุติการเชื่อมต่อ TCP เพื่อตรวจจับพอร์ตที่เปิดอยู่



ใน TCP การสิ้นสุดเซสชันหมายถึงการส่งแพ็กเก็ต TCP ที่มีการตั้งค่า FIN flag เป็น 1 ในการแลกเปลี่ยนตามปกติ เซิร์ฟเวอร์จะหยุดการสื่อสารทั้งหมดกับไคลเอนต์ (ไม่มีการตอบสนอง) หากเซิร์ฟเวอร์ไม่มีการเชื่อมต่อ TCP ที่ใช้งานกับไคลเอนต์ มันจะส่ง `RST/ACK` ดังนั้นเราจึงสามารถแยกแยะระหว่างพอร์ตที่เปิดและปิดได้โดยการส่งแพ็กเก็ต `TCP FIN` ไปยังชุดของพอร์ต:



![nmap-image](assets/fr/76.webp)



แผนภาพพฤติกรรมการสแกน tCP FIN สำหรับพอร์ตที่เปิดและปิด



ฉันได้ทำการจับภาพเครือข่ายอีกครั้งระหว่างการ _Stealth scan_ และนี่คือสิ่งที่คุณจะเห็นเมื่อพอร์ตที่สแกนเปิดอยู่:



![nmap-image](assets/fr/77.webp)



การจับภาพเครือข่ายระหว่างการสแกน TCP FIN สำหรับพอร์ตที่เปิดอยู่



เราสามารถเห็นได้ว่าลูกค้าส่งแพ็กเก็ตหนึ่งหรือสองแพ็กเก็ตเพื่อยุติการเชื่อมต่อ TCP และเซิร์ฟเวอร์ไม่ตอบสนอง มันเพียงแค่ยอมรับการสิ้นสุดของการเชื่อมต่อและหยุดการสื่อสาร



นี่คือสิ่งที่เราเห็นเมื่อเราสแกนพอร์ตที่ปิด:



![nmap-image](assets/fr/78.webp)



การจับภาพเครือข่ายระหว่างการสแกน TCP FIN สำหรับพอร์ตที่ปิด



เราเห็นว่าเซิร์ฟเวอร์ส่งแพ็กเก็ต `TCP RST/ACK` กลับมา ดังนั้นจึงมีความแตกต่างในพฤติกรรมระหว่างพอร์ตที่เปิดและปิด และเราสามารถแสดงรายการพอร์ตที่เปิดบนเซิร์ฟเวอร์ได้โดยการส่งแพ็กเก็ต TCP FIN โดยใช้ Nmap ตัวเลือก `-sF` (`scan FIN`) ถูกใช้เพื่อทำการสแกน TCP FIN:



```
# Execution of a TCP Fin Scan
nmap -sF 192.168.1.15
```



การสแกน TCP FIN ไม่ทำงานบนโฮสต์ Windows เนื่องจากระบบปฏิบัติการมักจะเพิกเฉยต่อแพ็กเก็ต TCP FIN เมื่อถูกส่งไปยังพอร์ตที่ไม่ได้เปิด ดังนั้นหากคุณทำการสแกน TCP FIN บนโฮสต์ Windows คุณจะได้รับความรู้สึกว่าพอร์ตทั้งหมดถูกปิด



นั่นคือเหตุผลที่ว่าทำไมการทำความคุ้นเคยกับวิธีการสแกนหลายๆ แบบจึงมีความสำคัญ และทำความเข้าใจถึงความแตกต่างระหว่างพวกมัน



เนื่องจากในทั้งสองกรณี TCP FIN จะไม่รอการตอบกลับ จึงไม่สามารถตรวจจับการมีอยู่ของไฟร์วอลล์ระหว่างโฮสต์เป้าหมายและแหล่งที่มาของการสแกนได้

![nmap-image](assets/fr/79.webp)

ในความเป็นจริง การไม่ตอบสนองจากโฮสต์บนพอร์ตที่กำหนดสามารถหมายความว่าพอร์ตนั้นถูกกรอง แต่ก็อาจหมายความว่าพอร์ตนั้นเปิดและใช้งานอยู่เช่นกัน



การสแกนนี้เรียกว่า "stealth" เนื่องจากมันไม่สร้างทราฟฟิก generate มากนักและโดยทั่วไปไม่ทำให้เกิดการบันทึกในระบบที่เป็นเป้าหมาย มันสามารถใช้เพื่อค้นหาพอร์ตในเครือข่ายอย่างเงียบๆ โดยไม่ทำให้เกิดการเตือน อย่างไรก็ตาม ดังที่กล่าวไว้ข้างต้น ประสิทธิภาพของมันอาจแตกต่างกันไปขึ้นอยู่กับระบบเป้าหมาย เช่นเดียวกับความรอบคอบของมันที่อาจแตกต่างกันไปขึ้นอยู่กับการกำหนดค่าของอุปกรณ์รักษาความปลอดภัย

สำหรับบทแรกจากสองบทเกี่ยวกับประเภทการสแกน TCP ที่แตกต่างกันที่เสนอโดย Nmap! ในบทถัดไป เราจะพิจารณาประเภทการสแกน TCP แบบ XMAS, Null และ ACK ซึ่งทำงานในรูปแบบที่แตกต่างกันเพื่อค้นหาพอร์ตที่เปิดอยู่บนโฮสต์





## 14 - การสแกนพอร์ตผ่าน TCP: XMAS, Null และ ACK



### I. การนำเสนอ



ในส่วนนี้ เราจะสำรวจวิธีการสแกน TCP ต่างๆ ที่เสนอโดย Nmap ต่อไป ที่นี่เราจะดูวิธีการ `XMAS`, `Null` และ `ACK` ซึ่งใช้คุณสมบัติเฉพาะของ TCP เพื่อดึงข้อมูลเกี่ยวกับพอร์ตและบริการที่เปิดอยู่บนเป้าหมายที่กำหนด



### II. TCP XMAS scan



XMAS Scan TCP ค่อนข้างแปลกตรงที่มันไม่ได้จำลองพฤติกรรมปกติของผู้ใช้หรือเครื่องบนเครือข่ายเลย ในความเป็นจริง XMAS Scan จะส่ง TCP packets ที่มีธง `URG` (Urgent), `PSH` (Push), และ `FIN` (Finish) ตั้งค่าเป็น 1 เพื่อหลีกเลี่ยงไฟร์วอลล์หรือกลไกการกรองบางอย่าง



ชื่อ XMAS มาจากความจริงที่ว่าการเห็นธงเหล่านี้เปิดอยู่นั้นเป็นเรื่องที่ไม่ปกติ เมื่อธงทั้งสามถูกตั้งพร้อมกันในแพ็กเก็ต TCP มันดูเหมือนต้นคริสต์มาสที่มีไฟสว่าง:



![nmap-image](assets/fr/80.webp)



ธง tCP ที่ใช้ใน XMAS scan



โดยไม่ต้องลงรายละเอียดเกี่ยวกับบทบาทของธงเหล่านี้ที่นี่ สิ่งสำคัญคือต้องรู้ว่าเมื่อส่งแพ็กเก็ตที่มีธงสามตัวนี้เปิดใช้งาน บริการที่ทำงานอยู่เบื้องหลังพอร์ตเป้าหมายจะไม่ส่งคืนแพ็กเก็ตใด ๆ ในทางกลับกัน หากพอร์ตถูกปิด เราจะได้รับแพ็กเก็ต TCP RST/ACK เราจะสามารถแยกแยะพฤติกรรมระหว่างพอร์ตที่เปิดและปิดเมื่อทำการลิสต์พอร์ตบนเครื่อง:



![nmap-image](assets/fr/81.webp)



แผนภาพพฤติกรรมการสแกน tCP XMAS สำหรับพอร์ตที่เปิดและปิด



ยังคงตามตรรกะเดิม การสแกนเครือข่ายบนพอร์ต TCP/80 ของโฮสต์ที่มีเว็บเซิร์ฟเวอร์ที่ใช้งานอยู่แสดงพฤติกรรมดังต่อไปนี้เมื่อพบพอร์ตที่เปิดอยู่ (แหล่งที่มาของการสแกน `10.10.14.84`):



![nmap-image](assets/fr/82.webp)



การจับภาพเครือข่ายระหว่างการสแกน TCP XMAS สำหรับพอร์ตที่เปิดอยู่



เราสามารถเห็นได้ว่าแหล่งสแกนส่งแพ็กเก็ต TCP XMAS สองแพ็กเก็ต (โดยมีธง `FIN`, `PSH` และ `URG` ตั้งค่าเป็น 1) ไปยังโฮสต์ `10.10.10.203` และไม่มีการตอบกลับจากเป้าหมาย ซึ่งบ่งชี้ว่าพอร์ตเปิด ในทางกลับกัน เมื่อทำการ `TCP XMAS Scan` บนพอร์ตที่ปิด จะสังเกตเห็นผลลัพธ์ดังต่อไปนี้:



![nmap-image](assets/fr/83.webp)



การจับภาพเครือข่ายระหว่างการสแกน TCP XMAS สำหรับพอร์ตที่ปิด



การตอบสนองต่อแพ็กเก็ต TCP ของเราคือ `TCP RST/ACK` ซึ่งบ่งชี้ว่าพอร์ตถูกปิดอยู่ ในการใช้เทคนิคนี้กับ Nmap ตัวเลือก `-sX` (`scan XMAS`) ช่วยให้คุณสามารถทำการสแกน TCP XMAS ได้:



```bash
# Execution of a TCP XMAS Scan
nmap -sX 192.168.1.15
```



สิ่งสำคัญที่ควรทราบคือการสแกน TCP XMAS ไม่สามารถตรวจจับไฟร์วอลล์ที่อาจอยู่ระหว่างเป้าหมายและเครื่องสแกนได้ ต่างจากการสแกนประเภทอื่น ๆ เช่น TCP SYN หรือ Connect อันที่จริง ไฟร์วอลล์ที่ทำงานอยู่ระหว่างโฮสต์ทั้งสองจะทำให้แน่ใจว่าไม่มีการส่งคืน TCP หากพอร์ตที่ถูกกำหนดเป้าหมายถูกกรอง (เช่น ได้รับการป้องกันโดยไฟร์วอลล์) ในกรณีที่ไม่มีการตอบสนอง จึงเป็นไปไม่ได้ที่จะทราบว่าพอร์ตนั้นได้รับการป้องกันโดยไฟร์วอลล์หรือเปิดและทำงานอยู่ คุณควรทราบด้วยว่า เช่นเดียวกับการสแกน TCP FIN แอปพลิเคชันหรือระบบปฏิบัติการบางอย่าง เช่น Windows สามารถบิดเบือนผลลัพธ์ของการสแกนประเภทนี้ได้



หมายเหตุ: การสนับสนุนสำหรับการสแกน XMAS/FIN/NULL บนเวอร์ชันล่าสุดของ Windows ยังคงมีจำกัด และผลลัพธ์อาจไม่สอดคล้องกันบนเป้าหมายประเภทนี้ (อัปเดต 2025)_



### III. TCP Null scan



ตรงกันข้ามกับการสแกน TCP XMAS การสแกน TCP Null จะส่งแพ็กเก็ตสแกน TCP โดยที่ไม่มีการตั้งค่าสถานะใดๆ เลย นี่ก็เป็นพฤติกรรมที่ไม่เคยพบในการแลกเปลี่ยนข้อมูลปกติระหว่างเครื่อง เนื่องจากการส่งแพ็กเก็ต TCP โดยไม่มีสถานะไม่ได้ถูกระบุใน RFC ที่อธิบายโปรโตคอล TCP นี่คือเหตุผลที่สามารถตรวจจับได้ง่ายกว่า



เช่นเดียวกับการสแกน TCP XMAS การสแกนนี้สามารถรบกวนไฟร์วอลล์หรือโมดูลการกรองบางอย่าง ทำให้แพ็กเก็ตสามารถผ่านได้:



![nmap-image](assets/fr/84.webp)



แผนภาพพฤติกรรมการสแกน tCP Null สำหรับพอร์ตที่เปิดและปิด



นี่คือสิ่งที่สามารถเห็นได้บนเครือข่ายระหว่างการสแกน TCP Null บนพอร์ตที่เปิด:



![nmap-image](assets/fr/85.webp)



การจับภาพเครือข่ายระหว่างการสแกน TCP Null สำหรับพอร์ตที่เปิดอยู่



เครื่องสแกนส่งแพ็กเก็ตที่ไม่มีธง (`[<None>]` ใน Wireshark) โดยไม่มีการตอบสนองจากเซิร์ฟเวอร์ ในทางกลับกัน เมื่อพอร์ตเป้าหมายถูกปิด:



![nmap-image](assets/fr/86.webp)



การจับภาพเครือข่ายระหว่างการสแกน TCP Null สำหรับพอร์ตที่ปิด



ในการทำการสแกน TCP Null ด้วย Nmap เพียงแค่ใช้ตัวเลือก `-sN` (`scan Null`):



```bash
# Execution of a TCP Null Scan
nmap -sN 192.168.1.15
```



เนื่องจากการตอบสนองเมื่อพอร์ตเปิดและเมื่อไฟร์วอลล์ทำงาน (ไม่มีการตอบกลับจากเซิร์ฟเวอร์ในทั้งสองกรณี) เหมือนกัน การสแกน TCP Null จึงไม่สามารถตรวจจับการมีอยู่ของไฟร์วอลล์ได้ ยิ่งไปกว่านั้น ไฟร์วอลล์อาจทำให้ผลลัพธ์ผิดพลาดโดยแนะนำว่าพอร์ตเปิด เนื่องจากมันไม่ตอบสนองต่อแพ็กเก็ต TCP ที่ไม่มีธง ถึงแม้ว่าพอร์ตจะถูกกรองก็ตาม นี่เป็นข้อมูลสำคัญที่ควรทราบเมื่อใช้การสแกนที่ไม่สามารถแยกแยะระหว่างพอร์ตที่เปิดและพอร์ตที่ถูกกรอง (ได้รับการป้องกันโดยไฟร์วอลล์) เช่น การสแกน `TCP Null`, `XMAS` หรือ `FIN` เพื่อให้การตีความผลลัพธ์ที่ได้รับมีความสอดคล้องกัน



### IV. TCP ACK scan



การสแกน TCP ACK ใช้เพื่อตรวจจับการมีอยู่ของไฟร์วอลล์บนโฮสต์เป้าหมายหรือระหว่างเป้าหมายและแหล่งที่มาของการสแกน



ไม่เหมือนกับการสแกนอื่น ๆ การสแกน TCP ACK ไม่ได้พยายามระบุว่าพอร์ตใดเปิดอยู่บนโฮสต์ แต่จะตรวจสอบว่าระบบกรองทำงานอยู่หรือไม่ โดยตอบสนองสำหรับแต่ละพอร์ตด้วย `filtered` หรือ `unfiltered` การสแกน TCP บางประเภท เช่น `TCP SYN` หรือ `TCP Connect` สามารถทำทั้งสองอย่างได้ในเวลาเดียวกัน ในขณะที่การสแกนอื่น ๆ เช่น `TCP FIN` หรือ `TCP XMAS` ไม่สามารถระบุการมีอยู่ของการกรองได้เลย นี่คือเหตุผลที่การสแกน TCP ACK สามารถมีประโยชน์:



![nmap-image](assets/fr/87.webp)



แผนภาพพฤติกรรมการสแกน tCP ACK สำหรับพอร์ตที่ถูกกรองและไม่ได้กรอง



เราจะใช้ตัวเลือก `-sA` ของ Nmap เพื่อทำการสแกนประเภทนี้ นี่คือผลลัพธ์ของการสแกน TCP ACK หากพอร์ตถูกกรอง คือ ถูกบล็อกและป้องกันโดยไฟร์วอลล์:



![nmap-image](assets/fr/88.webp)



nmap แสดงผลระหว่างการสแกน TCP ACK._



ตัวอย่างผลลัพธ์สำหรับโฮสต์ที่มีไฟร์วอลล์และไม่มีไฟร์วอลล์ Nmap ส่งคืน `filtered` บนพอร์ต TCP/80 และ TCP/81 ของโฮสต์ `10.10.10.203` ในการวิเคราะห์เครือข่ายผ่าน Wireshark พฤติกรรมเป็นดังนี้:



![nmap-image](assets/fr/89.webp)

เครื่องเป้าหมายจะไม่ส่งคืนอะไรเลยหากมีไฟร์วอลล์อยู่



ในการเริ่มการสแกนนี้ด้วย Nmap ให้ใช้ตัวเลือก `-sA` (`scan ACK`):



```bash
# Execution of a TCP ACK Scan
nmap -sA 192.168.1.15
```

เราได้พิจารณาวิธีการสแกนผ่าน TCP สามวิธีที่แตกต่างกันเพิ่มเติมจากที่ได้นำเสนอไปแล้ว วิธีการที่แตกต่างกันเหล่านี้จะถูกใช้ในเงื่อนไขและบริบทที่เฉพาะเจาะจงมาก โดยเฉพาะในบริบทของการทดสอบการเจาะระบบหรือปฏิบัติการ Red Team ซึ่งมีแนวคิดเรื่องความรอบคอบเข้ามาเกี่ยวข้อง



## 15 - Nmap CheatSheet - สรุปคำสั่งการสอน



### I. การนำเสนอ



นี่คือสรุปสั้น ๆ ของคำสั่งและกรณีการใช้งานของ Nmap ที่หลากหลาย เพื่อให้คุณสามารถค้นหาและนำไปใช้ซ้ำในชีวิตประจำวันได้อย่างรวดเร็ว



### II. Nmap: CheatSheet IT-Connect



นี่คือชีทชีทของคำสั่งที่นำเสนอ หน้านี้ทำให้ง่ายต่อการค้นหาการใช้งานทั่วไปของ Nmap มากที่สุด





- การสแกนพอร์ต




```bash
# Display installed Nmap version
nmap --version

# Scan for open specific ports on a single IP address
nmap --open -p 80 192.168.1.18

# Scan TCP ports on a selection of ports
nmap 192.168.1.19 -p 80
nmap 192.168.1.19 -p 22,80,1000-2000,3389

# Scan UDP services on an IP address
nmap -sU 192.168.1.19

# Scan UDP services on specific ports
nmap -sU 192.168.1.19 -p 161,23

# Scan the 100 most commonly used TCP ports
nmap 192.168.1.19 --top-ports 100

# Scan all ports on an IP address
nmap 192.168.1.19 -p-

# Scan multiple subnets with specific ports
nmap 192.168.0.0/24 192.168.1.0/24 -p 22

# Scan a subnet while excluding a specific IP address, scan all ports
nmap 192.168.0.0/24 --exclude 192.168.0.4 -p-
```





- การค้นหาโฮสต์ที่ใช้งานอยู่




```bash
# Scan on CIDR or IP ranges
nmap 192.168.0.0/24
nmap 192.168.0.0/24 192.168.1.0/24
nmap 192.168.1.2 192.168.1.3 192.168.1.10-20
nmap 192.168.0.0/24 192.168.1.3 192.168.1.10-20

# Host discovery scan (Ping Scan) on a network
nmap -sn 192.168.1.0/24
```



หมายเหตุ: ตัวเลือก `-sP` ได้ล้าสมัยไปหลายปีแล้วและควรเปลี่ยนเป็น `-sn` (อัปเดต 2025)_



```bash
# Host discovery scan without port scan
nmap 192.168.1.0/24 -sn

# Host discovery scan on a local network using various probe techniques
nmap -sn -PP -PS22,3389,445,139 -PA80 192.168.1.0/24

# Scan hosts listed in a text file
nmap -iL /tmp/mesCibles.txt

# Network scan with a specific IP excluded
nmap 192.168.1.0/24 --exclude 192.168.1.140

# Subnet scan excluding another subnet
nmap 10.0.0.0/16 --exclude 10.0.100.0/24
```





- การตรวจจับเวอร์ชัน




```bash
# Enable version detection
nmap 192.168.1.0/24 -sV

# Version detection + advanced trace
nmap 192.168.1.0/24 -sV --version-trace

# Version detection with maximum probe rarity of 2
nmap 192.168.1.0/24 -sV --version-intensity 2

# Version detection with all probes
nmap 192.168.1.0/24 -sV --version-intensity 9

# Enable OS detection
nmap -O 10.10.10.0/24
```





- สคริปต์ NSE: กำลังมองหาช่องโหว่




```bash
# Run default NSE scripts
nmap -sC 10.10.10.152

# Scan all TCP ports with default NSE scripts
nmap -sC -p- 10.10.10.152

# Display help for script by name
nmap --script-help=ftp-*

# Display help for a category
nmap --script-help=discovery

# Use NSE scripts by category
nmap --script default 10.10.10.152
nmap --script discovery 10.10.10.152

# Inclusion/exclusion filter for NSE script categories
nmap --script "discovery and safe" 10.10.10.152
nmap --script "not intrusive and not dos" 10.10.10.152
nmap --script "vuln and not intrusive and not dos" 10.10.10.152
nmap --script "(http and vuln) and not intrusive and not dos" 10.10.10.152

# Run a specific NSE script
nmap --script ftp-anon -p 21 10.10.10.152
nmap --script ftp-anon 10.10.10.152

# Pass arguments to an NSE script
nmap --script ssh-brute --script-args userdb=/tmp/userlist,passdb=/tmp/passlist 10.10.10.245 192.168.1.19
```





- การจัดการเอาต์พุต Nmap




```bash
# Save scan to text file
nmap 10.10.10.0/24 -sV -oN nmap_scan_10.10.10.0_24.txt

# Save scan to condensed text file
nmap 10.10.10.0/24 -sV -oG nmap_scan_10.10.10.0_24.gnmap

# Save scan to XML file
nmap 10.10.10.0/24 -sV -oX nmap_scan_10.10.10.0_24.xml
```





- การเพิ่มประสิทธิภาพการทำงาน




```bash
# Version detection scan with max rate of 300 packets per second
nmap -sV 10.10.10.0/24 --max-rate 300

# Version detection scan with default scripts, no retry, max host timeout 15min
nmap -sV -sC 10.10.10.0/24 --max-retries 0 --host-timeout 15m

# Version detection scan with the 2000 most common ports, aggressive scan speed (T4), debug output
nmap 10.10.10.0/24 -sV --top-ports 2000 -T4 -d
```





- ประเภทการสแกน TCP




```bash
# TCP SYN scan (fast, less stealthy)
nmap -sS 192.168.1.15

# TCP Connect scan (classic)
nmap -sT 192.168.1.15

# TCP FIN scan (stealth)
nmap -sF 192.168.1.15

# TCP XMAS scan
nmap -sX 192.168.1.15

# TCP Null scan
nmap -sN 192.168.1.15

# TCP ACK scan (detect firewall)
nmap -sA 192.168.1.15
```



ฉันหวังว่าคุณจะพบว่าคำสั่งเหล่านี้มีประโยชน์ อย่าลืมปรับเป้าหมายของการสแกนให้เข้ากับบริบทของคุณและอ้างอิงเอกสารอย่างเป็นทางการเพื่อเชี่ยวชาญการทดสอบที่ดำเนินการอย่างเต็มที่



### III. บทสรุป



บทแนะนำ Nmap เสร็จสมบูรณ์แล้ว ตอนนี้คุณมีพื้นฐานที่จำเป็นในการใช้เครื่องมือที่ครอบคลุมและทรงพลังนี้ เราขอแนะนำอย่างยิ่งให้คุณฝึกฝนในสภาพแวดล้อมที่ควบคุมได้ (Hack The Box, VulnHub, เครื่องเสมือน) ก่อนที่จะใช้ในสภาพแวดล้อมการผลิต



ยังคงมีอีกมากที่ต้องสำรวจเกี่ยวกับการทำงานภายในและคุณสมบัติขั้นสูงของเครื่องมือนี้ อย่างไรก็ตาม การเชี่ยวชาญคำสั่งและแนวคิดที่นำเสนอที่นี่จะช่วยให้คุณใช้ Nmap ได้อย่างมั่นใจและมีประสิทธิภาพ
