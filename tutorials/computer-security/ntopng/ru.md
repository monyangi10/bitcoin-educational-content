## I. Презентация



**Мониторинг сетевых потоков - важная часть корпоративной сети**, будь то проверка текучести сети, получение четкой картины использования или статистика производительности. Он помогает предвидеть изменения в инфраструктуре и обеспечивает качество использования для пользователей (также известное как QoE - *Quality of Experience*, в отличие от QoS).



Для этого существует множество методик и программ/протоколов. Например, Netflow, разработанный компанией Cisco, можно использовать для получения статистики IP-потоков с Interface, но его применение ограничено совместимым оборудованием.



Поэтому в этом руководстве я расскажу о **Ntop** и покажу, как использовать его в своей инфраструктуре, чтобы следить за использованием сети.



Ntop - это программное обеспечение с открытым исходным кодом, которое может быть установлено на любой машине под управлением Linux. Она бесплатна и может собирать следующие данные:





- Использование полосы пропускания
- Основные клиенты
- Основные направления
- Используемые протоколы
- Используемые приложения
- Используемые порты
- И т.д.



Переименованный в **Ntopng (New Generation)**, он предлагает множество базовых функций бесплатно. Также доступна коммерческая версия, позволяющая экспортировать настроенные оповещения в программное обеспечение типа SIEM или фильтровать трафик с помощью правил, задаваемых непосредственно в зонде.



## II. Пререквизиты



Установка датчика Ntop зависит от оборудования и условий эксплуатации. Поэтому я не буду давать вам пошаговое руководство, а расскажу о различных возможностях.



### A. Бортовой режим



Если у вас есть брандмауэр pfSense, OPNSense или Endian, или даже рабочая станция Linux с NFTables, хорошие новости! Вы можете установить Ntopng напрямую и начать мониторинг своих интерфейсов.



Преимущество этой техники в том, что она не требует дополнительного оборудования. С другой стороны, она увеличивает загрузку ресурсов, поэтому убедитесь, что у вас есть соответствующее оборудование или ВМ достаточного размера (минимум 2 ядра и 2 БГ ОЗУ).



### B. Режим TAP / SPAN



**tap** - это **сетевое устройство, которое дублирует проходящий через него трафик и отправляет его на другое устройство.** Преимущество этого устройства в том, что оно не требует модификации существующей инфраструктуры и может быть размещено в любом месте для мониторинга определенных участков сети или между коммутатором ядра и пограничным маршрутизатором для анализа трафика в/из Интернета.



Большим недостатком этого типа оборудования является его стоимость. В современных гигабитных сетях пассивный ответвитель не может должным образом улавливать сетевой трафик, поэтому вам необходимо активное устройство стоимостью около 200 евро (минимум).



Режим **SPAN**, также известный как **зеркалирование портов**, используется коммутатором для пересылки трафика с одного порта на другой. Это, безусловно, наиболее предпочтительный для меня метод, поскольку он прост в настройке и, как и tap, позволяет контролировать часть сети или всю сеть по своему усмотрению. Однако есть два недостатка: коммутатор должен интегрировать эту функцию, и ее использование может увеличить нагрузку на процессор коммутатора.



### C. Режим маршрутизатора



Вполне возможно смонтировать маршрутизатор под Linux и установить на него ntopng. Единственным недостатком этого метода является то, что он изменит вашу сеть, либо ее внутреннюю адресацию, либо адресацию между вашим "настоящим" маршрутизатором и тем, который вы добавляете.



Примечание: для читателей статьи [Create a Wifi router with Raspberry Pi and RaspAP](https://www.it-connect.fr/creer-un-routeur-wifi-avec-un-raspberry-pi-et-raspap/): для получения точной статистики вполне можно установить ntopng на ваш Rpi!



### D. Режим моста



Интересной альтернативой является использование **машины Linux в режиме моста.** Размещенная между двумя частями оборудования, она позволяет перехватывать весь трафик без необходимости вмешиваться в конфигурацию инфраструктуры или ее оборудования. Поскольку с этой задачей может справиться и старая машина, стоимость этого метода также может быть привлекательной. Обратите внимание, что для оптимальной работы устройство должно иметь три сетевые карты: две в режиме моста, одну для доступа к Interface и SSH. Можно использовать только две карты, но тогда будет перехватываться и трафик администрирования устройства...



Поэтому я буду использовать именно **этот последний режим**. По практическим соображениям я буду использовать виртуальные машины для демонстрации, но метод остается тем же самым для использования на физических машинах.



## III. Подготовка зонда с помощью мостика Interface



Для пробника я выбрал машину **Debian 11** с минимальной установкой.



Первый шаг, всегда один и тот же, - обновление:



```
apt-get update && apt-get upgrade
```



Затем установите пакет **bridge-utils**, который позволит нам создать наш мост:



```
apt-get install bridge-utils
```



После установки первое, что нужно отметить, это текущее имя наших сетевых карт. В Debian это имя может принимать несколько форм, и оно нам понадобится для настройки.



Простая команда **ip add** выдаст результат с этой информацией:



![Image](assets/fr/016.webp)



Здесь я вижу 3 интерфейса:





- Lo**: это loopback Interface; это виртуальный Interface, который "петляет" над оборудованием. По сути, этот Interface, чей Address - 127.0.0.1 (хотя подойдет любой Address в 127.0.0.0/8, поскольку этот диапазон зарезервирован для этой цели), используется для связи с самим оборудованием. Если вы установили на своей рабочей станции веб-сайт (например, с помощью WAMPP), вы, вероятно, использовали "*localhost*" Address для отображения сайта, размещенного на вашей собственной машине. Это имя хоста связано с Address 127.0.0.1 и, следовательно, со шлейфом Interface.
- ens33**: это мой первый Interface, который получил Address здесь от моего DHCP
- ens36**: мой второй Interface



Теперь, когда у меня есть вся информация, я могу изменить файл */etc/network/interfaces*, чтобы создать свой мост. Вот как он выглядит сейчас (может отличаться):



```
# This file describes the network interfaces available on your system
# and how to activate them. For more information, see interfaces(5).

source /etc/network/interfaces.d/*

# The loopback network interface
auto lo
iface lo inet loopback

# The primary network interface
allow-hotplug ens33
iface ens33 inet dhcp
# This is an autoconfigured IPv6 interface
iface ens33 inet6 auto
```



Первая часть касается моего loopback Interface, который я не буду трогать, а затем Interface ens33. Модификации включают в себя добавление второго Interface (ens36) и настройку моста с этими двумя интерфейсами:



```
# The primary network interface
auto ens33
iface ens33 inet manual

#The secondary network interface
auto ens36
iface ens36 inet manual
```



Вот некоторые пояснения к этим первым изменениям:





- auto *Interface***: будет автоматически "запускать" Interface при загрузке системы
- iface *Interface* inet manual**: для использования Interface без какого-либо IP Address. Например, ключевое слово "static" для определения статического IP Address или "dhcp" для использования динамической адресации



Модификации продолжаются:



```
# Bridge interface
auto br0
iface br0 inet static
address 192.168.1.23
netmask 255.255.255.0
gateway 192.168.1.1
bridge_ports ens33 ens36
bridge_stp off
```



И снова несколько объяснений:





- iface br0 inet static**: здесь я определил мой мост Interface (*br0*) со статическим Address.
- Address, netmask, gateway**: информация об адресации платы
- bridge_ports**: интерфейсы, которые должны быть включены в мост
- bridge_stp**: протокол Spanning Tree используется при объединении коммутаторов для обнаружения избыточных связей и предотвращения петель. Поскольку мост может быть вставлен между двумя сетевыми путями, он может стать источником петли, отсюда и возможность включения этого протокола. Здесь он мне не нужен, поэтому я его отключаю.



Сохраните изменения и перезапустите сеть:



```
systemctl restart networking
```



Чтобы проверить изменения, снова отобразите результат команды **ip** add:



![Image](assets/fr/021.webp)


Вы можете ясно видеть **мой новый Interface "*br0*" с IP Address, который я ему назначил.** Кстати, вы также можете видеть, что два моих физических интерфейса действительно "UP", но не имеют IP Address.



## IV. Установка NtopNG



Теперь, когда наш зонд способен обнюхивать трафик между моей сетью и маршрутизатором, осталось только установить ntopng. Для этого сначала измените файл */etc/apt/sources.list* и добавьте **contrib** в конце каждой строки, начинающейся с **deb** или **deb-src**.



По умолчанию источники пакетов содержат только совместимые с DFSG (*Debian Free Sotftware Guidelines*) пакеты, обозначенные ключевым словом **main**. Вы также можете добавить эти источники:





- contrib**: пакеты, содержащие DFSG-совместимое программное обеспечение, но использующие зависимости, не входящие в ветку **main**
- non-free**: содержит пакеты, не совместимые с DFSG



Пример строки в /etc/apt/sources.list:



```
deb http://deb.debian.org/debian/ bullseye main
```



Поэтому я просто добавлю слово **contrib** к таким строчкам.



Остальные шаги перечислены на сайте [NtopNG] (https://packages.ntop.org/apt/), где для Debian 11 вам нужно добавить исходники Ntop для будущей установки. Это добавление автоматизировано с помощью файла:



```
wget https://packages.ntop.org/apt/bullseye/all/apt-ntop.deb
apt install ./apt-ntop.deb
```



Затем наступает этап установки:



```
apt-get clean all
apt-get update
apt-get install ntopng
```



Первая команда удаляет кэш менеджера пакетов apt. Вторая обновит список пакетов, а третья установит NtopNG.



После установки программа **NtopNG становится доступной непосредственно на порту 3000 машины Debian**. Так что для меня это `http://192.168.1.23:3000`

Домашняя страница NtopNG



Логин и пароль по умолчанию отображаются на экране, так что вам остается только ввести их!



## V. Использование NtopNG



Когда вы впервые войдете в систему, первое, что вам будет предложено сделать, это изменить пароль администратора и язык по умолчанию. К сожалению, французского среди них нет.



Затем вы попадаете на приборную панель:



![Image](assets/fr/023.webp)



Приборная панель NtopNG



Не привыкайте к этому! Если вы обратите внимание на желтое поле в верхней части экрана, то увидите предложение: "*Лицензия истекает в 04:57*". По умолчанию при установке запускается пробная версия полной версии NtopNG, но на (очень) ограниченное время. По истечении этого "обратного отсчета" активируется версия для *сообщества*, и приборная панель меняется:



![Image](assets/fr/024.webp)



Новая приборная панель сообщества NtopNG



Первое, что нужно сделать, это **проверить, что нужный Interface прослушивается**. В левом верхнем углу выпадающий список доступных интерфейсов позволяет выбрать интересующий нас Interface: br0



![Image](assets/fr/025.webp)



Выбор Interface



В новом окне отображаются "Top Flaw Talkers", то есть устройства, которые общаются чаще всего. Здесь отображаются только две станции:



![Image](assets/fr/026.webp)



**Хосты-источники отображаются слева, адресаты - справа ** Это позволяет визуализировать использование общей полосы пропускания каждым хостом и получить общее представление о сетевом трафике. Неплохо, но мы можем пойти дальше...



Если я нажму, например, на "*Хосты*", то получу график, показывающий потребление энергии при отправке и получении каждым хостом в моей сети. Здесь, например, видно, что только на 192.168.1.37 приходится 80 % моего трафика:



![Image](assets/fr/027.webp)



Если я щелкну на IP Address рассматриваемого хоста, я буду перенаправлен на страницу сводки:



![Image](assets/fr/028.webp)



Например, я вижу, что это машина VMWare (распознав YES моего MAC Address), что она называется DESKTOP-GHEBGV1 (так что это наверняка хост Windows), и у меня есть **статистика полученных и отправленных пакетов, а также объем данных**.



Вы также заметите новое меню в верхней части этой сводки. Я предлагаю вам нажать на "**Приложения**", чтобы увидеть, что вызывает такой большой трафик:



![Image](assets/fr/017.webp)


Ха, похоже, мы нашли ответ! На графике слева **мы видим, что 76,6% трафика поступает с ... Windows Update**, так что этот хост загружает обновления!



NtopNG использует технологию DPI - *Deep Packet Inspection*, позволяющую классифицировать каждый сетевой поток и определять приложение (или семейство приложений), стоящее за ним.



Чтобы продемонстрировать это, я запускаю видеоролик YouTube на своем хосте:



![Image](assets/fr/018.webp)



**Трафик был немедленно распознан и классифицирован!



Примечание: по понятным причинам, подобное программное обеспечение может вызывать проблемы с конфиденциальностью, поэтому будьте осторожны и используйте его в правильных условиях. Обратите внимание, что можно **анонимизировать результаты**, опция находится в **Настройки > Параметры > Разное** и называется "**Mask Host IP Address**".



## VI. Обнаружения и предупреждения



NtopNG также способен выдавать предупреждения о безопасности на определенных каналах. Их можно найти в меню **Alerts**, расположенном на баннере слева. Здесь, например, у меня в общей сложности 2851 предупреждение, разделенное по степени важности: Уведомление, Предупреждение и Ошибка.



![Image](assets/fr/019.webp)



Давайте посмотрим на предупреждения о высокой критичности, у меня их 17!



Если щелкнуть на этом рисунке, появятся подробные сведения о предупреждениях. Здесь нет ничего тревожного, это ложное срабатывание, загрузка обновлений классифицируется как передача двоичного файла в потоке HTTP, что действительно может означать атаку.



![Image](assets/fr/020.webp)



Однако, поскольку я использую бесплатную версию, я не могу исключить домены или хосты, которые являются источником оповещений, поэтому вам придется следить за ними, чтобы не пропустить что-то гораздо более тревожное. NtopNG будет generate предупреждать в случае:





- Загрузка двоичных файлов по протоколу HTTP
- Подозрительный трафик DNS
- Использование нестандартного порта
- Обнаружение SQL-инъекций
- Межсайтовый скриптинг (XSS)
- И т.д.



## VII. Заключение



В этом уроке мы рассмотрели **как настроить зонд с помощью NtopNG**, позволяющий **анализировать сетевой трафик** для визуализации использования протоколов и занятости каждого узла, а также обнаружить подозрительный трафик.



К сожалению, в этой статье я не могу рассказать обо всех функциях и возможностях, но не стесняйтесь исследовать!



Это решение может быть внедрено на постоянной основе в корпоративную инфраструктуру. NtopNG также может экспортировать результаты в базу данных InfluxDB, что позволяет создавать собственные информационные панели с помощью сторонних инструментов, таких как Graphana.
