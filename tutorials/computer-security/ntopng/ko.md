## I. 프레젠테이션



**네트워크 유동성 확인, 사용량에 대한 명확한 파악, 성능 통계 등 네트워크 흐름 모니터링은 기업 네트워크의 중요한 부분입니다**. 인프라의 변화를 예측하고 사용자의 사용 품질을 보장하는 데 도움이 됩니다(QoS와 반대되는 *사용자 경험 품질*의 의미로 QoE라고도 함).



이를 위해 다양한 기술과 소프트웨어/프로토콜을 사용할 수 있습니다. 예를 들어 Cisco에서 개발한 Netflow는 Interface에서 IP 흐름 통계를 검색하는 데 사용할 수 있지만 호환되는 장비로 사용이 제한됩니다.



그렇기 때문에 이 튜토리얼에서는 **Ntop**을 소개하고 인프라에서 이를 사용하여 네트워크 사용량을 모니터링하는 방법을 보여드리려고 합니다.



Ntop은 모든 Linux 머신에 설치할 수 있는 오픈 소스 소프트웨어입니다. 무료이며 다음과 같은 데이터를 수집할 수 있습니다:





- 대역폭 사용률
- 주요 고객
- 주요 목적지
- 사용되는 프로토콜
- 사용 애플리케이션
- 사용 포트
- 기타



이제 **Ntopng(차세대)**로 이름이 변경되어 많은 기본 기능을 무료로 제공합니다. 상용 버전도 제공되므로 구성된 경고를 SIEM 유형 소프트웨어로 내보내거나 프로브에서 직접 정의한 규칙으로 트래픽을 필터링할 수 있습니다.



## II. 전제 조건



Ntop 프로브의 설치는 장비와 환경에 따라 다릅니다. 따라서 여기서는 단계별 가이드를 제공하지는 않겠지만 다양한 가능성을 간략하게 설명하겠습니다.



### A. 온보드 모드



프로덕션 환경에서 pfSense, OPNSense 또는 Endian 방화벽을 사용 중이거나 NFTables가 설치된 Linux 워크스테이션을 사용 중이라면 좋은 소식입니다! Ntopng를 직접 설치하여 인터페이스 모니터링을 시작할 수 있습니다.



이 기술의 장점은 추가 하드웨어가 필요하지 않다는 것입니다. 반면 리소스 사용률이 높아지므로 적절한 하드웨어 또는 충분한 크기의 가상 머신(최소 2코어 및 2GB RAM)이 있는지 확인해야 합니다.



### B. 탭/스팬 모드



탭**은 **자신을 통과하는 트래픽을 복제하여 다른 장치로 전송하는 네트워크 장치**로, 기존 인프라를 수정할 필요가 없으며 특정 네트워크 구간을 모니터링하거나 코어 스위치와 에지 라우터 사이에 배치하여 인터넷 송수신 트래픽을 분석할 수 있다는 장점이 있습니다.



이러한 유형의 장비의 가장 큰 단점은 비용입니다. 오늘날의 기가비트 네트워크에서는 패시브 탭으로는 네트워크 트래픽을 제대로 캡처할 수 없으므로 약 200유로(최소)의 액티브 디바이스가 필요합니다.



포트 미러링**이라고도 하는 **SPAN** 모드는 한 포트에서 다른 포트로 트래픽을 전달하기 위해 스위치에서 사용합니다. 설정이 간단하고 탭과 마찬가지로 네트워크의 일부 또는 전체 네트워크를 마음대로 모니터링할 수 있기 때문에 제가 가장 선호하는 방법입니다. 하지만 스위치에 이 기능이 통합되어 있어야 하고 이 기능을 사용하면 스위치의 프로세서 부하가 증가할 수 있다는 두 가지 단점이 있습니다.



### C. 라우터 모드



Linux에서 라우터를 마운트하고 ntopng를 설치하는 것은 완벽하게 가능합니다. 이 방법의 유일한 단점은 네트워크의 내부 주소 지정 또는 "실제" 라우터와 추가하는 라우터 간의 주소 지정이 수정된다는 것입니다.



참고: [라즈베리파이와 라스팹으로 와이파이 라우터 만들기](https://www.it-connect.fr/creer-un-routeur-wifi-avec-un-raspberry-pi-et-raspap/) 기사를 읽으신 분이라면, 정확한 통계를 얻기 위해 Rpi에 ntopng를 설치하는 것도 완벽하게 가능합니다!



### D. 브리지 모드



흥미로운 대안은 **브리지 모드에서 Linux 머신을 사용하는 것입니다.** 두 장비 사이에 배치하면 인프라나 장비의 구성에 개입할 필요 없이 모든 트래픽을 캡처할 수 있습니다. 구형 장비로도 이 작업을 수행할 수 있으므로 이 방법의 비용도 매력적일 수 있습니다. 최적의 결과를 얻으려면 장치에 3개의 네트워크 카드가 있어야 하며, 2개는 브리지 모드에서, 하나는 Interface 및 SSH에 액세스하는 데 사용해야 합니다. 두 장의 카드만 사용할 수도 있지만 그러면 장치 관리 트래픽도 캡처됩니다...



그래서 제가 사용하려고 하는 것은 **이 마지막 모드입니다**. 현실적인 이유로 데모에서는 가상 머신을 사용하지만 실제 머신에서 사용하는 방법은 동일하게 유지됩니다.



## III. Interface 브리지로 프로브 준비



프로브의 경우 최소한의 설치로 **Debian 11** 머신을 선택했습니다.



첫 번째 단계는 항상 동일합니다:



```
apt-get update && apt-get upgrade
```



그런 다음 브릿지를 생성할 수 있는 **bridge-utils** 패키지를 설치합니다:



```
apt-get install bridge-utils
```



설치가 완료되면 가장 먼저 주목해야 할 것은 네트워크 카드의 현재 이름입니다. Debian에서 이 이름은 여러 가지 형태를 취할 수 있으며 구성에 필요합니다.



간단한 **ip add** 명령은 이 정보가 포함된 출력을 반환합니다:



![Image](assets/fr/016.webp)



여기에는 3개의 인터페이스가 표시됩니다:





- Lo**: 이것은 루프백 Interface이며, 장비를 "루프"하는 가상 Interface입니다. 기본적으로 Address가 127.0.0.1(이 범위는 이 용도로 예약되어 있으므로 127.0.0.0/8의 모든 Address가 가능함)인 이 Interface은 장비 자체에 연결하는 데 사용됩니다. 워크스테이션에 웹사이트를 설치한 경우(예: WAMPP 사용) "*로컬호스트*"를 사용한 적이 있을 것입니다 Address를 사용하여 자신의 컴퓨터에서 호스팅되는 사이트를 표시한 적이 있을 것입니다. 이 호스트 이름은 Address 127.0.0.1과 연결되어 있으므로 Interface 루프백에 연결됩니다.
- ens33**: 이것은 나의 첫 번째 Interface이며, DHCP에서 여기에서 Address을 받았습니다
- ens36**: 나의 두 번째 Interface



이제 모든 정보를 얻었으니 */etc/network/interfaces* 파일을 수정하여 브리지를 만들 수 있습니다. 현재 모습은 다음과 같습니다(다를 수 있음):



```
# This file describes the network interfaces available on your system
# and how to activate them. For more information, see interfaces(5).

source /etc/network/interfaces.d/*

# The loopback network interface
auto lo
iface lo inet loopback

# The primary network interface
allow-hotplug ens33
iface ens33 inet dhcp
# This is an autoconfigured IPv6 interface
iface ens33 inet6 auto
```



첫 번째 부분은 루프백 Interface에 관한 것으로, 여기서는 다루지 않을 것이고 그다음은 Interface ens33에 관한 것입니다. 두 번째 Interface(ens36)을 추가하고 이 두 인터페이스로 브리지를 구성하는 것이 수정 사항입니다:



```
# The primary network interface
auto ens33
iface ens33 inet manual

#The secondary network interface
auto ens36
iface ens36 inet manual
```



다음은 이러한 첫 번째 변경 사항에 대한 몇 가지 설명입니다:





- 자동 **Interface**: 시스템 시작 시 Interface를 자동으로 "시작"합니다
- iface *Interface* inet **manual**: IP Address 없이 Interface을 사용하려면. 정적 IP Address을 정의하려면 "static" 키워드를, 동적 주소 지정을 사용하려면 "dhcp"를 사용합니다



수정은 계속됩니다:



```
# Bridge interface
auto br0
iface br0 inet static
address 192.168.1.23
netmask 255.255.255.0
gateway 192.168.1.1
bridge_ports ens33 ens36
bridge_stp off
```



다시 한 번 몇 가지 설명을 덧붙입니다:





- iface br0 inet static**: 여기서는 Interface 브리지(*br0*)를 정적 Address으로 정의했습니다.
- Address, 넷마스크, 게이트웨이**: 보드 주소 정보
- bridge_ports**: 브리지에 포함될 인터페이스
- bridge_stp**: 스패닝 트리 프로토콜은 스위치를 상호 연결할 때 중복 링크를 감지하고 루프를 피하기 위해 사용됩니다. 브리지는 두 네트워크 경로 사이에 삽입될 수 있으므로 루프의 원인이 될 수 있으므로 이 프로토콜을 활성화할 수 있습니다. 여기에서는 필요하지 않으므로 비활성화합니다.



변경 사항을 저장하고 네트워크를 다시 시작합니다:



```
systemctl restart networking
```



변경 사항을 확인하려면 **ip** add 명령의 결과를 다시 표시합니다:



![Image](assets/fr/021.webp)


제가 할당해 둔 IP Address과 함께 **새 Interface "*br0*"를 명확하게 볼 수 있습니다.** 덧붙여서, 두 개의 물리적 인터페이스가 실제로 "UP"이지만 IP Address이 없는 것을 볼 수 있습니다.



## IV. NtopNG 설치



이제 프로브가 내 네트워크와 라우터 간의 트래픽을 스니핑할 수 있게 되었으니, 남은 일은 ntopng를 설치하는 것입니다. 이렇게 하려면 먼저 */etc/apt/sources.list* 파일을 수정하고 **deb** 또는 **deb-src**로 시작하는 각 줄 끝에 **contrib**를 추가합니다.



기본적으로 패키지 소스에는 **main** 키워드로 식별되는 DFSG(*데비안 자유 소프트웨어 가이드라인*) 준수 패키지만 포함됩니다. 이러한 소스를 추가할 수도 있습니다:





- 기여**: DFSG 호환 소프트웨어를 포함하지만 **메인** 브랜치에 속하지 않는 종속성을 사용하는 패키지
- 비무료**: DFSG를 준수하지 않는 패키지가 포함되어 있습니다



Etc/apt/sources.list에 있는 줄의 예입니다:



```
deb http://deb.debian.org/debian/ bullseye main
```



따라서 이와 같은 줄에 **contrib**라는 단어를 추가하겠습니다.



나머지 단계는 [NtopNG] 사이트(https://packages.ntop.org/apt/)에 나열되어 있으며, Debian 11의 경우 나중에 설치할 수 있도록 Ntop 소스를 추가해야 합니다. 이 추가는 a를 사용하여 자동화됩니다:



```
wget https://packages.ntop.org/apt/bullseye/all/apt-ntop.deb
apt install ./apt-ntop.deb
```



그런 다음 실제 설치 단계가 시작됩니다:



```
apt-get clean all
apt-get update
apt-get install ntopng
```



첫 번째 명령은 apt 패키지 관리자의 캐시를 삭제합니다. 두 번째 명령은 패키지 목록을 업데이트하고 세 번째 명령은 NtopNG를 설치합니다.



설치 후 **NtopNG 소프트웨어는 데비안 머신의 포트 3000에서 바로 사용할 수 있습니다**. 따라서 저에게는 `http://192.168.1.23:3000`입니다



![Image](assets/fr/022.webp)



NtopNG 홈 페이지



기본 로그인과 비밀번호가 표시되므로 입력하기만 하면 됩니다!

![Image](assets/fr/025.webp)



Interface 선택



새 창에는 "상위 결함 토커", 즉 가장 많이 통신하는 디바이스가 표시됩니다. 여기에는 두 개의 스테이션만 표시됩니다:



![Image](assets/fr/026.webp)



**소스 호스트는 왼쪽에, 대상은 오른쪽에 표시 ** 이렇게 하면 각 호스트의 총 대역폭 사용량을 시각화하고 네트워크 트래픽을 전체적으로 파악할 수 있습니다. 나쁘지 않지만 더 나아갈 수 있습니다...



예를 들어 "*호스트*"를 클릭하면 내 네트워크에 있는 각 호스트의 송수신 전력 소비량을 보여주는 그래프가 표시됩니다. 예를 들어 여기에서는 192.168.1.37이 내 트래픽의 80%를 차지하는 것을 볼 수 있습니다:



![Image](assets/fr/027.webp)



해당 호스트의 IP Address를 클릭하면 요약 페이지로 리디렉션됩니다:



![Image](assets/fr/028.webp)



예를 들어, (내 MAC Address의 YES를 인식하여) VMWare 머신이라는 것을 알 수 있고, 데스크톱-GHEBGV1(확실히 Windows 호스트)이라는 것을 알 수 있으며, **수신 및 전송 패킷 통계와 데이터 양**을 확인할 수 있습니다.



또한 이 요약 상단에 새로운 메뉴가 표시됩니다. "**앱**"을 클릭하여 무엇이 많은 트래픽을 유발하는지 확인해 보세요:



![Image](assets/fr/017.webp)


하, 답을 찾은 것 같네요! 왼쪽 그래프에서 **트래픽의 76.6%가 ...에서 발생하는 것을 볼 수 있습니다. 윈도우 업데이트**에서 발생한 것으로, 이 호스트는 업데이트를 다운로드하고 있습니다!



NtopNG는 *심층 패킷 검사*를 위해 DPI라는 기술을 사용하여 각 네트워크 흐름을 분류하고 그 뒤에 있는 애플리케이션(또는 애플리케이션 제품군)을 정의할 수 있습니다.



데모를 위해 호스트에서 YouTube 동영상을 실행해 보겠습니다:



![Image](assets/fr/018.webp)



**트래픽이 즉시 인식되고 분류되었습니다!



참고: 이러한 종류의 소프트웨어는 명백한 이유로 개인정보 보호 문제를 일으킬 수 있으므로 올바른 조건에서 사용하도록 주의하세요. 또한 **결과를 익명화**할 수 있으며, 이 옵션은 **설정 > 환경설정 > 기타**에서 찾을 수 있으며 "**마스크 호스트 IP Address**"이라고 합니다.



## VI. 탐지 및 경고



NtopNG는 특정 피드에 대해 보안 경고를 발행할 수도 있습니다. 이러한 알림은 왼쪽 배너의 **알람** 메뉴에서 확인할 수 있습니다. 예를 들어, 여기에는 총 2851개의 알림이 심각도에 따라 나뉘어 있습니다: 알림, 경고 및 오류로 구분됩니다.



![Image](assets/fr/019.webp)



중요도가 높은 알림을 살펴봅시다. 17개가 있습니다!



이 그림을 클릭하면 경고에 대한 세부 정보가 표시됩니다. 업데이트 다운로드가 HTTP 스트림에서 바이너리 파일 전송으로 분류되어 실제로 공격을 의미할 수 있는 오탐으로, 걱정할 필요는 없습니다.



![Image](assets/fr/020.webp)



하지만 무료 버전을 사용 중이기 때문에 알림의 원인이 되는 도메인이나 호스트를 제외할 수 없으므로, 더 큰 걱정거리를 놓치지 않으려면 해당 도메인과 호스트를 주시해야 합니다. NtopNG는 다음과 같은 경우 generate 경고를 보냅니다:





- HTTP를 통한 바이너리 파일 다운로드
- 의심스러운 DNS 트래픽
- 비표준 포트 사용
- SQL 인젝션 탐지
- 크로스 사이트 스크립팅(XSS)
- 기타



## VII. 결론



이 튜토리얼에서는 네트워크 트래픽을 **분석**하여 프로토콜 사용량과 각 호스트의 점유율을 시각화하고 의심스러운 트래픽을 탐지할 수 있는 NtopNG로 프로브를 설정하는 **방법**을 살펴보았습니다.



안타깝게도 이 글에서 모든 기능과 가능성을 다룰 수는 없지만 자유롭게 탐색해 보세요!



이 솔루션은 기업 인프라 내에서 영구적으로 구현할 수 있습니다. 또한 NtopNG는 결과를 InfluxDB 데이터베이스로 내보낼 수 있으므로 Graphana와 같은 타사 도구를 사용하여 자신만의 대시보드를 만들 수 있습니다.
