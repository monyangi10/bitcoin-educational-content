## Внедряване на Graylog в Debian 12



### I. Презентация



Graylog е решение с отворен код, предназначено да централизира, съхранява и анализира логове от вашите машини и мрежови устройства в реално време. В този урок ще научим как да инсталираме безплатната версия на Graylog на машина Debian 12!



В рамките на една информационна система всеки **сървър**, независимо дали работи под **Linux** или **Windows**, и всяко **мрежово устройство** (комутатор, маршрутизатор, защитна стена и т.н.) **генерира свои собствени логове**, които се съхраняват локално. При наличието на локално съхранени логове на всяка машина анализът и корелацията на събитията са много трудни... Тук се намесва **Graylog**. Той ще действа като **поглътител на логове**, което означава, че **всичките ви машини ще му изпращат своите логове** (например чрез syslog). След това Graylog ще **съхранява и индексира тези логове**, като ви позволява да извършвате глобални търсения и да създавате сигнали.



Graylog е инструмент за анализ и наблюдение, който улеснява идентифицирането на подозрително поведение и различни проблеми (стабилност, производителност и др.).




![Image](assets/fr/034.webp)



**Забележка: безплатната версия, Graylog Open, не е SIEM, както Wazuh, особено поради липсата на реални функции за откриване на проникване.**



### II. Предварителни условия



**Стакът Graylog** се основава на **няколко компонента**, които ще трябва да инсталираме и конфигурираме. Тук ще инсталираме всички компоненти на един и същ сървър, но е възможно да се създадат клъстери, базирани на няколко възела, и да се разпределят ролите между няколко сървъра. За целите на този урок ще инсталираме **Graylog 6.1**, най-новата версия към момента.





- MongoDB 7**, текущата препоръчителна версия за Graylog (минимум 5.0.7, максимум 7.x)
- OpenSearch**, Fork с отворен код на Elasticsearch, създаден от Amazon (минимум 1.1.x, максимум 2.15.x)
- OpenJDK 17**



Сървърът **Graylog** работи на **Debian 12**, но е възможно да се инсталира и на други дистрибуции, включително чрез Docker. Виртуалната машина е оборудвана с **8 GB RAM** и **256 GB дисково пространство**, за да има достатъчно ресурси за всички компоненти (в противен случай това може да окаже значително влияние върху производителността). Въпреки това давам това като груб ориентир, тъй като **оразмеряването на архитектурата Graylog зависи от обема на информацията, която трябва да се обработва**. Graylog може да обработва 30 MB или 300 MB данни на ден, както и 300 GB данни на ден... Това е **мащабируемо решение**, способно да обработва **терабайти дневници** (вж. [тази страница](https://go2docs.graylog.org/current/planning_your_deployment/planning_your_deployment.html?tocpath=Plan%20Your%20Deployment%7C_____0)).



![Image](assets/fr/032.webp)



Източник: Graylog



Преди да започнете конфигурирането, задайте статичен IP адрес на машината Graylog и инсталирайте най-новите актуализации. Не забравяйте да зададете часовата зона на местната машина и да определите NTP сървър за синхронизиране на датата и часа. Ето командата, която трябва да изпълните:



```
sudo timedatectl set-timezone Europe/Paris
```



**Забележка:** Инсталацията на OpenSearch не е задължителна, ако вместо нея използвате **Graylog Data Node**.

```
sudo apt-get update
sudo apt-get install curl lsb-release ca-certificates gnupg2 pwgen
```

#### A. Инсталиране на MongoDB



След като това е направено, ще започнем да инсталираме MongoDB. Изтеглете GPG ключа, съответстващ на хранилището на MongoDB:



```
curl -fsSL https://www.mongodb.org/static/pgp/server-6.0.asc | sudo gpg -o /usr/share/keyrings/mongodb-server-6.0.gpg --dearmor
```



След това добавете хранилището на MongoDB 6 към машината Debian 12:



```
echo "deb [ signed-by=/usr/share/keyrings/mongodb-server-6.0.gpg] http://repo.mongodb.org/apt/debian bullseye/mongodb-org/6.0 main" | sudo tee /etc/apt/sources.list.d/mongodb-org-6.0.list
```



След това ще актуализираме кеша на пакетите и ще се опитаме да инсталираме MongoDB:



```
sudo apt-get update
sudo apt-get install -y mongodb-org
```



MongoDB не може да бъде инсталиран, защото липсва зависимост: **libssl1.1**. Ще трябва да инсталираме този пакет ръчно, преди да продължим, тъй като Debian 12 не разполага с него в своите хранилища.



```
Les paquets suivants contiennent des dépendances non satisfaites :
mongodb-org-mongos: Dépend: libssl1.1 (>= 1.1.1) mais il n'est pas installable
mongodb-org-server: Dépend: libssl1.1 (>= 1.1.1) mais il n'est pas installable
E: Impossible de corriger les problèmes, des paquets défectueux sont en mode « garder en l'état ».
```



Ще изтеглим DEB пакета с име "**libssl1.1_1.1.1f-1ubuntu2.23_amd64.deb**" (последна версия) с командата **wget**, след което ще го инсталираме с командата **dpkg**. Това води до следните две команди:



```
wget http://archive.ubuntu.com/ubuntu/pool/main/o/openssl/libssl1.1_1.1.1f-1ubuntu2.23_amd64.deb
sudo dpkg -i libssl1.1_1.1.1f-1ubuntu2.23_amd64.deb
```



![Image](assets/fr/028.webp)



Рестартирайте инсталацията на MongoDB:



```
sudo apt-get install -y mongodb-org
```



След това рестартирайте услугата MongoDB и разрешете тя да се стартира автоматично при стартиране на сървъра Debian.



```
sudo systemctl daemon-reload
sudo systemctl enable mongod.service
sudo systemctl restart mongod.service
sudo systemctl --type=service --state=active | grep mongod
```



След като е инсталирана MongoDB, можем да преминем към инсталирането на следващия компонент.



#### B. Инсталиране на OpenSearch



Нека преминем към инсталирането на OpenSearch на сървъра. Следната команда добавя ключа за подписване на пакетите OpenSearch:



```
curl -o- https://artifacts.opensearch.org/publickeys/opensearch.pgp | sudo gpg --dearmor --batch --yes -o /usr/share/keyrings/opensearch-keyring
```



След това добавете хранилището на OpenSearch, за да можем да изтеглим пакета с **apt** след това:



```
echo "deb [signed-by=/usr/share/keyrings/opensearch-keyring] https://artifacts.opensearch.org/releases/bundle/opensearch/2.x/apt stable main" | sudo tee /etc/apt/sources.list.d/opensearch-2.x.list
```



Актуализирайте кеша на пакета:



```
sudo apt-get update
```



След това **инсталирайте OpenSearch**, като внимавате да **определите паролата по подразбиране за акаунта на администратора на вашата инстанция**. Тук паролата е "**IT-Connect2024!**", но заменете тази стойност със силна парола. **Избягвайте слаби пароли** като "**P@ssword123**" и използвайте поне **8 символа** с поне по един символ от всеки вид (малка буква, главна буква, число и специален символ), в противен случай в края на инсталацията ще има грешка. **Това е задължително условие от OpenSearch 2.12.**



```
sudo env OPENSEARCH_INITIAL_ADMIN_PASSWORD=IT-Connect2024! apt-get install opensearch
```



Моля, бъдете търпеливи по време на инсталацията...



Когато приключите, отделете малко време, за да извършите минималната конфигурация. Отворете конфигурационния файл във формат YAML:



```
sudo nano /etc/opensearch/opensearch.yml
```



Когато файлът е отворен, задайте следните опции:



```
cluster.name: graylog
node.name: ${HOSTNAME}
path.data: /var/lib/opensearch
path.logs: /var/log/opensearch
discovery.type: single-node
network.host: 127.0.0.1
action.auto_create_index: false
plugins.security.disabled: true
```



Тази конфигурация на OpenSearch е предназначена за създаване на един възел. Тук са дадени някои обяснения на различните параметри, които използваме:





- cluster.name: graylog: този параметър дава име на клъстера на OpenSearch с името "**graylog**".
- node.name: **${HOSTNAME}**: името на възела се задава динамично, за да съвпадне с това на локалната Linux машина. Дори ако имаме само един възел, важно е да го именуваме правилно.
- path.data: /var/lib/opensearch: този път указва къде OpenSearch съхранява данните си на локалната машина, в този случай в "**/var/lib/opensearch**".
- path.logs: /var/log/opensearch: този път определя мястото, където се съхраняват файловете с дневници на OpenSearch, тук в "**/var/log/opensearch**".
- discovery.type: single-node: този параметър конфигурира OpenSearch да работи с един възел, поради което е избрана опцията "**single-node**".
- network.host: 127.0.0.1: тази конфигурация гарантира, че OpenSearch слуша само в локалния контур на Interface, което е достатъчно, тъй като той е на същия сървър като Graylog.
- action.auto_create_index: false**: като деактивирате автоматичното създаване на индекс, OpenSearch няма да създава автоматично индекс, когато се изпраща документ без съществуващ индекс.
- plugins.security.disabled: true**: тази опция деактивира приставката за сигурност на OpenSearch, което означава, че няма да има удостоверяване, управление на достъпа или криптиране на комуникацията. Тази настройка спестява време при настройката на Graylog, но трябва да се избягва в продукцията (вж. [тази страница](https://opensearch.org/docs/1.0/security-plugin/index/)).



Някои опции вече са налични, така че просто трябва да премахнете "#", за да ги активирате, и след това да посочите стойността си. Ако не можете да намерите опция, можете да я декларирате директно в края на файла.

Запишете и затворете този файл.



#### C. Конфигуриране на Java (JVM)



Необходимо е да конфигурирате виртуалната машина Java, използвана от OpenSearch, за да настроите количеството памет, което тази услуга може да използва. Редактирайте следния конфигурационен файл:



```
sudo nano /etc/opensearch/jvm.options
```



С разположената тук конфигурация **OpenSearch ще започне с 4 GB заделена памет и ще може да нарасне до 4 GB**, така че няма да има промени в паметта по време на работа. Тук конфигурацията отчита факта, че виртуалната машина разполага с общо **8 GB RAM**. И двата параметъра трябва да имат една и съща стойност. Това означава, че трябва да се заменят редовете:



```
-Xms1g
-Xmx1g
```



С тези редове:



```
-Xms4g
-Xmx4g
```



Ето изображение на промяната, която трябва да се направи:



![Image](assets/fr/022.webp)



Затворете този файл, след като го запазите.



Освен това трябва да проверим конфигурацията на параметъра "**max_map_count**" в ядрото на Linux. Той определя ограничението на областите памет, картографирани за един процес, за да отговори на нуждите на нашето приложение. **OpenSearch**, както и **Elasticsearch**, препоръчва да се зададе тази стойност на "262144", за да се избегнат грешки при управлението на паметта.



По принцип при току-що инсталирана машина Debian 12 стойността вече е правилна. Но нека да проверим. Изпълнете тази команда:



```
cat /proc/sys/vm/max_map_count
```



Ако получите стойност, различна от "**262144**", изпълнете следната команда, в противен случай това не е необходимо.



```
sudo sysctl -w vm.max_map_count=262144
```



Накрая разрешете автоматичното стартиране на OpenSearch и стартирайте свързаната услуга.



```
sudo systemctl daemon-reload
sudo systemctl enable opensearch
sudo systemctl restart opensearch
```



Ако покажете състоянието на системата си, трябва да видите процес на Java с 4 GB RAM.



```
top
```

#### D. Инсталиране на Graylog



За да **инсталирате Graylog 6.1** в последната му версия, изпълнете следните 4 команди, за да **изтеглите и инсталирате Graylog Server**:



```
wget https://packages.graylog2.org/repo/packages/graylog-6.1-repository_latest.deb
sudo dpkg -i graylog-6.1-repository_latest.deb
sudo apt-get update
sudo apt-get install graylog-server
```



Когато това стане, трябва да направим някои промени в конфигурацията на Graylog, преди да се опитаме да го стартираме.



Нека започнем с конфигурирането на тези две опции:





- password_secret**: този параметър се използва за определяне на ключ, използван от Graylog за защита на съхранението на потребителските пароли (в духа на ключа за осоляване). Този ключ трябва да бъде **уникален** и **случаен**.
- root_password_sha2**: този параметър съответства на паролата на администратора по подразбиране в Graylog. Тя се съхранява като Hash SHA-256.



Ще започнем с генерирането на 96-символен ключ за параметъра **парола_секрет**:



```
pwgen -N 1 -s 96
wVSGYwOmwBIDmtQvGzSuBevWoXe0MWpNWCzhorBfvMMhia2zIjHguTbfl4uXZJdHOA0EEb1sOXJTZKINhIIBm3V57vwfQV59
```



Копирайте върнатата стойност, след което отворете конфигурационния файл Graylog:



```
sudo nano /etc/graylog/server/server.conf
```



Поставете ключа в параметъра **password_secret** по следния начин:



![Image](assets/fr/027.webp)



Запишете и затворете файла.



След това трябва да зададете паролата за акаунта "**admin**", създаден по подразбиране. В конфигурационния файл трябва да се съхрани Hash на паролата, което означава, че тя трябва да се изчисли. Примерът по-долу дава Hash на паролата "**LogsWells@**": адаптирайте стойността към вашата парола.



```
echo -n "PuitsDeLogs@" | shasum -a 256
6b297230efaa2905c9a746fb33a628f4d7aba4fa9d5c1b3daa6846c68e602d71
```



Копирайте получената стойност като изход (без тирето в края на реда).



Отворете отново конфигурационния файл на Graylog:



```
sudo nano /etc/graylog/server/server.conf
```



Вмъкнете стойността в опцията **root_password_sha2** по следния начин:



![Image](assets/fr/026.webp)



Докато сте в конфигурационния файл, задайте опцията "**http_bind_address**". Посочете "**0.0.0.0:9000**", за да може да се осъществи достъп до уеб страницата на Graylog Interface на порт **9000** чрез всеки IP адрес на сървъра.



![Image](assets/fr/024.webp)



След това задайте опцията "**elasticsearch_hosts**" на `http://127.0.0.1:9200`, за да декларирате нашата местна инстанция на OpenSearch. Това е необходимо, тъй като не използваме **Graylog Data Node**. А без тази опция няма да е възможно да продължим по-нататък...



![Image](assets/fr/025.webp)



Запишете и затворете файла.



Тази команда активира Graylog, така че да се стартира автоматично при следващото зареждане, и незабавно стартира сървъра Graylog.



```
sudo systemctl enable --now graylog-server
```



След като това бъде направено, опитайте да се свържете с Graylog от браузър. Въведете IP адреса (или името) на сървъра и порт 9000.

## Изпращане на дневници към Graylog с rsyslog



### I. Презентация



В тази втора част ще научим как да конфигурираме машина с Linux да изпраща дневниците си към сървър Graylog. За тази цел ще инсталираме и конфигурираме Rsyslog в системата.



### II. Конфигуриране на Graylog за получаване на дневници за Linux



Ще започнем с конфигурирането на Graylog. Има три стъпки за изпълнение:





- Създаване на **вход** за създаване на входна точка, позволяваща на Linux машини да **изпращат Syslog логове чрез UDP**.
- Създаване на нов **индекс** за съхраняване и **индексиране на всички дневници на Linux**.
- Създаване на **поток** за **пренасочване** на дневниците, получени от Graylog, към новия индекс на Linux.



#### A. Създаване на вход за Syslog



Влезте в Graylog Interface, щракнете върху "**System**" (Система**) в менюто и след това върху "**Inputs**" (Входове**). В падащия списък изберете "**Syslog UDP**", след което щракнете върху бутона с надпис "**Launch new input**" (Стартиране на нов вход**). Възможно е също така да създадете TCP Syslog вход, но това изисква използването на TLS сертификат: плюс за сигурността, но не е разгледан в тази статия.



![Image](assets/fr/001.webp)



На екрана ще се появи съветник. Започнете, като дадете на този вход име, например "**Graylog_UDP_Rsyslog_Linux**", и изберете порт. По подразбиране портът е "**514**", но можете да го персонализирате. Тук е избран порт "**12514**".



![Image](assets/fr/016.webp)



Можете също така да поставите отметка пред опцията "**Store full message**" (Съхранявай цялото съобщение**), за да съхраните пълното съобщение в дневника в Graylog. Щракнете върху "**Създаване на вход**".



![Image](assets/fr/017.webp)



Новият вход е създаден и вече е активен. Graylog вече може да получава Syslog логове на порт 12514/UDP, но все още не сме завършили конфигурирането на приложението.



![Image](assets/fr/018.webp)


**Забележка:** Един вход може да се използва за съхраняване на дневници от няколко машини с Linux.



#### B. Създаване на нов индекс на Linux



Трябва да създадем индекс в Graylog, за да съхраняваме дневници от Linux машини. **Индекс** в Graylog е структура за съхранение, която съдържа получените логове, т.е. получените съобщения. Graylog използва OpenSearch като двигател за съхранение и съобщенията се индексират, за да се даде възможност за бързо и ефективно търсене.



В Graylog щракнете върху "**System**" (Система**) в менюто, след което върху "**Indices**" (Показатели**). На новата страница, която се появява, щракнете върху "**Създаване на набор от индекси**".



![Image](assets/fr/005.webp)



Дайте име на този индекс, например "**Linux Index**", добавете описание и префикс, преди да потвърдите. Тук ще **съхраняваме всички Linux логове в този индекс**. Възможно е също така да се създадат специфични индекси за съхраняване само на определени логове (само [SSH] логове (https://www.it-connect.fr/cours/comprendre-et-maitriser-ssh/ "SSH"), логове на уеб услуги и т.н.).



![Image](assets/fr/006.webp)



Сега трябва да създадем нов поток, който да насочва съобщенията към този индекс.



#### C. Създаване на поток



За да създадете нов поток, щракнете върху "**Предавания**" в главното меню на Graylog. След това щракнете върху бутона "**Създаване на поток**" вдясно. В появилия се прозорец дайте име на потока, например "**Linux Stream**", и изберете индекса "**Linux Index**" за полето, наречено "**Index Set**". Потвърдете избора си.



**Забележка: съобщенията, съответстващи на този поток, ще бъдат включени и в "Потока по подразбиране", освен ако не поставите отметка в опцията "Премахване на съвпаденията от "Потока по подразбиране"".**



![Image](assets/fr/002.webp)



След това в настройките на парата щракнете върху бутона "**Прибавяне на правило за поток**", за да добавите ново правило за маршрутизиране на съобщения. Ако не можете да намерите този прозорец, кликнете върху "**Streams**" в менюто, след това на реда, съответстващ на вашия поток, кликнете върху "**More**" (Повече**), а след това върху "**Manage Rules** (Управление на правилата**)".



Изберете типа "**match input**" и изберете предварително създадения **Rsyslog input in UDP**. Потвърдете с бутона "**Създай правило**". Всички съобщения към нашия нов вход сега ще се изпращат към индекса за Linux.



![Image](assets/fr/003.webp)



Новият ви поток трябва да се появи в списъка и да е в състояние "**Работи**". Пропускателната способност на съобщението показва "**0 msg/s**", тъй като в момента не изпращаме никакви логове на входа Rsyslog UDP. За да видите дневниците на даден поток, просто щракнете върху името му.



![Image](assets/fr/004.webp)



**Всичко е готово от страна на Graylog**. Следващата стъпка е да конфигурирате машината с Linux.



### III. Инсталиране и конфигуриране на Rsyslog под Linux



Влезте в машината с Linux, локално или отдалечено, и използвайте потребителски акаунт с права за повишаване на привилегиите му (чрез sudo). В противен случай използвайте директно акаунта "root".



#### A. Инсталиране на пакета Rsyslog



Започнете, като актуализирате кеша на пакетите и инсталирате пакета с име "**rsyslog**".



```
sudo apt-get update
sudo apt-get install rsyslog
```



След това проверете състоянието на услугата. В повечето случаи тя вече е стартирана.



```
sudo systemctl status rsyslog
```



#### B. Конфигуриране на Rsyslog



Rsyslog има основен конфигурационен файл, разположен тук:



```
/etc/rsyslog.conf
```



Освен това директорията "**/etc/rsyslog.d/**" се използва за съхраняване на допълнителни конфигурационни файлове за Rsyslog. В основния конфигурационен файл има директива Include за импортиране на всички файлове "**.conf**", разположени в тази директория. Това дава възможност да се използват допълнителни файлове за конфигуриране на Rsyslog, без да се променя основният файл.



В тази директория трябва да използвате цифри, за да определите реда на зареждане, тъй като файловете се зареждат по азбучен ред. Добавянето на цифров префикс ви позволява да управлявате приоритета между няколко конфигурационни файла. Тук имаме само един допълнителен файл, така че това не е проблем.



В тази директория ще създадем файл, наречен "**10-graylog.conf**":



```
sudo nano /etc/rsyslog.d/10-graylog.conf
```



В този файл въведете този ред:



```
*.* @192.168.10.220:12514;RSYSLOG_SyslogProtocol23Format
```



Ето как да тълкуваме този ред:





- `*.*`: означава изпращане на всички Syslog логове от Linux машината към Graylog.
- `@`: указва, че преносът се извършва в UDP. Трябва да посочите "**@@**", за да преминете към TCP.
- `192.168.10.220:12514`: указва адреса на сървъра Graylog и порта, на който се изпращат дневниците (съответстващ на Input).
- `RSYSLOG_SyslogProtocol23Format`: отговаря на формата на съобщенията, които ще се изпращат към Graylog.



След като приключите, запазете файла и рестартирайте Rsyslog.



```
sudo systemctl restart rsyslog.service
```



След това действие първите съобщения трябва да пристигнат на вашия сървър Graylog!



### IV. Показване на дневниците на Linux в Graylog



От Graylog можете да щракнете върху "**Streams**" и да изберете новия поток, за да видите свързаните с него съобщения. Алтернативно, щракнете върху "**Издирване**" и изберете своя Steam и стартирайте търсене.



Ето някои основни характеристики на Interface:



**1** - Изберете периода, за който да се показват съобщенията. По подразбиране се показват съобщенията от последните 5 минути.



**2** - Изберете потока(ите), който(ито) да бъде(ат) показан(и).



**3** - Активирайте автоматичното опресняване на списъка със съобщения (например на всеки 5 секунди). В противен случай той е статичен и ще трябва да го опреснявате ръчно.



**4** - Щракнете върху лупата, за да стартирате търсенето след промяна на периода, потока или филтъра.



**5** - Въвеждаща лента за задаване на филтри за търсене. Тук посочвам "**source:srv\-docker**", за да покажа само логовете на новата машина, на която току-що съм настроил Rsyslog.

### V. Идентифициране на отказ на SSH връзка



Силата на Graylog се състои в способността му да индексира дневници и да дава възможност за търсене по различни критерии: машина на източника, времеви печат, съдържание на съобщението и др. Можем да търсим да идентифицираме неуспешни SSH връзки.



От отдалечена машина (например от сървъра Graylog) се опитайте да се свържете с вашия Linux сървър, на който току-що сте конфигурирали Rsyslog. Например:



```
ssh [email protected]
```



След това умишлено въведете неправилно **име** и **парола**, за да получите **generate грешки при свързване**. Във файла "**/var/log/auth.log**" това ще доведе до съобщения в дневника generate, подобни на следните:



```
Failed password for invalid user it-connect from 192.168.10.199 port 50352 ssh2
```



Трябва да намерите тези съобщения в Graylog.



В Graylog използвайте следния филтър за търсене, за да покажете само съвпадащите съобщения:



```
message:Failed password AND application_name:sshd
```



Ако разполагате с няколко сървъра и желаете да анализирате дневниците на конкретен сървър, посочете допълнително неговото име:



```
message:Failed password AND application_name:sshd AND source:srv\-docker
```



Ето един преглед на резултата на машина, на която генерирах няколко грешки при свързване в различни часове на деня:



![Image](assets/fr/009.webp)



От машината с IP адрес "**192.168.10.199**" се правят неуспешни опити за свързване. Ако искате да научите повече за дейността на този хост, можете да **търсите този IP адрес**. Graylog ще изведе всички логове, в които се споменава този IP адрес, на всички хостове (за които е конфигурирано изпращането на логове).



В този случай филтърът, който трябва да се използва, може да бъде:



```
message:"192.168.10.199"
```



Получаваме допълнителни резултати (което не е изненадващо, тъй като не филтрираме приложението SSH):
