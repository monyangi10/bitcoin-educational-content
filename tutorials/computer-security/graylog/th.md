## กำลังปรับใช้ Graylog บน Debian 12



### I. การนำเสนอ



Graylog เป็นโซลูชัน "log sink" แบบโอเพนซอร์สที่ออกแบบมาเพื่อรวมศูนย์ จัดเก็บ และวิเคราะห์ล็อกจากเครื่องและอุปกรณ์เครือข่ายของคุณแบบเรียลไทม์ ในบทแนะนำนี้ เราจะเรียนรู้วิธีการติดตั้งเวอร์ชันฟรีของ Graylog บนเครื่อง Debian 12!



ภายในระบบสารสนเทศ, **เซิร์ฟเวอร์**แต่ละเครื่อง, ไม่ว่าจะใช้ระบบปฏิบัติการ **Linux** หรือ **Windows**, และอุปกรณ์เครือข่ายแต่ละตัว (สวิตช์, เราเตอร์, ไฟร์วอลล์, ฯลฯ...) **สร้างบันทึกของตัวเอง**, เก็บไว้ในเครื่องนั้นๆ การวิเคราะห์และเชื่อมโยงเหตุการณ์จึงเป็นเรื่องยากมาก... นี่คือที่ที่ **Graylog** เข้ามามีบทบาท มันจะทำหน้าที่เป็น **log sink**, หมายความว่า **เครื่องทั้งหมดของคุณจะส่งบันทึกไปให้มัน** (ผ่าน syslog, ตัวอย่างเช่น) จากนั้น Graylog จะ **เก็บและจัดทำดัชนีบันทึกเหล่านี้**, พร้อมทั้งให้คุณสามารถทำการค้นหาระดับโลกและสร้างการแจ้งเตือนได้



Graylog เป็นเครื่องมือวิเคราะห์และตรวจสอบที่ทำให้ง่ายต่อการระบุพฤติกรรมที่น่าสงสัยและปัญหาต่างๆ (ความเสถียร, ประสิทธิภาพ, ฯลฯ)




![Image](assets/fr/034.webp)



**หมายเหตุ: เวอร์ชันฟรี Graylog Open ไม่ใช่ SIEM เหมือน Wazuh โดยเฉพาะอย่างยิ่งเนื่องจากขาดฟังก์ชันการตรวจจับการบุกรุกที่แท้จริง**



### II. ข้อกำหนดเบื้องต้น



**stack Graylog** สร้างขึ้นจาก **หลายองค์ประกอบ** ที่เราจำเป็นต้องติดตั้งและกำหนดค่า ที่นี่เราจะติดตั้งองค์ประกอบทั้งหมดบนเซิร์ฟเวอร์เดียวกัน แต่สามารถสร้างคลัสเตอร์ที่ประกอบด้วยหลายโหนดและกระจายบทบาทไปยังเซิร์ฟเวอร์หลายเครื่องได้ สำหรับวัตถุประสงค์ของบทแนะนำนี้ เราจะติดตั้ง **Graylog 6.1** ซึ่งเป็นเวอร์ชันล่าสุดในปัจจุบัน





- MongoDB 7**, เวอร์ชันที่แนะนำในปัจจุบันสำหรับ Graylog (ขั้นต่ำ 5.0.7, สูงสุด 7.x)
- OpenSearch**, โอเพ่นซอร์ส Fork ของ Elasticsearch ที่สร้างโดย Amazon (ขั้นต่ำ 1.1.x, สูงสุด 2.15.x)
- OpenJDK 17**



**เซิร์ฟเวอร์ Graylog** กำลังทำงานบน **Debian 12** แต่สามารถติดตั้งบนดิสทริบิวชันอื่น ๆ ได้ รวมถึงผ่าน Docker เครื่องเสมือนนี้มี **RAM 8 GB** และ **พื้นที่ดิสก์ 256 GB** เพื่อให้มีทรัพยากรเพียงพอสำหรับทุกองค์ประกอบ (มิฉะนั้นอาจมีผลกระทบอย่างมากต่อประสิทธิภาพ) อย่างไรก็ตาม ฉันให้ข้อมูลนี้เป็นแนวทางคร่าว ๆ เนื่องจาก **การกำหนดขนาดของสถาปัตยกรรม Graylog ขึ้นอยู่กับปริมาณข้อมูลที่จะประมวลผล** Graylog สามารถประมวลผลข้อมูลได้ 30 MB หรือ 300 MB ต่อวัน รวมถึง 300 GB ต่อวัน... มันเป็น **โซลูชันที่สามารถขยายได้** ที่สามารถจัดการกับ **เทราไบต์ของล็อก** (ดู [หน้านี้](https://go2docs.graylog.org/current/planning_your_deployment/planning_your_deployment.html?tocpath=Plan%20Your%20Deployment%7C_____0))



![Image](assets/fr/032.webp)



แหล่งที่มา: Graylog



ก่อนเริ่มการกำหนดค่า ให้กำหนดที่อยู่ IP แบบคงที่ให้กับเครื่อง Graylog และติดตั้งการอัปเดตล่าสุด อย่าลืมตั้งค่าเขตเวลาของเครื่องท้องถิ่นและกำหนดเซิร์ฟเวอร์ NTP สำหรับการซิงโครไนซ์วันที่และเวลา นี่คือคำสั่งที่ต้องรัน:



```
sudo timedatectl set-timezone Europe/Paris
```



**หมายเหตุ:** การติดตั้ง OpenSearch เป็นทางเลือกหากคุณใช้ **Graylog Data Node** แทน

#### A. การติดตั้ง MongoDB



เมื่อเสร็จสิ้นแล้ว เราจะเริ่มติดตั้ง MongoDB ดาวน์โหลด GPG key ที่ตรงกับที่เก็บ MongoDB:



```
curl -fsSL https://www.mongodb.org/static/pgp/server-6.0.asc | sudo gpg -o /usr/share/keyrings/mongodb-server-6.0.gpg --dearmor
```



จากนั้นเพิ่มที่เก็บ MongoDB 6 ไปยังเครื่อง Debian 12:



```
echo "deb [ signed-by=/usr/share/keyrings/mongodb-server-6.0.gpg] http://repo.mongodb.org/apt/debian bullseye/mongodb-org/6.0 main" | sudo tee /etc/apt/sources.list.d/mongodb-org-6.0.list
```



ถัดไป เราจะอัปเดตแคชแพ็กเกจและพยายามติดตั้ง MongoDB:



```
sudo apt-get update
sudo apt-get install -y mongodb-org
```



MongoDB ไม่สามารถติดตั้งได้เนื่องจากขาดการพึ่งพา: **libssl1.1** เราจะต้องติดตั้งแพ็คเกจนี้ด้วยตนเองก่อนที่จะดำเนินการต่อ เพราะว่า Debian 12 ไม่มีในที่เก็บข้อมูลของมัน



```
Les paquets suivants contiennent des dépendances non satisfaites :
mongodb-org-mongos: Dépend: libssl1.1 (>= 1.1.1) mais il n'est pas installable
mongodb-org-server: Dépend: libssl1.1 (>= 1.1.1) mais il n'est pas installable
E: Impossible de corriger les problèmes, des paquets défectueux sont en mode « garder en l'état ».
```



เราจะดาวน์โหลดแพ็กเกจ DEB ที่ชื่อ "**libssl1.1_1.1.1f-1ubuntu2.23_amd64.deb**" (เวอร์ชันล่าสุด) ด้วยคำสั่ง **wget** จากนั้นติดตั้งด้วยคำสั่ง **dpkg** ซึ่งจะได้คำสั่งสองคำสั่งดังนี้:



```
wget http://archive.ubuntu.com/ubuntu/pool/main/o/openssl/libssl1.1_1.1.1f-1ubuntu2.23_amd64.deb
sudo dpkg -i libssl1.1_1.1.1f-1ubuntu2.23_amd64.deb
```



![Image](assets/fr/028.webp)



เริ่มการติดตั้ง MongoDB ใหม่:



```
sudo apt-get install -y mongodb-org
```



จากนั้นรีสตาร์ทบริการ MongoDB และเปิดใช้งานให้เริ่มต้นโดยอัตโนมัติเมื่อเซิร์ฟเวอร์ Debian ถูกเปิดใช้งาน



```
sudo systemctl daemon-reload
sudo systemctl enable mongod.service
sudo systemctl restart mongod.service
sudo systemctl --type=service --state=active | grep mongod
```



เมื่อได้ติดตั้ง MongoDB แล้ว เราสามารถดำเนินการติดตั้งส่วนประกอบถัดไปได้



#### B. การติดตั้ง OpenSearch



มาดำเนินการติดตั้ง OpenSearch บนเซิร์ฟเวอร์กันต่อ คำสั่งต่อไปนี้จะเพิ่มคีย์ลายเซ็นสำหรับแพ็กเกจ OpenSearch:



```
curl -o- https://artifacts.opensearch.org/publickeys/opensearch.pgp | sudo gpg --dearmor --batch --yes -o /usr/share/keyrings/opensearch-keyring
```



จากนั้นเพิ่มที่เก็บ OpenSearch เพื่อให้เราสามารถดาวน์โหลดแพ็กเกจด้วย **apt** ภายหลัง:



```
echo "deb [signed-by=/usr/share/keyrings/opensearch-keyring] https://artifacts.opensearch.org/releases/bundle/opensearch/2.x/apt stable main" | sudo tee /etc/apt/sources.list.d/opensearch-2.x.list
```



อัปเดตแคชแพ็คเกจของคุณ:



```
sudo apt-get update
```



จากนั้น **ติดตั้ง OpenSearch** โดยระมัดระวังในการ **กำหนดรหัสผ่านเริ่มต้นสำหรับบัญชี Admin ของอินสแตนซ์ของคุณ** ที่นี่ รหัสผ่านคือ "**IT-Connect2024!**" แต่ให้แทนค่าด้วยรหัสผ่านที่แข็งแรง **หลีกเลี่ยงรหัสผ่านที่อ่อนแอ** เช่น "**P@ssword123**" และใช้ตัวอักษรอย่างน้อย **8 ตัว** โดยมีตัวอักษรแต่ละประเภทอย่างน้อยหนึ่งตัว (ตัวพิมพ์เล็ก, ตัวพิมพ์ใหญ่, ตัวเลข และอักขระพิเศษ) มิฉะนั้นจะเกิดข้อผิดพลาดเมื่อสิ้นสุดการติดตั้ง **นี่เป็นข้อกำหนดเบื้องต้นตั้งแต่ OpenSearch 2.12.**



```
sudo env OPENSEARCH_INITIAL_ADMIN_PASSWORD=IT-Connect2024! apt-get install opensearch
```



โปรดอดทนระหว่างการติดตั้ง...



เมื่อคุณทำเสร็จแล้ว ให้ใช้เวลาสักครู่ในการกำหนดค่าขั้นต่ำ เปิดไฟล์การกำหนดค่าในรูปแบบ YAML:



```
sudo nano /etc/opensearch/opensearch.yml
```



เมื่อเปิดไฟล์ ให้ตั้งค่าตัวเลือกต่อไปนี้:



```
cluster.name: graylog
node.name: ${HOSTNAME}
path.data: /var/lib/opensearch
path.logs: /var/log/opensearch
discovery.type: single-node
network.host: 127.0.0.1
action.auto_create_index: false
plugins.security.disabled: true
```



การกำหนดค่า OpenSearch นี้ถูกออกแบบมาเพื่อการตั้งค่าโหนดเดียว นี่คือคำอธิบายของพารามิเตอร์ต่าง ๆ ที่เราใช้:





- cluster.name: graylog: พารามิเตอร์นี้ตั้งชื่อคลัสเตอร์ OpenSearch ด้วยชื่อ "**graylog**".
- node.name: **${HOSTNAME}**: ชื่อโหนดถูกตั้งค่าแบบไดนามิกให้ตรงกับชื่อของเครื่อง Linux ในท้องถิ่น แม้ว่าเราจะมีเพียงโหนดเดียว แต่การตั้งชื่อให้ถูกต้องก็เป็นสิ่งสำคัญ
- path.data: /var/lib/opensearch: เส้นทางนี้ระบุว่า OpenSearch เก็บข้อมูลไว้ที่ใดบนเครื่องท้องถิ่น ในกรณีนี้คือ "**/var/lib/opensearch**"
- path.logs: /var/log/opensearch: เส้นทางนี้กำหนดตำแหน่งที่เก็บไฟล์บันทึกของ OpenSearch ซึ่งอยู่ที่ "**/var/log/opensearch**"
- discovery.type: single-node: พารามิเตอร์นี้กำหนดค่า OpenSearch ให้ทำงานกับโหนดเดียว ดังนั้นจึงเลือกตัวเลือก "**single-node**"
- network.host: 127.0.0.1: การกำหนดค่านี้ทำให้ OpenSearch ฟังเฉพาะบน Interface local loop ของมันเอง ซึ่งเพียงพอแล้วเนื่องจากอยู่บนเซิร์ฟเวอร์เดียวกับ Graylog
- action.auto_create_index: false**: โดยการปิดการสร้างดัชนีอัตโนมัติ OpenSearch จะไม่สร้างดัชนีโดยอัตโนมัติเมื่อมีการส่งเอกสารโดยไม่มีดัชนีที่มีอยู่
- plugins.security.disabled: true**: ตัวเลือกนี้จะปิดการใช้งานปลั๊กอินความปลอดภัยของ OpenSearch ซึ่งหมายความว่าจะไม่มีการตรวจสอบสิทธิ์ การจัดการการเข้าถึง หรือการเข้ารหัสการสื่อสาร การตั้งค่านี้ช่วยประหยัดเวลาเมื่อทำการตั้งค่า Graylog แต่ควรหลีกเลี่ยงในการใช้งานจริง (ดู [หน้านี้](https://opensearch.org/docs/1.0/security-plugin/index/))



มีตัวเลือกบางอย่างอยู่แล้ว ดังนั้นคุณเพียงแค่ต้องลบ "#" เพื่อเปิดใช้งาน จากนั้นระบุค่าของคุณ หากคุณไม่พบตัวเลือก คุณสามารถประกาศได้โดยตรงที่ส่วนท้ายของไฟล์

#### C. กำหนดค่า Java (JVM)



คุณจำเป็นต้องกำหนดค่า Java Virtual Machine ที่ใช้โดย OpenSearch เพื่อปรับปริมาณหน่วยความจำที่บริการนี้สามารถใช้ได้ แก้ไขไฟล์การกำหนดค่าต่อไปนี้:



```
sudo nano /etc/opensearch/jvm.options
```



ด้วยการกำหนดค่าที่ปรับใช้ที่นี่ **OpenSearch จะเริ่มต้นด้วยหน่วยความจำที่จัดสรรไว้ 4 GB และสามารถเติบโตได้สูงสุดถึง 4 GB** ดังนั้นจะไม่มีการเปลี่ยนแปลงของหน่วยความจำระหว่างการทำงาน ที่นี่ การกำหนดค่าจะคำนึงถึงข้อเท็จจริงที่ว่าเครื่องเสมือนมีหน่วยความจำรวม **8 GB RAM** พารามิเตอร์ทั้งสองต้องมีค่าเท่ากัน ซึ่งหมายถึงการแทนที่บรรทัด:



```
-Xms1g
-Xmx1g
```



ด้วยบรรทัดเหล่านี้:



```
-Xms4g
-Xmx4g
```



นี่คือภาพของการแก้ไขที่จะทำ:



![Image](assets/fr/022.webp)



ปิดไฟล์นี้หลังจากบันทึกแล้ว



นอกจากนี้ เราจำเป็นต้องตรวจสอบการตั้งค่าของพารามิเตอร์ "**max_map_count**" ในเคอร์เนลของ Linux ซึ่งกำหนดขีดจำกัดของพื้นที่หน่วยความจำที่แมปต่อกระบวนการ เพื่อให้ตรงกับความต้องการของแอปพลิเคชันของเรา **OpenSearch** เช่นเดียวกับ **Elasticsearch** แนะนำให้ตั้งค่านี้เป็น "262144" เพื่อหลีกเลี่ยงข้อผิดพลาดในการจัดการหน่วยความจำ



ในหลักการแล้ว บนเครื่อง Debian 12 ที่ติดตั้งใหม่ ค่าได้ถูกต้องอยู่แล้ว แต่ลองตรวจสอบดู รันคำสั่งนี้:



```
cat /proc/sys/vm/max_map_count
```



หากคุณได้รับค่าอื่นที่ไม่ใช่ "**262144**" ให้รันคำสั่งต่อไปนี้ มิฉะนั้นไม่จำเป็นต้องทำ



```
sudo sysctl -w vm.max_map_count=262144
```



สุดท้าย เปิดใช้งานการเริ่มต้นอัตโนมัติของ OpenSearch และเปิดใช้บริการที่เกี่ยวข้อง



```
sudo systemctl daemon-reload
sudo systemctl enable opensearch
sudo systemctl restart opensearch
```



หากคุณแสดงสถานะระบบของคุณ คุณควรเห็นกระบวนการ Java ที่มี RAM 4 GB



```
top
```

## ส่งล็อกไปยัง Graylog ด้วย rsyslog



### I. การนำเสนอ



ในส่วนที่สองนี้ เราจะเรียนรู้วิธีการกำหนดค่าเครื่อง Linux เพื่อส่งบันทึกไปยังเซิร์ฟเวอร์ Graylog สำหรับการทำเช่นนี้ เราจะติดตั้งและกำหนดค่า Rsyslog บนระบบ



### II. การกำหนดค่า Graylog เพื่อรับบันทึกของ Linux



เราจะเริ่มต้นด้วยการกำหนดค่า Graylog มีสามขั้นตอนที่ต้องทำให้เสร็จ:





- การสร้าง **Input** เพื่อสร้างจุดเริ่มต้นที่อนุญาตให้เครื่อง Linux **ส่ง Syslog logs ผ่าน UDP**
- การสร้าง **ดัชนี** ใหม่เพื่อจัดเก็บและ **จัดทำดัชนีบันทึก Linux ทั้งหมด**
- การสร้าง **Stream** เพื่อ **route** บันทึกที่ได้รับจาก Graylog ไปยัง Linux Index ใหม่



#### A. สร้างอินพุตสำหรับ Syslog



เข้าสู่ระบบ Graylog Interface คลิกที่ "**System**" ในเมนูแล้วคลิกที่ "**Inputs**" ในรายการแบบดรอปดาวน์ เลือก "**Syslog UDP**" จากนั้นคลิกที่ปุ่มที่มีป้ายกำกับว่า "**Launch new input**" นอกจากนี้ยังสามารถสร้าง TCP Syslog Input ได้ แต่ต้องใช้ใบรับรอง TLS: เป็นข้อดีสำหรับความปลอดภัย แต่ไม่ได้ครอบคลุมในบทความนี้



![Image](assets/fr/001.webp)



พ่อมดจะปรากฏบนหน้าจอ เริ่มต้นด้วยการตั้งชื่อให้กับ Input นี้ เช่น "**Graylog_UDP_Rsyslog_Linux**" และเลือกพอร์ต โดยค่าเริ่มต้น พอร์ตคือ "**514**" แต่คุณสามารถปรับแต่งได้ ที่นี่เลือกพอร์ต "**12514**"



![Image](assets/fr/016.webp)



คุณยังสามารถเลือกตัวเลือก "**Store full message**" เพื่อจัดเก็บข้อความบันทึกทั้งหมดใน Graylog คลิกที่ "**Launch Input**"



![Image](assets/fr/017.webp)



อินพุตใหม่ได้ถูกสร้างขึ้นและใช้งานได้แล้ว Graylog สามารถรับบันทึก Syslog บนพอร์ต 12514/UDP ได้แล้ว แต่เรายังไม่ได้ทำการตั้งค่าแอปพลิเคชันให้เสร็จสิ้น



![Image](assets/fr/018.webp)


**หมายเหตุ:** อินพุตเดียวสามารถใช้เก็บบันทึกจากเครื่อง Linux หลายเครื่องได้



#### B. สร้างดัชนี Linux ใหม่



เราจำเป็นต้องสร้างดัชนีใน Graylog เพื่อจัดเก็บบันทึกจากเครื่อง Linux ดัชนีใน Graylog เป็นโครงสร้างการจัดเก็บที่มีบันทึกที่ได้รับ เช่น ข้อความที่ได้รับ Graylog ใช้ OpenSearch เป็นเครื่องมือจัดเก็บข้อมูล และข้อความจะถูกจัดทำดัชนีเพื่อให้สามารถค้นหาได้อย่างรวดเร็วและมีประสิทธิภาพ



จาก Graylog ให้คลิกที่ "**System**" ในเมนู จากนั้นคลิกที่ "**Indices**" ในหน้าที่ปรากฏขึ้นใหม่ ให้คลิกที่ "**Create index set**"



![Image](assets/fr/005.webp)



ตั้งชื่อดัชนีนี้ เช่น "**ดัชนี Linux**", เพิ่มคำอธิบายและคำนำหน้า ก่อนยืนยัน ที่นี่เราจะ **เก็บบันทึก Linux ทั้งหมดในดัชนีนี้** นอกจากนี้ยังสามารถสร้างดัชนีเฉพาะเพื่อเก็บบันทึกบางอย่างเท่านั้น (เฉพาะบันทึก [SSH] (https://www.it-connect.fr/cours/comprendre-et-maitriser-ssh/ "SSH"), บันทึกบริการเว็บ เป็นต้น)



![Image](assets/fr/006.webp)



ตอนนี้เราจำเป็นต้องสร้างสตรีมใหม่เพื่อส่งข้อความไปยังดัชนีนี้



#### ค. สร้างสตรีม



ในการสร้างสตรีมใหม่ ให้คลิกที่ "**Streams**" ในเมนูหลักของ Graylog จากนั้นคลิกที่ปุ่ม "**Create stream**" ทางด้านขวา ในหน้าต่างที่ปรากฏขึ้น ให้ตั้งชื่อสตรีม เช่น "**Linux Stream**" และเลือกดัชนี "**Linux Index**" สำหรับฟิลด์ที่ชื่อว่า "**Index Set**" ยืนยันการเลือกของคุณ



**หมายเหตุ: ข้อความที่สอดคล้องกับสตรีมนี้จะถูกรวมอยู่ใน "สตรีมเริ่มต้น" ด้วย เว้นแต่คุณจะเลือกตัวเลือก "ลบการจับคู่จาก 'สตรีมเริ่มต้น'"**



![Image](assets/fr/002.webp)



จากนั้น ในการตั้งค่า steam ของคุณ ให้คลิกที่ปุ่ม "**เพิ่มกฎสตรีม**" เพื่อเพิ่มกฎการกำหนดเส้นทางข้อความใหม่ หากคุณไม่พบหน้าต่างนี้ ให้คลิกที่ "**สตรีม**" ในเมนู จากนั้นในบรรทัดที่ตรงกับสตรีมของคุณ ให้คลิกที่ "**เพิ่มเติม**" แล้ว "**จัดการกฎ**"



เลือกประเภท "**match input**" และเลือก **Rsyslog input in UDP** ที่สร้างไว้ก่อนหน้านี้ ยืนยันด้วยปุ่ม "**Create Rule**" ข้อความทั้งหมดที่ส่งไปยัง Input ใหม่ของเราจะถูกส่งไปยัง Index สำหรับ Linux.



![Image](assets/fr/003.webp)



Stream ใหม่ของคุณควรปรากฏในรายการและควรอยู่ในสถานะ "**Running**" แบนด์วิดท์ข้อความแสดง "**0 msg/s**" เนื่องจากขณะนี้เราไม่ได้ส่งบันทึกใด ๆ ไปยังอินพุต Rsyslog UDP หากต้องการดูบันทึกของ Stream เพียงคลิกที่ชื่อของมัน



![Image](assets/fr/004.webp)



**ทุกอย่างพร้อมแล้วทางฝั่ง Graylog** ขั้นตอนต่อไปคือการกำหนดค่าของเครื่อง Linux



### III. การติดตั้งและกำหนดค่า Rsyslog บน Linux



เข้าสู่ระบบเครื่อง Linux ไม่ว่าจะเป็นในเครื่องหรือระยะไกล และใช้บัญชีผู้ใช้ที่มีสิทธิ์ในการยกระดับสิทธิ์ (ผ่าน sudo) มิฉะนั้น ให้ใช้บัญชี "root" โดยตรง



#### A. การติดตั้งแพ็กเกจ Rsyslog



เริ่มต้นด้วยการอัปเดตแคชแพ็กเกจและติดตั้งแพ็กเกจที่ชื่อ "**rsyslog**"



```
sudo apt-get update
sudo apt-get install rsyslog
```



จากนั้นตรวจสอบสถานะของบริการ ในกรณีส่วนใหญ่ มันจะทำงานอยู่แล้ว



```
sudo systemctl status rsyslog
```



#### B. การกำหนดค่า Rsyslog



Rsyslog มีไฟล์การตั้งค่าหลักที่ตั้งอยู่ที่นี่:



```
/etc/rsyslog.conf
```



นอกจากนี้ ไดเรกทอรี "**/etc/rsyslog.d/**" ถูกใช้เพื่อเก็บไฟล์การตั้งค่าเพิ่มเติมสำหรับ Rsyslog ในไฟล์การตั้งค่าหลัก จะมีคำสั่ง Include เพื่อดึงเข้าไฟล์ "**.conf**" ทั้งหมดที่อยู่ในไดเรกทอรีนี้ ซึ่งทำให้สามารถมีไฟล์เพิ่มเติมสำหรับการตั้งค่า Rsyslog ได้โดยไม่ต้องแก้ไขไฟล์หลัก



ในไดเรกทอรีนี้ คุณต้องใช้ตัวเลขเพื่อกำหนดลำดับการโหลด เพราะไฟล์จะถูกโหลดตามลำดับตัวอักษร การเพิ่มคำนำหน้าตัวเลขช่วยให้คุณจัดการลำดับความสำคัญระหว่างไฟล์การกำหนดค่าหลายไฟล์ได้ ที่นี่เรามีไฟล์เพิ่มเติมเพียงไฟล์เดียว ดังนั้นจึงไม่เป็นปัญหา



ในไดเรกทอรีนี้ เราจะสร้างไฟล์ที่ชื่อ "**10-graylog.conf**":



```
sudo nano /etc/rsyslog.d/10-graylog.conf
```



ในไฟล์นี้ ให้แทรกบรรทัดนี้:



```
*.* @192.168.10.220:12514;RSYSLOG_SyslogProtocol23Format
```



นี่คือวิธีการตีความบรรทัดนี้:





- `*.*`: หมายถึงส่งบันทึก Syslog ทั้งหมดจากเครื่อง Linux ไปยัง Graylog.
- `@`: ระบุว่าการส่งข้อมูลดำเนินการในรูปแบบ UDP คุณต้องระบุ "**@@**" เพื่อเปลี่ยนเป็น TCP
- `192.168.10.220:12514`: ระบุที่อยู่ของเซิร์ฟเวอร์ Graylog และพอร์ตที่ใช้ในการส่งบันทึก (สอดคล้องกับ Input)
- `RSYSLOG_SyslogProtocol23Format`: สอดคล้องกับรูปแบบของข้อความที่จะส่งไปยัง Graylog.



เมื่อเสร็จแล้ว ให้บันทึกไฟล์และรีสตาร์ท Rsyslog.



```
sudo systemctl restart rsyslog.service
```



หลังจากดำเนินการนี้ ข้อความแรกควรมาถึงเซิร์ฟเวอร์ Graylog ของคุณ!



### IV. การแสดงผลบันทึกของ Linux ใน Graylog



จาก Graylog คุณสามารถคลิกที่ "**Streams**" และเลือกสตรีมใหม่ของคุณเพื่อดูข้อความที่เกี่ยวข้อง หรือคลิกที่ "**Search**" และเลือก Steam ของคุณแล้วเริ่มการค้นหา



นี่คือคุณสมบัติหลักบางประการของ Interface:



**1** - เลือกช่วงเวลาที่จะแสดงข้อความ โดยค่าเริ่มต้นจะแสดงข้อความจาก 5 นาทีล่าสุด



**2** - เลือกสตรีมที่จะให้แสดง



**3** - เปิดใช้งานการรีเฟรชอัตโนมัติของรายการข้อความ (เช่น ทุก ๆ 5 วินาที) มิฉะนั้นจะเป็นแบบคงที่และคุณจะต้องรีเฟรชด้วยตนเอง



**4** - คลิกที่แว่นขยายเพื่อเริ่มการค้นหาหลังจากแก้ไขช่วงเวลา, สตรีมหรือฟิลเตอร์



**5** - แถบป้อนข้อมูลเพื่อระบุการค้นหาของคุณ ที่นี่ฉันระบุ "**source:srv\-docker**" เพื่อแสดงเฉพาะบันทึกของเครื่องใหม่ที่ฉันเพิ่งตั้งค่า Rsyslog ไว้เท่านั้น

### V. การระบุความล้มเหลวของการเชื่อมต่อ SSH



จุดแข็งของ Graylog อยู่ที่ความสามารถในการจัดทำดัชนีบันทึกและเปิดให้สามารถค้นหาได้ตามเกณฑ์ต่างๆ: เครื่องต้นทาง, เวลาประทับ, เนื้อหาข้อความ, ฯลฯ... เราอาจกำลังมองหาการระบุการเชื่อมต่อ SSH ที่ล้มเหลว.



จากเครื่องระยะไกล (เช่น เซิร์ฟเวอร์ Graylog) ให้ลองเชื่อมต่อกับเซิร์ฟเวอร์ Linux ของคุณที่คุณเพิ่งกำหนดค่า Rsyslog ตัวอย่างเช่น:



```
ssh [email protected]
```



จากนั้นใส่ **username** และ **password** ที่ไม่ถูกต้องโดยเจตนา เพื่อให้เกิด **generate connection errors** ในไฟล์ "**/var/log/auth.log**" จะมีข้อความ generate log ที่คล้ายกับต่อไปนี้:



```
Failed password for invalid user it-connect from 192.168.10.199 port 50352 ssh2
```



คุณควรค้นหาข้อความเหล่านี้บน Graylog.



บน Graylog ให้ใช้ตัวกรองการค้นหาต่อไปนี้เพื่อแสดงเฉพาะข้อความที่ตรงกัน:



```
message:Failed password AND application_name:sshd
```



หากคุณมีเซิร์ฟเวอร์หลายเครื่องและต้องการวิเคราะห์บันทึกของเซิร์ฟเวอร์เฉพาะ ให้ระบุชื่อเซิร์ฟเวอร์นั้นเพิ่มเติม:



```
message:Failed password AND application_name:sshd AND source:srv\-docker
```



นี่คือภาพรวมของผลลัพธ์บนเครื่องที่ฉันสร้างข้อผิดพลาดในการเชื่อมต่อหลายครั้ง ในเวลาต่างๆ ของวัน:



![Image](assets/fr/009.webp)



ความพยายามเชื่อมต่อที่ไม่สำเร็จเกิดขึ้นจากเครื่องที่มีที่อยู่ IP "**192.168.10.199**" หากคุณต้องการทราบข้อมูลเพิ่มเติมเกี่ยวกับกิจกรรมของโฮสต์นี้ คุณสามารถ **ค้นหาที่อยู่ IP นี้** Graylog จะส่งออกบันทึกทั้งหมดที่มีการอ้างอิงถึงที่อยู่ IP นี้ บนโฮสต์ทั้งหมด (สำหรับโฮสต์ที่มีการกำหนดค่าการส่งบันทึก)



ในกรณีนี้ ตัวกรองที่ใช้ได้คือ:



```
message:"192.168.10.199"
```



เราได้รับผลลัพธ์เพิ่มเติม (ไม่น่าแปลกใจ เนื่องจากเราไม่ได้กรองบนแอปพลิเคชัน SSH):



![Image](assets/fr/008.webp)
