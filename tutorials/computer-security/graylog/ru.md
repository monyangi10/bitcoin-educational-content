## Развертывание Graylog на Debian 12



### I. Презентация



Graylog - это решение с открытым исходным кодом, предназначенное для централизации, хранения и анализа журналов с ваших машин и сетевых устройств в режиме реального времени. В этом руководстве мы узнаем, как установить бесплатную версию Graylog на машину Debian 12!



В информационной системе каждый **сервер**, работающий под управлением **Linux** или **Windows**, и каждое **сетевое устройство** (коммутатор, маршрутизатор, брандмауэр и т. д.) **генерируют свои собственные журналы**, хранящиеся локально. Если журналы хранятся локально на каждой машине, анализ и корреляция событий очень затруднены... Именно здесь на помощь приходит **Graylog**. Он будет действовать как **поглотитель логов**, то есть **все ваши машины будут отправлять ему свои журналы** (например, через syslog). Затем Graylog будет **хранить и индексировать эти журналы**, позволяя вам выполнять глобальный поиск и создавать оповещения.



Graylog - это инструмент для анализа и мониторинга, который облегчает выявление подозрительного поведения и различных проблем (стабильность, производительность и т.д.).




![Image](assets/fr/034.webp)



**Примечание: бесплатная версия **Graylog Open** не является SIEM, как Wazuh, особенно потому, что в ней отсутствуют реальные функции обнаружения вторжений.



### II. Пререквизиты



В основе **стека Graylog** лежит **несколько компонентов**, которые нам нужно будет установить и настроить. Здесь мы установим все компоненты на один сервер, но можно создавать кластеры на базе нескольких узлов и распределять роли между несколькими серверами. Для целей этого руководства мы установим **Graylog 6.1**, самую последнюю версию на сегодняшний день.





- MongoDB 7**, текущая рекомендуемая версия для Graylog (минимум 5.0.7, максимум 7.x)
- OpenSearch**, Fork Elasticsearch с открытым исходным кодом, созданный Amazon (минимум 1.1.x, максимум 2.15.x)
- OpenJDK 17**



Сервер **Graylog** работает на **Debian 12**, но возможна установка и на другие дистрибутивы, в том числе через Docker. Виртуальная машина оснащена **8 ГБ оперативной памяти** и **256 ГБ дискового пространства**, чтобы хватило ресурсов для всех компонентов (в противном случае это может существенно повлиять на производительность). Однако я привожу эти данные в качестве приблизительного ориентира, поскольку **размер архитектуры Graylog зависит от объема обрабатываемой информации**. Graylog может обрабатывать как 30 МБ или 300 МБ данных в день, так и 300 ГБ данных в день... Это **масштабируемое решение**, способное обрабатывать **терабайты журналов** (см. [эту страницу](https://go2docs.graylog.org/current/planning_your_deployment/planning_your_deployment.html?tocpath=Plan%20Your%20Deployment%7C_____0)).

#### A. Установка MongoDB



После этого мы начнем установку MongoDB. Загрузите ключ GPG, соответствующий репозиторию MongoDB:

```
curl -fsSL https://www.mongodb.org/static/pgp/server-6.0.asc | sudo gpg -o /usr/share/keyrings/mongodb-server-6.0.gpg --dearmor
```



Затем добавьте репозиторий MongoDB 6 на машину Debian 12:



```
echo "deb [ signed-by=/usr/share/keyrings/mongodb-server-6.0.gpg] http://repo.mongodb.org/apt/debian bullseye/mongodb-org/6.0 main" | sudo tee /etc/apt/sources.list.d/mongodb-org-6.0.list
```



Далее мы обновим кэш пакетов и попытаемся установить MongoDB:



```
sudo apt-get update
sudo apt-get install -y mongodb-org
```



MongoDB не может быть установлен, потому что отсутствует зависимость: **libssl1.1**. Нам придётся установить этот пакет вручную, прежде чем мы сможем продолжить, потому что Debian 12 не имеет его в своих репозиториях.



```

```



Мы загрузим DEB-пакет под названием "**libssl1.1_1.1.1f-1ubuntu2.23_amd64.deb**" (последняя версия) с помощью команды **wget**, а затем установим его с помощью команды **dpkg**. Это приведет к появлению следующих двух команд:



```
wget http://archive.ubuntu.com/ubuntu/pool/main/o/openssl/libssl1.1_1.1.1f-1ubuntu2.23_amd64.deb
sudo dpkg -i libssl1.1_1.1.1f-1ubuntu2.23_amd64.deb
```

#### B. Установка OpenSearch



Перейдем к установке OpenSearch на сервер. Следующая команда добавляет ключ подписи для пакетов OpenSearch:



```
curl -o- https://artifacts.opensearch.org/publickeys/opensearch.pgp | sudo gpg --dearmor --batch --yes -o /usr/share/keyrings/opensearch-keyring
```



Затем добавьте репозиторий OpenSearch, чтобы впоследствии мы могли загрузить пакет с помощью **apt**:



```
echo "deb [signed-by=/usr/share/keyrings/opensearch-keyring] https://artifacts.opensearch.org/releases/bundle/opensearch/2.x/apt stable main" | sudo tee /etc/apt/sources.list.d/opensearch-2.x.list
```



Обновите кэш пакетов:



```
sudo apt-get update
```



Затем **установите OpenSearch**, позаботившись о том, чтобы **определить пароль по умолчанию для учетной записи Admin** вашего экземпляра. Здесь пароль "**IT-Connect2024!**", но замените это значение на надежный пароль. **Избегайте слабых паролей**, таких как "**P@ssword123**", и используйте не менее **8 символов**, по крайней мере по одному символу каждого типа (строчный, прописной, цифра и специальный символ), иначе в конце установки будет выдана ошибка. **Это обязательное условие, начиная с версии OpenSearch 2.12.**



```
sudo env OPENSEARCH_INITIAL_ADMIN_PASSWORD=IT-Connect2024! apt-get install opensearch
```



Пожалуйста, будьте терпеливы во время установки...



Когда вы закончите, уделите время минимальной настройке. Откройте файл конфигурации в формате YAML:



```
sudo nano /etc/opensearch/opensearch.yml
```



Когда файл открыт, установите следующие параметры:



```
cluster.name: graylog
node.name: ${HOSTNAME}
path.data: /var/lib/opensearch
path.logs: /var/log/opensearch
discovery.type: single-node
network.host: 127.0.0.1
action.auto_create_index: false
plugins.security.disabled: true
```



Эта конфигурация OpenSearch предназначена для настройки одного узла. Здесь приведены пояснения к различным параметрам, которые мы используем:





- cluster.name: graylog**: этот параметр называет кластер OpenSearch с именем "**graylog**".
- node.name: ${HOSTNAME}**: имя узла задается динамически, чтобы соответствовать имени локальной машины Linux. Даже если у нас всего один узел, важно правильно его назвать.
- path.data: /var/lib/opensearch**: этот путь указывает, где OpenSearch хранит свои данные на локальной машине, в данном случае в "**/var/lib/opensearch**".
- path.logs: /var/log/opensearch**: этот путь определяет, где хранятся файлы журналов OpenSearch, здесь в "**/var/log/opensearch**".
- discovery.type: single-node**: этот параметр настраивает OpenSearch на работу с одним узлом, поэтому выбирается опция "**single-node**".
- network.host: 127.0.0.1**: эта конфигурация гарантирует, что OpenSearch будет прослушивать только локальную петлю Interface, что вполне достаточно, поскольку он находится на том же сервере, что и Graylog.
- action.auto_create_index: false**: отключив автоматическое создание индекса, OpenSearch не будет автоматически создавать индекс, когда документ отправляется без существующего индекса.
- plugins.security.disabled: true**: эта опция деактивирует плагин безопасности OpenSearch, что означает отсутствие аутентификации, управления доступом и шифрования связи. Эта настройка экономит время при настройке Graylog, но ее следует избегать в производстве (см. [эту страницу](https://opensearch.org/docs/1.0/security-plugin/index/)).



Некоторые опции уже присутствуют, поэтому для их активации нужно просто убрать символ "#", а затем указать свое значение. Если вы не можете найти опцию, вы можете объявить ее непосредственно в конце файла.



![Image](assets/fr/023.webp)

#### C. Настройка Java (JVM)



Вам нужно настроить виртуальную машину Java, используемую OpenSearch, чтобы отрегулировать объем памяти, который может использовать эта служба. Отредактируйте следующий файл конфигурации:



```
sudo nano /etc/opensearch/jvm.options
```



В конфигурации, представленной здесь, **OpenSearch начнет работу с 4 ГБ выделенной памяти и сможет увеличить ее до 4 ГБ**, поэтому во время работы не будет никаких колебаний памяти. Здесь конфигурация учитывает тот факт, что виртуальная машина имеет в общей сложности **8 ГБ RAM**. Оба параметра должны иметь одинаковое значение. Это означает замену строк:



```
-Xms1g
-Xmx1g
```



С этими строками:



```
-Xms4g
-Xmx4g
```



Вот изображение модификации, которую нужно сделать:



![Image](assets/fr/022.webp)



Закройте этот файл после сохранения.



Кроме того, нам необходимо проверить конфигурацию параметра "**max_map_count**" в ядре Linux. Он определяет предельное количество областей памяти, отображаемых на один процесс, чтобы удовлетворить потребности нашего приложения. **OpenSearch**, как и Elasticsearch**, рекомендует установить это значение на "262144", чтобы избежать ошибок управления памятью.



В принципе, на свежеустановленной машине Debian 12 значение уже правильное. Но давайте проверим. Выполните эту команду:



```
cat /proc/sys/vm/max_map_count
```



Если вы получите значение, отличное от "**262144**", выполните следующую команду, в противном случае это необязательно.



```
sudo sysctl -w vm.max_map_count=262144
```



Наконец, включите автозапуск OpenSearch и запустите соответствующую службу.



```
sudo systemctl daemon-reload
sudo systemctl enable opensearch
sudo systemctl restart opensearch
```



Если вы отобразите состояние системы, вы должны увидеть процесс Java с 4 ГБ оперативной памяти.



```
top
```

Давайте начнем с настройки этих двух параметров:





- password_secret**: этот параметр используется для определения ключа, используемого Graylog для защиты хранения пользовательских паролей (в духе ключа засолки). Этот ключ должен быть **уникальным** и **случайным**.

## Отправка журналов в Graylog с помощью rsyslog



### I. Презентация



В этой второй части мы узнаем, как настроить Linux-машину на отправку журналов на сервер Graylog. Для этого мы установим и настроим Rsyslog в системе.



### II. Настройка Graylog для получения журналов Linux



Начнем с настройки Graylog. Для этого необходимо выполнить три шага:





- Создание **входа** для создания точки входа, позволяющей машинам Linux **отправлять журналы Syslog по UDP**.
- Создание нового **индекса** для хранения и **индексации всех журналов Linux**.
- Создание **потока** для **перенаправления** журналов, полученных Graylog, в новый индекс Linux.



#### A. Создание входа для Syslog



Войдите в систему Graylog Interface, нажмите на "**Система**" в меню, а затем на "**Входы**". В выпадающем списке выберите "**Syslog UDP**", затем нажмите на кнопку "**Launch new input**". Также можно создать вход Syslog по TCP, но для этого необходимо использовать сертификат TLS: это плюс для безопасности, но в данной статье он не рассматривается.



![Image](assets/fr/001.webp)



На экране появится мастер. Начните с присвоения имени этому входу, например "**Graylog_UDP_Rsyslog_Linux**", и выберите порт. По умолчанию используется порт "**514**", но вы можете настроить его. В данном случае выбран порт "**12514**".



![Image](assets/fr/016.webp)



Вы также можете отметить опцию "**Сохранить полное сообщение**", чтобы сохранить полное сообщение журнала в Graylog. Нажмите на кнопку "**Запустить ввод**".



![Image](assets/fr/017.webp)



Новый вход был создан и теперь активен. Теперь Graylog может получать журналы Syslog на порт 12514/UDP, но мы еще не закончили настройку приложения.



![Image](assets/fr/018.webp)


**Примечание: один вход может использоваться для хранения журналов с нескольких Linux-машин.



#### B. Создайте новый индекс Linux



Нам нужно создать индекс в Graylog для хранения журналов с машин Linux. Индекс** в Graylog - это структура хранения, содержащая полученные журналы, т.е. полученные сообщения. Graylog использует OpenSearch в качестве механизма хранения, и сообщения индексируются, чтобы обеспечить быстрый и эффективный поиск.



В Graylog щелкните в меню "**Система**", затем "**Индексы**". На появившейся странице нажмите "**Создать набор индексов**".



![Image](assets/fr/005.webp)



Назовите этот индекс, например, "**Linux Index**", добавьте описание и префикс перед подтверждением. Здесь мы будем **хранить все журналы Linux в этом индексе**. Также можно создать специальные индексы для хранения только определенных журналов (только журналы [SSH] (https://www.it-connect.fr/cours/comprendre-et-maitriser-ssh/ "SSH"), журналы веб-служб и т. д.).



![Image](assets/fr/006.webp)



Теперь нам нужно создать новый поток, который будет направлять сообщения в этот индекс.



#### C. Создайте поток



Чтобы создать новый поток, нажмите на "**Потоки**" в главном меню Graylog. Затем нажмите на кнопку "**Создать поток**" справа. В появившемся окне назовите поток, например, "**Linux Stream**" и выберите индекс "**Linux Index**" для поля "**Index Set**". Подтвердите свой выбор.



**Примечание: сообщения, соответствующие этому потоку, будут также включены в "**Поток по умолчанию**", если вы не отметите опцию "**Удалить совпадения из "Потока по умолчанию**"".



![Image](assets/fr/002.webp)



Затем в настройках steam нажмите на кнопку "**Добавить правило потока**", чтобы добавить новое правило маршрутизации сообщений. Если вы не можете найти это окно, нажмите на "**Потоки**" в меню, затем в строке, соответствующей вашему потоку, нажмите на "**Больше**", а затем на "**Управление правилами**".



Выберите тип "**match input**" и выберите ранее созданный **Rsyslog input in UDP**. Подтвердите выбор кнопкой "**Создать правило**". Теперь все сообщения на наш новый вход будут отправляться в Index для Linux.



![Image](assets/fr/003.webp)



Ваш новый поток должен появиться в списке и находиться в состоянии "**Running**". Пропускная способность сообщения показывает "**0 мс/с**", так как в настоящее время мы не отправляем никаких журналов на UDP-вход Rsyslog. Чтобы просмотреть журналы потока, просто щелкните на его имени.



![Image](assets/fr/004.webp)



**Все готово на стороне Graylog**. Следующим шагом будет настройка машины Linux.



### III. Установка и настройка Rsyslog в Linux



Войдите на машину Linux, локально или удаленно, и используйте учетную запись пользователя с правами на повышение привилегий (через sudo). В противном случае используйте учетную запись "root" напрямую.



#### A. Установка пакета Rsyslog



Начните с обновления кэша пакетов и установки пакета с именем "**rsyslog**".



```
sudo apt-get update
sudo apt-get install rsyslog
```



Затем проверьте состояние службы. В большинстве случаев она уже запущена.



```
sudo systemctl status rsyslog
```



#### B. Настройка Rsyslog



Rsyslog имеет основной файл конфигурации, расположенный здесь:



```
/etc/rsyslog.conf
```



Кроме того, каталог "**/etc/rsyslog.d/**" используется для хранения дополнительных конфигурационных файлов Rsyslog. В главном конфигурационном файле есть директива Include для импорта всех файлов "**.conf**", расположенных в этом каталоге. Это позволяет иметь дополнительные файлы для настройки Rsyslog без изменения основного файла.



В этом каталоге необходимо использовать цифры для определения порядка загрузки, поскольку файлы загружаются в алфавитном порядке. Добавление числового префикса позволяет управлять приоритетом между несколькими конфигурационными файлами. Здесь у нас только один дополнительный файл, так что это не проблема.



В этой директории мы создадим файл под названием "**10-graylog.conf**":



```
sudo nano /etc/rsyslog.d/10-graylog.conf
```



В этом файле вставьте следующую строку:



```
*.* @192.168.10.220:12514;RSYSLOG_SyslogProtocol23Format
```



Вот как следует интерпретировать эту строку:





- `*.*`: означает отправку всех журналов Syslog с Linux-машины в Graylog.
- `@`: указывает, что транспортировка осуществляется в UDP. Для переключения на TCP необходимо указать "**@@**".
- `192.168.10.220:12514`: указывает Address сервера Graylog и порт, на который отправляются журналы (соответствует Input).
- `RSYSLOG_SyslogProtocol23Format`: соответствует формату сообщений, которые будут отправляться в Graylog.



После этого сохраните файл и перезапустите Rsyslog.



```
sudo systemctl restart rsyslog.service
```



После этих действий на ваш сервер Graylog должны прийти первые сообщения!



### IV. Отображение журналов Linux в Graylog



В Graylog вы можете нажать на "**Streams**" и выбрать новый поток, чтобы просмотреть связанные с ним сообщения. Кроме того, нажмите на "**Поиск**", выберите свой Steam и запустите поиск.



Вот некоторые ключевые особенности Interface:



**1** - Выберите период, за который будут отображаться сообщения. По умолчанию отображаются сообщения за последние 5 минут.



**2** - Выберите поток(и), который(ые) будет(ут) отображаться.



**3** - Включите автоматическое обновление списка сообщений (например, каждые 5 секунд). В противном случае он будет статичным, и вам придется обновлять его вручную.



**4** - Нажмите на лупу, чтобы запустить поиск после изменения периода, потока или фильтра.



**5** - Строка ввода для указания фильтров поиска. Здесь я указываю "**source:srv\-docker**", чтобы отобразить только журналы новой машины, на которой я только что настроил Rsyslog.



Журналы отправляются машиной Linux:

### V. Идентификация сбоя соединения SSH



Сила Graylog заключается в способности индексировать журналы и осуществлять поиск по различным критериям: машина-источник, Timestamp, содержание сообщений и т. д.. Мы можем искать неудачные SSH-соединения.



С удаленной машины (например, с сервера Graylog) попробуйте подключиться к вашему серверу Linux, на котором вы только что настроили Rsyslog. Например:



```
ssh [email protected]
```



Затем намеренно введите неправильные **имя пользователя** и **пароль**, чтобы **generate ошибки подключения**. В файле "**/var/log/auth.log**" появятся сообщения журнала generate, похожие на следующие:



```
Failed password for invalid user it-connect from 192.168.10.199 port 50352 ssh2
```



Вы должны найти эти сообщения на Graylog.



В Graylog используйте следующий фильтр поиска, чтобы отобразить только подходящие сообщения:



```
message:Failed password AND application_name:sshd
```



Если у вас несколько серверов и вы хотите проанализировать журналы определенного сервера, дополнительно укажите его имя:



```
message:Failed password AND application_name:sshd AND source:srv\-docker
```



Вот обзор результатов на машине, где я выдал несколько ошибок подключения в разное время суток:



![Image](assets/fr/009.webp)



Неудачные попытки подключения предпринимаются с машины с IP Address "**192.168.10.199**". Если вы хотите узнать больше об активности этого узла, вы можете **поискать этот IP Address**. Graylog выведет все журналы, в которых упоминается этот IP Address, на всех хостах (для которых настроена отправка журналов).



В этом случае используемый фильтр может быть:



```
message:"192.168.10.199"
```
