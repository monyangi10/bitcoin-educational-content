## استقرار Graylog بر روی Debian 12



### I. ارائه



Graylog یک راه‌حل "log sink" متن‌باز است که برای متمرکز کردن، ذخیره و تحلیل لاگ‌ها از ماشین‌ها و دستگاه‌های شبکه شما به صورت لحظه‌ای طراحی شده است. در این آموزش، یاد خواهیم گرفت که چگونه نسخه رایگان Graylog را بر روی یک ماشین Debian 12 نصب کنیم!



در یک سیستم اطلاعاتی، هر **سرور**، چه با **لینوکس** و چه با **ویندوز** اجرا شود، و هر **دستگاه شبکه** (سوئیچ، روتر، فایروال، و غیره...) **لاگ‌های خود را تولید می‌کند** که به صورت محلی ذخیره می‌شوند. با ذخیره لاگ‌ها به صورت محلی روی هر ماشین، تحلیل و همبستگی رویدادها بسیار دشوار است... اینجاست که **Graylog** وارد می‌شود. این ابزار به عنوان یک **محل جمع‌آوری لاگ** عمل می‌کند، به این معنی که **تمام ماشین‌های شما لاگ‌های خود را به آن ارسال خواهند کرد** (به عنوان مثال از طریق syslog). سپس Graylog این لاگ‌ها را **ذخیره و فهرست‌بندی می‌کند** و به شما اجازه می‌دهد تا جستجوهای جهانی انجام داده و هشدارها ایجاد کنید.



Graylog یک ابزار تحلیل و نظارت است که شناسایی رفتارهای مشکوک و مشکلات مختلف (پایداری، عملکرد و غیره) را آسان‌تر می‌کند.




![Image](assets/fr/034.webp)



**توجه: نسخه رایگان، **Graylog Open**، مانند Wazuh یک SIEM نیست، به‌ویژه به دلیل عدم وجود عملکردهای واقعی تشخیص نفوذ.



### II. پیش‌نیازها



**پشته Graylog** بر اساس **چندین مؤلفه** است که باید نصب و پیکربندی کنیم. در اینجا، همه مؤلفه‌ها را روی همان سرور نصب خواهیم کرد، اما امکان ایجاد کلاسترهایی بر اساس چندین نود و توزیع نقش‌ها در چندین سرور وجود دارد. برای اهداف این آموزش، ما **Graylog 6.1**، جدیدترین نسخه تا به امروز را نصب خواهیم کرد.





- MongoDB 7**، نسخه فعلی توصیه‌شده برای Graylog (حداقل 5.0.7، حداکثر 7.x)
- OpenSearch**، یک Fork متن‌باز از Elasticsearch که توسط آمازون ایجاد شده است (حداقل 1.1.x، حداکثر 2.15.x)
- OpenJDK 17**



سرور **Graylog** بر روی **Debian 12** اجرا می‌شود، اما نصب آن بر روی توزیع‌های دیگر، از جمله از طریق Docker، نیز ممکن است. ماشین مجازی با **8 گیگابایت رم** و **256 گیگابایت فضای دیسک** مجهز شده است تا منابع کافی برای همه اجزا فراهم شود (در غیر این صورت می‌تواند تأثیر قابل توجهی بر عملکرد داشته باشد). با این حال، این را به عنوان یک راهنمای تقریبی ارائه می‌دهم، زیرا **اندازه‌گیری معماری Graylog به میزان اطلاعاتی که باید پردازش شود بستگی دارد**. Graylog می‌تواند 30 مگابایت یا 300 مگابایت داده در روز، و همچنین 300 گیگابایت داده در روز را پردازش کند... این یک **راه‌حل مقیاس‌پذیر** است که قادر به مدیریت **ترابایت‌های لاگ** می‌باشد (به [این صفحه](https://go2docs.graylog.org/current/planning_your_deployment/planning_your_deployment.html?tocpath=Plan%20Your%20Deployment%7C_____0) مراجعه کنید).



![Image](assets/fr/032.webp)



منبع: Graylog



قبل از شروع پیکربندی، یک IP استاتیک Address به ماشین Graylog اختصاص دهید و آخرین به‌روزرسانی‌ها را نصب کنید. حتماً منطقه زمانی ماشین محلی را تنظیم کرده و یک سرور NTP برای همگام‌سازی تاریخ و زمان تعریف کنید. در اینجا فرمانی که باید اجرا شود:



```
sudo timedatectl set-timezone Europe/Paris
```



**توجه: **نصب OpenSearch اختیاری است** اگر از **گرایلگ دیتا نود** استفاده کنید.



### III نصب مرحله به مرحله Graylog



بیایید با به‌روزرسانی حافظه پنهان بسته و نصب ابزارهایی که برای مراحل بعدی نیاز داریم، شروع کنیم.



```
sudo apt-get update
sudo apt-get install curl lsb-release ca-certificates gnupg2 pwgen
```



![Image](assets/fr/030.webp)



#### A. نصب MongoDB



پس از انجام این کار، نصب MongoDB را شروع خواهیم کرد. کلید GPG مربوط به مخزن MongoDB را دانلود کنید:



```
curl -fsSL https://www.mongodb.org/static/pgp/server-6.0.asc | sudo gpg -o /usr/share/keyrings/mongodb-server-6.0.gpg --dearmor
```



سپس مخزن MongoDB 6 را به ماشین Debian 12 اضافه کنید:



```
echo "deb [ signed-by=/usr/share/keyrings/mongodb-server-6.0.gpg] http://repo.mongodb.org/apt/debian bullseye/mongodb-org/6.0 main" | sudo tee /etc/apt/sources.list.d/mongodb-org-6.0.list
```



سپس، حافظه نهان بسته را به‌روزرسانی کرده و تلاش می‌کنیم MongoDB را نصب کنیم:



```
sudo apt-get update
sudo apt-get install -y mongodb-org
```



نصب MongoDB امکان‌پذیر نیست زیرا یک وابستگی وجود ندارد: **libssl1.1**. باید این بسته را به صورت دستی نصب کنیم قبل از اینکه بتوانیم ادامه دهیم، زیرا Debian 12 آن را در مخازن خود ندارد.



```
Les paquets suivants contiennent des dépendances non satisfaites :
mongodb-org-mongos: Dépend: libssl1.1 (>= 1.1.1) mais il n'est pas installable
mongodb-org-server: Dépend: libssl1.1 (>= 1.1.1) mais il n'est pas installable
E: Impossible de corriger les problèmes, des paquets défectueux sont en mode « garder en l'état ».
```



ما قصد داریم بسته DEB به نام "**libssl1.1_1.1.1f-1ubuntu2.23_amd64.deb**" (آخرین نسخه) را با فرمان **wget** دانلود کنیم، سپس آن را با فرمان **dpkg** نصب کنیم. این دو فرمان به صورت زیر هستند:



```
wget http://archive.ubuntu.com/ubuntu/pool/main/o/openssl/libssl1.1_1.1.1f-1ubuntu2.23_amd64.deb
sudo dpkg -i libssl1.1_1.1.1f-1ubuntu2.23_amd64.deb
```



![Image](assets/fr/028.webp)



نصب مجدد MongoDB:



```
sudo apt-get install -y mongodb-org
```



سپس سرویس MongoDB را مجدداً راه‌اندازی کنید و آن را طوری تنظیم کنید که به‌طور خودکار هنگام راه‌اندازی سرور Debian شروع به کار کند.



```
sudo systemctl daemon-reload
sudo systemctl enable mongod.service
sudo systemctl restart mongod.service
sudo systemctl --type=service --state=active | grep mongod
```



با نصب MongoDB، می‌توانیم به نصب مؤلفه بعدی بپردازیم.



#### ب. نصب OpenSearch



بیایید به نصب OpenSearch بر روی سرور بپردازیم. دستور زیر کلید امضای بسته‌های OpenSearch را اضافه می‌کند:



```
curl -o- https://artifacts.opensearch.org/publickeys/opensearch.pgp | sudo gpg --dearmor --batch --yes -o /usr/share/keyrings/opensearch-keyring
```



سپس مخزن OpenSearch را اضافه کنید تا بتوانیم بسته را با **apt** دانلود کنیم:



```
echo "deb [signed-by=/usr/share/keyrings/opensearch-keyring] https://artifacts.opensearch.org/releases/bundle/opensearch/2.x/apt stable main" | sudo tee /etc/apt/sources.list.d/opensearch-2.x.list
```



حافظه پنهان بسته خود را به‌روزرسانی کنید:



```
sudo apt-get update
```



سپس **OpenSearch را نصب کنید** و دقت کنید که **رمز عبور پیش‌فرض برای حساب Admin نمونه خود** را تعریف کنید. در اینجا، رمز عبور "**IT-Connect2024!**" است، اما این مقدار را با یک رمز عبور قوی جایگزین کنید. **از رمزهای عبور ضعیف** مانند "**P@ssword123**" اجتناب کنید و از حداقل **8 کاراکتر** با حداقل یک کاراکتر از هر نوع (حروف کوچک، حروف بزرگ، عدد و کاراکتر ویژه) استفاده کنید، در غیر این صورت در پایان نصب خطا رخ خواهد داد. **این یک پیش‌نیاز از نسخه OpenSearch 2.12 است.**



```
sudo env OPENSEARCH_INITIAL_ADMIN_PASSWORD=IT-Connect2024! apt-get install opensearch
```



لطفاً در طول نصب صبور باشید...



وقتی کارتان تمام شد، لحظه‌ای وقت بگذارید تا حداقل پیکربندی را انجام دهید. فایل پیکربندی را در قالب YAML باز کنید:



```
sudo nano /etc/opensearch/opensearch.yml
```



وقتی فایل باز است، گزینه‌های زیر را تنظیم کنید:



```
cluster.name: graylog
node.name: ${HOSTNAME}
path.data: /var/lib/opensearch
path.logs: /var/log/opensearch
discovery.type: single-node
network.host: 127.0.0.1
action.auto_create_index: false
plugins.security.disabled: true
```



این پیکربندی OpenSearch برای راه‌اندازی یک نود واحد طراحی شده است. در اینجا توضیحاتی درباره پارامترهای مختلفی که استفاده می‌کنیم آورده شده است:





- cluster.name: graylog**: این پارامتر نام کلاستر OpenSearch را با نام "**graylog**" تعیین می‌کند.
- node.name: ${HOSTNAME}**: نام گره به صورت پویا تنظیم می‌شود تا با نام ماشین محلی لینوکس مطابقت داشته باشد. حتی اگر فقط یک گره داشته باشیم، مهم است که آن را به درستی نام‌گذاری کنیم.
- مسیر path.data: /var/lib/opensearch**: این مسیر مشخص می‌کند که OpenSearch داده‌های خود را در کجا بر روی ماشین محلی ذخیره می‌کند، در این مورد در "**/var/lib/opensearch**".
- مسیر path.logs: /var/log/opensearch**: این مسیر تعیین می‌کند که فایل‌های لاگ OpenSearch کجا ذخیره می‌شوند، در اینجا در "**/var/log/opensearch**".
- discovery.type: single-node**: این پارامتر OpenSearch را برای کار با یک نود تنظیم می‌کند، بنابراین گزینه "**single-node**" انتخاب شده است.
- network.host: 127.0.0.1**: این پیکربندی تضمین می‌کند که OpenSearch فقط به حلقه محلی Interface خود گوش می‌دهد، که کافی است زیرا روی همان سرور با Graylog قرار دارد.
- action.auto_create_index: false**: با غیرفعال کردن ایجاد خودکار ایندکس، OpenSearch به‌طور خودکار ایندکس ایجاد نمی‌کند زمانی که یک سند بدون ایندکس موجود ارسال می‌شود.
- plugins.security.disabled: true**: این گزینه افزونه امنیتی OpenSearch را غیرفعال می‌کند، به این معنی که هیچ احراز هویت، مدیریت دسترسی یا رمزگذاری ارتباطی وجود نخواهد داشت. این تنظیم در زمان راه‌اندازی Graylog صرفه‌جویی می‌کند، اما باید در محیط تولید از آن اجتناب شود (به [این صفحه](https://opensearch.org/docs/1.0/security-plugin/index/) مراجعه کنید).



برخی گزینه‌ها از قبل موجود هستند، بنابراین تنها کافی است "#" را حذف کنید تا آن‌ها فعال شوند، سپس مقدار خود را مشخص کنید. اگر گزینه‌ای را پیدا نمی‌کنید، می‌توانید آن را مستقیماً در انتهای فایل اعلام کنید.



![Image](assets/fr/023.webp)



این فایل را ذخیره کرده و ببندید.



#### C. پیکربندی جاوا (JVM)



برای تنظیم مقدار حافظه‌ای که سرویس OpenSearch می‌تواند استفاده کند، باید ماشین مجازی جاوا (Java Virtual Machine) مورد استفاده توسط OpenSearch را پیکربندی کنید. فایل پیکربندی زیر را ویرایش کنید:



```
sudo nano /etc/opensearch/jvm.options
```



با پیکربندی که در اینجا پیاده‌سازی شده است، **OpenSearch با ۴ گیگابایت حافظه تخصیص‌یافته شروع به کار خواهد کرد و می‌تواند تا ۴ گیگابایت رشد کند**، بنابراین در طول عملیات هیچ تغییری در حافظه وجود نخواهد داشت. در اینجا، پیکربندی به این نکته توجه دارد که ماشین مجازی دارای **۸ گیگابایت رم** است. هر دو پارامتر باید دارای مقدار یکسانی باشند. این به معنای جایگزینی خطوط است:



```
-Xms1g
-Xmx1g
```



با این خطوط:



```
-Xms4g
-Xmx4g
```



تصویری از اصلاحاتی که باید انجام شود:



![Image](assets/fr/022.webp)



این فایل را پس از ذخیره کردن ببندید.



علاوه بر این، ما نیاز داریم که پیکربندی پارامتر "**max_map_count**" در هسته لینوکس را بررسی کنیم. این پارامتر محدودیت نواحی حافظه‌ای که به ازای هر فرآیند نگاشت می‌شوند را تعریف می‌کند، تا نیازهای برنامه ما را برآورده کند. **OpenSearch**، مانند **Elasticsearch**، توصیه می‌کند که این مقدار به "262144" تنظیم شود تا از خطاهای مدیریت حافظه جلوگیری شود.



به طور کلی، در یک ماشین Debian 12 تازه نصب شده، مقدار از قبل صحیح است. اما بیایید بررسی کنیم. این دستور را اجرا کنید:



```
cat /proc/sys/vm/max_map_count
```



اگر مقداری غیر از "**262144**" دریافت کردید، فرمان زیر را اجرا کنید، در غیر این صورت نیازی نیست.



```
sudo sysctl -w vm.max_map_count=262144
```



در نهایت، راه‌اندازی خودکار OpenSearch را فعال کرده و سرویس مرتبط را اجرا کنید.



```
sudo systemctl daemon-reload
sudo systemctl enable opensearch
sudo systemctl restart opensearch
```



اگر وضعیت سیستم خود را نمایش دهید، باید یک فرآیند جاوا با ۴ گیگابایت رم مشاهده کنید.



```
top
```

#### D. نصب Graylog



برای **نصب Graylog 6.1** در آخرین نسخه آن، ۴ دستور زیر را برای **دانلود و نصب Graylog Server** اجرا کنید:



```
wget https://packages.graylog2.org/repo/packages/graylog-6.1-repository_latest.deb
sudo dpkg -i graylog-6.1-repository_latest.deb
sudo apt-get update
sudo apt-get install graylog-server
```



وقتی این کار انجام شد، قبل از تلاش برای راه‌اندازی آن، باید تغییراتی در پیکربندی Graylog ایجاد کنیم.



بیایید با پیکربندی این دو گزینه شروع کنیم:





- **password_secret**: این پارامتر برای تعریف یک کلید استفاده می‌شود که توسط Graylog برای ایمن‌سازی ذخیره‌سازی رمزهای عبور کاربران استفاده می‌شود (در راستای یک کلید نمک‌گذاری). این کلید باید **منحصر به فرد** و **تصادفی** باشد.
- root_password_sha2**: این پارامتر مربوط به رمز عبور پیش‌فرض مدیر در Graylog است. این به صورت Hash SHA-256 ذخیره می‌شود.



ما با تولید یک کلید 96 کاراکتری برای پارامتر **password_secret** شروع خواهیم کرد:



```
pwgen -N 1 -s 96
wVSGYwOmwBIDmtQvGzSuBevWoXe0MWpNWCzhorBfvMMhia2zIjHguTbfl4uXZJdHOA0EEb1sOXJTZKINhIIBm3V57vwfQV59
```



مقدار بازگشتی را کپی کنید، سپس فایل پیکربندی Graylog را باز کنید:



```
sudo nano /etc/graylog/server/server.conf
```



کلید را در پارامتر **password_secret** جای‌گذاری کنید، به این صورت:



![Image](assets/fr/027.webp)



فایل را ذخیره کرده و ببندید.



سپس، باید رمز عبور برای حساب "**admin**" که به‌طور پیش‌فرض ایجاد شده است را تنظیم کنید. در فایل پیکربندی، باید Hash رمز عبور ذخیره شود، که به این معنی است که باید محاسبه شود. مثال زیر Hash رمز عبور "**LogsWells@**" را نشان می‌دهد: مقدار را با رمز عبور خود تطبیق دهید.



```
echo -n "PuitsDeLogs@" | shasum -a 256
6b297230efaa2905c9a746fb33a628f4d7aba4fa9d5c1b3daa6846c68e602d71
```



مقدار به‌دست‌آمده را به‌عنوان خروجی کپی کنید (بدون خط تیره در انتهای خط).



فایل پیکربندی Graylog را دوباره باز کنید:



```
sudo nano /etc/graylog/server/server.conf
```



مقدار را در گزینه **root_password_sha2** به این صورت جای‌گذاری کنید:



![Image](assets/fr/026.webp)



در حالی که در فایل پیکربندی هستید، گزینه "**http_bind_address**" را تنظیم کنید. مقدار "**0.0.0.0:9000**" را مشخص کنید تا وب Graylog's Interface از طریق هر IP سرور Address، روی پورت **9000** قابل دسترسی باشد.



![Image](assets/fr/024.webp)



سپس گزینه "**elasticsearch_hosts**" را به `http://127.0.0.1:9200` تنظیم کنید تا نمونه محلی OpenSearch ما اعلام شود. این ضروری است، زیرا ما از **Graylog Data Node** استفاده نمی‌کنیم. و بدون این گزینه، ادامه کار ممکن نخواهد بود...



![Image](assets/fr/025.webp)



فایل را ذخیره کرده و ببندید.



این دستور Graylog را فعال می‌کند تا در بوت بعدی به‌صورت خودکار شروع شود و بلافاصله سرور Graylog را راه‌اندازی می‌کند.



```
sudo systemctl enable --now graylog-server
```



پس از انجام این کار، سعی کنید از طریق یک مرورگر به Graylog متصل شوید. آدرس IP سرور Address (یا نام) و پورت 9000 را وارد کنید.



**برای اطلاع شما:**



نه چندان دور، یک پنجره احراز هویت مشابه با پنجره زیر زمانی که برای اولین بار به Graylog وارد می‌شدید ظاهر می‌شد. شما باید نام کاربری "**admin**" و رمز عبور خود را وارد می‌کردید. و سپس با ناراحتی متوجه می‌شدید که اتصال کار نمی‌کند.



![Image](assets/fr/031.webp)



لازم بود که به خط فرمان روی سرور Graylog برگردیم و لاگ‌ها را بررسی کنیم. سپس می‌توانستیم ببینیم که **برای اولین اتصال**، لازم است از **یک رمز عبور موقت** که در لاگ‌ها مشخص شده است، استفاده کنیم.



```
tail -f /var/log/graylog-server/server.log
```



![Image](assets/fr/021.webp)



سپس مجبور شدید با کاربر "**admin**" و رمز عبور موقت دوباره اتصال را امتحان کنید، و این به شما اجازه داد وارد شوید!



**این دیگر صدق نمی‌کند. تنها کاری که باید انجام دهید این است که با حساب کاربری مدیر خود و رمز عبوری که در خط فرمان تنظیم شده است وارد شوید.**

#### E. Graylog: ایجاد یک حساب کاربری جدید مدیر



به جای استفاده از حساب ادمین که به صورت بومی توسط Graylog ایجاد شده است، می‌توانید حساب مدیر شخصی خود را ایجاد کنید. روی منوی "**System**" کلیک کنید، سپس روی "**Users and Teams**" کلیک کنید تا دکمه "**Create user**" را بزنید. سپس فرم را پر کنید و نقش مدیر را به حساب خود اختصاص دهید.



![Image](assets/fr/020.webp)



یک حساب شخصی می‌تواند اطلاعات اضافی مانند نام و نام خانوادگی و ایمیل Address را شامل شود، برخلاف یک حساب مدیر بومی. علاوه بر این، این امر ردیابی بهتری را زمانی که هر فرد با یک حساب نام‌گذاری شده کار می‌کند، تضمین می‌کند.



## ارسال لاگ‌ها به Graylog با استفاده از rsyslog



### I. ارائه



در این بخش دوم، یاد خواهیم گرفت که چگونه یک ماشین لینوکس را برای ارسال لاگ‌های خود به یک سرور Graylog پیکربندی کنیم. برای انجام این کار، Rsyslog را بر روی سیستم نصب و پیکربندی خواهیم کرد.



### II. پیکربندی Graylog برای دریافت لاگ‌های لینوکس



ما با پیکربندی Graylog شروع خواهیم کرد. سه مرحله برای تکمیل وجود دارد:





- ایجاد یک **ورودی** برای ایجاد یک نقطه ورود که به ماشین‌های لینوکس اجازه می‌دهد **لاگ‌های Syslog را از طریق UDP ارسال کنند**.
- ایجاد یک **شاخص** جدید برای ذخیره و **فهرست‌بندی تمام لاگ‌های لینوکس**.
- ایجاد یک **Stream** برای **مسیر‌دهی** لاگ‌های دریافت شده توسط Graylog به ایندکس جدید لینوکس.



#### الف. ایجاد یک ورودی برای Syslog



وارد Graylog Interface شوید، در منو روی "**System**" کلیک کنید و سپس روی "**Inputs**" کلیک کنید. در لیست کشویی، "**Syslog UDP**" را انتخاب کنید و سپس روی دکمه‌ای که با "**Launch new input**" برچسب‌گذاری شده کلیک کنید. همچنین امکان ایجاد یک ورودی Syslog TCP وجود دارد، اما این نیاز به استفاده از یک گواهی TLS دارد: یک مزیت برای امنیت، اما در این مقاله پوشش داده نشده است.



![Image](assets/fr/001.webp)



یک جادوگر روی صفحه ظاهر خواهد شد. ابتدا به این ورودی یک نام بدهید، به عنوان مثال "**Graylog_UDP_Rsyslog_Linux**" و یک پورت انتخاب کنید. به طور پیش‌فرض، پورت "**514**" است، اما می‌توانید آن را سفارشی کنید. در اینجا، پورت "**12514**" انتخاب شده است.



![Image](assets/fr/016.webp)



شما همچنین می‌توانید گزینه "**ذخیره پیام کامل**" را انتخاب کنید تا پیام کامل لاگ را در Graylog ذخیره کنید. روی "**راه‌اندازی ورودی**" کلیک کنید.



![Image](assets/fr/017.webp)



ورودی جدید ایجاد شده و اکنون فعال است. Graylog اکنون می‌تواند لاگ‌های Syslog را روی پورت 12514/UDP دریافت کند، اما هنوز پیکربندی برنامه را به پایان نرسانده‌ایم.



![Image](assets/fr/018.webp)


**توجه: یک ورودی واحد می‌تواند برای ذخیره لاگ‌ها از چندین ماشین لینوکس استفاده شود.



#### ب. ایجاد یک شاخص جدید لینوکس



ما نیاز داریم که یک ایندکس در Graylog ایجاد کنیم تا لاگ‌های ماشین‌های لینوکس را ذخیره کنیم. یک **ایندکس** در Graylog یک ساختار ذخیره‌سازی است که شامل لاگ‌های دریافتی، یعنی پیام‌های دریافتی می‌باشد. Graylog از OpenSearch به عنوان موتور ذخیره‌سازی خود استفاده می‌کند و پیام‌ها برای جستجوهای سریع و کارآمد ایندکس می‌شوند.



از Graylog، در منو بر روی "**System**" کلیک کنید، سپس بر روی "**Indices**" کلیک کنید. در صفحه جدیدی که ظاهر می‌شود، بر روی "**Create index set**" کلیک کنید.



![Image](assets/fr/005.webp)



این شاخص را نام‌گذاری کنید، به عنوان مثال "**شاخص لینوکس**"، یک توضیح و پیشوند اضافه کنید، قبل از تأیید. در اینجا، ما **تمام لاگ‌های لینوکس را در این شاخص ذخیره خواهیم کرد**. همچنین امکان ایجاد شاخص‌های خاص برای ذخیره فقط برخی لاگ‌ها (فقط لاگ‌های [SSH] (https://www.it-connect.fr/cours/comprendre-et-maitriser-ssh/ "SSH")، لاگ‌های سرویس وب، و غیره) وجود دارد.



![Image](assets/fr/006.webp)



اکنون باید یک جریان جدید ایجاد کنیم تا پیام‌ها را به این شاخص هدایت کنیم.



#### ج. ایجاد یک جریان



برای ایجاد یک جریان جدید، در منوی اصلی Graylog روی "**Streams**" کلیک کنید. سپس روی دکمه "**Create stream**" در سمت راست کلیک کنید. در پنجره‌ای که ظاهر می‌شود، جریان را نام‌گذاری کنید، به عنوان مثال "**Linux Stream**" و شاخص "**Linux Index**" را برای فیلدی با نام "**Index Set**" انتخاب کنید. انتخاب خود را تأیید کنید.



**توجه: پیام‌های مربوط به این جریان نیز در "**جریان پیش‌فرض**" گنجانده خواهند شد، مگر اینکه گزینه "**حذف تطابق‌ها از 'جریان پیش‌فرض'**" را انتخاب کنید.**



![Image](assets/fr/002.webp)



سپس، در تنظیمات استیم خود، روی دکمه "**افزودن قانون استریم**" کلیک کنید تا یک قانون مسیریابی پیام جدید اضافه کنید. اگر نتوانستید این پنجره را پیدا کنید، در منو روی "**استریم‌ها**" کلیک کنید، سپس روی خط مربوط به استریم خود، روی "**بیشتر**" و سپس "**مدیریت قوانین**" کلیک کنید.



نوع "**match input**" را انتخاب کرده و **Rsyslog input in UDP** که قبلاً ایجاد شده است را انتخاب کنید. با دکمه "**Create Rule**" تأیید کنید. اکنون تمام پیام‌ها به ورودی جدید ما به ایندکس برای لینوکس ارسال خواهند شد.



![Image](assets/fr/003.webp)



استریم جدید شما باید در لیست ظاهر شود و باید در حالت "**Running**" باشد. پهنای باند پیام نشان می‌دهد "**0 msg/s**"، زیرا در حال حاضر هیچ لاگی به ورودی UDP Rsyslog ارسال نمی‌کنیم. برای مشاهده لاگ‌های یک استریم، به سادگی روی نام آن کلیک کنید.



![Image](assets/fr/004.webp)



**همه چیز در سمت Graylog آماده است**. مرحله بعدی پیکربندی ماشین لینوکس است.



### III. نصب و پیکربندی Rsyslog در لینوکس



وارد ماشین لینوکس شوید، چه به صورت محلی و چه از راه دور، و از یک حساب کاربری با مجوزهای لازم برای ارتقاء سطح دسترسی (از طریق sudo) استفاده کنید. در غیر این صورت، مستقیماً از حساب "root" استفاده کنید.



#### A. نصب بسته Rsyslog



با به‌روزرسانی حافظه پنهان بسته‌ها شروع کرده و بسته‌ای با نام "**rsyslog**" را نصب کنید.



```
sudo apt-get update
sudo apt-get install rsyslog
```



سپس وضعیت سرویس را بررسی کنید. در بیشتر موارد، این سرویس در حال اجرا است.



```
sudo systemctl status rsyslog
```



#### B. پیکربندی Rsyslog



Rsyslog دارای یک فایل پیکربندی اصلی است که در اینجا قرار دارد:



```
/etc/rsyslog.conf
```



علاوه بر این، دایرکتوری "**/etc/rsyslog.d/**" برای ذخیره فایل‌های پیکربندی اضافی Rsyslog استفاده می‌شود. در فایل پیکربندی اصلی، یک دستور Include وجود دارد که تمام فایل‌های "**.conf**" موجود در این دایرکتوری را وارد می‌کند. این امکان را فراهم می‌کند تا فایل‌های اضافی برای پیکربندی Rsyslog بدون تغییر فایل اصلی داشته باشید.



در این دایرکتوری، باید از اعداد برای تعریف ترتیب بارگذاری استفاده کنید، زیرا فایل‌ها به ترتیب حروف الفبا بارگذاری می‌شوند. افزودن پیشوند عددی به شما امکان می‌دهد اولویت بین چندین فایل پیکربندی را مدیریت کنید. در اینجا، ما فقط یک فایل اضافی داریم، بنابراین مشکلی نیست.



در این دایرکتوری، فایلی به نام "**10-graylog.conf**" ایجاد خواهیم کرد:



```
sudo nano /etc/rsyslog.d/10-graylog.conf
```



در این فایل، این خط را وارد کنید:



```
*.* @192.168.10.220:12514;RSYSLOG_SyslogProtocol23Format
```



در اینجا نحوه تفسیر این خط آمده است:





- `*.*`: به معنای ارسال تمام لاگ‌های Syslog از ماشین لینوکس به Graylog است.
- `@`: نشان می‌دهد که انتقال در UDP انجام می‌شود. باید "**@@**" را مشخص کنید تا به TCP تغییر دهید.
- `192.168.10.220:12514`: نشان‌دهنده Address سرور Graylog و پورتی است که لاگ‌ها به آن ارسال می‌شوند (مطابق با Input).
- `RSYSLOG_SyslogProtocol23Format`: مربوط به قالب پیام‌هایی است که باید به Graylog ارسال شوند.



هنگام اتمام، فایل را ذخیره کرده و Rsyslog را مجدداً راه‌اندازی کنید.



```
sudo systemctl restart rsyslog.service
```



پس از این اقدام، اولین پیام‌ها باید به سرور Graylog شما برسند!



### IV. نمایش لاگ‌های لینوکس در Graylog



از Graylog، می‌توانید روی "**Streams**" کلیک کرده و جریان جدید خود را برای مشاهده پیام‌های مرتبط انتخاب کنید. به‌طور جایگزین، روی "**Search**" کلیک کرده و Steam خود را انتخاب کرده و جستجو را آغاز کنید.



در اینجا برخی از ویژگی‌های کلیدی Interface آورده شده است:



**1** - دوره‌ای را که می‌خواهید پیام‌ها نمایش داده شوند، انتخاب کنید. به‌طور پیش‌فرض، پیام‌های 5 دقیقه گذشته نمایش داده می‌شوند.



**2** - جریان(های) نمایش داده شده را انتخاب کنید.



**3** - فعال‌سازی به‌روزرسانی خودکار لیست پیام‌ها (به عنوان مثال، هر 5 ثانیه). در غیر این صورت، لیست ثابت خواهد بود و باید به صورت دستی آن را به‌روزرسانی کنید.



**4** - برای راه‌اندازی جستجو پس از تغییر دوره، جریان یا فیلتر، روی ذره‌بین کلیک کنید.



**5** - نوار ورودی برای مشخص کردن فیلترهای جستجوی شما. در اینجا، من "**source:srv\-docker**" را مشخص می‌کنم تا فقط لاگ‌های ماشین جدیدی که به تازگی Rsyslog را روی آن تنظیم کرده‌ام نمایش داده شود.



لاگ‌ها توسط ماشین لینوکس ارسال می‌شوند:

### V. شناسایی یک خطای اتصال SSH



قدرت Graylog در توانایی آن برای فهرست‌بندی لاگ‌ها و امکان جستجو بر اساس معیارهای مختلف نهفته است: ماشین منبع، Timestamp، محتوای پیام و غیره... ما ممکن است به دنبال شناسایی اتصالات SSH ناموفق باشیم.



از یک ماشین راه دور (برای مثال، سرور Graylog)، سعی کنید به سرور لینوکسی که به تازگی Rsyslog را روی آن پیکربندی کرده‌اید، متصل شوید. برای مثال:



```
ssh [email protected]
```



سپس به‌طور عمدی یک **نام کاربری** و **رمز عبور** نادرست وارد کنید، تا **خطاهای اتصال generate** ایجاد شود. در فایل "**/var/log/auth.log**"، این کار پیام‌های لاگ generate مشابه زیر را ثبت خواهد کرد:



```
Failed password for invalid user it-connect from 192.168.10.199 port 50352 ssh2
```



شما باید این پیام‌ها را در Graylog پیدا کنید.



در Graylog، از فیلتر جستجوی زیر برای نمایش فقط پیام‌های مطابق استفاده کنید:



```
message:Failed password AND application_name:sshd
```



اگر چندین سرور دارید و می‌خواهید لاگ‌های یک سرور خاص را تحلیل کنید، نام آن را نیز مشخص کنید:



```
message:Failed password AND application_name:sshd AND source:srv\-docker
```



در اینجا یک نمای کلی از نتیجه در ماشینی که من چندین خطای اتصال را در زمان‌های مختلف روز ایجاد کردم، آمده است:



![Image](assets/fr/009.webp)



تلاش‌های ناموفق اتصال از ماشینی با IP Address "**192.168.10.199**" انجام می‌شود. اگر می‌خواهید درباره فعالیت این میزبان بیشتر بدانید، می‌توانید **این IP Address را جستجو کنید**. Graylog تمام لاگ‌هایی که در آن‌ها به این IP Address اشاره شده است را در تمام میزبان‌ها (که ارسال لاگ برای آن‌ها تنظیم شده است) نمایش خواهد داد.



در این مورد، فیلتری که باید استفاده شود می‌تواند باشد:



```
message:"192.168.10.199"
```



ما نتایج اضافی دریافت می‌کنیم (که تعجب‌آور نیست، زیرا ما بر روی برنامه SSH فیلتر نمی‌کنیم):



![Image](assets/fr/008.webp)
