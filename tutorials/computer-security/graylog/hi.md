## डेबियन 12 पर ग्रेलॉग तैनात करना



### I. प्रस्तुति



ग्रेलॉग एक ओपन सोर्स "लॉग सिंक" समाधान है जिसे आपकी मशीनों और नेटवर्क उपकरणों से लॉग को वास्तविक समय में केंद्रीकृत, संग्रहीत और विश्लेषण करने के लिए डिज़ाइन किया गया है। इस ट्यूटोरियल में, हम सीखेंगे कि डेबियन 12 मशीन पर ग्रेलॉग का मुफ़्त संस्करण कैसे इंस्टॉल करें!



एक सूचना प्रणाली में, प्रत्येक सर्वर, चाहे वह Linux चला रहा हो या Windows, और प्रत्येक नेटवर्क डिवाइस (स्विच, राउटर, फ़ायरवॉल, आदि) अपने लॉग स्वयं बनाता है, जिन्हें स्थानीय रूप से संग्रहीत किया जाता है। प्रत्येक मशीन पर स्थानीय रूप से संग्रहीत लॉग के साथ, घटना विश्लेषण और सहसंबंध बहुत मुश्किल होता है... यहीं पर ग्रेलॉग काम आता है। यह एक लॉग सिंक की तरह काम करेगा, जिसका अर्थ है कि आपकी सभी मशीनें इसे अपने लॉग भेजेंगी (उदाहरण के लिए, सिस्टलॉग के माध्यम से)। ग्रेलॉग इन लॉग को संग्रहीत और अनुक्रमित करेगा, साथ ही आपको वैश्विक खोज करने और अलर्ट बनाने की सुविधा भी देगा।



ग्रेलॉग एक विश्लेषण और निगरानी उपकरण है जो संदिग्ध व्यवहार और विभिन्न समस्याओं (स्थिरता, प्रदर्शन, आदि) की पहचान करना आसान बनाता है।




![Image](assets/fr/034.webp)



**नोट: निःशुल्क संस्करण, **ग्रेलॉग ओपन**, वाजुह की तरह SIEM नहीं है, विशेष रूप से क्योंकि इसमें वास्तविक घुसपैठ का पता लगाने वाले कार्यों का अभाव है।**



### II. पूर्वापेक्षाएँ



**ग्रेलॉग स्टैक** कई घटकों पर आधारित है जिन्हें हमें इंस्टॉल और कॉन्फ़िगर करना होगा। यहाँ, हम सभी घटकों को एक ही सर्वर पर इंस्टॉल करेंगे, लेकिन कई नोड्स पर आधारित क्लस्टर बनाना और कई सर्वरों में भूमिकाएँ वितरित करना संभव है। इस ट्यूटोरियल के लिए, हम **ग्रेलॉग 6.1** इंस्टॉल करेंगे, जो अब तक का सबसे नया संस्करण है।





- **MongoDB 7**, ग्रेलॉग के लिए वर्तमान अनुशंसित संस्करण (न्यूनतम 5.0.7, अधिकतम 7.x)
- **ओपनसर्च**, अमेज़ॅन द्वारा निर्मित इलास्टिकसर्च का एक ओपन सोर्स Fork (न्यूनतम 1.1.x, अधिकतम 2.15.x)
- **ओपनजेडीके 17**



**ग्रेलॉग सर्वर** डेबियन 12** पर चल रहा है, लेकिन इसे अन्य डिस्ट्रीब्यूशन पर भी इंस्टॉल किया जा सकता है, जिसमें डॉकर भी शामिल है। वर्चुअल मशीन में **8 जीबी रैम** और **256 जीबी डिस्क स्पेस** है, ताकि सभी कंपोनेंट्स के लिए पर्याप्त संसाधन उपलब्ध रहें (अन्यथा इसका प्रदर्शन पर काफी असर पड़ सकता है)। हालाँकि, मैं इसे एक मोटे गाइड के तौर पर दे रहा हूँ, क्योंकि **ग्रेलॉग आर्किटेक्चर का आकार प्रोसेस की जाने वाली जानकारी की मात्रा पर निर्भर करता है**। ग्रेलॉग प्रति दिन 30 एमबी या 300 एमबी डेटा, साथ ही प्रति दिन 300 जीबी डेटा संसाधित कर सकता है... यह एक **स्केलेबल समाधान** है जो **टेराबाइट्स लॉग** को संभालने में सक्षम है ([यह पृष्ठ](https://go2docs.graylog.org/current/planning_your_deployment/planning_your_deployment.html?tocpath=Plan%20Your%20Deployment%7C_____0) देखें)।



![Image](assets/fr/032.webp)



स्रोत: ग्रेलॉग



कॉन्फ़िगरेशन शुरू करने से पहले, ग्रेलॉग मशीन को एक स्थिर IP Address असाइन करें और नवीनतम अपडेट इंस्टॉल करें। स्थानीय मशीन का समय क्षेत्र सेट करना सुनिश्चित करें और दिनांक और समय सिंक्रनाइज़ेशन के लिए एक NTP सर्वर निर्धारित करें। चलाने के लिए यह कमांड है:



```
sudo timedatectl set-timezone Europe/Paris
```



**नोट: **यदि आप इसके बजाय **ग्रेलॉग डेटा नोड** का उपयोग करते हैं तो **ओपनसर्च इंस्टॉलेशन वैकल्पिक है**।



### III ग्रेलॉग की चरण-दर-चरण स्थापना



आइए पैकेज कैश को अपडेट करने और आने वाले समय के लिए आवश्यक टूल्स को इंस्टॉल करने से शुरुआत करें।



```
sudo apt-get update
sudo apt-get install curl lsb-release ca-certificates gnupg2 pwgen
```



![Image](assets/fr/030.webp)



#### A. MongoDB स्थापित करना



यह हो जाने के बाद, हम MongoDB इंस्टॉल करना शुरू करेंगे। MongoDB रिपॉजिटरी से संबंधित GPG कुंजी डाउनलोड करें:



```
curl -fsSL https://www.mongodb.org/static/pgp/server-6.0.asc | sudo gpg -o /usr/share/keyrings/mongodb-server-6.0.gpg --dearmor
```



फिर MongoDB 6 रिपॉजिटरी को Debian 12 मशीन में जोड़ें:



```
echo "deb [ signed-by=/usr/share/keyrings/mongodb-server-6.0.gpg] http://repo.mongodb.org/apt/debian bullseye/mongodb-org/6.0 main" | sudo tee /etc/apt/sources.list.d/mongodb-org-6.0.list
```



इसके बाद, हम पैकेज कैश को अपडेट करेंगे और MongoDB को स्थापित करने का प्रयास करेंगे:



```
sudo apt-get update
sudo apt-get install -y mongodb-org
```



MongoDB इंस्टॉल नहीं किया जा सकता क्योंकि एक निर्भरता गायब है: **libssl1.1**. आगे बढ़ने से पहले हमें इस पैकेज को मैन्युअल रूप से इंस्टॉल करना होगा, क्योंकि Debian 12 की रिपॉजिटरी में यह मौजूद नहीं है।



```
Les paquets suivants contiennent des dépendances non satisfaites :
mongodb-org-mongos: Dépend: libssl1.1 (>= 1.1.1) mais il n'est pas installable
mongodb-org-server: Dépend: libssl1.1 (>= 1.1.1) mais il n'est pas installable
E: Impossible de corriger les problèmes, des paquets défectueux sont en mode « garder en l'état ».
```



हम "**libssl1.1_1.1.1f-1ubuntu2.23_amd64.deb**" (नवीनतम संस्करण) नामक DEB पैकेज को **wget** कमांड से डाउनलोड करेंगे, फिर उसे **dpkg** कमांड से इंस्टॉल करेंगे। इससे निम्नलिखित दो कमांड उत्पन्न होंगे:



```
wget http://archive.ubuntu.com/ubuntu/pool/main/o/openssl/libssl1.1_1.1.1f-1ubuntu2.23_amd64.deb
sudo dpkg -i libssl1.1_1.1.1f-1ubuntu2.23_amd64.deb
```



![Image](assets/fr/028.webp)



MongoDB स्थापना पुनः आरंभ करें:



```
sudo apt-get install -y mongodb-org
```



फिर MongoDB सेवा को पुनः आरंभ करें और Debian सर्वर शुरू होने पर इसे स्वचालित रूप से प्रारंभ करने के लिए सक्षम करें।



```
sudo systemctl daemon-reload
sudo systemctl enable mongod.service
sudo systemctl restart mongod.service
sudo systemctl --type=service --state=active | grep mongod
```



MongoDB स्थापित होने के बाद, हम अगले घटक को स्थापित करने के लिए आगे बढ़ सकते हैं।



#### B. ओपनसर्च स्थापित करना



आइए सर्वर पर OpenSearch इंस्टॉल करना शुरू करें। निम्न कमांड OpenSearch पैकेज के लिए सिग्नेचर कुंजी जोड़ता है:



```
curl -o- https://artifacts.opensearch.org/publickeys/opensearch.pgp | sudo gpg --dearmor --batch --yes -o /usr/share/keyrings/opensearch-keyring
```



फिर OpenSearch रिपोजिटरी जोड़ें ताकि हम बाद में **apt** के साथ पैकेज डाउनलोड कर सकें:



```
echo "deb [signed-by=/usr/share/keyrings/opensearch-keyring] https://artifacts.opensearch.org/releases/bundle/opensearch/2.x/apt stable main" | sudo tee /etc/apt/sources.list.d/opensearch-2.x.list
```



अपना पैकेज कैश अपडेट करें:



```
sudo apt-get update
```



फिर **OpenSearch इंस्टॉल करें**, ध्यान रखें कि **अपने इंस्टेंस के एडमिन अकाउंट के लिए डिफ़ॉल्ट पासवर्ड** निर्धारित करें। यहाँ, पासवर्ड "**IT-Connect2024!**" है, लेकिन इस मान को एक मज़बूत पासवर्ड से बदलें। **कमज़ोर पासवर्ड** जैसे "**P@ssword123**" से बचें और कम से कम **8 अक्षरों** का इस्तेमाल करें, जिसमें हर प्रकार का कम से कम एक अक्षर (लोअरकेस, अपरकेस, नंबर और स्पेशल कैरेक्टर) हो, अन्यथा इंस्टॉलेशन के अंत में एक त्रुटि दिखाई देगी। **OpenSearch 2.12 के बाद से यह एक पूर्वापेक्षा है।**



```
sudo env OPENSEARCH_INITIAL_ADMIN_PASSWORD=IT-Connect2024! apt-get install opensearch
```



कृपया स्थापना के दौरान धैर्य रखें...



जब आपका काम पूरा हो जाए, तो न्यूनतम कॉन्फ़िगरेशन करने के लिए कुछ समय निकालें। कॉन्फ़िगरेशन फ़ाइल को YAML फ़ॉर्मैट में खोलें:



```
sudo nano /etc/opensearch/opensearch.yml
```



जब फ़ाइल खुली हो, तो निम्नलिखित विकल्प सेट करें:



```
cluster.name: graylog
node.name: ${HOSTNAME}
path.data: /var/lib/opensearch
path.logs: /var/log/opensearch
discovery.type: single-node
network.host: 127.0.0.1
action.auto_create_index: false
plugins.security.disabled: true
```



यह OpenSearch कॉन्फ़िगरेशन एकल नोड सेट अप करने के लिए डिज़ाइन किया गया है। हमारे द्वारा उपयोग किए जाने वाले विभिन्न पैरामीटर्स के कुछ स्पष्टीकरण यहां दिए गए हैं:





- cluster.name: graylog: यह पैरामीटर OpenSearch क्लस्टर को "**graylog**" नाम से नामित करता है।
- **node.name: ${HOSTNAME}**: नोड का नाम स्थानीय Linux मशीन के नाम से मेल खाने के लिए गतिशील रूप से सेट किया जाता है। भले ही हमारे पास केवल एक नोड हो, फिर भी उसका नाम सही रखना ज़रूरी है।
- path.data: /var/lib/opensearch: यह पथ निर्दिष्ट करता है कि OpenSearch स्थानीय मशीन पर अपना डेटा कहां संग्रहीत करता है, इस मामले में "**/var/lib/opensearch**" में।
- path.logs: /var/log/opensearch: यह पथ परिभाषित करता है कि OpenSearch लॉग फ़ाइलें कहाँ संग्रहीत हैं, यहाँ "**/var/log/opensearch**" में।
- **discovery.type: single-node**: यह पैरामीटर OpenSearch को एकल नोड के साथ काम करने के लिए कॉन्फ़िगर करता है, इसलिए "**single-node**" विकल्प का चयन किया जाता है।
- network.host: 127.0.0.1: यह कॉन्फ़िगरेशन सुनिश्चित करता है कि OpenSearch केवल अपने Interface लोकल लूप पर ही सुनता है, जो पर्याप्त है क्योंकि यह Graylog के समान सर्वर पर है।
- **action.auto_create_index: false**: स्वचालित अनुक्रमणिका निर्माण को अक्षम करके, OpenSearch स्वचालित रूप से अनुक्रमणिका नहीं बनाएगा जब कोई दस्तावेज़ बिना किसी मौजूदा अनुक्रमणिका के भेजा जाता है।
- **plugins.security.disabled: true**: यह विकल्प OpenSearch सुरक्षा प्लगइन को निष्क्रिय कर देता है, जिसका अर्थ है कि कोई प्रमाणीकरण, पहुँच प्रबंधन या संचार एन्क्रिप्शन नहीं होगा। यह सेटिंग ग्रेलॉग सेट अप करते समय समय बचाती है, लेकिन उत्पादन में इससे बचना चाहिए ([यह पृष्ठ](https://opensearch.org/docs/1.0/security-plugin/index/) देखें)।



कुछ विकल्प पहले से मौजूद हैं, इसलिए उन्हें सक्रिय करने के लिए आपको बस "#" हटाना होगा, फिर अपना मान बताना होगा। अगर आपको कोई विकल्प नहीं मिल रहा है, तो आप उसे सीधे फ़ाइल के अंत में घोषित कर सकते हैं।



![Image](assets/fr/023.webp)



इस फ़ाइल को सहेजें और बंद करें.



#### C. जावा (JVM) कॉन्फ़िगर करें



आपको OpenSearch द्वारा उपयोग की जाने वाली जावा वर्चुअल मशीन को कॉन्फ़िगर करना होगा ताकि यह सेवा कितनी मेमोरी का उपयोग कर सकती है, उसे समायोजित किया जा सके। निम्नलिखित कॉन्फ़िगरेशन फ़ाइल संपादित करें:



```
sudo nano /etc/opensearch/jvm.options
```



यहाँ लागू कॉन्फ़िगरेशन के साथ, **OpenSearch 4 GB आवंटित मेमोरी से शुरू होगा और 4 GB तक बढ़ सकता है**, इसलिए संचालन के दौरान मेमोरी में कोई बदलाव नहीं होगा। यहाँ, कॉन्फ़िगरेशन इस तथ्य को ध्यान में रखता है कि वर्चुअल मशीन में कुल **8 GB RAM** है। दोनों पैरामीटर का मान समान होना चाहिए। इसका अर्थ है इन पंक्तियों को बदलना:



```
-Xms1g
-Xmx1g
```



इन पंक्तियों के साथ:



```
-Xms4g
-Xmx4g
```



यहां किए जाने वाले संशोधन की एक छवि दी गई है:



![Image](assets/fr/022.webp)



इस फ़ाइल को सहेजने के बाद बंद कर दें।



इसके अलावा, हमें लिनक्स कर्नेल में "**max_map_count**" पैरामीटर के कॉन्फ़िगरेशन की जाँच करनी होगी। यह हमारे एप्लिकेशन की ज़रूरतों को पूरा करने के लिए, प्रत्येक प्रक्रिया के लिए मैप किए गए मेमोरी क्षेत्रों की सीमा निर्धारित करता है। **Elasticsearch** की तरह, OpenSearch भी मेमोरी प्रबंधन त्रुटियों से बचने के लिए इस मान को "262144" पर सेट करने की सलाह देता है।



सिद्धांततः, हाल ही में स्थापित डेबियन 12 मशीन पर, यह मान पहले से ही सही है। लेकिन आइए जाँच करते हैं। यह कमांड चलाएँ:



```
cat /proc/sys/vm/max_map_count
```



यदि आपको "**262144**" के अलावा कोई अन्य मान प्राप्त होता है, तो निम्न आदेश चलाएँ, अन्यथा यह आवश्यक नहीं है।



```
sudo sysctl -w vm.max_map_count=262144
```



अंत में, OpenSearch ऑटोस्टार्ट को सक्षम करें और संबंधित सेवा लॉन्च करें।



```
sudo systemctl daemon-reload
sudo systemctl enable opensearch
sudo systemctl restart opensearch
```



यदि आप अपनी सिस्टम स्थिति प्रदर्शित करते हैं, तो आपको 4 जीबी रैम वाली जावा प्रक्रिया दिखाई देगी।



```
top
```

ग्रेलॉग 6.1 को इसके नवीनतम संस्करण में स्थापित करने के लिए, ग्रेलॉग सर्वर को डाउनलोड और स्थापित करने के लिए निम्नलिखित 4 कमांड चलाएँ:

जब यह हो जाए, तो हमें इसे लॉन्च करने से पहले ग्रेलॉग के कॉन्फ़िगरेशन में कुछ बदलाव करने की आवश्यकता होगी।



आइए इन दो विकल्पों को कॉन्फ़िगर करके शुरू करें:





- **password_secret**: इस पैरामीटर का उपयोग ग्रेलॉग द्वारा उपयोगकर्ता पासवर्ड के भंडारण को सुरक्षित करने के लिए उपयोग की जाने वाली कुंजी को परिभाषित करने के लिए किया जाता है (सॉल्टिंग कुंजी की तरह)। यह कुंजी **अद्वितीय** और **यादृच्छिक** होनी चाहिए।
- **root_password_sha2**: यह पैरामीटर ग्रेलॉग में डिफ़ॉल्ट एडमिनिस्ट्रेटर पासवर्ड से मेल खाता है। इसे Hash SHA-256 के रूप में संग्रहीत किया जाता है।



हम **password_secret** पैरामीटर के लिए 96-वर्ण कुंजी उत्पन्न करके शुरू करेंगे:



```
pwgen -N 1 -s 96
wVSGYwOmwBIDmtQvGzSuBevWoXe0MWpNWCzhorBfvMMhia2zIjHguTbfl4uXZJdHOA0EEb1sOXJTZKINhIIBm3V57vwfQV59
```



लौटाए गए मान की प्रतिलिपि बनाएँ, फिर ग्रेलॉग कॉन्फ़िगरेशन फ़ाइल खोलें:



```
sudo nano /etc/graylog/server/server.conf
```

इसके बाद, आपको डिफ़ॉल्ट रूप से बनाए गए "**admin**" खाते के लिए पासवर्ड सेट करना होगा। कॉन्फ़िगरेशन फ़ाइल में, पासवर्ड का Hash संग्रहीत होना चाहिए, अर्थात इसकी गणना की जानी चाहिए। नीचे दिया गया उदाहरण "**LogsWells@**" पासवर्ड का Hash देता है: इस मान को अपने पासवर्ड के अनुसार समायोजित करें।

#### ई. ग्रेलॉग: एक नया व्यवस्थापक खाता बनाएँ

## rsyslog के साथ ग्रेलॉग को लॉग भेजें



### I. प्रस्तुति



इस दूसरे भाग में, हम सीखेंगे कि किसी Linux मशीन को अपने लॉग्स को Graylog सर्वर पर भेजने के लिए कैसे कॉन्फ़िगर किया जाए। ऐसा करने के लिए, हम सिस्टम पर Rsyslog इंस्टॉल और कॉन्फ़िगर करेंगे।



### II. लिनक्स लॉग प्राप्त करने के लिए ग्रेलॉग को कॉन्फ़िगर करना



हम ग्रेलॉग को कॉन्फ़िगर करके शुरुआत करेंगे। इसे पूरा करने के लिए तीन चरण हैं:





- एक इनपुट का निर्माण, जिससे एक प्रवेश बिंदु बनाया जा सके, जो लिनक्स मशीनों को यूडीपी के माध्यम से सिस्लॉग लॉग भेजने की अनुमति देता है।
- सभी लिनक्स लॉग को संग्रहीत और अनुक्रमित करने के लिए एक नए सूचकांक का निर्माण।
- ग्रेलॉग द्वारा प्राप्त लॉग को नए लिनक्स इंडेक्स पर रूट करने के लिए एक स्ट्रीम का निर्माण।



#### A. Syslog के लिए एक इनपुट बनाएँ



ग्रेलॉग Interface पर लॉग ऑन करें, मेनू में "सिस्टम" पर क्लिक करें और फिर "इनपुट" पर क्लिक करें। ड्रॉप-डाउन सूची में, "Syslog UDP" चुनें और फिर "नया इनपुट लॉन्च करें" बटन पर क्लिक करें। TCP Syslog इनपुट बनाना भी संभव है, लेकिन इसके लिए TLS प्रमाणपत्र की आवश्यकता होती है: सुरक्षा के लिए एक अतिरिक्त लाभ, लेकिन इस लेख में इसका उल्लेख नहीं किया गया है।



![Image](assets/fr/001.webp)



स्क्रीन पर एक विज़ार्ड दिखाई देगा। इस इनपुट को एक नाम देकर शुरू करें, उदाहरण के लिए "**Graylog_UDP_Rsyslog_Linux**" और एक पोर्ट चुनें। डिफ़ॉल्ट रूप से, पोर्ट "**514**" होता है, लेकिन आप इसे कस्टमाइज़ कर सकते हैं। यहाँ, पोर्ट "**12514**" चुना गया है।



![Image](assets/fr/016.webp)



आप ग्रेलॉग में पूरा लॉग संदेश संग्रहीत करने के लिए "**संपूर्ण संदेश संग्रहीत करें**" विकल्प भी चुन सकते हैं। "**इनपुट लॉन्च करें**" पर क्लिक करें।



![Image](assets/fr/017.webp)



नया इनपुट बन गया है और अब सक्रिय है। ग्रेलॉग अब पोर्ट 12514/UDP पर Syslog लॉग प्राप्त कर सकता है, लेकिन हमने अभी तक एप्लिकेशन को कॉन्फ़िगर करना पूरा नहीं किया है।



![Image](assets/fr/018.webp)


**नोट: एक एकल इनपुट का उपयोग कई लिनक्स मशीनों से लॉग संग्रहीत करने के लिए किया जा सकता है।**



#### B. एक नया Linux Index बनाएँ



हमें लिनक्स मशीनों से लॉग संग्रहीत करने के लिए ग्रेलॉग में एक इंडेक्स बनाना होगा। ग्रेलॉग में इंडेक्स एक स्टोरेज स्ट्रक्चर है जिसमें प्राप्त लॉग, यानी प्राप्त संदेश, संग्रहीत होते हैं। ग्रेलॉग अपने स्टोरेज इंजन के रूप में ओपनसर्च का उपयोग करता है, और तेज़, कुशल खोजों को सक्षम करने के लिए संदेशों को इंडेक्स किया जाता है।



ग्रेलॉग से, मेनू में "सिस्टम" पर क्लिक करें, फिर "इंडेक्स" पर। दिखाई देने वाले नए पेज पर, "इंडेक्स सेट बनाएँ" पर क्लिक करें।



![Image](assets/fr/005.webp)



इस इंडेक्स को नाम दें, उदाहरण के लिए "**Linux Index**", पुष्टि करने से पहले एक विवरण और एक उपसर्ग जोड़ें। यहाँ, हम **सभी Linux लॉग्स को इस इंडेक्स में संग्रहीत करेंगे**। केवल कुछ लॉग्स (केवल [SSH] लॉग्स (https://www.it-connect.fr/cours/comprendre-et-maitriser-ssh/ "SSH"), वेब सेवा लॉग्स, आदि) को संग्रहीत करने के लिए विशिष्ट इंडेक्स बनाना भी संभव है।



![Image](assets/fr/006.webp)



अब हमें इस इंडेक्स पर संदेशों को रूट करने के लिए एक नई स्ट्रीम बनाने की आवश्यकता है।



#### C. एक स्ट्रीम बनाएँ



एक नई स्ट्रीम बनाने के लिए, ग्रेलॉग के मुख्य मेनू में "स्ट्रीम्स" पर क्लिक करें। फिर दाईं ओर "स्ट्रीम बनाएँ" बटन पर क्लिक करें। दिखाई देने वाली विंडो में, स्ट्रीम को नाम दें, उदाहरण के लिए "लिनक्स स्ट्रीम" और "इंडेक्स सेट" फ़ील्ड के लिए "लिनक्स इंडेक्स" इंडेक्स चुनें। अपनी पसंद की पुष्टि करें।



**नोट: इस स्ट्रीम से संबंधित संदेश भी "डिफ़ॉल्ट स्ट्रीम" में शामिल किए जाएंगे, जब तक कि आप "डिफ़ॉल्ट स्ट्रीम से मिलान हटाएं" विकल्प को चेक न करें।**



![Image](assets/fr/002.webp)



फिर, अपनी स्टीम सेटिंग्स में, एक नया संदेश रूटिंग नियम जोड़ने के लिए "स्ट्रीम नियम जोड़ें" बटन पर क्लिक करें। अगर आपको यह विंडो नहीं मिलती है, तो मेनू में "स्ट्रीम" पर क्लिक करें, फिर अपनी स्ट्रीम से संबंधित लाइन पर, "अधिक" और फिर "नियम प्रबंधित करें" पर क्लिक करें।



"**मैच इनपुट**" प्रकार चुनें और पहले से बनाए गए **यूडीपी में आरसिस्टलॉग इनपुट** का चयन करें। "**नियम बनाएँ**" बटन से पुष्टि करें। हमारे नए इनपुट के सभी संदेश अब इंडेक्स फॉर लिनक्स को भेजे जाएँगे।



![Image](assets/fr/003.webp)



आपकी नई स्ट्रीम सूची में दिखाई देनी चाहिए और "चल रही है" स्थिति में होनी चाहिए। संदेश बैंडविड्थ "0 संदेश/सेकंड" दिखाता है, क्योंकि हम वर्तमान में Rsyslog UDP इनपुट पर कोई लॉग नहीं भेज रहे हैं। किसी स्ट्रीम के लॉग देखने के लिए, बस उसके नाम पर क्लिक करें।



![Image](assets/fr/004.webp)



**ग्रेलॉग की तरफ़ सब कुछ तैयार है**। अगला चरण लिनक्स मशीन को कॉन्फ़िगर करना है।



### III. Linux पर Rsyslog को स्थापित और कॉन्फ़िगर करना



लिनक्स मशीन पर स्थानीय या दूरस्थ रूप से लॉग ऑन करें और उसके विशेषाधिकार बढ़ाने के लिए अनुमति प्राप्त उपयोगकर्ता खाते का उपयोग करें (sudo के माध्यम से)। अन्यथा, सीधे "रूट" खाते का उपयोग करें।



#### A. Rsyslog पैकेज स्थापित करना



पैकेज कैश को अपडेट करके और "**rsyslog**" नामक पैकेज को इंस्टॉल करके प्रारंभ करें।



```
sudo apt-get update
sudo apt-get install rsyslog
```



फिर सेवा की स्थिति जांचें। ज़्यादातर मामलों में, यह पहले से ही चल रही होती है।



```
sudo systemctl status rsyslog
```



#### B. Rsyslog को कॉन्फ़िगर करना



Rsyslog की मुख्य कॉन्फ़िगरेशन फ़ाइल यहां स्थित है:



```
/etc/rsyslog.conf
```



इसके अतिरिक्त, "**/etc/rsyslog.d/**" निर्देशिका का उपयोग Rsyslog के लिए अतिरिक्त कॉन्फ़िगरेशन फ़ाइलों को संग्रहीत करने के लिए किया जाता है। मुख्य कॉन्फ़िगरेशन फ़ाइल में, इस निर्देशिका में स्थित सभी "**.conf**" फ़ाइलों को आयात करने के लिए एक "Include" निर्देश होता है। इससे मुख्य फ़ाइल को संशोधित किए बिना Rsyslog को कॉन्फ़िगर करने के लिए अतिरिक्त फ़ाइलें प्राप्त करना संभव हो जाता है।



इस निर्देशिका में, आपको लोडिंग क्रम निर्धारित करने के लिए संख्याओं का उपयोग करना होगा, क्योंकि फ़ाइलें वर्णानुक्रम में लोड होती हैं। संख्यात्मक उपसर्ग जोड़ने से आप कई कॉन्फ़िगरेशन फ़ाइलों के बीच प्राथमिकता प्रबंधित कर सकते हैं। यहाँ, हमारे पास केवल एक अतिरिक्त फ़ाइल है, इसलिए यह कोई समस्या नहीं है।



इस निर्देशिका में, हम "**10-graylog.conf**" नामक एक फ़ाइल बनाएंगे:



```
sudo nano /etc/rsyslog.d/10-graylog.conf
```



इस फ़ाइल में यह पंक्ति डालें:



```
*.* @192.168.10.220:12514;RSYSLOG_SyslogProtocol23Format
```



इस पंक्ति की व्याख्या इस प्रकार करें:





- `*.*`: का अर्थ है लिनक्स मशीन से सभी Syslog लॉग को ग्रेलॉग पर भेजना।
- `@`: यह दर्शाता है कि परिवहन UDP में किया जा रहा है। TCP पर स्विच करने के लिए आपको "**@@**" निर्दिष्ट करना होगा।
- `192.168.10.220:12514`: ग्रेलॉग सर्वर के Address को इंगित करता है, तथा वह पोर्ट जिस पर लॉग भेजे जाते हैं (इनपुट के अनुरूप)।
- `RSYSLOG_SyslogProtocol23Format`: ग्रेलॉग को भेजे जाने वाले संदेशों के प्रारूप से मेल खाता है।



जब काम पूरा हो जाए, तो फ़ाइल को सहेजें और Rsyslog को पुनः प्रारंभ करें।



```
sudo systemctl restart rsyslog.service
```



इस क्रिया के बाद, पहला संदेश आपके ग्रेलॉग सर्वर पर आना चाहिए!



### IV. ग्रेलॉग में लिनक्स लॉग प्रदर्शित करना



ग्रेलॉग से, आप "स्ट्रीम्स" पर क्लिक करके अपनी नई स्ट्रीम चुनकर संबंधित संदेश देख सकते हैं। वैकल्पिक रूप से, "खोज" पर क्लिक करके अपनी स्टीम चुनकर खोज शुरू करें।



Interface की कुछ प्रमुख विशेषताएं इस प्रकार हैं:



**1** - वह अवधि चुनें जिसके लिए संदेश प्रदर्शित करने हैं। डिफ़ॉल्ट रूप से, पिछले 5 मिनट के संदेश प्रदर्शित होते हैं।



**2** - प्रदर्शित की जाने वाली स्ट्रीम(स्ट्रीमों) का चयन करें।



**3** - संदेश सूची का स्वचालित रीफ़्रेश सक्षम करें (उदाहरण के लिए, हर 5 सेकंड में)। अन्यथा, यह स्थिर हो जाएगा और आपको इसे मैन्युअल रूप से रीफ़्रेश करना होगा।



**4** - अवधि, स्ट्रीम या फ़िल्टर को संशोधित करने के बाद खोज शुरू करने के लिए आवर्धक ग्लास पर क्लिक करें।



**5** - अपने खोज फ़िल्टर निर्दिष्ट करने के लिए इनपुट बार। यहाँ, मैं "**source:srv\-docker**" निर्दिष्ट करता हूँ ताकि केवल उस नई मशीन के लॉग प्रदर्शित हों जिस पर मैंने अभी Rsyslog सेट किया है।



लॉग्स लिनक्स मशीन द्वारा भेजे जाते हैं:

### V. SSH कनेक्शन विफलता की पहचान करना



ग्रेलॉग की ताकत लॉग को अनुक्रमित करने और विभिन्न मानदंडों के अनुसार खोज करने में सक्षम बनाने की इसकी क्षमता में निहित है: स्रोत मशीन, Timestamp, संदेश सामग्री, आदि... हम विफल SSH कनेक्शनों की पहचान करने की कोशिश कर सकते हैं।



किसी दूरस्थ मशीन (उदाहरण के लिए, ग्रेलॉग सर्वर) से, अपने उस Linux सर्वर से कनेक्ट करने का प्रयास करें जिस पर आपने अभी Rsyslog कॉन्फ़िगर किया है। उदाहरण के लिए:



```
ssh [email protected]
```



फिर जानबूझकर गलत **यूज़रनेम** और **पासवर्ड** डालें, ताकि **generate कनेक्शन त्रुटियाँ** उत्पन्न हों। "**/var/log/auth.log**" फ़ाइल में, यह generate लॉग संदेश निम्न के समान प्रदर्शित करेगा:



```
Failed password for invalid user it-connect from 192.168.10.199 port 50352 ssh2
```



आपको ये संदेश ग्रेलॉग पर मिल जाएंगे।



ग्रेलॉग पर, केवल मेल खाते संदेशों को प्रदर्शित करने के लिए निम्नलिखित खोज फ़िल्टर का उपयोग करें:



```
message:Failed password AND application_name:sshd
```



यदि आपके पास कई सर्वर हैं और आप किसी विशिष्ट सर्वर के लॉग का विश्लेषण करना चाहते हैं, तो इसके अतिरिक्त उसका नाम भी निर्दिष्ट करें:



```
message:Failed password AND application_name:sshd AND source:srv\-docker
```



यहां उस मशीन पर परिणाम का अवलोकन दिया गया है जहां मैंने दिन के अलग-अलग समय पर कई कनेक्शन त्रुटियां उत्पन्न कीं:



![Image](assets/fr/009.webp)



IP Address "**192.168.10.199**" वाली मशीन से कनेक्शन के असफल प्रयास किए गए हैं। यदि आप इस होस्ट की गतिविधि के बारे में अधिक जानना चाहते हैं, तो आप **इस IP Address को खोज सकते हैं**। ग्रेलॉग उन सभी होस्ट्स (जिनके लिए लॉग भेजना कॉन्फ़िगर किया गया है) पर, जहाँ इस IP Address का संदर्भ दिया गया है, सभी लॉग आउटपुट करेगा।



इस मामले में, उपयोग किया जाने वाला फ़िल्टर हो सकता है:



```
message:"192.168.10.199"
```



हमें अतिरिक्त परिणाम मिलते हैं (आश्चर्य की बात नहीं, क्योंकि हम SSH अनुप्रयोग पर फ़िल्टर नहीं करते हैं):



![Image](assets/fr/008.webp)

इस ट्यूटोरियल का पालन करके, आप एक Linux मशीन को उसके लॉग्स को Graylog सर्वर पर भेजने के लिए कॉन्फ़िगर कर पाएँगे। इस तरह, आप अपने Linux होस्ट्स के लॉग्स को अपने लॉग सिंक में केंद्रीकृत कर पाएँगे!
