## I. 프레젠테이션



**이 튜토리얼에서는 Lynis를 사용하여 Linux 시스템에서 보안 감사를 수행하는 방법을 배워보겠습니다! Lynis를 모르는 분들을 위해 설명하자면, **서버의 구성을 분석하고 컴퓨터의 보안을 개선하기 위한 권장 사항을 제시하는 작은 명령줄 유틸리티입니다**



Lynis는 **시스템 감사 및 강화**를 전문으로 하는 CISOFY의 오픈 소스 도구입니다. Linux와 인기 있는 서비스(SSH, Apache2 등)를 강화하고자 한다면 Lynis가 당신의 동맹입니다! Lynis는 무엇이 잘못되었는지 알려줄 뿐만 아니라 올바른 방향을 제시하고 시간을 절약할 수 있는 권장 사항도 제공합니다.



**Lynis**는 다음을 포함한 대부분의 Linux 배포판에서 작동합니다: **데비안, FreeBSD, HP-UX, NetBSD, NixOS, OpenBSD, Solaris**. Lynis는 리눅스/유닉스 사용자를 대상으로 하지만 **macOS**와도 호환됩니다. 설치가 매우 빠르며 패키지 수준에서 종속성 관리가 필요하지 않습니다.



다양한 용도로 사용됩니다:





- 안전 감사
- 규정 준수 테스트(PCI, HIPAA 및 SOX)
- 더 엄격한 시스템 구성
- 취약점 탐지



이 도구는 시스템 관리자, IT 감사자, 펜테스터 등 다양한 사용자가 폭넓게 사용합니다. 분석의 경우, 이 도구는 **CIS 벤치마크, NIST, NSA, OpenSCAP**와 같은 표준과 **데비안, 젠투, 레드햇**의 공식 권장 사항을 기반으로 합니다.



이 프로젝트는 **Github**의 Address에서 확인할 수 있습니다:





- [깃허브 - Lynis](https://github.com/CISOfy/lynis)
- [CISOFY에서 Lynis 다운로드](https://cisofy.com/lynis/#download)



이 단계별 튜토리얼에서는 **데비안 12**를 실행하는 VPS를 사용하며, 구성을 확인하고 개선하고 싶기 때문에 SSH 부분에 집중하겠습니다.



## II. 다운로드 및 설치



Lynis를 다운로드하고 설치하는 방법에는 여러 가지가 있습니다. 아래 목록에서 원하는 방법을 선택하세요.



### A. Debian 리포지토리에서 설치



이 설치 모드를 사용하면 디렉터리에 있어야 하는 소스에서 설치와 달리 시스템 어디에서나 **lynis** 명령을 사용할 수 있습니다.



SSH를 통해 서버에 연결하고 다음 명령을 입력하여 Lynis를 설치합니다:



```
sudo apt-get update
sudo apt-get install lynis -y
```



이것이 바로 여러분이 얻는 것입니다:



![Image](assets/fr/024.webp)



### B. 공식 웹사이트에서 다운로드



Cisofy 웹사이트에서도 다운로드할 수 있습니다:



```
sudo wget https://downloads.cisofy.com/lynis/lynis-3.0.9.tar.gz
```



이것이 바로 여러분이 얻는 것입니다:



![Image](assets/fr/032.webp)



다음으로 다음 명령을 사용하여 아카이브의 압축을 풉니다:



```
sudo tar -zxf lynis-3.0.9.tar.gz
```



이것이 바로 여러분이 얻는 것입니다:



![Image](assets/fr/020.webp)



Lynis** 폴더로 이동합니다:



```
cd lynis
```



다음 명령으로 업데이트를 확인할 수 있습니다:



```
./lynis update info
```



이것이 바로 여러분이 얻는 것입니다:



![Image](assets/fr/023.webp)



### C. GitHub에서 다운로드



다음 명령을 사용하여 공식 GitHub 리포지토리에서 **Lynis**를 다운로드합니다(*git*가 컴퓨터에 있어야 합니다):



```
git clone https://github.com/CISOfy/lynis.git
```



이것이 바로 여러분이 얻는 것입니다:



![Image](assets/fr/060.webp)



## III. Lynis 사용



Lynis가 저희 머신에 있으니 사용법을 배워보세요!

```
sudo lynis
# Si vous avez récupéré Lynis depuis les sources, utilisez cette syntaxe:
./lynis
```



이것이 바로 여러분이 얻는 것입니다:



![Image](assets/fr/039.webp)

![Image](assets/fr/040.webp)

### B. 시스템 감사 수행



Lynis**에게 시스템 감사를 요청하여 올바르게 구성된 부분과 개선할 수 있는 부분을 강조해 보겠습니다. 이렇게 하려면 다음 명령을 입력합니다:



```
sudo lynis audit system
# ou
./lynis audit system
```



기본적으로 이 명령을 실행할 때 루트로 로그인하지 않은 경우 현재 로그인한 사용자의 권한으로 도구가 실행됩니다. 일부 테스트는 이 컨텍스트에서 실행되지 않습니다:



![Image](assets/fr/052.webp)



이 모드에서 수행되지 않는 테스트는 다음과 같습니다:



![Image](assets/fr/051.webp)



명령이 실행되면 분석이 시작됩니다. 잠시만 기다려주세요.



감사가 끝나면 다음과 같은 메시지가 표시됩니다(**Lynis**가 **Debian 12** 시스템을 올바르게 식별했으며 분석에 **Debian 플러그인**을 사용한다는 것을 알 수 있습니다):



![Image](assets/fr/017.webp)



다음으로 Lynis는 시스템에서 확인한 모든 항목에 해당하는 일련의 포인트를 나열합니다. 이 목록은 카테고리별로 정리되어 있습니다. 권장 사항을 강조하기 위해 색상 코드가 사용된다는 점도 주목할 가치가 있습니다:





- 중요 Elements 또는 모범 사례 미준수(예: 누락된 패키지 등)의 경우 빨간색**, 즉 서버가 이 점을 준수하지 않습니다
- 노란색**은 권장 사항을 제안하거나 부분적으로 준수하는 경우(이 색상으로 강조 표시된 사항을 준수하는 것이 플러스(비우선순위)라고 가정)
- 서버 구성이 규정을 준수하는 지점용 Green**
- 화이트**, 중립적일 때



여기에서 Lynis가 **fail2ban** 설치를 권장하는 것을 볼 수 있습니다:



![Image](assets/fr/057.webp)



"**부팅 및 서비스**" 섹션에서 *systemd*를 통한 서비스 보호가 개선될 수 있음을 알 수 있습니다. 긍정적인 측면은 Grub2가 존재하며 권한 설정에 문제가 없다는 것입니다:



![Image](assets/fr/029.webp)



그런 다음 "**커널**" 및 "**메모리 및 프로세스**" 섹션이 있습니다:



![Image](assets/fr/037.webp)



다음으로 "**사용자, 그룹 및 인증**" 섹션으로 이동합니다. 이 도구는 "**/etc/sudoers.d**" 디렉터리의 권한에 대한 경고를 알려줍니다. 현재로서는 자세한 내용은 알 수 없지만 분석이 끝나면 권장 사항이 무엇인지 확인할 수 있습니다.



![Image](assets/fr/049.webp)



"**셸", "파일 시스템", "USB 장치"** 섹션에서 확인할 수 있는 내용은 다음과 같습니다. 무엇보다도 마운트 지점에 대한 제안이 있고 현재 이 머신에서 USB 장치가 허용된다는 것을 알 수 있습니다.



![Image](assets/fr/048.webp)



다음으로, 섹션: "**이름 서비스", "포트 및 패키지", "네트워킹".** 여기에는 더 이상 사용하지 않는 패키지가 제거될 수 있으며 자동 업데이트를 관리할 수 있는 유틸리티가 없음을 나타냅니다.



![Image](assets/fr/058.webp)



방화벽이 활성화되어 있는 것을 볼 수 있습니다(그리고 네, iptables가 있습니다!). 또한 사용되지 않는 규칙을 발견하고 Apache 웹 서버가 설치되어 있는 것을 확인할 수 있습니다.



![Image](assets/fr/055.webp)



그 다음에는 서비스가 식별되었으므로 웹 서버 자체에 대한 분석이 이어집니다.



Nginx** 구성 파일을 발견했으며, SSL/TLS 부분의 경우 **암호**가 안전하지 않은 프로토콜을 사용하여 구성되었음을 알 수 있습니다. 반면에 Lynis에 따르면 로그 관리는 올바른 것으로 나타났습니다.



![Image](assets/fr/038.webp)



내 VPS 서버에 SSH 서비스가 있습니다. 그 구성은 Lynis에 의해 분석됩니다. SSH 보안은 쉽게 개선할 수 있다고 말해야 합니다! 권장 사항을 확보한 후 이 부분에 대해 자세히 설명하겠습니다.



![Image](assets/fr/026.webp)



다음은 **"SNMP 지원", "데이터베이스", "PHP", "오징어 지원", "LDAP 서비스", "로깅 및 파일"** 섹션입니다.



로그 관리와 관련하여 한 가지 중요한 제안이 있는데, 로그를 컴퓨터에 로컬로 저장하지 않는 것이 좋습니다. 공격자가 머신을 손상시킨 경우, 공격자는 자신의 흔적을 지우려고 시도할 가능성이 높습니다. 따라서 로그를 외부화해야 합니다.



![Image](assets/fr/050.webp)



여기에는 취약한 서비스, 계정 관리, 예약 된 작업 및 NTP 동기화에 대한 감사가 있으며 배너 및 식별 부분에 레벨이 낮다는 표시가 있습니다. 시스템 버전을 표시하면 잠재적 인 공격자에게 표시되는 것으로 이해할 수 있습니다. 이것이 기본 설정입니다.



포렌식** 분석의 경우 **감사**를 활성화하여 로그를 확보하는 것이 흥미로울 것입니다. 로그를 효율적으로 검색하려면 시스템이 정시에 작동하는 것이 검색을 간소화하기 때문에 **NTP**도 확인합니다.



![Image](assets/fr/036.webp)



그런 다음 Lynis는 암호화 Elements, 가상화, 컨테이너 및 보안 프레임워크를 살펴봅니다. 분석 대상 시스템과 일치하는 항목이 없기 때문에 일부 섹션은 비어 있습니다. 하지만 만료된 SSL 인증서가 두 개 있고 **SELinux**를 활성화하지 않았음을 알 수 있습니다.



![Image](assets/fr/027.webp)



여기에도 개선의 여지가 있습니다. 안티바이러스 또는 안티멀웨어 스캐너가 없으며, 파일 권한에 대한 제안이 있습니다.



![Image](assets/fr/028.webp)



다음으로, Lynis는 Linux 커널 구성(IPv4 스택에 대한 규칙 포함)과 Linux 시스템의 "홈" 디렉터리 관리를 강화하는 데 중점을 둡니다.



![Image](assets/fr/035.webp)



분석의 마지막에 이르렀습니다... 이 마지막 요점은 이 머신에 ClamAV를 설치하면 얻을 수 있는 모든 것이 있다는 것을 보여줍니다.



![Image](assets/fr/030.webp)



## IV. 권장 사항



감사가 끝나면 이제 권장 사항을 읽고 분석할 차례입니다. 여기에서 Lynis가 수행한 각 테스트에 대한 권장 사항과 설명을 확인할 수 있습니다.



예를 들어 SSH 권장 사항을 살펴보세요. **각 권장 사항마다 권장 매개변수와 보안 포인트를 설명하는 링크가 제공됩니다 ** 상황과 사용법에 따라 결정은 사용자의 몫입니다.



감사 전반에 걸쳐 강조된 사항을 직접적으로 반영하는 몇 가지 권장 사항의 예를 살펴보세요...



### A. 권장 사항의 예





- 앞서 살펴본 것처럼 NTP는 타임스탬프 로그에 중요한 역할을 합니다:



![Image](assets/fr/043.webp)





- 또한 Lynis는 **apt.**를 통해 패키지 설치 중 중요한 버그에 대한 정보를 얻을 수 있도록 **apt-listbugs** 패키지를 설치할 것을 제안합니다



![Image](assets/fr/041.webp)





- 이 도구는 이전 버전의 라이브러리를 사용 중이므로 다시 시작해야 하는 프로세스를 확인할 수 있도록 **needrestart**를 설치하도록 제안합니다.



![Image](assets/fr/054.webp)





- 이 제안은 인증에 실패하는 호스트(특히 무차별 대입)를 자동으로 차단하기 위해 **fail2ban**을 설치하는 것을 제안합니다.



![Image](assets/fr/044.webp)





- 시스템 서비스를 강화하려면 컴퓨터의 각 서비스에 대해 파란색 명령을 실행하는 것이 좋습니다.



![Image](assets/fr/056.webp)





- 모든 보호된 계정의 비밀번호에 만료일을 설정할 것을 제안합니다.



![Image](assets/fr/031.webp)





- 이 제안은 비밀번호 사용 기간의 최소값과 최대값을 설정할 것을 제안합니다. 이렇게 하면 무엇보다도 비밀번호가 정기적으로 변경되도록 할 수 있습니다.



![Image](assets/fr/042.webp)





- 디스크 공간 문제가 한 파티션에 미치는 영향을 제한하기 위해 별도의 파티션을 사용하는 것이 좋습니다.



![Image](assets/fr/047.webp)





- 이 권장 사항은 데이터 유출을 방지하기 위해 USB 스토리지 및 파이어와이어를 비활성화할 것을 제안합니다



![Image](assets/fr/033.webp)





- 이 권장 사항을 충족하려면 **무인 업그레이드**를 설치 및 구성하기만 하면 됩니다.



![Image](assets/fr/053.webp)



### B. 권장 패키지 설치



시스템 구성을 개선하기 위해 Lynis에서 추천하는 패키지와 제가 개인적으로 추천하는 패키지를 컴퓨터에 설치하겠습니다.



약간의 시간만 투자하여 구성하면 작업할 수 있는 좋은 기반을 마련할 수 있습니다. 이렇게 하려면 IT-Connect 사이트, 웹의 다른 문서 및 도구 설명서를 참조하세요.



```
sudo apt-get install debsums apt-listbugs needrestart apt-show-versions fail2ban unattended-upgrades clamav clamav-daemon rkhunter
```



설치된 패키지에 대한 일부 정보입니다:





- Clamav**는 바이러스 백신입니다.
- 무인 업그레이드**를 사용하면 업데이트를 자동으로 관리하고 컴퓨터를 재부팅하거나 오래된 패키지를 자동으로 제거할 수도 있으며, 완벽하게 구성할 수 있습니다.
- rkhunter**는 파일 시스템을 검사하는 루트킷 방지 프로그램입니다.
- Fail2ban**은 사용자가 읽도록 지정한 내용에 따라 로그 파일을 기반으로 하며, 예를 들어 SSH에서 서버를 "무차별 공격"하려는 IP 주소를 차단하는 등 **iptables**와 함께 작동합니다.

그러겠습니다:

- 로그인 유예 시간**을 **2분**으로 설정합니다



2분 이내에 연결 정보를 입력하지 않으면 연결이 끊어집니다.





- 엄격한 모드** 활성화



연결을 확인하기 전에 'sshd'가 사용자 파일의 모드와 소유자, 사용자의 홈 디렉터리를 확인할지 여부를 지정합니다. 초보자가 실수로 디렉터리나 파일을 모든 사람이 액세스할 수 있게 두는 경우가 있으므로 일반적으로 이 옵션이 바람직합니다. 기본값은 "예"입니다.





- MaxAuthtries**를 3으로 설정합니다



통신이 종료되기 전까지 실패한 인증 시도 횟수를 나타냅니다.





- 최대 세션**을 2로 설정



이는 최대 동시 세션 수를 나타냅니다.





- 공개 키 인증 사용



```
PubkeyAuthentication yes
```





- 비밀번호 인증을 유지합니다:



```
PasswordAuthentication yes
```



빈 비밀번호와 **케르베로스** 인증, **직접 루트 인증**을 금지합니다



```
PermitEmptyPasswords no
PermitRootLogin no
```



"**PermitRootLogin no"가 있는지 확인하고, "yes"와 같으면 "절대 악"**입니다.





- TCP 연결 리디렉션 금지



```
AllowTcpForwarding no
```



TCP 리디렉션 허용 여부를 나타내며 기본 설정은 '예'입니다. 참고: 사용자가 셸에 액세스할 수 있는 경우 자체 리디렉션 도구를 계속 설정할 수 있으므로 TCP 리디렉션을 비활성화해도 보안이 강화되지는 않습니다.





- X11 리디렉션 금지



```
X11Forwarding no
```



X11 리디렉션 허용 여부를 나타내며 기본 설정은 '아니요'입니다. 참고: X11 리디렉션이 비활성화되어 있어도 사용자가 직접 리디렉터를 설정할 수 있으므로 보안이 강화되지는 않습니다. 로그인 사용**을 선택하면 X11 리디렉션이 자동으로 비활성화됩니다.





- 클라이언트와 sshd 간의 통신 시간 초과 설정



```
ClientAliveInterval  300
```



시간 초과를 초 단위로 정의하며, 그 후 클라이언트로부터 데이터가 수신되지 않으면 sshd 서비스에서 클라이언트에 응답을 요청하는 메시지를 보냅니다. 기본적으로 이 옵션은 "0"으로 설정되어 있어 이러한 메시지가 클라이언트에 전송되지 않습니다. SSH 프로토콜 버전 2에서만 이 옵션을 지원합니다.



```
ClientAliveCountMax 0
```



Sshd](https://www.delafond.org/traducmanfr/man/man5/sshd_config.5.html)의 [문서(*맨 페이지*)에 따르면 이 옵션의 의미는 다음과 같습니다: "**sshd**에 대해 클라이언트의 응답 없이 전송할 보류 메시지 수(위 참조)를 설정합니다. 보류 메시지가 전송되는 동안 이 임계값에 도달하면 **sshd**는 클라이언트의 연결을 끊고 세션을 종료합니다. 이러한 연결 유지 메시지는 **KeepAlive** 옵션(아래)과는 매우 다르다는 점에 유의해야 합니다. 연결 보류 메시지는 암호화된 터널을 통해 전송되므로 위조가 불가능합니다. KeepAlive**로 활성화된 TCP 수준의 연결 보류는 위조가 가능합니다. 연결 보류 메커니즘은 클라이언트나 서버가 연결이 유휴 상태인지 여부를 알아야 할 때 유용합니다."





- 모티프, 배너, 마지막 로그**를 비활성화하여 정보 공개를 방지하세요



```
PrintMotd no
```



사용자가 대화형 모드로 로그온할 때 sshd가 "/etc/motd" 파일의 내용을 표시할지 여부를 지정합니다. 일부 시스템에서는 셸에서 /etc/profile 또는 유사한 파일을 통해 이 내용을 표시할 수도 있습니다. 기본값은 "yes"입니다.



```
Banner none
```



일부 관할권에서는 인증 전에 메시지를 보내는 것이 법적 보호를 위한 전제 조건이 될 수 있다는 점에 유의할 필요가 있습니다. 연결 권한이 부여되기 전에 지정된 파일의 내용이 원격 사용자에게 전송됩니다. 기본적으로 메시지가 표시되지 않으므로 이 옵션을 구성해야 합니다.

## V. Lynis 자동화된 계획



**Lynis**는 예약된 작업을 통해 검사를 실행하는 옵션도 제공합니다. 실제로 **"--cronjob "** 옵션이 있는데, 이 옵션은 유효성 검사나 사용자 작업 없이 모든 Lynis 테스트를 실행합니다. 그런 다음 **Lynis**를 실행하는 스크립트를 아주 간단하게 만들고 해당 서버의 이름이 포함된 타임스탬프가 있는 파일에 출력을 넣을 수 있습니다. 다음은 */etc/cron.daily* 폴더에 넣을 수 있는 이러한 유형의 파일입니다:



```
#!/bin/sh
mkdir /var/log/lynis
NOM_AUDITEUR="tache_crontab"
DATE=$(date +%Y%m%d)
HOTE=$(hostname)
LOG_DIR="/var/log/Lynis"
RAPPORT="$LOG_DIR/rapport-${HOTE}.${DATE}"
DATA="$LOG_DIR/rapport-data-${HOTE}.${DATE}.txt"

cd /root/Lynis./Lynis -c --auditor "${NOM_AUDITEUR}" --cronjob > ${RAPPORT}
mv /var/log/lynis-report.dat ${DATA}
```



"AUDITOR_NAME"** 변수는 **Lynis**의 **"--auditor"** 옵션에서 이 이름이 보고서에 표시되도록 설정하는 변수일 뿐입니다:



![Image](assets/fr/059.webp)



또한 보고서의 이름을 지정하고 타임스탬프를 찍을 호스트 이름과 날짜, 보고서를 저장할 폴더의 경로와 같이 보다 체계적으로 정리하는 데 도움이 되는 몇 가지 상황별 변수를 만들 것입니다.



## VI. 결론



Lynis는 특히 보안 측면에서 Linux 서버의 구성 상태에 대해 자세히 알고 싶을 때 시간을 절약하고 효율성을 높이는 데 도움이 되는 매우 실용적인 도구입니다. 하지만 모든 수정 사항은 사용 환경과 상황을 고려하여 프로덕션에 적용하기 전에 반드시 테스트를 거쳐야 한다는 점을 잊지 마세요.



SSH 연결이 여러 개 필요한 **바스티온** 또는 **스케줄러**와 달리 (접속하는 사람이 한 명뿐이므로) SSH 연결이 하나만 필요한 인터넷에 노출된 VPS의 경우 동일한 구성을 적용하지 않을 수 있습니다



경화 측면에서 자신에게 적합한 구성을 찾은 후에는 시간이 많이 걸리고 오류가 발생하기 쉬운 작업을 수동으로 다시 실행할 필요가 없도록 자동화 도구를 사용하는 것이 좋습니다. 예를 들어 **를 사용할 수 있습니다: Puppet, Chef, Ansible 등...**를 사용할 수 있습니다



변경을 시행하기 전에 팀과 소통하는 것을 잊지 마세요. 단순히 변경을 시행한다고 말하는 것이 아니라 왜 이러한 변경을 시행하는지 팀원들이 이해할 수 있도록 해야 합니다. 결국 목표는 모범 사례를 전수하는 것이며, 이를 통해 성공 가능성을 높일 수 있습니다.



마지막으로 **Lynis**를 여러 가지 다른 도구와 비교할 수도 있습니다. 오픈 소스를 유지하면서 중앙 집중식 관리로 전환하고 싶으시다면 [Wazuh] 도구(https://wazuh.com/)를 추천합니다.
