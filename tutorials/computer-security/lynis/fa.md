## I. ارائه



**در این آموزش، ما یاد خواهیم گرفت که چگونه با استفاده از Lynis یک ممیزی امنیتی بر روی یک ماشین لینوکس انجام دهیم! برای کسانی که Lynis را نمی‌شناسند، این یک ابزار کوچک خط فرمان است که پیکربندی سرور شما را تحلیل کرده و توصیه‌هایی برای **بهبود امنیت ماشین شما ارائه می‌دهد.**



Lynis یک ابزار متن‌باز از شرکت CISOFY است که در **ممیزی سیستم و تقویت امنیت** تخصص دارد. اگر می‌خواهید در تقویت امنیت لینوکس و سرویس‌های محبوب (SSH، Apache2 و غیره) پیشرفت کنید، Lynis هم‌پیمان شماست! Lynis نه تنها به شما می‌گوید چه چیزی اشتباه است، بلکه توصیه‌هایی نیز ارائه می‌دهد تا شما را به مسیر درست هدایت کند (و در وقت شما صرفه‌جویی کند).



**Lynis** با اکثر توزیع‌های لینوکس کار می‌کند، از جمله: **Debian, FreeBSD, HP-UX, NetBSD, NixOS, OpenBSD, Solaris**. Lynis برای کاربران لینوکس / یونیکس طراحی شده است، اما با **macOS** نیز سازگار است. نصب آن بسیار سریع است و در سطح بسته نیازی به مدیریت وابستگی‌ها ندارد.



برای اهداف مختلفی استفاده می‌شود:





- ممیزی‌های ایمنی
- تست انطباق (PCI، HIPAA و SOX)
- پیکربندی‌های سخت‌تر سیستم
- تشخیص آسیب‌پذیری



این ابزار به طور گسترده توسط طیف وسیعی از کاربران، از جمله مدیران سیستم، حسابرسان IT و نفوذگران استفاده می‌شود. برای تحلیل‌ها، این ابزار بر اساس استانداردهایی مانند **CIS Benchmark, NIST, NSA, OpenSCAP** و توصیه‌های رسمی از **Debian, Gentoo, Red Hat** است.



پروژه در این Address در **Github** موجود است:





- [GitHub - Lynis](https://github.com/CISOfy/lynis)
- [دانلود Lynis از CISOFY](https://cisofy.com/lynis/#download)



در این آموزش گام‌به‌گام، قصد دارم از **یک VPS با سیستم‌عامل Debian 12 استفاده کنم** و تمرکز من بر روی بخش SSH خواهد بود، زیرا می‌خواهم تنظیمات آن را بررسی و بهبود دهم.



## II. دانلود و نصب



چندین روش برای دانلود و نصب Lynis وجود دارد. یکی را که ترجیح می‌دهید از لیست زیر انتخاب کنید.



### A. نصب از مخازن دبیان



این حالت نصب به شما اجازه می‌دهد تا از فرمان **lynis** از هر جایی در سیستم استفاده کنید، برخلاف نصب از منبع که نیاز دارید در دایرکتوری قرار داشته باشید.



از طریق SSH به سرور خود متصل شوید و دستورات زیر را برای نصب Lynis وارد کنید:



```
sudo apt-get update
sudo apt-get install lynis -y
```



این چیزی است که دریافت می‌کنید:



![Image](assets/fr/024.webp)



### B. از وب‌سایت رسمی دانلود کنید



شما همچنین می‌توانید آن را از وب‌سایت Cisofy دانلود کنید:



```
sudo wget https://downloads.cisofy.com/lynis/lynis-3.0.9.tar.gz
```



این چیزی است که دریافت می‌کنید:



![Image](assets/fr/032.webp)



سپس، آرشیو را با استفاده از فرمان زیر باز خواهیم کرد:



```
sudo tar -zxf lynis-3.0.9.tar.gz
```



این چیزی است که دریافت می‌کنید:



![Image](assets/fr/020.webp)



بیایید به پوشه **lynis** برویم:



```
cd lynis
```



می‌توانیم با استفاده از فرمان زیر به‌روزرسانی‌ها را بررسی کنیم:



```
./lynis update info
```



این چیزی است که دریافت می‌کنید:



![Image](assets/fr/023.webp)



### C. دانلود از GitHub



ما **Lynis** را از مخزن رسمی GitHub با استفاده از فرمان زیر دانلود خواهیم کرد (باید *git* روی دستگاه شما نصب باشد):



```
git clone https://github.com/CISOfy/lynis.git
```



این چیزی است که دریافت می‌کنید:



![Image](assets/fr/060.webp)



## III. استفاده از Lynis



Lynis در دستگاه ما موجود است، پس بیایید یاد بگیریم چگونه از آن استفاده کنیم!



### A. کنترل‌ها و گزینه‌های اصلی



برای نمایش دستورات موجود، به سادگی دستور زیر را وارد کنید:



```
sudo lynis
# Si vous avez récupéré Lynis depuis les sources, utilisez cette syntaxe:
./lynis
```



این چیزی است که دریافت می‌کنید:



![Image](assets/fr/039.webp)



و همچنین گزینه‌های زیر را دریافت می‌کنید:



![Image](assets/fr/040.webp)



برای نمایش تمام دستورات موجود، دستور زیر را وارد کنید:



```
sudo lynis show
```



این چیزی است که دریافت می‌کنید:



![Image](assets/fr/022.webp)



اگر می‌خواهید تمام گزینه‌ها را نمایش دهید، باید وارد کنید:



```
sudo lynis show options
```



این چیزی است که دریافت می‌کنید:



![Image](assets/fr/021.webp)



در ادامه این مقاله، یاد خواهیم گرفت که چگونه از گزینه‌های خاصی استفاده کنیم.



### ب. انجام ممیزی سیستم



ما قصد داریم از **Lynis** بخواهیم که سیستم ما را بررسی کند و مواردی که به درستی پیکربندی شده‌اند و مواردی که می‌توان بهبود بخشید را مشخص کند. برای انجام این کار، دستور زیر را وارد کنید:



```
sudo lynis audit system
# ou
./lynis audit system
```



به طور پیش‌فرض، اگر هنگام اجرای این دستور به عنوان ریشه وارد نشده باشید، ابزار با امتیازات کاربر فعلی وارد شده اجرا خواهد شد. برخی از آزمایش‌ها در این زمینه اجرا نخواهند شد:



![Image](assets/fr/052.webp)



در این حالت، آزمایش‌های زیر انجام نخواهند شد:



![Image](assets/fr/051.webp)



پس از اجرای فرمان، تحلیل آغاز می‌شود. فقط یک لحظه صبر کنید.



در پایان ممیزی، شما این را دریافت می‌کنید (ما می‌توانیم ببینیم که **Lynis** به درستی سیستم **Debian 12** را شناسایی کرده و از **پلاگین Debian** برای تحلیل استفاده خواهد کرد):



![Image](assets/fr/017.webp)



سپس، Lynis مجموعه‌ای از نکات را فهرست خواهد کرد که مربوط به همه چیزهایی است که در سیستم ما بررسی کرده است. این موارد بر اساس دسته‌بندی سازماندهی شده‌اند، همان‌طور که خواهیم دید. همچنین شایان ذکر است که از یک کد رنگی برای برجسته‌سازی توصیه‌ها استفاده می‌شود:





- قرمز** برای Elements بحرانی یا عدم رعایت بهترین شیوه‌ها (به عنوان مثال، یک بسته گم‌شده)، یعنی سرور شما این نکته را رعایت نمی‌کند.
- **زرد** برای پیشنهادات یا تطابق جزئی با توصیه (بیایید بگوییم که رعایت یک نکته که با این رنگ برجسته شده است یک امتیاز است (غیر اولویت‌دار))
- Green** برای نقاطی که پیکربندی سرور شما مطابق است
- سفید**، وقتی خنثی است



در اینجا می‌بینیم که Lynis توصیه می‌کند **fail2ban** را نصب کنید:



![Image](assets/fr/057.webp)



در بخش "**بوت و خدمات**"، می‌بینیم که حفاظت از خدمات از طریق *systemd* می‌تواند بهبود یابد. از جنبه مثبت، Grub2 حضور دارد و هیچ مشکلی با مجوزها در:



![Image](assets/fr/029.webp)



سپس بخش‌های "**هسته**" و "**حافظه و فرآیندها**" را دارید:



![Image](assets/fr/037.webp)



سپس، بخش "**Users, Groups and Authentication**". ابزار به ما در مورد هشداری در مورد مجوزهای دایرکتوری "**/etc/sudoers.d**" اطلاع می‌دهد. در حال حاضر، اطلاعات بیشتری نداریم، اما در پایان تحلیل قادر خواهیم بود ببینیم توصیه چیست.



![Image](assets/fr/049.webp)



در بخش‌های "**Shells"، "Files Systems"، و "USB Devices "** می‌توانید موارد زیر را پیدا کنید. از جمله، می‌توانیم ببینیم که پیشنهاداتی در مورد نقاط اتصال وجود دارد و اینکه دستگاه‌های USB در حال حاضر بر روی این دستگاه مجاز هستند.



![Image](assets/fr/048.webp)



بعدی، بخش‌ها: "**خدمات نام‌گذاری"، "پورت‌ها و بسته‌ها"، "شبکه‌سازی".** اینجا نشان می‌دهد که بسته‌هایی که دیگر استفاده نمی‌شوند می‌توانند پاک شوند و هیچ ابزاری قادر به مدیریت به‌روزرسانی‌های خودکار نیست.



![Image](assets/fr/058.webp)



ما می‌توانیم ببینیم که یک فایروال فعال شده است (و بله، iptables وجود دارد!). علاوه بر این، می‌توانیم ببینیم که قوانین استفاده نشده‌ای پیدا شده‌اند و یک سرور وب Apache نصب شده است.



![Image](assets/fr/055.webp)



این با تجزیه و تحلیل خود وب سرور دنبال می‌شود، زیرا سرویس شناسایی شده است.



می‌توانیم ببینیم که فایل‌های پیکربندی **Nginx** پیدا شده‌اند و برای بخش SSL/TLS، **رمزنگاری‌ها** با استفاده از پروتکلی که ناامن خواهد بود، پیکربندی شده‌اند. از طرف دیگر، طبق گزارش Lynis، مدیریت لاگ صحیح است.



![Image](assets/fr/038.webp)



سرویس SSH بر روی سرور VPS من موجود است. پیکربندی آن توسط Lynis تحلیل می‌شود. باید گفت که امنیت SSH به راحتی می‌تواند بهبود یابد! ما به این بخش به طور مفصل باز خواهیم گشت زمانی که توصیه‌ها را دریافت کنیم.



![Image](assets/fr/026.webp)



در اینجا بخش‌های **"پشتیبانی SNMP"، "پایگاه‌های داده"، "PHP"، "پشتیبانی Squid"، "خدمات LDAP"، "ثبت و فایل‌ها"** آمده است.



یک پیشنهاد واقعاً مهم درباره مدیریت لاگ وجود دارد: اکیداً توصیه می‌شود که لاگ‌ها را فقط به‌صورت محلی روی دستگاه خود ذخیره نکنید. اگر دستگاه توسط یک مهاجم خراب شود، احتمالاً او سعی می‌کند ردپای خود را پاک کند... بنابراین ما نیاز داریم که لاگ‌ها را به‌صورت خارجی ذخیره کنیم.



![Image](assets/fr/050.webp)



در اینجا، ما ممیزی خدمات آسیب‌پذیر، مدیریت حساب‌ها، وظایف زمان‌بندی‌شده و همگام‌سازی NTP را داریم. در بخش بنر و شناسایی نشان داده شده است که سطح پایین است: این قابل درک است، اگر نسخه سیستم را نمایش دهید، به یک مهاجم بالقوه اطلاعاتی می‌دهد. این تنظیمات پیش‌فرض است.



فعال‌سازی **auditd** برای داشتن لاگ‌ها در صورت نیاز به تحلیل **جرم‌شناسی** جالب خواهد بود. همچنین **NTP** بررسی می‌شود، زیرا برای جستجوی کارآمد لاگ‌ها، بهتر است سیستم‌ها به‌روز باشند که این امر جستجو را ساده‌تر می‌کند.



![Image](assets/fr/036.webp)



سپس Lynis به Elements رمزنگاری، مجازی‌سازی، کانتینرها و چارچوب‌های امنیتی نگاه می‌کند. برخی از بخش‌ها خالی هستند زیرا هیچ تطابقی با ماشین تحلیل‌شده وجود ندارد. با این حال، می‌توانیم ببینیم که من دو گواهینامه SSL منقضی‌شده دارم و **SELinux** را فعال نکرده‌ام.



![Image](assets/fr/027.webp)



در اینجا نیز جای بهبود وجود دارد: هیچ اسکنر ضد ویروس یا ضد بدافزاری وجود ندارد و ما پیشنهاداتی در مورد مجوزهای فایل داریم.



![Image](assets/fr/028.webp)



سپس، Lynis بر روی بهبود تنظیمات هسته لینوکس (شامل قوانین برای پشته IPv4) و همچنین مدیریت دایرکتوری‌های "Home" ماشین لینوکس تمرکز می‌کند.



![Image](assets/fr/035.webp)



به پایان تحلیل رسیده‌ایم... این نکته آخر نشان می‌دهد که با نصب ClamAV روی این دستگاه، همه چیز به نفع ما خواهد بود.



![Image](assets/fr/030.webp)



## IV. توصیه‌ها



پس از حسابرسی، زمان خواندن و تحلیل توصیه‌ها فرا می‌رسد. اینجاست که توصیه‌ها و توضیحات مربوط به هر یک از آزمایش‌های انجام‌شده توسط Lynis را دریافت می‌کنیم.



به عنوان مثال، توصیه‌های SSH را در نظر بگیرید. **برای هر پیشنهاد، پارامتر توصیه‌شده و لینکی که نقطه امنیتی را توضیح می‌دهد، خواهید یافت.** تصمیم‌گیری با شماست، بسته به زمینه و استفاده شما.



بیایید نگاهی به چند نمونه از توصیه‌هایی بیندازیم که به طور مستقیم نکات برجسته شده در طول ممیزی را منعکس می‌کنند...



### الف. نمونه‌هایی از توصیه‌ها





- همان‌طور که قبلاً دیدیم، NTP برای زمان‌گذاری گزارش‌ها مهم است:



![Image](assets/fr/043.webp)





- لینیس همچنین پیشنهاد می‌کند که بسته **apt-listbugs** را نصب کنید تا در هنگام نصب بسته‌ها از طریق **apt** اطلاعاتی درباره باگ‌های بحرانی دریافت کنید.



![Image](assets/fr/041.webp)





- ابزار پیشنهاد می‌کند که **needrestart را نصب کنیم تا بتوانیم** ببینیم کدام فرآیندها از نسخه قدیمی کتابخانه استفاده می‌کنند و نیاز به راه‌اندازی مجدد دارند.



![Image](assets/fr/054.webp)





- این پیشنهاد نصب **fail2ban** را برای مسدود کردن خودکار میزبان‌هایی که در احراز هویت شکست می‌خورند (به‌ویژه حملات جستجوی فراگیر) توصیه می‌کند.



![Image](assets/fr/044.webp)





- برای سخت کردن خدمات سیستم، او توصیه می‌کند که فرمان آبی را برای هر خدمت روی دستگاه خود اجرا کنیم.



![Image](assets/fr/056.webp)





- او پیشنهاد می‌دهد که برای تمام رمزهای عبور حساب‌های محافظت‌شده تاریخ انقضا تعیین شود.



![Image](assets/fr/031.webp)





- این پیشنهاد تعیین مقادیر حداقل و حداکثر برای سن رمز عبور را پیشنهاد می‌کند. در میان سایر موارد، این امر اطمینان حاصل می‌کند که رمزهای عبور به طور منظم تغییر می‌کنند.



![Image](assets/fr/042.webp)





- ما توصیه می‌کنیم از پارتیشن‌های جداگانه استفاده کنید تا تأثیر مشکلات فضای دیسک بر روی یک پارتیشن محدود شود.



![Image](assets/fr/047.webp)





- این توصیه پیشنهاد می‌کند که ذخیره‌سازی USB و فایروایر را غیرفعال کنید تا از نشت داده جلوگیری شود.



![Image](assets/fr/033.webp)





- برای برآورده کردن این توصیه، به سادگی **unnatended-upgrade** را نصب و پیکربندی کنید، به عنوان مثال.



![Image](assets/fr/053.webp)



### ب. نصب بسته‌های پیشنهادی



برای بهبود پیکربندی سیستم، قصد داریم تعدادی بسته روی دستگاه نصب کنیم: برخی که توسط Lynis توصیه شده‌اند و برخی که خودم شخصاً توصیه می‌کنم.



تا زمانی که کمی وقت صرف پیکربندی آن‌ها کنید، پایه خوبی برای کار خواهید داشت. برای انجام این کار، به سایت IT-Connect، مقالات دیگر در وب و مستندات ابزار مراجعه کنید.



```
sudo apt-get install debsums apt-listbugs needrestart apt-show-versions fail2ban unattended-upgrades clamav clamav-daemon rkhunter
```



برخی اطلاعات درباره بسته‌های نصب شده:





- Clamav** یک آنتی‌ویروس است.
- unattend-upgrades** به شما این امکان را می‌دهد که به‌روزرسانی‌های خود را به‌صورت خودکار مدیریت کنید و حتی ماشین را راه‌اندازی مجدد کنید یا بسته‌های قدیمی را به‌طور خودکار پاک کنید، این ابزار به‌طور کامل قابل تنظیم است.
- rkhunter** یک ابزار ضد روت‌کیت است که سیستم فایل شما را اسکن می‌کند.
- Fail2ban** بر اساس فایل‌های لاگ شما عمل خواهد کرد، مطابق با آنچه که به آن برای خواندن می‌دهید و با **iptables** کار خواهد کرد، به عنوان مثال برای مسدود کردن آدرس‌های IP که سعی می‌کنند به صورت "brute force" به سرور شما در SSH دسترسی پیدا کنند.



### ج. توصیه‌ها برای SSH



بیایید نگاهی به توصیه‌های SSH بیندازیم. آن‌ها در زیر فهرست شده‌اند. نگران نباشید، ما بلافاصله توضیح خواهیم داد که چگونه پیکربندی را بهبود بخشیم.



![Image](assets/fr/034.webp)



بیایید نگاهی دقیق‌تر به پیکربندی فعلی **SSH** من در:**/etc/ssh/sshd_config** بیندازیم.



![Image](assets/fr/018.webp)



پیکربندی پیشنهادی زیر هنوز می‌تواند بهینه‌سازی شود، اما یک پایه خوب به شما می‌دهد. *لطفاً توجه داشته باشید که برای خوانایی بیشتر تعدادی از نظرات را حذف کرده‌ام*.



ما خواهیم:





- تغییر پورت اتصال SSH (پورت پیش‌فرض 22 را فراموش کنید)
- افزایش سطح جزئیات لاگ‌ها، از **INFO به VERBOSE**
- **LoginGraceTime** را به **2 دقیقه** تنظیم کنید.



اگر اطلاعات اتصال ظرف دو دقیقه وارد نشود، اتصال قطع می‌شود.





- فعال کردن **strictModes**



مشخص می‌کند که آیا "sshd" باید حالت‌ها و مالک فایل‌های کاربر و همچنین دایرکتوری خانگی کاربر را قبل از اعتبارسنجی یک اتصال بررسی کند یا خیر. این معمولاً مطلوب است، زیرا گاهی اوقات مبتدیان به‌طور تصادفی دایرکتوری یا فایل‌های خود را به‌طور کامل برای همه قابل دسترسی می‌گذارند. مقدار پیش‌فرض "yes" است.





- تنظیم **MaxAuthtries** به 3



این نشان‌دهنده تعداد تلاش‌های ناموفق احراز هویت است قبل از اینکه ارتباط بسته شود.





- تنظیم **MaxSessions** به 2



این نشان‌دهنده حداکثر تعداد جلسات همزمان است.





- احراز هویت کلید عمومی را فعال کنید



```
PubkeyAuthentication yes
```





- احراز هویت با رمز عبور را حفظ کنید:



```
PasswordAuthentication yes
```



ممنوعیت استفاده از رمزهای عبور خالی و احراز هویت **Kerberos**، همچنین **احراز هویت مستقیم ریشه**



```
PermitEmptyPasswords no
PermitRootLogin no
```



مطمئن شوید که "**PermitRootLogin no" باشد، اگر برابر با "yes" است، این "شر مطلق" است **.





- ممنوعیت تغییر مسیر اتصال TCP



```
AllowTcpForwarding no
```



مشخص می‌کند که آیا تغییر مسیرهای TCP مجاز هستند یا خیر، با "بله" به عنوان تنظیم پیش‌فرض. لطفاً توجه داشته باشید: غیرفعال کردن تغییر مسیرهای TCP امنیت را افزایش نمی‌دهد اگر کاربران به یک شل دسترسی داشته باشند، زیرا آن‌ها همچنان می‌توانند ابزارهای تغییر مسیر خود را تنظیم کنند.





- ممنوعیت هدایت X11



```
X11Forwarding no
```



مشخص می‌کند که آیا ریدایرکت‌های X11 پذیرفته می‌شوند یا خیر، که به‌طور پیش‌فرض روی "no" تنظیم شده است. لطفاً توجه داشته باشید: حتی اگر ریدایرکت‌های X11 غیرفعال باشند، این امر امنیت را افزایش نمی‌دهد، زیرا کاربران همچنان می‌توانند ریدایرکتورهای خود را تنظیم کنند. ریدایرکت X11 به‌طور خودکار غیرفعال می‌شود اگر **UseLogin** انتخاب شود.





- تنظیم زمان انتظار ارتباط بین کلاینت و sshd



```
ClientAliveInterval  300
```



زمانی را به ثانیه تعریف می‌کند که پس از آن، اگر داده‌ای از مشتری دریافت نشود، سرویس sshd پیامی برای درخواست پاسخ از مشتری ارسال می‌کند. به طور پیش‌فرض، این گزینه روی "0" تنظیم شده است، به این معنی که این پیام‌ها به مشتری ارسال نمی‌شوند. تنها نسخه ۲ پروتکل SSH از این گزینه پشتیبانی می‌کند.



```
ClientAliveCountMax 0
```



طبق [مستندات (*صفحه man*) برای sshd](https://www.delafond.org/traducmanfr/man/man5/sshd_config.5.html)، این گزینه به این معناست: "تعداد پیام‌های نگه‌داری (به بالا مراجعه کنید) که بدون پاسخ از مشتری برای **sshd** ارسال می‌شوند را تنظیم می‌کند. اگر این آستانه در حالی که پیام‌های نگه‌داری ارسال شده‌اند، برسد، **sshd** مشتری را قطع کرده و جلسه را خاتمه می‌دهد. مهم است که توجه داشته باشید که این پیام‌های نگه‌داری بسیار متفاوت از گزینه **KeepAlive** (در زیر) هستند. پیام‌های نگه‌داری اتصال از طریق تونل رمزگذاری شده ارسال می‌شوند و بنابراین قابل جعل نیستند. نگه‌داری اتصال در سطح TCP که توسط **KeepAlive** فعال می‌شود، قابل جعل است. مکانیزم نگه‌داری اتصال زمانی مورد توجه است که مشتری یا سرور نیاز دارد بداند آیا اتصال بیکار است یا خیر."





- جلوگیری از افشای اطلاعات با غیرفعال کردن **motd, banner, lastlog**



```
PrintMotd no
```



مشخص می‌کند که آیا sshd باید محتوای فایل "/etc/motd" را زمانی که یک کاربر در حالت تعاملی وارد می‌شود نمایش دهد یا خیر. در برخی سیستم‌ها، این محتوا ممکن است توسط شل، از طریق /etc/profile یا فایل مشابهی نیز نمایش داده شود. مقدار پیش‌فرض "yes" است.



```
Banner none
```



شایان ذکر است که در برخی حوزه‌های قضایی، ارسال پیام قبل از احراز هویت ممکن است پیش‌نیازی برای حفاظت قانونی باشد. محتوای فایل مشخص شده قبل از اعطای مجوز اتصال به کاربر راه دور ارسال می‌شود. این گزینه نیاز به پیکربندی دارد، زیرا به‌طور پیش‌فرض هیچ پیامی نمایش داده نخواهد شد.

## برنامه‌ریزی خودکار V. Lynis



**Lynis** همچنین گزینه اجرای بررسی‌های خود را از طریق یک وظیفه زمان‌بندی شده ارائه می‌دهد. در واقع گزینه **"--cronjob "** وجود دارد که تمام تست‌های Lynis را بدون نیاز به اعتبارسنجی یا اقدام کاربر اجرا می‌کند. سپس می‌توانید به سادگی یک اسکریپت ایجاد کنید که **Lynis** را اجرا کرده و خروجی را در یک فایل با مهر زمانی و نام سرور مربوطه قرار دهد. در اینجا یک فایل از این نوع وجود دارد که می‌توانید در پوشه */etc/cron.daily* قرار دهید:



```
#!/bin/sh
mkdir /var/log/lynis
NOM_AUDITEUR="tache_crontab"
DATE=$(date +%Y%m%d)
HOTE=$(hostname)
LOG_DIR="/var/log/Lynis"
RAPPORT="$LOG_DIR/rapport-${HOTE}.${DATE}"
DATA="$LOG_DIR/rapport-data-${HOTE}.${DATE}.txt"

cd /root/Lynis./Lynis -c --auditor "${NOM_AUDITEUR}" --cronjob > ${RAPPORT}
mv /var/log/lynis-report.dat ${DATA}
```



متغیر **"AUDITOR_NAME "** به سادگی یک متغیر است که ما آن را در گزینه **"--auditor "** از **Lynis** تنظیم خواهیم کرد تا این نام در گزارش نمایش داده شود:



![Image](assets/fr/059.webp)



ما همچنین قصد داریم چند متغیر متنی ایجاد کنیم که به ما کمک می‌کنند بهتر سازماندهی کنیم، مانند نام میزبان و تاریخ برای نام‌گذاری گزارش و زمان‌گذاری آن، و مسیر پوشه‌ای که می‌خواهیم گزارش‌های خود را در آن قرار دهیم.



## VI. نتیجه‌گیری



Lynis یک ابزار بسیار کاربردی است که به شما کمک می‌کند تا در زمان صرفه‌جویی کرده و هنگامیکه می‌خواهید اطلاعات بیشتری درباره وضعیت پیکربندی یک سرور لینوکس، به‌ویژه از نظر امنیت، کسب کنید، کارآمدتر باشید. با این حال، فراموش نکنید که هر تغییری باید قبل از اعمال در محیط تولید، با توجه به استفاده و زمینه خودتان آزمایش شود.



احتمالاً همان پیکربندی را برای یک VPS که به اینترنت متصل است اعمال نخواهید کرد، جایی که فقط به یک اتصال SSH نیاز دارید (زیرا شما تنها فردی هستید که متصل می‌شوید)، برخلاف یک **bastion** یا **scheduler** که نیاز به چند برابر کردن اتصالات **SSH.** دارد.



هنگامی که یک پیکربندی مناسب از نظر سخت‌سازی دارید، توصیه می‌شود از یک ابزار اتوماسیون استفاده کنید تا نیازی به انجام دستی وظایف نداشته باشید، زیرا این کار زمان‌بر و مستعد خطا است. به عنوان مثال، می‌توانید از **: Puppet, Chef, Ansible, etc...** استفاده کنید.



فراموش نکنید که قبل از اجرا با تیم‌های خود ارتباط برقرار کنید: شما باید به آنها بفهمانید چرا این تغییرات را اعمال می‌کنید، نه فقط به آنها بگویید که این تغییرات را اعمال می‌کنید. در نهایت، هدف انتقال روش‌های خوب خواهد بود و این شانس موفقیت شما را افزایش خواهد داد.



در نهایت، شما همچنین می‌توانید **Lynis** را با ابزارهای دیگر مقایسه کنید که چندین مورد از آن‌ها وجود دارد. اگر می‌خواهید به سمت مدیریت متمرکز حرکت کنید و در عین حال منبع باز باقی بمانید، من ابزار [Wazuh] (https://wazuh.com/) را توصیه می‌کنم.
