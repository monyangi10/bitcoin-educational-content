## I.プレゼンテーション



**このチュートリアルでは、Lynisを使ってLinuxマシンのセキュリティ監査を行う方法を学びます！Lynisを知らない人のために言っておくと、**Lynisは小さなコマンドラインユーティリティで、サーバーの設定を分析し、マシンのセキュリティを**向上させるための推奨事項を提案します**。



Lynisは**システム監査とハードニング**を専門とするCISOFY社のオープンソースツールです。Linuxや一般的なサービス(SSH、Apache2など)の堅牢化を進めたいなら、Lynisはあなたの味方です！Lynis は、何が間違っているのかを教えてくれるだけでなく、正しい方向を指し示す（そして時間を節約する）推奨事項も提供してくれます。



**Lynis**はほとんどのLinuxディストリビューションで動作します： **Debian、FreeBSD、HP-UX、NetBSD、NixOS、OpenBSD、Solaris** です。Lynis は Linux / UNIX ユーザーを対象としていますが、**macOS** とも互換性があります。インストールは非常に迅速で、パッケージレベルでの依存関係管理はありません。



さまざまな用途に使われる：





- 安全監査
- コンプライアンス・テスト（PCI、HIPAA、SOX）
- よりタフなシステム構成
- 脆弱性の検出



このツールは、システム管理者、IT監査人、ペンテスターなど、幅広いユーザーに利用されています。分析に関しては、このツールは**CIS Benchmark、NIST、NSA、OpenSCAP**などの標準や、**Debian、Gentoo、Red Hat**の公式勧告に基づいています。



このプロジェクトは**Github**のAddressから入手できる：





- [GitHub - Lynis](https://github.com/CISOfy/lynis)
- [CISOFYからリニスをダウンロード](https://cisofy.com/lynis/#download)



このステップバイステップのチュートリアルでは、**Debian 12**を実行しているVPSを使用し、その設定をチェックして改善したいので、SSHの部分に集中するつもりです。

### B.推奨パッケージのインストール



システム構成を改善するために、いくつかのパッケージをマシンにインストールする。リニスが推奨するものもあれば、私が個人的に推奨するものもある。



設定に少し時間をかければ、十分な基礎ができる。そのためには、IT-Connectのサイトやウェブ上の他の記事、ツールのドキュメントを参照してください。



```
sudo apt-get install debsums apt-listbugs needrestart apt-show-versions fail2ban unattended-upgrades clamav clamav-daemon rkhunter
```



インストールされたパッケージに関するいくつかの情報 ：





- **Clamav**はアンチウイルスです。
- **unattend-upgrades**は、アップデートを自動的に管理し、マシンを再起動したり、古いパッケージを自動的に削除したりすることができます。
- **rkhunter**は、ファイルシステムをスキャンするアンチルートキットです。
- **Fail2ban**は、あなたが読み取るように与えたものに従って、ログファイルに基づいて動作し、例えば、SSHであなたのサーバーを「ブルートフォース」しようとするIPアドレスを禁止するために、**iptables**と連動します。



### C.SSHに関する推奨事項



SSHの推奨事項を見てみよう。以下にリストアップする。すぐに設定を改善する方法を説明しますので、ご心配なく。



![Image](assets/fr/034.webp)



etc/ssh/sshd_config**にある現在の**SSH**設定を詳しく見てみよう。



![Image](assets/fr/018.webp)



以下に提案するコンフィギュレーションは、まだ最適化することができますが、良いベースを与えてくれます。*読みやすくするため、多くのコメントを削除したことに注意してほしい。*



我々は：





- SSH接続ポートを変更する（デフォルトの22を忘れる）
- ログの冗長性レベルを**INFOからVERBOSE**に上げる。
- **LoginGraceTime**を**2分**に設定する。



2分以内に接続情報が入力されない場合、接続は切断される。





- **strictModes**をアクティブにする



sshd" が接続を認証する前に、ユーザのホームディレクトリだけでなく、 ユーザのファイルのモードと所有者をチェックするかどうかを指定します。初心者が誤って自分のディレクトリやファイル を完全に誰でもアクセス可能な状態にしてしまうことがあるため、通常はこれが望ま しい。デフォルトは「yes」である。





- **MaxAuthtries**を3に設定する。



これは、通信がクローズされるまでの認証失敗回数を表す。





- **MaxSessions**を2に設定する。



これは最大同時セッション数を表す。





- 公開鍵認証を有効にする



```
PubkeyAuthentication yes
```





- パスワード認証の保持 ：



```
PasswordAuthentication yes
```



空のパスワードと**Kerberos**認証、および**直接ルート認証**を禁止する。



```
PermitEmptyPasswords no
PermitRootLogin no
```



もし "yes "であれば、それは "絶対悪 "である。





- TCPコネクションのリダイレクトを禁止する



```
AllowTcpForwarding no
```



TCPリダイレクトを許可するかどうかを示し、デフォルト設定は「yes」である。注意: TCPリダイレクトを無効にしても、ユーザがシェルにアクセスできる場合はセキュリティは向上しない。





- X11リダイレクトの禁止



```
X11Forwarding no
```



デフォルトは "no "です。注意: X11リダイレクトを無効にしても、セキュリティは向上しません。**UseLogin**が選択されている場合、X11リダイレクトは自動的に無効になります。





- クライアントと sshd 間の通信タイムアウトを設定する。



```
ClientAliveInterval  300
```



クライアントからデータが受信されない場合、sshd サービスがクライアントからの応答を 要求するメッセージを送信するまでのタイムアウトを秒単位で定義します。既定では、このオプションは 「0」に設定されており、このメッセージはクライアントに送信されません。このオプションに対応しているのは、SSH プロトコルのバージョン 2 のみです。



```
ClientAliveCountMax 0
```



sshdのドキュメント(**manページ**)](https://www.delafond.org/traducmanfr/man/man5/sshd_config.5.html)によると、このオプションの意味は以下の通り："**sshd**に対して、クライアントからの応答なしに送信されるホールドメッセージ (上記参照) の数を設定します。ホールドメッセージの送信中にこのしきい値に達すると、**sshd** はクライアントを切断してセッションを終了します。これらのホールドメッセージは、 **KeepAlive**オプション(下記)とは大きく異なることに注意することが重要です。接続保留メッセージは暗号化トンネルを介して送信されるため、偽造できません。**KeepAlive**によって有効化されるTCPレベルのコネクションホールドは、偽造可能である。コネクションホールドの仕組みは、クライアントまたはサーバーがコネクションがアイドル





- motd、banner、lastlog**を無効にして情報漏洩を防ぐ。



```
PrintMotd no
```



ユーザが対話モードでログオンしたときに、sshd が「/etc/motd」ファイルの内容を 表示するかどうかを指定します。システムによっては、この内容は /etc/profile または類似のファイルを介してシェルでも表示される場合があります。デフォルト値は「yes」である。



```
Banner none
```



法域によっては、認証の前にメッセージを送信することが、法的保護の前提条件となる場合があることは注目に値する。指定されたファイルの内容は、接続認証が行なわれる前にリモートユーザに送信される。デフォルトではメッセージは表示されないので、このオプションを設定する必要がある。

## V.リニス自動プランニング



**Lynis**はスケジュールされたタスクによってチェックを実行するオプションも提供します。実際、**"-cronjob "** オプションがあり、これは検証やユーザ操作なしですべての Lynis テストを実行します。そして、**Lynis** を実行するスクリプトを作成し、その出力をタイムスタンプ付きのファイルに問題のサーバ名と一緒に保存することができます。以下は、*/etc/cron.daily* フォルダーに置くことができるこのタイプのファイルです：



```
#!/bin/sh
mkdir /var/log/lynis
NOM_AUDITEUR="tache_crontab"
DATE=$(date +%Y%m%d)
HOTE=$(hostname)
LOG_DIR="/var/log/Lynis"
RAPPORT="$LOG_DIR/rapport-${HOTE}.${DATE}"
DATA="$LOG_DIR/rapport-data-${HOTE}.${DATE}.txt"

cd /root/Lynis./Lynis -c --auditor "${NOM_AUDITEUR}" --cronjob > ${RAPPORT}
mv /var/log/lynis-report.dat ${DATA}
```



AUDITOR_NAME **変数は、単純に** Lynis **の** "--auditor" **オプションで設定する変数で、この名前がレポートに表示されるようにする：**



![Image](assets/fr/059.webp)



また、レポートの命名やタイムスタンプを付与するためのホスト名や日付、レポートを格納するフォルダーのパスなど、整理しやすくするためのコンテキスト変数もいくつか作成します。



## VI.結論



Lynisは、Linuxサーバーの設定の状態、特にセキュリティの状態についてもっと知りたいときに、時間を節約し、より効率的にするのに役立つ非常に実用的なツールである。しかし、すべての変更は、あなた自身の使い方とコンテキストを考慮して、本番環境に適用する前にテストされなければならないことを忘れないでください。



おそらく、**bastion**や**scheduler**のように複数の**SSH.**接続が必要な場合とは異なり、(接続するのは自分だけなので)1つのSSH接続しか必要としないネットに公開されたVPSに同じ設定を適用することはないでしょう。



ハードニングの面で自分に合ったコンフィギュレーションができたら、手作業でタスクをやり直す必要がないように、自動化ツールを採用することをお勧めする。例えば、**Puppet、Chef、Ansibleなど**を使うことができる。



実施前にチームとコミュニケーションをとることを忘れてはならない。ただ変更すると伝えるだけでなく、なぜ変更するのかを理解させる必要がある。最終的には、良い習慣を伝えることが目的であり、そうすることで成功の可能性が高まる。
