## I. การนำเสนอ



**ในบทแนะนำนี้ เราจะเรียนรู้วิธีการตรวจสอบความปลอดภัยบนเครื่อง Linux โดยใช้ Lynis! สำหรับผู้ที่ไม่รู้จัก** Lynis, **มันเป็นเครื่องมือบรรทัดคำสั่งขนาดเล็กที่จะวิเคราะห์การกำหนดค่าของเซิร์ฟเวอร์ของคุณและให้คำแนะนำในการปรับปรุงความปลอดภัยของเครื่องของคุณ**



Lynis เป็นเครื่องมือโอเพ่นซอร์สจาก CISOFY บริษัทที่เชี่ยวชาญด้าน **การตรวจสอบและเสริมความปลอดภัยของระบบ** หากคุณต้องการพัฒนาการเสริมความปลอดภัยให้กับ Linux และบริการยอดนิยม (SSH, Apache2, ฯลฯ) Lynis คือพันธมิตรของคุณ! Lynis ไม่เพียงแต่บอกคุณว่าอะไรผิดพลาด แต่ยังให้คำแนะนำเพื่อชี้แนะแนวทางที่ถูกต้อง (และประหยัดเวลาให้คุณ)



**Lynis** ทำงานร่วมกับลีนุกซ์ดิสทริบิวชันส่วนใหญ่ รวมถึง: **Debian, FreeBSD, HP-UX, NetBSD, NixOS, OpenBSD, Solaris**. Lynis มุ่งเป้าไปที่ผู้ใช้ Linux / UNIX แต่ก็สามารถใช้งานร่วมกับ **macOS** ได้เช่นกัน ติดตั้งได้อย่างรวดเร็วมาก ไม่มีการจัดการการพึ่งพาในระดับแพ็กเกจ



มันถูกใช้สำหรับวัตถุประสงค์ที่หลากหลาย:





- การตรวจสอบความปลอดภัย
- การทดสอบการปฏิบัติตามข้อกำหนด (PCI, HIPAA และ SOX)
- การกำหนดค่าระบบที่ยากขึ้น
- การตรวจจับช่องโหว่



เครื่องมือนี้ถูกใช้อย่างแพร่หลายโดยผู้ใช้หลากหลายกลุ่ม รวมถึงผู้ดูแลระบบ ผู้ตรวจสอบ IT และผู้ทดสอบเจาะระบบ สำหรับการวิเคราะห์ เครื่องมือนี้อ้างอิงตามมาตรฐานเช่น **CIS Benchmark, NIST, NSA, OpenSCAP** และคำแนะนำอย่างเป็นทางการจาก **Debian, Gentoo, Red Hat**.



โครงการนี้สามารถเข้าถึงได้ที่ที่อยู่นี้บน **Github**:





- [GitHub - Lynis](https://github.com/CISOfy/lynis)
- [ดาวน์โหลด Lynis จาก CISOFY](https://cisofy.com/lynis/#download)



ในบทแนะนำทีละขั้นตอนนี้ ฉันจะ **ใช้ VPS ที่รัน Debian 12** และฉันจะเน้นไปที่ส่วนของ SSH เนื่องจากฉันต้องการตรวจสอบการกำหนดค่าและปรับปรุงมัน



## II. ดาวน์โหลดและการติดตั้ง



มีหลายวิธีในการดาวน์โหลดและติดตั้ง Lynis เลือกวิธีที่คุณต้องการจากรายการด้านล่าง



### A. การติดตั้งจากที่เก็บข้อมูล Debian



โหมดการติดตั้งนี้ช่วยให้คุณสามารถใช้คำสั่ง **lynis** จากที่ใดก็ได้ในระบบ ซึ่งแตกต่างจากการติดตั้งจากซอร์สที่คุณจำเป็นต้องอยู่ในไดเรกทอรีนั้น



เชื่อมต่อกับเซิร์ฟเวอร์ของคุณผ่าน SSH และป้อนคำสั่งต่อไปนี้เพื่อติดตั้ง Lynis:



```
sudo apt-get update
sudo apt-get install lynis -y
```



นี่คือสิ่งที่คุณได้รับ:



![Image](assets/fr/024.webp)



### B. ดาวน์โหลดจากเว็บไซต์ทางการ



คุณยังสามารถดาวน์โหลดได้จากเว็บไซต์ Cisofy:



```
sudo wget https://downloads.cisofy.com/lynis/lynis-3.0.9.tar.gz
```



นี่คือสิ่งที่คุณได้รับ:



![Image](assets/fr/032.webp)



ถัดไป เราจะทำการแตกไฟล์เก็บถาวรโดยใช้คำสั่งต่อไปนี้:



```
sudo tar -zxf lynis-3.0.9.tar.gz
```



นี่คือสิ่งที่คุณได้รับ:



![Image](assets/fr/020.webp)



ย้ายไปที่โฟลเดอร์ **lynis**:



```
cd lynis
```



เราสามารถตรวจสอบการอัปเดตด้วยคำสั่งต่อไปนี้:



```
./lynis update info
```



นี่คือสิ่งที่คุณได้รับ:



![Image](assets/fr/023.webp)



### C. ดาวน์โหลดจาก GitHub



เราจะดาวน์โหลด **Lynis** จากที่เก็บ GitHub อย่างเป็นทางการ โดยใช้คำสั่งต่อไปนี้ (*git* ต้องติดตั้งอยู่บนเครื่องของคุณ):



```
git clone https://github.com/CISOfy/lynis.git
```



นี่คือสิ่งที่คุณได้รับ:

## III. การใช้ Lynis



Lynis อยู่ในเครื่องของเราแล้ว มาศึกษาวิธีการใช้งานกันเถอะ!



### A. การควบคุมหลักและตัวเลือก



หากต้องการแสดงคำสั่งที่มีอยู่ เพียงป้อนคำสั่งต่อไปนี้:



```
sudo lynis
# Si vous avez récupéré Lynis depuis les sources, utilisez cette syntaxe:
./lynis
```



นี่คือสิ่งที่คุณได้รับ:



![Image](assets/fr/039.webp)



และคุณยังได้รับตัวเลือกต่อไปนี้:



![Image](assets/fr/040.webp)



หากต้องการแสดงคำสั่งทั้งหมดที่มีอยู่ ให้ป้อนคำสั่งต่อไปนี้:



```
sudo lynis show
```



นี่คือสิ่งที่คุณได้รับ:



![Image](assets/fr/022.webp)



หากคุณต้องการแสดงตัวเลือกทั้งหมด คุณต้องป้อน:



```
sudo lynis show options
```



นี่คือสิ่งที่คุณได้รับ:



![Image](assets/fr/021.webp)



ในส่วนที่เหลือของบทความนี้ เราจะเรียนรู้วิธีการใช้ตัวเลือกบางอย่าง



### ข. การดำเนินการตรวจสอบระบบ



เรากำลังจะขอให้ **Lynis** ตรวจสอบระบบของเรา โดยเน้นสิ่งที่กำหนดค่าอย่างถูกต้องและสิ่งที่สามารถปรับปรุงได้ ในการทำเช่นนี้ ให้ป้อนคำสั่งต่อไปนี้:



```
sudo lynis audit system
# ou
./lynis audit system
```



ตามค่าเริ่มต้น หากคุณไม่ได้เข้าสู่ระบบในฐานะ root เมื่อคุณเรียกใช้คำสั่งนี้ เครื่องมือจะทำงานด้วยสิทธิ์ของผู้ใช้ที่เข้าสู่ระบบในปัจจุบัน การทดสอบบางอย่างจะไม่ถูกเรียกใช้ในบริบทนี้:



![Image](assets/fr/052.webp)



นี่คือการทดสอบที่จะไม่ดำเนินการในโหมดนี้:



![Image](assets/fr/051.webp)



เมื่อคำสั่งถูกดำเนินการแล้ว การวิเคราะห์จะเริ่มต้นขึ้น กรุณารอสักครู่



เมื่อสิ้นสุดการตรวจสอบ คุณจะได้รับสิ่งนี้ (เราสามารถเห็นว่า **Lynis** ได้ระบุระบบ **Debian 12** อย่างถูกต้องและจะใช้ **Debian plugin** สำหรับการวิเคราะห์):



![Image](assets/fr/017.webp)



ถัดไป Lynis จะระบุชุดของจุดที่สอดคล้องกับทุกสิ่งที่เขาได้ตรวจสอบในระบบของเรา ซึ่งจัดเรียงตามหมวดหมู่ตามที่เราจะเห็น นอกจากนี้ยังควรสังเกตว่ามีการใช้รหัสสีเพื่อเน้นคำแนะนำ:





- สีแดง** สำหรับองค์ประกอบที่สำคัญหรือแนวปฏิบัติที่ดีที่สุดที่ไม่ได้รับการเคารพ (เช่น แพ็คเกจที่ขาดหายไป) กล่าวคือ เซิร์ฟเวอร์ของคุณไม่เคารพจุดนี้
- สีเหลือง** สำหรับข้อเสนอแนะหรือการปฏิบัติตามคำแนะนำบางส่วน (สมมติว่าเป็นข้อดีที่จะปฏิบัติตามจุดที่เน้นด้วยสีนี้ (ไม่ใช่ลำดับความสำคัญ))
- Green** สำหรับจุดที่การกำหนดค่าของเซิร์ฟเวอร์ของคุณเป็นไปตามข้อกำหนด
- สีขาว**, เมื่อเป็นกลาง



ที่นี่ เราจะเห็นว่า Lynis แนะนำให้ติดตั้ง **fail2ban**:



![Image](assets/fr/057.webp)



ในส่วน "**Boot and services**" เราเห็นว่าการป้องกันบริการผ่าน *systemd* สามารถปรับปรุงได้ ในด้านบวก Grub2 มีอยู่และไม่มีปัญหาเกี่ยวกับสิทธิ์บน:



![Image](assets/fr/029.webp)



จากนั้นคุณจะมีส่วน "**Kernel**" และ "**Memory and Processes**":



![Image](assets/fr/037.webp)



ถัดไป ส่วน "**ผู้ใช้ กลุ่ม และการตรวจสอบสิทธิ์**" เครื่องมือแจ้งเตือนเราถึงคำเตือนเกี่ยวกับสิทธิ์ของไดเรกทอรี "**/etc/sudoers.d**" ในขณะนี้ เรายังไม่ทราบรายละเอียดเพิ่มเติม แต่เราจะสามารถดูคำแนะนำได้เมื่อสิ้นสุดการวิเคราะห์



![Image](assets/fr/049.webp)



นี่คือสิ่งที่คุณสามารถพบได้ในส่วน "**Shells", "Files Systems", และ "USB Devices "** นอกจากสิ่งอื่น ๆ แล้ว เราสามารถเห็นได้ว่ามีข้อเสนอแนะเกี่ยวกับจุดเชื่อมต่อและอุปกรณ์ USB ได้รับอนุญาตบนเครื่องนี้ในปัจจุบัน



![Image](assets/fr/048.webp)



ถัดไป ส่วน: "**Name services", "Ports and packages", "Networking".** ระบุไว้ที่นี่ว่าแพ็คเกจที่ไม่ได้ใช้งานอีกต่อไปสามารถลบออกได้ และไม่มีเครื่องมือที่สามารถจัดการการอัปเดตอัตโนมัติได้



![Image](assets/fr/058.webp)



เราสามารถเห็นได้ว่าไฟร์วอลล์ถูกเปิดใช้งาน (และใช่, มี iptables!) นอกจากนี้ เรายังเห็นว่ามันพบกฎที่ไม่ได้ใช้งานและมีการติดตั้งเซิร์ฟเวอร์เว็บ Apache



![Image](assets/fr/055.webp)



ตามด้วยการวิเคราะห์เว็บเซิร์ฟเวอร์เอง เนื่องจากบริการได้รับการระบุแล้ว



เราสามารถเห็นได้ว่ามันได้พบไฟล์การกำหนดค่าของ **Nginx** และสำหรับส่วนของ SSL/TLS นั้น **ciphers** ถูกกำหนดค่าด้วยการใช้โปรโตคอลที่อาจไม่ปลอดภัย ในทางกลับกัน ตามที่ Lynis ระบุ การจัดการบันทึกนั้นถูกต้อง



![Image](assets/fr/038.webp)



บริการ SSH มีอยู่บนเซิร์ฟเวอร์ VPS ของฉัน การกำหนดค่าของมันถูกวิเคราะห์โดย Lynis ต้องบอกว่าความปลอดภัยของ SSH สามารถปรับปรุงได้อย่างง่ายดาย! เราจะกลับมาที่ส่วนนี้อย่างละเอียดเมื่อเราได้รับคำแนะนำแล้ว



![Image](assets/fr/026.webp)



นี่คือส่วนต่างๆ **"SNMP Support", "Databases", "PHP", "Squid Support", "LDAP Services", "Logging and files "**.



มีข้อเสนอแนะที่สำคัญมากเกี่ยวกับการจัดการล็อก: ขอแนะนำอย่างยิ่งว่าอย่าเก็บล็อกไว้ในเครื่องของคุณเพียงอย่างเดียว หากเครื่องถูกโจมตีโดยผู้ไม่หวังดี มีความเป็นไปได้สูงว่าเขาจะพยายามลบหลักฐานของเขา... ดังนั้นเราจำเป็นต้องจัดเก็บล็อกภายนอก



![Image](assets/fr/050.webp)



ที่นี่ เรามีการตรวจสอบบริการที่มีช่องโหว่ การจัดการบัญชี งานที่กำหนดเวลา และการซิงโครไนซ์ NTP ระบุว่าระดับต่ำในส่วนแบนเนอร์และการระบุ: ซึ่งเข้าใจได้ หากคุณแสดงเวอร์ชันของระบบ มันจะให้ข้อมูลแก่ผู้โจมตีที่มีศักยภาพ นี่คือการตั้งค่าเริ่มต้น



การเปิดใช้งาน **auditd** จะน่าสนใจเพื่อให้มีบันทึกในกรณีที่ต้องทำการวิเคราะห์ **forensic** นอกจากนี้ยังมีการตรวจสอบ **NTP** ด้วย เพราะเพื่อให้การค้นหาบันทึกมีประสิทธิภาพ ควรให้ระบบมีเวลาเป็นปัจจุบัน ซึ่งจะช่วยให้การค้นหาง่ายขึ้น



![Image](assets/fr/036.webp)



Lynis จะพิจารณาองค์ประกอบการเข้ารหัสลับ, การจำลองเสมือน, คอนเทนเนอร์ และกรอบการทำงานด้านความปลอดภัย บางส่วนว่างเปล่าเพราะไม่มีความสอดคล้องกับเครื่องที่วิเคราะห์ อย่างไรก็ตาม เราสามารถเห็นได้ว่าฉันมีใบรับรอง SSL ที่หมดอายุสองใบและฉันยังไม่ได้เปิดใช้งาน **SELinux**.



![Image](assets/fr/027.webp)



ที่นี่ก็ยังมีพื้นที่ให้ปรับปรุง: ไม่มีโปรแกรมสแกนไวรัสหรือมัลแวร์ และเรามีข้อเสนอแนะเกี่ยวกับสิทธิ์ของไฟล์



![Image](assets/fr/028.webp)



ถัดไป, Lynis มุ่งเน้นไปที่การปรับแต่งการตั้งค่าของเคอร์เนล Linux (รวมถึงกฎสำหรับสแต็ก IPv4) รวมถึงการจัดการไดเรกทอรี "Home" ของเครื่อง Linux ด้วย



![Image](assets/fr/035.webp)



เราได้มาถึงจุดสิ้นสุดของการวิเคราะห์แล้ว... จุดสุดท้ายนี้แสดงให้เห็นว่าเราจะได้ประโยชน์ทุกอย่างจากการมี ClamAV บนเครื่องนี้



![Image](assets/fr/030.webp)



## IV. คำแนะนำ



หลังจากการตรวจสอบแล้ว ก็ถึงเวลาที่จะอ่านและวิเคราะห์คำแนะนำ นี่คือจุดที่เราได้รับคำแนะนำและคำอธิบายสำหรับการทดสอบแต่ละรายการที่ดำเนินการโดย Lynis



นำคำแนะนำ SSH มาเป็นตัวอย่าง **สำหรับแต่ละคำแนะนำ คุณจะพบพารามิเตอร์ที่แนะนำและลิงก์ที่จะอธิบายประเด็นด้านความปลอดภัย** ขึ้นอยู่กับคุณที่จะตัดสินใจ ขึ้นอยู่กับบริบทและการใช้งานของคุณ



ลองมาดูตัวอย่างคำแนะนำบางส่วนที่สะท้อนถึงประเด็นที่เน้นย้ำตลอดการตรวจสอบ...



### A. ตัวอย่างของคำแนะนำ





- ตามที่เราเห็นก่อนหน้านี้ NTP มีความสำคัญสำหรับการประทับเวลาล็อก:



![Image](assets/fr/043.webp)





- Lynis ยังแนะนำให้ติดตั้งแพ็กเกจ **apt-listbugs** เพื่อรับข้อมูลเกี่ยวกับบั๊กที่สำคัญในระหว่างการติดตั้งแพ็กเกจผ่าน **apt.**



![Image](assets/fr/041.webp)





- เครื่องมือแนะนำให้เราติดตั้ง **needrestart เพื่อให้สามารถ** ดูได้ว่ากระบวนการใดกำลังใช้เวอร์ชันเก่าของไลบรารีและจำเป็นต้องรีสตาร์ท



![Image](assets/fr/054.webp)





- คำแนะนำนี้แนะนำให้ติดตั้ง **fail2ban** เพื่อบล็อกโฮสต์ที่ไม่สามารถยืนยันตัวตนได้โดยอัตโนมัติ (โดยเฉพาะการโจมตีแบบ brute force)



![Image](assets/fr/044.webp)





- เพื่อเพิ่มความปลอดภัยให้กับบริการของระบบ เขาแนะนำให้เรารันคำสั่งสีน้ำเงินสำหรับแต่ละบริการบนเครื่องของเรา



![Image](assets/fr/056.webp)





- เขาแนะนำให้กำหนดวันหมดอายุสำหรับรหัสผ่านบัญชีที่ได้รับการป้องกันทั้งหมด



![Image](assets/fr/031.webp)





- ข้อเสนอแนะนี้แนะนำให้กำหนดค่าต่ำสุดและค่าสูงสุดสำหรับอายุของรหัสผ่าน สิ่งนี้จะช่วยให้มั่นใจได้ว่ารหัสผ่านจะถูกเปลี่ยนเป็นประจำ



![Image](assets/fr/042.webp)





- เราแนะนำให้ใช้พาร์ติชันแยกต่างหาก เพื่อจำกัดผลกระทบของปัญหาพื้นที่ดิสก์บนพาร์ติชันเดียว



![Image](assets/fr/047.webp)





- คำแนะนำนี้แนะนำให้ปิดใช้งานการจัดเก็บข้อมูล USB และ firewire เพื่อป้องกันการรั่วไหลของข้อมูล



![Image](assets/fr/033.webp)





- เพื่อตอบสนองคำแนะนำนี้ เพียงติดตั้งและกำหนดค่า **unnatended-upgrade** ตัวอย่างเช่น



![Image](assets/fr/053.webp)



### B. การติดตั้งแพ็คเกจที่แนะนำ



เพื่อปรับปรุงการกำหนดค่าระบบ เราจะติดตั้งแพ็คเกจบางอย่างบนเครื่อง: บางส่วนที่แนะนำโดย Lynis และบางส่วนที่ฉันแนะนำเอง



คุณจะมีพื้นฐานที่ดีในการทำงาน ตราบใดที่คุณใช้เวลาเล็กน้อยในการกำหนดค่าเหล่านั้น ในการทำเช่นนี้ ให้ดูที่เว็บไซต์ IT-Connect บทความอื่น ๆ บนเว็บ และเอกสารประกอบเครื่องมือ



```
sudo apt-get install debsums apt-listbugs needrestart apt-show-versions fail2ban unattended-upgrades clamav clamav-daemon rkhunter
```



ข้อมูลบางอย่างเกี่ยวกับแพ็คเกจที่ติดตั้ง:





- Clamav** เป็นโปรแกรมป้องกันไวรัส
- unattend-upgrades** จะช่วยให้คุณจัดการอัปเดตของคุณโดยอัตโนมัติและแม้กระทั่งรีบูตเครื่องหรือกำจัดแพ็คเกจเก่าโดยอัตโนมัติ ซึ่งสามารถปรับแต่งได้อย่างเต็มที่
- rkhunter** เป็นโปรแกรมป้องกัน rootkit ที่สแกนระบบไฟล์ของคุณ
- Fail2ban** จะอ้างอิงจากไฟล์บันทึกของคุณตามที่คุณให้มันอ่านและจะทำงานร่วมกับ **iptables** เช่น เพื่อแบนที่อยู่ IP ที่พยายาม "brute force" เซิร์ฟเวอร์ของคุณใน SSH



### ค. ข้อแนะนำสำหรับ SSH



มาดูคำแนะนำเกี่ยวกับ SSH กันเถอะ รายการมีดังต่อไปนี้ ไม่ต้องกังวล เราจะอธิบายทันทีว่าจะปรับปรุงการกำหนดค่าอย่างไร



![Image](assets/fr/034.webp)



มาดูการตั้งค่า **SSH** ปัจจุบันของฉันอย่างละเอียดใน:**/etc/ssh/sshd_config**



![Image](assets/fr/018.webp)



การกำหนดค่าที่แนะนำด้านล่างยังสามารถปรับให้เหมาะสมได้ แต่ให้ฐานที่ดีแก่คุณ *โปรดทราบว่าฉันได้ลบความคิดเห็นจำนวนหนึ่งเพื่อให้อ่านง่ายขึ้น*



เราจะ:





- เปลี่ยนพอร์ตการเชื่อมต่อ SSH (ลืมพอร์ตเริ่มต้น 22)
- เพิ่มระดับความละเอียดของบันทึกจาก **INFO เป็น VERBOSE**
- ตั้งค่า **LoginGraceTime** เป็น **2 นาที**



หากไม่มีการป้อนข้อมูลการเชื่อมต่อภายในสองนาที การเชื่อมต่อจะถูกตัดออก





- เปิดใช้งาน **strictModes**



ระบุว่า "sshd" ควรตรวจสอบโหมดและเจ้าของไฟล์ของผู้ใช้รวมถึงไดเรกทอรีบ้านของผู้ใช้ก่อนที่จะตรวจสอบการเชื่อมต่อหรือไม่ โดยปกติแล้วสิ่งนี้เป็นที่ต้องการ เพราะบางครั้งผู้เริ่มต้นอาจเผลอเปิดให้ไดเรกทอรีหรือไฟล์ของตนสามารถเข้าถึงได้โดยทุกคน ค่าเริ่มต้นคือ "yes"





- ตั้งค่า **MaxAuthtries** เป็น 3



นี่แสดงถึงจำนวนครั้งที่การยืนยันตัวตนล้มเหลวก่อนที่การสื่อสารจะถูกปิด





- ตั้งค่า **MaxSessions** เป็น 2



นี่แสดงถึงจำนวนเซสชันพร้อมกันสูงสุด





- เปิดใช้งานการตรวจสอบสิทธิ์ด้วยคีย์สาธารณะ



```
PubkeyAuthentication yes
```





- รักษาการยืนยันตัวตนด้วยรหัสผ่าน:



```
PasswordAuthentication yes
```



ห้ามใช้รหัสผ่านที่ว่างเปล่าและการตรวจสอบสิทธิ์ **Kerberos** รวมถึง **การตรวจสอบสิทธิ์ root โดยตรง**



```
PermitEmptyPasswords no
PermitRootLogin no
```



ตรวจสอบให้แน่ใจว่าคุณมี "**PermitRootLogin no", ถ้ามันเท่ากับ "yes", มันคือ "absolute evil "**.





- ห้ามการเปลี่ยนเส้นทางการเชื่อมต่อ TCP



```
AllowTcpForwarding no
```



ระบุว่าการเปลี่ยนเส้นทาง TCP ได้รับอนุญาตหรือไม่ โดยมี "yes" เป็นการตั้งค่าเริ่มต้น โปรดทราบ: การปิดใช้งานการเปลี่ยนเส้นทาง TCP ไม่ได้เพิ่มความปลอดภัยหากผู้ใช้สามารถเข้าถึงเชลล์ได้ เนื่องจากพวกเขายังสามารถตั้งค่าเครื่องมือเปลี่ยนเส้นทางของตนเองได้





- ห้ามการเปลี่ยนเส้นทาง X11



```
X11Forwarding no
```



ระบุว่าการเปลี่ยนเส้นทาง X11 ได้รับการยอมรับหรือไม่ โดยมีค่าเริ่มต้นเป็น "no" โปรดทราบ: แม้ว่าการเปลี่ยนเส้นทาง X11 จะถูกปิดใช้งาน แต่ก็ไม่ได้เพิ่มความปลอดภัย เนื่องจากผู้ใช้ยังสามารถตั้งค่าตัวเปลี่ยนเส้นทางของตนเองได้ การเปลี่ยนเส้นทาง X11 จะถูกปิดใช้งานโดยอัตโนมัติหากเลือก **UseLogin**





- ตั้งค่าการหมดเวลาการสื่อสารระหว่างไคลเอนต์และ sshd



```
ClientAliveInterval  300
```



กำหนดระยะเวลาเป็นวินาที หลังจากนั้นหากไม่ได้รับข้อมูลจากไคลเอนต์ บริการ sshd จะส่งข้อความขอการตอบกลับจากไคลเอนต์ โดยค่าเริ่มต้น ตัวเลือกนี้ถูกตั้งค่าเป็น "0" ซึ่งหมายความว่าจะไม่มีการส่งข้อความเหล่านี้ไปยังไคลเอนต์ มีเพียงเวอร์ชัน 2 ของโปรโตคอล SSH เท่านั้นที่รองรับตัวเลือกนี้



```
ClientAliveCountMax 0
```



ตาม [เอกสาร (*man page*) สำหรับ sshd](https://www.delafond.org/traducmanfr/man/man5/sshd_config.5.html) นี่คือความหมายของตัวเลือกนี้: "ตั้งค่าจำนวนข้อความถือ (ดูด้านบน) ที่จะส่งโดยไม่มีการตอบกลับจากลูกค้าสำหรับ **sshd** หากถึงเกณฑ์นี้ในขณะที่ข้อความถือถูกส่งไปแล้ว **sshd** จะตัดการเชื่อมต่อลูกค้าและยุติการเชื่อมต่อ สิ่งสำคัญคือต้องทราบว่าข้อความถือเหล่านี้แตกต่างจากตัวเลือก **KeepAlive** (ด้านล่าง) อย่างมาก ข้อความถือการเชื่อมต่อจะถูกส่งผ่านอุโมงค์ที่เข้ารหัส และดังนั้นจึงไม่สามารถปลอมแปลงได้ การถือการเชื่อมต่อระดับ TCP ที่เปิดใช้งานโดย **KeepAlive** สามารถปลอมแปลงได้ กลไกการถือการเชื่อมต่อเป็นที่สนใจเมื่อไคลเอนต์หรือเซิร์ฟเวอร์จำเป็นต้องทราบว่าการเชื่อมต่อว่างหรือไม่"





- ป้องกันการเปิดเผยข้อมูลโดยการปิดใช้งาน **motd, banner, lastlog**



```
PrintMotd no
```



ระบุว่า sshd ควรแสดงเนื้อหาของไฟล์ "/etc/motd" หรือไม่เมื่อผู้ใช้เข้าสู่ระบบในโหมดโต้ตอบ บนบางระบบ เนื้อหานี้อาจถูกแสดงโดย shell ผ่าน /etc/profile หรือไฟล์ที่คล้ายกัน ค่าเริ่มต้นคือ "yes"



```
Banner none
```



ควรสังเกตว่าในบางเขตอำนาจศาล การส่งข้อความก่อนการตรวจสอบสิทธิ์อาจเป็นข้อกำหนดเบื้องต้นสำหรับการคุ้มครองทางกฎหมาย เนื้อหาของไฟล์ที่ระบุจะถูกส่งไปยังผู้ใช้ระยะไกลก่อนที่จะได้รับอนุญาตการเชื่อมต่อ จำเป็นต้องกำหนดค่าตัวเลือกนี้ เนื่องจากโดยค่าเริ่มต้นจะไม่มีการแสดงข้อความใดๆ

## V. Lynis การวางแผนอัตโนมัติ



**Lynis** ยังมีตัวเลือกในการเรียกใช้การตรวจสอบผ่านงานที่กำหนดเวลาไว้ โดยมีตัวเลือก **"--cronjob "** ซึ่งจะเรียกใช้การทดสอบทั้งหมดของ **Lynis** โดยไม่จำเป็นต้องมีการตรวจสอบหรือการกระทำของผู้ใช้ จากนั้นคุณสามารถสร้างสคริปต์อย่างง่ายดายที่จะเรียกใช้ **Lynis** และใส่ผลลัพธ์ในไฟล์ที่มีการประทับเวลาและชื่อของเซิร์ฟเวอร์ที่เกี่ยวข้อง นี่คือตัวอย่างไฟล์ประเภทนี้ที่คุณสามารถใส่ในโฟลเดอร์ */etc/cron.daily*:



```
#!/bin/sh
mkdir /var/log/lynis
NOM_AUDITEUR="tache_crontab"
DATE=$(date +%Y%m%d)
HOTE=$(hostname)
LOG_DIR="/var/log/Lynis"
RAPPORT="$LOG_DIR/rapport-${HOTE}.${DATE}"
DATA="$LOG_DIR/rapport-data-${HOTE}.${DATE}.txt"

cd /root/Lynis./Lynis -c --auditor "${NOM_AUDITEUR}" --cronjob > ${RAPPORT}
mv /var/log/lynis-report.dat ${DATA}
```



ตัวแปร **"AUDITOR_NAME "** เป็นเพียงตัวแปรที่เราจะตั้งค่าในตัวเลือก **"--auditor "** ของ **Lynis** เพื่อให้ชื่อนี้แสดงในรายงาน:



![Image](assets/fr/059.webp)



เรายังจะสร้างตัวแปรเชิงบริบทบางตัวที่จะช่วยให้เราจัดระเบียบตัวเองได้ดีขึ้น เช่น ชื่อโฮสต์และวันที่สำหรับตั้งชื่อรายงานและประทับเวลา และเส้นทางไปยังโฟลเดอร์ที่เราต้องการใส่รายงานของเรา

Lynis เป็นเครื่องมือที่มีประโยชน์มากที่จะช่วยให้คุณประหยัดเวลาและมีประสิทธิภาพมากขึ้นเมื่อคุณต้องการทราบข้อมูลเพิ่มเติมเกี่ยวกับสถานะการกำหนดค่าของเซิร์ฟเวอร์ Linux โดยเฉพาะในด้านความปลอดภัย อย่างไรก็ตาม อย่าลืมว่าการปรับเปลี่ยนทุกครั้งต้องได้รับการทดสอบก่อนที่จะนำไปใช้ในสภาพแวดล้อมการผลิต โดยคำนึงถึงการใช้งานและบริบทของคุณเอง



คุณอาจจะไม่ใช้การตั้งค่าเดียวกันสำหรับ VPS ที่เปิดเผยต่ออินเทอร์เน็ต ซึ่งคุณต้องการเพียงการเชื่อมต่อ SSH หนึ่งครั้ง (เพราะคุณเป็นคนเดียวที่เชื่อมต่อ) ต่างจาก **bastion** หรือ **scheduler** ที่จะต้องมีการเชื่อมต่อ **SSH.** หลายครั้ง



เมื่อคุณได้การตั้งค่าที่เหมาะสมกับคุณในแง่ของการเพิ่มความปลอดภัยแล้ว ขอแนะนำให้ใช้เครื่องมืออัตโนมัติเพื่อที่คุณจะได้ไม่ต้องทำงานซ้ำด้วยตนเอง เนื่องจากจะใช้เวลานานและมีโอกาสเกิดข้อผิดพลาดได้ ตัวอย่างเช่น คุณสามารถใช้ **: Puppet, Chef, Ansible, etc...**



อย่าลืมสื่อสารกับทีมของคุณก่อนการดำเนินการ: คุณจำเป็นต้องทำให้พวกเขาเข้าใจว่าทำไมคุณถึงทำการเปลี่ยนแปลงเหล่านี้ ไม่ใช่แค่บอกพวกเขาว่าคุณกำลังทำอะไร สุดท้ายแล้ว เป้าหมายคือการถ่ายทอดแนวปฏิบัติที่ดี และสิ่งนี้จะเพิ่มโอกาสในการประสบความสำเร็จของคุณ
