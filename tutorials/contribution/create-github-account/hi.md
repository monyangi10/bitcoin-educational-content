प्लान ₿ का मिशन Bitcoin के बारे में शीर्ष-स्तरीय शैक्षिक संसाधन प्रदान करना है, जो यथासंभव अधिक से अधिक भाषाओं में उपलब्ध हो। साइट पर प्रकाशित सभी सामग्री ओपन-सोर्स है और GitHub पर होस्ट की गई है, जो किसी को भी प्लेटफ़ॉर्म के संवर्धन में योगदान करने का अवसर प्रदान करती है। योगदान विभिन्न रूपों में हो सकते हैं: मौजूदा पाठों को सही करना और प्रूफ़रीडिंग करना, अन्य भाषाओं में अनुवाद करना, जानकारी अपडेट करना, या हमारी साइट पर अभी तक उपलब्ध नहीं होने वाले नए ट्यूटोरियल बनाना।


यदि आप Plan ₿ Network में योगदान देना चाहते हैं, तो आपको Git और GitHub का उपयोग करना होगा, इसलिए, यदि ये उपकरण आपके लिए अपरिचित हैं या यदि उनकी कार्यप्रणाली अस्पष्ट लगती है, तो घबराएँ नहीं, यह लेख आपके लिए है! हम Git और GitHub के मूल सिद्धांतों के साथ-साथ संबंधित तकनीकी शब्दावली की समीक्षा करेंगे, ताकि आप बाद में उन्हें प्रभावी ढंग से उपयोग कर सकें।


## गिट क्या है?


Git एक संस्करण नियंत्रण प्रणाली है, जिसे विशेष रूप से सॉफ़्टवेयर प्रोजेक्ट्स को प्रबंधित करने के लिए डिज़ाइन किया गया है। 2005 में लिनस टोरवाल्ड्स द्वारा बनाया गया, Git जल्दी ही सॉफ़्टवेयर विकास उद्योग में संस्करण नियंत्रण के लिए मानक बन गया है। लेकिन इसका वास्तव में क्या मतलब है?

![git](assets/11.webp)

इसके मूल में, Git डेवलपर्स को समय के साथ किसी प्रोजेक्ट के सोर्स कोड में किए गए परिवर्तनों को ट्रैक करने की अनुमति देता है। इसका मतलब है कि कोड में हर बदलाव के साथ, Git प्रोजेक्ट का एक नया संस्करण रिकॉर्ड करता है। यदि कोई त्रुटि होती है या यदि कोई प्रायोगिक सुविधा अपेक्षित रूप से काम नहीं करती है, तो कोड की पिछली स्थिति पर वापस जाना संभव है, एक तरह की टाइम मशीन की तरह, लेकिन फ़ाइलों के लिए।


Git की प्रमुख विशेषताओं में से एक शाखा प्रबंधन है। शाखा एक तरह की समानांतर रेखा है जहाँ डेवलपर्स बाकी प्रोजेक्ट से स्वतंत्र रूप से काम कर सकते हैं। यह मुख्य कोड को बाधित किए बिना नई सुविधाओं को जोड़ने या बग को ठीक करने में बहुत सुविधा प्रदान करता है। एक बार संशोधनों का परीक्षण और सत्यापन हो जाने के बाद, उन्हें मुख्य शाखा के साथ विलय किया जा सकता है।


Git की एक ख़ासियत यह है कि यह वितरित तरीके से काम करने में सक्षम है। प्रत्येक डेवलपर के पास अपने कंप्यूटर के Hard ड्राइव पर प्रोजेक्ट की पूरी कॉपी होती है, जिससे वे ऑफ़लाइन काम कर सकते हैं और बाद में इंटरनेट कनेक्शन उपलब्ध होने पर बदलावों को मर्ज कर सकते हैं। इससे टकराव का जोखिम कम हो जाता है और कई डेवलपर्स एक ही प्रोजेक्ट पर एक साथ काम कर सकते हैं, बिना एक-दूसरे के पैरों में खटकने के।

शुरुआत में, Git को मुख्य रूप से सॉफ़्टवेयर डेवलपमेंट प्रोजेक्ट के लिए डिज़ाइन किया गया था। हालाँकि, इसका उपयोग कंटेंट राइटिंग प्रोजेक्ट को मैनेज करने के लिए भी किया जा सकता है। कोड पर सहयोग करने के बजाय, हम टेक्स्ट पर सहयोग करते हैं। और यह ठीक यही तरीका है जिसे Plan ₿ Network ने अपनी सामग्री को मैनेज करने के लिए अपनाया है! Git लेखन पाठ्यक्रम और ट्यूटोरियल पर सहयोग की सुविधा देता है, क्योंकि यह परिवर्तनों की सटीक ट्रैकिंग, कुशल संस्करण प्रबंधन की अनुमति देता है, और अन्य योगदानकर्ताओं द्वारा सामग्री की समीक्षा और सुधार को भी सक्षम बनाता है।


## GitHub क्या है?


GitHub एक सोर्स कोड प्रबंधन और होस्टिंग प्लेटफ़ॉर्म है जो Git संस्करण नियंत्रण प्रणाली पर आधारित है जिसकी हमने अभी चर्चा की है। 2008 में लॉन्च किए गए GitHub ने तेज़ी से लोकप्रियता हासिल की और दुनिया भर के डेवलपर्स के लिए एक ज़रूरी संदर्भ बन गया। लेकिन GitHub, Git से किस तरह अलग है और यह हमारी सामग्री उत्पादन पद्धति में इतना महत्वपूर्ण क्यों है?

![github](assets/12.webp)

सबसे पहले, यह समझना महत्वपूर्ण है कि GitHub Git पर बनाया गया है (जिस पर हमने पिछले अनुभाग में चर्चा की थी)। जबकि Git वह टूल है जो कोड परिवर्तनों को ट्रैक करता है, GitHub वह ऑनलाइन सेवा है जो इस कोड को होस्ट, शेयर और प्रबंधित करती है।


Git को एक तरह की लॉगबुक के रूप में कल्पना करें जिसका उपयोग प्रत्येक डेवलपर अपने कंप्यूटर पर अपने प्रोजेक्ट में सभी परिवर्तनों को रिकॉर्ड करने के लिए करता है। दूसरी ओर, GitHub एक सार्वजनिक लाइब्रेरी की तरह है जहाँ इन सभी लॉगबुक को साझा, तुलना और संयोजित किया जा सकता है।


Git और GitHub के बीच मूलभूत अंतर उनके कार्य में निहित है: Git प्रत्येक डेवलपर द्वारा अपने कोड संस्करणों को प्रबंधित करने के लिए स्थानीय रूप से उपयोग किया जाने वाला उपकरण है, जबकि GitHub एक ऑनलाइन प्लेटफ़ॉर्म है जो इन संस्करणों को होस्ट करता है और सहयोग की सुविधा प्रदान करता है।


GitHub सिर्फ़ एक कोड होस्टिंग सेवा से कहीं ज़्यादा है। यह एक सहयोग मंच है जो डेवलपर्स को कुशलतापूर्वक एक साथ काम करने की अनुमति देता है। और वास्तव में, Plan ₿ Network इस प्लेटफ़ॉर्म का उपयोग न केवल वेबसाइट को संचालित करने वाले सभी कोड को होस्ट करने के लिए करता है, बल्कि यह भी कि, और यही हमारी दिलचस्पी है, सभी सामग्री (ट्यूटोरियल, प्रशिक्षण, संसाधन...) को होस्ट करता है।


## कुछ तकनीकी शब्द


Git और GitHub पर, आपको ऐसे कमांड और फ़ीचर मिलेंगे जिनके नाम जटिल लग सकते हैं। इस अंतिम भाग में, मैं कुछ सरल परिभाषाएँ प्रदान करूँगा ताकि आप Git और GitHub का उपयोग करते समय आने वाले तकनीकी शब्दों को समझ सकें:



- फ़ेच ओरिजिन:** कमांड जो रिमोट रिपॉजिटरी से हाल की जानकारी और परिवर्तनों को आपके स्थानीय कार्य के साथ मर्ज किए बिना पुनर्प्राप्त करता है। यह आपके स्थानीय रिपॉजिटरी को रिमोट रिपॉजिटरी में मौजूद नई शाखाओं और कमिट्स के साथ अपडेट करता है।



- पुल ओरिजिन:** कमांड जो रिमोट रिपोजिटरी से अपडेट प्राप्त करता है और तुरंत उन्हें आपके स्थानीय शाखा में एकीकृत करता है ताकि इसे सिंक्रनाइज़ किया जा सके। यह फ़ेच और मर्ज के चरणों को एक ही कमांड में जोड़ता है।
- सिंक Fork:** GitHub पर एक सुविधा जो आपको स्रोत रिपॉजिटरी से नवीनतम परिवर्तनों के साथ किसी प्रोजेक्ट के Fork को अपडेट करने की अनुमति देती है। यह सुनिश्चित करता है कि प्रोजेक्ट की आपकी कॉपी मुख्य विकास के साथ अद्यतित रहे।
- पुश ओरिजिन:** यह कमांड आपके स्थानीय परिवर्तनों को दूरस्थ रिपोजिटरी में भेजने के लिए उपयोग किया जाता है।



- पुल अनुरोध:** किसी योगदानकर्ता द्वारा भेजा गया अनुरोध, जिसमें यह इंगित किया जाता है कि उन्होंने दूरस्थ रिपोजिटरी की एक शाखा में परिवर्तन किए हैं और वे चाहते हैं कि इन परिवर्तनों की समीक्षा की जाए और उन्हें संभवतः रिपोजिटरी की मुख्य शाखा में विलय कर दिया जाए।



- कमिट:** आपके परिवर्तनों को सहेजना। कमिट किसी निश्चित समय पर आपके काम का एक त्वरित स्नैपशॉट होता है, जो परिवर्तनों का इतिहास रखने की अनुमति देता है। प्रत्येक कमिट में एक वर्णनात्मक संदेश शामिल होता है जो बताता है कि क्या संशोधित किया गया है।



- शाखा:** रिपॉजिटरी का एक समानांतर संस्करण, जो आपको मुख्य शाखा (जिसे अक्सर "मुख्य" या "मास्टर" कहा जाता है) को प्रभावित किए बिना परिवर्तनों पर काम करने की अनुमति देता है। शाखाएँ स्थिर कोड को बाधित करने के जोखिम के बिना नई सुविधाओं के विकास और बगों के सुधार की सुविधा प्रदान करती हैं।



- मर्ज:** मर्जिंग में एक शाखा से दूसरे में किए गए परिवर्तनों को एकीकृत करना शामिल है। इसका उपयोग, उदाहरण के लिए, एक कार्यशील शाखा से किए गए परिवर्तनों को मुख्य शाखा में जोड़ने के लिए किया जाता है, जो विभिन्न योगदानों को जोड़ने की अनुमति देता है।



- Fork:** रिपॉजिटरी को फ़ॉर्क करने का मतलब है अपने खुद के GitHub अकाउंट पर उस रिपॉजिटरी की एक कॉपी बनाना, जो आपको मूल रिपॉजिटरी को प्रभावित किए बिना प्रोजेक्ट पर काम करने की अनुमति देता है। Fork या तो अपने तरीके से आगे बढ़ सकता है और मूल से अलग प्रोजेक्ट बन सकता है, या यह नियमित रूप से मूल प्रोजेक्ट के साथ सिंक्रोनाइज़ करके उसमें योगदान दे सकता है।



- क्लोन:** रिपॉजिटरी को क्लोन करने का मतलब है अपने कंप्यूटर पर एक स्थानीय प्रतिलिपि बनाना, जो आपको सभी फ़ाइलों और इतिहास तक पहुँच प्रदान करता है। यह आपको सीधे स्थानीय रूप से प्रोजेक्ट पर काम करने की अनुमति देता है।



- रिपॉजिटरी:** GitHub पर किसी प्रोजेक्ट के लिए स्टोरेज स्पेस। रिपॉजिटरी में सभी प्रोजेक्ट फ़ाइलें और साथ ही उसमें किए गए सभी बदलावों का इतिहास होता है। यह GitHub पर स्टोरेज और सहयोग का आधार है।



- समस्या:** GitHub पर कार्यों और बग्स को ट्रैक करने के लिए एक उपकरण। समस्याएँ समस्याओं की रिपोर्ट करने, सुधार का प्रस्ताव देने या नई सुविधाओं पर चर्चा करने की अनुमति देती हैं। प्रत्येक समस्या को असाइन किया जा सकता है, लेबल किया जा सकता है और उस पर टिप्पणी की जा सकती है।


यह सूची स्पष्ट रूप से संपूर्ण नहीं है। Git और GitHub के लिए कई अन्य तकनीकी शब्द भी हैं। हालाँकि, यहाँ बताए गए शब्द मुख्य हैं जो आपको अक्सर मिलेंगे।


इस लेख को पढ़ने के बाद, यह संभव है कि Git और GitHub के कुछ पहलू अभी भी आपके लिए अस्पष्ट हों, इसलिए मैं आपको इन उपकरणों का उपयोग स्वयं शुरू करने के लिए प्रोत्साहित करता हूँ। मशीन कैसे काम करती है, यह समझने का सबसे अच्छा तरीका अक्सर अभ्यास होता है! और आरंभ करने के लिए, आप ये 2 अन्य ट्यूटोरियल खोज सकते हैं:
