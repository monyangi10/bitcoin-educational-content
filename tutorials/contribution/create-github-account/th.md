พันธกิจของ Plan ₿ คือการจัดหาทรัพยากรการศึกษาชั้นนำเกี่ยวกับ Bitcoin ให้พร้อมใช้งานในหลายภาษามากที่สุด เนื้อหาทั้งหมดที่เผยแพร่บนเว็บไซต์เป็นโอเพ่นซอร์สและโฮสต์บน GitHub เปิดโอกาสให้ทุกคนมีส่วนร่วมในการพัฒนาแพลตฟอร์ม การมีส่วนร่วมสามารถทำได้หลายรูปแบบ: การแก้ไขและพิสูจน์อักษรข้อความที่มีอยู่ การแปลเป็นภาษาอื่น ๆ การอัปเดตข้อมูล หรือการสร้างบทเรียนใหม่ที่ยังไม่มีในเว็บไซต์ของเรา


หากคุณต้องการมีส่วนร่วมใน Plan ₿ Academy คุณจะต้องใช้ Git และ GitHub ดังนั้น หากเครื่องมือเหล่านี้ไม่คุ้นเคยกับคุณหรือหากการทำงานของมันดูคลุมเครือ อย่าตกใจ บทความนี้เหมาะสำหรับคุณ! เราจะทบทวนพื้นฐานของ Git และ GitHub รวมถึงศัพท์เทคนิคที่เกี่ยวข้อง เพื่อให้คุณสามารถใช้งานได้อย่างมีประสิทธิภาพในภายหลัง


## Git คืออะไร?


Git เป็นระบบควบคุมเวอร์ชันที่ออกแบบมาโดยเฉพาะเพื่อจัดการโครงการซอฟต์แวร์ สร้างขึ้นในปี 2005 โดย Linus Torvalds Git ได้กลายเป็นมาตรฐานในอุตสาหกรรมการพัฒนาซอฟต์แวร์สำหรับการควบคุมเวอร์ชันอย่างรวดเร็ว แต่มันหมายถึงอะไรจริงๆ?

![git](assets/11.webp)

ที่แก่นแท้ของมัน Git ช่วยให้นักพัฒนาสามารถติดตามการเปลี่ยนแปลงที่เกิดขึ้นกับซอร์สโค้ดของโปรเจกต์ตลอดเวลา ซึ่งหมายความว่าทุกครั้งที่มีการเปลี่ยนแปลงในโค้ด Git จะบันทึกเวอร์ชันใหม่ของโปรเจกต์ หากเกิดข้อผิดพลาดหรือฟีเจอร์ทดลองไม่ทำงานตามที่คาดหวัง สามารถย้อนกลับไปยังสถานะก่อนหน้าของโค้ดได้ เหมือนกับเครื่องย้อนเวลา แต่สำหรับไฟล์


หนึ่งในคุณสมบัติหลักของ Git คือการจัดการสาขา สาขาคือเส้นขนานชนิดหนึ่งที่นักพัฒนาสามารถทำงานได้อย่างอิสระจากส่วนที่เหลือของโครงการ สิ่งนี้ช่วยอำนวยความสะดวกอย่างมากในการเพิ่มคุณสมบัติใหม่หรือการแก้ไขข้อบกพร่องโดยไม่รบกวนโค้ดหลัก เมื่อการแก้ไขได้รับการทดสอบและตรวจสอบแล้ว สามารถรวมเข้ากับสาขาหลักได้


หนึ่งในลักษณะเฉพาะของ Git คือความสามารถในการทำงานในลักษณะกระจาย แต่ละนักพัฒนาจะมีสำเนาครบถ้วนของโครงการบนฮาร์ดไดรฟ์ของคอมพิวเตอร์ของตนเอง ทำให้สามารถทำงานแบบออฟไลน์และรวมการเปลี่ยนแปลงในภายหลังเมื่อมีการเชื่อมต่ออินเทอร์เน็ตได้ สิ่งนี้ช่วยลดความเสี่ยงของความขัดแย้งและอนุญาตให้นักพัฒนาหลายคนทำงานพร้อมกันในโครงการเดียวกันโดยไม่ก้าวก่ายกันและกัน

ในตอนแรก Git ได้รับการออกแบบมาเพื่อโครงการพัฒนาซอฟต์แวร์เป็นหลัก อย่างไรก็ตาม มันยังสามารถใช้จัดการโครงการเขียนเนื้อหาได้อีกด้วย แทนที่จะร่วมมือกันในโค้ด เราร่วมมือกันในข้อความ และนี่คือวิธีที่ Plan ₿ Academy นำมาใช้เพื่อจัดการเนื้อหาของตน! Git ช่วยอำนวยความสะดวกในการร่วมมือกันเขียนคอร์สและบทเรียน เนื่องจากมันช่วยให้สามารถติดตามการเปลี่ยนแปลงได้อย่างแม่นยำ จัดการเวอร์ชันได้อย่างมีประสิทธิภาพ และยังช่วยให้ผู้ร่วมงานคนอื่นๆ สามารถตรวจสอบและปรับปรุงเนื้อหาได้อีกด้วย


## GitHub คืออะไร?


GitHub เป็นแพลตฟอร์มการจัดการและโฮสต์ซอร์สโค้ดที่อิงตามระบบควบคุมเวอร์ชัน Git ที่เราเพิ่งพูดถึง เปิดตัวในปี 2008 GitHub ได้รับความนิยมอย่างรวดเร็วและกลายเป็นแหล่งอ้างอิงที่สำคัญสำหรับนักพัฒนาทั่วโลก แต่ GitHub แตกต่างจาก Git อย่างไร และทำไมมันถึงมีความสำคัญมากในวิธีการผลิตเนื้อหาของเรา?

![github](assets/12.webp)

ก่อนอื่น สิ่งสำคัญคือต้องเข้าใจว่า GitHub ถูกสร้างขึ้นบนพื้นฐานของ Git (ซึ่งเราได้พูดถึงในส่วนก่อนหน้า) ในขณะที่ Git เป็นเครื่องมือที่ติดตามการเปลี่ยนแปลงของโค้ด GitHub เป็นบริการออนไลน์ที่โฮสต์ แชร์ และจัดการโค้ดนี้


ลองจินตนาการว่า Git เป็นเหมือนสมุดบันทึกที่นักพัฒนาแต่ละคนใช้บนคอมพิวเตอร์ของตนเองเพื่อบันทึกการเปลี่ยนแปลงทั้งหมดในโครงการของพวกเขา ในทางกลับกัน GitHub เป็นเหมือนห้องสมุดสาธารณะที่สมุดบันทึกเหล่านี้สามารถแชร์ เปรียบเทียบ และรวมกันได้


ความแตกต่างพื้นฐานระหว่าง Git และ GitHub อยู่ที่หน้าที่ของพวกเขา: Git เป็นเครื่องมือที่ใช้ในเครื่องโดยนักพัฒนาแต่ละคนเพื่อจัดการเวอร์ชันของโค้ดของพวกเขา ในขณะที่ GitHub เป็นแพลตฟอร์มออนไลน์ที่โฮสต์เวอร์ชันเหล่านี้และอำนวยความสะดวกในการทำงานร่วมกัน


GitHub เป็นมากกว่าแค่บริการโฮสต์โค้ด มันเป็นแพลตฟอร์มการทำงานร่วมกันที่ช่วยให้นักพัฒนาสามารถทำงานร่วมกันได้อย่างมีประสิทธิภาพ และแน่นอนว่า Plan ₿ Academy ใช้แพลตฟอร์มนี้เพื่อโฮสต์ไม่เพียงแค่โค้ดทั้งหมดที่ขับเคลื่อนเว็บไซต์เท่านั้น แต่ยังรวมถึงเนื้อหาทั้งหมด (บทเรียน การฝึกอบรม ทรัพยากร...) ซึ่งเป็นสิ่งที่เราสนใจด้วย


## คำศัพท์ทางเทคนิคบางคำ


บน Git และ GitHub คุณจะพบกับคำสั่งและฟีเจอร์ที่มีชื่อซับซ้อน ในส่วนสุดท้ายนี้ ฉันจะให้คำจำกัดความง่ายๆ เพื่อทำความเข้าใจคำศัพท์ทางเทคนิคที่คุณอาจพบเมื่อใช้ Git และ GitHub:



- ดึงข้อมูลต้นทาง:** คำสั่งที่ดึงข้อมูลและการเปลี่ยนแปลงล่าสุดจากที่เก็บระยะไกลโดยไม่รวมเข้ากับงานในเครื่องของคุณ มันจะอัปเดตที่เก็บในเครื่องของคุณด้วยสาขาและคอมมิตใหม่ที่มีอยู่ในที่เก็บระยะไกล



- ดึงต้นทาง:** คำสั่งที่ดึงการอัปเดตจากที่เก็บระยะไกลและผสานเข้ากับสาขาท้องถิ่นของคุณทันทีเพื่อให้สอดคล้องกัน ซึ่งรวมขั้นตอนของการดึงข้อมูลและการผสานเข้าด้วยกันในคำสั่งเดียว
- Sync Fork:** ฟีเจอร์บน GitHub ที่ช่วยให้คุณอัปเดต fork ของโปรเจกต์ด้วยการเปลี่ยนแปลงล่าสุดจากที่เก็บต้นทาง ซึ่งจะช่วยให้สำเนาของโปรเจกต์ของคุณทันสมัยอยู่เสมอกับการพัฒนาหลัก
- Push origin:** คำสั่งที่ใช้เพื่อส่งการเปลี่ยนแปลงในเครื่องของคุณไปยังที่เก็บระยะไกล



- คำขอการดึง:** คำขอที่ส่งโดยผู้ร่วมพัฒนาเพื่อระบุว่าพวกเขาได้ส่งการเปลี่ยนแปลงไปยังสาขาในที่เก็บระยะไกลและต้องการให้มีการตรวจสอบการเปลี่ยนแปลงเหล่านี้และอาจรวมเข้ากับสาขาหลักของที่เก็บนั้น



- คอมมิต:** บันทึกการเปลี่ยนแปลงของคุณ คอมมิตเปรียบเสมือนภาพถ่ายทันทีของงานของคุณในช่วงเวลาหนึ่ง ซึ่งช่วยให้สามารถเก็บประวัติการเปลี่ยนแปลงได้ แต่ละคอมมิตจะมีข้อความอธิบายที่บอกว่ามีการแก้ไขอะไรบ้าง



- สาขา:** เวอร์ชันคู่ขนานของที่เก็บข้อมูล ช่วยให้คุณทำงานกับการเปลี่ยนแปลงโดยไม่กระทบต่อสาขาหลัก (มักเรียกว่า "main" หรือ "master") สาขาช่วยอำนวยความสะดวกในการพัฒนาฟีเจอร์ใหม่และการแก้ไขข้อบกพร่องโดยไม่เสี่ยงต่อการรบกวนโค้ดที่เสถียร



- ผสาน:** การผสานประกอบด้วยการรวมการเปลี่ยนแปลงจากสาขาหนึ่งไปยังอีกสาขาหนึ่ง ใช้เพื่อ, ตัวอย่างเช่น, เพิ่มการเปลี่ยนแปลงจากสาขาการทำงานไปยังสาขาหลัก, ซึ่งช่วยให้สามารถเพิ่มการมีส่วนร่วมต่างๆ ได้.



- Fork:** การ Forking repository หมายถึงการสร้างสำเนาของ repository นั้นบนบัญชี GitHub ของคุณเอง ซึ่งช่วยให้คุณสามารถทำงานบนโปรเจกต์ได้โดยไม่กระทบต่อ repository ต้นฉบับ fork สามารถเลือกที่จะพัฒนาไปในทิศทางของตัวเองและกลายเป็นโปรเจกต์ที่แตกต่างจากต้นฉบับ หรือสามารถซิงโครไนซ์กับโปรเจกต์ต้นฉบับเป็นประจำเพื่อมีส่วนร่วมกับมัน



- โคลน:** การโคลนที่เก็บข้อมูลหมายถึงการสร้างสำเนาในเครื่องบนคอมพิวเตอร์ของคุณ ซึ่งจะทำให้คุณเข้าถึงไฟล์ทั้งหมดและประวัติได้ สิ่งนี้ช่วยให้คุณทำงานกับโปรเจกต์ได้โดยตรงในเครื่องของคุณ



- ที่เก็บ:** พื้นที่จัดเก็บสำหรับโครงการบน GitHub ที่เก็บประกอบด้วยไฟล์ทั้งหมดของโครงการรวมถึงประวัติของการเปลี่ยนแปลงทั้งหมดที่ได้ทำไปแล้ว มันเป็นพื้นฐานของการจัดเก็บและการทำงานร่วมกันบน GitHub



- ปัญหา:** เครื่องมือสำหรับติดตามงานและบั๊กบน GitHub ปัญหาอนุญาตให้รายงานปัญหา เสนอการปรับปรุง หรืออภิปรายคุณสมบัติใหม่ แต่ละปัญหาสามารถกำหนดป้ายกำกับและแสดงความคิดเห็นได้


รายการนี้เห็นได้ชัดว่าไม่ครบถ้วนสมบูรณ์ ยังมีคำศัพท์ทางเทคนิคอื่น ๆ ที่เฉพาะเจาะจงกับ Git และ GitHub อีกมากมาย อย่างไรก็ตาม คำที่กล่าวถึงที่นี่คือคำหลักที่คุณจะพบเจอบ่อยครั้ง


หลังจากอ่านบทความนี้แล้ว อาจมีบางแง่มุมของ Git และ GitHub ที่ยังไม่ชัดเจนสำหรับคุณ ดังนั้นฉันขอแนะนำให้คุณเริ่มใช้เครื่องมือเหล่านี้ด้วยตัวคุณเอง การฝึกฝนมักเป็นวิธีที่ดีที่สุดในการเข้าใจว่าเครื่องทำงานอย่างไร! และเพื่อเริ่มต้น คุณสามารถค้นพบบทแนะนำอื่น ๆ อีก 2 รายการนี้:
