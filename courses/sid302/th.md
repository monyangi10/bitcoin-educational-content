# แนะนำเครือข่าย Liquid


เริ่มต้นการเดินทางทางการศึกษาที่ออกแบบมาเพื่อให้ความเข้าใจอย่างลึกซึ้งเกี่ยวกับโครงการ Liquid Network และ Elements บูตแคมป์นี้ผสมผสานทฤษฎีและการปฏิบัติเพื่อสอนคุณเกี่ยวกับพื้นฐานทางเทคนิค สถาปัตยกรรม และธุรกิจที่จำเป็นในการนำไปใช้และใช้ประโยชน์จากความสามารถของ Liquid ตั้งแต่ธุรกรรมที่เป็นความลับไปจนถึงการออกแบบระบบนิเวศ หลักสูตรนี้เหมาะสำหรับผู้ที่ต้องการขยายความรู้เกี่ยวกับเครื่องมือขั้นสูงภายในระบบนิเวศ Bitcoin


ด้วยการนำเสนอโดยผู้เชี่ยวชาญในอุตสาหกรรม หลักสูตรนี้ครอบคลุมหัวข้อต่างๆ เช่น สถาปัตยกรรม Liquid การประยุกต์ใช้โทเค็น แนวคิดทางเทคนิคของ Elements และกรณีการใช้งานนวัตกรรมเช่น Breez SDK ออกแบบมาเพื่อให้เข้าถึงได้สำหรับผู้เริ่มต้นและผู้ใช้ระดับกลาง หลักสูตรนี้ยังมอบคุณค่าให้กับนักพัฒนาที่มีประสบการณ์ที่ต้องการเชี่ยวชาญ Liquid ในฐานะแพลตฟอร์มสำหรับการเพิ่มประสิทธิภาพโครงการของพวกเขา

ยินดีต้อนรับสู่ค่ายฝึกอบรม Liquid การฝึกอบรมที่ครอบคลุมซึ่งออกแบบมาเพื่อให้คุณมีความรู้และทักษะในการใช้ประโยชน์จากโครงการ Liquid Network และ Elements อย่างมีประสิทธิภาพ หลักสูตรนี้นำเสนอการสำรวจคุณลักษณะเฉพาะของ Liquid อย่างครอบคลุม รวมถึง Confidential Transactions การออกสินทรัพย์ และสถาปัตยกรรมเครือข่ายแบบรวมศูนย์ พร้อมทั้งแนะนำแนวคิดพื้นฐานของ Elements ซอฟต์แวร์ที่ขับเคลื่อน Liquid


ตลอดระยะเวลาในค่ายฝึกอบรม คุณจะได้สำรวจการประยุกต์ใช้จริงของ Liquid Network ตั้งแต่การตั้งค่าและการดำเนินงานโหนดไปจนถึงการทำความเข้าใจการใช้งานในตลาดทุนของ Bitcoin และการโทเค็น คุณจะได้รับข้อมูลเชิงลึกจากผู้เชี่ยวชาญในอุตสาหกรรมเกี่ยวกับหัวข้อขั้นสูง เช่น HTLCs, Breez SDK และโครงการ Blockstream AMP


บูทแคมป์นี้เดิมทีจัดขึ้นเป็นกิจกรรมแบบพบปะกันตามตารางเวลาที่กำหนดไว้ (ตามที่แสดงในภาพ) ซึ่งออกแบบมาสำหรับการเรียนสด อย่างไรก็ตาม สำหรับการปรับหลักสูตรนี้ เนื้อหาได้ถูกจัดระเบียบใหม่เพื่อให้เหมาะสมกับรูปแบบออนไลน์และอำนวยความสะดวกในการทำความเข้าใจของนักเรียน ลำดับใหม่จะช่วยให้เกิดความก้าวหน้าอย่างมีเหตุผลจากแนวคิดพื้นฐานไปสู่หัวข้อที่มีความซับซ้อนและเทคนิคมากขึ้น ซึ่งจะเพิ่มประสิทธิภาพในการเรียนรู้สูงสุด


การเดินทางครั้งนี้ถูกออกแบบมาเพื่อรองรับผู้เข้าร่วมที่มีระดับความเชี่ยวชาญที่แตกต่างกัน โดยผสมผสานความรู้ทางทฤษฎีและประสบการณ์จริง เมื่อสิ้นสุดบูทแคมป์นี้ คุณจะมีความเข้าใจอย่างถ่องแท้เกี่ยวกับสถาปัตยกรรมของ Liquid การรวมเข้ากับ Bitcoin และวิธีการใช้คุณสมบัติที่เป็นนวัตกรรมของมันเพื่อสร้างและเพิ่มประสิทธิภาพโซลูชันทางการเงิน


ดำดิ่งสู่โลกของ Liquid sidechain และปลดปล่อยศักยภาพของมันอย่างเต็มที่ได้แล้วตอนนี้!

### สถาปัตยกรรมและโมเดลฉันทามติ Liquid Network


Liquid Network เป็นไซด์เชนแบบเฟเดอเรตที่สร้างขึ้นบนฐานรหัสของ Elements ออกแบบมาเพื่อขยายความสามารถของ Bitcoin ในขณะที่ยังคงพึ่งพาความปลอดภัยพื้นฐานของมัน แตกต่างจาก Bitcoin’s Proof-of-Work, Liquid ดำเนินการบนโมเดล Federated Consensus เครือข่ายนี้ได้รับการดูแลโดยกลุ่มสมาชิกที่กระจายอยู่ทั่วโลก รวมถึงการแลกเปลี่ยน โต๊ะซื้อขาย และผู้ให้บริการโครงสร้างพื้นฐาน จากสมาชิกเหล่านี้ จะมีการคัดเลือก "functionaries" จำนวนสิบห้าคนเพื่อทำหน้าที่เป็นผู้ลงนามบล็อก


เจ้าหน้าที่เหล่านี้สร้างบล็อกในรูปแบบการวนรอบแบบกำหนดล่วงหน้า โดยมีการสร้างบล็อกใหม่ทุกนาที การกำหนดเวลาที่แม่นยำนี้แตกต่างจากช่วงเวลาสิบ นาทีแบบสุ่มของ Bitcoin สำหรับบล็อกที่จะถือว่าถูกต้อง จะต้องมีลายเซ็นจากเจ้าหน้าที่อย่างน้อย 11 จาก 15 คน (เกณฑ์สองในสามบวกหนึ่ง) กลไกนี้ให้ Liquid มี "ความแน่นอนสองบล็อก" หมายความว่าเมื่อธุรกรรมได้รับการยืนยันสองครั้ง (ประมาณสองนาที) จะเป็นไปไม่ได้ทางคณิตศาสตร์ที่จะจัดระเบียบโซ่อีกครั้ง ความเร็วและความแน่นอนนี้มีความสำคัญต่อการเก็งกำไร การซื้อขายอัตโนมัติ และการชำระเงินระหว่างการแลกเปลี่ยนอย่างรวดเร็ว


สหพันธ์มีความไดนามิก ผ่านโปรโตคอล Dynamic Federation (Dynafed) เครือข่ายสามารถหมุนเวียนเจ้าหน้าที่หรืออัปเดตพารามิเตอร์โดยไม่ต้องใช้ fork ที่ยากลำบาก ซึ่งช่วยให้ระบบพัฒนาและเปลี่ยนฮาร์ดแวร์หรือสมาชิกได้อย่างราบรื่นในขณะที่ยังคงดำเนินการอย่างต่อเนื่อง


### Confidential Transactions และการจัดการสินทรัพย์


คุณสมบัติที่โดดเด่นของ Liquid คือการรองรับโดยธรรมชาติสำหรับ Confidential Transactions (CT) และสินทรัพย์หลายประเภท บนเครือข่ายหลัก Bitcoin รายละเอียดการทำธุรกรรมทั้งหมด—ผู้ส่ง ผู้รับ และจำนวน—เป็นสาธารณะ ใน Liquid, CT ใช้การผูกมัดทางเข้ารหัสเพื่อซ่อนประเภทสินทรัพย์และจำนวนจากบัญชีแยกประเภทสาธารณะ ในขณะที่ยังคงอนุญาตให้เครือข่ายตรวจสอบว่าการทำธุรกรรมนั้นถูกต้อง (เช่น ไม่มีการเกิดเงินเฟ้อ) เฉพาะผู้เข้าร่วมที่ถือกุญแจบังเท่านั้นที่สามารถดูค่าที่เฉพาะเจาะจงได้ ซึ่งให้ระดับความเป็นส่วนตัวทางการค้าที่จำเป็นสำหรับสถาบันที่เคลื่อนย้ายตำแหน่งขนาดใหญ่


Liquid ปฏิบัติต่อสินทรัพย์ทั้งหมดเป็น "พลเมืองพื้นเมือง" ของบล็อกเชน ซึ่งรวมถึง Liquid Bitcoin (LBTC), เหรียญที่มีเสถียรภาพเช่น USDT และโทเค็นความปลอดภัย การออกสินทรัพย์ไม่จำเป็นต้องใช้สัญญาอัจฉริยะที่ซับซ้อน; มันเป็นฟังก์ชันพื้นฐานของโปรโตคอล


#### สินทรัพย์ที่มีการควบคุมและ AMP

สำหรับสินทรัพย์ที่ต้องการการปฏิบัติตามข้อกำหนด เช่น โทเค็นความปลอดภัย Liquid ใช้แพลตฟอร์มการจัดการสินทรัพย์ Blockstream (AMP) ซึ่งแนะนำชั้นที่มีการอนุญาตที่ธุรกรรมต้องการลายเซ็นที่สองจากเซิร์ฟเวอร์การอนุญาต สิ่งนี้ช่วยให้ผู้ออกสามารถบังคับใช้กฎ—เช่น ข้อกำหนดการอนุญาตล่วงหน้าหรือ KYC—ในระดับที่ละเอียด โดยผสมผสานประสิทธิภาพของบล็อกเชนเข้ากับการควบคุมตามกฎระเบียบของการเงินแบบดั้งเดิม


### ระบบการผูกค่าแบบสองทางและโครงสร้างพื้นฐานด้านความปลอดภัย


การเชื่อมต่อระหว่าง Liquid และ Bitcoin ถูกรักษาผ่านการผูกสองทาง ในการ "peg-in" ผู้ใช้จะส่ง Bitcoin ไปยังที่อยู่ที่สร้างขึ้นบน mainchain เงินทุนเหล่านี้จะถูกล็อคไว้สำหรับการยืนยัน 102 ครั้ง (ประมาณ 17 ชั่วโมง) เพื่อลดความเสี่ยงจากการจัดระเบียบใหม่ เมื่อได้รับการยืนยันแล้ว จำนวน LBTC ที่เทียบเท่าจะถูกสร้างขึ้นบนไซด์เชน Liquid


กระบวนการ "peg-out" ช่วยให้ผู้ใช้สามารถแลก LBTC เป็น Bitcoin ได้ ซึ่งต้องการการเผา LBTC และการอนุญาตทางเข้ารหัสจากสหพันธ์ เพื่อป้องกันการโจรกรรม การทำ peg-out ถูกควบคุมอย่างเข้มงวดโดย Peg-out Authorization Keys (PAKs) ที่ถือโดยสมาชิกของสหพันธ์ นอกจากนี้ ยังสามารถแลกเปลี่ยนเงินได้ทันทีผ่านผู้ให้บริการบุคคลที่สามเช่น SideSwap เพื่อข้ามการล่าช้าในการชำระเงินสำหรับสภาพคล่องที่รวดเร็วขึ้น


#### โมดูลรักษาความปลอดภัยฮาร์ดแวร์ (HSMs)

มีการบังคับใช้ความปลอดภัยอย่างเข้มงวดผ่านฮาร์ดแวร์ เจ้าหน้าที่ไม่เก็บคีย์ส่วนตัวไว้บนเซิร์ฟเวอร์มาตรฐาน แต่จะใช้โมดูลความปลอดภัยฮาร์ดแวร์ (HSMs) แทน อุปกรณ์เหล่านี้ถูกแยกออกจากตรรกะของเซิร์ฟเวอร์โฮสต์และถูกตั้งโปรแกรมด้วยกฎที่เข้มงวด ตัวอย่างเช่น HSM จะปฏิเสธการลงนามบล็อกหากความสูงของมันไม่มากกว่าบล็อกก่อนหน้า เพื่อป้องกันการเขียนประวัติใหม่ โมเดลความปลอดภัยแบบ "ปฏิปักษ์" นี้ถือว่าเซิร์ฟเวอร์โฮสต์อาจถูกโจมตีได้ เพื่อให้มั่นใจว่าคีย์ยังคงปลอดภัยแม้ว่าที่ตั้งทางกายภาพจะถูกบุกรุกก็ตาม

### Elements: รากฐานของ Liquid


Elements เป็นแพลตฟอร์มบล็อกเชนแบบโอเพนซอร์สที่พัฒนามาจากฐานรหัสของ Bitcoin Core มันขยายความสามารถของ Bitcoin โดยการเปิดใช้งานไซด์เชน—บล็อกเชนอิสระที่สามารถโอนสินทรัพย์ไปและกลับจาก Bitcoin ในขณะที่ Bitcoin Core ขับเคลื่อนเครือข่าย Bitcoin, Elements เป็นซอฟต์แวร์เอนจินที่อยู่เบื้องหลัง Liquid Network และไซด์เชนที่กำหนดเองอื่นๆ


ความสัมพันธ์นั้นตรงไปตรงมา: Liquid เป็น "instance" เฉพาะของ sidechain Elements ที่ถูกกำหนดค่าสำหรับการใช้งานในสภาพแวดล้อมการผลิตด้วยโมเดลฉันทามติแบบ federated นักพัฒนาที่คุ้นเคยกับ Bitcoin จะพบว่า Elements นั้นใช้งานง่าย เนื่องจากยังคงรักษาอินเทอร์เฟซ RPC (Remote Procedure Call) และโครงสร้างบรรทัดคำสั่ง (`elements-cli`, `elements-d`, `elements-qt`) ไว้ ความแตกต่างหลักอยู่ที่การกำหนดค่า: การตั้งค่า `chain=liquidv1` จะเชื่อมต่อโหนดกับเครือข่ายหลัก Liquid ในขณะที่ `chain=elementsregtest` จะสร้างสภาพแวดล้อมการทดสอบการถดถอยในเครื่องที่นักพัฒนาสามารถ generate บล็อกได้ทันทีและทดสอบโดยไม่ต้องพึ่งพาภายนอก


#### การออกสินทรัพย์

คุณสมบัติเด่นของ Elements คือการออกสินทรัพย์ในตัวเอง แตกต่างจาก Ethereum ที่โทเค็นเป็นสัญญาอัจฉริยะที่ซับซ้อน สินทรัพย์ใน Elements เป็นพลเมืองชั้นหนึ่งที่สร้างขึ้นผ่านคำสั่ง RPC ที่ง่าย (`issueasset`)


- ตัวระบุเฉพาะ:** แต่ละสินทรัพย์จะได้รับรหัสประจำตัวเลขฐานสิบหกที่ไม่ซ้ำกันจำนวน 64 ตัวอักษร
- โทเค็นการออกใหม่:** ผู้ออกสามารถเลือกสร้างโทเค็นการออกใหม่ได้ ซึ่งให้สิทธิ์แก่ผู้ถือในการสร้างสินทรัพย์เพิ่มเติมในภายหลัง (มีประโยชน์สำหรับ stablecoins หรือโทเค็นความปลอดภัย)
- การลงทะเบียนสินทรัพย์:** เนื่องจากรหัสฐานสิบหกไม่สามารถอ่านได้โดยมนุษย์ การลงทะเบียนสินทรัพย์ของ Blockstream จะทำการจับคู่รหัสเหล่านี้กับชื่อและตัวย่อ (เช่น "USDT") ซึ่งคล้ายกับ DNS สำหรับสินทรัพย์


### ความเป็นส่วนตัวผ่าน Confidential Transactions


Elements แก้ไขข้อจำกัดหลักประการหนึ่งของบล็อกเชนสาธารณะ: การขาดความเป็นส่วนตัวทางการค้า บน Bitcoin จำนวนธุรกรรมทุกอย่างสามารถมองเห็นได้ทั่วโลก Elements แนะนำ **Confidential Transactions (CT)** ซึ่งใช้การเข้ารหัสเพื่อปกปิดจำนวนและประเภทสินทรัพย์ในขณะที่ยังคงให้เครือข่ายสามารถตรวจสอบความถูกต้องของธุรกรรมได้


สิ่งนี้ทำได้โดยใช้ **Pedersen Commitments** และ **Range Proofs**.


- Pedersen Commitments** แทนที่จำนวนที่มองเห็นได้ด้วยการผูกมัดทางการเข้ารหัสลับ เนื่องจากการเข้ารหัสแบบโฮโมมอร์ฟิก ผู้ตรวจสอบสามารถตรวจสอบว่า *Input Commitments = Output Commitments + Fees* โดยไม่ต้องเห็นค่าจริงเลย
- Range Proofs** ป้องกันไม่ให้ผู้ใช้สร้างเงินจากอากาศ (เช่น โดยการใช้ตัวเลขเชิงลบ) โดยการพิสูจน์ทางคณิตศาสตร์ว่าค่าที่ซ่อนอยู่เป็นจำนวนเต็มบวกภายในช่วงที่ถูกต้อง


สำหรับผู้สังเกตการณ์ภายนอก การทำธุรกรรมแบบ Confidential Transaction จะแสดงข้อมูลอินพุตและเอาต์พุตที่ถูกต้อง แต่จะปกปิด *สิ่งที่* ถูกส่งและ *จำนวนเท่าใด* เฉพาะผู้ส่งและผู้รับ (ที่มีคีย์บลายด์) เท่านั้นที่สามารถดูข้อมูลที่ชัดเจนได้


### การพัฒนาและขั้นตอนการทำงานของ RPC


การโต้ตอบกับโหนด Elements ทำได้หลัก ๆ ผ่านทางอินเทอร์เฟซ JSON-RPC ซึ่งช่วยให้งานที่เขียนด้วย Python, JavaScript หรือ Go สามารถสื่อสารกับบล็อกเชนได้



- การตั้งค่า:** นักพัฒนามักจะเริ่มต้นในโหมด `regtest` ซึ่งช่วยให้สามารถสร้างบล็อกได้ทันที (`generateblock`) เพื่อยืนยันธุรกรรมได้ทันที โดยไม่ต้องรอเวลาบล็อก 1 นาทีของเครือข่ายจริง
- คำสั่ง:** คำสั่งมาตรฐาน Bitcoin เช่น `getblockchaininfo` มีให้ใช้งาน พร้อมกับคำสั่งเฉพาะ Elements เช่น `dumpblindingkey` (สำหรับการตรวจสอบ CTs) หรือ `pegin` (สำหรับการย้าย BTC เข้าสู่ sidechain)
- นามแฝง:** เพื่อจัดการหลายโหนด (เช่น "ผู้ส่ง" และ "ผู้รับ" สำหรับการทดสอบ) นักพัฒนามักใช้เชลล์นามแฝงเช่น `e1-cli` และ `e2-cli` ชี้ไปยังไดเรกทอรีข้อมูลที่แตกต่างกัน จำลองเครือข่ายเพียร์ทูเพียร์บนเครื่องเดียว


สถาปัตยกรรมนี้ช่วยให้นักพัฒนาสามารถสร้างแอปพลิเคชันทางการเงินที่ซับซ้อน เช่น แพลตฟอร์มหลักทรัพย์หรือเกตเวย์การชำระเงินส่วนตัว โดยใช้เครื่องมือที่แข็งแกร่งและคุ้นเคยของระบบนิเวศ Bitcoin

### โครงสร้างพื้นฐาน Cross-Layer และ HTLCs


ระบบนิเวศ Bitcoin ได้พัฒนาเป็นสถาปัตยกรรมหลายชั้น โดยมี Mainchain ที่ให้การชำระบัญชี, Lightning ที่ให้ความเร็ว, และ Liquid ที่เปิดใช้งานความสามารถของสินทรัพย์ขั้นสูง การเคลื่อนย้ายมูลค่าระหว่างชั้นเหล่านี้โดยไม่ต้องใช้ตัวกลางที่รวมศูนย์ต้องการหลักการเข้ารหัสที่ไม่ต้องเชื่อถือ: Hash Time Locked Contract (HTLC)


HTLC สร้างช่องทางการชำระเงินแบบมีเงื่อนไขที่เชื่อมโยงระบบอิสระเข้าด้วยกันอย่างเป็นอะตอม มันทำงานผ่านข้อจำกัดหลักสองประการ: **hash lock** และ **time lock**


- Hash Lock:** สามารถใช้จ่ายเงินได้ทันทีหากผู้รับเปิดเผย "preimage" ลับที่ตรงกับแฮชเข้ารหัสเฉพาะ
- The Time Lock:** หากผู้รับไม่สามารถเปิดเผยความลับภายในระยะเวลาที่กำหนด (ความสูงของบล็อก) ผู้ส่งเดิมสามารถเรียกคืนเงินได้


โครงสร้างแบบสองเส้นทางนี้ช่วยให้มั่นใจในความปลอดภัย ในการแลกเปลี่ยนข้ามเลเยอร์ จะใช้การล็อกแฮชเดียวกันในทั้งสองเครือข่าย เมื่อผู้รับเปิดเผยความลับเพื่อเรียกร้องเงินในเลเยอร์หนึ่ง (เช่น Liquid) ความลับนั้นจะปรากฏให้ผู้ส่งเห็น ซึ่งสามารถใช้เพื่อเรียกร้องเงินตอบแทนในอีกเลเยอร์หนึ่ง (เช่น Lightning) การผูกมัดทางเข้ารหัสนี้รับประกันว่าการแลกเปลี่ยนจะสำเร็จสำหรับทั้งสองฝ่ายหรือไม่สำเร็จสำหรับทั้งสองฝ่าย ซึ่งช่วยขจัดความเสี่ยงที่ฝ่ายหนึ่งจะสูญเสียเงินในขณะที่อีกฝ่ายได้รับเงิน


### การอัปเกรด Taproot และ MuSig2


HTLCs แบบเก่า (SegWit v0) ทำงานได้ดีแต่มีข้อเสียในด้านความเป็นส่วนตัวและประสิทธิภาพ พวกมันต้องการการเผยแพร่ตรรกะของสคริปต์ทั้งหมด on-chain ทำให้การทำธุรกรรมสวอปสามารถระบุได้ง่ายโดยนักวิเคราะห์บล็อกเชนและมีค่าใช้จ่ายสูงขึ้นเนื่องจากขนาดข้อมูล การแนะนำของ Taproot (SegWit v1) และโปรโตคอล MuSig2 ได้ปฏิวัติสถาปัตยกรรมนี้


Taproot อนุญาตให้มี **การรวมคีย์** ผ่าน MuSig2 แทนที่จะเปิดเผยสคริปต์ที่ซับซ้อนที่มีคีย์สาธารณะหลายตัว MuSig2 ช่วยให้ผู้เข้าร่วมการแลกเปลี่ยนรวมคีย์ของพวกเขาเป็นคีย์สาธารณะที่รวมเป็นหนึ่งเดียว


- Cooperative "Key Path":** หากทั้งสองฝ่ายตกลงที่จะแลกเปลี่ยน (หรือ "happy path") พวกเขาจะร่วมลงนามในธุรกรรมนี้ สำหรับเครือข่ายแล้ว สิ่งนี้จะดูเหมือนกับการชำระเงินแบบมาตรฐานที่มีลายเซ็นเดียว มันใช้พื้นที่บล็อกน้อยที่สุดและไม่เปิดเผยข้อมูลใด ๆ เกี่ยวกับเงื่อนไขการแลกเปลี่ยนเลย
- เส้นทางสคริปต์ "Adversarial":** หากฝ่ายใดฝ่ายหนึ่งไม่ตอบสนองหรือมีเจตนาร้าย สคริปต์พื้นฐาน (ที่มีการล็อกแฮช/เวลา) จะถูกเปิดเผยในกรณีนั้นเท่านั้น ซึ่งถูกจัดระเบียบในรูปแบบต้นไม้เมอร์เคิล ทำให้ฝ่ายที่ซื่อสัตย์สามารถเปิดเผยเฉพาะสาขาที่จำเป็นในการกู้คืนเงินทุน โดยที่ตรรกะของสัญญาที่เหลือยังคงถูกซ่อนไว้


### การดำเนินการในทางปฏิบัติ: Liquid-Lightning Swaps


ในทางปฏิบัติ โปรโตคอลเหล่านี้ช่วยให้การแลกเปลี่ยนระหว่างเลเยอร์ของ Bitcoin เป็นไปอย่างราบรื่น การแลกเปลี่ยนจาก Liquid ไปยัง Lightning ทั่วไปเริ่มต้นด้วยลูกค้าที่ร้องขอการแลกเปลี่ยนจากผู้ให้บริการ ลูกค้าจะให้ใบแจ้งหนี้ Lightning และผู้ให้บริการจะล็อก Liquid Bitcoin (L-BTC) ที่เทียบเท่ากันไว้ในที่อยู่ HTLC ที่เปิดใช้งาน Taproot


ความเป็นอะตอมถูกบังคับใช้เมื่อการชำระเงินเสร็จสิ้น ในการเรียกร้อง L-BTC ผู้ให้บริการต้องมีภาพล่วงหน้า พวกเขาจะได้รับภาพล่วงหน้านี้ก็ต่อเมื่อพวกเขาชำระใบแจ้งหนี้ Lightning ของลูกค้าได้สำเร็จ เมื่อการชำระเงิน Lightning เสร็จสิ้น ภาพล่วงหน้าจะถูกเปิดเผย ทำให้ผู้ให้บริการสามารถปลดล็อกเงิน Liquid ได้


#### Wallet Abstraction และ UTXO Management

สำหรับผู้ใช้งานทั่วไป ความซับซ้อนนี้ถูกซ่อนอยู่เบื้องหลัง กระเป๋าเงินสมัยใหม่อย่าง Aqua จัดการการสร้างคีย์ การสร้างใบแจ้งหนี้ และกระบวนการลงนามในพื้นหลัง ผู้ใช้เพียงแค่ "ชำระเงิน" ใบแจ้งหนี้ Lightning โดยใช้ยอดคงเหลือ Liquid ของพวกเขา เบื้องหลัง บริการจะจัดการการรวม UTXO โดยกวาดล้างเอาท์พุตขนาดเล็กที่ยังไม่ได้เรียกร้องหรือคืนเงินเป็นระยะๆ เพื่อรักษาประสิทธิภาพของ wallet และลดผลกระทบของค่าธรรมเนียมในช่วงที่มีการใช้งานสูง

### สถาปัตยกรรมและฉันทามติ Liquid Network


Liquid Network ทำงานเป็นเฟเดอเรเต็ดไซด์เชนที่สร้างขึ้นบนฐานรหัส **Elements** ในขณะที่ Elements—fork ของ Bitcoin Core—ให้พื้นฐานซอฟต์แวร์ Liquid เป็นการใช้งานเครือข่ายการผลิต แตกต่างจาก Proof-of-Work ของ Bitcoin ซึ่งพึ่งพา mining ที่มีการแข่งขัน Liquid ใช้โมเดล **Federated Consensus**


เครือข่ายนี้ได้รับการดูแลโดย "เจ้าหน้าที่" สิบห้าคนที่กระจายอยู่ทั่วโลก องค์กรเหล่านี้ดำเนินการเซิร์ฟเวอร์เฉพาะทางที่ทำหน้าที่สำคัญสองประการ:

1.  **การผลิตบล็อก:** เจ้าหน้าที่จะผลัดกันสร้างบล็อกตามตารางรอบที่กำหนดไว้ล่วงหน้า โดยจะสร้างบล็อกใหม่ทุกๆ หนึ่งนาทีอย่างแม่นยำ

2.  **การลงนามบล็อก:** สำหรับบล็อกที่จะถือว่าถูกต้อง จะต้องมีการลงนามโดยเจ้าหน้าที่อย่างน้อย 11 จาก 15 คน


เกณฑ์ 11 จาก 15 นี้ช่วยให้เครือข่ายสามารถทนต่อความล้มเหลวของโหนดได้ถึงสี่โหนดโดยไม่หยุดชะงัก ข้อได้เปรียบหลักของการแลกเปลี่ยนนี้คือ **ความแน่นอนเชิงกำหนด** ในขณะที่ Bitcoin จัดการกับความน่าจะเป็น Liquid บรรลุความแน่นอนในการชำระหลังจากสองบล็อก (ประมาณสองนาที) เมื่อบล็อกมีการยืนยันเพียงครั้งเดียวอยู่ด้านบน โซ่จะไม่สามารถจัดระเบียบใหม่ได้ ทำให้มีประสิทธิภาพสูงสำหรับการเก็งกำไรและการชำระเงินที่รวดเร็ว


### Confidential Transactions และสินทรัพย์ดั้งเดิม


คุณสมบัติที่กำหนดของ Liquid คือการใช้ **Confidential Transactions (CT)** เป็นค่าเริ่มต้น บน Bitcoin mainchain ผู้ส่ง ผู้รับ และจำนวนเงินจะเป็นสาธารณะ Liquid ปรับปรุงสิ่งนี้โดยการทำให้จำนวนเงินและประเภทสินทรัพย์ถูกปิดบังด้วยการเข้ารหัส ในขณะที่ยังคงให้ที่อยู่ของผู้ส่งและผู้รับสามารถตรวจสอบได้


เพื่อให้แน่ใจว่าผู้ใช้ไม่สามารถพิมพ์เงิน (เช่น โดยการส่งจำนวนเงินติดลบ) Liquid ใช้ **Pedersen Commitments** และ **Range Proofs** เครื่องมือเข้ารหัสเหล่านี้ช่วยให้เครือข่ายสามารถตรวจสอบว่า *Inputs = Outputs + Fees* และค่าทั้งหมดเป็นจำนวนเต็มบวก โดยไม่ต้องเปิดเผยตัวเลขเฉพาะต่อบัญชีแยกประเภทสาธารณะ มีเพียงผู้เข้าร่วมที่ถือกุญแจบลายด์ดิ้งเท่านั้นที่สามารถดูข้อมูลที่ถอดรหัสได้


#### การออกสินทรัพย์

Liquid ปฏิบัติต่อสินทรัพย์ทั้งหมดเป็น "native" ไม่ว่าจะเป็น Liquid Bitcoin (L-BTC), stablecoin อย่าง USDT, หรือ security token ทั้งหมดนี้ใช้สถาปัตยกรรมเดียวกัน การออกสินทรัพย์ไม่จำเป็นต้องใช้ smart contracts—เพียงแค่การเรียก RPC อย่างง่าย ๆ เท่านั้น


- สินทรัพย์ที่ไม่ได้รับการควบคุม:** สามารถออกได้โดยไม่มีการขออนุญาตจากใครก็ได้
- สินทรัพย์ที่ถูกควบคุม:** การใช้แพลตฟอร์มการจัดการสินทรัพย์ของ Blockstream (AMP) ผู้ออกสามารถบังคับใช้กฎระเบียบการปฏิบัติตาม (เช่น KYC/AML) โดยการต้องการลายเซ็นที่สองจากเซิร์ฟเวอร์การอนุญาตก่อนที่สินทรัพย์จะสามารถถูกย้ายได้


### การตรึงค่าแบบสองทางและสหพันธ์แบบไดนามิก


สะพานระหว่าง Bitcoin และ Liquid คือ **Two-Way Peg** ซึ่งช่วยให้ผู้ใช้สามารถย้าย BTC ไปยังไซด์เชน (Peg-in) และกลับไปยัง mainchain (Peg-out) ได้


**กระบวนการ Peg-in:**

นี่เป็นแบบไม่ต้องขออนุญาต ผู้ใช้ส่ง BTC ไปยังที่อยู่ที่ควบคุมโดยสหพันธ์ เพื่อป้องกันการจัดระเบียบใหม่ของบล็อกเชน Bitcoin เงินเหล่านี้ต้องรอ **102 การยืนยัน** (ประมาณ 17 ชั่วโมง) ก่อนที่ L-BTC ที่เทียบเท่าจะถูกสร้างบนไซด์เชน


**กระบวนการ Peg-out:**

ในการกลับไปยัง Bitcoin จะมีการเผา L-BTC อย่างไรก็ตาม เพื่อป้องกันการโจรกรรมของทุนสำรอง Bitcoin การถอนเงินจะไม่ถูกทำให้เป็นอัตโนมัติอย่างเต็มที่ พวกเขาต้องการการอนุมัติจากสมาชิกที่ถือ **กุญแจอนุมัติการถอน (PAK)** ทุน Bitcoin เองถูกป้องกันในระบบ multisignature 11-of-15 ของ wallet โดยมีกุญแจที่ถูกเก็บไว้ในโมดูลความปลอดภัยฮาร์ดแวร์ (HSMs) ที่ถูกแยกออกจากเซิร์ฟเวอร์หลักของผู้ปฏิบัติงาน


**Dynamic Federation (Dynafed):**  
สหพันธ์แบบไดนามิก (Dynafed)

เพื่อให้มั่นใจถึงความยั่งยืน Liquid ใช้ Dynafed ซึ่งเป็นโปรโตคอลที่อนุญาตให้เครือข่ายหมุนเวียนผู้ทำหน้าที่หรืออัปเดตพารามิเตอร์โดยไม่ต้องใช้ fork แบบแข็ง หากจำเป็นต้องเปลี่ยนผู้ทำหน้าที่ เครือข่ายจะกระจายธุรกรรมการเปลี่ยนแปลง หลังจากช่วงเวลาซ้อนทับสองสัปดาห์ การกำหนดค่าใหม่จะเข้ามาแทนที่ ทำให้สหพันธ์สามารถพัฒนาได้อย่างราบรื่นในขณะที่ยังคงรักษาการทำงานต่อเนื่อง

### Liquid Network: กลยุทธ์ธุรกิจและระบบนิเวศ


Liquid เป็นมากกว่าซายด์เชนทางเทคนิค; มันเป็นชั้นโครงสร้างพื้นฐานที่เน้นธุรกิจซึ่งออกแบบมาเพื่อจัดการกับความต้องการที่ซับซ้อนของตลาดทุนที่ Bitcoin mainchain ไม่สามารถรองรับได้อย่างมีประสิทธิภาพ ในขณะที่ Lightning Network ปรับให้เหมาะสมสำหรับการชำระเงินที่มีความถี่สูงและมูลค่าต่ำ Liquid มุ่งเน้นไปที่การโอนมูลค่าสูง การออกสินทรัพย์ และการชำระบัญชีระหว่างการแลกเปลี่ยน


ระบบนิเวศนี้ขับเคลื่อนโดย **Liquid Federation** ซึ่งเป็นกลุ่มพันธมิตรของบริษัทประมาณ ~73 แห่ง รวมถึงตลาดหลักทรัพย์ โต๊ะซื้อขาย และผู้ให้บริการโครงสร้างพื้นฐาน โมเดลการทำงานร่วมกันนี้สะท้อนถึงสำนักหักบัญชีทางการเงินแบบดั้งเดิม แต่ดำเนินการด้วยความโปร่งใสมากขึ้นและลดเวลาการชำระบัญชีลงอย่างมาก (2 นาที เทียบกับ T+2 วัน)


### การแปลงสินทรัพย์ในโลกจริงเป็นโทเค็น (RWA)


กรณีธุรกิจหลักสำหรับ Liquid คือ "Tokenization"—การแทนค่ามูลค่าจริง (หุ้น, พันธบัตร, สัญญา mining) เป็นโทเค็นดิจิทัลบนบล็อกเชน ซึ่งมีข้อได้เปรียบหลักสามประการ:

1.  **ตลาดโลก 24/7:** ขจัดข้อจำกัดของเวลาทำการธนาคารและภูมิศาสตร์

2.  **ประสิทธิภาพการดำเนินงาน:** ขจัดข้อผิดพลาดในการกระทบยอดผ่านบัญชีแยกประเภทที่ใช้ร่วมกันและไม่เปลี่ยนแปลง

3.  **การชำระบัญชีแบบอะตอมมิก:** ลดความเสี่ยงของคู่สัญญาโดยการรับรองว่าการส่งมอบเกิดขึ้นพร้อมกับการชำระเงิน


#### สินทรัพย์ที่มีการควบคุมและ AMP

สินทรัพย์ส่วนใหญ่ของสถาบันไม่สามารถซื้อขายได้โดยไม่ต้องขออนุญาตเนื่องจากข้อกำหนดทางกฎหมาย **แพลตฟอร์มการจัดการสินทรัพย์ (AMP)** เป็นมาตรฐานทางเทคนิคที่บังคับใช้กฎเหล่านี้


- การอนุญาตพิเศษ:** ผู้ออกสามารถจำกัดการถือครองและการซื้อขายให้กับที่อยู่ที่ผ่านการตรวจสอบ KYC แล้ว
- Multisig Control:** การดำเนินการตามข้อกำหนด (เช่น การอายัดเงินที่ถูกขโมยหรือการออกโทเค็นที่สูญหายใหม่) จะถูกจัดการผ่านการอนุมัติแบบหลายลายเซ็น เพื่อให้แน่ใจว่าไม่มีพนักงานคนใดสามารถดำเนินการได้โดยลำพัง


โครงสร้างพื้นฐานนี้ใช้งานได้จริงในวันนี้ แพลตฟอร์มอย่าง **Stalker** ให้บริการโทเค็นแบบครบวงจรในยุโรป ในขณะที่ **Sideswap** ทำหน้าที่เป็นการแลกเปลี่ยนแบบกระจายศูนย์และ wallet ที่ไม่รับฝากทรัพย์สิน ช่วยให้การซื้อขายสินทรัพย์แบบเพียร์ทูเพียร์ เช่น **Blockstream Mining Note (BMN)** และหุ้น MicroStrategy ที่ถูกโทเค็น (CMSTR)


### ความสำเร็จในโลกแห่งความจริง: กรณีศึกษา Mayfell


หลักฐานที่น่าสนใจที่สุดเกี่ยวกับประโยชน์ของ Liquid มาจาก **Mayfell** ในเม็กซิโก ในตลาดที่การเงินแบบดั้งเดิมพึ่งพาโน้ตสัญญากระดาษ—ซึ่งมีแนวโน้มที่จะสูญหาย ถูกฉ้อโกง และการประมวลผลที่ช้า—Mayfell ได้ย้ายโครงสร้างพื้นฐานทั้งหมดไปยัง Liquid



- ขนาด:** ออกตั๋วสัญญาใช้เงินกว่า 25,000 ฉบับ คิดเป็นมูลค่า **$1.5 พันล้าน+**
- ความเป็นส่วนตัว:** โดยใช้ Liquid ของ Confidential Transactions จำนวนเงินกู้จะมองเห็นได้เฉพาะผู้ให้กู้และผู้กู้เท่านั้น ซึ่งรักษาความเป็นส่วนตัวทางการค้าในขณะที่อนุญาตให้ผู้ตรวจสอบสามารถตรวจสอบความถูกต้องของบัญชีแยกประเภทได้
- ผลกระทบ:** โดยการเชื่อมต่อผู้ให้กู้ที่ไม่ใช่ธนาคารกับตลาดทุนทั่วโลกผ่านบล็อกเชน Mayfell ลดต้นทุนและขยายการเข้าถึงเครดิตสำหรับ SMEs ในเม็กซิโก แสดงให้เห็นว่าข้อเสนอคุณค่าของ Liquid ขยายออกไปไกลกว่าการซื้อขายเพื่อเก็งกำไร


### การวางตำแหน่งเชิงกลยุทธ์เทียบกับเครือข่ายอื่น ๆ


Liquid แข่งขันในตลาดที่มีแพลตฟอร์มการโทเค็นมากมาย (Ethereum, Solana, ฯลฯ) แต่มีข้อได้เปรียบทางกลยุทธ์ที่แตกต่าง:


- ความชัดเจนด้านกฎระเบียบ:** Liquid ใช้ Bitcoin (L-BTC) เป็นสินทรัพย์พื้นเมืองของมันเอง ไม่มีการขุดล่วงหน้า token หรือ ICO ซึ่งช่วยหลีกเลี่ยงความเสี่ยง "หลักทรัพย์ที่ไม่ได้จดทะเบียน" ที่เป็นปัญหาของบล็อกเชน L1 อื่น ๆ
- ความเสถียร:** ต่างจากโมเดลบัญชีของ Ethereum ที่การทำธุรกรรมที่ล้มเหลวยังคงเผาผลาญค่าธรรมเนียมแก๊ส โมเดล UTXO ของ Liquid นั้นเป็นแบบกำหนดล่วงหน้าและเชื่อถือได้สำหรับข้อมูลทางการเงินที่สำคัญต่อภารกิจ
- ความเป็นส่วนตัว:** การรักษาความลับเป็นค่าเริ่มต้นเป็นข้อกำหนดสำหรับสถาบันการเงินส่วนใหญ่ ซึ่งเป็นคุณสมบัติที่ Liquid มีให้โดยธรรมชาติ ในขณะที่เครือข่ายสาธารณะต้องดิ้นรนในการนำไปใช้โดยไม่ต้องมีส่วนเสริมที่ซับซ้อน


สำหรับนักพัฒนา ระบบนิเวศนี้มอบโอกาสที่ชัดเจน: การสร้างเครื่องมือ (แดชบอร์ด, กระเป๋าเงิน, การผสานรวมการปฏิบัติตามข้อกำหนด) ที่เชื่อมโยงการเงินแบบดั้งเดิมกับชั้นการชำระเงินที่ปลอดภัยของ Bitcoin

### Blockstream AMP: การควบคุมสินทรัพย์ที่ได้รับอนุญาตบน Liquid


Blockstream AMP (Asset Management Platform) ทำหน้าที่เป็นชั้นโครงสร้างพื้นฐานที่สำคัญบน Liquid Network ซึ่งออกแบบมาโดยเฉพาะสำหรับผู้ออกหลักทรัพย์ดิจิทัลที่มีการควบคุมและ stablecoins ในขณะที่ Liquid ให้บริการชั้นฐานที่ไม่มีการอนุญาตพร้อมการออกสินทรัพย์เนทีฟ หน่วยงานที่มีการควบคุมมักต้องการการกำกับดูแลและความสามารถในการปฏิบัติตามข้อกำหนดอย่างเข้มงวด AMP เชื่อมช่องว่างนี้โดยการแนะนำชั้นควบคุมที่มีการอนุญาตเหนือสินทรัพย์เฉพาะโดยไม่ลดทอนประโยชน์ด้านความเป็นส่วนตัวของ Liquid’s Confidential Transactions


ข้อเสนอคุณค่าหลักของแพลตฟอร์มตั้งอยู่บนความสามารถหลักสองประการ: การมองเห็นผู้ออกที่ครอบคลุมและการอนุมัติธุรกรรม แตกต่างจากสินทรัพย์ Liquid มาตรฐานที่จำนวนและประเภทเป็น blinded สำหรับทุกคนยกเว้นผู้เข้าร่วม สินทรัพย์ AMP อนุญาตให้ผู้ออกสามารถรักษาร่องรอยการตรวจสอบ unblinded ได้อย่างสมบูรณ์ ความโปร่งใสนี้มีความสำคัญสำหรับการรายงานตามกฎระเบียบและการตรวจสอบภายใน นอกจากนี้ AMP ยังบังคับใช้โมเดลการอนุมัติที่เข้มงวดผ่านกลไกการร่วมลงนาม การโอนสินทรัพย์ AMP ทุกครั้งต้องการลายเซ็นจากเซิร์ฟเวอร์ AMP สิ่งนี้ช่วยให้ผู้ออกสามารถบังคับใช้กฎที่ซับซ้อน off-chain—เช่น การแช่แข็งบัญชี การอนุญาตนักลงทุนที่ได้รับการรับรอง หรือการกำหนดขีดจำกัดการโอน—ทำหน้าที่เป็นผู้รักษาประตูส่วนกลางภายในเครือข่ายแบบกระจายอำนาจได้อย่างมีประสิทธิภาพ


#### การแลกเปลี่ยนในการดำเนินงาน

สถาปัตยกรรมนี้แนะนำการแลกเปลี่ยนเฉพาะเจาะจง ระบบนี้พึ่งพาความพร้อมใช้งานของเซิร์ฟเวอร์ AMP; หากเซิร์ฟเวอร์ทำหน้าที่เป็นผู้ลงนามร่วมและออฟไลน์ สภาพคล่องของสินทรัพย์จะหยุดชั่วคราว นอกจากนี้ ในขณะที่ความเป็นส่วนตัวระหว่างผู้ใช้กับผู้ใช้ยังคงอยู่ นักลงทุนต้องยอมรับว่าผู้ออกมีความสามารถในการมองเห็นการถือครองของพวกเขาอย่างเต็มที่ โมเดลนี้เหมาะสำหรับโทเค็นความปลอดภัยที่เป็นไปตามข้อกำหนด แต่ไม่เหมาะสำหรับสกุลเงินดิจิทัลที่ต้านทานการเซ็นเซอร์


### วิวัฒนาการสถาปัตยกรรมและเส้นทางการบูรณาการ


ระบบนิเวศ AMP กำลังเปลี่ยนจากการทำซ้ำครั้งแรกไปสู่สถาปัตยกรรม "เวอร์ชัน 2" ที่ยืดหยุ่นและทำงานร่วมกันได้มากขึ้น ระบบเดิมต้องการให้ผู้ออกบัตรรักษาโหนด Elements ที่ซิงโครไนซ์อย่างเต็มที่และบังคับให้นักพัฒนาต้องพึ่งพา SDK Green ขนาดใหญ่ แม้ว่าจะใช้งานได้ แต่สิ่งนี้สร้างอุปสรรคทางเทคนิคสูงในการเข้าใช้งานและจำกัดตัวเลือก wallet


สถาปัตยกรรมรุ่นถัดไปทำให้ข้อกำหนดเหล่านี้ง่ายขึ้นโดยพื้นฐานโดยการย้ายความซับซ้อนไปยังเซิร์ฟเวอร์ ในโมเดลใหม่นี้ เซิร์ฟเวอร์ AMP จะจัดการงานหนักของการสร้างธุรกรรม การเลือก UTXO และการคำนวณค่าธรรมเนียม มันนำเสนอผู้ออกด้วยธุรกรรม Elements ที่ลงนามบางส่วน (PSET) ซึ่งเพียงแค่ต้องการลายเซ็น แนวทาง "เซิร์ฟเวอร์อัจฉริยะ, wallet โง่" นี้กำจัดความจำเป็นที่ผู้ออกจะต้องเรียกใช้โหนดเต็มรูปแบบและช่วยให้สามารถใช้กระเป๋าเงินฮาร์ดแวร์และการตั้งค่าลายเซ็นหลายมาตรฐานสำหรับการจัดการคลังได้


สำหรับนักพัฒนา การพัฒนานี้หมายถึงการเปลี่ยนจาก SDK ที่เป็นกรรมสิทธิ์ไปสู่ตัวบรรยายมาตรฐานและเวิร์กโฟลว์ PSET กระเป๋าเงินสามารถลงทะเบียนตัวบรรยาย multisignature กับเซิร์ฟเวอร์ AMP เพื่อสร้างสิทธิ์การอนุญาต ซึ่งสอดคล้องกับการพัฒนา AMP ตามมาตรฐาน Bitcoin wallet ที่กว้างขึ้น ทำให้การผสานรวมสามารถเข้าถึงได้สำหรับแอปพลิเคชันใด ๆ ที่สามารถจัดการรูปแบบ PSBT/PSET นักพัฒนาที่สร้างในวันนี้ได้รับการสนับสนุนให้ใช้เครื่องมือเช่น Liquid Wallet Kit (LWK) ซึ่งรองรับสถาปัตยกรรมที่ทันสมัยและอิงตามตัวบรรยาย เพื่อให้แน่ใจว่าแอปพลิเคชันของพวกเขาพร้อมสำหรับมาตรฐาน AMP ใหม่ในอนาคต


### ระบบนิเวศ Liquid Wallet และการรักษาความลับ


การสร้างแอปพลิเคชันบน Liquid ต้องการการนำทางผ่านสแต็กที่ซับซ้อนกว่าการพัฒนา Bitcoin มาตรฐาน เนื่องจากคุณสมบัติต่างๆ เช่น native assets และ Confidential Transactions ระบบนิเวศได้รับการสนับสนุนโดยสถาปัตยกรรมแบบเลเยอร์: ไลบรารีระดับต่ำเช่น `secp256k1-zkp` จัดการกับปริมิทีฟการเข้ารหัส ในขณะที่ชุดเครื่องมือระดับสูงจัดการกับตรรกะของ wallet


ในทางประวัติศาสตร์ Green Development Kit (GDK) มอบโซลูชันที่ครอบคลุมแต่เข้มงวด ทางเลือกสมัยใหม่คือ Liquid Wallet Kit (LWK) ซึ่งเสนอแนวทางแบบโมดูลาร์ LWK แยกข้อกังวลออกเป็นกล่องที่แยกจากกัน—จัดการกับตัวบรรยาย การลงนาม และการสื่อสารกับฮาร์ดแวร์อย่างอิสระ—ให้ความยืดหยุ่นแก่ผู้พัฒนาในการสร้างโซลูชันที่กำหนดเองโดยไม่ต้องมีภาระของไลบรารีแบบโมโนลิทิก


#### การจัดการ Confidential Transactions

ความท้าทายที่โดดเด่นที่สุดในระบบนิเวศนี้คือการจัดการวงจรชีวิตของการปิดบัง ใน Liquid ผลลัพธ์ของธุรกรรมจะถูกเข้ารหัสโดยใช้การแลกเปลี่ยนกุญแจ Elliptic Curve Diffie-Hellman (ECDH) ผู้ส่งใช้กุญแจสาธารณะของผู้รับในการปิดบังเพื่อเข้ารหัสจำนวนและประเภทของสินทรัพย์ สร้างหลักฐานช่วงที่ยืนยันความถูกต้องของธุรกรรมโดยไม่เปิดเผยค่า


สำหรับ wallet ที่จะทำงานได้ จะต้องย้อนกลับกระบวนการนี้ให้สำเร็จ เมื่อ wallet ตรวจพบธุรกรรมที่เข้ามา มันจะพยายามเปิดเผยผลลัพธ์โดยใช้กุญแจบอดส่วนตัวของมัน หากความลับที่ใช้ร่วมกันตรงกัน wallet สามารถถอดรหัสค่าและเพิ่มไปยังยอดคงเหลือของผู้ใช้ได้ กระบวนการนี้ต้องใช้การคำนวณที่เข้มข้นและต้องการการจัดการปัจจัยบอดที่แม่นยำเพื่อให้แน่ใจว่าธุรกรรมมีความสมดุลทางคณิตศาสตร์ ซึ่งเป็นความซับซ้อนที่เครื่องมืออย่าง LWK มุ่งหมายที่จะทำให้เป็นนามธรรมสำหรับนักพัฒนา

### บทนำสู่ Breez Liquid SDK


Breez Liquid SDK เป็นชุดเครื่องมือโอเพนซอร์สที่ผู้ใช้ดูแลเอง ออกแบบมาเพื่อเชื่อมช่องว่างระหว่าง Liquid Network และระบบนิเวศ Bitcoin ที่กว้างขึ้น ภารกิจหลักของมันคือการทำให้ความซับซ้อนของการจัดการโหนด Lightning Network และการแลกเปลี่ยนอะตอมเป็นเรื่องง่ายขึ้น ช่วยให้นักพัฒนาสามารถสร้างแอปพลิเคชันทางการเงินที่ไร้รอยต่อได้


สร้างขึ้นด้วยปรัชญา "mobile-first" โดยมีตรรกะหลักเขียนใน Rust เพื่อประสิทธิภาพและความปลอดภัย แต่ใช้ UniFFI (Unified Foreign Function Interface) เพื่อให้การเชื่อมต่อแบบเนทีฟสำหรับ Kotlin, Swift, Python, C#, Dart และ Flutter สิ่งนี้ทำให้นักพัฒนาสามารถรวมฟังก์ชันการทำงานของ Bitcoin เข้ากับสภาพแวดล้อมที่พวกเขาชื่นชอบได้โดยไม่ต้องจัดการกับการดำเนินการเข้ารหัสระดับต่ำ


**ประเภทธุรกรรมที่รองรับ:**

SDK ทำงานร่วมกับ Liquid เป็น "ฐานหลัก" ของมัน โดดเด่นในสามการดำเนินการเฉพาะ:

1.  **Liquid-to-Liquid:** โอน on-chain โดยตรง

2.  **Liquid-to-Lightning:** การชำระใบแจ้งหนี้ Lightning โดยใช้เงินทุน Liquid.

3.  **Liquid-to-Bitcoin:** การสับเปลี่ยนกองทุนโดยตรงไปยัง Bitcoin mainchain.


*หมายเหตุ: SDK ไม่รองรับการทำธุรกรรมแบบ Lightning-to-Lightning หรือ Bitcoin-to-Bitcoin โดยตรง เป็นเครื่องมือที่เน้น Liquid โดยเฉพาะ*


### สถาปัตยกรรมการชำระเงิน: การแลกเปลี่ยนใต้น้ำ


เพื่อให้เกิดการทำงานร่วมกันระหว่าง Liquid, Lightning และ Bitcoin โดยไม่ต้องมีผู้ดูแลส่วนกลาง Breez อาศัย **Submarine Swaps** ซึ่งเป็นการดำเนินการแบบอะตอมมิกที่การทำธุรกรรมจะสำเร็จบนทั้งสองเครือข่ายหรือไม่สำเร็จบนทั้งสองเครือข่าย เพื่อให้มั่นใจว่าเงินจะไม่สูญหายระหว่างการโอนย้าย


#### Lightning Send (Submarine Swap)  
ส่งสายฟ้า (การแลกเปลี่ยนใต้น้ำ)

เมื่อผู้ใช้ชำระเงินตามใบแจ้งหนี้ Lightning, SDK จะกระจายธุรกรรม "lock-up" บน Liquid Network ซึ่งจะทำให้เงินถูกเก็บไว้ในเอสโครว์ ผู้ให้บริการสวอป (Boltz) ตรวจพบสิ่งนี้ ชำระเงินตามใบแจ้งหนี้ Lightning และจากนั้นใช้ preimage การชำระเงิน (ใบเสร็จรับเงินเข้ารหัส) เพื่อเรียกร้องเงิน Liquid ที่ถูกล็อกไว้


#### Lightning Receive (Reverse Submarine Swap)  
รับสายฟ้า (การแลกเปลี่ยนเรือดำน้ำย้อนกลับ)

การรับ Lightning เป็นกระบวนการผกผัน ผู้ใช้สร้างใบแจ้งหนี้ และผู้ให้บริการสวอปล็อกเงินทุนบน Lightning Network SDK จะตรวจสอบกระบวนการนี้ผ่าน WebSockets เมื่อการล็อกได้รับการยืนยันแล้ว SDK จะดำเนินการทำธุรกรรมการเรียกร้องโดยอัตโนมัติเพื่อย้ายเงินทุนที่เทียบเท่าไปยัง Liquid wallet ของผู้ใช้


#### ครอสเชน Bitcoin

สำหรับการโอนจาก Liquid ไปยัง Bitcoin สถาปัตยกรรมใช้กลไก "dual-swap" การทำธุรกรรมล็อคอัพเกิดขึ้นบนทั้งสองเชนพร้อมกัน ผู้ส่งจะเรียกร้องเงินบน Bitcoin ในขณะที่ผู้รับจะเรียกร้องเงินบน Liquid สิ่งนี้ช่วยให้การเชื่อมต่อแบบไม่ต้องพึ่งพาความไว้วางใจโดยไม่ต้องพึ่งพา federated peg-outs หรือการแลกเปลี่ยนแบบรวมศูนย์


### นักพัฒนา Interface และการจัดการอัตโนมัติ


SDK Breez ช่วยให้ประสบการณ์ของนักพัฒนาง่ายขึ้นโดยการย่อกระบวนการทางการเงินที่ซับซ้อนให้เป็นกระบวนการมาตรฐานสามขั้นตอน: **เชื่อมต่อ, เตรียมการ, และดำเนินการ**.


1.  **Connect:** เริ่มต้นใช้งาน wallet, ซิงค์กับบล็อกเชนโดยใช้ชุด Liquid Wallet Kit (LWK), และสร้างการเชื่อมต่อ WebSocket สำหรับการตรวจสอบแบบเรียลไทม์

2. **Prepare:** ก่อนที่จะทำการจัดสรรเงิน ขั้นตอนนี้จะคำนวณและแสดงค่าธรรมเนียมเครือข่ายและค่าใช้จ่ายในการแลกเปลี่ยนทั้งหมดที่เกี่ยวข้อง เพื่อให้ UI สามารถแสดงยอดรวมที่ชัดเจนให้กับผู้ใช้ได้

3.  **ดำเนินการ:** ขั้นตอนนี้สร้างธุรกรรม, กระจายมัน, และเริ่มการสวอป.


#### กลไกความปลอดภัยอัตโนมัติ

หนึ่งในฟีเจอร์ที่สำคัญที่สุดของ SDK คือ **การจัดการการคืนเงินอัตโนมัติ** ในกรณีที่เกิดความล้มเหลวของเครือข่ายหรือคู่สัญญาที่ไม่ให้ความร่วมมือ เงินทุนอาจติดอยู่ในสัญญาที่ล็อคด้วยเวลาได้ในทางทฤษฎี SDK จะทำการสรุปตรรกะการกู้คืนทั้งหมด มันจะตรวจสอบสถานะของการแลกเปลี่ยนทุกครั้ง; หากการแลกเปลี่ยนล้มเหลวหรือล้มเหลวตามเวลา SDK จะสร้างและกระจายธุรกรรมการคืนเงินโดยอัตโนมัติเพื่อคืนเงินให้กับ wallet ของผู้ใช้


นอกจากนี้ SDK ยังจัดการกับ **การแก้ไขเมตาดาต้า** โดยจะรวมข้อมูลการสลับ off-chain (ที่จัดหาโดย Boltz swapper) เข้ากับประวัติการทำธุรกรรม on-chain ซึ่งจะช่วยให้ประวัติการทำธุรกรรมของผู้ใช้สามารถอ่านได้ง่ายขึ้น โดยแสดงรายละเอียดใบแจ้งหนี้และบริบทการชำระเงินแทนที่จะเป็นเพียงแฮชธุรกรรมเลขฐานสิบหกดิบๆ
