### 什麼是 Lightning 節點？



讓我們回到基本：在定義節點之前，我們需要瞭解 Lightning Network 是什麼。它是建立在 Bitcoin 之上的頂層協定，旨在實現 BTC 的[離鏈](https://planb.academy/resources/glossary/offchain)交易，交易速度快（近乎瞬間完成）且成本低廉。「離鏈 」是指在 Lightning 上進行的交易不會出現在主 Bitcoin [區塊鏈](https://planb.academy/resources/glossary/blockchain)上。Lightning 也是對 Bitcoin 使用量不斷增加以及[onchain](https://planb.academy/resources/glossary/onchain) 擁擠問題的部分回應，onchain 擁擠問題引起了人們對系統[可擴展性](https://planb.academy/resources/glossary/scalability)的擔憂。



為了運作，「閃電」依賴於參與者之間開放[支付通道](https://planb.academy/resources/glossary/payment-channel)，在這些通道中，交易幾乎可以立即進行，通常只需支付很少的費用，而無需在 Bitcoin 區塊鏈上逐一註冊。這些通道可以保持開放很長時間，只有在打開和關閉時才需要進行鏈上交易。



[Lightning 節點](https://planb.academy/resources/glossary/lightning-node)是 Lightning 網絡的參與者，與其他節點開通渠道並進行支付。具體來說，Lightning 節點是在電腦上運行並實施 Lightning Network 協定的軟件。例如 LND、Core Lightning 或 Eclair。該軟件的主要作用是：




- 連接至 [Bitcoin 節點](https://planb.academy/resources/glossary/full-node)，從主區塊鏈中獲取資訊；
- 與其他節點建立和管理雙向付款通道；
- 與整個 Lightning 網路交換訊息。



![Image](assets/fr/001.webp)



### 節點 vs. 閃電 Wallet：重要的區別



在 Bitcoin（onchain）上，「*[wallet](https://planb.academy/resources/glossary/wallet)*」是指管理您的[私人金鑰](https://planb.academy/resources/glossary/private-key)、根據您的 [UTXO](https://planb.academy/resources/glossary/utxo) 計算您的餘額以及建立您的交易的軟體。此 wallet 可能基於您自己的 Bitcoin 節點，也可能基於其他人的節點，但今天，節點的角色和鏈上 wallet 的角色明顯不同。



在 Lightning 上，要重用這種詞彙而不造成混亂是比較困難的。術語「*Lightning wallet*」相當含糊，因為實際上並不存在真正自我保管的 Lightning wallet，除非它是基於一個節點。因此，只有兩種情況是可能的：



- 要擁有一個真正的 Lightning 節點（即非監管）：您正在使用的軟件（例如像 Phoenix 這樣的移動應用程序或 Umbrel 上的 LND 實例）實際上正在運行一個節點，而您實際上持有鑰匙來取回您的比特幣。在這種情況下，您的 "*Lightning wallet*"實際上只是 Lightning 節點上面的一個用戶界面，無論是嵌入式的還是遠程的。



- 使用託管服務：您使用的應用程式在 Lightning 上顯示 [sats](https://planb.academy/resources/glossary/satoshi-sat) 的餘額，但在後台，資金在提供者的節點上（例如 Wallet of Satoshi）。您既沒有鑰匙，也沒有通道的控制權。您的餘額只是公司資料庫中的一個會計項目。這與將您的 bitcoins 放在交換平台上有相似之處，並帶有所有相關的風險。在這種情況下，您的 "*Lightning wallet*"只是一個由運營商管理的帳戶，而運營商則運行一個真正的 Lightning 節點。



因此，Lightning 沒有兩者之間的選擇：要麼你有一個節點 (即使是嵌入式節點)，所以你是自我保管；要麼你沒有，所以公司擁有你的 sats。但正如我們在接下來的章節中所看到的，這兩種用途有時候很難區分。例如，Phoenix 是一個嵌入了真正 Lightning 節點的移動應用程式，但用戶不一定知道它，因為其運作的全部複雜性幾乎完全被隱藏了。



### 提醒您 Lightning Network 的運作方式



在本節中，我會快速提醒您 Lightning 是如何運作的。再次重申，如果您想要更深入的理論概念介紹，我邀請您看看專門的 LNP 201 課程。



#### 付款管道：開啟、更新及關閉



Lightning 網絡的核心是基於雙向支付通道。通道可以打開（即創建），隨著閃電交易的發生而更新，最後關閉。從 onchain 的角度來看，通道不過是一個 2/2 [多重簽章](https://planb.academy/resources/glossary/multisig)[輸出](https://planb.academy/resources/glossary/output)。



![Image](assets/fr/002.webp)



從 Lightning 的角度來看，這是一個由兩個參與者瓜分[流動資金](https://planb.academy/resources/glossary/liquidity-lightning)的付款管道。



![Image](assets/fr/003.webp)





- 開啟頻道：**



兩個節點決定打開一個通道。其中一個在稱為 *funding transaction* 的 onchain 交易中投入 bitcoins。此交易根據 2 對 2 多重[簽名](https://planb.academy/resources/glossary/digital-signature)[腳本](https://planb.academy/resources/glossary/script)建立輸出，這表示在 Bitcoin 上花費這些資金需要通道中兩個節點的簽名。在發出此交易之前，提供資金的一方會要求另一方簽署*提款交易*，此交易並非在鏈上發出，但可讓其在發生問題時收回資金。



![Image](assets/fr/004.webp)





- Commitment 交易：**



通道的狀態（即 A 和 B 之間 sats 的分配）由 *[commitment transaction](https://planb.academy/resources/glossary/commitment-transaction)* 表示，兩個節點都知道，但不會立即在區塊鏈上廣播。此交易描述了如何根據 Lightning 上的支付在區塊鏈上重新分配通道資金。



每次 Lightning 付款時，兩個節點會簽署一個新的狀態，取代之前的狀態。舊的狀態會因為一個撤銷金鑰機制而被撤銷：如果其中一個參與者嘗試廣播舊的狀態，另一個參與者可以收回全額的資金作為懲罰。



這裏的重要概念是，節點總是有一個簽署的 Bitcoin 交易，不會在鏈上廣播，由節點來保存，並根據在 Lightning Network 上的支付來重新分配彼此的 sats。



![Image](assets/fr/005.webp)





- 通道關閉：**



當雙方就通道的最終狀態達成共識時，通道可以通過合作關閉的方式乾淨地關閉；如果其中一個參與者停止合作或無法聯繫，通道也可以單方面關閉（強制關閉）。在所有情況下，關閉的形式都是使用 2/2 輸出的 onchain 交易，並根據通道的最後有效狀態在參與者之間分配資金。



![Image](assets/fr/006.webp)



因此，Lightning 可作為錨定在 Bitcoin 上的次要層：只有特定事件（通道的開啟和關閉）才會出現在主區塊鏈上。中間的付款則保持在離鏈狀態。



在進一步了解之前，以下是了解如何管理 Lightning 頻道的兩個基本概念：




- Liquidity*：通道一側可用的 sats 數量；
- [容量](https://planb.academy/resources/glossary/lightning-channel-capacity)*：它是鎖定在 2/2 multisig 輸出中的總量，也就是通道兩端的流動性總和。



#### 渠道和流動性網路



通道不僅僅是用於兩個節點之間的支付：它是全球互聯通道網絡的一部分。您的節點可以通過自己的通道為其他用戶付款，您也可以發送 sats 到與您沒有直接通道的 Lightning 節點，只要在您的兩個節點之間能找到有效的路徑。



每個節點都透過[八卦](https://planb.academy/resources/glossary/gossip)通訊協定知道這個網路的地圖：哪些頻道存在、哪些節點有雙向頻道連接、哪些容量已公佈。若要將款項傳送給沒有直接通道的收款人，您的節點會計算一條包含數跳的路由：您的節點 → 節點 X → 節點 Y → 收款人節點。在每個跳躍點，付款都會經過一個在付款方向必須有足夠流動性的通道。



![Image](assets/fr/007.webp)



因此，通道的流動性不是對稱的：一邊可能負載很重，而另一邊幾乎是空的。管理這種流動性，也就是知道 sats 在哪裡，可以往哪個方向流動，是操作 Lightning 節點最重要的一環。在接下來的實用章節中，我們將詳細介紹這一點。



#### HTLC：運輸付款而不被搶劫



為了讓支付能夠在不需要信任的情況下通過中間節點，Lightning 使用稱為 *[HTLC](https://planb.academy/resources/glossary/htlc)*（*Hashed Time-Locked Contracts*）的[智能合約](https://planb.academy/resources/glossary/smart-contract)。簡單來說，HTLC 使資金轉移以揭露秘密為條件，並加入時間限制，以在交易失敗時保護寄件者。因此，每筆付款都必須出示預先影像 (其[雜湊值](https://planb.academy/resources/glossary/hash-function)對應於協定值的秘密)。如果最終收款人提供了這個預先影像，他或她就可以領回資金，這反过来又可以讓每個中介節點收回自己的資金。



![Image](assets/fr/008.webp)



關於 HTLC 工作原理的技術細節我就不多說了，因為這些細節對本課程的目的而言並非必要。您可以在 LNP 201 理論課程中找到深入的解釋。只需記住 HTLCs 可實現原子交換：轉移完成且路由中沒有人受到傷害，或者轉移失敗且每個參與者都能收回其初始資金。沒有兩者之間的選擇。



### 主要的 Lightning 節點實作



與 Bitcoin 一樣，Lightning 協定也有多種實作。許多獨立的團隊正在開發自己的版本，由於它們遵循相同的規格（[BOLT](https://planb.academy/resources/glossary/bolt)），因此所有版本都具有互操作性。以下是目前使用的主要實作。



#### LND (*Lightning Network Daemon*)



LND 是 Lightning 通訊協定的完整實作，以 Go 寫成，由 Lightning Labs 開發。



![Image](assets/fr/009.webp)



#### 核心閃電 (*CLN*)



Core Lightning（原名 *C-Lightning*）是 Blockstream 開發的實作。它以 C 寫成，部分元件以 Rust 寫成。



![Image](assets/fr/010.webp)



#### Eclair



Eclair 是用 Scala 寫成的實作，由法國公司 ACINQ 開發。ACINQ 使用 Eclair 運作 Lightning 網路中最重要的路由節點之一，並將此實作作為其自有產品 (例如 Phoenix 應用程式) 的軟體基礎。



![Image](assets/fr/011.webp)



#### LDK (*Lightning Development Kit*)



LDK (*Lightning Development Kit*) 是由 Spiral (Block, ex-Square) 所維護的 Rust 開發套件。它並不是像 LND 或 CLN 一樣即時可用的 daemon，而是為希望將 Lightning 直接整合到應用程式中的開發人員提供的函式庫。



![Image](assets/fr/012.webp)



在本 LNP202 課程中，我們將專注於 LND：最常用於私人客戶交鑰匙解決方案的實作，例如 Umbrel。

## 為什麼要運行自己的 Lightning 節點？


<chapterId>421db24e-511c-41ed-ad68-69b0662042ea</chapterId>



要回答這個問題其實很簡單，因為這是個反問句：如果沒有自己的節點，您就不再是真正在使用 Lightning，而只是在公司基礎架構中使用 Lightning 的假象。



使用 Lightning 託管 wallet 意味著比特幣在技術上屬於操作節點的公司。您不持有私鑰，也不控制通道。您的 wallet 結餘只是服務供應商資料庫中的一行。這種情況對於初學者來說當然非常方便，而且用戶體驗往往是流暢的，但根本的問題是：如果您最終放棄了 Bitcoin 和 Lightning 與傳統貨幣和銀行不同的地方，那麼使用它們又有什麼好處呢？



Bitcoin 的兩個主要價值主張是貨幣主權（不再依賴中央機構發行和持有）和抗審查（第三方無法阻止或過濾付款）。Lightning 上的託管系統正面違反了這兩個目標：您無法檢查平台內部的貨幣供應，而且根據定義，持有所有資金和所有管道的營運者可以審查、延遲、優先或攔截您的付款。在這種情況下，我們可以合理地問自己，**如果透過 Lightning 使用比特幣會重現傳統國家貨幣系統的信任與依賴模式，那它還有什麼意義**。



> 我們需要的是一個以加密證明而非信任為基礎的電子支付系統，讓任何願意交易的雙方都能直接進行交易，而不需要信任的第三方。
*Satoshi Nakamoto, Bitcoin 白皮書


撇開哲學不談，對您來說更具體的缺點如下。首先，您無法驗證公司是否真的持有與顯示餘額相對應的 bitcoins。它可能以部分儲備運作、被黑客入侵、破產或乾脆消失。在這種情況下，您只是另一個債權人，無法真正保證您能拿回您的錢。



其次，公司會面臨監管風險：禁制令、資金凍結、要求封鎖使用者或交易、加強監視，甚至在某些司法管轄區直接禁止活動。服務供應商所受到的每項限制都會影響到您。



在機密性方面，情況也好不到哪裡去。保管操作員可以看到您所有的資金流：金額、頻率、收款人、餘額、消費習慣。結合 Bitcoin 應用程式所提供的資訊以及可能的基礎連鎖分析，這些資訊可以提供您財務活動的精確輪廓。再次重申，這與 Bitcoin 減少財務監控的目的相去甚遠。



好消息是，今天，操作自己的 Lightning 節點不再是技術專家的專利，就像 2010 年代末期一樣。家庭用戶可以使用相對簡單的解決方案，我們將在下一章詳細解釋。




## 選擇合適的解決方案


<chapterId>615870e3-741d-4ec1-875d-a483e70f39d4</chapterId>



如今，您有可能在保持自我監管的情況下，獲得與 「閃電 」監管 wallet 非常接近的用戶體驗。本章的目的是幫助您選擇最適合您的路徑。



### 選項 1：不直接使用 Lightning



第一種解決方案就是不使用Lightning本機，而是使用嵌入[原子交換](https://planb.academy/resources/glossary/atomic-swap)的Bitcoin或[Liquid](https://planb.academy/resources/glossary/liquid-network) wallet。例如，Aqua 或 Bull Bitcoin Wallet 應用程式使用此方法，讓您無需自行操作 Lightning 節點，即可支付 [Lightning 發票](https://planb.academy/resources/glossary/invoice-lightning)，同時保持自我監控。



原理很簡單：您的資金停留在 Bitcoin 中的 on-chain 或 Liquid 上，您透過 wallet 存取資金，您在 wallet 中以傳統方式持有鑰匙。當您掃描 Lightning 發票時，您的 wallet 會向原子交換服務啟動一筆交易（on-chain 或 Liquid）。然后，该服务通过自己的节点，使用您提供的 on-chain 或通过 Liquid 提供的比特币管理闪电付款。因此，您不需要自己處理任何 Lightning 通道，但仍可以無縫結算 Lightning 發票。



![Image](assets/fr/013.webp)



與傳統的 Lightning 託管 wallet 相比，這種方式的主要優勢在於您始終 100% 擁有您的資金。比特幣在您的 onchain 或 Liquid wallet，有您自己的[助記短語](https://planb.academy/resources/glossary/seed)。即使在交換過程中，您仍然擁有您的資金，因為交換是原子式的。它依賴於加密機制，確保只有兩種可能的結果：一是交換完全成功，二是交換失敗，服務不能佔用您的資金。



大多數提供此類功能的組合都依賴 [Boltz](https://boltz.exchange/)來進行技術部分的交換。



此解決方案在保密性方面也有有趣的優勢，尤其是搭配 Liquid 時。對於初學者來說，它也非常容易設定和儲存：一個經典的記憶詞彙、沒有頻道、沒有需要平衡的流動性...



另一方面，這種方法也有其限制。首先，它不是不可替代的：您依賴於交換服務的可用性和善意。如果它不想再處理您的帳戶，或停止運作，您就無法再透過它支付 Lightning 發票。此外，您還要支付不小的費用：您既要支付onchain或Liquid的[交易費用](https://planb.academy/resources/glossary/transaction-fees)，也要支付交換服務的佣金。此外，如果 onchain 費用大幅增加，使用 Lightning 可能會變得非常昂貴。



因此偶爾使用還可以接受，但對於非常活躍的 Lightning 使用者來說，最好還是使用真正的 Lightning 節點。



### 選項 2：板載 Lightning 節點



第二類解決方案是以直接嵌入行動應用程式的 Lightning 節點為基礎。Phoenix Wallet 開創了這種模式，至今仍是業界的標竿。如今，其他專案也提供了類似的方法，例如 Zeus（嵌入式模式）或 BitKit。



這個想法很簡單：應用程式實際上是執行一個 Lightning 節點，但所有複雜的操作都在後台自動處理。您有一個 "*Lightning wallet*"介面，上面有備份的助記短語，您可以看到餘額和支付發票，但您不需要管理渠道、流動資金或大多數參數。



![Image](assets/fr/014.webp)



這些解決方案總是自我保管。控制資金的鑰匙是 generated 並儲存在您的手機上，備份則是透過 seed 或同等的機制。您並不是簡單地持有服務供應商的帳戶，您實際擁有的是鎖定在通道中的比特幣，這些比特幣屬於您，無法被盜取。



LN 板載節點的優點有很多：




- 非常容易安裝和使用；
- 使用者體驗接近於保管型 Lightning wallet，但具有自我保管功能；
- 無須手動管理通路或流動性；
- 相對簡單的備份。



但這些內嵌式 wallet 也有很大的限制。首先，就機密性而言，服務業者（例如 Phoenix 的 ACINQ）可以相當詳細地查看經過您節點的流量：數量、頻率、接收者，即使這一定會有所改善，尤其是隨著 *Trampoline 節點*的逐漸採用。其次，您嚴重依賴此操作員作為您的主要 Lightning 對等。如果 ACINQ 節點不可用（在 Phoenix 的情況下），如果該公司受到監管壓力或改變其業務模式，您的用戶體驗可能會嚴重下降，甚至受到影響。



最後，這種簡單性是有代價的。嵌入式 LN 節點服務一般會對存款、提款或某些通道管理操作收取特定費用。在我看來，就提供的服務而言，這種模式是合理的，但對於密集型使用而言，其成本可能遠高於管理完善的傳統 Lightning 節點。



### 選項 3：經典的 Lightning 節點



第三種解決方案，也就是我們將在本 LNP202 課程中深入探討的方案，是在專用伺服器或裝置上操作傳統 Lightning 節點。



我所說的 「傳統 」是指您在自己的 Bitcoin 節點之上自行安裝和配置 Lightning 實現（例如 LND）。您選擇您的對等、開啟您的通道、管理您的[入站和出站流動性](https://planb.academy/resources/glossary/inbound-capacity)，並設定您的路由費政策。



就主權而言，這是最好的解決方案。您的頻道或付款不再依賴於特定的公司：如果對方審查您或關閉頻道，您可以用不同的節點開啟另一個頻道。如果某項服務消失了，您的 sats 仍保留在您控制的頻道中，您可以在鏈上將其遣返。您也可以優化您的長期成本：一旦您的頻道有正確的大小和管理，整體的支付成本可以變得非常低。



在保密性方面，您顯然會受到 Lightning 自身模式的限制，但您不會將業務全部交給單一營運商。



相比之下，建立經典的 Lightning 節點顯然更加複雜：您必須安裝、配置、維護、監控更新，瞭解頻道和收費政策的邏輯，管理頻道和現金流等等。配置錯誤、備份馬虎或管理粗心更容易導致 sats 的損失。節點也必須持續運行。



這正是我在本課程中建議您遵循的路徑，陪伴您走過每一步，以限制風險並安排您的方法。



### 哪個解決方案適用於哪個使用者設定檔？



要為您的Lightning用戶檔案選擇合適的解決方案，您需要考慮兩個因素：您使用Lightning的頻率，以及您對技術管理的需求。



如果您偶爾進行的 Lightning 付款不多，但仍希望能不時用手機結算 LN 發票，而又不放棄自我保管：具有交換功能的 Bitcoin 或 Liquid wallet 可能是最佳選擇。您保留資金的所有權，不需要管理任何節點，並接受稍高的費用以換取簡單。



https://planb.academy/tutorials/wallet/mobile/bull-bitcoin-2c72127c-a228-4f50-b833-c6183d56aaf6

如果您經常使用 Lightning，並希望獲得真正節點的優勢，但又不想花時間管理渠道、費用或基礎設施，那麼移動嵌入式 Lightning 節點是一個很好的解決方案。您可以保留資金的保管權，UX仍然非常容易使用，所有的複雜性都被隱藏起來，但代價是對運營商的明顯依賴。

## 儲存您的 Lightning 節點


<chapterId>638fa75d-62af-4bf3-ab4a-b7d10ea75815</chapterId>



在發送您的第一個 sats 到您的節點之前，了解其備份工作原理以及相關風險非常重要。與簡單的 Bitcoin onchain 組合不同，備份 Lightning 節點相當複雜：錯誤的策略可能導致您的資金永久損失。在本章中，我們將探討真正需要備份的內容，以及 Umbrel 與 LND 如何處理這個流程。



在本課程中，我們將使用 LND (*Lightning Network Daemon*) 實作。雖然其他實作的原理類似，但我要談的復原檔案和程序是 LND 特有的。



### 在 Lightning 節點上我應該節省什麼？



在 Lightning 節點上，僅保存 seed 並希望一切恢復正常是不夠的。有幾種元素扮演著不同的角色，因此必須區分它們。



#### seed (*aezeed*)



初始化 LND 時，您會收到一個 24 字元的 seed。這是 LND 特有的格式，稱為「*aezeed*」。它不是經典的 seed BIP39，雖然看起來很像。從這個 seed 中，LND 可以得到您與 Lightning 節點相關的 onchain wallet 的私密金鑰，也就是在通道關閉後，您可以接收比特幣或將比特幣匯回的地址。



![Image](assets/fr/019.webp)



因此，此 seed 可用於重新創建與您節點相關的鏈上 wallet，以及取回已在鏈上遣返的資金（例如通道關閉後）。另一方面，單靠 seed 並不足以恢復您仍然開放的 Lightning 通道，因為它不包含通道狀態的必要資訊。



#### 頻道資料庫 (`channel.db`)



LND 會將頻道的詳細狀態儲存在名為 `channel.db` 的資料庫中。此資料庫包含




- 您開啟的頻道清單；
- 您的同伴及其識別碼；
- 每個通道的最後一個 commitment transaction（定義通道中誰擁有什麼的連續狀態，並可在發生問題時收回鏈上資金）；
- 懲罰試圖廣播舊報告的同伴所需的資訊。



這個資料庫的問題在於它會不斷改變：每次付款、每次路由、每次開啟或關閉通道都會修改其內容。這就是為什麼傳統的備份 (例如定期複製您的 `channel.db`)是危險的。如果您還原太舊版本的 `channel.db`，並用這個過時的狀態重新組裝節點，您的對等方可能會認為您在廣播舊的通道狀態。在這種情況下，協定會提供懲罰：您的對等者可以收回頻道的全部資金，就像您試圖作弊一樣。



實際上，`channel.db` 並不是備份媒介。它是您節點的生存狀態。使用它來還原您的節點唯一合理的情況是，您直接從剛發生故障的機器 (例如仍可讀取的磁碟) 復原此資料庫。在這種情況下，您可以復原最新的狀態，並根據此資料庫在另一台機器上重新啟動 LND，因為舊機器已不再執行，所以可以放心知道此狀態盡可能是最新的。這種方法可以作為相關備份的另一種情況是在雙磁盤配置中，從一個磁盤動態永久複製到另一個磁盤。不過，這種設定的實作較為複雜。



但定期複製 `channel.db`，並將它們儲存為稍後還原的備份是個非常糟糕的主意：當您使用它們的那一天，您就有可能懲罰自己，並失去所有的 sats。



#### 靜態通道備份 (SCB)



為了讓災難復原成為可能，LND 引進了 *Static Channel Backup* (SCB) 機制。這是一個特殊檔案，通常稱為 `channel.backup`，其中包含您頻道的靜態資訊：誰是您的對等人、如何聯絡他們以及您的頻道是什麼。



此檔案不包含詳細的頻道狀態或更新歷史。它不允許您以原來的狀態重新開啟頻道，也不允許您繼續操作這個 Lightning 節點。相反，它的作用是作為一個錨點，請您的同伴幫您乾淨地關閉所有頻道，從而接收您的 sats onchain，您可以透過 seed 擷取鑰匙。因此，與每次付款或路由都會修改的 `channel.db` 檔案不同，SCB 檔案只會在通道開啟或關閉時更新。



透過 SCB 復原時，流程如下：




- 您還原您的 seed (*aezeed*)，重新建立與 Lightning 節點相關的 onchain wallet；
- 您向 LND 提供您最近的 SCB；
- LND 使用 SCB 來尋找您的同伴清單以及您與他們之間的頻道；
- 它會聯繫每個對等人，告訴他們您遭受了資料遺失，並要求他們「強制關閉」您與他們的通道，這樣您就可以恢復您的 onchain 共用。



我們的想法是，您的同行注意到您報告資料遺失，就會廣播他們最後的 commitment transaction 並關閉力通道。一旦這些交易得到確認，您的資金就會重新出現在您的 onchain 投資組合中（與 seed 相關聯）。



然而，這種復原機制並非完美。首先，它不能真正恢復您的 Lightning 節點，因為所有通道都會關閉。您必須從頭建立一個新的 Lightning 節點。其次，它不能保證 100％ 復原，儘管它確實大大增加了在發生問題時復原您的鏈上餘額的機會。事實上，這種復原協議取決於您的對等方的合作和可用性：如果其中一個對等方離線、丟失了自己的資料或拒絕合作，您的資金可能會繼續被凍結，甚至永久丟失。這就是為什麼在您的 Lightning 節點上不要長時間與無法連線的對等方保持通道打開的原因。如果您在此時發生資料遺失，而對等端仍無法連線，則無法透過 SCB 進行復原，您的資金將一直遺失，直到該對等端重新連線為止（也許是永遠）。



總而言之，LND 上良好的 Lightning 備份策略有三大支柱：




- 您的 seed（*aezeed*），用於 onchain 層；
- 可靠的自動 SCB 備份；
- 良好的通路管理，選擇可靠的合作夥伴，並預防性地關閉那些經常無法聯絡的合作夥伴。



### Umbrel 如何管理 LND 節點的備份？



Umbrel 為 LND 節點提供了簡化的備份機制，完全基於 SCB。這樣做的目的是為了省去您自己操作此檔案的麻煩，將其備份並盡可能自動化。



當您在 Umbrel 上建立節點時，您會收到一個扮演雙重角色的 seed：




- 它會衍生出與您的 Lightning 節點相關的 Bitcoin onchain wallet；
- 它用於推導用於遠端 SCB 備份的備份識別碼和加密金鑰。



藉由此機制，Umbrel 會自動為您的 SCB 加密備份，並透過 Tor 將其儲存在伺服器上。SCB 是透過您的 seed 所產生的金鑰加密儲存的。因此，如果發生資料遺失，您只需在同一台或另一台機器上重新建立 Bitcoin 和 Umbrel 上的 Lightning 節點，然後輸入您的 seed。然後您就可以從 Umbrel 伺服器擷取最新的 SCB 狀態，將其解密並開始恢復資金的程序。



這些備份在傳送前會由您的節點進行本機加密，以保證資料的機密性：Umbrel 無法讀取 SCB 的內容。傳輸透過 Tor 進行，可防止您的 IP 位址外洩。更重要的是，您的 Umbrel 會在流量中加入雜訊（隨機填充和不定時傳送的錯誤備份），以防止伺服器精確推斷出您開啟或關閉通道的時間。



這種方法的主要優點是大大簡化了 Lightning 節點的備份：您只需要在節點初始化時備份一次 seed。這確實會帶來一些風險，因為這只是 SCB 的備份，但對於合理的數量來說，這是可以接受的折衷辦法。



### 限制損失風險的最佳作法



即使有 Umbrel 備份，幾個簡單的良好作法也能大幅降低遺失 sats 的風險：





- 監控同行的可用性：



如果某個重要通道經常與無法連線或不穩定的對等端相關聯，那麼在您的節點仍在運作時乾淨地關閉它會比較安全。預防性合作或強制關閉可消除 SCB 復原時的潛在麻煩來源。





- 避免將過多的流動資金集中在未知的同業身上：



對等體與您的通道越大，可靠度就越重要。請選擇認真、連線良好且活躍的節點，以便日後透過 SCB 進行的任何復原都能順利進行。





- 以本地備份補充 Umbrel：



除了 Umbrel 的自動備份外，您也可以在外部媒體上保留 SCB 檔案的加密副本 (`channel.backup`)，並定期更新。理想情況下，您應該在每次開啟或關閉通道時更新它。如果因為任何原因，Umbrel 的自動備份服務無法使用，您可以使用這個備份解決方案。





- 以正確的方式管理您的 seed



您的 seed Umbrel 不僅可以還原您的 onchain wallet，還可以取得備份的加密金鑰。因此，能存取 Umbrel 的攻擊者可以啟動復原，將您的資金轉移到他自己的 wallet，甚至不需要實體存取您的節點。



因此，如果您需要復原您的節點，但已沒有您的 seed，您將無法復原任何東西：您所有的 sats 都會遺失。因此，請務必小心保存您的 seed，只能保存在實體媒體 (紙張或金屬) 上，並將其存放在安全的地方。如需管理 seed 的詳細資訊，請參閱本教程：



https://planb.academy/tutorials/wallet/backup/backup-mnemonic-22c0ddfa-fb9f-4e3a-96f9-46e2a7954270

既然您已經瞭解了理論上的運作方式，現在讓我們進入實際的操作。從您的 Lightning Node 應用程式 (實際上是 LND)，按一下右上角的三個小圓點。

- 然後您可以使用 `Backup over Tor` 選項，選擇透過 Tor 或 clearnet 傳送。如前幾節所述，我強烈建議您使用 Tor 來保護您的機密。

## Watchtower：角色與設定


<chapterId>e6c654dd-26c5-4e4d-8d11-a215bac37812</chapterId>



在 Lightning 中，每個通道都是基於一連串的連續狀態，由未公開的 commitment transactions 表示。每一次 Lightning 付款或路由，通道中的兩個參與者都會建立一對新的 commitment transaction，反映通道中當前的資金分配。舊的 commitment transaction 會被淘汰。



如果其中一方發佈了過期的狀態，另一方有權對其進行制裁，並收回通道的全部資金。在本章中，我們將簡單介紹這個機制的運作方式，然後解釋如何建立一個***watchtower***：一個保護您的Lightning節點免受可能的作弊企圖的系統。



### 瞭解瞭望塔如何運作


在任何特定的時刻，頻道中的每一方都有一個 commitment transaction，如果公佈了這個 commitment transaction，他們就可以關閉頻道，收回他們應得的資金。這個過程稱為 * 強制關閉*。但如果他們嘗試發佈一個較舊的 commitment transaction，對應於通道之前持有較多 sats 的狀態，則此交易將被視為作弊嘗試。在這種情況下，交易對方可以使用與這個較舊狀態相關的撤銷金鑰來取回通道中的全部資金，而作弊者則會暫時被時間鎖封鎖。


這個系統意味著發佈舊的狀態，也就是試圖作弊，是非常危險的：如果對方在時間鎖到期前在 mempool 或區塊鏈上看到這筆交易，他們就可以使用撤銷金鑰，收回所有資金。 **因此，您的 Lightning 通道是否安全，取決於您是否有能力在時間鎖**的時間窗內偵測到作弊企圖。



#### 為什麼需要瞭望塔？



懲罰機制只有在受害方有能力的情況下才會起作用：




- 監控每個新的 Bitcoin 區塊，查看是否已發佈頻道 commitment transaction；
- 確定此交易是否對應於最後的有效狀態或已撤銷的狀態；
- 在撤銷狀態的情況下，及時廣播合法交易，使用撤銷金鑰在時間鎖到期前收回所有資金。



![Image](assets/fr/023.webp)



在理想情況下，您的 Lightning 節點全天候在線，它會同步並持續監控區塊鏈。因此，它可以單槍匹馬地偵測到作弊企圖，並作出反應。但實際上，個人 Lightning 節點可能會關閉，尤其是在長時間斷電或網際網路連線故障的情況下。



正是在這些宕機期間，風險才會變得真實：如果不誠實的同業在您的節點離線時公佈舊的狀態，而時間鎖用完時您也沒有任何反應，那麼作弊就變得有效了。您會失去部分或全部的通道資金。



引入 Watchtower 是為了降低此風險。watchtower 是一種外部服務，它代表您監控區塊鏈，掃描您的某個通道是否可能發布舊狀態，並在必要時代表您自動廣播懲罰交易。因此，即使您的 Lightning 節點長時間處於離線狀態，只要您使用的 watchtower 仍在運行，它就能夠通過監控任何作弊嘗試來保護您的資金，並在檢測到作弊嘗試時立即實施相應的處罰。



#### 瞭望塔如何運作



監視塔的設計是為了盡量減少它所得知有關您頻道的資訊，同時讓它有辦法在發生問題時採取行動：




- 對於與對等人的每個新通道狀態，您的節點都會事先準備一個潛在的懲罰交易。如果此對等人作弊，此交易將允許您收回通道中的所有資金；
- 然後，您的節點會使用對應的 commitment transaction 的 TXID (如果作弊者企圖詐騙時會使用的 TXID)，來加密這個懲罰交易。只要沒有閉鎖，守望塔就無法解密此交易，因為它不完全知道作弊交易的 TXID；
- 您的節點會傳送一個封包給 watchtower，這個封包包含加密的懲罰交易和潛在作弊交易的一半 TXID。



由於傳送給 watchtower 的 TXID 並不完整，因此它無法解密正義交易。但是，它可以監控區塊鏈，尋找與它擁有的部分相匹配的 TXID。如果偵測到這樣的交易，它就會嘗試使用該交易的完整 TXID 來解密您的懲罰交易。如果解密成功，它就知道這是一次作弊嘗試，並立即為您公佈罰款交易。



![Image](assets/fr/024.webp)



因此，瞭望塔無法看到您的頻道細節：無法看到您的對等人身份、餘額或交易結構。它只能看到加密的封包。它唯一能推斷的資訊是您的通道更新速度，因為它會收到每個新狀態的封包，但無法知道封包的內容。在作弊的情況下，它一定會透過解密懲罰交易來發現通道資訊，但至少您的 sats 會被保存下來。



此機制基於一種折衷辦法：您將發佈預先簽署的懲罰交易的能力委託給 watchtower，但此交易對 watchtower 來說仍是完全不透明的，直到發生某些作弊行為為止。watchtower 既不能修改收款人，也不能轉移資金，因為它只擁有已簽署的交易，而且輸出已凍結，對您有利。它也無法知道合法強制或合作關閉中通道的細節，因為 TXID 並不匹配。另一方面，watchtower 仍然是最低限度的可信第三方：您需要依賴它在線，並在您需要時正確地廣播您的正義交易。



#### 成為瞭望塔



理論上，任何 Lightning 節點都可以作為其他節點的瞭望塔 (如果這些節點使用相同的實作，例如 LND)，同時本身也受到其他節點的保護。在以下的實作部分，我會告訴您如何在 Umbrel 下的 LND 上設定這個簡單的機制。


因此，一個有趣的策略可能是與可信賴的 bitcoiner 朋友達成協議，互相監督。你監控他們的頻道，他們也監控你的頻道。

## 開啟您的第一個 Lightning 頻道


<chapterId>00642af7-8f3d-4a25-96d7-34e85de7bd5d</chapterId>



如果您已經瞭解到這一點，您已經知道沒有通道的 Lightning 節點有點像空的 wallet：它存在，但沒有用。為了能夠發送或接收付款，您的節點必須通過通道與 Lightning 網絡中的至少一個其他節點連接。在未來，我們強烈建議開通多個通道，以提高復原能力和路由效率。在接下來的章節中，我們還將瞭解如何管理您的流動資產、優化您的通道拓樸結構，以及在 Umbrel 上使用比基本 LND 介面更先進的工具。



但在這入門章節中，目的只是要了解如何選擇一個好的 Lightning 對等體、在哪裡找到選擇對等體所需的資訊，以及最後如何透過基本的 LND 介面開啟第一個頻道。



### 如何選擇好的 Lightning 同行？



當您開啟通道時，您需要選擇一個對等點：另一個Lightning節點，您的節點將通過通道直接與該節點連接。對等節點的選擇非常重要，因為它將直接影響到以下幾個方面：




- 讓您的付款更容易找到途徑；
- 隨著時間的推移，您的頻道的可靠性；
- 您的路由成本。



不存在適合所有人的完美匹配，但有許多可靠的標準可用來識別優秀的候選人。



#### 1.節點連線



一個好的對等節點是一個與 Lightning 網絡連接良好的節點。這意味著不僅要有大量的通道（雖然這可能是一個指標），最重要的是要與其他可靠的節點連接，並在網絡圖中佔據足夠的中心位置。



連線良好的節點可增加您找到通往大多數付款目的地的有效路由的機會。這也減少了所需的中介節點數量，從而降低成本。



相反地，開放您的第一個通道給一個孤立或弱連結的節點，可能會限制您的可能性：您可以付費給這個對等人，但要付費給網路的其他部分就難多了。



#### 2.資本化與通路容量



一個好的對等點也是一個資本充足的節點。這可以從它的總頻道容量 (投入所有頻道的 sats 總和) 和平均頻道容量 (通常只有幾個資本充足的頻道比很多小頻道要好) 看出來。



容量大得離譜或通道都很小的節點，將無法路由大金額的付款，導致您的路由失敗。



#### 3.費用政策



每個節點設定自己的路由費用 (「基本費用」和「費率」)。好的對等節點不會對戰略通道收取高昂的費用。對於第一個頻道，最好選擇收費較低的節點，以免影響自己的付款。



請記住，您自己的路由費用也會影響他人對您作為同儕的觀感：不斷變更費用或強加荒謬費用的節點，很少會受到讚賞。



#### 4.穩定性和年資



對等穩定度是一個非常重要的標準。一個好的節點有很高的正常運行時間 (很少離線)，它能長時間保持頻道開放，也不會無緣無故地不斷開放/關閉頻道。



舊且仍然活躍的通路是個好信號：它們顯示這種關係對對等人來說是有利可圖的，節點知道如何管理其資本，而且不會隨時關閉，因此您不必為關閉和重新開啟而不斷支付上鏈費用。



相反地，經常離線或很快就關閉通路的同儕，可能會為您帶來問題，並增加開闢新通路的成本。



即使有了這些標準，您也不會第一次就做出完美的選擇。這很正常：同儕的真正品質是在使用中顯露的。因此，重要的是：




- 監控通道活動（路由流量、可用性等）；
- 關閉沒有用處的頻道，或其對等人經常不在線的頻道；
- 隨著時間的推移，將您的資金重新分配給更好的同行。



我們的想法不是每天開啟和關閉通道（這會造成昂貴的 onchain 成本），而是逐步演進您的拓樸結構，以匯聚於一組可靠、連線良好且符合您需求的對等點。

## 定義您的節點操作員設定檔


<chapterId>d3b2e163-50f6-4d1d-a5fc-8fd177dfac76</chapterId>



現在您的 Lightning 節點已經啟動並運行，下一步就是定義您的交易者配置文件。這是一個重要的選擇，因為它決定了您的通道開啟策略、您喜歡的同行類型以及您管理流動性的方式。



在 Lightning 上，為了實現這一目標，您需要正確方向的流動資金。流出的流動資金對應於您的支付能力（sats 可在您一方的渠道上使用）。流入的流動資金對應於您的接收能力（sats 可用於您的同行端）。換句話說，您的資料可以歸結為一個簡單的問題：大部分時間，您的 sats 是離開您的節點，還是進入節點？



### 消費者



這就是絕大多數用戶的情況。您使用您的節點進行 Lightning 付款：購買商品和服務、支付帳單、寄送小費 - 簡而言之，使用 Lightning 作為一種快速的付款方式。另一方面，您從 Lightning 收到的款項很少，或只有很少，例如幾筆捐款、朋友之間的退款或幾筆微額付款。



這種狀況最容易管理，因為您的主要需求是能夠支付。實際上，這表示您需要流出的流動資金。一旦您開通了一個或多個大小合適的通道到穩定、連接良好的節點，您的流出付款將機械地把流動資金轉移到通道的另一端。正是這種移動自然而然地創造了合理數量的流入流動資金。因此，即使您不希望定期收款，您仍然可以不時收款，而無需實施複雜的策略。所以您不需要擔心您的流入流動資金。



在本 LNP202 課程中，我們將專注於這個「消費者」節點操作員設定檔，因為它幾乎對應 Lightning 上所有的 Bitcoin 使用者。



### 零售商



在某種程度上，商家與消費者正好相反。在這裡，主要目的不是付款，而是收款。一個接受 Lightning 的企業、服務供應商、網路商店或銷售點，會從這個節點收到許多進來的款項，並進行相對較少的支出。



這種情況要求較高，因為 Lightning 上的拒付可能代表銷售損失。因此，優先級變為入站流動資金。如果您的節點沒有足夠的入站流動資金，即使您的客戶有足夠的資金，他們的付款也會失敗，原因很簡單，因為沒有正確的途徑將流動資金傳遞給您。



商家面臨的主要挑戰也來自於通路的自然演進。如果您所做的只是接收，您這邊的通路就會逐漸填滿。因此，您需要機制來維持和更新您的流入流動性。



然而，還有一種更簡單的情況：消費者/商家混合模式。如果您在 「閃電 」上收款，同時也在 「閃電 」上消費（商業開支、支付給供應商的款項，甚至是個人開支），那麼您的出款自然會重新產生入款。由於資金流互相抵銷，管理變得更順暢，您也更不需要使用僅為恢復流入資金而設計的人工機制。



### 路由器



路由器是一個特定的配置文件。它不使用自己的 Lightning 節點來付款或收款，而是用來路由其他人的付款和收取費用。其目的是在 Lightning 圖形中成為一個有用、可靠且具有經濟競爭力的路由。



老實說，在 Lightning 上做路由器是一門比表面上更複雜的生意，盈利也很難實現。首先，打開和關閉通道的在線成本很高。為了有效率地進行路由，您經常需要調整拓樸、測試、關閉表現不佳的通道、開新通道，並定期重新平衡您的流動性。其次，競爭非常激烈：大型、已建立的節點自然會攫取大部分流量，若不在規模龐大的頻道中投入大量資金，就很難站穩腳步。



還有很高的作業要求。路由節點必須極其穩定、受到監控、有適當的備份，並受到嚴格的管理。您必須監控通道效能、調整費用政策、維持均衡的流動性、管理同業，以及快速解決技術問題。這種程度的參與可以是有趣的技術嗜好，或是對基礎建設的貢獻，但如果您的目標只是以主權的方式使用 Lightning，進入路由賺錢就很少相關了。這就是為什麼這門 LNP202 課程沒有深入涵蓋這個側面的原因。



### 在繼續前進之前，先清楚定位自己



如果您符合消費者的條件，您的優先考量就是能夠以簡單的管理方式可靠地付款。如果您是商家，您的優先考量將是無失敗地套現，這意味著您要採取內向式的流動資金取得策略。如果您正在考慮路由，您需要將其視為一項要求高、經濟上不確定且耗時的活動。



現在定義這個設定檔將幫助您避免一個典型的陷阱：應用為商家或路由器設計的通路策略，而您只是想要付款的使用者。

## 獲得流入的流動資金


<chapterId>b740c656-a897-4d95-af4b-116b718447cd</chapterId>



正如您所看到的，在 Lightning 上擁有流出的流動資金來進行支付並不特別複雜。您只需主動向其他節點打開通道，然後開始發送 sats。另一方面，在 Lightning 上接收付款的流入流動資金則比較複雜，因為您需要其他節點為您打開通道，或者您需要自己通過付款來移動流動資金。



如果您採用的是 「消費者 」配置文件，則通常無需擔心流入的流動資金。您對 Lightning 節點的使用將主要以付款為導向，而非現金流入。只需進行幾次 Lightning 付款，隨著時間的推移，您自然會產生流入的流動資金。



另一方面，如果您有 「商家 」個人資料，情況就會相反：您主要是收取款項，很少會付款。因此，您不能僅僅依賴自己的付款來獲得流動資金。因此，其他 Lightning 節點有必要為您打開通道。但如何才能做到這一點呢？如何讓其他運營商為您綁定資金？這正是本章要探討的問題。



### 購買入境流動資金



正如您所預期的，最有效的方法就是付錢給他們。對於進入您的 Lightning 節點的流動資金，原理完全相同。最簡單的方法是支付服務費用和相關的資金。



如果您是企業或零售商，這種方式意味著您可以快速取得可靠的管道，向客戶收取付款而不會產生摩擦。



買入流動性的方法有很多。我個人使用並推薦的是 Amboss 的 [Magma](https://magma.amboss.tech/) 平台。它非常容易使用，打開通道很快，而且費率一般都很合理。Magma 的運作方式就像一個市場，有製造商和接受商，但第二版已大大簡化了使用體驗：只要建立一個請求，透過 Lightning 支付價格，Magma 就會自動將該請求與可用的最佳報價進行匹配。在六次上鏈確認之後，您與流入流動資金的通道就開始運行了。以下是運作方式：



前往 [Magma 網站](https://magma.amboss.tech/buy)，在「購買頻道」一節。



![Image](assets/fr/052.webp)



如果您願意，您可以創建一個帳戶來追蹤您的購物，但這並非強制性的。如果您不創建帳戶，Magma只會向您提供一個會話ID，以便您日後檢索您的購買記錄。



進入網站後，填寫購買流動資金所需的資訊。若要一次性購買，請選擇「一次性」，然後輸入您願意為流入的流動資金支付的金額。支付的金額越高，向您的節點開放的通道容量就越大。下面顯示通道容量的估計值：這只是一個近似值，因為最終金額將取決於 Magma 找到的最佳出價，有時較高，有時較低。



![Image](assets/fr/053.webp)



然後輸入您的節點 ID。例如，您可以在 Umbrel 上的 `Lightning Node` 應用程式的 `Node ID` 功能表中找到它。



![Image](assets/fr/054.webp)



完成所有資訊後，按一下「檢視並開啟訂單」按鈕。



![Image](assets/fr/055.webp)



如果您尚未建立帳戶，Magma 會提供您一個會話金鑰和一個備份檔案。請將這兩樣東西放在安全的地方，因為它們可以讓您日後擷取訂單，或在發生問題時追蹤訂單狀態。保存檔案後，點擊 "Pay with Lightning "按鈕。



![Image](assets/fr/056.webp)



然後，Magma 會根據您選擇的金額開出一張 Lightning 發票。您必須支付。如果您的 Lightning 節點上已有通道，您可以直接從它支付，或使用另一個外部 Lightning wallet 支付。



![Image](assets/fr/057.webp)



付款完成後，交易會在 Magma 介面上顯示為正在處理中。



![Image](assets/fr/058.webp)



幾分鐘之後，請求就會被處理：一個 sats 的通道正在開啟到您的 Lightning 節點。一旦開啟的交易在鏈上得到確認，您就可以獲得相應的流入流動資金。



![Image](assets/fr/059.webp)



Magma 也直接整合到 ThunderHub 中。在 "Home 「標籤中，向下捲動到 」Liquidity 「部分，然後點擊 」Buy Inbound Liquidity"。您只需輸入所需的金額並確認即可。您不需要手動支付任何發票，因為 ThunderHub 會自動處理您節點的付款。



![Image](assets/fr/060.webp)



如果您是商家，這類服務尤其適合，因為它可以讓您快速從可靠的節點獲得大量流入的流動資金，保證您的客戶可以毫無困難地向您付款。另一方面，如果您是個人，或者您不想為流入的流動資金支付費用，也有免費的解決方案。讓我們來看看。



### 合作入境流動資金



如果您不是交易者，但仍然需要一些流入的流動資金（例如，在您等待一些付款的同時，為您的節點在啟動時充電），您不一定要為此支付費用。我的首選解決方案之一是與其他同樣需要入站流動資金的節點營運者合作，為彼此打開 Lightning 通道。這樣一來，打開通道為您帶來流出的流動資金，同時也為您提供流入的流動資金，而且是免費的（開放的鏈上費用除外）。



當然，您可以直接與比特幣同業安排。但是，有一個平台專門用於這種循環開戶：[Lightning Network +](https://lightningnetwork.plus/)。它的原則不是在同一個人之間開通兩個通道，而是建立至少有三個參與者，甚至更多參與的循環開通。



讓我們舉個例子來了解 Lightning Network + 如何運作：




- 一個名為 `A` 的節點操作員發佈了一則公告，說明他希望與另外兩個人開啟一個 100 萬 sats 通道；
- 廣告會維持有效，直到有更多參與者加入為止；
- 之後，兩個操作者 `B` 和 `C` 加入 `A` 節點公告。三角形現在已經完成，可以開始開放階段。
- 節點 `A` 開啟一個通道到節點 `B`；
- 節點 `B` 開啟一個通道到節點 `C`；
- 節點 `C` 開啟一個通道到節點 `A`。



操作結束時，每個節點有 100 萬 sats 的流出流動資金和 100 萬 sats 的流入流動資金。此方案可擴展至四或五位參與者。



當然，沒有任何技術機制可以保證參與者會真正打開他們承諾建立的頻道。這就是為什麼平台設立了一個聲譽系統，當經營者履行承諾時，就會獲得正面評價。在最壞的情況下，如果您遇到不合作的人，您不會損失任何金錢：您只會錯過一個獲得流動資金的機會。



此解決方案特別適合「消費者」，因為它允許您免費獲得流入的流動資金，同時將您的節點連接到其他小型運營商。另一方面，如果您是交易者，這種方法一般不適用：每一坐點的流入流動資金都需要阻擋一坐點的流出流動資金，而您對流入流動資金的大量需求會涉及到過多的資金負擔。



要使用 Lightning Network +，您有兩個選擇：使用 Umbrel 整合的應用程式，或使用經典網站，並從您的節點登入建立帳號。我建議使用後者，因為整合後的應用程式無法提供完整的可用功能。



進入 [Lightning Network +](https://lightningnetwork.plus/) 網站，點選介面右上方的「登入」按鈕。



![Image](assets/fr/061.webp)



要認證自己的身份，您需要使用您的 Lightning 節點的私人金鑰簽署所提供的訊息。使用ThunderHub，這個操作非常簡單。首先複製LN+顯示的訊息。



![Image](assets/fr/062.webp)



在 ThunderHub 中，移至「工具」標籤，然後按一下「訊息」部分的「簽名」按鈕。



![Image](assets/fr/063.webp)



貼上 LN+ 提供的驗證訊息，然後按一下`簽署`。



![Image](assets/fr/064.webp)



然後 ThunderHub 使用您節點的私人金鑰簽署此訊息。複製 generated 簽署。



![Image](assets/fr/065.webp)



將此簽名貼入 LN+ 網站的相應欄位，然後按一下「簽入」。



![Image](assets/fr/066.webp)



現在您已經用您的 Lightning 節點連接至 LN+。這個過程允許 LN+ 驗證您是您在其平台上聲稱的節點的合法擁有者。



![Image](assets/fr/067.webp)



如果您願意，可以個人化您的 LN+ 個人資料，例如加入簡短的自傳。



要參與您的第一次循環通道開啟，請前往介面上方的 `Swaps` 功能表。在這裡您可以找到目前所有可用的交換，這取決於您節點的特性。



![Image](assets/fr/068.webp)



要加入現有的交換要約，只需點擊該要約並註冊即可。但是，在 LN+ 上，交換的創造者可能會對參與者施加某些條件，例如最低頻道數或最低節點總容量。因此，特別是在早期，您的節點可能只有很少的優惠。在我的例子中，儘管有幾個頻道已經開放，但我的節點卻沒有出價。因此我建立了我自己的交換，如果您有同樣的情況，也可以這樣做。



按一下「開始 Liquidity 交換」，然後輸入您的報價參數：




- 選擇總參與人數 (3、4 或 5)；

- 在右邊，您可以設定參與者的限制：最小頻道數、最小總容量和接受的連線類型。



設定好所有參數後，按一下「Start Liquidity Swap」按鈕。



![Image](assets/fr/069.webp)



您的交換要約現在已經建立。現在您要做的就是等待其他節點經營者註冊。如果您的條件沒有太多限制，這應該不會花太長時間。請記住在接下來的幾個小時或幾天內監控您的交換狀態。



![Image](assets/fr/070.webp)



所有的交換位置都已就緒：我們現在進入頻道開啟階段。每位參與者都可以從 LN+ 介面看到他必須開啟 Lightning 通道到哪個節點。



![Image](assets/fr/084.webp)



在您這邊，使用 LN+ 提供的節點 ID 打開通道，並遵守指示的數量。正如我們在之前的章節中所看到的，您可以通過ThunderHub、另一個Lightning節點管理器或直接從基本的Lightning節點應用程序界面進行此操作。



![Image](assets/fr/085.webp)



一旦開啟啟動，您就可以看到它出現在等待頻道的部分。在我的例子中，它是節點為 `Plebian_fr` 的頻道。



![Image](assets/fr/086.webp)



然後您可以返回 LN+ 確認您已啟動頻道開啟。只要按一下「頻道開啟已開始」按鈕即可。



![Image](assets/fr/087.webp)



當所有其他參與者也打開他們承諾的管道時，記得給他們留下正面的評語。



![Image](assets/fr/088.webp)



如果遇到困難或延遲，您可以通過頁面底部的評論區直接與同行聯繫。



![Image](assets/fr/089.webp)



有些參與者可能希望從一開始就重新平衡循環渠道，向自己支付一筆款項。這可確保現金在每個渠道中的均衡分配。如果您是 「消費者 」型態，這並非必要，但如果您願意的話，您可以自己進行這項重新平衡，或是暫時將您的通道費用設定為零，讓想做這項工作的同業更容易。有時候沒有人想這麼做。

## 透過 Tailscale 連接行動版 wallet


<chapterId>5fefb222-3f50-4f9d-a170-2ea628be4437</chapterId>



就是這樣，您現在有了一個連接良好的 Lightning 節點，有流入和流出的流動資金。因此，您就可以在現實生活中使用您的 Lightning 節點了。到目前為止，我們一直都是直接使用 Umbrel 上的介面，無論是「Lightning Node」應用程式或「ThunderHub」介面。這些工具可用於發送和接收付款，但顯然未針對日常的 Lightning 付款進行最佳化。介面是專為在電腦上使用而設計，在智慧型手機上並不可行，而且還需要連線到同一個網路才能正常運作（雖然技術上可以透過 Tor 進行遠端連線）。



實際上，作為比特幣使用者，我們所尋求的是在智能手機上的經典 Lightning wallet 界面：通過 QR 碼掃描發票的能力，以及支付和兌現 sats 的簡單界面。這正是我們在本章和下一章要實現的功能。總的想法是在您的智能手機上安裝一個移動 Lightning wallet 應用程式，它可以在任何地方使用（不僅僅是本地網絡），但在後台，它依賴您自己的 Lightning 節點來發送和接收付款。



### 連接行動客戶的解決方案有哪些？



如今，無論是行動應用程式還是您的節點與此應用程式之間的連線類型，都有多種方式。三種主要的連線模式為




- 透過 ***Tor***；
- 透過 VPN ***Tailscale***；
- 透過 ***Nostr Wallet 連線****。



幾年前，我曾經透過 ***Tor*** 連線，但很快就停止了：失敗次數和通訊延遲實在太多了。理論上，它是可行的，但實際上，使用者的經驗是災難性的。因此，我建議您不要採用這種方式。



我當時採用的替代方案是使用 ***Tailscale*** VPN 來確保行動應用程式與節點之間的通訊。這個解決方案效果非常好：即使在吞吐量較低的行動網路中，我的付款也總能順利通過。這就是我要在本章首先介紹的 Zeus 應用程式的方法。



在下一章中，我們將探討另一種較新的解決方案，效果也非常好： ***Nostr Wallet Connect***。這一次，我們會使用 Alby Go 應用程式來為您介紹另一個選擇，不過如果您願意的話，Zeus 也可以相容於 NWC。



### 安裝和設定 Tailscale



對於第一種方法，我們需要 Tailscale。這是基於 WireGuard 的 VPN 解決方案，可讓您安全地連接分散在網際網路上的裝置，同時自動管理加密、驗證和 NAT 穿透。簡單來說，就好像您所有的裝置都和路由器連接到同一個 LAN，即使它們可能在網際網路上的任何地方。



要使用它，請先建立一個帳戶。前往 Tailscale 網站，然後按一下「Get Started」按鈕。



![Image](assets/fr/071.webp)



然後為您的 Tailscale 帳戶選擇一個身份提供者。就我個人而言，我使用其中一個 GitHub 帳戶登入。



![Image](assets/fr/072.webp)



登入後，系統會詢問您幾個關於使用情況的問題。簡短回答即可繼續。



![Image](assets/fr/073.webp)



Tailscale 接著會提供在您的機器上安裝用戶端程式。目前，我們對此不感興趣：直接前往 Umbrel 並從 App Store 安裝 Tailscale 應用程式。



![Image](assets/fr/074.webp)



當應用程式開啟時，按一下「Log In」（登入），然後按認證程序，使用與您建立帳戶時相同的方法。



![Image](assets/fr/075.webp)



按一下「連線」確認。您的 Umbrel 現在已連線至 VPN 網路。



![Image](assets/fr/076.webp)



然後將 Tailscale 應用程式下載到您的智慧型手機，並使用相同的帳號登入。請注意：在 Android 上，無法同時使用兩個 VPN。為了讓 Tailscale 正常運作，您需要停用任何其他使用中的 VPN。此外，每次您要透過 Zeus 使用您的 Lightning 節點時，您需要啟動 Tailscale VPN，否則連線將無法建立。



![Image](assets/fr/077.webp)



在 Tailscale 網站上，現在至少有兩個用戶端已連線，您可以存取管理主控台，上面有連線到網路的所有裝置清單，以及它們的 Tailscale IP 位址。



![Image](assets/fr/078.webp)

## 透過 NWC 連接行動裝置 wallet

如果您不信服 Tailscale 連線，或是管理雙 VPN 似乎太麻煩，本章建議另一種方式，使用遠端行動用戶端透過您的 Lightning 節點支付和接收 sats： ***Nostr Wallet 連線***。



在本範例中，我們將採用 Alby Go 行動應用程式，它的設計非常好，而且特別容易上手。雖然如此，您也可以使用 Zeus 或任何其他與 NWC 相容的行動應用程式。您可以在[`awesome-nwc` GitHub 套件庫](https://github.com/getAlby/awesome-nwc) 找到相容應用程式的清單。



### Nostr Wallet Connect 如何運作？



Nostr Wallet Connect 是一個標準化的通訊協定，允許應用程式或網站在遠端 Lightning 節點上觸發動作，而無需與該節點建立直接的網路連線（無 API LND 暴露、無 VPN、無 `.onion` 服務......）。NWC 定義了應用程式如何制定意向（例如 `pay_intece`）並接收結果。



它的運作相當簡單。您可以掃描 QR 碼或透過深層連結「nostr+walletconnect:」初始化階段作業。這個字串包含會話參數和授權秘訣。然後，當應用程式想要付款時，它會將要求序列化、加密，並在 Nostr 中繼上將其發佈為事件。節點讀取中繼器上的事件、解密、檢查作者是否已獲得此會話授權、執行付款，然後傳回加密回應（成功並帶有預設影像，或錯誤）。中繼只扮演傳輸中介的角色：它不能讀取內容，但可以觀察要求的時間和頻率。



與透過 Tailscale 或 Tor 連線相比，NWC 的主要優勢在於您的節點無法直接從外部接達。這大大簡化了行動使用：您的節點不需要接受進入的連線，只需要能夠與中繼器通訊即可。另一方面，您對 Nostr 中繼引入了功能上的依賴：如果中繼不可用，體驗就會降低。此外，即使訊息已加密，中繼站也能觀察到某種程度的活動元資料。



安全模式的差異也很重要。使用 Tailscale 或 Tor，您會暴露直接存取您的節點 (透過 LND 的 API)，並受到高度敏感機密的保護。這很強大，因為您可以管理一切，但這也是較低層的攻擊面。有了 NWC，存取的應用性更強：您委託會話 token 只授權特定的動作。



### 在您的 Lightning 節點上安裝 Alby Hub



之前，Umbrel App Store 有一個專門用於 NWC 連線的應用程式，但很遺憾，這個應用程式已不再可用。現在您需要使用 Alby Hub 來建立此類型的連線。要做到這一點，首先直接從商店安裝 Alby Hub 應用程式。



![Image](assets/fr/091.webp)



開啟時，跳過介紹畫面，然後按一下「Get Started (LND)」按鈕。重要的是要檢查括弧中是否寫著 `LND`，而不是 `LDK`。如果出現 `LND`，這表示 Alby Hub 已經正確偵測到您現有的 Lightning 節點，並會將自己設定為它的介面。另一方面，如果顯示了 `LDK`，這表示 Alby Hub 未偵測到您的節點，而且正要建立一個新的節點，這不是這裡的目的。



![Image](assets/fr/092.webp)



接下來會要求您建立 Alby 帳戶。對於僅限於 NWC 的使用，這是不必要的，但如果您希望利用 Alby 的特定服務，您可能希望這樣做。如果不需要，請按一下「稍後再說」繼續。



![Image](assets/fr/093.webp)



然後選擇一個強大且唯一的密碼。這將會保護您節點上 Alby Hub 的存取權。請記得將它儲存到您的密碼管理器中。



![Image](assets/fr/094.webp)



這會帶您進入 Alby Hub 介面。您不需要進行整個配置過程，除非您想使用它作為 Lightning 節點的主要管理員。正如我們前面所看到的，Alby Hub 事實上可以取代使用 ThunderHub 來管理您的節點。如果您想了解更多關於 Alby Hub 的選項，請參閱我們的專用教學：



https://planb.academy/tutorials/node/lightning-network/alby-hub-62e6356c-6a6d-4134-8f22-c3b6afb9882a

轉到 `連線 ` 功能表。



![Image](assets/fr/095.webp)



在這裡，您可以看到所有可以透過 NWC 連接到 Lightning 節點的應用程式。其中包括上一章已經提到的 Zeus。在這裡，我們將使用 Alby Go。點擊 Alby Go，然後點擊 `Connect to Alby Go` 按鈕開始連接過程。

## Lightning 長效自動化


<chapterId>691a0942-b46d-482a-8fbc-fe19b3814992</chapterId>



我們的 LNP202 實作課程到此為止。您現在已經掌握了以主權方式使用 Lightning Network 所需的基本知識：您瞭解了節點的真正作用、不同方法的取捨，並在 Umbrel 上以一致的備份和保護策略建立了 LND 實例。您也開通了第一個管道，學會了如何管理流動資金，讓您的付款每天都很可靠。



從操作的角度來看，您的節點現在應該進入維護節奏。最主要的是監控它（正常運行時間、同步、頻道狀態、付款失敗等），在穩定版本可用時套用 Umbrel 提供的更新，並定期檢查您的備份和 watchtower 設定是否仍然有效。



談到頻道時，請採取務實的方法：保留對您有用的頻道，關閉那些長期不活躍或與不穩定的同業有關聯的頻道，並逐步將您的資金重新分配，以建立更強大的拓樸架構。



**在此階段最常見的陷阱之一是為您的 Lightning 節點分配過多資金。請記住，您的 Lightning 節點的安全性遠遠低於硬體 wallet，而且承諾給您渠道的資金的可用性依賴於備份機制，而這些機制仍不完善。因此，保持合理的數額是非常重要的，在發生問題時，您可以承受損失，並始終保持您的 sats 的大部分在硬體 wallet 上。



就工具而言，我建議您保持好奇心並留意新的發展。在這次培訓中，我們發現了其中幾個，無論是用於管理您的節點、其連接性或遠端使用進行支付。然而，Lightning 是一個特別動態的領域。每年都會有新的相關工具出現，Umbrel 上也會出現許多新的應用程式。緊貼這些新發展，可能會讓您發現比本課程介紹的更強大或更實用的解決方案。
