### 什么是闪电节点？



让我们回到基础知识：在定义什么是节点之前，我们需要了解什么是 Lightning Network。它是建立在 Bitcoin 基础上的顶层协议，旨在实现 BTC 的[离链](https://planb.academy/resources/glossary/offchain)交易，这种交易速度快（近乎即时完成），而且成本低廉。"离链 "意味着在闪电上进行的交易不会出现在主 Bitcoin [区块链](https://planb.academy/resources/glossary/blockchain)上。闪电 "也是对 Bitcoin 使用量增加和[链上](https://planb.academy/resources/glossary/onchain)拥堵的部分回应，链上拥堵引发了对系统[可扩展性](https://planb.academy/resources/glossary/scalability)的担忧。



为了运行，"闪电 "依靠在参与者之间开通[支付通道](https://planb.academy/resources/glossary/payment-channel)，在这些通道内，交易几乎可以瞬间完成，通常只需支付极少的费用，而无需在 Bitcoin 区块链上逐一注册。这些通道可以保持开放很长时间，只有在打开和关闭时才需要进行链上交易。



[闪电节点](https://planb.academy/resources/glossary/lightning-node)是闪电网络的参与者，负责与其他节点打开通道并进行支付。具体而言，"闪电 "节点是在计算机上运行的、执行Lightning Network协议的软件。例如 LND、Core Lightning 或 Eclair。该软件的主要作用是




- 连接到 [Bitcoin 节点](https://planb.academy/resources/glossary/full-node)，从主区块链中获取信息；
- 创建并管理与其他节点的双向支付渠道；
- 与整个 Lightning 网络交换信息。



![Image](assets/fr/001.webp)



### 节点与闪电 Wallet：重要区别



在 Bitcoin（链上）上，"*[wallet](https://planb.academy/resources/glossary/wallet)*"指的是管理[私钥](https://planb.academy/resources/glossary/private-key)、根据 [UTXO](https://planb.academy/resources/glossary/utxo) 计算余额和建立交易的软件。这个 wallet 可能基于你自己的 Bitcoin 节点，也可能基于别人的节点，但如今，节点和链上 wallet 的作用已泾渭分明。



在 "闪电 "系统中，要重复使用这类词汇而不造成混淆就比较困难了。术语 "*Lightning wallet*"相当含糊，因为在现实中，除非基于节点，否则不存在真正意义上的自我监护 Lightning wallet。因此，只有两种情况是可能的：



- 要拥有一个真正的闪电节点（即非托管节点）：您正在使用的软件（例如 Phoenix 这样的移动应用程序或 Umbrel 上的 LND 实例）实际上正在运行一个节点，而您实际上持有取回比特币的密钥。在这种情况下，你的 "*闪电 wallet*"实际上只是闪电节点上的一个用户界面，无论是嵌入式节点还是远程节点。



- 使用托管服务：您使用的应用程序在 Lightning 上显示 [sats](https://planb.academy/resources/glossary/satoshi-sat) 的余额，但在后台，资金在提供商的节点（如 Wallet of Satoshi）上。您既没有密钥，也没有渠道控制权。你的余额只是公司数据库中的一个会计分录。这相当于把你的比特币留在交易所平台上，并承担所有相关风险。在这种情况下，你的 "*闪电 wallet*"只是一个由运营商管理的账户，而运营商则运行着一个真正的闪电节点。



因此，"闪电 "没有中间地带：要么你有一个节点（即使是嵌入式节点），所以你是自我监管的；要么你没有，所以一家公司拥有你的 sats。但正如我们在后续章节中所看到的，这两种用途有时很难区分。例如，Phoenix 是一个嵌入了真正的 "闪电 "节点的移动应用程序，但用户并不一定意识到这一点，因为其操作的全部复杂性几乎完全被隐藏起来了。



### 提醒大家注意 Lightning Network 的工作原理



在本节中，我将简要介绍 "闪电 "的工作原理。如果您想了解更深入的理论概念，我再次邀请您参加专门的 LNP 201 课程。



#### 支付渠道：打开、更新和关闭



闪电网络的核心是双向支付通道。通道可以打开（即创建），随着闪电交易的发生而更新，最后关闭。从链上的角度来看，通道只不过是一个 2/2 [多重签名](https://planb.academy/resources/glossary/multisig)[输出](https://planb.academy/resources/glossary/output)。



![Image](assets/fr/002.webp)



从 "闪电 "的角度来看，这是一个由两个参与者瓜分[流动资金](https://planb.academy/resources/glossary/liquidity-lightning)的支付渠道。



![Image](assets/fr/003.webp)





- 开设频道：**



两个节点决定打开一个通道。其中一个节点将比特币投入一个名为 *funding transaction* 的链上交易。该交易基于 2 对 2 多[签名](https://planb.academy/resources/glossary/digital-signature)[脚本](https://planb.academy/resources/glossary/script)创建输出，这意味着在 Bitcoin 上花费这些资金需要通道中两个节点的签名。在发布该交易之前，提供资金的一方会要求另一方签署一个*提款交易*，该交易不在链上发布，但可以在出现问题时收回资金。



![Image](assets/fr/004.webp)





- Commitment 交易：**



通道的状态（即 A 和 B 之间 sats 的分配）由 *[commitment transaction](https://planb.academy/resources/glossary/commitment-transaction)* 表示，两个节点都知道，但不会立即在区块链上广播。该交易描述了如何根据 Lightning 上的支付在链上重新分配通道资金。



每支付一次 "闪电"，两个节点就会签署一个新的状态，取代之前的状态。旧状态通过撤销密钥机制被撤销：如果其中一个参与者试图广播旧状态，另一个参与者可以收回全部资金作为惩罚。



这里最重要的一点是，节点之间始终保持着一个签名的 Bitcoin 交易（不在链上广播），这样就能根据 Lightning Network 上的支付情况重新分配彼此的 sats。



![Image](assets/fr/005.webp)





- 渠道关闭：**



当双方就通道的最终状态达成一致时，通道可以通过合作关闭的方式干净利落地关闭；如果其中一个参与者不再合作或无法联系，通道也可以单方面关闭（强制关闭）。在所有情况下，关闭都采用链上交易的形式，花费 2/2 输出，并根据通道的最后有效状态在参与者之间分配资金。



![Image](assets/fr/006.webp)



因此，"闪电 "作为锚定在 Bitcoin 上的二级层发挥作用：只有某些事件（通道的打开和关闭）才会出现在主区块链上。中间支付仍在链外进行。



在进一步了解如何管理 "闪电 "频道之前，这里有两个基本概念：




- Liquidity*：通道一侧可用的 sats 数量；
- [容量](https://planb.academy/resources/glossary/lightning-channel-capacity)*：它是 2/2 多重配对输出中锁定的总量，即通道两侧流动性的总和。



#### 渠道和流动性网络



通道不仅用于两个节点之间的支付：它还是全球互连通道网络的一部分。您的节点可以通过自己的通道为其他用户付款，您也可以向没有直接通道的 "闪电 "节点发送 sats，只要您的两个节点之间能找到有效路径。



通过[流言协议](https://planb.academy/resources/glossary/gossip)，每个节点都知道这个网络的地图：存在哪些信道，哪些节点通过双向信道连接，以及哪些容量已发布。要向没有直接信道的收款人发送付款，你的节点会计算一条由若干跳组成的路径：你的节点 → X 节点 → Y 节点 → 收款人节点。在每一跳，付款都要经过一个通道，该通道在付款方向必须有足够的流动性。



![Image](assets/fr/007.webp)



因此，通道的流动性并不是对称的：一边可能负荷很重，而另一边几乎是空的。管理这种流动性，即了解 sats 的位置和流动方向，是运营 Lightning 节点最重要的方面之一。我们将在后面的实践章节中详细介绍。



#### HTLC：在不被抢劫的情况下运送付款



为了使支付能够在无需信任的情况下通过中间节点，闪电公司使用了名为*[HTLC](https://planb.academy/resources/glossary/htlc)*（*Hashed Time-Locked Contracts*）的[智能合约](https://planb.academy/resources/glossary/smart-contract)。简单地说，HTLC 以泄露秘密作为资金转移的条件，并加入了时间限制，以便在交易失败时保护发送方。因此，每笔付款都需要提交一个预图像（其[哈希值](https://planb.academy/resources/glossary/hash-function)与约定值相对应的秘密）。如果最终收款人提供了这个预图像，他或她就可以要求获得资金，这反过来又使每个中间节点能够收回自己的资金。



![Image](assets/fr/008.webp)



关于 HTLC 如何工作的技术细节我就不多说了，因为它们对本课程的目的并不重要。您可以在 LNP 201 理论课程中找到深入的解释。请记住，HTLC 支持原子交换：要么转账完成，没有人在路由过程中受到伤害；要么转账失败，每个参与者都要收回自己的初始资金。这两者之间没有任何区别。



### 闪电节点的主要实现方式



与 Bitcoin 一样，"闪电 "协议也有多种实现方式。许多独立团队都在开发自己的版本，由于它们遵循相同的规范（[BOLT](https://planb.academy/resources/glossary/bolt)），因此都具有互操作性。以下是目前使用的主要实现方式。



#### LND (*Lightning Network Daemon*)



LND 是 Lightning 协议的完整实现，使用 Go 语言编写，由 Lightning Labs 开发。



![Image](assets/fr/009.webp)



#### 核心闪电 (*CLN*)



Core Lightning（前身为 *C-Lightning*）是 Blockstream 开发的实施方案。它使用 C 语言编写，部分组件使用 Rust 语言。



![Image](assets/fr/010.webp)



#### Eclair



Eclair 是由法国公司 ACINQ 开发的 Scala 语言实现。ACINQ 使用 Eclair 运行闪电网络中最重要的路由节点之一，并将该实现作为其自身产品（如 Phoenix 应用程序）的软件基础。



![Image](assets/fr/011.webp)



#### LDK（*闪电开发工具包）



LDK（*闪电开发工具包*）是一个Rust开发工具包，由Spiral（Block，前Square公司）维护。它不是像 LND 或 CLN 那样的即用型 daemon，而是一个库，供希望将 Lightning 直接集成到其应用程序中的开发人员使用。



![Image](assets/fr/012.webp)



在本 LNP202 课程中，我们将主要介绍 LND：在面向私人客户的交钥匙解决方案（如 Umbrel）中最常用的实现方式。



关于 "闪电 "如何工作的简要介绍就到此为止。再次重申，如果您有任何不理解的概念，或者想在实践之前深入了解一下，Fanis Michalakis 的课程是这方面的重要参考资料：



https://planb.academy/courses/34bd43ef-6683-4a5c-b239-7cb1e40a4aeb


## 为什么要运行自己的 "闪电 "节点？


<chapterId>421db24e-511c-41ed-ad68-69b0662042ea</chapterId>



回答这个问题非常简单，因为它是一个反问句：如果没有自己的节点，你就不再真正使用 Lightning，而只是在公司的基础设施上使用 Lightning 的假象。



使用闪电托管 wallet 意味着比特币在技术上属于运营节点的公司。你不持有私钥，也不控制通道。您的 wallet 余额只是服务提供商数据库中的一行。对于初学者来说，这种情况当然非常方便，用户体验通常也很流畅，但根本的问题是：如果你最终放弃了 Bitcoin 和 Lightning 与传统货币和银行不同的地方，那么使用它们又有什么好处呢？



Bitcoin 的两大价值主张是货币主权（不再依赖中央机构发行和持有）和抵制审查（第三方无法阻止或过滤支付）。闪电平台上的托管系统与这两个目标背道而驰：你无法检查平台内部的货币供应，而且顾名思义，持有所有资金和所有渠道的运营商可以审查、延迟、优先或阻止你的支付。在这种情况下，我们可以理直气壮地问自己：**如果通过闪电使用比特币会重现与传统国家货币系统相同的信任和依赖模式，那它还有什么意义**。



> 我们需要的是一种基于加密证明而非信任的电子支付系统，让任何愿意交易的双方都能直接进行交易，而无需信任的第三方。
*Satoshi Nakamoto、Bitcoin 白皮书


抛开理念不谈，对你来说更具体的缺点如下。首先，你无法证实该公司确实持有与显示余额相对应的比特币。它可能以部分准备金运作、被黑客攻击、破产或干脆消失。在这种情况下，你只是另一个债权人，并不能真正保证你能拿回你的钱。



其次，公司面临监管风险：禁令、冻结资金、要求阻止用户或交易、加强监控，甚至在某些司法管辖区直接禁止活动。服务提供商受到的每项限制都会机械地反映到您身上。



在保密性方面，情况也好不到哪里去。托管运营商可以看到您所有的资金流动：金额、频率、收款人、余额、消费习惯。这些信息与应用程序提供的信息以及 Bitcoin 上可能存在的基础链分析相结合，可以提供非常精确的金融活动概况。同样，这与 Bitcoin 减少财务监控的目标相去甚远。



好消息是，如今，操作自己的 "闪电 "节点不再是技术专家的专利，2010 年代后期可能还是如此。家庭用户可以使用相对简单的解决方案，我们将在下一章详细介绍。




## 为工作选择正确的解决方案


<chapterId>615870e3-741d-4ec1-875d-a483e70f39d4</chapterId>



如今，您有可能在自我监护的同时，获得与 "闪电 "监护型 wallet 非常接近的用户体验。本章旨在帮助您选择最适合自己的路径。



### 方案 1：不直接使用 "闪电



第一种解决方案是不直接使用Lightning，而是使用嵌入[原子交换](https://planb.academy/resources/glossary/atomic-swap)的Bitcoin或[Liquid](https://planb.academy/resources/glossary/liquid-network) wallet。例如，Aqua 或 BULL Wallet 应用程序就使用了这种方法，使您无需亲自操作 Lightning 节点即可支付 [Lightning 发票](https://planb.academy/resources/glossary/invoice-lightning)，同时保持自我监管。



原理很简单：您的资金留在 Bitcoin（on-chain 或 Liquid）中，您通过 wallet 访问这些资金，您在 wallet 中以传统方式持有密钥。当您扫描闪电发票时，您的 wallet 会向原子交换服务发起交易（on-chain 或 Liquid）。然后，该服务通过自己的节点，使用您提供的 on-chain 或 Liquid 比特币管理闪电付款。因此，您无需自己处理任何闪电通道，但仍可无缝结算闪电发票。



![Image](assets/fr/013.webp)



与传统的 "闪电 "托管 wallet 相比，这种方法的主要优势在于，您始终对您的资金拥有 100% 的所有权。比特币在你的链上或 Liquid wallet，有你自己的[记忆词组](https://planb.academy/resources/glossary/seed)。即使在交换过程中，您仍然拥有您的资金，因为交换是原子式的。它依靠一种加密机制，确保只有两种可能的结果：要么交换完全成功，要么交换失败，服务无法占用你的资金。



大多数提供此类功能的组合都依赖 [Boltz](https://boltz.exchange/)来实现交换的技术部分。



这种解决方案在保密性方面也具有令人感兴趣的优势，尤其是与 Liquid 结合使用时。对于初学者来说，它的设置和保存也非常简单：一个经典的记忆短语，没有通道，无需平衡流动性......



另一方面，这种方法也有其局限性。首先，它不是不可替代的：你依赖于交换服务的可用性和善意。如果它不再愿意处理你的账户，或者停止运营，你就不能再通过它支付闪电发票。此外还有不小的费用：您既要支付链上[交易费](https://planb.academy/resources/glossary/transaction-fees)或 Liquid 交易费，还要支付交换服务佣金。此外，如果链上费用急剧上涨，使用 Lightning 的费用也会变得非常昂贵。



因此，偶尔使用还可以接受，但对于非常活跃的 "闪电 "用户来说，最好使用真正的 "闪电 "节点。



### 方案 2：板载闪电节点



第二类解决方案基于直接嵌入移动应用程序的 "闪电 "节点。Phoenix Wallet 是这种模式的先驱，至今仍是基准。如今，其他项目也提供了类似的方法，如 Zeus（嵌入式模式）或 BitKit。



这个想法很简单：应用程序实际上运行一个 Lightning 节点，但所有复杂的操作都在后台自动处理。你有一个 "*Lightning wallet*"界面，上面有一个用于备份的助记词组，你可以看到余额并支付发票，但不能管理渠道、流动性或大多数参数。



![Image](assets/fr/014.webp)



这些解决方案始终是自我保管的。控制资金的密钥是 generated，存储在你的手机上，备份是通过 seed 或类似的机制。你并不只是在服务提供商那里持有一个账户，你实际上拥有锁定在属于你的通道中的比特币，而且不会被盗。



LN 机载节点具有众多优势：




- 极易安装和使用；
- 用户体验与托管 Lightning wallet 接近，但具有自我托管功能；
- 无需人工管理渠道或流动性；
- 相对简单的备份。



但这些嵌入式 wallet 也有很大的局限性。首先，在保密性方面，服务运营商（如 Phoenix 中的 ACINQ）对通过您节点的流量有相当详细的了解：数量、频率、收件人，即使这种情况一定会得到改善，尤其是随着 *Trampoline Nodes* 的逐步采用。其次，您在很大程度上依赖于该运营商作为您的主要 "闪电 "对等节点。如果 ACINQ 节点不可用（如 Phoenix），如果该公司面临监管压力或改变其业务模式，您的用户体验可能会严重下降，甚至受到影响。



最后，这种简单性是有代价的。嵌入式 LN 节点服务一般会对存款、取款或某些渠道管理操作收取特定费用。在我看来，就提供的服务而言，这种模式是合理的，但就密集使用而言，它可能比管理良好的传统闪电节点昂贵得多。



### 方案 3：经典的闪电节点



第三种解决方案是在专用服务器或设备上运行传统的 "闪电 "节点，我们将在本 LNP202 课程中对此进行深入探讨。



我所说的 "传统 "是指您在自己的 Bitcoin 节点上自行安装和配置 Lightning 实现（如 LND）。您可以选择您的对等方，打开您的通道，管理您的[入站和出站流动性](https://planb.academy/resources/glossary/inbound-capacity)，并设置您的路由收费政策。



就主权而言，这是最好的解决方案。你不再依赖于某个特定公司的渠道或支付：如果某个同行审查你或关闭某个渠道，你可以用不同的节点打开另一个渠道。如果某项服务消失了，您的 sats 仍会保留在您控制的渠道中，您可以在链上将其归还。您还可以优化您的长期成本：一旦您的渠道规模和管理正确，总体支付成本就会变得非常低。



就保密性而言，你显然会受到 "闪电 "模式的限制，但你不会把整个业务交给一个运营商。



相比之下，建立一个经典的 Lightning 节点显然要复杂得多：你必须安装、配置、维护、监控更新，了解渠道和收费政策的逻辑，管理渠道和现金流等等。配置错误、马虎的备份或粗心的管理更容易导致 sats 的丢失。节点还必须持续运行。



这正是我在本课程中建议你遵循的路径，我将陪伴你走过每一步，以限制风险并构建你的方法。

## 保存您的 "闪电 "节点


<chapterId>638fa75d-62af-4bf3-ab4a-b7d10ea75815</chapterId>



在向节点发送第一个 sats 之前，了解其备份工作原理和相关风险非常重要。与简单的Bitcoin链上投资组合不同，备份闪电节点相当复杂：错误的策略可能导致资金永久损失。在本章中，我们将了解真正需要备份的内容，以及 Umbrel 如何通过 LND 处理这一过程。



在本课程中，我们将使用 LND (*Lightning Network Daemon*) 实现。虽然其他实现的原理相似，但我要讲的恢复文件和程序是 LND 特有的。



### 在 "闪电 "节点上应该保存什么？



在 "闪电 "节点上，仅保存 seed 并希望一切恢复正常是不够的。有几种元素扮演着不同的角色，因此区分它们很重要。



#### seed (*aezeed*)



初始化 LND 时，会收到一个 24 个字符的 seed。这是 LND 特有的格式，称为 "*aezeed*"。它不是经典的 seed BIP39，尽管看起来很像。从这个 seed 中，LND 可以得到与闪电节点相关联的链上 wallet 的私钥，即在通道关闭后，您可以接收比特币或将比特币汇回的地址。



![Image](assets/fr/019.webp)



因此，该 seed 可用来重新创建与您的节点相关联的链上 wallet，并取回已在链上归还的资金（例如在通道关闭后）。另一方面，由于 seed 不包含有关渠道状态的必要信息，因此仅凭 seed 不足以恢复仍处于开放状态的闪电渠道。



#### 频道数据库 (`channel.db`)



LND 将通道的详细状态存储在名为 "channel.db "的数据库中。该数据库包含




- 打开频道列表；
- 您的同行及其标识符；
- 每个通道的最后一个 commitment transaction（定义通道中谁拥有什么的连续状态，并在出现问题时恢复链上资金）；
- 所需的信息，以惩罚试图广播旧报告的同行。



该数据库的问题在于它不断变化：每次付款、每次路由、每次打开或关闭通道都会修改其内容。这就是为什么传统的备份（如定期复制 `channel.db`）是危险的。如果您恢复了太旧版本的 `channel.db`，并用这种过时的状态重新组装节点，您的对等方可能会认为您在广播旧的通道状态。在这种情况下，协议会规定惩罚措施：你的对等方可以收回频道的全部资金，就像你试图作弊一样。



实际上，`channel.db` 并不是备份介质。它是节点的实时状态。使用它来恢复节点唯一有意义的情况是，您直接从刚发生故障的机器（如仍可读取的磁盘）上恢复该数据库。在这种情况下，您可以恢复最新的状态，并根据该数据库在另一台机器上重新启动 LND，因为旧机器已不再运行，所以可以放心地知道该状态尽可能是最新的。这种方法可以作为相关备份的另一种情况是双磁盘配置，从一个磁盘向另一个磁盘进行动态永久复制。不过，这种类型的设置实施起来比较复杂。



但是，定期复制 `channel.db`，并将其作为备份存储以备日后恢复，却是一个非常糟糕的主意：在使用它们的当天，就有可能惩罚自己并丢失所有的 sats。



#### 静态通道备份 (SCB)



为了实现灾难恢复，LND 引入了*静态频道备份* (SCB) 机制。这是一个特殊文件，通常称为 "channel.backup"，其中包含有关您的频道的静态信息：谁是您的同行、如何联系他们以及您的频道是什么。



该文件不包含详细的通道状态或更新历史。它既不允许您以原来的状态重新打开频道，也不允许您继续操作这个 "闪电 "节点。相反，它的作用是作为一个锚点，让你的同伴帮助你干净利落地关闭所有频道，从而接收你的 sats onchain，并通过 seed 获取密钥。因此，与每次付款或路由都会修改的 `channel.db` 文件不同，SCB 文件只在通道打开或关闭时才会更新。



通过 SCB 恢复时，流程如下：




- 您恢复了 seed (*aezeed*)，从而重新创建了与 "闪电 "节点相关的链上 wallet；
- 您向 LND 提供最近的 SCB；
- LND 使用 SCB 查找同行列表以及您与他们之间的频道；
- 它会联系每个对等节点，告诉它们你遭受了数据丢失，并要求它们 "强制关闭 "你与它们的通道，这样你就可以恢复链上共享。



这样做的目的是，你的同行在注意到你报告数据丢失后，会广播他们最后的 commitment transaction 并关闭力量通道。一旦这些交易得到确认，您的资金就会重新出现在您的链上投资组合中（与 seed 链接）。



不过，这种恢复机制并不完美。首先，它并不能真正恢复你的闪电节点，因为所有通道都会关闭。这样你就必须从头开始建立一个新的闪电节点。其次，它不能保证100%恢复，尽管在出现问题时，它确实大大增加了恢复链上余额的机会。事实上，这种恢复协议取决于同行的合作和可用性：如果其中一个同行离线、丢失了自己的数据或拒绝合作，您的资金可能会继续被冻结，甚至永久丢失。因此，重要的是不要在你的 "闪电 "节点上与无法连接的对等节点长时间保持通道打开。如果此时您的数据丢失，而该对等节点仍然无法连接，那么通过 SCB 恢复将是不可能的，您的资金将一直丢失，直到该对等节点重新上线（也许是永远）。



总而言之，LND 上良好的 "闪电 "备份战略有三大支柱：




- 您的 seed（*aezeed*），用于链上层；
- 可靠的 SCB 自动备份；
- 选择可靠的同行，并预防性地关闭那些经常联系不上的同行，从而实现良好的渠道管理。



### Umbrel 如何管理 LND 节点的备份？



Umbrel 以 SCB 为基础，为 LND 节点提供简化的备份机制。这样做的目的是为了省去您自己操作该文件的麻烦，尽可能实现备份和自动化。



在 Umbrel 上创建节点时，您会收到一个 seed，它具有双重作用：




- 它会导出与您的 "闪电 "节点相关联的 Bitcoin onchain wallet；
- 它用于生成用于远程 SCB 备份的备份标识符和加密密钥。



借助这一机制，Umbrel 会自动对 SCB 进行加密备份，并通过 Tor 将其存储在服务器上。由于密钥来自 seed，SCB 是加密存储的。因此，在数据丢失的情况下，您只需在同一台或另一台机器上重新创建一个 Bitcoin 和 Umbrel 上的 "闪电 "节点，然后输入您的 seed。然后，您就可以从 Umbrel 服务器上获取最新的 SCB 状态，解密后开始恢复资金。



这些备份在发送前由您的节点进行本地加密，从而保证了数据的机密性：Umbrel 无法读取 SCB 的内容。传输是通过 Tor 进行的，这可以防止您的 IP 地址泄露。此外，您的 Umbrel 还会在流量中添加噪音（随机填充和不定时发送的虚假备份），以防止服务器准确推断出您打开或关闭通道的时间。



这种方法的主要优点是大大简化了 "闪电 "节点的备份：只需在节点初始化时备份一次 seed。这确实会带来一些风险，因为这只是 SCB 的备份，但对于合理的数量来说，这是一个可以接受的折衷方案。



### 限制损失风险的最佳做法



即使有 Umbrel 备份，一些简单的良好做法也能大大降低丢失 sats 的风险：





- 监控同行的可用性：



如果一个重要通道经常与一个无法到达或不稳定的对等节点相关联，那么在节点仍在工作时干净利落地关闭该通道会更安全。预防性合作或强制关闭可在 SCB 恢复时消除潜在的麻烦源。





- 避免将过多的流动资金集中在未知的同行身上：



同行与您的渠道越大，其可靠性就越重要。选择严肃认真、连接良好和活跃的节点，这样将来通过 SCB 进行的任何恢复都能顺利进行。





- 用本地备份补充 Umbrel：

## Watchtower：作用和设置


<chapterId>e6c654dd-26c5-4e4d-8d11-a215bac37812</chapterId>



在 "闪电 "系统中，每个通道都基于一系列连续的状态，这些状态由未公布的 commitment transaction 表示。每次 "闪电 "付款或路由时，通道中的两个参与者都会建立一对新的 commitment transaction，反映通道中当前的资金分配情况。旧的 commitment transaction 将被淘汰。



如果其中一方发布了过时的状态，另一方有权对其进行制裁，并收回通道的全部资金。在本章中，我们将简要介绍这一机制的工作原理，然后介绍如何建立****监视塔****：一个保护你的 "闪电 "节点免遭可能的作弊企图的系统。



### 了解瞭望塔的工作原理


在任何时候，渠道中的每一方都有一个 commitment transaction，如果公布，他们就可以关闭渠道，收回自己的那部分资金。这个过程被称为*强制关闭*。但是，如果他们试图公布一个较早的 commitment transaction，对应于通道之前持有较多 sats 的状态，那么这笔交易将被视为作弊企图。在这种情况下，交易方可以使用与该旧状态相关的撤销密钥来恢复通道中的全部资金，而作弊者则会被时间锁暂时封锁。


这个系统意味着，发布旧状态（即试图作弊）是非常危险的：如果另一方在时间锁到期前在内存池或区块链上看到这笔交易，他们就可以使用撤销密钥收回所有资金。 **因此，你的闪电通道的安全性取决于你在时间锁规定的时间窗口内检测到作弊企图的能力**。



#### 为什么需要瞭望塔？



处罚机制只有在受害方有能力的情况下才能发挥作用：




- 监控每个新的 Bitcoin 块，查看是否已发布频道 commitment transaction；
- 确定该事务对应的是最后有效状态还是已撤销状态；
- 在状态被撤销的情况下，及时广播合法交易，使用撤销密钥在时间锁到期前收回所有资金。



![Image](assets/fr/023.webp)



在理想情况下，您的闪电节点全天候在线，同步并持续监控区块链。因此，它可以单独检测到作弊企图并做出反应。但实际上，个人闪电节点可能会关闭，特别是在长时间断电或互联网连接故障的情况下。



正是在这些停机期间，风险才会变得真实：如果一个不诚实的同行者在你的节点离线时发布了一个旧的状态，而你没有做出任何反应，时间锁就会耗尽，那么作弊行为就会生效。你将损失部分或全部资金。



引入 Watchtower 就是为了降低这种风险。Watchtower 是一种外部服务，它代表您监控区块链，扫描您的某个通道是否可能发布旧状态，并在必要时代表您自动广播处罚交易。因此，即使您的 "闪电 "节点长时间处于离线状态，只要您使用的 "瞭望塔 "还在运行，它就能通过监控任何作弊企图保护您的资金，并在检测到作弊企图时立即实施相应的惩罚。



#### 瞭望塔如何运作



瞭望塔的设计目的是最大限度地减少它所了解的有关通道的信息，同时在出现问题时提供采取行动的手段：




- 对于与对等节点的每个新通道状态，您的节点都会提前准备一个潜在的惩罚交易。如果该对等节点作弊，您可以通过该交易收回通道中的所有资金；
- 然后，您的节点会使用相应 commitment transaction 的 TXID（即作弊者试图欺诈时会使用的 TXID）对该惩罚交易进行加密。只要没有关闭，瞭望塔就无法解密此交易，因为它并不完全知道作弊交易的 TXID；
- 您的节点向监视塔发送一个数据包，其中包含加密的惩罚交易和潜在作弊交易的半个 TXID。



由于传输给瞭望塔的 TXID 不完整，它无法解密正义交易。不过，它可以监控区块链，寻找与其拥有的部分相匹配的 TXID。如果检测到这样的交易，它就会尝试使用该交易的完整 TXID 来解密您的惩罚交易。如果解密成功，它就会知道这是一次作弊尝试，并立即为您发布惩罚交易。



![Image](assets/fr/024.webp)



因此，瞭望塔无法看到您的通道细节：既看不到同行的身份，也看不到余额和交易结构。它只能看到加密数据包。它能推断出的唯一信息就是你们通道的更新速度，因为每个新状态都会收到一个数据包，但它无法知道数据包的内容。在作弊的情况下，它肯定会通过解密惩罚交易来发现通道信息，但至少您的 sats 会被保存下来。



这种机制基于一种折衷方案：你将发布预先签署的惩罚交易的能力委托给瞭望塔，但这种交易对瞭望塔来说是完全不透明的，直到发生了一些欺骗行为。瞭望塔既不能修改收款人，也不能挪用资金，因为它只掌握一笔已经签署的交易，其输出被冻结，对你有利。它也无法知道合法的强制或合作关闭通道的细节，因为 TXID 不匹配。另一方面，watchtower 仍然是最基本的可信第三方：您需要依靠它在线，并在需要时正确广播您的正义交易。



#### 成为瞭望塔

### 如何选择优秀的 "闪电 "同行？



打开通道时，需要选择一个对等体：另一个 "闪电 "节点，你的节点将通过通道与之直接连接。对等节点的选择非常重要，因为它将直接影响到以下方面：




- 让您的付款更容易找到方向；
- 随着时间的推移，您的渠道的可靠性；
- 您的路由成本。



世界上没有完美匹配的人，但有一些可靠的标准可以识别优秀的候选人。



#### 1.节点连接



好的对等节点是指与 "闪电 "网络连接良好的节点。这不仅意味着拥有大量通道（尽管这也是一个指标），更重要的是与其他可靠节点相连，并在网络图中占据足够中心的位置。



连接良好的节点可增加找到通往大多数支付目的地的有效路径的机会。它还能减少所需的中间节点数量，从而降低成本。



相反，向一个孤立或连接较弱的节点开通第一条通道可能会限制您的可能性：您可以向这个对等节点付款，但要向网络的其他节点付款就难上加难了。



#### 2.资本化和渠道能力



一个好的对等节点也是一个资本充足的节点。这体现在它的总频道容量（投入到所有频道的 sats 总和）和平均频道容量（拥有几个资本充足的频道往往比拥有许多小频道要好）。



如果节点的容量小得离谱，或者通道都很小，就无法路由大额支付，从而导致路由失败。



#### 3.支出政策



每个节点设置自己的路由费用（"基本费用 "和 "费用率"）。好的对等节点不会对战略通道收取过高的费用。对于第一个通道，最好选择收费适中的节点，以免影响自己的支付。



请记住，你自己的路由费用也会影响他人对你作为同行的看法：一个不断更改费用或强加荒谬成本的节点很少会受到赞赏。



#### 4.稳定性和资历



节点稳定性是一个非常重要的标准。一个好的节点有很高的正常运行时间（很少离线），能长期保持通道开放，不会无缘无故地不断打开/关闭通道。



仍然活跃的旧渠道是一个很好的信号：它们表明这种关系对同行来说是有利可图的，节点知道如何管理自己的资金，而且不会随时关闭，这样你就不必为关闭和重新打开支付链上费用。



反之，如果同行经常不在线，或很快关闭渠道，就会给你带来麻烦，并增加开设新渠道的成本。



即使有了这些标准，您也不可能第一次就做出完美的选择。这很正常：同类产品的真正质量是在使用中显现出来的。因此，重要的是：




- 监控通道活动（路由量、可用性等）；
- 关闭那些毫无用处或其同行经常不在线的渠道；
- 随着时间的推移，将你的资金重新分配给更好的同行。



这样做的目的不是每天打开和关闭通道（这样做的链上成本会很高），而是逐渐发展拓扑结构，以汇聚到一组可靠、连接良好且符合需求的对等设备上。

## 定义节点操作员配置文件


<chapterId>d3b2e163-50f6-4d1d-a5fc-8fd177dfac76</chapterId>



现在您的闪电节点已经启动并运行，下一步就是定义您的交易者配置文件。这是一个重要的选择，因为它决定了您的通道开放策略、您偏好的同行类型以及您管理流动性的方式。



在 "闪电 "平台上，要实现这一点，您需要正确方向的流动性。流出的流动性与您的支付能力相对应（sats 可在您的渠道上使用）。流入的流动性与您的接收能力相对应（sats 在您的同行一方可用）。换句话说，您的情况可以归结为一个简单的问题：大多数时候，您的 sats 是离开您的节点，还是进入您的节点？



### 消费者



这是绝大多数用户的特征。您使用您的节点进行 "闪电 "支付：购买商品和服务、支付账单、发送小费--总之，将 "闪电 "作为一种快速支付手段。另一方面，您从 "闪电 "中获得的很少，或者很少，例如一些捐款、朋友之间的退款或一些小额支付。



这种情况最容易管理，因为您的主要需求是能够支付。在实际操作中，这意味着你需要流出的流动资金。一旦您打开了一个或多个大小合适的通道，与稳定、连接良好的节点相连，您的对外支付就会机械地将流动性转移到通道的另一端。正是这种流动自然会产生合理数量的流入流动性。因此，即使您不希望定期收款，也可以不时收款，而无需实施复杂的策略。因此，您无需担心流入的流动性。



在本 LNP202 课程中，我们将重点介绍 "消费者 "节点操作员配置文件，因为几乎所有 Lightning 上的 Bitcoin 用户都使用这种配置文件。



### 零售商



在某种程度上，商人与消费者正好相反。在这里，主要目的不是支付，而是收款。接受 "闪电 "付款的企业、服务提供商、网店或销售点会收到很多付款，但从这个节点发出的付款相对较少。



这种情况要求更高，因为 Lightning 上的拒付可能意味着销售损失。因此，优先考虑的是入站流动性。如果您的节点没有足够的入站资金，您的客户即使有资金，也会看到他们的付款失败，原因很简单，因为没有正确的途径将流动资金送到您的节点。



商家面临的主要挑战还来自渠道的自然演变。如果你所做的只是接收，你这边的渠道就会逐渐填满。因此，您需要建立机制，以维持和更新流入的流动资金。



不过，还有一种更简单的情况：消费者/商户混合情况。如果您在 "闪电 "平台上收款，同时也在 "闪电 "平台上消费（业务支出、向供应商付款，甚至个人支出），那么您的出账自然会重新产生入账。由于资金流相互抵消，管理变得更加顺畅，您也就不需要诉诸仅为恢复入账流动性而设计的人为机制。



### 路由器



路由器是一个特定的配置文件。它不使用自己的 "闪电 "节点进行支付或收款，而是为他人的支付和收费提供路由。路由器的目标是成为 "闪电 "图谱中有用、可靠且具有经济竞争力的路由。



老实说，在 "闪电 "上做路由器比看上去要复杂得多，盈利也很难实现。首先，打开和关闭通道的链上成本很高。为了高效地进行路由，你经常需要调整拓扑结构、测试、关闭表现不佳的通道、开设新通道，并定期重新平衡流动性。其次，竞争非常激烈：大型、成熟的节点自然会占据很大的流量份额，如果不在规模较大的通道上投入大量资金，就很难站稳脚跟。



运行要求也很高。路由节点必须非常稳定、受到监控、有适当的备份和严格的管理。您必须监控通道性能、调整收费政策、保持平衡的流动性、管理同行并快速解决技术问题。这种程度的参与作为一种技术爱好或对基础设施的贡献可能会很有趣，但如果你的目标仅仅是以主权方式使用闪电，那么进入路由领域赚钱就很少有意义了。这就是本 LNP202 课程不深入介绍这一情况的原因。



### 在进一步行动之前，先明确自己的位置



如果您符合消费者的特征，您的首要任务就是能够通过简单的管理进行可靠的支付。如果您是商户，您的首要任务是顺利兑现，这就意味着您需要采取内部流动性获取战略。如果您正在考虑路由，则需要将其作为一项要求高、经济上不确定且耗时的活动来对待。



现在就定义这一特征将帮助您避免一个典型的陷阱：当您只是一个想要付费的用户时，却采用了为商家或路由器设计的渠道策略。

## 获得流入的流动资金


<chapterId>b740c656-a897-4d95-af4b-116b718447cd</chapterId>



正如您所看到的，在 "闪电 "平台上进行支付并不复杂。只需主动向其他节点打开通道，然后开始发送 sats。另一方面，在 "闪电 "上接收付款的流入流动资金则比较复杂，因为您要么需要其他节点为您打开通道，要么需要自己通过付款来转移流动资金。



如果您采用的是 "消费者 "模式，一般无需担心流入的流动资金。您对 "闪电 "节点的使用将主要以支付为导向，而不是以套现为导向。只需进行几次闪电支付，随着时间的推移，您自然会产生流入的流动资金。



另一方面，如果您是 "商户"，情况就会相反：您将主要负责收款，而很少进行付款。因此，您不能只依靠自己的支付来获得流动资金。因此，其他 "闪电 "节点有必要为您打开通道。但如何才能做到这一点呢？如何让其他运营商为您绑定资金？这正是本章要探讨的问题。



### 购买境内流动性



正如您所预料的，让别人采取行动的最有效方法就是付钱给他们。对于向您的 "闪电 "节点输入流动性而言，原理完全相同。让渠道进入节点的最简单方法就是支付服务费用和相关的资本绑定。



如果您是企业或零售商，这种方法意味着您可以快速获得可靠的渠道，无障碍地向客户收款。



购买境内流动性的方法有很多。我个人使用并推荐的是 Amboss 的 [Magma](https://magma.amboss.tech/) 平台。它非常容易使用，开辟渠道很快，费率一般也很合理。Magma 的工作方式就像一个市场，有制造者和接受者，但第二版大大简化了使用体验：只需创建一个请求，通过闪电支付价格，Magma 就会自动匹配最佳报价。经过六次链上确认后，您与流入流动性的通道就开始运行了。具体操作如下



访问 [Magma 网站](https://magma.amboss.tech/buy)，在 "购买频道 "部分。



![Image](assets/fr/052.webp)

进入网站后，填写购买流动性所需的信息。选择 "一次性 "进行一次性购买，然后输入您愿意为流入的流动性支付的金额。支付的金额越高，向您的节点开放的通道容量就越大。通道容量的估计值如下所示：这只是一个近似值，因为最终金额将取决于 Magma 找到的最佳报价，有时会更高，有时会更低。

### 合作入境流动资金



如果你不是交易者，但仍然需要一些流入的流动性（例如，在启动节点时，在你等待付款时，为你的节点充能），你并不一定要为此付费。我的首选解决方案之一是与其他同样需要流入流动性的节点运营商合作，为彼此打开 "闪电 "通道。这样，开通通道在为您带来流出流动性的同时，还能免费为您提供流入流动性（开通的链上费用除外）。



当然，你也可以直接与其他比特币玩家安排。不过，有一个平台专门用于这种循环开放：[Lightning Network +](https://lightningnetwork.plus/)。其原则是不在同一个人之间开辟两个渠道，而是建立至少有三个参与者，甚至更多的人参与的循环开放平台。



让我们举例说明 Lightning Network + 的工作原理：




- 一个名为 "A "的节点操作员发布了一则公告，称他希望与另外两人开通一个 100 万 sats 信道；
- 在有更多参与者加入之前，广告一直处于激活状态；
- 随后，`B`和`C`两个运算符加入`A`节点公告。三角形现在已经完成，可以开始开放阶段。
- 节点 `A` 向节点 `B` 开通一条通道；
- 节点 `B` 向节点 `C` 开通一条通道；
- 节点 `C` 向节点 `A` 开通一条通道。



操作结束时，每个节点都有 100 万 sats 的流出流动性和 100 万 sats 的流入流动性。该计划可扩展至四个或五个参与者。



当然，没有任何技术机制可以保证参与者会真正开通他们承诺创建的频道。因此，该平台建立了一套声誉系统，当运营商履行承诺时，就会获得正面评价。在最坏的情况下，如果你遇到不合作的人，你也不会损失任何金钱：你只是错过了一个流入流动性的机会。



这种解决方案特别适合 "消费者"，因为它允许您免费获得流入流动性，同时将您的节点与其他小型运营商连接起来。另一方面，如果您是交易商，这种方法一般不适用：每一坐流入的流动性都需要阻止一坐流出的流动性，而您对流入流动性的大量需求会占用过多资金。



要使用 Lightning Network +，您有两个选择：一是使用 Umbrel 中集成的应用程序，二是使用经典网站，并通过节点登录创建账户。我建议使用后者，因为集成的应用程序不能提供全部可用功能。

就这样，您现在就拥有了一个连接良好的 Lightning 节点，既有流入的流动资金，也有流出的流动资金。这样，您就可以在现实生活中使用您的 "闪电 "节点了。到目前为止，我们一直直接使用 Umbrel 上的接口，即 "闪电节点 "应用程序或 "ThunderHub "接口。这些工具可用于发送和接收付款，但显然没有针对日常 Lightning 付款进行优化。该界面专为在电脑上使用而设计，在智能手机上并不实用，而且还需要连接到同一网络才能正常运行（尽管在技术上可以通过 Tor 进行远程连接）。



实际上，作为比特币使用者，我们需要的是智能手机上的经典闪电 wallet 界面：通过二维码扫描发票的功能，以及支付和兑现 sats 的简单界面。这正是我们在本章和下一章要实现的功能。总体思路是在智能手机上安装一个移动 Lightning wallet 应用程序，可在任何地方（不仅是本地网络）使用，但在后台依靠自己的 Lightning 节点发送和接收付款。



### 连接移动客户的解决方案有哪些？



如今，无论是移动应用程序还是您的节点与该应用程序之间的连接类型，都有多种方法可以做到这一点。三种主要连接模式是




- 通过 ***Tor***；
- 通过 VPN ***Tailscale***；
- 通过 ***Nostr Wallet 连接***。



几年前，我曾通过 ***Tor*** 进行连接，但很快就停止了：失败的次数和通信延迟实在太多了。从理论上讲，它是可行的，但在实践中，用户体验是灾难性的。因此，我建议不要使用这种方法。



我当时采用的替代方案是使用 ***Tailscale*** VPN 来确保移动应用程序与节点之间的通信。这种解决方案效果非常好：即使在吞吐量较低的移动网络上，我的付款也总是顺利通过。这就是我将在本章首先介绍的使用 Zeus 应用程序的方法。



在下一章中，我们将介绍另一种最新的解决方案，它也非常有效： ***Nostr Wallet Connect***。这次，我们将使用 Alby Go 应用程序为您提供一个替代方案，不过如果您愿意，Zeus 也可以与 NWC 兼容。

### Nostr Wallet Connect 如何工作？



Nostr Wallet Connect 是一种标准化协议，允许应用程序或网站在远程 Lightning 节点上触发操作，而无需与该节点建立直接网络连接（无 API LND 暴露、无 VPN、无 `.onion` 服务......）。NWC定义了应用程序如何制定意图（如`pay_intece`）并接收结果。



它的工作原理非常简单。您可以通过扫描二维码或深度链接 "nostr+walletconnect: "来初始化会话。该字符串包含会话参数和授权密钥。然后，当应用程序要付款时，它会将请求序列化、加密，并将其作为一个事件发布到 Nostr 中继器上。节点读取中继器上的事件，对其解密，检查作者是否获得此会话授权，执行支付，然后返回加密响应（带预图像的成功或错误）。中继器只充当传输中介：它无法读取内容，但可以观察请求的时间和频率。



与通过 Tailscale 或 Tor 进行连接相比，NWC 的主要优势在于您的节点无法从外部直接到达。这大大简化了移动使用：你的节点不需要接受传入的连接，它只需要能与中继器通信即可。另一方面，您对 Nostr 中继站产生了功能依赖：如果它们不可用，体验就会下降。此外，即使信息是加密的，中继站也能观察到一定程度的活动元数据。



安全模式的不同也很重要。使用 Tailscale 或 Tor，您可以直接访问您的节点（通过 API 或 LND），并受到高度敏感机密的保护。这很强大，因为你可以管理一切，但这也是一个较低层次的攻击面。使用 NWC，访问更具应用性：您可以委托一个会话 token，该会话仅授权某些操作。

## 持久的 "闪电 "自动驾驶功能

LNP202 实践课程到此结束。现在，您已经掌握了以主权方式使用 Lightning Network 所需的基础知识：您了解了节点的真正作用、不同方法之间的权衡，并在 Umbrel 上建立了 LND 实例，同时制定了一致的备份和保护策略。您还开通了第一个渠道，学会了如何管理流动资金，使您的日常支付更加可靠。



从运行角度来看，您的节点现在应该进入维护节奏。最主要的是对其进行监控（正常运行时间、同步、通道状态、支付失败等），在稳定版本可用时应用 Umbrel 提供的更新，并定期检查备份和监视塔配置是否仍然有效。
