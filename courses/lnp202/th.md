# ก้าวแรกของคุณสู่ความเป็นอิสระบน Lightning



คุณได้ซื้อ sats เครื่องแรกของคุณแล้ว, รักษาเงินทุนด้วยตนเองและติดตั้งโหนด Bitcoin เพื่อให้มีอำนาจอธิปไตยในการใช้งาน on-chain ของคุณ ขั้นตอนต่อไปคือการได้รับอำนาจอิสระเดียวกันบน Lightning: นั่นคือเป้าหมายของหลักสูตรนี้



LNP202 มุ่งเป้าไปที่ผู้ใช้ระดับกลาง และแนะนำคุณทีละขั้นตอนผ่านการปรับใช้โหนด Lightning แรกของคุณ โดยไม่ต้องมีทักษะทางเทคนิคขั้นสูง



คุณจะได้เรียนรู้วิธีการติดตั้ง LND บน Umbrel, เปิดและจัดการช่องทางของคุณ, ควบคุมโหนดของคุณและเชื่อมต่อ wallet บนมือถือ, เพื่อให้คุณสามารถเพลิดเพลินกับประสบการณ์ที่เทียบเท่ากับ wallet แบบดูแลทรัพย์สิน, ในขณะที่ยังคงควบคุมเงินทุนของคุณได้อย่างเต็มที่

## ภาพรวมของหลักสูตร

LNP202 เป็นหลักสูตรเชิงปฏิบัติที่ออกแบบมาเพื่อให้คุณสามารถใช้งาน Lightning ได้ด้วยตนเองโดยการดำเนินการโหนดของคุณเอง วัตถุประสงค์นั้นง่าย: เริ่มต้นด้วยโหนด Bitcoin ที่มีอยู่แล้ว ติดตั้ง LND บน Umbrel รักษาความปลอดภัยให้เหมาะสม เปิดและจัดการช่องทางแรกของคุณ จากนั้นใช้โหนดของคุณในชีวิตประจำวันจาก wallet บนมือถือ หลักสูตรนี้ถือว่าคุณได้เรียน BTC 202 แล้ว เนื่องจากฉันถือว่าโหนด Bitcoin ของคุณบน Umbrel ได้ถูกติดตั้งและซิงโครไนซ์แล้ว เราจะไม่ย้อนกลับไปที่วิธีการตั้งค่าโหนด Bitcoin ที่นี่

ในส่วนแรกของหลักสูตร LNP202 นี้ เราจะมาดูกันว่าโหนด Lightning จริงๆ แล้วคืออะไร มันแตกต่างจาก wallet อย่างไร และทำไมการดำเนินการโหนดส่วนตัวจึงเป็นวิธีเดียวที่จะใช้ Lightning โดยไม่ต้องมอบหมาย sats ของคุณให้กับบุคคลที่สามที่เชื่อถือได้ ส่วนนี้จะจบลงด้วยการเลือกเชิงกลยุทธ์: โซลูชันใดที่เหมาะกับคุณ ตั้งแต่วิธีการที่ง่ายที่สุดไปจนถึงโหนด Lightning แบบคลาสสิก ซึ่งเป็นสิ่งที่เราจะนำไปใช้ในหลักสูตรนี้



ในส่วนที่ 2 ของหลักสูตร คุณจะติดตั้ง LND บน Umbrel จากนั้นวางองค์ประกอบที่ป้องกันความผิดพลาดที่มีค่าใช้จ่ายสูงที่สุด: กลยุทธ์การสำรองข้อมูลที่สมจริงและการป้องกันการโกงผ่านหอคอยเฝ้าระวัง เมื่อพื้นฐานเหล่านี้ถูกจัดวางแล้ว คุณจะเปิดช่องทางแรกของคุณ เพื่อให้คุณสามารถเริ่มชำระเงินบน Lightning ด้วยโครงสร้างพื้นฐานของคุณเอง



ดังนั้นคุณจะเห็นว่า: ในหลักสูตร LNP202 นี้ เราจะตั้งค่าโหนด Lightning ในโหมด plug-and-play ผ่านทาง Umbrel แทนที่จะใช้วิธีการแบบบรรทัดคำสั่งแบบคลาสสิก เพื่อให้เหมาะสมกับผู้ใช้ระดับกลาง หากคุณต้องการการติดตั้งแบบบรรทัดคำสั่ง ฉันขอแนะนำให้คุณทำตามบทแนะนำด้านล่าง หลักสูตร Lightning อื่นๆ ที่เน้นการใช้บรรทัดคำสั่งขั้นสูงก็กำลังอยู่ในระหว่างการเตรียมการเช่นกัน



https://planb.academy/tutorials/node/lightning-network/lightning-network-daemon-linux-59d777e9-72c8-4b32-8c50-e86cdae8f2f9

ส่วนที่ 3 จะนำคุณจากโหนดที่ใช้งานได้ไปยังโหนดที่คุณรู้วิธีควบคุม คุณจะเริ่มต้นด้วยการกำหนดโปรไฟล์ผู้ดำเนินการโหนด Lightning ของคุณ (ผู้บริโภค, ผู้ค้า หรือเราเตอร์) จากนั้นทำความเข้าใจกับชุดซอฟต์แวร์การจัดการที่สมบูรณ์ เพื่อให้สามารถติดตามช่องทางของคุณและดำเนินการอย่างสะอาดบนโทโพโลยีของคุณ สุดท้ายคุณจะจัดการกับจุดสำคัญของ Lightning: วิธีการรับสภาพคล่องขาเข้า ไม่ว่าจะผ่านทางโซลูชันที่ต้องชำระเงินหรือแบบร่วมมือ



ส่วนที่ 4 จะเน้นการใช้งานในชีวิตประจำวัน คุณจะตั้งค่าการเชื่อมต่อระหว่างโหนดของคุณกับลูกค้ามือถือ เพื่อให้คุณสามารถชำระเงินและเก็บเงินได้ง่ายๆ จากสมาร์ทโฟนของคุณ โดยไม่ต้องสละการควบคุมตนเอง เราจะดูวิธีการเครือข่ายผ่าน *Tailscale* ก่อน จากนั้นจะดูวิธีการโปรโตคอลผ่าน *Nostr Wallet Connect* พร้อมข้อดีและข้อเสียของแต่ละวิธี หลักสูตรจะสรุปด้วยบทสุดท้ายที่จะให้คุณมีนิสัยที่ถูกต้องในการรักษาการควบคุมตนเอง ทั้งในด้านการปฏิบัติการและการสอน



หากคุณทำตามหลักสูตร LNP202 นี้ตามลำดับที่ถูกต้อง เมื่อสิ้นสุดหลักสูตรคุณจะมีการตั้งค่าที่สมบูรณ์สำหรับโหนด Lightning ของคุณ ซึ่งสามารถใช้งานได้ในชีวิตประจำวันและที่สำคัญที่สุดคืออยู่ภายใต้การควบคุมของคุณ




## การทำความเข้าใจโหนด Lightning


<chapterId>8275dfd8-7a72-48cc-bf7f-bc2a46063003</chapterId>



ก่อนที่จะเปิดตัวโหนดของคุณเอง บทนี้จะทบทวนทฤษฎีพื้นฐานเบื้องหลัง [Lightning Network](https://planb.academy/resources/glossary/lightning-network) อย่างสั้นๆ การเข้าใจกลไกที่เกี่ยวข้องนั้นสำคัญมาก เพราะจะช่วยให้คุณระบุความเสี่ยงและนำแนวปฏิบัติที่ดีมาใช้เพื่อลดความเสี่ยงเหล่านั้น อย่างไรก็ตาม ฉันจะไม่ลงรายละเอียดในที่นี้ เนื่องจากไม่ใช่วัตถุประสงค์หลักของหลักสูตรนี้ หากคุณต้องการเจาะลึกในหัวข้อนี้ ฉันขอแนะนำอย่างยิ่งให้คุณศึกษาหลักสูตร LNP 201 ของ Fanis Michalakis ซึ่งเป็นแหล่งอ้างอิงในสาขานี้:

### Lightning node คืออะไร?



กลับไปที่พื้นฐาน: ก่อนที่จะกำหนดว่าโหนดคืออะไร เราจำเป็นต้องเข้าใจว่า Lightning Network คืออะไร มันเป็นโปรโตคอลชั้นบนสุดที่สร้างขึ้นบน Bitcoin ออกแบบมาเพื่อให้สามารถทำธุรกรรม BTC [นอกเชน](https://planb.academy/resources/glossary/offchain)ที่รวดเร็ว (ด้วยความแน่นอนเกือบจะทันที) และโดยทั่วไปมีค่าใช้จ่ายต่ำ "นอกเชน" หมายความว่าธุรกรรมที่ดำเนินการบน Lightning ไม่ได้มีเจตนาที่จะปรากฏบน[บล็อกเชน](https://planb.academy/resources/glossary/blockchain)หลักของ Bitcoin Lightning ยังเป็นการตอบสนองบางส่วนต่อการใช้ Bitcoin ที่เพิ่มขึ้นและความแออัด[บนเชน](https://planb.academy/resources/glossary/onchain) ซึ่งกำลังสร้างความกังวลเกี่ยวกับ[ความสามารถในการขยายตัว](https://planb.academy/resources/glossary/scalability)ของระบบ



ในการดำเนินการ Lightning อาศัยการเปิด[ช่องทางการชำระเงิน](https://planb.academy/resources/glossary/payment-channel)ระหว่างผู้เข้าร่วม ซึ่งภายในช่องทางเหล่านี้สามารถทำธุรกรรมได้เกือบจะทันที มักมีค่าธรรมเนียมต่ำ โดยไม่จำเป็นต้องลงทะเบียนทีละรายการบนบล็อกเชน Bitcoin ช่องทางเหล่านี้สามารถเปิดค้างไว้นานมาก โดยต้องการธุรกรรมบนเชนเฉพาะเมื่อเปิดและปิดเท่านั้น



[โหนด Lightning](https://planb.academy/resources/glossary/lightning-node) เป็นผู้เข้าร่วมในเครือข่าย Lightning ที่เปิดช่องทางและทำการชำระเงินกับโหนดอื่น ๆ ในแง่ที่เป็นรูปธรรม โหนด Lightning คือซอฟต์แวร์ที่ทำงานบนคอมพิวเตอร์และดำเนินการตามโปรโตคอล Lightning Network ตัวอย่างเช่น LND, Core Lightning หรือ Eclair บทบาทหลักของซอฟต์แวร์นี้คือการ:




- เชื่อมต่อกับ[โหนด Bitcoin](https://planb.academy/resources/glossary/full-node) เพื่อรับข้อมูลจากบล็อกเชนหลัก;
- สร้างและจัดการช่องทางการชำระเงินแบบสองทิศทางกับโหนดอื่น ๆ;
- แลกเปลี่ยนข้อความกับเครือข่าย Lightning ทั้งหมด

### Node vs. Lightning Wallet: an important distinction



บน Bitcoin (onchain), "*[wallet](https://planb.academy/resources/glossary/wallet)*" หมายถึงซอฟต์แวร์ที่จัดการ[คีย์ส่วนตัว](https://planb.academy/resources/glossary/private-key)ของคุณ คำนวณยอดคงเหลือจาก [UTXO](https://planb.academy/resources/glossary/utxo) ของคุณ และสร้างธุรกรรมของคุณ wallet นี้อาจอิงตามโหนด Bitcoin ของคุณเองหรือของคนอื่น แต่ในปัจจุบัน บทบาทของโหนดและบทบาทของ onchain wallet นั้นแตกต่างกันอย่างชัดเจน



บน Lightning การใช้คำศัพท์ประเภทนี้ซ้ำจะยากขึ้นโดยไม่สร้างความสับสน คำว่า "*Lightning wallet*" ค่อนข้างคลุมเครือ เพราะในความเป็นจริงไม่มีสิ่งที่เรียกว่า Lightning wallet ที่ดูแลตัวเองได้อย่างแท้จริง เว้นแต่ว่ามันจะอิงจากโหนด ดังนั้นจึงมีเพียงสองสถานการณ์ที่เป็นไปได้:



- ในการมีโหนด Lightning จริง (เช่น ไม่ใช่แบบรับฝาก): ซอฟต์แวร์ที่คุณใช้งาน (เช่น แอปมือถืออย่าง Phoenix หรืออินสแตนซ์ LND บน Umbrel) กำลังรันโหนดจริง ๆ และคุณถือกุญแจในการดึงบิตคอยน์ของคุณเอง ในกรณีนี้ "*Lightning wallet*" ของคุณเป็นเพียงแค่ส่วนติดต่อผู้ใช้บนโหนด Lightning ไม่ว่าจะเป็นแบบฝังหรือแบบระยะไกล



- ในการใช้บริการดูแลทรัพย์สิน: คุณใช้แอปพลิเคชันที่แสดงยอดคงเหลือใน [sats](https://planb.academy/resources/glossary/satoshi-sat) บน Lightning แต่ในเบื้องหลัง เงินทุนอยู่บนโหนดของผู้ให้บริการ (เช่น Wallet of Satoshi) คุณไม่มีทั้งกุญแจหรือการควบคุมช่องทาง ยอดคงเหลือของคุณเป็นเพียงรายการบัญชีในฐานข้อมูลของบริษัท มันเปรียบได้กับการทิ้งบิตคอยน์ของคุณไว้บนแพลตฟอร์มแลกเปลี่ยน พร้อมกับความเสี่ยงที่เกี่ยวข้องทั้งหมด ในกรณีนี้ "*Lightning wallet*" ของคุณเป็นเพียงการเข้าถึงบัญชีที่จัดการโดยผู้ดำเนินการซึ่งในทางกลับกันจะรันโหนด Lightning จริง



ดังนั้นจึงไม่มีตัวเลือกตรงกลางสำหรับ Lightning: คุณมีโหนด (แม้แต่โหนดฝังตัว) ดังนั้นคุณจึงควบคุมด้วยตนเอง หรือคุณไม่มี ดังนั้นบริษัทจึงเป็นเจ้าของ sats ของคุณ แต่ตามที่เราจะเห็นในบทต่อไป การใช้งานทั้งสองแบบนี้บางครั้งอาจแยกแยะได้ยาก ตัวอย่างเช่น Phoenix เป็นแอปพลิเคชันมือถือที่ฝังโหนด Lightning จริง แต่ผู้ใช้อาจไม่ทราบถึงมัน เนื่องจากความซับซ้อนทั้งหมดของการทำงานนั้นถูกซ่อนเกือบทั้งหมด



### การเตือนความจำเกี่ยวกับวิธีการทำงานของ Lightning Network



ในส่วนนี้ ฉันจะให้การทบทวนอย่างรวดเร็วเกี่ยวกับการทำงานของ Lightning อีกครั้ง หากคุณต้องการการนำเสนอเชิงลึกเกี่ยวกับแนวคิดทางทฤษฎี ฉันขอเชิญคุณดูที่หลักสูตร LNP 201 ที่จัดทำขึ้นโดยเฉพาะ



#### ช่องทางการชำระเงิน: เปิด อัปเดต และปิด



หัวใจสำคัญของเครือข่าย Lightning คือการใช้ช่องทางการชำระเงินแบบสองทิศทาง ช่องทางสามารถเปิด (เช่น สร้างขึ้น), อัปเดตเมื่อมีการทำธุรกรรม Lightning เกิดขึ้น, และสุดท้ายปิดได้ จากมุมมองของบล็อกเชน ช่องทางเป็นเพียง[เอาต์พุต](https://planb.academy/resources/glossary/output)[ลายเซ็นหลายตัว](https://planb.academy/resources/glossary/multisig) 2/2 เท่านั้น



![Image](assets/fr/002.webp)



จากมุมมองของ Lightning มันคือช่องทางการชำระเงินที่มี[สภาพคล่อง](https://planb.academy/resources/glossary/liquidity-lightning)แบ่งระหว่างผู้เข้าร่วมทั้งสองฝ่าย



![Image](assets/fr/003.webp)





- การเปิดช่องทาง:**



โหนดสองตัวตัดสินใจเปิดช่องทาง หนึ่งในนั้นให้คำมั่นบิตคอยน์ในธุรกรรมบนเชนที่เรียกว่า *ธุรกรรมการระดมทุน* ธุรกรรมนี้สร้างผลลัพธ์ตาม[สคริปต์](https://planb.academy/resources/glossary/script)มัลติซิกเนเจอร์ 2-of-2 ซึ่งหมายความว่าการใช้จ่ายเงินเหล่านี้บน Bitcoin ต้องการ[ลายเซ็น](https://planb.academy/resources/glossary/digital-signature)ของทั้งสองโหนดในช่องทาง ก่อนที่จะออกธุรกรรมนี้ ฝ่ายที่ให้เงินจะขอให้อีกฝ่ายเซ็น *ธุรกรรมการถอนเงิน* ซึ่งไม่ได้ออกบนเชน แต่ช่วยให้สามารถกู้คืนเงินได้ในกรณีที่เกิดปัญหา



![Image](assets/fr/004.webp)





- ธุรกรรม Commitment:**



สถานะของช่องทาง (เช่น การกระจายของ sats ระหว่าง A และ B) ถูกแสดงโดย *[commitment transaction](https://planb.academy/resources/glossary/commitment-transaction)* ซึ่งเป็นที่รู้จักของทั้งสองโหนดแต่ไม่ได้ถูกเผยแพร่บนบล็อกเชนทันที ธุรกรรมนี้อธิบายวิธีการแจกจ่ายเงินในช่องทางบนเชนใหม่ตามการชำระเงินที่ทำบน Lightning



ด้วยการชำระเงินผ่าน Lightning แต่ละครั้ง โหนดทั้งสองจะลงนามในสถานะใหม่ที่แทนที่สถานะก่อนหน้า สถานะเก่าจะถูกเพิกถอนด้วยกลไกกุญแจเพิกถอน: หากผู้เข้าร่วมคนใดคนหนึ่งพยายามเผยแพร่สถานะเก่า อีกฝ่ายสามารถกู้คืนจำนวนเงินทั้งหมดของกองทุนเป็นการลงโทษได้



แนวคิดสำคัญที่นี่คือจะมีธุรกรรม Bitcoin ที่ลงนามแล้วเสมอ ซึ่งไม่ได้ถูกเผยแพร่บนเชน แต่ถูกเก็บไว้โดยโหนด และช่วยให้สามารถกระจาย sats ของกันและกันได้ตามการชำระเงินที่ทำบน Lightning Network



![Image](assets/fr/005.webp)





- การปิดช่อง:**



ช่องทางสามารถปิดได้อย่างเรียบร้อยผ่านการปิดแบบร่วมมือกัน เมื่อทั้งสองฝ่ายตกลงในสถานะสุดท้ายของช่องทาง หรือปิดฝ่ายเดียว (การปิดแบบบังคับ) หากผู้เข้าร่วมคนใดคนหนึ่งหยุดให้ความร่วมมือหรือไม่สามารถติดต่อได้ ในทุกกรณี การปิดจะอยู่ในรูปแบบของธุรกรรมบนเชนที่ใช้จ่ายผลลัพธ์ 2/2 และแจกจ่ายเงินทุนระหว่างผู้เข้าร่วมตามสถานะสุดท้ายที่ถูกต้องของช่องทาง



![Image](assets/fr/006.webp)



ฟ้าผ่าจึงทำหน้าที่เป็นชั้นรองที่ยึดติดกับ Bitcoin: เฉพาะเหตุการณ์บางอย่าง (การเปิดและปิดช่องทาง) ที่ปรากฏบนบล็อกเชนหลัก การชำระเงินระหว่างกลางยังคงอยู่นอกเชน



ก่อนที่จะไปต่อ นี่คือสองแนวคิดสำคัญสำหรับการทำความเข้าใจวิธีการจัดการช่องทาง Lightning:




- Liquidity*: ปริมาณของ sats ที่มีอยู่ด้านหนึ่งของช่องทาง;
- *[ความจุ](https://planb.academy/resources/glossary/lightning-channel-capacity)*: คือจำนวนทั้งหมดที่ถูกล็อกไว้ในเอาต์พุต 2/2 multisig ซึ่งก็คือผลรวมของสภาพคล่องทั้งสองด้านของช่องทาง



#### เครือข่ายของช่องทางและสภาพคล่อง



ช่องทางไม่ได้มีไว้สำหรับการชำระเงินระหว่างสองโหนดเท่านั้น: มันเป็นส่วนหนึ่งของเครือข่ายทั่วโลกที่มีช่องทางเชื่อมต่อกัน โหนดของคุณสามารถส่งต่อการชำระเงินสำหรับผู้ใช้อื่นผ่านช่องทางของตัวเองได้ และคุณสามารถส่ง sats ไปยังโหนด Lightning ที่คุณไม่มีช่องทางตรงได้ ตราบใดที่สามารถหาวิถีที่ถูกต้องระหว่างโหนดทั้งสองของคุณได้



แต่ละโหนดรู้, ผ่านโปรโตคอล[กอสซิป](https://planb.academy/resources/glossary/gossip), แผนที่ของเครือข่ายนี้: ช่องทางใดบ้างที่มีอยู่, โหนดใดบ้างที่เชื่อมต่อด้วยช่องทางสองทิศทาง, และความจุใดบ้างที่ถูกเผยแพร่. เพื่อส่งการชำระเงินไปยังผู้รับโดยไม่มีช่องทางตรง, โหนดของคุณคำนวณเส้นทางที่ประกอบด้วยหลายฮอป: โหนดของคุณ → โหนด X → โหนด Y → โหนดผู้รับ. ในแต่ละฮอป, การชำระเงินจะผ่านช่องทางที่ต้องมีสภาพคล่องเพียงพอในทิศทางของการชำระเงิน.



![Image](assets/fr/007.webp)



สภาพคล่องของช่องทางจึงไม่สมมาตร: ด้านหนึ่งอาจมีการโหลดหนัก ในขณะที่อีกด้านเกือบจะว่างเปล่า การจัดการสภาพคล่องนี้ คือการรู้ว่า sats อยู่ที่ไหนและสามารถไหลไปในทิศทางใด เป็นหนึ่งในแง่มุมที่สำคัญที่สุดของการดำเนินงานโหนด Lightning เราจะพิจารณาในรายละเอียดเพิ่มเติมในบทปฏิบัติที่จะมาถึง



#### HTLC: การขนส่งเงินโดยไม่ถูกปล้น



เพื่อให้การชำระเงินผ่านโหนดกลางโดยไม่จำเป็นต้องมีความไว้วางใจ Lightning ใช้[สัญญาอัจฉริยะ](https://planb.academy/resources/glossary/smart-contract)ที่เรียกว่า *[HTLC](https://planb.academy/resources/glossary/htlc)* (*Hashed Time-Locked Contracts*) ในแง่ที่ง่าย HTLC ทำให้การโอนเงินมีเงื่อนไขตามการเปิดเผยความลับ และรวมข้อจำกัดด้านเวลาเพื่อปกป้องผู้ส่งในกรณีที่การทำธุรกรรมล้มเหลว การชำระเงินแต่ละครั้งจึงขึ้นอยู่กับการนำเสนอภาพก่อนหน้า (ความลับที่[แฮช](https://planb.academy/resources/glossary/hash-function)ตรงกับค่าที่ตกลงกัน) หากผู้รับปลายทางให้ภาพก่อนหน้านี้ เขาหรือเธอสามารถเรียกร้องเงินได้ ซึ่งจะทำให้โหนดกลางแต่ละโหนดสามารถกู้คืนเงินของตนเองได้



![Image](assets/fr/008.webp)



ฉันจะละเว้นรายละเอียดทางเทคนิคเกี่ยวกับวิธีการทำงานของ HTLC เพราะมันไม่จำเป็นสำหรับวัตถุประสงค์ของหลักสูตรนี้ คุณจะพบคำอธิบายเชิงลึกในหลักสูตรทฤษฎี LNP 201 เพียงจำไว้ว่าระบบ HTLC ช่วยให้เกิดการแลกเปลี่ยนเชิงอะตอม: ไม่ว่าการโอนจะเสร็จสมบูรณ์และไม่มีใครได้รับอันตรายในการกำหนดเส้นทาง หรือมันล้มเหลวและผู้เข้าร่วมแต่ละคนจะได้รับเงินทุนเริ่มต้นของตนคืน ไม่มีทางเลือกอื่นระหว่างนี้



### การใช้งานโหนด Lightning หลัก



เช่นเดียวกับ Bitcoin มีการใช้งานโปรโตคอล Lightning หลายแบบ ทีมงานอิสระหลายทีมกำลังพัฒนาเวอร์ชันของตนเอง ซึ่งทั้งหมดสามารถทำงานร่วมกันได้เนื่องจากปฏิบัติตามข้อกำหนดเดียวกัน ([BOLTs](https://planb.academy/resources/glossary/bolt)) นี่คือการใช้งานหลักที่ใช้ในปัจจุบัน



#### LND (*Lightning Network Daemon*)



LND เป็นการใช้งานโปรโตคอล Lightning อย่างสมบูรณ์ เขียนด้วยภาษา Go และพัฒนาโดย Lightning Labs.



![Image](assets/fr/009.webp)



#### Core Lightning (*CLN*)



Core Lightning (เดิมชื่อ *C-Lightning*) เป็นการพัฒนาที่พัฒนาโดย Blockstream มันถูกเขียนด้วยภาษา C โดยมีบางส่วนประกอบใน Rust.



![Image](assets/fr/010.webp)



#### เอแคลร์



Eclair เป็นการใช้งานที่เขียนด้วยภาษา Scala และพัฒนาโดยบริษัทฝรั่งเศส ACINQ ACINQ ดำเนินการหนึ่งในโหนดการกำหนดเส้นทางที่สำคัญที่สุดในเครือข่าย Lightning ด้วย Eclair และใช้การใช้งานนี้เป็นพื้นฐานซอฟต์แวร์สำหรับผลิตภัณฑ์ของตนเอง เช่น แอปพลิเคชัน Phoenix



![Image](assets/fr/011.webp)



#### LDK (*Lightning Development Kit*)



LDK (*Lightning Development Kit*) เป็นชุดพัฒนา Rust ที่ดูแลโดย Spiral (Block, อดีต-Square) มันไม่ใช่ daemon ที่พร้อมใช้งานเหมือน LND หรือ CLN แต่เป็นไลบรารีสำหรับนักพัฒนาที่ต้องการผสานรวม Lightning เข้ากับแอปพลิเคชันของตนโดยตรง



![Image](assets/fr/012.webp)



ในหลักสูตร LNP202 นี้ เราจะมุ่งเน้นไปที่ LND เป็นหลัก: การใช้งานที่ใช้กันมากที่สุดในโซลูชันแบบครบวงจรสำหรับลูกค้าเอกชน เช่น Umbrel



นี่คือการเตือนความจำสั้น ๆ เกี่ยวกับการทำงานของ Lightning อีกครั้ง หากมีแนวคิดใดที่คุณไม่เข้าใจ หรือหากคุณต้องการเจาะลึกเพิ่มเติมก่อนที่จะนำไปปฏิบัติ หลักสูตรของ Fanis Michalakis เป็นแหล่งอ้างอิงที่สำคัญในหัวข้อนี้:

## ทำไมต้องรันโหนด Lightning ของคุณเอง?


<chapterId>421db24e-511c-41ed-ad68-69b0662042ea</chapterId>



การตอบคำถามนี้ค่อนข้างง่าย เพราะมันเป็นคำถามเชิงวาทศิลป์: หากไม่มีโหนดของคุณเอง คุณจะไม่ได้ใช้งาน Lightning จริงๆ แต่เป็นเพียงภาพลวงตาของ Lightning บนโครงสร้างพื้นฐานของบริษัทเท่านั้น



การใช้ Lightning custodial wallet หมายความว่า bitcoins นั้นเป็นของบริษัทที่ดำเนินการโหนดในทางเทคนิค คุณไม่ได้ถือกุญแจส่วนตัว และคุณไม่ได้ควบคุมช่องทางต่างๆ ยอดคงเหลือ wallet ของคุณเป็นเพียงบรรทัดหนึ่งในฐานข้อมูลของผู้ให้บริการ นี่เป็นสถานการณ์ที่สะดวกมากสำหรับผู้เริ่มต้น และประสบการณ์ของผู้ใช้มักจะราบรื่น แต่คำถามพื้นฐานคือ: อะไรคือข้อได้เปรียบของการใช้ Bitcoin และ Lightning หากคุณต้องยอมสละแง่มุมที่ทำให้พวกมันแตกต่างจากสกุลเงินและธนาคารแบบดั้งเดิม?



ข้อเสนอคุณค่าหลักสองประการของ Bitcoin คือ อธิปไตยทางการเงิน (ไม่ต้องพึ่งพาอำนาจกลางในการออกและถือครอง) และการต้านทานการเซ็นเซอร์ (ความเป็นไปไม่ได้ที่บุคคลที่สามจะป้องกันหรือกรองการชำระเงิน) ระบบการดูแลบน Lightning ขัดแย้งกับวัตถุประสงค์ทั้งสองนี้โดยตรง: คุณไม่สามารถตรวจสอบอุปทานเงินภายในของแพลตฟอร์มได้ และโดยนิยามแล้ว ผู้ดำเนินการที่ถือครองเงินทุนทั้งหมดและช่องทางทั้งหมดสามารถเซ็นเซอร์, ล่าช้า, จัดลำดับความสำคัญ หรือบล็อกการชำระเงินของคุณได้ ภายใต้เงื่อนไขเหล่านี้ เราสามารถถามตัวเองได้อย่างถูกต้องว่า **จุดประสงค์ของการใช้บิตคอยน์ผ่าน Lightning คืออะไร หากมันจะทำให้เกิดรูปแบบความไว้วางใจและการพึ่งพาเช่นเดียวกับระบบสกุลเงินของรัฐแบบดั้งเดิม**



> สิ่งที่จำเป็นคือระบบการชำระเงินอิเล็กทรอนิกส์ที่อิงตามหลักฐานการเข้ารหัสแทนที่จะเป็นความไว้วางใจ ซึ่งช่วยให้ฝ่ายที่ต้องการสองฝ่ายสามารถทำธุรกรรมกันโดยตรงโดยไม่จำเป็นต้องมีบุคคลที่สามที่เชื่อถือได้
*Satoshi Nakamoto, Bitcoin เอกสารไวท์เปเปอร์


ปรัชญาไว้ทีหลัง ข้อเสียที่เป็นรูปธรรมมากขึ้นสำหรับคุณมีดังนี้ ประการแรก คุณไม่มีทางยืนยันได้ว่าบริษัทมีบิตคอยน์ที่สอดคล้องกับยอดคงเหลือที่แสดงอยู่จริงหรือไม่ บริษัทอาจดำเนินการด้วยการสำรองแบบเศษส่วน ถูกแฮ็ก ล้มละลาย หรือหายไปเฉยๆ ในกรณีนี้ คุณก็เป็นเพียงเจ้าหนี้รายหนึ่ง โดยไม่มีการรับประกันที่แท้จริงว่าคุณจะได้เงินคืน



ประการที่สอง บริษัทต้องเผชิญกับความเสี่ยงด้านกฎระเบียบ: คำสั่งห้าม การอายัดเงิน การขอให้บล็อกผู้ใช้หรือธุรกรรม การเฝ้าระวังที่เข้มงวดขึ้น หรือแม้กระทั่งการห้ามกิจกรรมโดยสิ้นเชิงในบางเขตอำนาจศาล ข้อจำกัดทุกประการที่ส่งผลต่อผู้ให้บริการจะสะท้อนกลับมาที่คุณโดยอัตโนมัติ



ในแง่ของความลับ สถานการณ์ไม่ได้ดีขึ้นเลย ผู้ดำเนินการดูแลสามารถเห็นการไหลของคุณทั้งหมด: จำนวนเงิน, ความถี่, ผู้รับ, ยอดคงเหลือ, พฤติกรรมการใช้จ่าย เมื่อรวมกับข้อมูลที่แอปพลิเคชันให้และอาจจะเป็นการวิเคราะห์เชนพื้นฐานบน Bitcoin ข้อมูลนี้สามารถให้โปรไฟล์ที่แม่นยำมากของกิจกรรมทางการเงินของคุณ อีกครั้งหนึ่ง มันห่างไกลจากเป้าหมายของ Bitcoin ที่จะลดการตรวจสอบทางการเงิน



ข่าวดีคือในปัจจุบัน การดำเนินการโหนด Lightning ของคุณเองไม่ใช่เรื่องที่สงวนไว้สำหรับผู้เชี่ยวชาญด้านเทคนิคอีกต่อไป เหมือนที่เคยเป็นในช่วงปลายปี 2010 มีโซลูชันที่ค่อนข้างง่ายสำหรับผู้ใช้ตามบ้าน ซึ่งเราจะอธิบายอย่างละเอียดในบทถัดไป




## การเลือกโซลูชันที่เหมาะสมสำหรับงาน


<chapterId>615870e3-741d-4ec1-875d-a483e70f39d4</chapterId>



วันนี้ เป็นไปได้ที่จะมีประสบการณ์ผู้ใช้ที่ใกล้เคียงกับ Lightning custodial wallet ในขณะที่ยังคงการดูแลด้วยตนเอง วัตถุประสงค์ของบทนี้คือเพื่อช่วยให้คุณเลือกเส้นทางที่เหมาะสมที่สุดกับโปรไฟล์ของคุณ



### ตัวเลือกที่ 1: อย่าใช้ Lightning โดยตรง



วิธีแก้ปัญหาแรกคือไม่ใช้ Lightning โดยตรง แต่ใช้ Bitcoin หรือ [Liquid](https://planb.academy/resources/glossary/liquid-network) wallet ที่ฝัง [atomic swaps](https://planb.academy/resources/glossary/atomic-swap) ตัวอย่างเช่น แอปพลิเคชัน Aqua หรือ Bull Bitcoin Wallet ใช้วิธีนี้ โดยช่วยให้คุณชำระเงิน[ใบแจ้งหนี้ Lightning](https://planb.academy/resources/glossary/invoice-lightning) ได้โดยไม่ต้องดำเนินการโหนด Lightning ด้วยตัวเอง ในขณะที่ยังคงควบคุมการดูแลด้วยตนเอง



หลักการนั้นตรงไปตรงมา: เงินทุนของคุณจะอยู่ใน Bitcoin, ไม่ว่าจะเป็น on-chain หรือบน Liquid และคุณเข้าถึงเงินทุนเหล่านั้นผ่าน wallet ที่คุณถือกุญแจในลักษณะดั้งเดิม เมื่อคุณสแกนใบแจ้งหนี้ Lightning, wallet ของคุณจะเริ่มต้นธุรกรรม (on-chain หรือ Liquid) ไปยังบริการแลกเปลี่ยนอะตอมิก บริการนี้จะจัดการการชำระเงิน Lightning ผ่านโหนดของตัวเอง โดยใช้บิตคอยน์ที่คุณให้ on-chain หรือผ่าน Liquid ด้วยเหตุนี้ คุณจึงไม่ต้องจัดการช่องทาง Lightning ด้วยตัวเอง แต่ยังสามารถชำระใบแจ้งหนี้ Lightning ได้อย่างราบรื่น



![Image](assets/fr/013.webp)



ข้อได้เปรียบหลักของวิธีการนี้เมื่อเปรียบเทียบกับ Lightning custodial wallet แบบดั้งเดิมคือคุณยังคงครอบครองเงินของคุณ 100% ตลอดเวลา บิตคอยน์จะอยู่ใน onchain หรือ Liquid wallet ของคุณ พร้อมกับ[วลีช่วยจำ](https://planb.academy/resources/glossary/seed)ของคุณเอง แม้ในระหว่างการสวอป คุณยังคงครอบครองเงินของคุณ เพราะการสวอปเป็นแบบอะตอมมิก มันอาศัยกลไกการเข้ารหัสที่รับประกันว่ามีเพียงสองผลลัพธ์ที่เป็นไปได้เท่านั้น: ไม่ว่าการสวอปจะสำเร็จทั้งหมด หรือมันล้มเหลวและบริการไม่สามารถยึดเงินของคุณได้



พอร์ตโฟลิโอส่วนใหญ่ที่มีฟังก์ชันการทำงานประเภทนี้พึ่งพา [Boltz](https://boltz.exchange/) สำหรับส่วนทางเทคนิคของการสวอป



โซลูชันนี้ยังมีข้อดีที่น่าสนใจในแง่ของความลับ โดยเฉพาะเมื่อใช้ร่วมกับ Liquid สำหรับผู้เริ่มต้น มันยังติดตั้งและบันทึกได้ง่ายมาก: วลีช่วยจำแบบคลาสสิก ไม่มีช่องทาง ไม่มีสภาพคล่องให้สมดุล...



ในทางกลับกัน วิธีการนี้มีข้อจำกัดของมันเอง ประการแรก มันไม่สามารถหลีกเลี่ยงได้: คุณต้องพึ่งพาความพร้อมใช้งานและความเต็มใจของบริการสวอป หากมันไม่ต้องการดำเนินการบัญชีของคุณอีกต่อไป หรือหยุดดำเนินการ คุณจะไม่สามารถชำระใบแจ้งหนี้ Lightning ผ่านมันได้ นอกจากนี้ยังมีค่าธรรมเนียมที่ไม่น้อย: คุณต้องจ่ายทั้ง[ค่าธรรมเนียมการทำธุรกรรม](https://planb.academy/resources/glossary/transaction-fees)บนเชนหรือ Liquid และค่าคอมมิชชั่นของบริการสวอป อีกทั้งหากค่าธรรมเนียมบนเชนเพิ่มขึ้นอย่างรวดเร็ว การใช้ Lightning อาจมีค่าใช้จ่ายสูงมาก



ดังนั้นสำหรับการใช้งานเป็นครั้งคราว มันยังคงยอมรับได้ แต่สำหรับผู้ใช้ Lightning ที่ใช้งานอย่างหนัก ควรทำสิ่งต่างๆ ให้ถูกต้องด้วยโหนด Lightning จริง



### ตัวเลือกที่ 2: โหนด Lightning บนบอร์ด



หมวดหมู่ที่สองของโซลูชันนั้นอิงจากโหนด Lightning ที่ฝังโดยตรงในแอปพลิเคชันมือถือ Phoenix Wallet เป็นผู้บุกเบิกโมเดลนี้และยังคงเป็นมาตรฐาน ปัจจุบันมีโครงการอื่น ๆ ที่เสนอวิธีการที่เทียบเคียงได้ เช่น Zeus (ในโหมดฝัง) หรือ BitKit



แนวคิดนั้นง่าย: แอปพลิเคชันจะทำงานเป็นโหนด Lightning จริง ๆ แต่การดำเนินการที่ซับซ้อนทั้งหมดจะถูกจัดการโดยอัตโนมัติในพื้นหลัง คุณมีอินเทอร์เฟซ "*Lightning wallet*" พร้อมวลีช่วยจำสำหรับการสำรองข้อมูล คุณจะเห็นยอดคงเหลือและชำระใบแจ้งหนี้ แต่คุณไม่ต้องจัดการช่องทาง สภาพคล่อง หรือพารามิเตอร์ส่วนใหญ่



![Image](assets/fr/014.webp)



โซลูชันเหล่านี้เป็นแบบดูแลตนเองเสมอ กุญแจที่ควบคุมเงินทุนคือ generated และถูกเก็บไว้ในโทรศัพท์ของคุณ และการสำรองข้อมูลผ่านทาง seed หรือกลไกที่เทียบเท่า คุณไม่ได้ถือบัญชีกับผู้ให้บริการเพียงอย่างเดียว แต่คุณเป็นเจ้าของบิตคอยน์ที่ถูกล็อกในช่องทางที่เป็นของคุณและไม่สามารถถูกขโมยได้



ข้อดีของโหนดบนบอร์ด LN มีมากมาย:




- ติดตั้งและใช้งานได้ง่ายมาก;
- ประสบการณ์ของผู้ใช้ใกล้เคียงกับ Lightning wallet ที่มีการดูแลโดยผู้ดูแล แต่มีการดูแลด้วยตนเอง;
- ไม่มีการจัดการช่องทางหรือสภาพคล่องด้วยตนเอง;
- การสำรองข้อมูลที่ค่อนข้างง่าย



แต่ wallet ที่ฝังอยู่เหล่านี้ก็มีข้อจำกัดที่สำคัญเช่นกัน ประการแรก ในแง่ของความลับ ผู้ให้บริการ (เช่น ACINQ ในกรณีของ Phoenix) มีมุมมองที่ค่อนข้างละเอียดเกี่ยวกับการไหลผ่านโหนดของคุณ: จำนวนเงิน ความถี่ ผู้รับ แม้ว่าสิ่งนี้จะมีแนวโน้มที่จะดีขึ้น โดยเฉพาะอย่างยิ่งกับการนำ *Trampoline Nodes* มาใช้ทีละน้อย ประการที่สอง คุณพึ่งพาผู้ให้บริการนี้อย่างมากในฐานะเพื่อน Lightning หลักของคุณ หากโหนด ACINQ ไม่สามารถใช้งานได้ (ในกรณีของ Phoenix) หากบริษัทตกอยู่ภายใต้แรงกดดันด้านกฎระเบียบหรือเปลี่ยนรูปแบบธุรกิจ ประสบการณ์การใช้งานของคุณอาจเสื่อมลงอย่างมาก หรือแม้กระทั่งถูกทำลาย



ในที่สุด ความเรียบง่ายนี้ก็มาพร้อมกับค่าใช้จ่าย บริการโหนดฝังตัว LN มักจะเรียกเก็บค่าธรรมเนียมเฉพาะในการฝาก ถอน หรือการดำเนินการจัดการช่องทางบางอย่าง ในความเห็นของฉัน โมเดลนี้มีเหตุผลในแง่ของบริการที่เสนอ แต่สำหรับการใช้งานอย่างเข้มข้น มันอาจจะแพงกว่าการจัดการโหนด Lightning แบบดั้งเดิมที่ดีมาก



### ตัวเลือกที่ 3: โหนด Lightning แบบคลาสสิก



โซลูชันที่สาม ซึ่งเป็นโซลูชันที่เราจะพิจารณาอย่างละเอียดในหลักสูตร LNP202 นี้ คือการใช้งานโหนด Lightning แบบดั้งเดิมบนเซิร์ฟเวอร์หรืออุปกรณ์เฉพาะ



โดย "คลาสสิก" ฉันหมายถึงการที่คุณติดตั้งและกำหนดค่าการใช้งาน Lightning (เช่น LND) ด้วยตัวคุณเองบนโหนด Bitcoin ของคุณเอง คุณเลือกเพียร์ของคุณ เปิดช่องทางของคุณ จัดการ[สภาพคล่องขาเข้าและขาออก](https://planb.academy/resources/glossary/inbound-capacity)ของคุณ และตั้งค่านโยบายค่าธรรมเนียมการกำหนดเส้นทางของคุณเอง



ในแง่ของอธิปไตย นี่คือทางออกที่ดีที่สุด คุณไม่ต้องพึ่งพาบริษัทใดบริษัทหนึ่งสำหรับช่องทางหรือการชำระเงินของคุณอีกต่อไป: หากเพื่อนร่วมงานเซ็นเซอร์คุณหรือปิดช่องทาง คุณสามารถเปิดช่องทางใหม่กับโหนดอื่นได้ หากบริการหายไป sats ของคุณยังคงอยู่ในช่องทางที่คุณควบคุม และคุณสามารถนำกลับมาใช้ใหม่บนเชนได้ คุณยังสามารถเพิ่มประสิทธิภาพค่าใช้จ่ายระยะยาวของคุณได้: เมื่อช่องทางของคุณมีขนาดและการจัดการที่ถูกต้อง ค่าใช้จ่ายโดยรวมของการชำระเงินสามารถลดลงได้มาก



ในแง่ของความลับ คุณย่อมอยู่ภายใต้ข้อจำกัดของโมเดลของ Lightning เอง แต่คุณไม่ได้มอบธุรกิจทั้งหมดของคุณให้กับผู้ดำเนินการเพียงรายเดียว



ในทางตรงกันข้าม การตั้งค่าโหนด Lightning แบบคลาสสิกนั้นซับซ้อนกว่าอย่างเห็นได้ชัด: คุณต้องติดตั้ง กำหนดค่า บำรุงรักษา ตรวจสอบการอัปเดต เข้าใจตรรกะของช่องทางและนโยบายการคิดค่าบริการ จัดการช่องทางและกระแสเงินสด และอื่นๆ การกำหนดค่าผิดพลาด การสำรองข้อมูลที่ไม่เรียบร้อย หรือการจัดการที่ไม่ระมัดระวังสามารถนำไปสู่การสูญเสีย sats ได้ง่ายขึ้น โหนดยังต้องทำงานอย่างต่อเนื่อง



นี่คือเส้นทางที่ฉันเสนอให้คุณปฏิบัติตามในหลักสูตรนี้ โดยจะอยู่เคียงข้างคุณทุกย่างก้าวเพื่อลดความเสี่ยงและจัดโครงสร้างวิธีการของคุณ



### โซลูชันใดสำหรับโปรไฟล์ผู้ใช้ใด?



ในการเลือกโซลูชันที่เหมาะสมสำหรับโปรไฟล์ผู้ใช้ Lightning ของคุณ คุณจำเป็นต้องพิจารณาสองปัจจัย: ความถี่ในการใช้ Lightning และความต้องการในการจัดการด้านเทคนิคของคุณ

## บันทึกโหนด Lightning ของคุณ


<chapterId>638fa75d-62af-4bf3-ab4a-b7d10ea75815</chapterId>



ก่อนที่จะส่ง sats ครั้งแรกไปยังโหนดของคุณ สิ่งสำคัญคือต้องเข้าใจว่าการสำรองข้อมูลทำงานอย่างไรและความเสี่ยงที่เกี่ยวข้องมีอะไรบ้าง ต่างจากพอร์ตโฟลิโอ Bitcoin บนเชนที่ง่าย การสำรองข้อมูลโหนด Lightning ค่อนข้างซับซ้อน: กลยุทธ์ที่ผิดพลาดอาจนำไปสู่การสูญเสียเงินของคุณอย่างถาวร ในบทนี้ เราจะดูว่าสิ่งใดที่จำเป็นต้องสำรองข้อมูลจริง ๆ และ Umbrel จัดการกระบวนการนี้กับ LND อย่างไร



ในหลักสูตรนี้ เราจะใช้การใช้งาน LND (*Lightning Network Daemon*) แม้ว่าหลักการจะคล้ายกันในงานอื่น ๆ แต่ไฟล์และขั้นตอนการกู้คืนที่ฉันจะพูดถึงนั้นเฉพาะเจาะจงสำหรับ LND



### สิ่งที่ควรบันทึกบนโหนด Lightning?



บนโหนด Lightning การบันทึก seed เพียงอย่างเดียวไม่เพียงพอและหวังว่าทุกอย่างจะกลับสู่ปกติ องค์ประกอบหลายอย่างมีบทบาทที่แตกต่างกัน ดังนั้นจึงสำคัญที่จะต้องแยกแยะระหว่างพวกมัน



#### seed (*aezeed*)



เมื่อคุณเริ่มต้น LND คุณจะได้รับ seed จำนวน 24 คำ นี่คือรูปแบบเฉพาะของ LND ที่เรียกว่า "*aezeed*" มันไม่ใช่ BIP39 seed แบบคลาสสิก แม้ว่ามันจะดูคล้ายกันมาก จาก seed นี้ LND จะสร้างกุญแจส่วนตัวของ wallet บนเชนที่เชื่อมโยงกับโหนด Lightning ของคุณ กล่าวคือ ที่อยู่ที่คุณสามารถรับหรือส่งคืนบิตคอยน์หลังจากการปิดช่องทางได้



![Image](assets/fr/019.webp)



seed นี้จึงสามารถใช้สร้าง wallet บนเชนที่เชื่อมโยงกับโหนดของคุณใหม่ และดึงเงินที่ได้ถูกส่งกลับบนเชนแล้ว (เช่น หลังจากการปิดช่องทาง) ในทางกลับกัน seed เพียงอย่างเดียวไม่เพียงพอที่จะกู้คืนช่องทาง Lightning ของคุณที่ยังเปิดอยู่ เนื่องจากมันไม่ได้มีข้อมูลที่จำเป็นเกี่ยวกับสถานะของช่องทางของคุณ



#### ฐานข้อมูลช่อง (`channel.db`)



LND เก็บสถานะรายละเอียดของช่องของคุณในฐานข้อมูลชื่อ `channel.db` ฐานข้อมูลนี้ประกอบด้วย:




- รายการช่องทางที่เปิดอยู่ของคุณ;
- เพื่อนร่วมงานของคุณและตัวระบุของพวกเขา;
- commitment transactions ล่าสุดสำหรับแต่ละช่อง (สถานะต่อเนื่องที่กำหนดว่าใครเป็นเจ้าของอะไรในช่องและช่วยให้สามารถกู้คืนเงินบนเชนได้ในกรณีที่เกิดปัญหา);
- ข้อมูลที่จำเป็นในการลงโทษเพื่อนที่พยายามเผยแพร่รายงานเก่า



ปัญหาของฐานข้อมูลนี้คือมันมีการเปลี่ยนแปลงอยู่ตลอดเวลา: ทุกการชำระเงิน, ทุกการกำหนดเส้นทาง, ทุกการเปิดหรือปิดช่องทางจะปรับเปลี่ยนเนื้อหาของมัน นี่คือเหตุผลที่การสำรองข้อมูลแบบดั้งเดิม (เช่น การคัดลอก `channel.db` เป็นระยะๆ) เป็นอันตราย หากคุณกู้คืนเวอร์ชันเก่าของ `channel.db` และประกอบโหนดใหม่ด้วยสถานะที่ล้าสมัยนี้ เพื่อนร่วมของคุณอาจพิจารณาว่าคุณกำลังเผยแพร่สถานะช่องทางเก่า ในกรณีนี้ โปรโตคอลจะมีบทลงโทษ: เพื่อนร่วมของคุณสามารถกู้คืนจำนวนเงินทั้งหมดของกองทุนในช่องทางได้ ราวกับว่าคุณพยายามโกง



ในทางปฏิบัติแล้ว `channel.db` ไม่ใช่สื่อสำรองข้อมูลในลักษณะนั้น มันคือสถานะที่มีชีวิตของโหนดของคุณ สถานการณ์เดียวที่มีเหตุผลในการใช้มันเพื่อกู้คืนโหนดของคุณคือเมื่อคุณกู้คืนฐานข้อมูลนี้โดยตรงจากเครื่องที่เพิ่งล้มเหลว (เช่น ดิสก์ที่ยังสามารถอ่านได้) ในกรณีนี้ คุณจะกู้คืนสถานะล่าสุดและสามารถเริ่มต้น LND บนเครื่องอื่นโดยอิงจากฐานข้อมูลนี้ได้อย่างปลอดภัย ด้วยความมั่นใจว่าสถานะนี้เป็นปัจจุบันที่สุดเท่าที่จะเป็นไปได้ เนื่องจากเครื่องเก่าไม่ได้ทำงานอีกต่อไป สถานการณ์อื่นที่วิธีนี้สามารถใช้เป็นการสำรองข้อมูลที่เกี่ยวข้องได้คือในโครงสร้างสองดิสก์ โดยมีการคัดลอกแบบไดนามิกและถาวรจากดิสก์หนึ่งไปยังอีกดิสก์หนึ่ง อย่างไรก็ตาม การตั้งค่าประเภทนี้มีความซับซ้อนมากขึ้นในการดำเนินการ



แต่การทำสำเนา `channel.db` เป็นระยะ ๆ และเก็บไว้เป็นข้อมูลสำรองเพื่อกู้คืนในภายหลังเป็นความคิดที่แย่มาก: วันที่คุณใช้มัน คุณเสี่ยงที่จะถูกลงโทษและสูญเสีย sats ทั้งหมดของคุณ



#### การสำรองข้อมูลช่องสัญญาณแบบคงที่ (SCB)



เพื่อให้การกู้คืนจากภัยพิบัติเป็นไปได้ LND ได้แนะนำกลไก *Static Channel Backup* (SCB) นี่คือไฟล์พิเศษที่มักเรียกว่า `channel.backup` ซึ่งมีข้อมูลสถิติเกี่ยวกับช่องของคุณ: ใครคือเพื่อนของคุณ วิธีติดต่อพวกเขา และช่องของคุณคืออะไร



ไฟล์นี้ไม่ได้มีสถานะช่องทางหรือประวัติการอัปเดตอย่างละเอียด มันไม่อนุญาตให้คุณเปิดช่องทางใหม่ในสถานะเดิมที่เคยเป็น หรือดำเนินการ Lightning node นี้ต่อไป แต่บทบาทของมันคือทำหน้าที่เป็นจุดยึดเพื่อขอให้เพื่อนของคุณช่วยปิดช่องทางทั้งหมดของคุณอย่างเรียบร้อย และรับ sats ของคุณบนเชน บนคีย์ที่คุณสามารถดึงข้อมูลได้ด้วย seed ดังนั้น แตกต่างจากไฟล์ `channel.db` ซึ่งถูกแก้ไขทุกครั้งที่มีการชำระเงินหรือการกำหนดเส้นทาง ไฟล์ SCB จะถูกอัปเดตเฉพาะเมื่อมีการเปิดหรือปิดช่องทางเท่านั้น



เมื่อกู้คืนผ่าน SCB กระบวนการเป็นดังนี้:




- คุณกู้คืน seed (*aezeed*) ของคุณ ซึ่งจะสร้าง wallet บนเชนที่เชื่อมโยงกับโหนด Lightning ของคุณขึ้นมาใหม่;
- คุณให้ LND พร้อม SCB ล่าสุดของคุณ;
- LND ใช้ SCB เพื่อค้นหารายชื่อเพื่อนของคุณและช่องทางที่คุณมีกับพวกเขา;
- มันติดต่อกับเพื่อนแต่ละคน บอกพวกเขาว่าคุณประสบปัญหาการสูญเสียข้อมูลและขอให้พวกเขา "ปิดช่องทาง" กับคุณเพื่อที่คุณจะสามารถกู้คืนส่วนแบ่งบนเชนของคุณได้



แนวคิดคือเพื่อนร่วมงานของคุณ เมื่อสังเกตเห็นว่าคุณรายงานการสูญเสียข้อมูล จะทำการกระจายสัญญาณ commitment transaction ล่าสุดของพวกเขาและปิดช่องทางบังคับ เมื่อธุรกรรมเหล่านี้ได้รับการยืนยันแล้ว เงินทุนของคุณจะปรากฏขึ้นอีกครั้งในพอร์ตโฟลิโอ onchain ของคุณ (เชื่อมโยงกับ seed)



อย่างไรก็ตาม กลไกการกู้คืนนี้ไม่สมบูรณ์แบบ ประการแรก มันไม่ได้กู้คืนโหนด Lightning ของคุณจริง ๆ เนื่องจากช่องทางทั้งหมดจะถูกปิด จากนั้นคุณจะต้องสร้างโหนด Lightning ใหม่ตั้งแต่ต้น ประการที่สอง มันไม่ได้รับประกันการกู้คืน 100% แม้ว่าจะเพิ่มโอกาสในการกู้คืนยอดคงเหลือบนเชนของคุณในกรณีที่เกิดปัญหาได้อย่างมากก็ตาม อันที่จริง โปรโตคอลการกู้คืนนี้ขึ้นอยู่กับความร่วมมือและความพร้อมใช้งานของเพื่อนของคุณ: หากคนใดคนหนึ่งออฟไลน์ สูญเสียข้อมูลของตนเอง หรือปฏิเสธที่จะร่วมมือ เงินของคุณอาจยังคงถูกบล็อก หรืออาจสูญหายอย่างถาวร นั่นคือเหตุผลที่สำคัญที่ไม่ควรเปิดช่องทางบนโหนด Lightning ของคุณกับเพื่อนที่ไม่สามารถเข้าถึงได้เป็นเวลานาน หากคุณสูญเสียข้อมูลในขณะนั้นและเพื่อนยังคงไม่สามารถเข้าถึงได้ การกู้คืนผ่าน SCB จะเป็นไปไม่ได้ และเงินของคุณจะยังคงสูญหายจนกว่าเพื่อนคนนั้นจะกลับมาออนไลน์ (อาจจะตลอดไป)



โดยสรุปแล้ว กลยุทธ์การสำรองข้อมูล Lightning ที่ดีบน LND ขึ้นอยู่กับสามเสาหลัก:




- seed ของคุณ (*aezeed*) สำหรับเลเยอร์บนเชน;
- การสำรองข้อมูล SCB อัตโนมัติที่เชื่อถือได้;
- การจัดการช่องทางที่ดีโดยการเลือกเพื่อนที่เชื่อถือได้และปิดช่องทางที่มักจะไม่สามารถเข้าถึงได้ล่วงหน้า



### Umbrel จัดการการสำรองข้อมูลของโหนด LND อย่างไร?



Umbrel นำเสนอระบบสำรองข้อมูลที่ง่ายขึ้นสำหรับโหนด LND โดยอิงตาม SCB อย่างแม่นยำ แนวคิดคือการช่วยให้คุณไม่ต้องยุ่งยากกับการจัดการไฟล์นี้ด้วยตัวเอง โดยทำการสำรองข้อมูลและทำให้กระบวนการเป็นอัตโนมัติมากที่สุดเท่าที่จะเป็นไปได้



เมื่อคุณสร้างโหนดของคุณบน Umbrel คุณจะได้รับ seed ซึ่งมีบทบาทสองประการ:




- มันได้มาจาก Bitcoin ของคุณบนเชน wallet ที่เชื่อมโยงกับโหนด Lightning ของคุณ;
- มันถูกใช้เพื่อสร้างตัวระบุสำรองและกุญแจการเข้ารหัสที่ใช้สำหรับการสำรองข้อมูล SCB ระยะไกล



ด้วยกลไกนี้ Umbrel จะทำการสำรองข้อมูล SCB ของคุณโดยอัตโนมัติและเก็บไว้บนเซิร์ฟเวอร์ของมันผ่าน Tor ข้อมูล SCB จะถูกเก็บไว้ในรูปแบบเข้ารหัสด้วยกุญแจที่ได้มาจาก seed ของคุณ ดังนั้นในกรณีที่ข้อมูลสูญหาย สิ่งที่คุณต้องทำคือสร้าง Bitcoin และโหนด Lightning บน Umbrel บนเครื่องเดียวกันหรือเครื่องอื่น จากนั้นป้อน seed ของคุณ คุณจะสามารถดึงสถานะ SCB ล่าสุดจากเซิร์ฟเวอร์ Umbrel ถอดรหัสและเริ่มกระบวนการกู้คืนเงินของคุณได้



การสำรองข้อมูลเหล่านี้ถูกเข้ารหัสในเครื่องโดยโหนดของคุณก่อนที่จะถูกส่ง ซึ่งรับประกันความลับของข้อมูลของคุณ: Umbrel ไม่สามารถอ่านเนื้อหาของ SCB ได้ การส่งข้อมูลเกิดขึ้นผ่าน Tor ซึ่งป้องกันไม่ให้ที่อยู่ IP ของคุณรั่วไหล ยิ่งไปกว่านั้น Umbrel ของคุณยังเพิ่มเสียงรบกวนให้กับการจราจร (การเติมแบบสุ่มและการสำรองข้อมูลเท็จที่ส่งในช่วงเวลาที่ไม่สม่ำเสมอ) เพื่อป้องกันไม่ให้เซิร์ฟเวอร์สรุปได้อย่างแม่นยำเมื่อคุณเปิดหรือปิดช่องทาง



ข้อได้เปรียบหลักของวิธีนี้คือมันช่วยให้การสำรองข้อมูลของโหนด Lightning ของคุณง่ายขึ้นอย่างมาก: คุณเพียงแค่ต้องสำรองข้อมูล seed ของคุณเพียงครั้งเดียวในระหว่างการเริ่มต้นโหนด นี่มีความเสี่ยงบางอย่าง เนื่องจากมันเป็นเพียงการสำรองข้อมูลของ SCB แต่สำหรับจำนวนที่สมเหตุสมผลมันเป็นการประนีประนอมที่ยอมรับได้



### แนวทางปฏิบัติที่ดีที่สุดในการจำกัดความเสี่ยงของการสูญเสีย



แม้จะมีการสำรองข้อมูล Umbrel แต่การปฏิบัติที่ดีเพียงไม่กี่อย่างสามารถลดความเสี่ยงในการสูญเสีย sats ได้อย่างมาก:





- ตรวจสอบความพร้อมใช้งานของเพื่อนร่วมงานของคุณ:



หากช่องทางที่สำคัญมักเกี่ยวข้องกับเพียร์ที่ไม่สามารถเข้าถึงได้หรือไม่เสถียร การปิดช่องทางนั้นอย่างสะอาดในขณะที่โหนดของคุณยังทำงานอยู่จะปลอดภัยกว่า การปิดแบบร่วมมือหรือบังคับล่วงหน้าจะช่วยขจัดแหล่งที่มาของปัญหาที่อาจเกิดขึ้นในกรณีที่มีการกู้คืน SCB





- หลีกเลี่ยงการกระจุกตัวของสภาพคล่องมากเกินไปกับเพื่อนที่ไม่รู้จัก:



ยิ่งช่องทางที่เพื่อนมีร่วมกับคุณใหญ่เท่าไหร่ ก็ยิ่งสำคัญที่ต้องมีความน่าเชื่อถือ เลือกโหนดที่จริงจัง มีการเชื่อมต่อที่ดี และมีความเคลื่อนไหว เพื่อให้การกู้คืนในอนาคตผ่าน SCB สามารถดำเนินไปได้อย่างราบรื่น





- เสริม Umbrel ด้วยการสำรองข้อมูลในเครื่อง:



นอกเหนือจากการสำรองข้อมูลอัตโนมัติของ Umbrel แล้ว คุณยังสามารถเก็บสำเนาที่เข้ารหัสของไฟล์ SCB ของคุณ (`channel.backup`) ไว้ในสื่อภายนอกและอัปเดตเป็นระยะๆ ควรต่ออายุทุกครั้งที่คุณเปิดหรือปิดช่องทาง ซึ่งจะทำให้คุณมีวิธีสำรองข้อมูลหากบริการสำรองข้อมูลอัตโนมัติของ Umbrel ไม่สามารถใช้งานได้ไม่ว่าด้วยเหตุผลใดก็ตาม





- การจัดการ seed ของคุณอย่างถูกวิธี



seed Umbrel ของคุณไม่เพียงแต่กู้คืน wallet บนเชนของคุณ แต่ยังสร้างคีย์การเข้ารหัสสำหรับการสำรองข้อมูลด้วย ผู้โจมตีที่สามารถเข้าถึงมันได้จึงสามารถเปิดการกู้คืนและโอนเงินของคุณไปยัง wallet ของเขาเองได้ โดยไม่จำเป็นต้องเข้าถึงโหนดของคุณทางกายภาพเลย



ดังนั้น หากคุณจำเป็นต้องกู้คืนโหนดของคุณแต่ไม่มี seed อีกต่อไป คุณจะไม่สามารถกู้คืนอะไรได้เลย: sats ทั้งหมดของคุณจะสูญหาย ดังนั้นจึงเป็นสิ่งสำคัญมากที่จะต้องบันทึก seed ของคุณด้วยความระมัดระวังสูงสุด เฉพาะบนสื่อทางกายภาพ (กระดาษหรือโลหะ) และเก็บไว้ในที่ปลอดภัย สำหรับข้อมูลเพิ่มเติมเกี่ยวกับการจัดการ seed โปรดดูบทแนะนำนี้:

### ฉันจะบันทึกโหนด Lightning ของฉันบน Umbrel ได้อย่างไร?



ตอนนี้ที่คุณเข้าใจแล้วว่าทฤษฎีทำงานอย่างไร มาเริ่มลงรายละเอียดกันเลย จากแอปพลิเคชัน Lightning Node ของคุณ (ซึ่งจริงๆ แล้วคือ LND) ให้คลิกที่จุดเล็กๆ สามจุดที่มุมขวาบน



![Image](assets/fr/022.webp)



มีสามองค์ประกอบที่น่าสนใจสำหรับการสำรองข้อมูล:




- ตรวจสอบให้แน่ใจว่าได้เปิดใช้งานตัวเลือก `Automatic backups` แล้ว ซึ่งจะส่ง SCB ที่เข้ารหัสของคุณไปยังเซิร์ฟเวอร์ของ Umbrel โดยอัตโนมัติ
- จากนั้นคุณสามารถเลือกที่จะส่งผ่าน Tor หรือ clearnet โดยใช้ตัวเลือก `Backup over Tor` ตามที่อธิบายไว้ในส่วนก่อนหน้า ฉันขอแนะนำอย่างยิ่งให้คุณใช้ Tor เพื่อรักษาความลับของคุณ
- ในที่สุดก็มีปุ่ม `Download channel backup file` ซึ่งช่วยให้คุณดาวน์โหลดไฟล์ `channel.backup` ซึ่งเป็นภาพรวมที่เข้ารหัสของ SCB ของคุณไปยังคอมพิวเตอร์ของคุณ สิ่งนี้ช่วยให้คุณสามารถทำการสำรองข้อมูลในเครื่องเพิ่มเติมได้เป็นครั้งคราว นอกเหนือจากที่ส่งไปยังเซิร์ฟเวอร์ Umbrel โดยอัตโนมัติ



ตอนนี้คุณรู้วิธีปกป้อง sats ของโหนด Lightning ของคุณจากการสูญหายของข้อมูลแล้ว ในบทถัดไป เราจะมาดูวิธีการรักษาความปลอดภัยโหนดของคุณจากความพยายามโกง




## Watchtower: บทบาทและการตั้งค่า


<chapterId>e6c654dd-26c5-4e4d-8d11-a215bac37812</chapterId>



ใน Lightning แต่ละช่องทางจะอิงตามลำดับของสถานะต่อเนื่อง ซึ่งแสดงโดย commitment transactions ที่ยังไม่ได้เผยแพร่ ด้วยการชำระเงินหรือการกำหนดเส้นทางของ Lightning แต่ละครั้ง ผู้เข้าร่วม 2 คนในช่องทางจะสร้างคู่ใหม่ของ commitment transactions ซึ่งสะท้อนถึงการกระจายเงินทุนในช่องทางปัจจุบัน commitment transactions เก่าจะกลายเป็นล้าสมัย



หากฝ่ายใดฝ่ายหนึ่งเผยแพร่สถานะที่ล้าสมัย อีกฝ่ายมีสิทธิ์ที่จะลงโทษและกู้คืนจำนวนเงินทั้งหมดของกองทุนของช่องทาง ในบทนี้ เราจะมาดูวิธีการทำงานของกลไกนี้อย่างคร่าว ๆ และจากนั้นอธิบายวิธีการตั้งค่า ***watchtower***: ระบบเพื่อปกป้องโหนด Lightning ของคุณจากความพยายามโกงที่อาจเกิดขึ้น



### การทำความเข้าใจเกี่ยวกับการทำงานของหอคอยเฝ้าระวัง


ในทุกช่วงเวลา แต่ละฝ่ายในช่องทางจะมี commitment transaction ซึ่งหากเผยแพร่จะทำให้พวกเขาสามารถปิดช่องทางและกู้คืนส่วนแบ่งของเงินทุนได้ กระบวนการนี้เรียกว่า *การปิดแบบบังคับ* แต่หากพวกเขาพยายามเผยแพร่ commitment transaction ที่เก่ากว่า ซึ่งสอดคล้องกับสถานะก่อนหน้าของช่องทางที่มี sats มากกว่า การทำธุรกรรมนี้จะถือว่าเป็นความพยายามในการโกง ในกรณีนี้ คู่สัญญาสามารถใช้กุญแจเพิกถอนที่เกี่ยวข้องกับสถานะเก่านี้เพื่อกู้คืนจำนวนเงินทั้งหมดในช่องทาง ในขณะที่ผู้โกงจะถูกบล็อกชั่วคราวโดยการล็อกเวลา


ระบบนี้หมายความว่าการเผยแพร่สถานะเก่า หรือพยายามโกงนั้นมีความเสี่ยงสูง: หากฝ่ายตรงข้ามเห็นธุรกรรมนี้ใน mempool หรือบนบล็อกเชนก่อนที่ timelock จะหมดอายุ พวกเขาสามารถใช้กุญแจเพิกถอนและกู้คืนเงินทั้งหมดได้ **ดังนั้น ความปลอดภัยของช่องทาง Lightning ของคุณขึ้นอยู่กับความสามารถของคุณในการตรวจจับความพยายามโกงภายในกรอบเวลาที่กำหนดโดย timelock**.



#### ทำไมหอคอยเฝ้าระวังจึงจำเป็น?



กลไกการลงโทษจะทำงานได้ก็ต่อเมื่อฝ่ายที่ได้รับบาดเจ็บสามารถ:




- ตรวจสอบแต่ละบล็อก Bitcoin ใหม่เพื่อดูว่ามีการเผยแพร่ช่อง commitment transaction หรือไม่;
- พิจารณาว่าธุรกรรมนี้สอดคล้องกับสถานะที่ถูกต้องล่าสุดหรือสถานะที่ถูกเพิกถอน;
- ในกรณีที่สถานะถูกเพิกถอน ให้กระจายธุรกรรมทางกฎหมายให้ทันเวลา โดยใช้กุญแจเพิกถอนเพื่อกู้คืนเงินทั้งหมดก่อนที่การล็อกเวลาจะหมดอายุ



![Image](assets/fr/023.webp)



ในสถานการณ์ที่เหมาะสม โหนด Lightning ของคุณจะออนไลน์ตลอด 24/7 มันจะซิงโครไนซ์และตรวจสอบบล็อกเชนอย่างต่อเนื่อง ด้วยเหตุนี้ มันสามารถตรวจจับความพยายามโกงและตอบสนองได้ด้วยตัวเอง อย่างไรก็ตาม ในทางปฏิบัติ โหนด Lightning ส่วนบุคคลอาจปิดตัวลง โดยเฉพาะอย่างยิ่งในกรณีที่ไฟฟ้าดับเป็นเวลานานหรือการเชื่อมต่ออินเทอร์เน็ตล้มเหลว



ในช่วงเวลาที่หยุดทำงานเหล่านี้เองที่ความเสี่ยงกลายเป็นเรื่องจริง: หากเพื่อนที่ไม่ซื่อสัตย์เผยแพร่สถานะเก่าในขณะที่โหนดของคุณออฟไลน์ และการล็อกเวลา (timelock) หมดลงโดยไม่มีการตอบสนองจากคุณ การโกงก็จะมีผล คุณจะสูญเสียเงินบางส่วนหรือทั้งหมดในช่องทางนั้น



Watchtowers ได้ถูกนำมาใช้เพื่อลดความเสี่ยงนี้ หอคอยเฝ้าระวังเป็นบริการภายนอกที่ตรวจสอบบล็อกเชนในนามของคุณ โดยสแกนหาการเผยแพร่สถานะเก่าที่อาจเกิดขึ้นในช่องทางของคุณ และหากจำเป็น จะทำการเผยแพร่ธุรกรรมการลงโทษโดยอัตโนมัติในนามของคุณ ดังนั้น แม้ว่าตัวโหนด Lightning ของคุณจะออฟไลน์เป็นเวลานาน ตราบใดที่หอคอยเฝ้าระวังที่คุณใช้งานยังคงทำงานอยู่ มันจะสามารถปกป้องเงินทุนของคุณได้โดยการตรวจสอบความพยายามในการโกงและใช้การลงโทษที่สอดคล้องกันทันทีที่ตรวจพบ



#### วิธีการทำงานของหอคอยเฝ้าระวัง



หอคอยเฝ้าระวังถูกออกแบบมาเพื่อลดข้อมูลที่ได้รับเกี่ยวกับช่องทางของคุณ ในขณะเดียวกันก็ให้วิธีการในการดำเนินการในกรณีที่เกิดปัญหา:




- สำหรับแต่ละสถานะช่องทางใหม่กับเพียร์ โหนดของคุณจะเตรียมธุรกรรมการลงโทษที่เป็นไปได้ล่วงหน้า ในกรณีที่เพียร์นี้โกง ธุรกรรมนี้จะช่วยให้คุณกู้คืนเงินทั้งหมดในช่องทางได้
- โหนดของคุณจะเข้ารหัสธุรกรรมการลงโทษนี้โดยใช้ TXID ของ commitment transaction ที่สอดคล้องกัน (ซึ่งจะถูกใช้หากผู้โกงพยายามที่จะทำการฉ้อโกง) ตราบใดที่ไม่มีการปิดบัญชีเกิดขึ้น หอคอยเฝ้าระวังจะไม่สามารถถอดรหัสธุรกรรมนี้ได้ เนื่องจากมันไม่ทราบ TXID ของธุรกรรมการโกงอย่างครบถ้วน
- โหนดของคุณส่งแพ็กเก็ตไปยังหอคอยเฝ้าระวังซึ่งมีการเข้ารหัสธุรกรรมการลงโทษและครึ่งหนึ่งของ TXID ของธุรกรรมการโกงที่อาจเกิดขึ้น



เนื่องจาก TXID ที่ส่งไปยังหอคอยเฝ้าระวังไม่สมบูรณ์ มันไม่สามารถถอดรหัสธุรกรรมยุติธรรมได้ อย่างไรก็ตาม มันสามารถตรวจสอบบล็อกเชนเพื่อหา TXID ที่ตรงกับส่วนที่มันเป็นเจ้าของได้ หากมันตรวจพบธุรกรรมดังกล่าว มันจะพยายามใช้ TXID เต็มของธุรกรรมนั้นเพื่อถอดรหัสธุรกรรมลงโทษของคุณ หากการถอดรหัสสำเร็จ มันจะรู้ว่าเป็นความพยายามโกงและจะเผยแพร่ธุรกรรมลงโทษให้คุณทันที



![Image](assets/fr/024.webp)



ดังนั้นหอคอยเฝ้าดูจึงไม่สามารถมองเห็นรายละเอียดของช่องทางของคุณ: ไม่ว่าจะเป็นตัวตนของเพื่อนร่วมงานของคุณ ยอดคงเหลือ หรือโครงสร้างของธุรกรรม มันจะเห็นเพียงแพ็กเก็ตที่เข้ารหัส ข้อมูลเดียวที่มันสามารถสรุปได้คืออัตราที่ช่องทางของคุณถูกอัปเดต เนื่องจากมันได้รับแพ็กเก็ตสำหรับแต่ละสถานะใหม่ แต่ไม่สามารถรู้เนื้อหาของมันได้ ในกรณีที่มีการโกง มันจะค้นพบข้อมูลช่องทางโดยการถอดรหัสธุรกรรมการลงโทษอย่างแน่นอน แต่อย่างน้อย sats ของคุณจะถูกบันทึกไว้



กลไกนี้ขึ้นอยู่กับการประนีประนอม: คุณมอบหมายความสามารถในการเผยแพร่ธุรกรรมการลงโทษที่ลงนามล่วงหน้าให้กับหอคอยเฝ้าระวัง แต่ธุรกรรมนี้ยังคงเป็นความลับต่อหอคอยเฝ้าระวังจนกว่าจะมีการโกงเกิดขึ้น หอคอยเฝ้าระวังไม่สามารถแก้ไขผู้รับหรือเบี่ยงเบนเงินทุนได้ เนื่องจากมีเพียงธุรกรรมที่ได้ลงนามแล้ว โดยมีผลลัพธ์ที่ถูกแช่แข็งในความโปรดปรานของคุณ นอกจากนี้ยังไม่สามารถทราบรายละเอียดของช่องทางในกรณีปิดที่ถูกบังคับหรือร่วมมือกันอย่างถูกต้องตามกฎหมาย เนื่องจาก TXIDs ไม่ตรงกัน ในทางกลับกัน หอคอยเฝ้าระวังยังคงเป็นบุคคลที่สามที่เชื่อถือได้ขั้นต่ำ: คุณจำเป็นต้องพึ่งพามันในการออนไลน์และเผยแพร่ธุรกรรมความยุติธรรมของคุณอย่างถูกต้องเมื่อคุณต้องการ



#### การเป็นหอคอยเฝ้าระวัง



ในทางทฤษฎี โหนด Lightning ใด ๆ สามารถทำหน้าที่เป็นหอคอยเฝ้าระวังสำหรับโหนดอื่น ๆ (หากพวกเขาใช้การใช้งานเดียวกัน เช่น LND) ในขณะที่ตัวมันเองได้รับการปกป้องโดยโหนดอื่น ๆ ที่ทำหน้าที่นี้ให้ ในส่วนปฏิบัติถัดไป ฉันจะแสดงวิธีการตั้งค่ากลไกง่าย ๆ นี้บน LND ของคุณภายใต้ Umbrel


ผลที่ตามมาคือ กลยุทธ์ที่น่าสนใจอาจเป็นการตกลงกับเพื่อนบิตคอยน์ที่ไว้ใจได้เพื่อทำหน้าที่เป็นหอคอยเฝ้าระวังให้กันและกัน คุณตรวจสอบช่องของพวกเขา และพวกเขาตรวจสอบช่องของคุณ



### ค้นหาหอคอยเฝ้าระวังที่มีความเสียสละ



หากคุณไม่รู้จักใครที่สามารถให้บริการหอคอยเฝ้าระวังได้ มีหอคอยเฝ้าระวังสาธารณะหลายแห่งที่คุณสามารถเชื่อมต่อได้ ตัวอย่างเช่น ในหลักสูตร LNP202 นี้ ฉันขอแนะนำให้คุณเชื่อมต่อกับบริการหอคอยเฝ้าระวังที่เสนอร่วมกันโดย LN+ และ Voltage ซึ่งเป็นหอคอยเฝ้าระวังสำหรับ LND


นี่คือรายละเอียดการเข้าสู่ระบบ:



- ผ่าน Tor:


```txt
023bad37e5795654cecc69b43599da8bd5789ac633c098253f60494bde602b60bf@iiu4epqzm6cydqhezueenccjlyzrqeruntlzbx47mlmdgfwgtrll66qd.onion:9911
```



- ผ่าน clearnet:


```txt
023bad37e5795654cecc69b43599da8bd5789ac633c098253f60494bde602b60bf@34.216.52.158:9911
```

เพื่อขอบคุณพวกเขาที่ให้บริการหอคอยเฝ้าระวังฟรีนี้ [คุณสามารถบริจาคผ่าน Lightning](https://lightningnetwork.plus/donation)


ตอนนี้ที่เราใช้บริการหอคอยเฝ้าระวังที่มีความเสียสละ ลองมาดูวิธีการตั้งค่าบนโหนด LND ภายใต้ Umbrel กันเถอะ


### การตั้งหอคอยเฝ้าระวัง


จากแอปพลิเคชัน `Lightning Node` ให้คลิกที่จุดสามจุดที่มุมขวาบนของอินเทอร์เฟซ จากนั้นเลือก `Advanced Settings`.


![Image](assets/fr/025.webp)


จากนั้นไปที่เมนู `Watchtower`


![Image](assets/fr/026.webp)



เปิดใช้งานตัวเลือก `Watchtower Client` จากนั้นคลิกที่ปุ่ม `SAVE AND RESTART NODE` รอให้ LND เริ่มใหม่อีกครั้ง


![Image](assets/fr/027.webp)


เมื่อการรีสตาร์ทเสร็จสิ้น ให้กลับไปที่เมนูเดิมและป้อน ID ของหอคอยเฝ้าระวังที่คุณเลือกในช่องที่ให้ไว้ จากนั้นคลิกที่ปุ่ม `ADD` เพื่อยืนยัน คุณยังสามารถปรับพารามิเตอร์ `Watchtower Client Sweep Fee Rate` ได้: นี่คืออัตราค่าธรรมเนียมที่คุณยินดีจ่ายสำหรับการทำธุรกรรมยุติธรรมที่อาจถูกกระจายโดยหอคอยเฝ้าระวัง ไม่จำเป็นต้องเลือกอัตราที่สูงเกินไป แต่คุณควรหลีกเลี่ยงอัตราที่ต่ำเกินไป มิฉะนั้นการทำธุรกรรมทางกฎหมายจะไม่ได้รับการยืนยันในเวลาที่เหมาะสม


รีสตาร์ทโหนดของคุณโดยใช้ปุ่ม `SAVE AND RESTART NODE` เพื่อใช้การเปลี่ยนแปลงเหล่านี้


![Image](assets/fr/028.webp)



หากคุณกลับมาที่เมนูเดิมนี้ คุณจะเห็นว่าโหนด Lightning ของคุณได้รับการปกป้องโดยหอคอยเฝ้าระวังที่คุณเพิ่งเพิ่มเข้าไปแล้ว



![Image](assets/fr/029.webp)



หอคอยเฝ้าระวังที่มีความเอื้อเฟื้อเผื่อแผ่โดยทั่วไปก็เพียงพอแล้ว โดยเฉพาะอย่างยิ่งหากคุณไม่ได้วางเงินจำนวนมากบนโหนด Lightning ของคุณและหากคุณจัดการโหนดของคุณได้ดี (อย่าปิดทิ้งไว้นานเกินไป) เพื่อความปลอดภัยที่มากยิ่งขึ้น คุณยังสามารถเพิ่มหลาย ๆ หอคอยได้โดยทำซ้ำกระบวนการเดียวกัน

หากคุณมาถึงจุดนี้แล้ว คุณคงทราบดีว่าโหนด Lightning ที่ไม่มีช่องทางก็เหมือนกับ wallet ที่ว่างเปล่า: มันมีอยู่ แต่ไม่มีประโยชน์ เพื่อที่จะสามารถส่งหรือรับการชำระเงินได้ โหนดของคุณต้องเชื่อมต่อกับโหนดอื่นอย่างน้อยหนึ่งโหนดในเครือข่าย Lightning ผ่านช่องทาง ในอนาคต เราขอแนะนำอย่างยิ่งให้เปิดหลายช่องทาง เพื่อเหตุผลด้านความยืดหยุ่นและประสิทธิภาพในการกำหนดเส้นทาง ในบทต่อไป เราจะดูวิธีจัดการสินทรัพย์สภาพคล่องของคุณ ปรับแต่งโครงสร้างช่องทางของคุณ และใช้เครื่องมือขั้นสูงกว่าที่อินเทอร์เฟซพื้นฐาน LND บน Umbrel



แต่ในบทนำนี้ เป้าหมายคือการทำความเข้าใจวิธีการเลือกเพื่อนร่วมเครือข่าย Lightning ที่ดี สถานที่ค้นหาข้อมูลที่คุณต้องการเพื่อเลือกเพื่อนร่วมเครือข่ายของคุณ และสุดท้ายคือวิธีการเปิดช่องทางแรกของคุณผ่านอินเทอร์เฟซพื้นฐาน LND



### วิธีเลือกเพียร์ Lightning ที่ดี?



เมื่อคุณเปิดช่อง คุณจำเป็นต้องเลือกเพียร์: โหนด Lightning อื่นที่โหนดของคุณจะเชื่อมต่อโดยตรงผ่านช่อง การเลือกเพียร์นี้มีความสำคัญ เนื่องจากจะมีผลกระทบโดยตรงต่อ:




- ความสะดวกในการชำระเงินของคุณให้ไปถึงที่หมาย;
- ความน่าเชื่อถือของช่องทางของคุณเมื่อเวลาผ่านไป;
- ค่าใช้จ่ายในการกำหนดเส้นทางของคุณ



ไม่มีสิ่งที่เรียกว่าคู่ที่สมบูรณ์แบบสำหรับทุกคน แต่มีเกณฑ์ที่น่าเชื่อถือหลายประการในการระบุผู้สมัครที่ดี



#### 1. การเชื่อมต่อของโหนด



เพียร์ที่ดีคือโหนดที่เชื่อมต่อกับเครือข่าย Lightning ได้ดี ซึ่งหมายถึงไม่เพียงแค่มีช่องทางจำนวนมาก (แม้ว่านี่อาจเป็นตัวบ่งชี้) แต่เหนือสิ่งอื่นใดคือการเชื่อมต่อกับโหนดที่เชื่อถือได้อื่น ๆ และครอบครองตำแหน่งที่เป็นศูนย์กลางเพียงพอในกราฟเครือข่าย



โหนดที่เชื่อมต่อกันอย่างดีจะเพิ่มโอกาสในการค้นหาเส้นทางที่มีประสิทธิภาพไปยังจุดหมายปลายทางการชำระเงินส่วนใหญ่ นอกจากนี้ยังลดจำนวนโหนดตัวกลางที่จำเป็น ซึ่งช่วยลดค่าใช้จ่ายลง



ในทางกลับกัน การเปิดช่องทางแรกของคุณไปยังโหนดที่แยกตัวหรือเชื่อมต่ออย่างอ่อนอาจจำกัดความเป็นไปได้ของคุณ: คุณจะสามารถชำระเงินให้กับเพียร์นี้ได้ แต่จะยากมากขึ้นในการชำระเงินให้กับเครือข่ายที่เหลือ



#### 2. การใช้ตัวพิมพ์ใหญ่และความจุของช่องสัญญาณ



เพียร์ที่ดีคือโหนดที่มีเงินทุนเพียงพอ ซึ่งแสดงโดยความจุช่องสัญญาณทั้งหมด (ผลรวมของ sats ที่ผูกพันกับช่องสัญญาณทั้งหมด) และความจุช่องสัญญาณเฉลี่ย (มักจะดีกว่าที่จะมีเพียงไม่กี่ช่องที่มีเงินทุนเพียงพอมากกว่ามีหลายช่องที่มีขนาดเล็ก)



โหนดที่มีความจุสูงเกินไป หรือช่องทางทั้งหมดมีขนาดเล็ก จะไม่สามารถส่งผ่านการชำระเงินที่มีจำนวนมากได้ ส่งผลให้เกิดความล้มเหลวในการส่งผ่านสำหรับคุณ



#### 3. นโยบายค่าใช้จ่าย



แต่ละโหนดจะกำหนดค่าธรรมเนียมการกำหนดเส้นทางของตัวเอง (`base fee` และ `fee rate`) เพียร์ที่ดีจะไม่เรียกเก็บค่าธรรมเนียมที่สูงเกินไปสำหรับช่องทางเชิงกลยุทธ์ สำหรับช่องทางแรก ควรเลือกโหนดที่มีค่าธรรมเนียมปานกลาง เพื่อไม่ให้เป็นอุปสรรคต่อการชำระเงินของคุณเอง



โปรดจำไว้ว่าค่าธรรมเนียมการกำหนดเส้นทางของคุณเองยังมีผลต่อการที่ผู้อื่นมองคุณในฐานะเพื่อนร่วมเครือข่าย: โหนดที่เปลี่ยนแปลงค่าธรรมเนียมบ่อยครั้งหรือกำหนดค่าใช้จ่ายที่ไร้เหตุผลมักจะไม่ได้รับการชื่นชมบ่อยนัก



#### 4. ความมั่นคงและอาวุโส



ความเสถียรของเพียร์เป็นเกณฑ์ที่สำคัญมาก โหนดที่ดีจะมีเวลาทำงานสูง (แทบจะไม่ออฟไลน์เลย) มันจะรักษาช่องทางของมันให้เปิดไว้นานและไม่เปิด/ปิดช่องทางบ่อยๆ โดยไม่มีเหตุผล



ช่องทางที่เก่าแต่ยังคงใช้งานอยู่เป็นสัญญาณที่ดี: พวกเขาบ่งบอกว่าความสัมพันธ์นั้นมีกำไรสำหรับเพื่อนร่วมเครือข่าย, ว่าโหนดรู้วิธีจัดการทุนของตนและมันจะไม่ปิดตัวลงในเวลาใด ๆ, ดังนั้นคุณไม่จำเป็นต้องจ่ายค่าธรรมเนียมบนเชนสำหรับการปิดและเปิดใหม่อยู่เสมอ.



ในทางกลับกัน เพื่อนที่มักจะออฟไลน์ หรือที่ปิดช่องทางอย่างรวดเร็ว อาจเป็นแหล่งปัญหาสำหรับคุณ และเป็นค่าใช้จ่ายเพิ่มเติมในการเปิดช่องทางใหม่



แม้จะมีเกณฑ์เหล่านี้ คุณก็จะไม่สามารถเลือกได้อย่างสมบูรณ์แบบในครั้งแรก นั่นเป็นเรื่องปกติ: คุณภาพที่แท้จริงของเพื่อนร่วมงานจะถูกเปิดเผยจากการใช้งาน ดังนั้นสิ่งสำคัญคือ:




- ตรวจสอบกิจกรรมของช่องทาง (ปริมาณที่ส่งผ่าน, ความพร้อมใช้งาน, ฯลฯ);
- ปิดช่องทางที่ไม่มีประโยชน์หรือที่มีเพื่อนร่วมงานออฟไลน์บ่อยเกินไป;
- จัดสรรเงินทุนของคุณใหม่ให้กับเพื่อนร่วมงานที่ดีกว่าเมื่อเวลาผ่านไป



แนวคิดคือไม่เปิดและปิดช่องทางทุกวัน (ซึ่งจะมีค่าใช้จ่ายสูงในด้านค่าใช้จ่ายบนเชน) แต่ค่อยๆ พัฒนาโครงสร้างของคุณเพื่อให้บรรจบกับชุดของเพื่อนที่เชื่อถือได้และเชื่อมต่อกันดีซึ่งสอดคล้องกับความต้องการของคุณ



### ฉันจะหาคนที่มีความสนใจเหมือนกันได้อย่างไร?



ในการใช้เกณฑ์เหล่านี้ คุณจะต้องมีเครื่องมือที่ให้การมองเห็นเครือข่าย Lightning มีตัวสำรวจและบริการหลายอย่างที่พร้อมใช้งานเพื่อทำสิ่งนี้ ในบรรดาตัวสำรวจ Lightning ที่เป็นที่รู้จักกันดี ได้แก่ [1ML](https://1ml.com/) และ [Amboss](https://amboss.space/)



https://planb.academy/tutorials/node/lightning-network/amboss-37044cad-0f85-41eb-af18-491384af1017

https://planb.academy/tutorials/node/lightning-network/1ml-37ada2ab-7a24-4473-87fd-007cb7640e7b

อย่างไรก็ตาม ที่นี่ฉันขอแนะนำให้คุณใช้ [เครื่องมือ Lightning Terminal จาก Lightning Labs](https://terminal.lightning.engineering/) ซึ่งให้การจัดอันดับ (ยอมรับว่าขึ้นอยู่กับเกณฑ์ที่เป็นอัตวิสัยบางส่วน) ของโหนด Lightning ที่ถือว่ามีความเกี่ยวข้องมากที่สุดสำหรับการเปิดช่องทาง



![Image](assets/fr/030.webp)



ปัญหาของโหนด Lightning ขนาดใหญ่มากที่อยู่ด้านบนของการจัดอันดับนี้คือส่วนใหญ่ไม่ยอมรับการเปิดช่องทางที่ต่ำกว่าจำนวนเงินที่สูงมาก หลายโหนดยังใช้กฎการจัดการเพื่อนที่เข้มงวด ซึ่งอาจทำให้ช่องทางของคุณถูกปิดได้ ในทางกลับกัน โหนดเหล่านี้โดยทั่วไปไม่มีความจำเป็นสำหรับสภาพคล่องขาเข้า เนื่องจากจำนวนการเชื่อมต่อที่มีอยู่



ดังนั้นฉันขอแนะนำให้คุณทำงานตามลำดับลงมาเพื่อหาน็อดที่เชื่อมต่อได้ดี, เชื่อถือได้ และอยู่ในตำแหน่งที่เป็นศูนย์กลางในกราฟเครือข่าย โดยไม่ใหญ่เกินไป ตัวอย่างเช่น ที่นี่ฉันได้ระบุ Lightning node บน stacker.news ซึ่งเชื่อมต่อได้ดีมาก, มีความจุสูง และอยู่ในตำแหน่งศูนย์กลางในกราฟเครือข่าย



![Image](assets/fr/031.webp)



อีกวิธีที่น่าสนใจคือการเปิดช่องทางไปยังโหนดที่ต้องการสภาพคล่องขาเข้า เช่น ร้านค้าที่คุณรู้จัก สมาคม หรือชุมชน กลยุทธ์นี้มีข้อดีสามประการ:




- เนื่องจากหน่วยงานที่เลือกต้องการสภาพคล่องขาเข้า จึงมีแรงจูงใจน้อยลงในการปิดช่องของคุณโดยไม่มีเหตุผล
- กิจกรรมทางเศรษฐกิจของมันเพิ่มโอกาสในการกำหนดเส้นทางบนช่องทางนี้ และดังนั้นจึงเพิ่มโอกาสในการเก็บค่าธรรมเนียมบางส่วน;
- คุณกำลังทำประโยชน์ให้กับระบบนิเวศและมีส่วนร่วมที่มีคุณค่าสำหรับผู้ใช้บิตคอยน์คนอื่น ๆ



เมื่อคุณระบุโหนดที่เกี่ยวข้องได้แล้ว คุณสามารถดึง Node ID ของมันได้ นี่คือการรวมกันของคีย์สาธารณะของโหนด ตัวคั่น `@` ที่อยู่ IP หรือที่อยู่ Tor และพอร์ตที่ใช้ คุณสามารถเข้าถึง ID นี้ได้ง่ายจากโปรไฟล์ของโหนดบน Lightning explorer ใด ๆ



### เปิดช่องแรกของคุณผ่าน LND



ตอนนี้ที่เราได้ระบุโหนดที่จะเปิดช่องแรกของเราแล้ว เราต้องให้ sats ล็อคเข้ากับมัน ในการทำเช่นนี้ คุณต้องป้อน Bitcoin onchain wallet ที่เชื่อมโยงกับ LND ของคุณ



จากอินเทอร์เฟซหลัก LND ให้ค้นหา `Bitcoin Wallet` จากนั้นคลิกที่ปุ่ม `Deposit` ที่อยู่รับแบบ onchain จะเป็น generated: ส่ง sats ไปยังที่อยู่นั้น จำนวนที่คุณต้องโอนขึ้นอยู่กับความจุที่คุณต้องการจัดสรรให้กับช่องแรกของคุณ แต่ควรจำไว้ว่าคุณต้องส่งมากกว่าความจุที่ตั้งเป้าหมายเล็กน้อย ตัวอย่างเช่น หากคุณต้องการเปิดช่อง sats ขนาด 500,000 อย่าส่งเพียง 500,000 sats แต่ให้ส่งจำนวนที่มากกว่า



![Image](assets/fr/032.webp)



เมื่อการทำธุรกรรมถูกกระจายออกไปแล้ว มันจะแสดงในอินเทอร์เฟซ รอการยืนยันก่อนเปิดช่องทาง คุณจะเห็นลูกศรสีเขียวข้างๆ เมื่อมันได้รับการยืนยัน



![Image](assets/fr/033.webp)



เลื่อนลงไปที่อินเทอร์เฟซหลัก จากนั้นคลิกที่ `+ OPEN CHANNEL`.



![Image](assets/fr/034.webp)



ป้อน `Node ID` ของโหนดที่คุณต้องการเปิดช่องทางด้วย ระบุจำนวนเงินที่คุณต้องการล็อคไว้ จากนั้นปรับค่าธรรมเนียมการทำธุรกรรมเปิดตามสถานะของตลาดค่าธรรมเนียมบนเชน แน่นอน ตรวจสอบให้แน่ใจว่าคุณมีเงินเพียงพอในพอร์ตโฟลิโอบนเชน LND ของคุณล่วงหน้า



เมื่อกำหนดพารามิเตอร์ทั้งหมดแล้ว ให้คลิกที่ปุ่ม `OPEN CHANNEL`



![Image](assets/fr/035.webp)



รอให้การทำธุรกรรมเปิดได้รับการยืนยันบนเชน ช่องแรกของคุณตอนนี้เปิดใช้งานอย่างเป็นทางการแล้ว: ขอแสดงความยินดี!



คุณสามารถเห็นได้ว่า ในขณะนี้ 100% ของสภาพคล่องของช่องทางอยู่ที่ฝั่งของฉัน: นั่นเป็นเรื่องปกติ เพราะฉันเป็นคนที่เปิดช่องทางนี้ เมื่อการชำระเงินและการกำหนดเส้นทางพัฒนาไป การกระจายนี้จะเปลี่ยนไป แต่ความจุรวมของช่องทางจะยังคงเหมือนเดิมเสมอ



![Image](assets/fr/036.webp)



ตอนนี้ที่คุณมีช่องทางแล้ว คุณสามารถทำการชำระเงิน Lightning ครั้งแรกได้ โดยมีเงื่อนไขว่าโหนดที่เลือกต้องเชื่อมต่อกับเครือข่ายอย่างถูกต้อง แน่นอนว่าในบทต่อๆ ไป เราจะมาดูวิธีการตั้งค่าที่สะดวกยิ่งขึ้นในการชำระใบแจ้งหนี้ Lightning จากมือถือของคุณ แต่สำหรับตอนนี้ ลองทำการชำระเงินครั้งแรกโดยตรงจาก LND ไปยัง Umbrel กันเถอะ



ในการทำเช่นนี้ ในส่วนของ `Lightning Wallet` ให้คลิกที่ปุ่ม `SEND` จากนั้นวางใบแจ้งหนี้ที่จะตั้งค่า



![Image](assets/fr/037.webp)



จำนวนเงินในใบแจ้งหนี้แสดงอยู่ ยืนยันการชำระเงินโดยคลิกที่ปุ่ม `SEND`



![Image](assets/fr/038.webp)



หากพบเส้นทางที่ถูกต้อง การชำระเงิน Lightning ครั้งแรกของคุณควรจะสำเร็จ



![Image](assets/fr/039.webp)



หากเราดูการกระจายของสภาพคล่องในช่องทาง เราจะเห็นว่าเพื่อนของฉันตอนนี้มี 5,002 sats อยู่ฝั่งของเขา ซึ่งสอดคล้องกับ 5,000 sats ของการชำระเงินที่ฉันเพิ่งทำ ซึ่งเขาได้ส่งต่อไปยังโหนดของผู้รับ ส่วน 2 sats ที่เพิ่มเติมนั้นคือค่าธรรมเนียมการส่งต่อที่ฉันจ่าย เนื่องจากผู้รับได้รับ 5,000 sats พอดี



![Image](assets/fr/040.webp)



เพื่อปรับปรุงความน่าเชื่อถือของการชำระเงินของเรา เห็นได้ชัดว่าจะต้องเปิดช่องทางอื่น ๆ ขึ้นอยู่กับวัตถุประสงค์ของเรา เราจะต้องหาวิธีทำให้สภาพคล่องที่เข้ามาพร้อมใช้งานเพื่อที่เราจะสามารถรับการชำระเงินบน Lightning ได้ นี่จะเป็นหัวข้อของส่วนถัดไป



# การจัดการโหนด Lightning ของคุณ


<partId>e27c3e1e-487b-4414-ad6b-d67bdb91c7c5</partId>



## กำหนดโปรไฟล์ผู้ดำเนินการโหนดของคุณ


<chapterId>d3b2e163-50f6-4d1d-a5fc-8fd177dfac76</chapterId>



ตอนนี้ที่โหนด Lightning ของคุณพร้อมใช้งานแล้ว ขั้นตอนต่อไปคือการกำหนดโปรไฟล์ผู้ค้า นี่เป็นการเลือกที่สำคัญ เนื่องจากมันจะกำหนดกลยุทธ์การเปิดช่องทาง ประเภทของเพื่อนที่คุณต้องการ และวิธีการจัดการสภาพคล่องของคุณ



บน Lightning สำหรับสิ่งนี้ที่จะทำงานได้ คุณจำเป็นต้องมีสภาพคล่องในทิศทางที่ถูกต้อง สภาพคล่องขาออกสอดคล้องกับความสามารถในการจ่ายของคุณ (sats ที่มีอยู่ในฝั่งของช่องทางของคุณ) สภาพคล่องขาเข้าสอดคล้องกับความสามารถในการรับของคุณ (sats ที่มีอยู่ในฝั่งของเพื่อนของคุณ) กล่าวอีกนัยหนึ่ง โปรไฟล์ของคุณสรุปลงในคำถามง่ายๆ: ส่วนใหญ่แล้ว sats ของคุณออกจากโหนดของคุณหรือเข้ามาในโหนดของคุณ?



### ผู้บริโภค



นี่คือโปรไฟล์ของผู้ใช้ส่วนใหญ่ คุณใช้โหนดของคุณเพื่อทำการชำระเงินผ่าน Lightning: เพื่อซื้อสินค้าและบริการ ชำระบิล ส่งทิป - กล่าวโดยสรุปคือการใช้ Lightning เป็นวิธีการชำระเงินที่รวดเร็ว ในทางกลับกัน คุณได้รับจาก Lightning เพียงเล็กน้อย หรือเพียงเล็กน้อยเท่านั้น เช่น การบริจาคเล็กน้อย การคืนเงินระหว่างเพื่อน หรือการชำระเงินขนาดเล็กไม่กี่ครั้ง



โปรไฟล์นี้จัดการได้ง่ายที่สุด เพราะความต้องการหลักของคุณคือความสามารถในการชำระเงิน ในทางปฏิบัติ หมายความว่าคุณต้องการสภาพคล่องขาออก เมื่อคุณเปิดช่องทางที่มีขนาดเหมาะสมกับโหนดที่เสถียรและเชื่อมต่อดีแล้ว การชำระเงินขาออกของคุณจะย้ายสภาพคล่องไปยังอีกด้านหนึ่งของช่องทางของคุณโดยอัตโนมัติ การเคลื่อนไหวนี้เองที่สร้างสภาพคล่องขาเข้าที่เหมาะสมตามธรรมชาติ ดังนั้น แม้ว่าคุณจะไม่ได้มองหาการรับเงินเป็นประจำ คุณก็ยังสามารถรับเงินได้เป็นครั้งคราวโดยไม่ต้องใช้กลยุทธ์ที่ซับซ้อน ดังนั้นคุณไม่จำเป็นต้องกังวลเกี่ยวกับสภาพคล่องขาเข้าของคุณ



ในหลักสูตร LNP202 นี้ เราจะมุ่งเน้นไปที่โปรไฟล์ผู้ดำเนินการโหนด "ผู้บริโภค" เนื่องจากเป็นโปรไฟล์ที่สอดคล้องกับผู้ใช้ Bitcoin เกือบทั้งหมดบน Lightning



### ผู้ค้าปลีก



ในทางหนึ่ง พ่อค้าเป็นสิ่งที่ตรงกันข้ามกับผู้บริโภค ที่นี่ วัตถุประสงค์หลักไม่ใช่การจ่าย แต่เป็นการเก็บรวบรวม ธุรกิจ ผู้ให้บริการ ร้านค้าออนไลน์ หรือจุดขายที่ยอมรับ Lightning จะได้รับการชำระเงินเข้ามามากมาย และทำการชำระเงินออกจากโหนดนี้ค่อนข้างน้อย



โปรไฟล์นี้มีความต้องการมากขึ้น เนื่องจากการชำระเงินที่ถูกปฏิเสธบน Lightning อาจหมายถึงการขายที่สูญเสียไป ดังนั้นความสำคัญจึงกลายเป็นสภาพคล่องขาเข้า หากโหนดของคุณไม่มีขาเข้าที่เพียงพอ ลูกค้าของคุณจะเห็นว่าการชำระเงินของพวกเขาล้มเหลว แม้ว่าพวกเขาจะมีเงินก็ตาม เพียงเพราะไม่มีเส้นทางที่จะนำสภาพคล่องมาหาคุณในทิศทางที่ถูกต้อง



ความท้าทายหลักสำหรับพ่อค้ายังมาจากการพัฒนาตามธรรมชาติของช่องทางด้วย หากสิ่งที่คุณทำคือการรับ ช่องทางของคุณจะค่อยๆ เต็มไปทางฝั่งของคุณ ดังนั้นคุณจำเป็นต้องมีกลไกในการรักษาและต่ออายุสภาพคล่องที่เข้ามาของคุณ



อย่างไรก็ตาม มีกรณีที่ง่ายกว่า: โปรไฟล์ผู้บริโภค/ผู้ค้าแบบผสม หากคุณเก็บเงินผ่าน Lightning แต่ก็ใช้จ่ายผ่าน Lightning ด้วย (ค่าใช้จ่ายทางธุรกิจ การชำระเงินให้กับซัพพลายเออร์ หรือแม้แต่ค่าใช้จ่ายส่วนตัว) การชำระเงินขาออกของคุณจะสร้างการรับขาเข้าใหม่โดยธรรมชาติ การจัดการจะราบรื่นขึ้น เนื่องจากการไหลเวียนของเงินจะชดเชยกัน และคุณมีความจำเป็นน้อยลงที่จะต้องใช้กลไกเทียมที่ออกแบบมาเพื่อฟื้นฟูสภาพคล่องขาเข้าเท่านั้น



### เราเตอร์



เราเตอร์เป็นโปรไฟล์เฉพาะ มันไม่ได้ใช้โหนด Lightning ของตัวเองในการชำระเงินหรือเก็บเงิน แต่ใช้ในการส่งต่อการชำระเงินของผู้อื่นและเก็บค่าธรรมเนียม วัตถุประสงค์คือการเป็นเส้นทางที่มีประโยชน์ เชื่อถือได้ และแข่งขันได้ในเชิงเศรษฐกิจภายในกราฟ Lightning



พูดตามตรง การเป็นเราเตอร์บน Lightning เป็นธุรกิจที่ซับซ้อนกว่าที่เห็น และการทำกำไรนั้นยากที่จะบรรลุได้ ประการแรก การเปิดและปิดช่องทางมีค่าใช้จ่ายสูงในด้านต้นทุนบนเชน เพื่อให้สามารถส่งข้อมูลได้อย่างมีประสิทธิภาพ คุณมักจะต้องปรับโครงสร้างทอพอโลยีของคุณ ทดสอบ ปิดช่องทางที่มีประสิทธิภาพต่ำ เปิดช่องทางใหม่ และปรับสมดุลสภาพคล่องของคุณเป็นประจำ ประการที่สอง การแข่งขันนั้นดุเดือด: โหนดขนาดใหญ่ที่มีชื่อเสียงมักจะครองส่วนแบ่งการจราจรจำนวนมาก และเป็นเรื่องยากที่จะสร้างฐานที่มั่นโดยไม่ต้องผูกมัดเงินทุนจำนวนมากในช่องทางที่มีขนาดเหมาะสม



มีข้อกำหนดในการดำเนินงานที่สูงมากเช่นกัน โหนดการกำหนดเส้นทางต้องมีความเสถียรอย่างยิ่ง ถูกตรวจสอบ สำรองข้อมูลอย่างเหมาะสม และจัดการอย่างเข้มงวด คุณต้องตรวจสอบประสิทธิภาพของช่องทาง ปรับนโยบายค่าธรรมเนียมของคุณ รักษาสภาพคล่องให้สมดุล จัดการเพื่อนร่วมงาน และแก้ไขปัญหาทางเทคนิคอย่างรวดเร็ว ระดับของการมีส่วนร่วมนี้อาจน่าสนใจในฐานะงานอดิเรกทางเทคนิคหรือเป็นการมีส่วนร่วมในโครงสร้างพื้นฐาน แต่ถ้าเป้าหมายของคุณคือการใช้ Lightning อย่างอธิปไตย การเข้าสู่การกำหนดเส้นทางเพื่อหาเงินมักไม่เกี่ยวข้อง นั่นคือเหตุผลที่หลักสูตร LNP202 นี้ไม่ได้ครอบคลุมโปรไฟล์นี้อย่างลึกซึ้ง



### ชัดเจนว่าตั้งตัวเองก่อนที่จะไปต่อ



หากคุณตรงตามโปรไฟล์ผู้บริโภค ความสำคัญของคุณจะเป็นการชำระเงินอย่างน่าเชื่อถือด้วยการจัดการที่ง่าย หากคุณเป็นพ่อค้า ความสำคัญของคุณจะเป็นการรับเงินสดโดยไม่ล้มเหลว ซึ่งหมายถึงกลยุทธ์การได้มาซึ่งสภาพคล่องขาเข้า หากคุณกำลังพิจารณาการกำหนดเส้นทาง คุณจำเป็นต้องมองว่าเป็นกิจกรรมที่ต้องการ ความไม่แน่นอนทางเศรษฐกิจ และใช้เวลามาก



การกำหนดโปรไฟล์นี้ในขณะนี้จะช่วยให้คุณหลีกเลี่ยงข้อผิดพลาดคลาสสิก: การใช้กลยุทธ์ช่องทางที่ออกแบบมาสำหรับผู้ค้า หรือเราเตอร์ เมื่อคุณเป็นเพียงผู้ใช้ที่ต้องการชำระเงินเท่านั้น



## การใช้ตัวจัดการโหนด Lightning


<chapterId>02eb4c09-d14b-4ff0-8b04-b90de3307d34</chapterId>



ในส่วนก่อนหน้าของหลักสูตรการฝึกอบรม LNP202 นี้ เราได้ใช้ส่วนติดต่อพื้นฐานของแอปพลิเคชัน `Lightning Node` บน Umbrel ส่วนติดต่อนี้เพียงพอสำหรับการดำเนินงานที่จำเป็น (การตรวจสอบยอดเงิน การดูการกระจายเงินสด การเปิดและปิดช่องทาง) แต่ถูกออกแบบให้เรียบง่ายมาก ความเรียบง่ายนี้จำกัดตัวเลือกที่มีอยู่และไม่ให้การเข้าถึงคุณสมบัติขั้นสูงหลายอย่างของโหนด LND ของคุณ ด้วยเหตุนี้ เราจะใช้ซอฟต์แวร์การจัดการโหนด Lightning ที่ครอบคลุมมากขึ้นอีกตัวหนึ่งในตอนนี้



ซอฟต์แวร์เพิ่มเติมนี้จะเป็นเพียงอินเทอร์เฟซการจัดการเสริมสำหรับโหนดของคุณ ซึ่งหมายความว่าคุณสามารถใช้ `Lightning Node` อินเทอร์เฟซควบคู่กันไปได้ และแม้กระทั่งใช้หลายๆ อินเทอร์เฟซที่แตกต่างกันหากคุณต้องการ เหล่านี้เป็นเพียงวิธีการที่แตกต่างกันในการโต้ตอบกับโหนด Lightning เดียวกัน



ในบรรดาโปรแกรมซอฟต์แวร์ที่รู้จักกันดี ได้แก่:




- [Alby Hub](https://albyhub.com/);
- [Ride The Lightning](https://www.ridethelightning.info/);

### ใช้ ThunderHub



ThunderHub เรียนรู้ได้ค่อนข้างง่าย เมนูทั้งหมดสามารถเข้าถึงได้จากคอลัมน์ด้านซ้ายของอินเทอร์เฟซ สรุปได้ดังนี้ว่าแต่ละเมนูทำอะไรบ้าง:




- หน้าแรก: ภาพรวมของโหนด, ยอดคงเหลือและการดำเนินการด่วน;
- แดชบอร์ด: แดชบอร์ดที่ปรับแต่งได้ด้วยวิดเจ็ตและเมตริก;
- เพื่อน: ดูและจัดการการเชื่อมต่อกับโหนด Lightning อื่น ๆ;
- channels': การจัดการช่องทางอย่างสมบูรณ์ (สภาพคล่อง, ค่าธรรมเนียม, การปิด, ฯลฯ);
- rebalance": เครื่องมือสำหรับปรับสมดุลช่องทางผ่านการชำระเงินแบบวงกลม;
- ธุรกรรม: ประวัติการชำระเงิน Lightning ที่ส่งและรับ;
- forwards`: สถิติการกำหนดเส้นทางและค่าใช้จ่าย generated โดยโหนดของคุณ;
- `Chain`: Bitcoin พอร์ตโฟลิโอออนเชน (UTXOs และธุรกรรม);
- การรวม gW-201 สำหรับการตรวจสอบและการสำรองข้อมูล;
- `Tools`: เครื่องมือขั้นสูง (การสำรองข้อมูล, รายงาน, มาการูน, ลายเซ็น, ฯลฯ);
- สลับ: การสลับ Lightning/onchain ผ่าน Boltz;
- `Stats`: สถิติและประสิทธิภาพโดยรวมของโหนด Lightning ของคุณ



ด้วยชุดฟังก์ชันนี้ คุณมีเครื่องมือทั้งหมดที่จำเป็นในการจัดการ Lightning node ของคุณอย่างมีประสิทธิภาพ ไม่จำเป็นต้องเชี่ยวชาญทุกตัวเลือกในรายละเอียดทันที: เราจะสำรวจพวกมันอย่างค่อยเป็นค่อยไปตลอดหลักสูตรนี้ อย่างไรก็ตาม หากคุณต้องการทำความเข้าใจกับซอฟต์แวร์ในเชิงลึกมากขึ้น ลองดูที่บทแนะนำ ThunderHub ของเรา:



https://planb.academy/tutorials/node/lightning-network/thunderhub-16909a39-2484-408e-a118-4e34e249bb9a

เมนูที่เราสนใจมากที่สุดที่นี่คือ `Channels` มันให้มุมมองรายละเอียดของช่องทางทั้งหมดในโหนดของคุณ พร้อมกับการกระจายสภาพคล่อง โดยเฉพาะอย่างยิ่ง คุณสามารถดูได้ว่าช่องทางใดเปิดอยู่ใน `Open` ช่องทางใดกำลังรอการเปิดหรือปิดใน `Pending` และช่องทางใดที่ปิดไปแล้วใน `Closed`

## การรับสภาพคล่องขาเข้า


<chapterId>b740c656-a897-4d95-af4b-116b718447cd</chapterId>



ตามที่คุณเห็น การมีสภาพคล่องขาออกเพื่อทำการชำระเงินบน Lightning ไม่ได้ซับซ้อนมากนัก เพียงแค่เปิดช่องทางด้วยความตั้งใจของคุณเองไปยังโหนดอื่น ๆ และเริ่มส่ง sats ในทางกลับกัน การมีสภาพคล่องขาเข้าเพื่อรับการชำระเงินบน Lightning นั้นซับซ้อนกว่า เนื่องจากคุณต้องการให้โหนดอื่นเปิดช่องทางมาหาคุณ หรือคุณต้องย้ายสภาพคล่องด้วยตัวเองโดยการทำการชำระเงิน



หากคุณใช้โปรไฟล์ "ผู้บริโภค" โดยทั่วไปแล้วไม่จำเป็นต้องกังวลเกี่ยวกับสภาพคล่องที่เข้ามา การใช้โหนด Lightning ของคุณจะเน้นไปที่การชำระเงินเป็นหลัก แทนที่จะเป็นการรับเงินสด โดยการทำการชำระเงินผ่าน Lightning เพียงไม่กี่ครั้ง คุณจะสร้างสภาพคล่องที่เข้ามาได้ตามธรรมชาติเมื่อเวลาผ่านไป



ในทางกลับกัน หากคุณมีโปรไฟล์ "พ่อค้า" สถานการณ์จะกลับกัน: คุณจะเก็บเงินเป็นหลักและจ่ายเงินเพียงเล็กน้อย ดังนั้นคุณไม่สามารถพึ่งพาการชำระเงินของคุณเองเพื่อสภาพคล่องที่เข้ามาได้ จึงจำเป็นที่โหนด Lightning อื่น ๆ จะต้องเปิดช่องทางไปยังโหนดของคุณ แต่จะทำได้อย่างไร? คุณจะทำให้นักปฏิบัติการอื่น ๆ ผูกทุนให้คุณได้อย่างไร? นั่นคือสิ่งที่เราจะสำรวจในบทนี้

หากคุณเป็นพ่อค้า บริการประเภทนี้เหมาะสมอย่างยิ่ง เนื่องจากช่วยให้คุณสามารถรับสภาพคล่องขาเข้าจำนวนมากจากโหนดที่เชื่อถือได้อย่างรวดเร็ว รับประกันว่าลูกค้าของคุณจะสามารถชำระเงินให้คุณได้โดยไม่มีปัญหา ในทางกลับกัน หากคุณเป็นบุคคลทั่วไป หรือหากคุณไม่ต้องการจ่ายค่าสภาพคล่องขาเข้า ก็มีโซลูชันฟรีเช่นกัน มาดูกันเถอะ



### สภาพคล่องขาเข้าที่ร่วมมือกัน



หากคุณไม่ใช่นักเทรด แต่ยังคงต้องการสภาพคล่องที่เข้ามา (เช่น เพื่อเตรียมพร้อมโหนดของคุณในช่วงเริ่มต้น ขณะที่คุณรอการชำระเงินบางส่วน) คุณไม่จำเป็นต้องจ่ายเงินสำหรับมัน หนึ่งในวิธีแก้ปัญหาที่ฉันชอบคือการร่วมมือกับผู้ดำเนินการโหนดคนอื่น ๆ ที่ต้องการสภาพคล่องขาเข้าเช่นกัน เพื่อเปิดช่องทาง Lightning ให้กันและกัน ด้วยวิธีนี้ การเปิดช่องทางจะทำให้คุณมีสภาพคล่องขาออก ในขณะเดียวกันก็ให้สภาพคล่องขาเข้ากับคุณโดยไม่เสียค่าใช้จ่าย (ยกเว้นค่าธรรมเนียมบนเชนสำหรับการเปิด)



คุณสามารถจัดการเรื่องนี้โดยตรงกับผู้ใช้บิทคอยน์คนอื่น ๆ ได้ อย่างไรก็ตาม มีแพลตฟอร์มที่อุทิศให้กับการเปิดแบบวงกลมประเภทนี้: [Lightning Network +](https://lightningnetwork.plus/) หลักการคือไม่เปิดช่องทางสองช่องระหว่างคนเดียวกัน แต่จัดตั้งการเปิดแบบวงกลมที่มีผู้เข้าร่วมอย่างน้อยสามคนหรือมากกว่านั้น



ลองมาดูตัวอย่างเพื่อทำความเข้าใจว่า Lightning Network + ทำงานอย่างไร:




- ผู้ดำเนินการโหนดชื่อ `A` ประกาศว่าเขาต้องการเปิดช่อง sats จำนวน 1 ล้านกับคนอื่นอีกสองคน;
- โฆษณาจะยังคงใช้งานได้จนกว่าจะมีผู้เข้าร่วมเพิ่มเติม;
- ต่อมา ผู้ปฏิบัติการสองคน `B` และ `C` เข้าร่วมการประกาศโหนด `A` ตอนนี้สามเหลี่ยมเสร็จสมบูรณ์แล้ว และสามารถเริ่มขั้นตอนการเปิดได้
- โหนด `A` เปิดช่องทางไปยังโหนด `B`;
- โหนด `B` เปิดช่องทางไปยังโหนด `C`;
- โหนด `C` เปิดช่องทางไปยังโหนด `A`



เมื่อสิ้นสุดการดำเนินการ แต่ละโหนดจะมีสภาพคล่องขาออก 1 ล้าน sats และสภาพคล่องขาเข้า 1 ล้าน sats แผนการนี้สามารถขยายไปยังผู้เข้าร่วมสี่หรือห้าคนได้



แน่นอนว่าไม่มีกลไกทางเทคนิคที่จะรับประกันว่าผู้เข้าร่วมจะเปิดช่องทางที่พวกเขาได้ให้คำมั่นว่าจะสร้างขึ้นจริง นั่นเป็นเหตุผลที่แพลตฟอร์มได้จัดตั้งระบบชื่อเสียงขึ้นมา โดยอิงจากรีวิวเชิงบวกเมื่อผู้ดำเนินการปฏิบัติตามคำมั่นสัญญาของพวกเขา และในกรณีที่เลวร้ายที่สุด หากคุณพบกับคนที่ไม่ให้ความร่วมมือ คุณจะไม่สูญเสียเงินใดๆ: คุณเพียงแค่พลาดโอกาสในการรับสภาพคล่องที่เข้ามาเท่านั้น



โซลูชันนี้เหมาะสมอย่างยิ่งสำหรับโปรไฟล์ "ผู้บริโภค" เนื่องจากช่วยให้คุณได้รับสภาพคล่องขาเข้าโดยไม่เสียค่าใช้จ่าย ในขณะที่เชื่อมต่อโหนดของคุณกับผู้ประกอบการรายย่อยอื่น ๆ ในทางกลับกัน หากคุณเป็นนักเทรด วิธีการนี้มักจะไม่เกี่ยวข้อง: ทุก ๆ sat ของสภาพคล่องขาเข้าต้องการการบล็อก sat ของสภาพคล่องขาออก และความต้องการสภาพคล่องขาเข้าจำนวนมากของคุณจะเกี่ยวข้องกับการผูกมัดเงินทุนมากเกินไป



ในการใช้ Lightning Network + คุณมีสองตัวเลือก: ใช้แอปพลิเคชันที่รวมอยู่ใน Umbrel หรือใช้เว็บไซต์แบบคลาสสิกและสร้างบัญชีโดยการเข้าสู่ระบบจากโหนดของคุณ ฉันแนะนำตัวเลือกหลัง เนื่องจากแอปพลิเคชันที่รวมอยู่นั้นไม่ได้มีฟังก์ชันทั้งหมดที่มีอยู่ครบถ้วน



ไปที่เว็บไซต์ [Lightning Network +](https://lightningnetwork.plus/) และคลิกที่ปุ่ม `Login` ที่มุมขวาบนของอินเทอร์เฟซ



![Image](assets/fr/061.webp)



ในการยืนยันตัวตนของคุณ คุณจำเป็นต้องลงนามในข้อความที่ให้ไว้โดยใช้กุญแจส่วนตัวของโหนด Lightning ของคุณ ด้วย ThunderHub การดำเนินการนี้เป็นเรื่องง่ายมาก เริ่มต้นโดยการคัดลอกข้อความที่แสดงโดย LN+



![Image](assets/fr/062.webp)



ใน ThunderHub ไปที่แท็บ `Tools` จากนั้นคลิกที่ปุ่ม `Sign` ในส่วน `Messages`

![Image](assets/fr/064.webp)



ThunderHub จากนั้นลงนามข้อความนี้โดยใช้กุญแจส่วนตัวของโหนดของคุณ คัดลอกลายเซ็น generated



![Image](assets/fr/065.webp)



วางลายเซ็นนี้ลงในช่องที่สอดคล้องบนเว็บไซต์ LN+ จากนั้นคลิก `Sign in`.



![Image](assets/fr/066.webp)



คุณเชื่อมต่อกับ LN+ ด้วยโหนด Lightning ของคุณแล้ว กระบวนการนี้ช่วยให้ LN+ ยืนยันว่าคุณเป็นเจ้าของที่ถูกต้องของโหนดที่คุณกำลังอ้างสิทธิ์บนแพลตฟอร์มของพวกเขา



![Image](assets/fr/067.webp)



หากคุณต้องการ คุณสามารถปรับแต่งโปรไฟล์ LN+ ของคุณได้ เช่น โดยการเพิ่มประวัติย่อ



ในการเข้าร่วมการเปิดช่องทางวงกลมครั้งแรกของคุณ ให้ไปที่เมนู `Swaps` ที่ด้านบนของอินเทอร์เฟซ ที่นี่คุณจะพบกับการแลกเปลี่ยนทั้งหมดที่มีอยู่ ขึ้นอยู่กับลักษณะของโหนดของคุณ



![Image](assets/fr/068.webp)



หากต้องการเข้าร่วมข้อเสนอการแลกเปลี่ยนที่มีอยู่ เพียงคลิกที่มันและลงทะเบียน อย่างไรก็ตาม บน LN+ ผู้สร้างการแลกเปลี่ยนอาจกำหนดเงื่อนไขบางประการสำหรับผู้เข้าร่วม เช่น จำนวนช่องขั้นต่ำหรือความจุของโหนดรวมขั้นต่ำ ดังนั้นจึงเป็นไปได้ โดยเฉพาะในช่วงแรกๆ ที่จะมีข้อเสนอไม่กี่รายการที่พร้อมใช้งานสำหรับโหนดของคุณ ในกรณีของฉัน แม้ว่าจะมีช่องเปิดอยู่ไม่กี่ช่อง แต่ก็ไม่มีข้อเสนอใดที่พร้อมใช้งานสำหรับโหนดของฉัน ดังนั้นฉันจึงสร้างการแลกเปลี่ยนของตัวเอง และคุณก็สามารถทำเช่นเดียวกันได้หากคุณอยู่ในสถานการณ์เดียวกัน



คลิกที่ `Start Liquidity Swap` จากนั้นป้อนพารามิเตอร์ข้อเสนอของคุณ:

- ระบุจำนวนช่องทางที่จะเปิด (ตรวจสอบให้แน่ใจว่าคุณมีอย่างน้อยจำนวนนี้ใน onchain wallet ของคุณ);
- กำหนดเวลาการเปิดช่องทาง นี่คือช่วงเวลาที่ผู้เข้าร่วมตกลงที่จะไม่ปิดช่องทาง;
- ทางด้านขวา คุณสามารถตั้งค่าข้อจำกัดสำหรับผู้เข้าร่วม: จำนวนช่องขั้นต่ำ ความจุรวมขั้นต่ำ และประเภทของการเชื่อมต่อที่ยอมรับได้



เมื่อกำหนดพารามิเตอร์ทั้งหมดแล้ว ให้คลิกที่ปุ่ม `Start Liquidity Swap`

![Image](assets/fr/084.webp)

![Image](assets/fr/085.webp)

![Image](assets/fr/086.webp)

## เชื่อมต่อมือถือ wallet ผ่าน Tailscale


<chapterId>5fefb222-3f50-4f9d-a170-2ea628be4437</chapterId>



นั่นแหละ ตอนนี้คุณมีโหนด Lightning ที่เชื่อมต่ออย่างดีแล้ว ทั้งในด้านสภาพคล่องขาเข้าและขาออก ดังนั้นคุณพร้อมที่จะใช้โหนด Lightning ของคุณในชีวิตจริงแล้ว จนถึงตอนนี้ เราได้ใช้ส่วนต่อประสานโดยตรงบน Umbrel เสมอ ไม่ว่าจะเป็นแอปพลิเคชัน `Lightning Node` หรือส่วนต่อประสาน `ThunderHub` เครื่องมือเหล่านี้ทำงานสำหรับการส่งและรับการชำระเงิน แต่ชัดเจนว่าไม่ได้ถูกออกแบบมาให้เหมาะสมสำหรับการชำระเงิน Lightning ในชีวิตประจำวัน ส่วนต่อประสานถูกออกแบบมาให้ใช้บนคอมพิวเตอร์ ซึ่งไม่สะดวกบนสมาร์ทโฟน และยังต้องการการเชื่อมต่อกับเครือข่ายเดียวกันเพื่อทำงานได้อย่างถูกต้อง (แม้ว่าจะสามารถเชื่อมต่อจากระยะไกลผ่าน Tor ได้ในทางเทคนิค)



ในทางปฏิบัติ สิ่งที่เรากำลังมองหาในฐานะผู้ใช้บิทคอยน์คืออินเทอร์เฟซ Lightning wallet แบบคลาสสิกบนสมาร์ทโฟน: ความสามารถในการสแกนใบแจ้งหนี้ผ่าน QR code และอินเทอร์เฟซที่เรียบง่ายสำหรับการชำระเงินและการถอนเงิน sats นี่คือสิ่งที่เราจะดำเนินการในบทนี้และบทถัดไป แนวคิดทั่วไปคือการมีแอปพลิเคชัน Lightning wallet บนมือถือในสมาร์ทโฟนของคุณ ซึ่งสามารถใช้งานได้จากทุกที่ (ไม่ใช่แค่เครือข่ายท้องถิ่นของคุณ) แต่ในเบื้องหลังจะพึ่งพาโหนด Lightning ของคุณเองในการส่งและรับการชำระเงิน



### โซลูชันสำหรับการเชื่อมต่อลูกค้ามือถือมีอะไรบ้าง?



วันนี้มีหลายวิธีในการทำเช่นนี้ ทั้งในแง่ของแอปพลิเคชันมือถือและประเภทของการเชื่อมต่อระหว่างโหนดของคุณกับแอปพลิเคชันนี้ โหมดการเชื่อมต่อหลักสามโหมดคือ:




- ผ่าน ***Tor***;
- ผ่าน VPN ***Tailscale***;
- ผ่าน ***Nostr Wallet Connect***.



ไม่กี่ปีที่ผ่านมา ฉันเคยเชื่อมต่อผ่าน ***Tor*** แต่ฉันก็หยุดอย่างรวดเร็ว: จำนวนความล้มเหลวและความล่าช้าในการสื่อสารมากเกินไป ในทางทฤษฎีมันใช้งานได้ แต่ในทางปฏิบัติ ประสบการณ์ของผู้ใช้นั้นเลวร้ายมาก ดังนั้นฉันจึงไม่แนะนำวิธีนี้



ทางเลือกที่ฉันเลือกใช้คือการใช้ ***Tailscale*** VPN เพื่อให้แน่ใจว่าการสื่อสารระหว่างแอปพลิเคชันมือถือและโหนดเป็นไปอย่างราบรื่น วิธีนี้ทำงานได้ดีมาก: แม้ในเครือข่ายมือถือที่มีความเร็วต่ำ การชำระเงินของฉันก็ผ่านไปได้โดยไม่มีปัญหา นี่คือวิธีที่ฉันจะนำเสนอเป็นอันดับแรกในบทนี้ โดยใช้แอปพลิเคชัน Zeus



ในบทถัดไป เราจะพิจารณาอีกหนึ่งวิธีแก้ปัญหาที่ใหม่กว่า ซึ่งก็ใช้งานได้ดีมากเช่นกัน: ***Nostr Wallet Connect***. คราวนี้ เราจะใช้แอปพลิเคชัน Alby Go เพื่อเสนอทางเลือกให้คุณ แม้ว่า Zeus จะสามารถใช้งานร่วมกับ NWC ได้หากคุณต้องการ



### การติดตั้งและกำหนดค่า Tailscale



สำหรับวิธีแรกนี้ เราจะต้องใช้ Tailscale นี่คือโซลูชัน VPN ที่อิงจาก WireGuard ซึ่งช่วยให้คุณเชื่อมต่ออุปกรณ์ที่กระจายอยู่ทั่วอินเทอร์เน็ตได้อย่างปลอดภัย ในขณะที่จัดการการเข้ารหัส การตรวจสอบสิทธิ์ และการข้าม NAT โดยอัตโนมัติ พูดง่ายๆ ก็คือ มันเหมือนกับว่าอุปกรณ์ทั้งหมดของคุณเชื่อมต่อกับ LAN เดียวกันกับเราเตอร์ของคุณ แม้ว่าพวกมันจะอยู่ที่ไหนก็ได้บนอินเทอร์เน็ตก็ตาม



ในการใช้งาน ก่อนอื่นให้สร้างบัญชี ไปที่เว็บไซต์ Tailscale จากนั้นคลิกที่ปุ่ม `Get Started`



![Image](assets/fr/071.webp)



จากนั้นเลือกผู้ให้บริการข้อมูลประจำตัวสำหรับบัญชี Tailscale ของคุณ ส่วนตัวแล้ว ฉันใช้หนึ่งในบัญชี GitHub ของฉันเพื่อลงชื่อเข้าใช้



![Image](assets/fr/072.webp)



เมื่อคุณเข้าสู่ระบบแล้ว คุณจะถูกถามคำถามสองสามข้อเกี่ยวกับการใช้งานของคุณ ตอบคำถามเหล่านั้นโดยย่อเพื่อดำเนินการต่อ



![Image](assets/fr/073.webp)



Tailscale จากนั้นเสนอให้ติดตั้งไคลเอนต์บนเครื่องของคุณ สำหรับตอนนี้ นั่นไม่ใช่สิ่งที่เราสนใจ: ไปที่ Umbrel โดยตรงและติดตั้งแอปพลิเคชัน Tailscale จาก App Store



![Image](assets/fr/074.webp)



เมื่อเปิดแอปพลิเคชัน ให้คลิกที่ `Log In` จากนั้นทำตามขั้นตอนการยืนยันตัวตน โดยใช้วิธีเดียวกับที่คุณใช้เมื่อตอนสร้างบัญชีของคุณ



![Image](assets/fr/075.webp)



คลิก `Connect` เพื่อยืนยัน Umbrel ของคุณเชื่อมต่อกับเครือข่าย VPN ของคุณแล้ว



![Image](assets/fr/076.webp)



จากนั้นดาวน์โหลดแอปพลิเคชัน Tailscale ลงในสมาร์ทโฟนของคุณและเข้าสู่ระบบโดยใช้บัญชีเดียวกัน โปรดทราบ: บน Android ไม่สามารถใช้ VPN สองตัวพร้อมกันได้ สำหรับ Tailscale ที่จะทำงาน คุณจะต้องปิดการใช้งาน VPN อื่นที่กำลังใช้งานอยู่ นอกจากนี้ ทุกครั้งที่คุณต้องการใช้ Lightning node ผ่าน Zeus คุณจะต้องเปิดใช้งาน Tailscale VPN มิฉะนั้นการเชื่อมต่อจะไม่ถูกสร้างขึ้น



![Image](assets/fr/077.webp)



บนไซต์ Tailscale ตอนนี้เมื่อมีลูกค้าอย่างน้อยสองรายเชื่อมต่อแล้ว คุณสามารถเข้าถึงคอนโซลการจัดการพร้อมรายการอุปกรณ์ทั้งหมดที่เชื่อมต่อกับเครือข่ายและที่อยู่ IP ของ Tailscale ของพวกเขาได้



![Image](assets/fr/078.webp)



### เชื่อมต่อ Zeus



ติดตั้งแอปพลิเคชัน Zeus บนโทรศัพท์ของคุณ เมื่อเปิดขึ้นมา ให้เลือก `Advanced Setup` จากนั้นเลือก `Create or connect a wallet`.



![Image](assets/fr/079.webp)



ในส่วน `Wallet interface` ให้เลือก `LND (REST)`. จากนั้นป้อนที่อยู่ Tailscale ของ Umbrel ของคุณ ซึ่งคุณสามารถหาได้จากแดชบอร์ด Tailscale ของคุณหรือโดยตรงในแอปพลิเคชัน Tailscale บน Umbrel. สำหรับพอร์ต ให้ป้อน `8080`.



![Image](assets/fr/080.webp)



จากนั้น Zeus ขอให้คุณจัดหา `Macaroon` นี่คือการอนุญาต token ซึ่งช่วยให้คุณสามารถกำหนดสิทธิ์ที่มอบให้กับแอปพลิเคชัน (ในกรณีนี้คือ Zeus) เพื่อโต้ตอบกับโหนด Lightning ของคุณได้อย่างแม่นยำ สามารถ generate macaroon จาก ThunderHub ได้ในเมนู `Tools` เมนูย่อย `Bakery` แต่สำหรับวัตถุประสงค์นี้ วิธีที่ง่ายที่สุดคือดึงข้อมูลโดยตรงจากแอปพลิเคชัน `Lightning Node`



คลิกที่จุดเล็ก ๆ สามจุดที่มุมขวาบนของอินเทอร์เฟซ จากนั้นคลิกที่ `Connect Wallet` เลือก `REST (Local Network)` จากนั้นคุณจะสามารถคัดลอก macaroon ที่มีสิทธิ์ที่เหมาะสมได้



![Image](assets/fr/081.webp)



วางลงในช่องที่สอดคล้องกันใน Zeus จากนั้นคลิกที่ปุ่ม `SAVE WALLET CONFIG`



![Image](assets/fr/082.webp)



คุณสามารถเข้าถึงโหนด Lightning ของคุณจากแอป Zeus ได้แล้ว ซึ่งหมายความว่าคุณสามารถ generate ใบแจ้งหนี้เพื่อรับการชำระเงินโดยตรงบนโหนด Lightning ของคุณจากสมาร์ทโฟนของคุณ และยังสามารถชำระใบแจ้งหนี้ Lightning ได้ทุกที่ที่คุณอยู่



![Image](assets/fr/083.webp)



เคล็ดลับ: Tailscale ไม่จำกัดเฉพาะการใช้โหนด Lightning ของคุณจากระยะไกลเท่านั้น มันยังช่วยให้คุณเข้าถึงเครื่องมือทั้งหมดของ Umbrel จากซอฟต์แวร์อื่น ๆ ได้ แม้จากระยะไกล ตัวอย่างเช่น คุณสามารถใช้ที่อยู่ IP ของ Tailscale ของ Umbrel เพื่อเชื่อมต่อโหนด Bitcoin ของคุณ (ผ่าน Electrs หรือ Fulcrum) ไปยัง Sparrow Wallet โดยไม่ต้องผ่าน Tor อีกครั้งที่นี่ช่วยหลีกเลี่ยงความช้าซึ่งเป็นลักษณะเฉพาะของ Tor เพียงแค่ติดตั้งไคลเอนต์ Tailscale บนคอมพิวเตอร์ของคุณและเชื่อมต่อกับเครือข่ายของคุณ



https://planb.academy/tutorials/computer-security/communication/tailscale-9acbd7de-04d9-40f6-ab80-35f0dfedb632

ในบทถัดไป เราจะพิจารณาวิธีการที่มีประสิทธิภาพเท่าเทียมกันในการเชื่อมต่อไคลเอนต์มือถือกับโหนด Lightning ของคุณ: Nostr Wallet Connect เราจะใช้แอปพลิเคชันที่แตกต่างจาก Zeus (แม้ว่า Zeus จะเข้ากันได้กับ NWC ด้วย) นั่นคือ Alby Go



## เชื่อมต่อมือถือ wallet ผ่าน NWC


<chapterId>f5c97e43-e66e-4ba3-bcc9-fee1a04fc7f4</chapterId>



หากคุณยังไม่มั่นใจกับการเชื่อมต่อ Tailscale หรือหากการจัดการ VPN คู่ดูเหมือนจะยุ่งยากเกินไป บทนี้แนะนำวิธีอื่นในการใช้ไคลเอนต์มือถือระยะไกลเพื่อชำระเงินและรับ sats ผ่านโหนด Lightning ของคุณ: ***Nostr Wallet Connect***.



สำหรับตัวอย่างนี้ เราจะใช้แอปพลิเคชันมือถือ Alby Go ซึ่งออกแบบมาอย่างดีและเรียนรู้ได้ง่ายมาก อย่างไรก็ตาม คุณยังสามารถใช้ Zeus หรือแอปพลิเคชันมือถืออื่น ๆ ที่เข้ากันได้กับ NWC ได้อีกด้วย คุณจะพบรายชื่อแอปพลิเคชันที่เข้ากันได้ใน [ที่เก็บ GitHub `awesome-nwc`](https://github.com/getAlby/awesome-nwc)



### Nostr Wallet Connect ทำงานอย่างไร?



Nostr Wallet Connect เป็นโปรโตคอลมาตรฐานที่อนุญาตให้แอปพลิเคชันหรือเว็บไซต์สามารถกระตุ้นการทำงานบนโหนด Lightning ระยะไกลได้ โดยไม่ต้องสร้างการเชื่อมต่อเครือข่ายโดยตรงกับโหนดนั้น (ไม่มี API LND เปิดเผย, ไม่มี VPN, ไม่มีบริการ `.onion`...) NWC กำหนดวิธีที่แอปพลิเคชันสร้างเจตนา (เช่น `pay_intece`) และรับผลลัพธ์



มันทำงานได้อย่างง่ายดาย คุณเริ่มต้นเซสชันโดยการสแกน QR โค้ดหรือผ่าน deeplink `nostr+walletconnect:` สตริงนี้ประกอบด้วยพารามิเตอร์เซสชันและความลับการอนุญาต จากนั้นเมื่อแอปพลิเคชันต้องการชำระเงิน มันจะทำการจัดลำดับคำขอ เข้ารหัส และเผยแพร่เป็นเหตุการณ์บน Nostr รีเลย์ โหนดจะอ่านเหตุการณ์บนรีเลย์ ถอดรหัส ตรวจสอบว่าผู้เขียนได้รับอนุญาตสำหรับเซสชันนี้ ดำเนินการชำระเงิน จากนั้นส่งกลับการตอบสนองที่เข้ารหัส (สำเร็จพร้อม preimage หรือข้อผิดพลาด) รีเลย์ทำหน้าที่เป็นเพียงตัวกลางในการขนส่ง: ไม่สามารถอ่านเนื้อหาได้ แต่สามารถสังเกตเวลาที่เกิดและความถี่ของคำขอได้



เมื่อเปรียบเทียบกับการเชื่อมต่อผ่าน Tailscale หรือ Tor ข้อได้เปรียบหลักของ NWC คือโหนดของคุณไม่สามารถเข้าถึงได้โดยตรงจากภายนอก ซึ่งทำให้การใช้งานบนมือถือง่ายขึ้นมาก: โหนดของคุณไม่จำเป็นต้องยอมรับการเชื่อมต่อขาเข้า เพียงแค่ต้องสามารถสื่อสารกับรีเลย์ได้ ในทางกลับกัน คุณได้แนะนำการพึ่งพาการทำงานบนรีเลย์ Nostr: หากพวกมันไม่พร้อมใช้งาน ประสบการณ์จะลดลง นอกจากนี้ แม้ว่าข้อความจะถูกเข้ารหัส รีเลย์ยังสามารถสังเกตข้อมูลเมตาระดับกิจกรรมบางอย่างได้



ความแตกต่างในรูปแบบความปลอดภัยก็สำคัญเช่นกัน ด้วย Tailscale หรือ Tor คุณเปิดเผยการเข้าถึงโดยตรงไปยังโหนดของคุณ (ผ่านทาง API หรือ LND) ซึ่งได้รับการปกป้องด้วยความลับที่มีความอ่อนไหวสูง สิ่งนี้มีพลังมากเพราะคุณสามารถจัดการทุกอย่างได้ แต่ก็เป็นพื้นผิวการโจมตีในชั้นที่ต่ำกว่าเช่นกัน ด้วย NWC การเข้าถึงจะเป็นแบบแอปพลิเคชันมากขึ้น: คุณมอบหมายเซสชัน token ที่อนุญาตเฉพาะการกระทำบางอย่างเท่านั้น



### ติดตั้ง Alby Hub บนโหนด Lightning ของคุณ

## อายุการใช้งานแบตเตอรี่ยาวนานบน Lightning


<chapterId>691a0942-b46d-482a-8fbc-fe19b3814992</chapterId>



เราได้มาถึงจุดสิ้นสุดของหลักสูตรปฏิบัติ LNP202 นี้แล้ว ตอนนี้คุณมีพื้นฐานที่จำเป็นในการใช้ Lightning Network อย่างมีอำนาจอธิปไตย: คุณเข้าใจบทบาทที่แท้จริงของโหนด, การแลกเปลี่ยนของวิธีการต่าง ๆ, และคุณได้ตั้งค่าอินสแตนซ์ LND บน Umbrel พร้อมกับกลยุทธ์การสำรองข้อมูลและการป้องกันที่สม่ำเสมอแล้ว คุณยังได้เปิดช่องทางแรกของคุณ, เรียนรู้วิธีจัดการสภาพคล่อง, เพื่อทำให้การชำระเงินของคุณเชื่อถือได้ในชีวิตประจำวัน



จากมุมมองการดำเนินงาน โหนดของคุณควรเข้าสู่จังหวะการบำรุงรักษาในขณะนี้ สิ่งสำคัญคือการตรวจสอบมัน (เวลาทำงาน, การซิงโครไนซ์, สถานะช่องทาง, ความล้มเหลวในการชำระเงิน ฯลฯ), ใช้การอัปเดตที่เสนอโดย Umbrel เมื่อมีเวอร์ชันที่เสถียร และตรวจสอบเป็นระยะว่าการสำรองข้อมูลและการกำหนดค่าหอคอยเฝ้าระวังของคุณยังคงใช้งานอยู่



เมื่อพูดถึงช่องทาง ให้ใช้วิธีการที่เป็นจริง: เก็บช่องทางที่มีประโยชน์ต่อคุณ ปิดช่องทางที่ไม่เคลื่อนไหวถาวรหรือเกี่ยวข้องกับเพื่อนที่ไม่เสถียร และค่อยๆ จัดสรรเงินทุนของคุณไปสู่โครงสร้างที่แข็งแกร่งยิ่งขึ้น



**หนึ่งในข้อผิดพลาดที่พบบ่อยที่สุดในขั้นตอนนี้คือการจัดสรรเงินทุนมากเกินไปให้กับโหนด Lightning ของคุณ โปรดจำไว้ว่าการรักษาความปลอดภัยของโหนด Lightning ของคุณนั้นน้อยกว่าฮาร์ดแวร์ wallet มาก และการที่เงินทุนที่ผูกพันกับช่องของคุณจะพร้อมใช้งานนั้นขึ้นอยู่กับกลไกการสำรองข้อมูลที่ยังคงไม่สมบูรณ์ ดังนั้นจึงเป็นเรื่องสำคัญมากที่จะต้องรักษาจำนวนเงินที่เหมาะสม ซึ่งคุณสามารถยอมรับการสูญเสียได้ในกรณีที่เกิดปัญหา และควรเก็บ sats ส่วนใหญ่ของคุณไว้ในฮาร์ดแวร์ wallet ที่อยู่บนเชนเสมอ



เท่าที่เกี่ยวข้องกับเครื่องมือ ฉันขอแนะนำให้คุณคงความอยากรู้อยากเห็นและใส่ใจต่อการพัฒนาใหม่ๆ ในการฝึกอบรมนี้ เราได้ค้นพบหลายอย่าง ไม่ว่าจะเป็นการจัดการโหนดของคุณ การเชื่อมต่อ หรือการใช้งานระยะไกลเพื่อทำการชำระเงิน อย่างไรก็ตาม Lightning เป็นสาขาที่มีความเคลื่อนไหวอย่างมาก ทุกปีมีเครื่องมือใหม่ๆ และมีประโยชน์เกิดขึ้น และมีแอปพลิเคชันใหม่ๆ ปรากฏขึ้นมากมายบน Umbrel การติดตามการพัฒนาใหม่ๆ เหล่านี้อาจทำให้คุณค้นพบโซลูชันที่ทรงพลังหรือใช้งานได้จริงมากกว่าที่นำเสนอในหลักสูตรนี้



ในด้านการศึกษา หากคุณยังไม่ได้ทำ ฉันขอแนะนำอย่างยิ่งให้คุณลงเรียนหลักสูตรทฤษฎี LNP 201 ของ Fanis Michalakis ซึ่งอุทิศให้กับการทำงานของ Lightning Network มันจะช่วยให้คุณเข้าใจการจัดการที่ดำเนินการในหลักสูตร LNP202 นี้ได้ดีขึ้น และให้ความมั่นใจมากขึ้นในการจัดการโหนดของคุณในแต่ละวัน



https://planb.academy/courses/34bd43ef-6683-4a5c-b239-7cb1e40a4aeb

ในแง่มุมที่แตกต่างออกไป แต่มีความสำคัญต่อการเดินทางของคุณในโลกบิตคอยน์ไม่แพ้กัน ฉันยังขอแนะนำหลักสูตรที่ยอดเยี่ยมของ Ludovic Lars เกี่ยวกับประวัติการสร้าง Bitcoin



https://planb.academy/courses/a51c7ceb-e079-4ac3-bf69-6700b985a082

แต่ก่อนที่จะดำเนินการต่อ คุณสามารถเขียนรีวิวเกี่ยวกับหลักสูตร LNP202 นี้ และแน่นอนว่าให้รับประกาศนียบัตรเพื่อยืนยันว่าคุณได้เข้าใจเนื้อหาทั้งหมดแล้ว
