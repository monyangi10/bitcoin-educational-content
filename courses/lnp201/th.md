# การเดินทางสู่ Bitcoin's Second Layer


ดำดิ่งสู่หัวใจของ Lightning Network ซึ่งเป็นระบบที่สำคัญสำหรับอนาคตของธุรกรรม Bitcoin LNP201 เป็นหลักสูตรทฤษฎีเกี่ยวกับการทำงานทางเทคนิคของ Lightning ซึ่งเปิดเผยรากฐานและกลไกของเครือข่ายชั้นที่สองนี้ ออกแบบมาเพื่ออำนวยความสะดวกในการชำระเงิน Bitcoin ที่รวดเร็ว ประหยัด และขยายขนาดได้


ต้องขอบคุณเครือข่ายของช่องทางการชำระเงิน Lightning ที่ทำให้สามารถทำธุรกรรมได้อย่างรวดเร็วและปลอดภัยโดยไม่ต้องบันทึกการแลกเปลี่ยนแต่ละครั้งบนบล็อกเชน Bitcoin ตลอดทั้งบท คุณจะได้เรียนรู้เกี่ยวกับวิธีการเปิด การจัดการ และการปิดช่องทาง วิธีการชำระเงินที่ถูกส่งผ่านโหนดตัวกลางอย่างปลอดภัยในขณะที่ลดความจำเป็นในการไว้วางใจ และวิธีการจัดการสภาพคล่อง คุณจะได้เรียนรู้เกี่ยวกับธุรกรรมการผูกมัด, HTLCs, กุญแจเพิกถอน, กลไกการลงโทษ, การส่งข้อมูลแบบหัวหอม, และใบแจ้งหนี้


ไม่ว่าคุณจะเป็นผู้เริ่มต้นกับ Bitcoin หรือผู้ใช้ที่มีประสบการณ์มากกว่า หลักสูตรนี้จะให้ข้อมูลที่มีค่าเพื่อช่วยให้คุณเข้าใจและใช้ประโยชน์จาก Lightning Network แม้ว่าเราจะครอบคลุมพื้นฐานบางอย่างของการทำงานของ Bitcoin ในส่วนแรก ๆ แต่สิ่งสำคัญคือต้องเชี่ยวชาญพื้นฐานของการประดิษฐ์ Satoshi ก่อนที่จะดำดิ่งสู่ LNP201


เพลิดเพลินกับการค้นพบของคุณ!

## ภาพรวมของหลักสูตร

<chapterId>f2e71062-5121-4114-a7f8-27df69884ce8</chapterId>


ยินดีต้อนรับสู่หลักสูตร LNP201!


การฝึกอบรมนี้มีเป้าหมายเพื่อให้คุณมีความเข้าใจเชิงลึกทางเทคนิคเกี่ยวกับ Lightning Network ซึ่งเป็นเครือข่ายซ้อนทับที่ออกแบบมาเพื่ออำนวยความสะดวกในการทำธุรกรรม Bitcoin ที่รวดเร็วและมักมีต้นทุนต่ำ คุณจะค่อยๆ ค้นพบแนวคิดพื้นฐานที่ควบคุมระบบนี้ ตั้งแต่การเปิดช่องทางการชำระเงินไปจนถึงเทคนิคการกำหนดเส้นทางและการจัดการสภาพคล่อง


**ส่วนที่ 1: พื้นฐาน**

เราจะเริ่มต้นด้วยการแนะนำทั่วไปเกี่ยวกับ Lightning Network วางรากฐานที่จำเป็นสำหรับการทำความเข้าใจ Bitcoin ที่อยู่ของมัน, UTXOs, และวิธีการทำงานของธุรกรรม การทบทวนพื้นฐานนี้จำเป็นเพื่อทำความเข้าใจว่า Lightning Network พึ่งพากลไกบล็อกเชนพื้นฐานอย่างไรในการดำเนินงานอย่างปลอดภัย


**ส่วนที่ 2: การเปิดและปิดช่องทาง**

ในส่วนนี้ เราจะสำรวจขั้นตอนการเปิดช่องทาง ซึ่งเป็นรากฐานสำคัญของ Lightning Network คุณจะได้เรียนรู้วิธีการสร้างธุรกรรมการผูกพัน บทบาทของกุญแจเพิกถอนเพื่อความปลอดภัย และวิธีการปิดช่องทางทั้งแบบร่วมมือกันหรือแบบฝ่ายเดียว แต่ละขั้นตอนจะถูกอธิบายอย่างแม่นยำและเชิงเทคนิคเพื่อช่วยให้คุณเข้าใจรายละเอียดทั้งหมดอย่างถ่องแท้


**ส่วนที่ 3: เครือข่ายสภาพคล่อง**

Lightning Network ไม่ได้จำกัดอยู่แค่ช่องทางเดียว; มันเป็นเครือข่ายการชำระเงินจริง เราจะเห็นว่าการทำธุรกรรมสามารถถูกส่งผ่านโหนดตัวกลางโดยใช้ HTLCs ได้อย่างไร ส่วนนี้ยังจะแนะนำคุณเกี่ยวกับความท้าทายของสภาพคล่องขาเข้าและขาออกด้วย


**ส่วนที่ 4: เครื่องมือ Lightning Network**

ส่วนนี้นำเสนอเครื่องมือที่ใช้งานได้จริงของ Lightning Network เช่น *Invoices*, *LNURL*, และ *Keysend* คุณยังจะได้เรียนรู้วิธีจัดการสภาพคล่องของช่องทางของคุณ ซึ่งเป็นแง่มุมที่สำคัญที่ช่วยให้การชำระเงินราบรื่นและเพิ่มประสิทธิภาพของการทำธุรกรรมของคุณบน Lightning Network


**Section 5: ก้าวไปข้างหน้า**

สุดท้ายนี้ เราจะสรุปการฝึกอบรมโดยการทบทวนแนวคิดที่ได้ครอบคลุมไปแล้ว และปูทางไปสู่หัวข้อขั้นสูงเพิ่มเติมสำหรับผู้ที่ต้องการเพิ่มพูนความรู้เกี่ยวกับ Lightning Network

## การทำความเข้าใจ Lightning Network

Lightning Network เป็นเครือข่ายของช่องทางการชำระเงินที่สร้างขึ้นบนโปรโตคอล Bitcoin โดยมีเป้าหมายเพื่อให้การทำธุรกรรมรวดเร็วและมีต้นทุนต่ำ ช่วยให้สามารถสร้างช่องทางการชำระเงินระหว่างผู้เข้าร่วม ซึ่งภายในช่องทางนี้สามารถทำธุรกรรมได้เกือบจะทันทีและมีค่าธรรมเนียมต่ำ โดยไม่จำเป็นต้องบันทึกการทำธุรกรรมแต่ละครั้งบนบล็อกเชน ดังนั้น Lightning Network จึงมีเป้าหมายเพื่อเพิ่มความสามารถในการขยายตัวของ Bitcoin และทำให้เหมาะสมยิ่งขึ้นสำหรับการทำธุรกรรมที่มีมูลค่าต่ำ


ก่อนที่จะสำรวจแง่มุมของ "เครือข่าย" จำเป็นต้องเข้าใจแนวคิดของ **[ช่องทางการชำระเงิน](https://planb.academy/resources/glossary/payment-channel)** บน Lightning การดำเนินงาน และรายละเอียดเฉพาะของมัน นี่คือหัวข้อของบทแรกนี้


### แนวคิดของช่องทางการชำระเงิน


ช่องทางการชำระเงินช่วยให้ทั้งสองฝ่าย, ที่นี่ **Alice** และ **Bob**, สามารถแลกเปลี่ยนเงินทุนผ่าน Lightning Network ได้ แต่ละตัวละครมีโหนด, แสดงด้วยวงกลม, และช่องทางระหว่างพวกเขาแสดงด้วยส่วนของเส้น.


![LNP201](assets/en/001.webp)


ในตัวอย่างของเรา Alice มี 100,000 satoshis อยู่ในฝั่งของเธอในช่องทาง และ Bob มี 30,000 รวมเป็น 130,000 satoshis ซึ่งถือเป็น **ความจุของช่องทาง**


**แต่ satoshi คืออะไร?**


**satoshi** (หรือ "sat") เป็นหน่วยของบัญชีบน Bitcoin คล้ายกับเซนต์สำหรับยูโร satoshi เป็นเพียงเศษส่วนของ Bitcoin หนึ่ง satoshi เท่ากับ **0.00000001 Bitcoin** หรือหนึ่งในร้อยล้านของ Bitcoin การใช้ satoshi จะมีความเป็นไปได้มากขึ้นเมื่อมูลค่าของ Bitcoin เพิ่มขึ้น


### การจัดสรรเงินทุนในช่องทาง


กลับมาที่ช่องทางการชำระเงินกันเถอะ แนวคิดสำคัญที่นี่คือ "**ด้านของช่องทาง**" ผู้เข้าร่วมแต่ละคนมีเงินทุนในด้านของตน: Alice มี 100,000 ซาโตชิ และ Bob มี 30,000 ดังที่เราได้เห็นแล้ว ผลรวมของเงินทุนเหล่านี้แสดงถึงความจุรวมของช่องทาง ซึ่งเป็นตัวเลขที่กำหนดเมื่อเปิดช่องทาง


![LNP201](assets/en/002.webp)


ลองมาดูตัวอย่างของการทำธุรกรรม Lightning กัน หาก Alice ต้องการส่ง 40,000 satoshis ไปยัง Bob สิ่งนี้เป็นไปได้เพราะเธอมีเงินเพียงพอ (100,000 satoshis) หลังจากการทำธุรกรรมนี้ Alice จะมี 60,000 satoshis ในฝั่งของเธอ และ Bob จะมี 70,000 satoshis


![LNP201](assets/en/003.webp)


**ความจุของช่องสัญญาณ** ที่ 130,000 satoshis ยังคงที่ สิ่งที่เปลี่ยนแปลงคือการจัดสรรเงินทุน ระบบนี้ไม่อนุญาตให้ส่งเงินมากกว่าที่มีอยู่ ตัวอย่างเช่น หาก Bob ต้องการส่งคืน 80,000 satoshis ให้กับ Alice เขาไม่สามารถทำได้ เพราะเขามีเพียง 70,000 เท่านั้น


อีกวิธีหนึ่งในการจินตนาการถึงการจัดสรรเงินทุนคือการนึกภาพ **เคอร์เซอร์** ที่บ่งบอกว่าเงินทุนอยู่ที่ใดภายในช่องทาง ในตอนแรก ด้วย 100,000 ซาโตชิสำหรับ Alice และ 30,000 สำหรับ Bob เคอร์เซอร์จะอยู่ทางด้านของ Bob มากกว่า เนื่องจาก Alice มีเงินทุนมากกว่า หลังจากการทำธุรกรรม 40,000 ซาโตชิ เคอร์เซอร์จะเลื่อนไปทาง Alice เล็กน้อย ซึ่งตอนนี้ถือ 60,000 ซาโตชิ


![LNP201](assets/en/004.webp)


การแสดงผลนี้สามารถเป็นประโยชน์ในการจินตนาการถึงความสมดุลของเงินทุนในช่องทาง


### กฎพื้นฐานของช่องทางการชำระเงิน


จุดแรกที่ต้องจำไว้คือ **ความจุของช่องสัญญาณ** นั้นคงที่ มันคล้ายกับเส้นผ่านศูนย์กลางของท่อ: มันกำหนดปริมาณสูงสุดของเงินทุนที่สามารถส่งผ่านช่องทางได้ในครั้งเดียว

ลองพิจารณาตัวอย่าง: ถ้า Alice มี 130,000 ซาโตชิในฝั่งของเธอ เธอสามารถส่งได้สูงสุด 130,000 ซาโตชิไปยัง Bob ในการทำธุรกรรมครั้งเดียว อย่างไรก็ตาม Bob สามารถส่งเงินเหล่านี้กลับไปยัง Alice ได้ทั้งบางส่วนหรือทั้งหมด


สิ่งที่สำคัญที่ต้องเข้าใจคือ ความจุคงที่ของช่องทางจำกัดจำนวนสูงสุดของธุรกรรมเดียว แต่ไม่จำกัดจำนวนธุรกรรมที่เป็นไปได้ทั้งหมด หรือปริมาณรวมของเงินที่แลกเปลี่ยนภายในช่องทาง


**คุณควรนำอะไรไปจากบทนี้?**



- ความจุของช่องทางถูกกำหนดไว้และกำหนดจำนวนสูงสุดที่สามารถส่งได้ในธุรกรรมเดียว
- เงินทุนในช่องทางจะถูกแจกจ่ายระหว่างผู้เข้าร่วมทั้งสอง และแต่ละคนสามารถส่งเงินให้กับอีกฝ่ายได้เฉพาะเงินที่พวกเขาเป็นเจ้าของในฝั่งของตนเองเท่านั้น
- Lightning Network จึงช่วยให้สามารถแลกเปลี่ยนเงินทุนได้อย่างรวดเร็วและมีประสิทธิภาพ ในขณะที่ยังคงเคารพข้อจำกัดที่กำหนดโดยความจุของช่องทาง


นี่คือจุดสิ้นสุดของบทแรก ซึ่งเราได้วางรากฐานสำหรับ Lightning Network ในบทต่อไป เราจะได้เห็นวิธีการเปิดช่องทางและเจาะลึกลงไปในแนวคิดที่ได้กล่าวถึงที่นี่


## Bitcoin, ที่อยู่, UTXO, และธุรกรรม


<chapterId>0cfb7e6b-96f0-508b-9210-90bc1e28649d</chapterId>


:::video id=75323eef-ea03-45ac-9a6e-46d73ca255de:::

บทนี้ค่อนข้างพิเศษเนื่องจากจะไม่ได้อุทิศให้กับ Lightning โดยตรง แต่จะเป็น Bitcoin แท้จริงแล้ว Lightning Network เป็นชั้นที่อยู่บน Bitcoin ดังนั้นจึงจำเป็นต้องเข้าใจแนวคิดพื้นฐานบางประการของ Bitcoin เพื่อให้เข้าใจการทำงานของ Lightning ในบทต่อไป ในบทนี้ เราจะทบทวนพื้นฐานของที่อยู่รับ Bitcoin, UTXOs รวมถึงการทำงานของธุรกรรม Bitcoin


### Bitcoin ที่อยู่, กุญแจส่วนตัว, และกุญแจสาธารณะ


ที่อยู่ Bitcoin เป็นชุดของอักขระที่ได้มาจาก **กุญแจสาธารณะ** ซึ่งคำนวณมาจาก **กุญแจส่วนตัว** อย่างที่คุณทราบดี มันถูกใช้ในการล็อคบิตคอยน์ ซึ่งเทียบเท่ากับการรับบิตคอยน์ใน wallet ของเรา


คีย์ส่วนตัวเป็นองค์ประกอบลับที่**ไม่ควรแชร์เด็ดขาด** ในขณะที่คีย์สาธารณะและที่อยู่สามารถแชร์ได้โดยไม่มีความเสี่ยงด้านความปลอดภัย (การเปิดเผยของพวกมันเป็นเพียงความเสี่ยงต่อความเป็นส่วนตัวของคุณ) นี่คือการแสดงทั่วไปที่เราจะใช้ตลอดการฝึกอบรมนี้:



- คีย์ส่วนตัวจะถูกแสดงในแนวตั้ง
- คีย์สาธารณะจะถูกแสดงในแนวนอน
- สีของพวกมันบ่งบอกว่าใครเป็นเจ้าของ (Alice ในสีส้มและ Bob ในสีดำ...)


### ธุรกรรม Bitcoin: การส่งเงินและสคริปต์


บน Bitcoin การทำธุรกรรมเกี่ยวข้องกับการส่งเงินจากที่อยู่หนึ่งไปยังอีกที่อยู่หนึ่ง ลองพิจารณาตัวอย่างของ Alice ที่ส่ง 0.002 Bitcoin ไปยัง Bob Alice ใช้คีย์ส่วนตัวที่เชื่อมโยงกับที่อยู่ของเธอเพื่อ **เซ็น** การทำธุรกรรม ซึ่งเป็นการพิสูจน์ว่าเธอสามารถใช้จ่ายเงินเหล่านี้ได้จริง แต่จริงๆ แล้วเกิดอะไรขึ้นเบื้องหลังการทำธุรกรรมนี้? เงินบนที่อยู่ Bitcoin ถูกล็อกโดย **สคริปต์** ซึ่งเป็นโปรแกรมขนาดเล็กที่กำหนดเงื่อนไขเฉพาะสำหรับการใช้จ่ายเงินเหล่านี้


สคริปต์ที่พบมากที่สุดต้องการลายเซ็นด้วยกุญแจส่วนตัวที่เชื่อมโยงกับที่อยู่ เมื่อ Alice เซ็นชื่อในธุรกรรมด้วยกุญแจส่วนตัวของเธอ เธอ **ปลดล็อกสคริปต์** ที่บล็อกเงินทุน และจากนั้นเงินทุนสามารถถูกโอนย้ายได้ การโอนย้ายเงินทุนเกี่ยวข้องกับการเพิ่มสคริปต์ใหม่ให้กับเงินทุนเหล่านี้ โดยกำหนดว่าการใช้จ่ายในครั้งนี้จะต้องใช้ลายเซ็นกุญแจส่วนตัวของ **Bob**


![LNP201](assets/en/005.webp)


### UTXOs: Unspent Transaction Outputs


บน Bitcoin สิ่งที่เราแลกเปลี่ยนจริงๆ ไม่ใช่บิตคอยน์โดยตรง แต่เป็น **[UTXOs](https://planb.academy/resources/glossary/utxo)** (_Unspent Transaction Outputs_) หมายถึง "เอาต์พุตของธุรกรรมที่ยังไม่ได้ใช้จ่าย"


UTXO คือหน่วยของบิตคอยน์ที่สามารถมีมูลค่าใดๆ ก็ได้ เช่น **2,000 บิตคอยน์**, **8 บิตคอยน์**, หรือแม้กระทั่ง **8,000 sats** แต่ละ UTXO ถูกล็อกด้วยสคริปต์ และเพื่อใช้จ่ายมัน จะต้องปฏิบัติตามเงื่อนไขของสคริปต์ ซึ่งมักจะเป็นลายเซ็นที่ใช้กุญแจส่วนตัวที่ตรงกับที่อยู่รับที่กำหนดไว้


UTXOs ไม่สามารถแบ่งได้ ทุกครั้งที่ใช้เพื่อใช้จ่ายจำนวนบิตคอยน์ที่พวกมันแทน มันต้องทำทั้งหมด เหมือนกับธนบัตร: ถ้าคุณมีธนบัตร €10 และคุณติดหนี้คนทำขนมปัง €5 คุณไม่สามารถตัดธนบัตรครึ่งหนึ่งได้ คุณต้องให้ธนบัตร €10 แก่เขา และเขาจะให้เงินทอน €5 แก่คุณ นี่คือหลักการเดียวกันสำหรับ UTXOs บน Bitcoin! ตัวอย่างเช่น เมื่อ Alice ปลดล็อกสคริปต์ด้วยกุญแจส่วนตัวของเธอ เธอจะปลดล็อก UTXO ทั้งหมด หากเธอต้องการส่งเพียงส่วนหนึ่งของเงินที่ UTXO แทนให้กับ Bob เธอสามารถ "แยก" มันออกเป็นหลายส่วนที่เล็กลง จากนั้นเธอจะส่ง 0.0015 BTC ให้กับ Bob และส่งส่วนที่เหลือ 0.0005 BTC ไปยัง **ที่อยู่เงินทอน**


นี่คือตัวอย่างของธุรกรรมที่มี 2 เอาท์พุต:



- UTXO จำนวน 0.0015 BTC สำหรับ Bob ถูกล็อกโดยสคริปต์ที่ต้องการลายเซ็นคีย์ส่วนตัวของ Bob
- UTXO จำนวน 0.0005 BTC สำหรับ Alice ถูกล็อกโดยสคริปต์ที่ต้องการลายเซ็นของเธอเอง


![LNP201](assets/en/006.webp)


### ที่อยู่หลายลายเซ็น


นอกเหนือจากที่อยู่แบบง่ายที่สร้างจากคีย์สาธารณะเดียวแล้ว ยังสามารถสร้าง**ที่อยู่หลายลายเซ็น**จากคีย์สาธารณะหลายตัวได้ กรณีที่น่าสนใจเป็นพิเศษสำหรับ Lightning Network คือ**ที่อยู่หลายลายเซ็น 2/2** ซึ่งสร้างจากคีย์สาธารณะสองตัว:


![LNP201](assets/en/007.webp)


ในการใช้จ่ายเงินที่ถูกล็อกด้วยที่อยู่หลายลายเซ็น 2/2 นี้ จำเป็นต้องลงนามด้วยกุญแจส่วนตัวสองชุดที่เชื่อมโยงกับกุญแจสาธารณะ


![LNP201](assets/en/008.webp)


ที่อยู่ประเภทนี้แสดงถึงช่องทางการชำระเงินบนบล็อกเชน Bitcoin อย่างแม่นยำ โดยเฉพาะช่องทางบน Lightning Network


**คุณควรนำอะไรไปจากบทนี้?**



- ที่อยู่ **Bitcoin** ได้มาจากคีย์สาธารณะ ซึ่งตัวมันเองได้มาจากคีย์ส่วนตัว
- เงินทุนบน Bitcoin ถูกล็อคโดย **scripts** และในการใช้จ่ายเงินทุนเหล่านี้ จะต้องปฏิบัติตาม script ซึ่งโดยทั่วไปจะเกี่ยวข้องกับการให้ลายเซ็นด้วยกุญแจส่วนตัวที่สอดคล้องกัน
- UTXOs** คือชิ้นส่วนของบิตคอยน์ที่ถูกล็อกโดยสคริปต์ และแต่ละธุรกรรมบน Bitcoin ประกอบด้วยการปลดล็อก UTXO และจากนั้นสร้างหนึ่งหรือมากกว่าใหม่เป็นการตอบแทน
- ที่อยู่แบบหลายลายเซ็น 2/2** ต้องการลายเซ็นจากกุญแจส่วนตัวสองชุดเพื่อใช้จ่ายเงิน ที่อยู่เฉพาะเหล่านี้ถูกใช้ในบริบทของ Lightning เพื่อสร้างช่องทางการชำระเงิน


บทนี้เกี่ยวกับ Bitcoin ได้เปิดโอกาสให้เราได้ทบทวนแนวคิดสำคัญบางประการสำหรับสิ่งที่จะตามมา ในบทถัดไป เราจะค้นพบโดยเฉพาะว่าการเปิดช่องทางบน Lightning Network ทำงานอย่างไร

## การเปิดช่องทาง


<chapterId>96243eb0-f6b5-5b68-af1f-fffa0cc16bfe</chapterId>


:::video id=6098fee1-735e-4d8d-9f57-0faf5fef6d76:::

ในบทนี้ เราจะตรวจสอบกระบวนการเปิดช่องทางการชำระเงินบน Lightning Network อย่างละเอียดมากขึ้นและสำรวจความเชื่อมโยงระหว่างการดำเนินการนี้กับระบบพื้นฐาน Bitcoin


### ช่องทางสายฟ้า


ตามที่เราเห็นในบทแรก, **ช่องทางการชำระเงิน** บน Lightning สามารถเปรียบเทียบได้กับ "ท่อ" สำหรับการแลกเปลี่ยนเงินทุนระหว่างผู้เข้าร่วมสองคน (**Alice** และ **Bob** ในตัวอย่างของเรา) ความจุของช่องทางนี้สอดคล้องกับผลรวมของเงินทุนที่มีอยู่ในแต่ละด้าน ในตัวอย่างของเรา, Alice มี **100,000 satoshis** และ Bob มี **30,000 satoshis** ทำให้มี **ความจุรวม** **130,000 satoshis**


![LNP201](assets/en/009.webp)


### ระดับของข้อมูล Exchange


สิ่งสำคัญคือต้องแยกแยะระดับการแลกเปลี่ยนที่แตกต่างกันบน Lightning Network อย่างชัดเจน:



- การสื่อสารแบบเพียร์ทูเพียร์ (โปรโตคอล Lightning)**: ข้อความเหล่านี้คือข้อความที่โหนด Lightning ส่งถึงกันเพื่อสื่อสาร เราจะแทนข้อความเหล่านี้ด้วยเส้นประสีดำในแผนภาพของเรา
- ช่องทางการชำระเงิน (โปรโตคอล Lightning)**: เหล่านี้คือเส้นทางสำหรับการแลกเปลี่ยนเงินทุนบน Lightning ซึ่งเราจะแสดงด้วยเส้นสีดำทึบ
- ธุรกรรม Bitcoin (โปรโตคอล Bitcoin)**: เหล่านี้คือธุรกรรมที่ทำโดย on-chain ซึ่งเราจะแสดงด้วยเส้นสีส้ม


![LNP201](assets/en/010.webp)


ควรสังเกตว่าโหนด Lightning สามารถสื่อสารผ่านโปรโตคอล P2P ได้โดยไม่ต้องเปิดช่องทาง แต่หากต้องการแลกเปลี่ยนเงิน จำเป็นต้องมีช่องทาง


### ขั้นตอนการเปิดช่องทาง Lightning



- การแลกเปลี่ยนข้อความ**: Alice ต้องการเปิดช่องทางกับ Bob เธอส่งข้อความถึงเขาซึ่งมีจำนวนเงินที่เธอต้องการฝากในช่องทาง (130,000 sats) และกุญแจสาธารณะของเธอ Bob ตอบกลับโดยแชร์กุญแจสาธารณะของเขาเอง


![LNP201](assets/en/011.webp)



- การสร้างที่อยู่ [multisignature](https://planb.academy/resources/glossary/multisig)**: ด้วยคีย์สาธารณะสองคีย์นี้ Alice สร้าง **ที่อยู่ multisignature 2/2** ซึ่งหมายความว่าเงินที่จะถูกฝากในที่อยู่นี้ในภายหลังจะต้องใช้ลายเซ็นทั้งสอง (Alice และ Bob) ในการใช้จ่าย


![LNP201](assets/en/012.webp)



- ธุรกรรมการฝาก**: Alice เตรียมธุรกรรม Bitcoin เพื่อฝากเงินในที่อยู่หลายลายเซ็นนี้ ตัวอย่างเช่น เธออาจตัดสินใจส่ง **130,000 satoshis** ไปยังที่อยู่หลายลายเซ็นนี้ ธุรกรรมนี้ถูก**สร้างขึ้นแต่ยังไม่ได้เผยแพร่**บนบล็อกเชน


![LNP201](assets/en/013.webp)



- การทำธุรกรรมการถอน**: ก่อนที่จะเผยแพร่การทำธุรกรรมการฝาก Alice จะสร้างการทำธุรกรรมการถอนเพื่อที่เธอจะสามารถกู้คืนเงินของเธอได้ในกรณีที่มีปัญหากับ Bob จริง ๆ แล้ว เมื่อ Alice เผยแพร่การทำธุรกรรมการฝาก sats ของเธอจะถูกล็อคในที่อยู่มัลติซิกเนเจอร์ 2/2 ซึ่งต้องการทั้งลายเซ็นของเธอและลายเซ็นของ Bob เพื่อปลดล็อค Alice ป้องกันความเสี่ยงจากการสูญเสียนี้โดยการสร้างการทำธุรกรรมการถอนที่ช่วยให้เธอกู้คืนเงินของเธอได้


![LNP201](assets/en/014.webp)



- ลายเซ็นของ Bob**: Alice ส่งธุรกรรมการฝากไปยัง Bob เป็นหลักฐานและขอให้เขาลงนามในธุรกรรมการถอน เมื่อได้รับลายเซ็นของ Bob ในธุรกรรมการถอนแล้ว Alice จะมั่นใจได้ว่าสามารถกู้คืนเงินของเธอได้ตลอดเวลา เนื่องจากตอนนี้ต้องการเพียงลายเซ็นของเธอเองเท่านั้นในการปลดล็อกมัลติซิกเนเจอร์


![LNP201](assets/en/015.webp)



- การเผยแพร่ธุรกรรมการฝาก**: เมื่อได้รับลายเซ็นของ Bob แล้ว Alice สามารถเผยแพร่ธุรกรรมการฝากบนบล็อกเชน Bitcoin ได้ ซึ่งจะเป็นการเปิดช่องทาง Lightning อย่างเป็นทางการระหว่างผู้ใช้ทั้งสองคน


![LNP201](assets/en/016.webp)


### ช่องเปิดเมื่อไหร่?


ช่องทางจะถือว่าเปิดเมื่อธุรกรรมการฝากถูกบรรจุในบล็อก Bitcoin และได้ถึงความลึกของการยืนยันในระดับหนึ่ง (เช่น จำนวนบล็อกที่ตามมาซึ่งถูกเพิ่มเข้าไปในบล็อกเชน)


**สิ่งที่คุณควรจำจากบทนี้คืออะไร?**



- การเปิดช่องทางเริ่มต้นด้วยการแลกเปลี่ยน **ข้อความ** ระหว่างสองฝ่าย (การแลกเปลี่ยนจำนวนเงินและกุญแจสาธารณะ)
- ช่องทางถูกสร้างขึ้นโดยการสร้าง **ที่อยู่ multisignature 2/2** และฝากเงินเข้าไปผ่านธุรกรรม Bitcoin
- บุคคลที่เปิดช่องทางจะต้องมั่นใจว่าพวกเขาสามารถ**กู้คืนเงินทุนของตน**ได้ผ่านการทำธุรกรรมการถอนที่ลงนามโดยอีกฝ่ายก่อนที่จะเผยแพร่การทำธุรกรรมการฝากเงิน


ในบทถัดไป เราจะสำรวจการทำงานทางเทคนิคของการทำธุรกรรม Lightning ภายในช่องทาง

![LNP201](assets/en/018.webp)


### ธุรกรรมที่ยังไม่ได้เผยแพร่: ธุรกรรม Commitment


เมื่อ Alice ทำธุรกรรมในช่องทางเพื่อส่งเงินไปยัง Bob จะมีการสร้างธุรกรรม Bitcoin ใหม่เพื่อสะท้อนการเปลี่ยนแปลงนี้ในการกระจายเงิน ธุรกรรมนี้เรียกว่า **[commitment transaction](https://planb.academy/resources/glossary/commitment-transaction)** ซึ่งไม่ได้ถูกเผยแพร่บนบล็อกเชนแต่แสดงถึงสถานะใหม่ของช่องทางหลังจากธุรกรรม Lightning


ลองมาดูตัวอย่างที่ Alice ส่ง 30,000 satoshis ไปยัง Bob:



- ในขั้นต้น**: Alice มี 130,000 ซาโตชิ
- หลังจากการทำธุรกรรม**: Alice มี 100,000 ซาโตชิ และ Bob มี 30,000 ซาโตชิ

ในการยืนยันการโอนนี้ Alice และ Bob สร้าง **ธุรกรรม Bitcoin ที่ยังไม่ได้เผยแพร่** ใหม่ที่จะส่ง **100,000 satoshis ไปยัง Alice** และ **30,000 satoshis ไปยัง Bob** จากที่อยู่มัลติซิกเนเจอร์ ทั้งสองฝ่ายสร้างธุรกรรมนี้อย่างอิสระ แต่ใช้ข้อมูลเดียวกัน (จำนวนและที่อยู่) เมื่อสร้างเสร็จแล้ว แต่ละฝ่ายจะลงนามในธุรกรรมและแลกเปลี่ยนลายเซ็นกับอีกฝ่าย ซึ่งจะทำให้ฝ่ายใดฝ่ายหนึ่งสามารถเผยแพร่ธุรกรรมได้ตลอดเวลา หากจำเป็นต้องกู้คืนส่วนแบ่งของช่องทางบนบล็อกเชนหลัก Bitcoin

![LNP201](assets/en/019.webp)


### กระบวนการโอน: The Invoice


เมื่อ Bob ต้องการรับเงิน เขาจะส่ง **_[invoice](https://planb.academy/resources/glossary/invoice-lightning)_** สำหรับ 30,000 ซาโตชิ ไปยัง Alice จากนั้น Alice จะดำเนินการชำระเงินตามใบแจ้งหนี้นี้โดยเริ่มการโอนภายในช่องทาง ดังที่เราได้เห็น กระบวนการนี้ขึ้นอยู่กับการสร้างและการลงนามของ **commitment transaction** ใหม่


แต่ละ commitment transaction แสดงถึงการกระจายเงินทุนใหม่ในช่องทางหลังจากการโอน ในตัวอย่างนี้ หลังจากการทำธุรกรรม Bob มี 30,000 ซาโตชิ และ Alice มี 100,000 ซาโตชิ หากผู้เข้าร่วมคนใดคนหนึ่งตัดสินใจที่จะเผยแพร่ commitment transaction นี้บนบล็อกเชน จะส่งผลให้ช่องทางถูกปิด และเงินทุนจะถูกกระจายตามการกระจายครั้งสุดท้ายนี้


![LNP201](assets/en/020.webp)


### สถานะใหม่หลังจากการทำธุรกรรมครั้งที่สอง


ลองพิจารณาตัวอย่างอื่น: หลังจากการทำธุรกรรมครั้งแรกที่ Alice ส่ง 30,000 satoshis ไปยัง Bob, Bob ตัดสินใจส่ง **10,000 satoshis กลับไปยัง Alice**. สิ่งนี้สร้างสถานะใหม่ของช่องทาง. **commitment transaction** ใหม่จะเป็นตัวแทนของการกระจายที่อัปเดตนี้:



- Alice** ตอนนี้มี **110,000 ซาโตชิ**.
- Bob** มี **20,000 ซาโตชิ**.


![LNP201](assets/en/021.webp)


อีกครั้ง การทำธุรกรรมนี้ไม่ได้เผยแพร่บนบล็อกเชน แต่สามารถทำได้ทุกเมื่อหากช่องทางถูกปิด


โดยสรุป เมื่อมีการโอนเงินภายในช่องทาง Lightning:



- Alice และ Bob สร้าง **commitment transaction** ใหม่ ซึ่งสะท้อนถึงการกระจายเงินทุนใหม่
- ธุรกรรม Bitcoin นี้ถูก **ลงนาม** โดยทั้งสองฝ่าย แต่ **ยังไม่ได้เผยแพร่** บนบล็อกเชน Bitcoin ตราบใดที่ช่องทางยังคงเปิดอยู่
- ธุรกรรมการผูกพันทำให้มั่นใจได้ว่าผู้เข้าร่วมแต่ละคนสามารถกู้คืนเงินของตนได้ตลอดเวลาบนบล็อกเชน Bitcoin โดยการเผยแพร่ธุรกรรมที่ลงนามล่าสุด


อย่างไรก็ตาม ระบบนี้มีข้อบกพร่องที่อาจเกิดขึ้น ซึ่งเราจะพูดถึงในบทถัดไป เราจะเห็นว่าผู้เข้าร่วมแต่ละคนสามารถป้องกันตนเองจากความพยายามที่จะโกงโดยอีกฝ่ายได้อย่างไร


## คีย์เพิกถอน


<chapterId>f2f61e5b-badb-5947-9a81-7aa530b44e59</chapterId>

:::video id=1d850f23-eff1-4725-b284-ce12456a2c26:::
ในบทนี้ เราจะเจาะลึกลงไปถึงวิธีการทำงานของธุรกรรมบน Lightning Network โดยการพูดคุยเกี่ยวกับกลไกที่มีอยู่เพื่อป้องกันการโกง เพื่อให้แน่ใจว่าทุกฝ่ายปฏิบัติตามกฎภายในช่องทาง


### เตือนความจำ: ธุรกรรม Commitment


ตามที่เห็นก่อนหน้านี้ การทำธุรกรรมบน Lightning ขึ้นอยู่กับ **ธุรกรรมการผูกมัด** ที่ไม่ได้เผยแพร่ ธุรกรรมเหล่านี้สะท้อนถึงการกระจายเงินทุนในช่องทางปัจจุบัน เมื่อมีการทำธุรกรรม Lightning ใหม่ จะมีการสร้าง commitment transaction ใหม่และลงนามโดยทั้งสองฝ่ายเพื่อสะท้อนสถานะใหม่ของช่องทาง


ลองมาดูตัวอย่างง่ายๆ:



- สถานะเริ่มต้น**: Alice มี **100,000 satoshis**, Bob **30,000 satoshis**.
- หลังจากการทำธุรกรรมที่ Alice ส่ง **40,000 satoshis** ไปยัง Bob, commitment transaction ใหม่จะแจกจ่ายเงินดังนี้:
  - Alice: **60,000 satoshis**
  - Bob: **70,000 satoshis**


![LNP201](assets/en/022.webp)


ทั้งสองฝ่ายสามารถเผยแพร่ **commitment transaction ล่าสุด** ที่ลงนามเพื่อปิดช่องทางและกู้คืนเงินของพวกเขาได้ตลอดเวลา


### The Flaw: Cheating by Publishing an Old Transaction


ปัญหาที่อาจเกิดขึ้นคือถ้าฝ่ายใดฝ่ายหนึ่งตัดสินใจที่จะ **โกง** โดยการเผยแพร่ commitment transaction เก่า ตัวอย่างเช่น Alice อาจเผยแพร่ commitment transaction เก่าที่เธอมี **100,000 satoshis** แม้ว่าในความเป็นจริงเธอมีเพียง **60,000** เท่านั้น ซึ่งจะทำให้เธอสามารถขโมย **40,000 satoshis** จาก Bob ได้


![LNP201](assets/en/023.webp)


ยิ่งไปกว่านั้น Alice อาจเผยแพร่ธุรกรรมการถอนครั้งแรก ซึ่งเป็นธุรกรรมก่อนที่ช่องทางจะถูกเปิด ซึ่งจะรวมถึง **130,000 satoshis** และด้วยเหตุนี้จึงขโมยเงินทั้งหมดในช่องทางนั้นได้


![LNP201](assets/en/024.webp)


### วิธีแก้ปัญหา: กุญแจเพิกถอนและการล็อกเวลา


เพื่อป้องกันการโกงประเภทนี้โดย Alice บน Lightning Network ได้มีการเพิ่ม **กลไกความปลอดภัย** ในธุรกรรมการให้คำมั่น:



- The timelock**: แต่ละ commitment transaction รวมถึง timelock สำหรับเงินทุนของ Alice. Timelock เป็น primitive ของ smart contract ที่ตั้งเงื่อนไขเวลา ซึ่งต้องถูกปฏิบัติตามเพื่อให้ธุรกรรมถูกเพิ่มลงในบล็อก. ซึ่งหมายความว่า Alice ไม่สามารถกู้คืนเงินทุนของเธอได้จนกว่าจะมีจำนวนบล็อกที่ผ่านไปตามที่กำหนด หากเธอเผยแพร่หนึ่งในธุรกรรมการผูกมัด. Timelock นี้เริ่มมีผลตั้งแต่การยืนยันของ commitment transaction. ระยะเวลาของมันมักจะเป็นสัดส่วนกับขนาดของช่องทาง แต่ก็สามารถกำหนดค่าได้ด้วยตนเอง.
- คีย์เพิกถอน**: เงินทุนของ Alice สามารถถูกใช้ได้ทันทีโดย Bob หากเขามี **คีย์เพิกถอน** คีย์นี้ประกอบด้วยความลับที่ถือโดย Alice และความลับที่ถือโดย Bob โปรดทราบว่าความลับนี้แตกต่างกันสำหรับแต่ละ commitment transaction

ด้วยกลไกทั้งสองนี้รวมกัน Bob มีเวลาที่จะตรวจจับความพยายามของ Alice ที่จะโกงและลงโทษเธอโดยการดึงผลลัพธ์ของเขากลับมาด้วยกุญแจเพิกถอน ซึ่งสำหรับ Bob หมายถึงการกู้คืนเงินทั้งหมดของช่องทาง commitment transaction ใหม่ของเราจะมีลักษณะดังนี้:


![LNP201](assets/en/025.webp)


มาลงรายละเอียดเกี่ยวกับการทำงานของกลไกนี้ด้วยกันเถอะ


### กระบวนการอัปเดตธุรกรรม


เมื่อ Alice และ Bob อัปเดตสถานะของช่องทางด้วยธุรกรรม Lightning ใหม่ พวกเขาจะแลกเปลี่ยน **ความลับ** ของตนล่วงหน้าสำหรับ commitment transaction ก่อนหน้า (ซึ่งจะกลายเป็นล้าสมัยและอาจอนุญาตให้คนใดคนหนึ่งโกงได้) ซึ่งหมายความว่า ในสถานะใหม่ของช่องทาง:



- Alice และ Bob มี commitment transaction ใหม่ที่แสดงการกระจายเงินทุนปัจจุบันหลังจากการทำธุรกรรม Lightning
- แต่ละคนมีความลับของอีกฝ่ายสำหรับธุรกรรมก่อนหน้า ซึ่งทำให้พวกเขาสามารถใช้กุญแจเพิกถอนก็ต่อเมื่อมีคนใดคนหนึ่งพยายามโกงโดยการเผยแพร่ธุรกรรมที่มีสถานะเก่าใน mempools ของโหนด Bitcoin จริง ๆ แล้ว เพื่อที่จะลงโทษอีกฝ่าย จำเป็นต้องถือทั้งสองความลับและ commitment transaction ของอีกฝ่าย ซึ่งรวมถึงอินพุตที่ลงนามแล้ว หากไม่มีธุรกรรมนี้ กุญแจเพิกถอนจะไร้ประโยชน์ในตัวเอง วิธีเดียวที่จะได้รับธุรกรรมนี้คือการดึงมันจาก mempools (เช่น ธุรกรรมที่รอการยืนยัน) หรือจากธุรกรรมที่ได้รับการยืนยันบนบล็อกเชนในช่วงเวลาล็อค ซึ่งพิสูจน์ได้ว่าอีกฝ่ายพยายามโกง ไม่ว่าจะโดยเจตนาหรือไม่ก็ตาม


ลองมาดูตัวอย่างเพื่อทำความเข้าใจกระบวนการนี้ให้ดี:



- สถานะเริ่มต้น**: Alice มี **100,000 satoshis**, Bob **30,000 satoshis**.


![LNP201](assets/en/026.webp)



- Bob ต้องการรับ 40,000 satoshis จาก Alice ผ่านช่องทาง Lightning ของพวกเขา ในการทำเช่นนี้:
   - เขาส่งใบแจ้งหนี้ให้เธอพร้อมกับความลับของเขาสำหรับกุญแจเพิกถอนของ commitment transaction ก่อนหน้านี้ของเขา
   - ในการตอบสนอง Alice ให้ลายเซ็นของเธอสำหรับ Bob's ใหม่ commitment transaction รวมถึงความลับของเธอสำหรับกุญแจเพิกถอนของธุรกรรมก่อนหน้าของเธอ
   - ในที่สุด Bob ก็ส่งลายเซ็นของเขาสำหรับ Alice's ใหม่ commitment transaction
   - การแลกเปลี่ยนเหล่านี้อนุญาตให้ Alice ส่ง **40,000 satoshis** ไปยัง Bob บน Lightning ผ่านช่องทางของพวกเขา และธุรกรรมการให้คำมั่นสัญญาใหม่ตอนนี้สะท้อนถึงการกระจายเงินทุนใหม่นี้แล้ว


![LNP201](assets/en/027.webp)



- หาก Alice พยายามที่จะเผยแพร่ commitment transaction เก่าที่เธอยังคงเป็นเจ้าของ **100,000 satoshis** Bob ซึ่งได้รับกุญแจเพิกถอนแล้ว สามารถกู้คืนเงินได้ทันทีโดยใช้กุญแจนี้ ในขณะที่ Alice ถูกบล็อกโดย timelock


![LNP201](assets/en/028.webp)


แม้ว่าในกรณีนี้ Bob จะไม่มีผลประโยชน์ทางเศรษฐกิจในการโกง แต่ถ้าเขาทำเช่นนั้น Alice ก็จะได้รับประโยชน์จากการป้องกันแบบสมมาตร ซึ่งให้การรับประกันแบบเดียวกันแก่เธอ


**คุณควรนำอะไรไปจากบทนี้?**


**ธุรกรรมการผูกพัน** บน Lightning Network รวมถึงกลไกความปลอดภัยที่ลดทั้งความเสี่ยงในการโกงและแรงจูงใจในการทำเช่นนั้น ก่อนที่จะเซ็น commitment transaction ใหม่ Alice และ Bob จะแลกเปลี่ยน **ความลับ** ของพวกเขาสำหรับธุรกรรมการผูกพันก่อนหน้า หาก Alice พยายามเผยแพร่ commitment transaction เก่า Bob สามารถใช้ **กุญแจเพิกถอน** เพื่อกู้คืนเงินทั้งหมดก่อนที่ Alice จะทำได้ (เพราะเธอถูกบล็อกโดยการล็อกเวลา) ซึ่งเป็นการลงโทษเธอสำหรับการพยายามโกง


ระบบรักษาความปลอดภัยนี้ทำให้มั่นใจได้ว่าผู้เข้าร่วมปฏิบัติตามกฎของ Lightning Network และพวกเขาไม่สามารถทำกำไรจากการเผยแพร่ธุรกรรมข้อผูกพันเก่าได้


ณ จุดนี้ในการฝึกอบรม คุณได้เรียนรู้แล้วว่าช่องทาง Lightning ถูกเปิดอย่างไรและธุรกรรมภายในช่องทางเหล่านี้ทำงานอย่างไร ในบทถัดไป เราจะสำรวจวิธีการต่างๆ ในการปิดช่องทางและการกู้คืนบิตคอยน์ของคุณบนบล็อกเชนหลัก

ในบทนี้ เราจะพูดถึง**การปิดช่องทาง**บน Lightning Network ซึ่งทำผ่านธุรกรรม Bitcoin เช่นเดียวกับการเปิดช่องทาง หลังจากที่ได้เห็นว่าธุรกรรมภายในช่องทางทำงานอย่างไร ตอนนี้ถึงเวลาที่จะดูวิธีการปิดช่องทางและกู้คืนเงินบนบล็อกเชน Bitcoin


### การเตือนเกี่ยวกับวงจรชีวิตของช่องทาง


**วงจรชีวิตของช่องทาง** เริ่มต้นด้วยการ **เปิด** ซึ่งเริ่มต้นผ่านธุรกรรม Bitcoin จากนั้นมีการทำธุรกรรม Lightning ภายในช่องทางนั้น สุดท้ายเมื่อฝ่ายต่างๆ ต้องการกู้คืนเงินของพวกเขา ช่องทางจะถูก **ปิด** ผ่านธุรกรรม Bitcoin ครั้งที่สอง ธุรกรรมระหว่างกลางที่ทำบน Lightning จะถูกแทนด้วย **ธุรกรรมการผูกพัน** ที่ไม่ได้เผยแพร่


![LNP201](assets/en/029.webp)


### ประเภทของการปิดช่องทางทั้งสาม


มีสามวิธีหลักในการปิดช่องทางนี้ ซึ่งสามารถเรียกว่า **the good, the brute, and the truant** (ได้รับแรงบันดาลใจจาก Andreas Antonopoulos ใน _Mastering the Lightning Network_):



- The Good**: the **cooperative closure**, where Alice and Bob agree to close the channel.
- The Bad**: the **forced closure**, where one of the parties decides to close the channel honestly, but without the other's agreement.
- The Ugly**: the **closure with cheating**, where one of the parties attempts to steal funds by publishing an old commitment transaction (any but not the last one, which reflects the actual and fair distribution of funds).


ลองมาดูตัวอย่าง:



- Alice เป็นเจ้าของ **100,000 satoshis** และ Bob **30,000 satoshis**.
- การกระจายนี้สะท้อนอยู่ใน **2 ธุรกรรมการผูกพัน** (หนึ่งรายการต่อผู้ใช้หนึ่งคน) ที่ไม่ได้เผยแพร่ แต่สามารถทำได้ในกรณีที่มีการปิดช่องทาง


![LNP201](assets/en/030.webp)


### สิ่งที่ดี: การปิดร่วมกัน


ใน **การปิดร่วมกัน**, Alice และ Bob ตกลงที่จะปิดช่องทาง นี่คือวิธีการ:



- Alice ส่งข้อความไปยัง Bob ผ่านโปรโตคอลการสื่อสาร Lightning เพื่อเสนอการปิดช่องทาง
- Bob ตกลง และทั้งสองฝ่ายไม่ทำธุรกรรมเพิ่มเติมในช่องทางนี้


![LNP201](assets/en/031.webp)



- Alice และ Bob เจรจาค่าธรรมเนียมของ **ธุรกรรมปิด** ร่วมกัน ค่าธรรมเนียมเหล่านี้มักจะคำนวณตามตลาดค่าธรรมเนียมของ Bitcoin ในขณะที่ปิดช่องทาง สิ่งสำคัญคือต้องทราบว่า **เป็นผู้ที่เปิดช่องทางเสมอ** (ในตัวอย่างของเรา Alice) ที่จะต้องจ่ายค่าธรรมเนียมการปิด
- พวกเขาสร้าง **ธุรกรรมการปิดใหม่** ธุรกรรมนี้คล้ายกับ commitment transaction แต่ไม่มีการล็อกเวลา (timelocks) หรือกลไกการเพิกถอน เนื่องจากทั้งสองฝ่ายให้ความร่วมมือและไม่มีความเสี่ยงในการโกง ดังนั้นธุรกรรมการปิดแบบร่วมมือนี้จึงแตกต่างจากธุรกรรมการผูกมัด


ตัวอย่างเช่น หาก Alice เป็นเจ้าของ **100,000 satoshis** และ Bob เป็นเจ้าของ **30,000 satoshis** การทำธุรกรรมปิดจะส่ง **100,000 satoshis** ไปยังที่อยู่ของ Alice และ **30,000 satoshis** ไปยังที่อยู่ของ Bob โดยไม่มีข้อจำกัดด้านเวลาล็อค เมื่อการทำธุรกรรมนี้ได้รับการลงนามโดยทั้งสองฝ่ายแล้ว จะถูกเผยแพร่โดย Alice เมื่อการทำธุรกรรมได้รับการยืนยันบนบล็อกเชนของ Bitcoin ช่องทาง Lightning จะถูกปิดอย่างเป็นทางการ


![LNP201](assets/en/032.webp)


**การปิดแบบร่วมมือ** เป็นวิธีการปิดที่นิยมใช้เพราะรวดเร็ว (ไม่มีการล็อคเวลา) และค่าธรรมเนียมการทำธุรกรรมถูกปรับตามสภาพตลาด Bitcoin ในปัจจุบัน วิธีนี้ช่วยหลีกเลี่ยงการจ่ายน้อยเกินไปซึ่งอาจเสี่ยงต่อการบล็อกธุรกรรมในเมมพูล หรือการจ่ายเกินความจำเป็นซึ่งส่งผลให้เกิดการสูญเสียทางการเงินที่ไม่จำเป็นสำหรับผู้เข้าร่วม


### ข้อเสีย: การปิดตัวลงอย่างบังคับ


เมื่อโหนดของ Alice ส่งข้อความไปยัง Bob เพื่อขอปิดแบบร่วมมือกัน หากเขาไม่ตอบสนอง (เช่น เนื่องจากการขัดข้องของอินเทอร์เน็ตหรือปัญหาทางเทคนิค) Alice สามารถดำเนินการ **ปิดแบบบังคับ** โดยการเผยแพร่ **commitment transaction ที่ลงนามล่าสุด** ได้

ในกรณีนี้ Alice จะเผยแพร่เฉพาะ commitment transaction ล่าสุด ซึ่งสะท้อนถึงสถานะของช่องทางในขณะที่การทำธุรกรรม Lightning ครั้งล่าสุดเกิดขึ้น พร้อมกับการกระจายเงินทุนที่ถูกต้อง


![LNP201](assets/en/033.webp)


ธุรกรรมนี้รวมถึง **timelock** สำหรับเงินทุนของ Alice ทำให้การปิดช้าลง


![LNP201](assets/en/034.webp)


นอกจากนี้ ค่าธรรมเนียมที่เกี่ยวข้องกับ commitment transaction อาจไม่เหมาะสมในขณะที่ปิดบัญชี เนื่องจากถูกกำหนดไว้เมื่อมีการสร้างธุรกรรม ซึ่งมักจะเป็นหลายเดือนก่อนหน้านี้ โดยทั่วไปแล้ว ลูกค้า Lightning มักจะประเมินค่าธรรมเนียมสูงเกินไปเพื่อหลีกเลี่ยงปัญหาในอนาคต แต่สิ่งนี้อาจนำไปสู่ค่าธรรมเนียมที่มากเกินไปหรือในทางกลับกัน ค่าธรรมเนียมที่ต่ำเกินไป


โดยสรุป, **การปิดแบบบังคับ** เป็นทางเลือกสุดท้ายเมื่อคู่สัญญาไม่ตอบสนองอีกต่อไป มันช้ากว่าและไม่ประหยัดเท่ากับการปิดแบบร่วมมือ ดังนั้นควรหลีกเลี่ยงเมื่อเป็นไปได้


### การโกง: การทุจริต


ในที่สุด การปิดบัญชีด้วย **การโกง** เกิดขึ้นเมื่อฝ่ายใดฝ่ายหนึ่งพยายามที่จะเผยแพร่ commitment transaction เก่า มักจะเกิดขึ้นเมื่อพวกเขาถือเงินมากกว่าที่พวกเขามีสิทธิ์ได้รับ ตัวอย่างเช่น Alice อาจเผยแพร่ธุรกรรมเก่าที่เธอเป็นเจ้าของ **120,000 satoshis** ในขณะที่เธอเป็นเจ้าของเพียง **100,000** ในขณะนี้


![LNP201](assets/en/035.webp)


Bob เพื่อตรวจสอบการโกงนี้ จะเฝ้าดูบล็อกเชน Bitcoin และ mempool เพื่อให้แน่ใจว่า Alice จะไม่เผยแพร่ธุรกรรมเก่า หาก Bob ตรวจพบความพยายามในการโกง เขาสามารถใช้ **กุญแจเพิกถอน** เพื่อกู้คืนเงินของ Alice และลงโทษเธอโดยการยึดเงินทั้งหมดของช่องทาง เนื่องจาก Alice ถูกบล็อกโดย timelock ในผลลัพธ์ของเธอ Bob จึงมีเวลาที่จะใช้จ่ายโดยไม่มี timelock ในฝั่งของเขา ทำให้เขาสามารถกู้คืนเงินทั้งหมดไปยังที่อยู่ที่เขาเป็นเจ้าของได้


![LNP201](assets/en/036.webp)


เห็นได้ชัดว่าการโกงอาจประสบความสำเร็จหาก Bob ไม่ดำเนินการภายในเวลาที่กำหนดโดยการล็อคเวลาบนผลลัพธ์ของ Alice ในกรณีนี้ ผลลัพธ์ของ Alice จะถูกปลดล็อค ทำให้เธอสามารถใช้มันเพื่อสร้างผลลัพธ์ใหม่ไปยังที่อยู่ที่เธอควบคุมได้


**คุณควรนำอะไรไปจากบทนี้?**


มีสามวิธีในการปิดช่อง:



- การปิดแบบร่วมมือ**: รวดเร็วและมีค่าใช้จ่ายน้อยกว่า โดยที่ทั้งสองฝ่ายตกลงที่จะปิดช่องทางและเผยแพร่ธุรกรรมการปิดที่ปรับแต่งแล้ว
- การปิดบังคับ**: ไม่น่าพึงประสงค์เท่าไหร่ เนื่องจากต้องพึ่งพาการเผยแพร่ commitment transaction ซึ่งอาจมีค่าธรรมเนียมที่ไม่เหมาะสมและการล็อกเวลา ซึ่งทำให้การปิดช้าลง
- การโกง**: หากฝ่ายใดฝ่ายหนึ่งพยายามขโมยเงินโดยการเผยแพร่ธุรกรรมเก่า อีกฝ่ายสามารถใช้กุญแจเพิกถอนเพื่อลงโทษการโกงนี้ได้


ในบทต่อๆ ไป เราจะสำรวจ Lightning Network จากมุมมองที่กว้างขึ้น โดยเน้นที่การทำงานในฐานะเครือข่าย


# เครือข่ายสภาพคล่อง

## Lightning Network


<chapterId>45a7252c-fa4f-554b-b8bb-47449532918e</chapterId>


:::video id=38419c23-5592-4573-b0a7-84824a5bfb77:::

ในบทนี้ เราจะสำรวจว่าการชำระเงินบน Lightning Network สามารถไปถึงผู้รับได้อย่างไร แม้ว่าพวกเขาจะไม่ได้เชื่อมต่อโดยตรงผ่านช่องทางการชำระเงินก็ตาม Lightning เป็น **เครือข่ายของช่องทางการชำระเงิน** ซึ่งอนุญาตให้ส่งเงินไปยังโหนดที่ห่างไกลผ่านช่องทางของผู้เข้าร่วมคนอื่น ๆ เราจะสำรวจว่าการชำระเงินถูกกำหนดเส้นทางอย่างไรในเครือข่าย การโอนสภาพคล่องระหว่างช่องทางเป็นอย่างไร และวิธีการคำนวณค่าธรรมเนียมการทำธุรกรรม


### เครือข่ายของช่องทางการชำระเงิน


บน Lightning Network การทำธุรกรรมสอดคล้องกับการโอนเงินระหว่างสองโหนด ดังที่เห็นในบทก่อนหน้า จำเป็นต้องเปิดช่องทางกับใครบางคนเพื่อทำธุรกรรม Lightning ช่องทางนี้อนุญาตให้ทำธุรกรรม off-chain ได้เกือบไม่จำกัดจำนวนก่อนที่จะปิดและเรียกคืนยอดคงเหลือ on-chain อย่างไรก็ตาม วิธีนี้มีข้อเสียที่ต้องการช่องทางตรงกับบุคคลอื่นเพื่อรับหรือส่งเงิน ซึ่งหมายถึงการทำธุรกรรมเปิดและปิดสำหรับแต่ละช่องทาง หากฉันวางแผนที่จะทำการชำระเงินจำนวนมากกับบุคคลนี้ การเปิดและปิดช่องทางจะคุ้มค่า ในทางกลับกัน หากฉันต้องการทำธุรกรรม Lightning เพียงไม่กี่ครั้ง การเปิดช่องทางตรงจะไม่เป็นประโยชน์ เนื่องจากจะมีการทำธุรกรรม on-chain สองครั้งสำหรับจำนวนธุรกรรม off-chain ที่จำกัด กรณีนี้อาจเกิดขึ้น ตัวอย่างเช่น เมื่อฉันต้องการชำระเงินด้วย Lightning ที่ร้านค้าโดยไม่วางแผนที่จะกลับมาอีก


ในการแก้ปัญหานี้ Lightning Network ช่วยให้สามารถกำหนดเส้นทางการชำระเงินผ่านช่องทางและโหนดตัวกลางหลายแห่ง ซึ่งทำให้สามารถทำธุรกรรมได้โดยไม่ต้องมีช่องทางตรงกับบุคคลอื่น


ตัวอย่างเช่น ลองจินตนาการว่า:



- Alice** (in orange) มีช่องทางกับ **Suzie** (in gray) โดยมี **100,000 satoshis** อยู่ฝั่งของเขาและ **30,000 satoshis** อยู่ฝั่งของ Suzie
- ซูซี่** มีช่องทางกับ **Bob** ซึ่งเธอมี **250,000 satoshis** และที่ Bob ไม่มี satoshis เลย


![LNP201](assets/en/037.webp)


หาก Alice ต้องการส่งเงินไปยัง Bob โดยไม่ต้องสร้างช่องทางโดยตรงกับเขา เธอจะต้องผ่าน Suzie และแต่ละช่องทางจะต้องปรับสภาพคล่องตามนั้น **ซาโตชิที่ส่งจะยังคงอยู่ภายในช่องทางของพวกเขาเอง**; พวกเขาไม่ได้ "ข้าม" ช่องทางจริง ๆ แต่การโอนจะทำผ่านการปรับสภาพคล่องภายในในแต่ละช่องทาง


สมมติว่า Alice ต้องการส่ง **50,000 satoshis** ไปยัง Bob:



- Alice** ส่ง 50,000 ซาโตชิให้กับ **Suzie** ในช่องทางที่พวกเขามีร่วมกัน
- ซูซี่** ทำการโอนนี้ซ้ำโดยการส่ง 50,000 ซาโตชิไปยัง **Bob** ในช่องของพวกเขา


![LNP201](assets/en/038.webp)


ดังนั้น การชำระเงินจะถูกส่งไปยัง Bob ผ่านการเคลื่อนย้ายสภาพคล่องในแต่ละช่องทาง เมื่อสิ้นสุดการดำเนินการ Alice จะมี 50,000 sats เธอได้โอน 50,000 sats จริง ๆ เนื่องจากในตอนแรกเธอมี 100,000 sats ในทางกลับกัน Bob จะมีเพิ่มขึ้นอีก 50,000 sats สำหรับซูซี่ (โหนดกลาง) การดำเนินการนี้เป็นกลาง: ในตอนแรกเธอมี 30,000 sats ในช่องทางของเธอกับ Alice และ 250,000 sats ในช่องทางของเธอกับ Bob รวมเป็น 280,000 sats หลังจากการดำเนินการ เธอถือ 80,000 sats ในช่องทางของเธอกับ Alice และ 200,000 sats ในช่องทางของเธอกับ Bob ซึ่งเป็นจำนวนรวมเดียวกันกับตอนเริ่มต้น


การโอนนี้จึงถูกจำกัดโดย **สภาพคล่องที่มีอยู่** ในทิศทางของการโอน


### การคำนวณเส้นทางและขีดจำกัดสภาพคล่อง


ลองมาดูตัวอย่างทางทฤษฎีของเครือข่ายอื่นที่มี:



- 130,000 satoshis** ในฝั่งของ Alice (สีส้ม) ในช่องของเธอกับ **Suzie** (สีเทา)
- 90,000 satoshis** บนฝั่งของ **Suzie** และ **200,000 satoshis** บนฝั่งของ **Carol** (ในสีชมพู)
- 150,000 satoshis** บนฝั่งของ **Carol** และ **100,000 satoshis** บนฝั่งของ **Bob**


![LNP201](assets/en/039.webp)


ในการกำหนดค่านี้ Alice สามารถส่งไปยัง Bob ได้สูงสุด **90,000 satoshis** เนื่องจากถูกจำกัดโดยสภาพคล่องที่เล็กที่สุดที่มีอยู่ในช่องทางจาก **ซูซี่ไปยังแครอล** ในทิศทางตรงกันข้าม (จาก Bob ไปยัง Alice) ไม่สามารถทำการชำระเงินได้เนื่องจากฝั่งของ **ซูซี่** ในช่องทางกับ **Alice** ไม่มี satoshis ดังนั้นจึง **ไม่มีเส้นทาง** ที่สามารถใช้สำหรับการโอนในทิศทางนี้

Alice ส่ง **40,000 satoshis** ไปยัง Bob ผ่านช่องทาง:



- Alice โอน 40,000 ซาโตชิไปยังช่องของเธอกับซูซี่
- ซูซี่โอน 40,000 ซาโตชิให้แครอลในช่องที่พวกเขาใช้ร่วมกัน
- แครอลโอน 40,000 ซาโตชิไปยัง Bob ในที่สุด


![LNP201](assets/en/040.webp)


**satoshis ที่ส่ง**ในแต่ละช่องทาง**ยังคงอยู่ในช่องทาง**ดังนั้น satoshis ที่ Carol ส่งไปยัง Bob จึงไม่เหมือนกับที่ Alice ส่งไปยัง Suzie การโอนจะทำได้โดยการปรับสภาพคล่องภายในแต่ละช่องทางเท่านั้น นอกจากนี้ ความจุรวมของช่องทางยังคงไม่เปลี่ยนแปลง


![LNP201](assets/en/041.webp)


เช่นเดียวกับในตัวอย่างก่อนหน้า หลังจากการทำธุรกรรม โหนดต้นทาง (Alice) มีซาโตชิลดลง 40,000 หน่วย โหนดกลาง (Suzie และ Carol) ยังคงมีจำนวนรวมเท่าเดิม ทำให้การดำเนินการเป็นกลางสำหรับพวกเขา สุดท้าย โหนดปลายทาง (Bob) ได้รับซาโตชิเพิ่มอีก 40,000 หน่วย


บทบาทของโหนดกลางจึงมีความสำคัญอย่างยิ่งในการทำงานของ Lightning Network พวกเขาช่วยอำนวยความสะดวกในการโอนโดยเสนอทางเลือกการชำระเงินหลายรูปแบบ เพื่อกระตุ้นให้โหนดเหล่านี้ให้สภาพคล่องและมีส่วนร่วมในการกำหนดเส้นทางการชำระเงิน **ค่าธรรมเนียมการกำหนดเส้นทาง** จะถูกจ่ายให้กับพวกเขา


### ค่าธรรมเนียมการกำหนดเส้นทาง


โหนดระดับกลางจะเรียกเก็บค่าธรรมเนียมเพื่ออำนวยความสะดวกในการชำระเงินที่ผ่านช่องทางของพวกเขา ค่าธรรมเนียมเหล่านี้ถูกกำหนดโดย **แต่ละโหนดสำหรับแต่ละช่องทาง** ค่าธรรมเนียมประกอบด้วย 2 องค์ประกอบ:



- "**Base fee**": จำนวนคงที่ต่อช่องทาง มักจะเป็น **1 sat** โดยค่าเริ่มต้น แต่สามารถปรับแต่งได้
- "**ค่าธรรมเนียมแปรผัน**": เปอร์เซ็นต์ของจำนวนเงินที่โอน คำนวณใน **ส่วนต่อหนึ่งล้าน (ppm)** โดยค่าเริ่มต้นคือ **1 ppm** (1 ซาตต่อซาตโทชิหนึ่งล้านที่โอน) แต่สามารถปรับได้เช่นกัน


ค่าธรรมเนียมจะแตกต่างกันไปขึ้นอยู่กับทิศทางของการโอน ตัวอย่างเช่น สำหรับการโอนจาก Alice ไปยัง Suzie จะใช้ค่าธรรมเนียมของ Alice ในทางกลับกัน จาก Suzie ไปยัง Alice จะใช้ค่าธรรมเนียมของ Suzie


ตัวอย่างเช่น สำหรับช่องทางระหว่าง Alice และ Suzie เราอาจมี:



- Alice**: ค่าธรรมเนียมพื้นฐาน 1 sat และ 1 ppm สำหรับค่าธรรมเนียมที่เปลี่ยนแปลงได้
- ซูซี่**: ค่าธรรมเนียมพื้นฐาน 0.5 sat และ 10 ppm สำหรับค่าธรรมเนียมแปรผัน


![LNP201](assets/en/042.webp)


เพื่อให้เข้าใจวิธีการทำงานของค่าธรรมเนียมได้ดียิ่งขึ้น มาศึกษา Lightning Network เดิมอีกครั้ง แต่คราวนี้มีค่าธรรมเนียมการส่งต่อดังนี้:



- ช่อง **Alice - ซูซี่**: ค่าธรรมเนียมพื้นฐาน 1 satoshi และ 1 ppm สำหรับ Alice.
- ช่อง **Suzie - Carol**: ค่าธรรมเนียมพื้นฐาน 0 satoshi และ 200 ppm สำหรับ Suzie.
- แครอล - Bob** ช่องทาง: ค่าธรรมเนียมพื้นฐาน 1 satoshi และ 1 ppm สำหรับซูซี่ 2.

![LNP201](assets/en/043.webp)


สำหรับการชำระเงินเดียวกันจำนวน **40,000 satoshis** ไปยัง Bob, Alice จะต้องส่งเพิ่มขึ้นเล็กน้อย เนื่องจากโหนดตัวกลางแต่ละตัวจะหักค่าธรรมเนียม:



- แครอล** หัก 1.04 ซาโตชิในช่องกับ Bob:

$$ f_{\text{Carol-Bob}} = \text{base fee} + \left(\frac{\text{ppm} \times \text{amount}}{10^6}\right) $$

$$ f_{\text{Carol-Bob}} = 1 + \frac{1 \times 40000}{10^6} = 1 + 0.04 = 1.04 \text{ sats} $$



- ซูซี่** หักค่าธรรมเนียม 8 ซาโตชิในช่องทางกับแครอล:

$$ f_{\text{Suzie-Carol}} = \text{ค่าธรรมเนียมพื้นฐาน} + \left(\frac{\text{ppm} \times \text{จำนวนเงิน}}{10^6}\right) $$

$$ f_{\text{Suzie-Carol}} = 0 + \frac{200 \times 40001.04}{10^6} = 0 + 8.0002 \approx 8 \text{ sats} $$


ค่าธรรมเนียมทั้งหมดสำหรับการชำระเงินในเส้นทางนี้คือ **9.04 satoshis** ดังนั้น Alice ต้องส่ง **40,009.04 satoshis** เพื่อให้ Bob ได้รับ **40,000 satoshis** พอดี


![LNP201](assets/en/044.webp)


สภาพคล่องจึงได้รับการอัปเดต:


![LNP201](assets/en/045.webp)


### ออเนียนรูทติ้ง


ในการส่งต่อการชำระเงินจากผู้ส่งไปยังผู้รับ Lightning Network ใช้วิธีที่เรียกว่า "**onion routing**" ซึ่งแตกต่างจากการส่งต่อข้อมูลแบบคลาสสิกที่แต่ละเราเตอร์จะตัดสินใจทิศทางของข้อมูลตามจุดหมายปลายทาง onion routing ทำงานแตกต่างออกไป:



- โหนดที่ส่งคำนวณเส้นทางทั้งหมด**: ตัวอย่างเช่น Alice กำหนดว่าการชำระเงินของเธอต้องผ่าน Suzie และ Carol ก่อนที่จะถึง Bob.
- โหนดตัวกลางแต่ละตัวจะรู้จักเพียงเพื่อนบ้านที่อยู่ติดกันเท่านั้น**: ซูซี่รู้เพียงว่าเธอได้รับเงินจาก Alice และเธอต้องโอนเงินนั้นไปให้แครอล อย่างไรก็ตาม ซูซี่ไม่รู้ว่า Alice เป็นโหนดต้นทางหรือโหนดตัวกลาง และเธอก็ไม่รู้ว่าแครอลเป็นโหนดผู้รับหรือเป็นเพียงโหนดตัวกลางอีกตัว หลักการนี้ยังใช้กับแครอลและโหนดอื่น ๆ ทั้งหมดบนเส้นทางด้วย การกำหนดเส้นทางแบบหัวหอมจึงรักษาความลับของธุรกรรมโดยการปิดบังตัวตนของผู้ส่งและผู้รับสุดท้าย

เพื่อให้แน่ใจว่าโหนดที่ส่งสามารถคำนวณเส้นทางที่สมบูรณ์ไปยังผู้รับใน onion routing ได้ จะต้องรักษา **กราฟเครือข่าย** เพื่อทราบโครงสร้างเครือข่ายและกำหนดเส้นทางที่เป็นไปได้

**คุณควรนำอะไรไปจากบทนี้?**



- บน Lightning การชำระเงินสามารถถูกส่งผ่านโหนดที่เชื่อมต่อกันทางอ้อมผ่านช่องทางกลาง แต่ละโหนดกลางเหล่านี้ช่วยอำนวยความสะดวกในการส่งต่อสภาพคล่อง
- โหนดตัวกลางได้รับค่าคอมมิชชั่นสำหรับบริการของพวกเขา ซึ่งประกอบด้วยค่าธรรมเนียมคงที่และค่าธรรมเนียมแปรผัน
- การกำหนดเส้นทางแบบหัวหอมช่วยให้โหนดที่ส่งสามารถคำนวณเส้นทางทั้งหมดได้โดยที่โหนดกลางไม่ทราบแหล่งที่มาหรือปลายทางสุดท้าย


ในบทนี้ เราได้สำรวจการกำหนดเส้นทางการชำระเงินบน Lightning Network แต่มีคำถามเกิดขึ้น: อะไรที่ป้องกันโหนดตัวกลางจากการยอมรับการชำระเงินที่เข้ามาโดยไม่ส่งต่อไปยังปลายทางถัดไป โดยมีจุดประสงค์เพื่อดักจับธุรกรรม? นี่คือบทบาทของ HTLCs ที่เราจะศึกษาในบทถัดไป


## HTLC – การล็อกเวลาด้วยแฮช Contract


<chapterId>4369b85a-1365-55d8-99e1-509088210116</chapterId>


:::video id=6f204b92-55a5-4939-9440-7c5b96a297bf:::

ในบทนี้ เราจะสำรวจว่า Lightning ช่วยให้การชำระเงินผ่านโหนดตัวกลางโดยไม่ต้องอาศัยความไว้วางใจได้อย่างไร ด้วยความช่วยเหลือของ **[HTLC](https://planb.academy/resources/glossary/htlc)** (_Hashed Time-Locked Contracts_) สัญญาอัจฉริยะเหล่านี้ทำให้มั่นใจได้ว่าโหนดตัวกลางแต่ละโหนดจะได้รับเงินจากช่องของตนก็ต่อเมื่อส่งต่อการชำระเงินไปยังผู้รับปลายทางเท่านั้น มิฉะนั้น การชำระเงินจะไม่ได้รับการยืนยัน


ปัญหาที่เกิดขึ้นสำหรับการกำหนดเส้นทางการชำระเงินคือความจำเป็นในการไว้วางใจในโหนดตัวกลาง รวมถึงความไว้วางใจระหว่างโหนดตัวกลางเหล่านี้ด้วย เพื่อแสดงให้เห็นสิ่งนี้ ลองกลับไปดูตัวอย่างเครือข่าย Lightning ที่เรียบง่ายของเราที่มี 3 โหนดและ 2 ช่องทาง:



- Alice มีช่องกับซูซี่
- ซูซี่มีช่องกับ Bob.


Alice ต้องการส่ง 40,000 sats ไปยัง Bob แต่เธอไม่มีช่องทางตรงกับเขาและไม่ต้องการเปิดช่องทางใหม่ เธอจึงมองหาช่องทางและตัดสินใจผ่านโหนดของซูซี่


![LNP201](assets/en/046.webp)


หาก Alice ส่ง 40,000 ซาโตชิให้ซูซี่โดยไม่คิดหน้าคิดหลัง หวังว่าซูซี่จะโอนจำนวนนี้ให้กับ Bob ซูซี่อาจเก็บเงินนี้ไว้เองและไม่ส่งต่ออะไรให้กับ Bob เลยก็ได้


![LNP201](assets/en/047.webp)

เพื่อหลีกเลี่ยงสถานการณ์นี้ บน Lightning เราใช้ HTLCs (Hashed Time-Locked Contracts) ซึ่งทำให้การชำระเงินไปยังโหนดตัวกลางมีเงื่อนไข หมายความว่า Suzie ต้องปฏิบัติตามเงื่อนไขบางประการเพื่อเข้าถึงเงินทุนของ Alice และโอนให้กับ Bob


### วิธีการทำงานของ HTLCs


HTLC เป็นสัญญาพิเศษที่อิงตามหลักการสองประการ:



- เงื่อนไขการเข้าถึง**: ผู้รับจะต้องเปิดเผยความลับเพื่อปลดล็อกการชำระเงินที่ถึงกำหนดแก่พวกเขา
- การหมดอายุ**: หากการชำระเงินไม่ได้รับการดำเนินการเสร็จสิ้นภายในระยะเวลาที่กำหนด การชำระเงินจะถูกยกเลิก และเงินจะถูกส่งกลับไปยังผู้ส่ง


นี่คือวิธีการทำงานของกระบวนการนี้ในตัวอย่างของเรากับ Alice, Suzie, และ Bob:


![LNP201](assets/en/048.webp)


**การสร้างความลับ**: Bob สร้างความลับแบบสุ่มที่เรียกว่า _s_ (ภาพก่อนแปลง) และคำนวณแฮชของมันที่เรียกว่า _r_ ด้วยฟังก์ชันแฮชที่เรียกว่า _h_ เรามี:


$$
r = h(s)
$$


การใช้ฟังก์ชันแฮชทำให้ไม่สามารถหาค่า _s_ ได้จาก _h(s)_ เพียงอย่างเดียว แต่ถ้าให้ค่า _s_ มา จะสามารถตรวจสอบได้ง่ายว่ามันสอดคล้องกับ _h(s)_ หรือไม่


![LNP201](assets/en/049.webp)


**การส่งคำขอชำระเงิน**: Bob ส่ง **ใบแจ้งหนี้** ไปยัง Alice เพื่อขอชำระเงิน ใบแจ้งหนี้นี้มีการระบุแฮช _r_ อย่างชัดเจน


![LNP201](assets/en/050.webp)


**การส่งการชำระเงินแบบมีเงื่อนไข**: Alice ส่ง HTLC จำนวน 40,000 satoshis ให้กับ Suzie เงื่อนไขที่ Suzie จะได้รับเงินนี้คือเธอต้องให้ความลับ _s'_ กับ Alice ที่ทำให้สมการต่อไปนี้เป็นจริง:


$$
h(s') = r
$$


![LNP201](assets/en/051.webp)


**การโอน HTLC ไปยังผู้รับสุดท้าย**: ซูซี่, เพื่อรับ 40,000 ซาโตชิจาก Alice, ต้องโอน HTLC ที่คล้ายกันจำนวน 40,000 ซาโตชิไปยัง Bob, ซึ่งมีเงื่อนไขเดียวกันคือเขาต้องให้ความลับ _s'_ แก่ซูซี่ที่ตอบสนองสมการ:


$$
h(s') = r
$$


![LNP201](assets/en/052.webp)


**การตรวจสอบโดยความลับ _s_**: Bob ให้ _s_ กับซูซี่เพื่อรับ 40,000 ซาโตชิที่สัญญาไว้ใน HTLC ด้วยความลับนี้ ซูซี่สามารถปลดล็อก Alice ของ HTLC และรับ 40,000 ซาโตชิจาก Alice การชำระเงินจึงถูกส่งไปยัง Bob อย่างถูกต้อง


![LNP201](assets/en/053.webp)

กระบวนการนี้ป้องกันไม่ให้ซูซี่เก็บเงินของ Alice โดยไม่ทำการโอนให้กับ Bob เนื่องจากเธอต้องส่งการชำระเงินไปยัง Bob เพื่อรับความลับ _s_ และปลดล็อก HTLC ของ Alice การดำเนินการยังคงเหมือนเดิมแม้ว่าจะมีเส้นทางที่รวมถึงโหนดตัวกลางหลายตัว: มันเป็นเพียงเรื่องของการทำซ้ำขั้นตอนของซูซี่สำหรับแต่ละโหนดตัวกลาง แต่ละโหนดได้รับการป้องกันโดยเงื่อนไขของ HTLCs เนื่องจากการปลดล็อก HTLC สุดท้ายโดยผู้รับจะทำให้เกิดการปลดล็อก HTLCs อื่น ๆ ทั้งหมดโดยอัตโนมัติในลักษณะลูกโซ่


### การหมดอายุและการจัดการ HTLCs ในกรณีที่เกิดปัญหา


หากในระหว่างกระบวนการชำระเงิน หนึ่งในโหนดตัวกลางหรือโหนดผู้รับหยุดตอบสนอง โดยเฉพาะในกรณีที่เกิดการขัดข้องของอินเทอร์เน็ตหรือไฟฟ้า การชำระเงินจะไม่สามารถเสร็จสมบูรณ์ได้เนื่องจากความลับที่จำเป็นในการปลดล็อก HTLCs ไม่ถูกส่งผ่าน ยกตัวอย่างเช่นในกรณีของ Alice, Suzie, และ Bob ปัญหานี้เกิดขึ้นหาก Bob ล้มเหลวในการส่งผ่านความลับ _s_ ไปยัง Suzie ในกรณีนี้ HTLCs ทั้งหมดที่อยู่ต้นทางของเส้นทางจะถูกบล็อก และเงินทุนที่พวกมันรักษาความปลอดภัยไว้ก็จะถูกบล็อกเช่นกัน


![LNP201](assets/en/054.webp)


เพื่อหลีกเลี่ยงสิ่งนี้ HTLCs บน Lightning มีการหมดอายุที่อนุญาตให้ลบ HTLC ออกหากไม่เสร็จสิ้นหลังจากเวลาที่กำหนด การหมดอายุจะเป็นไปตามลำดับเฉพาะ โดยเริ่มจาก HTLC ที่ใกล้กับผู้รับมากที่สุดและจากนั้นจึงไปยังผู้ออกธุรกรรม ในตัวอย่างของเรา หาก Bob ไม่เคยให้ความลับ _s_ แก่ Suzie สิ่งนี้จะทำให้ HTLC ของ Suzie ที่มุ่งไปยัง Bob หมดอายุเป็นอันดับแรก


![LNP201](assets/en/055.webp)


จากนั้น HTLC จาก Alice ถึง Suzie.


![LNP201](assets/en/056.webp)


หากลำดับการหมดอายุถูกย้อนกลับ Alice สามารถกู้คืนการชำระเงินของเธอก่อนที่ซูซี่จะสามารถป้องกันตัวเองจากการโกงที่อาจเกิดขึ้นได้ จริงๆ แล้ว หาก Bob กลับมาเพื่อเรียกร้อง HTLC ของเขาในขณะที่ Alice ได้ถอนของเธอไปแล้ว ซูซี่จะตกอยู่ในสถานการณ์ที่เสียเปรียบ ลำดับการหมดอายุของ HTLC ที่เป็นแบบลูกโซ่นี้จึงทำให้มั่นใจได้ว่าไม่มีโหนดตัวกลางใดที่ต้องประสบกับการสูญเสียที่ไม่เป็นธรรม


### การแสดงผลของ HTLCs ในธุรกรรมการผูกพัน


ธุรกรรม Commitment แสดงถึง HTLCs ในลักษณะที่เงื่อนไขที่พวกเขากำหนดใน Lightning สามารถถูกโอนไปยัง Bitcoin ในกรณีที่มีการปิดช่องทางบังคับในระหว่างอายุการใช้งานของ HTLC เพื่อเป็นการเตือนความจำ ธุรกรรมการผูกมัดแสดงถึงสถานะปัจจุบันของช่องทางระหว่างผู้ใช้สองคนและอนุญาตให้มีการปิดบังคับฝ่ายเดียวในกรณีที่มีปัญหา ด้วยแต่ละสถานะใหม่ของช่องทาง จะมีการสร้างธุรกรรมการผูกมัด 2 รายการ: หนึ่งรายการสำหรับแต่ละฝ่าย ลองกลับไปดูตัวอย่างของเรากับ Alice, ซูซี่, และ Bob แต่ดูอย่างใกล้ชิดมากขึ้นว่าเกิดอะไรขึ้นที่ระดับช่องทางระหว่าง Alice และซูซี่เมื่อ HTLC ถูกสร้างขึ้น

![LNP201](assets/en/057.webp)


ก่อนเริ่มการชำระเงิน 40,000 sats ระหว่าง Alice และ Bob, Alice มี 100,000 sats ในช่องของเธอกับ Suzie, ในขณะที่ Suzie ถือ 30,000. การทำธุรกรรมที่พวกเขาตกลงกันมีดังนี้:


![LNP201](assets/en/058.webp)


Alice เพิ่งได้รับใบแจ้งหนี้ของ Bob ซึ่งมี _r_ ซึ่งเป็นแฮชของความลับ เธอจึงสามารถสร้าง HTLC จำนวน 40,000 ซาโตชิกับซูซี่ได้ HTLC นี้แสดงในธุรกรรมการยืนยันล่าสุดเป็นเอาต์พุตที่เรียกว่า "**_HTLC Out_**" ในฝั่งของ Alice เนื่องจากเงินทุนกำลังออกไป และ "**_HTLC In_**" ในฝั่งของซูซี่ เนื่องจากเงินทุนกำลังเข้ามา


![LNP201](assets/en/059.webp)


เอาต์พุตเหล่านี้ที่เกี่ยวข้องกับ HTLC มีเงื่อนไขเดียวกันทุกประการ ได้แก่:



- หากซูซี่สามารถให้ความลับ _s_ ได้ เธอสามารถปลดล็อกผลลัพธ์นี้ได้ทันทีและโอนไปยังที่อยู่ที่เธอควบคุมได้
- หากซูซี่ไม่มีความลับ _s_ เธอจะไม่สามารถปลดล็อกผลลัพธ์นี้ได้ และ Alice จะสามารถปลดล็อกได้หลังจากการล็อกเวลาเพื่อส่งไปยังที่อยู่ที่เธอควบคุม การล็อกเวลาดังกล่าวจึงให้ช่วงเวลาหนึ่งแก่ซูซี่ในการตอบสนองหากเธอได้รับ _s_


เงื่อนไขเหล่านี้ใช้เฉพาะเมื่อช่องทางถูกปิด (เช่น commitment transaction ถูกเผยแพร่ on-chain) ในขณะที่ HTLC ยังคงใช้งานอยู่บน Lightning ซึ่งหมายความว่าการชำระเงินระหว่าง Alice และ Bob ยังไม่เสร็จสมบูรณ์ และ HTLCs ยังไม่หมดอายุ ด้วยเงื่อนไขเหล่านี้ ซูซี่สามารถกู้คืน 40,000 ซาโตชิของ HTLC ที่เป็นหนี้เธอได้โดยการให้ _s_ มิฉะนั้น Alice จะกู้คืนเงินหลังจากการหมดอายุของ timelock เพราะถ้าซูซี่ไม่รู้ _s_ หมายความว่าเธอยังไม่ได้โอน 40,000 ซาโตชิไปยัง Bob และดังนั้น เงินของ Alice จึงไม่เป็นหนี้เธอ


ยิ่งไปกว่านั้น หากช่องทางถูกปิดในขณะที่มี HTLCs หลายรายการที่ยังค้างอยู่ จะมีผลลัพธ์เพิ่มเติมมากเท่ากับจำนวน HTLCs ที่กำลังดำเนินการอยู่

หากช่องทางไม่ได้ถูกปิด การทำธุรกรรมการผูกพันใหม่จะถูกสร้างขึ้นหลังจากการหมดอายุหรือความสำเร็จของการชำระเงิน Lightning เพื่อสะท้อนถึงสถานะใหม่ที่เสถียรของช่องทาง ซึ่งตอนนี้ไม่มี HTLCs ที่ค้างอยู่แล้ว ดังนั้นเอาต์พุตที่เกี่ยวข้องกับ HTLCs สามารถถูกลบออกจากการทำธุรกรรมการผูกพันได้

![LNP201](assets/en/060.webp)


สุดท้ายนี้ ในกรณีที่มีการปิดช่องทางแบบร่วมมือกันในขณะที่ HTLC กำลังทำงานอยู่ Alice และ Suzie จะหยุดรับการชำระเงินใหม่และรอการแก้ไขหรือการหมดอายุของ HTLCs ที่กำลังดำเนินอยู่ ซึ่งจะช่วยให้พวกเขาสามารถเผยแพร่ธุรกรรมการปิดที่เบากว่าได้ โดยไม่มีผลลัพธ์ที่เกี่ยวข้องกับ HTLCs ซึ่งจะช่วยลดค่าธรรมเนียมและหลีกเลี่ยงการรอคอยสำหรับการล็อคเวลา (timelock) ที่อาจเกิดขึ้นได้


**คุณควรนำอะไรไปจากบทนี้?**


HTLCs ช่วยให้สามารถส่งผ่านการชำระเงินด้วย Lightning ผ่านหลายโหนดได้โดยไม่ต้องเชื่อถือในโหนดเหล่านั้น นี่คือประเด็นสำคัญที่ควรจำ:



- HTLCs รับรองความปลอดภัยของการชำระเงินผ่านความลับ (preimage) และเวลาหมดอายุ
- การแก้ไขหรือการหมดอายุของ HTLCs จะเป็นไปตามลำดับเฉพาะ: จากปลายทางไปยังแหล่งที่มา เพื่อปกป้องแต่ละโหนด
- ตราบใดที่ HTLC ยังไม่ได้รับการแก้ไขหรือหมดอายุ มันจะถูกเก็บรักษาไว้เป็นผลลัพธ์ในธุรกรรมข้อผูกพันล่าสุด


ในบทถัดไป เราจะค้นพบว่าน็อดที่ออกธุรกรรม Lightning ค้นหาและเลือกเส้นทางสำหรับการชำระเงินเพื่อไปยังน็อดผู้รับได้อย่างไร

ในบทก่อนหน้านี้ เราได้เห็นวิธีการใช้ช่องทางของโหนดอื่นเพื่อกำหนดเส้นทางการชำระเงินและเข้าถึงโหนดโดยไม่ต้องเชื่อมต่อโดยตรงกับมันผ่านช่องทาง นอกจากนี้เรายังได้พูดคุยเกี่ยวกับวิธีการรับรองความปลอดภัยของการโอนโดยไม่ต้องไว้วางใจโหนดตัวกลาง ในบทนี้ เราจะมุ่งเน้นไปที่การค้นหาเส้นทางที่เหมาะสมที่สุดเพื่อเข้าถึงโหนดเป้าหมาย


### ปัญหาการกำหนดเส้นทางใน Lightning


ตามที่เราได้เห็น ใน Lightning เป็นโหนดที่ส่งการชำระเงินที่ต้องคำนวณเส้นทางทั้งหมดไปยังผู้รับ เพราะเราใช้ระบบการส่งข้อมูลแบบหัวหอม โหนดตัวกลางไม่ทราบทั้งจุดเริ่มต้นหรือจุดหมายปลายทางสุดท้าย พวกเขารู้เพียงว่าการชำระเงินเริ่มต้นจากที่ใดและต้องโอนไปยังโหนดใดต่อไป ซึ่งหมายความว่าโหนดที่ส่งต้องรักษาโทโพโลยีท้องถิ่นแบบไดนามิกของเครือข่าย รวมถึงโหนด Lightning ที่มีอยู่และช่องทางระหว่างพวกเขา โดยคำนึงถึงการเปิด การปิด และการอัปเดตสถานะ


![LNP201](assets/en/061.webp)

แม้จะมีโทโพโลยีของ Lightning Network ข้อมูลสำคัญสำหรับการกำหนดเส้นทางยังคงไม่สามารถเข้าถึงได้สำหรับโหนดที่ส่ง โดยเฉพาะการกระจายสภาพคล่องที่แน่นอนในช่องทางต่างๆ ในช่วงเวลาใดเวลาหนึ่ง อันที่จริง แต่ละช่องทางจะแสดงเพียง **ความจุทั้งหมด** ของมัน แต่การกระจายเงินภายในเป็นที่รู้กันเฉพาะโหนดสองตัวที่เข้าร่วมเท่านั้น สิ่งนี้ก่อให้เกิดความท้าทายสำหรับการกำหนดเส้นทางอย่างมีประสิทธิภาพ เนื่องจากความสำเร็จของการชำระเงินขึ้นอยู่กับว่าจำนวนเงินนั้นน้อยกว่าสภาพคล่องต่ำสุดในเส้นทางที่เลือกหรือไม่ อย่างไรก็ตาม สภาพคล่องไม่ได้ทั้งหมดมองเห็นได้สำหรับโหนดที่ส่ง

![LNP201](assets/en/062.webp)


### อัปเดตแผนที่เครือข่าย


เพื่อให้แผนที่เครือข่ายของพวกเขาทันสมัยอยู่เสมอ โหนดจะแลกเปลี่ยนข้อความเป็นประจำผ่านอัลกอริทึมที่เรียกว่า "**_gossip_**" นี่คืออัลกอริทึมแบบกระจายที่ใช้ในการเผยแพร่ข้อมูลในลักษณะระบาดไปยังโหนดทั้งหมดในเครือข่าย ทำให้สามารถแลกเปลี่ยนและซิงโครไนซ์สถานะทั่วโลกของช่องทางได้ภายในไม่กี่รอบการสื่อสาร แต่ละโหนดจะเผยแพร่ข้อมูลไปยังเพื่อนบ้านหนึ่งหรือหลายคน ซึ่งเลือกแบบสุ่มหรือไม่ก็ได้; จากนั้นเพื่อนบ้านเหล่านี้จะเผยแพร่ข้อมูลไปยังเพื่อนบ้านอื่น ๆ ต่อไป และทำเช่นนี้ไปเรื่อย ๆ จนกว่าจะได้สถานะที่ซิงโครไนซ์ทั่วโลก


ข้อความหลัก 2 ข้อที่แลกเปลี่ยนระหว่างโหนด Lightning มีดังนี้:



- "**ประกาศช่อง**": ข้อความที่ส่งสัญญาณการเปิดช่องใหม่
- "**Channel Updates**": อัปเดตข้อความเกี่ยวกับสถานะของช่องทาง โดยเฉพาะการเปลี่ยนแปลงของค่าธรรมเนียม (แต่ไม่เกี่ยวกับการกระจายสภาพคล่อง)


โหนด Lightning ยังตรวจสอบบล็อกเชน Bitcoin เพื่อค้นหาธุรกรรมที่ปิดช่องทาง ช่องทางที่ปิดแล้วจะถูกลบออกจากแผนที่เนื่องจากไม่สามารถใช้ในการส่งผ่านการชำระเงินของเราได้อีกต่อไป


### กำลังจัดการการชำระเงิน


ลองพิจารณาตัวอย่างของ Lightning Network ขนาดเล็กที่มี 7 โหนด: Alice, Bob, 1, 2, 3, 4, และ 5 ลองจินตนาการว่า Alice ต้องการส่งการชำระเงินไปยัง Bob แต่ต้องผ่านโหนดตัวกลาง


![LNP201](assets/en/063.webp)


นี่คือการกระจายเงินทุนในช่องทางเหล่านี้:



- ช่องทางระหว่าง Alice และ 1**: 250,000 sats ที่ฝั่งของ Alice, 80,000 ที่ฝั่งของ 1 (ความจุรวม 330,000 sats)
- ช่องระหว่าง 1 และ 2**: 300,000 sats ฝั่งของ 1, 200,000 ฝั่งของ 2 (ความจุรวม 500,000 sats)
- Channel between 2 and 3**: 50,000 sats on 2's side, 60,000 on 3's side (total capacity of 110,000 sats).
- ช่องระหว่าง 2 และ 5**: 90,000 sats ด้านที่ 2, 160,000 ด้านที่ 5 (ความจุรวม 250,000 sats)
- ช่องระหว่าง 2 และ 4**: 180,000 sats ด้านที่ 2, 110,000 ด้านที่ 4 (ความจุรวม 290,000 sats)
- ช่องระหว่าง 4 และ 5**: 200,000 sats ด้านที่ 4, 10,000 ด้านที่ 5 (ความจุรวม 210,000 sats)
- ช่องระหว่าง 3 และ Bob**: 50,000 sats ด้านที่ 3, 250,000 ด้าน Bob (ความจุรวม 300,000 sats)
- ช่องทางระหว่าง 5 และ Bob**: 260,000 sats ด้าน 5, 100,000 ด้าน Bob (ความจุรวม 360,000 sats)


![LNP201](assets/en/064.webp)


ในการชำระเงินจำนวน 100,000 sats จาก Alice ไปยัง Bob ตัวเลือกการกำหนดเส้นทางถูกจำกัดโดยสภาพคล่องที่มีอยู่ในแต่ละช่องทาง เส้นทางที่เหมาะสมที่สุดสำหรับ Alice ตามการกระจายสภาพคล่องที่ทราบอาจเป็นลำดับ `Alice → 1 → 2 → 4 → 5 → Bob`:


![LNP201](assets/en/065.webp)


แต่เนื่องจาก Alice ไม่ทราบการกระจายเงินทุนที่แน่นอนในแต่ละช่องทาง เธอจึงต้องประมาณเส้นทางที่เหมาะสมที่สุดในเชิงความน่าจะเป็น โดยคำนึงถึงเกณฑ์ต่อไปนี้:



- ความน่าจะเป็นของความสำเร็จ**: ช่องทางที่มีความจุรวมสูงกว่ามีแนวโน้มที่จะมีสภาพคล่องเพียงพอมากกว่า ตัวอย่างเช่น ช่องทางระหว่างโหนด 2 และโหนด 3 มีความจุรวม 110,000 sats ดังนั้นจึงไม่น่าจะพบ 100,000 sats หรือมากกว่าที่ฝั่งของโหนด 2 แม้ว่าจะยังคงเป็นไปได้ก็ตาม
- ค่าธรรมเนียมการทำธุรกรรม**: ในการเลือกเส้นทางที่ดีที่สุด โหนดที่ส่งยังพิจารณาค่าธรรมเนียมที่ใช้โดยโหนดกลางแต่ละโหนดและพยายามลดต้นทุนการกำหนดเส้นทางทั้งหมดให้เหลือน้อยที่สุด
- การหมดอายุของ HTLCs**: เพื่อหลีกเลี่ยงการชำระเงินที่ถูกบล็อก เวลาหมดอายุของ HTLCs ก็เป็นพารามิเตอร์ที่ควรพิจารณาเช่นกัน
- จำนวนโหนดกลาง**: สุดท้ายนี้ โดยทั่วไปแล้ว โหนดที่ส่งจะพยายามหาช่องทางที่มีจำนวนโหนดน้อยที่สุดเท่าที่จะเป็นไปได้เพื่อลดความเสี่ยงของความล้มเหลวและจำกัดค่าธรรมเนียมการทำธุรกรรม Lightning


โดยการวิเคราะห์เกณฑ์เหล่านี้ โหนดที่ส่งสามารถทดสอบเส้นทางที่มีความเป็นไปได้มากที่สุดและพยายามปรับให้เหมาะสม ในตัวอย่างของเรา Alice อาจจัดอันดับเส้นทางที่ดีที่สุดดังนี้:



- `Alice → 1 → 2 → 5 → Bob`, เพราะเป็นเส้นทางที่สั้นที่สุดที่มีความจุสูงสุด
- `Alice → 1 → 2 → 4 → 5 → Bob` เพราะเส้นทางนี้มีความจุที่ดี แม้ว่าจะยาวกว่าเส้นทางแรกก็ตาม
- `Alice → 1 → 2 → 3 → Bob`, เพราะเส้นทางนี้รวมถึงช่อง `2 → 3` ซึ่งมีความจุจำกัดมาก แต่ยังคงสามารถใช้งานได้ในทางที่เป็นไปได้


### การดำเนินการชำระเงิน


Alice ตัดสินใจทดสอบเส้นทางแรกของเธอ (`Alice → 1 → 2 → 5 → Bob`) ดังนั้นเธอจึงส่ง HTLC จำนวน 100,000 sats ไปยังโหนด 1 โหนดนี้ตรวจสอบว่ามีสภาพคล่องเพียงพอกับโหนด 2 และดำเนินการส่งต่อ โหนด 2 รับ HTLC จากโหนด 1 แต่พบว่าไม่มีสภาพคล่องเพียงพอในช่องทางกับโหนด 5 เพื่อส่งการชำระเงินจำนวน 100,000 sats จากนั้นมันส่งข้อความแสดงข้อผิดพลาดกลับไปยังโหนด 1 ซึ่งส่งต่อไปยัง Alice เส้นทางนี้ล้มเหลว


![LNP201](assets/en/066.webp)


Alice พยายามที่จะส่งการชำระเงินของเธอโดยใช้เส้นทางที่สอง (`Alice → 1 → 2 → 4 → 5 → Bob`) เธอส่ง HTLC จำนวน 100,000 sats ไปยังโหนด 1 ซึ่งจากนั้นจะส่งต่อไปยังโหนด 2 และต่อไปยังโหนด 5 และสุดท้ายไปยัง Bob คราวนี้สภาพคล่องเพียงพอและเส้นทางทำงานได้ แต่ละโหนดปลดล็อก HTLC ของมันในลำดับโดยใช้ preimage ที่ให้โดย Bob (ความลับ _s_) ซึ่งทำให้การชำระเงินของ Alice ไปยัง Bob สำเร็จลุล่วงไปได้ด้วยดี


![LNP201](assets/en/067.webp)


การค้นหาเส้นทางดำเนินการดังนี้: โหนดที่ส่งเริ่มต้นโดยการระบุเส้นทางที่ดีที่สุดที่เป็นไปได้ จากนั้นพยายามชำระเงินทีละขั้นตอนจนกว่าจะพบเส้นทางที่ใช้งานได้


ควรสังเกตว่า Bob สามารถให้ข้อมูลแก่ Alice ใน **ใบแจ้งหนี้** เพื่ออำนวยความสะดวกในการกำหนดเส้นทางได้ ตัวอย่างเช่น เขาสามารถระบุช่องทางใกล้เคียงที่มีสภาพคล่องเพียงพอหรือเปิดเผยการมีอยู่ของช่องทางส่วนตัว การระบุเหล่านี้ช่วยให้ Alice หลีกเลี่ยงเส้นทางที่มีโอกาสสำเร็จน้อยและพยายามเส้นทางที่แนะนำก่อน


**คุณควรนำอะไรไปจากบทนี้?**



- โหนดจะรักษาแผนที่ของโครงสร้างเครือข่ายผ่านการประกาศและการตรวจสอบการปิดช่องทางบนบล็อกเชน Bitcoin
- การค้นหาเส้นทางที่เหมาะสมที่สุดสำหรับการชำระเงินยังคงเป็นแบบความน่าจะเป็นและขึ้นอยู่กับเกณฑ์หลายประการ
- Bob สามารถให้คำแนะนำใน **ใบแจ้งหนี้** เพื่อชี้นำการกำหนดเส้นทางของ Alice และช่วยให้เธอไม่ต้องทดสอบเส้นทางที่ไม่น่าจะเป็นไปได้


ในบทต่อไปนี้ เราจะศึกษาการทำงานของใบแจ้งหนี้โดยเฉพาะ รวมถึงเครื่องมืออื่น ๆ ที่ใช้บน Lightning Network

## Invoice, LNURL, and Keysend

ในบทนี้ เราจะพิจารณาการทำงานของ **ใบแจ้งหนี้** ของ Lightning อย่างละเอียด นั่นคือ คำขอชำระเงินที่ส่งโดยโหนดผู้รับไปยังโหนดผู้ส่ง เป้าหมายคือการทำความเข้าใจวิธีการชำระเงินและรับชำระเงินบน Lightning นอกจากนี้เราจะพูดถึงทางเลือก 2 ทางสำหรับใบแจ้งหนี้แบบคลาสสิก: LNURL และ Keysend


![LNP201](assets/en/068.webp)


### โครงสร้างของใบแจ้งหนี้ Lightning


ตามที่อธิบายไว้ในบทเกี่ยวกับ HTLCs การชำระเงินแต่ละครั้งเริ่มต้นด้วยการสร้าง **ใบแจ้งหนี้** โดยผู้รับ ใบแจ้งหนี้นี้จะถูกส่งไปยังผู้จ่าย (ผ่านทาง QR code หรือโดยการคัดลอกและวาง) เพื่อเริ่มต้นการชำระเงิน ใบแจ้งหนี้ประกอบด้วยสองส่วนหลัก:



- ส่วนที่มนุษย์อ่านได้**: ส่วนนี้ประกอบด้วยเมตาดาทาที่มองเห็นได้ชัดเจนเพื่อเพิ่มประสบการณ์ของผู้ใช้
- The Payload**: ส่วนนี้ประกอบด้วยข้อมูลที่ตั้งใจให้เครื่องจักรประมวลผลการชำระเงิน


โครงสร้างทั่วไปของใบแจ้งหนี้เริ่มต้นด้วยตัวระบุ `ln` สำหรับ "Lightning" ตามด้วย `bc` สำหรับ Bitcoin แล้วตามด้วยจำนวนเงินของใบแจ้งหนี้ ตัวคั่น `1` แยกส่วนที่มนุษย์อ่านได้ออกจากส่วนข้อมูล (payload)


ลองดูใบแจ้งหนี้ต่อไปนี้เป็นตัวอย่าง:


```invoice
lnbc100u1p0x7x7dpp5l7r9y50wrzz0lwnsqgxdks50lxtwkl0mhd9lslr4rcgdtt2n6lssp5l3pkhdx0cmc9gfsqvw5xjhph84my2frzjqxqyz5vq9qsp5k4mkzv5jd8u5n89d2yc50x7ptkl0zprx0dfjh3km7g0x98g70hsqq7sqqqgqqyqqqqlgqqvnv2k5ehwnylq3rhpd9g2y0sq9ujyxsqqypjqqyqqqqqqqqqqqsqqqqq9qsq3vql5f6e45xztgj7y6xw6ghrcz3vmh8msrz8myvhsarxg42ce9yyn53lgnryx0m6qqld8fql
```


เราสามารถแบ่งมันออกเป็น 2 ส่วนได้แล้ว ส่วนแรกคือ ส่วนที่มนุษย์อ่านได้:


```invoice
lnbc100u
```


จากนั้นส่วนที่ตั้งใจไว้สำหรับเพย์โหลด:


```invoice

p0x7x7dpp5l7r9y50wrzz0lwnsqgxdks50lxtwkl0mhd9lslr4rcgdtt2n6lssp5l3pkhdx0cmc9gfsqvw5xjhph84my2frzjqxqyz5vq9qsp5k4mkzv5jd8u5n89d2yc50x7ptkl0zprx0dfjh3km7g0x98g70hsqq7sqqqgqqyqqqqlgqqvnv2k5ehwnylq3rhpd9g2y0sq9ujyxsqqypjqqyqqqqqqqqqqqsqqqqq9qsq3vql5f6e45xztgj7y6xw6ghrcz3vmh8msrz8myvhsarxg42ce9yyn53lgnryx0m6qqld8fql
```


สองส่วนถูกแยกออกด้วย `1` ตัวคั่นนี้ถูกเลือกแทนที่จะใช้ตัวอักษรพิเศษเพื่อให้ง่ายต่อการคัดลอกและวางใบแจ้งหนี้ทั้งหมดโดยการดับเบิลคลิก


ในส่วนแรก เราจะเห็นว่า:



- `ln` หมายถึงว่ามันเป็นธุรกรรม Lightning
- `bc` ระบุว่าเครือข่าย Lightning อยู่บนบล็อกเชน Bitcoin (และไม่ได้อยู่บน testnet หรือบน Litecoin)
- `100u` หมายถึงจำนวนเงินในใบแจ้งหนี้ ซึ่งแสดงในรูปแบบ **ไมโครบิตคอยน์** (`u` หมายถึง "ไมโคร") ซึ่งในที่นี้เท่ากับ 10,000 sats.


ในการกำหนดจำนวนเงินชำระ จะถูกแสดงในหน่วยย่อยของบิตคอยน์ นี่คือหน่วยที่ใช้:



- มิลลิบิตคอยน์ (แสดงด้วย `m`):** แทนค่าหนึ่งในพันของบิตคอยน์


$$
1 \, \text{mBTC} = 10^{-3} \, \text{BTC} = 10^5 \, \text{satoshis}
$$



- ไมโครบิตคอยน์ (แสดงด้วย `u`):** บางครั้งเรียกว่า "บิต" แทนค่าหนึ่งในล้านของบิตคอยน์


$$
1 \, \mu\text{BTC} = 10^{-6} \, \text{BTC} = 100 \, \text{satoshis}
$$



- นาโนบิตคอยน์ (แสดงด้วย `n`):** แทนหนึ่งในพันล้านของบิตคอยน์


$$
1 \, \text{nBTC} = 10^{-9} \, \text{BTC} = 0.1 \, \text{satoshis}
$$



- Picobitcoin (denoted `p`):** แทนหนึ่งในล้านล้านของบิตคอยน์

$$
1 \, \text{pBTC} = 10^{-12} \, \text{BTC} = 0.0001 \, \text{satoshis}
$$


### น้ำหนักบรรทุกของ Invoice


ข้อมูลในใบแจ้งหนี้ประกอบด้วยข้อมูลหลายชิ้นที่จำเป็นสำหรับการดำเนินการชำระเงิน:



- The timestamp:** ช่วงเวลาที่สร้างใบแจ้งหนี้ แสดงในรูปแบบ Unix Timestamp (จำนวนวินาทีที่ผ่านไปตั้งแต่วันที่ 1 มกราคม 1970)
- การแฮชความลับ**: ตามที่เราเห็นในส่วนของ HTLCs โหนดที่รับจะต้องให้แฮชของภาพก่อนแก่โหนดที่ส่ง ซึ่งใช้ใน HTLCs เพื่อรักษาความปลอดภัยของธุรกรรม เราเรียกมันว่า "_r_"
- ความลับการชำระเงิน**: ความลับอีกอย่างหนึ่งถูกสร้างขึ้นโดยผู้รับ แต่คราวนี้จะถูกส่งไปยังโหนดที่ส่ง มันถูกใช้ใน onion routing เพื่อป้องกันไม่ให้โหนดกลางคาดเดาว่าโหนดถัดไปเป็นผู้รับสุดท้ายหรือไม่ สิ่งนี้จึงรักษาความลับสำหรับผู้รับเกี่ยวกับโหนดกลางสุดท้ายในเส้นทาง
- คีย์สาธารณะของผู้รับ**: ระบุให้ผู้ชำระเงินทราบถึงตัวระบุของบุคคลที่จะได้รับการชำระเงิน
- ระยะเวลาหมดอายุ**: เวลาสูงสุดสำหรับการชำระใบแจ้งหนี้ (ค่าเริ่มต้นคือ 1 ชั่วโมง)
- คำแนะนำการกำหนดเส้นทาง**: ข้อมูลเพิ่มเติมที่ผู้รับมอบให้เพื่อช่วยให้ผู้ส่งปรับเส้นทางการชำระเงินให้เหมาะสมยิ่งขึ้น
- ลายเซ็น**: รับประกันความสมบูรณ์ของใบแจ้งหนี้โดยการยืนยันความถูกต้องของข้อมูลทั้งหมด


ใบแจ้งหนี้จะถูกเข้ารหัสในรูปแบบ **bech32** ซึ่งเป็นรูปแบบเดียวกับที่อยู่ Bitcoin SegWit (รูปแบบที่เริ่มต้นด้วย `bc1`)


### การถอน LNURL


ในการทำธุรกรรมแบบดั้งเดิม เช่น การซื้อของในร้านค้า ใบแจ้งหนี้จะถูกสร้างขึ้นสำหรับจำนวนเงินทั้งหมดที่ต้องชำระ เมื่อใบแจ้งหนี้ถูกนำเสนอ (ในรูปแบบของ QR code หรือสตริงของตัวอักษร) ลูกค้าสามารถสแกนและทำธุรกรรมให้เสร็จสิ้นได้ การชำระเงินจะเป็นไปตามกระบวนการแบบดั้งเดิมที่เราได้ศึกษาในส่วนก่อนหน้า อย่างไรก็ตาม กระบวนการนี้บางครั้งอาจยุ่งยากมากสำหรับประสบการณ์ของผู้ใช้ เนื่องจากต้องให้ผู้รับส่งข้อมูลไปยังผู้ส่งผ่านใบแจ้งหนี้


สำหรับสถานการณ์บางอย่าง เช่น การถอนบิตคอยน์จากบริการออนไลน์ กระบวนการแบบดั้งเดิมอาจยุ่งยากเกินไป ในกรณีเช่นนี้ โซลูชันการถอน **LNURL** จะทำให้กระบวนการนี้ง่ายขึ้นโดยการแสดงรหัส QR ที่ wallet ของผู้รับสแกนเพื่อสร้างใบแจ้งหนี้โดยอัตโนมัติ จากนั้นบริการจะชำระใบแจ้งหนี้ และผู้ใช้จะเห็นการถอนเงินทันที


![LNP201](assets/en/069.webp)


LNURL เป็นโปรโตคอลการสื่อสารที่ระบุชุดของฟังก์ชันที่ออกแบบมาเพื่อทำให้การโต้ตอบระหว่างโหนด Lightning และลูกค้า รวมถึงแอปพลิเคชันของบุคคลที่สามง่ายขึ้น การถอน LNURL ดังที่เราได้เห็นแล้ว จึงเป็นเพียงตัวอย่างหนึ่งในบรรดาฟังก์ชันอื่นๆ

โปรโตคอลนี้อิงตาม HTTP และช่วยให้สามารถสร้างลิงก์สำหรับการดำเนินการต่างๆ เช่น คำขอการชำระเงิน คำขอการถอนเงิน หรือฟังก์ชันอื่นๆ ที่ช่วยเพิ่มประสบการณ์ของผู้ใช้ แต่ละ LNURL เป็น URL ที่เข้ารหัสแบบ bech32 พร้อมคำนำหน้า LNURL ซึ่งเมื่อสแกนแล้วจะกระตุ้นให้เกิดการดำเนินการอัตโนมัติหลายอย่างบน Lightning wallet

ตัวอย่างเช่น ฟีเจอร์ LNURL-withdraw (LUD-03) ช่วยให้สามารถถอนเงินจากบริการได้โดยการสแกน QR code โดยไม่จำเป็นต้อง generate ใบแจ้งหนี้ด้วยตนเอง ในทำนองเดียวกัน LNURL-auth (LUD-04) ช่วยให้สามารถเข้าสู่ระบบบริการออนไลน์โดยใช้คีย์ส่วนตัวบน Lightning wallet แทนการใช้รหัสผ่าน


### การส่งการชำระเงินด้วย Lightning โดยไม่มี Invoice: Keysend


กรณีที่น่าสนใจอีกกรณีหนึ่งคือการโอนเงินโดยไม่ได้รับใบแจ้งหนี้ล่วงหน้า ซึ่งรู้จักกันในชื่อ "**Keysend**" โปรโตคอลนี้อนุญาตให้ส่งเงินโดยการเพิ่มภาพล่วงหน้าไปยังข้อมูลการชำระเงินที่เข้ารหัส ซึ่งสามารถเข้าถึงได้โดยผู้รับเท่านั้น ภาพล่วงหน้านี้ช่วยให้ผู้รับสามารถปลดล็อก HTLC ได้ ดังนั้นจึงสามารถรับเงินได้โดยไม่ต้องสร้างใบแจ้งหนี้ล่วงหน้า


เพื่อให้ง่ายขึ้น ในโปรโตคอลนี้ ผู้ส่งจะเป็นผู้สร้างความลับที่ใช้ใน HTLCs แทนที่จะเป็นผู้รับ ในทางปฏิบัติ สิ่งนี้ช่วยให้ผู้ส่งสามารถทำการชำระเงินได้โดยไม่ต้องมีการโต้ตอบกับผู้รับล่วงหน้า


![LNP201](assets/en/070.webp)


**คุณควรนำอะไรไปจากบทนี้?**



- **Lightning Invoice** เป็นคำขอชำระเงินที่ประกอบด้วยส่วนที่มนุษย์อ่านได้และส่วนข้อมูลของเครื่อง
- ใบแจ้งหนี้ถูกเข้ารหัสใน **bech32** โดยมีตัวคั่น `1` เพื่ออำนวยความสะดวกในการคัดลอกและส่วนข้อมูลที่มีข้อมูลทั้งหมดที่จำเป็นในการดำเนินการชำระเงิน
- กระบวนการชำระเงินอื่น ๆ มีอยู่บน Lightning โดยเฉพาะอย่างยิ่ง **LNURL-Withdraw** เพื่ออำนวยความสะดวกในการถอนเงิน และ **Keysend** สำหรับการโอนเงินโดยตรงโดยไม่ต้องมีใบแจ้งหนี้


ในบทต่อไปนี้ เราจะได้เห็นว่าผู้ดำเนินการโหนดสามารถจัดการสภาพคล่องในช่องทางของพวกเขาได้อย่างไร เพื่อให้มั่นใจว่าพวกเขาจะไม่ถูกบล็อกและสามารถส่งและรับการชำระเงินบน Lightning Network ได้เสมอ

ในบทนี้ เราจะสำรวจกลยุทธ์ในการจัดการสภาพคล่องอย่างมีประสิทธิภาพบน Lightning Network การจัดการสภาพคล่องจะแตกต่างกันไปขึ้นอยู่กับประเภทของผู้ใช้และบริบท เราจะตรวจสอบหลักการสำคัญและเทคนิคที่มีอยู่เพื่อให้เข้าใจลึกซึ้งยิ่งขึ้นเกี่ยวกับวิธีการเพิ่มประสิทธิภาพในการจัดการนี้


### ความต้องการสภาพคล่อง


มีโปรไฟล์ผู้ใช้หลักสามประเภทบน Lightning แต่ละประเภทมีความต้องการสภาพคล่องเฉพาะ:



- The Payer**: นี่คือผู้ที่ทำการชำระเงิน พวกเขาต้องการสภาพคล่องขาออกเพื่อโอนเงินไปยังผู้ใช้อื่น ตัวอย่างเช่น อาจเป็นผู้บริโภค
- ผู้ขาย (หรือผู้รับเงิน)**: นี่คือผู้ที่ได้รับการชำระเงิน พวกเขาต้องการสภาพคล่องขาเข้าเพื่อรับการชำระเงินไปยังโหนดของพวกเขา ตัวอย่างเช่น อาจเป็นธุรกิจหรือร้านค้าออนไลน์
- เราเตอร์**: โหนดตัวกลาง ซึ่งมักมีความเชี่ยวชาญในการกำหนดเส้นทางการชำระเงิน ที่ต้องปรับปรุงสภาพคล่องในแต่ละช่องทางเพื่อกำหนดเส้นทางการชำระเงินให้ได้มากที่สุดและรับค่าธรรมเนียม


โปรไฟล์เหล่านี้ไม่ได้ถูกกำหนดตายตัว ผู้ใช้สามารถสลับระหว่างผู้จ่ายและผู้รับได้ขึ้นอยู่กับการทำธุรกรรม ตัวอย่างเช่น Bob อาจได้รับเงินเดือนผ่าน Lightning จากนายจ้างของเขา ทำให้เขาอยู่ในตำแหน่งของ "ผู้ขาย" ที่ต้องการสภาพคล่องขาเข้า จากนั้นหากเขาต้องการใช้เงินเดือนเพื่อซื้ออาหาร เขาจะกลายเป็น "ผู้จ่าย" และต้องมีสภาพคล่องขาออก


เพื่อความเข้าใจที่ดีขึ้น ลองพิจารณาเครือข่ายง่าย ๆ ที่ประกอบด้วยสามโหนด: ผู้ซื้อ (Alice), เราเตอร์ (Suzie), และผู้ขาย (Bob)


![LNP201](assets/en/071.webp)


ลองจินตนาการว่าผู้ซื้อต้องการส่ง sats จำนวน 30,000 หน่วยไปยังผู้ขาย และการชำระเงินจะต้องผ่านโหนดของเราเตอร์ แต่ละฝ่ายจะต้องมีสภาพคล่องขั้นต่ำในทิศทางของการชำระเงิน:



- ผู้ชำระเงินต้องมีอย่างน้อย 30,000 ซาโตชิในฝั่งของพวกเขาในช่องทางกับเราเตอร์
- ผู้ขายต้องมีช่องทางที่มี 30,000 ซาโตชิอยู่ฝั่งตรงข้ามเพื่อที่จะสามารถรับได้
- เราเตอร์ต้องมี 30,000 ซาโตชิในฝั่งของผู้จ่ายในช่องของพวกเขา และยังต้องมี 30,000 ซาโตชิในฝั่งของพวกเขาในช่องกับผู้ขาย เพื่อที่จะสามารถส่งผ่านการชำระเงินได้


![LNP201](assets/en/072.webp)


### กลยุทธ์การจัดการสภาพคล่อง


ผู้ชำระเงินต้องมั่นใจว่ามีสภาพคล่องเพียงพอในฝั่งของตนในช่องทางเพื่อรับประกันสภาพคล่องขาออก ซึ่งพิสูจน์ได้ว่าเป็นเรื่องค่อนข้างง่าย เนื่องจากเพียงพอที่จะเปิดช่องทาง Lightning ใหม่เพื่อให้มีสภาพคล่องนี้ อันที่จริง เงินทุนเริ่มต้นที่ถูกล็อกใน multisig on-chain นั้นอยู่ในฝั่งของผู้ชำระเงินในช่องทาง Lightning ตั้งแต่เริ่มต้น ความสามารถในการชำระเงินจึงได้รับการรับรองตราบใดที่มีการเปิดช่องทางด้วยเงินทุนเพียงพอ เมื่อสภาพคล่องขาออกหมดลง ก็เพียงพอที่จะเปิดช่องทางใหม่

ในทางกลับกัน สำหรับผู้ขาย งานนี้ซับซ้อนมากขึ้น ในการรับชำระเงิน พวกเขาต้องมีสภาพคล่องในฝั่งตรงข้ามของช่องทางของพวกเขา ดังนั้น การเปิดช่องทางไม่เพียงพอ พวกเขายังต้องทำการชำระเงินผ่านช่องทางนี้เพื่อโอนสภาพคล่องไปยังฝั่งตรงข้ามก่อนที่พวกเขาจะสามารถรับชำระเงินได้เอง สำหรับโปรไฟล์ผู้ใช้ Lightning บางประเภท เช่น ผู้ค้า มีความไม่สมดุลอย่างชัดเจนระหว่างสิ่งที่โหนดของพวกเขาส่งและสิ่งที่ได้รับ เนื่องจากเป้าหมายหลักของธุรกิจคือการเก็บเงินให้มากกว่าที่ใช้จ่ายเพื่อ generate กำไร โชคดีที่สำหรับผู้ใช้เหล่านี้ที่มีความต้องการสภาพคล่องขาเข้าที่เฉพาะเจาะจง มีหลายวิธีแก้ไข:



- ดึงดูดช่องทาง**: ผู้ค้าได้รับประโยชน์จากข้อได้เปรียบเนื่องจากปริมาณการชำระเงินที่คาดว่าจะเข้ามาที่โหนดของพวกเขา เมื่อคำนึงถึงสิ่งนี้ พวกเขาสามารถพยายามดึงดูดโหนดการกำหนดเส้นทางที่กำลังมองหารายได้จากค่าธรรมเนียมการทำธุรกรรมและอาจเปิดช่องทางไปยังพวกเขา ซึ่งจะทำให้สามารถกำหนดเส้นทางการชำระเงินและเก็บค่าธรรมเนียมที่เกี่ยวข้องได้



- การเคลื่อนไหวของสภาพคล่อง**: ผู้ขายยังสามารถเปิดช่องทางและโอนเงินบางส่วนไปยังฝั่งตรงข้ามโดยการทำการชำระเงินปลอมไปยังโหนดอื่น ซึ่งจะคืนเงินในวิธีอื่น เราจะสำรวจการดำเนินการนี้ในส่วนถัดไป



- การเปิดแบบสามเหลี่ยม**: มีแพลตฟอร์มสำหรับโหนดที่ต้องการเปิดช่องทางร่วมกัน ช่วยให้แต่ละโหนดได้รับประโยชน์จากสภาพคล่องขาเข้าและขาออกทันที ตัวอย่างเช่น [LightningNetwork+](https://lightningnetwork.plus/) ให้บริการนี้ หาก Alice, Bob และ Suzie ต้องการเปิดช่องทางด้วย 100,000 sats พวกเขาสามารถตกลงบนแพลตฟอร์มนี้ให้ Alice เปิดช่องทางไปยัง Bob, Bob ไปยัง Suzie และ Suzie ไปยัง Alice ด้วยวิธีนี้ แต่ละคนจะมี 100,000 sats ของสภาพคล่องทั้งขาออกและขาเข้า ในขณะที่มีเพียง 100,000 sats ที่ถูกล็อคไว้เท่านั้น


![LNP201](assets/en/073.webp)



- ช่องทางการซื้อ**: บริการสำหรับการเช่าช่องทาง Lightning ก็มีอยู่เพื่อให้ได้มาซึ่งสภาพคล่องขาเข้า เช่น [Bitrefill Thor](https://www.bitrefill.com/thor-lightning-network-channels/) หรือ [Lightning Labs Pool](https://lightning.engineering/pool/). ตัวอย่างเช่น Alice สามารถซื้อช่องทางที่มีหนึ่งล้านซาโตชิเพื่อเป็นทุนให้กับโหนดของเธอ ทำให้เธอสามารถรับการชำระเงินได้


![LNP201](assets/en/074.webp)


สุดท้าย สำหรับเราเตอร์ ซึ่งมีเป้าหมายเพื่อเพิ่มจำนวนการชำระเงินที่ดำเนินการและค่าธรรมเนียมที่เก็บรวบรวม พวกเขาต้อง:



- เปิดช่องทางที่ได้รับการสนับสนุนอย่างดีไปยังโหนดเชิงกลยุทธ์
- ปรับการกระจายเงินทุนในช่องทางต่าง ๆ อย่างสม่ำเสมอตามความต้องการของเครือข่าย


### บริการลูปเอาท์


บริการ [Loop Out](https://lightning.engineering/loop/) ที่นำเสนอโดย Lightning Labs ช่วยให้สามารถโอนสภาพคล่องไปยังฝั่งตรงข้ามของช่องทางในขณะที่เรียกคืนเงินบนบล็อกเชน Bitcoin ตัวอย่างเช่น Alice ส่ง 1 ล้านซาโตชิผ่าน Lightning ไปยังโหนดลูป ซึ่งจะคืนเงินเหล่านั้นให้เธอในรูปแบบบิตคอยน์ on-chain สิ่งนี้ทำให้ช่องทางของเธอสมดุลด้วย 1 ล้านซาโตชิในแต่ละด้าน เพิ่มประสิทธิภาพความสามารถในการรับชำระเงินของเธอ


![LNP201](assets/en/075.webp)


ดังนั้น บริการนี้จึงช่วยให้มีสภาพคล่องที่เข้ามาในขณะที่เรียกคืนบิตคอยน์ on-chain ซึ่งช่วยจำกัดการตรึงเงินสดที่จำเป็นในการรับชำระเงินด้วย Lightning


**คุณควรนำอะไรไปจากบทนี้?**



- ในการส่งการชำระเงินบน Lightning คุณต้องมีสภาพคล่องเพียงพอในช่องของคุณ เพื่อเพิ่มความสามารถในการส่งนี้ เพียงแค่เปิดช่องใหม่
- ในการรับชำระเงิน คุณจำเป็นต้องมีสภาพคล่องในฝั่งตรงข้ามในช่องทางของคุณ การเพิ่มความสามารถในการรับนี้มีความซับซ้อนมากขึ้น เนื่องจากต้องการให้ผู้อื่นเปิดช่องทางมาหาคุณ หรือทำการชำระเงิน (สมมติหรือจริง) เพื่อย้ายสภาพคล่องไปยังฝั่งตรงข้าม
- การรักษาสภาพคล่องในที่ที่ต้องการอาจเป็นเรื่องที่ท้าทายมากขึ้น ขึ้นอยู่กับการใช้ช่องทาง นั่นคือเหตุผลที่มีเครื่องมือและบริการเพื่อช่วยปรับสมดุลของช่องทางตามที่ต้องการ


ในบทถัดไป ฉันขอเสนอที่จะทบทวนแนวคิดที่สำคัญที่สุดของการฝึกอบรมนี้

## สรุปหลักสูตร



<chapterId>a65a571c-561b-5e1c-87bf-494644653c22</chapterId>


:::video id=5f4f4344-ef27-4765-8f09-8262e6833bde:::

ในบทสุดท้ายนี้ที่เป็นการสิ้นสุดการฝึกอบรม LNP201 ฉันขอเสนอให้เราทบทวนแนวคิดสำคัญที่เราได้ครอบคลุมร่วมกัน


เป้าหมายของการฝึกอบรมนี้คือการให้ความเข้าใจที่ครอบคลุมและเชิงเทคนิคเกี่ยวกับ Lightning Network เราได้ค้นพบว่า Lightning Network พึ่งพาบล็อกเชน Bitcoin ในการดำเนินการธุรกรรม off-chain ในขณะที่ยังคงรักษาลักษณะพื้นฐานของ Bitcoin ไว้ โดยเฉพาะอย่างยิ่งการไม่จำเป็นต้องไว้วางใจโหนดอื่น ๆ


### ช่องทางการชำระเงิน


ในบทแรก ๆ เราได้สำรวจว่าคู่สัญญาสองฝ่ายสามารถทำธุรกรรมนอกบล็อกเชน Bitcoin ได้อย่างไรโดยการเปิดช่องทางการชำระเงิน นี่คือขั้นตอนที่ครอบคลุม:



- การเปิดช่องทาง**: การสร้างช่องทางทำผ่านธุรกรรม Bitcoin ที่ล็อคเงินทุนในที่อยู่มัลติซิกเนเจอร์ 2/2 การฝากเงินนี้แสดงถึงช่องทาง Lightning บนบล็อกเชน


![LNP201](assets/en/076.webp) 2. **Transactions in the Channel**: In this channel, it is possible to carry out numerous transactions without publishing them on the blockchain. Each Lightning transaction creates a new state of the channel reflected in a commitment transaction.

![LNP201](assets/en/077.webp)



- การรักษาความปลอดภัยและการปิด**: ผู้เข้าร่วมให้คำมั่นสัญญาต่อสถานะใหม่ของช่องทางโดยการแลกเปลี่ยนกุญแจเพิกถอนเพื่อรักษาความปลอดภัยของเงินทุนและป้องกันการโกง ทั้งสองฝ่ายสามารถปิดช่องทางร่วมกันได้โดยการทำธุรกรรมใหม่บนบล็อกเชน Bitcoin หรือในกรณีสุดท้ายผ่านการปิดแบบบังคับ ตัวเลือกหลังนี้ แม้จะมีประสิทธิภาพน้อยกว่าเนื่องจากความยาวและบางครั้งการประเมินค่าธรรมเนียมที่ไม่ดี แต่ยังคงอนุญาตให้กู้คืนเงินทุนได้ ในกรณีที่มีการโกง เหยื่อสามารถลงโทษผู้โกงโดยการกู้คืนเงินทั้งหมดจากช่องทางบนบล็อกเชน


![LNP201](assets/en/078.webp)


### เครือข่ายของช่องทาง


หลังจากศึกษาช่องทางที่แยกออกมาแล้ว เราได้ขยายการวิเคราะห์ของเราไปยังเครือข่ายของช่องทาง:



- การกำหนดเส้นทาง**: เมื่อสองฝ่ายไม่ได้เชื่อมต่อกันโดยตรงผ่านช่องทาง เครือข่ายจะอนุญาตให้มีการกำหนดเส้นทางผ่านโหนดตัวกลาง จากนั้นการชำระเงินจะผ่านจากโหนดหนึ่งไปยังอีกโหนดหนึ่ง


![LNP201](assets/en/079.webp)



- HTLCs**: การชำระเงินที่ผ่านโหนดตัวกลางจะได้รับการรักษาความปลอดภัยโดย "_Hash Time-Locked Contracts_" (HTLC) ซึ่งอนุญาตให้เงินถูกล็อกจนกว่าการชำระเงินจะเสร็จสมบูรณ์จากต้นทางถึงปลายทาง


![LNP201](assets/en/080.webp)



- การส่งข้อมูลแบบ Onion Routing**: เพื่อให้มั่นใจในความลับของการชำระเงิน การส่งข้อมูลแบบ onion routing จะปกปิดปลายทางสุดท้ายจากโหนดกลาง โหนดที่ส่งจึงต้องคำนวณเส้นทางทั้งหมด แต่เนื่องจากขาดข้อมูลที่ครบถ้วนเกี่ยวกับสภาพคล่องของช่องทาง จึงต้องดำเนินการผ่านการทดลองต่อเนื่องเพื่อส่งผ่านการชำระเงิน


![LNP201](assets/en/081.webp)


### การจัดการสภาพคล่อง


เราได้สังเกตว่าการจัดการสภาพคล่องเป็นความท้าทายบน Lightning ซึ่งจำเป็นต้องมีการไหลเวียนของการชำระเงินอย่างราบรื่น การส่งการชำระเงินค่อนข้างง่าย: เพียงแค่ต้องเปิดช่องทางเท่านั้น อย่างไรก็ตาม การรับการชำระเงินจำเป็นต้องมีสภาพคล่องในฝั่งตรงข้ามของช่องทางของตน นี่คือกลยุทธ์บางส่วนที่ได้มีการพูดคุยกัน:



- ดึงดูดช่องทาง**: โดยการกระตุ้นให้โหนดอื่นเปิดช่องทางมาหาตนเอง ผู้ใช้จะได้รับสภาพคล่องขาเข้า



- การเคลื่อนย้ายสภาพคล่อง**: โดยการส่งการชำระเงินไปยังช่องทางอื่น สภาพคล่องจะเคลื่อนไปยังฝั่งตรงข้าม


![LNP201](assets/en/082.webp)



- การใช้บริการเช่น Loop และ Pool**: บริการเหล่านี้ช่วยให้สามารถปรับสมดุลหรือซื้อช่องทางที่มีสภาพคล่องในฝั่งตรงข้ามได้

![LNP201](assets/en/083.webp)


- การเปิดร่วมกัน**: ยังมีแพลตฟอร์มที่สามารถเชื่อมต่อเพื่อทำการเปิดแบบสามเหลี่ยมและมีสภาพคล่องที่เข้ามา


![LNP201](assets/en/084.webp)
