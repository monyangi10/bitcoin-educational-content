# สร้างบน Liquid และ Elements


ค้นพบคุณสมบัติและความสามารถขั้นสูงของ Liquid และ Elements และเรียนรู้วิธีการใช้เครื่องมือเหล่านี้อย่างมีประสิทธิภาพเพื่อเพิ่มประสิทธิภาพให้กับโครงการพัฒนาของคุณ การฝึกอบรมนี้ให้พื้นฐานทางทฤษฎีและปฏิบัติที่ครอบคลุม ช่วยให้คุณเชี่ยวชาญคุณสมบัติต่างๆ เช่น Confidential Transactions, Issued Assets และ Federated Block Signing


Liquid ซึ่งพัฒนาขึ้นบนพื้นฐานของกรอบงาน Elements ได้รับการออกแบบมาเพื่อปรับปรุงความเป็นส่วนตัว ความสามารถในการขยาย และฟังก์ชันการทำงานสำหรับโซลูชันทางการเงินและเทคนิค ในหลักสูตรนี้ คุณจะได้รับประสบการณ์จริงในการออกและจัดการสินทรัพย์ Federated 2-Way Peg และการใช้เครื่องมือเช่น elementsd และ elements-cli ซึ่งจะช่วยให้คุณสร้างโซลูชันที่เป็นนวัตกรรมใหม่ที่ปรับให้เหมาะกับความต้องการของคุณ


หลักสูตรนี้ได้รับการออกแบบมาเพื่อพัฒนาทักษะของนักพัฒนาทุกระดับประสบการณ์ ผู้เริ่มต้นและผู้ใช้ระดับกลางจะพบกับคำอธิบายที่เข้าใจง่ายและตัวอย่างที่ใช้งานได้จริง ในขณะที่ผู้ใช้ระดับสูงสามารถเจาะลึกลงไปในรายละเอียดทางเทคนิคและคุณสมบัติที่ไม่ค่อยมีคนรู้จักของ Liquid และ Elements


เข้าร่วมกับเราเพื่อยกระดับทักษะของคุณ ปลดล็อกศักยภาพเต็มรูปแบบของ Liquid และ Elements และสร้างเครื่องมือที่มีผลกระทบสำหรับอนาคตของนวัตกรรม Liquid

Elements เป็นแพลตฟอร์ม[บล็อกเชน](https://planb.academy/resources/glossary/blockchain)แบบโอเพนซอร์สที่สามารถใช้งานไซด์เชนได้ ซึ่งให้การเข้าถึงฟีเจอร์ที่ทรงพลังซึ่งพัฒนาโดยสมาชิกในชุมชน เช่น Confidential Transactions และ Issued Assets


Elements เป็นโปรโตคอลที่ช่วยให้เกิด[ฉันทามติ](https://planb.academy/resources/glossary/consensus)ในการสร้างประวัติการทำ[ธุรกรรม](https://planb.academy/resources/glossary/transaction-tx)และกฎเกณฑ์ที่ควบคุมการโอนและการสร้างสินทรัพย์ที่เก็บไว้ในบัญชีแยกประเภทบล็อกเชนแบบกระจาย


สามารถค้นหาข้อมูลเพิ่มเติมเกี่ยวกับ Elements ได้อย่างง่ายดายบนเว็บไซต์โครงการ Elements (https://elementsproject.org/), [บล็อก](https://planb.academy/resources/glossary/block)อย่างเป็นทางการของ Liquid (https://blog.liquid.net/), และพอร์ทัลนักพัฒนา (https://liquid.net/devs)


### Elements


เปิดตัวในปี 2015 Elements ช่วยลดต้นทุนการพัฒนาและวิจัยภายในและใช้ประโยชน์จากเทคโนโลยีบล็อกเชนล่าสุด เปิดโอกาสให้มีการใช้งานใหม่ๆ มากมายสำหรับการนำไปใช้ บล็อกเชนที่ใช้ Elements สามารถทำงานเป็น Blockchain แบบสแตนด์อโลนหรือเชื่อมโยงกับอีกอันหนึ่งและทำงานเป็น Sidechain การรัน Elements เป็น Sidechain ช่วยให้สามารถโอนสินทรัพย์ระหว่างบล็อกเชนต่างๆ ได้อย่างตรวจสอบได้


สร้างขึ้นบนฐานรหัสของ Bitcoin และขยายออกไป มันช่วยให้นักพัฒนาที่คุ้นเคยกับ bitcoind API สามารถสร้างบล็อกเชนที่ใช้งานได้และทดสอบโครงการต้นแบบได้อย่างรวดเร็วและคุ้มค่า การสร้างขึ้นบนฐานรหัสของ Bitcoin ยังช่วยให้ Elements ทำหน้าที่เป็นแพลตฟอร์มทดสอบสำหรับการเปลี่ยนแปลงในโปรโตคอลของ Bitcoin เองด้วย


คุณสมบัติหลักบางประการของ Elements มีดังต่อไปนี้


#### Confidential Transactions


ตามค่าเริ่มต้น ที่อยู่ทั้งหมดใน Elements จะเป็น blinded โดยใช้ Confidential Transactions การปิดบังเป็นกระบวนการที่จำนวนและประเภทของสินทรัพย์ที่ถูกโอนจะถูกซ่อนด้วยการเข้ารหัสจากทุกคน ยกเว้นผู้เข้าร่วมและผู้ที่พวกเขาเลือกที่จะเปิดเผยกุญแจการปิดบังให้


#### Issued Assets


Issued Assets บน Elements ช่วยให้สามารถออกและโอนประเภทสินทรัพย์หลายประเภทระหว่างผู้เข้าร่วมเครือข่าย สินทรัพย์ที่ออกแล้วยังได้รับประโยชน์จาก Confidential Transactions และสามารถออกใหม่หรือทำลายโดยใครก็ตามที่ถือ reissuance token ที่เกี่ยวข้อง


#### Federated 2-Way Peg


Elements เป็นแพลตฟอร์มบล็อกเชนที่มีวัตถุประสงค์ทั่วไปซึ่งสามารถ “ผูก” กับบล็อกเชนที่มีอยู่แล้ว (เช่น Bitcoin) เพื่อให้สามารถโอนสินทรัพย์สองทางจากเชนหนึ่งไปยังอีกเชนหนึ่งได้ การใช้งาน Elements เป็นไซด์เชนช่วยให้คุณสามารถทำงานรอบคุณสมบัติบางอย่างที่มีอยู่ในเชนหลัก ในขณะที่ยังคงรักษาระดับความปลอดภัยที่ดีที่สินทรัพย์ที่ถูกป้องกันบนเชนหลักมีให้


#### บล็อกที่ลงนามแล้ว


Elements ใช้ Strong Federation ของผู้ลงนามที่เรียกว่า Block Signers ซึ่งลงนามและสร้างบล็อกในลักษณะที่เชื่อถือได้และทันเวลา สิ่งนี้ช่วยลดความล่าช้าของธุรกรรมในกระบวนการ PoW mining ซึ่งมีความแปรปรวนของเวลาบล็อกสูงเนื่องจากการกระจายแบบสุ่มของ Poisson กระบวนการ Federated Block Signing ช่วยให้การสร้างบล็อกมีความน่าเชื่อถือโดยไม่ต้องแนะนำความจำเป็นในการไว้วางใจบุคคลที่สามหรือ `algorithm` คำนวณตาม mining


Elements เพิ่มคุณสมบัติทั้งหมดเหล่านี้บนฐานรหัสหลักของ Bitcoin ขยายความสามารถของโปรโตคอล mainchain และเปิดใช้งานกรณีการใช้งานทางธุรกิจใหม่ ๆ เมื่อปรับใช้เป็นไซด์เชนหรือเป็นโซลูชันบล็อกเชนอิสระ

## วิธีการทำงานของ Elements


<chapterId>05d88877-58b0-455b-9ae6-a72d19070525</chapterId>


:::video id=7c8c7981-11e5-47a2-a257-ef998f4892f5:::

Elements ให้โซลูชันทางเทคนิคสำหรับปัญหาที่ผู้ใช้บล็อกเชนพบเจอในแต่ละวัน: ความล่าช้าของการทำธุรกรรม, การขาดความเป็นส่วนตัว, และความเสี่ยงต่อ[การแลกเปลี่ยนได้](https://planb.academy/resources/glossary/fungibility)


Elements แก้ไขปัญหาเหล่านี้ผ่านการใช้ Federated Block Signing และ Confidential Transactions


ไม่เหมือนกับเครือข่าย Bitcoin กระบวนการของ block signing ภายใน Elements ไม่ได้พึ่งพา Dynamic Membership Multiparty Signatures (DMMS) และ Proof of Work (PoW) แต่ Elements ใช้ Strong Federation ของผู้ลงนามที่เรียกว่า Block Signers ซึ่งสามารถลงนามและสร้างบล็อกได้อย่างน่าเชื่อถือและทันเวลา สิ่งนี้ช่วยลดความล่าช้าของการทำธุรกรรมในกระบวนการ PoW mining ซึ่งมีความแปรปรวนของเวลาบล็อกสูงเนื่องจากการกระจายแบบสุ่มของ Poisson กระบวนการ Federated Block Signing ช่วยให้สามารถสร้างบล็อกที่น่าเชื่อถือได้โดยไม่ต้องพึ่งพาความไว้วางใจจากบุคคลที่สาม


Elements สามารถทำงานเป็นไซด์เชนของบล็อกเชนอื่น เช่น Bitcoin หรือเป็นบล็อกเชนอิสระที่ไม่มีการพึ่งพาเครือข่ายอื่น ๆ ได้


เมื่อใช้เป็นไซด์เชน Strong Federation ยังมีสมาชิกที่ช่วยให้การโอนสินทรัพย์ระหว่างเชนหลักและไซด์เชน Elements เป็นไปอย่างปลอดภัยและควบคุมได้ การโอนสินทรัพย์ที่ควบคุมนี้เรียกว่า Federated 2-Way Peg และสมาชิกที่ทำหน้าที่โอนสินทรัพย์เรียกว่า Watchmen.


กระบวนการที่เกี่ยวข้องในการดำเนินงานเครือข่าย Elements และบทบาทของผู้เข้าร่วมภายในเครือข่ายมีความสำคัญอย่างยิ่งต่อการทำความเข้าใจว่า Elements ทำงานอย่างไร


ไม่ว่าจะถูกนำมาใช้เป็นไซด์เชนหรือบล็อกเชนอิสระ Elements ใช้สหพันธ์ที่แข็งแกร่งของผู้ลงนามบล็อกเพื่อผลิตบล็อก


### สหพันธ์ที่แข็งแกร่ง


Elements ใช้โมเดลฉันทามติที่เสนอครั้งแรกโดย Blockstream ซึ่งรู้จักกันในชื่อ Strong Federations Strong Federation ไม่จำเป็นต้องใช้ Proof of Work (PoW) และแทนที่จะพึ่งพาการกระทำร่วมกันของกลุ่มผู้เข้าร่วมที่ไม่ไว้วางใจกัน เรียกว่า Functionaries


บทบาทที่ Functionary สามารถทำได้ภายใน Strong Federation ได้แก่: Block Signers และ Watchmen Block Signers จำเป็นต้องมีหากคุณใช้งาน Elements ในโหมด sidechain หรือ standalone blockchain ในขณะที่ Watchmen จำเป็นต้องมีเฉพาะในโหมด sidechain เท่านั้น


การกระทำที่สมาชิกของ Strong Federation สามารถทำได้ถูกแบ่งออกเป็นสองบทบาทที่แตกต่างกันเพื่อเพิ่มความปลอดภัยและจำกัดความเสียหายที่ผู้โจมตีสามารถก่อได้


เมื่อรวมกันแล้ว บทบาทของผู้เข้าร่วมเหล่านี้ทำให้ Elements สามารถสร้างบล็อกได้อย่างรวดเร็ว (การยืนยันธุรกรรมที่เร็วขึ้นและสุดท้าย) และสินทรัพย์ที่สามารถโอนได้อย่างมั่นใจ (สินทรัพย์ที่ตรึงซึ่งสามารถเชื่อมโยงโดยตรงกับบล็อกเชนอื่น)


คุณสามารถอ่าน[เอกสารไวท์เปเปอร์](https://planb.academy/resources/glossary/white-paper)ของ Strong Federations ได้ที่นี่: https://blockstream.com/strong-federations.pdf


### บล็อกผู้ลงนาม


บล็อกเชน เช่น ของ Bitcoin จะถูกขยายเมื่อใครก็ตามที่เป็นส่วนหนึ่งของกลุ่มผู้ลงนามบล็อกแบบไดนามิกขยายเชนโดยการแสดงให้เห็นว่า proof of work ถูกใช้ไปแล้ว ลักษณะแบบไดนามิกของชุดนี้ทำให้เกิดปัญหาความหน่วงที่มีอยู่ในระบบดังกล่าว


โดยการใช้ผู้ลงนามที่กำหนดไว้ล่วงหน้า โมเดลแบบ Federated จะเปลี่ยนชุดที่เปลี่ยนแปลงได้เป็นชุดที่รู้จัก ซึ่งเป็นรูปแบบการลงนามหลายลายเซ็น การลดจำนวนผู้เข้าร่วมที่จำเป็นในการขยายบล็อกเชนช่วยเพิ่มความเร็วและความสามารถในการขยายของระบบ ในขณะที่การตรวจสอบโดยทุกฝ่ายช่วยให้มั่นใจในความถูกต้องของประวัติการทำธุรกรรม


Federated block signing ประกอบด้วยหลายขั้นตอน:



- ขั้นตอนที่ 1 - ผู้ลงนามบล็อกเสนอผู้สมัครบล็อกในลักษณะวนรอบให้กับผู้ลงนามบล็อกคนอื่น ๆ ที่เข้าร่วมทั้งหมด



- ขั้นตอนที่ 2 - ผู้ลงนามบล็อกแต่ละคนแสดงเจตนาของตนโดยการให้คำมั่นล่วงหน้าที่จะลงนามในบล็อกผู้สมัครที่กำหนด



- ขั้นตอนที่ 3 - หากเกณฑ์ที่กำหนดสำหรับการผูกมัดล่วงหน้าถูกตอบสนอง ผู้ลงนามบล็อกแต่ละคนจะลงนามในบล็อก



- ขั้นตอนที่ 4 - หากเกณฑ์ลายเซ็น (ซึ่งอาจแตกต่างจากขั้นตอนที่ 3) ได้รับการตอบสนอง บล็อกจะได้รับการยอมรับและส่งไปยังเครือข่าย Strong Federation ได้บรรลุฉันทามติเกี่ยวกับบล็อกธุรกรรมล่าสุดแล้ว



- ขั้นตอนที่ 5 - บล็อกถัดไปจะถูกเสนอโดยผู้ลงนามบล็อกคนถัดไปในลำดับรอบ และกระบวนการจะทำซ้ำอีกครั้ง


เนื่องจากการสร้างบล็อกของ Strong Federation ไม่ได้เป็นแบบสุ่มตามความน่าจะเป็นและอิงตามชุดของผู้ลงนามที่กำหนดไว้ล่วงหน้า จึงไม่เคยต้องเผชิญกับการจัดระเบียบบล็อกหลายบล็อกใหม่ ซึ่งช่วยลดเวลารอที่เกี่ยวข้องกับการยืนยันธุรกรรมได้อย่างมาก นอกจากนี้ยังขจัดแรงจูงใจในการขุดเพื่อผลกำไร (เช่น รางวัลบล็อก) และแทนที่ด้วยแรงจูงใจในการมีส่วนร่วมอย่างมีประสิทธิภาพในเครือข่ายที่ผู้เข้าร่วมทุกคนมีเป้าหมายร่วมกัน นั่นคือ การทำให้แน่ใจว่าเครือข่ายยังคงทำงานในลักษณะที่เป็นประโยชน์ต่อทุกคน ซึ่งทำได้โดยไม่ต้องแนะนำจุดล้มเหลวเพียงจุดเดียวหรือไม่ต้องการความไว้วางใจที่สูงขึ้น


### Elements เป็น Sidechain - Watchmen และ Federated 2-Way Peg


เมื่อทำงานเป็นไซด์เชน สมาชิกบางคนของ Strong Federation มีบทบาทเพิ่มเติมที่ต้องทำ นั่นคือ Watchmen Watchmen มีหน้าที่ในการโอนสินทรัพย์เข้าและออกจากไซด์เชน Elements ซึ่งเป็นกระบวนการที่รู้จักกันในชื่อ `Peg-In` และ `Peg-Out`


สำหรับไซด์เชนที่จะดำเนินการในลักษณะที่น่าเชื่อถือ จะต้องทำให้ผู้เข้าร่วมสามารถตรวจสอบได้ว่าการจัดหาสินทรัพย์นั้นถูกควบคุมและสามารถตรวจสอบได้ ไซด์เชน Elements ใช้ 2-Way Federated Peg เพื่อให้สามารถโอนสินทรัพย์เข้าและออกจากบล็อกเชน Elements ได้อย่างสองทาง ซึ่งเป็นไปตามข้อกำหนดของการออกสินทรัพย์ที่พิสูจน์ได้และการโอนระหว่างเชน


คุณสมบัติ Federated 2-way Peg ช่วยให้อินเตอร์ออปเปอร์เรเบิลกับบล็อกเชนอื่น ๆ และแสดงถึงสินทรัพย์พื้นเมืองของบล็อกเชนอื่น ๆ ได้ โดยการผูกบล็อกเชนของคุณกับบล็อกเชนอื่น คุณสามารถขยายความสามารถของ mainchain และเอาชนะข้อจำกัดที่มีอยู่บางประการได้


ในระดับสูง การโอนเข้าสู่ไซด์เชนเกิดขึ้นเมื่อมีคนส่งสินทรัพย์ mainchain ไปยังที่อยู่ที่ควบคุมโดย Watchmen [wallet](https://planb.academy/resources/glossary/wallet) ที่มีหลายลายเซ็น ซึ่งจะทำให้สินทรัพย์บน mainchain ถูกแช่แข็ง Watchmen จะตรวจสอบความถูกต้องของธุรกรรมและปล่อยสินทรัพย์ในจำนวนเท่ากันภายในไซด์เชน สินทรัพย์ที่ถูกปล่อยจะถูกส่งไปยังไซด์เชน wallet ที่สามารถพิสูจน์การอ้างสิทธิ์ต่อสินทรัพย์ mainchain ดั้งเดิมได้ กระบวนการนี้ทำให้สินทรัพย์ย้ายจากเชนหลักไปยังไซด์เชนได้อย่างมีประสิทธิภาพ


ในการโอนสินทรัพย์กลับไปยังเชนหลัก ผู้ใช้จะเริ่มต้นธุรกรรมพิเศษที่เรียกว่า peg-out บนเชนด้านข้าง ธุรกรรมนี้จะถูกตรวจสอบโดย Watchmen ซึ่งจะลงนามในธุรกรรมที่ใช้จาก multi-signature wallet ที่พวกเขาควบคุมบนเชนหลัก ต้องมีจำนวนผู้เข้าร่วมในสหพันธ์ที่ถึงเกณฑ์ในการลงนามก่อนที่ธุรกรรม mainchain จะมีผล เมื่อ Watchmen ส่งสินทรัพย์กลับไปยังเชนหลัก พวกเขายังทำลายจำนวนที่สอดคล้องกันบนเชนด้านข้าง ซึ่งเป็นการโอนสินทรัพย์ระหว่างบล็อกเชนอย่างมีประสิทธิภาพ


## การตั้งค่าและการใช้งาน Elements


<chapterId>cc806e5a-81ab-457b-9531-9f863120a019</chapterId>

เนื่องจาก Elements มีพื้นฐานมาจากฐานรหัสของ Bitcoin ส่วนประกอบที่ทำให้เครือข่ายทำงานได้จึงมีความคล้ายคลึงกันมาก


ซอฟต์แวร์[โหนด](https://planb.academy/resources/glossary/node) Elements เองถูกเรียกว่า `elementsd` และทำงานเป็น daemon บนเครื่องของผู้ใช้ daemon (หรือบริการใน Windows) เป็นโปรแกรมที่ทำงานในพื้นหลังโดยไม่ต้องการการแทรกแซงโดยตรงจากผู้ใช้


หมายเหตุ: ตลอดทั้งเอกสารนี้ เราจะอ้างถึงองค์ประกอบต่างๆ เป็นเวอร์ชัน daemon เสมอ อย่างไรก็ตาม ทุกอย่างสามารถทำได้ด้วย elements-qt โดยมีเงื่อนไขว่าตัวเลือกเซิร์ฟเวอร์ถูกเปิดใช้งาน


Elements daemon เชื่อมต่อกับโหนดอื่นๆ บนเครือข่ายเพื่อให้สามารถแลกเปลี่ยนข้อมูลธุรกรรมและบล็อกได้ โดยการตรวจสอบและขยายสำเนาท้องถิ่นของบล็อกเชนของเครือข่าย


ซอฟต์แวร์ Elements ยังประกอบด้วยโปรแกรมไคลเอนต์ที่เรียกว่า `elements-cli` ซึ่งช่วยให้คุณสามารถส่งคำสั่ง Remote Procedure Call (RPC) ไปยัง elementsd จากบรรทัดคำสั่งได้ สิ่งนี้สามารถใช้เพื่อสอบถามยอดคงเหลือ wallet ดูข้อมูลธุรกรรมหรือบล็อก หรือกระจายธุรกรรมเป็นต้น การตั้งค่านี้ควรจะคุ้นเคยสำหรับทุกคนที่เคยใช้ Bitcoin ที่เทียบเท่า bitcoind และ bitcoin-cli


เนื่องจากโหนด Elements สามารถกำหนดค่าได้โดยการส่งพารามิเตอร์ในขณะเริ่มต้นหรือผ่านไฟล์การกำหนดค่า จึงเป็นไปได้ที่จะมีมากกว่าหนึ่งอินสแตนซ์ที่ทำงานบนเครื่องเดียวกัน สิ่งนี้มีประโยชน์สำหรับการทดสอบและการพัฒนา เนื่องจากช่วยให้คุณสามารถตั้งค่าเครือข่ายท้องถิ่นของคุณเองบนเครื่องเดียว แต่ละโหนด Elements มีสำเนาของข้อมูลบล็อกเชนของตัวเอง จัดการกลุ่มธุรกรรมที่ยังไม่ได้ยืนยันที่ถูกต้องของตัวเอง และฟังคำขอ RPC บนพอร์ตที่แตกต่างกัน


### ที่เก็บโค้ด Elements และชุมชน


Elements เป็นโครงการโอเพนซอร์ส และสามารถพบซอร์สโค้ดได้ในที่เก็บ GitHub ของ Elements ที่ https://github.com/ElementsProject/elements ที่เก็บนี้ประกอบด้วยซอร์สโค้ดสำหรับโปรแกรม elementsd และ elements-cli พร้อมด้วยเครื่องมือติดตั้งและสร้างสนับสนุน ชุดทดสอบ และเอกสารคำแนะนำบางส่วน


เพื่อเสริมคลังโค้ด ยังมีเว็บไซต์ https://elementsproject.org ซึ่งเป็นแหล่งข้อมูลที่เน้นชุมชน โดยมีคำอธิบายเกี่ยวกับ Elements ว่าคืออะไร ทำงานอย่างไร และมีส่วนของบทเรียนที่ครอบคลุม บทเรียนเน้นการเรียนรู้เกี่ยวกับ Elements โดยการทำตามตัวอย่างคำสั่งบรรทัดคำสั่ง และแสดงวิธีการสร้างแอปพลิเคชันเดสก์ท็อปและเว็บอย่างง่ายบนพื้นฐานของมัน เว็บไซต์ยังแสดงรายการฟอรัมสนทนาชุมชน Elements ที่ได้รับความนิยม และถูกโฮสต์บน GitHub ซึ่งช่วยให้ชุมชนสามารถมีส่วนร่วมในเนื้อหาของเว็บไซต์ได้


ในการรัน Elements บนเครื่องของคุณ คุณจะต้องโคลน (ดาวน์โหลดสำเนา) ซอร์สโค้ดก่อน ติดตั้งการพึ่งพาใดๆ ที่โค้ดต้องการ และสุดท้ายสร้าง daemon และไฟล์ปฏิบัติการของไคลเอนต์ จากนั้นซอฟต์แวร์ Elements ก็พร้อมที่จะถูกกำหนดค่าและรัน


## กำหนดค่าโหนดและเครือข่าย


<chapterId>df1ec0aa-84ea-4149-af7a-b4523d67e1d9</chapterId>


สามารถส่งการตั้งค่าการกำหนดค่าไปยังโหนด Elements เมื่อเริ่มต้นเพื่อเปลี่ยนวิธีการทำงาน ตรวจสอบความถูกต้องของข้อมูล เชื่อมต่อกับโหนดอื่น ๆ และเริ่มต้นข้อมูลบล็อกเชนของมัน


การตั้งค่าจะถูกโหลดจากไฟล์ `elements.conf` ที่กำหนดไว้หรือส่งผ่านเป็นพารามิเตอร์ผ่านทางบรรทัดคำสั่ง


บางสิ่งสามารถเปลี่ยนแปลงได้โดยใช้พารามิเตอร์เหล่านี้:



- ชื่อของ default asset ที่ใช้ในระบบบล็อกเชนแบบสแตนด์อโลน
- จำนวนสินทรัพย์เริ่มต้นที่สร้างขึ้น
- สินทรัพย์ที่จะใช้เมื่อชำระค่าธรรมเนียมการทำธุรกรรมบนเครือข่าย
- ตำแหน่งที่เก็บไฟล์ข้อมูลบล็อกเชน
- ข้อมูลประจำตัว RPC ใช้เชื่อมต่อกับโหนด Bitcoin
- `n ของ m` เกณฑ์ที่ต้องปฏิบัติตามและ[คีย์สาธารณะ](https://planb.academy/resources/glossary/public-key)ที่ถูกต้องที่สามารถเซ็นบล็อกได้
- [สคริปต์](https://planb.academy/resources/glossary/script)ที่จำเป็นต้องปฏิบัติตามเพื่อโอนสินทรัพย์เข้าและออกจากไซด์เชน
- ไม่ว่าจะเชื่อมต่อกับโหนด Bitcoin เป็นไซด์เชนหรือไม่


หลายสิ่งเหล่านี้เป็นส่วนหนึ่งของกฎฉันทามติของเครือข่าย ดังนั้นจึงเป็นสิ่งสำคัญที่ต้องนำไปใช้กับทุกโหนดเมื่อเริ่มต้น บางอย่างสามารถเปลี่ยนแปลงได้หลังจากที่เชนถูกเริ่มต้นแล้ว แต่บางอย่างจำเป็นต้องถูกแก้ไขหลังจากที่ถูกใช้เพื่อเริ่มต้นเชนแล้ว


การใช้พารามิเตอร์จะถูกครอบคลุมในภายหลังในหลักสูตรเมื่อและเมื่อพวกมันเกี่ยวข้องกับแต่ละส่วน


### การดำเนินการพื้นฐานโดยใช้บรรทัดคำสั่ง


หลักสูตรนี้จะแสดงตัวอย่างที่ใช้โปรแกรม `elements-cli` เพื่อทำการเรียก RPC ไปยังโหนด Elements หนึ่งหรือมากกว่า ซึ่งทำภายในเซสชันเทอร์มินัล และเพื่อทำให้คำสั่งสั้นลง จะมีการใช้ `alias` ตามธรรมเนียมนี้ เมื่อคุณเห็นบางอย่างเช่นคำสั่งต่อไปนี้:


```bash
e1-dae

e1-cli getnewaddress
```


`e1-dae` และ `e1-cli` เป็นทางลัดทางการพิมพ์ที่ใช้คุณสมบัติ `alias` ของเทอร์มินัลจริงๆ เมื่อคำสั่งถูกดำเนินการ `e1-dae` และ `e1-cli` จะถูกแทนที่ และคำสั่งที่จะรันจะคล้ายกับ:


```
$HOME/elements/src/elementsd -datadir=$HOME/elementsdir1

$HOME/elements/src/elements-cli -datadir=$HOME/elementsdir1 getnewaddress
```


สิ่งที่เราเห็นด้านบนคือการเรียกให้เริ่มต้น Elements daemon, การเรียกโปรแกรม elements-cli ที่อยู่ในไดเรกทอรี `$HOME/elements/src` และค่าของพารามิเตอร์ `datadir` พารามิเตอร์ `datadir` ช่วยให้เราสามารถระบุที่ตั้งของไฟล์คอนฟิกสำหรับทั้ง daemon และอินสแตนซ์ของไคลเอนต์ และในกรณีของ daemon ที่จะเก็บสำเนาของบล็อกเชน เนื่องจากพวกเขาใช้ไฟล์คอนฟิกเดียวกัน ไคลเอนต์จะสามารถทำการเรียก RPC ไปยัง daemon ได้


โดยการรันคำสั่งข้างต้นอีกครั้ง แต่ใช้ค่า `datadir` ที่แตกต่างกัน เราสามารถเริ่มต้นอินสแตนซ์ของ Elements ได้มากกว่าหนึ่งอินสแตนซ์ โดยแต่ละอินสแตนซ์จะมีสำเนาของบล็อกเชนและการตั้งค่าคอนฟิกของตัวเอง ตามธรรมเนียมนี้ เราจะใช้ชื่อเล่น `e2-dae` และ `e2-cli` ในหลักสูตรเพื่ออ้างถึงไดเรกทอรี datadir ที่แตกต่างจาก e1 ดังนั้นตัวอย่างข้างต้นสำหรับอินสแตนซ์ `e2` ที่สองของเราจะเป็น:


```
$HOME/elements/src/elementsd -datadir=$HOME/elementsdir2

$HOME/elements/src/elements-cli -datadir=$HOME/elementsdir2 getnewaddress
```


สิ่งนี้จะช่วยให้เราสามารถดำเนินการต่างๆ ได้ เช่น การทำธุรกรรมสินทรัพย์ระหว่างโหนด การออกสินทรัพย์ และการตรวจสอบการใช้การปิดบังใน Confidential Transactions ข้ามโหนดต่างๆ บนเครือข่ายเดียวกัน

## Confidential Transactions


<chapterId>263b1c5b-59ed-49e7-b811-95c354f41eae</chapterId>


:::video id=ea2121b6-24a8-458d-91e6-0c92eaf4dc65:::

ในส่วนนี้ คุณจะได้เรียนรู้วิธีการใช้คุณสมบัติ Confidential Transactions ของ Elements


ที่อยู่ทั้งหมดใน Elements จะเป็นค่าเริ่มต้น blinded โดยใช้ Confidential Transactions ซึ่งจะทำให้จำนวนและประเภทของสินทรัพย์ที่โอนสามารถมองเห็นได้เฉพาะผู้เข้าร่วมในธุรกรรม (และผู้ที่พวกเขาเลือกที่จะเปิดเผยกุญแจบอดให้) ในขณะที่ยังคงรับประกันทางการเข้ารหัสว่าจำนวนเหรียญที่ใช้จ่ายจะไม่เกินกว่าที่มีอยู่


### ที่อยู่ที่เป็นความลับและ Confidential Transactions


ตามค่าเริ่มต้น เมื่อคุณสร้างที่อยู่ใหม่ใน Elements โดยใช้คำสั่ง `getnewaddress` ที่อยู่นั้นจะถูกสร้างเป็นที่อยู่ที่เป็นความลับ


เพื่อสาธิตการทำธุรกรรมที่เป็นความลับ เราจะให้ e2 ส่งเงินให้ตัวเองและพยายามดูการทำธุรกรรมจาก e1 สิ่งนี้จะแสดงให้เห็นถึงลักษณะที่เป็นความลับของการทำธุรกรรมใน Elements


ทุกที่อยู่ใหม่ที่สร้างโดยโหนด Elements จะเป็นความลับโดยค่าเริ่มต้น เราสามารถแสดงให้เห็นสิ่งนี้โดยการให้ e2 ไปที่ generate เพื่อสร้างที่อยู่ใหม่


```
e2-cli getnewaddress
```


โปรดทราบว่าที่อยู่เริ่มต้นด้วย 'e1' ซึ่งระบุว่าเป็น Confidential Address การตรวจสอบที่อยู่อย่างละเอียดโดยใช้คำสั่ง getaddressinfo จะเผยรายละเอียดเพิ่มเติมเกี่ยวกับที่อยู่นี้


```
e2-cli getaddressinfo <address>
```


คุณสามารถเห็นได้ว่ามีคุณสมบัติ confidential_key ซึ่งบ่งบอกว่าเป็นที่อยู่ที่เป็นความลับ


confidential_key คือกุญแจบังตาสาธารณะ ซึ่งถูกเพิ่มเข้าไปในที่อยู่ที่เป็นความลับเอง นี่คือเหตุผลว่าทำไมที่อยู่ที่เป็นความลับจึงยาวมาก


นอกจากนี้ยังมีที่อยู่ที่ไม่เป็นความลับที่เกี่ยวข้อง หากคุณต้องการใช้ธุรกรรมปกติที่ไม่เป็นความลับภายใน Elements โปรดส่งสินทรัพย์ไปยังที่อยู่นี้แทนที่จะเป็นที่อยู่ที่มีคำนำหน้า 'lq1'


ตอนนี้เราสามารถให้ e2 ส่งเงินบางส่วนไปยังที่อยู่ที่มันสร้างขึ้นได้แล้ว ซึ่งจะเป็นการแสดงให้เห็นในภายหลังว่า e1 ซึ่งไม่ใช่หนึ่งในคู่สัญญา จะไม่สามารถดูรายละเอียดการทำธุรกรรมได้


```
e2-cli sendtoaddress <address>
```


โปรดทราบ transaction ID ยืนยันการทำธุรกรรม


```
e2-cli -generate 101
```


กำลังดูที่ธุรกรรม ซึ่ง e2 ส่งเงินบางส่วนให้กับตัวเองจากมุมมองของ e2 เอง


```
e2-cli gettransaction <txid>
```


เลื่อนขึ้นไปที่รายละเอียดการทำธุรกรรม คุณจะเห็นว่า e2 สามารถดูจำนวนเงินที่ส่งและรับได้ รวมถึงสินทรัพย์ที่ทำธุรกรรมด้วย คุณยังสามารถเห็นคุณสมบัติ amountblinder และ assetblinder ซึ่งใช้ในการปิดบังรายละเอียดจากโหนดอื่น ๆ ที่ไม่ได้เกี่ยวข้องกับการทำธุรกรรม


ในการตรวจสอบรายละเอียดของธุรกรรมเดียวกันจาก e1 เราจำเป็นต้องได้รับรายละเอียดธุรกรรมดิบก่อน


```
e1-cli getrawtransaction <txid>
```


ที่ส่งคืนรายละเอียดธุรกรรมดิบ หากคุณดูในส่วน vout คุณจะเห็นว่ามีสามอินสแตนซ์ อินสแตนซ์แรกสองอันคือจำนวนที่รับและจำนวนเงินทอน และอินสแตนซ์ที่สามคือค่าธรรมเนียมการทำธุรกรรม จากจำนวนทั้งสามนี้ ค่าธรรมเนียมเป็นเพียงอันเดียวที่คุณสามารถเห็นค่าได้ เนื่องจากค่าธรรมเนียมเองจะเป็น unblinded เสมอภายใน Elements


### Blinding Keys


สิ่งที่สองส่วนแรกของ VOUT แสดงคือ "ช่วง blinded" ของจำนวนค่าและข้อมูลการผูกพัน ซึ่งทำหน้าที่เป็นหลักฐานของจำนวนจริงและประเภทของสินทรัพย์ที่ทำธุรกรรม


แม้ว่าเราจะนำเข้า[คีย์ส่วนตัว](https://planb.academy/resources/glossary/private-key)ของ e2 ไปยัง wallet ของ e1 ก็ยังไม่สามารถเห็นจำนวนและประเภทของสินทรัพย์ที่ทำธุรกรรมได้ เพราะยังไม่มีความรู้เกี่ยวกับคีย์บลายด์ที่ใช้โดย e2 เราจะพิสูจน์สิ่งนี้โดยการนำเข้าคีย์ส่วนตัวที่ใช้โดย wallet ของ e2 ไปยัง e1 ก่อนอื่นเราต้องส่งออกคีย์จาก e2


```
e2-cli dumpprivkey <address>
```


จากนั้นนำเข้ามันเข้าไปใน e1.


```
e1-cli importprivkey <privkey>
```


ตอนนี้เราสามารถพิสูจน์ได้ว่า e1 ยังคงไม่สามารถเห็นค่าได้


```
e1-cli gettransaction <txid>
```


ในความเป็นจริง มันแสดง 0 เป็นจำนวน tx เมื่อมันควรจะเป็น 1


ในการดูค่า unblinded ที่แท้จริง เราจำเป็นต้องใช้กุญแจบลายด์ เพื่อทำเช่นนี้ เราต้องส่งออกกุญแจบลายด์จาก e2 ก่อน


```
e2-cli dumpblindingkey <address>
```


จากนั้นนำเข้ามันไปยัง e1.


```
e1-cli importblindingkey <address> <blinding key>
```


ตอนนี้ เมื่อเราได้รับรายละเอียดการทำธุรกรรมจาก e1.


```
e1-cli gettransaction <txid>
```


มันแสดงให้เห็นว่าเมื่อมีการนำเข้าคีย์บลายด์ เราสามารถดูค่าจริงของ 1 ภายในธุรกรรมได้แล้ว


ในส่วนนี้ เราได้เห็นว่าการใช้กุญแจบังตาสามารถซ่อนจำนวนและประเภทของสินทรัพย์ในธุรกรรมได้ และโดยการนำเข้ากุญแจบังตาที่ถูกต้อง เราสามารถเปิดเผยค่าดังกล่าวได้ ในการใช้งานจริง กุญแจบังตาอาจถูกมอบให้กับผู้ตรวจสอบบัญชี หากมีความจำเป็นต้องตรวจสอบจำนวนและประเภทของสินทรัพย์ที่ถือครองโดยฝ่ายหนึ่ง ฟีเจอร์ Confidential Transactions ของ Elements ยังอนุญาตให้มีการดำเนินการ “การพิสูจน์ช่วง” ได้อีกด้วย การพิสูจน์ช่วงสามารถพิสูจน์ได้ว่าจำนวนของสินทรัพย์อยู่ในช่วงที่กำหนด โดยไม่จำเป็นต้องเปิดเผยจำนวนที่แท้จริงเอง


เราได้เห็นแล้วว่า Confidential Transactions เป็นทางเลือก แต่จะเปิดใช้งานโดยค่าเริ่มต้นเมื่อมีการสร้างที่อยู่ใหม่


จบบทเรียนนี้แล้ว; ขอให้โชคดีในการทำแบบทดสอบ แล้วพบกันใหม่ในบทเรียนถัดไป!


## Issued Assets


<chapterId>c33c7020-5975-457a-99db-4f8b90d1fa1c</chapterId>


:::video id=7ac63148-d730-496d-85d4-0032aaf09be1:::

ในส่วนนี้ คุณจะได้เรียนรู้วิธีการใช้คุณสมบัติ Issued Assets ของ Elements


Issued Assets ช่วยให้สามารถออกและโอนสินทรัพย์หลายประเภทระหว่างผู้เข้าร่วมเครือข่าย Elements ได้ โหนดใด ๆ บนเครือข่ายสามารถออกสินทรัพย์ของตนเองได้ การออกอาจแสดงถึงความเป็นเจ้าของที่สามารถแลกเปลี่ยนได้ของสินทรัพย์ใด ๆ รวมถึงบัตรกำนัล คูปอง สกุลเงิน เงินฝาก พันธบัตร หุ้น และรายการอื่น ๆ ที่คล้ายกัน Issued Assets เปิดประตูสำหรับการสร้างการแลกเปลี่ยนที่ไม่ต้องอาศัยความไว้วางใจ ตัวเลือก และสัญญาอัจฉริยะขั้นสูงอื่น ๆ ที่เกี่ยวข้องกับสินทรัพย์ที่ออกด้วยตนเอง


สินทรัพย์ที่ออกแล้วยังได้รับประโยชน์จาก Confidential Transactions และสามารถออกใหม่ได้โดยใครก็ตามที่ถือ token ที่เกี่ยวข้อง


ขั้นตอนแรกคือเราจะต้องเข้าถึงโหนด Elements สองตัว ซึ่งเราจะเรียกว่า e1 และ e2 โหนดเหล่านี้ได้ถูกรีเซ็ตบล็อกเชนแล้ว และ default asset ถูกแบ่งระหว่างพวกมัน


โหนดทั้งสองอยู่บนเครือข่ายท้องถิ่นเดียวกันและเชื่อมต่อถึงกัน ดังนั้นพวกเขาจึงแชร์ธุรกรรมเดียวกันใน [mempool](https://planb.academy/resources/glossary/mempool) ธุรกรรมของพวกเขาและบล็อกเชนที่เหมือนกัน แม้ว่าพวกเขาจะทำงานบนเครื่องเดียวกัน แต่ก็ควรสังเกตว่าพวกเขาไม่ได้แชร์ไฟล์บล็อกเชนจริงเดียวกัน แต่ละโหนดจัดการสำเนาท้องถิ่นของบล็อกเชนของตัวเอง ซึ่งมีประวัติธุรกรรมเดียวกันเพราะพวกเขาอยู่ในฉันทามติและปฏิบัติตามกฎโปรโตคอลเดียวกัน


มาเริ่มต้นด้วยการตรวจสอบมุมมองของแต่ละโหนดเกี่ยวกับการออกสินทรัพย์ที่มีอยู่บนเครือข่ายกันเถอะ


สิ่งนี้ทำโดยใช้คำสั่ง listissuances


```
e1-cli listissuances

e2-cli listissuances
```


ตามที่คุณเห็น โหนดทั้งสองแสดงประวัติการออกสินทรัพย์เหมือนกัน พวกเขาทั้งสองแสดงสินทรัพย์หนึ่งรายการ การออกครั้งแรกของ 21 ล้าน Bitcoin ซึ่งถูกสร้างขึ้นในระหว่างการเริ่มต้น on chain คุณสามารถเห็นรหัส hex ของสินทรัพย์ในผลลัพธ์ของการรันคำสั่งข้างต้น รวมถึงป้ายกำกับที่กำหนดให้กับสินทรัพย์ซึ่งคือ 'bitcoin'


ควรสังเกตว่า default asset จะได้รับป้ายกำกับเสมอเมื่อมีการเริ่มต้นเชน เมื่อคุณออกสินทรัพย์ของคุณเอง คุณสามารถตั้งค่าป้ายกำกับสำหรับสินทรัพย์เหล่านั้นได้เอง ซึ่งเราจะทำในไม่ช้า ก่อนที่เราจะทำเช่นนั้น เราจำเป็นต้องออกสินทรัพย์ของเราเองก่อน


เราจะให้ e1 ออกสินทรัพย์ใหม่ ซึ่งทำได้โดยใช้คำสั่ง issueasset


```
e1-cli issueasset 100 1 false
```


`issueasset` ยอมรับ 3 พารามิเตอร์


จำนวนสินทรัพย์ใหม่ที่จะออกคือ 100 จำนวนโทเค็นที่จะสร้าง (โทเค็นใช้ในการออกจำนวนสินทรัพย์ใหม่) ซึ่งเราเลือก 1 พารามิเตอร์สุดท้ายบอก Elements ให้สร้างการออกสินทรัพย์เป็น blinded หรือ unblinded เราจะใช้ 'unblinded' เพราะเราต้องการดูจำนวนการออกจาก e2 ในอีกสักครู่ ดังนั้นเราจะป้อน 'false'


การรันคำสั่งจะส่งคืนข้อมูลเกี่ยวกับการออก ซึ่งรวมถึง transaction ID ที่คุณสามารถคัดลอกเพื่อใช้ในภายหลัง ค่าฮี๊กซ์ที่ไม่ซ้ำกันของสินทรัพย์ และค่าฮี๊กซ์ที่ไม่ซ้ำกันของ token ของสินทรัพย์


สร้างบล็อกเพื่อยืนยันการทำธุรกรรมการออก.


```
e1-cli -generate 1
```


เรียกใช้คำสั่ง `listissuances` กับ e1 อีกครั้ง


```
e1-cli listissuances
```


ซึ่งแสดงให้เราเห็นว่า e1 ตอนนี้รับรู้ถึงการออกสินทรัพย์ 2 รายการ การออกสินทรัพย์ครั้งแรกของ Bitcoin และสินทรัพย์ที่เราเพิ่งออกใหม่ ซึ่งเราสามารถเห็นจำนวน 100 โปรดสังเกตค่าฮีกซ์ของสินทรัพย์ใหม่และไม่มีป้ายกำกับสินทรัพย์ที่เกี่ยวข้อง แตกต่างจากการออกสินทรัพย์ของบิทคอยน์


ตรวจสอบรายการการออกที่ทราบของ e2 อีกครั้ง


```
e2-cli listissuances
```


นั่นแสดงให้เราเห็นว่า e2 ไม่ทราบเกี่ยวกับการออกสินทรัพย์ที่ e1 ทำ มันสามารถเห็นได้เพียงการออกบิตคอยน์ครั้งแรกที่มันสามารถเห็นได้อยู่แล้ว


เนื่องจาก e2 ไม่ทราบและไม่ได้เฝ้าดูที่อยู่ที่สินทรัพย์ใหม่ถูกส่งไปเมื่อมันถูกออกโดย e1


ควรสังเกตว่าแม้ว่า e2 จะไม่สามารถเห็นการออกสินทรัพย์เองได้ แต่ e1 ก็ยังสามารถส่งสินทรัพย์บางส่วนให้กับ e2 ได้ สินทรัพย์ใหม่จะปรากฏเป็นยอดคงเหลือที่ใช้ได้ใน wallet ของ e2 แม้ว่าจะไม่ทราบการออกสินทรัพย์เดิมก็ตาม


ในการเปิดใช้งาน e2 เพื่อดูการออกจริง (และดังนั้นจำนวนที่ออก) เราจำเป็นต้องเพิ่มที่อยู่ไปยัง e2 เป็นที่อยู่ที่เฝ้าดู


ในการทำเช่นนี้ เราจำเป็นต้องระบุที่อยู่ที่สินทรัพย์ถูกส่งไป สำหรับสิ่งนี้ เราจะใช้ transaction ID ที่เราคัดลอกมาก่อนหน้านี้และให้ e1 ดึงรายละเอียดการทำธุรกรรมเพื่อที่เราจะสามารถหาที่อยู่ที่ถูกต้องเพื่อเพิ่มไปยังรายการเฝ้าดู wallet ของ e2 ได้


```
e1-cli gettransaction <the-issuance-transaction-id>
```


เลื่อนขึ้นผ่านข้อมูลธุรกรรมในรูปแบบเลขฐานสิบหก คุณจะเห็นที่อยู่ที่ได้รับสินทรัพย์ใหม่ของเราจำนวน 100 หน่วย ซึ่งระบุโดยค่าเลขฐานสิบหกของมัน


คัดลอกที่อยู่นี้เพื่อที่เราจะสามารถนำเข้าไปใน e2 ได้


ตอนนี้เรามานำเข้าที่อยู่นั้นไปยัง e2 กันเถอะ ในการทำเช่นนี้ เราใช้คำสั่ง importaddress


```
e2-cli importaddress <the-issued-to-address>
```


หากเราตรวจสอบรายการการออกของ e2 ในขณะนี้


```
e2-cli listissuances
```


คุณสามารถเห็นได้ว่าสินทรัพย์ที่เราออกใหม่ได้ถูกเพิ่มในรายการแล้ว โหนด e2 ยังสามารถระบุจำนวนของสินทรัพย์ที่ถูกออก รวมถึงจำนวนของ token ที่เกี่ยวข้อง เนื่องจากการออกนี้เป็นการออก unblinded เพื่อเปิดใช้งานการใช้การจับคู่ชื่อกับรหัสสินทรัพย์ภายใน Elements ให้หยุด Elements ก่อน


```
e1-cli stop
```


จากนั้นรีสตาร์ทด้วยพารามิเตอร์เพิ่มเติมที่แมปเลขฐานสิบหกของสินทรัพย์ไปยังป้ายกำกับที่ให้ไว้ ซึ่งจะช่วยให้โหนดแสดงข้อมูลเกี่ยวกับสินทรัพย์ให้เราในรูปแบบที่มนุษย์อ่านได้มากขึ้น คุณสามารถเพิ่มสิ่งนี้ไปยังส่วนท้ายขององค์ประกอบ หากคุณต้องการ คุณสามารถละเว้นอาร์กิวเมนต์ไปยัง daemon ทุกครั้งที่คุณเริ่มต้นได้ ตัวอย่างเช่น:


```
assetdir=5186d0bc8ed15e6ef85571bd2d8070573adf0e06fd4507082694526975ce4f35:My new asset (MNA)
```


แต่เราจะใช้วิธีการโต้แย้งที่นี่


```
e1-dae -assetdir=<assetid-here>:<name-of-the-new-asset>
```


กำลังสอบถามโหนดสำหรับรายการการออกอีกครั้ง


```
e1-cli listissuances
```


นั่นบ่งชี้ว่าการจับคู่ค่าฮ็กซ์ของสินทรัพย์กับป้ายกำกับของมันทำงานได้อย่างถูกต้อง กำลังตรวจสอบรายการการออกของโหนด e2 อีกครั้ง


```
e2-cli listissuances
```


คุณสามารถเห็นได้ว่าโหนด e2 ไม่มีสิทธิ์เข้าถึงป้ายกำกับนี้ เนื่องจากป้ายกำกับจะสามารถใช้ได้เฉพาะกับโหนดที่ตั้งค่าเท่านั้น จริงๆ แล้ว เราสามารถกำหนดป้ายกำกับที่แตกต่างกันให้กับสินทรัพย์ hex เดียวกันบน e2 ได้เหมือนที่เราทำบน e1 ก่อนอื่น ให้หยุดโหนด e2 ก่อน


```
e2-cli stop
```


เริ่มต้นใหม่ด้วยการกำหนดป้ายชื่อที่แตกต่างให้กับเลขฐานสิบหกของสินทรัพย์ใหม่ของเรา


```
e2-dae -assetdir=<assetid-here>:<another-name-for-the-new-asset>
```


การออกใบแจ้งหนี้จาก e2.


```
e2-cli listissuances
```


ป้ายกำกับสินทรัพย์เป็นแบบท้องถิ่นสำหรับแต่ละโหนด; เฉพาะเลขฐานสิบหกของสินทรัพย์เท่านั้นที่ได้รับการยอมรับโดยโหนดอื่น ๆ บนเครือข่าย


การแมปป้ายกำกับไปยังสินทรัพย์ในรูปแบบเลขฐานสิบหกมีประโยชน์เมื่อดำเนินการต่างๆ เช่น การทำธุรกรรมและการสอบถามยอดคงเหลือ wallet เนื่องจากช่วยให้สามารถอ้างอิงสินทรัพย์ได้ในรูปแบบย่อ ตัวอย่างเช่น หากเราต้องการส่งสินทรัพย์ใหม่ของเรา (จำนวน 10) จาก e1 ไปยัง e2 โดยไม่ใช้ป้ายกำกับ


ก่อนอื่น เราจำเป็นต้องได้รับที่อยู่เพื่อส่งสินทรัพย์ไปให้


```
e2-cli getnewaddress
```


จากนั้นใช้คำสั่ง sendtoaddress


```
e1-cli sendtoaddress <address> 10 "" "" false false 1 UNSET false <asset-id-here>
```


ยืนยันการทำธุรกรรมโดยการสร้างบล็อก


```
generate 1
```


กำลังตรวจสอบว่าทรัพย์สินได้รับบน e2 แล้ว


```
e2-cli getwalletinfo
```


เราสามารถเห็นได้ว่าสินทรัพย์ได้รับแล้วจริงๆ


โปรดทราบว่า e2 จะแปลงค่า hex ของสินทรัพย์ที่ได้รับและแสดงโดยใช้ป้ายกำกับของตัวเอง วิธีที่ง่ายกว่าที่จะทำสิ่งเดียวกันคือการใช้ป้ายกำกับสินทรัพย์ของ e1 เมื่อทำการส่ง


```
e1-cli sendtoaddress <address> 10 "" "" false false 1 UNSET false <name-of-the-new-asset>
```


เบื้องหลัง, Elements ทำการแม็ปป้ายกำกับท้องถิ่นไปยังค่าเฮกซ์เพื่อช่วยให้การใช้สินทรัพย์ที่ออกง่ายขึ้น


ในส่วนนี้ เราได้เห็นวิธีการออกและติดฉลากสินทรัพย์แล้ว ในส่วนถัดไป เราจะตรวจสอบกระบวนการออกใหม่และทำลายจำนวนของสินทรัพย์ที่ออกแล้ว


## การออกสินทรัพย์ใหม่


<chapterId>78751b21-1dc8-4877-a406-e71bc80a95b0</chapterId>


:::video id=7df967b0-ffff-42e1-b1d5-868e76289faf:::

ในส่วนนี้ คุณจะได้เรียนรู้วิธีการออกสินทรัพย์ที่ออกแล้วเพิ่มเติม และวิธีการทำลายจำนวนที่กำหนดของสินทรัพย์ที่ออกแล้ว


ความจำเป็นในการออกใหม่ (หรือสร้างเพิ่มเติม) ของสินทรัพย์หรือทำลายส่วนหนึ่งของสินทรัพย์มีแนวโน้มที่จะเกิดขึ้นเมื่อสินทรัพย์นั้นแทนสิ่งที่ไม่มีอุปทานคงที่ สิ่งนี้อาจใช้กับสินทรัพย์ที่แทนทองคำที่เก็บไว้ในห้องนิรภัย ตัวอย่างเช่น เมื่อหน่วยของทองคำเคลื่อนเข้าและออกจากห้องนิรภัย สินทรัพย์ที่แทนอุปทานของห้องนิรภัยจำเป็นต้องได้รับการปรับตามนั้น


การออกจำนวนสินทรัพย์ใหม่ต้องมีความเป็นเจ้าของ token ที่เกี่ยวข้องซึ่งถูกสร้างขึ้นพร้อมกับสินทรัพย์เมื่อมีการออกครั้งแรก


เมื่อสร้างสินทรัพย์เพิ่มเติม ไม่สำคัญว่าสินทรัพย์นั้นถูกออกโดยโหนดใดในตอนแรก ตราบใดที่โหนดที่ออกสินทรัพย์เพิ่มเติมมีสิ่งที่เรียกกันทั่วไปว่า reissuance token ของสินทรัพย์นั้น เราจะตรวจสอบวิธีการสร้าง reissuance token ในตอนแรก วิธีการใช้มันเพื่อออกสินทรัพย์ในจำนวนที่กำหนดใหม่ และวิธีการโอน reissuance token ไปยังโหนดอื่น ๆ เพื่อให้สิทธิ์พวกเขาในการออกสินทรัพย์ใหม่


เราจะต้องเข้าถึงโหนด Elements สองโหนด ซึ่งเราจะเรียกว่า e1 และ e2 โหนดเหล่านี้ได้ถูกรีเซ็ตบล็อกเชนแล้ว และ default asset ได้ถูกแบ่งระหว่างพวกมัน


เราจะให้ e1 ออกจำนวน 100 ของสินทรัพย์ใหม่และสร้าง 1 reissuance token สำหรับสินทรัพย์เดียวกัน เราจะสร้างการออกเป็น unblinded เพื่อให้ง่ายต่อการทำความเข้าใจตัวอย่าง ดังนั้นให้เราออกสินทรัพย์และ reissuance token ที่เกี่ยวข้องกัน


```
e1-cli issueasset 100 1 false
```


โปรดทราบ ID ของสินทรัพย์และยังรวมถึงของ (การออกใหม่) token ด้วย


เนื่องจากเราจะออกสินทรัพย์เพิ่มเติมจาก e2 ในภายหลัง เราจำเป็นต้องจดบันทึก transaction ID ซึ่งเป็นหมายเลขที่สินทรัพย์ถูกออกและใช้เพื่อการนำเข้าที่อยู่ที่สินทรัพย์ถูกส่งไป


ยืนยันการทำธุรกรรม


```
e1-cli -generate 1
```


เราจะตรวจสอบรายละเอียดของธุรกรรมโดยใช้คำสั่ง gettransaction:


```
e1-cli gettransaction <txid>
```


เลื่อนขึ้นผ่านข้อมูล hex ของธุรกรรม คุณจะเห็นว่าในธุรกรรมนี้ e1 ได้รับ 1 reissuance token และ 100 ของสินทรัพย์ที่เกี่ยวข้อง


คัดลอกที่อยู่นี้เพื่อที่เราจะสามารถนำเข้าไปใน e2 ได้


และตอนนี้กำลังนำเข้าที่อยู่เข้าสู่ e2's wallet.


```
e2-cli importaddress <address>
```


ตอนนี้เราสามารถเห็นได้ว่า ทั้ง e1 และ e2 รับรู้เกี่ยวกับการออกสินทรัพย์แล้ว


```
e1-cli listissuances

e2-cli listissuances
```


ปัจจุบัน e1 ถือครองสินทรัพย์และ 1 reissuance token แต่ e2 ไม่ถือครอง


```
e1-cli getwalletinfo
```


โปรดทราบว่า e1 มี default asset น้อยลงกว่าเดิมเนื่องจากได้จ่ายจำนวนเล็กน้อยเพื่อครอบคลุมค่าธรรมเนียมการทำธุรกรรม จำนวนนี้จะถูกเก็บโดย e1 เมื่อบล็อกที่สร้างขึ้นนั้นเติบโตเต็มที่และมีความลึกมากกว่า 100 บล็อก


```
e2-cli getwalletinfo
```


ในขณะที่ e1 ถือ reissuance token, มันสามารถออกใหม่ได้มากขึ้น โดยใช้คำสั่ง reissueasset ให้ e1 ออกใหม่อีก 100 ของสินทรัพย์


```
e1-cli reissueasset <asset-id> 100
```


กำลังตรวจสอบว่าการออกใหม่ทำงานหรือไม่


```
e1-cli getwalletinfo
```


คุณสามารถเห็นได้ว่า e1 ตอนนี้ถือครองสินทรัพย์จำนวน 200 ตามที่คาดไว้


เนื่องจาก e2 ไม่มีจำนวนของ reissuance token พวกเขาจะได้รับข้อผิดพลาดหากพยายามออกสินทรัพย์ใหม่


```
e2-cli reissueasset <asset-id> 100
```


โปรดทราบข้อความแสดงข้อผิดพลาด


เราสามารถดูรายละเอียดของการออกใหม่จาก e1 โดยใช้คำสั่ง listissuances


```
e1-cli listissuances
```


สังเกตธง `is_reissuance`.


หากเราส่ง reissuance token จำนวนหนึ่งไปยัง e2 ตอนนี้ พวกเขาจะสามารถออกสินทรัพย์จำนวนหนึ่งได้ด้วยตนเอง ก่อนอื่นเราต้องมีที่อยู่เพื่อส่งไปให้ ควรสังเกตว่า reissuance token ถูกปฏิบัติเช่นเดียวกับสินทรัพย์อื่น ๆ ภายใน elements เมื่อส่งและแสดงยอดคงเหลือ และยังสามารถแบ่งออกเป็นหน่วยย่อยได้เช่นเดียวกับสินทรัพย์อื่น ๆ ดังนั้นเราไม่จำเป็นต้องส่ง 1 reissuance token ไปยัง e2 เพื่อให้สามารถออกสินทรัพย์ได้ การส่งในหน่วยใดก็เพียงพอแล้ว กำลังสร้างที่อยู่สำหรับ e2 เพื่อรับ reissuance token


```
e2-cli getnewaddress
```


จากนั้นส่งเศษส่วนของ RIT จาก e1 ไปยัง e2.


```
e1-cli sendtoaddress <address-of-e2> 0.1 "" "" false false 1 UNSET false <reissuance-token-id>
```


ยืนยันการทำธุรกรรม


```
e1-cli -generate 1
```


ตอนนี้เราสามารถเห็นได้ว่า e2 ถือ 0.1 ที่ถูกส่งไปแล้ว


```
e2-cli getwalletinfo
```


ซึ่งหมายความว่า e2 สามารถออกสินทรัพย์เพิ่มเติมที่เกี่ยวข้องกับ RIT ที่ถืออยู่ใน wallet ได้แล้ว เราจะให้ e2 ออกสินทรัพย์จำนวน 500 หน่วยใหม่


```
e2-cli reissueasset <asset-id> 500
```


ตรวจสอบผลการออกใหม่


```
e2-cli getwalletinfo
```


คุณสามารถเห็นได้ว่า e2 ขณะนี้ถือจำนวนที่ออกใหม่ในยอดคงเหลือ wallet และ RIT เองไม่ได้ถูกใช้ไปในกระบวนการออกสินทรัพย์ใหม่


การทำลายจำนวนสินทรัพย์เป็นสิ่งที่ใครก็ตามที่ถือครองอย่างน้อยในจำนวนนั้นสามารถทำได้; มันไม่ได้ถูกควบคุมโดย reissuance token.


```
e2-cli destroyamount <asset-id>

e2-cli getwalletinfo
```


ในส่วนนี้ เราได้เห็นวิธีการออกสินทรัพย์ พร้อมกับวิธีการใช้ reissuance token ที่ถูกสร้างขึ้นเป็นทางเลือกในกระบวนการออกสินทรัพย์ เราได้เห็นด้วยว่าการโอน reissuance token นั้นง่ายเหมือนกับการโอนสินทรัพย์อื่น ๆ และการถือครอง reissuance token ในจำนวนใด ๆ จะให้สิทธิ์แก่ผู้ถือในการออกสินทรัพย์ที่เกี่ยวข้องเพิ่มเติม ดังนั้นจึงเป็นสิ่งสำคัญที่จะควบคุมว่าใครมีสิทธิ์เข้าถึงโทเค็นการออกใหม่ภายในเครือข่ายของคุณ เราได้เห็นด้วยว่าการทำลายสินทรัพย์ในจำนวนหนึ่งนั้นไม่ถูกควบคุมโดยการครอบครอง reissuance token

## บล็อกการลงนาม


<chapterId>c47b217e-db14-4843-a66f-3e5f3a00a808</chapterId>


:::video id=c5a81820-77d7-4a0c-9a4e-9323386a74ac:::

Elements รองรับรูปแบบการลงนามแบบสหพันธ์ ซึ่งช่วยให้คุณสามารถระบุจำนวนสมาชิก Strong Federation ที่ต้องลงนามในบล็อกที่เสนอเพื่อสร้างบล็อกที่ถูกต้องได้


ก่อนหน้านี้ และเพื่อความง่ายในการยกตัวอย่าง เราได้สร้างบล็อกโดยใช้คำสั่ง `generate` ซึ่งไม่จำเป็นต้องมีการลงนามหลายลายเซ็นเพื่อให้บล็อกที่สร้างขึ้นได้รับการยอมรับจากเครือข่ายว่าเป็นบล็อกที่ถูกต้อง


เราจะตั้งค่าโหนดของเราให้ต้องการการสร้างบล็อกแบบ 2-of-2 multisig ซึ่งจะถูกตั้งค่าโดยใช้พารามิเตอร์ signblockscript ซึ่งสามารถเพิ่มลงในไฟล์ config หรือส่งเข้าไปในโหนดเมื่อเริ่มต้นได้ ในการเริ่มต้นเชนด้วยพารามิเตอร์นี้ เราจำเป็นต้องสร้างสคริปต์ที่ประกอบด้วยเชนก่อน


เราจะทำสิ่งนี้โดยใช้โหนดที่มีอยู่แล้วบางส่วน บันทึกข้อมูลที่พวกมันส่งออก จากนั้นล้างเชนเพื่อให้เราสามารถเริ่มต้นใหม่โดยใช้พารามิเตอร์ signblockscript นี่เป็นสิ่งจำเป็นเพราะสคริปต์เป็นส่วนหนึ่งของกฎฉันทามติของเครือข่ายและต้องตั้งค่าเป็น on-chain ในระหว่างกระบวนการเริ่มต้น ไม่สามารถเพิ่มเข้าไปในเชนที่มีอยู่ในภายหลังได้


เราจะต้องเข้าถึงโหนด Elements สองโหนด ซึ่งเราจะเรียกว่า e1 และ e2 โหนดเหล่านี้ได้ถูกรีเซ็ตบล็อกเชนแล้ว และ default asset ได้ถูกแบ่งระหว่างพวกมัน


ตรวจสอบให้แน่ใจว่าพารามิเตอร์ con_max_block_sig_size ถูกตั้งค่าเป็นค่าสูงในไฟล์ elements.conf ของคุณ มิฉะนั้น block signing จะล้มเหลวในภายหลังในส่วนนี้ เราได้ตั้งค่า con_max_block_sig_size เป็น 2000 สำหรับบทแนะนำนี้


เนื่องจากเราจะทำการรีเซ็ตบล็อกเชนและล้างกระเป๋าเงินที่เชื่อมโยงกับ e1 และ e2 เราจำเป็นต้องคัดลอกที่อยู่, กุญแจสาธารณะ, และกุญแจส่วนตัวที่ใช้กับ generate สคริปต์ block signing เพื่อให้เราสามารถใช้งานได้ในภายหลัง


ก่อนอื่น เราต้องให้แต่ละโหนด block signing ไปที่ generate เพื่อรับที่อยู่ใหม่ ซึ่งคุณต้องทำสำเนาไว้ด้วย


```
e1-cli getnewaddress

e2-cli getnewaddress
```


จากนั้นเราจำเป็นต้องดึงคีย์สาธารณะจากที่อยู่และบันทึกไว้เพื่อใช้ในภายหลัง


```
e1-cli getaddressinfo <e1-address>

e2-cli getaddressinfo <e2-address>
```


จากนั้น ให้ดึงคีย์ส่วนตัวออกมา ซึ่งเราจะนำเข้ากลับในภายหลัง เพื่อให้โหนดสามารถเซ็นชื่อบล็อกหลังจากที่เราเริ่มต้นบล็อกเชนและข้อมูล wallet ของเราใหม่


```
e1-cli dumpprivkey <e1-address>

e2-cli dumpprivkey <e2-address>
```


ตอนนี้เราจำเป็นต้อง generate สคริปต์การแลกด้วยข้อกำหนดลายเซ็นหลายตัวแบบ 2-of-2 เราทำสิ่งนี้โดยใช้คำสั่ง create-multisig โดยส่งพารามิเตอร์แรกเป็น 2 และจากนั้นให้คีย์สาธารณะสองคีย์ มันคือคีย์เหล่านี้ที่การเป็นเจ้าของต้องพิสูจน์ในภายหลังเมื่อบล็อกถูกสร้างขึ้น


โหนดใดก็ได้, e1 หรือ e2, สามารถ generate มัลติซิกได้


```
e1-cli createmultisig 2 '["<e1-pubkey>", "<e2-pubkey>"]'
```


นั่นทำให้เราได้สคริปต์การแลกของเรา ซึ่งคุณสามารถคัดลอกไปใช้ในภายหลังได้


ตอนนี้เราจำเป็นต้องล้างข้อมูลบล็อกเชนและข้อมูล wallet ที่มีอยู่เพื่อที่เราจะได้เริ่มต้นใหม่กับ signblockscript เป็นส่วนหนึ่งของกฎฉันทามติของเชน นี่คือเหตุผลที่เราจำเป็นต้องคัดลอกข้อมูลบางส่วนไว้ก่อน เช่น กุญแจส่วนตัวที่จะใช้ในเชนใหม่เพื่อเซ็นบล็อก คุณต้องทำขั้นตอนนี้ให้เสร็จก่อนที่จะดำเนินการต่อ


ด้วยข้อมูล wallet และ chain ที่มีอยู่ของเราถูกลบไปแล้ว ตอนนี้เราสามารถเริ่มต้น nodes ของเราและให้พวกมันเริ่มต้น chain ใหม่โดยใช้พารามิเตอร์ signblockscript ได้แล้ว ให้เราใส่ -evbparams=dynafed:0::: เพื่อปิดการใช้งาน dynafed เพราะเราไม่ต้องการฟีเจอร์ขั้นสูงนั้นสำหรับตัวอย่างนี้


```
e1-dae -signblockscript=<redeem-script> -evbparams=dynafed:0:::

e2-dae -signblockscript=<redeem-script> -evbparams=dynafed:0:::
```


ตอนนี้เราจำเป็นต้องนำเข้าคีย์ส่วนตัวที่เราได้บันทึกไว้ก่อนหน้านี้เพื่อให้โหนดของเราสามารถลงนามและช่วยในการทำบล็อกที่เสนอให้เสร็จสมบูรณ์ได้


```
e1-cli importprivkey <e1-priv-key>

e2-cli importprivkey <e2-priv-key>
```


การใช้คำสั่ง generate ควรแสดงข้อผิดพลาดในขณะนี้เนื่องจากไม่เป็นไปตามกฎ block signing ที่จำเป็นซึ่งขณะนี้ถูกบังคับใช้โดยโหนดของเรา


```
e1-cli -generate 1
```


ในการเสนอ block ใหม่ โหนดใด ๆ สามารถเรียกคำสั่ง getnewblockhex ได้ ซึ่งจะส่งคืนการแสดงผลในรูปแบบเลขฐานสิบหกของ block ใหม่ที่ต้องมีการลงนามก่อนที่จะได้รับการยอมรับโดยโหนดใด ๆ ในเครือข่ายของเรา


```
e1-cli getnewblockhex
```


โปรดจำไว้ว่าคำสั่งนี้เพียงแค่สร้างบล็อกที่เสนอ; มันไม่ได้ generate หนึ่ง


ในการยืนยันสิ่งนี้ เราสามารถเห็นได้ว่าในปัจจุบันไม่มีบล็อกใด ๆ ในบล็อกเชนของเรา


```
e1-cli getblockcount
```


หากเราลองส่งบล็อก hex โดยไม่เซ็นชื่อก่อน


```
e1-cli submitblock <block-hex>
```


เราได้รับข้อความแจ้งว่าหลักฐานบล็อกไม่ถูกต้อง เนื่องจากยังไม่ได้รับการลงนามโดย 2 ใน 2 ฝ่ายที่จำเป็น


ดังนั้นให้ e1 เซ็นชื่อบล็อกที่เสนอ


```
e1-cli signblock <block-hex>
```


ให้ e2 เซ็นชื่อใน hex.


```
e2-cli signblock <block-hex>
```


โปรดทราบว่า e2 ไม่ได้ลงนามในผลลัพธ์ที่สร้างจาก e1 ที่ลงนามในบล็อกที่เสนอ ทั้งสองลงนามในบล็อกที่เสนอในรูปแบบเลขฐานสิบหกโดยไม่ขึ้นกับผลลัพธ์ของกันและกัน


ตอนนี้เราจำเป็นต้องรวมลายเซ็นบล็อกของ e1 และ e2 ไม่ว่าจะเป็นโหนดใดก็สามารถทำการกระทำนี้ได้; สิ่งที่พวกเขาต้องการคือบล็อกที่ลงนามในรูปแบบเลขฐานสิบหกจากโหนดอื่น


```
e1-cli combineblocksigs <block-hex> '["<signed-hex-from-e1>", "<signed-hex-from-e2>"]'
```


คุณสามารถเห็นได้ว่า คำสั่ง combineblocksigs แสดงผลลัพธ์เป็นเลขฐานสิบหกของบล็อกที่ลงนามแล้ว รวมถึงสถานะ 'complete' ซึ่งบ่งบอกว่าเลขฐานสิบหกของบล็อกพร้อมที่จะถูกส่งแล้ว


ตอนนี้ โหนดใด ๆ ก็สามารถส่งบล็อกที่เสร็จสมบูรณ์ในรูปแบบเลขฐานสิบหกได้ เราจะให้ e1 ทำการส่ง


```
e1-cli submitblock <combined-signed-hex>
```


กำลังตรวจสอบว่าการส่งนั้นถูกต้องหรือไม่


```
e1-cli getblockcount

e2-cli getblockcount
```


คุณสามารถเห็นได้ว่า e1 และ e2 ทั้งสองได้ยอมรับบล็อกว่าเป็นบล็อกที่ถูกต้องและได้เพิ่มบล็อกนั้นไปยังปลายของสำเนาบล็อกเชนในเครื่องของพวกเขาแล้ว


เพื่อสรุปกระบวนการ ในส่วนนี้ เรามี:


- เสนอการบล็อก
- ให้แต่ละโหนดเซ็นชื่อ
- รวมลายเซ็นแล้ว
- ตรวจสอบแล้วว่าลายเซ็นถูกต้องและเป็นไปตามเกณฑ์ redeemscript ของเครือข่าย
- ส่งบล็อกแล้ว


โหนดแต่ละตัวบนเครือข่ายได้ตรวจสอบบล็อกและเพิ่มลงในสำเนาท้องถิ่นของบล็อกเชน


### บล็อกการลงนาม


แม้ว่ากระบวนการในตอนแรกจะดูซับซ้อน แต่ลำดับของ block signing ใน Elements จะเหมือนกันเสมอ และการตั้งค่าเริ่มต้นจำเป็นต้องทำเพียงครั้งเดียว:


1. การตั้งค่าเริ่มต้น (ทำครั้งเดียว)

2. มีการสร้างที่อยู่หลายลายเซ็นที่เรียกว่า `signblockscript` โดยใช้คีย์สาธารณะของ Federated Block Signers.

3. สคริปต์การแลกจากนี้ถูกใช้เพื่อเริ่มบล็อกเชนใหม่

4. การผลิตบล็อก (กำลังดำเนินการ)

5. บล็อกที่เสนอจะถูกสร้างและแลกเปลี่ยนเพื่อการลงนาม


เมื่อมีจำนวนผู้ลงนามถึงเกณฑ์ที่กำหนดแล้ว ข้อเสนอของบล็อกจะถูกผนวกและส่งไปยังเครือข่าย หากมันตรงตามเกณฑ์ของ `signblockscript` ของเชน โหนดจะยอมรับมันเป็นบล็อกที่ถูกต้อง

Elements เป็นแพลตฟอร์มบล็อกเชนแบบโอเพ่นซอร์สที่มีวัตถุประสงค์ทั่วไป ซึ่งสามารถ `pegged` กับบล็อกเชนที่มีอยู่แล้ว เช่น Bitcoin เมื่อ pegged กับบล็อกเชนอื่น Elements จะถือว่าทำงานเป็น `sidechain` Sidechains ช่วยให้สามารถโอนสินทรัพย์แบบสองทางจากเชนหนึ่งไปยังอีกเชนหนึ่ง การใช้งาน Elements เป็น sidechain ช่วยให้คุณสามารถหลีกเลี่ยงข้อจำกัดบางประการของ mainchain ในขณะที่ยังคงรักษาระดับความปลอดภัยที่ดีจากสินทรัพย์ที่ได้รับการรักษาความปลอดภัยบน mainchain


ในขณะที่ไซด์เชนรับรู้ถึง mainchain และประวัติการทำธุรกรรมของมัน mainchain ไม่รับรู้ถึงไซด์เชน และไม่จำเป็นต้องมีการรับรู้สำหรับการทำงานของมัน สิ่งนี้ทำให้ไซด์เชนสามารถนวัตกรรมได้โดยไม่มีข้อจำกัดหรือความล่าช้าที่เกี่ยวข้องกับข้อเสนอการปรับปรุงโปรโตคอล mainchain แทนที่จะพยายามเปลี่ยนแปลงมันโดยตรง การขยายโปรโตคอลหลักช่วยให้ mainchain เองยังคงปลอดภัยและมีความเชี่ยวชาญ สนับสนุนการทำงานที่ราบรื่นของไซด์เชน


โดยการขยายฟังก์ชันการทำงานของ Bitcoin และใช้ประโยชน์จากความปลอดภัยพื้นฐานของมัน sidechain ที่ใช้ Elements สามารถเสนอความสามารถใหม่ ๆ ที่ไม่เคยมีให้กับผู้ใช้ของ mainchain มาก่อน ตัวอย่างของ sidechain ที่ใช้ Elements ในการผลิตคือ Liquid Network


ในการเริ่มต้นบล็อกเชน Elements เป็นไซด์เชน เราจำเป็นต้องใช้พารามิเตอร์ federated peg script พารามิเตอร์นี้สามารถตั้งค่าในไฟล์ config ของโหนดหรือส่งผ่านในขณะเริ่มต้นระบบ


federated peg script กำหนดว่าสมาชิกคนใดในสหพันธ์ที่แข็งแกร่งมีอำนาจในการดำเนินการฟังก์ชัน peg-in และ peg-out ผู้ปฏิบัติงานเหล่านี้เรียกว่า `Watchmen` เพราะพวกเขาตรวจสอบ mainchain และ sidechain สำหรับธุรกรรม peg-in และ peg-out ที่ถูกต้องและดำเนินการกับพวกเขาหากพวกเขาถูกต้อง การ `peg-out` หมายถึงการย้ายสินทรัพย์ที่ถูกตรึงออกจาก sidechain และเข้าสู่ mainchain และการ `peg-in` หมายถึงการย้ายสินทรัพย์ที่ถูกตรึงเข้าสู่ sidechain จาก mainchain เมื่อเราพูดว่า `ย้ายเข้าสู่ sidechain` เราหมายถึงว่ากองทุนถูกล็อคในที่อยู่หลายลายเซ็นบน mainchain และจำนวนสินทรัพย์ที่สอดคล้องกันถูกสร้างขึ้นบน Elements sidechain เมื่อเราพูดว่า `ย้ายออกจาก sidechain` เราหมายถึงว่าสินทรัพย์ถูกทำลายบน Elements sidechain และจำนวนที่สอดคล้องกันถูกปล่อยจากกองทุนที่ถูกล็อคที่ถืออยู่บน mainchain การอนุญาตให้ดำเนินการฟังก์ชัน peg-in และ peg-out ต้องการให้ผู้ปฏิบัติงานพิสูจน์ความเป็นเจ้าของของคีย์สาธารณะที่ใช้ใน federated peg script ซึ่งทำได้โดยใช้คีย์ส่วนตัวที่สอดคล้องกัน


ในการสร้าง federated peg script เราจำเป็นต้องให้แต่ละโหนดของเรา generate คีย์สาธารณะก่อน นอกจากนี้เรายังต้องจัดเก็บคีย์ส่วนตัวที่เกี่ยวข้องไว้สำหรับการใช้งานในภายหลัง เนื่องจากเราจะต้องลบข้อมูลเชนที่มีอยู่และเริ่มต้นเชนใหม่โดยใช้ federated peg script นี่เป็นเพราะว่า federated peg script เป็นส่วนหนึ่งของกฎฉันทามติของไซด์เชน และไม่สามารถนำไปใช้กับบล็อกเชนที่มีอยู่ซึ่งไม่ได้เชื่อมโยงในภายหลังได้


ดังนั้นให้เรา generate ที่อยู่สำหรับแต่ละโหนดของเรา เก็บข้อมูลที่เกี่ยวข้องไว้ใช้ในภายหลัง และสร้าง federated peg script ซึ่งเราจะใช้ในการเริ่มต้น sidechain ของเราในภายหลัง


ก่อนอื่น เราต้องการให้แต่ละโหนดของเรา ซึ่งจะทำหน้าที่เป็น Watchmen ในเครือข่ายของเรา เพื่อ generate ที่อยู่ใหม่


```
e1-cli getnewaddress

e2-cli getnewaddress
```


จากนั้นเรายืนยันที่อยู่เพื่อรับคีย์สาธารณะ


```
e1-cli getaddressinfo <e1-address>

e2-cli getaddressinfo <e2-address>
```


จากนั้นดึงคีย์ส่วนตัวที่เกี่ยวข้องกับแต่ละที่อยู่ออกมา


```
e1-cli dumpprivkey <e1-address>

e2-cli dumpprivkey <e2-address>
```


จัดเก็บคีย์ส่วนตัวและคีย์สาธารณะเพื่อใช้งานในภายหลัง


ตอนนี้เราจำเป็นต้องล้างข้อมูลบล็อกเชนและข้อมูล wallet ที่มีอยู่ เนื่องจากเราจะเริ่มต้นเชนใหม่โดยใช้ federated peg script คุณสามารถทำได้เลยตอนนี้ อย่าลืมเริ่มต้น Bitcoin และ daemon ซึ่งเราจะต้องเชื่อมต่อด้วย


ตอนนี้เราสามารถเริ่มต้นเชนใหม่ด้วย federated peg script ที่สร้างขึ้นโดยใช้กุญแจสาธารณะที่เราเก็บไว้ก่อนหน้านี้ ตัวเลขที่เราใส่และที่ล้อมรอบกุญแจสาธารณะของเราจะกำหนดและจำกัดจำนวนกุญแจที่ใช้ รวมถึงการเป็นเจ้าของกุญแจที่ต้องพิสูจน์เพื่อเชื่อมต่อเข้าและออกจากไซด์เชนของเรา


```
e1-dae -fedpegscript=5221<e1-pubkey>21<e2-pubkey>52ae

e2-dae -fedpegscript=5221<e1-pubkey>21<e2-pubkey>52ae
```


ตอนนี้เราจะนำเข้าคีย์ส่วนตัวที่เราได้บันทึกไว้ก่อนหน้านี้ เพื่อให้โหนดของเราสามารถลงนามและทำการโอนสินทรัพย์ระหว่างเชนให้เสร็จสมบูรณ์ และตอบสนองความต้องการของ federated peg script ได้ในภายหลัง


```
e1-cli importprivkey <priv-key-1>

e2-cli importprivkey <priv-key-1>
```


ตอนนี้เราจำเป็นต้องทำให้บล็อกบางส่วนบนทั้งสองเชนเติบโตเต็มที่ การเติบโตเต็มที่ของบล็อกเป็นข้อกำหนดของกระบวนการ peg เนื่องจากมันป้องกันการจัดระเบียบบล็อกใหม่บนเชนหลัก ซึ่งอาจนำไปสู่[การเพิ่มขึ้นของอุปทาน](https://planb.academy/resources/glossary/inflation) pegged asset ภายในไซด์เชน


เพื่อให้ส่วนนี้มุ่งเน้นไปที่ federated peg เราจะบล็อก generate โดยไม่ใช้โมเดล block signing ที่ตรวจสอบในส่วนที่แล้วและกลับมาใช้คำสั่ง 'generate' เพื่อสร้างบล็อกใหม่


```
b-cli generate 101

e1-cli generate 1
```


เราไม่จำเป็นต้อง generate บล็อกสำหรับองค์ประกอบในขณะนี้ แต่ลอง generate หนึ่งอันดีกว่า มันเป็นการฝึกฝนที่ดีเพื่อหลีกเลี่ยงความไม่สอดคล้องที่อาจเกิดขึ้นได้


ตอนนี้เชนของเราพร้อมที่จะ peg in แล้ว ในการ peg-in เราจำเป็นต้องใช้ generate ซึ่งเป็นที่อยู่ชนิดพิเศษโดยใช้คำสั่ง getpeginaddress โปรดทราบว่าระยะเวลาระหว่างการสร้างที่อยู่ peg-in ด้วย getpeginaddress และการเคลมด้วย claimpegin ควรจะสั้นที่สุดเท่าที่จะเป็นไปได้ ที่อยู่ peg-in ไม่ทนทานในระยะยาวและไม่ควรนำกลับมาใช้ใหม่


```
e1-cli getpeginaddress
```


คุณสามารถเห็นได้ว่าคำสั่งนี้สร้างที่อยู่ mainchain ใหม่ รวมถึงสคริปต์ใหม่ที่ต้องปฏิบัติตามเพื่อเรียกร้องเงินทุน peg-in ที่อยู่ mainchain เป็นที่อยู่ `pay to script hash` ที่จะถูกใช้โดยผู้ทำหน้าที่เป็น Watchmen ภายในเครือข่าย Elements


เช่นเดียวกับ getnewaddress, getpeginaddress เพิ่มความลับใหม่ให้กับโหนดที่เรียกของ wallet ดังนั้นจึงเป็นสิ่งสำคัญที่จะต้องพิจารณาการสำรองไฟล์ wallet ในกระบวนการจัดการโหนดของคุณ


เราจะส่ง Bitcoin บางส่วนจากเชนหลักไปยังเชนด้านข้าง ขณะนี้การทดสอบถอยหลัง mainchain ของเรา wallet มีเงินทุนอยู่แล้ว


```
b-cli getwalletinfo
```


เราสามารถเห็นว่า wallet ถือ 50 Bitcoin เราจะส่งหนึ่งบิตคอยน์จากเชนหลักไปยังเชนข้าง เราจำเป็นต้องส่งเงินไปยังที่อยู่ mainchain ที่โหนดของเราสร้างขึ้นก่อนหน้านี้


```
b-cli sendtoaddress <e1-pegin-address>
```


เราจำเป็นต้องเก็บบันทึกของ transaction ID นี้ไว้ เนื่องจากจะต้องใช้เป็นหลักฐานการระดมทุนในภายหลัง


ตอนนี้เราสามารถเห็นได้ว่า mainchain wallet ยอดคงเหลือลดลงตามจำนวนที่เราส่งไป พร้อมกับจำนวนเล็กน้อยเพิ่มเติมเพื่อครอบคลุมค่าธรรมเนียมการทำธุรกรรม


```
b-cli getwalletinfo
```


เราจำเป็นต้องดำเนินการธุรกรรมใหม่อีกครั้ง


```
b-cli generate 101
```


ในการให้โหนด Elements ของเรารับสิทธิ์ในเงินที่ฝากเข้า เราจำเป็นต้องได้รับ `proof` ว่าธุรกรรมการฝากเข้านั้นเสร็จสมบูรณ์แล้ว หลักฐานทางการเข้ารหัสใช้การระดมทุน transaction ID เพื่อคำนวณเส้นทาง Merkle และพิสูจน์ว่าธุรกรรมนั้นอยู่ในบล็อกที่ได้รับการยืนยันแล้ว


```
b-cli gettxoutproof '["<tx-id>"]'
```


เรายังต้องการข้อมูลธุรกรรมดิบด้วย


```
b-cli getrawtransaction <tx-id>
```


ด้วยหลักฐานและข้อมูลดิบสำหรับธุรกรรม peg-in โหนด elements ของเราสามารถอ้างสิทธิ์ peg-in โดยใช้ธุรกรรมดิบและหลักฐานธุรกรรมได้แล้ว


```
e1-cli claimpegin <raw> <proof>
```


โปรดทราบว่ามีอาร์กิวเมนต์ที่สามที่เป็นทางเลือกซึ่งเราสามารถให้กับ claimpegin ได้ พารามิเตอร์ที่สามนี้สามารถใช้เพื่อระบุที่อยู่ของไซด์เชนที่เงินที่อ้างสิทธิ์ควรถูกส่งไป ในตัวอย่างของเราไม่จำเป็นต้องใช้ เนื่องจากเราเรียกคำสั่งจากโหนดเดียวกันที่เป็นเจ้าของที่อยู่ที่เงินที่อ้างสิทธิ์กำลังจะไป


กำลังตรวจสอบยอดคงเหลือของ e1.


```
e1-cli getwalletinfo
```


สร้างบล็อกเพื่อยืนยันการอ้างสิทธิ์


```
e1-cli generate 1
```


กำลังตรวจสอบยอดคงเหลือของ e1.


```
e1-cli getwalletinfo
```


เราสามารถเห็นได้ว่าการเชื่อมต่อได้ถูกอ้างสิทธิ์เรียบร้อยแล้ว


ในการถอนเงินออก ขั้นตอนจะคล้ายกัน โดยจะมีการสร้างที่อยู่ ส่งเงินไปยังที่อยู่นั้น และเงินจะถูกปล่อยออกหากธุรกรรมนั้นถูกต้อง เราจะไม่ครอบคลุมกระบวนการถอนเงินออกทั้งหมดเนื่องจากเกี่ยวข้องกับการทำงานบน mainchain ซึ่งเกินขอบเขตของหลักสูตรนี้ ขั้นตอนในแง่ของเหตุการณ์ Elements เกี่ยวข้องกับการสร้างที่อยู่บน mainchain


```
b-cli getnewaddress
```


มีการส่งเงินไปยังที่อยู่ mainchain จากโหนด Elements โดยใช้คำสั่ง sendtomainchain


```
e1-cli sendtomainchain <new-address> 1
```


กำลังสร้างบล็อกเพื่อยืนยันการทำธุรกรรม


```
e1-cli generate 1
```

```
e1-cli getwalletinfo
```


และเห็นว่ายอดคงเหลือลดลง


ในส่วนนี้ เราได้เห็นวิธีการ:


- สร้าง federated peg script
- เริ่มต้นเชนใหม่ที่ใช้สคริปต์เป็นกฎพารามิเตอร์ฉันทามติของเครือข่าย
- ส่งเงินจาก mainchain ไปยังไซด์เชน
- อ้างสิทธิ์เงินทุนภายใน Elements sidechain.
- ทำความเข้าใจว่าการส่งเงินกลับไปยัง mainchain เริ่มต้นอย่างไร


### FederatedPegScript



ในการเปิดใช้งาน Elements ให้ทำงานเป็นไซด์เชน [บล็อกกำเนิด](https://planb.academy/resources/glossary/genesis-block)ในบล็อกเชนของมันจะต้องถูกสร้างขึ้นพร้อมกับ `fedpegscript` ซึ่งสามารถทำได้โดยการส่งพารามิเตอร์ `fedpegscript` ในระหว่างการเริ่มต้นโหนด สคริปต์นี้จะกลายเป็นส่วนหนึ่งของกฎฉันทามติของบล็อกเชน Elements ทำให้สามารถตรวจสอบและดำเนินการคำขอ peg-in และ peg-out ได้


`fedpegscript` ประกอบด้วยคีย์สาธารณะที่ควบคุมโดยผู้ที่ได้รับอนุญาตให้ดำเนินการ peg การกระทำต่างๆ ด้านล่างนี้แสดงรูปแบบตัวอย่างของ fedpegscript แบบ multisignature 2-of-2:


```
fedpegscript=5221<public key 1>21<public key 2>52ae
```


หมายเหตุ: อักขระที่อยู่นอกคีย์สาธารณะเป็นตัวคั่นที่ระบุคีย์สาธารณะและข้อกำหนด `n of m` ตัวอย่างเช่น แม่แบบสำหรับ fedpegscript แบบ 1-of-1 จะเป็น '5121<pub key 1>51ae'


### Peg-in


ก่อนที่การ peg-in จะได้รับการยอมรับโดย Elements sidechain จะต้องมีการยืนยันที่เพียงพอบน mainchain นี่เป็นสิ่งจำเป็นเพื่อหลีกเลี่ยงการเพิ่มขึ้นของอุปทาน pegged asset บน Elements sidechain ซึ่งอาจเกิดจากการจัดระเบียบใหม่ของ mainchain


การจัดระเบียบใหม่แบบสั้น ๆ ของปลายบล็อกเชน Bitcoin เป็นสิ่งที่คาดหวังได้ในฐานะส่วนหนึ่งของการดำเนินงานปกติ เนื่องจากกลไกฉันทามติ Proof-of-Work (PoW) ดังนั้น Elements จะยอมรับการ peg-in ว่าถูกต้องก็ต่อเมื่อมีความลึกเพียงพอภายในบล็อกเชน Bitcoin สิ่งนี้ทำหน้าที่ป้องกันไม่ให้ Elements ยอมรับการ peg-in เดียวกันมากกว่าหนึ่งครั้ง


### เพ็ก-เอาท์


การถอนเงินเกิดขึ้นเมื่อโหนด Elements เรียกคำสั่ง `sendtomainchain` ซึ่งรับที่อยู่ mainchain (ปลายทางการถอนเงิน) และจำนวนของ pegged asset ที่จะ `ถอน` เป็นข้อมูลนำเข้า สิ่งนี้สร้างธุรกรรมการถอนเงินบนไซด์เชน เมื่อ Functionaries ที่ทำหน้าที่เป็น Watchmen ตรวจพบว่าธุรกรรมการถอนเงินได้รับการยืนยันบนไซด์เชนแล้ว พวกเขาจะดูแลการปล่อยสินทรัพย์บน mainchain ไปยังปลายทางการถอนเงิน ดังที่เราได้เรียนรู้ในส่วนก่อนหน้าของหลักสูตร


## Elements เป็น Blockchain แบบสแตนด์อโลน


<chapterId>50dff39b-2702-47d7-9c15-0b54b845e99f</chapterId>


:::video id=4955306b-4be3-429c-9d30-068f7644ea73:::

จนถึงตอนนี้ เราได้ดูวิธีการรัน Elements ในฐานะไซด์เชน อย่างไรก็ตาม มันยังสามารถทำงานเป็นโซลูชันบล็อกเชนแบบสแตนด์อโลนที่มีสินทรัพย์เนทีฟเริ่มต้นของตัวเองได้อีกด้วย ในการตั้งค่านี้ บล็อกเชน Elements ยังคงรักษาคุณสมบัติทั้งหมดของการใช้งานไซด์เชน เช่น Confidential Transactions และ Issued Assets แต่ไม่จำเป็นต้องใช้ peg-in หรือ peg-out เพื่อเพิ่มหรือลบจำนวน default asset ออกจากการหมุนเวียน


ในส่วนนี้ เราจะ:


เริ่มต้นบล็อกเชน Elements ใหม่ด้วย default asset ที่ชื่อว่า `newasset`.


ระบุ `newasset` จำนวน 1,000,000 รายการที่จะสร้างขึ้นพร้อมกับโทเค็นการออกใหม่ 2 รายการสำหรับมัน


อ้างสิทธิ์เหรียญ `newasset` ทั้งหมดของ anyone-can-spend


อ้างสิทธิ์โทเค็นการออกใหม่ anyone-can-spend ทั้งหมดสำหรับ 'newasset'


ส่งสินทรัพย์และ reissuance token ไปยัง wallet ของโหนดอื่น


ออก 'newasset' เพิ่มเติมจากทั้งสองโหนด


ในการเริ่มต้นเครือข่าย Elements เป็นบล็อกเชนแบบสแตนด์อโลน แต่ละโหนดจะต้องเริ่มต้นด้วยพารามิเตอร์พื้นฐานบางอย่าง ซึ่งใช้เพื่อสั่งให้โหนดไม่พยายามตรวจสอบ peg-ins จากบล็อกเชนอื่น ระบุชื่อของเครือข่าย default asset และระบุจำนวนของ default asset และ reissuance token ที่เกี่ยวข้องที่จะสร้างขึ้น


เราจะเริ่มต้นเชนใหม่โดยใช้พารามิเตอร์เหล่านี้บนโหนด Elements สองโหนดที่เชื่อมต่อกัน เราจะตั้งชื่อ default asset ว่า `newasset` ออกหนึ่งล้านหน่วย และสร้างโทเค็นการออกใหม่ `newasset` สองโทเค็น


```
e1-dae -validatepegin=0 -defaultpeggedassetname=newasset -initialfreecoins=100000000000000 -initialreissuancetokens=200000000

e2-dae -validatepegin=0 -defaultpeggedassetname=newasset -initialfreecoins=100000000000000 -initialreissuancetokens=200000000
```


โปรดทราบว่าจำนวนเงินที่ใช้ที่นี่อยู่ในหน่วยที่เล็กที่สุดที่เครือข่ายสามารถยอมรับได้ ดังนั้นโทเค็นการออกใหม่สองร้อยล้านโทเค็นจึงเท่ากับโทเค็นทั้งหมดสองโทเค็น เช่นเดียวกันกับหน่วยของเหรียญฟรีเริ่มต้น


ตรวจสอบยอดคงเหลือ wallet ปัจจุบันของโหนดของเรา


```
e1-cli getwalletinfo

e2-cli getwalletinfo
```


เราสามารถเห็นได้ว่าการเริ่มต้นทำงานอย่างถูกต้องแล้ว


เนื่องจากการออกสินทรัพย์เริ่มต้นถูกสร้างขึ้นเป็น `anyone can spend` เราจะให้ e1 อ้างสิทธิ์ทั้งหมดเพื่อที่เราจะสามารถลบการเข้าถึงของ e2 ได้


```
e1-cli getnewaddress

e1-cli sendtoaddress <e1-address> 1000000 "" "" true
```


โปรดทราบว่าเราไม่จำเป็นต้องระบุ 'newasset' เป็นสินทรัพย์ที่จะส่ง เนื่องจากมันเป็น default asset อยู่แล้ว และดังนั้นจึงเป็น default asset ที่ใช้ในการชำระค่าธรรมเนียมเครือข่ายด้วยเช่นกัน


ภายใน Elements คุณสามารถส่งสินทรัพย์หลายประเภทไปยังที่อยู่เดียวกัน ทำให้เราสามารถใช้ที่อยู่ที่เพิ่งสร้างขึ้นเพื่อรับ default asset และใช้เป็นที่อยู่ปลายทางสำหรับโทเค็นที่ออกใหม่ได้


```
e1-cli sendtoaddress <e1-address> 1 "" "" false false 1 UNSET false <reissuance-token-id>
```


ยืนยันการทำธุรกรรม


```
e1-cli generate 101
```


เราจะตรวจสอบว่า e1 เป็นผู้ถือครองทรัพย์สินเพียงผู้เดียวและเป็น reissuance token ของมัน


```
e1-cli getwalletinfo

e2-cli getwalletinfo
```


ซึ่งเราสามารถเห็นได้ว่าเป็นกรณีนี้จริงๆ


ตอนนี้เราจะส่ง 'newasset' บางส่วนไปยัง e2 ซึ่งปัจจุบันมียอดคงเหลือเป็นศูนย์


```
e2-cli getnewaddress

e1-cli sendtoaddress <e2-address> 500 "" "" false
```


โปรดทราบว่าเราไม่จำเป็นต้องระบุประเภทของสินทรัพย์ที่จะส่ง เนื่องจาก `newasset` ได้ถูกสร้างขึ้นเป็น default asset ของเครือข่าย


ลองส่งโทเค็นการออกใหม่บางส่วนสำหรับ `newasset` ไปที่ e2 ด้วย


```
e1-cli sendtoaddress <e2-address> 1 "" "" false false 1 UNSET false <reissuance-token-id>
```


ยืนยันการทำธุรกรรม


```
e1-cli generate 101
```


เราสามารถตรวจสอบได้ว่ากระเป๋าเงินได้รับการอัปเดตตามนั้นแล้ว


```
e1-cli getwalletinfo

e2-cli getwalletinfo
```


ตอนนี้เราจะออกสินทรัพย์เริ่มต้นบางส่วนจาก e1 ใหม่ โปรดทราบว่าความสามารถในการทำเช่นนี้ถูกเปิดใช้งานโดยพารามิเตอร์เริ่มต้นของโทเค็นการออกใหม่ครั้งแรก ซึ่งหากละเว้นหรือกำหนดเป็นศูนย์ จะส่งผลให้เกิด default asset ที่ไม่สามารถออกใหม่ได้ในภายหลัง


```
e1-cli reissueasset newasset 100
```


เราใช้ป้ายชื่อ `newasset` แทนการต้องระบุค่า hex ID ได้เพราะว่าโซ่ Elements มักจะติดป้าย default asset เสมอ


กำลังตรวจสอบว่าการออกใหม่ของ default asset ทำงานหรือไม่:


```
e1-cli generate 101

e1-cli getwalletinfo
```


เราจะพิสูจน์ในขณะนี้ว่าเนื่องจาก e2 ถือโทเค็นการออกใหม่บางส่วนของ `newasset` มันจึงสามารถออกใหม่ default asset ได้เช่นกัน


```
e2-cli reissueasset newasset 100
```


กำลังตรวจสอบว่าการออกใหม่ของ default asset โดย e2 สำเร็จหรือไม่


```
e2-cli generate 101

e2-cli getwalletinfo
```


ในส่วนนี้ เราได้ตั้งค่า Elements เป็นบล็อกเชนแบบสแตนด์อโลนและยืนยันว่าฟังก์ชันพื้นฐานของมันทำงานตามที่คาดไว้


เราใช้พารามิเตอร์เริ่มต้นเพื่อ:


เริ่มต้นบล็อกเชน Elements ใหม่ด้วย default asset ที่ชื่อว่า 'newasset'


ระบุจำนวนของ default asset เพื่อสร้างการเริ่มต้น on chain


สร้างโทเค็นการออกใหม่สำหรับ default asset และออกใหม่เพิ่มเติมของ default asset จากทั้งสองโหนด


บนเครือข่ายบล็อกเชนแบบสแตนด์อโลน Elements ของเรา การดำเนินการธุรกรรมอื่นๆ ทั้งหมดจะดำเนินการในลักษณะเดียวกับตัวอย่างที่ครอบคลุมในส่วนหลักของหลักสูตร แต่จะใช้ 'newasset' แทน `bitcoin` เป็นสินทรัพย์เริ่มต้นและค่าธรรมเนียม


### พารามิเตอร์การเริ่มต้นโหนดและการเริ่มต้นเชน


ในการสั่งให้โหนด Elements ทำงานเป็นบล็อกเชนแบบสแตนด์อโลน จำเป็นต้องใช้พารามิเตอร์หลายตัวร่วมกัน เราจะมาดูแต่ละตัวในตอนนี้และค้นหาว่าพวกมันทำอะไรบ้าง


#### `validatepegin=0`

ในฐานะที่เป็นบล็อกเชนแบบสแตนด์อโลน ไม่จำเป็นต้องตรวจสอบธุรกรรม peg-in หรือ peg-out เราจำเป็นต้องปิดการตรวจสอบเหล่านั้น ด้วยการตั้งค่านี้ คุณไม่จำเป็นต้องเรียกใช้ซอฟต์แวร์ไคลเอนต์ Bitcoin หรือเก็บสำเนาของบล็อกเชน Bitcoin เนื่องจากเครือข่าย Elements จะทำงานอย่างอิสระ


#### `defaultpeggedassetname`

สิ่งนี้ช่วยให้คุณระบุชื่อของ default asset ที่สร้างขึ้นเมื่อบล็อกเชนถูกเริ่มต้น


#### `initialfreecoins`

จำนวน (ในหน่วยเทียบเท่าของ Bitcoin ของ Satoshi) ของ default asset ที่จะสร้างขึ้น


#### `initialreissuancetokens`

จำนวน (ในหน่วยเทียบเท่าของ Bitcoin ของหน่วย Satoshi) ของโทเค็นการออกใหม่สำหรับ default asset เพื่อสร้างขึ้น หากไม่มีสิ่งนี้ จะเป็นไปไม่ได้ที่จะสร้าง default asset เพิ่มเติม หากคุณไม่ต้องการให้สามารถสร้าง default asset เพิ่มเติมได้ สามารถตั้งค่านี้เป็นศูนย์หรือเว้นว่างไว้ได้


โดยใช้พารามิเตอร์เหล่านี้ วิธีทั่วไปในการเริ่มต้นโหนดจะมีลักษณะดังนี้:


```
e1-dae -validatepegin=0 -defaultpeggedassetname=newasset -initialfreecoins=100000000000000 -initialreissuancetokens=200000000
```


### การดำเนินการพื้นฐาน


พารามิเตอร์ `defaultpeggedassetname` ใช้ในการตั้งชื่อให้กับ default asset หากไม่มีการตั้งค่านี้ default asset จะถูกตั้งชื่อเป็น `bitcoin` โดยอัตโนมัติ ในส่วนก่อนหน้านี้ เมื่อเราส่งสินทรัพย์ที่เราออกเองไปยังโหนดอื่น เราต้องระบุรหัส hex ของสินทรัพย์หรือชื่อสินทรัพย์ที่ตั้งไว้ในเครื่องเพื่อแจ้ง Elements เกี่ยวกับสินทรัพย์ที่ถูกส่ง เนื่องจาก `defaultpeggedassetname` ใช้กับทุกโหนด เราไม่จำเป็นต้องระบุชื่อเมื่อส่ง แม้ว่าชื่อจะไม่ใช่ `bitcoin` ทุกฟังก์ชันที่เคยส่ง `bitcoin` โดยค่าเริ่มต้นจะส่งสิ่งที่คุณเลือกตั้งชื่อให้กับ default asset แทน


ดังนั้นการส่งสินทรัพย์เริ่มต้นใหม่จำนวน 10 รายการไปยังที่อยู่หนึ่งนั้นง่ายเหมือน:


```
e1-cli sendtoaddress <destination address> 10 "" "" true
```


หากคุณได้ให้ค่า `initialreissuancetokens` กับโหนดที่มีค่ามากกว่าศูนย์ คุณก็จะสามารถออกใหม่ของ default asset ได้มากขึ้น ซึ่งสิ่งนี้จะไม่สามารถทำได้หากคุณรัน Elements เป็นไซด์เชน


ในการทำเช่นนี้ โหนดใด ๆ ที่ถือจำนวนของ token ที่เกี่ยวข้องกับ default asset เพียงแค่ต้องออกคำสั่งในรูปแบบ:


```
e1-cli reissueasset <default asset name> <amount>
```


โดยการใช้พารามิเตอร์ข้างต้น คุณสามารถดำเนินการ Elements เป็นบล็อกเชนแบบสแตนด์อโลนที่มี default asset ของตัวเอง แยกออกจาก Bitcoin และบล็อกเชนอื่น ๆ
