# ساخت بر روی Liquid و Elements


ویژگی‌ها و قابلیت‌های پیشرفته Liquid و Elements را کشف کنید و بیاموزید که چگونه از این ابزارها به طور مؤثر برای بهبود پروژه‌های توسعه خود استفاده کنید. این آموزش یک پایه نظری و عملی کامل ارائه می‌دهد و به شما امکان می‌دهد ویژگی‌هایی مانند Confidential Transactions، Issued Assets و Federated block signing را به خوبی فرا بگیرید.


Liquid، بر اساس چارچوب Elements، برای بهبود حریم خصوصی، مقیاس‌پذیری و عملکرد در راه‌حل‌های مالی و فنی طراحی شده است. در این دوره، شما تجربه عملی در صدور و مدیریت دارایی، Federated 2-Way Peg، و استفاده از ابزارهایی مانند elementsd و elements-cli کسب خواهید کرد که به شما امکان می‌دهد راه‌حل‌های نوآورانه‌ای متناسب با نیازهای خود ایجاد کنید.


این دوره برای توسعه‌دهندگانی با تمامی سطوح تجربه طراحی شده است. کاربران مبتدی و متوسط توضیحات قابل فهم و مثال‌های عملی را خواهند یافت، در حالی که کاربران پیشرفته می‌توانند به جزئیات فنی و ویژگی‌های کمتر شناخته شده Liquid و Elements عمیق‌تر بپردازند.


به ما بپیوندید تا مهارت‌های خود را ارتقا دهید، پتانسیل کامل Liquid و Elements را باز کنید و ابزارهای تاثیرگذار برای آینده نوآوری Liquid ایجاد کنید.

## بررسی اجمالی Elements


<chapterId>7a7f2712-5300-4a6d-b1ed-05eab731bc35</chapterId>


:::video id=eae666b4-eddc-4e00-adea-2a5f94396044:::


Elements یک پلتفرم متن‌باز، سازگار با Sidechain و [Blockchain](https://planb.academy/resources/glossary/blockchain) است که دسترسی به ویژگی‌های قدرتمندی که توسط اعضای جامعه توسعه یافته‌اند، مانند Confidential Transactions و Issued Assets را فراهم می‌کند.


Elements در اصل، یک پروتکل است که امکان ایجاد [اجماع](https://planb.academy/resources/glossary/consensus) در مورد تاریخچه [تراکنش‌ها](https://planb.academy/resources/glossary/transaction-tx) و قوانین حاکم بر انتقال و ایجاد دارایی‌های ذخیره‌شده در یک Blockchain Ledger توزیع‌شده را فراهم می‌کند.


اطلاعات بیشتر در مورد Elements به راحتی در وب‌سایت پروژه Elements (https://elementsproject.org/)، وبلاگ رسمی Liquid (https://blog.Liquid.net/) و پورتال توسعه‌دهنده (https://Liquid.net/devs) قابل یافتن است.


### Elements


راه‌اندازی شده در سال ۲۰۱۵، Elements هزینه‌های توسعه و تحقیق داخلی را کاهش می‌دهد و از جدیدترین فناوری Blockchain بهره می‌برد و بسیاری از موارد استفاده جدید برای پیاده‌سازی را باز می‌کند. یک Blockchain مبتنی بر Elements می‌تواند به عنوان یک Blockchain مستقل عمل کند یا به دیگری متصل شده و به عنوان یک Sidechain اجرا شود. اجرای Elements به عنوان یک Sidechain امکان انتقال قابل تأیید دارایی‌ها بین بلاکچین‌های مختلف را فراهم می‌کند.


بر اساس و گسترش کدبیس Bitcoin، این امکان را به توسعه‌دهندگان آشنا با API bitcoind می‌دهد تا به سرعت و با هزینه کم بلاکچین‌های کاربردی ایجاد کرده و پروژه‌های اثبات مفهوم را آزمایش کنند. ساخته شدن بر روی کدبیس Bitcoin همچنین به Elements اجازه می‌دهد تا به عنوان یک بستر آزمایشی برای تغییرات در پروتکل Bitcoin عمل کند.


برخی از ویژگی‌های اصلی Elements در ادامه فهرست شده‌اند.


#### Confidential Transactions


به طور پیش‌فرض، همه آدرس‌ها در Elements با استفاده از Confidential Transactions به blinded تبدیل می‌شوند. کورسازی فرآیندی است که در آن مقدار و نوع دارایی منتقل‌شده به‌صورت رمزنگاری‌شده از همه پنهان می‌شود، به‌جز شرکت‌کنندگان و کسانی که آن‌ها تصمیم می‌گیرند Blinding key را به آن‌ها نشان دهند.


#### Issued Assets


Issued Assets بر روی Elements اجازه می‌دهد که انواع مختلفی از دارایی‌ها بین شرکت‌کنندگان شبکه صادر و منتقل شوند. یک دارایی صادر شده همچنین از Confidential Transactions بهره‌مند می‌شود و می‌تواند توسط هر کسی که reissuance token مربوطه را در اختیار دارد، مجدداً صادر یا نابود شود.


#### Federated 2-Way Peg


Elements یک پلتفرم عمومی Blockchain است که می‌تواند به یک Blockchain موجود (مانند Bitcoin) "متصل" شود تا انتقال دو طرفه دارایی‌ها از یک زنجیره به زنجیره دیگر را امکان‌پذیر کند. پیاده‌سازی Elements به عنوان یک Sidechain به شما اجازه می‌دهد تا برخی از ویژگی‌های ذاتی زنجیره اصلی را دور بزنید، در حالی که همچنان درجه خوبی از امنیت ارائه شده توسط دارایی‌های امن شده بر روی زنجیره اصلی را حفظ می‌کنید.


#### بلوک‌های امضا شده


Elements از Strong Federation امضاکنندگان، به نام Block Signers، استفاده می‌کند که [بلوک](https://planb.academy/resources/glossary/block)‌ها را به صورت قابل اعتماد و به‌موقع امضا و ایجاد می‌کنند. این امر تأخیر تراکنش فرآیند PoW Mining را که به دلیل توزیع پواسون تصادفی دارای تغییرات زیاد در زمان بلوک است، از بین می‌برد. فرآیند Federated block signing ایجاد بلوک قابل اعتماد را بدون نیاز به اعتماد به شخص ثالث یا Mining مبتنی بر `الگوریتم` محاسباتی محقق می‌کند.


Elements تمام این ویژگی‌ها را بر روی کدبیس اصلی Bitcoin اضافه می‌کند، توانایی پروتکل mainchain را گسترش می‌دهد و موارد استفاده تجاری جدیدی را زمانی که به عنوان Sidechain یا به عنوان یک راه‌حل مستقل Blockchain مستقر می‌شود، امکان‌پذیر می‌سازد.

## نحوه عملکرد Elements


<chapterId>05d88877-58b0-455b-9ae6-a72d19070525</chapterId>


:::video id=7c8c7981-11e5-47a2-a257-ef998f4892f5:::


Elements یک راه‌حل فنی برای مشکلاتی که کاربران Blockchain روزانه با آن‌ها مواجه هستند ارائه می‌دهد؛ تأخیر در تراکنش، کمبود حریم خصوصی، و خطر برای [قابلیت تعویض‌پذیری](https://planb.academy/resources/glossary/fungibility).


Elements این مشکلات را از طریق استفاده از block signing و Confidential Transactions فدرال حل می‌کند.


برخلاف شبکه Bitcoin، فرآیند block signing درون Elements به امضاهای چندجانبه عضویت پویا (DMMS) و Proof of Work (PoW) وابسته نیست. در عوض، Elements از Strong Federation امضاکنندگان، به نام امضاکنندگان بلوک، استفاده می‌کند که می‌توانند بلوک‌ها را به صورت قابل اعتماد و به موقع امضا و ایجاد کنند. این امر تأخیر تراکنش فرآیند PoW Mining را که به دلیل توزیع پواسون تصادفی به تغییرات زیاد در زمان بلوک دچار است، از بین می‌برد. فرآیند فدرال block signing ایجاد بلوک قابل اعتماد را بدون نیاز به اعتماد به شخص ثالث محقق می‌کند.


Elements می‌تواند به عنوان یک Sidechain برای یک Blockchain دیگر، مانند Bitcoin، یا به عنوان یک Blockchain مستقل بدون وابستگی به شبکه‌های دیگر عمل کند.


هنگامی که به عنوان Sidechain استفاده می‌شود، Strong Federation همچنین شامل اعضایی است که انتقال امن و کنترل‌شده دارایی‌ها بین یک زنجیره اصلی و یک Elements Sidechain را امکان‌پذیر می‌سازند. انتقال کنترل‌شده دارایی‌ها Federated 2-Way Peg نامیده می‌شود و اعضایی که نقش انتقال دارایی را ایفا می‌کنند watchmen نامیده می‌شوند.


فرآیندهای درگیر در اجرای یک شبکه Elements و نقش‌های شرکت‌کنندگان در شبکه برای درک چگونگی عملکرد Elements مهم هستند.


چه به‌عنوان Sidechain پیاده‌سازی شود یا به‌صورت مستقل به‌عنوان Blockchain، Elements از فدراسیون‌های قوی امضاکنندگان بلوک برای تولید بلوک‌ها استفاده می‌کند.


### فدراسیون‌های قوی


Elements از یک مدل اجماع که ابتدا توسط Blockstream پیشنهاد شد، به نام Strong Federations استفاده می‌کند. یک Strong Federation به Proof of Work (PoW) نیاز ندارد و به جای آن به اقدامات جمعی گروهی از شرکت‌کنندگان که به یکدیگر اعتماد ندارند، به نام Functionaries، متکی است.


نقش‌هایی که یک کارمند می‌تواند در یک Strong Federation ایفا کند عبارتند از: امضاکنندگان بلوک و watchmen. امضاکنندگان بلوک در صورتی که Elements را در حالت Sidechain یا حالت مستقل Blockchain اجرا کنید، مورد نیاز هستند، در حالی که watchmen فقط در یک تنظیمات Sidechain مورد نیاز است.


اقداماتی که یک عضو Strong Federation می‌تواند انجام دهد بین دو نقش متمایز تقسیم می‌شود تا امنیت را افزایش داده و خسارتی که یک مهاجم می‌تواند وارد کند را محدود کند.


هنگامی که ترکیب می‌شوند، نقش‌های این شرکت‌کنندگان به Elements اجازه می‌دهد تا هم ایجاد سریع بلوک (تأیید تراکنش سریع‌تر و نهایی) و هم دارایی‌های قابل انتقال و تضمین‌شده (دارایی‌های متصل به طور مستقیم قابل پیوند به Blockchain دیگر) را ارائه دهد.


می‌توانید [وایت‌پیپر](https://planb.academy/resources/glossary/white-paper) Strong Federations را اینجا بخوانید: https://blockstream.com/strong-federations.pdf


### مسدود کردن امضاکنندگان


هنگامی که هر کسی که بخشی از یک گروه پویا از امضاکنندگان بلوک است زنجیره را با نشان دادن مصرف Proof of Work گسترش می‌دهد، یک Blockchain مانند Bitcoin گسترش می‌یابد. ماهیت پویا این مجموعه مسائل تأخیر ذاتی در چنین سیستم‌هایی را معرفی می‌کند.


با استفاده از یک مجموعه امضاکننده ثابت، مدل فدرال مجموعه پویا را با یک مجموعه شناخته‌شده و طرح چند امضایی جایگزین می‌کند. کاهش تعداد شرکت‌کنندگان مورد نیاز برای گسترش Blockchain سرعت و مقیاس‌پذیری سیستم را افزایش می‌دهد، در حالی که اعتبارسنجی توسط همه طرف‌ها یکپارچگی تاریخچه تراکنش را تضمین می‌کند.


فدراسیون block signing شامل چندین مرحله است:



- مرحله 1 - امضاکنندگان بلوک به صورت نوبتی بلوک‌های پیشنهادی را به سایر امضاکنندگان بلوک شرکت‌کننده پیشنهاد می‌دهند.



- مرحله 2 - هر block signer قصد خود را با پیش‌تعهد به امضای بلوک کاندیدای داده شده اعلام می‌کند.



- مرحله 3 - اگر آستانه تعیین‌شده برای پیش‌Commitment برآورده شود، هر block signer بلوک را امضا می‌کند.



- مرحله 4 - اگر آستانه امضا (که ممکن است با مرحله 3 متفاوت باشد) برآورده شود، بلوک پذیرفته شده و به شبکه ارسال می‌شود. Strong Federation به اجماع در مورد آخرین بلوک تراکنش‌ها رسیده است.



- مرحله 5 - بلوک بعدی سپس توسط block signer بعدی در سیستم چرخشی پیشنهاد می‌شود و فرآیند تکرار می‌شود.


از آنجا که تولید بلاک در Strong Federation احتمالی نیست و بر اساس مجموعه‌ای ثابت از امضاکنندگان انجام می‌شود، هرگز تحت تأثیر سازماندهی مجدد چند بلاکی قرار نمی‌گیرد. این امر باعث کاهش قابل توجه زمان انتظار مرتبط با تأیید تراکنش‌ها می‌شود. همچنین انگیزه استخراج برای سود (یعنی پاداش بلاک) را حذف کرده و آن را با انگیزه‌ای برای مشارکت سازنده در شبکه‌ای که همه شرکت‌کنندگان هدف مشترکی دارند جایگزین می‌کند؛ اطمینان از ادامه عملکرد شبکه به گونه‌ای که برای همه مفید باشد. این کار را بدون معرفی یک نقطه شکست واحد یا نیاز به اعتماد بیشتر انجام می‌دهد.


### Elements به عنوان Sidechain - watchmen و Federated 2-Way Peg


هنگامی که به عنوان Sidechain اجرا می‌شود، برخی از اعضای Strong Federation نقش اضافی برای ایفا کردن دارند، یعنی watchmen. watchmen مسئول انتقال دارایی‌ها به داخل و خارج از یک Elements Sidechain هستند، فرآیندهایی که به عنوان `Peg-In` و `Peg-Out` شناخته می‌شوند.


برای اینکه یک Sidechain به صورت قابل اعتماد عمل کند، باید به شرکت‌کنندگان اجازه دهد تا تأیید کنند که Supply دارایی‌ها کنترل و قابل تأیید است. یک Elements Sidechain از یک federated peg دوطرفه استفاده می‌کند تا انتقال دوطرفه دارایی‌ها به داخل و خارج از یک Elements Blockchain را ممکن سازد. این امر نیازهای صدور قابل اثبات و انتقالات بین زنجیره‌ای را برآورده می‌کند.


ویژگی Federated 2-Way Peg به یک دارایی اجازه می‌دهد تا با بلاکچین‌های دیگر سازگار باشد و نماینده دارایی بومی دیگری از Blockchain باشد. با متصل کردن Blockchain خود به دیگری، می‌توانید قابلیت‌های mainchain را گسترش دهید و بر برخی از محدودیت‌های ذاتی آن غلبه کنید.


در سطح بالا، انتقال‌ها به Sidechain زمانی رخ می‌دهند که کسی دارایی‌های mainchain را به یک Address ارسال کند که توسط یک watchmen چند امضایی [Wallet](https://planb.academy/resources/glossary/wallet) کنترل می‌شود. این عمل به طور مؤثر دارایی‌ها را در mainchain منجمد می‌کند. سپس watchmen تراکنش را تأیید کرده و همان مقدار از دارایی مرتبط را در Sidechain آزاد می‌کند. دارایی‌های آزاد شده به یک Sidechain Wallet ارسال می‌شوند که می‌تواند ادعای مالکیت بر دارایی‌های اصلی mainchain را اثبات کند. این فرآیند به طور مؤثر دارایی‌ها را از زنجیره والد به Sidechain منتقل می‌کند.


برای انتقال دارایی‌ها به mainchain، کاربر یک تراکنش ویژه‌ی خروج از Sidechain انجام می‌دهد. این تراکنش توسط watchmen بررسی می‌شود که سپس تراکنشی را امضا می‌کنند که از چند امضای Wallet که در mainchain کنترل می‌کنند، خرج می‌کند. تعداد مشخصی از شرکت‌کنندگان در فدراسیون باید امضا کنند تا تراکنش mainchain معتبر شود. هنگامی که watchmen دارایی را به mainchain بازمی‌گرداند، مقدار معادل را در Sidechain نیز از بین می‌برد و به طور مؤثر دارایی‌ها را بین بلاکچین‌ها منتقل می‌کند.


## راه‌اندازی و اجرای Elements


<chapterId>cc806e5a-81ab-457b-9531-9f863120a019</chapterId>


:::video id=1f73dfee-3623-483b-ab42-07d9286ed999:::


از آنجا که Elements بر اساس کدبیس Bitcoin ساخته شده است، اجزایی که یک شبکه کارآمد را تشکیل می‌دهند بسیار مشابه هستند.


نرم‌افزار [نود](https://planb.academy/resources/glossary/node) Elements خود به نام `elementsd` شناخته می‌شود و به عنوان daemon بر روی دستگاه کاربر اجرا می‌شود. یک daemon (یا سرویس در ویندوز) برنامه‌ای است که به عنوان یک سرویس پس‌زمینه اجرا می‌شود و نیازی به کنترل مستقیم کاربر وارد شده ندارد.


توجه: در سراسر این سند، ما همیشه به elementsd به عنوان نسخه daemon اشاره خواهیم کرد، اما همه چیز می‌تواند با Elements-qt انجام شود، به شرطی که گزینه سرور فعال باشد.


Elements daemon به سایر گره‌های شبکه متصل می‌شود تا بتواند داده‌های تراکنش و بلوک Exchange را اعتبارسنجی و گسترش دهد و نسخه محلی خود از Blockchain شبکه را توسعه دهد.


نرم‌افزار Elements همچنین شامل یک برنامه کلاینت به نام `elements-cli` است که به شما اجازه می‌دهد تا دستورات Remote Procedure Call (RPC) را از خط فرمان به elementsd ارسال کنید. این می‌تواند برای پرس‌وجوی یک موجودی Wallet، مشاهده داده‌های تراکنش یا بلوک یا پخش یک تراکنش استفاده شود. این تنظیمات باید برای هر کسی که از معادل‌های Bitcoin استفاده کرده است؛ bitcoind و bitcoin-cli، آشنا باشد.


از آنجا که یک نود Elements می‌تواند با عبور پارامترها در زمان راه‌اندازی یا از طریق یک فایل پیکربندی تنظیم شود، امکان اجرای بیش از یک نمونه بر روی یک ماشین وجود دارد. این برای اهداف آزمایش و توسعه مفید است زیرا می‌توانید شبکه محلی خود را بر روی یک ماشین واحد راه‌اندازی کنید، به طوری که هر نود Elements نسخه خود از داده‌های Blockchain را داشته باشد، استخر خود از تراکنش‌های معتبر تایید نشده را مدیریت کند و به درخواست‌های RPC بر روی پورت‌های مختلف گوش دهد.


### مخزن کد و جامعه Elements


Elements یک پروژه متن‌باز است و کد منبع آن را می‌توان در مخزن GitHub به آدرس https://github.com/ElementsProject/Elements یافت. این مخزن شامل کد منبع برنامه‌های elementsd و elements-cli به همراه ابزارهای نصب و ساخت پشتیبانی، مجموعه‌ای از تست‌ها و برخی مستندات آموزشی است.


برای تکمیل مخزن کد، وب‌سایت https://elementsproject.org نیز وجود دارد که یک منبع متمرکز بر جامعه است و شامل توضیحاتی درباره Elements، نحوه کارکرد آن و یک بخش آموزشی جامع می‌باشد. آموزش‌ها بر یادگیری درباره Elements از طریق دنبال کردن مثال‌های خط فرمان تمرکز دارند و به شما نشان می‌دهند که چگونه برنامه‌های ساده دسکتاپ و وب را بر روی آن بسازید. این سایت همچنین فهرستی از انجمن‌های محبوب بحث و گفتگوی جامعه Elements را ارائه می‌دهد و خود بر روی GitHub میزبانی می‌شود، که امکان مشارکت‌های جامعه در محتوای سایت را فراهم می‌کند.


برای اجرای Elements روی دستگاه خود، ابتدا باید کد منبع را کلون کنید (یک نسخه از آن را دانلود کنید)، هرگونه وابستگی که کد دارد را نصب کنید و در نهایت فایل‌های اجرایی daemon و کلاینت را بسازید. سپس نرم‌افزار Elements آماده پیکربندی و اجرا است.


## پیکربندی گره‌ها و شبکه‌بندی


<chapterId>df1ec0aa-84ea-4149-af7a-b4523d67e1d9</chapterId>


تنظیمات پیکربندی می‌توانند در زمان راه‌اندازی به یک گره Elements منتقل شوند تا نحوه اجرای آن، اعتبارسنجی داده‌ها، اتصال به سایر گره‌ها و مقداردهی اولیه داده‌های Blockchain را تغییر دهند.


تنظیمات یا از فایل تعیین‌شده `Elements.conf` بارگذاری می‌شوند یا به‌عنوان پارامترها از طریق خط فرمان وارد می‌شوند.


برخی از موارد را می‌توان با استفاده از این پارامترها تغییر داد:



- نام default asset در پیاده‌سازی‌های مستقل Blockchain استفاده می‌شود.
- تعداد دارایی اولیه ایجاد شده.
- دارایی که هنگام پرداخت کارمزد تراکنش‌ها در شبکه استفاده می‌شود.
- محل ذخیره‌سازی فایل‌های داده Blockchain.
- اعتبارنامه‌های RPC برای اتصال به یک گره Bitcoin استفاده شد.
- آستانه `n از m` که باید رعایت شود و [کلیدهای عمومی](https://planb.academy/resources/glossary/public-key) معتبر که می‌توانند بلوک‌ها را امضا کنند.
- [اسکریپتی](https://planb.academy/resources/glossary/script) که برای انتقال دارایی‌ها به داخل و خارج از یک Sidechain نیاز به رضایت دارد.
- آیا به یک گره Bitcoin به عنوان Sidechain متصل شویم یا خیر.


بسیاری از این‌ها بخشی از قوانین اجماع شبکه را تشکیل می‌دهند، بنابراین مهم است که در زمان راه‌اندازی در تمام نودها اعمال شوند. برخی از آن‌ها می‌توانند پس از راه‌اندازی یک زنجیره تغییر کنند اما برخی نیاز است که پس از استفاده برای راه‌اندازی یک زنجیره ثابت شوند.


استفاده از پارامترها در ادامه دوره و زمانی که به هر بخش مربوط می‌شوند، پوشش داده خواهد شد.


### عملیات پایه با استفاده از خط فرمان


این دوره مثال‌هایی را نشان خواهد داد که از برنامه `elements-cli` برای انجام تماس‌های RPC به یک یا چند گره Elements استفاده می‌کنند. این کار از یک جلسه ترمینال انجام می‌شود و برای خلاصه‌تر کردن دستورات از یک `alias` استفاده خواهد شد. بر اساس این قرارداد، زمانی که چیزی شبیه به دستورات زیر را مشاهده می‌کنید:


```bash
e1-dae

e1-cli getnewaddress
```


`e1-dae` و `e1-CLI` در واقع یک میانبر تایپوگرافی هستند که از ویژگی `alias` ترمینال استفاده می‌کنند. `e1-dae` و `e1-CLI` در واقع زمانی که فرمان اجرا می‌شود جایگزین خواهند شد و فرمانی که اجرا خواهد شد مشابه خواهد بود با:


```
$HOME/elements/src/elementsd -datadir=$HOME/elementsdir1

$HOME/elements/src/elements-cli -datadir=$HOME/elementsdir1 getnewaddress
```


آنچه در بالا می‌بینیم، یک فراخوانی برای شروع برنامه‌های Elements و daemon و یک فراخوانی برای برنامه‌های elements-cli است که در دایرکتوری `$HOME/Elements/src` قرار دارند و یک مقدار برای پارامتر `datadir`. پارامتر `datadir` به ما اجازه می‌دهد تا به نمونه‌های daemon و کلاینت بگوییم که فایل‌های پیکربندی خود را کجا پیدا کنند و در مورد daemon، کپی خود از Blockchain را کجا ذخیره کنند. از آنجا که آن‌ها یک فایل پیکربندی مشترک دارند، کلاینت قادر خواهد بود تا فراخوانی‌های RPC را به daemon انجام دهد.


با اجرای مجدد فرمان فوق، اما با یک مقدار `datadir` متفاوت، می‌توانیم بیش از یک نمونه از Elements را راه‌اندازی کنیم، هر کدام با کپی جداگانه‌ای از Blockchain و تنظیمات پیکربندی خود. بر اساس این قرارداد، ما از نام مستعار `e2-dae` و `e2-CLI` در دوره برای اشاره به یک دایرکتوری datadir متفاوت از e1 استفاده خواهیم کرد. بنابراین مثال فوق برای نمونه دوم `e2` ما به این صورت خواهد بود:


```
$HOME/elements/src/elementsd -datadir=$HOME/elementsdir2

$HOME/elements/src/elements-cli -datadir=$HOME/elementsdir2 getnewaddress
```


این به ما اجازه می‌دهد تا انواع عملیات‌ها مانند معامله دارایی‌ها بین نودها، صدور دارایی‌ها و بررسی استفاده از بلایندینگ در Confidential Transactions بین نودهای مختلف در همان شبکه را انجام دهیم.

## Confidential Transactions


<chapterId>263b1c5b-59ed-49e7-b811-95c354f41eae</chapterId>


:::video id=ea2121b6-24a8-458d-91e6-0c92eaf4dc65:::


در این بخش شما یاد خواهید گرفت که چگونه از ویژگی Confidential Transactions در Elements استفاده کنید.


همه آدرس‌ها در Elements به‌طور پیش‌فرض از blinded با استفاده از Confidential Transactions بهره می‌برند، که مقدار و نوع دارایی‌های منتقل‌شده را تنها برای شرکت‌کنندگان در تراکنش (و کسانی که آن‌ها تصمیم به افشای Blinding key به آن‌ها دارند) قابل مشاهده نگه می‌دارد، در حالی که همچنان به‌صورت رمزنگاری تضمین می‌کند که بیش از سکه‌های موجود نمی‌توان خرج کرد.


### آدرس‌های محرمانه و Confidential Transactions


به طور پیش‌فرض، زمانی که یک Address جدید را در Elements با استفاده از دستور `getnewaddress` ایجاد می‌کنید، به صورت یک Confidential Address ایجاد می‌شود.


برای نمایش Confidential Transactions، ما e2 را وادار می‌کنیم که به خودش مقداری وجه ارسال کند و سپس سعی می‌کنیم تراکنش را از e1 مشاهده کنیم. این کار ماهیت محرمانه تراکنش‌ها در Elements را نشان خواهد داد.


هر Address جدیدی که توسط یک گره Elements تولید می‌شود به‌طور پیش‌فرض محرمانه است. ما می‌توانیم این را با گرفتن e2 برای generate یک Address جدید نشان دهیم.


```
e2-cli getnewaddress
```


توجه داشته باشید که Address با e1 شروع می‌شود. این نشان می‌دهد که آن یک Confidential Address است. بررسی دقیق‌تر Address با استفاده از فرمان getaddressinfo جزئیات بیشتری از Address را نشان می‌دهد.


```
e2-cli getaddressinfo <address>
```


می‌توانید ببینید که یک ویژگی confidential_key وجود دارد که به ما می‌گوید این یک Confidential Address است.


کلید محرمانه کلید عمومی Blinding key است که به خود Confidential Address اضافه می‌شود. این دلیل طولانی بودن Confidential Address است.


همچنین دارای یک Unconfidential address مرتبط است. اگر مایل به استفاده از تراکنش‌های عادی و غیرمحرمانه در Elements هستید، دارایی‌ها باید به جای آنکه به پیشوند lq1 ارسال شوند، به این Address ارسال شوند.


اکنون می‌توانیم e2 را وادار کنیم تا مقداری وجه به Address که تولید کرده است ارسال کند. این بعداً نشان خواهد داد که e1، به عنوان یکی از طرف‌های معامله نیست، قادر به مشاهده جزئیات تراکنش نخواهد بود.


```
e2-cli sendtoaddress <address>
```


توجه داشته باشید transaction ID. تراکنش را تأیید کنید.


```
e2-cli -generate 101
```


نگاهی به تراکنشی که در آن e2 مقداری وجه به خودش ارسال کرده است از دیدگاه خود e2.


```
e2-cli gettransaction <txid>
```


با بالا رفتن در جزئیات تراکنش، می‌توانید ببینید که e2 می‌تواند مقادیر ارسال و دریافت شده و همچنین دارایی معامله شده را مشاهده کند. همچنین می‌توانید ویژگی‌های amountblinder و assetblinder را ببینید که برای مخفی کردن جزئیات از سایر نودهایی که در تراکنش دخیل نیستند، استفاده می‌شوند.


برای بررسی جزئیات همان تراکنش از e1 ابتدا باید جزئیات خام تراکنش را دریافت کنیم.


```
e1-cli getrawtransaction <txid>
```


این جزئیات خام تراکنش را برمی‌گرداند. اگر به بخش vout نگاه کنید، می‌توانید ببینید که سه مورد وجود دارد. دو مورد اول مقادیر دریافت و تغییر هستند و سومین مورد کارمزد تراکنش است. از این سه مقدار، کارمزد تنها مقداری است که می‌توانید یک مقدار را ببینید، زیرا خود کارمزد همیشه unblinded درون Elements است.


### کلیدهای کورکننده


آنچه که دو بخش اول vout نشان می‌دهند، "Blinded ranges" از مقادیر ارزش و داده‌های Commitment است که به عنوان اثبات مقدار واقعی و نوع دارایی معامله شده عمل می‌کند.


حتی اگر [کلید خصوصی](https://planb.academy/resources/glossary/private-key) e2 را به Wallet e1 وارد کنیم، باز هم قادر نخواهد بود مقادیر و نوع دارایی معامله شده را ببیند زیرا هنوز اطلاعاتی از Blinding key استفاده شده توسط e2 ندارد. ما این را با وارد کردن کلید خصوصی استفاده شده توسط Wallet e2 به e1 اثبات خواهیم کرد. ابتدا باید کلید را از e2 صادر کنیم.


```
e2-cli dumpprivkey <address>
```


سپس آن را به e1 وارد کنید.


```
e1-cli importprivkey <privkey>
```


اکنون می‌توانیم ثابت کنیم که e1 هنوز نمی‌تواند مقادیر را ببیند.


```
e1-cli gettransaction <txid>
```


در واقع، زمانی که مقدار واقعی 1 بود، مقدار tx را 0 نشان می‌دهد.


برای اینکه بتوانیم مقدار واقعی و بدون پوشش را ببینیم، به Blinding key نیاز داریم. برای انجام این کار ابتدا Blinding key را از e2 صادر می‌کنیم.


```
e2-cli dumpblindingkey <address>
```


سپس آن را به e1 وارد کنید.


```
e1-cli importblindingkey <address> <blinding key>
```


اکنون وقتی جزئیات تراکنش را از e1 دریافت می‌کنیم.


```
e1-cli gettransaction <txid>
```


این نشان می‌دهد که با وارد کردن Blinding key، اکنون می‌توانیم مقدار واقعی 1 را در تراکنش مشاهده کنیم.


در این بخش دیده‌ایم که استفاده از Blinding key مقدار و نوع دارایی‌ها در یک تراکنش را پنهان می‌کند و با وارد کردن Blinding key صحیح، می‌توانیم آن مقادیر را آشکار کنیم. در استفاده عملی، ممکن است یک Blinding key به یک حسابرس داده شود، در صورتی که نیاز به تأیید مقادیر و انواع دارایی‌های نگهداری شده توسط یک طرف باشد. ویژگی Confidential Transactions در Elements همچنین امکان انجام "اثبات‌های محدوده" را فراهم می‌کند. اثبات‌های محدوده می‌توانند ثابت کنند که مقدار یک دارایی در یک محدوده مشخص نگهداری می‌شود، بدون نیاز به افشای مقدار واقعی آن.


ما همچنین دیده‌ایم که Confidential Transactions اختیاری است، اما به‌طور پیش‌فرض زمانی که یک Address جدید تولید می‌شود فعال است.


این هم از این درس؛ موفق باشید در آزمون و در درس بعدی می‌بینمتان!


## Issued Assets


<chapterId>c33c7020-5975-457a-99db-4f8b90d1fa1c</chapterId>


:::video id=7ac63148-d730-496d-85d4-0032aaf09be1:::


در این بخش شما یاد خواهید گرفت که چگونه از ویژگی Issued Assets در Elements استفاده کنید.


Issued Assets امکان صدور و انتقال انواع مختلف دارایی را بین شرکت‌کنندگان شبکه Elements فراهم می‌کند. هر نودی در شبکه می‌تواند دارایی‌های خود را صادر کند. صدورها ممکن است نمایانگر Ownership قابل تعویض از هر دارایی از جمله ووچرها، کوپن‌ها، ارزها، سپرده‌ها، اوراق قرضه، سهام و غیره باشند. Issued Assets درب را برای ساخت Trustless مبادلات، گزینه‌ها و سایر قراردادهای هوشمند پیشرفته که شامل خود-Issued Assets می‌شوند، باز می‌کند.


یک دارایی صادر شده همچنین از Confidential Transactions بهره‌مند می‌شود و می‌تواند توسط هر کسی که token مرتبط را در اختیار دارد، مجدداً صادر شود.


اولین گام این است که به دو نود Elements دسترسی داشته باشیم، که آن‌ها را e1 و e2 می‌نامیم. بلاک‌چین‌های نودها ریست شده و default asset بین آن‌ها تقسیم شده است.


دو نود در یک شبکه محلی قرار دارند و به یکدیگر متصل هستند و بنابراین تراکنش‌های مشابهی را در تراکنش [Mempool](https://planb.academy/resources/glossary/mempool) و بلاک‌چین‌های یکسان به اشتراک می‌گذارند. اگرچه آن‌ها بر روی یک ماشین اجرا می‌شوند، شایان ذکر است که آن‌ها فایل‌های واقعی Blockchain یکسانی را به اشتراک نمی‌گذارند. هر نود نسخه محلی خود از Blockchain را مدیریت می‌کند که شامل تاریخچه تراکنش مشابهی است زیرا آن‌ها در اجماع هستند و از قوانین پروتکل مشابهی پیروی می‌کنند.


بیایید با بررسی دیدگاه هر گره نسبت به انتشار دارایی‌های موجود در شبکه شروع کنیم.


این با استفاده از فرمان listissuances انجام می‌شود.


```
e1-cli listissuances

e2-cli listissuances
```


همان‌طور که می‌بینید، هر دو نود تاریخچه انتشار یکسانی را نشان می‌دهند. هر دو یک دارایی را نمایش می‌دهند، انتشار اولیه 21 میلیون Bitcoin که در ایجاد on chain اولیه ایجاد شده‌اند. شما می‌توانید شناسه هگزادسیمال دارایی را در نتایج اجرای فرمان بالا مشاهده کنید و همچنین برچسب اختصاص داده شده به دارایی که 'Bitcoin' است.


شایان ذکر است که default asset همیشه زمانی که زنجیره راه‌اندازی می‌شود، یک برچسب دریافت می‌کند. هنگامی که دارایی‌های خود را صادر می‌کنید، می‌توانید برچسب‌هایی برای آن‌ها تعیین کنید که به زودی این کار را انجام خواهیم داد. قبل از اینکه بتوانیم این کار را انجام دهیم، باید دارایی خود را صادر کنیم.


ما e1 را موظف خواهیم کرد تا دارایی جدید را صادر کند. این کار با استفاده از فرمان issueasset انجام می‌شود.


```
e1-cli issueasset 100 1 false
```


`issueasset` سه پارامتر را می‌پذیرد.


مقدار دارایی جدید برای صدور، ما از 100 استفاده کرده‌ایم. مقدار توکن‌هایی که باید ایجاد شوند (توکن‌ها برای صدور مجدد مقادیر یک دارایی استفاده می‌شوند)، که ما 1 را انتخاب کرده‌ایم. پارامتر نهایی به Elements می‌گوید که دارایی را به عنوان blinded یا unblinded صادر کند. ما از unblinded استفاده خواهیم کرد زیرا می‌خواهیم مقادیر صدور را از e2 در یک دقیقه مشاهده کنیم، بنابراین false را وارد خواهیم کرد.


اجرای فرمان داده‌هایی درباره صدور را برمی‌گرداند. این‌ها شامل transaction ID است که می‌توانید یک نسخه از آن را برای استفاده بعدی بگیرید، مقدار هگز منحصر به فرد دارایی، و مقدار هگز منحصر به فرد token دارایی.


generate یک بلوک برای تأیید تراکنش صدور.


```
e1-cli -generate 1
```


دستور `listissuances` را دوباره بر روی e1 اجرا کنید.


```
e1-cli listissuances
```


این به ما نشان می‌دهد که e1 اکنون از 2 انتشار آگاه است، انتشار اولیه Bitcoin و دارایی جدید صادر شده ما، که می‌توانیم 100 عدد از آن را ببینیم. به مقدار هگزادسیمال دارایی جدید توجه کنید و اینکه هیچ برچسب دارایی مرتبطی وجود ندارد، همانطور که برای انتشار Bitcoin وجود دارد.


فهرست انتشارات شناخته شده e2 را دوباره بررسی کنید.


```
e2-cli listissuances
```


این به ما نشان می‌دهد که e2 از صدور دارایی e1 آگاه نیست. این تنها می‌تواند صدور اولیه Bitcoin را ببیند که قبلاً می‌توانست ببیند.


این به این دلیل است که e2 از Address که دارایی جدید به آن ارسال شده و توسط e1 صادر شده است، بی‌اطلاع است و آن را مشاهده نمی‌کند.


شایان ذکر است که حتی اگر e2 نتواند خود صدور را ببیند، e1 همچنان می‌تواند مقداری از دارایی را به e2 ارسال کند. دارایی جدید سپس به عنوان موجودی قابل دسترس در Wallet e2 ظاهر می‌شود، حتی اگر از صدور اصلی آگاه نباشد.


برای اینکه e2 بتواند انتشار واقعی (و بنابراین مقدار صادر شده) را ببیند، باید Address را به عنوان یک Address مشاهده شده به e2 اضافه کنیم.


برای انجام این کار، باید Address را که دارایی به آن ارسال شده پیدا کنیم. برای این منظور، از transaction ID که قبلاً کپی کرده‌ایم استفاده خواهیم کرد و e1 جزئیات آن تراکنش را بازیابی می‌کند تا بتوانیم Address صحیح را برای اضافه کردن به لیست نظارت Wallet از e2 پیدا کنیم.


```
e1-cli gettransaction <the-issuance-transaction-id>
```


با بالا رفتن از هگز داده‌های تراکنش، Address را خواهید دید که 100 از دارایی جدید ما را دریافت کرده است، که با مقدار هگز آن شناسایی می‌شود.


Address را بگیرید و آن را کپی کنید تا بتوانیم آن را به e2 وارد کنیم.


حالا بیایید Address را به e2 وارد کنیم. برای انجام این کار از دستور importaddress استفاده می‌کنیم.


```
e2-cli importaddress <the-issued-to-address>
```


اگر اکنون لیست انتشارهای e2 را بررسی کنیم.


```
e2-cli listissuances
```


می‌توانید ببینید که دارایی جدیداً صادر شده ما اکنون در لیست گنجانده شده است. گره e2 همچنین قادر است مقدار دارایی صادر شده را به همراه مقدار token مرتبط تعیین کند، زیرا صدور به صورت unblinded بوده است. برای فعال کردن استفاده از نگاشت شناسه دارایی به نام در Elements، ابتدا Elements را متوقف کنید.


```
e1-cli stop
```


سپس آن را با یک پارامتر اضافی که هگز یک دارایی را به برچسب ارائه شده نگاشت می‌کند، راه‌اندازی مجدد کنید. این به نود اجازه می‌دهد تا داده‌های مربوط به دارایی را به شکلی که برای ما قابل خواندن‌تر باشد نمایش دهد. اگر ترجیح می‌دهید، می‌توانید این را به انتهای Elements.conf اضافه کنید، سپس نیازی نیست هر بار که daemon را راه‌اندازی می‌کنید، آرگومان را اضافه کنید. برای مثال:


```
assetdir=5186d0bc8ed15e6ef85571bd2d8070573adf0e06fd4507082694526975ce4f35:My new asset (MNA)
```


اما ما در اینجا از روش استدلال استفاده خواهیم کرد.


```
e1-dae -assetdir=<assetid-here>:<name-of-the-new-asset>
```


درخواست مجدد از نود برای فهرست انتشارها.


```
e1-cli listissuances
```


این به ما نشان می‌دهد که نگاشت مقدار هگز دارایی به برچسب آن در حال کار است. دوباره فهرست انتشارهای گره e2 را بررسی می‌کنیم.


```
e2-cli listissuances
```


می‌توانید ببینید که گره e2 به این برچسب دسترسی ندارد، زیرا برچسب‌ها فقط برای گره‌ای که آن‌ها را تنظیم کرده است در دسترس هستند. در واقع، می‌توانیم یک برچسب متفاوت به همان دارایی hex روی e2 اختصاص دهیم که با e1 انجام دادیم. ابتدا گره e2 را متوقف کنید.


```
e2-cli stop
```


شروع مجدد با برچسب متفاوتی که به هگز دارایی جدید ما اختصاص داده شده است.


```
e2-dae -assetdir=<assetid-here>:<another-name-for-the-new-asset>
```


فهرست انتشارها از e2.


```
e2-cli listissuances
```


برچسب‌های دارایی به هر گره محلی هستند، تنها هگز دارایی توسط سایر گره‌ها در شبکه شناسایی می‌شود.


نقشه‌برداری از برچسب به هگز دارایی زمانی که اقداماتی مانند تراکنش‌ها و پرسش‌های موجودی Wallet انجام می‌شود، مفید است زیرا راهی کوتاه برای اشاره به یک دارایی فراهم می‌کند. به عنوان مثال، اگر بخواهیم مقداری از دارایی جدید خود (به میزان ۱۰) را از e1 به e2 بدون استفاده از برچسب ارسال کنیم.


اول باید یک Address تهیه کنیم تا دارایی را به آن ارسال کنیم.


```
e2-cli getnewaddress
```


سپس از فرمان sendtoaddress استفاده کنید.


```
e1-cli sendtoaddress <address> 10 "" "" false false 1 UNSET false <asset-id-here>
```


تراکنش را با تولید یک بلوک تأیید کنید.


```
generate 1
```


در حال بررسی دریافت دارایی در e2.


```
e2-cli getwalletinfo
```


ما می‌توانیم ببینیم که دارایی واقعاً دریافت شده است.


توجه داشته باشید که e2 هگز دارایی دریافت‌شده را نگاشت کرده و با استفاده از برچسب خود نمایش می‌دهد. یک روش ساده‌تر برای انجام همین کار این است که از برچسب دارایی e1 هنگام ارسال استفاده کنید.


```
e1-cli sendtoaddress <address> 10 "" "" false false 1 UNSET false <name-of-the-new-asset>
```


پشت صحنه، Elements برچسب‌های محلی را به مقادیر هگزادسیمال نگاشت می‌کند تا استفاده از Issued Assets را ساده‌تر کند.


در این بخش مشاهده کرده‌ایم که چگونه دارایی‌ها را صادر و برچسب‌گذاری کنیم. در بخش بعدی به بررسی صدور مجدد و از بین بردن مقادیر یک دارایی صادر شده خواهیم پرداخت.


## صدور مجدد دارایی‌ها


<chapterId>78751b21-1dc8-4877-a406-e71bc80a95b0</chapterId>


:::video id=7df967b0-ffff-42e1-b1d5-868e76289faf:::


در این بخش شما یاد خواهید گرفت که چگونه مقدار بیشتری از یک دارایی صادر شده را صادر کنید و همچنین چگونه مقدار مشخصی از یک دارایی صادر شده را از بین ببرید.


نیاز به صدور مجدد (ایجاد بیشتر) یک دارایی یا از بین بردن مقداری از دارایی چیزی است که احتمالاً زمانی رخ می‌دهد که دارایی نمایانگر چیزی باشد که Supply ثابتی ندارد. این می‌تواند به دارایی‌هایی که نمایانگر طلای نگهداری شده در یک خزانه هستند اعمال شود؛ همان‌طور که واحدهای طلا به داخل و خارج از خزانه منتقل می‌شوند، دارایی نمایانگر Supply خزانه باید به‌طور متناسب افزایش یا کاهش یابد.


صدور مجدد مقدار یک دارایی نیازمند Ownership از token مرتبط است که همراه با دارایی در زمان صدور اولیه ایجاد شده بود.


هنگام ایجاد بیشتر یک دارایی، مهم نیست که کدام نود ابتدا دارایی را صادر کرده است، تا زمانی که نودی که مقدار بیشتری از دارایی را مجدداً صادر می‌کند، در اختیار چیزی باشد که معمولاً به آن reissuance token دارایی گفته می‌شود. ما به چگونگی ایجاد اولیه reissuance token، نحوه استفاده از آن برای صدور مجدد مقدار یک دارایی و همچنین نحوه انتقال reissuance token به نودهای دیگر خواهیم پرداخت، به طوری که آنها نیز اجازه صدور مجدد دارایی را داشته باشند.


ما به دسترسی به دو نود Elements نیاز خواهیم داشت که آن‌ها را e1 و e2 می‌نامیم. بلاک‌چین‌های این نودها بازنشانی شده و default asset بین آن‌ها تقسیم شده است.


ما e1 را موظف خواهیم کرد که مقدار 100 از یک دارایی جدید صادر کند و 1 reissuance token برای همان دارایی ایجاد کند. ما صدور را به عنوان unblinded ایجاد خواهیم کرد تا مثال را ساده کنیم. بنابراین بیایید دارایی و reissuance token مرتبط با آن را صادر کنیم.


```
e1-cli issueasset 100 1 false
```


شناسه دارایی و همچنین شناسه (صدور مجدد) token را یادداشت کنید.


همانطور که بعداً دارایی بیشتری از e2 را مجدداً صادر خواهیم کرد، باید به transaction ID که دارایی در آن صادر شده توجه کنیم و از آن برای وارد کردن Address که دارایی به آن ارسال شده استفاده کنیم.


تراکنش را تأیید کنید.


```
e1-cli -generate 1
```


اکنون جزئیات تراکنش را با استفاده از فرمان gettransaction بررسی خواهیم کرد:


```
e1-cli gettransaction <txid>
```


با پیمایش به بالا از هگز داده‌های تراکنش، خواهید دید که در تراکنش e1، 1 reissuance token و 100 از دارایی مرتبط دریافت شده است.


یک نسخه از Address بگیرید تا بتوانیم آن را به e2 وارد کنیم.


و اکنون وارد کردن Address به Wallet در e2.


```
e2-cli importaddress <address>
```


اکنون می‌توانیم ببینیم که هم e1 و هم e2 از صدور دارایی آگاه هستند.


```
e1-cli listissuances

e2-cli listissuances
```


در حال حاضر e1 مقداری از دارایی و 1 reissuance token را در اختیار دارد اما e2 ندارد.


```
e1-cli getwalletinfo
```


همچنین توجه داشته باشید که e1 کمتر از default asset نسبت به قبل دارد زیرا مقدار کمی برای پوشش هزینه‌های تراکنش پرداخت کرده است. این مقدار قرار است توسط e1 جمع‌آوری شود زمانی که بلوک ایجاد شده بیش از 100 بلوک عمق پیدا کند.


```
e2-cli getwalletinfo
```


همانطور که e1 دارنده reissuance token است، می‌تواند مقدار بیشتری از آن را مجدداً صادر کند. این کار با استفاده از فرمان reissueasset انجام می‌شود. بیایید e1 را وادار کنیم تا 100 واحد دیگر از این دارایی را مجدداً صادر کند.


```
e1-cli reissueasset <asset-id> 100
```


بررسی شد که صدور مجدد کار کرد.


```
e1-cli getwalletinfo
```


شما می‌توانید ببینید که e1 اکنون 200 از دارایی را همان‌طور که انتظار می‌رفت، نگه می‌دارد.


از آنجا که e2 مقداری از reissuance token را در اختیار ندارد، در صورت تلاش برای صدور مجدد دارایی با خطا مواجه خواهند شد.


```
e2-cli reissueasset <asset-id> 100
```


پیام خطا را یادداشت کنید.


ما می‌توانیم جزئیات صدور مجدد را از e1 با استفاده از فرمان listissuances مشاهده کنیم.


```
e1-cli listissuances
```


به پرچم `is_reissuance` توجه کنید.


اگر اکنون مقدار reissuance token را به e2 ارسال کنیم، آن‌ها قادر خواهند بود که مقداری از دارایی را خودشان مجدداً صادر کنند. ابتدا نیاز داریم که یک Address برای ارسال آن داشته باشیم. شایان ذکر است که reissuance token دقیقاً مانند هر دارایی دیگری در Elements هنگام ارسال و نمایش موجودی‌ها رفتار می‌شود و می‌تواند به واحدهای کوچکتر نیز تقسیم شود، بنابراین نیازی نیست که 1 reissuance token را به e2 ارسال کنیم تا بتواند دارایی را مجدداً صادر کند. هر واحدی کافی خواهد بود. تولید یک Address برای e2 جهت دریافت reissuance token.


```
e2-cli getnewaddress
```


سپس کسری از RIT را از e1 به e2 ارسال کنید.


```
e1-cli sendtoaddress <address-of-e2> 0.1 "" "" false false 1 UNSET false <reissuance-token-id>
```


تراکنش را تأیید کنید.


```
e1-cli -generate 1
```


اکنون می‌توانیم ببینیم که e2 مقدار 0.1 که به آن ارسال شده بود را نگه می‌دارد.


```
e2-cli getwalletinfo
```


این بدان معناست که e2 اکنون می‌تواند مقدار بیشتری از دارایی مرتبط با RIT که در Wallet خود نگه می‌دارد را مجدداً صادر کند. ما از e2 می‌خواهیم که 500 از دارایی را مجدداً صادر کند.


```
e2-cli reissueasset <asset-id> 500
```


نتیجه صدور مجدد را بررسی کنید.


```
e2-cli getwalletinfo
```


می‌توانید ببینید که e2 اکنون مقدار مجدداً صادر شده را در تراز Wallet خود نگه می‌دارد و اینکه خود RIT در فرآیند صدور مجدد دارایی مصرف نمی‌شود.


از بین بردن مقداری از یک دارایی کاری است که هر کسی که حداقل به اندازه مقدار در حال از بین رفتن را دارد، می‌تواند انجام دهد، این کار تحت نظارت reissuance token نیست.


```
e2-cli destroyamount <asset-id>

e2-cli getwalletinfo
```


در این بخش مشاهده کرده‌ایم که چگونه یک دارایی را صادر کنیم و همچنین چگونه از reissuance token که به صورت اختیاری به عنوان بخشی از صدور دارایی ایجاد می‌شود، استفاده کنیم. همچنین دیده‌ایم که انتقال یک reissuance token به سادگی انتقال هر دارایی دیگری است و داشتن هر مقدار از یک reissuance token به دارنده این حق را می‌دهد که مقدار بیشتری از دارایی مرتبط را صادر کند. بنابراین بسیار مهم است که کنترل کنید چه کسی به توکن‌های صدور مجدد در شبکه شما دسترسی دارد. همچنین دیده‌ایم که چگونه می‌توان مقداری از یک دارایی را از بین برد و این فرآیند توسط مالکیت reissuance token کنترل نمی‌شود.

## block signing


<chapterId>c47b217e-db14-4843-a66f-3e5f3a00a808</chapterId>


:::video id=c5a81820-77d7-4a0c-9a4e-9323386a74ac:::


Elements از یک مدل امضای فدرال پشتیبانی می‌کند که به شما اجازه می‌دهد تعداد اعضای Strong Federation را که باید یک بلوک پیشنهادی را امضا کنند تا یک بلوک معتبر تولید شود، مشخص کنید.


قبلاً، و برای سهولت مثال، ما بلوک‌ها را با استفاده از دستور `generate` ایجاد کرده‌ایم، که نیازی به برآورده کردن یک الزام چند امضایی نداشته است تا بلوک‌های ایجاد شده توسط شبکه به عنوان معتبر پذیرفته شوند.


ما گره‌های خود را طوری تنظیم خواهیم کرد که نیاز به ایجاد بلوک 2-of-2 Multisig داشته باشند. این تنظیم با استفاده از پارامتر signblockscript انجام خواهد شد که می‌تواند به فایل پیکربندی اضافه شود یا در زمان راه‌اندازی به گره منتقل شود. برای راه‌اندازی یک زنجیره با این پارامتر، ابتدا باید اسکریپتی که از آن ساخته شده است را ایجاد کنیم.


ما این کار را با استفاده از برخی از نودهای موجود انجام خواهیم داد، داده‌هایی که آن‌ها خروجی می‌دهند را ذخیره می‌کنیم و سپس زنجیره را پاک می‌کنیم تا بتوانیم آن را با استفاده از پارامتر signblockscript دوباره راه‌اندازی کنیم. این کار ضروری است زیرا اسکریپت بخشی از قوانین اجماع شبکه را تشکیل می‌دهد و نیاز به تنظیم در زمان اولیه‌سازی on chain دارد. نمی‌توان آن را در تاریخ بعدی به زنجیره‌ای که از قبل وجود دارد اضافه کرد.


ما به دسترسی به دو گره Elements نیاز خواهیم داشت که آن‌ها را e1 و e2 می‌نامیم. بلاک‌چین‌های این گره‌ها بازنشانی شده و default asset بین آن‌ها تقسیم شده است.


مطمئن شوید که پارامتر con_max_block_sig_size در فایل Elements.conf به مقدار بالایی تنظیم شده است، در غیر این صورت، block signing در ادامه این بخش با شکست مواجه خواهد شد. ما برای این آموزش con_max_block_sig_size=2000 را تنظیم کرده‌ایم.


همانطور که ما Blockchain را بازنشانی کرده و کیف‌پول‌های مرتبط با e1 و e2 را پاک خواهیم کرد، نیاز داریم که یک نسخه از آدرس‌ها، کلیدهای عمومی و کلیدهای خصوصی که برای generate اسکریپت block signing استفاده می‌کنیم، تهیه کنیم تا بتوانیم بعداً از آنها استفاده کنیم.


ابتدا، ما به هر یک از آنچه که به عنوان گره‌های block signing خواهد بود نیاز داریم تا generate یک Address جدید، که شما نیاز دارید یک نسخه از آن بگیرید.


```
e1-cli getnewaddress

e2-cli getnewaddress
```


سپس باید کلیدهای عمومی را از آدرس‌ها استخراج کرده و آن‌ها را برای استفاده بعدی یادداشت کنیم.


```
e1-cli getaddressinfo <e1-address>

e2-cli getaddressinfo <e2-address>
```


سپس کلیدهای خصوصی را استخراج کنید، که بعداً آن‌ها را دوباره وارد خواهیم کرد تا گره‌ها بتوانند پس از بازنشانی داده‌های Blockchain و Wallet ما، بلوک‌ها را امضا کنند.


```
e1-cli dumpprivkey <e1-address>

e2-cli dumpprivkey <e2-address>
```


اکنون ما نیاز داریم که یک اسکریپت generate را با الزامات چند امضایی 2 از 2 به Redeem تبدیل کنیم. ما این کار را با استفاده از دستور createmultisig انجام می‌دهیم و اولین پارامتر را به عنوان 2 وارد کرده و سپس دو کلید عمومی ارائه می‌دهیم. این کلیدها همان‌هایی هستند که Ownership نیاز به اثبات آن‌ها دارد زمانی که بلوک ایجاد می‌شود.


یا گره e1 یا e2 می‌تواند generate را به Multisig تبدیل کند.


```
e1-cli createmultisig 2 '["<e1-pubkey>", "<e2-pubkey>"]'
```


این به ما redeemscript خودمان را می‌دهد که می‌توانید برای استفاده بعدی کپی کنید.


اکنون باید داده‌های موجود Blockchain و Wallet را پاک کنیم تا بتوانیم با signblockscript جدید به عنوان بخشی از قوانین اجماع زنجیره دوباره شروع کنیم. به همین دلیل بود که نیاز داشتیم قبلاً از برخی داده‌ها، مانند کلیدهای خصوصی که در زنجیره جدید برای امضای بلوک‌ها استفاده خواهند شد، نسخه‌ای تهیه کنیم. شما باید این کار را قبل از ادامه انجام دهید.


با حذف داده‌های موجود Wallet و زنجیره، اکنون می‌توانیم نودهای خود را راه‌اندازی کرده و یک زنجیره جدید با استفاده از پارامتر signblockscript ایجاد کنیم. بیایید -evbparams=dynafed:0::: را وارد کنیم تا فعال‌سازی dynafed را غیرفعال کنیم، زیرا برای این مثال به آن ویژگی پیشرفته نیازی نداریم.


```
e1-dae -signblockscript=<redeem-script> -evbparams=dynafed:0:::

e2-dae -signblockscript=<redeem-script> -evbparams=dynafed:0:::
```


اکنون باید کلیدهای خصوصی که قبلاً ذخیره کرده‌ایم را وارد کنیم تا گره‌های ما بتوانند امضا کنند و به تکمیل هر بلوک پیشنهادی کمک کنند.


```
e1-cli importprivkey <e1-priv-key>

e2-cli importprivkey <e2-priv-key>
```


استفاده از دستور generate اکنون باید خطا دهد زیرا نتوانسته است قوانین مورد نیاز block signing را که اکنون توسط نودهای ما اعمال می‌شود، رعایت کند.


```
e1-cli -generate 1
```


برای پیشنهاد یک بلوک جدید، هر یک از نودها می‌توانند فرمان getnewblockhex را فراخوانی کنند. این فرمان هگزادسیمال یک بلوک جدید را برمی‌گرداند که قبل از پذیرفته شدن توسط هر نودی در شبکه ما، نیاز به امضا دارد.


```
e1-cli getnewblockhex
```


به خاطر داشته باشید که این دستور فقط یک بلوک پیشنهادی ایجاد می‌کند، نه اینکه generate یکی را.


برای تأیید این موضوع می‌توانیم ببینیم که در حال حاضر هیچ بلوکی در Blockchain ما وجود ندارد.


```
e1-cli getblockcount
```


اگر سعی کنیم بلوک هگز را بدون امضا کردن آن ارسال کنیم.


```
e1-cli submitblock <block-hex>
```


ما پیامی دریافت می‌کنیم که به ما می‌گوید اثبات بلوک نامعتبر است. این به این دلیل است که هنوز توسط ۲ طرف از ۲ طرف مورد نیاز امضا نشده است.


پس بیایید e1 را برای امضای بلوک پیشنهادی بگیریم.


```
e1-cli signblock <block-hex>
```


هگز را به e2 بدهید تا امضا کند.


```
e2-cli signblock <block-hex>
```


توجه داشته باشید که e2 خروجی ایجاد شده از امضای بلوک پیشنهادی توسط e1 را امضا نمی‌کند. هر دوی آن‌ها بلوک پیشنهادی هگز را به‌طور مستقل از نتایج یکدیگر امضا می‌کنند.


اکنون ما نیاز داریم تا امضاهای بلوک e1 و e2 را ترکیب کنیم. هر کدام از نودها می‌توانند این کار را انجام دهند، تنها چیزی که نیاز دارند، هگز بلوک امضا شده از نود دیگر است.


```
e1-cli combineblocksigs <block-hex> '["<signed-hex-from-e1>", "<signed-hex-from-e2>"]'
```


می‌توانید ببینید که فرمان combineblocksigs خروجی‌های هگز بلاک امضا شده و همچنین وضعیت تکمیل را ارائه می‌دهد، که به ما می‌گوید هگز بلاک آماده ارسال است.


اکنون هر یک از نودها می‌توانند بلاک کامل شده را به صورت هگز ارسال کنند. ما این کار را با e1 انجام خواهیم داد.


```
e1-cli submitblock <combined-signed-hex>
```


بررسی اینکه ارسال معتبر بوده است.


```
e1-cli getblockcount

e2-cli getblockcount
```


می‌توانید ببینید که هم e1 و هم e2 بلوک را به عنوان معتبر پذیرفته و آن را به انتهای نسخه‌های محلی خود از Blockchain اضافه کرده‌اند.


برای خلاصه کردن فرآیند. در این بخش ما داریم:


- پیشنهاد یک بلوک.
- هر گره آن را امضا کرد.
- امضاها را ترکیب کرد.
- بررسی شد که امضاها معتبر هستند و به آستانه redeemscript زنجیره می‌رسند.
- بلوک ارسال شد.


هر گره در شبکه بلوک را تأیید کرد و آن را به نسخه محلی خود از Blockchain اضافه کرد.


### block signing


اگرچه این فرآیند در ابتدا پیچیده به نظر می‌رسد، توالی block signing در Elements همیشه یکسان است و تنظیم اولیه فقط یک بار باید انجام شود:


۱. تنظیم اولیه (یک بار انجام می‌شود)

۲. یک Address چندامضایی به نام `signblockscript` با استفاده از کلیدهای عمومی امضاکنندگان بلوک فدرال ایجاد می‌شود.

۳. اسکریپت Redeem از این برای شروع یک Blockchain جدید استفاده می‌شود.

۴. تولید بلوک (در حال انجام)

۵. بلوک‌های پیشنهادی تولید و برای امضا مبادله می‌شوند.


هنگامی که تعداد آستانه‌ای از امضاکنندگان، بلوک پیشنهادی را امضا کنند، این بلوک ترکیب شده و به شبکه ارسال می‌شود. اگر معیارهای زنجیره `signblockscript` را برآورده کند، نودها آن را به عنوان یک بلوک معتبر می‌پذیرند.


## عنصر به عنوان زنجیره جانبی


<chapterId>432d7a65-255f-44a3-8b38-78508202cb37</chapterId>


:::video id=c15e7eaf-9b5d-4696-bb36-bd10e7b56967:::


Elements یک پلتفرم متن‌باز و چندمنظوره Blockchain است که می‌تواند به یک Blockchain موجود، مانند Bitcoin، `متصل` شود. هنگامی که به یک Blockchain دیگر متصل می‌شود، گفته می‌شود که Elements به عنوان یک `Sidechain` عمل می‌کند. زنجیره‌های جانبی انتقال دوطرفه دارایی‌ها از یک زنجیره به زنجیره دیگر را امکان‌پذیر می‌سازند. پیاده‌سازی Elements به عنوان یک Sidechain به شما اجازه می‌دهد تا برخی از محدودیت‌های ذاتی mainchain را دور بزنید، در حالی که همچنان سطح خوبی از امنیت ارائه شده توسط دارایی‌های امن شده بر روی mainchain را حفظ می‌کنید.


در حالی که یک Sidechain از mainchain و تاریخچه تراکنش‌های آن آگاه است، mainchain هیچ آگاهی از Sidechain ندارد و نیازی به آن برای عملکردش نیست. این امر به زنجیره‌های جانبی اجازه می‌دهد بدون محدودیت یا تأخیرهای مرتبط با پیشنهادات بهبود پروتکل mainchain نوآوری کنند. به جای تلاش برای تغییر مستقیم آن، گسترش پروتکل اصلی اجازه می‌دهد تا خود mainchain امن و تخصصی باقی بماند و عملکرد روان Sidechain را پشتیبانی کند.


با گسترش قابلیت‌های Bitcoin و بهره‌گیری از امنیت زیربنایی آن، یک Sidechain مبتنی بر Elements قادر است قابلیت‌های جدیدی را ارائه دهد که قبلاً برای کاربران mainchain در دسترس نبود. نمونه‌ای از یک Sidechain مبتنی بر Elements که در تولید استفاده می‌شود، Liquid Network است.


برای راه‌اندازی یک Elements Blockchain به عنوان Sidechain، نیاز داریم از پارامتر federated peg script استفاده کنیم. این پارامتر می‌تواند در فایل پیکربندی یک نود تنظیم شود یا در زمان راه‌اندازی ارسال شود.


federated peg script تعیین می‌کند که کدام اعضای Strong Federation می‌توانند عملکردهای peg-in و peg-out را انجام دهند. این کارگزاران به عنوان `watchmen` شناخته می‌شوند زیرا آن‌ها mainchain و Sidechain را برای تراکنش‌های معتبر peg-in و peg-out مشاهده می‌کنند و در صورت معتبر بودن آن‌ها را اجرا می‌کنند. `peg-out` به معنای انتقال دارایی‌های متصل از Sidechain به mainchain و `peg-in` به معنای انتقال دارایی‌های متصل به Sidechain از mainchain است. وقتی می‌گوییم `انتقال به Sidechain`، منظورمان این است که وجوه در یک Address چند امضایی در mainchain قفل می‌شوند و مقدار معادل دارایی در Elements Sidechain ایجاد می‌شود. وقتی می‌گوییم `انتقال از Sidechain`، منظورمان این است که دارایی‌ها در Elements Sidechain نابود می‌شوند و مقدار معادل از وجوه قفل شده در mainchain آزاد می‌شود. اجازه انجام عملکردهای peg-in و peg-out نیازمند این است که کارگزاران Ownership کلیدهای عمومی استفاده شده در federated peg script را اثبات کنند. این کار با استفاده از کلیدهای خصوصی مربوطه انجام می‌شود.


برای ایجاد یک federated peg script، ابتدا باید هر یک از نودهای خود generate را با یک کلید عمومی تجهیز کنیم. همچنین نیاز داریم که کلیدهای خصوصی مرتبط را برای استفاده‌های بعدی ذخیره کنیم زیرا باید هر گونه داده زنجیره‌ای موجود را پاک کرده و یک زنجیره جدید با استفاده از federated peg script راه‌اندازی کنیم. این به این دلیل است که federated peg script بخشی از قوانین اجماع یک Sidechain را تشکیل می‌دهد و نمی‌توان آن را در تاریخ بعدی به یک Blockchain موجود و غیر متصل اعمال کرد.


بنابراین بیایید generate و Address را با هر یک از نودهای خود انجام دهیم، داده‌های مربوطه را برای استفاده بعدی ذخیره کنیم و generate را برای federated peg script انجام دهیم که بعداً برای راه‌اندازی Sidechain خود استفاده خواهیم کرد.


ابتدا نیاز داریم هر یک از نودهای ما، که به عنوان watchmen در شبکه ما عمل خواهند کرد، یک Address جدید را به generate کنند.


```
e1-cli getnewaddress

e2-cli getnewaddress
```


سپس Address را تأیید می‌کنیم تا کلیدهای عمومی را به دست آوریم.


```
e1-cli getaddressinfo <e1-address>

e2-cli getaddressinfo <e2-address>
```


سپس کلیدهای خصوصی مرتبط با هر Address را بازیابی کنید.


```
e1-cli dumpprivkey <e1-address>

e2-cli dumpprivkey <e2-address>
```


کلیدهای خصوصی و عمومی را برای استفاده بعدی ذخیره کنید.


اکنون باید داده‌های موجود Blockchain و Wallet را پاک کنیم زیرا زنجیره جدیدی را با استفاده از federated peg script راه‌اندازی خواهیم کرد. می‌توانید این کار را اکنون انجام دهید. فراموش نکنید که Bitcoin و daemon را راه‌اندازی کنید، که برای peg-in به آن‌ها نیاز خواهیم داشت.


اکنون می‌توانیم یک زنجیره جدید با یک federated peg script که با استفاده از کلیدهای عمومی که قبلاً ذخیره کرده‌ایم ایجاد شده است، مقداردهی اولیه کنیم. اعدادی که وارد می‌کنیم و کلیدهای عمومی ما را احاطه می‌کنند، تعداد کلیدهای استفاده‌شده و کلید Ownership را که باید برای ورود و خروج از Sidechain ما اثبات شود، تعریف و محدود می‌کنند.


```
e1-dae -fedpegscript=5221<e1-pubkey>21<e2-pubkey>52ae

e2-dae -fedpegscript=5221<e1-pubkey>21<e2-pubkey>52ae
```


اکنون ما کلیدهای خصوصی که قبلاً ذخیره کرده‌ایم را وارد خواهیم کرد، تا گره‌های ما بتوانند بعداً انتقال دارایی‌ها بین زنجیره‌ها را امضا و تکمیل کنند و الزامات federated peg script را برآورده سازند.


```
e1-cli importprivkey <priv-key-1>

e2-cli importprivkey <priv-key-1>
```


ما اکنون نیاز داریم که برخی از بلوک‌ها را در هر دو زنجیره بالغ کنیم. بلوغ بلوک‌ها یک نیازمندی از فرآیند پگ است زیرا از سازماندهی مجدد بلوک‌ها در mainchain جلوگیری می‌کند که منجر به [تورم](https://planb.academy/resources/glossary/inflation) pegged asset Supply درون Sidechain می‌شود.


برای متمرکز نگه‌داشتن این بخش بر روی federated peg، ما بلوک‌ها را بدون استفاده از مدل block signing که در بخش قبلی بررسی کردیم، تولید خواهیم کرد و به استفاده از فرمان 'generate' برای ایجاد بلوک‌های جدید بازخواهیم گشت.


```
b-cli generate 101

e1-cli generate 1
```


ما لزوماً نیازی به بلوک‌های generate برای Elements نداریم. اما بیایید به هر حال یک generate انجام دهیم. این کار خوبی است برای جلوگیری از ناسازگاری‌های احتمالی.


اکنون زنجیره ما آماده برای peg-in است. برای انجام peg-in، نیاز داریم تا یک نوع خاص از Address را با استفاده از فرمان getpeginaddress به generate تبدیل کنیم. توجه داشته باشید که مدت زمان بین تولید یک peg-in Address با getpeginaddress و ادعای آن با claimpegin باید تا حد امکان کوتاه نگه داشته شود. آدرس‌های peg-in به صورت بلندمدت پایدار نیستند و نباید مجدداً استفاده شوند.


```
e1-cli getpeginaddress
```


می‌توانید ببینید که این فرمان یک mainchain Address جدید ایجاد می‌کند، و همچنین یک اسکریپت جدید که برای مطالبه وجوه peg-in نیاز به برآورده شدن دارد. mainchain Address یک `pay to script Hash` Address است که توسط کارگزارانی که نقش watchmen را در شبکه Elements ایفا می‌کنند، استفاده خواهد شد.


مانند getnewaddress، getpeginaddress یک راز جدید به Wallet نود فراخوان اضافه می‌کند، بنابراین مهم است که پشتیبان‌گیری از فایل Wallet را در فرآیند مدیریت نود خود لحاظ کنید.


ما اکنون مقداری Bitcoin را از mainchain به Sidechain ارسال خواهیم کرد. تست رگرسیون mainchain ما، Wallet، در حال حاضر مقداری بودجه دارد.


```
b-cli getwalletinfo
```


می‌توانیم ببینیم که Wallet شامل 50 Bitcoin است. ما یک Bitcoin از mainchain به Sidechain ارسال خواهیم کرد. ما نیاز داریم که وجوه را به mainchain Address که گره ما قبلاً تولید کرده است، ارسال کنیم.


```
b-cli sendtoaddress <e1-pegin-address>
```


ما باید شناسه این تراکنش را نگه داریم زیرا بعداً برای اثبات تأمین مالی به آن نیاز داریم.


اکنون می‌توانیم ببینیم که تراز mainchain Wallet به میزان مبلغی که ارسال کردیم کاهش یافته است، به‌علاوه یک مقدار کوچک اضافی برای پوشش هزینه‌های تراکنش.


```
b-cli getwalletinfo
```


ما باید دوباره تراکنش را بالغ کنیم.


```
b-cli generate 101
```


برای اینکه گره Elements ما بتواند وجوه peg-in را مطالبه کند، باید `اثبات` انجام تراکنش peg-in را به دست آوریم. اثبات رمزنگاری از تامین مالی transaction ID برای محاسبه مسیر مرکل استفاده می‌کند و ثابت می‌کند که تراکنش در یک بلوک تایید شده وجود دارد.


```
b-cli gettxoutproof '["<tx-id>"]'
```


ما همچنین به داده‌های خام تراکنش نیاز داریم.


```
b-cli getrawtransaction <tx-id>
```


با داشتن اثبات و داده‌های خام برای تراکنش peg-in، نود Elements ما اکنون می‌تواند با استفاده از تراکنش خام و اثبات تراکنش، peg-in را ادعا کند.


```
e1-cli claimpegin <raw> <proof>
```


توجه داشته باشید که یک آرگومان سوم اختیاری وجود دارد که می‌توانستیم به claimpegin ارائه دهیم. از این پارامتر سوم می‌توان برای مشخص کردن Sidechain Address که وجوه ادعا شده به آن ارسال می‌شود، استفاده کرد. در مثال ما نیازی به این کار نبود زیرا ما فرمان را از همان نودی که مالک Address است و وجوه ادعا شده به آن می‌رود، فراخوانی می‌کردیم.


بررسی موجودی e1.


```
e1-cli getwalletinfo
```


تولید یک بلوک برای تأیید ادعا.


```
e1-cli generate 1
```


بررسی موجودی e1.


```
e1-cli getwalletinfo
```


ما می‌توانیم ببینیم که ورود موفقیت‌آمیز ادعا شده است.


برای خروج، فرآیند مشابه است. به این صورت که یک Address تولید می‌شود، وجوه به آن ارسال می‌شود و اگر تراکنش معتبر باشد، وجوه آزاد می‌شود. ما کل فرآیند خروج را پوشش نخواهیم داد زیرا شامل کار بر روی mainchain است که فراتر از محدوده این دوره است. مراحل از نظر رویدادهای Elements این است که یک Address بر روی mainchain تولید می‌شود.


```
b-cli getnewaddress
```

```

```


در حال تولید یک بلوک برای تأیید تراکنش.


```

```


موجودی گره Wallet را بررسی کنید.


```

```


و ببینید که موجودی کاهش یافته است.


در این بخش دیده‌ایم که چگونه:


- generate و federated peg script.
- یک زنجیره جدید را که از اسکریپت به عنوان قانون پارامتر اجماع شبکه استفاده می‌کند، مقداردهی اولیه کنید.
- ارسال وجوه از mainchain به Sidechain.
- مطالبۀ وجوه درون Elements Sidechain.
- درک کنید که چگونه ارسال وجوه به mainchain آغاز می‌شود.


### فدریتدپگ‌اسکریپت



برای اینکه Elements به عنوان Sidechain کار کند، [بلوک Genesis](https://planb.academy/resources/glossary/genesis-block) در Blockchain آن باید با یک `fedpegscript` ایجاد شود. این کار با وارد کردن پارامتر `fedpegscript` در زمان راه‌اندازی نود انجام می‌شود. سپس این اسکریپت بخشی از قوانین اجماع Elements Blockchain را تشکیل می‌دهد و اجازه می‌دهد درخواست‌های peg-in و peg-out تأیید و اجرا شوند.


`fedpegscript` از کلیدهای عمومی تشکیل شده است که توسط افرادی که مجاز به انجام اقدامات peg هستند کنترل می‌شود. در زیر فرمت نمونه‌ای از یک fedpegscript چند امضایی ۲ از ۲ نشان داده شده است:


```
fedpegscript=5221<public key 1>21<public key 2>52ae
```


توجه: کاراکترهای خارج از کلیدهای عمومی، جداکننده‌هایی هستند که الزامات کلید عمومی و `n از m` را نشان می‌دهند. به عنوان مثال، قالب یک fedpegscript از نوع 1-of-1 به صورت '5121<کلید عمومی 1>51ae' خواهد بود.


### پگ‌این


قبل از اینکه یک peg-in توسط یک Elements Sidechain پذیرفته شود، باید تأییدیه‌های کافی بر روی mainchain داشته باشد. این امر برای جلوگیری از تورم در Supply از pegged asset بر روی Elements Sidechain ضروری است که ممکن است به دلیل سازماندهی مجدد mainchain ایجاد شود.


انتظار می‌رود که سازماندهی‌های کوتاه در نوک Bitcoin Blockchain به عنوان بخشی از عملکرد عادی مکانیزم اجماع Proof of Work (PoW) رخ دهد. به همین دلیل، Elements تنها زمانی یک peg-in را معتبر می‌پذیرد که به عمق کافی درون Bitcoin Blockchain رسیده باشد. این کار برای جلوگیری از پذیرش چندباره همان peg-in توسط Elements انجام می‌شود.


### پگ-اوت


هنگامی که یک گره Elements فرمان `sendtomainchain` را فراخوانی می‌کند، یک peg-out رخ می‌دهد که به عنوان ورودی یک mainchain Address (مقصد peg-out) و همچنین مقدار pegged asset برای `برداشت` را می‌گیرد. این کار یک تراکنش peg-out بر روی Sidechain ایجاد می‌کند. هنگامی که Functionaries که به عنوان watchmen عمل می‌کنند تشخیص می‌دهند که تراکنش peg-out بر روی Sidechain تأیید شده است، آن‌ها مسئولیت آزادسازی واقعی دارایی بر روی mainchain به مقصد peg-out را بر عهده می‌گیرند، همان‌طور که در بخش‌های قبلی دوره یاد گرفتیم.


## Elements به عنوان یک Blockchain مستقل

تا کنون، ما بررسی کرده‌ایم که چگونه Elements را به عنوان یک Sidechain اجرا کنیم. با این حال، این می‌تواند به عنوان یک راه‌حل مستقل Blockchain با دارایی بومی پیش‌فرض خود نیز عمل کند. در این تنظیمات، یک Elements Blockchain همچنان تمامی ویژگی‌های پیاده‌سازی Sidechain، مانند Confidential Transactions و Issued Assets را حفظ می‌کند، اما نیازی به peg-in یا peg-out برای اضافه یا حذف مقادیر default asset از گردش ندارد.


در این بخش ما:


یک Elements Blockchain جدید با یک default asset به نام `newasset` مقداردهی اولیه کنید.


ایجاد 1,000,000 `newasset` به همراه 2 توکن بازانتشار برای آن مشخص کنید.


تمام سکه‌های anyone-can-spend `newasset` را مطالبه کنید.


تمام توکن‌های صدور مجدد anyone-can-spend را برای 'newasset' مطالبه کنید.


دارایی و reissuance token آن را به Wallet یک گره دیگر ارسال کنید.


صدور مجدد 'newasset' از هر دو نود.


برای راه‌اندازی یک شبکه Elements به‌گونه‌ای که به‌صورت مستقل به‌عنوان Blockchain عمل کند، لازم است هر گره با برخی پارامترهای پایه شروع شود. این پارامترها برای اطلاع دادن به گره استفاده می‌شوند تا تلاش نکند و اعتبارسنجی peg-ins از یک Blockchain دیگر را انجام دهد، نام default asset شبکه و مقدار default asset و reissuance token مرتبط برای ایجاد را مشخص کند.


ما اکنون یک زنجیره جدید با استفاده از این پارامترها بر روی دو گره متصل Elements خود شروع خواهیم کرد. ما default asset را `newasset` نامگذاری خواهیم کرد و یک میلیون از آنها و دو توکن بازصدور `newasset` صادر خواهیم کرد.


```
e1-dae -validatepegin=0 -defaultpeggedassetname=newasset -initialfreecoins=100000000000000 -initialreissuancetokens=200000000

e2-dae -validatepegin=0 -defaultpeggedassetname=newasset -initialfreecoins=100000000000000 -initialreissuancetokens=200000000
```


توجه داشته باشید که مبالغ استفاده شده در اینجا به کوچک‌ترین واحدی است که شبکه می‌تواند بپذیرد، بنابراین دویست میلیون توکن بازانتشار در واقع معادل دو توکن کامل است. همین امر در مورد واحد سکه‌های اولیه رایگان نیز صدق می‌کند.


موجودی‌های فعلی Wallet گره ما را بررسی کنید.


```
e1-cli getwalletinfo

e2-cli getwalletinfo
```


ما می‌توانیم ببینیم که مقداردهی اولیه به درستی کار کرد.


از آنجا که صدور اولیه دارایی‌ها به صورت `هر کسی می‌تواند خرج کند` ایجاد می‌شود، ما e1 را وادار می‌کنیم تا همه آنها را مطالبه کند تا بتوانیم دسترسی e2 به آنها را حذف کنیم.


```
e1-cli getnewaddress

e1-cli sendtoaddress <e1-address> 1000000 "" "" true
```


توجه داشته باشید که نیازی به مشخص کردن 'newasset' به عنوان دارایی برای ارسال نیست زیرا این دارایی قبلاً default asset است و بنابراین همچنین default asset برای پرداخت هزینه‌های شبکه استفاده می‌شود.


در Elements، شما می‌توانید انواع دارایی‌های مختلف را به همان Address ارسال کنید، بنابراین می‌توانیم از Address که به تازگی تولید کرده‌ایم برای دریافت default asset استفاده کنیم و از آن به عنوان مقصد Address برای توکن‌های بازانتشار استفاده کنیم.


```
e1-cli sendtoaddress <e1-address> 1 "" "" false false 1 UNSET false <reissuance-token-id>
```


تراکنش‌ها را تأیید کنید.


```
e1-cli generate 101
```


ما بررسی خواهیم کرد که e1 تنها دارنده دارایی و reissuance token آن است.


```
e1-cli getwalletinfo

e2-cli getwalletinfo
```


که می‌توانیم ببینیم واقعاً همین‌طور است.


اکنون مقداری از 'newasset' را به e2 ارسال خواهیم کرد، که در حال حاضر موجودی صفر دارد.



توجه داشته باشید که نیازی به مشخص کردن نوع دارایی برای ارسال نبود، زیرا `newasset` به عنوان default asset شبکه ایجاد شده است.


بیایید همچنین برخی از توکن‌های صدور مجدد برای `newasset` را به e2 ارسال کنیم.


```
e1-cli sendtoaddress <e2-address> 1 "" "" false false 1 UNSET false <reissuance-token-id>
```


تراکنش‌ها را تأیید کنید.


```
e1-cli generate 101
```


ما می‌توانیم بررسی کنیم که کیف پول‌ها به‌طور مناسب به‌روزرسانی شده‌اند.


```
e1-cli getwalletinfo

e2-cli getwalletinfo
```


اکنون برخی از default asset را از e1 مجدداً صادر خواهیم کرد. توجه داشته باشید که توانایی انجام این کار توسط پارامتر راه‌اندازی initialreissuancetokens فعال می‌شود. که اگر حذف شود یا روی صفر تنظیم شود، منجر به default asset می‌شود که نمی‌تواند در تاریخ بعدی مجدداً صادر شود.


```
e1-cli reissueasset newasset 100
```


ما توانستیم از برچسب `newasset` به جای ارائه مقدار شناسه هگز استفاده کنیم زیرا یک زنجیره Elements همیشه default asset خود را برچسب‌گذاری می‌کند.


بررسی اینکه صدور مجدد default asset کار کرد:


```
e1-cli generate 101

e1-cli getwalletinfo
```


اکنون ثابت خواهیم کرد که چون e2 دارای برخی از توکن‌های صدور مجدد `newasset` است، می‌تواند default asset را نیز مجدداً صادر کند.


```
e2-cli reissueasset newasset 100
```


بررسی اینکه صدور مجدد default asset توسط e2 کار کرد.


```
e2-cli generate 101

e2-cli getwalletinfo
```


در این بخش، ما Elements را به عنوان یک Blockchain مستقل تنظیم کرده‌ایم و بررسی کرده‌ایم که عملکرد پایه‌ای همان‌طور که انتظار داشتیم کار می‌کند.


ما از پارامترهای استارتاپ برای:


یک Elements Blockchain جدید با یک default asset به نام 'newasset' مقداردهی اولیه کنید.


مقدار default asset را برای ایجاد اولیه‌سازی on chain مشخص کنید.


توکن‌های تجدید انتشار برای default asset ایجاد کنید و مقدار بیشتری از default asset را از هر دو نود تجدید انتشار کنید.


در شبکه مستقل Blockchain Elements ما، تمامی عملیات تراکنشی دیگر به همان شیوه‌ای که در مثال‌های پوشش داده شده در بخش‌های اصلی دوره آمده است، عمل خواهند کرد، اما به جای `Bitcoin` از 'newasset' به عنوان دارایی پیش‌فرض و کارمزد استفاده خواهند کرد.


### پارامترهای راه‌اندازی نود و مقداردهی اولیه زنجیره


برای اینکه به یک گره Elements بگوییم به عنوان یک Blockchain مستقل عمل کند، باید چند پارامتر با هم استفاده شوند. اکنون به هر یک نگاهی می‌اندازیم و متوجه می‌شویم که چه کاری انجام می‌دهند.


#### `validatepegin=0`

به عنوان یک Blockchain مستقل نیازی به اعتبارسنجی هیچ تراکنش peg-in یا peg-out ندارد، ما باید این بررسی‌ها را غیرفعال کنیم. با این تنظیمات، نیازی به اجرای نرم‌افزار کلاینت Bitcoin یا ذخیره یک نسخه از Bitcoin Blockchain ندارید، زیرا شبکه Elements به صورت مستقل عمل خواهد کرد.


#### `defaultpeggedassetname`

این به شما اجازه می‌دهد نام default asset ایجاد شده در هنگام راه‌اندازی Blockchain را مشخص کنید.


#### `initialfreecoins`

تعداد (معادل واحد Bitcoin در واحد Satoshi) از default asset برای ایجاد.


#### `initialreissuancetokens`

تعداد (معادل واحد Bitcoin در واحد Satoshi) توکن‌های صدور مجدد برای ایجاد default asset. بدون این، ایجاد بیشتر default asset غیرممکن خواهد بود. اگر نمی‌خواهید امکان ایجاد بیشتر default asset وجود داشته باشد، می‌توانید این مقدار را به صفر تنظیم کنید یا آن را حذف کنید.


با استفاده از این پارامترها، معمولاً شروع یک نود به این شکل خواهد بود:


```
e1-dae -validatepegin=0 -defaultpeggedassetname=newasset -initialfreecoins=100000000000000 -initialreissuancetokens=200000000
```


### عملیات پایه


پارامتر `defaultpeggedassetname` یک برچسب به default asset اعمال می‌کند. بدون این تنظیم، default asset به‌طور خودکار به نام `Bitcoin` نام‌گذاری خواهد شد. در بخش‌های قبلی، زمانی که دارایی‌هایی را که خودمان صادر کرده بودیم به یک نود دیگر ارسال می‌کردیم، باید یا هگز دارایی یا برچسب دارایی محلی اعمال‌شده را مشخص می‌کردیم تا به Elements بگوییم کدام دارایی را ارسال می‌کنیم. از آنجا که `defaultpeggedassetname` در تمام نودها اعمال می‌شود، نیازی به نام‌گذاری آن هنگام ارسال نداریم، حتی اگر نام آن `Bitcoin` نباشد. هر تابعی که قبلاً به‌طور پیش‌فرض `Bitcoin` را ارسال می‌کرد، اکنون هر چیزی را که برای برچسب‌گذاری default asset انتخاب کرده‌اید ارسال خواهد کرد.


بنابراین ارسال 10 عدد از default asset جدید به یک Address به سادگی:


```
e1-cli sendtoaddress <destination address> 10 "" "" true
```


اگر شما همچنین به نود مقداری برای `initialreissuancetokens` بیشتر از صفر داده‌اید، در این صورت قادر خواهید بود که مقدار بیشتری از default asset را مجدداً صادر کنید، چیزی که در صورت اجرای Elements به عنوان Sidechain امکان‌پذیر نیست.


برای انجام این کار، هر گره‌ای که مقداری از token مرتبط با default asset را نگه می‌دارد، فقط نیاز دارد دستوری به شکل زیر صادر کند:


```
e1-cli reissueasset <default asset name> <amount>
```


با استفاده از پارامترهای فوق، می‌توانید Elements را به‌عنوان یک Blockchain مستقل با default asset خود، جدا از Bitcoin و سایر بلاک‌چین‌ها، راه‌اندازی کنید.
