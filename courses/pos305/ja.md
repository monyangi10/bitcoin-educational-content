### BTCPayサーバーとの連携


このコースブロックの目的は、BTCPay Server ソフトウェアの一般的な理解を得ることです。共有環境では、講師のデモンストレーションに従い、BTCPay Server コースブックを参照し、講師と一緒に進めていくことをお勧めします。Walletを複数の方法で作成する方法を学びます。例として、Hot Wallet セットアップや BTCPay Server Vault を通して接続されたハードウェアウォレットがあります。これらの目標は、コースのインストラクターによって表示され、アクセスできるデモ環境で発生します。


このコースに自分で参加する場合、https://directory.btcpayserver.org/filter/hosts にデモ用のサードパーティホストのリストがあります。これらのサードパーティオプションを本番環境として使用しないことを強くお勧めします。しかし、これらはBitcoinとBTCPay Serverの使用を紹介するための適切な目的を果たします。


BTCPay Server rockstarの研修生として、Bitcoinノードをセットアップした経験があるかもしれません。このコースは、特にBTCPayサーバーソフトウェアスタックに合わせています。


BTCPay Serverのオプションの多くは、他のBitcoin Wallet関連のソフトウェアにも何らかの形で存在する。


### BTCPayサーバーログイン画面


デモ環境にようこそ！」と歓迎されると、「ログイン」または「アカウントの作成」を求められます。サーバー管理者は、セキュリティ上の理由から新しいアカウントを作成する機能を無効にすることができます。BTCPayサーバーのロゴやボタンの色は、BTCPayサーバーがオープンソースソフトウェアであるため、変更することができます。サードパーティのホストは、ソフトウェアをホワイトラベル化し、全体の外観を変更することができます。


![image](assets/en/001.webp)


### アカウント作成ウィンドウ


BTCPay サーバーでアカウントを作成するには、有効な電子メール Address 文字列が必要です。電子メールには example@email.com が有効な文字列となります。


パスワードは、英字、数字、文字を含めて最低8文字必要です。パスワードを一度設定した後、そのパスワードが最初のパスワードフィールドに入力されたものと同じであることを確認する必要があります。


Eメールアドレスとパスワードの両方のフィールドが正しく入力されたら、「アカウントを作成」ボタンをクリックします。これで、講師のBTCPayサーバーインスタンスにメールアドレスとパスワードが保存されます。


![image](assets/en/002.webp)


**注


このコースを単独で受講する場合、このアカウントの作成はサードパーティのホスト上で行われる可能性が高いので、これらを本番環境として使用せず、トレーニング目的のみに使用することを再度強調します。


### BTCPayサーバー管理者によるアカウント作成


BTCPayサーバーインスタンスの管理者は、BTCPayサーバーのアカウントを作成することもできます。BTCPayサーバーインスタンスの管理者は、'Server Settings'をクリックし(1)、'Users'タブをクリックし(2)、Usersタブの右上にある "+ Add User "ボタン(3)をクリックします。目的(4.3)では、Accounts の管理者コントロールについて詳しく説明します。


![image](assets/en/003.webp)


管理者として、ユーザーの電子メール Address が必要であり、標準パスワードを設定する。管理者は、セキュリティ上の理由から、アカウントを使用する前にこのパスワードを変更するようユーザーに通知することを推奨する。管理者がパスワードを設定せず、SMTPがサーバーに設定されている場合、ユーザーは、アカウントを作成し、自分自身でパスワードを設定するための招待リンクが記載された電子メールを受け取ります。


### 例


インストラクターと一緒にコースを受講する場合、インストラクターが提供するリンクに従い、デモ環境でアカウントを作成してください。あなたのEメールAddressとパスワードが安全に保存されていることを確認してください。このコースの残りのデモ目標には、これらのログイン認証情報が必要です。


インストラクターは、この練習の前に、前もってEメールのAddressを集め、招待リンクを送っているかもしれません。指示があれば、Eメールを確認してください。

### Bitcoinキーの生成には何が関係しているのですか？


Bitcoinウォレットを作成すると、いわゆる「seed」を作成します。この前に生成された一連の単語は、Mnemonicフレーズとしても知られています。seed は個々の Bitcoin キーを導き出すために使用され、Bitcoin を送受信するために使用されます。seed フレーズは決して第三者や信頼できないピアと共有してはならない。


seedの生成は、「階層的決定論的（HD）」フレームワークとして知られる業界標準に従って行われる。


![image](assets/en/016.webp)


### 住所


BTCPayサーバーは、新しいAddressをgenerateするように構築されています。これにより、公開鍵やAddressの再利用の問題が軽減されます。同じ公開鍵を使用することで、支払い履歴全体の追跡が非常に簡単になります。鍵を一度しか使えないバウチャーと考えれば、プライバシーは大幅に改善されるでしょう。Bitcoinアドレスも使用していますが、公開鍵と混同しないでください。


Addressは、「ハッシュ・アルゴリズム」によって公開鍵から導き出される。しかし、ほとんどのウォレットやトランザクションでは、公開鍵ではなくアドレスが表示されます。一般的に、アドレスは公開鍵よりも短く、通常 `1`、`3`、`bc1` で始まるのに対し、公開鍵は `02`、`03`、`04` で始まる。



- 1......`で始まるアドレスは、今でも非常に一般的なアドレスです。新しいストアの作成」の章で述べたように、これらはレガシーアドレスである。このAddressタイプはP2PKHトランザクション用である。P2PkhはBase58エンコーディングを使用するため、Addressは大文字小文字を区別する。Addressの構造は、公開鍵に識別子として1桁追加したものである。



- bc1...`で始まるアドレスは、徐々にごく一般的なアドレスに移行しつつある。これらは（ネイティブ）SegWitアドレスとして知られている。これらのアドレスは、他のSegWitアドレスよりも優れた料金体系を提供しています。ネイティブSegWitアドレスはBech32エンコーディングを使用し、小文字しか使えません。



- 3...`で始まるアドレスは、取引所では預け入れアドレスとして今でも一般的に使用されている。これらのアドレスは、「新しいストアを作る」の章で述べた、ラップされた、あるいはネストされたSegWitアドレスである。しかし、これらは「Multisig Address」としても機能する。SegWit Address として使用する場合、取引手数料の節約になるが、これもネイティブ SegWit よりは少ない。P2SHアドレスはBase58エンコーディングを使用する。このため、レガシーAddressのように大文字と小文字が区別されます。



- 2...`で始まるアドレスはTestnetアドレスです。これらはTestnet Bitcoin (tBTC)を受け取るためのものです。これらのアドレスにBitcoinを送信することは絶対に避けてください。開発目的の場合は、Testnet Wallet を generate にすることができます。オンライン上にはTestnet Bitcoinを入手するための複数の蛇口があります。決してTestnet Bitcoinを購入しないでください。Testnet Bitcoinは採掘される。これは開発者が代わりにRegtestを使う理由になるかもしれない。これは開発者のための遊び場の環境であり、特定のネットワークコンポーネントが欠けています。しかし、Bitcoinは開発目的にはとても便利です。


### 公開鍵


現在、公開鍵はあまり使われていない。長い間、Bitcoinのユーザーは公開鍵をアドレスに置き換えてきた。公開鍵はまだ存在し、時々使われる。一般的に、公開鍵はアドレスよりもずっと長い文字列である。アドレスと同様に、公開鍵は特定の識別子で始まる。



- まず、`02...`と`03...`は、SEC形式でエンコードされた非常に標準的な公開鍵識別子である。これらを処理して受信用のアドレスにしたり、multi-sigアドレスの作成に使用したり、署名を検証したりすることができる。初期のBitcoinトランザクションでは、P2PKトランザクションの一部として公開鍵が使用された。



- しかし、HDウォレットは異なる構造を使用する。xpub...`、`ypub...`、`zpub...`は拡張公開鍵（xpub）と呼ばれる。これらの鍵は、HD Walletの一部として多くの公開鍵を導出するために使用される。xpubには、過去と未来の取引を意味するあなたの全履歴の記録が保存されているため、これらを信頼できない相手と決して共有しないでください。

### Bitcoinキーの保管


seedフレーズを生成した後、本書で生成される12～24単語のリストは、Walletへのアクセスを回復する唯一の方法であるため、適切なバックアップとセキュリティが必要です。HDウォレットの構造と、単一のseedを使用してアドレスを決定論的に生成する方法は、seedまたはリカバリーフレーズを表すMnemonic単語のこの1つのリストを使用して、作成されたすべてのアドレスがバックアップされることを意味します。


リカバリーフレーズを安全に保管してください。もし誰かが、特に悪意を持ってアクセスすれば、彼らはあなたの資金を動かすことができます。seedを安全に保管すること、同時にそれは相互のものであることを忘れないこと。Bitcoin秘密鍵の保管にはいくつかの方法があり、セキュリティ、プライバシー、利便性、物理的な保管など、それぞれに利点と欠点がある。秘密鍵の重要性から、Bitcoin の利用者は、銀行のような "保管 "サービスを利用するよりも、"自己保管 "で鍵を保管し、安全に保管する傾向がある。ユーザーによっては、Coldストレージ・ソリューションかHot Walletのどちらかを使用しなければならない。


### BitcoinキーのHotおよびCold保管


通常、BitcoinウォレットはHot WalletまたはCold Walletで表示されます。ほとんどのトレードオフは、利便性、使いやすさ、セキュリティ・リスクにある。これらの方法はそれぞれ、カストディアン・ソリューションにも見られる。しかし、ここでのトレードオフはほとんどセキュリティとプライバシーに基づくものであり、このコースの範囲を超えている。


### Hot Wallet


Hotウォレットは、モバイル、ウェブ、またはデスクトップ・ソフトウェアを通じてBitcoinとやりとりする最も便利な方法です。Walletは常にインターネットに接続されており、ユーザーはBitcoinを送受信することができる。しかし、これは弱点でもあります。Walletは常にオンラインであるため、ハッカーやマルウェアによる攻撃をデバイス上で受けやすくなっています。BTCPay Serverでは、Hotウォレットはインスタンス上に秘密鍵を保存します。BTCPay Serverストアにアクセスする者は、悪意があればこのAddressから資金を盗む可能性があります。BTCPay Server がホスト環境で実行される場合、常にセキュリティプロファイルでこれを考慮し、そのような場合には Hot Wallet を使用しないことが望ましいです。BTCPay Serverがあなたによって所有され、保護されたハードウェアにインストールされている場合、リスクプロファイルは大幅に下がりますが、完全になくなることはありません。


### Cold Wallet


BitcoinをCold Walletに移行することで、秘密鍵をインターネットから隔離し、潜在的なオンラインの脅威から守ることができるからです。方程式からインターネット接続を取り除くことで、マルウェア、スパイウェア、SIMスワップのリスクを減らすことができる。Coldストレージは、Bitcoin秘密鍵の紛失を防ぐために十分な予防措置が取られれば、セキュリティと自律性においてHotストレージより優れていると考えられている。Coldストレージは、Walletのセットアップが複雑なため、頻繁に使用することを想定していない大量のBitcoinに最も適している。


Bitcoinの鍵をColdのストレージに保存する方法は、ペーパーウォレット、ブレインウォレット、ハードウェアウォレット、あるいは最初からWalletファイルなど様々である。ほとんどのウォレットは、seedフレーズをgenerateするためにBIP 39を使用する。しかし、Bitcoin coreソフトウェアの中では、その使用についてまだコンセンサスが得られていません。Bitcoin coreソフトウェアはWallet.datファイルをgenerateします。

### トランザクション


管理者は、トランザクションビューで、この特定のストアに接続されている On-Chain Wallet の送受信トランザクションを確認できます。各トランザクションには、受信金額と送信金額の区別があります。受信トランザクションは Green となり、送信トランザクションは赤色となります。BTCPay サーバーのトランザクション・ビューでは、管理者は標準ラベルのセットも見ることができます。



| 取引タイプ | 説明                                               |
| ------------ | -------------------------------------------------- |
| アプリ        | アプリで作成された請求書を通じて支払いが受領されました |
| 請求書        | 請求書を通じて支払いが受領されました               |
| Payjoin       | 未払い、請求書のタイマーはまだ期限切れではありません |
| Payjoin公開   | 請求書のPayjoin提案を通じてUTXOが公開されました    |
| 支払いリクエスト | 支払いリクエストを通じて支払いが受領されました |
| 支払い        | 支払いまたは返金を通じて支払いが送信されました     |

### 送信方法


BTCPayサーバーの送信機能は、BTCPayサーバーOn-Chain Walletからトランザクションを送信します。BTCPay サーバーでは、資金を使用するトランザクションに複数の方法で署名することができます。トランザクションは次の方法で署名できます；



- Hardware Wallet
- PSBT対応財布
- HD秘密鍵または回復シード。
- Hot Wallet


#### Hardware Wallet


BTCPay ServerにはHardware Walletのサポートが組み込まれており、サードパーティのアプリやサーバーに情報を漏らすことなく、BTCPay VaultでHardware Walletを使用することができます。BTCPay Server内のHardware Wallet統合により、Hardware Walletをインポートし、デバイス上で簡単な確認を行うだけで、入金された資金を使用することができます。あなたの秘密鍵がデバイスから離れることはなく、すべての資金はあなたのFull nodeに対して検証されるため、データの漏洩はありません。


#### PSBTをサポートするWalletとの署名


PSBT（Partially Signed Bitcoin Transactions）は、完全な署名が必要なBitcoinトランザクションの交換フォーマットです。PSBTはBTCPayサーバーでサポートされており、互換性のあるハードウェアおよびソフトウェアウォレットで署名することができます。


完全に署名されたBitcoinトランザクションの構築は以下のステップを経る：



- PSBTは特定のインプットとアウトプットで構成されるが、シグネチャーはない。
- エクスポートされたPSBTは、このフォーマットをサポートするWalletでインポートすることができます。
- トランザクション・データは、Walletを使用して検査し、署名することができる。
- 署名されたPSBTファイルはWalletからエクスポートされ、BTCPayサーバーでインポートされます。
- BTCPayサーバーが最終的なBitcoinトランザクションを生成します。
- 結果を検証し、ネットワークにブロードキャストする。


#### HD秘密鍵またはMnemonic seedによる署名


BTCPayサーバーを使用する前にWalletを作成した場合、秘密鍵を適切なフィールドに入力することで資金を使用することができます。Wallet>設定にて、AccountKeyPathを適切に設定してください。


#### Hot Walletでの署名


ストアのセットアップ時に新しいWalletを作成し、Hot Walletとして有効にすると、自動的にサーバーに保存されたseedを使用して署名します。


### RBF（Replace-by-fee）


Replace-by-fee (RBF) は Bitcoin プロトコルの機能で、以前にブロードキャストされたトランザクションを（未確認のまま）置き換えることができる。これにより、Walletのトランザクション・フィンガープリントをランダムにしたり、より高い手数料率に置き換えたりして、確認（Mining）の優先順位のキューでトランザクションを上位に移動させることができる。これにより、より高い手数料率の取引が優先されるため、元の取引は効果的に置き換えられ、確認されると元の取引は無効になります（二重消費はありません）。


詳細設定 "ボタンを押すと、RBFのオプションが表示されます。

### Coinセレクション


Coinセレクションはプライバシーを強化する高度な機能で、トランザクションを作成する際に使用するコインを選択することができます。例えば、コンジョインミックスから生まれたばかりのコインで支払うことができます。


Coinセレクションは、Walletラベル機能とネイティブに連動します。これにより、UTXOの管理と支出をよりスムーズにするために、入金された資金にラベルを付けることができます。


BTCPay Serverは、ラベル管理のためにBIP-329をサポートしています。BIP-329をサポートするWalletから移行し、ラベルが設定されている場合、BTCPay Serverはそれらを自動的に認識し、インポートします。サーバーを移行する際には、この情報をエクスポートして新しい環境にインポートすることもできます。

#### 再スキャン


再スキャン機能はBitcoin core 0.17.0の「Scantxoutset」に依存し、設定された派生スキームに属するコインをBlockchainの現在の状態（UTXOセットと呼ばれる）でスキャンします。Wallet再スキャンは、BTCPayサーバーのユーザーが頻繁に遭遇する2つの一般的な問題に対処します。


1.ギャップリミットの問題 - ほとんどのサードパーティーウォレットは、多くのユーザー間でノードを共有するライトウォレットです。ライトウォレットやFull nodeに依存するウォレットは、パフォーマンスの問題を防ぐために、Blockchainで追跡する残高のないアドレスの数（通常は20）を制限しています。BTCPayサーバーは、Invoiceごとに新しいAddressを生成します。上記を念頭に置き、BTCPay Serverが20回連続して未払いの請求書を生成した後、外部のWalletは、新しいトランザクションが発生しなかったと仮定して、トランザクションのフェッチを停止します。外部Walletは、21日、22日などに請求書が支払われると、それらを表示しません。一方、内部的には、BTCPayサーバーのWalletは、かなり高いギャップ制限とともに、生成されたすべてのAddressを追跡します。これは第三者に依存せず、常に正しい残高を表示することができます。

2.ギャップ・リミットの解決策 - [外部/既存の Wallet](https://docs.btcpayserver.org/WalletSetup/#use-an-existing-Wallet)がギャップ・リミットの設定を許可している場合、簡単な解決策はそれを増やすことです。しかし、大半のウォレットはこれを許可していません。現在、ギャップリミット設定をサポートしているウォレットは、Electrum、Wasabi、Sparrow walletだけです。残念ながら、他の多くのウォレットでは問題が発生する可能性があります。最高のユーザーエクスペリエンスとプライバシーのために、外部ウォレットの代わりにBTCPayサーバーの内部Walletの使用を検討してください。


#### BTCPayサーバーは "mempoolfullrbf=1 "を使用しています。


BTCPayサーバーは "mempoolfullrbf=1 "を使用します。BTCPayサーバーのセットアップにデフォルトとしてこれを追加しました。これはBTCPayサーバーのセットアップのデフォルトとして追加されています。mempoolfullrbf=1 "がない場合、顧客がRBFを通知しないトランザクションで支払いを二重に使用した場合、マーチャントは確認後に初めて知ることになります。


管理者はこの設定をオプトアウトしたい場合があります。以下の文字列により、デフォルトの設定を変更することができます。


```
BTCPAYGEN_EXCLUDE_FRAGMENTS="$BTCPAYGEN_EXCL UDE_FRAGMENTS;opt-mempoolfullrbf"
. btcpay-setup.sh -i
```

### 内部ノードの接続（サーバー管理者）


サーバー管理者は、BTCPayサーバーの内部Lightningノードを使用できます。Lightningの実装にかかわらず、内部Lightningノードへの接続は同じです。


以前のセットアップストアに移動し、左メニューの「Lightning」Walletをクリックします。BTCPayサーバーには、内部ノード（デフォルトではサーバー管理者のみ）またはカスタムノード（外部接続）を使用する2つのセットアップ方法があります。サーバー管理者は、"内部ノードを使用 "オプションをクリックすることができます。これ以上の設定は必要ありません。保存」ボタンをクリックすると、「BTC Lightningノードが更新されました」という通知が表示されます。これで、ストアはLightning Networkの機能を取得しました。


### 外部ノードの接続（サーバーユーザー/ストアオーナー）


デフォルトでは、ストアオーナーはサーバー管理者のライトニングノードを使用することはできません。BTCPayサーバーをセットアップする前にストアオーナーが所有していたノード、サーバー管理者によって利用可能にされている場合はLNBankプラグイン、またはAlbyのようなカストディアンソリューションのいずれかの外部ノードに接続する必要があります。


設定済みのストアに移動し、左メニューのウォレットの下にある "Lightning "をクリックします。ストアオーナーはデフォルトで内部ノードを使用することができないため、このオプションはグレーアウトされています。カスタムノードを使用することは、ストアオーナーがデフォルトで使用できる唯一のオプションです。


BTCPay サーバーには接続情報が必要です。プリメイド（またはカストディアンソリューション）は、Lightning の実装に合わせて特別にこの情報を提供します。BTCPay Server内では、ストアオーナーは以下の接続を使用できます；



- TCPまたはUnixdomainsocketconnection経由のC-ライトニング。
- HTTPS経由のライトニングチャージ
- HTTPS経由のエクレア
- RESTプロキシ経由のLND
- REST API経由のLNDhub

## BTCPayサーバー - ストア設定


<chapterId>e8faef7b-278d-550e-a511-bc3a442daf64</chapterId>


BTCPay Serverソフトウェアには、2種類の設定があります。BTCPay Server Store固有の設定、ダッシュボードの下の左のメニューバーにある設定ボタン、そしてメニューバーの一番下、Accountのすぐ上にあるBTCPay Server設定です。BTCPayサーバー固有の設定は、サーバー管理者のみが閲覧できます。


店舗設定は、各設定セットを分類するための多くのタブで構成されています。



- 一般
- 料金
- チェックアウトの外観
- アクセス・トークン
- ユーザー
- 役割
- ウェブフック
- ペイアウト・プロセッサー
- 電子メール

### 一般


一般設定]タブでは、ストアのオーナーはブランディングと支払いのデフォルトを設定します。ストアの初期セットアップ時に、ストア名が設定されますが、これは一般設定のストア名に反映されます。ここで、ストアオーナーはブランディングと一致するウェブサイトを設定し、管理者がデータベースで認識するためのストアIDを設定することもできます。


#### ブランディング


BTCPay ServerはFOSSなので、ストアオーナーは自分のストアに合ったカスタムブランディングができます。ブランドカラーを設定し、ブランドのロゴを保存し、公開/顧客向けページ（請求書、支払いリクエスト、プルペイメント）にカスタムCSSを追加します。


#### 支払い


決済設定では、ストアオーナーはストアのデフォルト通貨（Bitcoinまたは任意の不換紙幣）を設定することができます。


#### 誰でも請求書を作成できるようにする


この設定は、BTCPay Server上の開発者またはビルダーのためのものです。あなたのストアでこの設定を有効にすると、外部からあなたのBTCPay Serverインスタンスで請求書を作成できるようになります。


#### 追加料金（ネットワーク料金）を請求書に追加する


BTCPay内の機能で、Dust攻撃からマーチャントを保護したり、マーチャントが一度に大量のBitcoinを移動する必要がある場合に、後で手数料の高コストが発生しないようにクライアントを保護するためのものです。例えば、顧客が20ドルのInvoiceを作成し、それを部分的に支払い、Invoiceが完全に支払われるまで1ドルを20回支払ったとします。マーチャントがその資金を後で移動させる場合、Miningのコストが増加します。デフォルトでは、BTCPayは、Invoiceが複数のトランザクションで支払われた場合、加盟店のその費用をカバーするために、Invoiceの合計金額に追加のネットワークコストを適用します。BTCPayには、この保護機能をカスタマイズするためのオプションがいくつかあります。ネットワーク費用を適用することができます：



- 顧客が Invoice の支払いを複数回行った場合のみ（上記の例では、顧客が 20$ で Invoice を作成し、1$ を支払った場合、Invoice の支払総額は 19$ + ネットワーク料となる。ネットワーク手数料は最初の支払いの後に適用される)
- 毎回の支払い（初回の支払いも含め、この例では初回の支払いでもすぐに合計20ドル＋ネットワーク手数料となる）
- ネットワーク手数料を追加しない（ネットワーク手数料を完全に無効にする）


Dust取引から保護する一方で、適切に伝えられなければビジネスに悪影響を及ぼす可能性もある。顧客は追加的な質問をしてくるかもしれないし、過剰な請求をしていると思うかもしれない。


#### Invoiceは、全額が支払われなかった場合、失効するのか？


Invoiceタイマーはデフォルトで15分に設定されている。このタイマーは、Bitcoin対フィアットExchangeレートに従ってBitcoin額をロックするため、変動に対する保護メカニズムとして機能する。顧客が定められた期間内にInvoiceを支払わない場合、Invoiceは期限切れとみなされる。Invoice は、取引が Blockchain に表示された時点で（確認がゼロの状態で）「支払われた」とみなされ、加盟店が定義した確認回数（通常は 1～6回）に達した時点で「完了」となります。タイマーは分単位でカスタマイズ可能です。


#### Invoiceの支払額が予想よりX％少なくても、支払われたと考えるか？


顧客が Exchange Wallet を使用して Invoice の代金を直接支払う場合、Exchange は少額の手数料を取る。これは、そのような Invoice が完全に完了したとはみなされないことを意味する。Invoice は "partially paid "と表示されます。加盟店が未払い請求書を受領したい場合、ここでパーセンテージを設定することができます。


### 料金


BTCPay サーバーでは、Invoice が生成されるとき、常に最新で正確な Bitcoin-to-fiat 価格が必要となります。BTCPay Serverで新しいストアを作成する際、管理者は希望する価格ソースを設定するよう求められます。ストアがセットアップされた後、ストアのオーナーはいつでもこのタブで価格ソースを変更することができます。

主にパワーユーザーが使用する。この機能をオンにすると、店舗オーナーは価格設定や課金方法に関するスクリプトを作成することができます。


#### テスト


お好みの通貨ペアを素早くテストできます。この機能には、RESTクエリを介してデフォルトの通貨ペアをチェックする機能も含まれています。

### ウェブフック


BTCPay Server内では、新しい "Webhook "を作るのはとても簡単です。BTCPay Server Store settings - Webhooksタブで、ストアオーナーは "+ Create Webhook "をクリックすることで、簡単に新しいWebhookを作成することができます。Webhookを利用することで、BTCPay Serverはストアに関連するHTTPイベントを他のサーバーやeコマース統合に送信することができます。

### ペイアウト・プロセッサー


ペイアウトプロセッサーは、BTCPayサーバーのペイアウトのコンセプトと連動します。ペイアウトアグリゲーターは、複数のトランザクションをバッチ処理し、一度に送信します。ペイアウトプロセッサーを使用すると、ストアオーナーはバッチされたペイアウトを自動化することができます。BTCPay Serverは自動ペイアウトの2つの方法を提供します：On-Chainとoff-chain（LN）です。


ストアオーナーは、両方のペイアウトプロセッサーを別々にクリックし、設定することができます。店舗オーナーは、On-ChainプロセッサーをX時間に1回だけ実行したいかもしれませんが、off-chainプロセッサーは数分に1回実行するかもしれません。On-Chainの場合、どのブロックに含めるかを設定することもできます。デフォルトでは、これは1（または利用可能な次のブロック）に設定されている。off-chainペイアウト・プロセッサーの設定はインターバル・タイマーだけで、ブロック・ターゲットはないことに注意してください。Lightning Networkの支払いは即時です。


![image](assets/en/063.webp)

![image](assets/en/064.webp)


店舗オーナーは、店舗にHot Walletが接続されている場合のみ、On-Chainプロセッサーを設定することができます。


![image](assets/en/065.webp)


ペイアウトプロセッサーを設定した後、BTCPayサーバーストア設定のペイアウトプロセッサータブに戻ることで、すぐに削除または変更することができます。


*注***。


ペイアウトプロセッサーOn-Chain - On-Chainペイアウトプロセッサーは、Hot Walletが接続されているストアでのみ動作します。Hot Walletが接続されていない場合、BTCPayサーバーはWalletキーを保持せず、自動的にペイアウトを処理することができません。

### フォーム

![image](assets/en/070.webp)

#### 高度なカスタムフォーム


BTCPay Serverでは、コードでフォームを構築することもできます。特にJSONです。ストアオーナーはエディターを見る代わりに、エディターのすぐ隣にあるCODEボタンをクリックしてフォームのコードに入ることができます。フィールド定義では以下のフィールドのみを設定できます。フィールドの値は Invoice のメタデータに保存されます：



| フィールド | 説明 |
| --------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| .fields.constant | true の場合、.value をフォーム定義で設定する必要があり、ユーザーはフィールドの値を変更できなくなります。（例：フォーム定義のバージョン） |
| .fields.type | HTML 入力タイプ：text, radio, checkbox, password, hidden, button, color, date, datetime-local, month, week, time, email, number, range, search, url, select, tel |
| .fields.options | .fields.type が select の場合、選択可能な値のリスト |
| .fields.options.text | このオプションに対して表示されるテキスト |
| .fields.options.value | このオプションが選択された場合のフィールドの値 |
| .fields.type=fieldset | 子要素 .fields.fields の周囲に HTML fieldset を作成します（以下を参照） |
| .fields.name | 請求書のメタデータに表示されるフィールドの JSON プロパティ名 |
| .fields.value | フィールドのデフォルト値 |
| .fields.required | true の場合、フィールドは必須になります |
| .fields.label | フィールドのラベル |
| .fields.helpText | フィールドの説明を提供するための追加テキスト。 |
| .fields.fields | フィールドを階層構造で整理し、請求書のメタデータ内に子フィールドをネストさせることができます。この構造により、収集した情報をより適切に整理・管理でき、アクセスや解釈が容易になります。たとえば、顧客情報を収集するフォームがある場合、customer という親フィールドの下に各フィールドをグループ化できます。この親フィールドの中に、name、Email、address といった子フィールドを含めることができます。 |

フィールド名は、Invoice のメタデータにユーザー提供値を格納する JSON プロパティ名を表します。よく知られている名前は、Invoice の設定を調整するために解釈し、変更することができます。



| フィールド名       | 説明                  |
| ---------------- | ---------------------- |
| invoice_amount   | 請求書の金額          |
| invoice_currency | 請求書の通貨          |

フォームのURLに"?your_field=value "のようなクエリー文字列を追加することで、Invoiceのフィールドを自動的にプリフィルすることができます。


この機能の使用例をいくつか紹介しよう：



- ユーザー入力の補助フォームへの入力を容易にするために、既知の顧客情報をあらかじめフィールドに入力しておきます。例えば、顧客のEメールAddressをすでに知っている場合、Eメールフィールドに事前に入力することで、時間を節約することができます。
- パーソナライゼーション：顧客の嗜好やセグメンテーションに基づいてフォームをカスタマイズします。例えば、異なる顧客層がある場合、会員レベルや特定のオファーなどの関連データをフォームに事前に入力することができます。
- トラッキング：隠しフィールドと事前入力値を使用して、顧客の訪問元を追跡します。例えば、各マーケティングチャネル（Twitter、Facebook、Eメールなど）ごとにutm_media値を事前に入力したリンクを作成できます。これは、マーケティング活動の効果を分析するのに役立ちます。
- A/Bテスト：異なるフォームバージョンをテストするために、異なる値でフィールドを事前に入力し、ユーザーエクスペリエンスとコンバージョン率を最適化することができます。

### ポリシー

#### 既存ユーザー設定


ここで利用可能なオプションは、カスタムロールとは別のものです。これらの追加権限は、ストアまたはその所有者を攻撃に対して脆弱にする可能性があります。既存ユーザに追加可能なポリシー



- 非管理者がストアで内部Lightningノードを使用できるようにする。
  - これにより、店舗オーナーはサーバー管理者のLightningノードを使用できるようになり、その結果、管理者の資金も使用できるようになる！これはLightningにアクセスするための解決策ではありませんのでご注意ください。
- 非管理者が自分のストア用にHotウォレットを作成できるようにする。
  - これにより、BTCPayサーバーインスタンスのアカウントを持つ誰もがHot-ウォレットを作成し、管理者のサーバーにリカバリseedを保存することができます。これにより、管理者は第三者の資金を保持する責任を負うことになるかもしれません！
- 非管理者が自分のストア用に Hot ウォレットをインポートできるようにする。
  - Hotウォレットの作成と同様に、このポリシーではHot Walletをインポートすることができますが、Hotウォレットの作成セクションで述べたような危険性があります。


![image](assets/en/075.webp)


#### 新規ユーザー設定


サーバに来る新規ユーザを管理するために、いくつかの重要な設定を行うことができます。新規登録の確認メールを設定したり、ログイン画面からの新規ユーザ作成を無効にしたり、API経由でのユーザ作成への管理者以外のアクセスを制限したりすることができます。



- 登録には確認メールが必要です。
  - サーバー管理者がEメールサーバーをセットアップしている必要があります。
- サーバーでの新規ユーザー登録を無効にする
- ユーザー作成 API エンドポイントへの管理者以外のアクセスを無効にします。


デフォルトでは、BTCPayサーバーは "Disable new user registration on the server "をオンにし、ユーザー作成APIエンドポイントへの管理者以外のアクセスをオフにしています。これはセキュリティのためで、あなたのBTCPayログインを偶然見つけた不特定多数の人がアカウントを作成できないようにするためです。

Invoiceは、売主が買主に代金を回収するために発行する書類である。


BTCPay サーバーでは、Invoice は、固定 Exchange レートで定義された時間間隔内に支払われなければならない文書を表します。請求書には有効期限がありますが、これはExchangeレートを指定された期間内に固定し、受信者を価格変動から保護するためです。


BTCPay Serverの中核は、Bitcoin Invoice管理システムとして機能する機能です。Invoiceは、受け取った支払いを追跡・管理するために不可欠なツールです。


組み込みの[Wallet](https://docs.btcpayserver.org/Wallet/)を使用して手動で支払いを受け取らない限り、ストア内のすべての支払いは請求書ページに表示されます。このページは日付ごとに支払いを累積的にソートし、Invoice管理と支払いのトラブルシューティングのための中心的なリソースとして機能します。

### 一般


#### Invoice ステータス


以下の表は、BTCPayの標準的なInvoiceステータスと、推奨される一般的なアクションを一覧にして説明しています。アクションはあくまで推奨事項です。各ユースケースやビジネスに最適なアクションを定義するのはユーザー次第です。



| 請求書ステータス | 説明 | アクション |
| -------------------------- | --------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------- |
| New | 未払い。請求書タイマーはまだ期限切れではありません | なし |
| New (paidPartial) | 部分的な支払い済み。請求書タイマーはまだ期限切れではありません | なし |
| Expired | 未払い。請求書タイマーが期限切れです | なし |
| Expired (paidPartial) ** | 部分的な支払い済みですが、期限切れです | 購入者に連絡して返金を調整するか、不足分の支払いを依頼してください。オプションで請求書を settled または invalid に設定できます |
| Expired (paidLate) | 請求書タイマーの期限切れ後に全額支払い済み | 購入者に連絡して返金を調整するか、遅延確認が許容される場合は注文を処理してください。 |
| Settled (paidOver) | 請求額以上の支払い済み、決済完了、十分な承認数を受信済み | 購入者に連絡して超過分の返金を調整するか、オプションで購入者からの連絡を待ちます |
| Processing | 全額支払い済みですが、ストア設定で指定された十分な承認数を受信していません | 購入者に連絡して超過分の返金を調整するか、オプションで購入者からの連絡を待ちます |
| Processing (paidOver) | 請求額以上の支払い済みですが、十分な承認数を受信していません | 決済完了まで待ち、その後購入者に超過分の返金を調整するか、オプションで購入者からの連絡を待ちます |
| Settled | 全額支払い済み。ストアで十分な承認数を受信しました | 注文を履行してください |
| Settled (marked) | ステータスが processing または invalid から手動で settled に変更されました | ストア管理者が支払いを settled としてマークしました |
| Invalid* | 支払い済みですが、ストア設定で指定された時間内に十分な承認数を受信できませんでした | ブロックチェーンエクスプローラーで取引を確認し、十分な承認がある場合は settled としてマークしてください |
| Invalid (marked) | ステータスが settled または expired から手動で invalid に変更されました | ストア管理者が支払いを invalid としてマークしました |
| Invalid (paidOver) | 請求額以上の支払い済みですが、ストア設定で指定された時間内に十分な承認数を受信できませんでした | ブロックチェーンエクスプローラーで取引を確認し、十分な承認がある場合は settled としてマークしてください |

#### Invoiceの有効期限は、全額が支払われていない場合......分。


Invoiceのタイマーはデフォルトで15分に設定されている。このタイマーは、暗号対フィアットレートに基づいて暗号通貨額をロックするため、ボラティリティに対する保護メカニズムとして機能する。顧客が定められた期間内にInvoiceを支払わない場合、Invoiceは期限切れとみなされる。Invoiceは、取引がBlockchainに表示された時点で（確認がゼロの状態で）「支払い済み」とみなされ、加盟店が定義した確認回数（通常は1～6回）に達した時点で「完了」とみなされます。タイマーはカスタマイズ可能です。


#### Invoiceは、支払額が予想より.%少なくても支払われたとみなす。

### リクエスト


支払いリクエストは、BTCPayのストアオーナーが長期間の請求書を作成できる機能です。資金は、支払い時に有効なExchangeレートを使用して、支払いリクエストに従って支払われます。これにより、ユーザーは支払い時にストアオーナーとExchangeレートを交渉または確認する必要がなく、都合の良い時に支払いを行うことができます。


ユーザーはリクエストの支払いを部分的に行うことができます。支払いリクエストは、全額支払われるか、ストアオーナーが有効期限を要求するまで有効です。アドレスが再利用されることはありません。ユーザが支払いをクリックするたびに新しいAddressが生成され、支払いリクエスト用のInvoiceが作成されます。


ストアのオーナーは、記録管理や会計のために支払い依頼を印刷（またはInvoiceデータをエクスポート）することができます。BTCPayは、ストアのInvoiceリストで請求書を自動的に支払依頼としてラベル付けします。


#### 支払いリクエストのカスタマイズ



- Invoice 金額 - 要求支払額の設定

- 支払回数 - 一回払いまたは部分払いのみを許可する。
- 有効期限 - 期日まで、または期限なしで支払いを許可する。
- 説明 - テキストエディタ、データテーブル、写真とビデオの埋め込み
- 外観 - CSSテーマによる色とスタイル

### 支払いの引き上げ


従来、受信者は Bitcoin の Address を共有して Bitcoin の支払いを行い、送信者は後でこの Address に送金する。このようなシステムはプッシュペイメントと呼ばれ、受信者が利用できない間に送信者が支払いを開始し、受信者に支払いをプッシュする。


しかし、役割を逆にするのはどうだろう？


もし、送り手が支払いをプッシュする代わりに、送り手が受け手の都合の良い時に支払いをプルできるようにしたらどうだろうか？これがプルペイメントのコンセプトである。これにより、以下のような新しいアプリケーションが可能になる：



- サブスクリプションサービス（契約者が、XX期間ごとにお金を引き出すことを許可するサービス）
- 払い戻し（加盟店は、顧客が適切と判断したときに払い戻し金をWalletに引き出せるようにする。）
- フリーランサーのための時間ベースの課金（雇う側は、時間が報告されるにつれて、フリーランサーが自分のWalletにお金を引き込むことを許可します。）
- パトロネージ（パトロンが毎月資金を引き出すことを許可し、彼らの活動を支援し続ける）
- 自動販売（Exchangeの顧客がExchangeにWalletから資金を引き出して毎月自動的に販売することを許可する。）
- 残高出金システム（大量出金サービスでは、ユーザーが残高からの出金を要求すると、サービスは簡単に一定の間隔で多くのユーザーにすべての支払いをバッチすることができます。）


### 支払い


ペイアウト機能は[Pull Payments](https://docs.btcpayserver.org/PullPayments/)機能に関連しています。この機能により、BTCPay内でペイアウトを作成することができます。この機能により、プルペイメント（払い戻し、給与の支払い、または引き出し）を処理することができます。


#### 例1：払い戻し


返金の例から始めましょう。顧客があなたの店で商品を購入したが、残念ながら返品しなければならなくなった。返金を希望しています。BTCPayの中で、[Refund](https://docs.btcpayserver.org/Refund/)を作成し、資金を請求するためのリンクを顧客に提供することができます。顧客がAddressを提供し、資金を請求すると、それはPayoutsセクションに表示されます。


最初のステータスは「承認待ち」である。店員は複数の承認待ちがあるかどうかを確認することができ、選択した後、アクションボタンを使用する。


アクションボタンのオプション



- 選択した支払いを承認する
- 選択した支払いを承認して送信する
- 選択したペイアウトをキャンセルする


次のステップは、顧客に返金したいので、選択したペイアウトを承認して送信することです。顧客のAddressをチェックし、金額と手数料を返金額から差し引くかどうかを確認します。チェックを終えたら、あとは取引にサインするだけです。


顧客は請求ページで更新されます。顧客は、Block explorerと取引へのリンクが提供されるため、取引を追跡することができます。取引が確認されると、ステータスは「完了」に変わります。


#### 例2：給与


では、給与の支払いについて説明しよう。この支払いは、顧客からのリクエストに従うのではなく、店舗内部から行われる。基本的なコンセプトは同じで、プルペイメントを利用します。しかし、払い戻しの代わりに[Pull Payment](https://docs.btcpayserver.org/PullPayments/)を行います。


BTCPayサーバーのPull Paymentsタブに移動します。右上の「Create Pull Payment」ボタンをクリックします。


ペイアウトの作成に入ります。ペイアウトの名前と、選択した通貨での希望額を入力します。Description"（説明）を記入し、従業員が何の支払いかわかるようにします。次の部分は払い戻しと同様です。従業員は、宛先AddressとこのPayoutから請求したい金額を記入する。2回に分けて請求したり、別の住所に請求したり、部分的に光に請求したりすることもできる。


複数のペイアウトが待機している場合は、一括して署名して送信することができます。署名が完了すると、ペイアウトは進行中タブに移動し、トランザクションが表示されます。ネットワークに承認されると、「完了」タブに移動します。完了]タブは純粋に履歴のためのものです。処理済みのペイアウトとそれに属するトランザクションが保存されます。

### BTCPayサーバー - 販売時点情報管理


#### 寄付金


店舗オーナーは、特典の表示方法、並べ方、他の特典との順位付けなどを選択することができます。しかし、クラウドファンディングの目標額に達した後、ストアオーナーはこのファンドレイザーへの寄付の流れを止めたいと思うかもしれません。そこで、「目標達成後の追加寄付を許可しない」をオンにすることができる。これにより、クラウドファンディングは寄付を受け付けなくなる。


##### クラウドファンディングの行動


Crowdfundの標準では、Crowdfundで作成された請求書のみがゴールにカウントされます。しかし、ストアのオーナーは、このストアで作成されたすべての請求書をクラウドファンドにカウントしたい場合があるかもしれません。


#### カスタマイズのための追加オプション


BTCpay Serverでは、いくつかのカスタマイズが可能です。クラウドファンディングにサウンド、アニメーション、ディスカッションスレッドを追加することができます。ストアオーナーは、独自のカスタムCSSを入力することで、クラウドファンドのルック＆フィールを変更することもできます。


#### このアプリを削除する


店舗オーナーがBTCPayサーバーからクラウドファンドを完全に削除したい場合、クラウドファンドの更新の一番下にある「このアプリを削除」ボタンをクリックすることで、クラウドファンドアプリを完全に削除することができます。Delete this app "をクリックすると、BTCPayサーバーは "DELETE`"と入力し、"Delete "ボタンをクリックして確認します。削除後、ストアオーナーはBTCPayサーバーのダッシュボードに戻ります。


### BTCPayサーバー - 支払いボタン


簡単に埋め込み可能なHTMLと高度にカスタマイズ可能な支払いボタンにより、ストアオーナーはチップや寄付を受け取ることができます。BTCPayサーバーの左メニューバー、プラグインセクションの下にある "Pay Button "をクリックし、"Enable "をクリックすると支払いボタンを作成することができます。


#### 一般設定


支払いボタンの一般設定内で、ストアオーナーは以下を設定できます。



- 標準価格
- デフォルト通貨
- デフォルトの支払い方法
  - 店舗のデフォルトを使用する
  - BTC On-Chain
  - BTC off-chain（ライトニング）
  - BTC off-chain (LNURL-pay)
- チェックアウトの説明
- オーダーID


#### 表示オプション


BTCPay Serverの支払いボタンは、様々なスタイルに合わせて設定することができます。ボタンは固定またはカスタム金額を持つことができ、スライダーまたはプラスとマイナスのトグルで表示されます。


#### モーダルを使う


支払いボタンを作成する際、ストアオーナーは顧客がクリックした時の動作を選択し、モーダルで表示したり、新しいページとして表示したりすることができます。


**注


警告支払いボタンは、チップや寄付のためにのみ使用してください。


ユーザーが注文関連情報を変更できるため、eコマース統合に支払いボタンを使用することは推奨されません。eコマースには、弊社のGreenfield APIを使用してください。このストアが商取引を処理する場合、支払いボタンを使用する前に別のストアを作成することをお勧めします。


#### 支払いボタンテキストのカスタマイズ


デフォルトでは、BTCPay Serverの支払いボタンは "Pay With BTCPay "と表示されます。ストアオーナーは、このテキストを自由に設定し、BTCPay Serverのロゴを変更することができます。テキストは "Pay Button Text "で設定し、画像のURLは "Pay Button Image URL "の下に貼り付けてください。


##### 画像サイズ


ボタン内の画像サイズは3つのデフォルトにのみ設定可能。



- 146x40px
- 168x46px
- 209x57px


#### ボタンタイプ


BTCPayサーバーは、支払いボタンの3つの状態を知っています。



- 定額
  - 以前の設定価格は、ボタンの一般設定にあります。
- カスタム金額
  - BTCPayサーバーのPayボタンには、カスタム価格を設定するための+と-のトグルがあります。
  - カスタム金額を使用する場合、BTCPayサーバーは、最小、最大、およびどの程度徐々に増加させるかを要求します。
  - ボタンは「シンプルな入力スタイルを使用」に設定することができます。
  - ボタンとトグルがインラインに表示されるように、ボタンをインラインに合わせる。
- スライダー
  - カスタム量と似ているが、+/-トグルの代わりにスライダーがあり、視覚的に異なる。
  - スライダーを使用する場合、BTCPayサーバーは、最小値、最大値、およびどの程度徐々に増加させるかを要求します。


**注


支払いボタンは、警告の説明の上部で削除することができます。


#### 支払い通知


サーバーIPN（即時支払い通知）は、ウェブフック用に設計されており、購入後のデータへのURLを設定することができます。


#### メール通知


支払いが行われると、BTCPayサーバーはストアオーナーに通知することができます。


#### ブラウザのリダイレクト


顧客が購入を完了すると、ストアオーナーによって設定されている場合は、このリンクにリダイレクトされます。


#### 高度な支払いボタンオプション


Invoiceが作成されたら、チェックアウトページに追加されるクエリー文字列パラメータを指定します。例えば、`lang=da-DK`はデフォルトでデンマーク語でチェックアウトページをロードします。


#### アプリをエンドポイントとして使用する


以前使用したことのあるPOSアプリやクラウドファンディングアプリで、支払いボタンをアイテムに直接リンクさせることができます。


ストアオーナーはドロップダウンメニューをクリックし、希望のアプリを選択することができます。アプリを選択したら、ストアオーナーはリンクさせるアイテムを追加することができます。

ダッシュボードの左上にある「ノード」をクリックします。ここでBTCPayサーバーインスタンスの次の部分を設定できます。ノードの概要で「新規作成」をクリックします。先ほどと同じような画面が表示されます。ここで、Lightning Nodeの代わりにBTCPay Serverを選択します。


VoltageはBTCPayサーバーのジオロケーションを表示します。ここではサーバーのホスティング費用も表示されます。Create をクリックし、BTCPay サーバーに名前を付けます。Lightningを有効にすると、Voltageは前のステップで作成したLightningノードを表示します。Createをクリックすると、VoltageがBTCPay Serverインスタンスを作成します。


![image](assets/en/121.webp)


作成ボタンを押すと、ボルテージはデフォルトのユーザー名とパスワードを表示します。これらは以前にボルテージで設定したパスワードと同様です。アカウントにログインボタンをクリックすると、BTCPayサーバーにリダイレクトされます。


新しいBTCPayサーバーインスタンスへようこそ。作成プロセスですでにLightningを設定しているため、Lightningはすでに有効になっています！
