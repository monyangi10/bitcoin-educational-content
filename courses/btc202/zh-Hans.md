## 什么是比特币节点？


<chapterId>0a9fd4e0-94ab-405e-924c-023397393027</chapterId>



正如其创造者 Satoshi Nakamoto（中本聪）所描述的，比特币是一个点对点的电子现金系统。这句简单的话，也就是白皮书的标题，蕴含着许多关于比特币性质的线索：




- 首先，中本聪将比特币描述为一个 "系统"，换句话说，是一整套硬件和软件组件，它们相互作用以提供特定服务或执行特定功能；
- 接着，他解释说，该系统可以允许电子现金的使用，即一种无形货币；
- 最后，他指出，该系统不依赖于任何中央实体：它是 "点对点"（P2P）的，即由用户自己来操作该系统。



由于比特币是一个系统，因此必须在计算机上运行。而且，由于其点对点性质，运行这些计算机的责任由用户自己承担。我们所说的 "比特币节点" 正是指运行着比特币协议软件（如 Bitcoin Core，但我们稍后再谈论）的计算机。这就是比特币能在没有中央机构的情况下运行的原因：验证是通过分布式方式进行的，由属于成千上万用户的成千上万台独立机器执行。



![Image](assets/fr/047.webp)



Nakamoto, S. (2008). *Bitcoin：A Peer-to-Peer Electronic Cash System*。https://Bitcoin.org/Bitcoin.pdf



正是这些用户确保了比特币的安全性。正如 Eric Voskuil 在他的著作《加密经济学》（*Cryptoeconomics*）中讲述的，比特币的安全性既不依赖于区块链，也不依赖于哈希能力，更不依赖于验证、去中心化、密码学、开源或博弈论。比特币的安全性主要取决于愿意承担个人风险的个人。去中心化使这种风险分散到大量个人身上，只有他们的抵御能力才能确保系统的稳健性。



这一原则很容易理解：如果比特币依靠的是一个人拥有的单个节点，那么监禁这个人就足以关闭网络，因为他们将独自承担所有风险。如果有数以万计的节点分布在世界各地，风险就会扩散：要关闭比特币，就必须消灭每一个操作员。



![Image](assets/fr/048.webp)



因此，我们可以区分并列出几个概念，以便将接下来的课程内容更加清晰：




- 比特币货币：本系统内交易使用的记账单位；
- 比特币网络：所有相互连接的节点集合；
- 比特币节点：运行比特币实现的机器；
- 比特币实现：将协议转化为可执行指令的软件；
- 比特币协议：一套规范系统运行的规则；
- 比特币系统：所有这些元素系统的协调组合。



### 比特币节点的作用



比特币节点共同组成了所谓的比特币网络。它们使整个系统能够自主运行，而无需求助于中央机构或层级服务器。



从一开始，比特币的设计就允许每个用户运行一个个人节点。这种情况在今天的 Bitcoin Core 软件中仍然有效，它结合了钱包和节点的作用。但如今，这一功能经常被分离：许多现代比特币钱包只是连接外部节点（无论是否为同一人所有）的钱包。



### 保存区块链



节点的首要任务是存储区块链的本地副本。为了在不涉及中央机构的情况下防止比特币上出现多重花费，每个用户都必须检查系统中是否存在任何交易。确保这一点的唯一方法就是了解在比特币网路上进行的所有交易。因此，所有交易都会带有时间戳，并被分组成块，每个节点都会存储整个比特币区块链。



> 确认没有交易的唯一方法就是了解所有的交易。

Nakamoto, S. (2008). *Bitcoin：A Peer-to-Peer Electronic Cash System*。https://Bitcoin.org/Bitcoin.pdf



因此，区块链是一个不断更新的列表：每当矿工发布一个新区块，节点都会检查其有效性，然后将其添加到自己的本地区块链副本中。截至今天（2025 年 7 月），完整的区块链大小已超过 675 GB，而且由于平均每 10 分钟就会添加一个新区块，这一规模还在继续增长。



![Image](assets/fr/049.webp)



该节点还保存一个本地记录，其中包含在任何给定时间内存在的所有 UTXO（未花费的交易输出），即**UTXO集**。该数据库包含所有未花费的比特币“碎片”。我们将在课程的最后一部分详细讨论这个问题。



### 验证和分发交易



节点的第二个作用是确保交易的验证和传播。当新交易到达节点（通过钱包软件或其他节点）时，节点将检查该交易是否符合一系列规则（共识规则和中继规则）。例如：




- 所花费的比特币必须存在于 UTXO 集（未花费的输出数据库）中；
- 签名必须有效，并且必须满足所有支出条件（有效脚本）；
- 输出总量不得超过输入总量，这意味着费用不能为负。



![Image](assets/fr/050.webp)



经过验证后，交易会被存储在节点的内存池（Mempool）中，这是一个为未经确认的交易预留的临时内存空间，然后转发给与之相连的其他网络对等节点。这种分发和验证机制在节点之间持续进行。通过这种方式，交易在比特币网络中传播，每个节点都将其存储在 Mempool 中，直到矿工将其纳入一个有效区块，然后对其进行首次确认。



### 检查和分发区块



节点的第三个作用是管理挖出的区块。当矿工发现一个带有有效工作证明（Proof-of-Work）的新区块时，它就会在网络上广播。节点收到消息后，它将检查其是否符合所有协议规则，如果有效，则将其整合到自己本地的区块链副本中。与交易一样，新验证的区块会被转发给与节点相连的所有对等节点。这个过程一直持续到比特币网络上的所有节点都知道新区块为止。



![Image](assets/fr/051.webp)



## 节点和钱包有什么区别？


<chapterId>de5af634-a628-4b90-b869-468c208e178b</chapterId>



使用比特币时，必须区分两种不同类型的软件：节点和钱包。



如上所述，比特币节点是一个积极参与点对点网络的软件。它主要执行三项任务：




- 区块链的备份、
- 交易验证和传播、
- 区块验证和传播。



另一方面，比特币钱包是一款用于存储和管理私钥的软件。有了这些密钥，您就可以通过满足锁定脚本（通常是通过签名）来使用比特币。钱包可以连接到一个节点（无论是本地节点还是远程节点），以查询区块链的状态并广播它所建立的交易，但它本身并不是网络的参与者。



在某些情况下，这两种功能共存于同一个软件中，Bitcoin Core 就是如此，它既是全节点也是一个钱包。不过，许多流行的钱包程序（Sparrow、BlueWallet 等）都需要连接到外部节点（无论是自己的还是第三方的节点），以传播交易并确定钱包余额。



![Image](assets/fr/052.webp)



## 节点和矿工有什么区别？


<chapterId>d2992614-7ab7-4bf9-81b1-f548cda67257</chapterId>



节点和矿工的概念经常被混淆。然而，这两种元素在系统中发挥着截然不同的作用。



比特币最初由 Satoshi Nakamoto（中本聪）于 2009 年推出，他希望每个用户都能参与到整个网络中。因此，最初的比特币软件同时兼具几种功能：它既是一个节点钱包，又是一个能够生成新区块的矿机。当时，挖矿的难度非常低。您只需在电脑上运行比特币软件，就能找到区块并获得比特币作为奖励。



然而，随着比特币的逐渐普及和采矿者数量的增加，挖矿的竞争格局发生了很大的变化。如今，挖矿已成为一项竞争异常激烈的活动，由配备专业基础设施的行业参与者主导。现在，挖出一个新区块所需的能量非常大，个人用户仅靠一台传统计算机几乎不可能实现这一目标。因此，挖矿现在主要使用名为 ASIC（*特定应用集成电路*）的专用机器。这些芯片已经过优化，专门用于在比特币上运行双 SHA-256 哈希函数（比特币挖矿使用的算法）。



![Image](assets/fr/053.webp)



在这一演变过程中，比特币节点和挖矿节点的作用已泾渭分明。如上所述，比特币节点的作用纯粹是提供信息和验证。而挖矿节点的作用则不同：




- 它选择 Mempool（内存池）中的待处理交易。
- 它会建立一个候选区块，将这些交易整合在一起。
- 他通过反复试验寻找有效的工作证明（Proof of Work）。
- 如果找到有效证明，它就会通过自己的节点向其他节点广播这个区块。



矿工需要一个比特币节点才能与网络互动。



矿工的作用有时也有别于grind。grind是一种机器，其任务是将模板区块经过哈希函数，由区块池服务器提供，寻找满足为份额定义的难度目标的哈希值，而不是比特币的难度目标。挖矿流程的其余部分，包括实际区块构建、交易选择或根据比特币自身难度进行工作证明搜索以及分配，均由程序池直接执行。



![Image](assets/fr/054.webp)



最后，在经济激励方面，矿机和节点之间存在重要差异。运行比特币节点不会带来直接的金钱收益。另一方面，参与挖矿可为发现的每个区块带来奖励（补贴和交易费）。



在第二部分中，我们将更详细地探讨安装和使用比特币节点的实际好处和个人利益，而不仅仅是经济上的好处。



## Bitcoin Core 和协议实施


<chapterId>72381876-9317-4faa-8d41-2b252a945b8a</chapterId>



比特币协议不是软件：它是一套网络用户之间共享的默许规则。它定义了交易有效性条件、货币创建机制、区块格式、工作证明条件和许多其他规范。要与该协议交互，用户必须运行实现这些规则的软件：这被称为比特币的**实现**。



因此，实施方案就是节点软件：一个能够与比特币网络上的其他机器连接，下载、验证、存储和传播区块和交易，并在本地执行共识和中继规则的程序。每个实施方案都是对协议的具体诠释，用特定的编程语言编写，有自己的架构、性能和人体工程学设计。每个实施方案也都有自己的开发组织和职责分工。



在这些实施方案中，有一种方案遥遥领先： **Bitcoin Core**。



![Image](assets/fr/055.webp)



### 成为基准的历史性实施



Bitcoin Core 是比特币协议的基准软件。它源于 Satoshi Nakamoto 在 2008-2009 年编写的原始代码，是其直接延续。最初称为 "*Bitcoin*"，后又称为 "*Bitcoin QT*"（由于通过 Qt 库添加了图形化 Interface），2014 年更名为 "*Bitcoin Core*"，以明确区分软件与网络。自 0.5 版起，该软件发布了两个组件：Bitcoin-qt"（图形页面）和 "bitcoind"（命令行页面）。



从理论上讲，Bitcoin Core 并不代表比特币协议；相反，它只是众多实施方案中的一种。不过，它的与众不同之处在于它的大量采用、悠久历史、稳健代码和严谨的开发过程。因此，在实践中，Bitcoin Core 所应用的规则事实上就是比特币协议的规则：用户、开发者、矿工和生态系统服务几乎只将它视为标准。



### 目前的实施分布情况



根据[Luke Dashjr 于 2025 年 8 月收集的数据](https://luke.dashjr.org/programs/Bitcoin/files/charts/software.html)（生态系统中的知名开发者），网络公共节点的实施分布情况如下：




- **Bitcoin Core**： 87.3%的所有节点
- **Bitcoin Knots**： 12.5%的所有节点
- **其他实施**： 0.2% (btcsuite, Bcoin, BTCD等)



![Image](assets/fr/056.webp)



换句话说，10 个公共节点中大约有 9 个在运行 Bitcoin Core。网络的其他部分则依赖于更少见的客户端（尽管近几个月 Knots 的份额急剧上升，可能是因为有关的 "OP_RETURN" 大小限制的争论）。这些替代实现通常由一个人或一个小团队维护。



**注意：** 这些数字仍是估算值，因为它们主要基于*监听节点*（listening node），即接受传入连接的节点（8333 端口已打开）。*非监听节点*（non-listening node）的计算要复杂得多，因为不可能直接连接到它们：您必须等待它们以传出连接的形式主动提供连接。Luke Dashjr 的网站声称也在尝试统计这些*非监听节点*，但仍然不可能获得关于它们的完全准确的数据，而且这些统计数据的更新不可避免地落后于现实。



### Bitcoin Core 的内部运行



Bitcoin Core 是用 C++ 编写的。它也是一个开源项目，由开发人员社区维护，这些开发人员是志愿者，或由不同实体（通常是生态系统中对 Core 的开发有既得利益的公司）支付报酬。[代码托管在 GitHub 上可见](https://github.com/Bitcoin/Bitcoin)，开发遵循严格的流程：




- **贡献者**以 "拉动请求"（Pull Request）的形式提交建议。原则上，任何人都可以提出修改建议，但必须经过测试、记录并通过同行评审。
- 维护者**有权批准和合并 PR。他们负责保证项目的连贯性和稳定性。2025 年 7 月，共有五位维护者：Hennadii Stepanov、Michael Ford、Andrew Chow、Gloria Zhao 和 Ryan Ofsky。
- 自2023年2月以来，一直没有**首席维护者**。这一角色最初由 Bitcoin 启动时的中本聪（Satoshi Nakamoto）担任，随后在 2011 年初中本聪离开后由 Gavin Andresen 担任，最后在 2014 年至 2023 年期间由 Wladimir J. Van Der Laan 担任。



![Image](assets/fr/057.webp)



Bitcoin Core 的开发遵循 "任人唯贤" 的逻辑：鼓励新的贡献者在提出任何修改之前先对代码进行审查和测试。决策以技术共识为基础，重大修改（尤其是在共识领域的修改）需要在公共渠道（如邮件列表或公关审查俱乐部）上进行上游讨论。



### 其他比特币实施方案



尽管在采用方面还处于边缘地位，但确实存在其他客户端。其中最主要的是由 Luke Dashjr 开发的 Bitcoin Knots，它是 Bitcoin Core 的分叉，包含了更多选项和更保守的开发方法。其中包括对交易格式的更严格限制。



![Image](assets/fr/058.webp)



我们还可以提到：




- **Libbitcoin**: 由 Amir Taaki 开发、Eric Voskuil 维护的模块化 C++ 库；
- **Bcoin**：JavaScript 实现，已停止维护；
- **BTCD/btcsuit**e: Go 语言的实现。



这些项目为生态系统的多样性做出了贡献，但它们的采用仍然非常有限，使得 Bitcoin Core 难以独立发展。



### Core 开发人员的力量



您可能认为 Bitcoin core 开发者可以直接控制比特币，但事实并非如此。他们不能强行更改协议。他们的职责是提出代码。每个用户都可以通过自己的节点决定是否使用这些代码。



这意味着，如果 Bitcoin Core 中的某项变更未达成共识，节点可以忽略它，要么不更新 Bitcoin Core，要么直接改变实现方式。相反，如果用户所需的功能在 Core 开发过程中受阻，则随时可以切换到其他实现或分叉项目。



正如我们将在本课程后面讨论的内容，节点（即商家）根据其经济权重，通过接受遵守其规则的单位，为某一版本的协议（以及相应的货币）赋予效用。因此，管理比特币的真正权力在于这些商家，而不是开发者。

## 为什么要运行自己的节点？


<chapterId>39c0cd19-67f9-4c64-bfb3-dbd6eec0bf42</chapterId>



人们普遍认为，运营比特币节点是一种纯粹的利他主义行为，没有任何个人利益，完全是为了网络去中心化的贡献。一些人认为，比特币支持者有责任支持这个系统，并对比特币的存在表示感谢。



事实上，正如我们在前几章中指出的，运行节点并没有直接的经济收益。因此，人们可能会认为这种做法没有任何个人利益。然而，运行自己的节点却能带来许多个人利益。为了让大家相信这一点，我将在本章中从技术和战略两方面阐述为什么要安装和使用自己的比特币节点。



### 更加保密地传播交易信息



当钱包软件连接到外部节点时，它会将其交易传输到不受您控制的基础设施。这会产生明显的监控风险：远程节点的操作员可以分析您的交易细节，包括金额和频率，并通过交叉检查某些元数据（如 IP 地址、时间和地点），将其与您的身份联系起来。



事实上，正如前一章所指出的，钱包与比特币网络之间的通信并不是靠魔法实现的；它们必须连接到一个节点，才能查询余额或广播交易。如果您从未设置过自己的节点，这意味着您的钱包依赖于第三方（通常是软件背后的公司）的基础设施。第三方（尤其是公司）可能会观察、利用甚至披露这些数据：无论是出于商业原因、法律限制还是盗版行为。



![Image](assets/fr/059.webp)



通过使用自己的节点，您可以绕过中间商，直接向网络广播您的交易。只要您妥善保护自己的节点（我们稍后会讨论）或遵守某些标准，就不会暴露任何信息：您的 IP 地址和交易细节都不会通过您不控制的实体广播。这是在比特币上为您保密的基本前提。



https://planb.academy/courses/65c138b0-4161-4958-bbe3-c12916bc959c

### 不可监控的交易



出于上述同样的原因，基于第三方节点的钱包软件容易受到审查风险的影响：远程节点的运营商可能会出于各种原因拒绝转发某些交易。它可能认为这些交易可疑或违反其政策。如果交易不符合节点的中继规则，也可能被阻止。最后，操作员可能会专门针对您的 IP 地址，阻止您的交易广播。



相反，通过使用自己的节点，您可以确保您的交易在点对点网络中的传播。这意味着您可以完全控制交易的传播，而无需依赖中介。只要交易符合与您的节点相连的节点的共识和中继规则，它就会在网络上广播，然后，只要包含足够的费用，就会被矿工整合到一个区块中。拥有自己的节点可确保交易得到中立、无权限的确认。



### 独立数据验证



如果没有个人节点，您仍然需要依赖第三方实体才能获取信息，例如您的地址余额、交易确认状态和区块有效性。这意味着对外部节点的准确性和完整性的隐性信任。



![Image](assets/fr/060.webp)



运行全节点意味着您可以自行检查每笔交易和每个区块的所有协议规则。因此，钱包显示的余额不是从远程服务器接收的数据，而是根据区块链的完整副本在本地计算得出的结果，并逐块进行验证。这种方法充分体现了比特币支持者的格言：



> 不要相信，要验证。

### 更好地分配系统的安全



每一个加入网络的节点都加强了比特币的冗余和韧性。它促进了信息的传播，并使新的对等节点能够相互连接。没有这些节点，系统根本无法运行。



正如我们所见，比特币的安全性并非基于去中心化、挖矿过程或密码学：与任何系统一样，它依赖于个人。更确切地说，它依赖于节点操作员抵制胁迫的能力。



像比特币的去中心化系统的与众不同之处在于其运行风险在所有参与者之间的分配。运行自己的比特币节点意味着通过确保自己实例的安全来分担风险；这个做法也减轻了其他节点运营商的风险负担。



因此，这并不是直接的个人利益：运行一个节点会让您对网络安全承担部分责任。最重要的是，这是一种集体利益，因为您的参与有助于分散风险。反过来，您也提高了自己可靠使用比特币的能力。



### 加深对系统的理解



安装全节点绝非易事。它涉及安装软件、了解基本操作、监控同步、在出现问题时检查日志，甚至还要使用终端。这必然会加深您对协议的理解。这是一个间接但并非微不足道的优势。



掌握这些知识可以增强您对工具的信心，并减少出错或遭遇诈骗的风险。运行自己的节点也是一种学习过程。



### 选择要遵守的规则



一个经常被误解的重要方面是，运行节点允许您选择本地应用的规则。规则主要有两种：





- **共识规则（Consensus rules）**：



这些规则是比特币协议的基本规则，可确保系统的真实性，并建立验证交易和区块的标准。任何不符合这些共识规则的交易都不会被纳入有效区块。例如，一个交易的某个条目上的签名无效，该交易就会被系统排除在外。



改变这些规则就等于改变协议，进而改变货币（导致硬分叉）。然而，即使不试图修改这些规则，严格执行现有规则这一简单事实也赋予了某种力量：如果某个区块违反了规则，节点就会立即拒绝传播该区块。





- **中断规则（Relay rules）**：



这些规则是每个比特币节点特有的，它们被添加到共识规则中，以定义 Mempool 接受并转发给对等节点的未确认交易的结构。每个节点都在本地配置和应用这些规则，这就是为什么不同节点的规则可能不同。这些规则只适用于未经确认的交易：如果某个节点认为一笔交易是 “非标准” 的，它会拒绝将其加入内存池，但如果这笔交易已经出现在有效区块中，节点仍然会接受。



例如，根据共识规则，一笔没有费用的交易是完全有效的，但根据 Bitcoin Core 中继策略，它将被默认拒绝，因为 `minRelayTxFee` 参数被设置为 `0.00001`（单位：BTC/kB）。不过，您可以在自己的节点上降低该阈值，以中继费用较低的交易，或者相反，增加限额，例如，增加到 2 Sats/vB，以避免中继费用较低的交易。



运行自己的节点意味着断言："我根据自己采用的规则验证我选择验证的内容"。这样，您就成为了系统管理的参与者，可以拒绝您认为不可接受的演变，也可以根据自己的标准接受更新。



因此，我们可以很快了解到，由于节点的存在，您对规则有多大的控制权。这种权力的大小取决于规则的类型。



#### 关于中断规则



就中继规则而言，最重要的是拥有一个节点，无论其经济活动如何。关键是您是否同意转发某些类型的交易。



例如，如果您认为比特币应该接受费用低于 1 sat/vB 的交易，您可以调整节点上的这一规则，使其广播这些交易，从而促进它们在网络上的传播，直到矿工最终将其纳入一个有效区块。从根本上说，这是一个交易传播权力的问题：每个节点都有决策权，因为同意转发某种类型的交易就等于促进比特币网络接受这种交易。因此，如果您运行着多个节点，那么您对中继政策的影响力就会更大，因为每个节点都有自己的连接和对网络有影响的区域。



事实上，一个或多个节点配置特定的中继规则，意味着要确定网络的哪一部分接受传播特定类型的交易。在点对点图中传播信息，如比特币交易，遵循的是渗滤理论（Percolation theory）的逻辑。把每个节点想象成一个站点，它可能是活跃的（`p` = 它会中继），也可能是不活跃的（`1-p`）。一旦 `p` 的比例超过临界阈值（`p_c`），网络中会出现一个巨大的连通部分：交易能够顺利在网络中传播，并很有可能被矿工接收到。在比特币的网络中，每个节点平均保持 8 个向外连接，因此 `p_c` 临界值一般只设定为几个百分点，如果某些节点拥有大量连接，这个临界值甚至会更低。



![Image](assets/fr/061.webp)



只要 `p` 低于 `p_c`，交易就会局限在孤立的小区内，不会达到矿工的节点。一旦超过这个阈值，交易几乎会瞬间扩散到整个网络。



最终，矿工会决定是否要将交易纳入区块。然而，节点通过影响交易的分布进行上游干预：它们决定矿工是否会知道某笔交易的存在。如果交易没有被转发给矿工，矿工显然不可能将其纳入区块。



因此，如果网络在某类交易上已经处于渗流阶段，再增加少量节点只会产生微弱的影响；但当网络接近渗流阈值时，这些新增节点可能会起到关键作用。拥有或影响几个节点，特别是如果这些节点连接良好，可以增加或减少 `p` 的值，从而间接调整中继规则，决定矿工看到并最终接受哪些交易。



#### 共识规则



说到您的节点对共识规则的影响，它的经济权重是决定性的。这是一个至关重要的概念：任何货币的价值都与其促进交换的能力直接相关。事实上，如果一个物品在交换过程中不被任何人接受为商品或服务，那么理论上它就没有货币效用。例如，如果没有商人接受鹅卵石作为支付手段，那么鹅卵石就没有货币用途。当然，效用仍然是个人的主观概念，但在特定领土上，接受某种物品作为交换支付手段的商人数量越多，这种物品就越有可能对生活在这片领土上的人们具有货币效用。



以一个村庄为例，在这个村庄里，许多商人接受使用黄金换取商品：黄金很可能对村民具有货币效用。这表明，一种货币的效用直接取决于商人接受或拒绝这种货币的决定。



这一概念对于理解比特币系统中的权力动态至关重要。中本聪明确指出：比特币是一个电子现金系统；换句话说，它提供一种服务，提供一种货币形式，即比特币（或 BTC）。当协议规则以一种不向后兼容的方式进行修改时（导致硬分叉），这相当于创建了一个新的系统，因此也是一种新的货币。这种分叉的成败取决于其经济规模，而经济规模又取决于接受这种新形式货币的商家数量。



![Image](assets/fr/062.webp)



让我们举个例子：假设比特币遭遇了分叉。那么就会出现两种不同形式的货币：BTC-1（原始的、没有改变的版本）和 BTC-2（具有不同共识规则的新货币）。如果所有接受 BTC-1 的商家都继续这样做，但拒绝接受 BTC-2，那么理论上，后者的货币效用将非常有限。作为用户，我没有兴趣保留和使用 BTC-2，因为我知道在交换过程中，没有商家愿意用它来购买商品或服务。相反，如果 50% 的商家选择只接受 BTC-2，而剩下的 50% 只接受 BTC-1，那么理论上，BTC-1 的效用将减半。我之所以使用 "理论上" 这个词，是因为效用在个人层面上仍然是主观的，取决于多种因素（如地域和消费习惯），很难逐一理解。



在比特币中，"商家" 的角色被理解为任何具有一定经济权重的实体，包括企业（实体商店、在线销售网站、服务提供商等），但也包括交易所平台（因为它们在此可用其他货币买卖比特币）和矿工（因为它们通过在交换中收取费用接受比特币，以获得将交易纳入区块的服务）。



就共识规则而言，您的节点允许您将经济活动导向一种或另一种货币。例如，如果您家里有 10 个全节点，但没有重要的经济活动，那么您在发生分叉期间的影响力几乎为零。相反，用于管理 200 家连锁店的单个节点在接受比特币时会产生重大经济影响。



因此，重要的不是节点的数量，而是它们所支持的经济活动的重要性。更重要的是，如果您的经济活动依赖于一个您无法控制的节点，那么只要你与该节点保持连接，它的所有者就会决定您使用什么货币。这就是为什么运行和使用自己的节点在系统治理中尤为重要：



> 不是你的节点，就不是你的规则。


## 不同类型的比特币节点


<chapterId>be8f0baa-41f2-4b54-b011-092f4ccc93aa</chapterId>



比特币节点就是运行比特币协议的机器。在节点的这一通用定义的背后，存在着几种可能的配置，但并非所有配置都能提供相同程度的自主性、资源消耗和网络实用性。在本章中，我们将了解这些差异，以帮助您选择适合您的用途和硬件限制的节点架构。



### 全节点（Full node）



全节点只是一个比特币节点，它从创世区块下载整个比特币区块链，独立验证每个区块，并在本地存储所有区块链的历史记录。这就是中本聪所想象的比特币节点的 "正常" 形式。



![Image](assets/fr/063.webp)



全节点不需要信任任何人，因为它验证并了解系统中的所有信息。这种节点能为您提供最大的保证：您无需依赖第三方，就能知道付款是否有效、区块是否有效、重组是否合法等等。



实际上，全节点需要的资源并不少，包括数百 GB 的块文件、能够验证脚本的处理器、Mempool 和 Cache的 RAM 以及稳定的带宽。第一次同步（*IBD*）会读取并验证完整的历史记录：这需要大量时间，但只进行一次。全节点会积极参与网络，中继区块和交易，并接受传入连接以协助其他对等节点。



根据需要，您可以为全节点添加索引器。Bitcoin Core 将交易索引作为可选功能提供（默认情况下停用），这对特定用途非常有用。但是，它不包括地址索引器，而这往往是个人用户最需要的功能。要解决这个问题，可以在节点上安装专用软件，如 Electrs 或 Fulcrum，以加快相关 UTXO 的地址余额验证查询。我们将在另一章中详细介绍索引器的作用。



### 剪枝节点（Pruned node）



剪枝节点会像全节点一样验证从创世区块到工作量最多的链头的所有内容，但**只保留最新部分的区块文件**。一旦检查完旧的区块，它就会逐渐删除它们，以保持低于您可以设置的空间限制。如果磁盘空间有限，这种配置是理想的选择：您可以保留区块验证的独立性，而无需存储完整的 区块链历史档案。如果您只想在个人电脑上安装 Bitcoin Core，而不想使用专用机器，该选项尤其有用。



![Image](assets/fr/064.webp)



该方案的技术含义相当简单：pruned 节点完全能够广播您的交易、参与中继、验证区块和交易并跟踪链。另一方面，它不能为其他应用（如全探索器、索引器、钱包）提供超出其限制的历史数据源。因此，需要存档（或全局索引）的功能将无法使用。



实际上，您可以使用剪枝节点连接钱包管理软件（如 Sparrow Wallet）。但是，您将无法扫描钱包上剪枝限制之前的交易。例如，如果您在区块 901 458 中注册了一个交易，但您的节点只保留区块 905 402 及以上（因为最旧的区块已被剪枝保存），您将无法扫描到该交易。另一方面，如果您在节点仍有该区块高度时就已扫描了该交易，那么您的钱包管理软件就会存储该信息，并正确显示相应 UTXO 的余额。



简而言之，如果在软件已连接到剪枝节点的情况下创建新的钱包，钱包跟踪就能在该节点上顺利运行。另一方面，恢复旧的钱包时可能会遇到困难，因为节点不再保留的过去交易显然无法检索。



### 轻节点（Light Node）/ SPV节点（SPV Node）



SPV（*简单支付验证*）节点或轻量级节点只保留区块头，而不保留交易的细节，并依靠其他完整节点来获得交易在其拥有头的区块中的证明（通过树的梅克尔证明（Merkle Proof））。简单支付验证的概念并不全新，中本聪本人在《白皮书》第 8 部分中就提出了这一概念。



![Image](assets/fr/066.webp)



Nakamoto, S. (2008). *Bitcoin：A Peer-to-Peer Electronic Cash System.*。https://Bitcoin.org/Bitcoin.pdf



与全节点和剪枝节点相比，这种类型的节点在存储和 CPU 占用方面显然要轻得多。因此，SPV 节点非常适合小型设备和间歇性连接。事实上，它通常直接集成到钱包中，尤其是 Blockstream App 等移动软件。



这种做法的代价是信任度和保密性：SPV 客户端本身并不检查脚本或验证策略；它假定工作最多的链是有效的，并依赖一个或多个全节点的响应。因此，使用这种类型的节点比连接到第三方节点更好；不过，与全节点和剪枝节点对比，它的优势仍然较小。



![Image](assets/fr/065.webp)



### 哪个节点满足哪种需求？





- 手机/初级用户



对于只有一个钱包移动应用程序的新手用户来说，使用 SPV 节点无疑是最佳的入门方式。安装快捷，所需资源少，体验简单流畅。这意味着您可以自己验证某些信息，从而减少对第三方节点的依赖，同时在广播交易时更加独立。





- 电脑/中级用户



拥有个人电脑的中级用户可以安装一个剪枝节点，享受全节点的几乎所有优势，而无需每天超负荷工作：完全验证、适度的磁盘使用和简单的维护。它是连接桌面钱包和保持交易分发独立性的理想解决方案，无需投资专用机或超负荷使用磁盘空间。





- 主权比特币支持者 / 高级用户



如果您想完全独立地使用比特币，而不是限制自己以后使用索引器、闪电节点甚至区块浏览器等高级用途，那么全节点仍然是最佳的解决方案。这正是我们将在本课程中探讨的内容！



## 软件解决方案概述


<chapterId>0d48b89a-e8b5-441e-a707-537a035fc15e</chapterId>



在软件方面，运行比特币节点有两种主要方式：




- 直接安装协议实现，如 Bitcoin Core（推荐）或 Bitcoin Knots、
- 或者使用 "交钥匙" 发行版（通常被称为"_node-in-a-box_"），该发行版以同样的方式集成了比特币实现，但也包括页面管理系统、应用程序商店和随时可用的工具（闪电网络、浏览器、索引服务器，甚至比特币外部的自托管应用程序......）。



这两种方法的目标相同：拥有自己的节点，但在页面的安装和使用、维护、可扩展性和成本方面有所不同。这就是我们将在本章探讨的内容。



### 原始比特币节点实现



安装原始实施方案意味着直接使用比特币协议实施方案（如 Core）的软件，而无需任何附加软件层面。您可以根据自己的需要自行管理配置、更新和相关服务（索引、API、闪电、备份等）。



这是最主要也是最灵活的方法：您可以清楚地了解正在运行什么、数据在哪里以及一切是如何工作的。另一方面，一旦你想超越比特币节点的简单操作，它就会变得更加复杂。如果您的目的仅仅是拥有一个节点，那么其复杂程度与盒装节点相当，甚至更低，因为只需安装软件即可。



#### Bitcoin Core（极端多数客户）



[Bitcoin Core是网络的超多数客户端](https://bitcoincore.org/)。它能下载、验证和维护区块链，提供 RPC/REST API，并能集成到钱包中。如果您更喜欢标准工具，并愿意自己添加服务（如 Electrum 服务器、资源管理器和 LND），最好还是使用 Core。



**优势：** 最高稳定性、可预测行为、原始体验、易于安装和配置。



**缺点：** 您必须手动构建堆栈的其他部分，以创建一个完整的应用环境，而不仅仅是一个比特币节点。



https://planb.academy/tutorials/node/bitcoin/bitcoin-core-linux-568c13a6-8746-4d63-8e95-f4a61c5ae0ed

https://planb.academy/tutorials/node/bitcoin/bitcoin-core-mac-windows-9684ab02-e0af-41c9-8102-86ac7c7727f3

#### Bitcoin Knots（主要替代客户端）



[Bitcoin Knots 是 Bitcoin Core 的分叉](https://bitcoinknots.org/)，由 Luke Dashjr 维护。它是执行比特币协议的 Core 客户端的主要替代客户端。它与网络的其他部分完全兼容（与 Bitcoin Cash 类似，它绝不是硬分叉），但它提供了更多的功能，包括 Core 中没有的中继策略选项，或在默认情况下更严格地限制某些人认为的垃圾交易。



选择 Knots 而不是选 Core 可能有两个原因：




- **技术**：与 Core 不同的选项，特别是在中继管理方面，通过决定节点接受和广播哪些交易。
- **政策**：有些人出于非技术原因喜欢使用 Knots 等替代客户端，主要是为了支持 Core 的替代方案，从而减少 Core 的垄断。如果 Core 遭到破坏，那么不仅要有可靠、维护良好的替代客户端，还要知道如何有效利用它们。还有一些人使用 Knots 是出于抗议目的，因为他们对 Core 开发人员失去了信心，或者不赞成客户端的大多数管理人员。


https://planb.academy/tutorials/node/bitcoin/bitcoin-knots-e04b2196-4df2-4246-86ef-c02269c29098

我个人建议您选择 Core，主要是为了更快地从安全补丁中受益。事实上，Knots 中发现的一些漏洞迟迟得不到修正。更广泛地说，Core 的开发过程结构稳固，得到了大量贡献者的支持，而 Knots 仅由一个人维护，社区规模小得多。另一方面，中继规则在今天往往会失去其效用，尤其是当只有极少数网络应用时（如渗流理论）。



### Node-in-a-box （盒中节点）分布



Node-in-a-box 将 Bitcoin Core（或 Knots）与预配置操作系统、页面网站和自托管服务应用程序商店（闪电网络、浏览器、Electrum 服务器、Mempool、BTCPay Server、Nextcloud 等）结合在一起。只需点击一下，您就可以安装、更新和互联这些不同的模块。



对于日常启动和管理众多辅助应用程序来说，这是一个简单得多的解决方案。缺点是，当出现问题时（例如，Docker 映像冲突、错误更新、数据库损坏），调试可能会变得非常复杂，因为您需要依赖发行版自身的集成。此外，社区或官方支持通常也很复杂。



因此，只要一切运行正常，"盒中节点" 就非常容易使用，但如果出现错误，您就必须准备好进行长时间的搜索、等待帮助并弄脏自己的手。



这些解决方案大多有两种格式：




- 预装机：已安装操作系统的完整计算机。这些 "即用即付" 的计算机只需插上电源并连接互联网即可运行。如果您的预算允许，这种选择的优点是安装非常简单，通常提供优先支持，并有助于为开发提供资金，因为这些公司的商业模式通常是以销售硬件为基础的。
- 自我动手：在自己的机器（旧 PC、NUC、Raspberry Pi、家庭服务器......）上安装发行版操作系统。这是最经济的解决方案，因为您可以回收旧机器，或选择与您的需求和预算完全匹配的硬件。这也是最灵活的方案，配置起来也最令人满意。我们将在课程的实践部分探讨这种方法。



以下是现有（2025 年）主要的盒中节点解决方案概览：



### Umbrel（umbrelOS 和 Umbrel Home）



[如今，Umbrel 已成为 "盒中节点" 解决方案 (https://umbrel.com/) 的领导者。它的成功主要归功于其简单的安装方式（当时是在一个简单的 Raspberry Pi 上推出的）、优雅直观的页面以及迅速发展并现已极为广泛的应用生态系统。

关于它的工作原理和具体功能，我在这里就不再赘述了，因为我们将在下一部分的第一章中进行更深入的探讨。事实上，为了达到本 BTC 202 课程的目的，我选择了使用 UmbrelOS，我认为它是目前最适合初级和中级用户的节点开箱即用解决方案。

### Start9（StartOS）



[Start9 提供 StartOS（https://start9.com/），这是一个专为 "主权计算 "设计的系统：其目的是让每个人都能拥有和管理自己的私人服务器，并通过自托管应用程序市场来增强其功能。您可以购买 Start9 服务器（Server One 售价 619 美元，Server Pure 售价 899 美元），也可以在自己的机器上以 DIY 模式组装自己的服务器。



在比特币方面，StartOS 可以让您安装全节点、Lightning 节点、BTCPay Server、Electrs 和许多其他服务。然而，Start9 的魅力远不止于此：它提供了以统一方式发现、配置和公开各种软件（文件云、消息传递、监控）的可能性，并具有完全的控制权。因此，该项目的目标用户是那些想要一个强大的自托管平台，而不仅仅是一个简单的比特币节点的用户。它可能是继 Umbrel 之后最完整的生态系统。

简而言之，如果您只需要比特币的基本应用程序，对 Umbrel 非常丰富的生态系统没有特别兴趣，而且页面用户不是优先考虑的因素，那么 Start9 是更好的选择。否则，Umbrel 是更好的选择。

## 硬件解决方案概览


<chapterId>245d6add-9cda-46b9-9343-31dcdd70456e</chapterId>



既然我们已经探究了所有软件的方案，那么让我们来关注一下节点所需的硬件。我将为您提供一些选择组件的具体建议，以及适合不同预算的配置。当然，这只是我的个人意见和反馈：除了这里所讲述的内容，肯定还有其他相关的替代方案。此外，我将不再赘述节点盒项目提供的预组装机器，我们在上一章中已经介绍过这些机器。在这里，我们将只关注自我手动的解决方案。



### 您真的需要一台专用机吗？



在过去的几年里，比特币支持者们越来越意识到一个常见的误解，尤其是随着 2020 年代初 "盒中节点" 的普及：比特币节点必须运行在专门用于此目的的机器上。但事实并非如此。运行比特币节点并不一定需要专用计算机：Bitcoin Core完全可以在您的日常电脑上运行。如果您有足以安装比特币区块链的磁盘空间，或您选择启用剪枝功能，您就可以验证区块链、连接钱包，甚至在使用完毕后关闭程序。这种方法的优势非常明显：零初始投资和非常小的复杂性。



![Image](assets/fr/074.webp)



尽管如此，使用专用机往往更合适。它可以连续运行（24/7），随时可以远程访问，不会耗掉主机器的资源，最重要的是，可以隔离使用（这是一种很好的安全做法：如果您的个人电脑遇到问题，您的节点可以继续运行，反之亦然）。因此，问题不在于 "我是否需要专用机"，而在于 "我是否需要一个持续在线、可被其他设备访问并能不断发展的节点"（闪电、索引器、附加应用程序......）。如果您需要的话，那么选择独立的机器会让事情变得简单得多。



### 三种购买方法：回收设备、二手设备和全新设备



#### 回收旧计算机



这是最经济的解决方案。大多数人家里或亲朋好友家里都有一台老式个人计算机：这正好是重新启用它的好机会！要将其改装成比特币节点，只需添加 2TB 的固态硬盘（SSD），并根据需要更换或添加内存条以增加内存。一台功能齐全的机器预计售价在 100 至 200 欧元之间。



在购买任何硬件之前，请检查可用的磁盘插槽数量、连接类型（M.2 或 SATA）、内存格式（SODIMM 或 DIMM）及其世代（DDR4 等）。您还应借此机会清洁机器，尤其是风扇，以确保最佳的性能。



不过，如果您使用的是笔记本电脑，您要更加谨慎：随着时间的推移，电池可能会出问题（本章后面将详细介绍）。



#### 翻新或二手设备



市场上有很多翻新的商用微型电脑，如*联想 ThinkCentre Tiny*、*HP EliteDesk Mini* 或*D ell OptiPlex Micro*。这些机器坚固、小巧、静音、节能。它们的价格远低于新机，而且很容易找到配备第六代至第十代 i5/i7 处理器和 8 GB 至 16 GB 内存的机型，价格非常低廉，一般在 70 欧元至 200 欧元之间，视配置而定。在我看来，如果您需要用来运行比特币节点的专用机，那么这可能是最佳的选择。



![Image](assets/fr/075.webp)



在这里还能找到几年前的二手个人电脑和笔记本电脑，它们配置有趣，性价比极高。



**注意：** 公司机群中的机器，如*ThinkCentre Tiny*，通常只配备一个*DisplayPort*（DP）端口用于屏幕，没有 HDMI 输出。因此，如果需要的话，不要忘了带适配器或 DP 转 HDMI 电缆。



#### 购买新设备



如果预算足够，也可以选择购买新设备。如果您想要拥有性能良好的最新硬件，尤其是如果您打算使用 Umbrel 或 Start9 以及比特币生态系统之外的其他应用程序进行自托管，这就是一个不错的选择。



### 我应该选择哪种类型的机器？



#### 微型计算机 "NUC"/裸机



在我看来，微型电脑是在家中托管比特币节点的最佳选择。它们所占用空间不大，可轻松放在架子上，耗电量极低，而且易于进行硬件修改，如添加内存或更换 SSD。



就我个人而言，我更喜欢*Lenovo ThinkCentre Tiny*，它在二手市场上非常普遍（来自企业车队）；它们特别坚固耐用，而且易于改装。当然，其他制造商也有许多同类产品： *Dell OptiPlex Micro*、*HP ProDesk / EliteDesk Mini / Micro*、*英特尔 NUC*、*Gigabyte BRIX*、*MSI Cubi*......。



![Image](assets/fr/001.webp)



**优点：** 占地面积小、功耗适中、噪音低、可扩展性强（取决于型号）、可靠性高。



**缺点：** 比 Raspberry Pi- 型 SBC 稍贵，没有内置屏幕（远程访问或通过外部显示器），没有电池（断电时会突然关闭）。



#### 专用笔记本电脑



它是微型电脑的绝佳低成本替代方案：如今，您可以以低廉的价格找到二手甚至全新的笔记本电脑，它们配备了不错的处理器、众多端口以及集成屏幕和键盘（对于初次安装非常实用）。最重要的是，电池可以充当天然的 UPS：在断电的情况下，节点不会突然关闭，甚至可以保持运行数小时。



![Image](assets/fr/076.webp)



**优点：** 一体化解决方案，电池可作为 UPS（不会停电），集成显示屏和键盘简化了安装过程，集成 Wi-Fi 卡，二手市场和新市场选择广泛（这通常意味着可以议价）。



**缺点：** 功耗略高于裸机 Mini-PC，全天候运行时电池会逐渐损耗，导致容量下降，电池老化后极少出现膨胀或热失控的风险。我之所以认为 Mini-PC 比笔记本电脑更好，主要是考虑到了这一点：电池的逐渐老化和相关风险。



如果您选择这种解决方案，我建议您密切关注电池的状况，以防发生任何危险。注意电池是否过热、有异常气味、不稳定或外壳变形。如果出现警报，请立即关闭电脑并拔掉电源插头，然后到专门的回收机构处理电池。



**提示：** 如果 BIOS/UEFI 或制造商的工具允许，可设置负载限制（如 60% 或 80%）以延长电池寿命。



#### Raspberry Pi 和其他单板机：错误的想法



20 世纪 20 年代初，随着 "盒装节点" 软件的兴起，Raspberry Pi 作为运行比特币节点的一种方式也出现了。这个想法似乎很有吸引力：价格低廉、结构紧凑、易于使用。



![Image](assets/fr/073.webp)



实际上，如果您的目标仅仅是运行一个比特币节点，而不需要额外的应用程序，那么 Raspberry Pi 可能就足够了。但一旦您想使用 Umbrel、Start9 或更丰富的生态系统（区块浏览器、地址索引器、闪电节点、自托管应用程序......），这台机器会有所限制。



Raspberry Pi 有一些缺点：




- 处理器过于纤薄，ARM 架构有时与某些软件不兼容或需要更多处理；
- 焊接内存，无法升级，配置有限（通常最多 8 GB）；
- 通过电缆连接的固态硬盘外置盒，经常出现错误，需要购买特定的卡才能获得稳定的固态硬盘；
- 容易迅速升温，难以确保正确冷却；
- 需要购买额外的硬件（机箱、风扇、固态硬盘卡等）；
- 连接性非常有限。



从历史上看，Raspberry Pi 等单板机的最大优势在于其价格：几十欧元就能买到一台专用机。然而，如今价格已大幅上涨，一旦添加了所有必要的附加硬件，成本就接近于第一台二手或翻新的 x86 微型电脑，而我认为后者的优势要大得多。因此，我不建议您选择 SBC。



### 选择组件



#### 磁盘存储：必须使用固态硬盘，至少 2 TB



从技术上讲，在硬盘上运行比特币节点是可行的。问题是一切都会大大减慢，尤其是 IBD（初始化区块下载），由于 Bitcoin Core 将磁盘作为缓存密集使用（尤其是 UTXO 集），IBD 会变得非常耗时。这就是我强烈建议不要使用硬盘的原因：硬盘会造成真正的瓶颈，严重限制未来的发展（如闪电节点），甚至可能导致与比特币磁头的同步不匹配。此外，机械磁盘上的持续压力会增加过早磨损的风险。



固态硬盘从根本上改变了用户体验：一切都变得更快、更流畅、更可靠。因此，使用固态硬盘是节点的（几乎）必修课，您不会后悔的，尤其是现在大容量型号的固态硬盘价格相对低廉。



![Image](assets/fr/077.webp)



就容量而言，2 TB 正逐渐成为新的合理最小值。2025 年夏天，比特币区块链的容量已经接近 700 GB，如果再加上 Umbrel、Address 索引器和一些应用程序，1 TB 的固态硬盘很快就会饱和。如果使用 2TB SSD，在未来几年内（大致估计在 5 到 15 年之间），您将有一个舒适的余量。如果您计划在 Umbrel 上使用许多应用程序，在自托管中存储大型文件，或者您想在很大程度上预测您的磁盘空间需求，您也可以选择 4 TB。



![Image](assets/fr/078.webp)



至于格式，这取决于你机器上可用的端口；不过，如果可能的话，我建议使用 NVMe M.2 SSD。



#### 内存（RAM）8 至 16 GB



对于单独使用 Bitcoin Core（无 Umbrel 叠加）的情况，开发者建议在将设置调整到最低时至少使用 256 MB 内存，在使用默认设置时至少使用 512 MB 内存，正常使用时至少使用 1 GB 内存。



另一方面，如果您使用的是 Umbrel 或 Start9 这样的 "盒中节点" 系统，对内存的要求就会高得多。Umbrel 开发者建议至少使用 4 GB 内存。这也许足以只运行 Core，但很快就会受到限制。因此他们建议使用 8 GB 内存，我也认为这是比特币基本配置（Core、LND、索引器和一些应用程序）的最低要求。根据我的经验，在使用 Umbrel 和一些附加服务时，8 GB 还是不太宽裕。我建议使用 16 GB 内存，运行更顺畅，也有一定余量。



#### 处理器（CPU）



对于 Umbrel 节点而言，最低要求是 Intel 或 AMD 的双核 64 位处理器。如还想使用 Bitcoin Core 之外的一些应用程序，那么四核以上处理器将在流畅性方面发挥真正的作用。例如，在二手市场上，第六代至第十代 i5/i7 处理器是很好的选择。

## Umbrel: 不仅仅是一个比特币节点


<chapterId>dd4c04f1-924a-43e1-94f3-ea9fbc83dd43</chapterId>



Umbrel 是一个个人服务器操作系统，旨在实现自助托管：您只需安装 Umbrel，在`umbrel.local`上打开浏览器，然后通过一个简单的远程页面管理一切。



该项目首先普及了一键式比特币和闪电节点的理念，然后扩展为名副其实的 "家庭云"：文件和照片存储、多媒体流媒体、网络工具、家庭自动化、本地人工智能，以及可从集成 App Store 安装的数百个应用程序。



在 Umbrel 中，每个应用程序都在 Docker 容器中运行（隔离、原子更新、独立启动/停止）。Interface 可以集中访问所有这些应用程序，提供单点登录（可选 2FA）、一键更新操作系统和应用程序、实时监控机器（CPU、RAM、温度、存储）、应用程序之间的权限管理以及应用程序消耗概览。



因此，Umbrel 的目标是在不依赖云服务的情况下，让您重新获得对数据的控制权和保密性，而不仅仅是操作一个 Bitcoin 节点。



### Umbrel Home 与 umbrelOS



Umbrel 提供两种不同的方法：





- [**Umbrel Home**](https://umbrel.com/umbrel-home)：这是一款即用型微型服务器，专为 umbrelOS 设计和优化。它小巧、静音、带有以太网连接，配备 NVMe 固态硬盘（最高 4TB 可选）、16GB 内存和四核 CPU。订购后，插上电源，然后访问 `umbrel.local`。几分钟内，您就可以启动并运行 Umbrel。这就是即插即用选项。



![Image](assets/fr/081.webp)





- [**umbrelOS**](https://umbrel.com/umbrelos)：这是您可以在自己的硬件（Mini-PC、NUC、塔式电脑、专用笔记本电脑......）上自行安装的操作系统。您拥有与 Umbrel Home 相同的页面和App Store。



![Image](assets/fr/080.webp)



在这两种情况下，软件方面的用户体验是相同的：基于浏览器的管理、一键更新、按需安装应用程序...自我手动 解决方案通常比购买 Umbrel Home 更经济（取决于所使用的机器）。不过，我并不一定建议您总是选择自我手动方案，**因为购买 Umbrel Home 直接为项目开发提供资金**，因为其商业模式是基于硬件销售。老实说，2TB 的存储空间只需 389 欧元，从机器的质量来看，价格还是非常合理的。



![Image](assets/fr/079.webp)



下一章，我们将探讨如何在自己的机器上安装 umbrelOS DIY。不过，如果您选择了 Umbrel Home，也可以以同样的方式学习本 BTC 202 课程。



### 使用案例：从比特币节点到家庭云



Umbrel 可以保持极简风格，只专注于比特币，也可以发展成为真正的多功能个人服务器，具体取决于您的需求。以下是 Umbrel 的主要用途：





- **简单的比特币节点**：这是 Umbrel 从一开始就依赖的基础用途。您可以运行 Bitcoin Core（或 Knots），将您的钱包直接连接到您的节点，暴露一个 Electrum 服务器，托管您的内存池区块链浏览器以查看区块链，并估算费用...在本课程中，我们将重点介绍这些用途。



![Image](assets/fr/082.webp)





- **闪电网络**：Umbrel 还允许您部署 LND 或 Core Lightning（闪电网络的两种实现方式）来管理您自己的闪电节点。您将能够打开通道、管理您的流动性、进行支付、自动平衡、提供服务、连接远程钱包，或利用许多可用的应用程序对页面进行高级管理。我们将在下一期 LNP 202 课程中介绍这一特定用例。



![Image](assets/fr/083.webp)





- **一般自托管**：与 Nextcloud、Immich、Jellyfin/Plex、DNS 范围内的广告拦截器（Pi-hole/AdGuard）、VPN（WireGuard、Tailscale）、家庭自动化（家庭助理）、备份、笔记管理、办公工具、本地人工智能（Ollama + Open WebUI）...Umbrel 可以成为您的个人服务器，让您重新掌控自己的数据。您可以自行托管每天使用的服务，获得与外部解决方案极为相似的完美用户体验，同时保留对数据和隐私的完全控制。



通过在容器中部署应用程序，您可以随心所欲地塑造 Umbrel：从一个简单的比特币节点和几个与其生态系统相关联的应用程序开始，然后在比特币节点旁安装一个闪电节点，并逐渐用你需要的自托管应用程序来丰富你的实例。



### 社区和互助



与竞争对手相比，Umbrel 的主要优势之一是其庞大而高度活跃的用户社区。您可以主要通过 [他们的 Discord](https://discord.gg/efNtFzqtdx) 和 [他们的在线论坛](https://community.umbrel.com/) 与他们取得联系。在这里，您不仅能找到实用的建议，最重要的是，还能找到解决问题或修复漏洞的方法。这里是您开始使用 Coin 的好地方，也是您不断进步的好地方，最终还能帮助其他用户，让您不再一个人往前走。

### UmbrelOS 许可证



Umbrel 的代码是公开的（您可以查看、创造分叉和修改它），但它并不是根据真正的开源许可证发布的。事实上，umbrelOS 是根据[*PolyForm Noncommercial 1.0*]许可证（https://polyformproject.org/licenses/noncommercial/1.0.0/）发布的，尽管一些相关的开发工具是根据 MIT 许可证发布的。



在实践中，只要是出于个人和非商业用途，您几乎可以用 umbrelOS 做任何事情：修改、为非盈利目的重新分发、为自己或非盈利组织创建衍生产品，但必须遵守法律声明。



但是，禁止出售 Umbrel 或其衍生产品（例如，预装了 umbrelOS 的预组装机器），禁止提供与 Umbrel 相关的商业服务，禁止将其代码集成到产品中以牟利。



从技术上讲，本许可证不限制安装、审核或改编 Umbrel 供个人使用。从法律上讲，它保护项目免遭未经授权的转售或商业托管，尤其是云提供商的转售或商业托管。因此，尽管 Umbrel 的代码仍可公开访问，但它并不是开放源代码。



不过，商店中的每个应用程序都有自己的许可证，而且它们通常是开源的。

## 索引器：作用、运行和解决方案


<chapterId>4f93c07a-f0cb-435f-8b68-162f316d7039</chapterId>



如果您在学习本课程之前已经探索过比特币节点，那么您可能会遇到 "索引器"（Indexer）这个术语。这些工具，如 Electrs 或 Fulcrum，可以被添加到 Bitcoin Core 节点中。但它们的作用究竟是什么？它们在实践中是如何工作的？您是否应该在新的比特币节点上安装？这就是本章要探讨的问题。



### 索引器是什么？



一般来说，索引器是一个程序，它可以扫描一组原始数据，提取相关的关键字（如单词、标识符和地址），并建立一个称为 "索引" 的辅助文件，其中每个关键字都指向数据在语料库中的确切位置。这一预处理阶段需要占用 CPU 时间和一定的磁盘空间，但它消除了每次查询数据库时处理整个语料库的需要；只需查找索引，几乎就能立即得到响应。



通俗地说，这与书中的索引原理相同：如果要查找某条信息，与其重读整本书，不如查阅索引，直接找到要查找的信息所在的页面。



在比特币节点（如 Bitcoin Core）中，区块链数据以原始的、按时间顺序排列的形式存储。每个区块都包含交易，交易又包含输入和输出，没有按照地址、标识符或钱包进行任何特定分类。这种线性组织方式最适合区块验证，但不适合有针对性的搜索。例如，如果您想在非索引节点中找到与特定地址相关联的所有交易，就必须逐块逐交易地手动查看整个区块链。这正是比特币节点上的索引器的用武之地。



![Image](assets/fr/085.webp)



索引器是一种专门的软件程序，用于分析大量原始数据（区块链、Mempool、UTXO 集）并提取交易标识符、地址和区块高度等关键信息。它根据这些关键字建立索引，将每个关键字与节点存储中信息的确切位置关联起来。



![Image](assets/fr/086.webp)



通过索引器，您可以快速、准确、高效地搜索节点上的信息。例如，当您将钱包（如 Sparrow）连接到您的节点时，它几乎可以立即显示地址的余额。具体来说，它可以通过类似以下的请求来查询索引器："_哪些 UTXO 与某些Script-Hash相关联？_"索引器几乎会立即做出响应，而无需重新读取整个区块链，因为这些数据已列在其数据库中。



### Bitcoin Core 带有索引器吗？



如果不需要额外的软件，严格来说，Bitcoin Core 并不能提供与 Electrs 或 Fulcrum 等软件中的索引器相媲美的完整地址索引器。不过，Bitcoin Core 确实包含了几种内部索引机制，以及用于扩展查询功能的可选选项。为了全面了解情况，我们需要绕道了解一下该项目的历史。



在 Bitcoin Core 0.8.0 版之前，交易验证基于全局事务索引（称为 `txindex`）。该索引引用所有区块链交易及其输出。当节点收到一个新交易时，它会查阅该索引，以验证消耗的输出（在输入中）是否确实存在，并且尚未花费。因此，`txindex` 在当时的交易验证中是不可或缺的。



然而，这种方法有其局限性：速度慢、存储成本高、信息冗余。为了弥补这一缺陷，0.8.0 版本对验证模型进行了全面修改，称为 ***Ultraprune*** 。Bitcoin Core 不再以事务索引的形式存储所有信息，而是维护一个专门用于UTXO 的简单数据库，称为 `chainstate`（在日常语言中称为 "UTXO set" 或 "UTXO 集"），并在消耗和创建输出时更新其列表。



这种方法速度更快，而且只存储寄存器的当前状态，无需使用 `txindex` 索引器。不过，开发人员并没有删除 `txindex` 代码，而是选择将此功能保留在一个简单参数（`txindex=1`）后面。在节点上启用该选项后，就可以从其 `txid` 中查询任何事务。



与普遍的看法相反，Bitcoin Core 并不像 Electrs 或 Fulcrum 那样提供基于地址的索引。这个决定是出于多方面的原因：





- Bitcoin Core 的作用不是成为一个完整的区块浏览器，也不是提供一个针对每种用途的应用程序接口。整合基于地址的索引意味着需要长期维护，这超出了软件的初始范围。





- 大多数用例已经可以通过其他方式解决。例如，要估算地址的余额，可以使用 `scantxoutset` 命令，它可以直接查询 UTXO 集，而不需要完整的索引。





- 每个软件程序对要索引的数据格式或类型（地址、哈希脚本、专有标签等）都有特定的要求。让这些程序建立自己的定制索引，比在 Bitcoin Core 中固定一个通用解决方案更灵活、更合理。



Bitcoin Core 确实有一个可选的交易索引器（`txindex`），这是其历史操作的遗留物，但它不提供地址索引，也不直接提供用于复杂搜索的页面索引。因此，在某些情况下，添加一个外部索引器可能是有助的做法。



### 您是否应该在节点上添加一个地址索引器？



您并非必须添加 Electrs 或 Fulcrum 等地址索引器；这取决于您的具体需求。



如果您只是想将钱包（如 Sparrow）连接到您的节点，以查看余额和广播交易，完全可以直接通过 Bitcoin Core 的 Interface RPC，在本地或通过 Tor 进行远程连接。



另一方面，如果您想要使用更高级的软件，如运行本地 Mempool，安装地址索引器就成为区块浏览器必不可少的配置。



索引器需要一定的同步时间（少于 IBD），并会占用额外的磁盘空间。如果您的固态硬盘在下载区块链后仍有足够的可用空间，您可以轻松添加索引器。



### 选择哪种索引器？



有两个软件程序常用于建立此类地址索引并使其易于访问： **Electrs** 和 **Fulcrum**。这些工具根据哈希脚本（地址）为区块链编制索引，然后提出一个标准化的页面（Electrum 协议），许多钱包（如 Electrum Wallet、Sparrow 或 Phoenix）都与之连接。



![Image](assets/fr/087.webp)



简而言之，Electrs 相当紧凑：它索引区块链的速度更快，占用的磁盘空间更少，但查询性能略逊于 Fulcrum。相比之下，Fulcrum 消耗的磁盘空间更大，索引时间更长，但查询性能更优。



对于个人用户，我推荐使用 Electrs：它占用空间少，维护良好，虽然在某些请求上比 Fulcrum 稍慢，但对于日常使用还是绰绰有余的。如果您有时间和磁盘空间，也可以试试 Fulcrum，它的性能会好很多，尤其是在有大量地址需要验证的钱包上。



具体而言，到 2025 年 8 月，Electrs 将需要约 56 GB 的存储空间，而 Fulcrum 则需要约 178 GB。因此，索引器的选择也取决于您的存储容量：




- 如果磁盘空间非常有限，则只能使用 Bitcoin Core，而不能使用外部地址索引器。
- 如果您想使用索引器，但仍受容量限制，请选择 Electrs。
- 如果您有足够的磁盘空间，Fulcrum 正是您所需要的。



在本 BTC 202 课程的其余部分，我将使用 Electrs，但您也可以很容易地使用 Fulcrum：安装程序是相同的，页面与钱包的连接也是相同的，因为两者都暴露了一个 Electrum 服务器。

### Tailscale



我特别喜欢的另一个 Umbrel 应用程序是 Tailscale。它是一款 VPN 应用程序，旨在简化多台设备之间安全网络的创建，无论这些设备位于世界何处。与依赖中央服务器的传统 VPN 不同，Tailscale 利用 WireGuard 协议在不同机器之间建立端到端加密连接。这意味着您可以在几分钟内部署一个可用的 VPN，而无需复杂的网络配置。



在 Umbrel 上，安装 Tailscale 会将比特币节点连接到自己的虚拟专用网络。配置完成后，您的节点将获得一个私有的 Tailscale IP 地址，只有连接到同一 Tailscale 网络的其他设备（如电脑、智能手机和平板电脑）才能访问。这种连接是端到端加密的，不会通过未受保护的公共网络，与未加密连接相比，大大提高了安全性。



![Image](assets/fr/090.webp)



具体来说，在使用 Umbrel 时，Tailscale 可为您提供多种优势：





- 您可以在任何地方管理 Interface Umbrel 或访问与您的节点相关联的应用程序（如 Mempool、Ride The Lightning、ThunderHub等），就像在同一个本地网络上一样，无需在互联网上暴露端口，也无需通过速度非常慢的 Tor 网络；





- 您可以通过 VPN 连接到您的 Electrum 服务器（Electrs 或 Fulcrum），或者绕过 Tor 直接连接到 Bitcoin Core。这提供了一个安全的连接，与使用 Tor 相当，但速度更快，延迟更短。简而言之，您既能保留 Tor 的隐私和安全优势，又能享受 Clearnet 连接的速度。对于链上钱包来说，这种好处似乎微不足道，但如果您打算日后建立自己的闪电节点，这种差别就相当大了。事实上，在 Tor 上通过您的节点进行移动支付，由于需要进行大量交换，速度非常慢，而在 Tailscale 上则可以完美运行。





- 无需配置 NAT 规则、开放端口或设置传统的 VPN 服务器。一旦在 Umbrel 和您的设备上已安装了应用程序，网络就会自动建立。



因此，如果您想在保护隐私或安全的同时，从世界任何地方以安全、高性能和易于配置的方式访问你的节点，那么 Umbrel 上的 Tailscale 就是一个非常有趣的解决方案。

### Nostr



Nostr是 "*Notes and Other Stuff Transmitted by Relays（由中继器传输的笔记和其他东西）*"的首字母缩写，是一个开放的、去中心化的协议，旨在使信息能够在互联网上发布和交换，而无需依赖中心化平台。每个用户都有一对加密密钥：公钥（`npub`）用作标识符，私钥（`nsec`）用于签署信息并保证其真实性。



信息通过独立的中继网络传输。这种分布式架构使 Nostr 能够抵御审查：没有一个服务器可以控制访问或分发，用户可以随意连接多个中继器。



该协议在比特币社区非常受欢迎，因为与比特币一样，Nostr 也涉及数字主权和数据控制问题。该协议的创建者 Fiatjaf 是一名开发者，他的众多贡献已经得到了生态系统的认可。



有了您的 Umbrel，您可以优化 Nostr 的使用。通过安装***Nostr Relay***应用程序，您可以直接在您的机器上托管自己的私人中继器，确保您在 Nostr 上的所有帖子和互动都保存在本地，不会因公共中继的删除而丢失。



Nostr 客户端 ***noStrudel*** 或 ***Snort*** 也可在 Umbrel 上使用。有了这些应用程序，您就可以直接从 Umbrel 上的页面网络发布、阅读、搜索配置文件并与 Nostr 生态系统互动。

### 保持设备健康



可靠的节点始于稳定的硬件。确保安装节点的机器通风良好、无灰尘，并安装在干燥的环境中，远离任何热源和湿源。避免将其安装在狭窄的空间内，应选择通风良好的位置。



在 Raspberry Pi 和微型电脑上，灰尘最终会堵塞散热片，导致温度升高和节流（自愿限制资源使用），进而导致节点效率下降。因此，我建议定期清洁进气口和风扇，最好每隔几个月清洁一次。



确保使用高质量的电源，因为不稳定的电压会导致系统损坏，甚至造成火灾危险。最好使用机器制造商提供的原装电源。此外，还要小心电源插座的焦耳效应导致的过热：始终遵守最大允许的功率，切勿串联多个电源插座。



我还建议投资购买不间断电源。这可以保护您的节点免受突然关机的影响，使 Umbrel 能够在断电时干净利落地关闭，并确保在微停机或短期故障时的连续运行。



在存储方面，请密切关注进度：如果磁盘接近饱和，请考虑释放空间（卸载不使用的应用程序、调整索引器设置）或迁移到更大的固态硬盘。比特币全节点的缺点是其存储需求会持续增加，因为每 10 分钟就会生成一个新区块，而且旧块无法删除（除非是剪枝节点）。因此，我建议您在购买硬件时准备足够大的容量（至少 2 TB）。



### 更新



节点更新之所以重要，主要有三个原因：第一，安全性（包括漏洞修补、网络强化以及拒绝服务攻击（DoS）防护）；第二，兼容性（中继策略更改、格式更改和协议升级）；第三，可靠性和性能（错误修复、资源消耗和其他改进）。因此，请定期检查 UmbrelOS 和您的应用程序是否为最新版本：





- 更新系统打开设置选单，然后点击 "*UmbrelOS*" 参数旁边的 "*Check for Update*"按钮。



![Image](assets/fr/042.webp)





- 更新应用程序：前往 App Store。如果有任何应用程序需要更新，页面的右上角会出现一个带红色的按钮。只需单击该按钮，您就可以更新所有应用程序。



定期执行此操作可使操作系统和应用程序保持最新。



### 备份



如果您只使用比特币节点来验证和分发交易，但您的钱包是在 Umbrel 外部管理的（例如，使用硬件钱包和 Sparrow wallet），则无需直接备份到 Umbrel 上。在这种情况下，基本备份仍然是外部钱包的恢复短语和描述符，无论您是否使用自己的节点，这一点都适用。因此，与之前的配置相比没有任何变化。



另一方面，根据您在 Umbrel 上使用的其他应用程序，可能需要进一步备份。如果您在 Umbrel 上运行一个闪电节点，情况尤其如此。在这种情况下，绝对有必要备份安装闪电节点时提供的种子（seed）。除了种子，您还需要一个最新的***静态通道备份（SCB）***，以便在出现问题时恢复您的闪电节点。SCB 允许您通过强行关闭通道来恢复资金。如果种子或 SCB 丢失，则无法恢复闪电节点。



Umbrel 还提供通过 Tor 在其服务器上自动动态备份 SCB 的选项，以确保始终提供最新文件。在这种情况下，只需要种子就可以恢复节点。

## 了解 IBD 和同行发现过程


<chapterId>175ac9d1-ea23-45d9-9918-d3e7352435cd</chapterId>



您的比特币节点启动时，事先并不知晓任何交易历史记录。最初，它只是一台运行软件（Bitcoin Core 或类似软件）的计算机。要成为一个完全同步运行的比特币节点，它必须通过检查自创世区块（中本聪在 2009 年 1 月 3 日发布的第 0 个区块）以来发布的所有区块，在本地重建账本的状态。这一步骤称为 **IBD（_初始区块下载_）**。



IBD 包括下载和逐个验证每个区块和交易，应用共识规则，建立自己版本的区块链。其目的不是简单地检索未经验证数据的副本，而是与网络中诚实的大多数人一样，完全独立地得出相同的结论。



![Image](assets/fr/092.webp)



### IBD 阶段进度



区块同步从_**先下载区块头**_步骤开始。节点向多个对等节点请求区块头的序列，并对每个区块头验证工作量证明（Proof-of-Work）、难度调整、语法结构（syntax）以及时间戳和版本号规则。总之，它要确保收到的每个区块头都符合共识规则。



![Image](assets/fr/093.webp)



需要提醒的是，比特币数据块由一个 80 字节的区块头和一个交易列表组成。区块的 “指纹”（即区块哈希）是通过对区块头进行两次 SHA-256 哈希运算得到的。这个区块头包含 6 个字段：




- 版本
- 上一个区块的哈希
- 交易的哈希树根
- 时间戳（大于前 11 个区块的中位时间）
- 难度目标
- Nonce



![Image](assets/fr/094.webp)



区块中的交易会被写入到哈希树中。这是一种数据结构，用于汇总大量数据（在这里指区块中的所有交易），方法是将所有交易的哈希值两两合并，逐层向上汇总，直到形成一个唯一的“根”（哈希树根）。这样，对交易的任何修改也会生成不同的哈希树根值，从而修改区块头的指纹。SegWit 为签名引入了引入了一个独立的额外承诺，并将该承诺放入 Coinbase 交易中。



![Image](assets/fr/095.webp)



这一_***先下载区块头***步骤使节点能够识别工作量最大的分支（无论其区块数多少），也就是比特币节点同步的分支。一旦确定了这个分支，节点就会从多个连接并行下载区块的内容，然后验证每个交易：格式、脚本的有效性（`assumevalid=1` 除外）、金额以及是否存在重复支出。每次成功检查后，`chainstate/`数据库都会更新当前未消费的币（UTXO 集）的状态：删除已消费的输出，同时添加新的有效输出。



另一方面，Mempool 只有在接近链顶端时才会发挥作用：只要节点仍然处于滞后状态，它就没有待处理交易可存储。



IBD 完成后，节点进入正常阶段：验证发布的新区块，根据中继规则维护 Mempool 与待处理交易，中继交易和区块，并管理任何链重组。



### AssumeValid



Bitcoin Core 采用了一种机制，旨在缩短节点全面运行前所需的时间，同时保留自主验证原则的精髓：AssumeValid。



`assumevalid`　参数基于过去的参考区块，其哈希值已集成到每个软件版本中。在 IBD 期间，如果节点发现该区块确实是工作量最大的分支的一部分，就可以忽略在此之前所有交易的脚本验证。



所有其他规则（区块结构、Proof of Work、大小限制、交易金额、UTXO 等）仍完全通过验证。只有该参考区块之前的脚本计算会被忽略。由于签名验证占 CPU 负载的很大一部分，因此 IBD 的性能提升非常显著。在此参考块之后，验证将恢复到正常状态。



您可以使用 `Bitcoin.conf` 文件中的 `assumevalid=0` 参数，禁用这一机制，强制对所有脚本进行全面验证，但代价是 IBD 要长得多。



### AssumeUTXO



`assumeutxo` 是另一个现有参数，但与 `assumevalid` 不同，它不是默认激活的。这种机制使软件能够加载 UTXO 集及其元数据的快照，并在确认区块头确实对应于工作量最大的区块链之后，暂时将该快照视为参考状态。



这样，节点就能迅速投入正常使用状态（RPC、连接钱包等），同时在后台对自己的 UTXO 数据集进行完整的、经过验证的重建。一旦这一阶段已完成，最初加载的快照会被本地重新构建的状态所取代。这种做法将节点的快速启动与完整验证分离开来，同时不会削弱验证的安全性。


### 发现对等节点：您的节点如何找到比特币网络？



当节点第一次启动时，它还不认识任何对等节点。但是，它必须找到互联网上的其他比特币节点，请求区块头，然后请求区块，以完成其 IBD。为了启动这些连接，Bitcoin Core 遵循优先逻辑。



![Image](assets/fr/096.webp)



当节点在使用后重新启动时，Core 会首先尝试重新连接在关闭前注册的传出对等节点（信息存储在 `anchors.dat` 文件中）。然后，它会查阅其 IP 地址 书 **`peers.dat`**，其中存储了以前遇到的对等设备列表，以便与它们重新连接。这只是一个本地文件，由 Core 更新和保存。另一方面，对于刚刚启动的新节点，这两个文件是空的，因为它从未与其他比特币节点通信过。



在这种情况下，软件会查询 _**DNS 种子**_。这些是[由公认的生态系统开发者维护的服务器](https://github.com/Bitcoin/Bitcoin/blob/master/src/kernel/chainparams.cpp)，它们会返回推定活跃节点的 IP 地址列表。有了这些地址，新节点就能启动首次连接，并向 IBD 申请必要的数据。以下是迄今为止（2025 年 8 月）活跃的 *DNS 种子*列表：




- Pieter Wuille: `seed.Bitcoin.sipa.be.`
- Matt Corallo: `dnsseed.bluematt.me.`
- Luke Dashjr: `dnsseed.Bitcoin.dashjr-list-of-P2P-nodes.us.`
- Jonas Schnelli: `seed.Bitcoin.jonasschnelli.ch.`
- Peter Todd: `seed.btc.petertodd.net.`
- Sjors Provoost: `seed.Bitcoin.sprovoost.nl.`
- Stephan Oeste: `dnsseed.emzy.de.`
- Jason Maurice: `seed.Bitcoin.wiz.biz.`
- Ava Chow：`seed.Mainnet.achownodes.xyz.`



在绝大多数情况下，*DNS 种子* 步骤足以建立与其他节点的首次连接。例外情况下，如果这些服务器在 60 秒内没有响应，节点就会切换到另一种方法：Bitcoin Core 的代码中内置了 "种子节点" 的[超过 1000 个地址的静态列表](https://github.com/Bitcoin/Bitcoin/blob/master/src/chainparamsseeds.h)，并定期更新。如果前两种获取 IP 地址的方法都失败了，最后一种方法可以建立初始连接，然后节点可以从该连接请求新的 IP 地址。



![Image](assets/fr/097.webp)



最后，您可以通过 `peers.dat` 文件手动输入 IP 地址来强制特定连接。



启动后，内部地址管理器会将来源多样化（独立的自治网络、clearnet 和 Tor，以及不同的地理区域），以降低拓扑隔离的风险。节点建立这些出站连接（连接由它自己选择，因此更加安全）。



如果您的节点正在寻找一个开放端口（默认为 8333），它就会接受传入连接。这些连接为新节点提供了一个联络点，从而加强了网络的整体弹性，但不会给你自己的 IBD 带来任何特别的好处。如果您的节点是在 Tor 上运行的，则逻辑保持不变，但所使用的地址为 `.onion` 服务。




## 比特币节点的结构


<chapterId>b420bd9d-7e2a-4984-bc70-2b732a94c8ce</chapterId>



节点完成初始同步后，会在本地存储几个互补数据集，使其能够验证区块和交易、为网络对等节点提供服务，并在保持状态的同时快速重启。节点上有 3 块必不可少的部分：




- gW-402 **块**，存储在磁盘上、
- 在键值数据库中维护**UTXO 集**、
- 而 **Mempool** 则存储在 RAM 中，并定期序列化。



此外，还有一些辅助文件（对等点、费用估算、排除列表、钱包等）使整个系统更加完善。让我们来了解一下这些文件的作用。



### 节点数据的实际位置在哪里？



默认情况下，Bitcoin Core 会将数据保存在特定的工作目录中。在 GNU/Linux 下，通常位于 `~/.Bitcoin/`；在 Windows 下，位于 `%APPDATA%\Bitcoin/`；在 macOS 下，位于 `~/Library/Application Support/Bitcoin/`。如果您使用的是打包解决方案（例如，在节点分发版中），该目录可能挂载在其他地方，但其结构保持不变。下面描述的重要子文件夹和文件仍位于此处。



![Image](assets/fr/098.webp)



### 区块



区块链是区块的集合。全节点将这些数据块存储为顺序平面文件，并维护一个并行索引，以便快速检索。需要时（重组、Wallet 重新扫描、对等服务），这些数据将按原样重新读取。



**请注意：** 重组或重新同步是指区块链因同一高度上存在相互竞争的区块而改变其结构的现象。当区块链的一部分被另一条累积工作量更大的链取代时，就会发生这种情况。这些重新同步是比特币运行的自然组成部分，不同的矿工几乎可以同时挖出新区块，从而将比特币网络一分为二。在这种情况下，网络可能会暂时分裂成相互竞争的链。最终，随着其中一条链积累了更多的工作量，其他链就会被节点抛弃，它们的区块就会被称为 "过时区块 "或 "孤块"。这种用一条链取代另一条链的过程称为重新同步。



#### Blk*.dat 文件（原始区块数据）



接收并验证的区块会被写入名为 `blkNNNN.dat` 的顺序文件中，并存储在 `blocks/` 文件夹中。每个文件按顺序填充，直到达到 128 MiB 的最大容量，此时 Core 会打开下一个文件。在文件中，每个数据块都以网络格式序列化，前面加上魔法标识符和长度。这种组织方式可实现向磁盘的快速写入，并有助于区块服务同步同行。



![Image](assets/fr/099.webp)



在剪枝模式下，节点只保留这些文件的最近窗口，以限制磁盘占用空间。一旦达到配置的空间目标，它就会删除最旧的 `blk*.dat` 文件，同时保留足够的历史记录，以确保与当前已知的主链保持一致。区块索引和 UTXO 集保持正常，以便验证下一个交易和区块。



#### Rev*.dat 文件（取消数据）



为了在重组发生时回溯时间，Core 会在每个 `blk` 文件的同时，在 `blocks/` 中保存一个 `revNNNN.dat` 文件。该文件包含撤销程序块对 UTXO 数据集影响所需的信息：对于程序块消耗的每个输出，都会存储相应 UTXO 的先前状态（数量、脚本、高度等）。在区块中止的情况下，节点可以快速重建之前的状态，而无需重新扫描整个区块链。



![Image](assets/fr/100.webp)



#### 区块索引（区块/索引）



直接在平面文件中搜索区块太过耗时。因此，Core 在 `blocks/index/` 中维护了一个 LevelDB 数据库，其中列出了每个已知数据块的元数据，如哈希值、高度、验证状态、`blk` 文件以及它所在的偏移量。当对等程序请求区块或内部组件需要访问特定区块时，该索引可提供快速访问。如果没有这个索引，就需要进行过多的操作。



![Image](assets/fr/101.webp)



#### 可选索引 (indexes/)



有些索引是可选的，默认情况下是禁用的，因为它们会增加磁盘占用空间：




- 我们已经提到过的 `indexes/txindex/` 提供了一个交易 → 位置映射表，因此可以在未知包含该交易的区块的情况下检索任何已确认的交易。这对于钱包外的 `getrawtransaction` 类型的 RPC 查询非常有用，但成本相当高。
- `indexes/blockfilter/` 可包含用于瘦客户机的紧凑型块过滤器 (BIP157/158)。这些结构加快了客户端验证速度，但索引器节点上的存储空间却有所增加。



### UTXO 集（chainstate）



UTXO（*未花费的交易输出*）模型是比特币的会计表示：每个未用输出都是可用的 “币”，可用作未来交易的输入。



![Image](assets/fr/102.webp)



这些 UTXO 在某一特定时刻的总和构成了 UTXO 集：一个包含所有可用部件的大列表。节点正是参考这个状态来决定交易是否花费了合法单位，而这些合法单位尚未在之前的交易中使用过（以避免双重支付）。



![Image](assets/fr/103.webp)



UTXO 集作为一个紧凑的 LevelDB 数据库存储在 `chainstate/` 文件夹中。每个部分都关联了一个从交易的哈希导出的键和输出索引，其值包含：金额、`scriptPubKey` 锁、创建区块的高度和一个 Coinbase 指标。



![Image](assets/fr/104.webp)



节点会在 LevelDB 上方维护一个内存缓存，以吸收频繁的读写操作。可以使用 `dbcache` 参数来修改缓存的大小：缓存越大，IBD 和当前验证受益的内存访问就越多，但代价是需要消耗更多的 RAM。当矿工发现一个新数据块时，节点会从 UTXO 中删除该数据块中包含的事务花费（或消耗）的输出，并添加新创建的输出。



理论上，我们可以通过重新扫描区块历史记录来验证交易，以检查输出是否从未被花费过。但在实际操作中，这将耗费大量时间，因为每一笔新交易都必须扫描整个区块链。因此，UTXO 数据集提供了在合理时间内就地证明不存在双重支付所需的最小视图。



请注意，UTXO 集往往是比特币分散化问题的核心，因为其规模自然会迅速扩大。这一方面是由于比特币的价格不断上涨，推动了 UTXO 的分散，另一方面是由于该系统的采用率越来越高：用户越多，对 UTXO 的需求就越大。



![Image](assets/fr/105.webp)



UTXO 集的增长还源于比特币上简单支付交易的结构。事实上，当您进行支付时，您会消耗一个 UTXO 作为输入，并创建两个新的 UTXO 作为输出（一个用于支付，另一个用于找钱）。最后，被称为 CIOH（*所有权分析法*）的连锁分析启发式进一步鼓励人们避免 UTXO 合并。



https://planb.academy/courses/65c138b0-4161-4958-bbe3-c12916bc959c

由于用户必须将其中的一部分保存在 RAM 中才能在合理的时间内验证交易，UTXO 集可能会逐渐导致全节点的运行成本变得过高。一些建议已出现以解决这个问题，特别是 [Utreexo](https://planb.academy/resources/glossary/utreexo)。



### Mempool（内存池）



Mempool 是已收到但尚未确认的本地有效交易集。需要提醒的是，"已确认交易"是指已包含在有效区块中的交易。每个节点都维护自己的 Mempool，它可能与网络中其他节点的 Mempool 有所不同，具体取决于以下因素：




- 通过 `maxmempool` 参数为内存池（Mempool）分配的大小：：拥有较大 Mempool 的节点比拥有较小 Mempool 的节点能容纳更多的交易（除非后者的内存池已经被清空）；
- gW-433 规则：这些规则是节点中继规则的子集，规定了未经确认的交易必须符合的要求，以能在 Mempool 中被接受；
- 交易渗透：由于各种因素，某项交易可能已经分发到网络的某一部分节点，但尚未到达其他节点。



需要注意的是，节点 mempool 没有共识值。即使每个节点都有不同的 Mempool，比特币也能完美地正常运行。最终，被网络共识认可总是那些添加到区块链中的区块。例如，即使某个节点最初拒绝了其 Mempool 中的某个交易（其根据共识规则是有效的），但如果该交易最终被包含在一个具有有效 Proof-of-Work 的区块中，那么该节点就务必接受该交易。如果某个节点错误地拒绝了一个其实符合共识规则的区块，那么它就会触发一次硬分叉（Hard Fork）——也就是说，它会创造出一个新的、独立的比特币链，而且只有它自己在那条链上运行。



#### 内存政策和管理



收到交易时，Core 会根据共识规则（语法结构、有效脚本、无双重支付等）和 Mempool 规则（手续费替换(RBF)、最低收费阈值、`OP_RETURN` 中的数据限制等）进行一系列检查。如果交易符合这些规则，就会被存储在内存中。



Mempool 的大小受 `Bitcoin.conf` 文件中的 `maxmempool` 参数限制（将在下一章详细介绍）。默认情况下，该限制为 300 MB。当内存池满时，节点会提高最低收费阈值，并首先驱逐利润最低的交易（即保留应首先挖掘的交易）。时间过长的交易也会在配置的延迟后过期。



#### Mempool 在磁盘上的持久性



为了加快重启速度，当节点关闭时，Core 会定期将 Mempool 的状态序列化到 `Mempool.dat` 文件中。除了保留在内存中的实际 Mempool 外，Core 还将此 `Mempool.dat` 文件存储在磁盘上。下一次启动节点时，它会重新加载该快照，并删除对当前区块链无效的内容。



### 辅助文件和数据库



在与 `block/`、`chainstate/` 和 `indexes/` 目录同一层级的位置，还有其他几个文件，它们也在节点的正常运行中发挥着重要作用：




- `peers.dat` 保留了潜在对等节点的 IP 地址记录，由初始 DNS 发现、网络交换和手动添加提供。当节点启动时，它可以利用该文件建立向外连接。
- 当节点关闭时，`anchors.dat` 会保存传出的对等节点的地址，以便下次启动时可以快速尝试再次与它们连接。
- `banlist.json` 包含操作员或节点决定的本地禁令（重复无效行为），以防止节点重新连接或接受来自这些特定对等节点的连接。
- `fee_estimates.dat` 保存了在不同时间范围内观察到的交易确认统计数据，这些数据会被手续费估算器使用，以便在用户创建交易时，根据期望的确认延迟目标，给出相应的手续费建议。
- `gW-446.conf` 包含节点的配置参数。您可以在这里调整中继规则。下一章我会详细介绍。
- `settings.json` 包含 `Bitcoin.conf` 的附加参数。
- `debug.log` 是诊断文本日志，可用于在出现错误时了解节点活动。
- `gW-448.pid` 在运行时存储进程标识符，以便其他应用程序或脚本轻松识别 bitcoind (*Bitcoin daemon*)，并在必要时与其交互。它在节点启动时创建，并将在节点关闭时删除。
- `ip_asn.map` 是一个 IP → ASN 映射表（独立系统），用于分组和对等多样化（`-asmap` 选项）。
- 当启用 `-listenonion` 选项时，`onion_v3_private_key` 会存储 Tor v3 服务的私钥，以便在重启之间保持稳定的洋葱地址（Onion Address）。
- 当使用 `-i2psam=` 时，`i2p_private_key` 会存储 I2P 私钥，以便在 I2P 上建立传出连接和可能的传入连接。
- 当使用 cookie 身份验证时，`.cookie` 包含一个短暂的 RPC 身份验证令牌（启动时创建，关闭时删除）。例如，这可用于连接钱包软件。
- `.lock` 是数据目录锁，用于防止多个实例同时向同一数据目录写入数据。
- `guisettings.ini.bak` 是使用 `resetguisettings` 选项时自动保存的图形用户界面设置（*Bitcoin Qt*）。



正如我们在本 BTC 202 课程的第一部分中所看到的，Bitcoin Core 既是比特币节点软件，也是钱包软件。但是，它并不一定是我推荐的管理钱包的解决方案，因为与 Sparrow 或 Liana 等现代软件相比，它的页面仍然是最基本的，功能也很有限。Core 还包含用于管理钱包的文件：





- `wallets/` 是存放一个或多个的默认目录；
- `wallets/<name>/Wallet.dat` 是 Wallet 的 SQLite 数据库（键、描述符、交易元数据等）；
- `wallets/<name>/Wallet.dat-journal` 是 SQLite 回滚日志。

### 新区块的验证路径



收到新区块后，节点会检查工作证明，更广泛地说，检查是否符合共识规则。如果一切正常，它就会在其 UTXO 集上逐个交易应用更改：检查每个条目是否用有效脚本花费现有的 UTXO，删除这些 UTXO，并添加新的输出。如果一切正常，则将更改提交到 `chainstate/`。



同时，撤消数据将被写入 `rev*.dat` 文件，元数据被写入 `blocks/index/` 索引。然后，该区块姜被序列化到正确的 `blk*.dat` 文件中。如果发生重组，节点会反向读取 `rev*.dat`，以干净利落地断开废弃的区块，恢复 UTXO 集，然后连接新的最佳链的区块。





## 了解 Bitcoin.conf


<chapterId>c54a629a-ddb1-41cb-9a88-21dfd9be50ca</chapterId>



`Bitcoin.conf` 文件是 Bitcoin Core 的主要页面配置文件。它允许您调整节点的行为和参数，而无需重新编译源代码或修改命令行。具体来说，它是一个以键值对为结构的纯文本文件，这意味着文件中的每一行都引用了一个特定参数（键）及其相关值，可以通过修改该值来调整该参数。



网络、交易中继、性能、索引、日志和 RPC 访问参数可在 `Bitcoin.conf` 中定义。不过，该配置文件从不修改协议的共识规则：它只设定节点的本地策略（中继规则）、连接、索引和公开服务的方式。



### 地点和优先次序



默认情况下，`Bitcoin.conf` 位于 Bitcoin Core 数据目录下。这就是我们在上一章中提到的著名目录。不过，Bitcoin Core 不会自动创建该文件，除非在 Umbrel 等特定环境下。如果该文件不存在，则必须自行生成创建一个名为 `Bitcoin.conf` 的文件，然后在文本编辑器中打开该文件进行修改。



`Bitcoin.conf` 中定义的参数可由 2 层覆盖：




- `settings.json`（由页面图形或某些 RPC 动态编写）、
- 以及通过命令行修改的选项。



请注意，对 `Bitcoin.conf` 的任何修改都需要重新启动节点才能生效。



### 格式和结构



`Bitcoin.conf` 的格式非常简单：每个选项一行，格式为 `option=value`。不必要的空格和空行将被忽略，代码注释以 `#` 开头。



几乎所有布尔选项都可以用 `no` 前缀禁用。例如，`listen=0` 和 `nolisten=1` 是等价的，取决于版本。


若想要按网络划分配置，可以使用部分：`[main]`、`[test]、`[testnet3]`、`[testnet4]`、`[bookmark]`、`[regtest]`。或者，也可以在选项名称前加上 `regtest.maxmempool=100`。



### Bitcoin.conf 能做到的与做不到的事情



如上所述，共识规则显然不能在 `Bitcoin.conf` 中配置，因为这会产生硬分叉。另一方面，许多其他方面是可以配置的。这些可配置项大致可分为三类：




- 纯本地参数。这些参数只影响你的节点：缓存大小 (`dbcache`)、剪枝模式 (`prune`)、可选索引等。它们会影响您的节点性能，但不会影响网络。
- 中继和 Mempool 策略。这些策略决定了节点在确认前接受、保留和中继的交易：最低费用阈值（`minrelaytxfee`）、Mempool 大小和保留时间（`maxmempool`、`mempoolexpiry`）、交易替换（手续费替换，即 RBF）。这些规则不是共识的一部分，因此两个不同的节点可以有不同的策略，但仍然完全兼容。另一方面，这些参数会对比特币网络产生影响（正如第一部分所解释的，尤其是渗滤理论）。
- 网络连接。这些选项决定了节点如何查找对等点、监听、穿越 NAT、使用 Tor 或代理，或限制其带宽。它们塑造了您的拓扑结构，但不会改变交易的中继。



理解这种分离至关重要：如果交易不符合共识规则，您的节点无论如何都会拒绝它。但是，更严格的本地策略可能会拒绝转发在共识意义上有效的交易。



### 网络和拓扑结构



首先，我们必须明确区分比特币节点的两种连接类型：




- 传出连接，由我们的节点向另一个节点发起；



![Image](assets/fr/106.webp)





- 传入连接，由另一个节点向我们的节点发起。



![Image](assets/fr/107.webp)



这两种类型的连接完全可以在两个方向上交换相同的数据；这不是限制流量方向的问题，而只是连接发起者的不同。从我们节点的角度来看，传出连接通常被认为是更安全的，因为我们发起了连接，并精确地选择了要连接的节点，这使得恶意连接的可能性很小。默认情况下，Bitcoin Core 维护 10 个传出连接（其中 8 个为“完整转发连接（full-relay）”，另外 2 个为“仅区块转发连接（block-relay-only）”。）。



全节点通过接受传入连接为网络增加更多价值。使用 `listen=1` 参数可以监听相关网络的默认端口 8333，从而在清网上接收这些传入连接。为使其正常工作，路由器上的该端口也必须打开。如果没有打开，您的节点将继续只使用传出连接，这对您个人使用比特币没有任何影响。您可以选择是否允许传入连接；没有 "最佳选择"。



如果您不想在路由器上打开端口，但仍然接受传入连接，可以激活 `listenonion=1` 参数。这将达到相同的效果，但只能通过 Tor 网络而不是 clearnet。



在网络层面，我们还有以下参数：




- `addnode`：除通常的发现外，还添加一个友好的对等节点进行连接（可多次指定）。
- `connect`：严格限制连接到所提供的地址（可多次指定）。Core 不会连接到任何其他节点。
- `seednode`: 它仅用于在连接节点时填充地址簿，然后断开连接。
- `maxconnections`：定义传入和传出连接的全局上限。默认情况下，该参数设置为 125，这意味着您的节点永远不会接受超过 125 个连接。
- `maxuploadtarget`：设定上传上限，以限制 24 小时滑动窗口的带宽。此上限不会影响近期数据的基本传播。
- `onlynet`： 仅将传出连接限制到选定的网络（`ipv4`、`ipv6`、`onion`、`i2p`、`cjdns`）。例如，如果希望节点只通过 Tor 连接到比特币网络，可以启用 `onlynet=onion` 参数并禁止传入连接（或也只允许通过 Tor 连接）。
- `dnsseed`：允许或禁止 _DNS 种子_ 在本地地址池不足时请求对等体（默认值：`1`，除非 `-connect` 或 `-maxconnections=0`）。
- `forcednsseed`: 强制在启动时请求 _DNS 种子_，即使已有地址库存（默认值：`0`）。
- `fixedseeds`：如果 DNS 种子失败或被禁用，允许使用固定种子节点（即内置的地址列表，默认值：`1`）。
- `dns`：授权一般 DNS 解析（例如，用于 `-addnode`/`-seednode`/`-connect`）。



默认情况下，您的节点通过 clearnet、Tor 和 I2P 进行通信。这意味着它在 clearnet 上连接的对等网络可以看到您的公共 IP 地址，而您的 ISP 很可能会检测到您正在运行一个比特币节点（尽管 P2P Transport V2 让 ISP 更难窃听）。这不一定是个问题，但如果您想避免这些信息的泄露，您可以只通过 Tor 网络连接您的节点。



如果您想要完全启用 Tor，您需要强制 Bitcoin Core 只使用该网络，并为传入连接创建一个隐藏服务（如果您想启用的话）。您需要在 `Bitcoin.conf` 中添加以下配置：




- `onlynet=onion`、
- `proxy=127.0.0.1:9050`,
- `listenonion=1`、
- `torcontrol=127.0.0.1:9051`,
- `proxyrandomize=1`、
- `listen=1`、
- `bind=127.0.0.1`、
- `upnp=0`、
- `natpmp=0`。



您的所有 P2P 连接都会通过 Tor 传输。您的节点会收到一个 `.onion` 地址，用于传入连接，因此无需在路由器上打开端口。您的 ISP 只能看到 Tor 流量，而您的对等方则不知道您的实际公共 IP 地址。



为避免 DNS 解析不明，您可以在配置中添加 `dnsseed=0` 和 `dns=0`。然后，您需要通过 `seednode=` 或 `addnode=` 手动提供 `.onion` 对等节点，否则很难发现新节点。



显然，如果您是初学者，我建议您先不要使用这些网络设置。默认配置通常就足够了。



### Mempool 和中继政策



#### 基本参数



以下是您可以在 `Bitcoin.conf` 中修改的有关 Mempool 管理和转发未确认交易的基本参数：





- `maxmempool=<n>`：将本地 Mempool 的最大容量限制为 `<n>` 兆字节（默认值： `300`）。当达到限制时，您的节点会提高其有效费用阈值，并优先处理利润最低的交易（基于费率，而非绝对值），以保持低于限制。您可以将此设置设为默认值。如果您是独立的矿工，或者您想更准确地了解 Mempool 的拥堵情况并改进费用估算，那么提高有效阈值可能很有用。相反，减少该阈值可以节省 RAM，并在一定程度上节省其他系统资源。





- `mempoolexpiry=<n>`：Mempool 中未确认交易的最长保留时间（以小时为单位，默认为 `336`）。过了这段时间，即使仍有可用空间，交易也会被删除。





- `persistmempool=1`：保存 Mempool 停机时的快照，并在重启时重新加载（默认值： `1`）。这样可以加快重启后的恢复速度，避免通过网络重新学习状态。





- `maxorphantx=<n>`：保留的孤儿交易的最大数量（来自 UTXO 集尚未看到的 UTXO 的依赖输入，默认值：`100`）。超过此阈值后，最旧的交易将被删除，以避免缓存不受控制地增长。





- `blocksonly=1`：禁止接受和重传从对等节点收到的未确认交易（除非有特殊权限）。节点现在只上传和广播区块。本地创建的交易仍可广播（将节点与钱包软件一起使用）。这大大降低了对带宽和内存的要求，但代价是降低了对中继的实用性，以及对 Mempool 的完全不熟悉。





- `minrelaytxfee=<n>`：最低费率（以 BTC/kvB 为单位），低于此费率的交易不会被节点的 Mempool 接受，也不会转发给对等节点（默认值：`0.00001` = 1 sat/vB）。该值越高，节点过滤低成本交易的力度就越大。





- `mempoolfullrbf=1`：接受 RBF 交易，即使被替换交易中没有明确的 RBF 信号。使用这种 "*full-RBF*"策略，如果满足其他替换条件，提供较高费率的交易可以替换 Mempool 中的其他交易。



需要提醒的是，RBF 是一种交易机制，它能让发送方用手续费更高的交易来替换原来的交易，以加快确认速度。如果手续费过低的交易仍然被阻止，发送方可以使用 *Replace-by-fee（手续费替换）* 来提高手续费，并在内存池和矿工中优先处理其替代交易。



#### 高级和特定设置



下面是 Mempool 和中继策略的高级设置。如果您是初学者，应该不需要修改这些设置：





- `datacarrier=1`：允许通过 `OP_RETURN` 输出转发和（如果使用节点挖矿）包含携带非财务数据的交易（默认值：`1`）。停用此参数会略微减少非财务数据垃圾邮件的表面积，但代价是降低了与某些用途的兼容性。在任何情况下，您都必须接受已挖出的 `OP_RETURN`。





- `datacarrierize=<n>`：节点中继的 `OP_RETURN` 的最大大小（以字节为单位）（默认值： `83`）。降低该值可限制通过 `OP_RETURN` 传输的有效载荷。请注意，在未来的 Bitcoin Core 版本中，该限制将被默认取消。





- `bytespersigop=<n>`：参数，用于将签名交易转换为等效字节，以便进行中继限制评估（默认值：`20`）。这将根据本地策略规则影响对 `sigops` 的交易的接受。





- `permitbaremultisig=1`：允许中继*裸多签 （Bare Multisig）* P2MS 交易（默认：`1`）。这是在 UTXO 上建立多重签名条件的最古老脚本模板（由 Gavin Andresen 于 2011 年发明）。





- `whitelistrelay=1`：自动向白名单传入的对等节点授予中继权限（默认：`1`）。即使您的节点不处于一般中继模式，中继也会接受这些对等节点的交易。





- `whitelistforcerelay=1`：将 "*forcerelay*" 权限分配给具有默认权限（默认值：`0`）的白名单对等节点。这样，即使这些节点已经存在于 Mempool 中，它也会转发它们的交易，从而绕过反冗余机制。





- `whitebind=<[permissions@]addr>` / `whitelist=<[permissions@]CIDR>`：绑定页面或地址范围并为相应的对等设备分配细粒度权限：`relay`、`forcerelay`、`Mempool`（Mempool 内容请求）、`noban`、`download`、`addr`、`bloomfilter`。这对于向受信任的对等（如网关、局域网和内部服务）授予特权待遇非常有用。





- `peerbloomfilters=1`：启用对布隆过滤器 (BIP37) 的支持，以便向轻节点提供过滤后的区块/交易（默认值：`0`）。警告：这会增加资源负荷。





- `peerblockfilters=1`：向对等设备提供 BIP157 (*Neutrino*) 紧凑型过滤器（默认值：`0`）。





- `blockreconstructionextratxn=<n>`：为重建区块而保留在内存中的额外交易数量（默认：`100`）。在紧凑同步期间提高重建的成功率，但代价是占用少量内存。



需要提醒的是，这些中继规则都不会影响有效区块中包含的交易的有效性。它们的作用是调整您对中继的贡献，保护您的资源，并使您的节点在受限环境中具有可预测性，但绝不允许您拒绝遵守共识规则的区块。



### 钱包



您还可以在 `Bitcoin.conf` 文件中调整钱包的管理方式。如果您不是直接在 Core 中使用钱包，而是使用外部管理软件（如 Sparrow 或 Liana），以下参数对您来说不太重要：





- `addresstype=<legacy|P2SH-SegWit|bech32|bech32m>`：定义钱包生成的接收地址格式。





- `changetype=<legacy|P2SH-SegWit|bech32|bech32m>`：用于强制指定找零地址的格式（即单笔交易中输入的找零部分的地址类型）。





- `Wallet=<path>`：启动时加载现有的钱包（可重复加载多个钱包）。





- `walletdir=<dir>`：指定钱包文件所在的目录。默认情况下，如果 `<datadir>` 下存在 `wallets` 文件夹，则使用 `<datadir>/wallets`；否则，就直接使用 `<datadir>` 作为钱包目录。





- `walletbroadcast=1`：自动广播已加载钱包创建的交易（默认：`1`）。如果希望通过其他渠道管理广播，则设置为 `0`。





- `walletrbf=1`：启用后，如果交易长时间未被打包，可以通过提高手续费来重新广播该交易，从而加快确认速度。





- `txconfirmtarget=<n>`：交易的确认目标（区块数，默认：`6`）。钱包将自动为在此区块数内确认的交易设置费用。





- `paytxfee=<amt>`：适用于钱包交易的固定费率（BTC/kvB）。注意：一般来说，请通过 `txconfirmtarget` 使用自适应估算。





- `fallbackfee=<amt>`：指定一个备用手续费率（以 BTC/kvB 为单位），当手续费估算器（fee estimator）缺乏足够数据时，就会使用这个费率。默认值是 `0.00`。如果将该值设为 `0`，则完全禁用备用手续费。





- `mintxfee=<amt>`：钱包创建交易的最小费用阈值（BTC/kvB）（默认值：`0.00001`）。钱包将拒绝创建低于此阈值的交易。





- `maxtxfee=<amt>`：钱包交易总费用的绝对上限（默认值：`0.10`BTC）。防止因异常设置而支付过高的手续费，避免大量的比特币损失。





- `avoidpartialspends=1`：按地址集群选择 UTXO，以避免部分支出。





- `spendzeroconfchange=1`：允许未确认的 UTXO 在新的交易中再次被使用（默认：`1`）。





- `consolidatefeerate=<amt>`：指定一个最高手续费率上限（以 BTC/kvB 为单位）。当当前手续费高于这个上限时，钱包会避免为了 “整合输入”（consolidation）而添加额外输入。





- `maxapsfee=<n>`：为启用 “避免部分花费” 选项设定的额外手续费预算上限（单位：BTC，绝对值）。





- `discardfee=<amt>`：指定一个丢弃费率（以 BTC/kvB 为单位），表示钱包愿意为“舍弃小额找零”支付的容忍度。





- `keypool=<n>`：预生成的地址池的大小（默认值：`1000`）。数值太小会增加还原不完整的风险。





- `disablewallet=1`：在不使用钱包子系统的情况下启动 Bitcoin Core，并禁用相关的 RPC。如果节点仅用于验证/释放，则可减少攻击面和足迹。



### 存储、索引和性能



Bitcoin Core 的配置文件还允许您调整与机器相关的参数。如果您的资源有限，或者可用容量较大，这一点尤为重要：





- `datadir=<dir>`：设置 Bitcoin Core 的主数据目录。





- `blocksdir=<dir>`：将区块文件（`blocks/blk*.dat` 和`blocks/rev*.dat`）的位置与`datadir`分离。这样做的好处是，可以将区块存档放在另一块存储设备上，同时将状态基础（`chainstate/`）保留在更快的介质上，例如 SSD，以提高性能。





- `dbcache=<n>`：为区块索引和 `chainstate` 使用的数据库缓存（*LevelDB*）分配 `<n>` MiB（默认值：`450`）。数值越大，IBD 和当前验证的速度越快，但代价是需要消耗更多内存。





- `prune=<n>`：启用区块文件剪枝并设置磁盘空间目标值（单位：MB，默认：`0` = 禁用；`1` = 通过 RPC 手动剪枝；`>=550` = 低于目标值自动剪枝）。与 `txindex=1` 不兼容。节点仍是完全验证节点，但不能再提供旧的历史记录。如果磁盘空间有限，例如在家用电脑上安装节点时，该选项尤其有用。





- `txindex=1`：建立并维护已确认交易的全局索引。对于某些查询（`getrawtransaction` 对非钱包交易的查询）和探索目的必不可少，但会显著增加磁盘占用。与剪枝模式不兼容。





- `assumevalid=<hex>`：表示假定有效的区块，允许跳过对其祖先的脚本检查（设置 `0` 则检查所有内容）。更多信息请参阅上一章。





- `reindex=1`：从磁盘上的 `blk*.dat` 文件重建区块索引和状态（`chainstate`）。同时重建可选的活动索引。使用该操作修复损坏的数据库或激活/停用重索引非常耗时。





- `reindex-chainstate=1`：只重建当前区块索引中的 `chainstate`。区块文件健康时优先使用。





- `blockfilterindex=<type>`：维护轻节点 (BIP157/158) 和某些 RPC 使用的紧凑型区块过滤器（例如，`basic`）的索引。默认为已禁用 (`0`)。占用额外的磁盘空间和索引时间。





- `coinstatsindex=1`：维护由 `gettxoutsetinfo` 调用操作的 UTXO 集统计索引。这对审计和度量非常有用，可避免昂贵的重新计算。默认情况下，该功能是关闭的。





- `loadblock=<file>`：启动时从外部 `blk*.dat` 文件导入区块。用于从离线源（本地副本、外部媒体）预载历史记录，以加快初始化。





- `par=<n>`：设置脚本验证线程数（取值范围：`-10` 到 `15`，`0` 表示自动，负数表示保留相应数量的 CPU 核心不参与）。这个参数可以让你调整验证交易和区块时的 CPU 并行度。





- `debuglogfile=<file>`：指定 `debug.log` 日志的位置。





- `shrinkdebugfile=1`：减小启动时 `debug.log` 的大小（默认值：当 `debug` 未激活时为 `1`）。





- `settings=<file>`：动态设置文件 `settings.json` 的路径。



### RPC 访问参数和操作安全



最后，`Bitcoin.conf` 文件还允许您配置节点的访问参数。不过在修改这些设置时要格外小心，尤其是刚开始接触时：如果在不了解其影响的情况下随意更改，可能会带来安全漏洞。





- `server=1`：激活 JSON-RPC 服务器。如果您通过 `bitcoin-cli` 或第三方应用程序来驱动 `bitcoind`，则此功能必不可少。在不公开任何 API 或已使用 Electrum 服务器的纯验证节点上禁用 (`0`)。





- `rpcbind=<addr>[:port]`：RPC 服务器监听地址/端口。默认情况下，只在本地监听（`127.0.0.1` 和 `::1`）。如果未同时定义 `rpcallowip` 则忽略此参数。可以通过此参数明确限制 RPC 接口的可访问范围。





- `rpcport=<port>`：RPC 端口（默认值：Mainnet 为 `8332`，Testnet 为 `18332`，bookmark 为 `38332`，regtest 为 `18443`）。





- `rpcallowip=<ip|cidr>`：允许来自指定 IP 或子网的 RPC 客户端（可重复）。与 `rpcbind` 结合使用，只向受信任的网段（LAN/VPN）公开 API。





- `rpcauth=<USERNAME>:<SALT>$<Hash>`：推荐的 RPC 身份验证方法（散列密码）。允许多个条目，并避免以明文存储秘密。





- `rpccookiefile=<path>`：身份验证 cookie 的路径（默认：`datadir/`下的`.cookie`文件）。用于同一用户的本地访问，无需管理持久密码。例如，您可以使用它将 Liana Wallet 连接到同一台机器上的 Bitcoin Core。





- `rpcuser=<user>` / `rpcpassword=<pw>`：使用明文密码的传统 RPC 身份验证。请避免使用，而应使用 `rpcauth` 或 cookie。





- `rpcthreads=<n>`：为 RPC 调用提供服务的线程数（默认：`4`）。如果监控/外部工具方面的调用峰值较高，请增加该值。





- `rpcwhitelist=<USERNAME>:<rpc1>,<rpc2>,...`：授权 API 的白名单。通过限制可访问的方法来减少攻击面。





- `rpcwhitelistdefault=1|0`：设置 RPC 白名单的默认行为。启用后，未列入白名单的调用将被拒绝；如果白名单为空，则默认不允许任何调用。





- `rest=1`：启用公共 REST API（默认：禁用）。仅在受信任的网络中公开（与 JSON-RPC 的注意事项相同）。





- `conf=<file>`：仅在命令行中指定只读配置文件。可用于在操作侧冻结执行配置文件（不可变）。





- `includeconf=<file>`：加载额外的配置文件（路径相对于 `datadir/`）。允许角色分离：通用基础 + 敏感的本地重载。





- `daemon=1` / `daemonwait=1`：在后台启动 `bitcoind`，并使用 `daemonwait` 等待初始化完成后再移交。这有助于与监控程序（systemd、runit）集成。





- `pid=<file>`：PID 文件的位置。





- `sandbox=<log-and-abort|abort>`：启用实验性系统调用沙箱：只允许预期的系统调用。





- `startupnotify=<cmd>` / `shutdownnotify=<cmd>`：在启动或关闭时执行命令。





- `alertnotify=<cmd>`：收到警报时将触发命令。





- `blocknotify=<cmd>`：为每个新区块执行一条命令。





- `debug=<category>|1` / `debugexclude=<category>`：启用/禁用详细日志类别（例如，`net`、`Mempool`、`RPC`、`validation`等）。





- `logips=1`：记录 IP 地址。





- `logsourcelocations=1` / `logthreadnames=1` / `logtimestamps=1`：分别为日志添加源位置、线程名称和精确的时间戳。





- `printtoconsole=1`：将追踪与调试信息输出到控制台 (*stdout*)。





- `help-debug=1`：显示调试选项的帮助信息后退出程序。





- `uacomment=<cmt>`：在节点的 User-Agent（P2P 标识）中添加自定义注释。




现在我们已经列出了大部分配置参数。因此，这个 `Bitcoin.conf` 文件构成了节点的真正仪表板：它定义了网络配置、Mempool 管理、磁盘和内存使用、索引和一般管理。如果您想进一步了解该文件，并根据自己的需要创建一个文件，我建议您使用 [Jameson Lopp 的生成器](https://jlopp.github.io/Bitcoin-core-config-generator/)。
