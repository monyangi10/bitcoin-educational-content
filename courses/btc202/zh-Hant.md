## 什麼是 Bitcoin 節點？


<chapterId>0a9fd4e0-94ab-405e-924c-023397393027</chapterId>



正如其創造者 Satoshi Nakamoto 所描述的，Bitcoin 以點對點電子現金系統自居。這個簡單的句子，也就是白皮書的標題，包含了許多關於 Bitcoin 本質的線索：




- 首先，Satoshi 將 Bitcoin 描述為「系統」，換句話說，是一套連貫的硬體和軟體元件，這些元件互動以提供特定服務或執行特定功能；
- 接下來，他解釋這個系統可以使用電子現金，也就是一種無形的貨幣；
- 最後，他指出這個系統並不依賴任何中央實體：它是「點對點」的，意即由使用者自己來操作系統。



由於 Bitcoin 是一個系統，因此必須在電腦上執行。而且，由於其點對點的性質，這些電腦的運行是由使用者自己負責的。我們稱之為「Bitcoin 節點」的電腦，正是執行 Bitcoin 通訊協定 (如 Bitcoin core，但我們稍後會再談到) 軟體的電腦。這就是 Bitcoin 能夠在沒有中央機關的情況下運作的原因：驗證是以分散的方式進行，由屬於數以千計使用者的數千台獨立機器執行。



![Image](assets/fr/047.webp)



Nakamoto, S. (2008). *Bitcoin：點對點電子現金系統*。https://Bitcoin.org/Bitcoin.pdf



正是這些使用者確保了 Bitcoin 的安全性。正如 Eric Voskuil 在他的著作 *Cryptoeconomics* 中所解釋的，Bitcoin 的安全性既不依靠 Blockchain，也不依靠雜湊能力，更不依靠驗證、去中心化、加密學、開放源碼或博弈理論。Bitcoin 的安全性主要取決於願意承擔個人風險的個人。去中心化允許這種風險分散到大量個人身上，只有他們的抵抗能力才能確保系統的穩健性。



這個原則很容易理解：如果 Bitcoin 只依賴一個人所擁有的單一節點，監禁這個人就足以關閉網路，因為他們將獨自承受所有風險。如果數以萬計的節點散佈在世界各地，風險就會擴散：要關閉 Bitcoin，就必須瓦解每個操作者。



![Image](assets/fr/048.webp)



因此，我們可以區分並命名幾個概念，以便在本課程的其餘部分釐清事情：




- Bitcoin 貨幣：本系統內交易使用的會計單位；
- Bitcoin 網路：所有連線節點的集合；
- Bitcoin 節點：執行 Bitcoin 實作的機器；
- Bitcoin 實作：將協定轉譯為可執行指令的軟體；
- Bitcoin 通訊協定：規範系統作業的一組規則；
- Bitcoin 系統：所有這些 Elements 的連貫組合。



### Bitcoin 節點的作用



Bitcoin 節點共同組成所謂的 Bitcoin 網路。這些節點可讓整個系統自主運作，無須求助中央機關或伺服器層級。



從一開始，Bitcoin 的設計就允許每位使用者執行一個個人節點。這種情況在今天的 Bitcoin core 軟體中仍然有效，它結合了 Wallet 和節點的角色。但現在，這個功能經常被分離出來：許多現代的 Bitcoin 錢包只是連接外部節點（無論是否為同一人所有）的錢包。



### 保持 Blockchain



節點的第一個任務是維護 Blockchain 的本機副本。為了防止 Bitcoin 上的 Double-spending 而不涉及中央機關，每個使用者都必須檢查系統中是否存在任何交易。要確定這一點的唯一方法是知道在 Bitcoin 上進行的所有交易。基於這個原因，所有的交易都有時間戳記，並歸組成區塊，每個節點都會儲存整個 Blockchain。



> 確認沒有交易的唯一方法，就是注意所有的交易。

Nakamoto, S. (2008)。 *Bitcoin：點對點電子現金系統*。https://Bitcoin.org/Bitcoin.pdf。



因此，Blockchain 是一個不斷演進的暫存器：每次 Miner 發佈新區塊時，節點都會檢查其有效性，然後將其加入到自己的本地區塊鏈副本中。截至今日 (2025 年 7 月)，完整的 Blockchain 已超過 675 GB，而且這個大小還在持續增加，因為平均每 10 分鐘就會新增一個新區塊。



![Image](assets/fr/049.webp)



該節點還會維護在任何特定時間存在的所有 UTXO 的本地記錄，稱為 **UTXO 集**。這個資料庫包含所有未使用的 Bitcoin 碎片。我們會在課程的最後部分再詳細討論這個主題。



### 驗證和分配交易



節點的第二個角色是確保交易的驗證和傳播。當新的交易到達節點 (透過 Wallet 軟體或其他節點)，節點會檢查它是否符合一組規則 (共識規則和中繼規則)。例如：




- 已花費的 bitcoins 必須存在於其 UTXO 集 (未花費輸出的資料庫)；
- 簽章必須有效，且必須符合所有支出條件 (有效的腳本)；
- 輸出總量不得超過投入總量，這表示成本不能為負。



![Image](assets/fr/050.webp)



驗證完成後，交易會儲存在節點的 Mempool (預留給未驗證交易的臨時記憶體空間)，然後再轉發給與之相連的其他網路對等體。此分發與確認機制在節點與節點之間持續進行。如此一來，交易會在 Bitcoin 網路中傳播，每個節點都會將交易儲存在 Mempool 中，直到 Miner 將交易納入有效區塊中，再由 Miner 對其進行第一次確認。



### 檢查和分配區塊



節點的第三個角色涉及管理已挖出的區塊。當 Miner 發現具有有效 Proof of Work 的新區塊時，就會在網路中廣播。節點會接收它，檢查它是否符合所有通訊協定規則，如果有效，就把它整合到自己本地的 Blockchain 副本中。與交易一樣，新驗證的區塊隨後會轉發給與節點連接的所有對等節點。這個過程一直持續到 Bitcoin 網路上的所有節點都知道新區塊為止。



![Image](assets/fr/051.webp)



## 弓箭和 Wallet 有什麼不同？


<chapterId>de5af634-a628-4b90-b869-468c208e178b</chapterId>



使用 Bitcoin 時，必須區分兩種截然不同的軟體：節點和 Wallet。



如上所述，Bitcoin 節點是一個積極參與點對點網路的軟體。它主要執行三項任務：




- Blockchain 的備份、
- 交易驗證和中繼、
- 區塊驗證及中繼。



另一方面，Bitcoin Wallet 是專門用來儲存和管理您的私人金鑰的軟體。這些金鑰可讓您滿足鎖定腳本 (通常是透過簽名) 的要求來花費您的比特幣。Wallet 可以連接到節點（無論是本機或遠端），以便查詢 Blockchain 的狀態，並廣播其建立的交易，但它本身不是網路的參與者。



在某些情況下，這兩種功能會共存於同一軟體中，例如 Bitcoin core，它同時兼具 Full node 和 Wallet 的功能。不過，許多流行的 Wallet 程式 (Sparrow、BlueWallet 等) 都需要連線到外部節點 (無論是您自己的節點或第三方的節點)，才能廣播交易並決定 Wallet 的餘額。



![Image](assets/fr/052.webp)



## 節點和 Miner 有什麼不同？


<chapterId>d2992614-7ab7-4bf9-81b1-f548cda67257</chapterId>



節點和 Miner 的概念經常被混淆。然而，這兩種 Elements 在系統中卻執行著截然不同的功能。



最初，Bitcoin 由 Satoshi Nakamoto 於 2009 年推出時，期望每位使用者都能整體參與該網路。因此，最初的 Bitcoin 軟體同時結合了幾種功能：它既是 Wallet，一個節點，也是 Miner，能夠產生新區塊。當時，Mining 的難度非常低。您只需在電腦上執行 Bitcoin 軟體，就能找到區塊並獲得比特幣作為獎勵。



然而，隨著 Bitcoin 的逐漸普及以及礦工數量的增加，Mining 的競爭格局也發生了翻天覆地的變化。如今，Mining 已經成為一種競爭極為激烈的活動，由配備了專業基礎設施的產業玩家主導。現在，開採一個新區塊所需的能量已大到個人使用者僅使用傳統電腦幾乎不可能達成的地步。因此，Mining 現在主要使用稱為 ASIC（*特定應用集成電路*）的專用機器進行。這些晶片經過最佳化，專門用來在 Bitcoin 上執行雙重 SHA-256，也就是 Mining 所使用的演算法。



![Image](assets/fr/053.webp)



面對這樣的演進，Bitcoin 節點和 Miner 的角色已明顯區別。如上所示，Bitcoin 節點的角色純粹是資訊性和驗證性的。Miner 的作用則有所不同：




- 它選擇 Mempool 中的待處理交易。
- 它會建立一個整合這些交易的候選區塊。
- 他透過試驗和錯誤來尋找有效的 Proof of Work。
- 如果它找到有效的證明，它會透過自己的節點向其他節點廣播該區塊。



Miner 需要 Bitcoin 節點才能與網路互動。



Miner 的角色有時亦會與剁肉機有所區別。剁手機的任務是 Hash 模版區塊，由池的伺服器提供，尋找滿足為股份定義的難度目標的哈希值，而不是 Bitcoin 的哈希值。Mining 程序的其餘部分，包括實際區塊建構、交易選擇或根據 Bitcoin 本身的難度進行 Proof-of-Work 搜尋，以及分發，則直接由池執行。



![Image](assets/fr/054.webp)



最後，Miner 與節點在經濟誘因上有一個重要的差異。運行 Bitcoin 節點不會帶來直接的金錢利益。另一方面，參與 Mining 會為找到的每個區塊帶來獎勵 (補貼和交易費用)。



在第二部分中，我們將更詳細地探討安裝和使用 Bitcoin 節點所帶來的實際和個人效益，而不只是純粹的財務效益。



## Bitcoin core 與通訊協定實作


<chapterId>72381876-9317-4faa-8d41-2b252a945b8a</chapterId>



Bitcoin 協定不是軟體：它是一組網路使用者之間共享的默認規則。它定義了交易有效性條件、貨幣創建機制、區塊格式、Proof-of-Work 條件以及許多其他規格。若要與此通訊協定互動，使用者必須執行實作這些規則的軟體：這稱為 Bitcoin 的**實作**。



因此，實作就是節點軟體：一個能夠與 Bitcoin 網路上的其他機器互動、下載、驗證、儲存和傳播區塊與交易，以及在本地執行共識和中繼規則的程式。每個實作都是協定的具體詮釋，以特定的程式語言撰寫，擁有自己的架構、效能和人體工學。每個實作也都有自己的開發組織，有自己的職責分工。



在這些實作中，有一種實作至今仍佔主導地位： **Bitcoin core**。



![Image](assets/fr/055.webp)



### 成為基準的歷史性實作



Bitcoin core 是 Bitcoin 協定的參考軟體。它源自 Satoshi Nakamoto 在 2008-2009 年所寫的原始程式碼，是其直接延續。最初被稱為 "*Bitcoin*「，後來又被稱為 」*Bitcoin QT*「（由於通過 Qt 函式庫增加了圖形化的 Interface），2014 年更名為 」*Bitcoin core*"，以明確區分軟體與網路。自版本 0.5 起，它以兩個元件發行：`Bitcoin-qt`（圖形化 Interface）和`bitcoind`（命令列 Interface）。



理論上，Bitcoin core 並不代表 Bitcoin 通訊協定；相反地，它只是眾多實作中的一種。不過，它的特色在於被大量採用、歷史悠久、代碼穩健、開發過程嚴謹。因此，在實務中，Bitcoin core 所應用的規則實際上就是 Bitcoin 通訊協定的規則：使用者、開發者、礦商和生態系統服務幾乎完全參考 Bitcoin 通訊協定。



### 目前的實作分佈



根據 [Luke Dashjr 於 2025 年 8 月收集的資料](https://luke.dashjr.org/programs/Bitcoin/files/charts/software.html)（生態系統中的知名開發人員），網路公共節點間的實作分佈如下：




- Bitcoin core**： 87.3% 的節點
- Bitcoin Knots**： 12.5
- 其他累計實現**： 0.2% (btcsuite, Bcoin, BTCD...)



![Image](assets/fr/056.webp)



換句話說，10 個公共節點中約有 9 個在執行 Bitcoin core。其餘的網路則仰賴較為邊緣化的用戶端 (雖然近幾個月來 Knots 的佔有率急遽上升，尤其是在「OP_RETURN」大小限制的爭論之後)。這些替代實作通常由一個人或一個小團隊維護。



**註：** 這些數字仍是估計值，因為它們主要是基於*聆聽節點*，也就是接受傳入連線的節點 (開啟了 8333 連接埠)。非監聽節點*的計算要複雜得多，因為您不可能直接連線到這些節點：您必須等待這些節點以傳出連線的形式主動提供連線。Luke Dashjr 的網站聲稱也在嘗試計算這些 * 非監聽節點 *，但仍無法取得完全精確的相關資料，而且這些統計資料的更新也不可避免地落後於現實。



### Bitcoin core 的內部運作



Bitcoin core 以 C++ 寫成。它也是一個開放原始碼專案，由志願或由不同單位（通常是由生態系統中對 Core 的開發有既得利益的公司）付費的開發人員社群維護。[Code is hosted on GitHub](https://github.com/Bitcoin/Bitcoin), and development follows a rigorous：




- 貢獻者**以_pull request_ (PR) 的形式提交提案。原則上，任何人都可以提出變更，但必須經過測試、記錄並通過同儕審查程序。
- 維護者**有權批准和合併 PR。他們保證專案的連貫性和穩定性。在 2025 年 7 月，共有五位維護員：Hennadii Stepanov、Michael Ford、Andrew Chow、Gloria Zhao 和 Ryan Ofsky。
- 自 2023 年 2 月起就沒有**首席維護員。這個角色最初在 Bitcoin 推出時由 Satoshi Nakamoto 擔任，然後在 2011 年初 Nakamoto 離任後由 Gavin Andresen 擔任，最後在 2014 年至 2023 年間由 Wladimir J. Van Der Laan 擔任。



![Image](assets/fr/057.webp)



Bitcoin core 的開發遵循任人唯才的邏輯：鼓勵新的貢獻者在自己提出任何修改之前先審閱並測試程式碼。決策以技術共識為基礎，重大修改 (特別是在有共識的領域) 需要在公共管道 (如郵件列表或公關審查俱樂部) 上進行上游討論。



### 其他 Bitcoin 實作



儘管在採用方面微不足道，但其他用戶端確實存在。最主要的是由 Luke Dashjr 開發的 Bitcoin Knots，它是 Bitcoin core 的 Fork，加入了額外的選項和更保守的開發方式。其中包括對交易格式更嚴格的限制。



![Image](assets/fr/058.webp)



我們也可以提一提：




- Libbitcoin**: 由 Amir Taaki 開發、Eric Voskuil 維護的模組化 C++ 函式庫；
- Bcoin**: JavaScript 實作，已不再積極維護；
- BTCD/btcsuit**e: Go 中的實作。



這些專案貢獻了生態系統的多樣性，但其採用仍非常有限，使得 Bitcoin core 難以獨立演進。



### 核心開發人員的力量



您可能認為 Bitcoin core 開發人員可以直接控制 Bitcoin，但事實並非如此。他們無法強行更改通訊協定。他們的角色是提出程式碼。每個使用者都可以透過他們的節點來決定是否使用這些程式碼。



這表示，如果 Bitcoin core 中的變更未達成共識，節點可以忽略它，不更新 Bitcoin core 或直接改變實作。相反地，如果使用者想要的功能在 Core 開發過程中受阻，隨時可以改用其他實作或 Fork 專案。



正如我們稍後在本課程中所討論的，是節點（即商家）根據其經濟權重，通過接受尊重其規則的單位，為某個版本的協定（以及相應的貨幣）賦予效用。因此，治理 Bitcoin 的真正權力在於這些商家，而非開發者。

## 為什麼要自己打結？


<chapterId>39c0cd19-67f9-4c64-bfb3-dbd6eec0bf42</chapterId>



有一種廣泛的信念認為，操作 Bitcoin 節點是一種純粹利他的行為，沒有任何個人利益，完全是為網絡去中心化服務。有些人認為這是比特幣使用者支持系統的一種責任，並對 Bitcoin 表示感謝。



事實上，正如我們在前幾章所指出的，打結並沒有直接的經濟利益。因此，人們可能會認為這樣做沒有任何個人利益。然而，經營自己的結點卻能帶來許多個人利益。為了讓您相信這一點，我將在本章中提出所有的理由，包括技術上和策略上的理由，為何您應該安裝並使用自己的 Bitcoin 節點。



### 更保密的交易傳播



當 Wallet 軟體連線至外部節點時，會將交易傳輸至不受您控制的基礎架構。這會產生明顯的監控風險： 遠端節點的操作者可以分析您的交易細節，包括金額和頻率，並透過交叉檢查某些元資料 (例如 IP 位址、時間和地點)，可能將這些資料與您的身份聯繫起來。



事實上，正如前一章所指出的，錢包與 Bitcoin 網路之間的溝通並不是靠魔法；它們必須連接到一個節點，才能查詢餘額或廣播交易。如果您從未建立過自己的節點，這就意味著您的 Wallet 依賴於第三方（通常是軟件背後的公司）的基礎設施。這個第三方，特別是如果它是一家公司，可能會觀察、利用、甚至洩露這些資料：無論是出於商業原因、法律約束，還是盜版的結果。



![Image](assets/fr/059.webp)



透過使用您自己的節點，您可以繞過中介直接向網路廣播您的交易。只要您妥善保護您的節點（我們稍後會討論）或遵守某些標準，就不會暴露任何資訊：您的 IP Address 或您的交易細節都不會經過您無法控制的實體。這是保護您在 Bitcoin 上的機密性的基本先決條件。



https://planb.network/courses/65c138b0-4161-4958-bbe3-c12916bc959c

### 不可申報的交易



基於上述相同原因，基於第三方節點的 Wallet 軟體容易受到審查風險的影響：遠端節點的操作者可能會基於各種原因拒絕轉發某些交易。它可能認為這些交易可疑或違反其政策。如果交易不符合節點的中繼規則，也可能會被攔截。最後，營運商可能會特別針對您的 IP Address 來封鎖您交易的廣播。



相反地，透過使用自己的節點，您可以確保交易在點對點網路中的傳播。這表示您可以完全控制交易的傳播，而不需依賴中介。只要交易符合連接到您的節點的共識和中繼規則，它就會在網路中廣播，然後在包含足夠費用的情況下，由 Miner 整合到一個區塊中。擁有自己的節點可保證您的交易得到中立、無權限的確認。



### 獨立資料驗證



如果沒有個人節點，您仍需依賴第三方才能存取資訊，例如您的 Address 結餘、交易確認狀態和區塊有效性。這意味著對外部節點的準確性和完整性的隱含信任。



![Image](assets/fr/060.webp)



執行 Full node 表示您可以針對每筆交易和每個區塊，自行檢查所有通訊協定規則。因此，您的 Wallet 所顯示的餘額不是從遠端伺服器接收的資料，而是從完整的 Blockchain 副本本地計算出來的結果，並經過一個區塊一個區塊的驗證。這種方法充分體現了比特幣玩家的格言：



> 不要相信，要驗證。

### 更佳的系統安全分配



每個加入網路的節點都能強化 Bitcoin 的備援與彈性。它促進了資訊的傳播，並使新的對等體能夠彼此連接。如果沒有節點，系統根本無法運作。



正如我們所見，Bitcoin 的安全性並非基於分散化、Mining 或密碼學：與任何系統一樣，它依賴於個人。更確切地說，它取決於節點操作者抵抗強迫的能力。



Bitcoin 之類的分散式系統的特點在於在所有參與其運作的人之間分配風險。運行您自己的 Bitcoin 節點意味著透過確保您的實例的安全性來承擔部分風險；在這樣做的同時，您也減輕了其他節點運營者的風險負擔。



因此，這並非直接的個人利益：執行節點會讓您對網路安全負上部分責任。最重要的是，這是一種集體利益，因為您的參與有助於分散風險。反過來，您也增加了自己可靠使用 Bitcoin 的能力。



### 加深對系統的了解



安裝 Full node 並不是簡單的操作。它包括安裝軟體、瞭解基本操作、監控同步化、在出現問題時檢查日誌，甚至還要使用終端機。這必然會讓您加深對通訊協定的了解。這是間接的優勢，但並非微不足道。



掌握這些知識可以增強您對工具的信心，並可以降低出錯或受騙的風險。自己打結也是一種學習方式。



### 選擇要應用的規則



一個經常被誤解的重要方面是，操作節點允許您選擇本機應用的規則。有兩種主要的規則：





- 共識規則**：



這些是 Bitcoin 協議的基本規則，可確保系統的完整性，並建立交易和區塊的驗證標準。任何不符合這些共識規則的交易，永遠無法被包含在有效區塊中。例如，某筆交易的其中一個項目簽章無效，就會被系統排除在外。



變更這些規則等於變更協議，因此也等於變更貨幣 (Hard Fork)。然而，即使沒有試圖修改這些規則，嚴格應用現有規則這個簡單的事實就賦予了一定的權力：如果一個區塊違反了規則，節點就會立即拒絕它。





- 接力規則**：



這些是每個 Bitcoin 節點特有的規則，被加入到共識規則中，以定義在 Mempool 中接受並轉達給對等方的未確認交易的結構。每個節點在本機配置並應用這些規則，這也解釋了為什麼不同節點的規則可能不同。這些規則僅適用於未確認的交易：被節點視為「非標準」的交易，只有在已經出現在有效區塊中時才會被接受。變更這些規則並不會將該節點排除在 Bitcoin 系統之外。



例如，根據共識規則，無費用的交易是完全有效的，但根據 Bitcoin core 中繼政策，它將被默認拒絕，因為 `minRelayTxFee` 參數被設置為 `0.00001`（以 BTC/kB 為單位）。但是，在您自己的節點上，可以降低此門檻以中繼費用較低的交易，或者反過來，增加限制，例如，增加到 2 Sats/vB，以避免中繼費用低的交易。



編織你自己的節點意味著：「根據我自己採用的規則，我驗證我選擇驗證的東西 」*。您因此成為系統管理的參與者，能夠拒絕您無法接受的演進，或根據您自己的標準核准更新。



因此，我們可以快速嘗試了解由於您的節點，您對規則有多大的權力。而這種權力的大小將取決於規則的類型。



#### 針對接力規則



就中繼規則而言，最基本的事情只是擁有一個節點，不論其經濟活動如何。這裡的關鍵在於您是否同意中繼某些類型的交易。



舉例來說，如果您認為 Bitcoin 應該接受費用低於 1 sat/vB 的交易，您可以調整節點上的這項規則，讓它廣播這些交易，從而促進它們在網路上的傳播，直到 Miner 最終將其納入有效區塊。基本上，這就是交易傳播的權力問題：每個節點都有決策權，因為同意轉播某種類型的交易，就等於促進 Bitcoin 網路接受這種交易。因此，如果您操作數個節點，您對中繼政策的影響力就會更大，因為每個節點都有自己的連線和對網路有影響的區域。



事實上，有一個或多個節點配置了特定的中繼規則，意味著決定網路的哪一部分接受傳播特定類型的交易。在點對點圖形中傳播訊息，就像 Bitcoin 交易一樣，遵循滲透理論的邏輯。將每個節點想像成一個網站，它可以是活躍的 (`p`=它中繼) 或非活躍的 (`1-p`)。只要比例 `p` 跨過臨界值 (`p_c`)，就會出現一個巨大的元件：交易成功穿越網路，並完全有機會到達 Miner。在 Bitcoin 這樣的網路中，每個節點平均維護 8 個向外連線，`p_c` 臨界值一般只設定為百分之幾，如果某些節點有非常多的連線，這個臨界值甚至會更低。



![Image](assets/fr/061.webp)



只要 `p` 維持在 `p_c` 以下，交易就會被限制在孤立的區塊中，不會達到 Miner。一旦超過這個臨界值，它幾乎會在瞬間傳遍整個網路。



最終，總是由礦工來決定是否將交易包含在區塊中。然而，節點透過影響交易的分佈來介入上游：它們決定礦工是否會知道某筆交易。如果某筆交易沒有傳達給礦工，礦工顯然不可能將其納入區塊中。



因此，如果網路已處於特定交易類型的滲透階段，則增加幾個節點只會產生微不足道的影響，但隨著滲透臨界值的接近，它可能會被證明是決定性的。擁有或影響幾個節點，尤其是當它們有良好的連線時，可以增加或減少 `p` 的值，進而間接引導中繼規則，決定哪些交易會被看到並最終被礦工接受。



#### 對於共識規則



當談到您的節點對共識規則的影響力時，其經濟權重首先是決定性的。這是一個關鍵的概念：任何貨幣的價值都與其促進 Exchange 的能力直接相關。事實上，如果一個物品在 Exchange 中不被任何人接受為商品或服務，理論上它就沒有任何貨幣效用。舉例來說，如果沒有商人接受卵石作為付款方式，卵石就沒有錢的用途。當然，效用仍是個人尺度上的主觀概念，但在特定領域中，接受某物品作為 Exchange 支付手段的商人越多，該物品對生活在此領域中的人們而言就越有可能具有貨幣效用。



讓我們以一個村莊為例，許多商人接受黃金以 Exchange 換取商品：黃金有可能對村民有貨幣效用。這表示一種貨幣的效用直接取決於商人接受或拒絕它的決定。



這個概念對於理解 Bitcoin 系統中發揮作用的權力動力至關重要。Satoshi 說得很清楚：Bitcoin 是一個電子現金系統；換句話說，它提供一種服務，提供一種貨幣形式，即 Bitcoin（或 BTC）。當協定規則以不向後相容的方式修改時 (Hard Fork)，這就等於創造了一個新的系統，因此也創造了一種新的貨幣。這個 Fork 的成敗就取決於其經濟規模，而經濟規模又取決於接受這種新形式貨幣的商家數量。



![Image](assets/fr/062.webp)



讓我們舉個例子：假設 Bitcoin 遭到 Hard Fork 的襲擊。那麼就會有 2 種不同形式的貨幣：BTC-1 (原始的、沒有改變的版本) 和 BTC-2 (具有不同共識規則的新貨幣)。如果所有接受 BTC-1 的商家繼續這樣做，但拒絕 BTC-2，那麼理論上，後者的貨幣效用將非常有限。作為使用者，我沒有興趣保留和使用 BTC-2，因為我知道在 Exchange 中沒有商家會想要用它來購買商品或服務。相反地，如果50%的商家選擇只接受BTC-2，而其餘50%的商家只接受BTC-1，那麼理論上，BTC-1的效用將減半。我之所以使用 「理論上 」這個詞，是因為效用在個人層面上仍然是主觀的，並且取決於多種因素（例如領土和消費習慣），而這些因素是很難逐一理解的。



在 Bitcoin 上，「商家 」的角色可理解為任何具有一定經濟份量的實體，當然包括企業（實體商店、線上銷售網站、服務供應商等），但也包括 Exchange 平台，因為他們接受 Bitcoin 在 Exchange 中的其他貨幣，以及礦工，因為他們通過在 Exchange 中收取費用來接受 Bitcoin，以獲得將交易包含在區塊中的服務。



就共識規則而言，您的節點允許您將您的經濟活動導向一種或另一種貨幣。舉例來說，如果您家中有 10 個完整的節點，但沒有顯著的經濟活動，您在 Fork 期間的影響力幾乎為零。相反地，一個用來管理 200 間連鎖店的節點在接受 Bitcoin 時會賦予顯著的經濟影響力。



因此，重要的不是節點的數量，而是它們所支援的經濟活動的重要性。更重要的是，如果您的經濟活動依賴於您無法控制的節點，只要您仍與該節點連線，它的擁有者就會決定您使用何種貨幣。這就是為什麼運行和使用自己的節點在系統治理方面特別重要的原因：



> 不是你的結，不是你的規則。


## 不同類型的 Bitcoin 節點


<chapterId>be8f0baa-41f2-4b54-b011-092f4ccc93aa</chapterId>



因此，Bitcoin 節點就是運行 Bitcoin 通訊協定實作的機器。在這個共同的節點定義背後，存在幾種可能的配置，但並非所有的配置都能提供相同程度的自主性、資源消耗以及對網路的效用。在本章中，我們將嘗試瞭解這些差異，以協助您選擇適合您的用途和硬體限制的節點架構。



### 完整的結



Full node 只是一個 Bitcoin 節點，它從 Genesis 區塊下載整個 Blockchain，獨立驗證每個區塊，並在本地儲存所有該 Blockchain 的歷史。這是 Satoshi Nakamoto 所想像的 Bitcoin 節點的「正常」形式。



![Image](assets/fr/063.webp)



Full node 不需要信任任何人，因為它驗證並知識系統中的所有資訊。它是能給您最大保證的節點類型：您無需依賴第三方，就能知道付款是否有效、區塊是否有效、重組是否合法等等。



實際上，Full node 需要相當多的資源，包括數百 GB 的區塊檔案、能夠驗證腳本的處理器、Mempool 的 RAM 和快取記憶體，以及穩定的頻寬。第一次同步 (*IBD*) 會讀取並驗證完整的歷史記錄：它很密集，但只會發生一次。Full node 會積極參與網路，中繼區塊和交易，並可接受進入的連線，以協助其他對等體。



根據您的需求，您可以在 Full node 中加入索引器。Bitcoin core 提供交易索引作為選購功能 (預設為停用)，對於特定用途很有用。但是，它不包含 Address 索引器，而這通常是個人使用者最想要的功能。為了彌補這一缺點，您可以在節點上安裝專用軟體，例如 Electrs 或 Fulcrum，以加速相關 UTXO 的 Address 結餘驗證查詢。我們會在另一章節再詳細討論索引器的角色。



### pruned 結



pruned 節點會以 Full node 的方式驗證所有東西，從 Genesis 區塊到工作最多的鏈頭，但 ** 只會保留區塊檔案的最新部分。一旦檢查過舊區塊，它會逐漸刪除它們，以保持低於您可以設定的空間限制。如果您有磁碟空間限制，此設定是理想的選擇：您可以保留區塊驗證的獨立性，而無需儲存完整的 Blockchain 歷史存檔。如果您只想在個人電腦上安裝 Bitcoin core，而不使用專用機器，此選項特別有用。



![Image](assets/fr/064.webp)



此方案的技術意義相當直接：pruned 節點完全有能力廣播您的交易、參與中繼、驗證區塊和交易，以及追蹤鏈。另一方面，它無法為其他應用程式 (如完整探索者、索引器、錢包) 充當超出其限制的歷史資料來源。因此，需要存檔（或全局索引）的功能將無法使用。



實際上，您可以使用 pruned 節點連接 Wallet 管理軟體，例如 Sparrow wallet。但是，您將無法掃描 Wallet 上早於修剪限制的交易。例如，如果您有一個交易登記在區塊 901 458，但您的節點只保留區塊 905 402 以上（因為最舊的已經是 pruned），您將無法掃描這個交易。另一方面，如果您在節點仍有此區塊高度時已掃描，那麼您的 Wallet 管理軟體將會儲存資訊，並正確顯示相對應的 UTXO 的餘額。



簡而言之，如果您在軟體已連接到 pruned 節點時建立新的 Wallet，Wallet 追蹤就可以在該節點上順利運作。另一方面，如果您還原舊的 Wallet，可能會遇到困難，因為節點不再保留的過去交易顯然無法檢索。



### 光結 / SPV



SPV (* 簡化支付驗證*) 節點，或稱輕量級節點，只保留區塊標頭，而不保留交易細節，並依賴其他完整節點來取得交易在其擁有標頭的區塊中的證明 (透過樹的 Merkle 證明)。簡化支付驗證的概念並不新穎，Satoshi Nakamoto 本人已在白皮書第 8 部分中提出。



![Image](assets/fr/066.webp)



Nakamoto, S. (2008)。 *Bitcoin：點對點電子現金系統*。https://Bitcoin.org/Bitcoin.pdf



這種類型的節點在儲存和 CPU 使用量方面，顯然比 Full node 甚至 pruned 節點輕得多。因此，SPV 節點非常適合較小的裝置和間歇性連線。事實上，它通常會直接整合至 Wallet，尤其是 Mobile 軟體，例如 Blockstream App。



取捨的關鍵在於信任和機密性：SPV 用戶端本身不會檢查腳本或驗證政策；它會假設工作最多的鏈是有效的，並依賴一個或多個完整節點的回應。因此，使用這種類型的節點是比連接第三方節點更好的選擇；然而，它的優勢仍比不上擁有 Full node 甚至 pruned 節點。



![Image](assets/fr/065.webp)



### 哪個節點符合哪種需求？





- 行動/新手使用者



對於只有行動應用程式 Wallet 的新手使用者而言，使用 SPV 節點肯定是最佳的入門方式。安裝快速、所需資源少，而且體驗簡單流暢。這意味著您可以自行驗證某些資訊，因此可以減少對第三方節點的依賴，同時在廣播交易時更加獨立。





- PC / 中階使用者



擁有個人電腦的中級用戶可以安裝 pruned 節點，以受益於幾乎所有 Full node 的優點，而不會讓他們的機器每天負荷過重：完全驗證、適度的磁碟使用量和簡單的維護。這是一個理想的解決方案，可連接您的桌上型錢包，並在交易分發時保持獨立，而無需投資專用機器或超載磁碟空間。





- 主權 Bitcoiner / 進階



如果您想要完全獨立使用 Bitcoin，而不限制自己日後進階的使用，例如索引器、Lightning 節點，甚至是 Block explorer，那麼 Full node 仍然是最佳的解決方案。這正是我們要在本課程中探討的！



## 軟體解決方案概述


<chapterId>0d48b89a-e8b5-441e-a707-537a035fc15e</chapterId>



在軟體方面，有 2 種主要方式可執行 Bitcoin 節點：




- 直接安裝通訊協定實作，例如 Bitcoin core（推薦）或 Bitcoin Knots、
- 或使用交鑰匙發行版（通常稱為「_node-in-a-box_」），它以相同的方式整合 Bitcoin 實作，但也包含 Interface 管理系統、應用程式儲存庫，以及隨時可用的工具（Lightning、瀏覽器、索引伺服器，甚至是 Bitcoin 外部的自我託管應用程式...）。



這兩種方法的目標相同：擁有自己的節點，但在 Interface 的安裝和使用、維護、擴充性和成本方面有所不同。這就是本章要探討的問題。



### 原始 Bitcoin 節點實作



安裝原始實作是指直接使用 Bitcoin 通訊協定實作的軟體 (例如 Core)，不需要任何附加軟體 Layer。您可以根據自己的需求，自行管理設定、更新和相關服務 (索引、API、Lightning、備份等)。



這是最主權、最靈活的方法：您可以清楚知道正在執行什麼、資料在哪裡，以及一切是如何運作的。另一方面，只要您想超越 Bitcoin 節點的簡單運作，它就會變得更複雜。如果您的目的只是要擁有一個節點，複雜度就會與盒中節點的複雜度相若，甚至更低，因為這只是安裝軟體的問題。



#### Bitcoin core (超大多數客戶)



[Bitcoin core 是網路的超大多數用戶端](https://bitcoincore.org/)。它會下載、驗證和維護 Blockchain，提供 RPC/REST API，並能整合 Wallet。如果您偏好標準工具，並覺得可以自行新增服務 (例如 Electrum 伺服器、explorer 和 LND)，您最好還是原封不動地使用 Core。



**優點：** 最大的穩定性、可預測的行為、原始的體驗、容易安裝和設定。



**缺點：** 您必須手動建立堆疊的其他部分，才能建立完整的應用環境，而不只是 Bitcoin 節點。



https://planb.network/tutorials/node/bitcoin/bitcoin-core-linux-568c13a6-8746-4d63-8e95-f4a61c5ae0ed

https://planb.network/tutorials/node/bitcoin/bitcoin-core-mac-windows-9684ab02-e0af-41c9-8102-86ac7c7727f3

#### Bitcoin Knots (主要替代客戶)



[Bitcoin Knots是Bitcoin core的Fork](https://bitcoinknots.org/)，由Luke Dashjr維護。它是實作Bitcoin通訊協定的主要替代Core用戶端。它與網路的其他部分完全相容 (它絕對不是像 Bitcoin Cash 一樣的 Hard Fork)，但它提供了額外的功能，包括 Core 沒有的中繼政策選項，或在預設情況下應用得更嚴格，以限制某些人認為的垃圾郵件。



選擇 Knots 而非 Core 可能有 2 個原因：




- 技術**：與 Core 不同的選項，特別是在中繼管理方面，透過決定您的節點接受和廣播哪些交易。
- 政策**：有些人基於非技術上的原因，喜歡使用 Knots 等替代用戶端，主要是為了支持 Core 的替代方案，從而減少 Core 的壟斷。如果 Core 遭到攻擊，擁有可靠、維護良好的替代用戶端，而且知道如何有效地使用它們，將是非常有用的。其他人使用 Knots 是為了抗議目的，因為他們對 Core 的開發人員失去信心，或不認同用戶端的大多數管理人員。


https://planb.network/tutorials/node/bitcoin/bitcoin-knots-e04b2196-4df2-4246-86ef-c02269c29098

就個人而言，我建議您選擇 Core，主要是為了更快地從安全修補程式中獲益。事實上，Knots 發現的某些漏洞會延遲修正。更一般來說，Core 的開發過程結構穩固，並有許多貢獻者的支持，而 Knots 則只由一個人維護，社群規模也小得多。另一方面，中繼規則在今日往往會失去其效用，尤其是當只有極小部分的網路應用時 (就像 per percolation 理論一樣)。



### 節點在方塊中的分佈



這個_node-in-a-box_結合了 Bitcoin core（或 Knots）與預設的作業系統、Interface Web 以及自託管服務的 App Store（Lightning、explorers、Electrum 伺服器、Mempool、BTCPay 伺服器、Nextcloud 等）。只需輕輕一按，您就能安裝、更新並將這些不同的模組互相連結。



對於日常啟動和管理眾多輔助應用程式來說，這是一個簡單得多的解決方案。缺點是當發生問題時（例如 Docker 映像衝突、錯誤更新、資料庫損壞），除錯可能會變得非常複雜，因為您必須仰賴發行版本身的整合。更重要的是，社區或官方支援通常都很複雜。



因此，只要一切運作正常，盒中節點是非常容易使用的，但如果發生錯誤，您就必須準備進行長時間的搜尋、等待協助，並且弄髒您的手。



這些解決方案大多有兩種形式：




- 預裝機：已安裝作業系統的完整電腦。這些隨用隨付的機器只需插入主電源並連上網際網路即可運作。如果您的預算允許，此選項的優點是設定非常簡單，通常提供優先支援，並有助於開發資金，因為這些公司的商業模式通常是以銷售硬體為基礎。
- DIY: 在您自己的機器 (舊 PC、NUC、Raspberry Pi、家用伺服器...) 上安裝發行版作業系統。這是最經濟的解決方案，因為您可以回收舊機器或選擇符合您需求和預算的硬體。這也是最靈活的方案，配置起來也最令人滿意。我們將在課程的實務部分探討這種方法。

### 您真的需要專用機器嗎？



在過去的幾年裡，比特幣玩家越來越意識到一個常見的誤解，特別是隨著 2020 年代初 node-in-a-box 的普及：Bitcoin 節點必然應該運行在一台專門用於此目的的機器上。但事實並非如此。您不一定需要專用電腦來執行 Bitcoin 節點：Bitcoin core 完全可以在您的日常 PC 上運行。如果您有足夠的磁碟空間給 Blockchain 或啟用修剪功能，您就可以驗證鏈結、連接您的 Wallet，甚至在使用完畢後關閉程式。這種方式的優勢相當可：零初始投資，複雜性最低。



![Image](assets/fr/074.webp)



雖然如此，使用專用機器通常會比較舒適。它可以連續運行 (24/7)、隨時可遠端存取、不會壟斷您主機的資源，最重要的是，可以隔離使用 (良好的安全作法：如果您的個人電腦遇到問題，您的節點會繼續運作，反之亦然)。因此，問題不是「我需要專用一台機器嗎？」，而是「我需要一個持續線上、可由其他裝置存取，且能不斷演進的節點嗎？」（Lightning、索引器、附加應用程式...）。如果答案是肯定的，選擇獨立的機器會讓事情變得更簡單。



### 3 種取得方法：回收、二手和全新



#### 回收舊電腦



這是最經濟的解決方案。我們大多數人的家中或親友家中都有一台舊的 PC 收集 Dust：這是讓它重新投入使用的絕佳機會！若要將其改裝成 Bitcoin 節點，只需新增 2TB SSD，並視您的需求，更換或新增 RAM 棒以增加 RAM。一台功能完備的機器預計需要支付 100 到 200 歐元。



在購買任何硬體之前，請檢查可用的磁碟插槽數量、連線類型 (M.2 或 SATA)、記憶體格式 (SODIMM 或 DIMM) 及其世代 (DDR4 等)。您也應該趁此機會清潔機器，尤其是風扇，以確保最佳效能。



不過，如果您使用的是筆記型電腦，請務必小心：電池可能會隨著時間的推移而產生問題（本章稍後將詳細說明）。



#### 翻新或二手



市場上有許多翻新的商用迷你電腦，例如 *Lenovo ThinkCentre Tiny*、*HP EliteDesk Mini* 或 *Dell OptiPlex Micro*。這些機器堅固、小巧、安靜且節能。它們的價格遠低於新機，而且很容易找到配備第 6 到第 10 代 i5/i7 處理器和 8 到 16 GB 記憶體的機型，價格都非常吸引人，一般在 70 到 200 歐元之間，視配置而定。在我看來，如果您要為 Bitcoin 節點尋找專用機器，這可能是最佳選擇。



![Image](assets/fr/075.webp)



也可以找到幾年前的二手 PC 和筆記型電腦，配置有趣，性價比極高。



**註：** 企業機隊的機器，例如*ThinkCentre Tiny*，通常只配備*DisplayPort* (DP) 連接埠連接螢幕，沒有 HDMI 輸出。因此，如果您需要，別忘了攜帶轉換器或 DP 至 HDMI 連接線。



#### 購買新的



如果您的預算允許，您也可以選擇新的機器。如果您希望擁有性能良好的最新硬體，這是一個不錯的選擇，尤其是如果您打算使用 Umbrel 或 Start9 與 Bitcoin 生態系統以外的其他應用程式進行自我託管。



### 我應該選擇何種類型的機器？



#### 迷你電腦 "NUC" / 裸機



我認為，迷你電腦是在家中架設 Bitcoin 節點的最佳折衷方案。它節省空間，可輕鬆放在架子上，耗電量極低，而且硬體修改也很容易，例如增加 RAM 或更換 SSD。



就我個人而言，我比較喜歡 *Lenovo ThinkCentre Tiny*，這種產品在二手市場上非常普遍（來自企業機群）；它們特別堅固，而且容易改裝。當然，其他製造商也有許多同類產品： *Dell OptiPlex Micro*、*HP ProDesk / EliteDesk Mini / Micro*、*Intel NUC*、*Gigabyte BRIX*、*MSI Cubi*...。



![Image](assets/fr/001.webp)



**Highlights:** 基底面小、功耗適中、低噪音、可擴充性 (視機型而定) 及可靠性。



**缺點：** 比 Raspberry Pi- 類型的 SBC 稍貴，沒有內建螢幕 (遠端存取或透過外部顯示器)，沒有電池 (斷電時會突然關機)。



#### 專用筆記型電腦



它是迷你電腦的絕佳低成本替代方案：如今，您可以找到價格低廉的二手筆記型電腦，甚至是全新的筆記型電腦，配備正規的處理器、許多連接埠，以及整合式螢幕和鍵盤 (對於初次安裝非常實用)。最重要的是，電池可當作天然的 UPS：在電源中斷時，節點不會突然關機，甚至可以維持運作數小時。



![Image](assets/fr/076.webp)



**Highlights:** All-in-one 解決方案，電池可充當 UPS (不會停電)，整合式顯示器和鍵盤簡化了安裝程序，整合式 Wi-Fi 卡，二手和全新市場選擇眾多 (這通常意味著您可以議價)。



**缺點：**耗電量比裸機 Mini-PC 略高，24/7 運作中電池會逐漸耗損而失去容量，電池隨著使用年限腫脹或熱失控的風險雖然罕見但確實存在。主要是這方面讓我認為迷你電腦比筆記型電腦更好的選擇：電池的漸進式耗損及相關風險。



如果您選擇此解決方案，我建議您密切注意電池的狀況，以免發生危險。注意電池是否過熱、有異常氣味、不穩定或外殼變形。如果發生警報，請立即關閉電腦並拔下插頭，然後到專門的回收機構棄置電池。



**Tip:** 如果 BIOS/UEFI 或製造商的工具允許，請設定負載限制 (例如 60% 或 80%)，以延長電池使用壽命。



#### Raspberry Pi 和其他 SBC：錯誤的想法



2020 年代初期，隨著節點整合軟體的興起，Raspberry Pi 的熱潮也應運而生，成為運行 Bitcoin 節點的一種方式。這個想法似乎很吸引人：價格便宜、小巧、容易取得。



![Image](assets/fr/073.webp)



實際上，如果您的目標只是執行 Bitcoin 節點而不需要額外的應用程式，Raspberry Pi 可能就足夠了。但只要您想使用 Umbrel、Start9 或更豐富的生態系統 (Block explorer、Address 索引器、Lightning 節點、自主控應用程式...)，這台機器很快就會達到極限。



Raspberry Pi 有許多缺點：




- 處理器過於纖薄，ARM 架構有時與某些軟體不相容，或需要更多的處理；
- 固態記憶體，無法升級，配置有限（通常最多 8 GB）；
- 透過纜線連接固態硬碟的外接盒，經常出現錯誤訊息，需要購買特定的卡才能使用穩定的固態硬碟；
- 有快速升溫的傾向，且難以確保正確冷卻；
- 需要購買額外的硬體 (機殼、風扇、SSD 卡等)；
- 連接性非常有限。



從歷史上來看，像 Raspberry Pi 這種 SBC 的最大優勢在於價格：花幾十歐元就能買到一台專用機器。然而，今天的價格已經大幅攀升，一旦您添加了所有必要的附加硬體，其成本已經接近第一台二手或翻新 x86 迷你電腦的價格，而在我看來，這些迷你電腦的優勢要大得多。因此，我不建議選擇 SBC。



### 選擇元件



#### 磁碟儲存：必須使用 SSD，最低 2 TB



技術上來說，在硬碟上執行 Bitcoin 節點是可能的。問題是一切都會變慢，尤其是 IBD，因為 Bitcoin core 會大量使用磁碟作為快取記憶體（尤其是 UTXO 套件），所以 IBD 會變得非常長。這就是我強烈建議不要使用 HDD 的原因：它會造成真正的瓶頸，嚴重限制未來的演進（例如，對 Lightning 節點而言），甚至可能導致與 Blockchain 磁頭的同步失配。此外，機械磁碟上的持續壓力會增加過早磨損的風險。



固態硬碟機徹底改變您的使用者經驗：一切都變得更快、更順暢，可靠性也更高。因此，使用 SSD 對於您的節點來說是（幾乎）必須的，而且您不會後悔，尤其是現在大容量機型的價格相對較低。



![Image](assets/fr/077.webp)



就容量而言，2 TB 已逐漸成為新的合理最小值。2025 年夏天，Blockchain 已經接近 700 GB，如果再加上 Umbrel、Address 索引器和一些應用程式，1 TB 的 SSD 很快就會飽和。有了 2 TB，您在未來幾年（概略估計為 5 到 15 年）都有舒適的剩餘空間。如果您打算在 Umbrel 上使用許多應用程式、在自寄存中儲存大型檔案，或是您想要預估很大程度的磁碟空間需求，您也可以選擇 4 TB。



![Image](assets/fr/078.webp)



至於格式，這將取決於您機器上可用的連接埠；但是，如果可能的話，我建議使用 NVMe M.2 SSD。



#### 記憶體 (RAM)：8 至 16 GB



對於單獨使用 Bitcoin core（無 Umbrel 覆疊），開發人員建議在設定調至最低時至少需要 256 MB RAM，在預設設定時至少需要 512 MB，正常使用時至少需要 1 GB。



另一方面，如果您使用的是 Umbrel 或 Start9 之類的 node-in-a-box 系統，則對 RAM 的要求會高很多。Umbrel 開發人員建議至少使用 4 GB 記憶體。這也許只夠執行 Core，但很快就會受到限制。因此，他們建議使用 8 GB，我也認為這是圍繞 Bitcoin (Core、LND、索引器和一些應用程式) 的基本配置的最低值。根據我的經驗，如果使用 Umbrel 和一些額外的服務，8 GB 的空間還是有點緊。若要真正舒適並有一些餘量，我建議使用 16 GB RAM。



#### 處理器 (CPU)



對於 Umbrel 節點，最低要求是 Intel 或 AMD 的雙核 64 位元處理器。如果您想在 Bitcoin core 以外使用一些應用程式，四核心 (或更高階) 會讓流暢度產生真正的差異。例如，在二手市場上，第 6 至第 10 代 i5/i7 處理器是極佳的選擇。



### 具體配置範例



以下，我提出三種具體的配置，適用於不同的預算和需求，並有精確的模型來支援它們。這些選擇是為了說明本章的資訊而提供的範例；您沒有義務完全選擇這些機型。由於我認為 Mini-PC 是長期的最佳選擇，因此我將依據此格式來提供三種建議的配置。



*以下所示價格僅供參考，可能因地區、供應商和時期而異*



首先，您需要的固態硬碟必須大到足以容納 Blockchain，同時還要留有操作空間。就寫入週期和寫入資料總量而言，固態硬碟的壽命有限。然而，Bitcoin 節點在寫入時會對磁碟造成很大的負荷。這就是我不推薦入門級機種的原因；相反地，我建議使用 NVMe SSD，因為它的效能明顯更好。



以本課程為例，我選擇了以下機型： *Samsung 990 EVO Plus NVMe M.2 SSD 2Tb*，在 Amazon 上的售價約為 120 歐元。您也可以選擇其他知名品牌，例如 Crucial、Western Digital 或 Kingston。



![Image](assets/fr/046.webp)



#### 低預算配置



很明顯，如果您的預算非常有限（低於 200 歐元），我建議您不要投資專用機器，而是直接將 Bitcoin core 安裝在您的日常電腦上（如果磁碟空間不足，則安裝在 pruned 模式下）。



否則，對於入門級的預算，我推薦 *HP EliteDesk 800 G2 Mini*。我在 Amazon 找到一款翻新機型，售價 96 歐元，配備第六代 Intel Core i5 處理器和 8 GB 記憶體。對於初學者來說，這是一個特別有趣的選擇：這個處理器和這個記憶體容量足以在 Umbrel 上運行 Core，以及同時運行幾個應用程式，例如 Electrs 索引器、Lightning 節點和 Mempool 實例，前提是您不要為 Core 分配太多快取。更重要的是，如果有需要，這款迷你電腦可以輕鬆將記憶體增加到 16 GB（預計需要額外花費約 30-40 歐元購買一或兩根高品質的記憶棒）。



![Image](assets/fr/045.webp)



然後只要將 SSD 加入預算即可。從 Samsung 2TB 的 120 歐元開始，我們可以得到一台完整且功能齊全的機器，總成本為 216 歐元。



#### 中等預算配置



如果您有 300 歐元左右的平均預算來購買承載節點的機器，我建議您選擇 *Lenovo ThinkCentre Tiny*，例如配備高效能處理器和足夠的記憶體。我在 Amazon 找到一台翻新機型，價格為 180 歐元，搭載第六代 Intel Core i7 處理器和 16GB 記憶體。加上 120 歐元的 2TB SSD，總成本為 300 歐元。



![Image](assets/fr/044.webp)



有了這台機器，您就能擁有舒適的配置：快速的 IBD，並能在 Umbrel 或 Start9 上毫不費力地執行許多應用程式。這正是我在這次 BTC 202 課程中所使用的配置。



#### 高階配置



有了更多的預算，可能性就變得更大。您可以選擇 DIY 配置，甚至可以選擇由節點盒專案直接提供的預先組裝機器。



舉例來說，*華碩 NUC 14 Pro* 在亞馬遜有售，新價為 540 歐元。以這個價格，您可以獲得 Intel Core Ultra 5 處理器 (最新且特別高效能)，以及 16 GB DDR5 記憶體。有了這樣的配置，您就能在最短的時間內完成 IBD，並輕鬆安裝要求嚴苛的應用程式。



如果最初的目標只是執行 Bitcoin 節點，這是一個非常舒適的配置，甚至可以說是矯枉過正。另一方面，如果您想充分利用 Umbrel 和 Start9 上的所有自託管應用程式，這個功率等級對您來說正合適。



![Image](assets/fr/043.webp)



依據您的預期用途，您可以選擇 2TB SSD，就像其他配置一樣；如果您也想儲存個人檔案並擴大自助託管用途，也可以直接選擇 4TB SSD，價格為 260 歐元。使用 2TB SSD 的配置總成本為 660 歐元，而使用 4TB SSD 則高達 800 歐元。

## Umbrel: 遠超過 Bitcoin 節點


<chapterId>dd4c04f1-924a-43e1-94f3-ea9fbc83dd43</chapterId>



Umbrel 是一個個人伺服器作業系統，設計目的是讓自我託管變得容易：您安裝 Umbrel，在 `umbrel.local` 上開啟瀏覽器，並透過簡單的遠端 Interface 管理一切。



該專案首先普及了一鍵式 Bitcoin 和 Lightning 節點的概念，隨後擴展為名副其實的「家庭雲」：檔案和相片儲存、多媒體串流、網路工具、家庭自動化、本機 AI，以及可從整合式 App Store 安裝的數百個應用程式。



在 Umbrel 中，每個應用程式都在 Docker 容器中執行 (隔離、原子式更新、獨立啟動/停止)。Interface 集中管理所有這些應用程式的存取，提供單一登入 (可選擇 2FA)、作業系統與應用程式的一鍵更新、機器即時監控 (CPU、RAM、溫度、儲存)、應用程式間的權限管理，以及消耗總覽。



因此，Umbrel 的目標是讓您在不依賴雲端服務的情況下，除了簡單地操作 Bitcoin 節點之外，還能重新獲得資料的控制權與保密性。



### Umbrel Home vs umbrelOS



Umbrel 提供兩種截然不同的方法：





- [**Umbrel Home**](https://umbrel.com/umbrel-home)：這是一台即開即用的迷你伺服器，專為 umbrelOS 設計與最佳化。小巧、無聲、連接乙太網路，配備 NVMe SSD (最高可選 4TB)、16GB RAM 和四核心 CPU。您只需訂購、插上電源，然後前往 `umbrel.local`。幾分鐘之內，您就可以讓 Umbrel 運作起來。這就是隨插即用的選項。



![Image](assets/fr/081.webp)





- [**umbrelOS**](https://umbrel.com/umbrelos): 這是您可以在自己的硬體 (迷你電腦、NUC、塔式電腦、專用筆記型電腦...) 上自行安裝的作業系統。您擁有與 Umbrel Home 相同的 Interface 與 App Store。



![Image](assets/fr/080.webp)



在這兩種情況下，軟體端的使用者體驗是相同的：以瀏覽器為基礎的管理、一鍵更新、依需求安裝應用程式...DIY 解決方案通常比購買 Umbrel Home 更經濟（取決於所使用的機器）。不過，我不一定建議您總是選擇 DIY 方案，因為**購買 Umbrel Home 會直接資助專案的開發**，因為它的商業模式是以銷售硬體為基礎。坦白說，2TB 儲存空間的價格為 389 歐元，以該機器的品質而言，價格仍然非常合理。



![Image](assets/fr/079.webp)



在下一章，我們將探討如何在您自己的機器上安裝 umbrelOS DIY。然而，如果您選擇了 Umbrel Home，您也可以以相同的方式學習本 BTC 202 課程。



### 使用個案：從 Bitcoin 節點到家庭雲端

### UmbrelOS 授權



Umbrel 的程式碼是公開的 (您可以檢視、Fork 並修改它)，但它並不是在真正的開放原始碼授權下。事實上，umbrelOS 是根據 [*PolyForm Noncommercial 1.0*] 授權 (https://polyformproject.org/licenses/noncommercial/1.0.0/) 發佈的，雖然一些相關的開發工具是根據 MIT 授權提供的。



實際上，只要是個人、非商業用途，您可以用 umbrelOS 做任何事：修改、為非營利目的再散布、為自己或為非營利組織製作衍生產品，只要您尊重法律聲明。



然而，禁止出售 Umbrel 或其衍生產品 (例如，預先安裝了 umbrelOS 的預裝機器)、提供與 Umbrel 相關的商業服務，或將其程式碼整合至產品中圖利。



技術上，此授權並不限制個人使用 Umbrel 的安裝、審計或改編。在法律上，它保護專案免於未經授權的轉售或商業託管，尤其是雲端服務供應商。因此，Umbrel 並非開放原始碼，但其程式碼仍可公開存取。



不過，商店中的每個應用程式都保留自己的授權，通常是開放原始碼。

### 步驟 1 - 安裝電腦



根據您所選擇的硬體，第一步是組裝電腦的各個元件。舉例來說，以我的情況為例，原本的固態硬碟容量只有256 GB，因此我會回收它以作他用，並換成2 TB的固態硬碟。如果你也想更換記憶體模組，現在就是時候了。



### 步驟 2：準備可開機 USB 密鑰



在您的電腦上安裝 UmbrelOS 之前，您需要建立一個包含作業系統的可開機 USB key。步驟 2 的所有步驟都必須在您的個人電腦上執行 (而不是直接在要成為您節點的電腦上)。





- 首先下載 USB 格式的最新版 UmbrelOS：



前往 [Umbrel官方網站下載ISO映像檔](https://download.umbrel.com/release/latest/umbrelos-amd64-usb-installer.iso)，透過USB key進行安裝。確保您選擇的版本相容於 x86_64 架構 (檔案名稱為 `umbrelos-amd64-usb-installer.iso`)。由於映像檔較大，下載可能需要一些時間。

- 安裝 Balena Etcher：



要製作可開機 USB 隨身碟，您會使用一個簡單的跨平台工具 [Balena Etcher](https://www.balena.io/etcher/)。下載並安裝到您的電腦上。



![Image](assets/fr/003.webp)





- 插入至少 4 GB 的空白 USB 密鑰：



將 USB 密鑰插入您的電腦（即您剛下載 UmbrelOS 和 Balena Etcher 映像的電腦）。 **警告：鑰匙上的所有資料將被刪除**。請確保它不包含任何重要檔案。





- 使用 Balena Etcher 將 ISO 映像燒錄至 USB 隨身碟：



啟動 Balena Etcher 並選擇您剛下載的 `umbrelos-amd64-usb-installer.iso` ISO 檔案，按一下「*Flash from file*」按鈕。然後選擇 USB key 作為目標裝置，點選「*Flash!*」開始寫入。



![Image](assets/fr/004.webp)



操作完成後，您將擁有一個包含 UmbrelOS 的可開機 USB 密鑰，可隨時開機並在您的機器上安裝 Umbrel。



![Image](assets/fr/005.webp)



### 步驟 3：從 USB 隨身碟啟動電腦



現在包含 UmbrelOS 的可開機 USB 隨身碟已準備就緒，您可以將電腦開機到 USB 隨身碟上，開始系統安裝。從主電腦拔除 USB 隨身碟，並將其插入您要安裝 Umbrel 和 Bitcoin 節點的裝置。



如本章開頭所述，要完成安裝，您需要一個顯示裝置和一個輸入裝置。透過 HDMI（或其他連接埠，視您的電腦而定）連接顯示器，並透過 USB 將鍵盤連接到您的電腦。這些裝置只在安裝時需要；之後就不需要了，因為 Umbrel 會從另一台電腦遠端存取。將這兩個裝置連接到您的電腦。



**祕訣：** 如果您家中沒有週邊螢幕，您可以使用電視。透過 HDMI（或其他）輸入，它可以在您安裝作業系統時用作臨時螢幕。



Umbrel 顯然需要網際網路連線。在裝置和路由器之間連接 RJ45 乙太網路線。



![Image](assets/fr/006.webp)



開啟您的電腦。在大多數情況下，它應該會自動偵測到 USB key 並從它開機。然後您會看到 UmbrelOS Interface 安裝畫面出現。



如果裝置在其他系統上開機或顯示錯誤訊息，這可能表示裝置並非從 USB 隨身碟自動開機。在這種情況下，請重新開機並進入 BIOS/UEFI 設定 (通常可透過按下「DEL」、「F2」、「F12」或「ESC」進入，視電腦製造商而定)。然後變更開機順序，讓 USB key 優先。然後重新啟動裝置以啟動 UmbrelOS。



### 步驟 4：在電腦上安裝 UmbrelOS



裝置從 USB 隨身碟開機後，您會看到 Interface UmbrelOS 安裝。此步驟會將系統直接安裝到您的機器內部 Hard 磁碟上。



出現的畫面會列出電腦偵測到的所有內部儲存裝置。每個磁碟都附有編號、名稱和儲存容量。找到要安裝 Umbrel 的磁碟。 **警告：此磁碟上的所有檔案都會被永久刪除。



![Image](assets/fr/007.webp)



確定正確的磁碟 (通常是容量最大的磁碟，以放置 Blockchain) 後，請記下分配給它的編號。例如，如果您選擇的磁碟出現在號碼`2`之下，只要輸入`2`，然後按下鍵盤上的`Enter`鍵即可。



![Image](assets/fr/008.webp)



程式會格式化選取的磁碟，安裝 UmbrelOS，並自動設定系統。這可能需要幾分鐘。讓這個過程不中斷地執行。



![Image](assets/fr/009.webp)



安裝完成後，系統會提示您關閉裝置。按任意鍵關閉電腦。



![Image](assets/fr/010.webp)



現在您可以移除 USB 鑰匙、鍵盤和螢幕，因為您的 Umbrel 不再需要這些東西。您的節點只剩下電源 Supply 和 RJ45 乙太網路線。



![Image](assets/fr/011.webp)



重新啟動裝置前，請檢查下列兩點：





- USB 隨身碟已拔除**：如果 USB 隨身碟仍保持連接，系統可能會在 USB 隨身碟上而非內接式磁碟上重新啟動；
- 乙太網路線已插入**：裝置必須連接到路由器才能運作。



按下電源按鈕。系統會自動從安裝 UmbrelOS 的內部磁碟啟動。第一次開機可能需要大約 **5 分鐘**。在此期間，Umbrel 會初始化其服務和 Interface。



從另一台連接到**相同區域網路**的電腦（您的日常電腦），開啟網頁瀏覽器（Firefox、Chrome...）並移至：



```
http://umbrel.local
```



此 Address 用於遠端存取 Umbrel Interface 圖形使用者 Interface 並開始設定。



如果等待至少 5 分鐘後，Address `http://umbrel.local` 在您的瀏覽器上仍無法運作，只需嘗試一下：



```
http://umbrel
```



如果還是不行，請直接在瀏覽器中輸入 Umbrel 的本機 IP Address。例如 (將 `42` 改為您在區域網路上託管 Umbrel 的機器號碼)：



```
http://192.168.1.42
```



要識別您的 Umbrel IP Address，有幾種方法，從最簡單到最先進的都有：





- 存取路由器的管理 Interface，並找出本端網路上 Umbrel 裝置的 IP Address。





- 使用網路掃描軟體（例如 Angry IP Scanner）來偵測連線的裝置，並找出 Umbrel 的 IP Address。



![Image](assets/fr/012.webp)



https://planb.network/tutorials/computer-security/communication/angry-ip-scanner-47f7c943-53b7-4098-b167-4cec8e747b5d

```
hostname -I
```



現在您可以使用 Umbrel 了！

### 步驟 6：在 Umbrel 上安裝 Bitcoin 節點



現在 UmbrelOS 已經正確地安裝和設定在您的機器上，您可以繼續安裝 Bitcoin 節點。沒什麼比這更簡單了：前往 App Store，開啟「*Bitcoin*」類別，然後選擇「*Bitcoin 節點*」應用程式（它實際上是 Bitcoin core）。



![Image](assets/fr/022.webp)



然後按一下「*安裝*」按鈕。



![Image](assets/fr/023.webp)



安裝完成後，您的 Bitcoin 節點將啟動其 IBD（*初始區塊下載*）：它將下載並驗證自 2009 年 Bitcoin 建立以來的所有交易和區塊。



![Image](assets/fr/024.webp)



這個階段特別耗時，因為其持續時間取決於幾個因素，包括分配給節點快取記憶體的 RAM 容量、磁碟速度、網路連線速度和處理器功率。因此，持續時間的範圍非常廣泛，視配置而定。使用高效能 PC（NVMe SSD、+32 GB RAM、強大的處理器和良好的網際網路連線），IBD 可在約 10 小時內完成。另一方面，老舊的處理器、低記憶體，或更糟的是，機械式 Hard 磁碟 (強烈不建議使用) 可能會將此作業延長至數週。



使用一般配置的電腦 (像樣的處理器、8 到 16 GB 記憶體和 SSD)，約可使用 2 到 7 天。



要稍微加快 IBD 的速度，您可以透過 `dbcache` 參數增加分配給節點快取記憶體 (主要用於 UTXO 集，我們稍後會再討論) 的 RAM。在 Umbrel 上，這項修改是在節點參數的「*最佳化*」標籤中進行。



![Image](assets/fr/025.webp)



預設情況下，Bitcoin core 中的 `dbcache` 參數值設定為 450 MiB，即約 472 MB。藉由增加此值，您可以稍微加快 IBD 的速度。但是，我不建議將此參數設定得太高：即使設定為 4 GiB 也只能使同步速度提高約 10%，而且可能會在 IBD 中斷時造成時間損失。



請小心不要分配一個對您的機器來說太大的值。如果 UmbrelOS 可用的 RAM 用盡，您的節點可能會突然停止，中斷 IBD 並需要您手動重新啟動，造成相當大的時間損失。



要了解更多關於 `dbcache` 參數對初始同步的影響，我推薦 Jameson Lopp 的這篇分析： [*Effects of DBcache Size on Bitcoin Node Sync Speed*](https://blog.lopp.net/effects-dbcache-size-Bitcoin-node-sync-speed/)



當您的節點的 IBD 完成後 (100% 同步化)，您就擁有了一個完全運作的 Bitcoin 節點。恭喜您，您現在已經是 Bitcoin 網路不可或缺的一部分了！



![Image](assets/fr/027.webp)



在下一部分，我們將探討您的新節點的實際使用：如何將您的 Wallet 連接到它，以及您應該安裝哪些應用程式才能成為主權 Bitcoiner。

## 索引器：角色、操作和解決方案


<chapterId>4f93c07a-f0cb-435f-8b68-162f316d7039</chapterId>



如果您在學習本課程之前已經探索過 Bitcoin 節點，您可能會遇到「索引器」這個名詞。這些工具如 Electrs 或 Fulcrum，可以新增到 Bitcoin core 節點。但它們的作用到底是什麼？它們實際上如何運作？您應該在新的 Bitcoin 節點上安裝一個嗎？這就是本章要探討的問題。



### 什麼是索引器？



一般而言，索引器是一個掃描原始資料集、擷取相關關鍵（如字詞、識別符和位址）並建立稱為「索引」的輔助檔案的程式，其中每個關鍵都是指資料在語料庫中的確切位置。這個預處理階段會耗費 CPU 的時間，並需要一些磁碟空間，但它免除了每次查詢資料庫時處理整個語料庫的需要；只要查詢索引，幾乎就能立即得到回應。



通俗來說，這與書本中的索引原理相同：如果您要尋找特定的資訊，與其重新閱讀整本書，不如參考索引，直接找到您要找的資訊出現的頁面。



在 Bitcoin 節點（例如 Bitcoin core）中，Blockchain 資料以原始的時序形式儲存。每個區塊包含交易，交易又包含輸入和輸出，沒有任何特定的 Address、識別符或 Wallet 分類。這種線性組織方式最適合區塊驗證，但不適合目標搜尋。例如，如果您想在非索引節點中找到與特定 Address 相關聯的所有交易，您必須手動逐區塊、逐交易檢閱整個 Blockchain。這正是 Bitcoin 節點上的索引器的用武之地。



![Image](assets/fr/085.webp)



索引器是一種專門的軟體程式，用來分析大量原始資料 (Blockchain、Mempool、UTXO 集)，並擷取交易識別碼、位址和區塊高度等鑰匙。從這些鑰匙建立索引，將每個鑰匙與節點儲存區中資訊的確切位置相關聯。



![Image](assets/fr/086.webp)



索引可讓您快速、準確、有效率地搜尋節點上的資訊。例如，當您將 Wallet 如 Sparrow 連接到您的節點時，它幾乎可以立即顯示 Address 的餘額。具體來說，它會以類似的請求來查詢索引器："_Which UTXOs are associated with this script-Hash?_" 索引器幾乎會立即回應，而不需要重新讀取整個 Blockchain，因為這些資料已經列在它的資料庫中。



### Bitcoin core 有索引器嗎？



在不需要額外軟體的情況下，嚴格來說，Bitcoin core 並未提供完整的 Address 索引器，無法媲美 Electrs 或 Fulcrum 等軟體中的索引器。儘管如此，Bitcoin core 仍然整合了多種內部索引機制，以及擴充查詢功能的選購選項。為了充分了解情況，我們需要繞道進入這個專案的歷史。



在 Bitcoin core 版本 0.8.0 之前，事務驗證基於全局事務索引，稱為 `txindex`。此索引參考所有 Blockchain 交易及其輸出。當節點收到一個新的交易時，它會參考這個索引來驗證消耗的輸出（在輸入中）是否真的存在，而且還沒有被花掉。因此，`txindex` 對當時的交易驗證是不可或缺的。



然而，這種方法有其限制：速度慢、儲存成本高、資訊冗餘。為了彌補這一缺點，0.8.0 版引入了稱為 ***Ultraprune*** 的驗證模型大修。Bitcoin core 不再以交易索引的形式儲存所有資料，而是維護一個專門用於 UTXO 的簡單資料庫，稱為「chainstate」（在日常語言中稱為「UTXO set」），並隨著輸出的消耗和建立更新其清單。



這個方法更快，而且只儲存暫存器的目前狀態，使得 `txindex` 索引器變得不必要。然而，開發人員並沒有刪除 `txindex` 程式碼，而是選擇將此功能保留在一個簡單的參數 (`txindex=1`)。只要在節點上啟用此選項，您就可以從其 `txid`查詢任何交易。



與普遍的看法相反，Bitcoin core 並不提供像 Electrs 或 Fulcrum 般基於 Address 的索引。這個選擇有幾個原因：





- Bitcoin core 的作用不是成為完整的 Block explorer，也不是提供針對各種用途量身打造的 API。整合基於 Address 的索引意味著需要長期維護 Commitment，這超出了軟體的初始範圍。





- 大多數的用例已經可以用其他方式涵蓋。例如，若要估算 Address 的餘額，您可以使用 `scantxoutset` 指令，它可以直接查詢 UTXO 集，而不需要完整的索引。





- 每個軟體程式對於要建立索引的資料格式或類型 (Address、Hash script、專屬標籤等) 都有特定的要求。比起在 Bitcoin core 中固定一個通用的解決方案，讓這些程式建立自己的客製化索引更有彈性，也更合乎邏輯。



Bitcoin core 確實有一個可選的交易索引器 (`txindex`)，這是其歷史操作的殘留物，但它不提供 Address 索引，也不直接提供 Interface 進行複雜的搜尋。因此，在某些情況下，增加一個外部索引器可能是有用的。



### 您是否應該在節點上新增 Address 索引器？



添加 Address 索引器（例如 Electrs 或 Fulcrum）並非必須；這取決於您的特定需求。



如果您只是想將 Wallet (例如 Sparrow) 連接到您的節點來檢視餘額和廣播交易，這完全可以直接透過 Bitcoin core 的 Interface RPC 來實現，無論是在本機或透過 Tor 遠端。



另一方面，若要使用更進階的軟體，例如執行 Mempool.Loc，安裝 Address 索引器成為空間 Block explorer 不可或缺的部分。



索引器需要一定的同步時間 (比 IBD 少)，並且會佔用額外的磁碟空間。如果您的 SSD 在下載 Blockchain 之後仍有足夠的可用空間，您可以輕鬆地新增索引器。



### 選擇哪一種索引器？



有兩種軟體程式常用於建立此類 Address 索引，並使其可被存取： **Electrs** 和 **Fulcrum**。這些工具根據腳本-Hash（地址）索引 Blockchain，然後建議一個標準化的 Interface（Electrum 協定），許多錢包，如 Electrum Wallet、Sparrow 或 Phoenix，都與之連接。



![Image](assets/fr/087.webp)



簡單來說，Electrs 相當精簡：它索引 Blockchain 的速度更快，佔用的磁碟空間更少，但在查詢上的表現卻稍遜於 Fulcrum。相比之下，Fulcrum 消耗的磁碟空間較多，索引時間較長，但查詢效能優異。



對於個人使用，我推薦 Electrs：它佔用的空間較少，維護良好，雖然在某些請求上比 Fulcrum 稍慢，但對於日常使用還是綽綽有餘的。如果您有時間和磁碟空間，也可以試試 Fulcrum，它的表現會好很多，尤其是在有許多地址需要驗證的錢包上。



具體而言，在 2025 年 8 月，Electrs 將需要約 56 GB 的儲存空間，而 Fulcrum 則需要約 178 GB。因此，您對索引器的選擇也取決於您的儲存容量：




- 如果您的磁碟空間非常有限，您只能使用 Bitcoin core，而無法使用外接式 Address 索引器。
- 如果您想使用索引器，但仍受限於容量，請選擇 Electrs。
- 如果您有足夠的磁碟空間，Fulcrum 可能正是您要找的。



在此 BTC 202 課程的其餘部分，我將使用 Electrs，但您也可以很容易地跟著使用 Fulcrum：安裝程序完全相同，Interface 與 Wallet 的連線也是一樣，因為兩者都暴露了一個 Electrum 伺服器。



### 如何在 Umbrel 上安裝索引器？



要在 Umbrel 上安裝 Electrs (或 Fulcrum)，步驟很簡單：前往 App Store，搜尋相關的應用程式 (位於 Bitcoin 標籤)，然後按一下「*安裝*」按鈕。



![Image](assets/fr/028.webp)



安裝完成後，Electrs 將進入同步化 (建立索引) 階段，此階段可能需要數小時。



![Image](assets/fr/029.webp)



同步化完成後，您就可以將 Wallet 軟體連接到託管在 Umbrel 上的 Electrum 伺服器。



## 如何將 Wallet 連接到 Bitcoin 節點？


<chapterId>35519b1a-f681-4a69-a652-9fbe510cd17f</chapterId>



現在您已經擁有一個完整的 Bitcoin 節點，是時候好好利用它了！在下一章中，我們將探討 Umbrel 實例的其他潛在用途。不過，讓我們從最基本的開始：連結您的 Wallet 軟體，利用您自己 Blockchain 的資訊，並透過您自己的節點分發交易。



如上所述，主要有兩種連接介面：




- 透過 RPC 直接連線至 Bitcoin core；
- 或連接到 Electrum 伺服器 (Electrs 或 Fulcrum)。



在本教程中，我們將專注於透過 Tor 連接到您的節點，因為對於初學者來說，這是一個既簡單又安全的解決方案。我強烈建議不要明確暴露您節點的 RPC 連接埠，因為錯誤設定會對您資料的安全性和機密性造成重大風險。透過 Tor 進行通訊的主要缺點是速度太慢。在下一章，我們將探討 Tor 以外另一種快速又安全的遠端存取節點方式：VPN.



在本章中，我們將以 Sparrow 為例，但對於所有其他接受與 Electrum 伺服器連線的 Wallet 管理軟體，步驟都是一樣的。只要在您的應用程式參數中找到相應的設定即可（通常在「*Server*」、「*Network*」、「*Node*」...）。



在 Sparrow 上，開啟「*檔案*」標籤，進入「設定」功能表。



![Image](assets/fr/030.webp)



然後按一下「*Server*」以存取連線參數。



![Image](assets/fr/031.webp)



接下來您會發現有三種選項可以將軟體連結到 Bitcoin 節點：




- Public Server* (黃色)：預設情況下，如果您不擁有 Bitcoin 節點，這個選項會將您連線到您不擁有的公共節點 (通常是公司的)。此選項與此無關，因為您在 Umbrel 上有自己的節點。
- Bitcoin core* (Green)：此選項對應於透過 Interface RPC 連線，即直接與 Bitcoin core 連線。
- Private Electrum*（藍色）：此選項可讓您透過索引器的 Interface Electrum 伺服器（Electrs 或 Fulcrum）連線。



### 連接至 Bitcoin core RPC



如果您的 Umbrel 節點沒有索引器，這是您需要選擇的選項。在 Sparrow 上，點選「*Bitcoin core*」。



![Image](assets/fr/032.webp)



接下來您需要輸入幾項資訊，以建立與節點的連線。所有這些資料都可以從 Umbrel 上的「*Bitcoin Node*」應用程式中取得，只要點選 Interface 右上角的「*Connect*」按鈕即可。



![Image](assets/fr/033.webp)



*RPC 詳細資訊*」標籤會顯示連線所需的所有資訊。選擇透過 Tor Address 連線 (在 `.onion` 中)。



![Image](assets/fr/034.webp)



在 Sparrow wallet 上的相應欄位輸入這些資料，然後按一下「*測試連線*」按鈕。



![Image](assets/fr/035.webp)



如果連線成功，則會出現 Green 剔號及確認訊息。



![Image](assets/fr/036.webp)



Interface Sparrow wallet 右下方的勾號現在會變成 Green（表示與 Bitcoin core 直接連線）。



**Note:** 若要連線成功，您的節點必須 100% 同步。如果不是這樣，請等到 IBD 結束。



### 連接至電器



如果您的節點有索引器，與其直接使用 Bitcoin core，不如連接到索引器，因為您的查詢會處理得更快。



在 Sparrow 上，移至「*私人電子幣*」標籤。



![Image](assets/fr/037.webp)



然後，您需要輸入幾項資訊，以建立與索引器的連線。您可以在 Umbrel 的「*Electrs*」應用程式（或在適用的情況下，「*Fulcrum*」）中找到這些資料。



選擇 "*Tor*"標籤以取得`.onion`連線 Address。如果您希望連接行動版 Wallet 軟體，也可以直接掃描 QR code。



![Image](assets/fr/038.webp)



只要在「*URL*」欄位中輸入您的 Electrum 伺服器的 Tor Address，然後點選「*測試連線*」按鈕即可。



![Image](assets/fr/039.webp)



如果連線成功，則會顯示核取標記和確認訊 息。



![Image](assets/fr/040.webp)



Interface Sparrow wallet 右下角的勾會變成藍色（與連線到 Electrum 伺服器相關的顏色）。



**Note:** 要使連接正常運作，您的索引器必須 100% 同步。如果不是這樣，請等待索引程序完成。

如前所述，探索器可以在網站上線託管，也可以在您的機器上本機執行。線上服務的一大缺點是可能會損害您的隱私。在沒有 VPN 或 Tor 的情況下，託管探索器的伺服器可以將您的 IP Address 與您正在檢視的交易連結起來，這可以為連鎖分析提供理想的切入點。

這就是為什麼最好使用您自己的本機 Block explorer。這樣一來，與您的搜尋活動有關的資料就不會外洩，因為所有查詢都是直接在您控制的機器上處理，而不會經過網際網路。更重要的是，本機探索器依賴您自己的 Bitcoin 節點所提供的資料，這些資料都是您根據自己的規則自行驗證過的，值得您信任。

### 尾尺



我特別喜歡 Umbrel 的另一個應用程式是 Tailscale。它是一個 VPN 應用程式，設計用來簡化多台裝置之間安全網路的建立，不論它們在世界何處。與依賴集中式伺服器的傳統 VPN 不同，Tailscale 利用 WireGuard 通訊協定在您的不同機器之間建立端對端加密連線。這表示您可以在短短幾分鐘內部署可用的 VPN，而不需要複雜的網路設定。



在 Umbrel 上，Tailscale 安裝會將您的 Bitcoin 節點連接至您自己的虛擬私人網路。配置完成後，您的節點會取得私人 Tailscale IP Address，只有連接到相同 Tailscale 網路的其他裝置 (例如電腦、智慧型手機和平板電腦) 才能存取。此連線經過端對端加密，不會經過未受保護的公用網路，與未加密的連線相比，安全性大幅提升。



![Image](assets/fr/090.webp)



具體而言，在使用 Umbrel 時，Tailscale 可為您提供多項優勢：





- 您可以在任何地方管理 Interface Umbrel 或存取與您的節點連結的應用程式 (例如 Mempool、Ride The Lightning、ThunderHub...)，就像在同一個區域網路中一樣，不需要在網際網路上暴露連接埠，也不需要經過速度非常慢的 Tor；





- 您可以連接到您的 Electrum 伺服器 (Electrs 或 Fulcrum) 或直接透過您的 VPN 連接到 Bitcoin core，繞過 Tor。這提供了一個安全的連線，與使用 Tor 相當，但速度更快，延遲時間更短。簡而言之，您可以保留 Tor 的隱私和安全優點，同時享受 Clearnet 連線的速度。對於 On-Chain Wallet 來說，這一收益看似微不足道，但如果您打算日後建立自己的 Lightning 節點，則差別就相當大。事實上，在 Tor 上透過您的節點進行移動支付，由於需要進行多次交換，因此速度極慢，而在 Tailscale 上則可以完美運作。





- 無需設定 NAT 規則、開啟埠或設定傳統的 VPN 伺服器。只要在 Umbrel 和您的裝置上安裝應用程式，網路就會自動建立。



因此，如果您想要以安全、高效能且容易設定的方式，從世界上任何地方存取您的節點，而又不犧牲隱私或安全性，那麼 Umbrel 上的 Tailscale 就是非常有趣的解決方案。

## 維護您的 Umbrel 結


<chapterId>06d77d09-bf24-4555-b2ba-c08bbda477c7</chapterId>



為了揭開這最後一節的序幕，在進入更進階的理論之前，我想在這短短的一章中，檢視一旦您的 Umbrel 節點安裝、同步化並正確設定後，您可以採取的最佳實作和具體行動。您如何進行日常維護？



### 保持設備健康



可靠的節點從穩定的硬體開始。確保安裝節點的機器通風良好、無 Dust，並安裝在乾燥的環境中，遠離任何熱源和濕氣。避免將它塞入狹小的空間，並選擇通風良好的位置。



在 Raspberry Pi 和 mini-PC 上，Dust 最終會堵塞散熱器、升高溫度並導致節流（自動限制資源使用），進而導致節點效率下降。這就是我建議定期清理進氣口和風扇的原因，最好每隔幾個月清理一次。



確保使用高品質的電源 Supply，因為不穩定的電壓可能導致系統損壞，甚至造成火災危險。最好使用機器製造商提供的原廠電源 Supply。此外，請小心因電源插座的焦耳效應而造成的過熱現象：務必遵守最大允許功率，切勿串接多個電源插座。



我也建議您投資一台 UPS。這可保護您的節點免於突然關機，讓 Umbrel 在發生停電時能乾淨地關機，並確保在微小停電或短期故障時能持續運作。



在儲存方面，請密切注意進度：如果磁碟已接近飽和，請考慮釋放空間 (卸載不使用的應用程式、調整索引器設定) 或遷移至更大的固態硬碟。完整 Bitcoin 節點的缺點在於其儲存需求會持續增加，因為每 10 分鐘就會產生一個新區塊，而且舊區塊無法刪除 (除非節點是 pruned)。因此，我建議您在購買硬體時規劃足夠大的容量 (至少 2 TB)。



### 更新



節點更新很重要，主要有三個原因：第一，安全性（漏洞修補、網路加固和 DoS 保護）；第二，相容性（中繼政策變更、格式變更和通訊協定升級）；第三，可靠性和效能（錯誤修補、資源消耗和其他改進）。因此，請定期檢查 UmbrelOS 和您的應用程式是否為最新版本：





- 要更新系統開啟設定功能表，然後按一下「*UmbrelOS*」參數旁邊的「*檢查更新*」按鈕。



![Image](assets/fr/042.webp)





- 更新應用程式：前往 App Store。如果您有任何應用程式需要更新，Interface 的右上角會出現紅色泡泡的按鈕。只要按一下按鈕，即可更新各應用程式。



定期執行此操作，以保持作業系統和應用程式為最新版本。



### 備份



如果您只使用您的 Bitcoin 節點來驗證和分發您的交易，但您的錢包是在 Umbrel 外部管理的 (例如，使用 Hardware Wallet 和 Sparrow wallet)，就沒有什麼可以直接備份到 Umbrel。在這種情況下，基本備份仍然是您外部 Wallet 的復原短語和 Descriptor，無論您是否使用自己的節點，這一點都適用。因此，您之前的配置沒有任何改變。



另一方面，根據您在 Umbrel 上使用的其他應用程式，可能需要進一步備份。如果您在 Umbrel 上操作 Lightning 節點，情況尤其如此。在這種情況下，備份您安裝 Lightning 節點時提供的 seed 是絕對必要的。除了seed，您還需要一個最新的***靜態通道備份(SCB)***，以便在發生問題時恢復您的Lightning節點。SCB 可讓您透過強制關閉通道來恢復資金。如果缺少seed或SCB，就無法還原Lightning節點。



Umbrel 也提供透過 Tor 在其伺服器上自動動態備份 SCB 的選項，以確保可隨時取得最新檔案。在這種情況下，只需要 seed 就可以還原節點。

## 瞭解 IBD 和同儕發現程序


<chapterId>175ac9d1-ea23-45d9-9918-d3e7352435cd</chapterId>



您的 Bitcoin 節點啟動時，事先並不知道任何交易歷史。起初，它只是一台執行軟體 (Bitcoin core 或類似的軟體) 的電腦。要成為完全同步運作的 Bitcoin 節點，它必須透過檢查自 Genesis 區塊（2009 年 1 月 3 日由 Satoshi Nakamoto 發佈的區塊 0）以來所發佈的所有區塊，在本機重建 Ledger 的狀態。這個步驟稱為 **IBD（_初始區塊下載_）**。



IBD 包括下載並個別驗證每個區塊和交易，應用共識規則，以建立自己版本的 Blockchain。這樣做的目的不是簡單地擷取一份未經驗證的資料，而是完全獨立地得出相同的結論，就像網路中誠實的大多數人一樣。



![Image](assets/fr/092.webp)



### IBD 里程碑



同步從 _**headers-first**_ 步驟開始。您的節點會向多個對等端要求區塊標頭順序，並針對每個標頭驗證 Proof of Work、難度調整、語法以及 Timestamp 和版本號規則。簡而言之，它會確保收到的每個標頭都符合共識規則。



![Image](assets/fr/093.webp)



提醒一下，Bitcoin 區塊由一個 80 位元組的標頭和交易清單組成。該區塊的指紋是將雙 SHA-256 Hash 應用於此標頭（包含 6 個欄位）而得到的：




- 版本
- 上一區塊的 Hash
- 交易的 Merkle Root
- Timestamp（大於前 11 個區塊的中位時間）
- 難度目標
- Nonce



![Image](assets/fr/094.webp)



交易提交到 Merkle Tree。這是一種總結大型資料集（在此情況下，區塊中的所有交易）的結構，方法是將資料集的哈希值以兩個對兩個的方式逐步累積到單一「根」，從而證明元素屬於資料集（並偵測任何修改）。如此一來，對交易的任何修改也會修改 Merkle Tree 的根，因此也會修改區塊標頭的指紋。SegWit 為 Cookie (簽名) 引進了獨立的附加 Commitment，放置在 coinbase 中。



![Image](assets/fr/095.webp)



這個 _**headers-first**_ 步驟讓節點能夠識別工作量最多的分支 (不論其區塊數)，也就是 Bitcoin 節點同步的分支。一旦識別出這個分支，節點就會從數個連線平行下載區塊的內容，然後驗證每筆交易：格式、腳本的有效性 (`assumevalid=1`除外)、金額，以及是否有雙重花費。每次成功檢查後，`chainstate/` 資料庫中未花費的硬幣 (UTXO 集) 的當前狀態就會更新：已花費的輸出會被移除，而新的有效輸出則會被加入。



另一方面，Mempool 只有在接近鏈頂端時才會發揮作用：只要節點仍然遲到，它就沒有待定交易要儲存。



一旦 IBD 完成，節點就會進入正常階段：在發佈新區塊時進行驗證，根據其中繼規則維護其 Mempool 與待處理交易，中繼交易和區塊，並管理任何鏈重組。



### 假設有效



Bitcoin core 納入了一種機制，旨在減少節點完全運作之前所需的時間，同時保留自主驗證原則的精髓：AssumeValid.



assumevalid」參數是基於過去的參考區塊，其 Hash 已整合到每個軟體版本中。在 IBD 期間，如果您的節點發現此區塊確實在工作最多的分支上，就可以忽略此點之前所有交易的腳本驗證。



所有其他規則（區塊結構、Proof of Work、大小限制、交易金額、UTXOs 等）仍完全經過驗證。只有此參考區塊之前的腳本計算會被忽略。由於簽章驗證佔據了 CPU 負載的主要部分，因此 IBD 的效能增益非常顯著。在此參考區塊之後，驗證會回到正常狀態。



您可以使用 `Bitcoin.conf` 檔案中的 `assumevalid=0` 參數，停用此機制，強制對所有腳本進行完全驗證，但代價是 IBD 要長得多。



### 假設UTXO



`assumeutxo` 是另一個現有的參數，但與 `assumevalid` 不同，它不是預設啟動的。此機制可讓軟體載入 UTXO 集的快照，以及其元資料，並在驗證標頭確實引導到工作最多的 Blockchain 之後，暫時將其視為參考狀態。



如此一來，節點就能快速地運作一般用途 (RPC、連接錢包等)，同時在背景中啟動其本身 UTXO 設定的完整、經驗證的重構。一旦此階段完成，初始快照就會被本機重建的狀態取代。此方法將快速節點提供與完整驗證分開，但不會影響後者。



### 對等體發現：您的節點如何找到 Bitcoin 網路？



當節點第一次啟動時，它還不認識任何對等體。但是，它必須找到網際網路上的其他 Bitcoin 節點來請求標頭，然後是區塊，以便完成其 IBD。為了啟動這些連線，Bitcoin core 遵循優先順序的邏輯。



![Image](assets/fr/096.webp)



當節點在使用過後重新啟動時，Core 會首先嘗試重新連線到關機前註冊的對等網路，這些資訊儲存在 `anchors.dat` 檔案中。接著，它會查閱其 IP Address 帳本 **`peers.dat`**，其中儲存了之前遇到的對等體清單，以便與它們重新連線。這只是一個本地檔案，由 Core 更新和保存。另一方面，對於剛啟用的新節點，這兩個檔案是空的，因為它從未與其他 Bitcoin 節點通訊過。



在這種情況下，軟體會查詢 _**DNS 種子**_。這些是 [由公認的生態系統開發者所維護的伺服器](https://github.com/Bitcoin/Bitcoin/blob/master/src/kernel/chainparams.cpp)，會傳回一組假定活躍節點的 IP 位址清單。這些位址可讓新節點啟動第一次連線，並向 IBD 請求必要的資料。以下是迄今為止 (2025 年 8 月) 活躍的 *DNS 種子*清單：




- Pieter Wuille：`seed.Bitcoin.sipa.be.`
- Matt Corallo: `dnsseed.bluematt.me.`
- Luke Dashjr：`dnsseed.Bitcoin.dashjr-list-of-P2P-nodes.us.`
- Jonas Schnelli: `seed.Bitcoin.jonasschnelli.ch.`
- Peter Todd：`seed.btc.petertodd.net.`
- Sjors Provoost：`seed.Bitcoin.sprovoost.nl.`
- Stephan Oeste：`dnsseed.emzy.de.`
- Jason Maurice：`seed.Bitcoin.wiz.biz.`
- Ava Chow：`seed.Mainnet.achownodes.xyz.`。



在絕大多數情況下，*DNS 種子*步驟足以建立與其他節點的第一次連線。如果在特殊情況下，這些伺服器在 60 秒內沒有回應，節點就會改用其他方法：Bitcoin core 的程式碼內建了 [超過 1,000 個位址的靜態清單](https://github.com/Bitcoin/Bitcoin/blob/master/src/chainparamsseeds.h) _種子節點_，並會定期更新。如果前兩種取得 IP 位址的方法失敗，最後一種解決方案會建立初始連線，接著節點就可以從中要求新的 IP 位址。



![Image](assets/fr/097.webp)



最後，您可以透過 `peers.dat` 檔案手動 Supply IP 位址來強制特定連線。



啟動後，內部 Address 管理員會分散來源 (獨立的自治網路、clearnet 和 Tor，以及不同的地理區域)，以降低拓樸隔離的風險。節點會建立這些向外連線 (它自己選擇的連線，因此更安全)。



如果您的節點在開放的連接埠（預設為 8333）上聆聽，它就會接受傳入的連線。這些連線透過提供新節點的接觸點，加強網路的整體復原能力，但不會為您自己的 IBD 帶來任何特別的好處。如果您的節點在 Tor 上執行，邏輯仍然相同，但使用的位址是 `.onion` 服務。




## 剖析您的 Bitcoin 結


<chapterId>b420bd9d-7e2a-4984-bc70-2b732a94c8ce</chapterId>



當您的節點完成初始同步後，它會在本機儲存幾個互補資料集，使它能夠驗證區塊和交易、服務網路對等機構，並在維持狀態的同時快速重新啟動。節點上有 3 個主要的磚塊是不可或缺的：




- gW-402 ** 區塊**儲存在磁碟上、
- **UTXO 集**維護在鍵值資料庫中、
- 和 **Mempool** 儲存在 RAM 中，並定期序列化。



此外，還有幾個輔助檔案（同行、費用估算、排除清單、錢包等），使整個畫面更加完整。讓我們來瞭解所有這些檔案的作用。



### 節點的資料實際位於何處？



預設情況下，Bitcoin core 會將資料儲存在特定的工作目錄中。在 GNU/Linux 下，這通常是 `~/.Bitcoin/`，在 Windows 下是 `%APPDATA%/Bitcoin/`，在 macOS 下是 `~/Library/Application Support/Bitcoin/`。如果您使用的是套件解決方案 (例如，在節點發行版中)，此目錄可能會掛載到其他地方，但其結構保持不變。下面描述的重要子資料夾和檔案仍然位於此處。



![Image](assets/fr/098.webp)



### 磚塊



因此，Blockchain 是區塊的集合。Full node 會將這些區塊儲存為順序的平面檔案，並維護平行索引以進行快速檢索。需要時 (重組、Wallet 重新掃描、同儕服務)，這些資料會原封不動地重新讀取。



**註：** 重組或重新同步化是一種現象，Blockchain 會因為同一高度存在競爭區塊而發生結構修改。當 Blockchain 的一部分被另一個累積工作量較多的鏈取代時，就會發生這種情況。這些重新同步化是 Bitcoin 運作的自然部分，不同的礦工幾乎可以同時找到新區塊，進而將 Bitcoin 網路一分為二。在這種情況下，網路可能會暫時分裂為相互競爭的區塊鏈。最後，當其中一條鏈累積了更多工作時，其他鏈就會被節點放棄，它們的區塊就會被稱為「過時區塊」或「孤兒區塊」。這種以一條鏈取代另一條鏈的過程稱為重新同步。



#### Blk*.dat 檔案（原始區塊資料）



接收並驗證的區塊會寫入命名為 `blkNNNN.dat` 的順序容器中，儲存於 `blocks/` 資料夾。每個檔案都會依序填滿，直到達到 128 MiB 的最大容量為止，此時 Core 會開啟下一個檔案。在檔案中，每個區塊都以網路格式序列化，前面加上魔法識別碼和長度。這種組織方式可快速寫入磁碟，並方便區塊服務同步化對等。



![Image](assets/fr/099.webp)



在 pruned 模式中，節點只保留這些檔案的最近視窗，以限制磁碟佔用空間。一旦達到設定的空間目標，它會立即刪除最舊的「blk*.dat」容器，同時保留足夠的歷史記錄，以便與最知名的鏈保持一致。索引和 UTXO 集仍保持正常，使下一個交易和區塊得以驗證。



#### Rev*.dat 檔案（取消資料）



為了能夠在重組過程中回溯時間，Core 會在每個 `blk` 檔案的同時，在 `blocks/` 中儲存一個 `revNNNN.dat` 檔案。此檔案包含撤銷區塊對 UTXO 設定的影響所需的資訊： 區塊消耗的每個輸出，都會儲存相對應的 UTXO 先前的狀態（數量、腳本、高度...）。在區塊中止的情況下，節點可以快速重建之前的狀態，而無需重新掃描整個鏈。



![Image](assets/fr/100.webp)



#### 區塊索引 (區塊/索引)



直接在平面檔案中搜尋區塊會太費時。因此，Core 在 `blocks/index/` 中維護一個 LevelDB 資料庫，其中列出每個已知區塊的元資料，例如 Hash、高度、驗證狀態、`blk` 檔案，以及區塊所在的偏移量。當對等人請求區塊，或內部元件需要存取特定區塊時，此索引可提供快速存取。如果沒有這個索引，就需要進行太多的作業。



![Image](assets/fr/101.webp)



#### 可選索引 (indexes/)



某些索引是可選的，預設為停用，因為它們會增加磁碟佔用空間：




- 我們已經提到過的 `indexes/txindex/` 提供了一個交易 → 位置對應表，讓我們可以在不知道包含該交易的區塊的情況下擷取任何已確認的交易。這對非 Wallet `getrawtransaction` 類型的 RPC 查詢很有用，但相當昂貴。
- indexes/blockfilter/` 可包含精簡型用戶端的精簡區塊篩選器 (BIP157/158)。這些結構可以加速用戶端驗證，但卻要犧牲索引節點上的額外儲存空間。



### UTXO 套件 (連鎖狀態)



UTXO (* 未動用交易輸出*) 模型是 Bitcoin 的會計表示：每項未動用輸出都是可用的「Coin」，可用來作為未來交易的輸入。



![Image](assets/fr/102.webp)



所有這些零件在某個特定時刻 T 的總和構成 UTXO 集：一個現在可用的所有零件的大清單。節點會參考這個狀態，以決定交易是否花費合法的單位，而這些單位尚未用於之前的交易（以避免 Double-spending）。



![Image](assets/fr/103.webp)



UTXO 集作為精簡的 LevelDB 資料庫儲存在 `chainstate/` 資料夾中。每個部分關聯一個由交易的 Hash 和輸出索引衍生出來的關鍵，其值包含：金額、`scriptPubKey` 鎖、創建區塊的高度和一個 coinbase 指示器。



![Image](assets/fr/104.webp)



節點在 LevelDB 上方維護一個記憶體快取，以吸收頻繁的讀寫作業。可以使用 `dbcache` 參數來修改這個快取記憶體的大小：快取記憶體越大，IBD 和目前的驗證就會受益於更多的記憶體存取，但代價是更高的 RAM 消耗。當 Miner 發現一個新區塊時，節點會從 UTXO 設定中刪除該區塊所包含的交易花費（或消耗）的輸出，並加入新建立的輸出。



理論上，我們可以透過重新掃描區塊歷史記錄來驗證交易，以檢查輸出是否從未被花費。但實際上，這將會非常耗時，因為每筆新交易都必須掃描整個 Blockchain。因此，UTXO 集提供所需的最低限度檢視，可在合理的時間內就地證明 Double-spending 不存在。



請注意，UTXO 套件經常是關於 Bitcoin 分散性的核心問題，因為其規模自然會快速增加。這一方面是因為 Bitcoin 的價格不斷上漲，鼓勵零件分散，另一方面則是因為系統的採用率越來越高：使用者越多，對 UTXO 的需求就越大。



![Image](assets/fr/105.webp)



UTXO 集的成長也源自於 Bitcoin 上簡單付款交易的結構。事實上，當您進行付款時，您會消耗一個 UTXO 作為輸入，並產生 2 個新的 UTXO 作為輸出 (一個用於付款，另一個用於 Exchange)。最後，一個稱為 CIOH (*Common Input Ownership Heuristic*) 的連鎖分析啟發式，提供了避免 Coin 整合的進一步誘因。



https://planb.network/courses/65c138b0-4161-4958-bbe3-c12916bc959c

由於其中一部分必須保留在 RAM 中，才能在合理的時間內驗證交易，因此 UTXO 集可能會逐漸使 Full node 的操作成本過高。為了解決這個問題，目前已有一些建議，特別是 [Utreexo](https://planb.network/resources/glossary/utreexo)。



### Mempool



Mempool 是已收到但尚未確認的本地有效交易集。提醒一下，「已確認交易」是指已包含在有效區塊中的交易。每個節點都維護自己的 Mempool，它可能與網路中其他節點的 Mempool 有所不同，這取決於以下因素：




- 透過 `maxmempool` 參數分配給 Mempool 的大小：擁有較大 Mempool 的節點將能夠比擁有較小 Mempool 的節點容納更多的交易 (除非後者變空)；
- gW-433 規則：這些規則是節點中繼規則的子集，定義了未確認交易必須符合的特性，才能在 Mempool 中被接受；
- 交易滲透：由於各種因素，特定的交易可能已分佈到網路的一部分，但尚未到達另一部分。



必須注意的是，節點 mempool 沒有共識值。即使每個節點都有不同的 Mempool，Bitcoin 也能完美運作。最終，權威區塊永遠是那些加入 Blockchain 的區塊。例如，即使節點最初在其 Mempool 中拒絕了某個交易（根據共識規則是有效的），但如果該交易最終被包含在具有有效 Proof of Work 的區塊中，它就有義務接受該交易。如果它沒有這樣做，拒絕了這個區塊，即使它遵守共識規則，也會觸發 Hard Fork，也就是建立一個新的、獨立的 Bitcoin，在這個區塊上只有它一個節點。



#### 記憶體政策與管理



收到交易時，Core 會根據共識規則（語法、有效腳本、無重複支出等）和 Mempool 規則（RBF、最低收費臨界值、`OP_RETURN`中的資料限制等）進行一系列檢查。如果交易遵守這些規則，就會儲存於記憶體中。



Mempool 的大小受到 `Bitcoin.conf` 檔案中的 `maxmempool` 參數限制 (下一章將詳細介紹)。預設限制為 300 MB。當它滿了時，節點會動態地提高最低收費臨界值，並先驅逐最不獲利的交易 (也就是保留應該先挖的交易)。太舊的交易也會在設定的延遲時間後過期。



#### Mempool 在磁碟上的持久性



為了加快重新啟動的速度，當節點關閉時，Core 會定期將 Mempool 的狀態序列化為 `Mempool.dat` 檔案。除了保留在記憶體中的實際 Mempool 外，Core 還會將這個 `Mempool.dat` 檔案儲存在磁碟上。下次啟動節點時，會重新載入此快照，並刪除對目前 Blockchain 不再有效的內容。



### 輔助檔案和資料庫



與 `blocks/`、`chainstate/` 和 `indexes/` 同級的其他幾個檔案也參與了正常運作：




- `peers.dat` 保存了潛在對等點的 IP Address 紀錄，由初始 DNS 發現、網路交換和手動新增提供。節點啟動時，可利用此檔案建立向外連線。
- 關閉節點時，`anchors.dat` 會儲存外寄對等體的位址，以便您下次啟動時可以嘗試快速地再次與他們聯絡。
- `banlist.json` 包含由操作員或節點 (重複無效的行為) 決定的本地封禁，以防止節點重新連線或接受來自這些特定對等點的連線。
- `fee_estimates.dat` 儲存觀察到的確認的時間範圍統計資料，費用估算器使用這些資料來提出與建立交易時選擇的延遲目標一致的費用比率。
- gW-446.conf` 包含您節點的設定參數。您可以在這裡調整中繼規則。我會在下一章告訴您更多關於這方面的資訊。
- `settings.json` 包含 `Bitcoin.conf` 的附加參數。
- `debug.log` 是診斷文字日誌，在發生 bug 時可用來瞭解節點活動。
- gW-448.pid` 會在執行時儲存進程識別碼，讓其他應用程式或指令碼可以輕鬆識別 bitcoind (*Bitcoin daemon*)，並在必要時與之互動。它會在節點啟動時建立，並在關機時刪除。
- `ip_asn.map` 是 IP → ASN 對應表 (獨立系統)，用於分桶和對等分散 (`-asmap`選項)。
- 當 `-listenonion` 選項啟用時，`onion_v3_private_key`會儲存 Tor v3 服務的私密金鑰，以便在重新開機之間保持穩定的洋蔥 Address。
- 當使用 `-i2psam=` 時，`i2p_private_key` 會儲存 I2P 私密金鑰，以便在 I2P 上建立傳出和可能傳入的連線。
- 當使用 cookie 驗證時，`.cookie` 包含短暫的 RPC 驗證 token（啟動時建立，關機時刪除）。例如，這可用於連接 Wallet 軟體。
- `.lock` 是資料目錄鎖，可防止多個實體同時寫入相同的資料目錄。
- `guisettings.ini.bak` 是使用 `-resetguisettings` 選項時自動儲存的 GUI 設定 (*Bitcoin Qt*)。



正如我們在 BTC 202 課程的第一部分所看到的，Bitcoin core 同時是 Bitcoin 節點軟體和 Wallet。但是，它不一定是我推薦的管理您的錢包的解決方案，因為它的 Interface 仍然是基本的，與 Sparrow 或 Liana 等現代軟體相比，它的功能有限。Core 也包含用於管理您的錢包的檔案：





- `wallets/` 是存放一個或多個的預設目錄；
- `wallets/<name>/Wallet.dat`是 Wallet 的 SQLite 資料庫（金鑰、描述符、交易元資料等）；
- wallets/<name>/Wallet.dat-journal` 是 SQLite 回滾日誌。

### 新區塊的驗證路徑



在收到新區塊時，您的節點會檢查 Proof of Work，更廣泛而言，檢查是否符合共識規則。如果一切正常，它會逐筆交易將變更套用到它的 UTXO 集：它會檢查每個條目是否以有效的腳本花費現有的 UTXO，刪除這些 UTXO，並加入新的 exits。如果一切有效，變更就會提交到 `chainstate/`。



同時，撤銷資料會寫入 `rev*.dat` 並將 metadata 寫入 `blocks/index/` 索引。然後區塊會序列化到正確的 `blk*.dat` 檔案。在重組的情況下，節點會反向讀取 `rev*.dat` 來乾淨地斷開被遺棄的區塊，還原 UTXO 集，然後連接新的最佳鏈的區塊。





## 瞭解 Bitcoin.conf

Bitcoin.conf ` 檔案是 Bitcoin core 的主要 Interface 設定檔案。它允許您調整節點的行為和參數，而無需重新編譯其原始碼或修改指令列。具體來說，它是一個以 key-value 對為結構的純文字檔案，意即檔案的每一行都參照一個特定的參數 (key) 及其相關值，可以透過修改來調整該參數。



網路、交易中繼、效能、索引、日誌和 RPC 存取參數可以在 `Bitcoin.conf`中定義。但是，這個組態檔從來不會修改通訊協定的共識規則：它只會設定節點的本機政策 (中繼規則)、連線方式、索引和揭露服務。



### 地點與優先順序



預設情況下，`Bitcoin.conf` 會存放在 Bitcoin core 資料目錄中。這是我們在前一章提到的著名目錄。不過，Bitcoin core 並不會自動建立這個檔案，除非是在特定的環境中，例如 Umbrel。如果它不存在，您必須自己 generate 它，只要建立一個名為 `Bitcoin.conf`的檔案，然後在文字編輯器中打開它進行修改即可。



在 `Bitcoin.conf` 中定義的參數可由 2 層覆寫：




- `settings.json` (由 Interface 圖形或某些 RPC 動態寫入)、
- 以及透過指令行修改的選項。



請注意，任何對 `Bitcoin.conf` 的修改都需要重新啟動節點才能生效。



### 格式與結構



因此，`Bitcoin.conf` 的格式非常簡單：每個選項一行，格式為 `option=value`。不必要的空格和空行會被忽略，而代碼註解則以 `#` 開頭。



幾乎所有的布林選項都可以用 `no` 前綴來停用。例如，根據版本不同，`listen=0` 和 `nolisten=1`是等效的。



若要依網路區分設定，您可以使用區段：`[main]`、`[test]` (testnet3)、`[testnet4]`、`[bookmark]`、`[regtest]`。或者，您可以在選項名稱前加上 `regtest.maxmempool=100`。



### Bitcoin.conf 能做和不能做的事



如上所述，共識規則顯然無法在 `Bitcoin.conf` 中設定，因為這可能會產生 Hard Fork。另一方面，許多其他方面是可以設定的。有 3 個有用的類別需要記住：




- 純本地參數。這些參數只會影響您的節點：快取大小 (`dbcache`)、pruned 模式 (`prune`)、可選索引...它們會影響您電腦的效能，但不會影響網路。
- 中繼和 Mempool 政策。這些決定您的節點在確認前接受、保留和中繼的內容：最低費用臨界值 (`minrelaytxfee`)、Mempool 大小和保留時間 (`maxmempool`、`mempoolexpiry`)、交易替換 (RBF)...這些規則不是共識的一部分，因此兩個不同的節點可以有不同的政策，而且仍然完全相容。另一方面，這些參數會對 Bitcoin 網路產生影響 (如第一部分所解釋，主要是滲透理論)。
- 網路連線。這些選項決定您的節點如何尋找對等、聆聽、穿越 NAT、使用 Tor 或代理，或限制其頻寬。它們會塑造您的拓樸結構，但不會改變交易的中繼。



理解這種分離是很重要的：如果交易不遵守共識規則，您的節點在任何情況下都會拒絕它。但是，更嚴格的本地政策可能會拒絕轉發在共識意義上有效的交易。



### 網路與拓樸



首先，必須清楚區分 Bitcoin 節點所能擁有的 2 種連線類型：




- 出站連線，由我們的節點啟動到另一個節點；



![Image](assets/fr/106.webp)





- 傳入連線，由其他節點啟動到我們的節點。



![Image](assets/fr/107.webp)



這兩種連線完全可以在兩個方向上交換相同的資料；這並不是限制流量方向的問題，而只是連線發起者的不同。從我們節點的觀點來看，向外連線通常被認為是比較安全的，因為我們啟動連線，並精確選擇連線到哪個節點，使得連線不太可能是惡意的。預設情況下，Bitcoin core 會維護 10 個傳出連線 (8 個「*full-relay*」+ 2 個「*block-relay-only*」)。



Full node 透過接受傳入的連線，為網路增加更多的價值。listen=1」參數會啟用有關網路的預設連接埠 8333 的聆聽功能，讓 Clearnet 可以接收這些傳入的連線。若要執行此功能，您的路由器也必須開啟此連接埠。如果沒有開啟，您的節點將繼續只使用傳出連線，這對您個人使用 Bitcoin 並無影響。您可以選擇是否允許傳入連線；沒有「最佳選擇」。



如果您不想在路由器上開啟連接埠，但仍然接受進入的連線，您可以啟用 `listenonion=1`參數。這將達到相同的結果，但只能透過 Tor 網路而非 clearnet。



在網路層級，我們也有




- `addnode`: 除了一般的發現之外，還會新增一個友好的對等端點來聯絡 (可以指定數次)。
- connect`：嚴格限制連線至所提供的 Address（可多次指定）。Core 將不會連線到任何其他節點。
- `seednode`: 僅用於連線到節點時填入書本-Address，然後斷開連線。
- `maxconnections`: 定義傳入 + 傳出連線的全局上限。預設情況下，此參數設定為 125，表示您的節點永遠不會接受超過 125 個連線。
- maxuploadtarget`: 設定上傳上限，以限制 24 小時滑動視窗的頻寬。此上限不會犧牲基本的最近 Elements 的傳播。
- `onlynet`：僅限制出站連線至選定的網路 (`ipv4`、`ipv6`、`onion`、`i2p`、`cjdns`)。例如，如果您希望您的節點只透過 Tor 連線到 Bitcoin 網路，您可以啟用 `onlynet=onion` 參數，並禁止傳入連線 (或也只允許透過 Tor 連線)。
- `dnsseed`: 允許或禁止 _DNS 種子_ 在您的本機 Address pool 不足時要求對等體 (預設值：`1`，除非 `-connect` 或 `-maxconnections=0`).
- `forcednsseed`: 強制在啟動時請求 _DNS 種子_，即使您已有網址 (預設值: `0`)。
- `fixedseeds`：如果 _DNS seeds_ 失敗或停用，允許使用 *seed 節點* (硬碼 Address 清單) (預設值: `1`)。
- `dns`：授權一般的 DNS 解析 (例如，用於「-addnode」/「-seednode」/「-connect」)。



預設情況下，您的節點會透過 clearnet、Tor 和 I2P 進行通訊。這表示它在 clearnet 上連線的對等端可以看到您的公開 IP Address，而您的 ISP 很可能會偵測到您正在執行 Bitcoin 節點 (雖然 P2P Transport V2 會讓 ISP 更難竊聽)。這不一定是個問題，但如果您想避免這些資訊外洩，您可以完全透過 Tor 網路連接您的節點。



若要完全啟用 Tor，您需要強制 Bitcoin core 只使用此網路，並為傳入的連線建立隱藏服務 (如果您要啟用的話)。您需要在 `Bitcoin.conf` 中加入此設定：




- `onlynet=onion`、
- `proxy=127.0.0.1:9050`,
- `listenonion=1`、
- `torcontrol=127.0.0.1:9051`,
- proxyrandomize=1`、
- `listen=1`、
- bind=127.0.0.1`、
- `upnp=0`、
- natpmp=0`。



您所有的 P2P 連線都會經過 Tor。您的節點會接收一個 `.onion` Address 來傳入連線，因此不需要在路由器上開啟任何連接埠。您的 ISP 只會看到 Tor 流量，而您的對等方並不知道您的實際公共 IP Address。



為了避免 DNS 解析不明，您可以在設定中加入 `dnsseed=0` 和 `dns=0`。然後您需要透過 `seednode=` 或 `addnode=` 來手動提供 `.onion` 對等點，否則很難發現新的節點。



很明顯，如果您是新手，我建議您暫時不要使用這些網路設定。預設設定通常就足夠了。



### Mempool 與中繼政策



#### 基本參數



以下是您可以在「Bitcoin.conf」中修改的基本參數，這些參數與 Mempool 的管理和未確認交易的轉送有關：





- `maxmempool=<n>`：將本機 Mempool 的最大大小限制為 `<n>`MB (預設值：`300`)。當達到限制時，您的節點會動態提高其有效費用臨界值，並優先處理獲利最少的交易 (根據費用比率，而非絕對值)，以維持在限制以下。您可以將此設定保留為預設值。如果您是 Mining 的單獨使用者，或如果您想要更精確地瞭解 Mempool 擁擠情況並改進費用估算，則提高它會很有用。相反地，減少它可以節省 RAM，其次也可以節省其他系統資源。





- `mempoolexpiry=<n>`：Mempool 中未確認交易的最長保留時間 (以小時為單位，預設：`336`)。超過這段時間後，即使仍有可用空間，交易也會被移除。





- `persistmempool=1`：儲存 Mempool 在靜止時的快照，並在重新開機時重新載入 (預設值：`1`)。這可以加快重新開機後的復原速度，避免需要透過網路重新學習狀態。





- `maxorphantx=<n>`：保留無主交易的最大數量 (來自 UTXO 集尚未看到的 UTXO 的依賴輸入，預設值：`100`)。超過此臨界值後，最舊的交易會被刪除，以避免快取記憶體不受控制地成長。





- blocksonly=1`：停用接受和重新傳送從對等點收到的未確認交易 (除非授予特殊權限)。節點現在只上傳和公告區塊。本機建立的交易仍可廣播 (以使用您的節點與您的 Wallet 軟體)。這大大降低了頻寬和 RAM 的需求，儘管代價是降低了中繼的實用性，以及完全不熟悉 Mempool。





- `minrelaytxfee=<n>`：最低費率 (以 BTC/kvB 為單位)，低於此費率的交易將不會被節點的 Mempool 接受，也不會轉送至對等節點 (預設值: `0.00001` = 1 sat/vB)。此值越高，您的節點過濾低費用交易的力度就越大。





- `mempoolfullrbf=1`：接受 RBF 交易，即使被取代的交易中沒有明確的 RBF 訊號。使用此「*full-RBF*」政策，如果符合其他取代條件，提供較高費率的交易可以取代 Mempool 中的其他交易。



提醒一下，RBF 是一種交易機制，可讓寄件者以費用較高的交易取代原來的交易，以加快確認速度。如果費用太低的交易仍然被凍結，寄件者可以使用 *Replace-by-fee* 來增加費用，並在 Mempool 和礦工處優先處理其替代交易。



#### 進階及特定設定



以下是 Mempool 和中繼政策的進階設定。如果您是初學者，應該不需要修改這些設定：





- datacarrier=1`：允許透過 `OP_RETURN` 輸出中繼和（如果 Mining 透過節點）包含載有非財務資料的交易（預設：`1`）。停用此參數可稍微減少非財務資料垃圾郵件的表面面積，但代價是降低與某些用途的相容性。在任何情況下，您都必須接受已挖掘的 `OP_RETURN`。





- `datacarriersize=<n>`：節點轉送的 `OP_RETURN` 的最大大小 (位元組)(預設值: `83`)。降低此值可限制經由 `OP_RETURN` 傳輸的有效負載。請注意，在 Bitcoin core 未來的版本中，預設會移除此限制。





- `bytespersigop=<n>`：參數，可將簽章交易轉換成等效位元組，以便進行中繼限制評估 (預設：`20`)。這會根據當地的政策規則，影響對 `sigops` 豐富交易的接受。





- `permitbaremultisig=1`：允許中繼 *bare-Multisig* P2MS 交易 (預設值：`1`)。這是在 UTXO 上建立多重簽章條件的最古老腳本範本 (2011 年由 Gavin Andresen 發明)。





- `whitelistrelay=1`：自動授予進入白名單的對等端中繼權限 (預設：`1`)。即使您的節點不是在一般中繼模式，這些對等點的交易也會被中繼接受。





- `whitelistforcerelay=1`：指派「*forcerelay*」權限給具有預設權限 (預設：`0`) 的白名單對等點。如此一來，即使他們已經出現在 Mempool 中，節點也會中繼他們的交易，從而繞過防冗餘機制。





- `whitebind=<[permissions@]addr>` / `whitelist=<[permissions@]CIDR>`：綁定 Interface 或 Address 範圍，並將細粒度權限指派給對應的對等者：relay`、`forcerelay`、`Mempool` (Mempool 內容要求)、`noban`、`download`、`addr`、`bloomfilter`。這對於授予受信任對等體 (例如閘道、區域網路和內部服務) 特權待遇很有用。





- peerbloomfilters=1`：啟用對 Bloom 過濾器 (BIP37) 的支援，將過濾過的區塊/交易提供給瘦客戶端 (預設值：`0`)。警告：這會增加您的資源負載。





- peerblockfilters=1`：將 BIP157 (*Neutrino*) 精簡過濾器提供給對等（預設值：`0`）。





- `blockreconstructionextratxn=<n>`：重建緊湊區塊時保留在記憶體中的額外事務數 (預設：`100`)。提高緊湊同步期間重建的成功率，代價是少許記憶體。



提醒您，所有這些中繼規則對包含在有效區塊中的交易的有效性沒有影響。它們的作用是調整您對中繼的貢獻、保護您的資源，並讓您的節點在受限制的環境中具有可預測性，但絕不允許您拒絕尊重共識規則的區塊。



### 皮夾



您也可以在 `Bitcoin.conf` 檔案中調整您的錢包管理方式。如果您不是直接在 Core 中使用 Wallet，而是使用外部管理軟體，例如 Sparrow 或 Liana，這些參數就不太重要了：





- addresstype=<legacy|P2SH-SegWit|bech32|bech32m>`：定義接收 Wallet 產生的位址格式。





- `changetype=<legacy|P2SH-SegWit|bech32|bech32m>`：強制 Exchange Address 格式 (單一付款上輸入的剩餘部分)。





- `Wallet=<path>`：啟動時載入現有的 Wallet（可重複載入多個錢包）。





- `walletdir=<dir>`：包含錢包的目錄 (預設: `<datadir>/wallets` 如果存在，否則 `<datadir>`)。如果您希望將錢包儲存到專用或加密的磁碟區，這將非常有用。





- `walletbroadcast=1`：自動廣播已載入錢包建立的交易 (預設: `1`)。如果您希望透過其他管道管理廣播，請設定為 `0`。





- `walletrbf=1`：啟用 RBF 選項，在所有交易上發出 RBF 訊號 (預設：`1`)。允許您在交易受阻時稍後增加費用。





- `txconfirmtarget=<n>`：交易的確認目標 (區塊數，預設：`6`)。Wallet 會自動設定交易的費用，以便在此區塊數內確認。





- `paytxfee=<amt>`：應用於 Wallet 交易的固定費率 (BTC/kvB)。一般避免：透過 `txconfirmtarget` 使用自適應估算。





- fallbackfee=<amt>`：當估計器耗盡資料時使用的回退費率 (BTC/kvB)（預設：`0.00`）。將其設定為 0 會完全停用回退。





- `mintxfee=<amt>`：Wallet建立交易的最小門檻（BTC/kvB）（默認：`0.00001`）。Wallet 會拒絕建立低於此臨界值的交易。





- `maxtxfee=<amt>`：Wallet 交易總費用的絕對上限 (預設: `0.10` BTC)。防止異常高的費用不必要地破壞比特幣。





- `avoidpartialspends=1`：按 Address 集群選擇 UTXO，以避免部分花費。





- `spendzeroconfchange=1`：允許未確認的 UTXO Exchange 重複使用，作為新交易的項目（默認：`1`）。





- consolidatefeerate=<amt>`：最大速率 (BTC/kvB)，超過此速率時，Wallet 會避免增加超過整合所需的投入。這允許在低價時擇機合併，並在成本高時降低成本。





- `maxapsfee=<n>`：Wallet 同意支付以啟動「*避免部分花費*」選項的額外費用預算 (BTC，絕對值)。





- `discardfee=<amt>`：費率 (BTC/kvB)，表示您可以容忍將 Exchange 加入費用而丟棄。在此比率下，成本超過其價值三分之一的輸出都會被丟棄。





- `keypool=<n>`：預先產生的 Address 池的大小 (預設值：`1000`)。太小的值會增加還原不完整的風險。





- `disablewallet=1`：在沒有 Wallet 子系統的情況下啟動 Bitcoin core，並停用相關的 RPC。如果節點只用於驗證/釋放，則可減少攻擊面和足跡。



### 儲存、索引和效能



設定檔也允許您調整與機器相關的參數。如果您的資源有限，或相反地，可用容量很大，這可能特別相關：





- `datadir=<dir>`：設定 Bitcoin core 的主要資料目錄。





- `blocksdir=<dir>`：將區塊檔案 (`blocks/blk*.dat`和`blocks/rev*.dat`) 的位置與 `datadir` 解耦。舉例來說，這對於將區塊存檔放置在不同的磁碟區，同時將狀態基礎 (`chainstate/`)保留在速度較快的媒體上很有用。





- `dbcache=<n>`：分配 `<n>` MiB 給區塊索引和 `chainstate` 使用的資料庫快取 (*LevelDB*) (預設值: `450`)。值越大，IBD 和目前驗證的速度越快，代價是較高的 RAM 消耗。





- `prune=<n>`：啟用區塊檔案修剪，並設定磁碟空間目標 (單位為 MiB)（預設：`0` = 停用；`1` = 透過 RPC 手動修剪；`>=550` = 低於目標值自動修剪）。與 `txindex=1` 不相容。節點仍是完全驗證的節點，但無法再提供舊的歷史記錄。如果您的磁碟空間有限，例如在家用電腦上安裝節點時，這個選項特別有用。





- txindex=1`：建立並維護已確認交易的全局索引。對於某些查詢 (`getrawtransaction`非 Wallet) 和探索目的而言是必要的，但會大幅增加磁碟佔用空間。與 pruned 模式不相容。





- `assumevalid=<hex>`：表示假定為有效的區塊，允許您跳過對其祖先的腳本檢查 (設定 `0` 可檢查所有內容)。更多資訊請參閱前一章。





- `reindex=1`：從磁碟上的 `blk*.dat` 檔案重建區塊索引和狀態 (`chainstate`)。也會重建可選的作用中索引。這是一個耗時的操作，用來修復損毀的資料庫或乾淨地啟動/停用重索引。





- `reindex-chainstate=1`：只重建目前區塊索引的 `chainstate`。區塊檔案健康時的首選。





- `blockfilterindex=<type>`：維護精簡用戶端 (BIP157/158) 和某些 RPC 使用的精簡區塊篩選器 (例如，`basic`) 的索引。預設為停用 (`0`)。消耗額外的磁碟空間和索引時間。





- `coinstatsindex=1`：維護由 `gettxoutsetinfo` 呼叫操作的 UTXO 設定統計索引。對於稽核和度量非常有用，省去了重新計算的昂貴費用。預設為停用。





- `loadblock=<file>`：啟動時從外部的 `blk*.dat` 檔案匯入區塊。用來從離線來源 (本機副本、外部媒體) 預載歷史，以加快初始化。





- `par=<n>`：設定腳本驗證線程數 (從 `-10` 到 `15`, `0` = 自動, `<0` = 保留此核心數)。允許您在驗證期間調整 CPU 的平行度。自動模式適用於大多數情況。





- `debuglogfile=<file>`：指定 `debug.log` 日誌的位置。





- `shrinkdebugfile=1`：在啟動時減少 `debug.log` 的大小 (當 `debug` 未啟動時，預設值：`1`)。





- `settings=<檔案>`：動態設定檔案 `settings.json` 的路徑。



### RPC 存取與操作安全



最後，`Bitcoin.conf` 檔案也允許您設定節點的存取參數。請謹慎使用這些設定，尤其是當您剛開始使用時：在沒有透徹了解其影響之前，請避免變更這些設定，因為這可能會引入漏洞。





- `server=1`：啟動 JSON-RPC 伺服器。如果您透過 `bitcoin-cli` 或第三方應用程式來驅動 `bitcoind`，這是必要的。停用 (`0`)於不揭露任何 API 或已使用 Electrum 伺服器的純驗證節點。





- `rpcbind=<addr>[:port]`：RPC 伺服器監聽 Address/port。預設情況下，只在本機進行監聽 (`127.0.0.1`和`::1`)。如果未同時定義 `rpcallowip` 則會忽略此參數。使用它來明確限制 Interface。





- `rpcport=<port>`：RPC 連接埠（預設：Mainnet `8332`、Testnet `18332`、bookmark `38332`、regtest `18443`）。





- `rpcallowip=<ip|cidr>`：允許來自指定 IP 或子網路 (可以重複) 的 RPC 用戶端。與 `rpcbind` 搭配使用，可讓 API 僅暴露給受信任的區段 (LAN/VPN)。





- `rpcauth=<USERNAME>:<SALT>$<Hash>`：建議的 RPC 驗證方法 (散列密碼)。允許多重輸入，並避免以明確文字儲存秘密。





- `rpccookiefile=<path>`：認證 cookie 的路徑 (預設：`datadir/` 下的 `.cookie` 檔案)。這用於同一使用者的本機存取，而不需管理永久密碼。例如，您可以使用它將 Liana Wallet 連接到同一台機器上的 Bitcoin core。





- `rpcuser=<user>` / `rpcpassword=<pw>`：使用明文密碼的經典 RPC 認證。避免使用此功能，改用 `rpcauth` 或 cookie。





- `rpcthreads=<n>`：服務 RPC 呼叫的線程數 (預設：`4`)。如果您在監控/外部工具端有較高的呼叫峰值，請增加它。





- `rpcwhitelist=<USERNAME>:<rpc1>,<rpc2>,...`：授權 API 的白名單。透過限制可存取的方法來減少攻擊面。





- `rpcwhitelistdefault=1|0`：預設白名單行為：如果啟用且使用白名單，則會拒絕未列出的呼叫。這也可以強制預設空集 (不允許任何呼叫)，只要沒有任何東西被明確列出。





- `rest=1`：啟用公開 REST API (預設為停用)。僅在可信賴的網路中公開 (與 JSON-RPC 的注意事項相同)。





- `conf=<file>`：僅在命令列上指定唯讀設定檔。有助於在作業端凍結執行設定檔（不可變）。





- `includeconf=<file>`：載入額外的設定檔案 (路徑相對於 `datadir/`)。允許角色分離：共同基礎 + 敏感的本地重載。





- `daemon=1` / `daemonwait=1`：在背景中啟動 `bitcoind`，並配合 `daemonwait` 等待初始化完成後才交出。這有助於與監督程式 (systemd, runit) 整合。





- `pid=<file>`：PID 檔案的位置。





- `sandbox=<log-and-abort|abort>`：啟用實驗性的 syscalls 沙箱：只允許使用預期的 syscalls。





- `startupnotify=<cmd>` / `shutdownnotify=<cmd>`：在啟動或關機時執行指令。





- `alertnotify=<cmd>`：收到警示時觸發指令。





- `blocknotify=<cmd>`：為每個新區塊執行指令。





- `debug=<category>|1` / `debugexclude=<category>`：啟用/停用詳細的日誌類別 (例如 `net`、`Mempool`、`RPC`、`validation`...)。





- `logips=1`：記錄 IP 位址。





- `logsourcelocations=1` / `logthreadnames=1` / `logtimestamps=1`：分別在日誌中加入來源位置、線程名稱和精確的時間戳記。





- `printtoconsole=1`：將追蹤/debugs 傳送至控制台 (*stdout*)。





- `help-debug=1`：顯示 debug 選項說明，並退出。





- `uacomment=<cmt>`：在 User-Agent P2P 中加入註解。



我們現在已經列出大部分的組態參數。這個 `Bitcoin.conf` 檔案因此構成您節點真正的儀表板：它定義網路組態、Mempool 管理、磁碟和記憶體使用、索引和一般管理。如果您想瞭解更多關於這個檔案的資訊，並建立一個符合您需求的檔案，我建議您使用 [Jameson Lopp 的產生器](https://jlopp.github.io/Bitcoin-core-config-generator/)。
